# AI Agent 学习路径与知识图谱

## 学习定位

AI Agent 这几年很热，但它特别容易被学偏。  
很多人一上来就沉迷框架名词，比如 ReAct、Tool Calling、Planner、Multi-Agent、Memory、Browser Agent，看起来很丰富，结果学完之后还是说不清三件最关键的事：

- AI Agent 和普通的大模型问答到底差在哪
- 什么问题适合做 Agent，什么问题根本不该做
- 真正落地时，系统边界、状态管理、工具权限、成本控制、可观测性应该怎么设计

所以这条路线不会从框架 API 开始，而是先建立一个牢固的工程视角：

**AI Agent 本质上不是“会聊天的模型”，而是“围绕目标，在环境里观察、决策、行动，并根据反馈持续推进任务的系统”。**

你有后端开发经验，这会是很大的优势。  
因为学 AI Agent 时，很多关键问题其实和你熟悉的后端系统设计强相关：

- 如何拆职责
- 如何管理状态
- 如何封装外部能力
- 如何控制副作用
- 如何做重试、超时、权限、审计和回滚
- 如何评估系统到底是真的有效，还是只是 demo 看起来聪明

学完这条路线后，你应该逐步具备这些能力：

- 能准确区分普通 LLM 应用、工作流系统、RAG 系统和 Agent 系统。
- 能用“目标 - 状态 - 工具 - 决策 - 反馈闭环”的视角分析一个 Agent。
- 能设计单 Agent 的基本架构，包括上下文、工具调用、状态存储、记忆和终止条件。
- 能判断什么时候用固定工作流更合适，什么时候才值得引入 Agent 自主决策。
- 能理解多 Agent 协作、长期记忆、规划执行、评测与安全控制的真实价值与局限。
- 能从后端工程角度实现一个可控、可观测、可逐步迭代的 Agent 系统。

## 知识图谱（从基础到进阶）

1. 先理解 AI Agent 到底是什么
   - Agent 与普通 Prompt 调用的区别
   - Agent 与 Workflow 的区别
   - Agent 的最小闭环：目标、观察、决策、行动、反馈
2. 工具使用：让模型不只是“回答”，还能“做事”
   - Function Calling / Tool Calling
   - 工具描述为什么决定可用性
   - 工具选择、参数生成、执行结果回灌
3. 状态与记忆：Agent 为什么不能只靠一次对话上下文
   - 短期上下文
   - 结构化状态
   - 长期记忆
   - 任务历史与审计日志
4. 推理与执行循环
   - ReAct 思路
   - Plan-and-Execute
   - 单轮决策与多轮迭代
   - 终止条件、失败恢复与回退策略
5. Workflow 与 Agent 的边界
   - 固定流程的优势
   - 什么时候需要把“下一步怎么做”交给模型
   - 人工审批、规则引擎与 Agent 的协作边界
6. RAG 与 Agent 的关系
   - 检索增强是不是 Agent
   - Agent 如何利用知识库
   - 检索、推理、执行之间如何协同
7. 工程落地
   - Agent 服务的模块划分
   - 工具注册中心
   - 会话状态与任务状态存储
   - 幂等、超时、重试、权限控制、审计
8. 评测与迭代
   - 如何定义任务成功
   - 如何评估工具选择是否正确
   - 如何定位“看起来会，其实不稳定”的问题
   - 离线评测、线上观测与回放
9. 多 Agent 协作
   - 为什么会想到多 Agent
   - 多 Agent 的收益与额外复杂度
   - 协作协议、上下文共享、任务拆分
10. 安全与边界控制
   - Prompt Injection
   - 工具越权
   - 敏感数据泄露
   - 成本失控与死循环
11. 实战搭建
   - 从零做一个“研发助理”或“工单处理助手”
   - 把需求拆成：目标、工具、状态、执行循环、评测、护栏
12. 进阶视角
   - Agent 平台化
   - 记忆系统设计
   - 长任务编排
   - 人机协作与审批流整合

## 建议学习顺序背后的原因

很多教程会直接从某个框架开始，比如 LangGraph、AutoGen、CrewAI、OpenAI Agents SDK。  
这种学法容易出现一个问题：你学会了“怎么拼装组件”，但没有先建立判断标准，于是后面会出现这些典型误区：

- 明明一个固定工作流就能解决的问题，硬要做成自主 Agent
- 明明需要结构化状态，却只把所有东西塞进对话上下文
- 明明真正难点是权限控制和异常恢复，却把精力都花在 prompt 润色上
- 明明单 Agent 都没稳定，却上来就做多 Agent

所以这条路线会严格按下面的顺序推进：

1. 先搞懂 Agent 的本质和边界
2. 再理解工具、状态、记忆这些基础构件
3. 然后进入执行循环和系统架构
4. 最后讨论评测、安全、多 Agent 与实战

这样学，你最后拿到的不只是概念表，而是一套工程判断力。

## 学习方法

- 每一课都先回答“这个东西在解决什么问题”，再进入术语和实现。
- 会优先用后端工程例子，比如工单处理、告警排查、知识检索、配置变更建议、研发助手，帮助你把抽象概念落到真实系统。
- 每课结尾都会有检查站。你答完后，我会根据你的掌握程度决定是推进、纠偏还是补充一课。
- 目标不是让你背定义，而是让你能自己设计一个简单但靠谱的 Agent 系统。

## 推进规则

- 完全掌握：进入下一课，并逐步增加工程实现和架构权衡。
- 基本掌握：简短纠正后进入下一课。
- 部分理解：生成补充课，例如 `01b_Agent与Workflow补充.md`。
- 尚未理解：先退回更基础的问题，用更直白的例子重建概念，不跳过地基。
