# AI Agent - 第 8 课：评测与可观测性：怎么判断 Agent 真的有用

## 学习目标

- 理解 Agent 为什么不能只靠“看起来挺聪明”来评估。
- 区分离线评测、线上观测、人工复核三类方法各自的作用。
- 知道评测 Agent 时不能只看最终答案，还要看过程质量。
- 建立一套适合工程落地的指标意识：成功率、工具命中率、时延、成本、稳定性。
- 明白可观测性不是附属功能，而是 Agent 能不能迭代的基础设施。

## 内容讲解

### 1. Agent 最容易骗人的地方是什么

就是它特别容易让人产生一种错觉：

“它刚刚做对了几次，所以应该已经可以用了。”

但 Agent 系统和普通功能模块不一样，它的不稳定性往往体现在：

- 同样的问题，不同时间表现不同
- 一条链路里某一步偶尔出错
- 看上去回答合理，但工具其实调错了
- 最终结果对了，但过程不可复用

所以评估 Agent 不能只靠演示成功案例。  
真正的问题是：

**它在足够多真实任务上，是否稳定、可控、成本可接受。**

### 2. 评测 Agent，不能只看“最后答得对不对”

因为 Agent 不是纯问答系统。  
它经常包含：

- 工具选择
- 参数构造
- 多步执行
- 中途状态切换
- 终止判断

所以只看最终答案，信息是不够的。

举个例子：

某个 Agent 最终确实答对了，但它中途：

- 调错了 2 次工具
- 多绕了 4 步
- 成本比正常情况高 5 倍

这说明它“能做出来”，但还远远谈不上“好用”。

### 3. 一个更完整的评测视角

评测一个 Agent，至少要看三层。

#### 3.1 结果层

它最后有没有完成任务。

例如：

- 是否找到正确答案
- 是否生成了符合要求的报告
- 是否真的完成了工单创建

#### 3.2 过程层

它是怎么完成的。

例如：

- 选的工具对不对
- 步数是否合理
- 是否有无效重试
- 是否在关键时刻停下来了

#### 3.3 资源层

它完成任务花了多大代价。

例如：

- 总耗时
- token 成本
- 工具调用次数
- 外部 API 成本

如果只盯结果层，很容易把一个“很贵、很慢、偶尔还飘”的系统误判成有效。

### 4. 离线评测在解决什么问题

离线评测更像做题集。

它适合回答的问题是：

- 这个 Agent 在一组已知任务上表现大概如何
- A 版本和 B 版本谁更好
- 某个 prompt、某个工具描述是否真的优化了效果

离线评测的好处是可重复、可对比。  
你可以固定一批样本，反复测试不同版本。

但它也有明显边界：

- 测不出真实流量噪声
- 测不出用户真实输入的变化
- 测不出线上依赖抖动

所以离线评测很重要，但它不是全部。

### 5. 线上观测在解决什么问题

线上观测更像看生产系统指标。

它回答的是：

- 真正用户在用的时候，成功率怎么样
- 哪类任务经常失败
- 成本有没有失控
- 哪个工具最不稳定
- 哪一步最容易成为瓶颈

很多 Agent 团队后面会发现：

离线看起来不错，线上却问题不断。  
原因通常不是评测没做，而是只做了离线，没有把线上观测建起来。

### 6. 一套比较实用的核心指标

如果你要做一个 Agent 服务，比较常见的指标大概有这些。

#### 6.1 任务成功率

最基础的指标。  
但要注意先定义清楚“成功”是什么。

比如：

- 找到正确答案算成功
- 创建工单成功才算成功
- 只是给出建议，不算真正完成动作

#### 6.2 工具调用正确率

看模型是否选对了工具、参数是否合理。

因为很多 Agent 的失败，不是最后回答差，而是中途工具用错。

#### 6.3 平均步数和重试率

如果步数越来越多、重试越来越频繁，说明系统可能在绕路。

#### 6.4 平均时延和尾时延

Agent 系统一定不能只看平均耗时。  
尾部慢请求更容易影响真实体验。

#### 6.5 单任务成本

包括：

- token 消耗
- 外部工具成本
- 搜索和数据库调用成本

#### 6.6 人工接管率

如果一个 Agent 常常需要人工补救，那它的真实自动化价值就要重新评估。

### 7. 为什么 Trace 和回放特别重要

Agent 的问题很难只靠日志关键词排查。  
因为它失败往往不是某个接口 500，而是：

- 这一步为什么选了这个工具
- 工具返回后它为什么理解错了
- 为什么本来该停却继续跑了

这时候如果没有完整 trace，你几乎没法复盘。

一个比较理想的 trace 至少要包含：

- 本轮输入
- 装配后的上下文
- 模型的每次决策
- 每次工具调用及结果
- 中间状态变化
- 最终输出

有了它，团队才可能真正调优，而不是靠猜。

### 8. 人工评审为什么仍然必要

很多任务很难只用机器自动打分。

比如：

- 报告是否有说服力
- 回复是否符合组织风格
- 方案是否真正可执行

这些时候，人工评审仍然很有价值。

所以成熟的评测体系往往不是单一方法，而是组合：

- 离线自动评测
- 线上指标观测
- 抽样人工评审

三者一起，才比较接近真实质量。

### 9. 评测不只是“验收”，更是“迭代方向盘”

很多团队把评测理解成最后上线前打一遍分。  
其实更好的理解是：

**评测体系决定你知道该优化什么。**

比如：

- 成功率低 -> 看看是不是工具选择问题
- 成功率高但成本高 -> 优化上下文和步数
- 某类任务总失败 -> 针对这类任务补样本、补工具、补规则

没有评测，调优就会退化成凭感觉。

### 10. 一个很现实的建议：先建立粗指标，再逐步细化

一上来想做完美评测体系，通常容易拖很久。  
更实际的路线是：

第一阶段先有这些基础能力：

- 成功 / 失败统计
- 平均时延
- 单任务成本
- 工具错误统计
- 基础 trace

等系统有了真实使用，再逐步补：

- 分类任务评测
- 线上回放集
- 人工标注样本
- 版本对比实验

这条路线更符合真实团队节奏。

## 小结

这一课最重要的认识是：

**Agent 不能只靠演示来判断价值，必须靠评测和可观测性来判断稳定性、收益和边界。**

真正靠谱的 Agent 团队，不是“会做 demo”，而是知道：

- 什么叫成功
- 为什么失败
- 成本花在哪
- 下一步该优化什么

## 问题

3. 如果一个 Agent 结果正确，但过程绕路很多、成本很高，你会把它判断成“成功”吗？为什么？
4. 为什么说 Trace 和回放能力，是 Agent 系统迭代的基础设施？
