# AI Agent - 第 9 课：上下文工程：Context Engineering、GSSC、笔记与长任务

## 学习目标

- 理解为什么 Prompt Engineering 不足以支撑复杂 Agent，为什么要引入 Context Engineering。
- 搞清楚“上下文”不是一句 system prompt，而是模型当前能看到的全部状态。
- 建立 GSSC 这条主线：Gather、Select、Structure、Compress。
- 认识上下文腐蚀、注意力预算、长任务断裂这些问题为什么会成为 Agent 的真实瓶颈。
- 能为一个长时程 Agent 设计一套基本的上下文装配、笔记和压缩策略。

## 内容讲解

### 1. 什么叫上下文工程

如果说提示工程关注的是：

**“提示词该怎么写，模型才更听话？”**

那上下文工程关注的是：

**“在每次调用模型之前，到底该把哪些信息放进它的可见世界里？”**

这两者不是对立关系，而是包含关系。

在真实 Agent 系统中，模型看到的并不只是 prompt，还包括：

- system prompt
- 工具定义
- 最近对话
- 当前任务状态
- 检索到的资料
- 长期记忆
- 中间笔记
- 输出格式约束

所以“上下文”更像一个完整输入状态，而不是一段文案。

### 2. 为什么 Agent 越复杂，越需要上下文工程

因为 Agent 的复杂度不是线性长出来的，而是乘法长出来的。

当系统同时具备下面这些能力时：

- 会话历史
- 工具调用
- Memory
- RAG
- 长任务执行
- 多轮思考

你会很快碰到一个现实问题：

**模型能看的 token 是有限的，但候选信息几乎无限。**

这时最关键的能力就不是“还能不能多塞一点”，而是：

**如何筛出真正值得被看到的那一小部分。**

### 3. 上下文腐蚀到底是什么

很多人以为上下文窗口大了，问题就自动解决了。  
但实践里常见的情况恰恰相反：

- 上下文越长，噪声越多
- 信息越多，注意力越分散
- 关键约束被埋进长文本里
- 模型开始“看起来都看了，其实没抓住重点”

这就是为什么很多长任务系统不是死在“窗口不够大”，而是死在“上下文太脏”。

可以把上下文窗口理解成一种**注意力预算**：

- 不是能装多少字的问题
- 而是模型能在多大程度上持续抓住真正重要的信号

### 4. 一个好上下文，通常至少包含六类东西

一个比较稳的上下文骨架，通常会有这些部分：

1. **角色与规则**
   - 你是谁、你能做什么、不能做什么
2. **当前任务**
   - 这轮到底要解决什么问题
3. **状态摘要**
   - 已经做过什么、现在到哪一步了
4. **证据材料**
   - 检索结果、工具结果、结构化数据
5. **辅助笔记**
   - 待办、假设、风险、未决问题
6. **输出约束**
   - 返回格式、语言、风格、必须覆盖的点

如果缺了这些中的关键部分，Agent 往往会出现：

- 方向漂移
- 重复劳动
- 忽略历史决策
- 说得对但格式不对
- 工具结果用不上

### 5. GSSC 是最容易落地的一条主线

一个很实用的上下文工程心智模型是 GSSC：

- **Gather**：先把候选信息尽量收集上来
- **Select**：从候选信息里选最值得保留的
- **Structure**：把信息摆成模型容易读的形状
- **Compress**：超预算时做压缩，不是直接硬砍

可以画成这样：

```mermaid
flowchart LR
    A["候选信息宇宙"] --> B["Gather 收集"]
    B --> C["Select 筛选"]
    C --> D["Structure 结构化"]
    D --> E["Compress 压缩"]
    E --> F["最终上下文"]
```

### 6. Gather：先别急着拼 prompt，先定义候选信息池

Gather 的核心不是“多拿”，而是“把来源明确”。

候选上下文通常来自：

- 当前用户输入
- 最近几轮历史
- 当前任务状态
- 长期记忆召回
- RAG 检索结果
- 工具最近输出
- 系统配置和护栏

如果这些来源没有被清楚分层，后面几步就会越来越乱。

### 7. Select：不是相关就一定该进上下文

筛选时至少应该看四个维度：

1. **相关性**
2. **新近性**
3. **可信度**
4. **成本**

举个例子，同样是“和当前任务有关”的信息：

- 昨天刚写入的待办事项，可能优先级高
- 半年前的旧结论，即使相关，也可能已过期
- 一个模型自己猜出来的中间假设，可信度可能低
- 一大段原始日志虽然相关，但 token 成本太高

所以 Select 从来不是单纯做相似度检索，而是做**价值排序**。

### 8. Structure：模型不是数据库，别把信息原封不动扔进去

很多系统效果差，不是因为没查到信息，而是因为查到了之后摆得太差。

模型更容易读懂的信息形式通常是：

- 标题 + 摘要
- 分区块展示
- 显式标签
- 结构化字段
- 有层级的 bullet

模型更容易被拖垮的形式通常是：

- 大段原始日志
- 无标记的长文拼接
- 多来源材料混成一坨
- 不区分“事实”“假设”“要求”“结果”

所以 Structure 其实是在做一件事：

**把“机器存储友好”的内容，转换成“模型阅读友好”的内容。**

### 9. Compress：压缩不是删，而是重写信息密度

很多人一看到超 token，就直接截断。  
这种方式最容易把关键约束截没。

更稳的做法通常是：

- 删掉重复内容
- 把长工具输出摘要化
- 把早期历史压成阶段总结
- 只保留最近的高价值明细

压缩的目标不是“更短”，而是：

**在更少 token 里保住决策所需的信息密度。**

### 10. 长任务为什么特别依赖笔记系统

长任务不是“把对话拉长”这么简单。

一旦任务跨多个小时、多次调用、多轮工具执行，就会遇到：

- 早期背景信息被挤出窗口
- 已做决策无法稳定继承
- 下一轮不知道上一轮为什么这么做
- 任务中途失败后很难恢复

这时就需要外化笔记。

笔记系统最重要的不是“记录一切”，而是记录这几类高价值信息：

- 当前目标
- 已完成里程碑
- 关键证据
- 未决问题
- 下一个动作
- 风险和假设

### 11. 为什么长任务里的笔记比长聊天更重要

因为聊天历史是按时间排序的，笔记是按任务价值组织的。

聊天历史的问题是：

- 冗余多
- 可检索性差
- 信息重要性不均匀
- 不适合快速恢复任务状态

笔记则更像是“任务控制面板”。

比如一个代码迁移 Agent，笔记里最值得保留的不是完整过程，而是：

- 已迁移哪些模块
- 哪些接口仍未适配
- 已发现哪些兼容性风险
- 下一次要从哪里继续

### 12. 长任务的一个核心策略：阶段性压缩 + 可恢复检查点

如果任务明显会跑很久，一个实用模式是：

1. 每完成一个阶段，生成阶段摘要
2. 把阶段摘要写入外部存储
3. 为任务建立 checkpoint
4. 后续新窗口只带“阶段摘要 + 当前局部明细”

这比死扛全量上下文稳定得多。

### 13. 上下文工程不是越智能越好，而是越可控越好

很多系统失败，不是因为检索能力不够，而是因为上下文装配不可预测。

常见坏味道包括：

- 什么信息进上下文完全靠拍脑袋
- 每次调用装进去的东西都不稳定
- 查到了证据，却没有标明来源和可信度
- 每一轮都把全部历史再塞一遍
- 过度相信模型会自己从噪声中抓重点

所以成熟系统会把上下文构建当成一个显式模块，而不是 prompt 之前的小脚本。

### 14. 一个更深的区别：Prompt 是语言层优化，Context 是状态层优化

这句话特别关键。

很多人做了半年 Agent，最后还在“改 prompt”。

但真正难的问题往往不是：

- 这句话怎么说更优雅

而是：

- 这轮到底该给模型看哪些信息
- 哪些信息不该给
- 哪些信息应该先压缩再给
- 哪些信息应该延迟到工具调用后再给

Prompt engineering 更像“措辞优化”。  
Context engineering 更像“状态分配”。

### 15. 上下文工程和工具工程、记忆工程是互相咬合的

一个很容易忽略的事实是：

上下文工程做得好不好，不只取决于 prompt。

它还取决于：

- 工具返回结果是不是 token 友好
- 记忆检索结果是不是干净
- RAG 的 chunk 和召回是不是合理
- 状态是不是有结构化表示

所以很多团队最后会发现：

**上下文工程其实是 Agent 系统的中控层。**

它把 prompt、工具、memory、RAG、任务状态都串起来。

### 16. 常见误区

- 误区一：把 prompt engineering 当成 context engineering
- 误区二：上下文越全越好
- 误区三：只做召回，不做结构化
- 误区四：长任务靠全量历史硬撑
- 误区五：没有笔记和 checkpoint，却想做长期自主任务

## 一句话总结

**上下文工程的本质不是“把更多信息塞给模型”，而是“在有限注意力预算里，把最值得看的状态以最容易读懂的形状交给模型”。**
