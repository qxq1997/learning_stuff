# AI Agent - 第 10 课：多 Agent 协作：什么时候值得，什么时候是复杂度陷阱

## 学习目标

- 理解多 Agent 不是“越多越强”，而是一种带来额外复杂度的系统设计选择。
- 知道多 Agent 常见的几种协作模式：经理-执行者、专家分工、辩论、接力交接。
- 能判断一个问题到底值不值得拆成多 Agent。
- 认识到协议、消息格式、共享状态、冲突处理为什么会突然变难。
- 建立“单 Agent 先跑稳，多 Agent 后考虑”的工程判断力。

## 内容讲解

### 1. 为什么大家一学 Agent 就容易想到多 Agent

因为它很符合人的直觉。

人类组织里，复杂任务经常靠多人协作完成：

- 一个负责规划
- 一个负责检索
- 一个负责写报告
- 一个负责审核

所以大家自然会想：

“那我是不是也该弄几个 Agent 分工合作？”

这个想法不算错，但有一个前提：

**拆开之后，收益真的要大于协调成本。**

### 2. 多 Agent 真正带来的收益是什么

多 Agent 不是为了显得高级，它真正可能带来的收益通常有三种。

#### 2.1 角色分工更清晰

不同 Agent 只负责自己擅长的事。

例如：

- 研究员负责搜资料
- 写作者负责整合报告
- 审核员负责找问题

#### 2.2 上下文更聚焦

一个大 Agent 什么都做，容易上下文过载。  
拆成多个 Agent 后，每个 Agent 只看自己关心的信息，反而更稳定。

#### 2.3 更接近组织流程

如果你的真实业务本来就是分岗位协作，那多 Agent 有时更贴近业务结构。

### 3. 多 Agent 带来的代价是什么

这部分经常被低估。

一旦你拆成多个 Agent，系统马上会多出这些问题：

- 谁来分配任务
- 谁来判断什么时候交接
- 上下文怎么传
- 状态怎么共享
- 两个 Agent 结论冲突怎么办
- 成本怎么算
- trace 怎么看

你会发现，原来一个 Agent 的复杂度，突然变成了“Agent 之间的协作复杂度”。

所以多 Agent 的本质不是能力叠加，而是：

**把单体复杂度换成协作复杂度。**

### 4. 常见的几种协作模式

#### 4.1 经理 - 执行者模式

一个 Agent 负责拆任务和调度，其他 Agent 负责具体执行。

这很像项目经理带几个专员。

优点是结构清晰。  
缺点是经理 Agent 容易变成瓶颈。

#### 4.2 专家分工模式

每个 Agent 负责一个专业领域。

比如：

- 法务 Agent
- 技术 Agent
- 销售 Agent

适合领域边界清晰的任务。

#### 4.3 接力交接模式

前一个 Agent 的输出是后一个 Agent 的输入。

比如：

- 检索 Agent 找资料
- 总结 Agent 提炼观点
- 报告 Agent 写成文稿

这种模式最接近流水线。

#### 4.4 辩论 / 评审模式

一个 Agent 产出方案，另一个 Agent 负责批评和修正。

它适合需要审校的任务，但也容易增加成本和时延。

### 5. 什么场景值得用多 Agent

经验上，更适合多 Agent 的任务通常有这些特征：

- 明显存在可分离的子专业
- 单个 Agent 上下文已经过载
- 任务链条较长，且阶段边界明确
- 每一阶段的输入输出可以结构化定义

比如：

- 深度研究
- 多角色内容生产
- 复杂咨询式任务
- 与真实组织结构高度一致的流程

### 6. 什么场景不值得用多 Agent

如果是这些情况，往往先别拆：

- 任务本身并不复杂
- 单 Agent 已经能稳定完成
- 拆完之后没有明确边界
- 你连单 Agent 的 trace 都还没做好

一句很实用的话是：

**如果你还说不清为什么一个 Agent 不够，那大概率先不要上多 Agent。**

### 7. 多 Agent 最难的其实不是“多”，而是“协作协议”

Agent 之间怎么交流，是多 Agent 成败的关键。

它至少要回答：

- 传什么格式
- 谁对消息负责
- 是否共享同一份状态
- 如何标记任务完成或失败

这也是为什么业界会出现一堆协议和框架思路，比如：

- 工具协议
- Agent 间通信协议
- 服务发现
- 任务交接约定

你不一定要背这些名字，但要理解背后问题：

**一旦从单体 Agent 变成网络化协作体，协议和消息模型就会变得重要。**

### 8. 共享状态是多 Agent 的真正难点之一

单 Agent 时，状态基本在自己手里。  
多 Agent 时就会出现：

- 谁能看哪些状态
- 谁能改哪些状态
- 谁写的是最终结论
- 如何避免互相覆盖

如果这层设计不好，就会出现非常混乱的情况：

- A 认为任务完成了
- B 觉得还缺信息
- C 还在根据旧状态继续跑

所以多 Agent 系统比单 Agent 更依赖显式状态管理，而不是只靠自然语言传递。

### 9. 一个现实建议：先把“多角色”做成“单 Agent + 多工具/多模式”

很多团队其实并不是真的需要多 Agent。  
他们需要的只是：

- 一个会检索的模式
- 一个会总结的模式
- 一个会审查的模式

这种时候，不一定非要拆成三个 Agent。  
完全可以先做成：

- 一个单 Agent
- 带几个明确模式
- 或几个清晰的子任务节点

这样更容易控，也更容易调。

### 10. 一个成熟的判断框架

你可以这样判断是否要上多 Agent：

#### 10.1 单 Agent 是否已经被上下文压垮

如果没有，先别拆。

#### 10.2 子任务边界是否明确

如果边界模糊，拆开只会互相打架。

#### 10.3 协作成本是否可接受

包括：

- 更多 token
- 更多请求
- 更多 trace
- 更多状态同步

#### 10.4 你是否真的有能力调试它

如果现在连单 Agent 的失败原因都看不清，多 Agent 只会更难。

## 小结

这一课最核心的结论是：

**多 Agent 不是“更高级的默认答案”，而是一种只有在分工收益大于协调成本时才值得采用的架构选择。**

单 Agent 先跑稳，是更健康的路线。  
当你发现单 Agent 确实出现了上下文过载、角色冲突或任务链太长的问题，再去拆多 Agent，通常才比较合理。
