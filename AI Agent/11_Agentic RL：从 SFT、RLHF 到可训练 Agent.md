# AI Agent - 第 11 课：Agentic RL：从 SFT、RLHF 到可训练 Agent

## 学习目标

- 理解为什么单靠提示工程和工作流拼装，并不能无限提升 Agent 能力。
- 建立一张从预训练、SFT、偏好对齐到 Agentic RL 的完整训练地图。
- 把“可训练 Agent”理解成一个序贯决策问题，而不是单轮问答优化问题。
- 理解 Agentic RL 的核心难点：长时程、延迟奖励、工具噪声、奖励作弊、环境漂移。
- 能判断哪些场景值得考虑 Agentic RL，哪些场景完全不值得碰。

## 内容讲解

### 1. 为什么我们会开始讨论 Agentic RL

当 Agent 还比较简单时，我们主要靠这些手段提升效果：

- 改 prompt
- 补 few-shot
- 换更强模型
- 增加工具
- 改工作流

这些方法都很重要，而且大多数团队其实还远没用到上限。

但当任务开始变成：

- 长时程
- 多步规划
- 复杂工具使用
- 需要试错和恢复
- 需要在环境中持续行动

你就会发现一个问题：

**我们现在优化的，大多还是“说得更好”，不是“做得更好”。**

Agentic RL 想解决的，就是这个落差。

### 2. 先把大模型训练的全景图补齐

很多人一听到 RL，就会直接跳到“强化学习训练 Agent”。  
但中间其实隔了不少层。

一个大模型从“会补全文本”到“像助手一样工作”，大致会经过这些阶段：

1. **预训练**
   - 学语言规律、世界知识、基础推理先验
2. **SFT**
   - 学会按指令说话，模仿人类示范
3. **偏好对齐**
   - 用 RLHF、RLAIF、DPO 一类方法让输出更符合偏好
4. **Agentic RL**
   - 不再只优化单轮回答，而是优化多步任务表现

这条链路里最关键的变化是：

- 前三步更像在优化“输出质量”
- 最后一步更像在优化“任务行为”

### 3. SFT 到底给了 Agent 什么

SFT 的价值不是让模型更聪明，而是让它：

- 学会遵循指令
- 学会某种输出风格
- 学会某类问题的常见解法
- 学会基本的工具调用格式

所以 SFT 更像给 Agent 打一个行为底座。

没有这个底座，后面的 RL 往往不稳定。  
因为如果模型连基本动作都不会，强化学习就会变成在噪声里乱试。

### 4. RLHF、RLAIF、DPO 这些方法和 Agentic RL 的关系

这几类方法的共同点，是都在做“对齐”。

它们大多在解决：

- 哪种回答更像人喜欢的回答
- 哪种输出更安全
- 哪种表达更有帮助

但它们通常还是偏向单轮样本。

也就是说，它们优化的是：

**给定输入，输出一个更好的回答。**

而 Agentic RL 更关注的是：

**给定目标，在一段连续行动中，最后能不能把事做成。**

### 5. Agentic RL 的本质：把 Agent 看成一个策略

当我们把 Agent 放进 RL 语境里，可以这样理解：

- **状态（state）**：当前上下文、环境反馈、工具结果、任务进度
- **动作（action）**：生成文本、调用工具、选择下一步、结束任务
- **奖励（reward）**：任务是否成功，中间步骤是否合理，成本是否可接受
- **轨迹（trajectory）**：从开始到结束的一连串状态和动作

这时 Agent 就不再只是“生成器”，而更像一个策略函数：

**在当前状态下，决定下一步动作。**

### 6. 为什么 Agentic RL 比普通 LLM 对齐难很多

因为它从“单步预测”变成了“长期决策”。

这里会同时冒出几个难题：

1. **奖励延迟**
   - 最终任务成败，可能要很多步之后才知道
2. **奖励稀疏**
   - 中间步骤往往没有明确标签
3. **环境噪声**
   - 工具返回、网页变化、系统波动都会影响结果
4. **动作空间复杂**
   - 不只是吐 token，还可能要调工具、做规划、终止任务
5. **信用分配困难**
   - 最后成了，到底是哪一步做对了？

这也是为什么训练一个“真正能干活的 Agent”远比训练一个“会说话的模型”更难。

### 7. Agentic RL 最适合什么样的任务

它比较适合这类任务：

- 可验证
- 有明确目标
- 能自动判分
- 多步完成
- 不只是靠语言风格取胜

比如：

- 数学题求解
- 工具调用任务
- 代码修复与测试通过
- 浏览器操作任务
- 游戏 / 仿真环境任务

这些任务的共同点是：

**你比较容易定义“做成了没”。**

这非常关键，因为没有稳定 reward，RL 就会变成瞎练。

### 8. 奖励函数为什么是 Agentic RL 的命门

很多人会把重点放在算法名词上，但实际最致命的通常是 reward 设计。

如果 reward 定错了，模型会学到非常奇怪的行为。

这叫 reward hacking，常见表现包括：

- 学会投机取巧而不是真的完成任务
- 用奇怪格式骗过评估器
- 为了得分而重复无意义步骤
- 选择成本极高但高分的策略

所以一个好的 reward 往往不是单一指标，而是组合指标。

比如一个工具使用 Agent，reward 可能同时看：

- 最终任务是否成功
- 工具是否选对
- 参数是否正确
- 步数是否合理
- 成本是否过高
- 是否触发安全违规

### 9. 训练数据在 Agentic RL 里依然重要

哪怕用了 RL，也不代表可以完全摆脱数据。

通常还是需要：

- **SFT 轨迹**
  - 给模型打底
- **环境交互轨迹**
  - 让模型探索
- **合成任务**
  - 扩大训练规模
- **高质量验证任务**
  - 防止只在训练环境里看起来变强

所以现实里更常见的不是“纯 RL”，而是：

**SFT + RL 的组合拳。**

### 10. 为什么现在大家会提到 GRPO

你会发现这两年 Agent 训练讨论里，GRPO 出现得比较频繁。

从非常实用的角度讲，你可以把它理解成：

**在一组候选答案里做相对比较，而不是依赖一个特别昂贵、特别稳定的 value 估计。**

它之所以受关注，主要是因为在大模型训练语境里，这种做法在某些场景里更省事、更稳定，也更适合和现有推理采样流程结合。

你不一定要把算法细节全背下来，但至少要抓住一个判断：

**Agentic RL 的很多算法改造，都是在想办法降低长时程训练的不稳定性和成本。**

### 11. LoRA 在这里为什么重要

因为很多团队根本不可能每次都全参训练。

所以在工程实践里，常见路径是：

- 先拿一个基础模型
- 做少量 SFT
- 用 LoRA 这类参数高效微调方式做适配
- 再叠加 RL 优化某类任务能力

这条路的现实意义是：

它让“训练 Agent 行为”从大厂专属，变成中小团队也可以尝试的事情。

### 12. Agentic RL 和工作流优化，不是替代关系

一个很重要的边界是：

如果你的问题本质上是：

- 工具没接好
- 任务没拆清
- 上下文工程做得很差
- 评估体系没有建立

那上 RL 往往只是在放大混乱。

所以 Agentic RL 不是第一步。  
更像是在你已经把系统基本骨架搭稳之后，用训练去继续逼近上限。

### 13. 什么时候值得认真考虑 Agentic RL

通常至少满足下面一些条件时，才值得开始：

- 任务是高价值、可重复、可规模化的
- 单靠 prompt 和 workflow 已经很难继续提升
- 任务结果可验证，不然 reward 难以定义
- 有足够样本和算力做迭代
- 有评估基线，知道“变强”到底是什么意思

### 14. 什么时候完全不值得

如果你还处于这些阶段，通常不值得急着上：

- 还没把单 Agent 跑稳
- 任务本身没有清晰成功标准
- 数据量小到训练没有意义
- 成本敏感，无法承担 rollout 和评估开销
- 真正问题是系统工程，不是模型行为

这种时候，更高 ROI 的通常还是：

- 改上下文工程
- 改工具设计
- 改记忆与检索
- 改评估和护栏

### 15. 一个更高阶的理解：Agentic RL 在训练“决策风格”

它不只是训练知识，也不只是训练格式。  
它在训练一种更像“策略”的东西：

- 什么时候该继续想
- 什么时候该查资料
- 什么时候该调用工具
- 什么时候该停
- 遇到错误时如何恢复

从这个意义上说，Agentic RL 比普通对齐更接近“行为训练”。

### 16. 常见误区

- 误区一：以为 RL 一上，Agent 就会自己进化
- 误区二：还没把系统工程做好，就想用 RL 硬救
- 误区三：只看最终准确率，不看 reward hacking
- 误区四：把单轮偏好优化等同于长时程任务优化
- 误区五：忽略环境稳定性，导致训练出来的策略不可迁移

## 一句话总结

**Agentic RL 不是在教模型“说得更像助手”，而是在教它作为一个行动者，在多步任务里学会更好的决策策略；但它只有在任务可验证、系统骨架稳定、评估闭环完整时才真正值得投入。**
