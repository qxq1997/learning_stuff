# AI Agent - 第 12 课：综合复盘：如何判断你的业务该用多少 Agent

## 学习目标

- 用一套更成熟的判断框架，回看 Agent 到底应该用在什么地方。
- 学会在“0 Agent、Workflow、单 Agent、多 Agent”之间做取舍。
- 理解 Agent 引入的真实收益和真实代价，不再被概念热度带着走。
- 建立一个适合业务落地的渐进式路线图。
- 完成这一专题的整体收束，把前面所有课串成工程判断力。

## 内容讲解

### 1. 学到最后，最重要的不是会做，而是会克制

学 Agent 的过程里，最容易出现一种状态：

越学越觉得什么都能做成 Agent。

但真正成熟的工程判断，通常不是“这里也能上 Agent”，而是：

**这里到底值不值得上 Agent。**

因为每引入一点自主性，都会带来一整套额外成本：

- 上下文管理
- 工具设计
- 状态存储
- trace 和评测
- 安全和权限
- 成本和终止控制

所以最后最重要的能力，其实是做减法。

### 2. 四种系统形态，你要能选得出来

很多业务问题，大体都可以落在下面四种形态里。

#### 2.1 0 Agent：普通系统就够了

如果问题本身就是：

- 规则清晰
- 输入结构化
- 逻辑稳定

那一个普通后端系统就够了。

比如：

- 权限校验
- 账务处理
- 状态流转
- 报表聚合

这里不需要模型，更不需要 Agent。

#### 2.2 Workflow：固定流程 + 局部模型能力

如果流程稳定，但中间某几个步骤适合用模型，那就做 Workflow。

比如：

- 工单分类
- 内容审核辅助
- 文本提取和总结

这类系统通常已经很有价值。

#### 2.3 单 Agent：任务路径有一定动态性

如果任务中途确实要根据观察结果调整路径，那单 Agent 就有价值。

比如：

- 研发排查助手
- 深度研究助手
- 多轮信息整合助手

#### 2.4 多 Agent：分工收益真的大于协作成本

只有在角色边界清晰、任务链长、单体上下文过载时，多 Agent 才值得上。

### 3. 一个很实用的判断表

你可以用下面几个维度来判断。

#### 3.1 步骤是否稳定

- 很稳定 -> Workflow 或普通系统
- 不稳定 -> 更可能需要 Agent

#### 3.2 中途观察是否会改变路径

- 不会 -> Workflow
- 会 -> Agent

#### 3.3 风险是否很高

- 很高 -> 先规则系统 + 审批，少放权
- 中低风险 -> 可以逐步增加 Agent 自主性

#### 3.4 子任务边界是否清楚

- 不清楚 -> 先别做多 Agent
- 清楚 -> 才有可能拆多 Agent

### 4. 如何判断一个 Agent 项目是不是“假需求”

很常见的假需求信号有这些：

- 其实只是想做一个问答系统，却硬叫 Agent
- 其实步骤都固定，却觉得“不上 Agent 不够先进”
- 其实数据和权限问题还没解决，却先上复杂编排
- 其实没有任何评测和 trace，却想直接搞多 Agent

这些项目很容易在 demo 阶段看起来有趣，但真正落地时收益很小。

### 5. 如何判断一个 Agent 项目是不是“真机会”

比较像真机会的信号通常有：

- 人工流程本身已经很费时间
- 任务里存在明显的信息收集和整理成本
- 决策路径确实依赖中途观察
- 人工专家也常常按“先查 A，再看 B，再决定 C”这种方式工作
- 任务失败代价可控，适合先做辅助

这种场景通常更值得投入。

### 6. 一条更现实的落地路线图

如果你真的要推动业务落地，一个比较健康的路线通常是这样：

#### 6.1 第一阶段：证明价值

先选一个窄场景，目标是：

- 节省时间
- 提高信息整理效率
- 降低人工重复劳动

#### 6.2 第二阶段：把系统做稳

补齐：

- 状态
- trace
- 评测
- 护栏

#### 6.3 第三阶段：适度放权

从纯建议，逐步走到低风险自动执行。

#### 6.4 第四阶段：再谈平台化和多 Agent

只有前面几步都走顺了，平台化和多 Agent 才比较有意义。

### 7. 为什么“先证明单点价值”比“先搭平台”更重要

很多团队会一开始就想搭 Agent 平台。  
但如果你连一个真实场景都还没跑顺，平台大概率会变成空壳。

更好的顺序通常是：

- 先做一个真实可用的 Agent
- 在这个过程中抽象共性
- 再慢慢沉淀成平台能力

平台应该是结果，不该是起点。

### 8. 回头看，前面每一课到底在帮你形成什么能力

如果把整个专题压缩成一句话，其实是在帮你形成三种判断力。

#### 8.1 概念判断力

你要能分清：

- LLM
- Workflow
- RAG
- Agent
- Multi-Agent

#### 8.2 系统判断力

你要能设计：

- 工具
- 状态
- 循环
- 护栏
- 评测

#### 8.3 业务判断力

你要能回答：

- 值不值得做
- 先做哪块
- 放权到什么程度
- 风险怎么控

真正拉开差距的，往往不是谁背了更多术语，而是谁有这三种判断力。

### 9. 一个最后的建议：把 Agent 当成系统能力，不是魔法能力

Agent 这几年之所以容易被误解，一个原因就是大家太容易把它想成某种“突然涌现出来的智能魔法”。

但从工程视角看，它其实更像：

- 模型推理
- 工具系统
- 状态管理
- 任务编排
- 风险控制
- 可观测性

这些能力拼起来的一个系统。

一旦你这样看它，很多东西都会变得清楚：

- 哪里该乐观
- 哪里该保守
- 哪里值得投入
- 哪里其实没必要上

## 小结

这一课最核心的结论是：

**不是所有业务都该用 Agent，也不是用了 Agent 就更高级。真正成熟的选择，是在普通系统、Workflow、单 Agent、多 Agent 之间做出合适的取舍。**

如果一个场景能用更简单的方式解决，就先别急着上复杂架构。  
如果一个场景确实存在动态决策价值，再逐步引入 Agent，并且先把单点价值做出来、做稳定。
