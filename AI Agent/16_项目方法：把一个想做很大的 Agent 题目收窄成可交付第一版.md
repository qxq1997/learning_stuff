# AI Agent - 扩展课 16：项目方法：把一个想做很大的 Agent 题目收窄成可交付第一版

## 学习目标

- 学会用"任务形态"而不是"项目名字"来理解一个 Agent 题目。
- 拿到一个模糊宽泛的题目，能快速判断它是不是好题目。
- 能把"全能助手"式的大题目收窄成一个可在 1-2 周内交付的第一版。
- 能为课程项目、内部试点、毕业设计找到现实的切入点。
- 知道 Agent 项目最常踩的"野心-范围-评估"三角失衡陷阱。

## 先给结论

**Agent 项目失败最常见的原因不是技术不到位，而是题目太大、范围太散、成功标准模糊。**

真正能交付的 Agent 项目，第一版几乎都长得很窄、很无聊：

- 只读，不写
- 只一类任务，不全能
- 只一类用户，不通用
- 只一种成功定义，不多目标

把"很酷的想法"压成"第一版能落地的窄场景"，是这一课的全部要点。

## 1. 学项目时先看任务形态，不要看项目名字

听到"旅行助手 / 研究助手 / 客服 Agent"这种名字，很容易先入为主想"这是个很有趣的应用"，然后开始堆功能。这条路一上来就走偏。

更稳的视角是：**看它属于哪一类任务形态**。任务形态决定了你真正要练的是哪些工程能力。

下面是几种常见 Agent 项目的任务形态拆解。

### 1.1 旅行助手（任务形态：多源查询 + 信息整合 + 结构化输出）

它本质上是：

- 多工具查询（航班 / 酒店 / 天气 / 路线）
- 信息整合
- 结构化结果生成

**它真正训练你**：
- 工具编排（参数怎么组合、顺序怎么定）
- 输出结构化（行程表、预算表）
- 多数据源整合（不同来源结果的去重、对齐、置信度）

### 1.2 深度研究助手（任务形态：长任务 + 笔记 + 阶段推进）

它本质上是：

- 长任务
- 多轮搜索
- 中间笔记
- 最终报告生成

**它真正训练你**：
- 上下文工程（哪些进 prompt、哪些进笔记）
- 笔记系统（pinned / non-pinned、待办 / 已确认 / 已排除）
- 计划和阶段推进（任务状态机）

### 1.3 研发助手 / 工单助手（任务形态：内部知识 + 受控工具 + 审计）

它本质上是：

- 公司内部知识检索
- 受限工具集（创建工单、查日志、查监控）
- 强审计 / 强权限边界

**它真正训练你**：
- RAG 工程（chunk 策略、混合检索、权限边界）
- 工具系统设计（schema、幂等、副作用控制）
- 审计与合规（每一步都可追溯）

### 1.4 AI 小镇 / NPC 协作（任务形态：多 Agent + 角色记忆 + 世界状态）

它本质上是：

- 多 Agent 协作
- 角色长期记忆
- 共享世界状态
- 连续交互

**它真正训练你**：
- 多 Agent 状态管理
- 角色 / 用户画像
- 长期一致性（同一个 NPC 一周后还记得你是谁）

### 1.5 一个很有用的对照：你到底在训练哪类肌肉

| 项目形态 | 主要训练 | 次要训练 |
| --- | --- | --- |
| 多源查询整合 | 工具编排、结构化输出 | 上下文装配 |
| 长任务研究 | 上下文工程、笔记系统 | 阶段状态机 |
| 内部知识助手 | RAG、工具受控、审计 | 权限边界 |
| 多 Agent 协作 | 多 Agent 编排、记忆 | 世界状态、角色一致性 |

选项目时，先想清楚**你想练哪块肌肉**，再倒推选哪种形态——而不是看哪个名字最酷。

## 2. 一个题目是不是好题目，主要看五件事

把题目放进这五个问题里过一遍，能快速判断它能不能落地。

### 2.1 问题真实存在吗

很多 Agent 项目失败的第一个信号是：**写题目时只能想象出"潜在用户可能会用"**，而说不出"昨天有个真实的人因为这个问题花了半小时"。

判断方法很简单：能不能找到 3 个真实用户，他们最近一周确实遇到过这个问题。如果找不到，先别动。

### 2.2 目标说得清楚吗

不能是"做一个智能助手帮用户做研究"这种话。

应该是：**"输入一个研究主题，输出一份不超过 2000 字的结构化综述（背景 / 主要观点 / 三个最相关来源），耗时不超过 5 分钟，token 成本不超过 $1"**。

具体到这个程度，团队才知道在做什么，做完才知道能不能验收。

### 2.3 成功标准能衡量吗

成功标准必须是**可观测的、第三方能复核的**。

好的成功标准：
- 5 个真实用户在一周内主动用了 3 次以上
- 80% 的输出用户标注为"可直接使用，无需大改"
- 对比 baseline（用户自己 google + 整理）能省 40% 时间

坏的成功标准：
- "用户觉得有用"（怎么证明？）
- "效果还不错"（不可验证）
- "比 GPT 更聪明"（不可对比）

### 2.4 风险可控吗

Agent 一旦能写、能调外部 API、能花钱、能影响他人，风险就指数级上升。

第一版项目应该几乎都是**只读的**：
- 只查不写
- 只建议不执行
- 只输出文档不调真实 API
- 真的需要写时，强制人工确认

把"自动执行"放到第二版以后，是省麻烦的最有效方式。

### 2.5 第一版能做窄吗

如果你描述第一版功能列表的时候停不下来，几乎肯定题目还太大。

一个好题目，第一版功能描述应该一句话能说完：

- "**输入论文 PDF，输出一段 200 字的摘要 + 5 个关键引用。**"
- "**输入 K8s Pod 名字，输出过去 24 小时事件 + 资源使用 + 可能原因 3 条。**"
- "**输入一个公司内部 wiki 链接，输出一段总结 + 3 个关联文档。**"

如果第一版做不到一句话能说完，先回去砍。

## 3. 一个想做很大的题目，怎么收窄

最容易踩的坑是：题目本身没问题，但你想做的"第一版"太大。

举两个真实的收窄例子。

### 3.1 旅行助手怎么收窄

**初始野心**（典型失控版）：
- 查路线
- 查天气
- 查酒店
- 出预算
- 订票
- 自动改签
- 推送提醒
- 集成支付

这版第一版几乎不可能完成。

**收窄到第一版**：
- **只做"生成可编辑行程建议"**
- 不做真正预订
- 只读，不写
- 只输出结构化计划（用户可复制粘贴到 Notion）
- 用户范围：只服务"提前 2 周以上的国内 3-5 天行程"
- 工具：航班查询 + 酒店查询 + 天气查询，三个就够

第一版用 1-2 周就能做出来。**真正预订、自动改签、支付，全部留到第二版以后**。

### 3.2 深度研究助手怎么收窄

**初始野心**：
- 自动调研一个领域
- 自动产出完整研究报告
- 自动比较多个观点
- 自动给出独立结论

第一版完全不应该做"自动产出完整研究报告"，因为：
- 成功标准模糊（什么叫"完整"？）
- 评估困难（你怎么判断研究报告好不好？）
- 容易幻觉（agent 不知道的东西会瞎编）

**收窄到第一版**：
- **只做"信息收集 + 去重 + 摘要 + 引文整理"**
- 不下结论，让人来下
- 输出物：5-10 篇相关文献的摘要 + 3 个不同观点的并列陈列 + 完整可点击引用
- 用户范围：只服务"已有明确研究主题"的用户（不帮你想题目）

这个版本容易做、容易评估（用户能验证引用真伪 + 摘要是否准确）、价值清晰。

### 3.3 收窄的通用方法：四把刀

把任何"想做很大"的题目按这四个维度切：

| 维度 | 怎么切 | 例子 |
| --- | --- | --- |
| **用户切窄** | 只服务一类用户 | "国内 3-5 天提前 2 周行程" 而不是 "所有旅行者" |
| **任务切窄** | 只做完整链路里的一段 | "整理引用" 而不是 "写完整报告" |
| **副作用切窄** | 只读不写 | "建议" 而不是 "执行" |
| **场景切窄** | 只覆盖一类输入 | "已知 K8s Pod 名" 而不是 "任意运维问题" |

任何一个题目都至少要被这四把刀各砍一次。砍完看看还剩什么，那就是第一版。

## 4. 项目收窄模板（五个问题压缩成一句话）

每次开题时，先把题目压成这五个问题的答案。回答不出来就别开工。

1. **这个系统到底帮谁省时间？**
   - 一个具体的人 / 角色，名字 / 岗位说得出来
2. **它第一版只完成哪一件事？**
   - 一句话能说完，多一件就不算第一版
3. **它需要哪些只读工具？**
   - 列出工具清单，越少越好（3-5 个最佳）
4. **它哪些动作绝对不能自动做？**
   - 写、删、付钱、发邮件、改配置——明确写出禁止清单
5. **我怎么判断它真的比人工流程更省力？**
   - 一个可衡量的指标 + baseline 对比

只要这五个问题答不清，大概率题目还太大。**这五问就是 Agent 项目的开题模板**。

## 5. 课程项目 / 毕设：哪些题更稳

更稳的方向（按推荐度排序）：

1. **文档问答 + 检索 + 结构化输出**——RAG 主线，评估清晰
2. **研发助手 / 工单助手 / 知识助手**——内部场景，工具有限，边界清晰
3. **报告生成 / 研究摘要 / 信息整合**——用户能验证输出，容易评估
4. **有限工具调用的多步任务**——比如"3 个 API 串起来生成结构化结果"

**不太适合一开始就做的**：

- 全自动交易 / 投资助手（金融合规 + 高风险，且你也不该做）
- 全自动生产运维（一个动作错就是事故）
- 完全自治的多 Agent 世界（评估几乎不可能）
- 需要大量底层训练才能成立的项目（Agentic RL 训练是另一个量级的工作量，参考第 11 课）

这些项目难，主要不是因为技术量大，而是**成功标准模糊、调试链路太长、风险太高**。这三件事任意一件就足以让一个学期的项目废掉。

## 6. 一条很现实的项目推进路线

如果你真的要做项目，建议按这个节奏。

### 第一步：先做一个窄助手（1-2 周）

- 能回答、能检索、能给结构化结果
- 工具不超过 5 个
- 只读、只服务一类用户
- 部署到本地能 demo

这一步完成时，你会发现：**就算什么花活都没做，能稳定跑 50 个真实任务的窄助手已经比 80% 的 demo 强**。

### 第二步：补状态和笔记（1 周）

- 引入任务状态（planning / running / done / blocked）
- 引入笔记（已确认事实、已排除假设、待办）
- 让它能处理跨会话长任务

这一步你会真正理解第 9 课讲的上下文工程不是抽象概念。

### 第三步：补评测和 trace（1 周）

- 沉淀 20 个真实失败案例做回归集
- 每次改 prompt / 工具 / 检索都跑一次
- 加 trace（每次 LLM 调用 + 每次工具调用 + 中间状态）
- 这样它从 demo 变成可迭代系统

这一步是 demo 和"真正能上线的东西"之间的分水岭，对应第 12 课的评估闭环。

### 第四步：才考虑多 Agent 或更复杂协议

如果第一到三步走完了，再决定要不要拆 Agent、要不要上 A2A / MCP。**90% 的情况你会发现根本不需要**——单 Agent + 5 个工具 + checkpointer 已经够用了。

这条路线的核心思想：**先把一件事做稳，再扩；不要一上来就做大**。

## 7. 三个最常见的"野心-范围-评估"失衡陷阱

总结几个真实见过的失败模式，避开它们能救你很多时间。

### 7.1 陷阱一：野心 >> 范围

题目想得很大（"做一个全能旅行助手"），但实际范围说不清第一版到底做什么。**症状是：开了三周还在改架构图，没有写代码**。

修复：用第 3 节的"四把刀"砍。

### 7.2 陷阱二：范围 >> 评估

第一版功能写得很清楚，但成功标准是"用户觉得不错"。**症状是：做完了不知道是不是真的好用，团队各执一词**。

修复：用第 2.3 节的"可观测、可复核"两条标准重写成功定义。

### 7.3 陷阱三：评估 >> 真实用户

成功标准定得很学术（"在 GAIA 子集上 70% 准确率"），但根本没有真实用户来验证。**症状是：benchmark 跑得不错，给真人用就一团糟**。

修复：每周必须有 3 个真实用户用过，否则 benchmark 数据没意义。

## 8. 学前沿最稳的心态

Agentic RL（见第 11 课）、协议生态（见第 10 课）、多 Agent 社会模拟，这些都很值得了解。但更稳的学习顺序不是：

- 先追最前沿名词

而是：

- **先把真实系统做稳**
- 再去理解前沿为什么难

只有你真的做过 Agent 系统、踩过状态管理 / 工具失控 / 评估虚高这些坑，才会清楚：

- 哪些前沿问题确实重要（往往是 evaluation、long-horizon、tool-use 训练）
- 哪些只是暂时很热（往往是 multi-agent simulation、各种新 framework）

很多团队的真实经验是：**做完一个真实落地的窄助手后，再回头看前沿论文，理解力会强 5 倍**。

---

## 小结

这一课最重要的结论是：

**Agent 项目的核心难题不是技术，而是题目选择和范围控制。先收窄，再迭代；先做稳，再做大；先满足真实用户，再追求架构优雅。**

一句话开题模板：

> 我要帮 [具体的人] 做 [一件具体的事]，用 [3-5 个只读工具]，第一版禁止 [写 / 付钱 / 发邮件 / 改配置]，成功标准是 [一个可观测、可复核的指标]。

任何 Agent 项目，能用这个模板把第一版讲清楚，就有 70% 概率做成。
