# AI Agent - 第 22 课：工具系统设计：Schema、鲁棒性、幂等与结果标准化

## 学习目标

- 理解为什么很多 Agent 的问题表面看是“模型不聪明”，本质却是工具系统设计得太差。
- 建立一套工具工程视角：不是“能调一次”就算完成，而是要可选、可解释、可重试、可审计、可恢复。
- 掌握工具 schema、参数生成、幂等、副作用控制、结果标准化这几个最常见的工程要点。
- 能识别工具接入中的常见失败模式，并知道应该怎么修。

## 内容讲解

### 1. 工具为什么是 Agent 系统的第一事故源

很多团队在做 Agent 时，最开始会把重点放在：

- prompt 怎么写
- 模型怎么选
- memory 怎么接

但一旦进入真实生产，最容易出事故的往往是工具系统。

原因很简单：

- 模型决定“要不要用工具”
- 工具决定“能不能真正做成事”

如果工具设计不稳，模型再强也只是一个会指挥但不会落地的调度器。

常见症状包括：

- 模型老选错工具
- 参数经常不合法
- 同一个动作重复执行两次
- 工具返回太乱，模型没法继续推理
- 外部系统明明成功了，Agent 却以为失败
- 失败了但无法恢复

### 2. 工具系统的真正职责不是“封个函数”

一个成熟的工具系统至少要同时解决五件事：

1. 能力表达
2. 参数约束
3. 副作用边界
4. 错误语义
5. 结果可消费性

所以工具不是普通 SDK 方法的裸暴露。  
它更像给模型提供的一种“受控行动接口”。

### 3. schema 为什么不是装饰，而是行为引导

工具 schema 往往包含：

- 名称
- 描述
- 参数字段
- 参数类型
- 是否必填
- 枚举限制
- 示例
- 返回结构

很多团队把 schema 当“给 OpenAI API 填个 JSON”。  
这会直接低估它的重要性。

从模型角度看，schema 不是元数据，而是**动作空间说明书**。

也就是说，schema 写得越含糊，模型越容易：

- 误解工具用途
- 乱填参数
- 在相似工具之间选错

### 4. 一个差 schema 会怎么害死系统

比如你提供两个工具：

- `search_user`
- `get_user_profile`

如果描述都只是“查询用户信息”，模型很难分清：

- 一个是模糊搜索
- 一个是精确读取

这时它可能会：

- 明明拿到了 user_id，还继续走 search
- 本来要查 profile，却把手机号丢给精确查询接口

所以 schema 设计时最重要的不是“写得规范”，而是：

**让模型能在相似能力之间稳定做出正确区分。**

### 5. 工具描述应该重点写什么

一个好工具描述通常至少要讲清这几件事：

1. 这个工具是干什么的
2. 什么时候该用它
3. 什么时候不该用它
4. 参数各自代表什么
5. 调用前的前置条件是什么
6. 调用后的返回结果该怎么理解

比如比起写：

- “查询订单”

更好的写法是：

- “通过已知 order_id 精确查询单个订单详情。若只有用户名或手机号，请先调用 `search_orders` 获取候选订单，再使用本工具读取详情。”

这类描述会显著降低错误调用率。

### 6. 参数约束越清楚，系统越稳定

模型最擅长“大致对”，但工程系统往往要求“字段精确合法”。

常见风险包括：

- 日期格式错
- 枚举值错
- 单位错
- 字段缺失
- 把展示字段当内部 ID

所以参数层面最好做三道防线：

1. schema 约束
2. 服务端校验
3. 自动修正 / 友好报错

不要把“模型应该会填对”当作校验策略。

### 7. 工具返回结果为什么也要设计

很多工具系统前半段做得还行，后半段崩在返回值。

典型问题：

- 返回原始 HTML
- 返回一大坨未清洗日志
- 返回深层嵌套 JSON
- 返回码和业务语义不一致
- 返回结果里混杂技术字段和业务字段

这会让模型后续推理质量快速下降。

从 Agent 视角，好的工具结果应该满足：

- 结构稳定
- 字段语义明确
- 尽量少噪声
- 同时保留必要证据

### 8. 结果标准化是工具系统的核心增益之一

假设你有三个不同来源的搜索工具：

- 数据库搜索
- 文档搜索
- 第三方 API 搜索

如果它们返回格式各不相同，模型就得每次重新适应。

一个常见做法是做一层结果标准化：

```json
{
  "status": "success",
  "items": [
    {
      "id": "doc_123",
      "title": "支付排障文档",
      "summary": "连接池耗尽会导致超时放大",
      "source": "internal_docs",
      "confidence": 0.86
    }
  ],
  "raw_ref": "trace://tool-run/abc"
}
```

这样模型看到的是统一语义，真正原始细节留给 trace 和审计。

### 9. 幂等性为什么是 Agent 的救命绳

Agent 和普通按钮式系统很不一样。  
它天然容易出现：

- 模型不确定是否执行成功，于是再试一次
- 网络超时后重放
- orchestrator 崩溃恢复后重试
- 人工点击“继续执行”

如果工具没有幂等设计，就很容易出现：

- 重复扣款
- 重复发券
- 重复发邮件
- 重复创建工单

所以一切有副作用的工具，都应该默认从“可能会被重复执行”来设计。

### 10. 幂等通常怎么做

常见手段包括：

- 幂等 key
- 去重表
- 请求指纹
- 业务唯一约束
- 结果缓存

例如“创建退款申请”这种工具，不应该只暴露：

- `create_refund(order_id, amount)`

更稳的接口是：

- `create_refund(order_id, amount, request_id)`

然后由服务端基于 `request_id` 做幂等去重。

### 11. 副作用工具要和只读工具明确区分

这件事在安全和稳定性上都非常关键。

工具大体可以分成两类：

- 只读工具
- 有副作用工具

只读工具例如：

- 查询用户
- 搜索文档
- 查看日志

副作用工具例如：

- 发邮件
- 创建工单
- 修改配置
- 下单
- 删除资源

这两类工具在系统里不应该被一视同仁。

副作用工具通常需要额外机制：

- 更高权限
- 人工确认
- 幂等控制
- 更严格审计
- 回滚 / 补偿设计

### 12. 错误语义必须对模型友好

很多工具失败时只返回一句：

- `Internal Server Error`

这对工程师来说都不够用，对模型更没有意义。

工具错误至少应该分清：

- 参数错误
- 权限错误
- 不存在
- 临时不可用
- 超时
- 业务冲突
- 不可重试错误

一个更适合 Agent 的错误结构可能是：

```json
{
  "status": "error",
  "error_type": "retryable_timeout",
  "message": "Order service timed out after 3000ms",
  "suggestion": "retry with backoff or ask user to narrow query"
}
```

这能帮助 orchestrator 决定：

- 自动重试
- 改参数再试
- 直接终止
- 请求人工接管

### 13. 工具应该暴露“动作”，而不是暴露底层系统复杂性

模型不是系统集成工程师。

如果一个业务动作需要：

- 先查用户
- 再查订单
- 再查地址
- 再拼请求体
- 再调第三方接口

那你要认真判断：

是应该暴露 5 个底层工具，
还是暴露一个更高层的业务工具。

工具粒度太底层，会带来：

- 调用链变长
- 错误点变多
- 上下文更长
- 成功率下降

但工具粒度过粗，也会带来：

- 复用性差
- 调试困难
- 权限边界模糊

一个常见经验是：

**优先暴露“业务动作级工具”，必要时再保留底层读工具。**

### 14. 工具系统最常见的失败模式

1. 工具太多，模型选不明白
2. 工具名字像，语义没分清
3. 参数要求复杂，模型经常填错
4. 返回结果太乱，后续推理失真
5. 有副作用工具缺少确认和幂等
6. 重试无策略，导致风暴
7. 错误语义差，系统不会恢复
8. trace 不全，出了问题无法回放

### 15. 一个实用的工具设计检查清单

每接一个新工具，至少问这些问题：

- 这个工具的职责边界清楚吗？
- 它和其他工具的区别够明显吗？
- 参数字段是不是能被模型稳定理解？
- 是否需要前置校验？
- 是否有副作用？
- 是否需要幂等？
- 是否需要人工确认？
- 返回结果是不是结构化、稳定、可继续使用？
- 错误是否能被系统分类处理？
- 是否接入 trace、审计和指标？

## 小结

很多 Agent 项目的真实瓶颈，不在模型推理本身，而在工具系统太脆。

你可以把工具系统理解成 Agent 的“手和脚”。  
手脚如果长得不清楚、不稳定、不受控，那大脑再聪明也做不好事。

所以一个好的工具系统，要追求的不是“接入速度最快”，而是：

- 模型容易选
- 参数容易填
- 结果容易用
- 副作用可控制
- 错误可恢复
- 整体可审计
