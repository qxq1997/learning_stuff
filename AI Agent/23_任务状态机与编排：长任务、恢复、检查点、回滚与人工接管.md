# AI Agent - 第 23 课：任务状态机与编排：长任务、恢复、检查点、回滚与人工接管

## 学习目标

- 理解为什么 Agent 一旦走出单轮同步调用，就必须引入任务状态机和编排层。
- 搞清楚长任务为什么容易跑丢、重复执行、卡死和无法恢复。
- 掌握检查点、幂等、补偿、人工接管这些关键机制在 Agent 里的位置。
- 能为一个长任务 Agent 设计基本的状态流转和恢复策略。

## 内容讲解

### 1. 为什么长任务是 Agent 从 demo 到工程的分水岭

一个 demo 常见路径是：

- 用户提问
- 模型思考
- 调工具
- 返回答案

这类流程都在一次请求里完成。  
看上去系统很顺。

但真实业务很快会碰到：

- 任务持续几分钟到几小时
- 需要多轮工具调用
- 依赖异步事件回调
- 中途可能失败
- 用户可能中途修改目标

这时如果还把 Agent 当成一个同步 HTTP 接口，系统很快就会失控。

### 2. 任务编排层在解决什么

任务编排层最核心的作用是：

**把“模型的连续推进行为”变成一个可管理、可中断、可恢复的状态过程。**

它通常需要负责：

- 保存任务状态
- 调度下一步动作
- 管理超时与重试
- 落检查点
- 处理人工中断
- 恢复失败任务

也就是说，编排层不是锦上添花，而是 Agent 系统的控制平面。

### 3. 为什么状态机比“写个 while 循环”更重要

很多早期实现会写成：

- while 未完成
- 调模型
- 看下一步
- 执行

这种写法在本地跑 demo 很方便，但一旦进入生产就会出问题：

- 进程挂了，任务丢了
- 中途成功到哪一步不知道
- 重启后无法恢复
- 某一步重复执行可能有副作用

状态机的意义，就是把任务执行过程显式化。

例如一个任务至少可以有：

- `created`
- `planning`
- `waiting_tool_result`
- `waiting_human_approval`
- `running`
- `retrying`
- `completed`
- `failed`
- `cancelled`

### 4. 状态机真正的价值不是流程图，而是恢复点

状态机最大的好处不是“看起来专业”，而是：

**每个状态都意味着一个可恢复边界。**

例如：

- 如果任务停在 `waiting_human_approval`，恢复后不应重跑前面的动作
- 如果停在 `waiting_tool_result`，恢复后应先查询外部工具是否已成功
- 如果停在 `retrying`，恢复后要判断重试预算是否耗尽

### 5. 检查点为什么比聊天历史更重要

长任务里，聊天历史只能告诉你“发生过什么”，  
但检查点要告诉你：

- 现在任务处在哪个阶段
- 哪些动作已经确定完成
- 哪些动作绝不能重复
- 下次该从哪里继续

一个好的检查点一般会包含：

- 任务 ID
- 当前状态
- 已完成步骤列表
- 每步结果摘要
- 外部副作用记录
- 待执行动作
- 重试计数
- 最后更新时间

### 6. 检查点不能只存在模型上下文里

这是最常见的工程错误之一。

如果你把任务推进信息只放在：

- 对话历史
- scratchpad
- 模型输出文本

那系统一旦发生：

- 进程重启
- context 裁剪
- 多实例切换
- 任务转移

这些状态就都不可靠了。

所以检查点一定要外化到系统状态存储。

### 7. 长任务恢复时最怕什么

最怕的是“看起来恢复了，实际上已经重复执行了副作用动作”。

例如：

- 已经发过邮件，但恢复时又发一次
- 已经创建过工单，但恢复时又创建一个
- 已经完成审批，但状态没写回，恢复后重新申请

所以恢复逻辑必须建立在：

- 状态持久化
- 副作用幂等
- 外部结果可查询

这三件事之上。

### 8. 补偿和回滚为什么要区分

很多人会自然地说“失败了就回滚”。  
但在 Agent 系统里，真正的数据库式回滚往往做不到。

因为你经常面对的是：

- 发过邮件
- 下游 API 已调用
- 人工审批已完成
- 第三方系统已接受请求

这些动作无法像数据库事务一样简单撤销。

所以更现实的做法通常是：

- **回滚**：只适用于可逆的本地状态
- **补偿**：用另一个动作抵消已产生的副作用

例如：

- 已创建错误工单 -> 再执行关闭工单
- 已错误发券 -> 再执行作废
- 已修改配置 -> 再写回旧版本

### 9. 人工接管为什么必须被当成一等能力

很多 Agent 系统的一个错误幻想是：

**一旦流程自动化了，就不需要人工。**

实际上越复杂的任务，越需要人工接管点。

典型场景包括：

- 模型置信度低
- 多次重试仍失败
- 高风险副作用动作
- 涉及业务规则解释
- 用户目标发生变化

一个成熟系统应该支持：

- 暂停任务
- 注入人工备注
- 修改任务状态
- 指定从某个检查点继续
- 直接终止或转人工处理

### 10. 任务编排常见的实现层次

可以粗略分成三层：

1. **任务定义层**
   - 定义阶段、状态、动作类型
2. **执行控制层**
   - 调度模型、工具、重试、等待
3. **恢复与治理层**
   - 检查点、审计、人工介入、监控

这三层不要揉在一个 prompt 里。

### 11. 一个长任务常见的失败模式

1. 状态只保存在上下文里
2. 没有明确状态机
3. 没有检查点
4. 副作用动作不可幂等
5. 恢复逻辑默认重跑
6. 没有人类接管点
7. 没有终止条件，任务空转
8. 没有超时预算，任务无限挂起

### 12. 为什么终止条件必须显式定义

Agent 很擅长继续做事，但不擅长自己优雅停下。

如果没有终止条件，常见结果是：

- 循环调用工具
- 反复总结
- 一直搜证据但不收敛
- 一直尝试微调答案

终止条件通常至少要有：

- 成功完成
- 达到最大步数
- 达到最大耗时
- 达到最大失败次数
- 被人工取消
- 风险超阈值

### 13. 长任务里“等待”也是一种重要状态

很多编排系统只关注执行，不关注等待。

但现实里大量任务都会进入等待：

- 等待外部审批
- 等待异步回调
- 等待数据库任务完成
- 等待用户补充信息

如果等待状态建模不好，就会出现：

- 死轮询
- 无意义重试
- 任务永远不结束

### 14. 一个比较实用的设计原则

对于长任务，可以把每一步动作都当成：

- 输入明确
- 输出明确
- 副作用明确
- 恢复策略明确

如果一段逻辑无法说清这四件事，它大概率还不适合上线。

## 小结

长任务 Agent 真正难的，不是让模型“持续思考”，而是：

- 让系统知道它此刻在哪
- 知道它做过什么
- 知道它失败时如何继续
- 知道什么时候该停
- 知道什么时候该交给人

所以任务编排的核心，不是 while loop，而是：

- 状态机
- 检查点
- 幂等
- 补偿
- 人工接管
