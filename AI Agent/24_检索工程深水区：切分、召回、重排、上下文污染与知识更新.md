# AI Agent - 第 24 课：检索工程深水区：切分、召回、重排、上下文污染与知识更新

## 学习目标

- 理解为什么“接了向量库”不等于“RAG 做好了”。
- 搞清楚 RAG 系统最常见的工程问题出在哪里。
- 掌握 chunk 切分、召回、重排、上下文装配、知识更新这些关键环节的真实取舍。
- 能对 RAG 效果差的问题进行结构化排查。

## 内容讲解

### 1. 为什么 RAG 这么容易看起来正确

RAG 很容易给人一种幻觉：

- 文档入库了
- embedding 也做了
- 检索也返回内容了
- 模型看上去还引用了资料

于是大家会觉得系统“已经有知识增强了”。

但生产里的问题通常不是“完全没检索到”，而是：

- 检索到了一些相关内容，但不是最关键的
- 召回顺序不对
- chunk 被切坏
- 上下文被污染
- 文档过期
- 模型没有真正依赖证据作答

### 2. RAG 是一条链，而不是一个组件

一个完整 RAG 至少包括：

1. 文档采集
2. 清洗解析
3. 切分 chunk
4. 索引构建
5. 查询理解
6. 候选召回
7. 重排
8. 上下文拼装
9. 生成
10. 引用与评估

任何一环做差，系统最终都可能“像是检索问题”，但根因完全不同。

### 3. 切分为什么是第一大隐形问题

很多团队把切分做成了：

- 固定每 500 字一个 chunk

这在简单场景里能跑，但一旦文档变复杂，问题就出来了：

- 语义边界被截断
- 表格拆碎
- 标题与正文分离
- 一个结论依赖多个 chunk 才完整

切分本质上是在做两个目标之间的平衡：

- chunk 足够小，召回精准
- chunk 足够完整，保留语义闭包

### 4. 什么叫语义闭包

一个 chunk 至少应该尽量保证：

- 读这一段时，关键结论的前提仍在
- 模型不需要额外再猜上下文

比如“退款需要主管审批”这句话，
如果前半段“超过 500 元”被切到上一个 chunk，
那模型看到这句时就会误用规则。

### 5. 召回差，不一定是 embedding 差

常见误判是：

- 检索效果差 -> 肯定是 embedding 模型不够强

实际上更常见的原因是：

- chunk 切错
- query 太泛
- metadata 没利用
- 召回 topk 太小或太大
- 过滤条件设置不对
- 同义表达没有被覆盖

embedding 当然重要，但它通常不是第一修复点。

### 6. Query 理解为什么经常被忽略

用户问题往往不是检索友好的。

例如：

- “这个接口为什么会偶发超时？”

真正适合检索的可能是：

- 支付接口超时排查
- 数据库连接池耗尽
- 限流策略与重试

所以 query 在送进检索前，常常需要：

- 改写
- 拆分
- 扩展关键词
- 显式抽取实体

### 7. 混合检索为什么常常优于纯向量检索

纯向量检索适合语义相近。  
但很多工程文档、代码文档场景里，关键词同样重要。

例如：

- 错误码
- 接口名
- 配置项
- 类名
- 表名

这类信息纯语义不一定召回得最好。

所以实践里常见的更稳组合是：

- 向量召回
- 关键词召回
- metadata 过滤
- 混合排序

### 8. 重排为什么经常是效果跃升点

召回的目标是“别漏掉”。  
重排的目标是“把最该看的排前面”。

这是两个不同问题。

很多系统 top20 其实已经有答案，但 top3 不够好，于是模型吃到的上下文全是次优证据。

这时引入 reranker 常常会比盲目换 embedding 更有效。

### 9. 上下文污染到底是什么

上下文污染指的是：

明明召回到了有点关系的内容，但这些内容混在一起之后，反而误导了模型。

常见场景：

- 旧文档和新文档一起进上下文
- FAQ 和设计文档冲突
- 中文问答混入英文实现细节
- 不同业务线的相似规则混在一起

这时模型不是“没看到资料”，而是“看到了太多互相打架的资料”。

### 10. 知识更新为什么比建库更难

第一次建库通常不难，难的是后面：

- 文档改版
- wiki 搬家
- 规则失效
- FAQ 更新
- 代码仓库迭代

如果知识更新链路没有设计好，就会出现：

- 旧 chunk 长期残留
- 同一知识多版本冲突
- 检索结果里旧数据和新数据并存

所以一个成熟 RAG 系统必须有：

- 文档版本意识
- 增量更新机制
- 失效清理策略
- 可追踪的来源标识

### 11. 一个实用的排障顺序

当你发现 RAG 不好用时，不要直接问“模型怎么这么笨”，而是按这条链路排：

1. 问题是否适合检索
2. query 是否被改写得合理
3. chunk 是否切得有语义闭包
4. topk 里有没有正确证据
5. 正确证据是否排得太后
6. 上下文里是否混入冲突证据
7. 模型是否真正使用了证据
8. 文档是否已经过期

### 12. RAG 里最常见的几个反模式

1. 什么都入库
2. 原始 PDF 不清洗直接切
3. chunk 固定长度硬切
4. 只做向量召回
5. 不做 rerank
6. 不做引用，无法验证模型用了什么证据
7. 不做版本治理
8. 评估只看最终回答，不看检索链路

## 小结

RAG 不是“一个向量库 + 一个 prompt”。  
它是一条完整证据链工程。

真正的关键问题是：

- 你有没有把对的知识切出来
- 有没有把它召回来
- 有没有排在前面
- 有没有干净地塞进上下文
- 模型有没有真的据此作答
