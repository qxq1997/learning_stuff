# AI Agent - 第 25 课：记忆工程深水区：写入判定、冲突、遗忘、用户画像与治理

## 学习目标

- 理解长期记忆系统为什么常常是 Agent 后期最脏、最难管的部分。
- 搞清楚记忆系统的关键问题不是“能不能存”，而是“该不该存、怎么更新、怎么失效”。
- 掌握写入判定、读取预算、冲突解决、遗忘机制、用户画像治理这些核心问题。
- 能设计一个不会越用越脏、越用越危险的记忆系统。

## 和第 3 课的边界

第 3 课解决的是概念分层：

- 什么是上下文
- 什么是状态
- 什么是短期记忆
- 什么是长期记忆
- 为什么聊天历史不能代替这些层

本课解决的是工程治理：

- 长期记忆到底分几类
- 一条记忆什么时候该写入
- 旧记忆和新记忆冲突怎么办
- 过期记忆怎么失效
- 用户画像怎么避免变成隐私和偏见风险

一句话区分：

**第 3 课讲“记忆是什么”，第 25 课讲“记忆系统怎么不变脏”。**

## 内容讲解

### 1. Memory 系统最容易被浪漫化

很多人一谈记忆，会想象成：

- Agent 越用越懂你
- 用得越久越聪明
- 会逐渐形成“长期关系”

这些想象不是完全错，但工程上更真实的画面通常是：

- 记进去了太多没用信息
- 旧偏好和新偏好冲突
- 一次错误回答被当成长期事实
- 用户画像逐渐污染
- 敏感信息被长期保存
- 过期记忆反复被召回

所以 Memory 的核心挑战，从来不是存储，而是治理。

### 2. 长期记忆系统最常见的坏掉方式

1. 什么都记
2. 记住了错误事实
3. 旧记忆不失效
4. 用户偏好和任务事实混在一起
5. 检索时总召回过时内容
6. 敏感信息误入长期存储
7. 用户无法查看、修改、删除自己的记忆
8. 没有来源，出错后不知道是谁写进去的

这些问题的根因通常不是向量库不好，而是系统没有定义记忆生命周期。

一个成熟记忆系统至少要回答：

- 谁可以写
- 写什么
- 写到哪里
- 什么时候读
- 怎么更新
- 怎么失效
- 怎么删除
- 怎么审计

### 3. 长期记忆不是一块：四个子类型

长期记忆不要只说成一个“memory store”。  
更实用的拆法是四类：

```text
Profile      = "你是谁"           -> 稳定、全量注入
Semantic     = "你知道什么事实"    -> 较稳定、按需检索
Procedural   = "你会做什么套路"    -> 较稳定、按任务模板命中
Episodic     = "你经历过什么"      -> 大量增长、按相关性检索
```

这四类的读写方式完全不同。

如果混在一个表或一个向量库里，后面一定会出现：

- 用户偏好和业务事实混召回
- 历史事件被当成稳定规则
- 过期流程继续影响新任务
- 用户画像越来越不可解释

### 4. Episodic Memory：情景记忆

Episodic Memory 存的是具体发生过的事件。

例如：

- 用户 2026-03-15 问过一次 K8s OOM 排查
- Agent 当时调用了 metrics、events、logs 三个工具
- 最后结论是 memory limit 设置过低
- 用户否决过某个方案，原因是上线风险太高

它适合的存储方式通常是组合：

| 介质 | 存什么 | 作用 |
| --- | --- | --- |
| 对象存储 | 原始对话、工具 trace、长日志 | 审计和回放 |
| OLTP | 摘要、时间、用户、重要度、来源 | 精确过滤 |
| 向量库 | 摘要 embedding | 语义召回 |

这不是“同一条记忆存三份浪费空间”，而是同一件事的三个视图。

典型表结构可以是：

```sql
CREATE TABLE episodic_memories (
  episode_id     UUID PRIMARY KEY,
  user_id        BIGINT,
  session_id     UUID,
  summary        TEXT,
  raw_uri        TEXT,
  embedding      VECTOR(1536),
  importance     NUMERIC(4,3),
  last_accessed  TIMESTAMP,
  created_at     TIMESTAMP
);
```

Episodic Memory 的治理重点是：

- 不要把每轮对话都永久保存
- 原始内容要有访问权限
- 摘要要带来源
- 长期不用的 episode 要衰减或归档

### 5. Semantic Memory：语义事实

Semantic Memory 存的是去掉具体对话语境后仍然成立的事实。

例如：

- 用户偏好 Python
- 公司告警阈值是延迟超过 500ms 升 P1
- 某业务系统的主键字段是 `tenant_id + order_id`

这类记忆最关键的是：

**事实有时间性。**

“用户偏好 Python”可能三个月后变成“用户偏好 Go”。  
如果你只是覆盖旧值，审计时看不到变化历史；如果你只是追加新值，召回时会打架。

更稳的做法是带有效期：

```sql
CREATE TABLE semantic_facts (
  fact_id      UUID PRIMARY KEY,
  subject      VARCHAR(128),
  predicate    VARCHAR(64),
  object       TEXT,
  confidence   NUMERIC(4,3),
  source_ref   TEXT,
  valid_from   TIMESTAMP,
  valid_to     TIMESTAMP,
  embedding    VECTOR(1536),
  UNIQUE(subject, predicate, object, valid_from)
);
```

当用户偏好变化时，不是简单删除旧事实，而是：

- 旧事实 `valid_to = now()`
- 新事实 `valid_from = now()`
- 保留来源和时间线

这就是很多 Temporal KG / memory 系统解决“用户改主意”的核心思路。

### 5b. 语义事实和用户画像为什么看起来很像

你会觉得 Semantic Memory 和 Profile Memory 像，是非常正常的。

因为它们都可能保存类似这种信息：

- 用户偏好 Go
- 用户喜欢简洁回答
- 用户是后端工程师
- 用户当前项目是 Talent Hub

但它们的系统角色不同。

一句话区分：

**Semantic Memory 是事实库，Profile Memory 是主体摘要。**

再具体一点：

| 维度 | Semantic Memory | Profile Memory |
| --- | --- | --- |
| 核心问题 | 关于某个主体，有哪些可追溯事实？ | 当前要如何快速理解这个用户 / 组织？ |
| 数据形态 | 多条 fact，带来源、时间、置信度 | 小体积画像摘要，通常是 JSON |
| 读取方式 | 按 query / subject / predicate 检索 | 每轮按 scope 全量加载 |
| 更新方式 | ADD / UPDATE / 失效旧事实 | 聚合多个事实后更新摘要字段 |
| 作用 | 事实推理、冲突处理、追溯来源 | 个性化、默认约束、交互风格 |
| 风险 | 事实过期、冲突、来源不明 | 过度概括、标签化、隐私风险 |

举个例子。

用户在三次对话里分别说：

```text
2026-03-01：我平时主要写 Java。
2026-04-10：最近项目里开始用 Go。
2026-05-07：以后 Talent Hub 相关示例尽量用 Go。
```

Semantic Memory 更适合保存成事实时间线：

```text
fact_1: user prefers Java for general backend work, valid_from=2026-03-01
fact_2: user uses Go in recent project, valid_from=2026-04-10
fact_3: user prefers Go examples in Talent Hub scope, valid_from=2026-05-07
```

Profile Memory 更适合保存成当前摘要：

```json
{
  "default_language": "Java",
  "project_preferences": {
    "talent_hub": {
      "example_language": "Go"
    }
  }
}
```

所以 Profile 通常不是原始事实，而是从多条 Semantic Facts 里归纳出来的“当前可用摘要”。

这也解释了为什么 Profile 必须谨慎：

- Semantic Fact 可以保留多版本和来源
- Profile 是高压缩摘要，容易把复杂事实压扁
- Profile 每轮都可能进上下文，影响面更大

### 5c. 一条信息应该进 Semantic 还是 Profile

可以用下面这组问题判断。

适合写入 Semantic Memory：

- 这是一个明确事实吗？
- 需要保留来源和时间吗？
- 未来可能和新事实冲突吗？
- 需要按 subject / predicate 检索吗？
- 需要支持“以前是这样，现在不是”吗？

适合写入 Profile Memory：

- 这条信息是否稳定到可以作为默认行为？
- 体积是否足够小，适合每轮加载？
- 用户是否应该能直接看到和编辑它？
- 它是否会影响交互风格、默认工具、默认约束？
- 它是否已经被多次验证，而不是一次性推断？

一个实用规则是：

**先写 Semantic Fact，稳定后再汇总进 Profile。**

比如：

- 第一次听到“我最近用 Go” -> 写 Semantic，置信度中等
- 多次在同一项目里确认“示例用 Go” -> 更新 project-scoped Profile
- 用户明确说“以后都默认用 Go” -> 可以直接更新 Profile，但仍保留 Semantic 来源

这样系统既能有个性化，又不会把一次对话里的临时偏好误当成长期画像。

### 6. Procedural Memory：过程记忆

Procedural Memory 存的是做一类任务的套路。

例如：

- 排查 Pod OOM：先看 metrics，再看 events，再看 logs
- 发布流程：改配置，跑测试，提 PR，审批，部署
- 代码 review：先看 diff，再看测试，再看风险文件

它本质上和 prompt 版本化、工作流模板、playbook 很接近。

适合的存储方式通常是：

- Git / 文档库保存模板
- OLTP 记录 procedure metadata
- 向量索引用来按任务相似度召回

小规模系统不一定要上向量检索。  
如果 procedure 只有几十个，用标签、任务类型、规则匹配就够。

### 6b. 为什么有了过程记忆，还需要情景记忆

这个问题很关键：

**如果 Procedural Memory 已经记录了“怎么做”，Episodic Memory 还有什么用？**

答案是：

- Procedural Memory 记录的是“通用套路”
- Episodic Memory 记录的是“具体发生过的案例”

它们解决的问题不一样。

过程记忆更像 SOP：

```text
排查 Pod OOM：
1. 看 metrics
2. 看 events
3. 看 logs
4. 看 deployment yaml
```

情景记忆更像历史事故案例：

```text
2026-03-15，talent-hub 的 worker Pod OOM。
当时 metrics 看起来只是轻微上涨，但 events 里反复出现 OOMKilled。
最后发现不是代码泄漏，而是新导入任务把 batch size 调大了。
用户当时拒绝了“直接扩容”方案，因为成本预算紧。
```

这两类信息进入 Agent 后，作用完全不同。

| 维度 | Procedural Memory | Episodic Memory |
| --- | --- | --- |
| 核心问题 | 通常应该按什么步骤做？ | 过去类似场景发生过什么？ |
| 形态 | 流程、模板、playbook | 案例、经历、trace、复盘 |
| 稳定性 | 相对稳定 | 大量增长，容易过期 |
| 召回方式 | 按任务类型 / 标签命中 | 按语义相似度 + 时间 + 重要度召回 |
| 对当前任务的价值 | 给执行顺序 | 给异常线索、上下文偏好、历史坑点 |

所以只保留过程记忆会有一个问题：

**Agent 知道“标准流程”，但不知道“这个用户 / 这个项目 / 这类事故以前怎么特殊”。**

举几个例子就清楚了。

#### 例子 1：研发助手

Procedural Memory 告诉 Agent：

- review PR 先看 diff
- 再看测试
- 再看数据库迁移
- 最后总结风险

Episodic Memory 告诉 Agent：

- 上次这个项目的 PR 出问题，是因为迁移脚本没有 rollback
- 用户上次特别在意“不要引入新依赖”
- 某个模块过去两次 bug 都和时区处理有关

前者让 Agent 会做 review。  
后者让 Agent review 得更像“熟悉这个项目的人”。

#### 例子 2：运维排障

Procedural Memory 告诉 Agent：

- 先看指标
- 再看日志
- 再看事件
- 再看最近发布

Episodic Memory 告诉 Agent：

- 上次类似延迟升高不是数据库问题，而是下游限流
- 当时错误日志关键词是 `upstream timeout`
- 值班同学曾经排除过某个方向

前者提供排障路径。  
后者提供排障直觉。

#### 例子 3：用户偏好

Procedural Memory 告诉 Agent：

- 做系统设计题时按“需求 -> 架构 -> 存储 -> 一致性 -> 可观测性”展开

Episodic Memory 告诉 Agent：

- 用户上次练系统设计时卡在“容量估算”
- 用户不喜欢一上来画大图，希望先讲业务边界
- 上次某个答案被用户认为“太模板化”

前者是通用答题套路。  
后者是这个用户的学习轨迹。

### 6c. 情景记忆什么时候可以不存

不是所有案例都值得写成 Episodic Memory。

如果一个事件只是普通执行过程，没有复用价值，就不该长期保存。

比如：

- 一次普通工具调用成功
- 一次没有新信息的问答
- 一个临时查询结果
- 一段没有后续价值的闲聊

Episodic Memory 应该优先保存这些情况：

- 这次经历暴露了一个未来容易复用的坑
- 用户明确否决或偏好过某个方案
- 过程里有可复盘的工具调用链路
- 这次结果和标准流程不同
- 这次经历可以帮助未来避免重复排查

一句话判断：

**如果它只能告诉你“当时发生了什么”，价值不够；如果它能帮助未来类似任务少走弯路，才值得存。**

### 6d. 情景记忆怎么变成过程记忆

还有一个很重要的关系：

**多个高价值情景记忆，可能会沉淀成新的过程记忆。**

例如系统积累了三次类似 episode：

- OOM 案例 1：batch size 变大导致内存超限
- OOM 案例 2：图片解析任务没有分块处理
- OOM 案例 3：worker 并发数提高后触发 OOMKilled

这时可以把它们总结成新的 Procedural Memory：

```text
排查 Worker OOM 时，除了常规 metrics/events/logs，还要检查：
1. 最近 batch size 是否变化
2. 单任务输入是否变大
3. worker concurrency 是否被调高
4. 是否有大对象未分块处理
```

所以两者不是替代关系，而是上下游关系：

```text
Episodic Memory = 原始案例和经验素材
Procedural Memory = 从多个案例里提炼出的稳定套路
```

情景记忆是“经验原料”，过程记忆是“经验提炼”。

### 7. Profile Memory：用户 / 组织画像

Profile 存的是相对稳定的用户或组织属性。

例如：

- 用户角色
- 回复风格偏好
- 常用技术栈
- 组织合规约束
- 禁用工具列表

它通常不需要向量检索，因为体积小，每次可以全量加载：

```sql
CREATE TABLE user_profile (
  user_id      BIGINT PRIMARY KEY,
  preferences  JSONB,
  attributes   JSONB,
  constraints  JSONB,
  updated_at   TIMESTAMP
);
```

但 Profile 是最危险的记忆之一。

因为它很容易从“用户偏好”滑向“系统对用户的猜测”。

治理原则是：

- 用户可见
- 用户可编辑
- 用户可删除
- 来源可追溯
- 推断类画像要标注置信度
- 敏感画像默认不写入

### 8. 写入判定为什么是第一关键点

如果没有写入判定，系统通常会退化成：

- 会话日志仓库
- 噪声垃圾堆
- 过期事实集合
- 隐私风险池

所以不是“每轮都写”，而是要做一条写入流水线：

```text
原始交互
  ↓
候选提取
  ↓
价值判断
  ↓
类型分类
  ↓
结构化归一
  ↓
敏感信息检查
  ↓
冲突检测
  ↓
最终写入 / 更新 / 忽略 / 删除
```

这个流水线里最重要的是：写入动作要有 CRUD 语义。

也就是说，模型或规则系统不应该只能 append，还应该能输出：

- `ADD`
- `UPDATE`
- `DELETE`
- `NOOP`

例如：

```json
{
  "operation": "UPDATE",
  "memory_type": "semantic",
  "subject": "user:123",
  "predicate": "preferred_language",
  "object": "Go",
  "reason": "User explicitly said they now prefer Go for backend work",
  "source_ref": "episode:abc"
}
```

这比“发现一句话就 append 到 memory”靠谱很多。

### 9. 哪些信息通常值得记

比较适合长期沉淀的包括：

- 稳定偏好
- 长期约束
- 重复出现的实体关系
- 高价值经验总结
- 跨会话任务进展
- 用户明确要求“记住”的信息
- 被多次验证过的事实

一个简单判断是：

**未来复用概率高、过期风险可控、权限风险可控，才值得进长期记忆。**

### 10. 哪些信息通常不应该记

不建议进入长期记忆的包括：

- 一次性情绪表达
- 临时故障状态
- 未验证推断
- 冗长原始对话
- 高敏感原文数据
- 密钥、token、cookie
- 不必要的身份信息
- 模型自己猜出来但用户没确认的画像

尤其要警惕一句话：

**“看起来有用”不等于“应该长期保存”。**

### 11. 读取策略：不要把记忆全塞进上下文

一次推理前，要决定读哪些记忆进上下文。

常见组合是：

1. **Always-load**
   - Profile 体积小，每轮全量注入
2. **Retrieval**
   - Semantic + Episodic 按当前 query 做 top-k 检索
3. **Procedure match**
   - Procedural 按任务类型或相似度命中
4. **Salience score**
   - Episodic 按重要度、近因性、访问频率综合排序

一个常用公式可以是：

```text
score = semantic_similarity * importance * recency_decay * permission_factor
```

记忆窗口也要有预算。

经验上，长期记忆不要占满上下文，通常控制在上下文的 10-20% 更稳。  
否则记忆会干扰当前任务，把 Agent 拉回旧上下文里。

### 12. 记忆冲突为什么会越来越频繁

随着使用时间增长，你几乎必然会遇到：

- 用户以前喜欢中文，现在要求英文
- 老项目结构变了
- 旧 SOP 被新流程替代
- 上次总结和本次事实不一致
- 组织制度发生了版本变更

如果记忆系统没有冲突解决机制，就会频繁召回错误信息。

### 13. 冲突解决的常见策略

可以从四个维度判断：

- 时间新旧
- 来源可信度
- 明确覆盖关系
- 是否需要人工确认

常见策略包括：

| 策略 | 适合场景 |
| --- | --- |
| 新事实覆盖旧事实 | 用户偏好、短期项目状态 |
| 保留版本时间线 | 业务规则、组织制度、流程 SOP |
| 按来源可信度选择 | 用户显式确认 > 模型抽取 > 推测 |
| 人工确认 | 法务、财务、合规、高风险画像 |

一个典型例子：

> 用户半年前说喜欢 Python，现在说“我后端主要用 Go 了”。

更稳的处理不是直接删除 Python 偏好，而是：

- 让 Python 偏好失效
- 写入 Go 偏好
- 保留两个事实的来源和时间

这样未来如果需要回溯，就知道偏好什么时候发生了变化。

### 14. 遗忘机制为什么是必须的

记忆系统如果只会增长，不会遗忘，最终一定出问题。

遗忘的原因包括：

- 过时信息继续召回会误导决策
- 记忆无限增长会降低召回精度
- 用户隐私和合规要求必须支持删除
- 旧事实和新事实会持续冲突

常见遗忘策略包括：

1. **TTL**
   - 临时偏好、低价值 episode 到期自动归档
2. **Decay scoring**
   - 长期未访问的记忆分数自然下降
3. **Explicit delete**
   - 用户或合规工单触发明确删除
4. **Invalidation**
   - 旧事实不物理删除，但标记 `valid_to`
5. **Consolidation**
   - 多条相似记忆合并成一条摘要

工程里最容易漏的是级联删除。

如果向量库删了但原文没删，会留下隐私风险。  
如果原文删了但向量没删，会召回一个指向空内容的 id。

所以删除应该覆盖：

- 原始 trace
- OLTP 元数据
- 向量索引
- 画像字段
- 知识图谱边
- 派生摘要

### 15. 用户画像为什么危险

用户画像看起来很有价值，但也最容易变成风险源。

常见风险：

- 用户画像过度推断
- 私密信息长期保存
- 模型把“猜测”当成事实
- 业务策略与真实用户意图不一致
- 用户不知道系统记住了什么
- 不同场景画像混用，例如工作偏好影响生活咨询

所以用户画像应尽量：

- 结构化
- 可解释
- 可删除
- 可覆盖
- 可审计
- 分作用域

一个实用做法是给画像加 scope：

```json
{
  "scope": "work_project:talent_hub",
  "preference": "prefers concise architecture summaries",
  "source_ref": "episode:abc",
  "confidence": 0.86,
  "updated_at": "2026-05-07T10:00:00Z"
}
```

这样不会把某个项目里的偏好无脑带到所有对话里。

### 16. 记忆系统和 RAG 最大的区别

RAG 面向外部知识。  
Memory 面向内部持续关系和状态。

如果把这两者混在一起，系统就会出现：

- 外部知识和用户偏好相互污染
- 召回时难以做优先级判断
- 一旦错误写入，影响范围过大
- 公共文档和个人画像权限边界混乱

可以这么记：

- RAG 回答“世界 / 组织里有什么知识”
- Memory 回答“这个用户 / 这个 Agent / 这类任务过去发生过什么”

### 17. 真实系统怎么做的

几个有代表性的设计可以作为参考：

| 系统 | 分层 | 核心机制 | 值得学习的点 |
| --- | --- | --- | --- |
| MemGPT / Letta | Core memory + Archival memory | 模型显式 recall 外部记忆 | 把记忆当成可调度资源 |
| Mem0 | Episodic + Semantic | 抽取后做 `ADD / UPDATE / DELETE / NOOP` | 写入有 CRUD 语义 |
| Zep | 消息层 + 事实层 + 时间知识图谱 | 每条事实有时间边界 | 解决“用户改主意”问题 |
| ChatGPT Memory | 用户级 fact list | 用户可见可编辑 | 合规和用户控制 |
| Cursor / Windsurf | Repo-scoped rules | 项目级规则文件化 | 作用域隔离 |

不要为了“看起来先进”照搬全套。

更重要的是偷它们的设计原则：

- 记忆要有类型
- 写入要有判断
- 事实要有来源和时间
- 用户要能管理自己的记忆
- 记忆要能失效和删除

### 18. 一条原始事件进来，怎么落地

假设发生了这次交互：

```text
用户问：K8s Pod OOM 怎么排查？
Agent 调 metrics / events / logs，结论是 memory limit 太低。
用户顺口说：我一般用 Go 写运维脚本。
```

记忆 extractor 可以这样处理：

```text
1. 这是一次具体交互？
   -> 写 Episodic：摘要、trace、embedding

2. 对话里有稳定事实？
   -> 写 Semantic：user prefers Go for ops scripts

3. 走通了可复用流程？
   -> 写 Procedural：OOM 排查流程

4. 用户画像要更新？
   -> 如果 Go 偏好多次出现，更新 Profile
```

注意：不是把整段对话塞进长期记忆。  
而是把同一个事件拆成不同性质的记忆条目。

### 19. 记忆系统最稳的设计原则

1. 分层存储
2. 只沉淀高价值信息
3. 记忆条目带来源
4. 记忆条目带时间
5. 支持覆盖和失效
6. 不保存不必要的敏感原文
7. 用户画像分作用域
8. 删除必须级联
9. 读取必须有预算
10. 高风险记忆写入需要人工确认或明确来源

### 20. 记忆写入流水线应该怎么拆

如果要把 Memory 做成工程系统，不建议让模型在对话里直接“随手写库”。

更稳的方式是把写入拆成一条独立 pipeline：

```text
conversation / trace
        ↓
候选提取 Candidate Extraction
        ↓
价值评分 Value Scoring
        ↓
类型分类 Type Classification
        ↓
敏感信息和权限检查 Safety Check
        ↓
冲突检测 Conflict Resolution
        ↓
写入决策 ADD / UPDATE / DELETE / NOOP
        ↓
审计日志 Audit Log
```

每一步的职责不同：

| 阶段 | 解决什么问题 |
| --- | --- |
| 候选提取 | 从原始对话中找出“可能值得记”的信息 |
| 价值评分 | 判断这条信息未来是否值得复用 |
| 类型分类 | 决定写入 Profile、Semantic、Procedural 还是 Episodic |
| 安全检查 | 判断是否包含敏感信息、越权内容、不可保存内容 |
| 冲突检测 | 查已有记忆，看是否需要更新或失效旧记忆 |
| 写入决策 | 输出明确的 CRUD 动作 |
| 审计日志 | 记录为什么写、谁触发、来源是什么 |

这条 pipeline 最好是异步的。

原因是：

- 不阻塞用户当前对话
- 可以批量处理，降低 LLM 抽取成本
- 可以接人工审核队列
- 可以在写入前做更完整的冲突检测

但有一类场景可以同步写：

- 用户明确说“记住这件事”
- 用户明确修改自己的偏好
- 当前任务必须依赖这条记忆继续执行

也就是说，默认异步，显式记忆同步。

### 21. 记忆条目的最小字段

不管底层是 Postgres、向量库还是知识图谱，抽象到系统层，记忆条目至少应该有这些字段：

```text
memory_id
memory_type
scope
subject
predicate
content
confidence
source_ref
source_type
created_by
valid_from
valid_to
sensitivity_level
status
embedding_ref
created_at
updated_at
```

这些字段分别解决不同问题：

| 字段 | 作用 |
| --- | --- |
| `memory_type` | 区分 Profile / Semantic / Procedural / Episodic |
| `scope` | 限定记忆在哪个范围内可用 |
| `subject` | 记忆绑定的主体，例如 user / org / project |
| `predicate` | 事实关系，例如 prefers / uses / forbids |
| `content` | 可给模型看的自然语言摘要 |
| `confidence` | 这条记忆有多可信 |
| `source_ref` | 回指到对话、文档、工具结果或人工编辑 |
| `created_by` | 是用户、系统、模型还是管理员写入 |
| `valid_from / valid_to` | 支持事实生效和失效 |
| `sensitivity_level` | 控制是否能进入模型上下文 |
| `status` | active / expired / deleted / pending_review |

这里最容易被低估的是 `scope`。

例如用户在某个工作项目里说“我喜欢回答简短一点”，不一定代表他在学习笔记、生活咨询、面试模拟里都喜欢简短。  
如果 scope 不清楚，记忆会跨场景污染。

一个更具体的 scope 可以长这样：

```json
{
  "scope_type": "project",
  "scope_id": "talent_hub",
  "tenant_id": "org_123",
  "user_id": "u_456"
}
```

### 22. 写入判定可以做成评分模型

写入判定不要只靠一句 prompt：

> “请判断这条信息是否值得记住。”

这太粗了。

更稳的是拆成几个维度打分：

| 维度 | 问题 | 高分含义 |
| --- | --- | --- |
| 复用价值 | 未来还会用到吗 | 多任务、多会话都可能复用 |
| 稳定性 | 它会不会很快过期 | 短期内比较稳定 |
| 可信度 | 来源可靠吗 | 用户明确说过或工具验证过 |
| 作用域清晰度 | 适用范围清楚吗 | 明确绑定用户、组织、项目或任务类型 |
| 敏感风险 | 保存后风险高吗 | 分数越高风险越大 |
| 冲突风险 | 是否和已有记忆打架 | 分数越高越需要谨慎 |

然后用一个决策规则：

```text
write_score = reuse_value * stability * confidence * scope_clarity
risk_score  = sensitivity_risk + conflict_risk

if risk_score >= high:
    pending_review
elif write_score >= threshold and risk_score < medium:
    write_or_update
else:
    noop
```

这类规则的好处是：

- 可解释
- 可调参
- 可审计
- 能逐步接评测

例如一条“用户今天有点烦”：

- 复用价值低
- 稳定性低
- 敏感风险中
- 结果应该是 `NOOP`

一条“用户明确说以后这个项目默认用 Go 写后端示例”：

- 复用价值高
- 稳定性中高
- 来源可信
- scope 清楚
- 结果可以是 `ADD / UPDATE`

### 23. 记忆读取链路也要分层

写入治理很重要，但读取治理同样重要。

很多记忆系统变脏，不是因为写错了，而是因为读的时候不加区分，把不该出现的记忆塞进了上下文。

一条稳妥的读取链路是：

```text
当前任务
  ↓
构造 memory query
  ↓
按 tenant / user / project 做权限过滤
  ↓
按 memory_type 分路召回
  ↓
重排和去重
  ↓
敏感级别过滤
  ↓
压缩成上下文片段
  ↓
注入 prompt
```

注意顺序：

**权限过滤必须尽量前置。**

如果你先在全库向量检索，再在结果里过滤权限，理论上仍可能出现侧信道风险，比如模型看到相似但无权访问的标题或摘要。  
更稳的做法是向量检索也带 metadata filter：

- `tenant_id`
- `user_id`
- `project_id`
- `scope_type`
- `visibility`
- `status = active`

然后再做二次权限过滤。

### 24. 不同类型记忆的读取方式不同

不要对四类记忆都用同一个 top-k 向量检索。

| 类型 | 推荐读取方式 |
| --- | --- |
| Profile | 小体积，全量加载，但要按 scope 过滤 |
| Semantic | 精确过滤 + 向量检索混合 |
| Procedural | 先按任务类型 / 标签匹配，再按相似度补充 |
| Episodic | 语义检索 + 时间衰减 + 重要度排序 |

举个例子：

用户问：“帮我 review 这个 Go 项目的 PR。”

比较合理的读取组合是：

- Profile：用户偏好“先说风险，再说建议”
- Semantic：该项目约束“不能引入新的外部依赖”
- Procedural：代码 review 的固定流程
- Episodic：过去同类 PR review 中踩过的坑

这些记忆进入上下文时也不应该混成一坨。

更好的 prompt 片段是分块：

```text
[User Profile]
- prefers concise review summaries

[Project Constraints]
- no new external dependencies without approval

[Relevant Procedures]
- review order: diff -> tests -> risk files -> migration impact

[Similar Past Episodes]
- previous PR failed because migration script missed rollback
```

分块可以帮助模型理解每类信息的权重。

### 25. 记忆进入上下文前要做压缩

长期记忆如果原样塞进 prompt，会带来两个问题：

- token 很快失控
- 模型被历史细节带偏

所以读取结果最好再做一次上下文压缩。

压缩原则是：

- Profile 保留短句
- Semantic 保留事实和来源
- Procedural 保留步骤，不保留长解释
- Episodic 保留结论、时间和和当前任务的关系

例如原始 episode 是：

```text
用户上次排查 OOM 时，Agent 先查了 metrics，发现 memory usage 持续接近 limit，
然后查 events 发现 OOMKilled，最后查 deployment yaml，确认 memory limit 设置过低。
```

进入上下文可以压成：

```text
Past relevant episode:
- Similar OOM case was caused by memory limit being too low.
- Useful path: metrics -> events -> deployment yaml.
- Source: episode_123.
```

压缩不是为了“少说”，而是为了让记忆变成当前任务可用的线索。

### 26. 记忆系统要有评估指标

Memory 如果没有评估，会很快变成玄学。

你至少要看这些指标：

| 指标 | 含义 |
| --- | --- |
| 写入准确率 | 写进去的记忆有多少真的该写 |
| 误写率 | 不该记的内容被写入的比例 |
| 漏写率 | 应该记的重要信息没有写入的比例 |
| 过期召回率 | 已失效记忆仍被召回的比例 |
| 有用召回率 | 被召回记忆对最终回答有帮助的比例 |
| 权限违规召回数 | 无权记忆被读出的次数 |
| 删除完成时间 | 用户删除请求多久能全链路生效 |
| 记忆膨胀率 | 每个用户 / 项目的记忆增长速度 |

其中最关键的不是“记忆数量”，而是：

- 写得准
- 读得准
- 删得干净
- 过期的不再影响决策

### 27. 记忆评测集怎么做

一个小型 memory eval 可以这样构造：

1. 准备 20 到 50 段模拟对话
2. 每段对话标注应该写入哪些记忆
3. 标注不应该写入哪些内容
4. 加入偏好变化、事实冲突、敏感信息、项目 scope 等干扰项
5. 让 extractor 输出 CRUD 决策
6. 人工或规则评估是否正确

样例可以这样设计：

| 对话内容 | 正确记忆动作 |
| --- | --- |
| “以后这个项目示例都用 Go。” | ADD project-scoped preference |
| “我今天有点累。” | NOOP |
| “把我的手机号记一下：138...” | pending_review 或拒绝长期保存 |
| “之前说用 Python，现在这个项目改用 Go。” | UPDATE 旧偏好并失效旧事实 |
| “这次故障只是临时缓存抖动。” | 不进入长期记忆，最多写 episode 摘要 |

这套评测集不需要一开始很大，但必须持续跑。  
否则每次改 prompt 或阈值，都不知道 memory 系统到底变好了还是变脏了。

### 28. 记忆系统的权限和合规设计

Memory 的权限风险比普通 RAG 更隐蔽。

因为 RAG 通常是读外部知识，而 Memory 会沉淀用户长期行为和偏好。

最低限度要做到：

- 用户能查看系统记住了什么
- 用户能删除自己的记忆
- 用户能关闭某个 scope 的记忆
- 记忆条目带来源
- 敏感记忆默认不写入
- 管理员访问要有审计
- 删除要级联到向量库和派生摘要

对于企业系统，还要注意多租户隔离：

- 不同 tenant 的 memory 不能共用检索空间
- embedding metadata 必须带 tenant filter
- 管理后台不要默认展示原文
- 跨项目记忆共享必须显式配置

一个保守默认值是：

**记忆默认私有，跨 scope 使用必须显式授权。**

### 29. 记忆系统的落地路线

不要一上来就做完整 Memory 平台。

比较稳的路线是四步：

#### 第一阶段：用户显式记忆

只支持用户明确说：

- “记住我偏好 X”
- “以后这个项目都用 Y”
- “忘掉刚才那条”

这阶段重点是：

- 用户可见
- 用户可删
- scope 清楚
- 不自动推断敏感信息

#### 第二阶段：异步抽取

会话结束后异步抽取候选记忆。

这阶段重点是：

- `ADD / UPDATE / DELETE / NOOP`
- 写入评分
- 冲突检测
- 敏感信息过滤

#### 第三阶段：读取重排

让记忆真正参与上下文，但必须有读取预算和权限过滤。

这阶段重点是：

- 按 type 分路召回
- metadata filter
- 记忆压缩
- 有用召回率评估

#### 第四阶段：治理平台

当记忆量上来后，再做：

- 用户记忆管理页
- 管理员审计
- 删除工单
- 记忆质量 dashboard
- 高风险记忆审核队列

这条路线的好处是：先保证记忆不会伤害系统，再追求越用越聪明。

### 30. 面试和系统设计里的表达框架

如果面试里被问：

> “Agent 的长期记忆怎么设计？”

不要只回答“用向量库存起来”。

更好的回答结构是：

1. 先分类型
   - Profile、Semantic、Procedural、Episodic
2. 再讲写入
   - 候选提取、评分、敏感检查、冲突检测、CRUD 决策
3. 再讲读取
   - 按 scope 过滤、按类型召回、重排、压缩、预算控制
4. 再讲治理
   - 来源、时间、失效、删除、用户可见、审计
5. 最后讲评估
   - 误写率、漏写率、有用召回率、过期召回率、删除 SLA

30 秒版本可以这样说：

> “我不会把 memory 当成一个简单向量库。长期记忆先分 Profile、Semantic、Procedural、Episodic 四类；写入走候选抽取、价值评分、敏感检查、冲突检测，最后输出 ADD/UPDATE/DELETE/NOOP；读取时先按 tenant/user/project 做权限过滤，再按类型召回和压缩；每条记忆都有 source、confidence、valid_from/valid_to 和 scope；同时支持用户查看、删除和级联清理。评估上看误写率、漏写率、有用召回率、过期召回率和删除完成时间。”

## 小结

长期记忆不是“越多越好”。  
真正成熟的记忆系统，更像一个带治理能力的知识面板。

第 3 课里你只需要记住：长期记忆是跨任务可复用的信息。  
到了本课，你要进一步掌握：长期记忆必须有类型、来源、时间、作用域、冲突处理和删除机制。

如果这些都没有，Memory 不会让 Agent 越用越聪明，只会让它越用越脏。
