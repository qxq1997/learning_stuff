# AI Agent - 第 26 课：稳定性工程：超时、重试、限流、背压、降级与熔断

## 学习目标

- 理解为什么 Agent 比传统同步接口更容易出现稳定性事故。
- 掌握超时、重试、限流、背压、降级、熔断这些稳定性手段在 Agent 场景里的具体用法。
- 学会识别“看起来像模型问题，实际上是系统稳定性问题”的典型症状。

## 内容讲解

### 1. Agent 为什么天然更脆

因为它同时放大了三类不稳定性：

- 模型推理时间不稳定
- 工具调用链长
- 上下文和外部依赖更多

所以一个 Agent 请求，常常不是一次 RPC，而是一串不稳定步骤的组合。

### 2. 超时为什么是第一守门员

没有超时，系统很容易：

- 卡死
- 队列堆积
- 资源耗尽
- 用户长时间无响应

Agent 场景通常至少要有三层超时：

- 单次模型调用超时
- 单次工具调用超时
- 整个任务生命周期超时

### 3. 重试为什么不是越多越好

很多故障不是因为没重试，而是因为重试太狠。

常见后果：

- 下游服务被打爆
- 队列放大
- 成本暴涨
- 同一副作用动作重复执行

所以重试一定要配：

- 可重试错误分类
- 指数退避
- 最大次数
- 幂等保证

### 4. 限流和背压在 Agent 里尤其重要

传统接口通常一进一出。  
Agent 往往一进多出：

- 一个请求可能触发多个工具
- 一个任务可能触发多个模型调用

所以当流量上来时，放大的不是 QPS，而是系统内部工作量。

这时如果没有背压机制，最常见结果就是：

- 上游还在接请求
- 下游已经排不动
- 整体时延雪崩

### 5. 降级为什么不是失败，而是系统成熟

Agent 场景非常适合做分级降级。

例如：

- 关闭次要工具
- 从多步规划退化为单轮回答
- 从大模型退到小模型
- 从在线检索退到本地缓存知识

如果没有降级，系统只能在“完全可用”和“完全不可用”之间跳。

### 6. 熔断为什么能防止事故扩大

当某个依赖已经明显不稳时，继续调用它只会让全链路更差。

熔断的意义是：

- 快速失败
- 保护核心资源
- 等待依赖恢复

尤其在 Agent 多工具链路里，熔断可以防止单个坏依赖把整个系统拖死。

### 7. 常见稳定性事故图谱

1. 模型时延突增
2. 检索服务抖动
3. 工具依赖 429
4. 队列积压
5. 任务超时未终止
6. 重试风暴
7. 降级缺失导致整条链崩

## 小结

Agent 的稳定性工程，本质上不是“让模型稳定”，而是：

- 让系统知道什么时候该等
- 什么时候该停
- 什么时候该少做一点
- 什么时候该快速失败
