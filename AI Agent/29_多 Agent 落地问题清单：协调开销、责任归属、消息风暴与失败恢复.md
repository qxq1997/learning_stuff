# AI Agent - 第 29 课：多 Agent 落地问题清单：协调开销、责任归属、消息风暴与失败恢复

## 学习目标

- 理解多 Agent 为什么常常不是“能力翻倍”，而是“复杂度翻倍”。
- 掌握多 Agent 系统最常见的失败模式。
- 学会判断什么时候该拆 Agent，什么时候其实应该合并回单 Agent 或工作流。

## 内容讲解

### 1. 多 Agent 最常见的误区

很多系统上多 Agent，不是因为问题真的需要协作，而是因为：

- 听起来先进
- 结构上看起来像组织分工
- demo 更容易显得复杂

但在工程里，多 Agent 的代价包括：

- 更多上下文同步
- 更多消息传递
- 更多失败点
- 更难调试
- 更难归责

### 2. 多 Agent 真正适合的场景

通常至少满足：

- 角色边界明确
- 输入输出契约清晰
- 子任务相对独立
- 失败可隔离

### 3. 最常见的失败模式

1. 角色漂移
2. 职责重叠
3. 消息风暴
4. 彼此重复检索
5. 互相甩锅
6. 故障恢复链过长

### 4. 责任归属为什么在多 Agent 中尤其麻烦

单 Agent 出错，至少知道“这一条链错了”。  
多 Agent 出错时，往往要追：

- 谁做的决策
- 谁传错了上下文
- 谁丢了关键信号
- 谁在下游执行失败却没上报

### 5. 恢复为什么更难

因为你恢复的不再是一个任务，而是一个协作图。

可能需要回答：

- 从哪个 Agent 的哪个检查点恢复？
- 上游结果还有效吗？
- 下游是否已经产生副作用？
- 要不要整个子图回滚？

## 小结

多 Agent 真正值钱的不是“多个脑子”，而是：

- 有必要的分工
- 清晰的边界
- 可控的协作成本

## 你现在应该能回答的问题

1. 多 Agent 为什么常常会让复杂度先于收益到来？
2. 角色漂移和职责重叠分别会导致什么问题？
3. 什么是消息风暴？
4. 多 Agent 的故障恢复为什么比单 Agent 难很多？
5. 什么时候应该把多 Agent 收回成单 Agent？
