# AI考试平台系统设计学习路径

## 学习目标

这个主题不是为了把“LLM API + Web 页面 + 数据库”拼成一个看起来很新的系统，而是为了把你训练到这样一种程度：

- 你能把“AI 出题、AI 判卷、用户可查看考试记录”的平台，讲成一个边界清楚、链路清楚、风险清楚的系统设计题。
- 你能识别这道题真正的难点不是普通 CRUD，也不是单纯高并发，而是`模型不确定性 + 评分公平性 + 异步任务可靠性 + 可审计性`。
- 你能区分“把大模型接进来”和“把大模型变成可上线能力”之间的差别，知道为什么需要约束、校验、回放、人工复核和版本化。
- 你能在面试里把业务链路、技术选型、落地难点和 trade-off 讲得像一个真的设计过 AI 平台的人，而不是停留在“这里调一下 GPT”。

## 重要说明

这道题表面上像一个在线考试系统，但一旦加上 AI 出题和 AI 判卷，系统性质就变了。  
它不再只是“题库 + 作答 + 打分”，而是一个包含以下三类能力的组合系统：

1. `考试交易链路`
   - 创建考试
   - 生成或装配试卷
   - 用户开考、答题、提交
   - 结果发布与考试记录查询

2. `AI 能力链路`
   - 基于大模型生成题目、答案、解析、评分 Rubric
   - 基于大模型对主观题进行评分、摘要和解释
   - 对模型输出做格式校验、质量评估和安全过滤

3. `治理与审计链路`
   - 模型版本、Prompt 版本、评分规则版本管理
   - 失败重试、人工复核、申诉处理
   - 成本控制、风控、防作弊、可观测性

如果你只把它答成“前端一个考试页，后端调大模型出题和判卷”，面试官很容易继续追问：  
怎么保证题目质量？怎么防止同一场考试里不同用户拿到难度完全不同的题？怎么避免评分漂移？模型挂了怎么办？用户申诉时怎么追溯？  
这些才是这道题真正拉开差距的地方。

## 知识图谱

建议把这个主题拆成 6 个层次来学：

1. 业务与对象层
   - 考试模板、试卷、题目、题目版本、评分 Rubric
   - 考试场次、考试实例、答卷、评分结果、考试记录
   - 模型调用日志、评分解释、人工复核记录

2. 主链路设计层
   - AI 出题什么时候发生
   - 试卷如何发布和冻结
   - 用户答题如何自动保存和最终提交
   - 判卷如何异步执行并回写结果

3. AI Agent 约束层
   - 不是让模型自由发挥，而是做受约束的多阶段流水线
   - Prompt 模板、知识检索、结构化输出、规则校验、二次审核
   - 为什么正式考试不适合完全在线即席生成

4. 稳定性与治理层
   - 模型输出漂移
   - 评分公平性
   - 成本和限流
   - 审计与申诉
   - Provider 故障和降级

5. 面试表达层
   - 如何先定义题目边界
   - 如何说清楚同步链路和异步链路
   - 如何讲技术选型背后的取舍
   - 如何把 AI 风险讲成工程问题而不是抽象担忧

6. 实现细节层
   - RAG 具体由哪些包拼起来
   - 文档切块、token 预算、embedding 和 rerank 怎么落地
   - 向量库怎么建模、怎么查、怎么做混合检索
   - 为什么 RAG 经常不是“接上向量库就好了”

4. `04_技术选型、容量估算与Trade_off.md`

## 推荐学习方式

- 先不要急着讨论用哪个模型，先回答“这是练习平台、招聘考试平台，还是高风险认证考试平台”。业务风险不同，系统设计完全不同。
- 每一课都强制自己回答三个问题：谁是真相源、哪些链路必须同步、哪些能力可以异步。
- 对所有 AI 能力都问一句：如果模型错了、慢了、挂了、输出脏数据了，系统怎么处理。
- 讲方案时尽量把`模型输出`和`平台最终状态`分开。模型给的是候选结果，平台要决定什么能进正式数据。

## 学完后你应该能回答的问题

- 为什么正式考试通常不应该在用户点击“开始考试”时临时让大模型现生成整套题？
- 为什么主观题 AI 判卷必须做版本固化、结果留痕和人工复核兜底？
- 用户查看“我的考试记录”时，到底应该看到哪些层次的数据，哪些应该对用户隐藏？
- 当模型升级后，旧考试记录为什么不能被新模型静默重算覆盖？
- 这道题最关键的 trade-off，为什么是公平性、可审计性和成本，而不只是延迟？

## 推进规则

- 我们先从第 1 课开始，把题目边界定清楚
- 每一课末尾都有检查站
- 你回答后，我会判断是继续推进，还是补一节特定追问
- 原则是不跳过关键理解点，尤其是`试卷冻结`、`评分版本化`、`人工复核`和`考试记录可追溯`
