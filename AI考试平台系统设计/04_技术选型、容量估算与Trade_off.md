# AI考试平台系统设计 - 第 4 课：技术选型、容量估算与 Trade-off

## 学习目标（本节结束后你能做到什么）

1. 能根据这道题的核心矛盾给出一套务实的技术选型，而不是只堆热门组件。
2. 能做轻量容量估算，识别真正的压力点在异步判卷和模型成本，而不一定在 API QPS。
3. 能主动讲出几组关键 trade-off，包括预生成与动态生成、规则评分与模型评分、同步出分与异步出分。
4. 能把前几课内容压缩成一段适合面试的完整回答。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

技术选型一定要围绕问题，而不是围绕名词。  
这道题真正要解决的是：

- 试卷和答卷等核心状态要稳定落库
- AI 出题和 AI 判卷要受控、可回放、可治理
- 用户提交和查记录体验要稳定
- 成本和模型延迟不能把主业务拖垮

基于这个目标，一个比较稳的首版技术栈可以是：

- 前端：`React + TypeScript`
- 核心业务后端：`Spring Boot`
- AI 编排服务：`Python FastAPI` 或独立 Worker
- 关系数据库：`MySQL` 或 `PostgreSQL`
- 缓存：`Redis`
- 消息队列：`Kafka` 或 `RocketMQ`
- 对象存储：`S3 / OSS / MinIO`
- 检索能力：`OpenSearch / Elasticsearch` 或 `pgvector`
- 模型网关：自建统一网关，屏蔽不同 Provider 差异

为什么核心业务和 AI 编排可以分开？  
因为这两类服务的工程形态不同。  
考试主业务更看重事务状态、幂等、权限和稳定 API；  
AI 编排更看重 Prompt 模板、模型路由、异步任务、实验迭代和结构化解析。  
如果团队以 Java 为主，主业务放 Spring Boot 很自然；AI 编排单独用 Python，也更方便接模型 SDK、做数据分析和 prompt 实验。当然，如果团队要求统一栈，也可以都用 Java，但要把 AI 流程和交易主链路的边界保持清楚。

数据库怎么选？  
这题我通常优先关系数据库，因为核心状态非常结构化：

- 考试模板
- 试卷快照
- 考试实例
- 答卷状态
- 评分结果
- 复核记录

这些都很适合关系模型、事务更新和索引查询。  
如果答案内容或题目结构本身是半结构化的，可以在关系库里放 JSON 字段，或者把原始大对象放对象存储，数据库里只留元数据和索引字段。

Redis 的职责要讲清楚。  
它适合做：

- 登录态和短期考试会话
- 防重复提交 token
- 热点考试配置缓存
- 限流计数
- 短期计时状态

但它不适合做考试结果真相源，更不适合单独承载正式答卷。  
因为考试平台最怕的是掉数据、结果不一致和无法审计。

消息队列在这题里很关键，因为 AI 链路天然适合异步。  
比如：

- 出题任务
- 解析生成
- 主观题判卷
- 复核任务
- 成绩发布通知

这些都应该通过事件解耦，而不是挂在用户接口同步等待。  
你可以这样总结：`开始考试`、`保存答案`、`提交答卷`属于交易动作；`出题`、`主观题评分`、`解析生成`、`复核`属于异步能力动作。

再做一个简单容量估算。  
假设这是一个招聘考试平台：

- 日活考生：2 万
- 高峰 1 小时内参加考试用户：4000
- 同时在线考试用户：2000
- 每人一场考试 30 题，其中 10 道主观题
- 客观题提交后立即规则评分，主观题异步 AI 评分

如果只看 API 层，压力其实不算夸张。  
比如自动保存按每 10 秒一次估算，2000 人同时在线，大约是 `200 QPS`。  
开始考试、提交答卷、查询记录这些接口，通常也都在普通中后台系统可承受范围内。  
真正的压力点反而是 AI 评分任务量：  
`4000 场 * 10 道主观题 = 4 万次主观题评分调用`。  
这还没算复核、重试、解析生成和实验流量。  
所以这道题更像`异步 AI 任务吞吐和成本控制题`，而不是单纯的接口高并发题。

也正因为如此，你在面试里最好主动讲成本控制。比如：

- 客观题不用模型
- 短答案优先规则或小模型
- 长答案再调用大模型
- 同一场考试的解析可以预生成，不必提交后现算
- 低风险练习模式可用便宜模型，高风险正式考试用更稳的模型

接下来是几组最值得主动讲的 trade-off。

### 1. 预生成试卷 vs 动态生成试卷

预生成的好处是公平、可审计、开考延迟低。  
坏处是个性化弱，占用题库准备成本。  
动态生成的好处是个性化强。  
坏处是难度不一致、质量不可控、审计困难。  
正式考试我优先预生成，练习模式才考虑动态生成。

### 2. 规则评分 vs 模型评分

规则评分便宜、稳定、可解释。  
但它对开放题型覆盖差。  
模型评分覆盖广、能处理复杂语义。  
但成本高、波动大、解释不一定可靠。  
所以稳妥策略是：能规则化的先规则化，不能规则化的再引入模型。

### 3. 同步出分 vs 异步出分

同步出分体验更直接。  
但一旦遇到主观题或模型拥堵，接口延迟和失败率都会上来。  
异步出分更稳，也方便复核和重试。  
通常适合设计成：客观题可同步出部分结果，主观题异步补齐。

### 4. 单一模型 Provider vs 模型网关多路由

单一 Provider 接入快，系统更简单。  
但一旦限流、涨价、故障，平台会很被动。  
模型网关接入成本更高，但能统一鉴权、打点、降级和多模型策略。  
如果这是正式平台，我更倾向一开始就做统一模型网关抽象。

### 5. 全自动评分 vs 自动评分 + 人工复核

全自动评分成本低、效率高。  
但对边缘样本、高价值考试、用户申诉不够稳。  
自动 + 人工复核的成本更高，但更适合正式考试。  
你可以明确说：高风险场景里，AI 是提效工具，不是绝对裁判。

最后给你一段适合面试复述的 60 秒版本：

“这道题我会先把它定义成考试交易系统和 AI 能力平台的组合。正式考试模式下，我会把题目生成和判卷都做成异步流水线，而不是把模型调用直接塞进开始考试和提交接口。核心状态如考试模板、试卷快照、答卷、评分结果放在 MySQL 或 PostgreSQL，Redis 只做会话、限流和热点缓存，Kafka 或 RocketMQ 承接出题、主观题评分和通知等异步任务。AI 层我会做统一模型网关和受约束的编排服务，保证 Prompt、模型版本和评分 Rubric 可追溯。容量上这题通常不是 API 极限吞吐问题，真正压力在主观题评分任务量和 token 成本，所以我会尽量让客观题规则评分、正式试卷预生成、主观题异步出分，并对边缘结果引入人工复核。”  

## 小结（3-5 条关键点）

1. 这道题最稳的架构通常是关系数据库承载真相源，消息队列承接 AI 异步任务，模型网关统一对接外部 LLM。
2. API QPS 往往不是主矛盾，真正压力点在异步判卷吞吐、模型延迟和 token 成本。
3. 正式考试更适合预生成试卷、异步出分和人工复核；练习模式可以更激进地用动态生成。
4. 能规则化的评分尽量不用模型，模型应该只解决规则覆盖不了的问题。
5. 面试里要主动讲 trade-off，尤其是公平性、成本和可审计性的取舍。
