# AI 辅助开发 - 第 1 章：AI 如何帮助开发：SDD、Skills 与生产落地问题

## 学习目标

这章不是讲“AI 可以补全代码、解释代码、写注释”这些基础能力。  
面试里如果只讲这些，很容易显得停留在工具使用层。

这一章要解决的是：

- 如何把 AI 辅助开发讲成一种工程方法论，而不是个人效率小技巧。
- 如何解释 SDD、Agentic Coding、Skills / Superpowers 这些更前沿的实践。
- 如何说明 AI 进入生产研发流程后，会带来哪些新的问题。
- 如何在面试中讲出“我会用 AI，但我也知道怎么控制 AI 风险”的成熟感。

如果只记一句话：

**AI 对开发的真正价值，不是让人少写几行代码，而是把开发过程从“直接实现驱动”推向“规格、上下文、验证和知识沉淀驱动”。**

---

## 先给结论

面试里问“AI 怎么帮助开发”，一个比较有含金量的回答可以是：

> 我不会把 AI 只看成代码补全工具。真正有价值的方向是把 AI 放进完整软件工程流程里：先用 spec 明确目标和验收标准，再让 coding agent 在受控上下文里实现，最后通过测试、review、CI、安全扫描和人工判断做验证。对于可复用的流程，还可以沉淀成 skill 或团队规则，让 AI 不只是一次性回答问题，而是稳定执行团队认可的开发方法。

这里有四个关键词：

| 关键词 | 解决的问题 | 面试里的表达价值 |
| --- | --- | --- |
| SDD | 需求模糊、实现跑偏 | 说明你知道先定义问题，再让 AI 写代码 |
| Agentic Coding | 多文件修改、执行命令、跑测试、开 PR | 说明你了解 AI 工具从 chat 变成开发代理 |
| Skills / Superpowers | 团队经验难复用、每次都要重新 prompt | 说明你关注流程沉淀和工程标准化 |
| Verification | AI 生成快，但正确性难保证 | 说明你有质量意识和生产意识 |

所以这章的核心不是“AI 能做什么”，而是：

**怎么让 AI 在工程约束下可靠地帮助开发。**

## 1. 面试官真正想听什么

当面试官问 AI 辅助开发时，他通常不是想听你列工具名。

他更可能在观察这些能力：

- 你是不是知道 AI 输出有不确定性。
- 你是不是能把任务拆小，而不是把一整个复杂需求丢给 AI。
- 你是不是能定义验收标准，而不是只看代码能不能跑。
- 你是不是会审查 AI 生成的 diff、测试和依赖。
- 你是不是理解生产环境里的安全、权限、成本和责任边界。
- 你是不是能把个人使用经验上升成团队流程。

一个普通回答是：

> 我用 AI 帮我写代码、查 bug、生成测试。

一个更好的回答是：

> 我会按风险等级使用 AI。低风险任务可以让 AI 快速实现；涉及核心业务、数据一致性、安全或生产配置时，我会先写清 spec 和验收标准，把上下文限制在相关模块，让 AI 小步修改，再通过测试、review 和 CI 验证。对于重复流程，比如写单测、做 PR review、处理迁移，我会沉淀成 checklist 或 skill，避免每次靠临时 prompt。

差别在于：  
前者说的是工具能力，后者说的是工程控制力。

## 2. SDD：Spec-Driven Development

SDD 可以理解成 Spec-Driven Development，也就是规格驱动开发。

它的核心不是“多写文档”，而是：

**先把模糊需求变成可审查、可执行、可验证的规格，再让 AI 围绕规格推进实现。**

传统 AI 编码里，很多人会这样做：

```text
一句需求
  -> AI 直接改代码
  -> 人看结果能不能跑
```

SDD 更像这样：

```text
需求意图
  -> specify：写清目标、范围、约束、非目标、验收标准
  -> plan：拆方案、模块、数据流、风险点
  -> tasks：拆成可执行的小任务
  -> implement：AI 执行具体修改
  -> verify：测试、review、CI、人工验收
```

### 2.1 为什么 SDD 适合 AI 时代

因为 LLM 的输出高度依赖上下文。  
如果上下文只有一句“帮我优化一下登录逻辑”，AI 就会用自己的假设补洞。

这些假设可能看起来合理，但不一定符合真实业务。

比如：

- 它可能不知道登录失败次数要触发风控。
- 它可能不知道某个字段是兼容老客户端用的。
- 它可能不知道某个接口不能改返回结构。
- 它可能不知道线上有灰度、审计、权限和监控要求。

SDD 的价值就是减少这种“AI 自行脑补”。

一份好的 spec 至少应该包含：

| 内容 | 作用 |
| --- | --- |
| 背景 | 为什么要做这个需求 |
| 目标 | 做成后应该解决什么问题 |
| 非目标 | 明确这次不做什么，防止范围膨胀 |
| 约束 | 兼容性、性能、安全、数据一致性、接口契约 |
| 验收标准 | 怎么判断做完了 |
| 风险点 | 哪些地方必须人工确认 |
| 测试要求 | 单测、集成测试、回归范围 |

### 2.2 SDD 面试表达

可以这样说：

> 我比较认可 spec-driven 的 AI 开发方式。因为 AI 生成代码的成本很低，但如果需求本身不清楚，生成越快，返工也越快。所以我会先让 AI 帮我把需求整理成 spec，包括目标、非目标、接口约束、兼容性和验收标准。spec 先由人审一遍，再让 AI 拆 plan 和 tasks。这样 AI 执行时不是靠猜，而是围绕明确规格工作。

如果面试官追问“这和普通写需求文档有什么区别”，可以回答：

> 普通文档很多时候只是给人看的，写完后和代码脱节。SDD 更强调 spec 是后续实现、测试、review 的 source of truth。AI 后续生成代码、测试和 checklist 都围绕这份 spec 展开，所以 spec 不只是文档，而是开发过程里的控制面。

### 2.3 SDD 的边界

SDD 不是银弹。

它也会有问题：

- spec 写错了，AI 会更稳定地朝错误方向执行。
- spec 太大太泛，会变成形式主义。
- spec 没有验收标准，还是无法判断产物是否正确。
- spec 没有人 review，会把需求评审问题后移到代码阶段。

所以 SDD 的关键不是“先写一大段文档”，而是：

**把最容易误解的约束提前显式化。**

> SDD 具体怎么落地——四份产物（constitution/spec/plan/tasks）长什么样、用什么工具、端到端走一遍——见 `02_SDD规格驱动开发：从需求到代码的具体落地.md`。本节只讲面试表达，那一章讲实操。

## 3. Agentic Coding：从聊天助手到 issue-to-PR

AI 编程工具正在从“聊天式助手”变成“开发代理”。

聊天助手主要做：

- 回答问题
- 解释代码
- 生成片段
- 给出建议

开发代理会进一步做：

- 读取代码库
- 搜索相关文件
- 修改多文件
- 运行命令
- 跑测试
- 根据报错修复
- 生成提交或 PR

也就是说，AI 不只是给建议，而是进入了实际研发链路。

### 3.1 哪些任务适合交给 coding agent

比较适合：

- 边界清晰的小功能。
- 有明确测试保护的 bug fix。
- 重复性重构，比如改 API 调用方式、补日志、补指标。
- 根据现有模式新增类似模块。
- 补充单测、集成测试、mock 数据。
- 根据 CI 报错做定位和修复。
- 文档和代码同步更新。

不太适合直接放手：

- 需求还没有澄清的核心功能。
- 涉及资金、权限、安全、风控的数据逻辑。
- 牵涉大规模架构调整的改造。
- 没有测试、没有日志、没有可回放样例的复杂 bug。
- 需要大量业务隐性知识的历史系统改动。

可以用一句话概括：

**AI 适合执行边界清晰、有反馈闭环的任务；不适合替人做价值判断和责任判断。**

### 3.2 一个更稳的 agentic coding 流程

```text
1. 明确任务边界
2. 收集相关上下文
3. 让 AI 先给 plan，不直接改
4. 人审 plan，缩小范围
5. AI 小步修改
6. 跑测试 / lint / typecheck
7. 人审 diff
8. 必要时补充测试和文档
9. 合并前走正常 code review
```

这里最重要的是第 3 步和第 7 步。

不要让 AI 一上来就大改。  
也不要因为测试通过就不看 diff。

测试只能证明一部分行为，不能证明：

- 设计是否合理
- 抽象是否过度
- 是否破坏隐含兼容性
- 是否引入安全风险
- 是否符合团队风格

### 3.3 面试表达

可以这样说：

> 我会把 coding agent 当成一个能执行任务的 junior engineer，而不是全自动负责人。它可以读代码、改代码、跑测试，但任务边界、设计取舍、最终 review 仍然由人负责。我的习惯是先让它做代码检索和方案草稿，再让它按小任务实现，每一步都要求有测试或可验证输出，最后我自己审 diff。

这个表达的好处是：  
既承认 AI 的能力，也没有把责任甩给 AI。

## 4. Skills / Superpowers：把开发经验封装成可复用流程

如果 SDD 解决的是“这次要做什么”，那么 skill 解决的是：

**遇到某类任务时，AI 应该按什么流程做。**

这里的 skill 可以理解成一套可复用的任务能力包，通常包含：

- 任务触发条件
- 操作步骤
- 上下文读取规则
- 检查清单
- 示例
- 脚本
- 参考资料
- 禁止事项

Superpowers 这类实践的价值，也不在于某个工具名本身，而在于它把开发流程模块化了。

例如：

| Skill | 解决的问题 |
| --- | --- |
| TDD skill | 先写失败测试，再实现，再重构 |
| Code review skill | 按正确性、边界、安全、可维护性审查 |
| Debug skill | 先复现、再假设、再最小验证、最后修复 |
| Migration skill | 数据迁移前后校验、回滚、兼容窗口 |
| PR finish skill | 跑测试、整理变更、写 PR 描述、列风险 |
| Incident postmortem skill | 还原时间线、根因、影响面、改进项 |

### 4.1 为什么 skill 比临时 prompt 更有价值

临时 prompt 的问题是不可复用。

今天你告诉 AI：

```text
做 code review 时优先找逻辑 bug，不要只改格式。
```

下次你还得再说一遍。

如果把它写成 skill，AI 在触发 code review 任务时就会自动加载这套规则。  
这就把个人经验变成了可复用的团队资产。

这背后的思想可以叫：

**Knowledge as Code。**

也就是把团队知识像代码一样管理：

- 版本化
- 可 review
- 可迭代
- 可复用
- 可被 AI 自动加载

### 4.2 SDD 和 skill 的关系

二者不是替代关系。

| 维度 | SDD | Skill |
| --- | --- | --- |
| 核心问题 | 这次需求到底是什么 | 这类任务应该怎么做 |
| 产物 | spec、plan、tasks、验收标准 | workflow、checklist、脚本、规则 |
| 作用范围 | 单个需求或项目 | 可复用任务类型 |
| 面试表达 | 需求和实现对齐 | 流程和经验沉淀 |

可以这样理解：

```text
SDD 管目标和边界
Skill 管执行方法和质量纪律
```

### 4.3 面试表达

可以这样说：

> 我觉得 AI 研发后面会从 prompt engineering 走向 workflow engineering。个人临时 prompt 很难稳定复用，所以更好的方式是把团队认可的流程沉淀成 skill，比如 review skill、debug skill、TDD skill、migration skill。这样 AI 每次处理同类任务时，不是重新猜流程，而是执行团队沉淀过的工程方法。

这比单纯说“我会写 prompt”更有工程味。

## 5. AI 不只参与生成，也要参与验证

很多人谈 AI 辅助开发，只讲生成代码。  
但生产里真正贵的是验证。

AI 可以帮助做这些验证工作：

- 根据 spec 生成测试用例。
- 找边界条件和异常路径。
- 做 PR 风险扫描。
- 对比新旧实现行为差异。
- 解释 CI 失败原因。
- 根据日志和 trace 梳理故障链路。
- 检查文档、接口、测试是否同步更新。

但要注意：

**AI 生成的测试不等于可靠测试。**

常见问题有：

- 只覆盖 happy path。
- 测试跟着实现写，断言没有业务意义。
- mock 太重，真实集成问题被遮住。
- 为了让测试通过，反向修改测试预期。
- 漏掉并发、时序、兼容性、权限等非功能问题。

所以测试也要有“真值来源”。

常见真值来源包括：

- spec 里的验收标准
- 老版本行为
- 线上样本
- 数据库约束
- API contract
- 业务规则文档
- 人工确认过的边界案例

面试可以这样说：

> AI 可以帮我生成测试，但我不会只看测试数量。我更关注测试的 oracle 来自哪里：是 spec、老行为、接口契约，还是人工确认的业务规则。如果测试只是根据 AI 自己写的实现反推出来的，那它可能只是证明代码符合 AI 的假设，而不是证明业务正确。

这句话很加分。  
因为它讲到了测试的本质。

## 6. 生产落地中的真实问题

AI 进入真实研发流程后，最大变化是：

**生成成本下降了，但验证、治理和责任成本上升了。**

下面这些问题，比“模型会不会写代码”更关键。

### 6.1 需求模糊会被放大

人写代码时，遇到不清楚的地方可能会停下来问。  
AI 更容易直接补一个“看起来合理”的假设。

风险：

- 改错业务语义
- 漏掉非目标
- 破坏历史兼容
- 让错误实现看起来很完整

应对：

- 先写 spec。
- 明确非目标。
- 对关键假设要求 AI 列出来。
- 让 AI 在不确定时提问，而不是自行决定。

### 6.2 上下文丢失和上下文污染

AI 不是“读过整个仓库就永久理解项目”。  
它只是在当前上下文里推理。

风险：

- 读到了过期文档。
- 只看局部文件，没看调用链。
- 长会话里混入无关信息。
- 多轮修改后忘记最初约束。

应对：

- 控制上下文范围。
- 给出相关文件和禁止修改范围。
- 用 spec / task list 作为持久上下文。
- 长任务阶段性总结，不让上下文无限膨胀。

### 6.3 Diff 审查压力变大

AI 让代码生成速度变快，但人类审查速度没有同等提高。

风险：

- 一次生成几百上千行 diff，人看不过来。
- 逻辑问题藏在看似合理的代码里。
- Reviewer 因为“AI 应该懂”而降低警惕。

应对：

- 小步提交。
- 限制单次修改范围。
- 要求 AI 解释关键 diff。
- 对高风险逻辑增加人工 review 和测试。

### 6.4 架构漂移

AI 很擅长模仿局部代码，但不一定理解全局架构边界。

风险：

- 在错误层写业务逻辑。
- 绕过现有抽象。
- 重复造 helper。
- 把一次性逻辑抽象成过度设计。
- 不同 agent / 不同轮次生成风格不一致。

应对：

- 提供架构规则。
- 提供已有模式示例。
- 明确“不要新增抽象，除非有重复需求”。
- review 时重点看职责边界，而不只是语法。

### 6.5 测试质量虚高

AI 很容易快速生成大量测试。  
但测试多不代表测试好。

风险：

- 测试覆盖率上升，但真实缺陷没减少。
- 测试复制了实现逻辑。
- 测试为了通过而弱化断言。

应对：

- 看测试是否覆盖业务边界。
- 看断言是否有业务含义。
- 看是否包含失败路径、权限路径、兼容路径。
- 对关键逻辑补人工设计的测试用例。

### 6.6 工具权限和副作用风险

Coding agent 可以跑命令、读文件、改文件，甚至调用外部服务。  
这意味着它有真实副作用。

风险：

- 执行危险命令。
- 误删文件。
- 修改生产配置。
- 读取或暴露敏感信息。
- 重复执行数据库写操作。

应对：

- 沙箱执行。
- 最小权限。
- 写操作审批。
- 生产凭证不进入 AI 环境。
- 对数据库、部署、云资源操作要求 dry-run 和人工确认。

### 6.7 安全、合规和依赖风险

AI 可能为了快速解决问题引入新依赖或代码片段。

风险：

- 引入有漏洞的依赖。
- 引入许可证不兼容代码。
- 泄漏 secret、token、内部 URL。
- 生成不安全的鉴权、加密、SQL 拼接逻辑。

应对：

- 依赖变更必须人工确认。
- 跑 SCA / SAST / secret scanning。
- 安全敏感代码不能只靠 AI review。
- 对 auth、crypto、payment、privacy 逻辑提高审核等级。

### 6.8 成本和治理问题

Agent 能自己跑很多轮。  
这既是能力，也是成本风险。

风险：

- token 成本失控。
- CI 被反复触发。
- 多 agent 重复做同一件事。
- 长任务空转。

应对：

- 设置预算。
- 设置最大轮次。
- 设置终止条件。
- 记录 trace，方便复盘。
- 对高成本任务做人工批准。

### 6.9 责任归属不能交给 AI

最终进入主干的代码，责任不属于 AI。  
责任属于提交者、reviewer 和团队。

所以生产流程里不能说：

> 这是 AI 写的，所以出问题不是我的问题。

更成熟的说法是：

> AI 可以参与生成和分析，但 merge 前必须经过和人工代码一样的质量门禁。谁批准合入，谁就要对结果负责。

## 7. 一个可复用的面试回答模板

### 7.1 30 秒版本

> 我觉得 AI 对开发最大的帮助不是简单补全代码，而是进入完整研发流程。我的使用方式通常是：先用 spec 明确目标、约束和验收标准，再让 coding agent 在受控上下文里做小步实现，然后通过测试、CI、code review 和安全扫描验证。对于重复任务，比如 debug、TDD、PR review，我会倾向于沉淀成 skill 或 checklist，让 AI 按稳定流程执行，而不是每次临时 prompt。

### 7.2 2 分钟版本

> 我会把 AI 辅助开发分成三层。第一层是规格驱动，也就是 SDD：先把需求写成 spec，明确目标、非目标、接口约束和验收标准，避免 AI 在需求模糊时自行假设。第二层是 agentic coding：让 AI 读代码、改代码、跑测试、根据 CI 反馈修复，但任务要小步推进，人要审 plan 和 diff。第三层是流程沉淀：把常见任务沉淀成 skill，比如 TDD、debug、review、migration，让 AI 每次按团队认可的流程工作。生产里我最关注的是验证和治理，比如测试是否有业务 oracle、是否引入不安全依赖、是否越权访问数据、是否让 reviewer 承担过大的 diff 审查压力。总的来说，我认为 AI 可以提升开发效率，但不能替代工程师对需求、设计和质量的责任。

### 7.3 如果被问：你怎么看 AI 替代程序员

可以答：

> 我不太把它理解成简单替代。AI 确实会替代一部分低上下文、重复性、边界清晰的编码工作，但复杂系统里的需求澄清、架构取舍、质量责任、跨团队协调和生产风险判断，仍然需要工程师负责。变化更像是工程师的工作重心前移：从逐行写代码，变成定义 spec、组织上下文、设计验证方式、审查 AI 产物。

### 7.4 如果被问：怎么保证 AI 代码质量

可以答：

> 我会从四层保证。第一是输入质量，先写清 spec 和验收标准。第二是执行控制，限制任务范围，小步修改，不让 AI 一次性大改。第三是自动化验证，跑单测、集成测试、lint、typecheck、安全扫描。第四是人工 review，重点看业务语义、职责边界、兼容性和安全风险。AI 生成的代码不能因为“看起来合理”就合并，必须走和人工代码一样甚至更严格的质量门禁。

### 7.5 如果被问：你了解哪些前沿方法或工具

可以答：

> 方法论上我比较关注 SDD、Agentic Coding 和 Skills。SDD 强调先定义 spec，再让 AI 围绕 spec 生成 plan、tasks 和实现；Agentic Coding 指 Claude Code、Codex、Copilot coding agent、Cursor agent 这类能读仓库、改代码、跑测试的开发代理；Skills / Superpowers 这类实践则是把 TDD、debug、review 等流程封装成可复用能力。它们共同的方向是：AI 不再只是聊天式助手，而是被放进工程流程里受控执行。

### 7.6 如果被问：AI 开发最大风险是什么

可以答：

> 我觉得最大风险不是 AI 写不出代码，而是它写出看起来很合理但实际上偏离业务约束的代码。因为生成能力变强后，团队可能低估验证成本。具体风险包括需求假设错误、上下文不完整、测试虚高、架构漂移、安全依赖风险和 reviewer 审查压力上升。解决方式不是不用 AI，而是用 spec、权限控制、小步提交、自动化测试、人工 review 和知识沉淀把它纳入工程治理。

## 8. 项目里可以怎么落地

如果要把这章和自己的项目经历结合，可以这样组织：

```text
背景：项目里有重复开发 / 历史代码理解 / 测试补齐 / 文档维护 / bug 定位成本高的问题。

做法：
1. 对边界清晰的需求先写 spec。
2. 让 AI 根据 spec 拆 plan 和 tasks。
3. 让 AI 在独立分支上小步实现。
4. 每步跑测试并审 diff。
5. 对常见流程沉淀 checklist 或 skill。

结果：
- 开发反馈周期缩短。
- 测试和文档补齐更系统。
- 新人理解模块更快。
- 但核心业务逻辑仍然保留人工 review 和责任边界。
```

注意不要吹成：

> 我们用 AI 自动完成了整个项目。

更可信的表达是：

> AI 主要帮助我降低重复劳动和上下文理解成本，但最终设计、验收和合并决策还是由人负责。

这更像真实生产环境。

## 9. 使用 AI 开发前的检查清单

每次把任务交给 AI 前，可以先过一遍：

- 这次任务的目标是什么？
- 非目标是什么？
- 哪些文件可以改，哪些不能改？
- 哪些接口、数据结构、兼容性不能破坏？
- 有没有验收标准？
- 有没有现成测试？
- 没有测试时，真值来源是什么？
- 这次任务是否涉及权限、资金、安全、生产配置？
- AI 可以执行哪些命令？
- 哪些操作必须人工确认？
- 最终需要哪些验证：单测、集成测试、lint、typecheck、CI、安全扫描？
- 这次经验是否值得沉淀成 rule、checklist 或 skill？

这份清单背后的核心是：

**AI 可以提高执行速度，但工程师要提高控制质量。**

## 10. 本章总结

AI 辅助开发的面试表达，不要停在“我会用工具”。

更好的主线是：

```text
需求清晰化：SDD
  -> 执行自动化：Agentic Coding
  -> 流程复用：Skills / Superpowers
  -> 质量保障：测试、review、CI、安全扫描
  -> 生产治理：权限、成本、审计、责任边界
```

最后可以收束成一句话：

**AI 时代的优秀工程师，不是把判断交给 AI，而是更擅长定义问题、组织上下文、设计验证，并把经验沉淀成可复用的工程流程。**
