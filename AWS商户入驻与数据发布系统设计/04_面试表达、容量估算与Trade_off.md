# AWS商户入驻与数据发布系统设计 - 第 4 课：面试表达、容量估算与 Trade-off

## 学习目标（本节结束后你能做到什么）

1. 能把前面三课的内容压缩成一段 5 到 8 分钟的面试回答，而不是讲成散乱的技术点。
2. 学会对商户入驻场景做轻量容量估算，说明为什么这里的重点是可靠推进而不是极限吞吐。
3. 掌握几个关键 trade-off：Standard vs Express、DynamoDB vs Aurora、同步 vs 异步、直接入仓 vs 先落 S3。
4. 知道这道题最容易被追问的稳定性问题，并能给出成熟回答。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

系统设计面试最常见的问题，不是“不会设计”，而是“知道不少，但讲不成一段有层次的话”。前面三课讲的是知识点，这一课要把它们变成面试表达。你可以把这道题稳定压成五段。

第一段，澄清问题。你可以先说：“我先把题目收敛一下。这里我理解的是一个平台商户入驻系统，商户要提交主体信息并绑定一个或多个仓库，流程中既有自动校验，也可能有人工审核。除此之外，系统还要把入驻过程数据同步到数仓，生成 Dashboard 和邮件报表。”这一段的价值，是把业务对象和两条链路先说清楚。

第二段，讲核心矛盾。你可以说：“这个系统的核心矛盾不是高吞吐，而是长流程编排、跨服务协同、状态可追踪、失败可恢复；分析侧则要求与主流程解耦，保证最终一致和可回放。”这句话非常重要，因为它在告诉面试官：你知道系统的关键压力点在哪里。很多候选人会下意识把所有题都往高并发方向讲，但入驻系统通常不是秒杀系统，真正难点是流程复杂性，而不是每秒十万请求。

第三段，给高层方案。这里可以简洁地说：“主链路上，入口用 API Gateway 和 Onboarding Service 接收申请，Step Functions Standard 负责编排，Lambda 或独立微服务执行校验、风控、仓库检查和人工审核回调，交易态落到 DynamoDB。分析链路上，通过 DynamoDB Streams 捕获变更，经 Lambda 标准化后落 S3，再用 Glue 加工到 Redshift，最后给 Dashboard 和邮件系统使用。”这一段讲完，面试官已经能在脑子里形成你的整体图。

第四段，深挖一个主流程。你可以挑最关键的几个动作讲：“提交时先做幂等，避免重复申请；然后做基础信息和主体去重校验；风控、KYC、合规之类的检查可以并行；多个仓库用 Map 状态逐个处理；命中人工审核时使用 task token callback 挂起并恢复工作流；最终创建正式商户档案和仓库绑定，并发布领域事件。失败场景按 saga 思路做补偿或转人工处理。”注意，这一段不要试图把每个 Lambda 都讲一遍，重点是让面试官看到你会抓主干。

第五段，讲 trade-off 和扩展。比如：“我这里选 Standard 而不是 Express，因为这是长事务流程，需要可追踪和人工回调；我选 DynamoDB 作为交易存储，是因为主访问模式更像按主键查询和状态推进，且和 serverless 编排耦合自然；数据发布不直接写 Redshift，而是先落 S3，主要是为了回放和解耦；如果后续规则越来越复杂，也可以把某些校验从 Lambda 演进为独立微服务。”这样一收束，整个回答就比较完整。

再说容量估算。你不需要把数字算得特别精细，但要体现量级感。比如你可以做这样一个假设：平台每天新增 5 万个入驻申请，平均每个商户绑定 3 个仓库，80% 在白天 10 小时内提交。那平均提交速率大约是每秒 1 到 2 个申请，高峰按 10 倍放大也就十几到几十 QPS。再考虑每个申请触发多个状态更新、外部校验和仓库子任务，工作流内部的执行动作会更多，但整体仍远低于秒杀类系统。也就是说，这里的设计重点应该是“流程稳定推进”和“外部依赖可靠隔离”，而不是为了极致吞吐过早引入特别重的分布式组件。

但不要因此忽视扩展性。因为虽然申请提交 QPS 不高，事件链路和分析链路可能有积压风险。比如每个申请平均产生 10 次状态变更，那每天就是几十万条到上百万条 CDC 事件。这个量级对 Streams、Lambda、S3、Glue、Redshift 来说并不夸张，但你需要有清晰的分层：主流程稳住交易，数据链路吸收分析负载。只要把边界讲清楚，面试官会觉得你是“有规模感，但不过度设计”。

下面是这道题最值得主动讲的几个 trade-off。

第一，`Step Functions Standard vs Express`。Standard 的优点是执行历史完整、适合长流程、支持人工回调，缺点是单次成本可能更高、超高吞吐短流程不是它的强项。Express 的优点是吞吐高、适合事件驱动短流程，但在这个题里，长流程和审计更关键。所以我会首选 Standard。

第二，`DynamoDB vs Aurora`。DynamoDB 的优点是按键访问性能稳定、天然适合 serverless、扩缩容友好；Aurora 的优点是关系模型强、复杂查询和事务更自然。如果入驻系统后续要承载很多复杂运营查询、联表审核规则、强关系型约束，那么 Aurora 也可能成立；但如果我们把交易读写模式控制在“按申请单、按商户、按状态”的路径上，DynamoDB 会更轻。

第三，`同步调用 vs 异步事件`。像提交申请、查询进度这种面向用户即时反馈的动作，应该保留同步接口；像通知、分析、报表、弱依赖下游触发，应该异步化。不要为了“纯事件驱动”把所有东西都改成异步，否则用户连提交结果都拿不到，也不利于排障。

第四，`直接入 Redshift vs 先落 S3`。直接入仓看起来简单，但恢复能力差、Schema 变化难管、对 Redshift 友好度也一般。先落 S3 多了一层，但换来了重放、解耦和多下游复用能力。对于面试题，这是更稳的工程答案。

稳定性问题也一定要准备。你至少要覆盖这些追问：

- 如果风控服务超时怎么办：有限次重试，超过阈值后转人工或挂起待处理。
- 如果同一申请重复触发事件怎么办：靠幂等键和版本号去重。
- 如果状态机执行中断怎么办：依赖持久化状态继续恢复，必要时人工补单。
- 如果数据发布链路挂了怎么办：不影响交易主流程，依赖 S3 原始层回放。
- 如果 Dashboard 数字和交易后台短时间不一致怎么办：明确说明交易查询和分析报表是两套 SLA，报表允许最终一致。

最后，给你一段可以直接复述的短版本：

“这个题我会拆成交易工作流和数据发布两条链。交易侧的核心难点不是高并发，而是长流程编排、跨服务协同、人工审核和失败恢复，所以我会用 Step Functions Standard 做流程总控，Lambda 或微服务做校验、风控和仓库检查，交易状态落 DynamoDB，并通过幂等、重试和 saga 补偿保证流程可靠推进。分析侧我不会让 Redshift 进入主交易路径，而是通过 DynamoDB Streams 捕获状态变化，经 Lambda 标准化后落到 S3，再用 Glue 处理进 Redshift，最终生成 Dashboard 和邮件报表。这样主流程保证正确性和可追踪，分析链路保证解耦、可回放和最终一致。” 

如果你能把这段话自然讲出来，再配合 1 到 2 个关键追问展开，这道题的完成度就已经很高了。

## 小结（3-5 条关键点）

1. 这道题的核心矛盾是长流程编排和可恢复，不是极限高并发。
2. 一个成熟回答要先讲需求收敛，再讲整体架构，然后深挖工作流，最后说 trade-off。
3. 轻量容量估算的意义不是算得极准，而是证明你知道系统真正的压力点在哪里。
4. Standard vs Express、DynamoDB vs Aurora、同步 vs 异步、直接入仓 vs 先落 S3，是这道题最值得主动讲的四组取舍。
5. 面试里要主动补失败路径和 SLA 边界，否则架构图再完整也容易显得偏浅。
