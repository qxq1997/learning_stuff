# AWS商户入驻与数据发布系统设计 - 第 5 课：项目难点与解决方案

## 学习目标（本节结束后你能做到什么）

1. 能把这个项目的真正难点讲清楚，而不是停留在“用了哪些 AWS 服务”。
2. 知道哪些难点属于流程编排问题，哪些属于一致性问题，哪些属于数据发布和可观测性问题。
3. 能针对每个难点给出合理解决方案，并解释为什么这样设计。
4. 能把这些内容组织成一段适合面试的“项目难点与解决方案”表达。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

面试里讲项目难点，最怕两种回答。第一种是把难点讲成“技术栈很多”，比如说这个项目用了 Step Functions、Lambda、Glue、Redshift，所以很复杂。第二种是把难点讲成“流量很大”，但其实这个场景的核心矛盾并不在于极限吞吐。真正成熟的表达，应该先抓住这个系统的本质：这是一个长流程、跨服务、带人工审核、还需要同步分析数据的商户入驻系统。难点来自流程复杂性、状态一致性和链路解耦，而不是 AWS 服务名本身。

第一个难点，是长流程编排复杂。商户入驻不是一个提交接口写完数据库就结束了，它会经过基础资料校验、主体去重、风控检查、合规校验、仓库校验、人工审核、正式开户、结果通知等多个步骤。有些步骤可以立即返回结果，有些要依赖外部系统，有些甚至需要等待人工几小时后才有结论。也就是说，这个项目天然不是短事务，而是长事务工作流。这里最容易出的问题是：流程写散了，推进逻辑藏在多个服务里，导致没人说得清当前卡在哪、下一步该谁接手、失败后怎么恢复。

这个难点的解决思路，就是用 `Step Functions Standard` 统一做流程编排。它的价值不只是“能调 Lambda”，而是把流程中的步骤、分支、并行、等待、重试、异常捕获都显式表达出来。比如基础资料校验后进入主体去重，再并行触发风控和合规检查；多个仓库则通过 `Map` 状态逐个处理；如果命中人工审核，则通过 `task token callback` 挂起等待；审核结束后再恢复流程。这种设计的本质，是把“流程控制权”从分散的业务服务里收回来，让流程推进变得可视、可追踪、可审计。

第二个难点，是跨服务一致性。入驻链路里经常会依赖多个微服务，比如商户服务、仓库服务、风控服务、审核服务、通知服务。每个服务都有自己的数据存储和失败模式，所以你不能指望用一个数据库事务把所有步骤包起来。很多人在这里会本能地想到分布式事务，但面试里这样答通常不够成熟，因为跨服务长流程更现实的做法不是强事务，而是 `Saga + 幂等 + 补偿`。比如正式商户已经创建了，但仓库绑定失败了，这时不是回滚整个世界，而是根据业务规则决定执行补偿、标记异常待处理，或者进入人工介入流程。

所以这个难点的解决方案，是把“强一致”改写成“可恢复的一致性”。具体做法包括：每一步操作都尽量做成幂等；状态机记录当前进度和失败原因；跨服务调用区分业务失败和瞬时失败；瞬时失败做有限次重试和退避；确定性失败则直接落状态或转人工。这样设计的核心价值，不是让系统永远不出错，而是保证出错后系统知道自己错在哪、能否自动恢复、不能恢复时由谁接手。

第三个难点，是重复提交和重复消费，也就是幂等问题。商户前端可能因为网络抖动多次点击提交，API 调用方也可能在超时后自动重试。即使上游只发了一次，消息链路或流处理链路也可能因为重试导致重复事件。如果你没有在系统入口和异步链路上做幂等控制，就很容易出现多个相同申请单、重复创建商户、重复绑定仓库、报表重复计数等问题。

这个问题的解决方案，是在不同层次分别做幂等。同步入口层，通过 `idempotencyKey` 或唯一申请号保证同一个申请只被创建一次；工作流执行层，以 `onboardingId` 作为唯一流程实例标识，防止重复启动；下游服务操作层，通过条件写、唯一约束或版本号控制，避免重复落正式数据；数据发布链路里，则通过 `eventId`、版本号或“主键 + 变更时间”的组合键做去重。面试里如果你能主动说出“幂等不是只做一层，而是入口、流程、事件都要做”，会非常加分。

第四个难点，是一个商户可能绑定多个仓库，导致部分成功和部分失败场景复杂。比如一个商户提交了 5 个仓库，其中 4 个校验通过，1 个仓库因为状态异常失败。如果你的设计是在校验通过一个就立即写一个正式绑定，那么系统就很容易进入半成功状态。后面如果整单被拒绝，回滚会很痛苦；如果整单通过，但有 1 个仓库失败，业务语义也会变得模糊。

这里更稳妥的处理方式，是把“校验阶段”和“正式落库阶段”拆开。多个仓库可以通过 `Map` 并发校验，但校验结果先汇总到申请单上下文里，不要急着创建正式绑定。等所有仓库校验都完成后，再根据业务规则统一决定：是全部通过后再正式绑定，还是允许部分通过并把失败仓库标记为待补充。这样做的本质，是先收敛结果，再落正式态，避免过早写入导致补偿复杂度飙升。

第五个难点，是人工审核和外部依赖的不确定性。风控、合规、仓库等外部系统都有可能超时、限流、返回不一致结果，而人工审核更可能几小时甚至一天后才有结论。如果处理不好，工作流要么长期占住资源轮询，要么因为超时而中断，最后形成大量“悬挂中的申请”。

这个问题的解决方案有两个关键点。第一个关键点，是对外部依赖调用进行分级处理。像网络抖动、暂时性 5xx 这种瞬时错误，适合做有限次指数退避重试；像业务拒绝、参数错误、主体已存在这类确定性错误，不应该重试，而应该直接结束流程或转人工。第二个关键点，是人工审核用 callback 模式。也就是状态机在需要人工审核时挂起，等待审核系统完成后带着 `task token` 回调恢复，而不是让 Lambda 一直轮询审核结果。这样既节省资源，也更符合长流程的业务形态。

第六个难点，是交易链路和分析链路的解耦。这个系统不仅要让商户成功入驻，还要把状态变化、结果数据发布到 Redshift、Dashboard 和邮箱。如果你把分析处理直接放进主交易流程，比如申请通过后同步写 Redshift、同步刷新报表，那分析系统一旦变慢或失败，就可能反向拖垮主业务。这是很多系统在演进中容易踩的坑。

解决方案就是严格解耦。交易链路只关心正确推进申请状态，并把最终状态写入 DynamoDB；分析链路通过 `DynamoDB Streams` 增量捕获变更，由独立的 Lambda 做 CDC 标准化，再落到 S3 raw 层，之后由 Glue 处理进 Redshift，最终供 Dashboard 和邮件使用。这样做的好处是：即使 Glue 或 Redshift 出了问题，商户入驻仍然可以正常走完；分析链路恢复后，还可以从 S3 原始数据回放补数。也就是说，交易和分析之间是事件连接，而不是同步强耦合。

第七个难点，是可观测性和审计。长流程系统最怕的不是某一步失败，而是失败之后没人知道为什么失败、失败在哪、下一步该怎么办。如果面试官追问“有一单卡了 3 小时，你怎么排查”，而你只能回答“去看日志”，那就显得比较浅。

更成熟的做法，是把可观测性设计进系统里。申请单表里要显式记录当前步骤、当前状态、更新时间、失败原因码、重试次数；Step Functions 保存完整执行历史，便于定位卡点；CloudWatch 和告警系统监控超时、失败率、积压量；关键领域事件最好有审计记录，方便追溯是谁在什么时候推进了哪个状态。这样，当申请卡住时，运营看业务状态表，研发看状态机执行历史和日志，双方都能快速定位问题。可观测性的本质，不是多打几行日志，而是让状态和动作都有迹可循。

如果把这些难点收成一句适合面试的话，可以这样说：这个项目最大的挑战，不是 AWS 服务怎么拼，而是如何把一个跨服务、带人工审核、包含多仓库处理和数据发布的长流程系统做得可推进、可恢复、可审计。我的做法是用 `Step Functions Standard` 做流程编排，用 `DynamoDB` 保存过程态和结果态，用幂等和 `Saga` 补偿解决重复和跨服务失败，用 `task token callback` 处理人工审核，用 `DynamoDB Streams + S3 + Glue + Redshift` 解耦分析链路，并通过状态表、执行历史和告警体系提升可观测性。

## 小结（3-5 条关键点）

1. 这个项目的核心难点是长流程、跨服务、人工审核、多仓库处理和分析链路解耦，而不是单纯的技术栈数量。
2. 长流程编排适合交给 `Step Functions Standard`，业务规则仍由微服务承载，避免把流程写散。
3. 跨服务失败要靠幂等、状态机和 `Saga` 补偿处理，而不是依赖分布式强事务。
4. 多仓库场景要先汇总校验结果，再落正式绑定，尽量避免半成功状态。
5. 数据发布链路必须和交易主链路解耦，并配套可回放、可告警、可审计能力。
