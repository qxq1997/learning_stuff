# AWS商户入驻与数据发布系统设计 - 第 7 课：DF 入驻工作流编排系统本体复盘

## 学习目标（本节结束后你能做到什么）

1. 能把这个项目讲成清晰的“两层结构”：底层是 Direct Fulfillment 的 vendor / warehouse onboarding 工作流编排系统，上层是工作流可观测性与 BI 报表平台（第 6 课讲的那一层）。
2. 能讲清楚 DF 这种业务模式下，onboarding 为什么天然是长事务、跨服务、带人工节点的编排场景，而不是一个简单的提交接口。
3. 能从“它解决了什么问题”出发，把架构、技术栈、技术挑战、改进空间、生产问题串成一条可追问的主线。
4. 能在面试里把编排层（Step Functions + 微服务）和观测层（事件驱动 BI）分开讲，而不是混成一锅。

## 一、为什么要把项目理解成两层

很多人讲这个项目时，会下意识地只讲“我们做了一个监控大盘 / BI 报表”，但这样会丢掉一半价值，因为报表只是结果，真正撑起这个系统的是底层那套**长流程编排引擎**。更成熟的表达是把项目拆成两层：

- **底层：onboarding 工作流编排系统**。基于 AWS Step Functions + 一组微服务，负责把一个 vendor / warehouse 从“提交申请”一步步推进到“正式可跑 DF 业务”，中间穿插自动校验、外部系统回调和人工审核。
- **上层：工作流可观测性 & BI 报表平台**。基于 EventBridge → Lambda → DynamoDB → Glue → Redshift → QuickSight，把底层流程的状态变化采集出来，回答“卡在哪、卡了多久、平均时长、卡点分布”这类问题，也就是第 6 课讲的内容。

这两层的关系是：底层负责“把事做对”，上层负责“把事看清”。面试里先抛出这个两层视角，会立刻显得你对系统有全局结构感，而不是只盯着某一个服务。

## 二、项目背景：Direct Fulfillment 是什么，onboarding 又是什么

Amazon Direct Fulfillment（DF）是一种区别于 FBA 自营仓配的模式。FBA 是 Amazon 自己的仓配，DF 则是 Amazon 把订单流转给大型 vendor，由 **vendor 自己维护仓库、库存与 carrier 履约**，Amazon 负责订单流转、对账、SLA 监督和消费者体验。换句话说，履约的“手”是 vendor 的，但订单和体验的“账”还是 Amazon 来管。

要让一个 vendor 真正能跑 DF 业务，必须走完一整套 onboarding 流程，它至少包含三类：

- **Vendor onboarding（主体级）**：公司主体审核、税务与支付配置、合同与 SLA 签署、EDI 或 API 通道对接、SKU catalog 接入、价格与履约协议、合规与安全审查。
- **Warehouse onboarding（仓库级）**：仓库地址与服务区域注册、carrier 对接、cut-off time 与 SLA 配置、库存上报通道打通、test order 验证、灰度放量。
- **Carrier onboarding（承运商级）**：承运商资质、tracking 集成、label 规范、退货流程。

这些流程横跨多个微服务——Vendor Service、Catalog Service、Tax Service、Contract Service、Warehouse Service、Carrier Service、Compliance Service……既有自动校验，也有人工审核，还有外部系统回调。**单个 onboarding 通常持续数小时到数天**。这正是典型的长事务、异步、跨服务编排场景，也是为什么不能用一个同步接口加一个数据库事务搞定。

## 三、它解决了什么问题

在没有这套系统之前，这类流程往往是“邮件 + 工单 + 一堆脚本”驱动的，问题集中在五个方面：

1. **流程不可见**：vendor 卡在哪一步、谁负责、为什么卡，Ops 要到处问人。
2. **流程不可复用**：每来一个新 vendor 类型（欧洲 vs 美国、3P vs 1P）就重写一套脚本。
3. **失败不可恢复**：某一步挂了往往要从头跑，代价大。
4. **SLA 不可量化**：回答不了“平均 onboarding 时长”“卡点分布在哪一步”。
5. **合规不可审计**：监管要求可追溯每个决策点的输入、输出、操作人。

这套系统通过 **Step Functions 编排 + 微服务执行 + 事件驱动可观测性**，把上述五个问题系统性解决：编排让流程可见可复用可恢复，事件采集让 SLA 可量化、决策可审计。

## 四、整体架构

可以按职责分成六块来讲：

- **编排层**：AWS Step Functions（Standard workflow）。选 Standard 是因为 onboarding 是长事务，可能跑几天甚至更久，Standard 支持最长 1 年执行时长，而 Express 上限只有 5 分钟。每一种 onboarding 类型对应一个 state machine 定义（用 ASL，即 Amazon States Language，本质是 JSON），内部用 Choice / Parallel / Map / Wait / Task 这些状态组合出流程。
- **执行层**：多个 Java 17 + Spring Boot 3 微服务，通过 Lambda 或直接的 service integration 被 Step Functions 调用。长任务用 **callback pattern（taskToken）**——Step Functions 发起后挂起，微服务处理完（包括等待人工审核完成）再回调 `SendTaskSuccess` / `SendTaskFailure` 唤醒流程。
- **人工节点**：涉及合规审核、合同签署的步骤，用 taskToken + 内部审批系统集成，审批人在 UI 上点通过 / 拒绝后回调。
- **外部对接**：EDI / SFTP / API 对接 vendor 系统，通过专门的 integration service 屏蔽协议差异，让上层流程不用关心底层是哪种协议。
- **存储**：每个微服务持有自己的领域数据库（DynamoDB / Aurora），Step Functions 自身持有 execution state。
- **可观测层**：EventBridge → Lambda → DynamoDB → Glue → Redshift → QuickSight，即第 6 课的内容。
- **基础设施**：CDK / CloudFormation 做 IaC，CodePipeline + CodeBuild 做 CI/CD，各 region 独立部署。

## 五、技术栈

> AWS Step Functions（Standard）、AWS Lambda、Amazon EventBridge、Amazon SQS、Amazon SNS、DynamoDB、Aurora、S3、AWS Glue（PySpark）、Amazon Redshift、Amazon QuickSight、CloudWatch、AWS X-Ray、AWS IAM、AWS KMS、AWS Secrets Manager、AWS CDK；微服务侧 Java 17 + Spring Boot 3，部分用 Python；CI/CD 走 CodePipeline + CodeBuild。

## 六、技术挑战（简历和面试都用得上）

**1. 长事务与人工节点的状态保持**

Onboarding 可能跑 3 天，中间有人工审核。用 Step Functions 的 callback pattern with taskToken：`.waitForTaskToken` 让 Step Functions 挂起当前 state，把 token 传给下游服务，审批完成后调 `SendTaskSuccess(taskToken, output)` 唤醒。这样状态完全托管在 Step Functions，微服务保持无状态，重启不丢失流程。这是整套设计的核心——把流程状态从分散的服务里收回到编排引擎手里。

**2. 工作流版本演化**

ASL 改了之后，正在运行的老 execution 怎么办？Step Functions 不支持热替换运行中的 execution。方案：state machine 加版本号，新 execution 走新版本，老 execution 跑完自然消亡（让版本自然“收敛”而不是强行迁移）；Lambda 端的业务逻辑做向后兼容，通过 input 里的 `schemaVersion` 字段分支处理。

**3. 幂等性**

任何一个 task 都可能因为 timeout / retry 被执行多次。每个微服务接口都要做幂等：用 `executionArn + stateName + attempt` 作为幂等 key，DynamoDB conditional write 保证同一 key 的副作用只生效一次。要强调“幂等不是只做一层”——入口、流程、下游写入都要做。

**4. 失败处理与补偿**

Step Functions 的 `Retry` + `Catch` 处理可重试错误；但对于已经产生的不可逆下游副作用（比如已经在 vendor 那边创建了 catalog），需要 **Saga 模式**——为每一步定义补偿操作，失败时进入 compensation branch 反向执行。关键认知：跨服务长流程不追求强一致，而追求“可恢复的一致性”。

**5. 跨服务数据传递与 payload 限制**

Step Functions 的 state input/output 有 256KB 上限。SKU catalog、税务文档这种大对象不能直接塞进 state，要 **存 S3，state 里只传 reference（S3 key + version）**，下游服务自己拉。这既绕开了大小限制，也避免敏感数据在 state 里裸奔。

**6. 并发与限流**

某些 vendor 一次性提交几千个 warehouse，会并发打爆下游服务。在 Step Functions 里用 `Map` state 的 `MaxConcurrency` 控制并发；微服务侧再用 token bucket + 按 vendor 维度的限流，避免一个大客户挤占小客户的资源。

**7. 多 region / 多账号部署**

DF 业务全球运营，NA / EU / FE 各 region 独立合规、资源隔离。用 CDK 做 multi-region multi-account 部署，跨 region 的全局数据（比如 vendor 主数据）走专门的 global 服务，工作流执行严格 region-local。

**8. 可观测性**

跨服务异步流程的黑盒问题，详见第 6 课。要点是：execution 原生事件和 task 业务事件能力边界不同，不要混讲。

## 七、改进空间（你来做会怎么改）

1. **编排层抽象化**：现在每个 onboarding 类型都要手写一个 ASL，改动成本高。可以做一层 **DSL → ASL 编译器**，用 YAML 描述业务流程自动生成 Step Functions 定义，业务方不接触底层细节。
2. **工作流即配置**：把“vendor 类型 → 需要走哪些步骤”做成配置中心，而不是写死在 ASL 里。新增一个 vendor 类型不发版，改配置即可。
3. **引入 LLM 辅助卡点处理**：历史失败 case + Ops 处理记录沉淀成知识库；新 onboarding 卡住时自动 RAG 推荐处理方案，甚至自动生成回复邮件给 vendor。
4. **端到端 trace**：把 Step Functions execution、下游微服务 API call、外部 vendor 系统调用通过 X-Ray / OpenTelemetry 串成一条 trace，任何卡点能下钻到具体服务的具体调用。
5. **实时报表**：把现在的 T-1 报表改造成 Kinesis Data Streams + Flink 做近实时聚合，Ops 能看到“当前进行中”“过去 1 小时新增卡点”。
6. **自助式 onboarding**：给 vendor 一个 portal，让 vendor 自己查进度、传材料、回应审核问题，而不是邮件来回。Step Functions 的进度信息直接驱动 portal UI。
7. **成本治理**：Standard workflow 按 state transition 计费，长事务 + 多步骤累计起来不便宜。可以把纯校验类、无人工节点的子流程拆出来用 Express workflow，大幅降本。

## 八、生产问题及应对

1. **Step Functions execution 历史只保留 90 天，但合规审计要求 7 年**。对策：EventBridge 全量归档执行事件到 S3（配 Glacier 分层存储降本），需要时按 executionArn 检索回放。
2. **下游服务挂导致大量 execution 堆积**。对策：`Retry` 配合 exponential backoff；超过阈值进入 manual intervention state，由 Ops 介入；同时 CloudWatch alarm 触发 SNS 通知。
3. **误删 / 误改 state machine**。对策：state machine 走 GitOps 流程，改动必须 PR + code review；关键改动先在 pre-prod 跑 shadow execution 验证。
4. **人工审核长时间不响应**。对策：taskToken 设置 timeout（比如 48h），超时进入升级流程通知 manager；审批任务有 SLA dashboard 给审核团队 leader 看。
5. **重复提交 / 同一 vendor 多个并发 execution**。对策：启动 execution 时用 `vendor_id + onboarding_type` 做 idempotency key（Step Functions 的 execution name 有唯一性约束，重名直接拒绝），前置一个去重 Lambda。
6. **敏感数据（税号、合同）处理**。对策：S3 SSE-KMS 加密，Secrets Manager 管密钥，IAM 最小权限 + CloudTrail 全量审计；state 里绝不存明文 PII，只存 S3 reference。
7. **跨 region 故障切换**。对策：关键 onboarding 流程做 active-passive，DynamoDB Global Tables 同步 vendor 主数据，故障时切流到备 region 重新发起 execution——因为幂等，重新发起是安全的。
8. **工作流变更对历史数据兼容**。对策：维度建模时引入 **SCD Type 2（缓慢变化维）**，把 state machine 版本作为维度属性，报表按版本切片或归并，避免新增 / 改名 state 导致老数据维度对不齐。

## 九、面试表达提示

- 开场先抛“两层结构”，把编排层和观测层分开，再说底层是核心、上层是衍生价值。
- 讲底层时，锚定三个关键词：**长事务（taskToken callback）、可恢复的一致性（Saga + 幂等）、大 payload 外置（S3 reference）**。这三点最能体现你理解了 DF 场景的本质，而不是堆服务名。
- 被追问“怎么排查一单卡了 3 小时”时，分两路答：Ops 看业务状态 / SLA dashboard，研发看 Step Functions execution 历史 + X-Ray trace。
- 被追问成本时，主动提 Standard vs Express 的取舍，以及把无人工节点子流拆成 Express 降本，这是加分项。
