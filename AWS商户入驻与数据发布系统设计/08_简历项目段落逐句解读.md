# 第一段:Vendor Onboarding 工作流(编排层)

## 1.1 总览句:"以 Step Functions 编排跨微服务长事务流程,协调数十个下游微服务"

**这句在说什么**

商家入驻不是一个 HTTP 接口写完数据库就结束的事情,而是一条要走几小时到几天的长流程,中间穿插自动校验、外部系统回调、人工审核。这条流程横跨数十个下游微服务(Vendor、Catalog、Tax、Contract、Warehouse、Carrier、Compliance……),你需要一个"流程总指挥"统一推进。这里选的总指挥就是 AWS Step Functions。

**为什么是 Step Functions,不是别的**

- 不是同步串行调用:某一步挂了整条流程崩,而且占线程。
- 不是消息队列 + 状态机自研:状态、重试、超时、补偿都要自己写,出错没人能审计。
- 不是 Airflow:Airflow 偏批处理调度,不擅长"长事务 + 人工节点 + 跨服务回调"这种业务流。
- Step Functions Standard 的杀手特性:支持单次 execution 跑 1 年(Express 只能 5 分钟);所有状态托管在 AWS 侧,自己不用维护任何 state store。

**面试官可能追问**

- "为什么不用 Express?" —— DF onboarding 是长事务,中间有人工审核,5 分钟根本撑不到。但纯校验类、无人工节点的子流我会拆出来用 Express,因为它便宜两个数量级(按请求计费而不是按状态转换计费)。
- "协调几十个下游服务,怎么避免变成巨型 ASL?" —— 按 onboarding 类型拆 state machine(vendor / warehouse / carrier),复杂步骤用 nested workflow 嵌套调用,而不是平铺写一个超长 ASL。

---

## 1.2 长事务 + 人工审核:"callback pattern + taskToken 支持跨人工审核的多日流程"

**这句在说什么**

合规审核、合同签署这类步骤,机器干不了,要等人。比如审核员可能 2 小时后来点"通过",也可能明天才点。这种情况下你不能让 Lambda 在那里轮询——既费钱,又有 15 分钟超时上限。

Step Functions 的解法叫 callback pattern:当 task 配置成 `.waitForTaskToken` 时,Step Functions 把当前 state 挂起,生成一个 taskToken 传给下游。下游把这个 token 存进自己的审批系统,审核员在 UI 上点"通过",审批系统拿着 token 调 `SendTaskSuccess(taskToken, output)`,Step Functions 才唤醒并继续往后走。

**关键认知**

- 挂起期间不消耗任何计算资源,只占一个 state(Standard 按 state transition 计费,挂起本身不花钱)。
- 流程状态完全托管在 Step Functions,微服务保持无状态;微服务重启、redeploy、扩缩容,流程不丢。
- taskToken 必须设 timeout(比如 48h),超时自动走升级分支通知 manager,否则会有"卡住的悬挂申请"永远不结束。

**面试官可能追问**

- "taskToken 丢了怎么办?" —— token 一旦发出 Step Functions 不会重发,所以审批系统必须把 token 落库持久化;丢了只能等 timeout 后走异常分支重新发起。
- "为什么不用 DynamoDB 自己存一个待审核队列,审完了发 SNS 通知主流程?" —— 可以,但你就要自己实现挂起 / 唤醒 / 超时 / 失败 / 审计这一整套,Step Functions 已经免费给你了。

---

## 1.3 大对象外置:"S3 reference 规避 256KB payload 上限"

**这句在说什么**

Step Functions 的 state 之间通过 input/output 传数据,单次 payload 上限 256KB。但 onboarding 流程里经常要传:几千个 SKU 的 catalog、整份税务文档、扫描件 PDF、几百个仓库的清单。这些动辄几 MB 甚至几十 MB,塞不进 state。

解法是 S3 reference 模式:大对象写到 S3,state 里只传 `{"s3_bucket": "...", "s3_key": "...", "version_id": "..."}`,下游 Lambda 自己去 S3 拉。

**为什么不只是"传 reference"这么简单**

- 要带 version:S3 对象可能被覆盖,流程跑到一半 reference 指向的内容变了就出大事。带 version_id 强制读取那一刻的快照。
- PII 不能直传:税号、身份证这种敏感数据,state 里出现就违反合规审计。所以哪怕只有几 KB,也走 S3 reference + SSE-KMS 加密。
- 避免 state log 泄漏:Step Functions 的 execution history 默认保留所有 input/output,大对象直传会让 history 体积爆炸,也会把敏感数据写进 history。

**面试官可能追问**

- "S3 拉取失败怎么办?" —— Lambda 内做 retry,重试耗尽进 catch 分支;由于 S3 reference + version 是幂等的,重试安全。

---

## 1.4 幂等性:"DynamoDB conditional write 做幂等"

**这句在说什么**

Step Functions 的每个 task,在网络抖动、Lambda 超时、Retry 触发时都可能被执行多次。如果某一步是"在 vendor 那边创建 catalog 条目",重复执行会创建多份,业务就脏了。

每个微服务接口必须做幂等。幂等 key 用 `executionArn + stateName + attempt`(execution 唯一 ARN + 当前 state 名 + 第几次尝试),DynamoDB 写入时用 conditional write:`PutItem` 时加 `ConditionExpression: attribute_not_exists(idempotency_key)`,同一个 key 第二次写直接被拒,业务副作用只生效一次。

**为什么是 DynamoDB conditional write 而不是别的**

- 单条 write 原生原子,不需要分布式锁。
- 延迟稳定(P99 < 10ms),不会成为瓶颈。
- conditional write 失败是显式异常(`ConditionalCheckFailedException`),业务可以直接判定为"这次是重试,跳过实际操作,直接返回上次结果"。

**面试官可能追问**

- "幂等就 DynamoDB 这一层够吗?" —— 不够。幂等是分层的:入口层(申请提交去重,vendor_id + onboarding_type 做 execution name)、编排层(Step Functions execution name 唯一性约束,重启同名 execution 直接拒)、执行层(每个 task 的 DynamoDB conditional write)、事件链路(下游消费用 eventId 去重)。要主动说出这四层,而不是只答一句"做了幂等"。

---

## 1.5 并发治理:"Map state MaxConcurrency + vendor 维度 token bucket 限流"

**这句在说什么**

某个大 vendor 一口气提交 5000 个 warehouse,如果不加控制,Step Functions 的 Map state 会并行展开 5000 个分支同时打下游服务,直接打挂(Lambda 并发上限、下游服务 QPS 上限、外部 EDI 通道带宽都会爆)。

两层限流:

- Step Functions 侧:Map state 配 `MaxConcurrency: 50`,保证同一个 execution 最多 50 个分支同时跑。
- 微服务侧:按 vendor_id 维度做 token bucket(令牌桶),即"每个 vendor 每秒最多消耗 100 个 token",防止单个大 vendor 把整个下游池子吃光。

**关键认知:为什么两层都要**

只靠 Step Functions 侧 MaxConcurrency 不够:如果同时来 10 个大 vendor,每个并发 50,加起来还是 500,会挤占小 vendor。只靠微服务侧 token bucket 也不够:Step Functions 还是会把所有任务调度起来导致 Lambda 并发爆。前者控总量,后者保公平。

**面试官可能追问**

- "为什么按 vendor 维度而不是全局限流?" —— 全局限流会导致小 vendor 被大 vendor 拖死(头部效应)。按 vendor 维度是 multi-tenant 系统的标准做法,保证公平性。
- "超出 token bucket 怎么办?" —— 不丢任务,而是返回 retryable error,Step Functions 自动 exponential backoff,既不丢单也不雪崩。

---

## 1.6 补偿:"不可逆步骤引入 Saga 补偿保障最终一致性"

**这句在说什么**

跨服务长流程,没有分布式事务可以一次性回滚一切。比如流程跑到第 7 步,前 6 步已经在 6 个不同的下游服务里改了状态,第 7 步失败了——你不可能跨 6 个服务 rollback。

Saga 模式的核心:每一步操作都配一个补偿操作,失败时反向执行补偿,把世界还原回"逻辑上没发生过"的状态。比如"已在 vendor 那边创建 catalog"的补偿是"调用 vendor 的删除 catalog API"。

Step Functions 的实现方式:用 `Catch` 捕获异常,跳进 compensation branch,反向依次执行已成功步骤的补偿。

**关键认知**

- 不是所有步骤都需要补偿:只读校验、只在 Step Functions 内部传数据的步骤不需要,只有产生了外部副作用的步骤才需要。
- 补偿本身也要幂等:补偿可能也会失败重试,所以"删除 catalog"要能处理"已经被删了"的情况。
- 补偿失败兜底:补偿也失败时,进入 manual intervention 状态,落事件给 Ops,人工接手。这是面试加分点——表明你知道"自动化兜底之外,还要有人工兜底"。

**面试官可能追问**

- "为什么不用 2PC?" —— 2PC 要求所有参与者都支持 prepare/commit 协议,跨外部 vendor 系统根本做不到;而且 2PC 是阻塞的,长事务用 2PC 等于把整个分布式系统锁住几天。Saga 牺牲了"过程一致性"换"最终一致性",这是长事务的标准取舍。

---

## 1.7 基础设施:"CDK 统一管理,核心链路接入 CloudWatch 告警"

**这句在说什么**

- CDK 是 AWS 的 IaC 工具,用 TypeScript / Python 写代码定义基础设施,生成 CloudFormation 模板部署。优点是可以把"创建 state machine、Lambda、IAM role、DynamoDB 表"这些一次性声明,跨 region / 跨账号复用同一份代码。
- CloudWatch 告警监控关键指标:execution 失败率、平均时长、卡在某 state 的数量、Lambda 错误率;超阈值通过 SNS 通知 Ops。

**为什么这块在简历里也要写**

光有架构没有运维信号,等于"系统能跑但没人知道它在不在跑"。这条传达的是"我考虑了生产可运维性",是高级工程师和初级工程师的分水岭。

# 第二段:工作流观测平台(观测层)

## 2.1 总览句:"采集 Step Functions 全链路执行事件并沉淀为数据资产"

**这句在说什么**

底层 Step Functions 自带 console,但 console 只解决"我现在看一个 execution 长什么样"。它解决不了:

- 跨 execution 聚合("过去 30 天 vendor onboarding 平均时长是多少")
- 历史归档(execution history 只保留 90 天,合规要求 7 年)
- 多维分析("按 region + vendor 类型 + 卡点 state 切片")
- 自助 BI(Ops 不会用 AWS console,要看 QuickSight 大盘)

所以要把 execution 事件采集出来沉淀到数据湖 + 数仓,让它从"运行时状态"变成"可分析的数据资产"。

---

## 2.2 采集架构:"EventBridge + Lambda 订阅 state change 事件并落入 DynamoDB"

**这句在说什么**

Step Functions 每次 state 变更(execution started / state entered / state exited / failed / succeeded……)都会自动发事件到 EventBridge。订阅一条 EventBridge rule,匹配 source = `aws.states`,把事件路由到 Lambda,Lambda 解析后写 DynamoDB。

**为什么先落 DynamoDB**

- 写入快:单条 P99 < 10ms,跟得上事件洪峰。
- schema-less:不同 event type 字段不一样,无 schema 约束的 KV 存储天然适配。
- TTL:DynamoDB 可以设 TTL 自动清理热数据,避免和冷数据混在一起膨胀。
- DynamoDB 不是终态:它是"热缓冲层",最终数据要导到 S3 + Redshift 做分析。

**"打破 90 天保留期与单 execution 视角的可见性瓶颈"**

这句是这整段最该被记住的价值陈述:90 天保留是 AWS 硬性限制,单 execution 视角是 console UI 局限,这两个瓶颈是项目存在的根本理由。面试时要先把"瓶颈"摆出来,再讲方案,不要直接堆服务名。

**面试官可能追问**

- "为什么不直接从 S3 export 拉 execution history?" —— Step Functions 没有官方 history export API,只能靠 EventBridge 事件实时采集,或者用 `ListExecutions` + `GetExecutionHistory` 接口拉,但后者有 throttle 限制并且只能拉最近 90 天。
- "EventBridge 投递保证什么级别?" —— at-least-once,所以下游必须做幂等,衔接到下一段。

---

## 2.3 可靠性:"conditional write 幂等 + sparse update + SQS 批量消费 + DLQ 重放,实现 effectively-once"

这一行密度极高,拆四个子点。

### 2.3.1 conditional write 幂等

EventBridge 是 at-least-once,同一个 event 可能被投递 2 次甚至 3 次。Lambda 用 `event_id` 做 DynamoDB conditional write:`PutItem` 时 `ConditionExpression: attribute_not_exists(event_id)`,重复 event 写第二次会失败,业务无副作用。

### 2.3.2 sparse update 写入模型

这是这段最容易被追问、也最体现细节的点。

Step Functions 的一次 execution 会发出多个事件:`ExecutionStarted` → `StateEntered(校验)` → `StateExited(校验)` → `StateEntered(审核)` → ... → `ExecutionSucceeded`。如果你想把这条 execution 的完整状态视图沉淀到一行(execution_id 作为主键),你不能等所有事件齐了再写——事件是异步乱序到达的。

sparse update 的做法是:每个事件到达时,只更新它能确定的字段,其他字段不动。比如 `ExecutionStarted` 事件到达,只写 start_time,不动 end_time;`ExecutionSucceeded` 到达,只写 end_time 和 status,不动 start_time。

具体实现:DynamoDB 的 `UpdateItem` + UPDATE action,而不是 `PutItem`(后者会整行覆盖)。配合每个字段独立的 `ConditionExpression`(比如"只有当 end_time 不存在时才写入 end_time")处理乱序。

**为什么这一招重要**

- 解决了乱序到达问题:`ExecutionSucceeded` 可能比某个中间 `StateExited` 先到,sparse update 不会被乱序覆盖。
- 解决了部分失败问题:某些字段写失败可以单独重试,不影响其他字段。

**面试官可能追问**

- "如果两个事件同时到达更新同一字段怎么办?" —— UpdateItem 是单行原子,加 conditional expression(比如"只在 attempt_count < incoming.attempt 时更新")就能保证后到的胜出 / 先到的胜出,看业务语义。

### 2.3.3 SQS 批量消费 + DLQ

Lambda 直接订阅 EventBridge 当流量突增(比如某个 region 大批 onboarding 同时启动)会被 Lambda 并发上限打挂。中间加一层 SQS 做缓冲:EventBridge → SQS → Lambda 批量拉(`BatchSize: 100`),把瞬时洪峰摊平到几分钟内消化。

DLQ(Dead Letter Queue):Lambda 处理失败超过 maxReceiveCount(比如 3 次),消息进 DLQ,不会无限重试堵住主队列。Ops 看 DLQ 大小告警,从 DLQ 重放消息修问题。

**面试官可能追问**

- "为什么不用 Kinesis?" —— Kinesis 适合"有序、按 shard partition、高吞吐"场景,这里事件乱序本来就要处理,顺序不重要;SQS 更简单、运维成本更低,而且 SQS 原生集成 DLQ。

### 2.3.4 effectively-once 是什么

at-least-once + 幂等 = effectively-once。不是真的"恰好一次"(分布式系统理论上不存在严格的 exactly-once),而是业务侧观察不到重复。说"effectively-once"比说"exactly-once"准确,也是面试加分点——说明你懂 message delivery semantics。

## 2.4 ETL 链路:"DynamoDB Export to S3 + Glue(PySpark)构建天级增量 ETL,维度建模后写入 Redshift"

**这句在说什么**

DynamoDB 是热缓冲,适合点查 / 高频写,不适合扫描分析(全表 scan 又慢又贵)。所以每天定时把 DynamoDB 导出到 S3,在 S3 上用 Glue(PySpark)做 ETL,最终写到 Redshift 做分析。

**为什么是 DynamoDB Export to S3 而不是 DynamoDB Streams**

- Streams 是实时增量:适合做实时 CDC,但有 24h 保留期,跑 ETL 不需要这么实时。
- Export to S3 是离线快照:T-1 一次,无 RCU 消耗(不占 DynamoDB 读容量),最适合天级 ETL。
- 用 incremental export(2023 年才出的功能)只导前一天的变更,数据量进一步压缩。

**Glue(PySpark)做什么**

- 反范式化:把 event 表展开成 execution 视角的宽表。
- 维度建模:输出"事实表 + 维度表",事实表是 execution 维度(每行一条 execution,带 start_time / end_time / duration / status / region),维度表是 vendor 维 / state 维 / region 维。这是 Kimball 维度建模法,数仓标配。
- 写 Redshift 用 `COPY` from S3,比 INSERT 快 10x+。

**面试官可能追问**

- "为什么不直接在 DynamoDB 上跑分析?" —— DynamoDB 不支持 JOIN / GROUP BY,做不了多维聚合;且 scan 整表成本极高。
- "为什么用 Redshift 而不是 Athena?" —— Athena 适合 ad-hoc 查询,QuickSight 接 Athena 大盘刷新慢;Redshift 列存预聚合后 dashboard 响应在秒级,且支持物化视图。
- "维度建模为什么不直接宽表?" —— state machine 会演化(SCD Type 2),用维度表才能历史可追溯;宽表对 join 性能友好但失去版本能力。

---

## 2.5 价值闭环:"QuickSight 自助式 Dashboard,运营周报从小时级降至分钟级"

**这句在说什么**

Redshift 接 QuickSight 出大盘:执行耗时分布、失败 Top N(按 state、按 vendor、按 region)、SLA 达成率(承诺 X 天内完成 onboarding 的比例)、卡点漏斗(每个 state 的流转率)。

**"自助式 Dashboard 替代人工拉数"**

之前 Ops 要 onboarding 数据,得发邮件给数据团队,数据团队写 SQL、跑、整理 Excel、回邮件,一来一回 2-4 小时。现在 Ops 自己在 QuickSight 上拖筛选条件,分钟级出图。

**为什么这句重要**

简历项目最容易被诟病的是"技术做了一堆但说不出业务价值"。最后这一句把价值落到可量化的运营效率提升——这是面试官最爱听的"技术 → 业务影响"闭环表达。

**面试官可能追问**

- "周报到底节省了多少人时?" —— 提前准备好真实数字(比如"每周节省 6 个数据分析师小时 + Ops 当天能自己看到数据,决策时延从 T+1 降到 T+0")。这种数字最好真实,需要补齐实际指标。

# 整体面试串讲建议

讲这两段时,先抛"两层结构":

> 这个项目我做了两层。底层是 vendor onboarding 工作流编排系统,核心矛盾是"长事务 + 跨服务 + 人工节点";上层是工作流观测平台,核心矛盾是"Step Functions console 只能看单 execution、保留 90 天,且 Ops 拉数据效率低"。两层加起来回答了"流程怎么跑起来"和"流程跑得好不好"两个问题。

然后讲底层用三个关键词锚定:**长事务 taskToken / 可恢复一致性 Saga + 幂等 / 大 payload 外置 S3 reference**。

讲上层用三个关键词锚定:**事件采集打破 90 天 + 单 execution 瓶颈 / sparse update 解决乱序 + at-least-once / 维度建模 + Redshift + QuickSight 实现自助 BI**。

这样不管面试官从哪里切入,都能 30 秒落到你最熟的那条主线上。

## 常见误区与避雷提示

1. 不要一上来就堆服务名(Step Functions、Lambda、Glue、Redshift……),先抛"瓶颈和核心矛盾",再用服务做答案。
2. 不要把幂等讲成"做了 DynamoDB conditional write",要主动分四层讲(入口 / 编排 / 执行 / 事件)。
3. 不要把 Saga 讲成"分布式事务",要明确说"放弃过程一致性,换最终一致性"。
4. 不要把 effectively-once 和 exactly-once 混说,前者是工程上的实现,后者是理论上不存在的目标。
5. 不要把 sparse update 讲成"部分字段写入",要明确说它解决的是"事件乱序到达 + 部分字段失败重试"两个具体问题。
6. 不要忘了维度建模时 state machine 演化的兼容(SCD Type 2),这是最容易被追问的细节。
7. 不要把价值收尾讲成"提升了体验",要落到可量化的数字(分钟级出图、节省 X 人时、SLA 达成率从 X 升到 Y)。
