# AWS商户入驻与数据发布系统设计学习进度

## 学习主题

AWS 商户入驻工作流与数据发布系统设计

## 学习者画像

- 背景：5 年后端开发经验
- 目标：把一个偏 AWS 技术栈的商户入驻题，讲成结构化、可追问、可落地的系统设计面试答案
- 当前诉求：不写代码，重点整理设计思路、选型理由、流程编排和数据发布链路
- 学习重点：主交易链路与分析链路分离、Step Functions 工作流、DynamoDB 状态建模、Glue/Redshift 数据发布、面试表达

## 当前判断

- 优势假设：对微服务、数据库、异步化、分布式事务与补偿已经有基础理解
- 风险假设：容易把 AWS 服务直接堆出来，但没有先定义业务边界、状态模型和取舍理由
- 当前阶段：已完成该专题的第一轮结构化整理，适合继续做口头演练和追问强化

## 学习路径

| 课次 | 主题 | 目标 | 状态 |
| --- | --- | --- | --- |
| 01 | 需求澄清与总体架构 | 先把题目拆成交易链路和分析链路，明确边界和高层图 | 已完成 |
| 02 | 基于 Step Functions 的入驻工作流设计 | 讲清状态机步骤、人工审核、补偿、幂等和进度查询 | 已完成 |
| 03 | DynamoDB Streams 到 Redshift 的数据发布链路 | 讲清 CDC、S3 raw、Glue、Redshift、Dashboard、邮件发布 | 已完成 |
| 04 | 面试表达、容量估算与 Trade-off | 把前三课压缩成面试回答，并准备好关键追问 | 已完成 |
| 05 | 项目难点与解决方案 | 能总结这个项目真正的技术难点，并讲清楚对应设计手段 | 已完成 |

## 下一步建议

1. 先不急着继续扩展新知识点，优先把第 4 课里的短版本答案练到能自然复述。
2. 每次练习时，强制自己先说“这个题的核心矛盾是什么”，避免一开口就讲服务名。
3. 先把第 5 课练熟，重点练“长流程、一致性、幂等、解耦、可观测性”五个关键词的串联表达。
4. 接下来可以继续补两类追问：一类是安全与合规，另一类是多区域部署与容灾。
