# Claude Code 源码学习 · 第 01 课:Claude Code 的项目定位与"Agent"的本质

## 内容讲解

### 一、表面答案:它就是一个能调 Claude 的命令行工具?

如果你只看 npm 上的 `@anthropic-ai/claude-code`,Claude Code 表面上就是一个再普通不过的 Node CLI。装上之后,在终端里敲 `claude`,就进入了一个交互界面。你输入自然语言,它回答你。

如果故事到此为止,那它就是一个"终端版 ChatGPT",顶多比 web 版 Claude 多了一个 TUI 包装。

但如果你真的用过几次 Claude Code,你会发现一些**让 web 版 Claude 永远做不到的事情**:

- 你说"帮我把这个项目里所有 `console.log` 改成 `logger.info`",它真的去改了。
- 你说"看一下当前分支落后 main 多少",它真的执行了 `git rev-list --count main..HEAD`,然后把数字告诉你。
- 你说"把刚刚的改动提交并推上去",它真的跑 `git add && git commit && git push`,而且如果有钩子失败,它会读 hook 输出再决定下一步怎么办。
- 你说"这个 bug 怎么解",它会先 `grep` 一遍代码、读几个文件,**然后再开口**——而不是凭空猜。

这些动作的共同点是:**它在你的机器上,以你的身份,做了一件之前只有你才能做的事**。

这一刻,它就跨过了 Chat 和 Agent 的分水岭。

### 二、Agent 与 Chat 的边界:决定性变量是"动作权"

把 LLM 应用按"自主性"排个序,大致是这样:

```
              更可控                                 更"智能"
   ┌─────────┬───────────┬──────────────┬────────────────┐
   │ 模板填空 │ Chatbot    │ Tool-using    │ Agentic Loop    │
   │(GPT 写邮件)│(Claude.ai) │ Chat(早期 GPT-4│(Claude Code/    │
   │          │           │ Function Call)│ Devin/Codex 等) │
   └─────────┴───────────┴──────────────┴────────────────┘
```

- **模板填空**:用户输入就是 prompt,模型一次输出就结束。整个过程是 *一次函数调用*。
- **Chatbot**:多轮对话,每轮还是"用户问→模型答"。模型不"动手",只"动嘴"。
- **Tool-using Chat**:模型可以请求调用工具,**但执行权仍在外部框架**,通常一两次工具调用就结束。
- **Agentic Loop**:模型在一个**循环**中持续输出 → 调用工具 → 看到结果 → 继续输出,直到自己判断"任务完成"。这个循环的退出由模型自己决定。

Claude Code 是上面这条线最右侧的形态。它和 Claude.ai 的区别**不是模型不同**(底层是同一个 Claude 模型),而是:

> Claude Code = **Claude 模型** + **一个能在你机器上执行动作并把结果喂回去的循环**。

后者那个"循环"——业内叫 **Agent Loop** 或 **ReAct Loop**——才是真正让它"成为 Agent"的东西。模型本身没有变,它仍然是一个无记忆、无副作用的纯函数 `f(messages) → message`。但只要外面套上一个能执行工具的循环,它就突然能"做事"了。

这是一个工程上极其优雅的事实:**Agent 的智能,不在模型里,而在循环里。**

### 三、给 Agent 一个可操作的工程定义

很多人对"Agent"的理解停留在"很高级的 ChatGPT"。这种定义没有任何工程指导价值。我们换一个能写出代码的定义:

> **Agent = 一个由 LLM 驱动的、能在受控环境中持续观察-决策-行动直至任务完成的循环系统。**

把它拆成五个必备零件:

| 零件 | 含义 | Claude Code 中的对应 |
| --- | --- | --- |
| **大脑(Reasoner)** | 一个 LLM,负责把"当前状态"映射到"下一步动作" | Claude 模型(claude-opus / sonnet / haiku) |
| **手脚(Tools)** | 一组明确定义的、可执行的动作 | Read / Edit / Write / Bash / Grep / Task / WebFetch / MCP 工具 |
| **眼睛(Observation)** | 把工具结果反馈给大脑的通道 | `tool_result` 消息 |
| **心跳(Loop)** | 让"想 → 做 → 看 → 再想"持续下去的控制流 | 主循环里的 while:只要最后一条 assistant 消息里还有 `tool_use`,就继续 |
| **门闸(Policy/Permission)** | 哪些动作能做、哪些必须问、哪些一定不能做 | 权限模式、Hooks、安全规则、提示词注入防御 |

**任何缺一个的,都不算 Agent。** 这就是判断一个所谓"Agent 框架"是否名副其实的清单。

举几个反例:

- 一个用 LangChain 写的,只会 `RetrievalQA.run(question)` 的脚本——没有 Loop,**不是 Agent**,是 RAG。
- 一个 ChatGPT 插件,模型每次最多调一次工具——没有真正的 Loop(只有一跳),**算半个 Agent**。
- 一个用 GPT 自动生成 SQL 的工具——没有 Tool 反馈、没有 Loop,**完全不是 Agent**。

而 Claude Code 这五样**都有,而且都做到了生产级**。这是它值得我们深入研究的根本原因。

### 四、为什么"动作权"这件事这么关键

这里要扎进去一点。后端工程师看待"AI 集成"通常有两条思路:

- **路线 A:把 AI 当函数**。后端给一段 prompt,LLM 返回一段 JSON,后端反序列化后继续走业务流程。模型只是函数调用链上的一环。
- **路线 B:把 AI 当主控**。LLM 拥有调度权,它决定接下来该调哪个函数、读哪个文件、跑哪个命令,后端只负责"提供能力"和"兜底约束"。

路线 A 是确定性的、可测试的、好回滚的。绝大多数生产系统用的就是这个。
路线 B 是不确定的、长尾的、出错时责任难界定的。它换来的是**未知任务的处理能力**——你不需要预先把流程写死,只要把"工具"和"约束"写好,模型自己拼出流程。

Claude Code 是彻底的路线 B。它的内部并没有一张"用户想干什么 → 该走什么流程"的状态机。它只有:

```
工具集 + 权限策略 + 系统提示 + 用户输入 → 让模型自己决定每一步
```

你可以把它和你正在做的 EMF 流程编排平台对比着看:

- **EMF**:DAG 是**人写好的**,运行时引擎只是按图执行。AI 顶多帮你"生成 DAG"或"诊断异常",但执行的骨架是确定的。
- **Claude Code**:没有 DAG。每一步要不要调工具、调哪个、参数是什么,**全是模型在每一轮 inference 里现场决定的**。

这两条路线没有谁优谁劣,只有适不适合。EMF 适合**重复的、可被精准描述的**业务流;Claude Code 适合**长尾的、难以提前枚举的**编程任务。后面的课程我们会反复回到这个对比。

### 五、Claude Code 在"用户 / 模型 / 机器"三角里的位置

画一张图把它在系统中的位置摆清楚:

```
        ┌──────────────────────────┐
        │          用户            │
        │ (在终端里敲自然语言)      │
        └────────────┬─────────────┘
                     │ 自然语言 / Slash 命令
                     ▼
        ┌──────────────────────────┐
        │        Claude Code       │
        │  (本地 Node 进程, 主循环) │
        │                          │
        │  ┌────────────────────┐  │
        │  │   Agent Loop       │  │
        │  │  (while tool_use)  │  │
        │  └────────────────────┘  │
        │                          │
        │  ┌────────────┐  ┌────┐  │
        │  │ 工具执行器  │  │权限│  │
        │  │ Read/Edit/  │  │门闸│  │
        │  │ Bash/Task...│  │     │  │
        │  └─────┬──────┘  └─────┘  │
        └────────┼─────────────┬────┘
                 │ HTTPS       │ 文件 IO / 子进程
                 ▼             ▼
        ┌──────────────┐  ┌──────────────────┐
        │ Anthropic API│  │  用户本机环境    │
        │ (Claude 模型)│  │ (源码/git/shell) │
        └──────────────┘  └──────────────────┘
```

这张图里有一个关键事实容易被忽略:**Anthropic API 那一侧是无副作用的**。模型不会动你的文件,它只会"说话"——说"我想调 Read 工具读文件 X"。真正去读文件的是**本地 Claude Code 进程**。

也就是说,网络上传输的全部是**消息**(文本、tool_use 请求、tool_result),没有任何"动作"穿过那条 HTTPS。所有动作都发生在本地。

这个设计有两个直接后果:

1. **隐私**:你的代码内容确实会被发到 Anthropic API,但**执行权一直在本地**。模型再怎么"想"删你硬盘,也得先发一个 `tool_use` 回来,本地循环再去执行——而本地有权限门闸。
2. **离线能力的天花板**:一旦没网,模型就罢工。这也是为什么 Claude Code 不能像传统 IDE 那样"离线工作"。

### 六、与传统 CLI、IDE 插件的关键差异

把 Claude Code 和它**最像但不是**的几类东西对比一下,边界就清楚了:

| 对照对象 | 表面相似点 | 本质区别 |
| --- | --- | --- |
| 传统 CLI 工具(如 `git`、`make`) | 都是终端里敲命令 | 传统 CLI 命令语义是**确定的**,Claude Code 的"命令"是自然语言,语义由模型每次重新解释 |
| Shell 脚本 | 都能编排多个命令 | 脚本是**预先写死的流程**,Claude Code 是**在线决策的循环** |
| IDE 插件(Copilot 类) | 都能改代码 | Copilot 是**补全器**,只在光标处吐字符;Claude Code 是**智能体**,自己决定该改哪个文件 |
| Web 版 Claude.ai | 都能对话 | Web 版没有访问你机器的能力,**没有 Tool**,所以**没有 Agent Loop** |

记住这一句话就够了:

> **Claude Code 是一个把"模型"和"机器"用循环粘起来的薄壳。** 它的"智能"全部来自模型,它的"价值"全部来自那个粘合层。

这个粘合层正是我们后面 19 课要逐层拆开的东西。

### 七、几个常见误解,提前消毒

**误解 1:"Claude Code 比 Claude.ai 更聪明,所以它能写代码。"**
错。模型一样。聪明的是**循环**:它能反复调工具、看结果、纠偏,而 Claude.ai 只能一次性输出。

**误解 2:"Claude Code 内部有一个很复杂的状态机来理解任务。"**
错。它没有显式状态机,**所有"理解"都发生在每一轮模型的注意力里**。它内部的"复杂"主要是工程性的——权限、流式、注入防御、上下文压缩——而不是认知性的。

**误解 3:"Claude Code 提示词都是写死的,只要拿到 system prompt 就能复刻。"**
半对半错。System prompt 确实有可观察的稳定结构,但**真正难复刻的是工具集的设计 + 权限模型 + 上下文管理策略**——这些是几年工程沉淀,不是抄一段 prompt 就能搞定的。

**误解 4:"Agent 就是一堆 prompt,没什么技术含量。"**
最常见也最致命的一种偏见。Prompt 只是 Agent 系统**最薄的一层**,它下面是循环控制、工具协议、流式协议、权限引擎、注入防御、持久化、TUI 渲染、扩展机制……每一层都是工程问题。这门课就是要把这些层一层层翻给你看。

---

## 小结(关键点抽出来)

1. **Claude Code = 模型 + 工具 + 循环 + 权限**。 缺一个都不算 Agent。
2. **智能不在模型里,在循环里**。模型仍然是无状态纯函数,Agent 性来自外部那层 while。
3. **动作权全部在本地**。Anthropic API 只传消息,所有副作用动作都在你的进程里执行,这决定了它的隐私边界和权限设计起点。
4. **它走的是"AI 主控"路线**,与你日常做的"AI 当函数"业务系统是相反方向的设计。理解这点,后面所有"为什么 Claude Code 这么设计"才有答案。
5. **它和 EMF 编排平台正好成对照**:一个是预先写死的 DAG,一个是在线决策的循环。两者的取舍不一样。
