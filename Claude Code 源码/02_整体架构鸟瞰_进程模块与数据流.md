## 内容讲解

### 一、为什么先看全景?

后端工程师看陌生系统,有一种本能是直接钻进某个模块去读代码。这在分层清晰的传统服务里没问题——一个 Spring Boot 项目有 Controller / Service / Dao,看 Controller 就能猜出全貌。

但 Claude Code 不是这种"水平分层 + 请求穿透"的系统。它的真正复杂性集中在**那个循环**——一个请求会**反复**穿过 API、Tool、Permission、Render,可能穿十几次才结束。这种系统如果你不先把"循环"和"分叉点"画在脑子里,每次读到某段代码都不知道它处于第几次回环。

所以这节我们做的事很简单:**画三张图**——进程拓扑图、逻辑分层图、数据流时序图——让你脑子里有一份**坐标系**。后面所有具体机制,都会被钉到这份坐标系上。

### 二、进程拓扑:谁是谁的"邻居"

Claude Code 不是一个单体进程,但它**主体确实是一个 Node 进程**。围绕这个主进程,会按需拉起若干子进程或外部连接:

```
                            ┌──────────────────────────────────────────┐
                            │      Claude Code 主进程 (Node.js)         │
                            │                                          │
                            │  ┌───────┐  ┌───────────┐  ┌──────────┐  │
                            │  │ TUI   │  │ Agent     │  │ Tool     │  │
                            │  │ (Ink) │  │ Loop      │  │ Registry │  │
                            │  └───┬───┘  └────┬──────┘  └────┬─────┘  │
                            │      │           │              │        │
                            │  ┌───┴────┐  ┌───┴────┐  ┌──────┴─────┐  │
                            │  │Session │  │API     │  │Permission/ │  │
                            │  │Store   │  │Client  │  │Hooks       │  │
                            │  └────────┘  └────────┘  └────────────┘  │
                            └──┬───────┬─────────┬──────────┬──────────┘
                               │HTTPS  │stdio    │stdio     │socket/HTTP
                               │       │(spawn)  │(spawn)   │(MCP)
                               ▼       ▼         ▼          ▼
                        ┌─────────┐ ┌──────┐ ┌────────┐ ┌──────────┐
                        │Anthropic│ │ Bash │ │ Task   │ │ MCP      │
                        │ API     │ │子进程 │ │ 子代理 │ │ Server  │
                        │(网络)   │ │(本地) │ │(同进程)│ │(可远程)  │
                        └─────────┘ └──────┘ └────────┘ └──────────┘
```

几个**容易踩坑的事实**值得在这里强调一下:

1. **Anthropic API 是远程的,但它无副作用**——它只决定"接下来想干什么",真正动手发生在主进程及其子进程里。这点我们第 01 课已经埋伏过,后面权限模型(第 15 课)和注入防御(第 16 课)的所有设计都建立在这个事实上。
2. **Bash 工具是 spawn 出来的子进程**,不是在 Node 里 eval 一段 shell。这意味着每次 Bash 调用都有一次 fork/exec 开销,但换来的是清晰的隔离边界——子进程崩了不会拖死主循环。
3. **Subagent (Task 工具) 默认在主进程内**,只是"再起一个 Agent Loop 实例"。它的"隔离"不是进程级的,是**上下文级**的(开一份新的 messages 数组)。这层抽象的精妙之处会留到第 11 课。
4. **MCP Server 是协议级的外部进程**——它甚至可以跨机器。MCP 的存在让"Tool"这个概念脱离了 Claude Code 的代码仓库,变成一个开放生态。第 14 课专门讲。

进程图最关键的洞察是:**Claude Code 主进程的角色是"调度者",而不是"执行者"**。绝大多数实际工作(API 推理、shell 命令、MCP 工具)都在外部完成,主进程只负责拉数据、做决策、写文件、改终端。

### 三、逻辑分层:每一层只做一件事

把主进程内部按职责拆开,大致是九层。读这张图的方法:**从上往下看是请求方向,从下往上看是响应方向**:

```
┌────────────────────────────────────────────────┐
│ ① Input 层  (TUI / Ink, 也即"前端")             │
│   职责:读 stdin、渲染 stdout、解析 / 与 @       │
├────────────────────────────────────────────────┤
│ ② Session 层                                    │
│   职责:维护 messages 数组、transcript 文件、   │
│         resume 时复原状态                       │
├────────────────────────────────────────────────┤
│ ③ Prompt 构造层                                 │
│   职责:把静态 system 模板 + CLAUDE.md +        │
│         环境信息 + 工具清单 拼成最终 system     │
├────────────────────────────────────────────────┤
│ ④ Agent Loop  (本系统的"心脏")                  │
│   职责:while not done: 调 API → 派工具 →       │
│         喂结果 → 再调 API                       │
├────────────────────────────────────────────────┤
│ ⑤ API Client 层                                 │
│   职责:HTTP/SSE 流式协议、重试、prompt cache,  │
│         把字节流解码成事件流                    │
├────────────────────────────────────────────────┤
│ ⑥ Tool Registry & Dispatcher                    │
│   职责:工具元数据(JSON Schema)、按名分发、     │
│         参数校验                                │
├────────────────────────────────────────────────┤
│ ⑦ Permission & Hooks                            │
│   职责:allow / ask / deny 三态决策、           │
│         PreToolUse / PostToolUse 钩子           │
├────────────────────────────────────────────────┤
│ ⑧ Tool Executor                                 │
│   职责:真正去 Read / Edit / Bash / fetch / MCP │
├────────────────────────────────────────────────┤
│ ⑨ Render & Persist                              │
│   职责:把流式事件变成屏幕字符;把每条消息      │
│         落到 .jsonl                             │
└────────────────────────────────────────────────┘
```

**这套分层的核心设计哲学是:把"循环"集中在第 ④ 层,其余各层都是"请求-响应"式的单向函数。**

- 第 ① 层只关心字符进出,不知道什么是 LLM。
- 第 ⑤ 层只关心 HTTP/SSE,不知道什么是工具。
- 第 ⑥ 层只关心"这个工具叫什么、参数对不对",不知道权限是什么。
- 第 ⑦ 层只关心"这个动作能不能做",不知道这个动作是 Bash 还是 Edit。
- 第 ⑧ 层只关心"把这个动作执行了,把结果回传",不关心循环什么时候停。

**只有第 ④ 层 Agent Loop 知道"循环"这件事。** 它是唯一一个"有状态机概念"的模块。这是非常重要的工程取舍——把控制复杂度收敛在一个地方,其他模块全都做成无状态/低状态的纯函数。

后面 18 节课,我们其实就是在轮流放大这九个盒子:

| 盒子 | 对应课次 |
| --- | --- |
| ① Input 层 | 13(Slash 命令)、18(TUI)|
| ② Session 层 | 17(持久化)、19(Memory) |
| ③ Prompt 构造层 | 06(System Prompt)、07(上下文压缩)|
| ④ Agent Loop | 03(主循环)、04(Tool Use 循环)|
| ⑤ API Client | 05(流式响应)|
| ⑥ Tool Registry | 08(工具定义)|
| ⑦ Permission & Hooks | 12(Hooks)、15(权限)、16(注入防御)|
| ⑧ Tool Executor | 09(调度)、10(文件类工具)、11(Subagent)、14(MCP)|
| ⑨ Render & Persist | 18(TUI)、17(持久化)|

把这张映射表存进脑子,你后面随时知道你在学**整张图的哪一块**。

### 四、一次完整交互的数据流(时序视角)

举一个最朴素的例子。用户输入:

> "看一下 src/auth.ts 这个文件,告诉我它做什么的"

我们跟着这一句话走完一遍完整循环:

```
[用户敲回车]
    │
    ▼
①  Input 层
    └─ 既不是 / 开头也没有 @,纯文本
    └─ 把"看一下 src/auth.ts ..."包装成一条 user message
    │
    ▼
②  Session 层
    └─ 追加到 messages,同时写一行进 transcript .jsonl
    │
    ▼
③  Prompt 构造层  (只有第一次循环或 system 变了才重算)
    └─ 拼出 system_prompt:
       静态模板 + 工具清单(Read/Edit/Bash/...) +
       CLAUDE.md 内容 + 当前工作目录 + git 分支等
    └─ 给 system 部分打上 cache_control(下一轮还能命中)
    │
    ▼
④  Agent Loop 第 1 次进入
    │
    ▼
⑤  API Client
    └─ POST /v1/messages?stream=true
    └─ 拿到 SSE 字节流,边读边解码成事件
        │
        │  逐事件返回:
        │   - message_start
        │   - content_block_start(type=text 或 tool_use)
        │   - content_block_delta * N
        │   - content_block_stop
        │   - message_stop
        │
        ▼ (一边解一边送)
    ⑨  Render 层
        └─ text_delta:打到 TUI
        └─ tool_use:渲染一个"准备调用 Read(src/auth.ts)..."占位
    │
    ▼
④  Agent Loop:本轮 assistant 消息组装完成
    └─ 检查最后一条 assistant 消息里有没有 tool_use?
       ├── 没有 ──→ 把控制权交回 ① Input,等用户下一句  (退出 loop)
       └── 有  ──→ 进入工具派发分支 ↓
    │
    ▼
⑥  Tool Registry & Dispatcher
    └─ 工具名 = "Read" → 找到 handler
    └─ 参数 = { file_path: "src/auth.ts" } → JSON Schema 校验通过
    │
    ▼
⑦  Permission & Hooks
    └─ Read 默认 allow(只读) → 不打断用户
    └─ 触发 PreToolUse hook(若用户配了)
    │
    ▼
⑧  Tool Executor
    └─ fs.readFileSync("src/auth.ts") → 文件内容
    └─ 触发 PostToolUse hook
    │
    ▼
②  Session 层
    └─ 把内容包装成 tool_result,追加到 messages,写 transcript
    │
    ▼
④  Agent Loop 第 2 次进入  ←──────  (循环!)
    │
    ▼
⑤  POST /v1/messages?stream=true   再发一次
    └─ 这次 system + 之前的轮次都命中 prompt cache,只重算最后一条
    └─ 模型这回拿到了文件内容,不再 tool_use,而是开始解释代码
    │
    ▼
⑨  Render
    └─ 流式打字效果输出 "src/auth.ts 是一个登录模块..."
    │
    ▼
④  Agent Loop:assistant 消息无 tool_use → 退出 loop → 回 ①
```

这张时序图里**最值得反复看的两个点**:

1. **循环的入口和出口都是同一个判定:"最后一条 assistant 消息里有没有 tool_use?"** 这就是 Agent Loop 的全部停止条件。它没有像传统状态机那样"达到 success 状态才停",而是把停止权完全交给模型。模型说不调工具了,循环就退。这种设计极简,但也极依赖模型的判断力——这正是为什么 Anthropic 在系统提示里要花大力气塑造"什么时候停"的行为。
2. **Prompt Cache 的存在让循环代价可控**。如果没有 cache,每次循环都把整个 messages 历史 + system prompt 重新付费送上去,几个 tool 来回就吃掉几十万 tokens。Cache 把"已经发过的部分"标记为可复用,让一次循环的边际成本接近"只发增量"。第 07 课会展开。

### 五、热路径上的关键设计取舍

把上面这条数据流再抽象一层,你会发现有几个关键设计**反复出现**,值得现在就点出来:

**(a) 流式优先,而不是请求-响应**
Claude Code 几乎所有路径都是流式的——API 返回是 SSE,TUI 渲染是逐 token,工具结果哪怕是一次性返回的也用同样的事件接口注入。流式带来的好处是**首字节延迟低**,但代价是工程复杂度——任何一段半成品 assistant 消息(只到一半的 tool_use 块)都得能被中断、回退、续传。第 05 课专门讲。

**(b) 状态尽量集中,模块尽量无状态**
真正持有"对话状态"的只有 ② Session 层那个 messages 数组。其他所有模块都是函数式的:输入消息,输出新消息或副作用。这让重放(replay)和 resume 变得简单——只要重新把 messages 喂给 Agent Loop,系统就回到了那个时刻。

**(c) 权限是切面,不是分支**
Permission 没有埋在每个工具的实现里,而是统一在 ⑦ 层挡一刀。这是一个 AOP 风格的设计:工具实现者完全不需要写"我这个动作要不要问用户"的代码,所有这类决策被外置成规则配置。这种切面化让 Permission 模型可以独立演化(允许列表、模式切换、Hook 干预),不污染工具实现。

**(d) 扩展走"协议"而不是"插件"**
Claude Code 没有传统意义上的"插件 API",而是通过 MCP(Model Context Protocol)这样一份开放协议把外部工具拉进来。这意味着 Anthropic 不需要把第三方代码塞进自己的进程,第三方也不需要懂 Claude Code 内部结构。两边只需要遵守 MCP 协议。这是一个非常成熟的"边界设计"——用协议定义边界,而不是用 SDK。

### 六、与 EMF 的对照(刻意做的对比)

你正在做的 EMF 流程编排平台,在这套架构里大致对应什么?我们逐层对一下:

| Claude Code 层 | EMF 类比 | 关键差异 |
| --- | --- | --- |
| ① Input | DAG 提交入口(YAML / DSL / UI)| EMF 输入是结构化 DAG,Claude Code 是自然语言 |
| ② Session | DAG run 的 metadata 表 | 都是状态持有者,但 EMF 状态是"任务执行进度",Claude Code 状态是"对话上下文" |
| ③ Prompt 构造层 | (没有对应) | 这是 Agent 系统独有的——把上下文塞给"决策者"|
| ④ Agent Loop | DAG Scheduler | 都是循环调度,但 EMF 调度的"下一步"由 DAG 拓扑决定,Claude Code 的"下一步"由模型决定 |
| ⑤ API Client | (没有对应) | 因为 EMF 的"决策者"是写死的 DAG,不需要外部推理 |
| ⑥ Tool Registry | Operator 注册表 | **几乎一对一**——都是"按名称分发的可执行单元"|
| ⑦ Permission | RBAC + 审批流 | EMF 通常是"提前审批好的 DAG",Claude Code 是"运行时即时审批"|
| ⑧ Tool Executor | Operator Runner | **几乎一对一**——执行隔离、超时、重试机制是相通的 |
| ⑨ Render & Persist | 任务日志 + run history | 都是 append-only 的事件流 |

如果你硬要给 EMF 加一个 AI 赋能层,把它升级成"半 Agent",最自然的做法是:

> **保留 ④ DAG Scheduler,加一个旁路的"Agent 决策器"——只在 DAG 节点失败时被唤起,以"诊断/修复/补偿"为目标启动一个 mini Agent Loop。**

也就是说,EMF 的常规路径继续走"写死 DAG"的可靠路线,在异常分支才启用"Agent 主控"的探索能力。这是路线 A 和路线 B 的混合,正好取两者所长。

这条思路我们会在第 11 课(Subagent)和第 20 课(综合设计)再回来精讲。

## 小结

1. **三张图记住即可**:进程拓扑图、九层逻辑分层图、一次完整循环的时序图。
2. **主进程是调度者**,真正干活的是 API、子进程、MCP server。
3. **九层里只有 Agent Loop 一层有"循环"概念**,其他层尽量做成无状态函数。这是控制复杂度的关键招式。
4. **Agent Loop 的停止条件极简**:模型最后一条消息里没有 tool_use,就停。停止判断权完全交给模型。
5. **流式 + 缓存 + 切面权限 + 协议扩展**——这四条贯穿整个架构,后面每一课都会反复遇到它们。
6. **EMF 与 Claude Code 在 ⑥⑧ 层(Tool Registry / Executor)几乎一一对应**,真正不同在 ④⑤ 层——一个用 DAG 决策,一个用 LLM 决策。
