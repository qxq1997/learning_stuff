# 05 · 流式响应处理

> **核心问题:SSE 怎么解析成增量 UI?**
>
> 前两课反复出现"流式"这个词,但都一笔带过。这课我们把镜头对准数据流本身:
> 一帧一帧地看从"bytes 进来"到"Markdown 渲染出去"中间发生了什么,
> 以及当流在错误的位置被截断时,Agent Loop 要怎么善后。

---

## 一、为什么要流式?

先退一步问这个问题——不流式不行吗?

完全可以不流式:等模型把整条 assistant 消息生成完毕,一次性返回,然后 Agent 再处理。这是最简单的实现。

但这个方案的用户体验很糟。Claude Code 的回答动辄几百到几千 token,按 token/second 估算:
- Sonnet 4.x 大约 80~120 tokens/s
- 一个 500 token 的回答要等 4~6 秒才开始显示
- 一个 2000 token 的分析报告要等 15~25 秒才出来

用户在这段时间里只能盯着光标,不知道模型在干什么、有没有卡死。这是不可接受的。

流式的好处不只是"早点看到结果"。它还解决了几个工程问题:
- **中断能力**:用户能在流进行中按 Esc 打断,而不是等整个响应完成后才能干预。
- **tool_use 识别时机**:在流里尽早识别到 `tool_use` 块的开头,就能提前告知用户"准备调工具"。
- **超长响应的容错**:流式解析让截断的检测从"批量处理后发现"变成"实时发现"。

## 二、SSE 是什么

**SSE(Server-Sent Events)** 是 HTTP 长连接上的单向推送协议,格式极简:

```
data: {"type":"message_start","message":{"id":"msg_abc","model":"claude-sonnet-4-6",...}}\n\n

data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}\n\n

data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"我来"}}\n\n

data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"看一下"}}\n\n

data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"目录结构"}}\n\n

data: {"type":"content_block_stop","index":0}\n\n

data: {"type":"message_delta","delta":{"stop_reason":"end_turn","stop_sequence":null},...}\n\n

data: {"type":"message_stop"}\n\n
```

每行以 `data: ` 开头,后跟一个 JSON 对象,以 `\n\n` 结尾表示这帧结束。客户端逐行读,每读到 `\n\n` 就处理一帧。

Anthropic Messages API 的流式事件类型有这几种:

| 事件类型 | 含义 |
|---------|------|
| `message_start` | 本次响应的元信息(id、model、usage 初始值) |
| `content_block_start` | 一个新的 content 块开始(text 或 tool_use) |
| `content_block_delta` | 向当前 content 块追加增量内容 |
| `content_block_stop` | 当前 content 块结束 |
| `message_delta` | 消息级别的元信息更新(stop_reason 在这里出现) |
| `message_stop` | 整条消息结束 |
| `ping` | 心跳,无业务意义 |

## 三、流式解析的状态机

读完上面的事件列表,你会意识到流式解析本质上是一个**状态机**。客户端需要维护当前在"哪个 content 块里"、"这个块是 text 还是 tool_use"、"text 的当前累积内容是什么"。

用最简的伪代码描述这个状态机:

```python
state = {
    "current_block_index": None,
    "current_block_type": None,   # "text" or "tool_use"
    "text_buffer": "",
    "tool_use_json_buffer": "",   # 累积 tool_use 的 JSON 参数
    "tool_use_id": None,
    "tool_use_name": None,
    "stop_reason": None,
    "content_blocks": [],         # 最终收集的所有块
}

for event in sse_stream:
    if event.type == "content_block_start":
        state["current_block_index"] = event.index
        state["current_block_type"] = event.content_block.type
        if event.content_block.type == "tool_use":
            state["tool_use_id"] = event.content_block.id
            state["tool_use_name"] = event.content_block.name
            state["tool_use_json_buffer"] = ""

    elif event.type == "content_block_delta":
        if state["current_block_type"] == "text":
            state["text_buffer"] += event.delta.text
            render_incremental(event.delta.text)   # ← 实时渲染

        elif state["current_block_type"] == "tool_use":
            state["tool_use_json_buffer"] += event.delta.partial_json

    elif event.type == "content_block_stop":
        if state["current_block_type"] == "text":
            state["content_blocks"].append({"type": "text", "text": state["text_buffer"]})
            state["text_buffer"] = ""
        elif state["current_block_type"] == "tool_use":
            parsed_input = json.parse(state["tool_use_json_buffer"])
            state["content_blocks"].append({
                "type": "tool_use",
                "id": state["tool_use_id"],
                "name": state["tool_use_name"],
                "input": parsed_input,
            })
            state["tool_use_json_buffer"] = ""

    elif event.type == "message_delta":
        state["stop_reason"] = event.delta.stop_reason

    elif event.type == "message_stop":
        break   # 整条消息收完
```

两个关键点:

**1. text 块是增量渲染的,tool_use 块不是。**

`text_delta` 每来一帧就立刻 `render_incremental`,用户能实时看到文字出现。

但 `tool_use` 块的参数 JSON 是**流式到来的碎片**(`partial_json`),必须等 `content_block_stop` 来了、JSON 完整了,才能解析并执行。在此之前只是把碎片拼起来缓存在 `tool_use_json_buffer` 里。

为什么不边到边解析 JSON?因为 JSON 是结构化的——一个 partial JSON `{"command": "git lo` 还没法告诉你 `command` 的值是什么。只有完整的 `{"command": "git log --oneline -10"}` 才能被 parse。

**2. `stop_reason` 出现在 `message_delta` 里,不是 `message_stop` 里。**

`message_delta` 事件先来,里面带 `stop_reason`。`message_stop` 是最后一帧,是纯粹的"结束信号",没有业务信息。这个顺序很重要:等 `message_delta` 来了你就知道 stop_reason,不必等最后那帧。

## 四、TUI 的渲染管道

流式数据到了客户端,还要变成终端里的 Markdown 渲染。这是另一个有意思的工程问题。

### 4.1 增量 Markdown 渲染的难点

Markdown 是"批量解析"友好的格式——你把整篇文本交给 parser,它输出 AST,然后渲染。但流式场景要求"拿到一个字符就能渲染一点"——这和传统 Markdown parser 的设计是矛盾的。

难点在哪?来看一个例子:

```
模型在流式输出:

"这里有一个**重要"  ← 第一帧到达时,粗体还没闭合
"**的细节"         ← 第二帧到达,粗体闭合了
```

如果你在第一帧就解析渲染,`**重要` 是不完整的 Markdown——`**` 没有闭合。你没法知道该把它渲染成普通文字还是粗体开始。

经典解法:**推迟渲染**。不是"每来一个字符就渲染一个字符",而是维护一个小窗口 buffer,等到"安全可渲染的边界"才真正渲染。

"安全边界"的判断规则大致是:
- 遇到换行符:当前行可以渲染了(一行 Markdown 通常是完整语义单元)
- 遇到代码块闭合符(` ``` `):代码块可以渲染了
- 流结束:剩余 buffer 全部渲染

这让 Claude Code TUI 里看到的效果是"逐行出现",而不是"逐字出现"——因为渲染单元是"一行确定的 Markdown",不是"一个字符"。

### 4.2 代码块的特殊处理

代码块(` ```lang\n...\n``` `)有语法高亮需求,这让渲染更复杂:

- 开头 ` ```lang ` 到来时,客户端知道接下来是代码,可以切换到"代码模式"
- 代码内容逐行到来,可以逐行高亮渲染(代码的行是独立的)
- 结尾 ` ``` ` 到来时,代码块结束

代码块里的逐行高亮相对好处理——代码语言的 tokenizer 通常能处理"到当前位置为止"的增量高亮。

### 4.3 工具调用的 UI 呈现

流里出现 `content_block_start` 且 type 是 `tool_use` 时,TUI 不需要等 JSON 完整就能开始更新 UI——它已经知道了工具名字(`event.content_block.name`)。

所以 Claude Code 的 TUI 会在这个时刻立刻在界面上出现类似:

```
⏳ Bash
```

然后等 `content_block_stop` 来了、JSON parse 完成、工具执行开始后,变成:

```
⚙ Bash: git log --oneline -10
```

执行完成后再变成:

```
✓ Bash: git log --oneline -10  (0.3s)
```

这种三段式 UI 反馈就是流式解析和 TUI 渲染紧密协作的产物:**工具名字来了立刻显示,参数来了追加显示,结果来了完成显示**——每一步用户都能看到进度。

## 五、截断场景的处理

现在来正面处理上两课都提到、但故意留到这里讲的问题:**流被截断时怎么办?**

### 5.1 截断发生在哪里

有两类截断:

**A. 网络截断**:HTTP 连接中途断掉,SSE 流突然停了,没有 `message_stop`。

**B. max_tokens 截断**:模型生成达到 token 上限,API 正常发了 `message_delta`(里面 `stop_reason: "max_tokens"`),然后发 `message_stop`。从协议层看这是正常结束,但语义上内容是不完整的。

这两类的处理方式不同。

### 5.2 网络截断的处理

网络截断:流异常中止,最后一帧不是 `message_stop`。

这时候 `state["stop_reason"]` 还是 None(因为 `message_delta` 还没来),`state["tool_use_json_buffer"]` 可能也是半截的字符串。

处理方式:
1. **丢弃这条不完整的 assistant 消息**——不写进 messages,不写进 transcript。
2. **给用户提示**:"连接中断,请重试"。
3. **messages 数组回滚到本轮 API 调用之前的状态**。

这等价于"本轮推理作废,用户重新发指令"。因为 messages 没有被修改,重试是安全的。

### 5.3 max_tokens 截断的处理

max_tokens 截断更复杂,因为要看截断的位置:

**位置 A:截断在 text 块里**

```
"content": [
  {"type": "text", "text": "我分析了一下代码结构,发现问题在于...【截断】"}
]
stop_reason: "max_tokens"
```

处理相对简单:把这条不完整的 assistant 消息写进 messages(保留已有分析),然后告诉用户"输出被截断"。用户可以让模型继续。

**位置 B:截断在 tool_use 块的 JSON 里**

```
"content": [
  {"type": "text", "text": "我来执行这个命令:"},
  {"type": "tool_use", "id": "toolu_abc", "name": "Bash",
   "input": "【JSON 不完整,parse 失败】"}
]
stop_reason: "max_tokens"
```

这是危险情况:
- `tool_use_json_buffer` 里的字符串 parse 失败
- 这个 tool_use 块无法执行
- 但 assistant 消息已经表达了"我打算调工具"的意图

Claude Code 的处理:
1. **不执行这个 tool_use**——参数不完整,执行是危险的。
2. **保留这条 assistant 消息进 messages**。
3. **追加一条 user 消息**,内容类似:

```
[tool_result tool_use_id="toolu_abc" is_error=true content=
"工具调用因 max_tokens 截断而参数不完整,无法执行。请重新发起。"]
```

4. **再调一次模型**——让模型看到这条 error result,自主决定怎么处理(通常是重新发一个完整的 tool_use)。

这个处理方式体现了一个一致性原则:**每个 tool_use 都必须有对应的 tool_result**,哪怕这个 tool_result 的内容是"无法执行"的错误。违反这个约束会让 messages 数组处于 API 不接受的状态(有 tool_use 没有对应 result),后续所有 API 调用都会报错。

### 5.4 截断场景的统一心智模型

把三种截断情形放在一起:

| 情形 | messages 状态 | 用户看到什么 | 循环怎么继续 |
|------|--------------|------------|------------|
| 网络中断 | 回滚,不写入 | "连接断开" | 用户重新输入 |
| max_tokens 截断 text | 写入不完整 assistant 消息 | "输出被截断" | 用户让模型继续 |
| max_tokens 截断 tool_use | 写入 assistant 消息 + 错误 tool_result | 工具调用失败提示 | 模型自主重试 |

共同点:**截断永远不是沉默的**——它一定会在 messages 里或用户界面上留下痕迹,让模型和用户都知道发生了什么。

## 六、流式和 tool_use 共存的完整时序

把"一条既有 text 又有 tool_use 的 assistant 消息"的完整流式时序画出来:

```
SSE 帧                          客户端行为

message_start                   初始化 state
content_block_start(text)       state: 进入 text 块
content_block_delta(text)×N     逐帧渲染到 TUI ← 用户实时看到
content_block_stop              text 块完成,写入 content_blocks
content_block_start(tool_use)   state: 进入 tool_use 块
                                TUI 显示 "⏳ Bash"
content_block_delta(partial_json)×M  累积 JSON buffer,不渲染
content_block_stop              JSON 完整,parse input
                                TUI 显示 "⚙ Bash: git log..."
message_delta(end_turn/tool_use)  记录 stop_reason
message_stop                    流结束

              ↓ 流结束后

if stop_reason == "tool_use":
    并发执行所有 tool_use 块
    TUI 显示工具执行进度
    收集 tool_result
    messages.append(user 消息含所有 tool_result)
    → 下一轮 API 调用(重新走上面流程)

if stop_reason == "end_turn":
    messages.append(assistant 消息)
    → 等待用户输入
```

注意时序的两个"等待点":

1. **text 渲染不等 tool_use**:text 块结束了就完成,不等后面的 tool_use 块出现才渲染。
2. **tool_use 执行要等流结束**:必须等 `message_stop` 来了,确认所有 tool_use 块都完整收到,才能开始并发执行工具。不能"tool_use 块一出现就立刻执行"——因为同一条消息里可能还有后续 tool_use 块,需要一起 fork-join。

## 七、usage 和 token 计数

流式响应里有一个经常被忽视的细节:**token 用量在哪里获取**?

`message_start` 事件的 `message` 字段里有 `usage` 的初始值(通常是输入 tokens):

```json
{
  "type": "message_start",
  "message": {
    "usage": {"input_tokens": 4523, "output_tokens": 0}
  }
}
```

`message_delta` 事件里有 `usage` 的更新(输出 tokens 在这里确定):

```json
{
  "type": "message_delta",
  "delta": {"stop_reason": "end_turn"},
  "usage": {"output_tokens": 312}
}
```

Claude Code 的 TUI 里实时显示的 token 计数和费用估算,就是从这两个地方读的。

这个设计有个有意思的含义:用户在流还没结束的时候就能看到"已用 input_tokens: 4523",但 output_tokens 要等流结束才知道精确值。这和你在超市购物——结账前已知食材价格(input),结账时才知道袋子费(output)——有点类似。

## 八、与 EMF 的对照

EMF 里有类似"流式"的概念吗?

有——日志流。EMF 的 Operator 执行过程中会持续写日志,Scheduler 可以实时拉取这些日志展示给用户。但这是"日志流",不是"控制流的流式"。

日志流的特点是:
- **只用于展示**,不影响 Scheduler 的决策
- Operator 的成功/失败状态还是批量判断(等 operator.execute 返回)

而 Claude Code 的 SSE 流是"控制流的流式":
- `stop_reason` 在流里出现,决定循环走哪条分支
- `tool_use` 的识别在流里发生,触发工具执行
- 流的中断会改变 Agent Loop 的状态

**EMF 的流式是可观测性工具;Claude Code 的流式是控制路径的一部分。** 这是两者最大的结构性区别。

如果你给 EMF 加 Agent 能力,流式处理会是一个必须面对的新问题——模型的输出流需要被实时解析,里面既有"给用户看的文字",也有"要执行的工具调用",还有"决定循环走向的 stop_reason"。这三件事都从同一个字节流里来,只能用状态机处理。

---

## 小结

1. **流式是必须的**:不流式意味着用户等待、无法中断、截断检测滞后。
2. **SSE 是帧协议**:`data: {...}\n\n` 逐帧到来,客户端维护状态机逐帧处理。
3. **text 块增量渲染,tool_use 块等完整才执行**:两者在流里并存,处理方式截然不同。
4. **截断分两类**:网络截断→丢弃本轮;max_tokens 截断→看位置分类处理,text 截断保留,tool_use 截断注入 error tool_result。
5. **截断永远不沉默**:截断信息必须写进 messages 或展示给用户——对模型和用户都不隐藏。
6. **工具执行等流结束**:必须确认整条消息的所有 tool_use 块都完整收到,才能开始 fork-join 执行。
7. **与 EMF 对比**:EMF 的流式是可观测性附加品;Claude Code 的流式是控制路径本身。

**问题 1**(状态机):
一条 assistant 消息里有 2 个 content 块:先是一个 text 块,然后是一个 tool_use 块。
SSE 帧的顺序是:
`block_start(text)` → `delta×3` → `block_stop` → `block_start(tool_use)` → `delta×5` → `block_stop` → `message_delta` → `message_stop`
(a) 工具在哪一帧之后才能开始执行?
(b) 用户在哪一帧之后才能看到 text 内容?
(c) 如果用户在 `delta×5` 的第 3 帧时按下 Esc,tool_use 会被执行吗?此时 messages 里会留下什么?

**问题 2**(截断处理):
模型在生成一个复杂 Edit 工具调用时被 max_tokens 截断,JSON 参数的 `new_string` 字段只输出了一半。
你作为 Agent Loop 设计者,决定注入一条 `is_error: true` 的 tool_result 让模型重试。
但有一个风险:如果模型重试时生成了一个略有不同的 `new_string`,它可能写入一个和原意不同的文件内容。
(a) 你能在 Agent Loop 层面消除这个风险吗?还是这个风险只能靠模型推理来处理?
(b) 如果这个 Edit 操作的重要性很高(比如修改的是 schema 文件),你会给 system prompt 加什么样的指令来降低风险?

**问题 3**(并发时序):
一条 assistant 消息里有 3 个 tool_use 块(A、B、C)。
SSE 帧的顺序是:
`block_start(tool_use A)` → `A 的 partial_json×10` → `block_stop(A)` →
`block_start(tool_use B)` → `B 的 partial_json×8` → `block_stop(B)` →
`block_start(tool_use C)` → `C 的 partial_json×6` → `block_stop(C)` →
`message_delta` → `message_stop`
(a) A、B、C 在流里是串行到来的。它们的执行是串行还是并发?
(b) 如果你在 `block_stop(A)` 之后立刻开始执行工具 A(不等 B、C),会有什么问题?
(c) 有没有办法让 A 提前执行同时保证语义正确?给出你的方案和前提条件。
