# DDD 学习路径与知识图谱

## 学习定位

DDD 是 Domain-Driven Design 的缩写，中文常译为“领域驱动设计”。

很多人一听 DDD，会立刻想到几个词：实体、值对象、聚合、仓储、领域事件。  
这些词当然重要，但如果你一开始就把 DDD 理解成“面向对象术语大全”或者“某种固定代码模板”，后面很容易学偏。

DDD 更准确的定位是：

**当业务复杂度高到不能只靠表结构和 CRUD 思维来驾驭时，用一套面向业务语义的建模方法，让代码结构、团队语言和业务规则尽量对齐。**

学完这条路线后，你应该逐步具备这些能力：

- 识别一个系统的复杂度到底主要来自技术，还是来自业务规则本身。
- 区分“简单数据增删改查”和“真正需要领域建模”的场景。
- 理解领域、子域、核心域、限界上下文、通用语言这些战略设计概念。
- 能在代码层面区分实体、值对象、聚合、仓储、领域服务、应用服务各自的职责。
- 能把 DDD 和微服务、六边形架构、CQRS、事件驱动这些常见架构话题正确关联起来，而不是混为一谈。
- 能用 DDD 视角分析订单、支付、库存、结算、营销规则等典型后端业务场景。

## 知识图谱（从基础到进阶）

1. 先理解 DDD 在解决什么问题
   - 为什么很多系统“功能越来越多，代码越来越难改”
   - 业务复杂度和技术复杂度有什么不同
   - DDD 与 CRUD、脚本式开发、贫血模型的区别
2. 战略设计：先画业务地图，再谈代码实现
   - 领域、子域、核心域、支撑域、通用域
   - 通用语言
   - 限界上下文
   - 上下文映射
3. 战术设计：把业务规则装进模型
   - 实体与值对象
   - 聚合与聚合根
   - 仓储、工厂、领域服务、应用服务
   - 领域事件
4. 工程落地：模型如何进入真实项目
   - 分层架构与 DDD 的关系
   - 六边形架构、端口适配器
   - DDD 与微服务拆分
   - DDD 与 CQRS、事件驱动的边界
5. 进阶权衡
   - 聚合边界与一致性边界
   - 大事务与最终一致性
   - 性能、复杂度、可维护性的平衡
   - 什么场景不该硬上 DDD
6. 实战演练
   - 订单、支付、库存案例建模
   - 识别模型误区
   - 从“表设计思维”过渡到“业务语义思维”

## 建议学习顺序背后的原因

这条路线不是先背术语，而是先建立“为什么需要 DDD”的问题意识。  
原因很简单：如果你没先看到业务复杂度带来的痛点，那么实体、值对象、聚合这些概念就会变成抽象名词，学完也不会用。

所以顺序是：

1. 先理解 DDD 解决的核心矛盾
2. 再理解业务边界和语言边界
3. 然后进入代码级建模
4. 最后讨论架构整合、工程权衡和实战

## 学习方法

- 每一课都先回答“它在解决什么问题”，再进入定义和术语。
- 学习过程中持续把抽象概念映射到真实业务，例如订单创建、履约、支付、退款、库存锁定、优惠叠加。
- 你有后端经验，所以我会优先用服务拆分、数据一致性、规则演化、多人协作这些工程问题来解释 DDD，而不是只讲抽象理论。
- 每一课结束先完成检查站，我会根据你的回答决定是推进、补充还是回到更基础的位置重讲。

## 推进规则

- 完全掌握：进入下一课，并逐步提高案例难度。
- 基本掌握：简短纠正后进入下一课。
- 部分理解：生成补充课，例如 `04b_聚合补充.md`。
- 尚未理解：回到更基础的问题重建概念，不跳过地基。
