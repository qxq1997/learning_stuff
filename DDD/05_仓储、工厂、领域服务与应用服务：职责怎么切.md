# DDD - 第 5 课：仓储、工厂、领域服务与应用服务：职责怎么切

## 学习目标（本节结束后你能做到什么）

- 理解为什么有了实体、值对象、聚合之后，还需要仓储、工厂、领域服务、应用服务这四个角色。
- 能说清楚仓储（Repository）和你平时写的 DAO / Mapper 到底差在哪。
- 知道工厂（Factory）在解决什么问题，以及什么时候根本不需要它。
- 能稳定区分领域服务（Domain Service）和应用服务（Application Service）——这是面试最爱追问、也最容易写成“Service 大杂烩”的地方。
- 能判断一段业务逻辑应该放进聚合、放进领域服务，还是放进应用服务。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先接上上一课：聚合解决了规则的“落点”，但没解决规则的“协作”

上一课我们说清楚了一件事：

业务规则要有稳定落点，核心不变量收进聚合，由聚合根统一守住。

但聚合根只能守住**一个聚合内部**的规则。真实业务里马上会冒出三类聚合根自己搞不定的问题：

1. **聚合怎么从数据库里捞出来、又怎么存回去？**
   聚合根不应该自己懂 SQL，否则领域模型又被数据库细节污染了。
2. **聚合一开始怎么被正确地创建出来？**
   有些聚合创建过程很复杂，光靠 `new` 出来很容易造出一个“非法的聚合”。
3. **一条规则同时牵扯多个聚合，该谁来管？**
   比如“从账户 A 转账到账户 B”，这条规则不属于 A，也不只属于 B。

这三类问题,正好对应三个角色:**仓储、工厂、领域服务**。
而把这些角色串成一个完整用例(一次请求)的,是**应用服务**。

所以这一课你可以理解成:**给聚合配齐它的"配套班子"**。

### 2. 仓储（Repository）：让领域层假装数据库不存在

#### 2.1 仓储到底在解决什么问题

领域模型最怕被两样东西污染:UI 和数据库。

如果聚合根里到处是 `SELECT`、`INSERT`、`connection.commit()`,那它就不再是业务模型,而是一个裹着业务壳的 DAO。

仓储的目的就是:

**让领域层用“取出聚合 / 放回聚合”的语义来操作持久化,而不关心背后是 MySQL、Redis 还是一个远程服务。**

你可以把仓储想象成一个“聚合的集合”。对领域层来说,它的心智模型是:

- `orderRepository.findById(orderId)` —— 给我那个订单聚合
- `orderRepository.save(order)` —— 把这个订单聚合存回去

至于背后是一张表、三张表、还是表 + 缓存,领域层一概不知,也不该知道。

#### 2.2 仓储和 DAO / Mapper 的真正区别

这是面试高频点。很多人觉得“仓储不就是换个名字的 DAO 吗”,这就误解了。

| 维度 | DAO / Mapper | Repository |
| --- | --- | --- |
| 操作对象 | 表 / 行 / 字段 | 聚合(整体) |
| 粒度 | 单表 CRUD,常有 `updateStatus`、`updateAmount` 这种字段级方法 | 以聚合为单位存取,通常只有 `save` / `findById` / `remove` |
| 归属 | 数据访问层,面向存储 | 领域层定义接口,面向业务 |
| 心智 | “我要更新这几列” | “我把这个聚合的当前状态整体保存下来” |

关键差别是**粒度**和**归属**:

- DAO 的世界里有 `updateOrderStatus(orderId, status)` 这种方法,它鼓励你绕过聚合直接改字段——这恰恰是上一课说的“破坏聚合保护罩”。
- 仓储只接收整个聚合 `save(order)`,你想改状态,得先 `order.pay()` 让聚合根自己改,再整体存回。

所以仓储不只是技术封装,它在**逼你走聚合根这道门**。

#### 2.3 一个常被忽略的点:仓储只为聚合根服务

不是每个实体都配一个仓储。

**仓储只针对聚合根。** 聚合内部的子实体(比如订单明细),不单独建仓储,它们随聚合根一起被取出、一起被存回。

如果你发现自己在写 `orderItemRepository.save(item)`,基本就说明你把聚合拆散了,明细脱离订单被单独操作——这是伪 DDD 的典型信号。

#### 2.4 仓储接口放在哪一层(依赖倒置)

这点很重要,也是后面六边形架构的伏笔:

**仓储的接口属于领域层,实现属于基础设施层。**

```
领域层:        interface OrderRepository { Order findById(id); void save(Order o); }
基础设施层:    class MysqlOrderRepository implements OrderRepository { ... 真正的 SQL ... }
```

为什么要这么反过来?因为我们希望**领域层不依赖数据库,而是数据库实现去依赖领域层定义的契约**。这就是依赖倒置:高层(业务)不依赖低层(存储)的细节,而是双方都依赖一个由业务定义的抽象。

记住这句话,第 7 课讲六边形架构时它就是主角。

### 3. 工厂（Factory）：保证“一出生就是合法的聚合”

#### 3.1 为什么 new 不够用

简单聚合直接 `new Order(...)` 没问题。但有些聚合的创建本身就是一段业务逻辑:

- 创建过程要校验一堆前置规则
- 要根据输入装配出一组内部对象(明细、快照、初始状态)
- 要保证创建出来的瞬间,聚合的不变量就已经成立

如果把这些逻辑散在调用方的 service 里,就会出现一个危险局面:**有人能造出一个“非法的聚合”**——比如一个总金额对不上明细的订单。

工厂的职责就是:

**把“如何正确地创建一个聚合”这段知识收敛到一个地方,保证产出的聚合从第一刻起就是合法的。**

#### 3.2 工厂不是一定要有

这点要特别强调,因为很多人学了 DDD 就给每个聚合配个 `XxxFactory`,纯属过度设计。

判断要不要工厂,就一个问题:

**创建过程是不是复杂到、或重要到值得被单独保护?**

- 创建逻辑简单 → 构造函数或聚合根的静态方法 `Order.create(...)` 就够了,不需要独立工厂类。
- 创建逻辑复杂、有多种装配路径、或者涉及多个对象协同 → 才考虑独立工厂。

所以工厂在四个角色里是**最可选**的一个。别为了凑齐 DDD 术语硬造工厂。

### 4. 领域服务（Domain Service）：规则不属于任何单个聚合时,它的家

#### 4.1 什么时候需要领域服务

绝大多数业务规则,应该放在实体或聚合里(这叫“充血”)。但确实有一类规则,放进任何一个聚合都别扭:

**这条规则天然涉及多个聚合,或者它表达的是一个领域概念,但不属于某一个具体对象。**

经典例子:**转账**。

“从账户 A 转出 100,转入账户 B”这条规则:

- 放进账户 A?它凭什么知道、还能修改账户 B?
- 放进账户 B?同样别扭。

它涉及两个对等的聚合,不属于其中任何一个。这种逻辑就交给领域服务:

```java
class TransferService {           // 领域服务
    void transfer(Account from, Account to, Money amount) {
        from.withdraw(amount);    // 各自的规则还是各自聚合守
        to.deposit(amount);
    }
}
```

注意:领域服务**不抢聚合的活**。`withdraw` / `deposit` 里的“余额不能为负”这类单聚合规则,仍然由账户聚合自己守。领域服务只负责**协调多个聚合之间那条单聚合管不了的规则**。

#### 4.2 领域服务的三个识别信号

一段逻辑适合放领域服务,通常满足下面之一:

1. 它**跨多个聚合**,不属于任何一个(转账、调拨、撮合)。
2. 它是一个**重要的领域计算/策略**,放进实体会让实体职责不纯(比如复杂的计费规则、风控评分)。
3. 它表达的是一个**动词式的领域概念**,本身没有身份和状态(`PricingService`、`SettlementCalculator`)。

#### 4.3 领域服务最大的陷阱:别变成贫血模型的垃圾场

这是真实项目里最常见的退化:

一开始大家图省事,把所有逻辑都写进 `XxxDomainService`,实体退化成只有 getter/setter 的数据壳。结果就是**换了个名字的 transaction script**——表面上有领域服务,实质是贫血模型。

判断口诀:

**能放进聚合的规则,优先放聚合;只有真的“无家可归”的规则,才放领域服务。**

领域服务应该是“少而精”的,如果你的领域服务比聚合还胖、还多,方向多半错了。

### 5. 应用服务（Application Service）：把一次用例编排起来,但自己不写业务规则

#### 5.1 应用服务在干什么

前面几个角色都在领域层内部。应用服务则站在领域层之上,它对应的是**一个用例 / 一次请求**:

“用户支付一个订单”“生成本月结算单”“发起一笔转账”。

应用服务的典型动作就四步,非常机械:

1. 通过仓储把需要的聚合**取出来**
2. 调用聚合根或领域服务**执行业务动作**
3. 通过仓储把结果**存回去**
4. 管理**事务**、发布**领域事件**、返回结果

```java
class OrderAppService {           // 应用服务
    @Transactional
    void payOrder(PayCommand cmd) {
        Order order = orderRepo.findById(cmd.orderId());   // 1 取
        order.pay(cmd.payInfo());                           // 2 调业务动作(规则在聚合里)
        orderRepo.save(order);                              // 3 存
        eventPublisher.publish(new OrderPaidEvent(...));    // 4 发事件
    }
}
```

#### 5.2 应用服务的铁律:它自己不做业务决策

这是整节课最该记住的一句话:

**应用服务负责“编排流程”,不负责“业务规则”。**

也就是说,应用服务里**不应该出现 if/else 形式的核心业务判断**。

反面例子(常见的坏味道):

```java
// 坏:业务规则泄漏到了应用服务
void payOrder(PayCommand cmd) {
    Order order = orderRepo.findById(cmd.orderId());
    if (order.getStatus() == CLOSED) throw ...;     // ← 这条规则应该在 order.pay() 里
    if (order.getAmount() <= 0) throw ...;          // ← 同上
    order.setStatus(PAID);                          // ← 直接改字段,绕过聚合
    orderRepo.save(order);
}
```

这段代码把“已关闭不能支付”“金额必须为正”这些**业务不变量泄漏到了应用层**,而且用 `setStatus` 绕过了聚合根。一旦还有别的入口(job、mq、另一个接口)也要支付订单,这些规则就得再抄一遍——回到了第 4 课开头那个“同一规则散在 5 个地方”的噩梦。

正确做法是把规则压回 `order.pay()` 里,应用服务只剩“取—调—存—发事件”的骨架。

#### 5.3 应用服务还负责什么(容易漏的)

除了编排,应用服务通常还兜着这些“非业务但必要”的职责:

- **事务边界**:一个用例就是一个事务,`@Transactional` 通常加在这一层。
- **领域事件的发布**:聚合产生事件,应用服务在事务提交前后负责发出去(第 6 课细讲)。
- **DTO ↔ 领域对象的转换**:对外接收 Command/DTO,对内操作聚合,不让外部数据结构渗进领域层。
- **安全、权限、日志**等横切关注点(或者交给更外层)。

### 6. 四个角色放在一起:一张职责分配图

把这一课的角色按“谁该干什么”排一下,你就有了一张可以套用的检查表:

| 角色 | 所在层 | 核心职责 | 不该做的事 |
| --- | --- | --- | --- |
| 聚合根 / 实体 | 领域层 | 守住单聚合内部的业务规则和不变量 | 不该懂 SQL、不该懂 HTTP |
| 值对象 | 领域层 | 表达不可歧义的值,承接细小规则 | 不该有身份、不该可变(通常) |
| 工厂 | 领域层 | 保证聚合“一出生就合法” | 别为简单创建过度设计 |
| 领域服务 | 领域层 | 承接跨聚合、无家可归的业务规则 | 别变成贫血模型的逻辑垃圾场 |
| 仓储 | 接口在领域层,实现在基础设施层 | 以聚合为单位存取 | 不做字段级 update、不为子实体单独建仓储 |
| 应用服务 | 应用层 | 编排用例:取—调—存—发事件,管事务 | 绝不写核心业务规则,不绕过聚合改字段 |

### 7. 实战判断:一段逻辑到底该放哪?

给你一个可以现场用的决策顺序。拿到一段逻辑,按这个顺序问:

1. **它只关于一个聚合内部的不变量吗?** → 放进那个聚合(实体/聚合根方法)。
2. **它跨了多个聚合,或不属于任何具体对象吗?** → 放进领域服务。
3. **它是创建一个合法聚合的复杂过程吗?** → 放进工厂(或聚合根静态方法)。
4. **它是“取数据、调用、存回去、管事务、发事件”这种流程编排吗?** → 放进应用服务。
5. **它是 SQL / 缓存 / 外部 API 这种技术细节吗?** → 放进基础设施层(仓储实现、防腐层等)。

用结算中心举个完整例子,“生成本月结算单”这个用例怎么拆:

- **应用服务** `SettlementAppService.generateMonthlyBill(cmd)`:开启事务,取出本月需要结算的数据,调用下面的角色,最后存回并发事件。
- **领域服务** `SettlementCalculator`:跨多个明细/账户做复杂的金额汇总和分摊计算(不属于单个结算单聚合的纯计算)。
- **工厂 / 聚合根静态方法** `SettlementBill.create(...)`:用算好的结果装配出一张合法的结算单(总额等于明细汇总、初始状态为草稿)。
- **聚合根** `bill.confirm()`:守住“已确认不能改回草稿”等不变量。
- **仓储** `settlementBillRepository.save(bill)`:把整张结算单聚合存回去。

你会发现,每个角色都很薄、很专一,没有哪个变成大杂烩——这就是职责切对了的样子。

## 小结（3-5 条关键点）

- 仓储以**聚合为单位**存取,只服务聚合根;它和字段级的 DAO 本质不同,接口属于领域层、实现属于基础设施层(依赖倒置)。
- 工厂保证聚合“一出生就合法”,但它是最可选的角色,简单创建别硬上工厂。
- 领域服务承接**跨聚合、无家可归**的业务规则;最大风险是退化成贫血模型的逻辑垃圾场——能进聚合的规则优先进聚合。
- 应用服务只做**用例编排**(取—调—存—发事件)和管事务,**绝不写核心业务规则、绝不绕过聚合改字段**。
- 判断逻辑该放哪,按“单聚合不变量 → 跨聚合规则 → 复杂创建 → 流程编排 → 技术细节”的顺序问一遍即可。
