# DDD - 第 6 课：领域事件：模型如何和流程协作

## 学习目标（本节结束后你能做到什么）

- 理解领域事件在解决什么问题：上一课聚合切小了，跨聚合、跨上下文的流程靠什么串起来。
- 能说清楚领域事件和你平时用的 MQ 消息、普通回调有什么区别。
- 知道领域事件为什么要用“过去式 + 业务语义”命名，以及它和命令（Command）的区别。
- 理解发布时机这个大坑：事务内发还是事务后发，会带来什么一致性问题。
- 知道生产里怎么保证“聚合落库”和“事件发出”不丢不乱（Outbox / 事务消息）。
- 能区分进程内事件和跨上下文事件，知道什么时候该用哪种。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 接上一课：聚合切小之后，流程怎么办？

第 4 课我们反复强调一句话：**聚合要尽量小，只包住必须强一致的那部分**。

这句话很对，但它马上带来一个新问题：

如果聚合都切小了，那些**横跨多个聚合、多个上下文的业务流程**，靠什么把它们串起来？

举个结算中心的例子，一个真实流程往往是：

> 订单支付成功 → 触发结算计算 → 生成结算单 → 触发出款 → 通知外部系统

这条链路跨了**订单、结算、出款、通知**好几个聚合甚至好几个上下文。第 5 课我们说过，这种东西**不该塞进同一个聚合**（不是一个一致性边界），也不该让应用服务里写一长串硬编码的 if/else 调用——那样耦合会爆炸。

DDD 给的答案就是：**领域事件**。

**领域事件 = “领域里发生了一件值得被其他部分关心的事实”。**

发生了这件事的聚合，只负责“喊一声：我这边发生了 XXX”，至于谁关心、要做什么，由订阅方自己决定。发布方和订阅方**不直接认识彼此**。

### 2. 领域事件的本质：从“我去叫别人做事”变成“我宣布发生了一件事”

理解领域事件，关键是一个思维转变。

**没有领域事件时（命令式、紧耦合）：**

```
订单支付成功后，订单服务亲自去调：
  结算服务.算一笔()
  通知服务.发个消息()
  积分服务.加积分()
```

订单服务变成了一个“指挥中心”，它必须**知道所有下游**、知道它们的接口、知道调用顺序。每加一个下游，订单服务就得改一次——这就是耦合。

**有领域事件时（事件式、解耦）：**

```
订单支付成功后，订单只宣布一个事实：
  发布 OrderPaidEvent（订单已支付）

谁关心这个事实，自己去订阅：
  结算上下文：收到 OrderPaidEvent → 触发结算
  积分上下文：收到 OrderPaidEvent → 加积分
  通知上下文：收到 OrderPaidEvent → 发消息
```

订单**完全不知道**下游有谁。加一个新下游（比如风控也想监听支付），订单一行代码都不用改。

这就是领域事件的核心价值：**把“流程编排”从命令式的强耦合，变成基于事实的发布-订阅解耦。**

### 3. 领域事件怎么命名：过去式 + 业务语义

领域事件有一个几乎是铁律的命名约定：

**用过去式，描述一个已经发生、不可撤销的事实。**

- ✅ `OrderPaid`（订单已支付）
- ✅ `SettlementBillConfirmed`（结算单已确认）
- ✅ `AccountFrozen`（账户已冻结）
- ❌ `PayOrder`（这是命令，不是事件）
- ❌ `OrderUpdate` / `OrderChanged`（太泛，没有业务语义，等于没说）

为什么强调过去式？因为事件代表的是**既成事实**。事实是不能被拒绝的——你不能“拒绝一件已经发生的事”。这和命令（Command）有本质区别：

| | 命令 Command | 领域事件 Domain Event |
| --- | --- | --- |
| 语气 | 祈使句：“给我做 X” | 陈述句：“X 已经发生了” |
| 时态 | 现在/将来 | 过去式 |
| 能否被拒绝 | 能（校验不过就拒绝） | 不能（事实已发生） |
| 接收方 | 明确的一个 | 任意多个，发布方不关心 |
| 例子 | `PayOrder` | `OrderPaid` |

命名上别偷懒用 `OrderChanged` 这种泛词。事件名应该让人一看就知道**业务上发生了什么**，这也是通用语言（第 3 课）落地的一部分。

### 4. 领域事件和 MQ 消息是一回事吗？

这是面试和实战都容易混的点。答案是：**不是一回事，但常常借 MQ 来实现。**

要分清两层：

- **领域事件是一个领域概念**：它属于业务模型，表达“领域里发生了什么”，是设计层面的东西。
- **MQ 消息是一个技术手段**：Kafka/RocketMQ 里的一条消息，是传输层面的东西。

关系是：领域事件**可以**通过 MQ 投递出去（尤其是跨上下文、跨服务时），但领域事件不等于 MQ 消息。

几个关键区别：

1. **领域事件先有业务含义，MQ 消息可以只是技术信号**。`OrderPaid` 是领域事件；“缓存刷新通知”这种纯技术消息就不算领域事件。
2. **领域事件可以是进程内的**（见第 7 节），根本不碰 MQ。
3. **一个领域事件落到 MQ 时，往往要做一层转换**：领域事件对象（带聚合内部细节）→ 对外的消息 DTO（只暴露必要字段）。不要把领域模型对象直接序列化扔进 MQ，否则上下文之间又耦合了。

一句话：**领域事件是“设计意图”，MQ 是“落地管道”之一。**

### 5. 最关键的坑：事件什么时候发？（一致性问题）

这是整节课最值钱、也最容易在生产出事的地方。

聚合状态变更和事件发布，是两个动作：

1. 订单聚合状态改成“已支付”，落库（写 MySQL）
2. 发布 `OrderPaid` 事件（比如写 Kafka）

这两个动作**不在同一个事务里**（MySQL 事务管不了 Kafka）。于是就有两种失败姿势：

**姿势 A：先发事件，再落库** —— 事件发出去了，下游开始算结算了，结果订单落库失败回滚了。**下游基于一个不存在的事实做了事**（脏数据、对不上账）。

**姿势 B：先落库，再发事件** —— 订单落库成功了，但发事件那一步进程挂了 / Kafka 超时。**事实发生了，但没人知道**（结算永远不会被触发，订单卡死）。

两种都会出事。这就是著名的“**双写问题**”（dual write）：你要同时写两个存储（DB + MQ），却没有一个横跨两者的事务。

### 6. 生产怎么解：Outbox（事务发件箱）模式

业界最常用的稳妥解法是 **Outbox（事务发件箱）**，核心思路：

**把“发事件”这个动作，先变成一次普通的数据库写入，和业务数据放进同一个本地事务。**

具体做法：

1. 在同一个 DB 事务里，**既改订单表，也往一张 `outbox` 事件表插一条事件记录**。
   - 这俩都是 MySQL 写，同一个本地事务，要么都成功要么都失败。**双写问题没了**。
2. 事务提交后，由一个**独立的投递进程**（轮询 outbox 表，或订阅 binlog）把事件表里的记录捞出来，发到 MQ。
3. 发成功后把这条 outbox 记录标记为已发送。

```
   ┌─────────── 一个本地事务 ───────────┐
   │  UPDATE orders SET status='PAID'   │
   │  INSERT INTO outbox (OrderPaid...) │
   └────────────────────────────────────┘
                    ↓ 事务提交后
        投递进程轮询 outbox → 发到 MQ → 标记已发
```

这样换来的代价是：**事件投递从“实时”变成“最终一致”**——下游可能晚几百毫秒甚至几秒才收到。但它保证了**不丢、不出现幻觉事实**。

几个要点：

- 投递进程要保证“**至少一次**”投递（at-least-once），所以**下游必须做幂等**——可能收到重复事件（比如标记已发那步失败、下次重投）。幂等通常靠事件里的唯一 `eventId` 去重。
- 很多消息中间件提供的“事务消息”（如 RocketMQ 事务消息）本质上是 Outbox 的另一种实现形态，目标一样：让本地事务和消息发送达成一致。

### 7. 进程内事件 vs 跨上下文事件：别一上来就上 MQ

不是所有领域事件都要走 MQ。要先分清两种场景：

**进程内事件（同一个服务、同一个进程）：**

- 同一个限界上下文内，几个聚合之间的协作。
- 可以用框架的进程内事件总线（比如 Spring 的 `ApplicationEventPublisher`）。
- 甚至可以和发布方在同一个事务里同步处理。
- 优点：简单、强一致；缺点：耦合在同一进程，扩不出去。

**跨上下文 / 跨服务事件：**

- 跨限界上下文（订单上下文 → 结算上下文），尤其是跨微服务。
- 这时才需要 MQ + Outbox，接受最终一致。
- 优点：彻底解耦、能独立伸缩；缺点：复杂度高、要处理重复和顺序。

一个常见的误区是：**还在单体里、明明是进程内协作，却为了“DDD 范儿”硬上 Kafka**，把简单问题搞复杂。判断标准还是回到第 4 课的一致性边界：**需要强一致就别拆开异步，能接受最终一致才上 MQ。**

### 8. 别把领域事件写成“贫血通知”

最后提醒一个退化方向，和第 5 课“领域服务变垃圾场”是一类病。

坏味道是：事件里只塞一个 id，下游收到后还得反查一堆数据：

```java
// 偏弱：下游几乎什么都得自己反查
class OrderPaidEvent { String orderId; }
```

到底事件里该带多少数据，是个权衡：

- **带太少**（只有 id）：下游被迫回查发布方，又产生了运行时耦合和额外调用。
- **带太多**（把整个订单聚合塞进去）：事件和发布方内部模型强绑定，发布方一改模型，所有下游都受影响。

经验做法：**事件里带上“描述这个事实所需的、相对稳定的关键字段”**，比如 `OrderPaid` 带上 `orderId`、`paidAmount`、`paidAt`、`buyerId`，让大多数下游不用回查就能干活，同时不暴露发布方的内部结构。

事件是上下文之间的**契约**，要像设计 API 一样认真设计它的字段，而不是随手塞个 id。

### 9. 串一遍：结算流程用领域事件长什么样

回到开头那条链路，用领域事件重写一遍，你能感受到解耦：

```
订单上下文：
  order.pay() 成功
  → 同一事务内：更新订单 + 写 outbox(OrderPaid)
  → 投递进程把 OrderPaid 发到 MQ

结算上下文（订阅 OrderPaid）：
  收到 OrderPaid（幂等校验）
  → SettlementAppService 触发结算计算、生成结算单
  → bill.confirm() 成功
  → 同一事务内：更新结算单 + 写 outbox(SettlementBillConfirmed)

出款上下文（订阅 SettlementBillConfirmed）：
  收到事件 → 触发出款……
```

注意每一环：

- 上游只发**事实**，完全不知道下游是谁。
- 每个上下文内部该强一致的（改自己的聚合 + 写自己的 outbox）放同一本地事务。
- 上下文之间靠事件做**最终一致**。
- 加一个新下游（风控、对账、数据分析）只需新增一个订阅者，**已有代码一行不改**。

这就是“模型如何和流程协作”的完整图景：**聚合管强一致的小边界，领域事件管跨边界的流程协作。**

## 小结（3-5 条关键点）

- 聚合切小后，跨聚合、跨上下文的流程靠**领域事件**串联：发布方只宣布“发生了什么事实”，订阅方自己决定怎么响应，二者互不认识，实现解耦。
- 领域事件用**过去式 + 业务语义**命名（`OrderPaid` 而非 `OrderChanged`）；它和命令的区别是“既成事实 vs 可被拒绝的请求”。
- 领域事件是**设计概念**，MQ 是**落地管道之一**；跨上下文才需要 MQ，进程内协作未必要。
- 最大的坑是**发布时机的双写问题**；生产用 **Outbox / 事务消息**把“写业务数据”和“记事件”放进同一本地事务，换来最终一致 + 不丢，代价是下游必须**幂等**。
- 事件字段要像 API 契约一样设计：带够关键字段避免下游回查，又不暴露发布方内部模型，别退化成只有一个 id 的“贫血通知”。
