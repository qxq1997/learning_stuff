# DDD - 第 7 课：DDD 与分层架构、六边形架构、微服务的关系

## 学习目标（本节结束后你能做到什么）

- 理解 DDD 的四层架构和你熟悉的传统三层架构差在哪，尤其是依赖方向。
- 把第 5 课埋的伏笔（仓储接口在领域层、实现在基础设施层）展开成完整的依赖倒置图。
- 理解六边形架构（端口-适配器）到底在说什么，它和 DDD 是什么关系。
- 知道防腐层（ACL）是干什么的，它怎么把第 3 课的上下文映射落到代码里。
- 能正确回答“限界上下文和微服务是不是一回事”——这是面试高频陷阱。
- 知道什么时候该把上下文拆成独立微服务，什么时候不该。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先看你最熟悉的传统三层架构，它有什么问题

你做后端，肯定写过这套：

```
Controller（表现层）
    ↓ 依赖
Service（业务逻辑层）
    ↓ 依赖
DAO / Mapper（数据访问层）
    ↓ 依赖
Database
```

这套结构本身没错，但它有一个**致命的依赖方向问题**：

**业务逻辑层（Service）依赖了数据访问层（DAO），而 DAO 是面向数据库的。**

结果就是：**你的核心业务，最终依赖了数据库**。

这带来几个后果，都是你大概率遇到过的：

- Service 里到处是 `mapper.selectByXxx`、`mapper.updateStatus`，业务规则和 SQL 纠缠在一起。
- 想换存储（MySQL 换成别的、或加一层缓存），核心业务代码得跟着改。
- 想给业务逻辑写单元测试，绕不开数据库，必须连库或者 mock 一大堆 DAO。
- 时间一长，“业务规则”其实是长在 DAO 周围的，而不是独立存在的——这就是为什么很多“三层架构”项目本质是**面向数据库编程**。

DDD 的架构演进，核心就是要把这个依赖方向**掰过来**。

### 2. DDD 四层架构：把“业务”放到正中央

DDD 经典的分层是四层（注意从上到下的角色变化）：

```
用户接口层 / 表现层 (Interface / UI)
   —— Controller、RPC 接口、消息监听器；负责协议转换、参数校验
        ↓
应用层 (Application)
   —— 应用服务；编排用例：取—调—存—发事件，管事务（第 5 课）
        ↓
领域层 (Domain)              ← 整个系统的核心，业务规则的家
   —— 聚合、实体、值对象、领域服务、领域事件、仓储【接口】
        ↑（依赖倒置！）
基础设施层 (Infrastructure)
   —— 仓储【实现】、MQ、缓存、外部 API 调用、防腐层
```

和三层比，关键变化有两个：

1. **多了一个“领域层”，而且它是中心**。业务规则不再寄生在 Service+DAO 里，而是有了自己独立的一层。
2. **基础设施层的依赖箭头是反过来的**——它依赖领域层，而不是领域层依赖它。

第 2 点就是第 5 课埋的伏笔，现在正式展开。

### 3. 依赖倒置：为什么基础设施要“反过来”依赖领域

回顾第 5 课那段代码：

```
领域层:        interface OrderRepository { Order findById(id); void save(Order o); }
基础设施层:    class MysqlOrderRepository implements OrderRepository { ...SQL... }
```

注意发生了什么：

- **接口** `OrderRepository` 定义在**领域层**——它是业务说“我需要能存取订单聚合”的契约。
- **实现** `MysqlOrderRepository` 在**基础设施层**——它去实现这个契约。

于是依赖方向变成：

```
领域层  →  定义 OrderRepository 接口（业务说了算）
              ↑ implements
基础设施层  →  MysqlOrderRepository 来满足这个契约
```

**领域层不知道 MySQL 的存在**，它只知道有个 `OrderRepository` 接口。具体是 MySQL、是 MongoDB、是内存实现，对它完全透明。

这就是**依赖倒置原则（DIP）**：高层模块（业务）不依赖低层模块（存储）的细节，而是**低层去依赖高层定义的抽象**。

带来的直接好处：

- 核心业务**不再依赖数据库**，可以独立编译、独立测试（测试时给个内存实现的 `OrderRepository` 就行，不用连库）。
- 换存储、加缓存，只改基础设施层，**领域层一行不动**。
- 业务规则真正“独立存在”了，而不是长在 SQL 周围。

记住这句话：**依赖倒置是 DDD 架构的发动机，六边形架构只是把它画得更清楚的一张图。**

### 4. 六边形架构（端口-适配器）：把依赖倒置画成一张图

六边形架构（Hexagonal Architecture），又叫**端口-适配器架构（Ports and Adapters）**，是对上面那个思想的可视化表达。

它的画法是：把**业务核心放在六边形正中间**，外界的一切（数据库、MQ、HTTP、第三方 API、UI）都在六边形外面，中间靠“端口”和“适配器”连接。

```
        [HTTP Controller]   [MQ 消费者]        ← 驱动侧适配器（外部调我）
                │                │
            ┌───┴────────────────┴───┐
            │      入站端口（接口）    │
            │   ┌──────────────────┐  │
            │   │                  │  │
            │   │    领域核心        │  │   ← 聚合、领域服务，纯业务，不依赖任何外部
            │   │  (业务规则)       │  │
            │   │                  │  │
            │   └──────────────────┘  │
            │      出站端口（接口）    │
            └───┬────────────────┬───┘
                │                │
        [MySQL 适配器]    [外部 API 适配器]      ← 被驱动侧适配器（我调外部）
```

核心概念：

- **端口（Port）= 一个接口**，定义“核心需要什么 / 核心提供什么”。
  - **入站端口**：外界怎么驱动核心（比如应用服务暴露的接口）。
  - **出站端口**：核心需要外界提供什么（比如 `OrderRepository`、`PaymentGateway` 接口）。
- **适配器（Adapter）= 接口的具体实现**，对接某种具体技术。
  - **驱动侧适配器**（左/上）：HTTP Controller、MQ 监听器——它们把外部请求“翻译”成对入站端口的调用。
  - **被驱动侧适配器**（右/下）：`MysqlOrderRepository`、`AlipayGateway`——它们实现出站端口，去对接真实的数据库、第三方。

**所有箭头都指向核心**。核心不认识任何适配器，它只认识自己定义的端口（接口）。

你会发现：**六边形架构 = DDD 四层 + 依赖倒置的图形化**。出站端口就是仓储接口那一套，被驱动适配器就是基础设施层的实现。它俩说的是一回事，只是六边形强调“核心被一圈适配器包围、内外完全解耦”这个画面感。

> 补充：洋葱架构（Onion）、整洁架构（Clean Architecture）和六边形是“一家人”，核心思想都一样——**依赖只能由外向内，业务核心在最里面、不依赖任何外部细节**。面试被问到三者区别，记住“同一个思想的不同画法”就够了，别陷进术语。

### 5. 防腐层（ACL）：把上下文映射落到代码里

第 3 课讲限界上下文时提过“上下文之间要做映射”，但没说怎么落地。防腐层就是其中最重要的一种落地手段。

**问题场景**：你的结算上下文要调用一个外部系统（比如老的支付系统、或者一个第三方）。这个外部系统的模型很烂、字段命名很怪、概念和你完全不一样。如果你直接在领域层用它的模型，**它的烂模型就会渗透进你干净的领域**——你的通用语言被污染了。

**防腐层（Anti-Corruption Layer, ACL）** 的作用就是：

**在你的上下文和外部之间架一道“翻译 + 隔离”层，把外部模型翻译成你自己的领域模型，挡住外部的脏东西渗进来。**

```
你的结算领域（干净的模型）
        ↑ 用你自己的领域对象
   ┌─────────────┐
   │  防腐层 ACL  │  ← 翻译：外部 DTO ↔ 我的领域对象，挡住外部概念
   └─────────────┘
        ↓ 用外部的烂模型
外部支付系统（你管不了的模型）
```

在六边形架构里，防腐层通常就活在**被驱动侧适配器**里：出站端口定义“我需要什么”（用你自己的语言），适配器内部去调外部、并做翻译。

实际好处：

- 外部系统改了字段、换了接口，**只改防腐层**，你的领域不受影响。
- 你的领域模型保持纯净，不被外部的历史包袱污染。
- 哪天要换一个外部供应商，换个适配器实现就行。

这是“上下文映射”里最常考、也最实用的一种模式，记住它的关键词：**翻译 + 隔离**。

### 6. 限界上下文和微服务是一回事吗？（面试高频陷阱）

先给结论：**不是一回事。**

- **限界上下文是逻辑边界**（第 3 课）：它是“一套通用语言和领域模型成立的范围”，是**设计概念**。
- **微服务是物理/部署边界**：它是一个能独立部署、独立伸缩的进程，是**架构落地概念**。

两者的关系是：**限界上下文是划分微服务的最佳参考依据，但不是一一对应。**

常见的几种组合：

1. **一个上下文 = 一个微服务**：最理想、最常见的情况，边界清晰。
2. **一个上下文，内部仍是单体的一个模块**：业务量没到、团队没到，没必要拆进程。**可以先在单体里用模块/包把上下文隔离好，将来再拆**——这恰恰是 DDD 的价值：先把逻辑边界划对，物理拆分是水到渠成的事。
3. **多个上下文 = 一个微服务**：几个小上下文关系紧密、又都很小，合在一个服务里部署也合理。

**反过来最要命的错误**：不按上下文边界、而是**按表或按技术分层拆微服务**（“用户表一个服务、订单表一个服务、所有 Controller 一个服务”）。这种拆法会导致一个业务用例横跨一堆服务、到处分布式事务，是“分布式单体”的典型病因。

所以正确顺序是：**先用 DDD 划清限界上下文（逻辑边界），再根据团队规模、业务量、伸缩需求决定怎么映射到微服务（物理边界）。** 别一上来就拆服务。

### 7. 拆不拆微服务：判断清单

承接上一节，给你一个可以现场用的判断清单。一个上下文**值得拆成独立微服务**，通常要满足几条：

- **伸缩需求不同**：这块流量/计算量和其他部分明显不一样，需要独立扩容（比如结算的批量计算 vs 下单的高并发）。
- **变更频率不同 / 团队不同**：由不同团队负责、独立发版，拆开能减少互相踩踏。
- **可用性要求不同**：核心域要高可用、和支撑域隔离开，避免一个挂全挂。
- **技术栈差异**：某块确实需要不同的语言/存储。

如果都不满足——团队就几个人、流量也不大——那**别拆**。在单体里用 DDD 分层 + 上下文模块隔离，把边界划清楚，是更省成本的选择。微服务的复杂度（分布式事务、服务治理、链路追踪、数据一致性）是有实打实代价的，第 6 课讲的双写/Outbox 就是这些代价之一。

记住：**DDD 帮你划逻辑边界，微服务是可选的物理落地。先有清晰的边界，再谈要不要拆。**

### 8. 串一遍：结算中心的一张完整架构图

把这一课的东西套到结算中心上，你能看到所有角色各就各位：

```
                    [HTTP 接口]      [监听 OrderPaid 的 MQ 消费者]   ← 用户接口层（驱动适配器）
                         │                      │
                    ┌────┴──────────────────────┴────┐
                    │   应用层：SettlementAppService   │  ← 编排用例、管事务、发事件
                    ├──────────────────────────────────┤
                    │   领域层（核心）                  │
                    │   - SettlementBill（聚合根）      │
                    │   - SettlementCalculator（领域服务）
                    │   - Money/BillingCycle（值对象）  │
                    │   - SettlementBillRepository【接口】← 出站端口
                    │   - PaymentGateway【接口】 ───────┼─→ 出站端口
                    └────┬──────────────────────┬──────┘
                         │ implements           │ implements
              [MysqlSettlementBillRepo]   [外部支付系统适配器 + 防腐层]  ← 基础设施层（被驱动适配器）
                         ↓                       ↓
                      MySQL                  外部支付系统（烂模型被 ACL 挡在外面）
```

读这张图的方式：

- **核心（领域层）在正中间，不认识 MySQL、不认识外部支付系统**，它只认识自己定义的接口（出站端口）。
- 所有技术细节都被推到外圈的适配器里。
- 防腐层守在和外部系统的交界处，挡住污染。
- 这一整块如果将来要独立部署，就是一个“结算微服务”；现在不拆，它就是单体里一个干净隔离的模块。

这就是 DDD、分层、六边形、微服务**拼在一起的完整图景**：DDD 给你模型和边界，分层/六边形给你依赖方向，微服务是你按需选择的物理落地。

## 小结（3-5 条关键点）

- 传统三层的病根是**业务依赖数据库**；DDD 四层把领域层放到中心，并用**依赖倒置**让基础设施反过来依赖领域（仓储接口在领域层、实现在基础设施层）。
- **六边形架构（端口-适配器）= 依赖倒置的图形化**：业务核心在中间、不认识任何外部，所有箭头指向核心；洋葱/整洁架构是同一思想的不同画法。
- **防腐层（ACL）= 翻译 + 隔离**，把外部的烂模型挡在领域之外，是上下文映射最实用的落地手段。
- **限界上下文（逻辑边界）≠ 微服务（物理边界）**：上下文是划分微服务的最佳依据，但可以一对一、也可以先在单体里做模块隔离；最忌按表/按技术分层拆服务。
- 拆不拆微服务，看**伸缩、团队、可用性、技术栈**是否真有差异；没差异就别拆，微服务的分布式复杂度是实打实的代价。
