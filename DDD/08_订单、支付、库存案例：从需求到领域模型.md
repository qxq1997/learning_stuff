# DDD - 第 8 课：订单、支付、库存案例：从需求到领域模型

## 学习目标（本节结束后你能做到什么）

- 走一遍完整的建模流程：从一段自然语言需求，到一套限界上下文 + 聚合 + 事件的领域模型。
- 学会用“事件风暴”的简化版，从业务流程里捞出领域事件、命令、聚合。
- 把前 7 课的概念（上下文、聚合、仓储、领域服务、领域事件、依赖倒置）全部用上一次。
- 啃下电商最经典的建模难题：**下单扣库存到底强一致还是最终一致、订单和库存能不能放一个聚合、超卖怎么防**。
- 形成一套“拿到需求该怎么下手”的可复用套路。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 拿到需求别急着建表，先建模

这一课是实战，所以我会**把思考过程摊开给你看**，而不是直接甩一张最终的类图。

先约定一个习惯，这是 DDD 实战最重要的起手式：

**拿到需求，第一反应不该是"建几张表"，而是"业务上到底发生了哪些事、有哪些边界、哪些规则必须一起成立"。**

表是模型的结果，不是模型的起点。很多人一上来就 `CREATE TABLE order(...)`，模型就被表结构绑死了，这正是第 7 课说的"面向数据库编程"。

我们这一课就反过来：**需求 → 流程 → 边界 → 聚合 → 事件 → 最后才落代码/表**。

### 2. 需求：一个简化版电商下单履约场景

先把需求摆出来（故意写成产品口吻，模拟真实场景）：

> 用户在 App 里挑选商品，加入购物车，提交订单。提交时要校验库存够不够、计算订单金额（含优惠）。下单成功后用户去支付，支付成功后订单进入待发货，仓库发货后订单进入已发货，用户确认收货后订单完成。任何时候用户在支付前可以取消订单，取消后要释放占用的库存。

这段话信息量不小。直接建模会乱，我们一步步拆。

### 3. 第一步：用"事件风暴"捞出领域事件

事件风暴（Event Storming）的简化玩法：**通读需求，把"已经发生的事实"全部用过去式列出来**（就是第 6 课的领域事件命名法）。

从上面那段话里捞，按时间顺序：

```
订单已提交        OrderPlaced
库存已占用        StockReserved
订单金额已计算     （内部动作，未必是对外事件）
支付已成功        PaymentSucceeded
订单已支付        OrderPaid
订单已发货        OrderShipped
订单已完成        OrderCompleted
订单已取消        OrderCancelled
库存已释放        StockReleased
```

捞事件的好处：**事件就是业务的"骨架时间线"**。一旦时间线清楚了，每个事件"谁产生的、谁关心"自然浮现。

接着给每个事件配上**触发它的命令**和**它涉及的对象**，就得到一张"命令→聚合→事件"的草表：

| 命令 | 哪个对象处理 | 产生的事件 |
| --- | --- | --- |
| 提交订单 PlaceOrder | 订单 | OrderPlaced |
| 占用库存 ReserveStock | 库存 | StockReserved |
| 支付 Pay | 支付单 | PaymentSucceeded |
| 发货 Ship | 订单 | OrderShipped |
| 取消订单 CancelOrder | 订单 | OrderCancelled |
| 释放库存 ReleaseStock | 库存 | StockReleased |

到这一步，三个核心对象已经自己冒出来了：**订单、库存、支付单**。

### 4. 第二步：划限界上下文

看上面那张表，三个对象明显属于三套不同的"语言和职责"：

- **订单上下文**：管订单的生命周期（提交、支付状态、发货、完成、取消）。
- **库存上下文**：管商品可用量、占用、释放、扣减。
- **支付上下文**：管支付单、对接渠道、回调。

为什么这么分？回到第 3 课的判据——**通用语言在哪里发生变化，边界就在哪里**：

- "订单"在订单上下文里是核心聚合，但在库存上下文里它只是一个"占用来源的引用"。
- "库存"在库存上下文里是核心，在订单上下文里只是下单时校验一下的"外部能力"。
- "支付"在支付上下文里是支付单聚合，在订单上下文里只是"订单的一个状态来源"。

同一个词在不同上下文含义不同 → 这就是三个上下文的铁证。

```
┌──────────────┐   OrderPlaced    ┌──────────────┐
│ 订单上下文     │ ───────────────→ │ 库存上下文     │
│ Order(聚合根)  │ ←─StockReserved─ │ Stock(聚合根)  │
└──────┬───────┘                  └──────────────┘
       │ OrderPaid ↑ PaymentSucceeded
┌──────┴───────┐
│ 支付上下文     │
│ Payment(聚合根)│
└──────────────┘
```

### 5. 第三步：每个上下文里识别聚合、实体、值对象

现在进到每个上下文内部，用第 4 课的判据（看身份 / 看值 / 看一致性边界）建模。

**订单上下文：**

- `Order`：聚合根（有身份、有生命周期、守状态流转规则）。
- `OrderItem`：聚合内部实体（明细，依附订单存在，不单独建仓储）。
- `Money`、`Address`（收货地址）、`OrderStatus`：值对象。
- 守在 `Order` 里的不变量：
  - 订单总金额 = 明细金额之和（减优惠）
  - 只有待支付订单能取消
  - 已支付不能回到待支付
  - 状态机：待支付→已支付→待发货→已发货→已完成；任意支付前可→已取消

**库存上下文：**

- `Stock`（某个 SKU 的库存）：聚合根。
- 关键值/字段：可用量 `available`、占用量 `reserved`。
- 守在 `Stock` 里的不变量：
  - `available >= 0`（不能超卖）
  - 占用：`available -= n; reserved += n`，且 `available` 扣减后不能为负
  - 释放：`reserved -= n; available += n`

**支付上下文：**

- `Payment`（支付单）：聚合根。
- 不变量：金额一致、同一支付单不能重复成功、状态流转受控。

注意我们**没有把订单、库存、支付塞进一个大聚合**——它们是三个上下文里各自的聚合根。这就引出本课最硬的难题。

### 6. 第四步（核心难题）：下单扣库存，是强一致还是最终一致？

这是电商建模绕不开、面试必问的一题。摊开思考过程：

**自问：订单和库存能不能放进同一个聚合，下单时一个事务里既建订单又扣库存？**

技术上能，但代价极大：

- 库存是**热点**。爆款 SKU 下单并发极高，如果订单和库存在一个聚合/一个事务里，库存行锁会把整个下单链路卡死。
- 订单和库存是**两个上下文**（第 4 步已论证），把它们焊死违背了上下文边界。

所以**不该放一个聚合**。那它们之间用什么一致性？两条主流路线：

**路线 A：同步扣减（下单事务里 RPC 调库存，强一致倾向）**

```
下单应用服务（一个用例）：
  1. 调库存上下文：ReserveStock(sku, n)   ← 同步，扣成功才继续
  2. 成功 → 创建 Order，落库
  3. 失败 → 直接拒绝下单（库存不足）
```

- 优点：用户提交时就知道库存够不够，体验直接，不会"下单成功又被取消"。
- 缺点：下单和库存强耦合，库存挂了下单也挂；跨服务调用，要处理"订单建失败但库存已占用"的补偿。
- **超卖防护靠库存聚合内部**：`UPDATE stock SET available=available-n WHERE sku=? AND available>=n`，靠数据库行级原子更新 + `available>=n` 条件，保证不超卖。这是库存聚合守不变量的真实落地。

**路线 B：异步占用 + 最终一致（先建订单，事件驱动扣库存）**

```
1. 创建 Order（状态：待确认），落库 + 写 outbox(OrderPlaced)
2. 库存上下文订阅 OrderPlaced → 尝试占用库存
3a. 占用成功 → 发 StockReserved → 订单转"待支付"
3b. 占用失败 → 发 StockReserveFailed → 订单自动取消
```

- 优点：下单链路极快、不被库存拖住，能抗超高并发（秒杀常用）。
- 缺点：最终一致——可能出现"提示下单成功，几百毫秒后因无货被取消"，体验要产品配合（比如显示"占位中"）。

**怎么选？** 回到第 4 课那句话：**需要强一致就别异步，能接受最终一致才上事件。**

- 普通电商、下单量不极端 → **路线 A**（同步扣减），简单可靠、体验好。
- 秒杀/大促、库存是极端热点 → **路线 B** 或更激进的方案（Redis 预扣 + 异步落库），用最终一致换吞吐。

**这道题没有标准答案，关键是你能说清两条路线的权衡和适用场景**——面试官要的就是这个。

### 7. 第五步：用领域事件把跨上下文流程串起来

把三个上下文用第 6 课的领域事件连起来（这里用偏最终一致的写法）：

```
订单上下文：  order.pay() 成功
              → 同一本地事务：更新订单 + 写 outbox(OrderPaid)

库存上下文：  订阅 OrderPaid（幂等）
              → 把"占用"转为"真实扣减"（reserved -= n，库存真正出账）

物流/发货：   仓库发货 → OrderShipped → 订单转已发货

取消流程：    用户取消 → order.cancel()（聚合校验：仅待支付可取消）
              → OrderCancelled → 库存上下文订阅 → 释放库存(StockReleased)
```

每个上下文内部该强一致的（改自己的聚合 + 写自己的 outbox）放本地事务；上下文之间靠事件做最终一致。**取消→释放库存**这条链尤其典型：它绝不该是订单上下文直接去改库存表，而是订单只发 `OrderCancelled`，库存自己决定怎么释放。

### 8. 第六步：落到代码骨架（这时候才碰技术）

到这里模型已经清楚，最后才映射到代码（用第 5、7 课的分层）。以订单上下文为例：

```java
// —— 应用层：编排用例，不写业务规则 ——
class OrderAppService {
    @Transactional
    OrderId placeOrder(PlaceOrderCommand cmd) {
        stockClient.reserve(cmd.sku(), cmd.qty());      // 路线A：同步占用（出站端口）
        Order order = Order.create(cmd);                // 工厂/静态方法：保证一出生就合法
        orderRepo.save(order);                          // 仓储：整聚合存
        return order.id();
    }

    @Transactional
    void pay(PayCommand cmd) {
        Order order = orderRepo.findById(cmd.orderId());
        order.pay(cmd.payInfo());                        // 规则在聚合：已关闭不能支付等
        orderRepo.save(order);
        eventPublisher.publish(new OrderPaid(...));      // 经 outbox 发出
    }
}

// —— 领域层：聚合根守不变量 ——
class Order {                       // 聚合根
    private OrderStatus status;
    private List<OrderItem> items;  // 内部实体
    private Money totalAmount;      // 值对象

    static Order create(PlaceOrderCommand cmd) { /* 校验 + 装配，保证总额=明细和 */ }
    void pay(PayInfo info)   { if (status == CLOSED) throw ...; this.status = PAID; }
    void cancel()            { if (status != UNPAID) throw new IllegalStateException("仅待支付可取消"); this.status = CANCELLED; }
}

// —— 领域层定义接口，基础设施层实现（依赖倒置）——
interface OrderRepository { Order findById(OrderId id); void save(Order o); }
interface StockClient { void reserve(Sku sku, int qty); }   // 出站端口，实现里含防腐层
```

注意这套代码把前 7 课全用上了：聚合根守规则、应用服务只编排、仓储整聚合存取、依赖倒置定义端口、`StockClient` 的实现里藏着对库存上下文的调用 + 防腐层、跨上下文靠 `OrderPaid` 事件。

### 9. 复盘：这一套套路你可以直接复用

把整个建模流程抽象成一个可复用的清单，下次拿到任何业务都能套：

1. **别建表，先列事件**：通读需求，用过去式把"发生了什么事实"全列出来（事件风暴）。
2. **配命令和对象**：每个事件配"什么命令触发、哪个对象处理"，核心聚合会自己冒出来。
3. **划上下文**：看通用语言在哪里变味，把对象归到不同上下文。
4. **建聚合**：每个上下文内用"身份/值/一致性边界"识别聚合根、实体、值对象，写清不变量。
5. **定一致性**：聚合之间哪些强一致（同事务/同步）、哪些最终一致（事件），逐对决定——这是最考功力的一步。
6. **串事件**：用领域事件把跨上下文流程连起来，每段强一致放本地事务 + outbox。
7. **最后落代码**：套分层 + 依赖倒置，聚合守规则、应用服务编排、仓储存取、端口对接外部。

记住整条路线的灵魂：**模型来自业务，不是来自表；边界来自语言，不是来自技术；一致性来自规则，不是来自习惯。**

## 小结（3-5 条关键点）

- 建模起手式是"先列事件、后建表"：用事件风暴从需求里捞出领域事件，命令和核心聚合会自然浮现。
- 订单/库存/支付是**三个限界上下文**（同一个词在各上下文含义不同），各有自己的聚合根，**不该焊成一个大聚合**。
- 下单扣库存是核心难题：**同步扣减（强一致、体验好、耦合高）vs 异步占用（最终一致、抗并发、体验需配合）**，按是否需要强一致和并发量级来选，没有唯一答案。
- 超卖防护落在**库存聚合内部**的原子条件更新（`available>=n`），取消→释放库存靠**事件**而非直接改库存表。
- 整个流程可抽象成"列事件→配命令→划上下文→建聚合→定一致性→串事件→落代码"的七步套路，灵魂是"模型来自业务、边界来自语言、一致性来自规则"。
