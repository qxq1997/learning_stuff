# DDD - 第 9 课：聚合划分、一致性边界与性能权衡

## 学习目标（本节结束后你能做到什么）

- 把第 4 课的"聚合是一致性边界"和第 8 课的实战权衡，提炼成几条可落地的聚合设计经验法则。
- 掌握聚合设计四条经典规则，尤其是"聚合之间用 ID 引用"和"一个事务只改一个聚合"。
- 理解为什么聚合大小会直接影响**并发冲突**和**性能**，会用乐观锁保护聚合不变量。
- 能在"强一致大事务"和"最终一致 + 事件"之间做有理有据的权衡。
- 知道这些规则不是教条，什么时候可以、应该打破它们。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 这一课在回答一个问题：聚合到底该画多大

前面几课你已经反复碰到聚合：第 4 课说它是"一致性边界"，第 8 课实战里我们纠结"订单和库存能不能放一个聚合"。

但有一个问题一直没系统回答：

**一个聚合，边界到底画到哪？画大画小，会带来什么后果？**

这一课就专门啃这个。它是 DDD 战术设计里**最难、也最能体现功力**的部分——因为它不是"对错题"，而是"权衡题"。画错了不会立刻报错，但会在半年后以"并发冲突频发""一个接口慢到要命""事务死锁"的形式找上门。

### 2. 聚合设计的四条经典规则

业界（尤其 Vaughn Vernon 的《实现领域驱动设计》）总结过四条聚合设计经验法则。先全列出来，再逐条拆：

1. **在一致性边界内建模真正的不变量**——聚合里只放"必须同时成立"的规则。
2. **设计小聚合**——聚合尽量小。
3. **聚合之间通过 ID 引用，而不是对象引用**。
4. **一个事务只修改一个聚合实例；跨聚合用最终一致性**。

这四条是相互支撑的一套，不是孤立的口号。下面逐条讲。

### 3. 规则一：聚合里只放真正的不变量

第 4 课讲过，再强化一遍，因为它是另外三条的地基。

**不变量 = 一条必须在一次修改后立刻成立、不容许中间态违反的业务规则。**

只有真正的不变量，才有资格进同一个聚合。例如订单聚合里：

- ✅ "订单总金额 = 明细金额之和" —— 改明细的同一瞬间总额必须对得上，这是不变量，进聚合。
- ❌ "订单支付成功后要加积分" —— 晚几秒加积分没人会死，这不是不变量，**不该**进订单聚合，用事件做最终一致。

判断标准就一句：**这条规则如果晚一会儿才成立，业务能不能接受？能接受 → 不是不变量 → 别进聚合。**

很多人聚合越画越大，根子就是把"相关的"误当成"必须强一致的"。**相关 ≠ 一致性边界**。

### 4. 规则二：设计小聚合（为什么大聚合是灾难）

第 4 课说过"聚合越大越不好"，这里讲清**到底坏在哪**——两个硬伤：

**硬伤一：并发冲突。**

聚合是"一起被维护"的单位，意味着对聚合的修改要**整体加锁/整体校验版本**（见第 6 节乐观锁）。聚合越大，包的东西越多，"撞车"概率越高。

举个反例：假设你把"一个用户的所有订单"塞进一个 `Customer` 大聚合。那么用户 A 同时下两个单，这两次操作都要改 `Customer` 聚合 → 版本冲突 → 一个必须重试。明明是两个互不相干的订单，却互相阻塞。聚合太大，把本来无关的操作**人为串行化**了。

**硬伤二：加载性能。**

聚合通常要整体加载（仓储 `findById` 取的是整个聚合）。大聚合意味着：改一个小字段，也得把一大坨关联数据全捞进内存。`Customer` 聚合如果包含几千个历史订单，每次操作都加载几千条 —— 性能直接崩。

所以经验是：**聚合尽量小，理想情况下一个聚合就是一个聚合根 + 少量紧密依附的内部对象。** 拿不准时，倾向于拆小，而不是合大。

### 5. 规则三：聚合之间用 ID 引用，不用对象引用

这条很具体，也最容易在代码里违反。

**反例（对象引用）：**

```java
class Order {
    private Customer customer;     // ← 直接持有另一个聚合的对象引用
    private List<OrderItem> items;
}
```

**正例（ID 引用）：**

```java
class Order {
    private CustomerId customerId; // ← 只持有另一个聚合的 ID
    private List<OrderItem> items; // 内部实体仍是对象引用（这是同一聚合内部，OK）
}
```

为什么聚合之间只能用 ID 引用？三个理由：

1. **逼出清晰的聚合边界**。如果 `Order` 直接持有 `Customer` 对象，加载订单时很容易顺手把整个客户聚合也加载进来，边界就糊了。只持有 `CustomerId`，物理上就隔开了。
2. **防止误改别的聚合**。手里只有 `customerId`，你没法在订单的事务里顺手 `customer.setXxx()` —— 这正好呼应规则四（一个事务只改一个聚合）。
3. **性能可控**。需要客户信息时，显式地用 `customerId` 去 `customerRepository.findById()` 加载，加载什么、加载多少由你说了算，不会被对象图牵着走。

注意区分：**聚合内部**的实体（如 `Order` 和它的 `OrderItem`）用对象引用是对的，它们本来就是一个整体。ID 引用的规则只针对**跨聚合**。

### 6. 规则四 + 并发：一个事务只改一个聚合，靠乐观锁守不变量

**规则四：一个数据库事务，只修改一个聚合实例；需要改多个聚合时，用最终一致性（事件）。**

为什么这么严？因为这条规则把"事务边界"和"聚合边界"对齐了，好处：

- 事务小 → 锁的范围小 → 并发好、不容易死锁。
- 每个聚合是自洽的一致性单元，跨聚合的事不强求同步，自然走第 6 课的事件 + Outbox。

第 8 课的下单就是活例子：订单聚合和库存聚合不在一个事务里改，而是订单改完发 `OrderPaid`，库存上下文异步响应。

**那聚合自己的不变量，在并发下怎么守？** 答案是**乐观锁（版本号）**：

```sql
UPDATE orders
   SET status = 'PAID', version = version + 1
 WHERE id = ? AND version = ?   -- ← 版本不匹配说明被人改过，更新 0 行
```

机制：聚合带一个 `version` 字段，每次保存 `version + 1`，保存时校验"我读出来时的 version 和现在库里的是否一致"。如果别人在我之前改了，version 对不上，更新失败 → 应用层捕获冲突 → 重试整个用例（重新读聚合、重新执行）。

为什么用乐观锁而不是悲观锁（`SELECT ... FOR UPDATE`）？因为聚合操作通常"读出来 → 在内存里跑业务规则 → 存回去"，中间隔着应用逻辑，悲观锁会把行锁持有很久，并发差。乐观锁让大家先各干各的，只在提交时检测冲突，更适合聚合这种"内存里改、最后存回"的模式。

**聚合 + 乐观锁，是聚合在并发下守住不变量的标准武器。** 这也反过来解释了规则二：聚合越小，乐观锁冲突越少。

### 7. 核心权衡：强一致大事务 vs 最终一致 + 事件

前面的规则都指向同一个权衡，单独拎出来讲透，因为这是面试和架构评审最常被追问的。

**选项 A：强一致大事务**——把多个聚合塞进一个大事务，一次性改完。

- 优点：实现简单直白，不用处理中间态、不用幂等、不用补偿。读出来永远是一致的。
- 缺点：事务大、锁多、并发差、容易死锁；跨服务时根本做不到（分布式事务又重又脆）。

**选项 B：最终一致 + 事件**（第 6 课那套 Outbox）——每个聚合各自小事务，跨聚合用事件。

- 优点：事务小、并发好、能跨服务、能独立伸缩。
- 缺点：复杂——有中间态（订单已支付但积分还没加）、要幂等、要处理事件丢失/重复、要补偿、要给产品解释"为什么有几秒延迟"。

**怎么选？** 给你一个判断顺序：

1. **这些规则是不是真不变量**（晚一会儿成立行不行）？不是 → 直接选 B，别犹豫。
2. **是不是跨服务/跨上下文**？是 → 基本只能 B（分布式强一致代价太大）。
3. **并发量级高不高**？高 → 倾向 B（大事务扛不住）。
4. **都不满足**（单服务内、确实是强一致需求、并发不高）→ 可以选 A，别为了"DDD 范儿"硬上事件，把简单问题复杂化。

一句话：**强一致是有成本的（并发、伸缩），最终一致也是有成本的（复杂度、心智）。没有免费的一致性，只有适合场景的取舍。**

### 8. 规则不是教条：什么时候可以打破

最后必须说清——上面四条是**经验法则，不是物理定律**。Vaughn Vernon 自己也强调它们可以违反，前提是你**知道自己在违反、并为此负责**。

几个常见的合理打破场景：

- **小聚合 + 性能反而更差时**：如果两个对象几乎总是一起读写、又确实需要强一致，拆成两个聚合反而要两次加载、还要处理最终一致，那合成一个聚合可能更合理。
- **"一个事务改一个聚合"在落地初期太重时**：有些团队在单体早期，为了快，会在一个事务里改两个聚合。只要你清楚这会增加锁范围、将来拆服务时要还债，作为阶段性选择是可以的。
- **历史包袱**：老系统改造，没法一步到位，先把边界画清、逐步演进，比一次性推倒重来务实。

判断能不能破的标准只有一个：**你是不是清楚地知道破了哪条规则、换来了什么、欠下了什么债。** 不知道而违反，叫踩坑；知道而违反，叫权衡。

### 9. 把第 8 课的下单用这一课的规则复盘一遍

回头看第 8 课的模型，验证它其实严格遵守了这四条：

- **规则一**：订单聚合里只放"总额=明细和""状态流转"这些真不变量；"加积分"被踢出去用事件。✅
- **规则二**：订单、库存、支付各自是小聚合，没有"用户的所有订单"这种大聚合。✅
- **规则三**：`Order` 里持有的是 `customerId`、`sku`，不是 `Customer`、`Stock` 对象。✅
- **规则四**：下单事务只改订单（或只占库存），订单↔库存↔支付之间全靠 `OrderPaid`/`StockReserved`/`OrderCancelled` 事件做最终一致；每个聚合用 version 乐观锁守自己的不变量。✅

你看，第 8 课当时是凭"权衡直觉"做的决定，这一课给了它**系统的理论支撑**。以后你做聚合设计，就能从"凭感觉"升级到"有法则、能解释"。

## 小结（3-5 条关键点）

- 聚合设计四法则：**只放真不变量、设计小聚合、聚合间用 ID 引用、一个事务只改一个聚合（跨聚合走最终一致）**——它们是一套相互支撑的整体。
- 大聚合的两个硬伤是**并发冲突**（无关操作被人为串行化）和**加载性能**（改小字段也要全量加载）；拿不准就拆小。
- 聚合在并发下守不变量靠**乐观锁（版本号）**：读时记 version、存时校验，冲突则重试整个用例；它比悲观锁更适合"内存里改、最后存回"的聚合模式。
- 核心权衡是**强一致大事务 vs 最终一致 + 事件**：按"是不是真不变量 / 是否跨服务 / 并发量级"决定，两边都有成本，没有免费的一致性。
- 这四条是**经验法则不是教条**，可以在"知道破了什么、欠了什么债"的前提下合理违反——知道而违反是权衡，不知道而违反是踩坑。
