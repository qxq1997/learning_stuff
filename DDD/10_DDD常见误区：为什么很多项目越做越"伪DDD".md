# DDD - 第 10 课：DDD 常见误区：为什么很多项目越做越"伪 DDD"

## 学习目标（本节结束后你能做到什么）

- 系统收口前 9 课里散落的各种坏味道，认清"伪 DDD"长什么样。
- 识别最常见、危害最大的几个误区：贫血模型、Service 大杂烩、把 DDD 当目录模板、跳过战略设计、过度设计。
- 知道每个误区的"病根"和"怎么改回来"，而不只是知道它是错的。
- 能在 code review 或面试里，一眼指出一个号称 DDD 的项目其实是伪 DDD，并说清理由。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么"伪 DDD"特别普遍

先说一个扎心的现实：**大多数号称"我们用了 DDD"的项目，其实是伪 DDD。**

原因不难理解：DDD 的术语（实体、值对象、聚合、仓储）很好抄，目录结构（domain/application/infrastructure）也很好建，但**它真正的价值——业务语义对齐、规则有稳定落点、边界清晰——是看不见摸不着的**。于是大量项目抄了形，丢了神：

- 建了 `domain` 包，里面装的全是只有 getter/setter 的数据类。
- 建了 `Repository`，里面全是 `updateStatus` 这种字段级方法。
- 建了一堆 `XxxDomainService`，业务逻辑全堆在里面。

**形似神不似**，这就是伪 DDD。这一课把前面散落提过的坏味道集中收口，逐个点名。

### 2. 误区一（头号杀手）：贫血模型

这是最常见、危害最大的伪 DDD。

**贫血模型（Anemic Domain Model）= 领域对象只有数据（字段 + getter/setter），没有行为；所有业务逻辑都堆在 Service 里。**

```java
// 贫血的"领域对象"（其实是数据袋）
class Order {
    private OrderStatus status;
    private Money amount;
    // 一堆 getter / setter，没有任何业务方法
    public void setStatus(OrderStatus s) { this.status = s; }
}

// 所有规则都在 Service 里
class OrderService {
    void cancel(Order order) {
        if (order.getStatus() == PAID) throw ...;   // 规则在这
        order.setStatus(CANCELLED);                  // 裸改字段
    }
}
```

**为什么这是病？** 回到第 4、5 课：

- 规则没有稳定落点。`cancel` 的规则在 `OrderService`，但别的入口（job、mq、另一个 service）想取消订单，要么重抄一遍规则，要么忘了抄 → 同一规则散在多处，对不齐。
- 对象失去了自我保护能力。任何人都能 `order.setStatus(任意值)`，聚合的不变量形同虚设。

这本质上就是 Martin Fowler 说的——**贫血模型是个反模式，它顶着面向对象的壳，干的是面向过程的事**。

**怎么改回来（充血）：** 把规则搬进对象，用业务方法替代 setter：

```java
class Order {
    void cancel() {
        if (status != UNPAID) throw new IllegalStateException("仅待支付可取消");
        this.status = CANCELLED;
    }
}
```

判断口诀：**如果你的领域对象全是 getter/setter、没有一个有业务含义的方法，那它不是领域模型，是数据袋，你写的是伪 DDD。**

### 3. 误区二：Service 大杂烩（transaction script 穿了件 DDD 外套）

这个和贫血模型是孪生兄弟。第 5 课点过名，这里再强调。

症状：一个 `XxxService`（或 `XxxDomainService`）几千行，什么都往里塞——业务规则、流程编排、SQL 拼装、参数校验、调外部接口，全混在一起。

**病根**：没有分清第 5 课讲的角色。该进聚合的规则进了 Service，该是应用服务编排的混进了业务规则，该是基础设施的技术细节也混进来了。

**怎么改回来**：按第 5 课的职责切——

- 业务规则 → 聚合 / 实体
- 跨聚合的规则 → 领域服务（且要"少而精"）
- 流程编排、事务 → 应用服务
- SQL、外部调用 → 基础设施层

记住第 5 课那个反例：领域服务也会退化成"贫血模型的逻辑垃圾场"，所以**不是把逻辑从 Service 挪到 DomainService 就叫 DDD**，挪错了还是大杂烩。

### 4. 误区三：把 DDD 当成目录结构 / 代码模板

很多团队的"DDD 落地"，约等于：

> 建好 `domain` / `application` / `infrastructure` 三个包，给每个表配一个 `Entity`、一个 `Repository`、一个 `Service`，完事。

**病根**：把 DDD 理解成一套**文件夹模板**，而不是一种**建模方法**。

这种项目的特征是：包结构很 DDD，但——

- `domain` 里全是贫血对象（误区一）。
- `Entity` 和数据库表一一对应（其实是 ORM 实体冒充领域模型，见误区六）。
- 没有真正的聚合边界，每张表一个"聚合"。

**怎么识别**：看包结构没用，要看**行为有没有落进对象、聚合边界是不是按一致性画的、有没有真正的通用语言**。目录是 DDD 最不重要的部分。

一句话：**DDD 是建模方法，不是脚手架模板。能 new 出三个文件夹，不等于在做 DDD。**

### 5. 误区四：跳过战略设计，直接上战术设计

这是另一类很隐蔽的伪 DDD：团队很认真地写聚合、值对象、领域事件（战术设计做得有模有样），但**从来没认真做过战略设计**——没划限界上下文、没统一通用语言。

**后果**（回到第 2、3 课）：

- 上下文边界混乱，一个聚合的概念在不同模块含义不一样，越改越乱。
- "订单"这个词在交易、履约、结算里其实是不同的东西，但被强行做成了一个共享的大模型 → 各方都来改它 → 没人敢动。
- 战术设计做得再漂亮，建在错误的边界上，等于在流沙上盖楼。

**病根**：DDD 的难点和价值，**七成在战略设计（划边界、统一语言），三成在战术设计（写代码）**。但战术设计有现成代码可抄，战略设计要啃业务，于是大家都跳过最难也最值钱的部分。

**怎么改回来**：顺序不能反——**先战略后战术**。先用第 8 课那套"列事件→划上下文"把边界和语言定下来，再进聚合。第 1 课开头反复强调的"先建立问题意识、先画业务地图"，针对的就是这个误区。

### 6. 误区五：仓储退化成 DAO

第 5 课详细讲过，这里作为坏味道清单收个尾。

症状：`Repository` 里出现 `updateOrderStatus(id, status)`、`selectByCondition(...)`、`batchUpdateAmount(...)` 这种**字段级、单表 CRUD** 的方法。

**病根**：换了个名字的 DAO。它鼓励调用方绕过聚合根直接改字段，聚合的保护罩被捅穿。

**怎么识别 / 改回来**：仓储只该有 `save(聚合)` / `findById` / `remove` 这类**以聚合为整体**的方法。想改状态，得 `order.pay()` 后整体 `save`。看到字段级 update 出现在仓储里，基本就是伪 DDD 信号。

### 7. 误区六：把 ORM 实体当领域模型

这个很隐蔽，但很致命。

症状：领域对象上挂满了 ORM 注解（`@Entity` `@Table` `@Column` `@OneToMany`），字段和表列一一对应，关系映射决定了对象结构。

**病根**：**对象结构被数据库表结构绑架了**。你以为在按业务建模，其实在按表建模——又回到了第 7 课批判的"面向数据库编程"。表怎么设计，你的"领域模型"就长什么样，业务语义根本没机会表达。

典型连锁反应：因为对象 = 表，所以聚合边界 = 表边界 = 每张表一个聚合（误区三的同款病）。

**怎么改回来**：让领域模型**纯净**，不带任何持久化注解；持久化的事交给基础设施层——要么用单独的持久化对象（PO）做映射 + 转换，要么用支持"充血对象与表分离"的方式。核心是第 7 课的依赖倒置：**领域模型不认识数据库**。

> 务实提醒：完全分离 PO 和领域对象会增加转换成本，小项目里把 ORM 实体当领域对象用、但坚持往上加业务方法，也是常见的折中。关键是**别让表结构反过来决定业务模型**。

### 8. 误区七：贫血领域事件 / 事件满天飞

两个相关的事件坏味道，第 6 课提过，收口一下：

- **贫血事件**：事件里只有一个 id（`OrderPaid{orderId}`），下游被迫回查 → 运行时耦合。改法：带上描述事实所需的关键稳定字段。
- **事件满天飞**：明明进程内、明明该强一致的协作，也硬拆成异步事件，结果到处是中间态、幂等、补偿，复杂度爆炸。改法：回到第 9 课的权衡——**真不变量就同步，能最终一致才发事件**。

事件是好东西，但**不是越多越解耦**。该同步的强行异步，是另一种过度设计。

### 9. 误区八（反方向）：过度设计——简单 CRUD 也硬上 DDD

前面七个误区都是"抄了形丢了神"，这个误区相反：**神也想要，但用错了地方**。

不是所有系统都需要 DDD。回到第 1 课的定位：**DDD 是用来驾驭"业务复杂度"的，不是用来驾驭"技术复杂度"或"简单增删改查"的。**

如果一个模块就是：

- 一堆配置项的增删改查
- 一个没什么业务规则的数据录入表单
- 一个纯报表查询

那给它套上聚合、仓储、领域服务、领域事件、防腐层，纯属**用大炮打蚊子**。开发成本翻几倍，收益是零，团队还会因为"DDD 这么麻烦"而反感它。

**判断标准**：这块的复杂度主要来自**业务规则**（规则多、易变、需要多人对齐语言）→ 值得 DDD；主要是**简单数据搬运**→ 直接 CRUD / 事务脚本更划算。

**好的架构师会在一个系统里混用**：核心域（复杂业务）上 DDD，支撑域/通用域（简单 CRUD）就用最朴素的方式。这正好呼应第 2 课的核心域/支撑域划分——**把 DDD 的力气花在核心域上**。

### 10. 一张"伪 DDD 自检表"

把这一课压成一张你能直接拿去 review 的清单。一个项目命中越多，越像伪 DDD：

| # | 坏味道 | 真 DDD 应该是 |
| --- | --- | --- |
| 1 | 领域对象全是 getter/setter，没业务方法 | 规则进对象，用业务方法（充血） |
| 2 | Service 几千行，啥都往里塞 | 按角色切：聚合/领域服务/应用服务/基础设施 |
| 3 | "DDD" = 建了三个包 + 每表一套 Entity/Repo/Service | DDD 是建模方法，看行为和边界，不看目录 |
| 4 | 没划上下文、没统一语言，直接写聚合 | 先战略后战术，先画业务地图 |
| 5 | 仓储里有 updateStatus 这种字段级方法 | 仓储以聚合为整体存取 |
| 6 | 领域对象挂满 ORM 注解、字段=表列 | 领域模型纯净，不认识数据库 |
| 7 | 事件只带 id / 该同步的也硬异步 | 事件带够字段；真不变量同步，最终一致才发事件 |
| 8 | 简单 CRUD 也硬套全套 DDD | 把 DDD 用在核心域，支撑域用朴素方式 |

**最锋利的一句自检**：去问"这个项目的业务规则到底落在哪里？"

- 答得出"在聚合里、由聚合根守"→ 大概率是真 DDD。
- 答"在 Service 里"或者支支吾吾 → 大概率是伪 DDD。

规则的落点，是真伪 DDD 的试金石。

## 小结（3-5 条关键点）

- 伪 DDD 的本质是**抄了形（术语、目录）丢了神（语义对齐、规则落点、清晰边界）**；它特别普遍，因为形好抄、神难学。
- 头号杀手是**贫血模型**（对象只有 getter/setter，规则全在 Service）；它顶着 OO 的壳干面向过程的事，连带产生 Service 大杂烩。
- 几个隐蔽误区：**把 DDD 当目录模板、跳过战略设计直接写聚合、ORM 实体冒充领域模型、仓储退化成 DAO**——共同点是让技术/表结构反过来绑架了业务模型。
- 反方向的误区是**过度设计**：简单 CRUD 也硬上 DDD；正确做法是把 DDD 的力气花在**核心域**，支撑域用朴素方式。
- 最锋利的自检问题是"**业务规则到底落在哪里**"：答得出在聚合里就是真 DDD，答在 Service 里或答不上来就是伪 DDD。
