# DDIA - 第 2 课：可靠性、可扩展性、可维护性

## 学习目标（本节结束后你能做到什么）

- 准确区分可靠性、可扩展性、可维护性，而不是把它们混成“系统做得好不好”这种模糊判断。
- 理解这三个词在工程上分别关注什么问题、解决什么风险。
- 知道为什么“系统现在能跑”不代表它可靠，也不代表它可扩展，更不代表它可维护。
- 能用一个具体业务系统例子，判断某个问题究竟属于可靠性问题、可扩展性问题，还是可维护性问题。
- 能用 MTBF/MTTR/SLO/错误预算等量化指标去讨论可靠性，而不是只停留在形容词。
- 理解为什么平均响应时间是骗人的，掌握尾延迟（tail latency）的数学原理和它在扇出系统里的放大效应。
- 理解 Amdahl 定律和通用可扩展性定律（USL）给出的硬性天花板，知道为什么“加机器”并不总能线性扩展。
- 建立 DDIA 的第一套分析框架：看一个系统时，先从这三个维度切开来看，每个维度都有可量化的工程语言。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 这三个词为什么是全书总纲

DDIA 开篇先讲的不是数据库模型，也不是复制和事务，而是可靠性、可扩展性、可维护性。这个顺序不是偶然的。

因为后面所有技术选型，本质上都在回答同一个问题：
**我们为什么要这样设计系统？**

如果没有这三个总目标，你后面学复制、分区、事务、一致性时，就很容易变成“背方案”：

- 主从复制是什么
- 分库分表是什么
- 事务隔离级别是什么
- Kafka 为什么快

但如果你先有这三个标准，你看后面任何方案都会更清晰：

- 这是在增强可靠性，还是在提升可扩展性？
- 这个设计是不是让维护成本变高了？
- 它解决了一个问题，是不是又引入了另一个问题？

这就是 DDIA 的阅读方式：
**先有评估标准，再看技术手段。**

### 2. 可靠性：不是“永远不坏”，而是“坏了还能顶住”

很多初学者把可靠性理解成“系统不宕机”。这个理解太窄了。

更准确地说，可靠性是在问：

- 某台机器坏了，系统还能不能继续服务？
- 某个服务超时了，是否会引发整条链路雪崩？
- 数据库主机挂了，数据会不会丢？
- 程序员写错代码、运维误操作、用户重复提交时，系统有没有保护机制？

所以可靠性的关键词是：
**容错、恢复、保护、降级、冗余。**

你可以把可靠性理解成“系统面对异常时的生存能力”。

#### 2.1 一个生活类比

假设你经营一家餐厅。

- 厨房里有一台冰箱坏了，餐厅还能不能营业？
- 收银系统一时卡顿，服务员有没有手工兜底方案？
- 某个员工把订单写错了，系统能不能发现异常并纠正？

如果这些情况一发生，整个餐厅立刻瘫痪，那这家餐厅就不可靠。
所以可靠性的重点不在“绝对不会出问题”，而在“出问题时有没有缓冲和恢复能力”。

#### 2.2 故障的三种来源：硬件、软件、人

DDIA 把可靠性威胁明确分成三类，而且告诉你哪一类被低估得最严重。

**硬件故障** —— 大家最先想到，也最容易解决。
不是“万一发生”，而是“按概率必然发生”。几个工业界的经验数字：

- 磁盘年失效率（AFR）：Backblaze 每季度的公开报告，典型消费盘在 1%–2%/年，企业盘 0.5%–1%/年。一个 10000 块盘的集群，每天都会坏 0.1–0.5 块。
- DRAM 错误：Google 2009 年那篇著名的 DIMM 研究（Schroeder et al.）发现，**每年约 8% 的 DIMM 会遇到可纠正错误**，大规模集群里不开 ECC 内存是在自杀。
- 电源、风扇、网卡、光模块都有各自的 AFR，没有一个是“几乎不坏”。

所以“加冗余、做备份、做健康检查”在硬件层面是**预算问题，不是思考问题**。

**软件故障** —— 比硬件更难缠。
特点是：同一个 bug 会**在所有节点上同时触发**，冗余救不了你。常见模式：

- 某个边界条件触发级联放大（2012 AWS Elastic Beanstalk 的一次故障就是这种）
- 进程吃满某项资源（内存泄漏、文件描述符不回收、goroutine 泄漏）
- 依赖的第三方服务返回了“合法但非预期”的数据
- 时间、时区、闰秒这类语义坑（2012 年闰秒引发过全球 Java 进程 CPU 100%）

**人为错误** —— 根据 Google 和 AWS 公开的事后复盘，**大规模事故里配置变更和运维操作引起的占 40%–70%**。比硬件和纯代码 bug 都多。典型形态：

- 改配置时写错一个参数（上一章提到的 2017 S3 典型案例）
- 上线时回滚路径没测过
- 有 root 权限的人执行了有副作用的命令
- 灰度、分流、限流等保护被误关

DDIA 对此给出的治理思路不是“让人不犯错”，而是**把系统设计成让错误变难、让影响变小、让恢复变快**：

- 设计上：把生产环境和沙箱环境用权限和 API 分开
- 流程上：危险操作需要人工双签，或者“do nothing"-first 的 dry run
- 技术上：任何部署都带自动回滚、任何高权限命令都留 audit trail
- 文化上：无指责复盘（blameless postmortem），让大家敢把教训说出来

这也是为什么后面很多看似“运维问题”的话题（监控、灰度、配置管理）会反复出现——它们都是在降低第三类故障的成本。

#### 2.3 可靠性不是形容词，要用数学讲

你会在行业交流里经常听到“几个 9”——99.9%、99.99%、99.999%。这不是吹牛，是可量化的。

**可用性（Availability）** 最基本定义：

$$ A = \frac{MTBF}{MTBF + MTTR} $$

其中 `MTBF`（Mean Time Between Failures，平均无故障时间）和 `MTTR`（Mean Time To Recovery，平均恢复时间）。

这条公式的含义非常工程化：**想提升可用性，有两条路——要么让坏的间隔更长，要么让坏了恢复得更快。** 实践中，后者几乎总是更划算。把 MTTR 从 1 小时降到 5 分钟，比把 MTBF 从半年拉到一年要容易得多，且效果更好。

每多一个 9 对应的“允许停机时间”大致是这样：

| 可用性 | 每年允许不可用 | 每月允许不可用 |
|---|---|---|
| 99%     | ~3.65 天   | ~7.3 小时  |
| 99.9%   | ~8.76 小时 | ~43.8 分钟 |
| 99.99%  | ~52.6 分钟 | ~4.4 分钟  |
| 99.999% | ~5.26 分钟 | ~26 秒     |

注意一个反直觉事实：**99.999% 的系统每月只剩 26 秒可以出问题**。这意味着你根本没有“人肉响应”的时间窗口，只能靠自动化。这就是为什么大厂的五个 9 系统一定是靠自动故障切换 + 多活部署，而不是靠运维值班手速。

#### 2.4 SLA / SLO / SLI 与错误预算

光讲 MTBF 不够用，Google SRE 在工程化里把可靠性落成了三层指标：

- **SLI（Service Level Indicator）**：服务的真实测量值，比如“过去 5 分钟 P99 延迟”“HTTP 5xx 错误率”。
- **SLO（Service Level Objective）**：你给自己设的目标，比如“每个月 P99 延迟要低于 300ms 的时间占 99.9%”。
- **SLA（Service Level Agreement）**：对外承诺，通常比 SLO 留一档冗余（SLO 99.95% → SLA 99.9%），因为 SLO 破了是内部警报，SLA 破了要赔钱。

然后引入一个工程杠杆——**错误预算（Error Budget）**。

如果你的 SLO 是 99.9%，那就意味着一个月里允许有 0.1% 的时间出错。这 0.1% 就是你这个月的“预算”。

这张预算表直接决定开发节奏：

- 预算还多 → 可以发大版本、可以做迁移、可以做压力测试
- 预算快用完 → 冻结非必要上线，把资源投向稳定性
- 预算已透支 → 只许修 bug、不许发新功能

这就把“可靠性”从“一个抽象的好东西”变成了**可以和业务增长对冲的货币**。DDIA 没展开讲 SRE 这套语言，但你读后面章节时如果带着这个视角，会对很多设计决策（比如为什么要搞限流、为什么要搞金丝雀）理解得更透。

#### 2.5 在数据系统里，可靠性通常表现为哪些设计

- 多副本：一台机器坏了，还有其他副本顶上
- 持久化：断电重启后数据还能恢复
- 重试与幂等：消息重复投递时不会造成多次扣款
- 限流与熔断：下游故障时，上游先保护自己
- 舱壁（bulkhead）：不同类型的流量走不同的线程池/连接池，一类被打爆不会拖垮另一类
- 监控与告警：出问题能尽早发现
- 备份与恢复：极端情况下还能把数据找回来
- 混沌工程：主动往生产环境里注入故障，让恢复路径常演常有

注意，复制和备份都和可靠性有关，但它们不是一个东西：

- 复制更偏向在线可用性和故障切换
- 备份更偏向灾难恢复和历史回滚

一个简单判别：**"rm -rf 把主库删了，复制救不了你，备份才救得了你。"** 因为复制会忠实地把删除也同步过去。

### 3. 可扩展性：不是“代码以后好不好改”，而是“负载变大还能不能撑”

你刚才的回答里，把“后面更多需求，方便适应更新需求”也算进了可扩展性。
这其实更接近可维护性里的“可演进”。
DDIA 里说的可扩展性，重点更偏向：**负载增长时，系统还能不能继续保持可接受的服务质量。**

这里的“负载”不只是用户数量，还可能是：

- QPS 增长
- 写入吞吐增长
- 存储数据量增长
- 某个热点 key 被疯狂访问
- 某类查询越来越重

所以可扩展性的核心问题是：

1. 负载是什么？
2. 瓶颈在哪？
3. 增长后怎么扩？

#### 3.1 一个生活类比

还是餐厅的例子。

现在中午突然从 50 个客人变成 500 个客人，问题就来了：

- 厨房能不能同时出更多菜？
- 收银台会不会堵死？
- 座位够不够？
- 外卖单会不会挤爆打印机？

如果客流一大，整个流程就卡死，这就是可扩展性不足。

注意，这和“这家餐厅装修好不好改”不是一回事。
前者是“负载涨了还扛不扛得住”，后者是“系统以后好不好调整”。
这就是可扩展性和可维护性的分界。

#### 3.2 如何描述负载：找对那个“load parameter”

DDIA 用 Twitter 的早期 home timeline 作为经典案例，值得在这里展开，因为它把“描述负载”这件事讲得非常透。

Twitter 的关键查询是：**某个用户打开首页时，要看到自己关注的所有人最新的 tweet，按时间倒序。**

最朴素的做法（方案 A，fanout-on-read）：

```sql
SELECT tweets.* FROM tweets
JOIN follows ON follows.followee_id = tweets.author_id
WHERE follows.follower_id = $user
ORDER BY created_at DESC LIMIT 50;
```

读的时候现 JOIN follow 表 + tweets 表。写 tweet 简单（一次 insert），但每次读主页都要做一次大 JOIN。

**方案 B，fanout-on-write（推模式）**：每当有人发一条 tweet，就把它复制进所有粉丝的“收件箱”表里，主页直接查自己的收件箱。读极快，但写一条 tweet 要触发 N 次写（N = 粉丝数）。

Twitter 选了 B。但这就引出一个问题：**Lady Gaga 有几千万粉丝，她发一条推就要写几千万次。** 这个写扇出（write fanout）成了新的负载参数。

最终解决方案是混合：

- 普通用户走 B（推模式）
- 名人走 A（拉模式），他们的 tweets 在读时才合并进来

这个案例的深刻之处在于：

1. **"负载"不是单一数字，而是一个有结构的分布**。Twitter 的关键负载参数不是 QPS，是**每个 tweet 作者的粉丝数分布**。
2. **读优化和写优化会相互转化**。B 方案把读延迟优化了，代价是写的扇出变大。
3. **真正的工程方案通常是两种极端的混合**。你会在 DDIA 后面章节反复看到这个模式。

当你描述任何系统的负载时，别只说 "10w QPS"，要说清楚：

- 读写比例
- 请求大小分布
- 热点分布（是均匀的还是有长尾？）
- 扇出结构（一次请求背后触发几次下游调用？）
- 工作集（working set）能不能装进内存？

#### 3.3 如何描述性能：平均值是骗人的

另一个容易错的地方是**用平均响应时间衡量性能**。

考虑一个服务，1000 个请求里：

- 990 个请求 10ms 返回
- 10 个请求 2000ms 返回

平均响应时间 = (990×10 + 10×2000) / 1000 ≈ 30ms。

这个“30ms”是工程上毫无意义的数字。**对那 10 个倒霉用户来说，真实体验是 2 秒，不是 30 毫秒。** 而且这 10 个用户往往就是最活跃的付费用户（因为他们的数据最多、查询最重）。

所以行业标准是看**百分位延迟（percentile latency）**：

- P50（中位数）：一半请求比它快，一半比它慢
- P95：95% 请求比它快
- P99：99% 请求比它快
- P999：99.9% 请求比它快

**一个产品的体感好坏，往往决定于 P99 或 P999，而不是 P50。** Amazon 的内部经验是：100ms 的额外延迟对转化率的影响是肉眼可见的，而这 100ms 往往藏在 P99 那 1% 的请求里。

#### 3.4 尾延迟在扇出系统里的放大（Tail at Scale）

Jeff Dean 和 Luiz Barroso 2013 年在 CACM 上有一篇非常有名的文章《The Tail at Scale》，揭示了一个反直觉的数学现象：

**一个请求越要扇出到越多后端，它的整体延迟就越会被尾部支配。**

直觉推导：假设单个后端的 P99 是 1 秒（99% 的请求都在 1 秒内返回）。你发起一个需要并行打到 100 个后端、等全部返回的请求。这 100 个中**至少有一个超过 1 秒**的概率是：

$$ 1 - 0.99^{100} \approx 63\% $$

也就是说，**单后端看起来很健康的 P99，到了 100 路扇出的查询身上，有 63% 的请求都要吃这个"尾巴"**。再扇出到 1000 个后端时，基本是 100%。

这解释了为什么：

- 搜索引擎、推荐系统、微服务大量存在“对冲请求”（hedge request）、“尾部截断”等怪招
- 很多公司的架构决定不是“怎么让 P50 更快”，而是“怎么把 P99 的长尾压下来”

这个洞见后面在讲分布式事务、共识、读修复时都会再次出现。

#### 3.5 扩展的硬性天花板：Amdahl 定律与 USL

很多人以为“加机器就能扩展”，直到撞上数学上的天花板。

**Amdahl 定律**（1967）：如果一个任务里有比例 `p` 的部分可以并行，`(1-p)` 的部分必须串行，那么用 `n` 个处理器能得到的最大加速比是：

$$ S(n) = \frac{1}{(1-p) + p/n} $$

极限情况下（n → ∞），加速比上限是 `1/(1-p)`。如果你的任务里有 5% 必须串行，那么哪怕用一百万台机器，你也只能加速 20 倍，再也上不去了。

**通用可扩展性定律 USL**（Neil Gunther，1990s）更残酷：它在 Amdahl 基础上加了一项**coherency cost（同步代价）**，模型变成：

$$ S(n) = \frac{n}{1 + \alpha(n-1) + \beta n(n-1)} $$

其中 α 是串行比例，β 是节点之间一致性/同步的开销。关键结论：

- 有 α 项 → 有加速上限（和 Amdahl 一样）
- 有 β 项 → **加速比不是单调递增的，到某个临界点后反而会下降**

这个临界点的本质是：**节点之间交换信息的代价，超过了新增节点带来的收益**。

一个具体例子：分布式事务要做两阶段提交，机器越多，协调开销越大，过了某个数就**越加越慢**。很多人踩过的坑——"我们加了一倍机器，TPS 只涨了 20%"——往往就是 β 项在起作用。

这解释了为什么 DDIA 后面章节要花大力气讲**共享无状态（shared-nothing）、分区、异步复制、最终一致性**：都是为了尽量压低 α 和 β，让系统能线性扩展。

#### 3.6 Little's Law 和为什么队列是万恶之源

排队论里一条最简单但最有用的定律——**Little's Law**：

$$ L = \lambda \times W $$

在稳态下，系统里平均排队数量 L = 到达速率 λ × 平均等待时间 W。

推论（非常反直觉）：**当系统利用率接近 100% 时，等待时间会爆炸式增长，不是线性。** 对于 M/M/1 队列模型，利用率 ρ 下的平均等待时间是 $W = \frac{1/\mu}{1-\rho}$。

- ρ = 50% 时，等待时间 = 2 倍服务时间
- ρ = 80% 时，等待时间 = 5 倍服务时间
- ρ = 95% 时，等待时间 = 20 倍服务时间
- ρ = 99% 时，等待时间 = 100 倍服务时间

这就是为什么老运维都有个口诀：**"别把 CPU 跑到 80% 以上。"** 不是浪费钱，是数学上没有安全余量。

这也决定了很多架构选择的形态：线程池要留余量、数据库连接池不能打满、Kafka 消费者必须有 lag 预警、下游重试必须带指数退避。都是在不让队列失控。

#### 3.7 常见扩展手段

- **垂直扩展（scale-up）**：升级单机配置，CPU 更强、内存更大、磁盘更快。优点是一致性语义不变；缺点是有物理天花板（最大内存、最快 CPU 是有限的）。
- **水平扩展（scale-out）**：加更多机器，把流量和数据分散出去。优点是理论无上限；缺点是引入分布式一致性问题。
- **读副本**：读扩展容易（加从库），写扩展难（主库只有一个）。
- **缓存**：减少重复访问数据库，命中率决定效益。
- **异步化**：把不必同步完成的工作后移，用消息队列吸收峰值。
- **预计算 / 物化视图**：把昂贵计算提前做掉。
- **分区（sharding）**：把原本单点承载的数据拆到多个节点。
- **边缘化**：CDN、边缘计算、本地副本，把数据搬到离用户更近的地方。

这里你可以先记住一句话：
**可靠性更关心“坏了怎么办”，可扩展性更关心“多了怎么办”。**

### 4. 可维护性：不是只指“代码可读”，而是“这个系统对人是否友好”

很多人第一次听“可维护性”，会只想到代码风格、命名规范、好不好 debug。
这些当然是其中一部分，但 DDIA 说的可维护性更宽。

它更像是在问：
**这个系统能不能被人长期、安全、低成本地理解、修改、排障和演进。**

DDIA 在这里很强调“人”的因素。
因为系统不是建好就完了，它要持续被开发、上线、排障、迁移、接手。

#### 4.1 可维护性的三层意思

DDIA 把可维护性明确分成三个子目标：

1. **可运维（Operability）**
   出问题能定位，日常运行不折磨人。好的系统让运维可以专注在"非常规"的事情上。
2. **简洁性（Simplicity）**
   管理复杂度（accidental complexity），而不是本质复杂度（essential complexity）。Fred Brooks 在《No Silver Bullet》里区分过这两种。好的抽象可以消灭偶然复杂度。
3. **可演进（Evolvability）**
   新需求来了，不需要推倒重来。书里用了 "agility"（敏捷性）作为同义词。

这三条不是独立的。一个没法运维的系统，演进的成本会指数级增长（没人敢改）；一个复杂到看不懂的系统，运维成本和演进成本一起爆炸。

#### 4.2 用 DORA 指标把可维护性量化

Google DORA 团队做了多年研究，把“团队交付能力”提炼成四个指标，这四个指标很大程度就是在量化 DDIA 意义上的可维护性：

- **Deployment Frequency（部署频率）**：精英团队多次/天，低效团队几个月一次
- **Lead Time for Changes（从提交到上线的时长）**：精英团队 < 1 小时，低效 > 6 个月
- **Change Failure Rate（变更失败率）**：精英 0%–15%，低效 > 45%
- **Mean Time to Restore（故障恢复时间）**：精英 < 1 小时，低效 > 6 个月

这四个指标背后是一组相互咬合的能力：版本控制、自动化测试、CI/CD、监控可观测性、灰度、回滚。它们描述的是**"系统让人能走多快"**，这正是可维护性的工程本质。

一个反常识的观察：**部署频率越高的团队，事故恢复时间反而越短**。因为他们天天上线，回滚路径常用常新；低频部署的团队每次上线都是"大事件"，回滚路径反而年久失修。

#### 4.3 可观测性：不是 "日志 + 监控"，是三个正交维度

要做到可运维，系统必须是**可观测的（observable）**。工业界把可观测性拆成三根支柱：

- **Metrics（指标）**：数值型时间序列，适合做聚合、告警、趋势。代价：基数（cardinality）有限制，不能带太多标签。
- **Logs（日志）**：离散事件记录，适合细节排查。代价：存储和检索成本高，结构化程度参差。
- **Traces（追踪）**：一次请求在多个服务间的完整调用链。解决"这个慢请求到底慢在哪"。代价：采样率和侵入性之间要权衡。

三者解决的问题不重叠：
- 看到 P99 飙了 → Metrics 告警
- 想知道是哪些请求慢 → Traces
- 想知道慢的请求里具体发生了什么 → Logs（按 trace ID 关联）

没有这三样，排障只能靠"感觉"。这就是为什么可维护性差的系统，事故时间总比你预期的长：不是没人会修，是没人知道哪里坏了。

#### 4.4 一个反例最容易理解

想象一个订单系统：

- 逻辑分散在 12 个服务里
- 没有统一日志 ID
- 告警噪音很多，真正故障反而看不见
- 谁也说不清某个字段是从哪里写进去的
- 线上问题只能靠“熟人经验”排查

这种系统可能还能跑，也可能性能不差，但它非常不可维护。
因为团队对它没有控制感。

#### 4.5 "复杂度"是另一种隐形成本

Rich Hickey（Clojure 作者）有个很有穿透力的观察：**"complexity is complected"**——复杂是"东西被纠缠在一起"。

大量可维护性问题，本质上是这种纠缠：

- 状态和行为纠缠：业务逻辑散在 Service/Controller/DAO/Listener 各处
- 时间和空间纠缠：一个接口的返回值依赖"最近 30 秒谁调用过"
- 数据和展示纠缠：schema 字段取名叫 `show_on_mobile_v2_enabled`

DDIA 主张用**好的抽象**来解开这些纠缠——关系模型解开了"程序逻辑"和"存储布局"的纠缠，Kafka 解开了"生产者"和"消费者"的时序纠缠，Raft 解开了"日志复制"和"业务逻辑"的纠缠。

这解释了为什么书里反复出现**"把 X 抽象成 Y"**的讨论：这不是炫技，是在降低可维护性成本。

### 5. 三者最容易混淆的地方，必须掰开看

下面是最容易搞混的几组边界。

#### 5.1 可靠性 vs 可扩展性

场景 A：一台机器坏了，服务不可用。
这是可靠性问题。

场景 B：双十一流量翻 20 倍，系统响应时间从 50ms 飙到 5 秒。
这是可扩展性问题。

两者都可能导致“系统不好用”，但原因不同：

- 可靠性关注故障、异常、容错
- 可扩展性关注增长、负载、瓶颈

但两者也有交叉地带：**规模上来了之后，小概率故障变成高频故障**。单机失效概率 1%/年，10000 台机器集群每天都有 0.3 台坏。规模本身就是可靠性挑战。

#### 5.2 可扩展性 vs 可维护性

场景 A：新活动上线，数据库扛不住写入高峰。
这是可扩展性问题。

场景 B：新活动规则要改，但订单状态机写得过于耦合，谁都不敢动。
这是可维护性问题。

所以：

- “量大了扛不住”是扩展问题
- “要改时改不动”是维护问题

这两个在你刚才的回答里有一点重叠，现在你要故意把它们分开。

#### 5.3 可靠性 vs 可维护性

场景 A：半夜系统挂了，告警响了，有人接到电话十分钟内恢复了。
**看起来**是可靠性出问题，但恢复很快——其实是**可维护性救了可靠性**。

场景 B：系统半年没出过故障，但一旦出故障，恢复要 4 小时。
这是典型的"可靠性得分虚高、可维护性欠债"——统计平均是好的，P99 损失是灾难的。

**记住这个杠杆：MTTR 主要由可维护性决定**。你的监控、文档、回滚能力、灰度能力，最终在事故时决定你能多快恢复。

### 6. 用电商订单系统把三者放到同一个场景里

我们拿“订单创建”这条链路做例子：

用户点击下单后，系统要做：

- 校验商品与库存
- 创建订单
- 锁定库存
- 调用支付
- 发送消息给商家、用户、物流系统

如果从三个维度看，同一条链路会冒出不同性质的问题：

#### 6.1 可靠性问题

- 订单刚写入，数据库主机宕机，数据会不会丢？（依赖同步复制与持久化语义）
- 支付回调重复到达，会不会重复扣库存？（依赖幂等设计）
- MQ 短暂不可用，会不会导致订单状态永远卡住？（依赖死信队列和人工兜底）
- 下游支付服务超时时，订单是"成功"还是"失败"？（这是著名的**不确定结果（indeterminate outcome）**问题，DDIA 第 8 章会讲透）

#### 6.2 可扩展性问题

- 大促时每秒 10 万下单请求，单库单表还能扛住吗？（写扩展问题）
- 某个爆款商品访问异常集中，会不会打爆单个分区？（**热点 key** 问题，第 6 章分区）
- 查询订单列表的人数暴涨，读流量会不会把主库拖死？（读写分离的必要性）
- 订单表 1 亿行后，二级索引维护代价是否变得不可接受？（第 3 章索引成本）

#### 6.3 可维护性问题

- 一个订单状态改动，要同时改 6 个服务，是否太脆弱？（耦合、契约演化，第 4 章）
- 出现“用户已付款但商家未接单”时，排查链路是否清晰？（端到端 trace）
- 新人接手系统时，能不能在一周内建立基本心智模型？（概念简洁性）
- 订单状态机上能不能加一个新状态，不影响历史订单？（schema 演化，第 4 章）

你会发现：
同一个系统，不同问题分属不同维度。
这就是为什么 DDIA 要先教你“分类看问题”。

### 7. 三个目标之间的拉扯：没有免费的午餐

真正的工程难点在这里：
很多设计不是同时提升三者，而是提高一个，牺牲另一个。

比如：

- 加更多副本，可靠性和读扩展性可能提高了，但一致性和维护复杂度也上来了
- 拆更多服务，也许局部扩展更灵活了，但可维护性和排障复杂度可能下降
- 强事务能提升正确性与部分可靠性，但吞吐和扩展能力可能受影响
- 增加观测埋点能提升可维护性，但会带来性能开销和存储成本
- 上混沌工程能暴露可靠性弱点，但**增加了上线时的团队心理负担**（这也是维护成本）

所以 DDIA 不会给你“万能答案”，它会训练你做权衡。

你以后每学到一种设计，都可以问：

1. 它主要在优化哪一个目标？
2. 它牺牲了哪一个目标？
3. 这个牺牲在当前业务里值不值得？

#### 7.1 这些拉扯在后续章节的对应关系

先给一个预告表，让你读后续章节时看到取舍能立即归类：

| 设计决策 | 升级了 | 代价是 | 相关章节 |
|---|---|---|---|
| 异步复制 | 可用性、写吞吐 | 一致性（读到旧数据） | 第 5 章 |
| 同步复制 | 一致性、持久性 | 延迟、可用性 | 第 5 章 |
| 哈希分区 | 负载均衡 | 范围查询难做 | 第 6 章 |
| 范围分区 | 范围查询快 | 易出热点 | 第 6 章 |
| 强隔离事务 | 正确性 | 吞吐、死锁 | 第 7 章 |
| 弱隔离（RC、SI） | 吞吐 | 写偏斜、幻读 | 第 7 章 |
| 线性化一致性 | 程序员心智简单 | 延迟、可用性 | 第 9 章 |
| 最终一致性 | 可用性、吞吐 | 冲突解决复杂 | 第 5、9 章 |
| 批处理 | 幂等、可回放 | 延迟高 | 第 10 章 |
| 流处理 | 延迟低 | 恰好一次难 | 第 11 章 |

你会发现：**右侧"代价"栏，几乎就是下一章要讲的内容**。这就是全书的节奏。

### 8. 先记一个判断口诀

为了帮你快速建立直觉，你可以先记这个不严谨但很好用的口诀：

- 出故障了还能不能顶住：看可靠性
- 量上来了还能不能撑住：看可扩展性
- 人来维护时还能不能管住：看可维护性

再压缩成一句更短的话：

**坏了怎么办，看可靠性；多了怎么办，看可扩展性；改了怎么办，看可维护性。**

这句话你后面会反复用到。

#### 8.1 一个工程判别题训练

下面几个问题，你试着在读下一章前自己归类一下（答案多分散在全书后面）：

1. 数据库主从同步延迟 500ms，用户刚改完资料刷新看不到新值。
2. 我们生产流量从 1000 QPS 涨到了 50 万 QPS，Redis 单节点开始 CPU 打满。
3. 上线一个字段增加需求，要同时改 ORM、缓存协议、消息体、下游系统，3 天没动完。
4. 凌晨 3 点告警响了，但没人知道哪条链路最先出问题，查了 2 小时才定位。
5. 交易高峰时某个热门商家的订单全部打到同一个分片，该分片 CPU 100%。
6. 我们做压测，TPS 在 1000 时是线性涨的，到 1500 之后就再也上不去了。
7. 故障切换到从库后，有 30 秒的写入丢了，因为复制是异步的。

（分别对应：可靠性延伸到一致性 / 扩展性 / 维护性 / 维护性 / 扩展性的热点 / 扩展性的 USL 天花板 / 可靠性和一致性的取舍）

## 小结（5 条关键点）

- **可靠性**关注系统在硬件故障、软件 bug、人为错误下的生存能力；可以用 MTBF/MTTR/SLO/错误预算等指标量化，而 MTTR 往往比 MTBF 更值得优化。
- **可扩展性**关注负载增长下的性能保持能力；描述负载要找对 load parameter（不是单一 QPS），描述性能要用百分位而不是平均值；尾延迟在扇出系统里会被显著放大。
- **可扩展性有数学上的硬天花板**：Amdahl 定律限制了并行加速比上限，USL 进一步揭示了"加机器反而更慢"的临界点；Little's Law 说明了为什么利用率超过 80% 就会爆炸。
- **可维护性**包含可运维、简洁性、可演进三层；可以用 DORA 四指标量化；三支柱可观测性（metrics/logs/traces）是事故时压缩 MTTR 的关键。
- 三者不是独立目标，经常互相拉扯；DDIA 后续章节每个设计决策的"代价"栏，几乎就是下一章的主题——这是全书的基本节奏。
