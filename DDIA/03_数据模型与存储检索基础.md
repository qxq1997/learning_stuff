# DDIA - 第 3 课：数据模型与存储检索基础

## 学习目标（本节结束后你能做到什么）

- 理解“数据模型”到底是什么，知道它不是语法细节，而是系统如何组织现实世界信息的方式。
- 能区分关系型模型、文档模型、图模型各自更擅长解决什么问题。
- 建立对“存储引擎”最基础的直觉，知道为什么同样是数据库，底层组织数据的方式会不一样。
- 初步理解 B-Tree 和 LSM Tree 这两条主流思路分别更适合什么读写模式。
- 形成一个关键认识：应用层看到的“表、文档、索引、查询”，背后都对应具体的存储与检索代价。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 DDIA 在讲完总目标后，马上转到“数据模型”

很多人学系统设计时，一上来就想问：

- 用 MySQL 还是 MongoDB？
- 要不要上 Elasticsearch？
- Redis 到底该放哪里？

但 DDIA 不会先让你背“技术选型口诀”，而是先问更底层的问题：

**你到底在表示什么数据？这些数据之间是什么关系？你最常做什么样的查询和更新？**

这就是“数据模型”的核心。

因为数据库不是一个抽象黑箱。  
你用什么模型去描述数据，会直接影响：

- 数据怎么存
- 查询怎么写
- 更新是否方便
- 一致性边界怎么划
- 后续扩展是否痛苦

所以“数据模型”不是一个纯学术名词，它决定了你后面很多工程选择。

### 2. 什么是数据模型

最简单地说，数据模型就是：  
**你用什么结构去表达现实世界里的对象、属性和关系。**

举个电商例子。现实世界里有：

- 用户
- 商品
- 订单
- 店铺
- 评论

这些对象之间还有各种关系：

- 一个用户可以下多个订单
- 一个订单包含多个商品
- 一个商品属于某个店铺
- 用户和商品之间可能有“浏览过”“收藏过”“买过”的关系

你怎么把这些信息放进系统里？  
这就需要数据模型。

数据模型不是现实世界本身，而是现实世界在系统里的“翻译版本”。

### 3. 三种最重要的数据模型：关系型、文档型、图模型

DDIA 这一部分很重要，因为它会帮你建立一个认识：  
**没有一种模型在所有场景下都最好。**

你需要根据数据形态和访问模式来选。

#### 3.1 关系型模型：表格化、结构清晰、适合强约束

关系型数据库最典型的表示方式是“表”。

比如订单系统里可以有这些表：

- `users`
- `orders`
- `order_items`
- `products`

每张表里是若干行记录，每行记录代表一个实体实例，每列代表一个字段。

例如：

| order_id | user_id | status | total_amount |
| --- | --- | --- | --- |
| 1001 | 88 | paid | 199.00 |

关系型模型的强项在于：

- 结构清晰
- 约束明确
- 适合事务处理
- 可以通过 join 表达复杂关联查询

这很适合订单、支付、账户、库存这类“结构相对稳定、正确性要求高”的数据。

但它也有代价：

- 表结构变更有成本
- 多表 join 在大规模场景下可能很重
- 如果你的数据天然是嵌套结构，硬拆成很多表可能会很绕

#### 3.2 文档模型：更贴近对象结构，适合聚合读写

文档模型常见于 MongoDB 这类系统。  
它把一条记录看成一个文档，常常是 JSON 风格的嵌套结构。

比如一个订单，可以直接长这样：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "status": "paid",
  "items": [
    {"product_id": 10, "name": "耳机", "count": 1, "price": 99},
    {"product_id": 20, "name": "鼠标", "count": 1, "price": 100}
  ],
  "address": {
    "city": "上海",
    "detail": "xx 路 xx 号"
  }
}
```

这种模型的直观优势是：

- 更接近应用程序里的对象结构
- 一个聚合对象可以整体读写
- 某些场景下不需要频繁 join

它常适合：

- 内容管理系统
- 用户画像
- 商品详情页
- 配置类数据
- 数据结构天然层级化的业务

但它也有代价：

- 文档之间复杂关联不如关系型自然
- 数据冗余更常见
- 一旦很多地方都嵌套复制同样信息，更新会麻烦

所以你可以先记：  
**关系型更强调规范化与关联；文档型更强调聚合对象的整体读写。**

#### 3.3 图模型：关系本身就是重点

有些业务的重点不在“对象字段很多”，而在“对象之间关系很复杂”。  
例如：

- 社交网络中的关注关系
- 推荐系统中的“用户看过商品”“商品属于类目”“用户喜欢品牌”
- 风控系统中的设备、账号、银行卡、IP 之间关系

这时候图模型会更自然。

图模型通常由两部分组成：

- 点：实体，如用户、商品、设备
- 边：关系，如关注、购买、登录、关联

如果一个问题需要频繁地做“多跳关系查询”，例如：

- 某账号关联过哪些设备？
- 这些设备还关联过哪些账号？
- 某人朋友的朋友是否买过这本书？

图模型会很强。

但图模型并不是通用替代品。  
如果你的业务主要是订单写入、支付扣款、库存事务，图数据库往往不是第一选择。

### 4. 选模型时不要问“先进不先进”，要问“查询模式是什么”

这是非常重要的一点。

很多人选型时会问：

- 哪个数据库性能更高？
- 哪个模型更现代？

这些问法太粗。

更好的问法是：

- 你的主要读写模式是什么？
- 数据是一条一条独立查询，还是经常聚合读取？
- 关联关系多不多？
- 数据结构变化频繁吗？
- 正确性约束是不是很强？

例如：

- 订单支付：通常更偏关系型
- 商品详情聚合展示：文档模型会更顺手
- 社交关系或风控关系图谱：图模型更自然

所以模型选择不是“宗教战争”，而是“查询模式匹配”。

### 5. 光有模型还不够，系统还要回答：数据在磁盘上怎么放，查的时候怎么找

前面讲的是“逻辑视角”。

- 表
- 文档
- 图

这些更像是你在应用层看到的数据组织方式。

但数据库还要做一件更底层的事：  
**把数据真正存起来，并且在你查询时尽快找到它。**

这就进入“存储与检索”的问题。

最简单的直觉是：

- 存数据时，希望写得快
- 查数据时，希望找得快

问题是，这两件事常常会互相拉扯。  
因为“写入方便”的结构，不一定“查询方便”；“查询高效”的结构，也可能让写入更贵。

### 6. 从最朴素的存储开始理解：追加写日志

先别急着看复杂树结构。  
先想一个最简单的方法：每次写入都直接追加到文件末尾。

比如：

- `set x = 3`
- `set y = 8`
- `set x = 5`

磁盘上就不断追加：

```text
x=3
y=8
x=5
```

这个方法的优点很明显：

- 写入很简单
- 顺序追加通常很快
- 不需要每次都改动原位置

但问题也很明显：

- 如果你现在问 `x` 是多少，要从头或某种辅助结构里找最后一次写入
- 数据会越来越多
- 老版本记录会堆积

所以很多系统会基于“追加写”再配合索引、压缩、合并等机制。

这类思路对理解后面的 LSM Tree 很有帮助。

### 7. 为什么需要索引

如果没有索引，数据库查数据就像你在一大堆纸质档案里一页页翻。

索引本质上就是：  
**为了更快找到数据，而额外维护的一套辅助结构。**

代价是：

- 查询更快了
- 但写入时要顺便维护索引，所以写可能更慢，占用更多空间

这又是一个典型 trade-off。

举个最简单例子：

如果订单表按 `order_id` 建了索引，那么你查某个订单时就不用扫描整张表。  
但每次插入新订单时，这个索引也要同步更新。

所以“加索引”不是零成本优化，而是“拿写入代价和空间换查询效率”。

### 8. 两条主流底层思路：B-Tree 和 LSM Tree

你现在不需要记特别细的实现，只要先抓住直觉。

#### 8.1 B-Tree：更像一本排好序、可以快速翻到位置的字典

B-Tree 家族是很多传统数据库常见的索引/存储结构。  
你可以把它想成一本非常厚但排好序的字典。

它的核心思路是：

- 数据或索引按有序方式组织
- 查找时能快速缩小范围
- 更新时通常会在树的某些节点上做修改

它比较适合：

- 范围查询
- 随机读取
- 读写都比较均衡的场景

但如果写入特别猛，频繁在磁盘上改结构，也会有成本。

#### 8.2 LSM Tree：更像“先快速记草稿，再后台慢慢整理”

LSM Tree 的直觉可以理解成：

1. 新写入先放到内存中的有序结构
2. 达到一定规模后批量刷到磁盘
3. 磁盘上可以有多层有序文件
4. 后台不断做合并整理（compaction）

这条路线非常适合写多场景，因为它把很多“零散小写”变成“顺序批量写”。

你可以把它想成办公室记账：

- 收到单据先记在手边的小本子上
- 攒一批后再统一整理进正式账本

LSM Tree 常见于：

- 写入量很大
- 顺序写优势明显
- 愿意接受后台合并成本

但代价可能包括：

- 读路径可能更复杂，因为数据可能散在多层文件里
- 后台 compaction 会消耗资源
- 某些情况下读放大、写放大、空间放大会成为问题

你现在只要记住一个粗粒度结论：

- B-Tree：传统、均衡、查询友好
- LSM Tree：写入友好，但后台整理和读取代价要管

这个结论不够严谨，但对入门很有用。

### 9. 为什么同样是“数据库”，有的适合事务，有的适合日志写入，有的适合搜索

因为它们优化目标不同。

比如：

- MySQL 更强调事务、约束、通用查询能力
- Cassandra/HBase 这类系统更偏大规模写入和分布式扩展
- Elasticsearch 更偏全文检索和倒排索引

它们处理的不是同一种“查找问题”。

这也是 DDIA 很重要的一点：  
不要把“存数据”看成一件单一的事。  
不同系统之所以不同，是因为它们在底层为不同工作负载做了不同优化。

### 10. 用一个订单系统把这一课串起来

假设你在做电商订单系统。

你会遇到不同性质的数据：

- 订单主记录：结构清晰、事务要求高
- 商品详情：字段多、结构常变、适合聚合展示
- 用户关系推荐：关系网络复杂
- 搜索商品标题：需要全文检索

如果你硬要求“一种模型、一个库搞定一切”，往往会越来越别扭。

更常见的现实是：

- 核心交易数据在关系型数据库
- 某些聚合展示或配置数据在文档型存储
- 搜索单独走搜索引擎
- 某些关系分析走图或离线系统

这不是“技术栈乱”，而是因为不同数据问题需要不同表示和检索方式。

当然，这也会带来系统复杂度。  
所以真正的工程能力不是看到多系统就兴奋，而是知道：

- 为什么要拆
- 拆了后数据怎么同步
- 哪个才是事实来源

这些问题在后面复制、数据流、一致性章节会继续展开。

### 11. 这一课真正要建立的底层意识

你现在最需要建立的意识是下面这句：

**应用层看到的数据结构，背后总对应具体的存储组织和检索代价。**

也就是说：

- 不是“我想怎么查就怎么查”，底层一定要为这种查询付代价
- 不是“我想怎么存就怎么存”，后面检索和更新也要承担后果

所以好的系统设计，从来不是先拍脑袋定技术，而是先弄清：

1. 数据长什么样？
2. 最常见的读写模式是什么？
3. 哪些查询最关键？
4. 哪些正确性约束最重要？
5. 我愿意用什么代价换什么能力？

这就是这一课和后面所有课之间的连接点。

## 小结（3-5 条关键点）

- 数据模型是系统表达现实世界对象与关系的方式，它会直接影响后续存储、查询、更新和扩展成本。
- 关系型、文档型、图模型各有擅长场景，关键不在“谁更先进”，而在“谁更匹配你的数据形态和访问模式”。
- 数据库存储不仅是“把数据放进去”，还包括如何高效检索，因此索引是核心辅助结构。
- B-Tree 更偏向均衡读写与有序查询；LSM Tree 更偏向高写入吞吐和顺序写，但会引入后台合并与读路径复杂度。
- 技术选型本质上是在不同数据模型、查询模式和底层代价之间做权衡。
