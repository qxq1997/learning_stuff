# DDIA - 第 3 课：数据模型与存储检索基础

## 学习目标（本节结束后你能做到什么）

- 理解"数据模型"到底是什么，知道它不是语法细节，而是系统如何组织现实世界信息的方式。
- 能区分关系型模型、文档模型、图模型各自更擅长解决什么问题，并理解它们背后的历史演化。
- 知道"对象-关系阻抗失配"（impedance mismatch）究竟指什么，为什么它不是嘴上的痛点而是决定模型选择的核心。
- 用"多对多关系"作为切分点，判断什么时候文档模型会把你反噬。
- 理解存储引擎的**物理代价模型**：磁盘顺序写 vs 随机写的数量级差距，以及为什么这一个差距催生了 LSM Tree。
- 真正理解 B-Tree 和 LSM Tree 的内部结构（页、WAL、SSTable、memtable、compaction），而不是停留在"一个像字典、一个像草稿"的比喻。
- 掌握**写放大、读放大、空间放大**三角取舍，知道为什么所有存储引擎都在这三者之间做折中。
- 理解布隆过滤器在 LSM 里扮演的关键角色及其数学（假阳率与 bit 数的关系）。
- 区分 OLTP 和 OLAP 的根本差异，知道为什么列存与行存是两种完全不同的物种。
- 初步理解全文倒排索引、二级索引（clustered vs non-clustered）的构造和代价。
- 形成一个关键认识：应用层看到的"表、文档、索引、查询"，背后都对应具体的磁盘 I/O 模式与物理代价。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 DDIA 在讲完总目标后，马上转到"数据模型"

很多人学系统设计时，一上来就想问：

- 用 MySQL 还是 MongoDB？
- 要不要上 Elasticsearch？
- Redis 到底该放哪里？

但 DDIA 不会先让你背"技术选型口诀"，而是先问更底层的问题：

**你到底在表示什么数据？这些数据之间是什么关系？你最常做什么样的查询和更新？**

这就是"数据模型"的核心。

因为数据库不是一个抽象黑箱。
你用什么模型去描述数据，会直接影响：

- 数据怎么存
- 查询怎么写
- 更新是否方便
- 一致性边界怎么划
- 后续扩展是否痛苦

所以"数据模型"不是一个纯学术名词，它决定了你后面很多工程选择。

#### 1.1 数据模型是一层层叠加的抽象

Kleppmann 在本章开篇做了一件容易被忽视的事：他把"数据模型"定义为一组**层层嵌套的抽象**。

从下往上看：

1. **硬件层**：磁盘块、字节、电压、磁头
2. **存储引擎层**：页、记录、B-Tree 节点、SSTable 行
3. **数据模型层**：表、行、文档、节点、边
4. **应用模型层**：用户、订单、商品、关注关系
5. **概念层**：业务里的"客户"到底指什么

每一层的存在目的都是：**让上一层不用关心下一层的细节**。

这件事看起来抽象，但它解释了为什么数据模型选型的后果这么深远：**你选的是一个抽象接口，而这个接口限定了上层能表达什么、下层能优化什么。** 选了关系模型，你获得了声明式查询和优化器空间，但也承诺了"我的数据能被拉成行列"；选了文档模型，你获得了嵌套表达的自由，但放弃了某些跨文档优化机会。

这一课关注的是"数据模型层"和"存储引擎层"。后面章节的复制、分区、事务、一致性，都会叠在这两层之上。

### 2. 什么是数据模型

最简单地说，数据模型就是：
**你用什么结构去表达现实世界里的对象、属性和关系。**

举个电商例子。现实世界里有：

- 用户
- 商品
- 订单
- 店铺
- 评论

这些对象之间还有各种关系：

- 一个用户可以下多个订单
- 一个订单包含多个商品
- 一个商品属于某个店铺
- 用户和商品之间可能有"浏览过""收藏过""买过"的关系

你怎么把这些信息放进系统里？
这就需要数据模型。

数据模型不是现实世界本身，而是现实世界在系统里的"翻译版本"。

#### 2.1 阻抗失配：为什么有文档数据库这件事

2006 年前后，Martin Fowler 写过一系列关于 **"object-relational impedance mismatch"**（对象-关系阻抗失配）的文章。这个词是整个 NoSQL 运动的思想起点之一。

问题是这样的：面向对象编程里，一个业务对象长得像嵌套结构：

```java
class Order {
  long id;
  User user;                   // 嵌套对象
  List<OrderItem> items;       // 嵌套集合
  Address shippingAddress;     // 嵌套对象
  List<StatusChange> history;  // 嵌套集合
}
```

但关系数据库把它硬拆成 `orders / order_items / users / addresses / status_changes` 五张表，然后每次读写都要靠 ORM 把对象和表来回翻译。这个翻译层：

- 写代码累（配置复杂、JPA/Hibernate 那些注解）
- 读路径慢（N+1 查询、懒加载陷阱）
- 容易错（事务边界、脏检测、缓存一致性）
- 对象身份和行身份不对齐

MongoDB 2009 年起势的卖点就是**"把这个翻译层干掉，对象长什么样就存什么样"**。这是文档模型最真实的价值主张。

但故事没完。后来大家发现：**只要你的数据有真正的多对多关系，文档模型就会把你拉回到同样的翻译地狱**，只是换成了手动 JOIN。所以今天的主流观点是：**模型选择不是宗教，而是根据"数据关系是不是聚合形的"来判断**。这个判断点我们 3.4 节会展开。

### 3. 三种最重要的数据模型：关系型、文档型、图模型

DDIA 这一部分很重要，因为它会帮你建立一个认识：
**没有一种模型在所有场景下都最好。**

你需要根据数据形态和访问模式来选。

#### 3.1 关系型模型：表格化、结构清晰、适合强约束

关系型数据库最典型的表示方式是"表"。

比如订单系统里可以有这些表：

- `users`
- `orders`
- `order_items`
- `products`

每张表里是若干行记录，每行记录代表一个实体实例，每列代表一个字段。

例如：

| order_id | user_id | status | total_amount |
| --- | --- | --- | --- |
| 1001 | 88 | paid | 199.00 |

关系型模型的强项在于：

- 结构清晰
- 约束明确（外键、非空、唯一、check）
- 适合事务处理
- 可以通过 JOIN 表达复杂关联查询
- **声明式查询语言**：你写 `WHERE x=1`，优化器自己决定用哪个索引、按什么顺序执行

这很适合订单、支付、账户、库存这类"结构相对稳定、正确性要求高"的数据。

但它也有代价：

- 表结构变更有成本（大表 `ALTER` 要小心锁）
- 多表 JOIN 在大规模场景下可能很重
- 如果你的数据天然是嵌套结构，硬拆成很多表可能会很绕

**历史背景补一句**：关系模型不是自古就有的。它是 Edgar Codd 在 1970 年《A Relational Model of Data for Large Shared Data Banks》里提出的。它当年挑战的是更早的**层次模型**（IBM IMS，1960s）和**网状模型**（CODASYL，1960s–70s）。这两个模型最大的问题是：**查询路径是手动编码的**，你要沿着指针一层层 navigate。Codd 的革命性洞察是把"导航"升级为"声明"——你说你要什么，让查询优化器决定怎么找。

这个洞察今天听起来平平无奇，但它是数据库工业最重要的一次范式转移。**所有后来的 NoSQL 系统在大数据兴起后，又被迫把 SQL 查询语言加回去**（Hive、Presto、Spark SQL、CQL），就是因为声明式的威力太大，人类工程师离开它就活不下去。

#### 3.2 文档模型：更贴近对象结构，适合聚合读写

文档模型常见于 MongoDB 这类系统。
它把一条记录看成一个文档，常常是 JSON 风格的嵌套结构。

比如一个订单，可以直接长这样：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "status": "paid",
  "items": [
    {"product_id": 10, "name": "耳机", "count": 1, "price": 99},
    {"product_id": 20, "name": "鼠标", "count": 1, "price": 100}
  ],
  "address": {
    "city": "上海",
    "detail": "xx 路 xx 号"
  }
}
```

这种模型的直观优势是：

- 更接近应用程序里的对象结构
- 一个聚合对象可以整体读写（**存储局部性**，data locality）
- 某些场景下不需要频繁 JOIN
- Schema 灵活，字段可以按需增减

它常适合：

- 内容管理系统
- 用户画像
- 商品详情页
- 配置类数据
- 数据结构天然层级化的业务

##### 3.2.1 Schema-on-read vs Schema-on-write

文档模型常被说成"无 schema"（schemaless），这句话是错的。正确的说法是 **schema-on-read**（读时校验）vs **schema-on-write**（写时校验）：

- 关系库：写入时必须符合 schema，不符合直接报错。好处是数据质量天然有保证；代价是 `ALTER TABLE` 有物理成本。
- 文档库：写入时不强制校验，schema 隐含在读取这份文档的代码里。好处是演化灵活；代价是**同一个集合里可能混着新老格式的文档**，读取方必须手动处理各种版本。

这个差别在真实项目里会变成一条明显的分水岭：

- 新功能迭代快、字段频繁变的业务（MVP 阶段、用户画像）：schema-on-read 更省事
- 数据要被很多下游系统消费、要做长期分析的业务：schema-on-write 才能避免下游一个个踩坑

##### 3.2.2 局部性（locality）：文档模型的底牌

把订单和订单项放一个文档里，不只是写法漂亮，还有一个物理好处：**它们在磁盘上也挨着**。

读一次订单详情，磁盘 I/O 可能只是一次随机寻址 + 顺序读；如果是关系库拆成三张表，你可能需要三次随机 I/O 或一个 JOIN。

这个优势非常真实，但也**只在"整体读取"时成立**。如果你只想更新订单的某一个字段，很多文档库会重写整个文档，反而更贵。所以局部性在"读远多于写、且读总是整体读"时是最强的。

##### 3.2.3 文档模型的代价

- 文档之间复杂关联不如关系型自然（多对多是死穴，见 3.4）
- 数据冗余更常见（用户名复制进每份订单里）
- 一旦很多地方都嵌套复制同样信息，更新会麻烦（用户改名 → 遍历更新）
- 不支持跨文档原子性事务（老版 MongoDB 只有单文档原子性，4.0 之后才有多文档事务，且代价不小）

所以你可以先记：
**关系型更强调规范化与关联；文档型更强调聚合对象的整体读写。**

#### 3.3 图模型：关系本身就是重点

有些业务的重点不在"对象字段很多"，而在"对象之间关系很复杂"。
例如：

- 社交网络中的关注关系
- 推荐系统中的"用户看过商品""商品属于类目""用户喜欢品牌"
- 风控系统中的设备、账号、银行卡、IP 之间关系
- 知识图谱
- 基因/蛋白质相互作用

这时候图模型会更自然。

图模型通常由两部分组成：

- 点（vertex / node）：实体，如用户、商品、设备
- 边（edge）：关系，如关注、购买、登录、关联

##### 3.3.1 属性图 vs 三元组（triple store）

工业界有两套主流的图模型：

**属性图（Property Graph）**：Neo4j、JanusGraph、Nebula Graph 是代表。每个点和每条边都可以带任意键值对的属性。查询语言最流行的是 **Cypher**（Neo4j）和 **Gremlin**。

```cypher
-- 找"某用户关注的人关注的人"（经典二跳）
MATCH (me:User {id: 88})-[:FOLLOWS]->()-[:FOLLOWS]->(fof)
RETURN DISTINCT fof
```

**三元组（Triple Store / RDF）**：每条信息表达为 `(主语, 谓语, 宾语)` 三元组。背后是语义网（Semantic Web）和 W3C 标准。查询语言是 **SPARQL**。

```sparql
SELECT ?fof WHERE {
  :user_88 :follows ?friend .
  ?friend :follows ?fof .
}
```

两者的差别不仅是语法。RDF 的设计目的是**跨组织的数据互通**（链接开放数据 Linked Open Data），每个 URI 全球唯一，天然适合做知识图谱与互联网规模的推理；属性图更工程化，对单库内的复杂关系查询更直接。

##### 3.3.2 为什么图模型不是通用银弹

图模型的杀手锏是 **"任意跳数的多对多遍历"**——这在关系库里要写一堆递归 CTE 或存储过程，图库一句话搞定。

但代价：

- 图数据库的事务、复制、分区生态远不如 MySQL 成熟
- 对"宽表过滤聚合"（OLAP 式查询）并不擅长
- 学习曲线高，团队心智模型要重建
- 很多"看起来像图"的问题其实用关系表 + 递归 CTE 足以解决

一个经验判别：**如果你的核心查询里会问"X 和 Y 之间的最短路径"或"X 关联了多少层的 Y"，那是图；如果你只是"X 关联了 Y 的属性"，两层关联，用关系库的 JOIN 就够。**

##### 3.3.3 Datalog：图模型的理论底色

DDIA 还特别提到了 Datalog——一种 1970 年代诞生的逻辑式查询语言，比 SQL 更早。它的基本单位也是三元组，支持**递归规则**，这使得它在表达图查询时非常优雅：

```prolog
ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).
```

这两行就定义了"祖先"这个概念。Cascalog、LogicBlox、部分现代数据库（如 Datomic）底层用的都是 Datalog 思路。了解它的价值在于：**你能看清 Cypher/SPARQL 背后的通用结构，而不是觉得它们是凭空长出来的语法**。

### 4. 多对多关系：选型的真正分界线

这是本章最容易被表层"关系 vs 文档"口水战掩盖的真正洞见。

#### 4.1 三类关系

在数据建模里，你遇到的关系只有三类：

- **一对一**（User → Profile）
- **一对多**（User → Orders）
- **多对多**（User ↔ Product，经"购买""浏览""收藏"等关系连接）

文档模型对一对多非常友好——直接嵌套。但它**对多对多极其不友好**。

#### 4.2 一个被反复重演的故事：LinkedIn 的职业经历

假设你在存一份简历：

```json
{
  "name": "Bill Gates",
  "positions": [
    {"company": "Microsoft", "title": "CEO"},
    {"company": "Gates Foundation", "title": "Co-Chair"}
  ]
}
```

看起来很合理。但接下来：

- 产品经理说："我希望用户能从公司页看到所有在这家公司工作过的人。"
- 然后："公司名规范化——同一家公司不同拼写要统一。"
- 再："公司本身有介绍、logo、产业分类，需要独立管理。"

这时候你要么：

1. 把 `company` 从字符串改成引用（`company_id`），然后手动 JOIN——**你已经在用关系模型了，只是 DB 不帮你**
2. 把公司数据复制进每份简历——**更新时要遍历所有简历**

这是文档模型设计里最经典的反模式。**一旦多对多进来，文档模型就开始漏水**。

#### 4.3 规范化与反规范化：永恒的折中

关系模型的标准做法是 **规范化（normalization）**，按第三范式（3NF）把重复数据抽离，用外键引用。优势：

- 单一真相源，更新只改一处
- 节省空间
- 数据一致性由 DB 保证

代价：

- 查询要 JOIN，性能代价跟数据量和 JOIN 次数挂钩
- 应用代码更啰嗦

**反规范化（denormalization）** 是主动违反范式、冗余存储，目的是把 JOIN 消灭掉换性能。这在 OLAP、缓存、读副本里非常常见。**文档模型在某种意义上就是"语言级别鼓励反规范化"**。

真正的工程判断是：

- 一对一、一对多、且写少读多 → 可以反规范化 / 文档化
- 多对多、需要频繁更新"共享实体" → 规范化 / 关系化
- 业务还没定型、字段经常加 → schema-on-read 更省事

### 5. 数据局部性与聚合根

上面提到"局部性"时顺带有一个跨领域的联系：**DDD（领域驱动设计）里的"聚合根"概念，和文档模型的边界几乎一一对应**。

聚合根的意思是：**某些对象是紧密绑在一起、生命周期与事务边界必须一致的**。比如订单+订单项+收货地址，就是一个聚合。

- 存储选择上，聚合根 → 一个文档
- 事务边界上，聚合根 → 一次 commit
- 对外 API 上，聚合根 → 一个资源

这个视角帮你做模型选择时有一个明确标尺：**先画清楚聚合边界，再判断聚合之间的关系密度。聚合内是嵌套（文档/JSON 列），聚合之间是引用（外键/关联表）。**

这也解释了为什么 Postgres 最近几年"复兴"——它同时支持关系建模（表）和嵌套建模（JSONB 列 + GIN 索引），**让你在同一个库里把聚合内和聚合间用合适的工具分别处理**。现代很多团队的默认选择就是 Postgres + JSONB，而不是纯关系或纯文档。

### 6. 光有模型还不够，系统还要回答：数据在磁盘上怎么放，查的时候怎么找

前面讲的是"逻辑视角"。

- 表
- 文档
- 图

这些更像是你在应用层看到的数据组织方式。

但数据库还要做一件更底层的事：
**把数据真正存起来，并且在你查询时尽快找到它。**

这就进入"存储与检索"的问题。

最简单的直觉是：

- 存数据时，希望写得快
- 查数据时，希望找得快

问题是，这两件事常常会互相拉扯。
因为"写入方便"的结构，不一定"查询方便"；"查询高效"的结构，也可能让写入更贵。

#### 6.1 物理代价模型：为什么所有存储设计都绕着"磁盘物理"转

要讨论存储引擎，先记住几个数量级。这些数字决定了一切。

**机械硬盘（HDD）** 的物理特性：

- 顺序读写：100 MB/s 左右
- 随机寻址：~10 ms（7200 RPM 磁头物理移动 + 旋转等待）
- 顺序与随机 I/O 带宽比：**近 100 倍**

**固态硬盘（SSD / NVMe）** 改善了这一点，但并未消除：

- 顺序读写：几 GB/s
- 随机读：µs 级别，和顺序差距小
- 随机写：比顺序写慢，且有**写放大**（page erase 前要整 block 擦除）

**内存（DRAM）**：

- 带宽：几十 GB/s
- 随机访问：~100 ns
- 和磁盘差 **10^4–10^5** 倍

**CPU L1 缓存**：~1 ns，又比 DRAM 快 100 倍。

这些数字决定了一切存储设计的底层朝向：

- **磁盘上能顺序写就不要随机写** —— LSM Tree 的起点
- **能把热数据放内存就放内存** —— Buffer Pool、Memcache、Redis 的起点
- **批量化、向量化、列式存储** —— 为了让 CPU 缓存命中率最高

一张 Jeff Dean 式的"每个工程师都该记的数字"表：

| 操作 | 大致耗时 |
|---|---|
| L1 cache 访问 | 0.5 ns |
| 分支预测失败 | 5 ns |
| L2 cache 访问 | 7 ns |
| Mutex 加解锁 | 25 ns |
| 主存访问 | 100 ns |
| 压缩 1KB（Snappy） | 3 µs |
| 1Gbps 网络发 1KB | 10 µs |
| SSD 随机读 4KB | 150 µs |
| 内存顺序读 1MB | 250 µs |
| 同机房 RTT | 500 µs |
| SSD 顺序读 1MB | 1 ms |
| HDD 寻址 | 10 ms |
| HDD 顺序读 1MB | 20 ms |
| 跨地域 RTT（美-欧） | 150 ms |

这张表你之后会在共识、复制、缓存设计里反复用到。**当有人说"加一次 Redis 调用没什么"，你脑子里应该立刻浮现 500µs 那一行**。

### 7. 从最朴素的存储开始理解：追加写日志

先别急着看复杂树结构。
先想一个最简单的方法：每次写入都直接追加到文件末尾。

比如：

- `set x = 3`
- `set y = 8`
- `set x = 5`

磁盘上就不断追加：

```text
x=3
y=8
x=5
```

这个方法的优点很明显：

- 写入很简单
- 顺序追加通常很快（机械盘顺序 ≈ 100MB/s，快到惊人）
- 不需要每次都改动原位置
- 崩溃恢复简单：扫一遍日志就能还原

但问题也很明显：

- 如果你现在问 `x` 是多少，要从头或某种辅助结构里找最后一次写入
- 数据会越来越多
- 老版本记录会堆积

几乎所有现代数据库都**建立在"追加写日志"这个基础之上**：

- Postgres/MySQL 的 **WAL**（Write-Ahead Log）
- Kafka 整个系统就是个持久化的日志
- LSM Tree 的底层就是一堆顺序写的 SSTable
- Git 的底层也是"追加对象 + 引用指向"

追加写的本质是**用顺序 I/O 换了随机 I/O 的代价**。DDIA 会不断回到这个主题。

#### 7.1 Bitcask：最小可工作的键值引擎

Riak 的 Bitcask 是理解"追加写 + 索引"的最简原型：

- 数据全部追加写到一个 append-only 文件
- 内存里维护一个哈希表：`key → (file_id, offset, size)`
- 读：查哈希表找到 offset，一次随机读取返回值
- 写：追加一条记录 + 更新内存哈希

优点：写极快（纯顺序）、读一次磁盘 I/O。
限制：**所有 key 必须能装进内存**。

这是你理解后续 LSM 的起点。LSM 做的事情就是：**把内存里的哈希表换成一个有序结构（SSTable），允许 key 超过内存**。

### 8. 为什么需要索引

如果没有索引，数据库查数据就像你在一大堆纸质档案里一页页翻（这叫**全表扫描**）。

索引本质上就是：
**为了更快找到数据，而额外维护的一套辅助结构。**

代价是：

- 查询更快了
- 但写入时要顺便维护索引，所以写可能更慢，占用更多空间

这又是一个典型 trade-off。

举个最简单例子：

如果订单表按 `order_id` 建了索引，那么你查某个订单时就不用扫描整张表。
但每次插入新订单时，这个索引也要同步更新。

所以"加索引"不是零成本优化，而是"拿写入代价和空间换查询效率"。

#### 8.1 索引的基本类型

- **主键索引 / 聚簇索引（Clustered Index）**：索引的叶子节点就是数据本身。InnoDB、SQL Server 默认如此。好处：按主键查询不需要"回表"。代价：表物理上按主键排序，主键设计不当代价很大。
- **非聚簇索引（Non-clustered / Secondary Index）**：索引叶子只存主键值，查到后要再回主键索引查真实行。MyISAM、Postgres 默认如此。好处：同一张表可以有多个二级索引，互不干扰。代价：每次查询都要"回表"。
- **覆盖索引（Covering Index）**：索引包含了查询所需的全部列，根本不需要回表。经典例子：建 `(user_id, order_id, status)` 索引后，`SELECT status WHERE user_id=? AND order_id=?` 就只读索引，一次 I/O。

#### 8.2 索引的真实工程成本

索引是免费的吗？完全不是。一条记录写入时的代价，在 InnoDB 这种系统里大致是：

- 1 次聚簇索引更新（数据本身）
- N 次二级索引更新（每个二级索引一次）
- 加日志、加缓冲池、加复制流

**一张业务表如果有 10 个二级索引，写入放大就是 11 倍。** 这就是为什么 DBA 对"乱加索引"很警惕：查询优化的人看到变快很开心，背后写入和磁盘空间却在默默膨胀。

所以索引设计的黄金法则是：**先看查询模式，按查询频次和性能敏感度排序，只为真正需要的查询建索引，多列索引按最左前缀设计**。

### 9. 两条主流底层思路：B-Tree 和 LSM Tree

这两条路不是"随便选的"，而是**从 70 年代到现在所有存储引擎工业实践的两个主干**。

- B-Tree 家族（1970s，Bayer & McCreight）：Postgres、MySQL InnoDB、Oracle、SQL Server、BoltDB、LMDB
- LSM Tree 家族（1996，O'Neil et al.）：LevelDB、RocksDB、Cassandra、HBase、InfluxDB、ScyllaDB、TiKV、Pebble

#### 9.1 B-Tree 内部：页、节点、WAL

B-Tree 不是一个抽象的树，它是一个具体的磁盘结构。要理解它，先理解"页"。

**页（page）**：磁盘和内存之间的 I/O 基本单位。InnoDB 默认 16KB，Postgres 默认 8KB。为什么不是 4KB？因为：

- 4KB 是 OS 页大小，一次 I/O 也得读这么多
- 但 4KB 装不下多少记录和索引分支（fanout 太小）
- 16KB 在 fanout 和 I/O 对齐之间找了个平衡

B+Tree（InnoDB 实际用的变体）的结构：

```
          [根节点: 10 | 50 | 200]
          /           |           \
   [1..9]        [11..49]     [51..199]   <- 内部节点
     |               |               |
   ...             ...             ...     <- 叶子节点（有序链表连起来）
```

关键点：

- 内部节点存**键 + 子节点指针**，不存数据
- 叶子节点存**键 + 数据（或主键值）**
- 叶子之间用双向链表串起来，方便范围扫描
- 一棵高度为 h 的树，一次查询最多 h 次 I/O。一个 100GB 的表，深度通常只有 3–4 层

**页分裂（page split）**：插入时如果目标页已满，要分裂成两页，并在父节点加一个新指针。**分裂是 B-Tree 写入代价最大的来源之一**。

**一个致命问题**：如果分裂到一半断电了怎么办？父节点改了、子节点没写完，磁盘上就是个损坏的树。

**B-Tree 的应对方案：WAL（Write-Ahead Log）**。每次改动先顺序追加到 WAL，然后才改 B-Tree 本体。崩溃恢复时根据 WAL 重做。这是所有"就地修改"型存储引擎的标配。

**另一种应对**：Copy-on-Write B-Tree（LMDB、BoltDB、Btrfs 用的思路）。不直接改旧页，而是写一个新页，再原子地替换父指针。没有 WAL，但空间换安全。

#### 9.2 LSM Tree 内部：memtable、SSTable、compaction

LSM 把"随机写 B-Tree"这件事换成了"顺序追加 + 后台整理"，核心三件套：

1. **Memtable（内存有序结构，通常是跳表或红黑树）**：新写入先进这里。查找也优先查它。内存里有序，所以能支持范围查询。
2. **SSTable（Sorted String Table，磁盘上不可变的有序键值文件）**：memtable 满了后一次性刷盘成一个 SSTable。SSTable 内部按 key 有序，写完就不再修改。
3. **Compaction（后台合并）**：随着 SSTable 越来越多，后台进程定期挑几个 SSTable 合并成一个更大的、更干净的 SSTable（淘汰老版本和墓碑）。

**读路径**：查一个 key 时，要从 memtable 开始，依次查每个 SSTable（新的优先），直到找到为止。

这里有个性能灾难：如果一个 key 不存在，要扫完所有 SSTable 才能确认。解决方案是**布隆过滤器**。

##### 9.2.1 布隆过滤器（Bloom Filter）：LSM 的性能救星

布隆过滤器是一个**概率性数据结构**：

- 你问它"这个 key 在不在"
- 它回答"肯定不在"或"可能在"
- 从不会把"在"误报成"不在"（没有假阴性），但可能把"不在"误报成"可能在"（有假阳性）

内部实现：一个 m 位的 bit 数组 + k 个哈希函数。插入 key 时，k 个哈希函数映射到 k 个位置，都置 1。查找时，k 个位置都是 1 才返回"可能在"。

关键数学：给定 m 个 bit、k 个哈希、n 个元素，假阳率约为：

$$ P \approx \left(1 - e^{-kn/m}\right)^k $$

最优 k ≈ `(m/n) × ln2`。实践中：

- 每个 key 分配 10 bits，假阳率约 1%
- 每个 key 分配 20 bits，假阳率约 0.01%

**LSM 的每一个 SSTable 都配一个布隆过滤器**。读 key 时先问布隆过滤器"这个文件里可能有吗"，如果否，直接跳过，零 I/O。这个优化让 LSM 的点查性能从"扫 N 个文件"降到"接近 1 次 I/O"。

##### 9.2.2 Compaction 策略：两个主流选项

Compaction 不是随便合，工业界有两大流派：

- **Leveled Compaction**（LevelDB、RocksDB、Cassandra 新版）：SSTable 组织成 L0、L1、L2...，每一层容量约为上一层的 10 倍，每层内 key 范围不重叠。空间放大低（~1.1x），读放大低，但**写放大高**（一个 key 可能被反复重写到更高层）。
- **Tiered Compaction / Size-Tiered**（Cassandra 老版、HBase）：相同大小的 SSTable 一起合并成更大的。写放大低（~4x），但空间放大高（最多 2x，合并峰值内存更高），读放大也更高。

**写密集型选 tiered，读密集型选 leveled**。RocksDB 的默认配置和推荐参数就是围绕这两个策略展开。

##### 9.2.3 墓碑（tombstone）：删除的特殊处理

LSM 里删除不是"擦掉"，而是写一条特殊的"墓碑"记录。Compaction 时墓碑被保留到足够久后才真正清理。

这带来一个真实的线上陷阱：**在 Cassandra 里频繁写入和删除同一批 key，墓碑会堆积。查询时要跳过大量墓碑，延迟飙升**。业界把这叫"tombstone hell"，有公司因此下线过表。

### 10. B-Tree vs LSM Tree：三种放大的三角

真正能一句话抓住两者区别的，是**三种放大（amplification）**这个视角：

- **写放大（write amplification）**：一次逻辑写入，在磁盘上实际写了多少次
- **读放大（read amplification）**：一次查询，实际读了多少次
- **空间放大（space amplification）**：实际占用的存储空间 / 逻辑数据大小

典型对比（经验数字，实际随配置变化）：

| | B-Tree | LSM Tree (leveled) | LSM Tree (tiered) |
|---|---|---|---|
| 写放大 | ~2–5x（WAL + 随机写页） | ~10–30x（多次 compaction） | ~4–6x |
| 读放大 | 低（一次树查找） | 中（布隆+多 SSTable） | 高 |
| 空间放大 | 低（~1.3x，含碎片） | ~1.1x | ~2x |
| 写吞吐 | 中 | 高（顺序写） | 最高 |
| 范围扫描 | 好 | 好 | 中 |

**没有"更好"，只有"更适合你的工作负载"**：

- 读多写少、范围扫描多、事务要求高 → B-Tree（InnoDB、Postgres）
- 写远多于读、能接受后台 compaction 带来的延迟抖动 → LSM（RocksDB、Cassandra）
- 时序数据、日志类数据、IoT 数据 → 几乎一定是 LSM 或专门的 TSDB

### 11. 全文索引：倒排索引是另一套世界

Elasticsearch/Lucene 不是在"关系库上加索引"，它本身是一个**专门为全文搜索优化的存储引擎**，核心结构叫**倒排索引（inverted index）**。

基本想法：

- 正排：`doc_id → [term1, term2, ...]`（即文档内容）
- 倒排：`term → [doc_id1, doc_id2, ...]`（某个词出现在哪些文档里）

查询"apple iphone"时，取两个词的 posting list 做**交集**（intersection），再按相关性打分。

难点在哪：

- **分词**：中文、日文没有空格，要上分词器（jieba、IK、mecab）
- **压缩**：posting list 可以非常长（高频词几百万），用 delta encoding + variable byte encoding 压缩
- **打分**：TF-IDF、BM25 是主流相关性算法
- **近实时**：每次写入都更新索引代价大，Lucene 用 segment 机制（类似 LSM 的 SSTable），后台合并

近年向量检索（HNSW、IVF-PQ）又加了一层：文本先 embed 成向量，查询时算近邻。这是另一个大话题，原理上和倒排是两套思路，但工程上经常部署在同一个系统里（Elasticsearch 8、Milvus、Qdrant）。

### 12. 内存数据库：不是"磁盘 + 缓存"，是完全不同的物种

Redis、Memcached、VoltDB、Tarantool 这些系统把数据全部放内存。你可能以为它们和"磁盘 DB + 大缓冲池"效果差不多，其实不是。

差别有两层：

1. **设计假设不同**：磁盘 DB 要处理"页在内存和磁盘之间换入换出"，而内存 DB 根本不管这件事。这让索引结构、锁、事务协议都能用更激进的设计。
2. **数据结构更丰富**：内存里可以轻松实现链表、跳表、哈希、有序集合等复杂结构；磁盘结构受限于页和顺序写，选择少得多。

持久化通常通过**快照（snapshot）+ 操作日志（AOF）** 或**异步复制到磁盘**实现，而不是让内存和磁盘同步。

结论：**内存数据库不是快一点的磁盘数据库，它是另一种设计哲学**。用 Redis 当 KV 缓存是大材小用——它真正的价值是 `ZADD` `LPUSH` `GEOADD` 这些复杂数据结构原语。

### 13. 为什么同样是"数据库"，有的适合事务，有的适合日志写入，有的适合搜索

因为它们优化目标不同。

比如：

- MySQL/Postgres：强调事务、约束、通用查询（行存 + B-Tree）
- Cassandra/HBase：偏大规模写入和分布式扩展（LSM + 宽列）
- Elasticsearch：偏全文检索（倒排 + 分段）
- ClickHouse/BigQuery：偏分析查询（列存 + 向量化）
- Redis：偏低延迟热数据（内存 + 丰富数据结构）
- InfluxDB：偏时序数据（LSM + 时间分片）
- Neo4j：偏关系遍历（邻接表 + 免索引查找）

它们处理的不是同一种"查找问题"。

这也是 DDIA 很重要的一点：
不要把"存数据"看成一件单一的事。
不同系统之所以不同，是因为它们在底层为不同工作负载做了不同优化。

### 14. OLTP vs OLAP：行存与列存的物种分化

上面这些系统里，最深的一条分界线不是"关系 vs NoSQL"，而是 **OLTP vs OLAP**。

| | OLTP（在线事务处理） | OLAP（在线分析处理） |
|---|---|---|
| 典型用户 | 终端用户、应用服务 | 业务分析师、报表 |
| 典型查询 | 按主键读写少量行 | 扫描大量行做聚合 |
| 数据量 | GB–TB | TB–PB |
| 延迟要求 | ms | 秒–分钟 |
| 写入模式 | 高频小事务 | 低频大批量（ETL） |
| 存储布局 | **行存** | **列存** |

#### 14.1 行存 vs 列存的根本差别

假设有一张 1 亿行、每行 100 列的订单表，你要做 `SELECT AVG(amount) FROM orders WHERE region='US'`。

**行存**（InnoDB 式）：每一行 100 个字段连续存。读这条查询要扫完整张表、每行读 100 列但只用 2 列 → **浪费 98% 的 I/O**。

**列存**（C-Store、ClickHouse、Parquet、ORC 的思路）：每一列独立存储。查询只读 `region` 和 `amount` 两列 → I/O 立刻少 50 倍。

#### 14.2 列存的其他优势

把同类型的数据放一起后，能做关系行存做不了的优化：

- **压缩**：同列数据重复度高（性别只有 2 个值，国家只有几百个值），**字典编码（dictionary encoding）+ 位图（bitmap）+ RLE（run-length encoding）** 压缩率可以到 10:1 甚至更高。
- **向量化执行（vectorized execution）**：一次处理一个 column batch（比如 1024 个值），用 SIMD 指令并行。相比行存一行行处理，CPU 缓存命中率和吞吐都上一个台阶。
- **Zone maps / min-max indexes**：每个块记录这块数据的最小最大值。`WHERE region='US'` 时，直接跳过不含 'US' 的整块。
- **列级索引**：某些列做 bitmap 索引，过滤条件多个时做位图 AND/OR 极快。

这些优势结合起来，OLAP 查询在列存上比行存可以快 **10–100 倍**。这就是 ClickHouse、BigQuery、Snowflake、Redshift 的存在理由。

#### 14.3 为什么不用一个库搞定两件事

理论上 Postgres 可以做分析，MySQL 也能跑报表。但现实里：

- OLTP 要小事务、低延迟、高并发——必须行存 + B-Tree
- OLAP 要大扫描、高压缩、向量化——必须列存 + 批量

**它们是两个相反方向的工程优化**。HTAP 数据库（TiDB、Unistore、Snowflake Unistore）尝试在同一系统里维护两套存储，但本质上是"一套数据、两种存储"，而不是"一套存储两个用途"。

### 15. 用一个订单系统把这一课串起来

假设你在做电商订单系统。

你会遇到不同性质的数据：

- **订单主记录**：结构清晰、事务要求高、多对多（订单 ↔ 商品） → 关系库 + B-Tree + 聚簇索引
- **商品详情**：字段多、结构常变、整体读多 → Postgres JSONB 或文档库
- **用户关系推荐**：关系网络复杂、需要多跳 → 图数据库或专门的图计算
- **搜索商品标题**：需要全文检索 → Elasticsearch
- **订单历史分析、运营大盘**：批量扫描 + 聚合 → 列存数仓（ClickHouse / BigQuery）
- **实时热点榜、购物车** → Redis / 内存库
- **用户行为日志** → Kafka + 落到时序库或数据湖

如果你硬要求"一种模型、一个库搞定一切"，往往会越来越别扭。

更常见的现实是多存储架构。这不是"技术栈乱"，而是因为**不同数据问题需要不同表示和检索方式**，而且大多数情况下，用一个"通用"存储去对付所有场景，总代价反而更高。

当然，这也会带来系统复杂度。
所以真正的工程能力不是看到多系统就兴奋，而是知道：

- 为什么要拆
- 拆了后数据怎么同步（后面章节的 CDC、数据流）
- 哪个才是事实来源（Source of Truth）
- 每个系统自己的一致性和可用性语义是什么

这些问题在后面复制、数据流、一致性章节会继续展开。

### 16. 这一课真正要建立的底层意识

你现在最需要建立的意识是下面这句：

**应用层看到的数据结构，背后总对应具体的存储组织和检索代价。**

也就是说：

- 不是"我想怎么查就怎么查"，底层一定要为这种查询付代价
- 不是"我想怎么存就怎么存"，后面检索和更新也要承担后果
- 不是"索引加上就变快"，索引本身是写入放大和空间放大的来源
- 不是"哪个数据库更先进"，而是"哪种物理代价模型最匹配我的访问模式"

所以好的系统设计，从来不是先拍脑袋定技术，而是先弄清：

1. 数据长什么样？（聚合边界、关系类型）
2. 最常见的读写模式是什么？（读写比、点查/范围/扫描、扇出）
3. 哪些查询最关键？（SLA 所在的路径）
4. 哪些正确性约束最重要？（事务边界）
5. 我愿意用什么代价换什么能力？（放大 vs 延迟 vs 空间）

这就是这一课和后面所有课之间的连接点。

## 小结（5 条关键点）

- **数据模型**是系统表达现实的抽象接口，决定了上层能表达什么、下层能优化什么；关系/文档/图并非替代关系，而是对不同关系密度的优化——**多对多是判断文档是否适合的真正分界**。
- **Schema-on-read vs schema-on-write** 和**局部性**是文档模型两个最常被忽视的真正特征；DDD 的聚合根概念和文档边界几乎一一对应。
- **磁盘物理代价模型**（顺序 100× 随机、内存 10^4× 磁盘）决定了所有存储引擎的底层形态；"追加写日志"是大半个现代数据库的共同基石。
- **B-Tree vs LSM Tree** 不是"哪个更好"，而是在**写放大、读放大、空间放大**三角里做不同折中；布隆过滤器、页/WAL、compaction 策略是两种架构各自关键的工程细节。
- **OLTP 与 OLAP 是行存与列存的根本分化**；列存靠压缩 + 向量化 + zone map 在分析场景下可以快 10–100 倍；没有哪一种存储能同时赢两边。
