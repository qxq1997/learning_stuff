# DDIA - 第 4 课：编码、模式演化与数据流

## 学习目标（本节结束后你能做到什么）

- 理解"编码"在 DDIA 里不是字符编码，而是数据如何被序列化、传输、存储、再还原。
- 能说清楚为什么系统升级时最容易出问题的不是"代码能不能编译"，而是"旧数据和新代码还能不能互相理解"。
- 理解向后兼容、向前兼容这两个概念，并能从 wire format 层面（字段编号、tag、默认值）解释它们是如何被技术实现保证的。
- 区分"语言内置序列化"、"文本格式"、"二进制 schema 格式"三类主流编码，知道为什么大厂几乎都用第三类。
- 理解 Protobuf 的 varint、tag 编号、wire type 和 Avro 的 writer's / reader's schema 解析，知道这两套完全不同的兼容性哲学。
- 掌握 schema 演化的工程范式（expand-contract 模式、schema registry、契约治理），而不是只停留在"新增字段不会崩"。
- 理解三种数据流链路（数据库、RPC、消息）各自的兼容性挑战和典型陷阱。
- 知道"远程调用就像本地调用"（location transparency）为什么是谎言，这个谎言怎么一路发展到 REST/gRPC/GraphQL。
- 初步建立"事件驱动 / log-centric"的架构观，为后面章节的流处理打底。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 DDIA 在讲完数据模型后，会马上讲"编码与演化"

前一课我们讲了数据模型，回答的是：
**现实世界里的信息，在系统里应该怎样组织。**

但系统设计还要继续追问一个问题：

**这些数据一旦要写入磁盘、通过网络发送、在不同服务之间传递，它们到底是以什么形式存在的？**

这就是"编码"的问题。

注意，这里的"编码"不是 UTF-8、GBK 那种字符编码，而更接近：

- 一个对象如何变成字节流
- 一段字节流如何被另一个程序还原成对象

这个过程通常叫：

- **序列化（serialization / encoding / marshalling）**
- **反序列化（deserialization / decoding / unmarshalling）**

（在不同生态里术语不一样：Java 说 marshalling、C# 说 serialization、Go 说 encoding。它们指的都是同一回事。）

如果一个系统永远只有一个进程、一个版本、一个程序员，那这件事可能没那么棘手。
但现实系统不是这样。现实里经常是：

- 数据已经写进数据库几年了
- 某个服务正在做新版本发布
- 消息队列里还有老格式消息没消费完
- 手机端 App 还停留在旧版本
- 外部合作伙伴用的是半年前你们发布的 SDK

这时候真正困难的问题就来了：
**新旧版本之间还能不能互相理解数据？**

这就是"模式演化（schema evolution）"。

#### 1.1 DDIA 把这一章放在这里的深意

Kleppmann 把"编码与演化"放在数据模型之后、复制分区之前，这个位置不是随意的。

原因是：**一旦系统跨进程、跨机器、跨时间地流动数据，编码就是绕不开的边界**。

- 跨进程：RPC 和消息队列都要先序列化
- 跨机器：复制和分区都涉及 wire format
- 跨时间：磁盘上躺几年的旧数据、消息队列里积压的旧消息

后面讲复制、分区、流处理、批处理时，都会默认你知道"编码层是解决什么问题的"。否则你会发现每一章都绕不开这个坎，却不知道为什么绕不开。

### 2. 先理解"模式"是什么

你可以先把"模式"理解成：
**数据长什么样、有哪些字段、字段含义是什么、哪些字段是必须的，以及这些定义如何演化。**

比如一个用户对象最开始是：

```json
{
  "user_id": 1001,
  "name": "张三"
}
```

后来业务升级，加了一个字段：

```json
{
  "user_id": 1001,
  "name": "张三",
  "email": "a@example.com"
}
```

再后来觉得 `name` 要拆成 `first_name` 和 `last_name`：

```json
{
  "user_id": 1001,
  "first_name": "三",
  "last_name": "张",
  "email": "a@example.com"
}
```

这就是模式在变化，也就是模式演化。

问题在于：
旧代码可能只认 `name`，新代码可能开始依赖 `email`，再新一点的代码可能连 `name` 都不读了。
如果你改得太猛，系统不同部分就会互相听不懂。

#### 2.1 "模式"有显式与隐式之分

一个经常被忽视的点：**所有数据都有 schema，只是定义方式不同**。

- **显式 schema**：Postgres 的表结构、Protobuf 的 `.proto`、Avro 的 `.avsc`。你在数据库或定义文件里写清楚。
- **隐式 schema**：JSON、MongoDB 文档。它"没有 schema"是个错觉——真正的 schema 藏在读数据的代码里：`order.items[0].price` 这一行本身就在假设一个结构。

这个区分直接决定了演化时的工作分工：

- 显式 schema：schema 是合同，工具链（编译器、迁移器、optimizer）帮你校验
- 隐式 schema：schema 是假设，校验责任落在每个读取方的代码里（还要跨版本兼顾）

很多"schema-free 灵活"带来的痛苦，本质上是**把 schema 的复杂度从"建库的一次性工作"转移到了"每次读数据的永恒工作"**。

### 3. 为什么模式演化是分布式系统里的高频痛点

很多初学者以为"改个字段名不就是重构吗"。
在单体本地代码里，很多时候确实只是重构。
但到了数据系统里，字段演化远比重构复杂，因为数据会脱离代码、长期存在。

它会出现在这些地方：

- 数据库里已经存了海量旧数据
- 日志文件里保留着旧格式记录
- Kafka 或其他消息队列里还有旧消息
- 调用方和被调用方不一定同时升级
- 不同服务由不同团队维护，发布时间不一致
- 上游第三方 SDK 发给你的数据不受你控制

所以"模式变更"不是你一个人改代码那么简单，而是牵涉整个数据流链路的兼容问题。

#### 3.1 滚动升级（rolling upgrade）决定了兼容性不是可选项

大规模线上系统绝不可能"所有节点一起重启升级"。标准做法是**滚动升级**：

1. 从负载均衡摘下一台旧节点
2. 升级它到新版本
3. 健康检查通过后放回
4. 重复这个过程，直到所有节点升级完成

这个过程可能持续几小时甚至几天。这段时间里：

- 新版本节点和旧版本节点**同时在线**
- 它们可能互相调用（A 服务升级过程中，A 的新旧版本都会调 B）
- 消息队列里有新旧版本节点产生的消息混杂
- 数据库里有新旧版本写入的数据

**这就从根本上要求**：新版本能读懂旧版本写的东西（向后兼容），旧版本必须至少不崩于新版本写的东西（向前兼容）。**兼容性不是"最佳实践"，是"发布系统的硬约束"**。

一个反例：Facebook 早年就有过一次故障，上游服务发新字段，下游旧版本解析器直接崩，灰度发布推到 1% 流量时整个服务挂了。事后把兼容性写进了 Protobuf 的强制规则。

### 4. 三类主流编码方式：先看全景

在讨论兼容性规则前，先把编码格式的大谱系建立起来。

#### 4.1 语言内置序列化：方便但危险

Java 的 `Serializable`、Python 的 `pickle`、Ruby 的 `Marshal`、PHP 的 `serialize`——几乎每个语言都有一套"把对象变字节"的原生机制。

优点：调用一行代码就行，**不需要写 schema**。

致命问题：

1. **绑死语言**：Java 序列化的字节流只有 Java 能读。异构系统一上来就废了。
2. **安全灾难**：反序列化会实例化任意类，可以触发任意构造函数和 `readObject` 副作用。**历史上导致过多次 RCE 漏洞**（Apache Commons Collections、Log4Shell 都和反序列化链有关）。
3. **版本脆弱**：字段加个或去个、类包名一改，旧字节流就崩。Java 的 `serialVersionUID` 机制本身就是给这套坑打的补丁。
4. **性能一般**：既不紧凑也不快。Java 序列化甚至会带上整个类的元信息。

**结论**：除非纯进程内缓存、永远不跨进程，否则别用语言内置序列化。这是业界的共识，DDIA 花了不短篇幅说这个，就是因为**还有很多团队在生产里踩这个坑**。

#### 4.2 文本格式：JSON / XML / CSV

好处是**人类可读、跨语言、工具链成熟**。这是你前面最熟悉的一类。

但 DDIA 对它们有一段冷静的批评，你应该记住：

- **JSON 的数字精度问题**：JavaScript 里所有数字都是 IEEE 754 双精度浮点，整数只能精确表示到 2^53 - 1。Twitter 的 tweet ID 超过这个数后，JS 客户端直接拿不到正确 ID，最后发 API 时要同时返回 `id` 和 `id_str` 两个字段。这个坑坑过太多团队。
- **JSON/XML 没有原生二进制**：要发图片、音频、证书，只能 Base64 编码，体积膨胀 33%。
- **CSV 的本质是"没有规范的协议"**：字符串里能不能有逗号？换行符怎么处理？引号嵌套？每家实现不一样。
- **XML 的过度设计**：命名空间、DTD、XSD、XPath、XSLT，学习成本高，性能差。
- **所有文本格式的共同代价**：字段名重复出现，占用带宽；解析慢（要扫字符、处理转义）。

文本格式的正确定位是：**对外 API、调试接口、配置文件**——需要人读、容忍开销的场景。内部高频 RPC 和消息流，基本都会换成二进制格式。

##### 4.2.1 "二进制 JSON" 家族：BSON / MessagePack / CBOR

它们试图在"JSON 的数据模型"上做二进制编码，拿掉冗余字段名和解析开销：

- **BSON**：MongoDB 的内部存储格式
- **MessagePack**：应用最广，Redis 的 `msgpack` 支持也用它
- **CBOR**：IETF 标准化的那个，被 COSE（物联网安全）和 WebAuthn 用

它们的本质是"压缩的 JSON"，但都**保留了字段名**。比起下面真正的 schema-based 二进制格式，压缩率和性能都还差一个量级，因为字段名每条消息都带一次。

#### 4.3 Schema-based 二进制格式：Thrift / Protobuf / Avro

这是真正的大厂主力：

- **Thrift**：Facebook 2007 开源，接口定义语言（IDL）+ 二进制格式 + RPC 框架一体
- **Protocol Buffers**：Google 内部 2001 起用，2008 开源。v2 和 v3 语义差别较大
- **Avro**：Hadoop 社区出身，Doug Cutting 2009 设计，Kafka 生态首选

这三个的共同思想：

1. 先写一份 **schema 定义文件**（`.thrift` / `.proto` / `.avsc`）
2. 用**代码生成器**生成目标语言的对象类
3. 序列化时只写"字段编号/位置 + 值"，**不写字段名**
4. 反序列化时根据 schema 解读

字段名不在 wire 上传输，这就是它们压缩率高、解析快的根本原因。但代价是：**两端必须能共享 schema 知识**。兼容性规则就是围绕"schema 变了之后还能解码"这件事展开的。

### 5. 两个必须吃透的概念：向后兼容与向前兼容

这是本课最核心的知识点之一。

#### 5.1 向后兼容（backward compatibility）

向后兼容指的是：
**新代码能够读取旧数据。**

比如：

- 数据库里以前没有 `email` 字段
- 现在新版本程序上线了
- 它在读取老数据时，发现没有 `email`，仍然能正常工作

这就叫向后兼容。

为什么它重要？
因为旧数据常常是大量长期存在的，你不可能每次升级都先把全量历史数据立刻改完。

#### 5.2 向前兼容（forward compatibility）

向前兼容指的是：
**旧代码能够读取新数据。**

比如：

- 旧版本消费者还在运行
- 新版本生产者已经开始发送带 `email` 的消息
- 旧消费者看到这个额外字段时，不会直接崩掉，而是忽略自己不认识的部分

这就叫向前兼容。

为什么它重要？
因为在现实里，系统升级往往不是"全员同时切换"，而是逐步发布、灰度发布、分批发布。

#### 5.3 一个很容易记的口诀

你可以先这么记：

- **新程序读老数据：向后兼容**
- **老程序读新数据：向前兼容**

这两个方向不要搞反。

#### 5.4 两者缺一不可

很多人忽视向前兼容，觉得"老代码迟早会被淘汰，为什么要管它读不读得懂新数据"。

但真实系统里的场景：

- 你发消息到 Kafka，消费者集群正在滚动升级，你不可能暂停生产
- 移动端 App 用户不会都升级到最新版，**后端必须同时支持所有仍活跃的客户端版本**
- 一次发布出问题要回滚，回滚回老代码时**新代码已经写了一批新格式数据在数据库里**，老代码必须至少能读懂不崩

所以**一个合格的数据格式，必须双向兼容**。这是所有 schema-based 二进制格式在设计时都绕不开的根本约束。

### 6. Protobuf 是如何在 wire format 层面实现兼容的

仅仅说"新增字段安全"是不够的，你应该理解**为什么它安全**。这决定了你在边界情况下能不能正确判断一个改动是否兼容。

#### 6.1 Protobuf 的 wire 长什么样

以这个定义为例：

```proto
message Order {
  int32 order_id = 1;
  string user_name = 2;
  int32 amount = 3;
}
```

序列化后在线上的字节，大致是一系列 `(tag, wire_type, value)` 三元组：

- **Field number**（就是等号后面的 1、2、3）：用来标识是哪个字段
- **Wire type**（3 bits）：告诉解码器这个值占多长（varint、64-bit、length-delimited、32-bit 几种）
- **Value**：实际编码后的值

字段名 `order_id` `user_name` 根本不在 wire 上！wire 上只有数字 1、2、3。

这决定了兼容性规则：

**只要字段编号不变，字段在代码里叫什么名都无所谓。**

所以改名对兼容性完全无影响，改编号则会灾难性地让所有老数据/老消息解析失败。

#### 6.2 新增字段为什么自动前后兼容

- 老数据没有新字段的编号 → 新代码读时发现缺了字段 5，用默认值（0 / 空字符串）→ 向后兼容 ✓
- 新数据有新字段编号 7 → 老代码读时看到编号 7，查自己的 schema 不认识 → **按 wire type 跳过这几个字节，继续读下一个字段** → 向前兼容 ✓

这个"跳过未知字段"的能力就是靠 wire type 实现的——wire type 告诉解码器"这个值占几字节"，所以即使不认识字段含义，也能跨过它。

#### 6.3 哪些改动会破坏兼容

| 改动 | 向后兼容？ | 向前兼容？ | 说明 |
|---|---|---|---|
| 新增 optional 字段 | ✓ | ✓ | 最安全 |
| 删除 optional 字段 | ✓ | ✓（老代码不再依赖该字段时） | 但字段编号永远不能再复用 |
| 改字段名 | ✓ | ✓ | wire 上不含字段名 |
| 改字段编号 | ✗ | ✗ | 灾难 |
| 改字段类型（int32 → int64） | 看情况 | 看情况 | Protobuf 有兼容表，int32/int64/uint32/uint64/bool 之间相对宽松 |
| 改 repeated → 单数 | ✗ | ✗ | wire type 变了 |
| 新增 required 字段（proto2） | ✗ | ✗ | **这就是为什么 proto3 废掉了 required** |

**Protobuf 3 的一个重要设计决策**：完全废除 `required` 关键字。原因就是：一旦某个字段标为 required，它就永远不能被删除（删了老数据就读不进来）。这被视作"错误的默认值不可逆"，和兼容性哲学相悖。

这个决策在社区引起过很大争议，但它体现的思想是："不要让现在做的决定，锁死未来十年的演化自由"。

#### 6.4 Protobuf 的 varint 编码：压缩小整数

```
int32 = 150
```

如果硬编码 4 字节，就是 `00 00 00 96`。太浪费了，因为大多数业务里数字都小。

Protobuf 用 **varint**：每字节的最高位作为"是否还有后续字节"的标记，剩下 7 位存数据。小数字只需 1 字节：

```
150 的 varint = 0x96 0x01   (两个字节)
1 的 varint   = 0x01        (一个字节)
```

这个小设计让 "大部分字段是小整数" 的真实业务数据压缩效果很好，是 Protobuf 体积优势的主要来源之一。负数怎么办？用 **zigzag encoding**：把正负数交错映射到自然数，再走 varint。-1 变 1，1 变 2，-2 变 3……这样小负数也能用 1–2 字节。

（了解这些细节的好处是：你以后看 wireshark 抓包、调试 RPC 问题，不会一脸懵。）

### 7. Avro 的另一种哲学：writer's schema vs reader's schema

Avro 和 Protobuf 在兼容性实现上走的是完全不同的路，了解这个对比能让你真正理解"schema 是什么"。

#### 7.1 Avro 的关键设计

Avro 序列化时**不带字段编号**。wire 上只有按 schema 顺序的二进制值：

```
int32 int32 string    (原始 schema 定义的顺序)
1001  88    "张三"     (实际数据)
```

那没有字段编号怎么定位字段？靠的是 **writer's schema 必须随数据一起被消费者知道**。

#### 7.2 两种 schema 的配对

Avro 引入了两个概念：

- **Writer's schema**：数据写入时用的 schema
- **Reader's schema**：数据读取时代码期望的 schema

Avro 运行时会做一次 **schema resolution**（模式解析），自动将 writer 的格式映射到 reader 的期望：

- writer 有但 reader 没有的字段 → 跳过
- reader 有但 writer 没有的字段 → 用 reader schema 里定义的默认值
- 字段顺序可以不同（Avro 按名字匹配）

这个设计的代价：**消费者必须拿到 writer's schema**。否则它连"字段占几字节"都不知道，因为 wire 上没有 wire type 标记。

#### 7.3 Schema Registry：Kafka 生态的解决方案

当消息量巨大时，每条消息都带一份完整 schema 代价太高。Confluent 的 **Schema Registry** 解决了这个：

1. 所有 schema 版本在 registry 注册，获得一个全局 ID
2. Kafka 消息里只写 schema ID（4 字节）+ 数据
3. 消费者第一次见到某 ID 时去 registry 拉取对应 schema，本地缓存

这样：

- wire 开销极小（4 字节开销/消息）
- schema 集中管理，演化有审计和校验
- 可以在 registry 侧强制"向后/向前兼容"规则（BACKWARD、FORWARD、FULL），非兼容变更直接拦下

**Schema Registry 本质上是把 schema 从"代码"变成了"基础设施"**。这是 Avro 在 Kafka 场景里越来越普及的关键原因。

#### 7.4 动态生成 schema：Avro 比 Protobuf 灵活的地方

Avro 的另一个优势：schema 可以在运行时动态生成，不需要先 `protoc` 生成代码。这在"把数据库表结构自动导出为数据管道 schema"这类场景里非常好用——Hive、Pinot、Iceberg、Hudi 这些系统大量依赖这个能力。

#### 7.5 Protobuf vs Avro 哲学总结

| 维度 | Protobuf | Avro |
|---|---|---|
| 字段标识 | **字段编号**（wire 里显式） | **字段名**（wire 里省略，靠 schema 对齐） |
| 未知字段处理 | 按 wire type 跳过 | 靠 schema resolution |
| schema 分发 | 代码生成，编译期绑定 | 消费时动态拿（+ registry） |
| 改字段名 | 无影响 | 要处理（但支持 alias） |
| 动态 schema | 弱 | 强 |
| 典型用途 | 同步 RPC（gRPC） | 异步事件流（Kafka） |

**一句话选型**：**RPC 偏 Protobuf，异步事件流偏 Avro**。这个结论对大多数团队成立。Thrift 在 Facebook 内部仍是主力，但外部生态比前两者弱。

### 8. 兼容性不是空谈，它和字段变更动作直接相关

下面这些改动，有些比较安全，有些很危险。

#### 8.1 通常相对安全的改动

- 新增一个可选字段（Protobuf 3 默认 optional）
- 删除没人读的旧字段（字段编号仍保留，不能复用）
- 重命名字段（只在代码层面，wire 不受影响）

这是很多兼容设计的基础思路。

#### 8.2 通常风险较高的改动

- 直接删除还有人读的字段
- 改字段含义但字段名不变
- 把数字改成字符串，或者把字符串改成结构体
- 重复使用字段编号或改变编号语义
- 把一个字段从 "单数" 改成 "repeated"（wire type 变了）
- 在 Protobuf 2 新增 required 字段（Protobuf 3 已禁）

其中最危险的一种，其实不是"少个字段"，而是：
**字段还在，但含义偷偷变了。**

比如原来 `status = 1` 表示"已支付"，后来某团队改成"已创建"。
这类问题非常隐蔽，因为程序不一定会立刻报错，但业务语义已经错了。

#### 8.3 Expand-Contract 模式：安全演化的工程范式

当一个改动不是"纯兼容的新增"时（比如要重命名字段、要换类型、要拆字段），业界有一套被验证过的安全做法叫 **expand-contract**（或 parallel change、branch by abstraction）：

1. **Expand（扩展）**：新增目标字段，与旧字段**并存**。代码同时写两份。
2. **Migrate（迁移）**：后台任务把旧数据回填到新字段；所有读取方切到新字段。
3. **Contract（收缩）**：确认没有读取方还在用旧字段后，删除旧字段。

这三步之间都有**显式的等待期**（通常是一个发布周期 + 一个观察周期）。整个过程可能跨几周。

这个模式的核心思想：**任何破坏性变更都要拆成"一串纯兼容的变更序列"**。它是数据库迁移、RPC 契约变更、消息 schema 演化通用的武器。

一个具体例子：把 `name` 拆成 `first_name` + `last_name`：

1. 加 `first_name`, `last_name`，保留 `name`。写入时三个字段都写。
2. 回填历史数据。所有读取切到 `first_name` + `last_name`。
3. 等一段时间，确认无人读 `name`，删除它。

如果直接跳到第 3 步，老代码全崩。expand-contract 就是把"不兼容"拆成"若干兼容 + 等待"。

### 9. 数据是如何流动的：数据库、服务调用、消息传递

DDIA 在这里的一个重要视角是：
不要只盯着数据库表结构，要看到数据在整个系统里流动。

Kleppmann 把数据流归纳为三类主要模式：

- **通过数据库**（database）：一方写，另一方读
- **通过服务调用**（RPC / REST）：调用方请求，被调用方响应
- **通过异步消息**（message / event）：生产者发，消费者订阅

每种模式的兼容性挑战不一样。

#### 9.1 通过数据库流动

应用把数据写进数据库，另一个程序再去读取。
这种模式里，数据库像一个长期共享状态的中心。

问题在于：

- **数据在数据库里的寿命远长于任何一个应用版本**：你的 App 发布了 50 次，最老的那条记录还在数据库里等着被读
- **通用数据库没 schema registry**：行级 JSON 字段的演化全靠应用代码约定
- **同一张表被多个服务读写**时，谁能改 schema？

##### 9.1.1 在线 DDL 与大表迁移

数据库 schema 迁移自身就是个大坑。直接 `ALTER TABLE` 对亿级表：

- InnoDB 早期版本要**完全重建表**，期间锁表，线上直接不可用几十分钟到几小时
- Postgres 的某些 DDL 要 `ACCESS EXCLUSIVE` 锁，阻塞读写

工业界的做法：

- **MySQL 侧**：Percona `pt-online-schema-change`、Facebook `OnlineSchemaChange`、GitHub `gh-ost`。思路类似：创建影子表、逐步同步、切换。
- **Postgres 侧**：`pg_repack`、`concurrently` 索引、分阶段加字段
- **现代做法**：pt-osc 和 gh-ost 都把 expand-contract 工业化了

每次大表改 schema，都是一次需要 SRE 参与的工程事件，不是写一行 DDL 就完事。

##### 9.1.2 数据映射层（ORM）的兼容陷阱

ORM 把"对象"映射到"行"。但对象版本、行版本、库版本三者如果不一致，可能产生极隐蔽的 bug：

- 新代码读 `user.premium_level`，数据库里是 `premium_level = NULL`（旧数据）→ ORM 可能返回 0 或抛 NPE，取决于注解
- 新代码通过部分更新保存对象，ORM 把所有字段都写回去 → 覆盖了并发另一台机器刚写入的新字段

所以 "应用层对象 schema" 和 "数据库 schema" 的演化节奏必须同步考虑。

#### 9.2 通过服务调用流动

比如 A 服务调用 B 服务的 HTTP 或 RPC 接口。

这里的兼容问题通常体现为：

- 请求体加字段，旧服务能不能忽略？
- 响应里多出字段，旧客户端会不会崩？
- 某字段从必填变成选填，调用方是否都跟上了？

##### 9.2.1 远程调用的思想史

RPC 的思想可追溯到 1984 年 Birrell & Nelson 的经典论文《Implementing Remote Procedure Calls》。核心理想是 **"让远程调用看起来像本地函数调用"**，也叫 **location transparency**。

这个理想贯穿了：

- **SUN RPC / ONC RPC**（80 年代）
- **CORBA**（90 年代，对象远程调用，复杂到劝退）
- **DCOM / .NET Remoting**（微软栈）
- **RMI**（Java）
- **SOAP**（2000 年代前半）
- **REST**（Roy Fielding 2000 博士论文提出，反潮流）
- **JSON-RPC / gRPC / Thrift / GraphQL**（现代）

这个几十年的演化，本质上就是**一次次的"location transparency 是谎言"的认清过程**。Waldo 等人 1994 年那篇《A Note on Distributed Computing》就把这事说透了：远程调用和本地调用有四个不可消除的差异：

1. **延迟不同**：本地调用 ns 级，远程 ms 级（甚至 100ms+）
2. **可能失败**：本地调用不会网络断，远程会
3. **并发性不同**：远程方可能同时被多人调用，产生并发问题
4. **内存空间不同**：本地可以传指针/引用，远程必须值传递

这四条差异就是后面章节"分布式系统的 8 个谬误"的雏形。DDIA 第 8 章会展开。

##### 9.2.2 REST 为什么赢了 SOAP

SOAP 曾是 2000 年代的标准：HTTP 之上套 XML 信封 + WSDL 契约 + WS-* 扩展家族。工业级，但太重。

REST（Fielding 2000）反其道而行：
- 用 HTTP 本身的动词（GET/POST/PUT/DELETE）
- 资源用 URL 标识
- 无状态、可缓存
- 格式不强制，JSON 逐渐成事实标准

REST 赢的本质是**拥抱 HTTP 而不是覆盖 HTTP**。不过 REST 本身很多原教旨主义概念（HATEOAS、Richardson Maturity Model Level 3）在工程里很少真正落地——实际大家用的是"类 REST 的 HTTP JSON API"。

##### 9.2.3 gRPC 与 GraphQL：现代服务通信的两个方向

- **gRPC**：Google 2015 年开源。Protobuf + HTTP/2 + 双向流。强 schema、高性能、代码生成。适合**内部服务间**。
- **GraphQL**：Facebook 2015 年开源。客户端主导查询，解决"REST 里客户端要么拿太多要么跑太多次"的问题。适合**面向多端（web + iOS + Android）的 BFF**。

它们不是替代 REST，是在不同约束下的优化。

##### 9.2.4 RPC 契约演化的主要陷阱

- 加必填参数 → 所有老调用方崩
- 改变返回值结构 → 老调用方反序列化可能失败
- 改错误码含义 → 灰难（用户代码通常写死 `if code == 42`）
- 删除 endpoint → 要先加版本路径（`/v2/...`），保留 `/v1/...` 过渡期

实践上一般遵循：**endpoint 加版本、字段 optional + default、破坏变更走 expand-contract**。

#### 9.3 通过异步消息流动

比如订单服务发一条消息给库存服务、营销服务、物流服务。

这时兼容性更敏感，因为：

- 消息可能在队列里停留一段时间
- 生产者和消费者升级节奏不同
- 同一个 topic 可能被多个消费者读取
- 消息一旦发出就收不回来

##### 9.3.1 消息队列 vs 事件日志

Kleppmann 在这里区分了两个完全不同的形态：

| | 消息队列（传统 MQ） | 事件日志（Kafka 式） |
|---|---|---|
| 代表 | RabbitMQ、ActiveMQ、SQS | Kafka、Pulsar |
| 消息被消费后 | 删除 | 保留（可回放） |
| 消费模型 | Push（或 pull）到若干消费者 | 消费者按 offset 自己推进 |
| 重播 | 很难 | 随意，按 offset 回拨 |
| 扇出 | N 份复制 | 零成本（每个消费者各自读同一份日志） |

这个差别在**事件驱动架构**里至关重要。Kafka 式日志不仅是"更好的 MQ"，它改变了架构——数据可以被多次、按需、任意时间消费，这使得：

- 新加一个下游消费者无需生产者改动
- 出 bug 时可以回放事件修数据
- 数据湖和 OLAP 侧可以直接订阅同一个流，保持同源

这就是 **"turning the database inside-out"**（数据库解构）思想的工业化落地。LinkedIn、Uber、Netflix 都大量基于这个模式构建数据管道。

##### 9.3.2 消息的兼容性更严苛

为什么比 RPC 更严？因为：

- RPC 失败可以返回错误码让调用方看到
- 消息发出去就"埋"进队列里，消费者几分钟或几小时后解不出来，你可能都不知道

所以消息格式的演化必须：

- **永远双向兼容**
- **schema 集中治理**（Schema Registry、契约中心）
- **生产者先升级、消费者后升级**（向前兼容覆盖过渡期）
- **有 dead letter queue**（解析失败的消息不要阻塞 pipeline）

### 10. 用订单系统举一个完整例子

假设订单消息最开始长这样：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "amount": 199
}
```

后来业务想加一个优惠券信息：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "amount": 199,
  "coupon_id": 555
}
```

如果库存服务、营销服务、风控服务都在消费这条消息，问题就来了：

- 老版本消费者看到 `coupon_id` 会不会报错？（**向前兼容**问题）
- 新版本消费者如果依赖 `coupon_id`，那老历史消息里没有这个字段怎么办？（**向后兼容**问题）
- 如果以后把 `amount` 改成带币种结构体 `{value, currency}`，旧逻辑还能不能工作？（**类型变更**问题）
- 消息 schema 归谁管？谁有权变更？（**契约治理**问题）

如果用 Avro + Schema Registry，这些问题被系统性解决：

- 新增 optional 字段，兼容性规则自动校验
- 消费者自动读到正确 schema
- 破坏性变更被注册中心拒绝提交

如果用 JSON + 约定俗成，每一步都靠人为谨慎，出事概率高得多。

这就是为什么字节、阿里、Uber、Netflix 这些数据密集型公司，内部都在把 JSON 消息逐步迁移到 Avro/Protobuf + Registry。**不是性能问题，是治理问题**。

### 11. 学这部分时最重要的工程意识

这一课真正想让你建立的，不是背格式名词，而是下面这个意识：

**数据一旦离开当前进程，就会拥有自己的生命周期。**

它可能：

- 在数据库里躺三年
- 在日志里保留半年
- 在消息队列里停十分钟
- 被旧客户端继续发送
- 被新服务重新解释
- 被数据湖用于半年后的分析

所以你设计数据格式时，不能只考虑"当前版本好不好写"，而必须考虑：

1. 旧数据怎么办？
2. 新旧版本如何共存？
3. 谁来保证 schema 变更规则？
4. 一旦改错，怎么回滚？
5. 格式本身是不是支持双向兼容？

这就是 DDIA 强调"演化"的原因。
很多系统不是死在不会传数据，而是死在"版本一多，大家互相看不懂数据"。

### 12. 一个实用判断框架

以后你看到一个字段或消息格式要改，可以先问六个问题：

1. **旧程序看到新字段会怎么样？**（向前兼容）
2. **新程序看到旧数据缺字段会怎么样？**（向后兼容）
3. **这个字段是新增、删除、改名，还是改语义？**（变更种类）
4. **这次变更影响的是数据库、RPC，还是消息队列？**（数据流路径）
5. **这是"纯新增"变更，还是需要 expand-contract 拆成多步？**（演化方法）
6. **滚动升级过程中新老版本混跑时会发生什么？**（过渡期行为）

只要这六个问题没想清楚，就不要轻易说"这个改动很简单"。

## 小结（5 条关键点）

- **编码**是数据跨进程、跨时间流动时的 wire format 问题；所有数据都有 schema，只是有的显式（数据库/Protobuf）、有的隐式（JSON/MongoDB，schema 在读取代码里）。
- **向后兼容 = 新代码读旧数据，向前兼容 = 旧代码读新数据**；滚动升级、异步消息、长寿数据使"双向兼容"成为发布系统的硬约束，而不是最佳实践。
- **三类主流编码**：语言内置（方便但危险）、文本（易读但冗余）、schema-based 二进制（紧凑严谨）；工业主力是 Protobuf（wire 里带字段编号，跳过未知字段）和 Avro（wire 不带编号，靠 writer's/reader's schema 匹配，常配 Schema Registry）。
- **破坏性变更必须拆成一串纯兼容变更 + 等待期**，这就是 expand-contract 模式；数据库大表 DDL、RPC 契约演化、消息 schema 演化都依赖这一范式。
- **三类数据流**（数据库、RPC、消息）各有兼容性特点；Kafka 式事件日志不仅是"更好的 MQ"，它通过可回放、多消费者、同源数据把架构推向事件驱动，是"数据库解构"思想的工业化落地。
