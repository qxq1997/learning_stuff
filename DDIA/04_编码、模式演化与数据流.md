# DDIA - 第 4 课：编码、模式演化与数据流

## 学习目标（本节结束后你能做到什么）

- 理解“编码”在 DDIA 里不是字符编码，而是数据如何被序列化、传输、存储、再还原。
- 能说清楚为什么系统升级时最容易出问题的不是“代码能不能编译”，而是“旧数据和新代码还能不能互相理解”。
- 理解向后兼容、向前兼容这两个概念，并能结合真实系统举例。
- 知道 JSON、Protocol Buffers、Avro 这类格式各自的大致特点，不把它们简单理解成“都能传数据所以差不多”。
- 初步建立“数据流”视角：数据不仅存在数据库里，还会在服务调用、消息队列、缓存、日志等链路中流动。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 DDIA 在讲完数据模型后，会马上讲“编码与演化”

前一课我们讲了数据模型，回答的是：  
**现实世界里的信息，在系统里应该怎样组织。**

但系统设计还要继续追问一个问题：

**这些数据一旦要写入磁盘、通过网络发送、在不同服务之间传递，它们到底是以什么形式存在的？**

这就是“编码”的问题。

注意，这里的“编码”不是 UTF-8、GBK 那种字符编码，而更接近：

- 一个对象如何变成字节流
- 一段字节流如何被另一个程序还原成对象

这个过程通常叫：

- 序列化
- 反序列化

如果一个系统永远只有一个进程、一个版本、一个程序员，那这件事可能没那么棘手。  
但现实系统不是这样。现实里经常是：

- 数据已经写进数据库几年了
- 某个服务正在做新版本发布
- 消息队列里还有老格式消息没消费完
- 手机端 App 还停留在旧版本

这时候真正困难的问题就来了：  
**新旧版本之间还能不能互相理解数据？**

这就是“模式演化”。

### 2. 先理解“模式”是什么

你可以先把“模式”理解成：  
**数据长什么样、有哪些字段、字段含义是什么、哪些字段是必须的。**

比如一个用户对象最开始是：

```json
{
  "user_id": 1001,
  "name": "张三"
}
```

后来业务升级，加了一个字段：

```json
{
  "user_id": 1001,
  "name": "张三",
  "email": "a@example.com"
}
```

再后来觉得 `name` 要拆成 `first_name` 和 `last_name`：

```json
{
  "user_id": 1001,
  "first_name": "三",
  "last_name": "张",
  "email": "a@example.com"
}
```

这就是模式在变化，也就是模式演化。

问题在于：  
旧代码可能只认 `name`，新代码可能开始依赖 `email`，再新一点的代码可能连 `name` 都不读了。  
如果你改得太猛，系统不同部分就会互相听不懂。

### 3. 为什么模式演化是分布式系统里的高频痛点

很多初学者以为“改个字段名不就是重构吗”。  
在单体本地代码里，很多时候确实只是重构。  
但到了数据系统里，字段演化远比重构复杂，因为数据会脱离代码、长期存在。

它会出现在这些地方：

- 数据库里已经存了海量旧数据
- 日志文件里保留着旧格式记录
- Kafka 或其他消息队列里还有旧消息
- 调用方和被调用方不一定同时升级
- 不同服务由不同团队维护，发布时间不一致

所以“模式变更”不是你一个人改代码那么简单，而是牵涉整个数据流链路的兼容问题。

### 4. 两个必须吃透的概念：向后兼容与向前兼容

这是本课最核心的知识点之一。

#### 4.1 向后兼容

向后兼容指的是：  
**新代码能够读取旧数据。**

比如：

- 数据库里以前没有 `email` 字段
- 现在新版本程序上线了
- 它在读取老数据时，发现没有 `email`，仍然能正常工作

这就叫向后兼容。

为什么它重要？  
因为旧数据常常是大量长期存在的，你不可能每次升级都先把全量历史数据立刻改完。

#### 4.2 向前兼容

向前兼容指的是：  
**旧代码能够读取新数据。**

比如：

- 旧版本消费者还在运行
- 新版本生产者已经开始发送带 `email` 的消息
- 旧消费者看到这个额外字段时，不会直接崩掉，而是忽略自己不认识的部分

这就叫向前兼容。

为什么它重要？  
因为在现实里，系统升级往往不是“全员同时切换”，而是逐步发布、灰度发布、分批发布。

### 5. 一个很容易记的口诀

你可以先这么记：

- 新程序读老数据：向后兼容
- 老程序读新数据：向前兼容

这两个方向不要搞反。  
你之后学消息队列、服务调用、数据库迁移时会不断碰到。

### 6. 最常见的几种编码格式，各自大概解决什么问题

这一部分你先抓大方向，不用一开始就背格式细节。

#### 6.1 JSON：好读、灵活、通用，但不够严格

JSON 的优点很明显：

- 人能直接看懂
- 调试方便
- 前后端、服务间都很常见
- 增加字段通常比较自然

但它也有明显问题：

- 类型表达不够严谨
- 字段名重复出现，体积偏大
- 没有强约束时，容易出现“看起来一样，实际不完全一样”的数据

比如某个字段今天是字符串 `"123"`，明天变成数字 `123`，表面上都像“123”，但对程序来说可能不是一回事。

所以 JSON 很适合：

- 对外 API
- 调试友好场景
- 对强约束要求没那么极致的系统边界

但它不一定是所有高性能或强 schema 场景的最优解。

#### 6.2 Protocol Buffers：更紧凑、更严格、适合服务间通信

Protocol Buffers 常被简称为 Protobuf。  
它的核心特点是：

- 有明确 schema
- 编码后二进制更紧凑
- 通信效率通常比 JSON 更高
- 对字段编号和兼容性规则要求更明确

你可以把它理解成：  
比 JSON 更“工程化”、更“机器友好”的数据表示方式。

它很适合：

- 服务间 RPC
- 对性能和带宽更敏感的场景
- 需要比较清晰 schema 管理的系统

但它的代价也很明显：

- 人眼不容易直接读
- 调试不像 JSON 那么自然
- schema 管理要更谨慎

#### 6.3 Avro：很强调 schema 演化，常见于数据管道与大数据场景

Avro 的特点之一是它对 schema 演化支持得比较体系化。  
它常出现在：

- Kafka 生态
- 数据仓库
- 批处理与流处理链路

为什么它在这些地方常见？  
因为这类系统特别在意“今天生成的数据，明天后天还能不能被不同版本程序读懂”。

你现在不需要死记 Avro 细节，只要先建立这个印象：

- JSON：直观灵活
- Protobuf：高效严格
- Avro：对 schema 演化管理很重要

### 7. 兼容性不是空谈，它和字段变更动作直接相关

下面这些改动，有些比较安全，有些很危险。

#### 7.1 通常相对安全的改动

- 新增一个可选字段
- 旧程序不认识新字段时选择忽略
- 新程序读不到旧字段时提供默认值

这是很多兼容设计的基础思路。

#### 7.2 通常风险较高的改动

- 直接删除旧字段
- 改字段含义但字段名不变
- 把数字改成字符串，或者把字符串改成结构体
- 重复使用字段编号或改变编号语义

其中最危险的一种，其实不是“少个字段”，而是：  
**字段还在，但含义偷偷变了。**

比如原来 `status = 1` 表示“已支付”，后来某团队改成“已创建”。  
这类问题非常隐蔽，因为程序不一定会立刻报错，但业务语义已经错了。

### 8. 数据是如何流动的：数据库、服务调用、消息传递

DDIA 在这里的一个重要视角是：  
不要只盯着数据库表结构，要看到数据在整个系统里流动。

常见的数据流场景有三类：

#### 8.1 通过数据库流动

应用把数据写进数据库，另一个程序再去读取。  
这种模式里，数据库像一个长期共享状态的中心。

问题在于：

- 新程序写入的新字段，老程序是否读得懂？
- 一次模式迁移要不要回填历史数据？
- 多个程序共用一张表时，谁能改 schema？

#### 8.2 通过服务调用流动

比如 A 服务调用 B 服务的 HTTP 或 RPC 接口。

这里的兼容问题通常体现为：

- 请求体加字段，旧服务能不能忽略？
- 响应里多出字段，旧客户端会不会崩？
- 某字段从必填变成选填，调用方是否都跟上了？

#### 8.3 通过异步消息流动

比如订单服务发一条消息给库存服务、营销服务、物流服务。

这时兼容性更敏感，因为：

- 消息可能在队列里停留一段时间
- 生产者和消费者升级节奏不同
- 同一个 topic 可能被多个消费者读取

所以消息格式设计通常更强调长期兼容，而不是“我当前代码能跑就行”。

### 9. 用订单系统举一个完整例子

假设订单消息最开始长这样：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "amount": 199
}
```

后来业务想加一个优惠券信息：

```json
{
  "order_id": 1001,
  "user_id": 88,
  "amount": 199,
  "coupon_id": 555
}
```

如果库存服务、营销服务、风控服务都在消费这条消息，问题就来了：

- 老版本消费者看到 `coupon_id` 会不会报错？
- 新版本消费者如果依赖 `coupon_id`，那老历史消息里没有这个字段怎么办？
- 如果以后把 `amount` 改成带币种结构体，旧逻辑还能不能工作？

这就是模式演化和数据流结合后的真实复杂度。  
你会发现，字段变化不再是“一个类文件改一下”这么简单，而是整个系统协作问题。

### 10. 学这部分时最重要的工程意识

这一课真正想让你建立的，不是背格式名词，而是下面这个意识：

**数据一旦离开当前进程，就会拥有自己的生命周期。**

它可能：

- 在数据库里躺三年
- 在日志里保留半年
- 在消息队列里停十分钟
- 被旧客户端继续发送
- 被新服务重新解释

所以你设计数据格式时，不能只考虑“当前版本好不好写”，而必须考虑：

1. 旧数据怎么办？
2. 新旧版本如何共存？
3. 谁来保证 schema 变更规则？
4. 一旦改错，怎么回滚？

这就是 DDIA 强调“演化”的原因。  
很多系统不是死在不会传数据，而是死在“版本一多，大家互相看不懂数据”。

### 11. 先给你一个实用判断框架

以后你看到一个字段或消息格式要改，可以先问四个问题：

1. 旧程序看到新字段会怎么样？
2. 新程序看到旧数据缺字段会怎么样？
3. 这个字段是新增、删除、改名，还是改语义？
4. 这次变更影响的是数据库、RPC，还是消息队列？

只要这四个问题没想清楚，就不要轻易说“这个改动很简单”。

## 小结（3-5 条关键点）

- 编码在 DDIA 里指的是数据如何被序列化、存储、传输和还原，而不是字符编码。
- 模式演化的核心难点在于：数据会长期存在，而系统各部分升级并不同步，因此必须考虑兼容性。
- 向后兼容是“新程序读老数据”，向前兼容是“老程序读新数据”。
- JSON、Protobuf、Avro 都能表示数据，但它们在可读性、效率、schema 严格性和演化管理上各有取舍。
- 数据会通过数据库、服务调用、消息队列等多条链路流动，因此 schema 设计必须站在整个数据流视角上考虑。
