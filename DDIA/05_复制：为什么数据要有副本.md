# DDIA - 第 5 课：复制：为什么数据要有副本

## 学习目标（本节结束后你能做到什么）

- 理解"复制"不是简单的备份，而是让同一份数据在多个节点上保存和服务。
- 说清楚为什么系统要做复制，以及复制主要在解决哪些问题。
- 区分主从复制、多主复制、无主复制三种基本思路，理解它们各自适合什么场景。
- 掌握**复制日志的四种实现方式**（语句、WAL、逻辑行、触发器），知道为什么现代系统几乎都走逻辑行这条路。
- 理解同步/异步/半同步复制的工程差别，知道它们在"故障时丢多少数据"上的具体含义。
- 掌握**复制延迟引发的三类一致性问题**（读自己写、单调读、前缀一致读）和它们的工程解法。
- 真正理解故障切换（failover）的真实风险：**脑裂、fencing、STONITH、见证节点**；能从 GitHub 2018 那次 24 小时脑裂里看出每一步的决策陷阱。
- 理解冲突解决的工程层次：Last-Write-Wins、版本向量、CRDT；以及"时钟漂移为什么让 LWW 变成数据杀手"。
- 掌握 Dynamo 式无主复制的 **W+R>N 法定人数**、宽松法定人数（sloppy quorum）、读修复（read repair）和反熵（anti-entropy）机制。
- 能结合真实业务（订单、社交、账户、跨国商务）判断某个系统应选哪种复制方式，以及它在"故障时丢多少、故障切换要几秒、一致性多强"上的具体权衡。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么系统要有多个副本

前面我们已经反复提到可靠性和可扩展性。
复制就是这两个目标在数据层面最常见的一种手段。

所谓复制，简单说就是：

**把同一份数据保存到多台机器上，而不是只放在一台机器上。**

比如订单数据不只存在主库里，还同步到其他副本机器。
这样做不是为了"显得高级"，而是因为单机天生有很多问题：

- 机器会坏（第 2 章的硬件 AFR 数据）
- 磁盘会坏
- 网络会抖
- 单台机器读能力有限
- 单点一旦挂掉，整个系统风险很大

所以，复制最直观的出发点是：

- 一台坏了，别的还在（**故障容忍**）
- 一台太忙了，别的能分担（**读扩展**）
- 一地出故障，别的机房还能接住（**地理容灾**）
- 用户就近访问（**降低延迟**）

### 2. 复制不等于备份，这两个概念要分开

这是最容易混淆的一点。

很多人听到"多份数据"，第一反应就是备份。
但复制和备份虽然都在做"多份"，目标并不一样。

#### 2.1 复制更偏在线可用性

复制强调的是：

- 线上系统持续可用
- 某台节点坏了，能尽快切走
- 多个副本一起承担查询

也就是说，它更像"正在执勤的多个人"。

#### 2.2 备份更偏灾难恢复

备份强调的是：

- 数据被误删了怎么办
- 整个库写坏了怎么办
- 我想恢复到昨天凌晨的状态怎么办

它更像"放在档案馆里的历史副本"。

#### 2.3 一个致命区分：复制会忠实地同步"错误"

复制通常**不能**代替备份。举几个你以后会踩的坑：

- 应用代码 bug 把 `salary` 字段改成 0，这条更新会**同步到所有副本**，几秒内全集群的工资都清零
- DBA 执行 `DROP TABLE orders;` 忘了 `WHERE`，这条 DDL 会同步到所有副本
- 勒索软件加密了主库数据，加密操作本身也会同步

这时候救你的**只能是备份**，因为备份在**某个时间点是"凝固"的**，它不会实时跟着错误走。

再说一个维度：
- 复制存的是"当前最新状态"
- 备份存的是"某个历史时间点的状态快照"

所以成熟团队的做法是：**复制保在线 + 定期物理备份 + 定期逻辑备份 + 定期恢复演练**。"3-2-1" 备份原则（3 份副本、2 种介质、1 份异地离线）就是工业界的经验。

### 3. 复制能带来什么好处

复制最核心的好处通常有四个。

#### 3.1 提升可用性

如果只有一台数据库，那么这台机器挂了，服务就可能直接不可用。
有副本后，主节点挂掉时，可以让另一个副本接替。

这就是复制对可靠性的直接帮助。

#### 3.2 提升读性能

很多业务是读多写少的。
例如商品详情、文章详情、用户主页，这类查询非常多。

如果只有一个节点来扛所有读取，很容易成为瓶颈。
复制后，可以把读请求分散到多个副本上。

#### 3.3 降低地域延迟

如果你的用户分布在不同地区，把副本放在更接近用户的地方，读取延迟通常会更低。

例如：
- 华东用户读上海机房副本
- 华南用户读广州机房副本
- 美西用户读 us-west 副本

**物理定律是硬约束**：光纤里光速约 20 万 km/s，中美跨洋 RTT 理论下限 ~130ms，实际 150–200ms。这个延迟你无法通过软件优化掉，只能把副本搬近。

#### 3.4 用于实时分析或冷备读

专门用一个"只读副本"挂接分析系统、搜索引擎或冷备：
- 实时分析副本跑重查询不影响主库
- CDC（Change Data Capture）把变更流到数据湖
- 搜索引擎从副本拉数据，不干扰 OLTP

这个模式把"复制"从"同构副本"扩展到**"异构数据流"**——是第 4 章"数据库解构"思想的延续。

### 4. 复制为什么又会变复杂

如果复制只有好处，那所有系统早就无脑上很多副本了。
真正难的地方在于：

**多份数据之间怎么保持一致。**

只要副本超过一个，就会出现下面这些问题：

- 新写入先到哪个副本？
- 其他副本什么时候才能看到这次写入？
- 某个副本落后了怎么办？
- 如果两个地方都改了同一条数据，谁算准？
- 故障切换时会不会丢最近几秒的数据？

所以复制的核心不是"拷贝几份"，而是：

**如何在多个副本之间传播更新，并处理传播过程中的延迟、故障和冲突。**

### 5. 复制日志的四种实现方式

在进入主从 / 多主 / 无主三大模式前，先搞清楚一个底层问题：**主节点的更新到底以什么形式传给副本？**

这个实现细节看起来"只是工程问题"，其实决定了你能用什么复制拓扑、会踩什么坑。

#### 5.1 基于语句（Statement-Based Replication）

最朴素的做法：主库把执行的 SQL 语句原样发给副本，副本再执行一遍。

MySQL 5.1 之前默认就是这个模式。它的致命问题：

- **非确定性函数**：`NOW()`、`RAND()`、`UUID()` 在主从两边执行结果不一样
- **自增 ID**：`AUTO_INCREMENT` 在并发插入下每台机器算出来的 ID 可能不同步
- **副作用**：触发器、存储过程、UDF 在复制时需要保证两边行为完全一致
- **顺序敏感**：两个并发事务执行顺序可能不同，结果不一样

结论：**除了简单场景基本不该用**。MySQL 5.1 之后默认切到了 ROW 格式。

#### 5.2 基于 WAL（Write-Ahead Log Shipping）

主库把自己的 WAL（Write-Ahead Log，预写日志）字节流原样发给副本，副本从日志里重放。Postgres 的流复制（streaming replication）就是这么干的。

优点：
- **极高保真**：同一份日志在两边产生同样的结果，没有语句级别的非确定性问题
- **性能好**：字节流传输，副本重放只做 I/O，不走 SQL 解析层

致命限制：
- **WAL 格式和存储引擎版本绑定**。主库升级 Postgres 大版本，副本必须同步升级。不能做"主库 PG14 副本 PG15"这种小版本滚动升级。
- **跨引擎不可能**：Postgres 的 WAL 给不了 MySQL 用。

这就是为什么 Postgres 复制很"硬"——它传的是引擎内部数据结构的变化，而不是逻辑语义。

#### 5.3 基于逻辑（行）复制（Logical / Row-Based Replication）

主库把"哪一行被改成什么样"的逻辑描述发给副本。这是今天的工业主流：

- MySQL binlog 的 ROW 格式
- Postgres 10+ 的 Logical Decoding / Logical Replication
- Oracle Streams / GoldenGate

它记录的不是"执行了什么语句"，也不是"WAL 字节"，而是：

```
Table: orders
Operation: UPDATE
Primary Key: order_id=1001
Before: {status: "paid", amount: 199}
After:  {status: "shipped", amount: 199}
```

这个格式的好处：

- 决策确定：没有非确定性问题
- 跨版本、跨引擎友好：可以做滚动升级
- 可以被第三方消费：**Debezium、Canal 这些 CDC 工具就是在读逻辑 binlog**，然后把变更流到 Kafka、ES、数仓

**CDC（Change Data Capture）是数据库解构时代的关键粘合剂**。它让数据库变成一个"发布事件的系统"，下游可以按需订阅。这在 Netflix、Uber、Shopify 的数据管道里都是基础设施。

#### 5.4 基于触发器（Trigger-Based）

在主库上加触发器，每次写入时把变更写入一张"变更日志表"，然后由外部程序读取并分发。

今天已经不太用，原因：
- 性能差（触发器侵入每次写入）
- 容易出 bug
- 工具链弱

但它有一个特殊用途：**跨异构数据库做复制**（比如 Oracle → MySQL），在 CDC 成熟前是唯一选择。Bucardo、Golden Gate 的早期版本用过这种方式。

#### 5.5 为什么要知道这四种

当你遇到下面问题时，知道底层实现就能立刻抓住要点：
- "为什么 MySQL ROW 格式的 binlog 比 STATEMENT 大得多？" → 因为记录了完整的前后镜像
- "Postgres 滚动升级能不能让副本先升级主库后升级？" → 不能，WAL 格式绑定
- "Debezium 凭什么能把 binlog 变成 Kafka 事件流？" → 因为逻辑复制是暴露接口的

### 6. 同步、异步、半同步：提交等几个副本

另一个底层决定：**主库在返回"写成功"之前，要等几个副本确认？**

| 模式 | 主库何时返回成功 | 故障时可能丢数据 | 写延迟 | 可用性 |
|---|---|---|---|---|
| 同步复制 | 至少一个副本确认持久化 | 0（理论上） | 高 | 任何副本故障都可能阻塞写 |
| 异步复制 | 主库本地持久化即可 | 有（几 ms 到几秒） | 低 | 主库单独不受副本影响 |
| 半同步复制 | 至少 1 个副本确认，其他异步 | 极少（但非零） | 中 | 折中 |

#### 6.1 为什么纯同步复制几乎没人用

纯同步复制意味着**每一次写操作都要等待所有副本的网络往返**。3 副本、跨机房 2ms RTT 的场景，每次写至少多 2ms。这听起来不多，但考虑：

- 一个副本暂时卡顿 → 主库也卡了
- 一个副本挂了 → 主库写入直接不可用
- 网络抖一下 → 写延迟长尾飙升

所以实践中"同步复制"几乎都是半同步：**至少有一个副本确认就返回**。这就能保证"主库不是孤家寡人"，但不会被最慢的副本拖死。

MySQL 有 `rpl_semi_sync_master_wait_for_slave_count` 参数控制等几个。Postgres 有 `synchronous_commit` 和 `synchronous_standby_names`。

#### 6.2 异步复制的真实成本：故障时数据丢失

异步复制的优势是主库极快，但代价明确：**主库突然挂掉时，没来得及传给副本的那部分写会丢**。

丢多少？取决于：
- 副本的复制延迟（lag）有多大
- 故障切换策略（强制切还是等复制追上再切）

一个典型数字：跨机房异步复制，正常情况下 lag < 100ms；**网络抖动或副本 IO 压力大时，可能瞬时飙到 10 秒**。这 10 秒的写如果主库此时挂了就全丢。

这就是为什么很多"金融级"系统要求至少半同步：宁可牺牲一点延迟，也要保证跨机房至少一个副本持有数据。

### 7. 最常见的复制方式一：主从复制

主从复制也常叫：

- Leader-Follower
- Primary-Secondary
- Master-Replica
- Active-Passive

它的基本思路是：

1. 一个主节点负责接收写入
2. 主节点把写入同步或异步传给从节点
3. 从节点通常负责读取

你可以把它想成一个"总账本 + 多个抄写本"：

- 所有正式修改先写总账本
- 然后再把内容抄到其他账本

#### 7.1 主从复制的优点

- 结构相对容易理解
- **写入入口单一，没有并发冲突**：这是最大的简化
- 读请求可以分散到从库
- 是几乎所有传统数据库（MySQL、Postgres、Oracle、SQL Server）、以及很多现代分布式数据库（MongoDB、Elasticsearch、Kafka、Redis）的默认模式

#### 7.2 主从复制的典型问题

- 主库压力可能大，因为所有写都要先进主库
- **从库会有延迟**
- 用户刚写完，马上去从库读，可能读不到最新数据
- 主库挂掉时，切主过程可能复杂、可能有脑裂风险

这就是我们常说的"主从延迟"问题。
它会导致一种很常见的现象：

**写已经成功了，但某些读取却还看不到。**

### 8. 复制延迟引发的三类一致性问题

这是 DDIA 这一章最值得刻进脑子里的部分。它告诉你"最终一致"不是一个形容词，而是**至少三类具体问题**，每一类都有典型症状和工程解法。

#### 8.1 读自己写（Read-Your-Writes / Read-After-Write）

**症状**：用户更新了自己的资料，刷新页面却看到旧资料。

为什么：写打到主库成功返回；紧接着的读打到一个还没同步的从库。

**为什么这个问题特别严重**：用户对"自己"的操作有强一致预期。看到"别人的"旧数据能忍，看到"自己刚写的"旧数据会觉得系统坏了。

**工程解法**：

1. **Sticky routing**：用户自己的会话在一段时间内把读请求路由到主库（或者到写入时连接的那个副本）。在 session 表里记一个"最近写入时间"，读请求在这个时间窗内走主库。
2. **按用户 hash 路由**：同一用户的读写固定到同一个副本。
3. **带时间戳的读**：客户端记住"我上次写成功的 replication log position"，读请求带上这个位置，路由层只把它发给至少追到这个位置的副本。

方案 3 最严谨，但实现复杂。MySQL 的 GTID（Global Transaction Identifier）就是为此而生。方案 1 最常用。

#### 8.2 单调读（Monotonic Reads）

**症状**：用户连续刷新两次页面，第一次看到数据 A，第二次反而看到比 A 更旧的数据 B。

为什么：第一次读打到一个已经同步的副本，第二次读打到一个还没同步的副本，时间上"倒流"了。

**为什么尤其烦人**：用户会怀疑自己是不是手抖、是不是系统错乱。

**工程解法**：
- **一个用户 pin 到同一个副本**：最简单，用 user_id 做 hash，路由到固定副本。
- **副本后追赶失败时，拒绝服务**：让客户端重试，而不是给出"倒流"的数据。

单调读是比"读自己写"弱的一致性保证，但在"时间线"类场景（消息列表、订单历史）里同样很重要。

#### 8.3 前缀一致读（Consistent Prefix Reads）

**症状**：用户看到的事件顺序错乱。经典例子：两人对话，先问"你吃了吗？"后答"吃了"。某个观察者看到的顺序是先"吃了"后"你吃了吗？"，像答非所问。

为什么：在分区数据库里，问题和答案可能在不同分区，两个分区独立复制，同步速率不同，造成观察者看到的跨分区顺序错乱。

**工程解法**：
- 因果相关的写入放进同一分区
- 使用**因果一致性协议**（causal consistency、版本向量）

这是比前两者更难的问题，涉及**跨分区的因果顺序**。DDIA 第 9 章会深入。

#### 8.4 最重要的意识：一致性不是二元的

DDIA 的一个核心洞察：**一致性不是"有"或"无"两个选项，而是一个光谱**。

从强到弱：

- 线性一致（Linearizability）：所有读写像单机一样
- 顺序一致（Sequential Consistency）
- 因果一致（Causal Consistency）：只保证因果关联的事件顺序
- 会话一致（Session / Read-Your-Writes / Monotonic）
- 最终一致（Eventual Consistency）：最弱

**越强的一致性，代价越高**（延迟、可用性、实现复杂度）。真实系统选哪个，取决于业务容忍度：

| 场景 | 典型需求 |
|---|---|
| 银行账户余额 | 线性一致 |
| 订单状态 | 读自己写 + 单调读 |
| 评论点赞数 | 最终一致 |
| 社交 feed | 因果一致（起码要看到别人的评论在 post 之后） |

### 9. 故障切换（Failover）：复制里最容易翻车的环节

主库挂了怎么办？理论上很简单：把一个副本提升为新主库，让客户端连新主库。实际上这是复制链条中**最容易造成数据丢失和系统瘫痪**的地方。

#### 9.1 故障切换的四大难题

1. **确认主库真的挂了**：网络抖动 vs 真死机，怎么区分？
2. **选谁当新主**：如果副本之间有 lag 差异，应该选最新的那个（减少数据丢失）还是最稳定的那个？
3. **通知客户端**：所有客户端什么时候知道有了新主？期间写到老主库的数据怎么处理？
4. **防脑裂（split-brain）**：老主库还没真挂，新主库已经选出来了。两边都在接写。

这四个问题里，**脑裂是最致命的**。

#### 9.2 GitHub 2018 年的 24 小时脑裂事件（深度复盘）

这是教科书级别的案例。

**时间**：2018 年 10 月 21 日 22:52 UTC。

**触发**：GitHub 美东（主）和美西（备）数据中心之间的网络分区，持续 43 秒。

**自动故障切换的决策**：GitHub 用了 **Orchestrator**（MySQL 故障切换工具）。Orchestrator 在 43 秒后检测到美东主库"不可达"，触发切换：美西副本提升为新主。

**问题出现在网络恢复后**：
- 老主库（美东）以为自己还活着，上面已经有了 43 秒内接到的新写入
- 新主库（美西）也开始接受新写入
- 两边写入的是**相互不知道的数据**
- 部分应用仍然连老主库（因为 DNS/路由没完全切过来）

**后果**：几小时内两边都写了，数据分叉。GitHub 最后决定：

- 回滚到老主库，废弃新主库在这几小时里的写入
- 对用户公布"我们丢失了这段时间的某些数据"
- **花了 24 小时手动合并、对账、通知受影响的 PR 和 issue**

**这次事故里的关键教训**：

1. 网络分区是**部分的、短暂的**，不是干净的断电。判断主库死没死非常难。
2. Orchestrator 选主时没有 fencing 机制：它只能"告诉新主'你是主了'"，但没法"强制老主停下来"。
3. 异步复制意味着：故障切换时新主一定比老主旧，你永远在选"丢多少"。

事后 GitHub 改了架构：**不再做完全自动的故障切换，引入人工确认步骤**。这个选择本身也是一种权衡——牺牲部分 MTTR 换彻底消除脑裂风险。

#### 9.3 防脑裂的几种工程手段

**1. Fencing Token**：新主被选出来时获得一个**单调递增的 token**（比如 epoch number）。所有对存储的写入都要带 token，存储端拒绝 token 小于最新值的写入。这样即使老主库复活，它的 token 已经过期，写不进去。

这个思路在共识协议（Paxos、Raft）里是标配，**Chubby、ZooKeeper 的 session ID 也是这个思想**。

**2. STONITH（Shoot The Other Node In The Head）**：新主被选出来时，**主动关掉老主库**（通过 IPMI、PDU 断电、关机命令）。这是高可用集群软件（Pacemaker、Corosync）的经典做法。粗暴有效。

**3. Quorum-based leader election**：要选主，必须得到多数派副本同意。网络分区时少数派那边选不出主，自动"禁音"。这是共识算法（Raft、Paxos）的核心思想，第 9 章会讲透。

**4. 见证节点（Witness / Tiebreaker）**：在两机房部署时加第三方的小节点，它本身不存数据，但参与投票。避免两机房各 50%、无法决出多数的情况。

**5. 手动 failover**：最保守，最不容易脑裂，MTTR 最长。适合"宁可多停 30 分钟也不丢数据"的场景（比如金融）。

#### 9.4 选择自动还是手动切换

这是一个永恒争论：

- **自动**：MTTR 短（秒级），但脑裂风险和错误切换风险更高
- **手动**：MTTR 长（分钟到小时级），但决策质量更高

经验上：

- 非关键业务 + 多副本 + 明确共识算法 → 自动
- 金融、核心交易 → 经常倾向手动或半自动（自动故障检测 + 人工确认 + 一键执行）

工具链：MHA、Orchestrator、Patroni、Amazon RDS Multi-AZ、MongoDB Replica Set Election、Vitess 的 `vtorc`。每个都有自己的 fencing 策略和局限。

### 10. 第二种复制方式：多主复制

多主复制的思路是：

**不只有一个主节点，多个节点都可以接收写入。**

例如两个机房各有一个主节点：

- 上海能写
- 北京也能写

然后它们互相同步更新。

#### 10.1 多主复制的典型用例

- **多数据中心部署**：每个 DC 本地写入避免跨洋延迟
- **离线客户端**（协作编辑、移动 App）：本地写入，联网后合并
- **实时协同编辑**（Google Docs、Figma 底层）
- **异构系统间双向同步**（比如 CRM 和 ERP 双向同步）

#### 10.2 多主复制的拓扑

- **环形（Circular）**：A → B → C → A。MySQL 5.5 以前多主用过。问题：一个节点挂了整条链断。
- **星形（Star）**：一个中心节点连所有其他节点。简单但中心节点是单点。
- **全对全（All-to-All）**：每个节点连所有其他节点。最健壮也最复杂。

#### 10.3 冲突问题：多主复制的阿喀琉斯之踵

一旦多个地方都能写，就会出现冲突：

- 节点 A 把用户名改成 `Alice`
- 节点 B 同时把用户名改成 `Alicia`

最终保留哪个？

**冲突检测**本身就已经不简单。不同节点的时钟不同步，你不能简单说"谁时间晚谁赢"。

#### 10.4 冲突解决方案

**方案 1：Last Write Wins（LWW）**
选"时间戳最大"的写。简单，但**时间戳不可靠**：
- 不同节点时钟不同步（NTP 误差 10–100ms）
- 时钟可能回拨（leap second、管理员手动改）
- 同一毫秒内两次写随机决定谁赢

**LWW 的后果**：部分写会**静默丢失**。Cassandra 的一个经典坑：生产环境时钟漂移了 5 秒，在这 5 秒里所有写入都"失败了"——不是报错，是被后面一个看起来更早的时间戳盖掉了。

**方案 2：版本向量（Version Vector / Vector Clock）**
每个节点维护自己的写入计数器，更新时带上所有节点的计数。合并时可以机器化地判断"是同一个版本被并发修改"还是"B 是 A 的后续版本"。能准确**检测**冲突，但冲突还是需要应用层决定怎么合并。

**方案 3：CRDT（Conflict-free Replicated Data Types）**
从数学上保证"任何顺序合并结果都一致"的数据结构。典型：
- **G-Counter**：单调递增计数器
- **PN-Counter**：可增可减的计数器
- **OR-Set**：添加和删除都能正确合并的集合
- **LWW-Register**、**Map**、**Sequence**

Figma 的实时协作、Redis Enterprise CRDB、Riak 2.0+、Automerge、Yjs 都用 CRDT。CRDT 的限制是：**它只能表达某些数据结构**，你的业务语义必须能映射到 CRDT 原语。

**方案 4：Operational Transformation (OT)**
Google Docs 经典算法。允许并发编辑，通过"变换"使不同顺序到达的操作在每个节点都产生一致结果。比 CRDT 早，但实现复杂。

**方案 5：应用层自定义合并**
系统检测到冲突后保留所有版本，让应用决定。Riak 的"siblings"、CouchDB 的冲突文档、Git 的 conflict marker 都是这个路子。灵活，但责任推给了应用开发者。

#### 10.5 为什么很多核心交易系统不上多主

答案在"一致性 vs 可用性"的取舍里。多主能做到**多机房都能写 + 本地写快 + 断网也能继续写**，代价是**冲突和一致性复杂度陡增**。

核心交易（订单、支付、库存）的要求是**正确性优先**，宁可偶尔不可写，也不能冲突导致业务状态错乱。所以这些场景通常用**单主 + 多副本 + 自动/半自动切换**，或者**共识协议**（Raft、Spanner 的 TrueTime + Paxos）。

社交、协作、IM 这类场景，可用性和本地写入体验更重要，多主+CRDT 是合适的。

### 11. 第三种复制方式：无主复制（Dynamo 式）

无主复制的思路更进一步：

**没有一个固定主节点，客户端（或协调者）把写请求并行发给多个副本。**

这个思路来源于 **Amazon 2007 年发表的 Dynamo 论文**，被 Cassandra、Riak、Voldemort 广泛采纳。Kleppmann 本人深度参与过这个时代的工作。

#### 11.1 基本机制

假设 N 个副本（比如 3）：

- 写入：客户端**并行**发给全部 N 个副本，收到 W 个确认就算成功
- 读取：客户端**并行**发给 N 个副本，收到 R 个响应后合并（取最新版本）

**法定人数（quorum）公式**：只要 W + R > N，读写一定至少有一个副本重叠，读能读到最新写。

常见配置（N=3）：
- W=2, R=2 → 任意 2 个副本同意即可，1 个副本故障不影响服务
- W=3, R=1 → 写慢读快
- W=1, R=3 → 写快读慢

#### 11.2 版本冲突与读修复

无主复制里**没有单一"写顺序"**，不同副本可能看到不同顺序的写入。所以：

- 每次写带一个版本号（vector clock 或逻辑时钟）
- 读取时如果多个副本返回不同版本，客户端合并
- 客户端发现某副本数据旧，回写新版本到它（**read repair**）

#### 11.3 反熵（Anti-entropy）：后台对账

读修复只能修复"被读到"的数据。冷数据可能长期在某个副本上不一致。解决方法是**定期运行对账进程**，扫描所有副本找差异。

这里有个经典数据结构：**Merkle Tree**。把每个副本的数据分块哈希，组成一棵哈希树。对账时只比较根节点：

- 根一样 → 全部一致，不用再比
- 根不一样 → 递归比对子树

Merkle Tree 让对账的通信量从 O(n) 降到 O(log n)。**Cassandra、Dynamo、比特币、IPFS、Git 都用 Merkle Tree**，只是各自的业务场景不同。

#### 11.4 宽松法定人数（Sloppy Quorum）与 Hinted Handoff

如果某个副本所在节点故障，写入凑不齐 W 个怎么办？

**严格法定人数**：写入失败，整体不可用（牺牲可用性保一致）。

**宽松法定人数**：暂时把写入给"邻居节点"（不属于这个 key 原本的 N 个副本），让它们**代为保管（hinted handoff）**。故障节点恢复后，邻居把"寄存"的数据还给它。

**代价**：宽松法定人数破坏了 W+R>N 的严格保证——寄存期间，真正的副本可能没拿到新写入，读到旧数据的风险上升。

**Cassandra 默认用宽松法定人数**，优先保可用性。这也是为什么 Cassandra 的官方文档反复提醒"一致性是调优问题，不是默认保证"。

#### 11.5 为什么无主复制没能成为主流

2007 Dynamo 论文引发的浪潮在 2010 年代初达到顶峰，之后逐渐退潮。原因：

- **应用开发者对"最终一致"不熟**，很多 bug 来自假设了强一致性
- **LWW 带来的数据丢失**很难 debug
- **事务支持弱**：很多 NoSQL 后来才加上事务
- **Spanner 证明了强一致也能扩展**（靠 TrueTime + Paxos）
- **NewSQL 崛起**（CockroachDB、TiDB）提供了单机 SQL + 水平扩展 + 强一致

**今天的工业现实**：纯无主系统越来越少。Cassandra 仍活跃，但多用于"可以接受最终一致"的场景（IoT、时序、推荐数据）。核心业务更多选择**单主 + 共识**（Spanner 式、Raft 式）。

但 Dynamo 的思想没死——**DynamoDB 本身就是 Dynamo 论文思想的产品化演进**，只是它对外提供的是"按分区的单主"视图，内部的多副本机制仍在。

### 12. 用订单系统看复制的选型决策

假设你在做电商订单系统。核心问题不是"要不要复制"，而是"哪里上、上什么类型"。

**核心库存、订单、支付（强一致关键）**：
- 单主 + 半同步复制
- 跨机房部署，同机房 2 副本同步、跨机房 1 副本异步
- 故障切换半自动（监控自动触发 + 人工确认）
- 正确性第一，MTTR 可以接受分钟级

**商品详情、商品搜索（读量大、容忍短暂旧读）**：
- 主从复制 + 多只读副本
- 异步复制没问题
- 用户刚改自己商品的描述 → sticky session 路由到主库避免"改完刷新没生效"

**用户购物车（需要跨设备、跨地域）**：
- 可以考虑多主 + CRDT（集合类数据结构天然适配）
- 手机端断网也能改，联网后自动合并

**推荐、点赞数、阅读量（弱一致）**：
- Cassandra / ScyllaDB 这类无主系统
- 最终一致足够，极高写吞吐

**跨境跨地域场景（亚洲 + 美洲）**：
- 近用户的区域自治 + CRDT 或异步复制
- 每区域自己做主从
- 全局视图通过数据流（CDC → Kafka → 全局数仓）聚合

这就是 DDIA 教给你的思路：**不是"主从 vs 多主 vs 无主哪个好"，而是"这个具体数据流，能容忍什么、需要什么"**。

### 13. 一致性 / 可用性 / 延迟：真正的三角

CAP 定理（Brewer 2000, Gilbert & Lynch 2002）说："网络分区时，一致性和可用性只能二选一"。这个定理是对的，但**它对实践的指导意义有限**，因为：

1. 网络分区在单一集群内其实很少真正发生（同机房内），CAP 强制二选一的场景其实是边缘情况
2. 没有网络分区时，系统仍然在权衡

Daniel Abadi 提出了更实用的 **PACELC**：
- 有分区（Partition）时：在可用性（A）和一致性（C）间选
- 否则（Else）：在延迟（L）和一致性（C）间选

PACELC 揭示了一个更重要的事实：**绝大多数时候系统权衡的不是 CAP，是延迟 vs 一致性**。

这就是为什么异步复制如此普遍——为了降延迟，放弃了"写立刻在副本可见"的一致性。**这个权衡每个写操作都在做，不仅仅是分区时**。

| 系统 | PAC 倾向 | ELC 倾向 |
|---|---|---|
| Dynamo/Cassandra | PA | EL |
| MongoDB（默认） | PA | EL（近年改变） |
| BigTable/HBase | PC | EC |
| Spanner | PC | EC |
| VoltDB | PC | EC |

### 14. 复制不是"要不要上"，而是"愿意接受什么代价"

这一点很重要。

几乎所有真实系统都会在某种程度上使用复制。
问题通常不是"要不要复制"，而是：

1. **用哪种复制方式？**（主从 / 多主 / 无主）
2. **同步还是异步？**（延迟 vs 丢失窗口）
3. **复制日志用什么格式？**（语句 / WAL / 逻辑行）
4. **允许多大的复制延迟？**
5. **读请求可不可以接受短暂旧数据？**
6. **故障切换时可不可以接受极短的数据丢失风险？**
7. **脑裂怎么防？fencing？STONITH？quorum？**
8. **冲突怎么解决？**（LWW / 版本向量 / CRDT / 应用层）

这些问题的答案，是复制方案设计的真正内容。

## 小结（6 条关键点）

- **复制不等于备份**：复制是"多人执勤"保在线可用性，备份是"档案副本"救灾难和回滚；复制会忠实同步错误操作，所以两者不能互相取代。
- **复制日志的四种实现**（语句、WAL、逻辑行、触发器）决定了能做什么——逻辑行格式是今天的工业主流，也是 CDC 管道的基础；"数据库解构"在复制层面的落地就是逻辑日志。
- **同步 vs 异步的核心代价是"故障时丢多少"**：纯同步几乎没人用，半同步是主流；异步下"跨机房 lag 可能瞬时飙到 10 秒"是工程常态。
- **复制延迟不是一个问题，是三类问题**：读自己写、单调读、前缀一致读，每类都有对应症状和工程解法；"一致性不是二元的，是光谱"是 DDIA 最重要的洞察之一。
- **故障切换最危险的敌人是脑裂**：GitHub 2018 用 43 秒网络分区换了 24 小时手动修复；工业界的防线是 fencing token、STONITH、quorum-based leader election 和见证节点，根本上要靠共识算法（第 9 章）。
- **多主和无主是"本地写入、可用性优先"的选项**：多主依赖 LWW/版本向量/CRDT 解决冲突（LWW 在时钟漂移下是数据杀手）；无主靠 W+R>N 法定人数、读修复和反熵（Merkle Tree）；但对正确性要求高的核心业务，单主+共识仍是主流。
