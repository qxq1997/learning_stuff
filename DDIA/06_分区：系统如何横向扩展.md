## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么学完复制，下一步一定要学分区

复制和分区经常一起出现，所以很多人刚开始会把它们混在一起。
但它们其实在解决两类不同的问题：

- **复制**：同一份数据放多份，主要解决可用性、读扩展、异地访问
- **分区（partitioning / sharding）**：把不同数据拆开存，主要解决**写扩展、存储扩展、负载拆分**

你可以先用一个很实用的口诀记住：

- **复制是"同样的数据放多台"**
- **分区是"不同的数据放不同台"**

例如一个订单系统：

- 复制是在说：订单 1001 这条数据除了主库，还可以在副本库上也有一份
- 分区是在说：订单 1 到 1000 可能在机器 A，订单 1001 到 2000 可能在机器 B

这两件事听起来都像"多台机器"，但本质不同。

#### 1.1 术语地狱：同一件事，每家叫法不同

分区这个话题在不同系统里术语差异极大。第一次接触时非常容易被混淆：

| 系统 | 分区的叫法 |
|---|---|
| MongoDB / Elasticsearch | **Shard** |
| HBase / BigTable | **Region** / **Tablet** |
| Spanner / CockroachDB | **Tablet** / **Range** |
| Cassandra / Riak | **Partition** / **Virtual Node (vnode)** |
| Kafka | **Partition** |
| Couchbase | **vBucket** |
| DynamoDB | **Partition** |
| Vitess | **Shard** / **Keyspace** |
| Redis Cluster | **Slot** / **Hash Slot**（共 16384 个） |

DDIA 统一用 **partition**。你在工作中看到 shard、tablet、region、slot 时，记住它们本质上指的是**同一个东西：一份数据的一个片段**。

#### 1.2 水平分区 vs 垂直分区

DDIA 主要讨论的是**水平分区（horizontal partitioning）**——按行切开，每个分区含一个行子集。

但还有一个常被忽视的兄弟：**垂直分区（vertical partitioning）**——按列切开。Facebook 早年把用户的主档案表和用户的大字段（私信历史、动态 feed）拆成不同库，就是垂直分区。

微服务架构本质上也是一种垂直分区：**按业务能力切分数据库**（订单库、用户库、库存库）。只是它在服务边界同时拆分了计算。

水平和垂直通常一起用：先按服务拆库（垂直），再在库内按 key 拆分区（水平）。

### 2. 为什么只有复制还不够

上一课我们讲过复制可以把读流量分散出去。
但如果你把所有写请求仍然压在一个主节点上，就会出现一个很现实的问题：

- 读可能扛住了
- 但写还是单点瓶颈

而且不只是写请求，单机还有其他天然上限：

- **磁盘容量**：最大企业级 SSD 约 30TB 单盘，但随机 I/O 和并发受限
- **内存**：2TB 是典型单机上限，热数据工作集塞不下就要换出
- **网卡带宽**：25GbE/100GbE 单机
- **CPU**：核数有天花板，垂直扩 CPU 的成本非线性
- **索引维护**：单表几百亿行后，B-Tree 深度、锁竞争、备份窗口都成问题

所以，哪怕你已经做了主从复制，系统也可能仍然扛不住：

- 每秒几十万写入
- 几十 TB 的数据量
- 某些特别集中的热点访问

这时候只靠"再加几个副本"往往不够，因为副本只是多了几份同样的数据，不会减少主写节点的根本压力。

这就是为什么系统要分区。

#### 2.1 从"一台很大的机器"到"一堆普通机器"：shared-nothing 哲学

Kleppmann 在这一章暗含一个哲学立场：**shared-nothing architecture**。

- **共享内存（shared-memory）**：多 CPU 共享同一块内存。扩展受限于单机硬件，昂贵。
- **共享磁盘（shared-disk）**：多机器共享一个 SAN/NAS。扩展到一定程度磁盘会成瓶颈。Oracle RAC 是代表。
- **共享无（shared-nothing）**：每台机器有自己的 CPU、内存、磁盘；节点通过网络协作。

**shared-nothing 是水平扩展最彻底的形态**，也是 Google、Amazon、Facebook 在 2000 年代早期就下定决心走的路。它的硬代价是：一切协调必须走网络，分布式一致性问题全部浮出水面。但它是目前我们已知的、扩展性最好的选择。

分区 + 复制 + 共识算法，就是 shared-nothing 的三大支柱，分别对应 DDIA 的第 5、6、9 章。

### 3. 什么是分区

分区可以简单理解成：

**把一大份数据拆成很多小块，每块放到不同节点上。**

这和你把一整个仓库拆成很多库房有点像。

如果所有货都堆在一个库房里，问题会很多：

- 空间不够
- 出货口会堵
- 一着火影响全部

如果你把货分到多个库房：

- 空间更容易扩展
- 不同库房可以并行处理
- 整体吞吐会提升

数据系统里的分区，本质上就是在做这件事。

### 4. 分区主要在解决哪三个问题

#### 4.1 存储容量不够

如果所有数据都放在一台机器上，数据量大到一定程度后，磁盘和内存就扛不住了。
把数据拆到多台机器，整体容量就能上去。

#### 4.2 写入吞吐不够

很多场景不是读多写少，而是写非常多：

- 日志写入
- 用户行为上报
- 订单创建
- 物联网设备上报

如果所有写都进同一个节点，很快就会打满。
分区后，不同数据可以同时写到不同节点，吞吐就能横向扩展。

#### 4.3 负载太集中

哪怕总流量不算天文数字，如果请求都集中打到某一小部分数据，也会把局部打爆。
合理的分区策略可以帮助你把流量更均匀地摊开。

但要注意：**不合理的分区也可能让这个问题反而变严重**。后面 7–9 节专门讨论。

### 5. 两种最基础的分区思路：按 key 范围、按哈希

这部分是分区里最核心的入门知识。

#### 5.1 按 key 范围分区

范围分区就是：

**按照某个键的连续区间来拆分数据。**

例如：

- `order_id` 1 到 100 万在分区 A
- `order_id` 100 万到 200 万在分区 B

或者按时间：

- 2026 年 1 月的数据在分区 A
- 2026 年 2 月的数据在分区 B

**代表系统**：HBase、BigTable、MongoDB（范围模式）、RethinkDB、Spanner。

##### 5.1.1 范围分区的优点

- **范围查询很自然**：`BETWEEN ? AND ?` 直接打到少数几个分区
- 按时间、按 ID 扫描会比较方便
- 对有序数据的处理较友好（时间序列、日志、消息）

例如你要查：
- 最近 7 天订单
- 某个时间段的日志
- 某个区间内的交易记录

范围分区通常很顺手。

##### 5.1.2 范围分区的致命缺点：尾部热点

比如按时间分区时，最新一天的数据往往最热。
**所有新写入都打到最新分区**，那个分区就特别容易成为瓶颈。

这叫"尾部热点"，在时序类数据里几乎是默认会发生的事。

HBase 早期就因此吃过大亏：用户表按 `user_id` 范围分区，`user_id` 是自增的，**所有新注册用户都写到最后一个 region**，导致那个 region server 长期被打爆。

**工业解法**：
- 在 key 前加**哈希前缀**："2026-04-17#orders" → "a3#2026-04-17#orders"
- 用**复合键**：`(geo_zone, timestamp)` 让不同区域的数据落到不同分区
- **动态 region 分裂**：BigTable/HBase 监控 region 大小，超过阈值自动分裂

##### 5.1.3 动态分区分裂（Dynamic Partitioning）

BigTable 和 HBase 的 region 是动态的：

- 初始一个 region 覆盖整个 key 空间
- 数据写入增多，region 增大
- 超过配置的阈值（例如 10GB），region 自动分裂成两个
- 反过来，合并（compaction）过程中太小的 region 也可以合并

这个机制让运维从"手动规划分区边界"中解脱出来。但代价是：

- 分裂和合并本身是昂贵操作，期间性能有抖动
- 在写入极快的热分区，"分裂不够快"仍然会导致持续过载

**MongoDB 和 CockroachDB 也都采用这类动态分裂思想**。这是 2010 年代之后的主流。

#### 5.2 按哈希分区

哈希分区就是：

**先对 key 做哈希，再根据哈希结果决定去哪个分区。**

最朴素的实现：

```
partition = hash(key) % N
```

##### 5.2.1 哈希分区的优点

- 分布通常更均匀（好的哈希函数让不同 key 的映射接近随机）
- 不容易因为 ID 连续增长而把所有新写入打到同一台机器
- 更适合追求负载平均摊开的场景

##### 5.2.2 哈希分区的缺点

- **范围查询变难**：相邻 key 被打散到不同分区，`BETWEEN ?` 查询必须 scatter-gather
- **再平衡代价大**（见下节）

所以哈希分区常见于：
- 单 key 点查为主
- 希望尽量均匀分散写入和读取
- 不强依赖范围查询

##### 5.2.3 为什么 "hash mod N" 在生产里几乎不能用

新手常想当然："直接 `hash(key) % N` 不就好了？"

**致命问题**：N 变化时，几乎所有 key 的分区都会变。

例子：原本 4 个分区，用 `hash(key) % 4` 分布。扩容到 5 个分区时：
- 原来 `hash=10` 的 key 在 `10 % 4 = 2` 号分区
- 扩容后应在 `10 % 5 = 0` 号分区

几乎所有 key 都要搬家。这意味着扩容时要重新 shuffle 几乎**全部数据**，整个集群不可用的时间可能是几小时甚至几天。这在生产中是不可接受的。

所以工程上需要一套更好的哈希映射方式。

#### 5.3 一致性哈希（Consistent Hashing）：Akamai 的贡献

**一致性哈希**是 MIT 1997 年 David Karger 的论文提出的，最初用于 Akamai 的 CDN。2007 年 Dynamo 论文把它带进数据系统主流。

##### 5.3.1 基本思路

1. 想象一个**环**：从 0 到 2^32 - 1
2. 把每个节点通过 hash(node_id) 映射到环上某个点
3. 把每个 key 通过 hash(key) 映射到环上某个点
4. key 归属于**顺时针方向最近的那个节点**

```
        0
        |
     Node A
    /
   /   <- key1 在这里，归 A
  /
Node D ---- Node B
  \         /
   \       /
    Node C
```

##### 5.3.2 它解决了什么问题

**扩容时只搬部分数据**：新增 Node E 到环上某个位置，只有落在"E 和它逆时针前邻居之间"的 key 需要从某一个旧节点搬到 E。

**平均只搬 1/N 的数据**，而不是 (N-1)/N。这是一致性哈希的数学本质。

##### 5.3.3 虚拟节点（Virtual Nodes / Vnodes）

朴素的一致性哈希有个新问题：**数据分布不均**。如果某个节点正好落在环上一个稀疏区域，它负责的 key 范围就大，压力失衡。

解决方案：**每个物理节点映射到环上的 K 个虚拟节点**（例如 K=256）。这样每个物理节点负责的总范围更接近平均值。

更妙的是，当增减节点时：
- 新增节点：256 个 vnode 均匀插入环上，从现有节点"各偷一点"过来
- 删除节点：它的 256 个 vnode 均匀分给其他节点

**Cassandra、ScyllaDB、Riak、DynamoDB 都用 vnode**。Cassandra 默认 256 个 vnode/节点。Redis Cluster 的 16384 个 hash slot 是另一种"固定虚拟节点数"的实现。

##### 5.3.4 注意：Kleppmann 的一个澄清

DDIA 里 Kleppmann 专门澄清："一致性哈希"这个词在业界用得很混乱，不同系统的"一致性哈希"实现细节差别很大。**Cassandra 和 Riak 的 "一致性哈希" 实际上是加 vnode 后的变体**，和 1997 年原版有差异。

另外，Kleppmann 指出：**很多实践中所谓的"一致性哈希"其实并不涉及 Karger 那个环**，而只是取了"移动最少数据的哈希分配"这个语义。所以看到这个词时要去看具体实现。

##### 5.3.5 跳跃哈希（Jump Consistent Hash）

2014 年 Google 在论文《A Fast, Minimal Memory, Consistent Hash Algorithm》里提出了一个极简的替代方案：

```python
def jump_hash(key, num_buckets):
    b, j = -1, 0
    while j < num_buckets:
        b = j
        key = key * 2862933555777941757 + 1
        j = int((b + 1) * (1 << 31) / ((key >> 33) + 1))
    return b
```

短短 7 行，性能比一致性哈希更好，不需要维护环，不需要 vnode。唯一限制：节点数只能从末尾增减（不能随意删除中间节点）。

**适合分区数严格递增/递减的场景**。Vitess、TiKV 部分场景使用。

##### 5.3.6 会合哈希（Rendezvous Hashing / Highest Random Weight）

1996 年 David Thaler 提出，比一致性哈希更早，但关注度低很多。思路：

- 对每个 (key, node) 组合算一个权重：`weight = hash(key, node_id)`
- key 归属于权重最大的那个 node

优点：天然均衡、扩缩容只搬 1/N、无 vnode 复杂度。
缺点：每次查找要算所有 node 的 hash（但 N 不大时代价可忽略）。

**Akamai、Apache Druid、Ceph 的 CRUSH 算法都用这个思路**。某些场景下比一致性哈希更优。

### 6. 按 key 哈希 vs 按 key 范围的选择：一张表总结

| 维度 | 范围分区 | 哈希分区 |
|---|---|---|
| 范围查询 | 极友好 | 需要 scatter-gather |
| 写入分布 | 易出尾部热点 | 均匀 |
| 动态扩缩 | 动态 split/merge 机制 | 一致性哈希 / 跳跃哈希 |
| 代表系统 | HBase、Spanner、MongoDB 范围模式 | Cassandra、DynamoDB、Redis Cluster |
| 典型场景 | 时序、日志、按时间查询 | 用户点查、KV 缓存 |

**混合方案：复合键（compound key）**

很多系统支持**复合分区键**：第一个字段做哈希分区，第二个字段在分区内按范围排序。

例如 Cassandra 的主键 `((user_id), created_at)`：
- `user_id` 做分区（均匀分布到所有节点）
- 在每个 user 内部 `created_at` 排序（支持该用户的范围查询）

这样"查某用户最近 30 天的记录"只打到一个分区，且内部是有序扫描。两者优点都得到。

这是 Cassandra、DynamoDB 这类系统非常标志性的数据模型。

### 7. 热点和倾斜：分区最残酷的现实

理论上你把数据拆开了，负载应该更均匀。
但现实里，数据访问往往**极度不均衡**。

#### 7.1 真实世界里的 Zipf 分布

人类活动产生的数据访问**几乎总是倾斜的**。典型规律：

- 推特关注数分布：极少人有上亿粉丝，大多数人只有几十粉丝
- 电商商品访问：爆款占总访问的 80%
- 视频平台：头部 0.1% 的视频占 50% 的播放量
- 游戏：头部主播房间并发百万，普通房间几人

这就是 **Zipf 分布**（幂律分布）。如果分区键分布也是 Zipf 的，均匀哈希也救不了你——每个分区内部仍然存在热点。

#### 7.2 Twitter 的 Justin Bieber 问题

Twitter 早期有个著名的工程问题叫 "Justin Bieber problem"：Justin Bieber 发一条推，要扇出到几千万粉丝。无论怎么分区，这个操作都会集中打到"存 Bieber 的推文"的那个分区，以及"粉丝 feed"所在的各个分区。

Twitter 最后的方案（见第 2 章提过）：**名人用户走 fan-out-on-read，普通用户走 fan-out-on-write**。这本质上是**为 Zipf 分布的两端采用完全不同的路径**。

这种"对热 key 特殊对待"的思路在很多系统里都出现：
- Pinterest：热门 board 单独缓存层
- YouTube：热门视频进 edge CDN
- Netflix：超热内容预加载到所有 region

#### 7.3 实用热点处理方法

**方法 1：Key 前缀加随机数（Salting）**

把热 key 拆成多个子 key：原本所有人访问 `post:justinbieber`，现在访问 `post:justinbieber_salt_0` ~ `post:justinbieber_salt_99`，100 个副本分散到不同分区。读取时需要聚合 100 份结果。

代价：读要聚合；写有冗余。但把极热 key 完全摊平了。

**方法 2：应用层缓存**

热 key 基本只读 → 用 Redis/Memcache 挡在前面。代价：可能吃到缓存击穿、缓存雪崩。而且**缓存本身也会热**（Redis 单分片最多扛几十万 QPS）。

**方法 3：专属热 key 分区**

监控系统识别出超热 key，把它们单独放到一个"热 key 分区"里，这个分区用更强的硬件或更多副本。**DynamoDB 的 adaptive capacity** 就是类似思路——动态识别热 partition 并给它更多配额。

**方法 4：应用分片（Sub-sharding）**

直播/游戏房间这类场景，房间号本身就是 key，但单个房间可能有百万用户。解法：把一个房间内部再拆成多个 "sub-channel"，消息随机打到其中一个，读时合并。本质上是 key 空间被主动细化。

#### 7.4 Foursquare 2010 事故：MongoDB shard 倾斜

2010 年 10 月，Foursquare 全站宕机了 17 小时，根因是一个经典的分区倾斜事故：

- MongoDB shard 2 达到了 67GB，塞满了 66GB 内存的那台机器
- 一旦工作集不能全部进内存，性能断崖式下跌（磁盘随机 I/O 是内存的几万倍慢）
- 重启和迁移都需要"先把数据倒出来"，但数据一落磁盘就变成更慢的操作
- 最后花了 17 小时才重建到可用状态

**教训**：
1. 分区大小要监控，接近内存就要提前扩
2. 单个分区塞满内存是性能断层点，不是线性劣化
3. 运维工具（分区迁移、重建）必须在健康时演练

### 8. 分区上的二级索引：两条完全不同的路

这是分区里经常被忽视但至关重要的话题。

假设你有订单数据，按 `order_id` 哈希分区。现在要支持查询 `WHERE user_id = ?`——`user_id` 是二级索引。在分区系统里怎么做？

#### 8.1 路线 1：本地索引（Document-Partitioned Index）

每个分区内**各自维护**自己的 `user_id` → `order_id` 倒排。

- 写入：订单写到一个分区，更新该分区的本地索引。**一次写只影响一个分区**。
- 查询 `WHERE user_id = ?`：不知道命中哪些分区，只能**查所有分区（scatter）+ 合并结果（gather）**。

**代表系统**：MongoDB、Cassandra、Elasticsearch、Riak。

**关键代价**：Scatter-gather 查询的延迟被**最慢分区**决定。这就是第 2 章讲的尾延迟在这里的直接体现——查 10 个分区，每个 P99=100ms，整体 P99 很可能是 200ms+。

#### 8.2 路线 2：全局索引（Term-Partitioned Index）

**索引本身是一个独立分区的数据结构**。`user_id` → `order_id` 的映射按 `user_id` 哈希分区。

- 写入：订单数据落一个分区，但要**跨分区更新全局索引**（因为索引分区可能在别的机器上）。
- 查询：按 `user_id` 算出索引在哪个分区 → 一次读取拿到所有 `order_id` 列表 → 再去订单分区读详情。

**代表系统**：DynamoDB 的 GSI（Global Secondary Index）、某些新型 NewSQL。

**关键代价**：写入涉及多分区 → 要么异步（索引和主数据短暂不一致），要么分布式事务（慢且复杂）。

#### 8.3 两者对比

| 维度 | 本地索引 | 全局索引 |
|---|---|---|
| 写入 | 一次只改一个分区，快 | 跨分区写，慢或异步 |
| 查询 | Scatter-gather，P99 看尾延迟 | 直接命中一个分区，快 |
| 实现复杂度 | 简单 | 需要分布式事务或容忍延迟 |
| 代表 | ES, Cassandra, Mongo | DynamoDB GSI |

**实践经验**：
- 索引少、查询频繁 → 全局索引更合适
- 索引多、更新频繁 → 本地索引更合适
- 时间允许短暂不一致 → 全局异步索引
- 强一致要求高 → 本地索引 + scatter-gather

### 9. 请求路由：数据分散后，怎么找到正确的那片

数据分散到多台机器后，应用还要知道：

**这次请求应该发到哪一台？**

#### 9.1 三种实现思路

**思路 1：智能客户端（Client-side routing）**

客户端自己知道分区规则，直接计算 `partition = hash(key) % N`，直连目标节点。

- 优点：一跳，延迟低
- 缺点：客户端要维护分区元数据；规则变更所有客户端同步更新

**代表**：Cassandra（客户端 driver 内置 token-aware 路由）、Kafka（producer 自己根据 partition 策略决定发哪个 broker）

**思路 2：路由层（Routing tier / Proxy）**

应用打到一个中间代理，代理知道路由规则并转发。

- 优点：客户端无需感知分区；规则变更只改代理
- 缺点：多一跳延迟；代理是潜在单点

**代表**：Vitess（MySQL 前面的 VTGate）、MongoDB 的 mongos、Redis Cluster 的智能代理、ShardingSphere、MyCat

**思路 3：元数据服务（Metadata-based routing）**

所有节点向一个独立元数据服务（ZooKeeper、etcd、Consul）注册自己的分区信息。客户端/代理查元数据服务找到目标。

- 优点：分区变更立即对所有人可见
- 缺点：元数据服务本身要高可用

**代表**：Kafka 老版（ZooKeeper）、HBase（ZooKeeper）、ClickHouse（ZooKeeper）

#### 9.2 为什么 ZooKeeper/etcd 成为了基础设施

这类系统做的事情看起来很朴素：**存一些元数据（节点状态、分区映射），通知变更**。但它们解决了分布式系统一个硬核问题：**多节点就某份元数据达成一致**。

这靠的是共识算法——ZooKeeper 用 ZAB，etcd 用 Raft。第 9 章会讲透。

今天几乎所有需要"集群级别协调"的系统，要么直接用 ZooKeeper/etcd，要么在内部实现一套 Raft。

#### 9.3 Kafka 3.x 的新动向：KRaft 替换 ZooKeeper

2022 年 Kafka 3.0 正式发布了 KRaft 模式，用 **Kafka 内部自己的 Raft 实现**替代 ZooKeeper。这反映了一个趋势：**大型分布式系统越来越倾向于"自己内嵌共识"，而不是依赖外部协调服务**。

这背后的权衡是：自包含系统运维更简单（少一个组件）、延迟更低，但实现复杂度高。

### 10. 再平衡（Rebalancing）：分区数据搬家的艺术

刚开始你可能只有 4 个分区，后来流量继续增长，你想扩到 8 个。
这时就会遇到"再平衡"问题，也就是：

**原来已经分好的数据，怎么重新分布到新的节点上。**

这件事听起来像搬家，实际上确实很像搬家。

#### 10.1 再平衡的三种工程范式

**范式 1：固定数量分区（Fixed number of partitions）**

一开始就创建**远大于节点数**的分区（比如 1000 个分区、10 个节点，每节点管 100 个分区）。加节点时只需要"从现有节点均匀搬一些分区过去"。

- 优点：搬家单位清晰（整个分区），路由简单
- 缺点：分区数一旦确定很难改；单分区大小会随时间变化

**代表**：Elasticsearch（index 的 shard 数创建时确定）、Couchbase、早期 Riak

**范式 2：动态分区（Dynamic partitioning）**

分区数随数据量自动增减：分区超过阈值自动分裂，低于阈值自动合并。

- 优点：不用预估总规模
- 缺点：分裂/合并本身有性能成本

**代表**：HBase、MongoDB、CockroachDB、TiKV

**范式 3：按节点数比例分区（Partitioning proportionally to nodes）**

每个节点维护固定数量 vnode（例如 256 个）。加节点时，新节点"从现有节点各偷一些 vnode"。

- 优点：随节点数自动扩展
- 缺点：实现复杂

**代表**：Cassandra、ScyllaDB

#### 10.2 再平衡的操作陷阱

**不要让再平衡自动全速运行**。经验教训：

- 如果新节点加入立即触发大规模数据迁移，网络和磁盘 I/O 可能冲击线上请求
- 自动触发的再平衡在"抖动"场景（网络临时问题）下会误判节点离线、疯狂搬数据
- 搬到一半断电的分区可能产生不一致

实际做法：

- **限速迁移**（limit bandwidth）
- **人工审批后触发**
- **只在维护窗口进行**
- **双写 + 追赶 + 切换** 三步法（类似第 5 章的故障切换）

**Cassandra 的 nodetool rebalance、HBase 的 balance_switch 都是手动触发的**。自动再平衡只在非常成熟的托管服务（AWS DynamoDB）里敢默认开。

#### 10.3 再平衡时的正确性保证

再平衡期间的数据读写是最危险的：

- 数据已搬到新节点，但路由表还指旧节点 → 读到旧副本
- 数据还没搬完，路由表已更新 → 写到新节点但数据丢了

典型的安全做法：

1. **Copy phase**：后台异步复制数据到新分区，持续追赶写入
2. **Catchup phase**：追到差距很小时，短时间锁写
3. **Switchover phase**：原子切换路由，释放锁
4. **Cleanup phase**：旧分区上的数据延迟删除（保留回滚机会）

这几乎就是 gh-ost / pt-online-schema-change 的跨分区版本。

### 11. 分区和复制几乎总是一起用

真实系统里很少只有分区、没有复制，或者只有复制、没有分区。
更常见的是两者一起上。

例如：

- 先把订单数据按 `order_id` 哈希分成 64 个分区
- 每个分区再做 3 副本复制

这样系统就同时得到：

- 分区带来的写扩展和容量扩展
- 复制带来的可用性和读扩展

但这也意味着复杂度叠加了。整个数据流需要回答：

1. **key → partition**：分区规则（哈希、范围、一致性哈希...）
2. **partition → replicas**：该分区的 N 个副本在哪些节点上？
3. **replicas → leader**：哪个是 leader？是不是每次 leader 都一样？
4. **请求路由**：客户端怎么找到 leader？
5. **副本同步**：leader 怎么把写同步到 follower？
6. **故障切换**：leader 挂了怎么重新选？
7. **再平衡**：扩节点时怎么搬？

这 7 个问题组合起来，决定了一个分布式数据库的"性格"。

#### 11.1 工业主流系统的分区+复制哲学

| 系统 | 分区策略 | 副本模型 | 一致性 |
|---|---|---|---|
| MySQL（分库分表） | 应用层哈希 | 主从 | 异步，最终一致 |
| Vitess | 范围/哈希 | MySQL 主从 | 异步 |
| MongoDB | 范围或哈希 | Primary-Secondary | 默认最终，可调强一致 |
| Cassandra | 一致性哈希+vnode | 无主 | 可调（ONE/QUORUM/ALL） |
| HBase | Key 范围（region） | 主从（基于 HDFS 副本） | 强一致（单 region 单 leader） |
| DynamoDB | 哈希 + 范围键 | 多副本 Paxos 式 | 可选强/最终 |
| Spanner | 范围 | Paxos | 线性一致（TrueTime） |
| CockroachDB | 范围 | Raft | 可串行化 |
| TiKV | 范围（region） | Raft | 线性一致 |
| Kafka | Partition（固定数） | ISR 主从 | 可选 at-least-once / exactly-once |
| Elasticsearch | 固定哈希 | Primary-Replica | 近实时 |
| Redis Cluster | 16384 hash slot | 主从 | 异步 |

**你会发现工业界 2015 之后的新系统（CockroachDB、TiKV、Spanner）几乎全部走"范围分区 + Raft/Paxos 多副本共识"的路**。这反映了一个行业共识：**强一致 + 线性扩展是技术上可行的，不用再在 Dynamo 式最终一致里受苦**。

### 12. 用订单系统把分区问题真正想透

假设你在做一个大型电商订单系统。

为什么可能需要分区？

- 订单量太大，单机存不下（**容量**）
- 大促时每秒创建订单很多，单机写入扛不住（**写吞吐**）
- 查询订单详情、用户订单列表、商家订单列表压力很大（**读分散**）

那你可以怎么分？

**选项 A：按 `order_id` 哈希分区**
- 写入分布均匀 ✓
- 查单个订单方便 ✓
- 查某用户所有订单：要 scatter 到所有分区 ✗
- 查某商家所有订单：同样 scatter ✗

**选项 B：按 `user_id` 哈希分区**
- 查用户订单列表：单分区直达 ✓
- 写入均匀（假设用户量级足够） ✓
- 查商家订单：scatter ✗
- 查单个订单：需要先知道 user_id 才能定位，否则 scatter ✗

**选项 C：复合键 `((user_id), order_id)`**
- 分区按 `user_id` 哈希
- 分区内按 `order_id` 排序
- 查用户所有订单 ✓
- 查单个订单：要带 user_id，否则仍需 scatter

**选项 D：多份数据（CQRS 思路）**
- 主数据按 `user_id` 分区（用户侧查询为主）
- 商家视角的订单通过 CDC 异步同步到"商家订单库"，按 `merchant_id` 分区
- 全局订单号查询走独立索引表

**选项 D 在大规模电商里是主流**。它放弃了"一份数据一次查"的简洁，换来了"每种访问模式都有最优路径"。代价是多份数据的一致性要靠异步数据流保证。

这也是 DDIA 全书后面反复出现的主题：**真正大规模系统很少靠一份完美数据模型解决所有问题；它们是"一份事实 + 多份为不同查询优化的派生视图"**。

#### 12.1 分区方案的七要素清单

回答一个分区方案的完整性，要答全这七条：

1. **分区键**：用哪个字段？为什么？
2. **分区策略**：哈希 / 范围 / 复合？具体算法？
3. **分区数**：初始多少？上限多少？动态还是固定？
4. **副本布局**：每分区几副本？放在哪些机房？
5. **路由方式**：智能客户端 / 代理 / 元数据服务？
6. **再平衡策略**：何时、怎么、谁来触发？
7. **热点处理**：怎么监控？怎么兜底？
8. **二级索引**：本地还是全局？查询模式能覆盖吗？

这八个问题能全部回答清楚，你的分区方案基本就是经过思考的，而不是拍脑袋的。

### 13. 先建立一个非常实用的判断口诀

你可以先记这些：

- 单机放不下、写不动、热点太集中：考虑分区
- 数据查找依赖连续区间：优先想范围分区
- 数据更需要均匀摊开：优先想哈希分区（配合一致性哈希/vnode）
- 热点 key 用 salting、应用缓存、专属分区三板斧
- 再平衡宁可手动，不要让它在错误的时间全速跑
- 二级索引要么本地（scatter-gather）、要么全局（跨分区写）
- 复制和分区是正交的两个维度，真实系统几乎总是一起用
- 分区键一旦选定，改起来极痛——务必想清楚主要查询路径

## 小结（6 条关键点）

- **分区的本质**是 shared-nothing 架构下"不同数据放不同机器"，解决单机在存储/写入/负载上的天花板；术语在业界有 shard/region/tablet/partition/slot 等多种叫法，但都是同一概念。
- **两大分区策略是 key 范围和哈希**：范围查询友好但易出尾部热点；哈希分布均匀但范围查询要 scatter-gather；复合键（先哈希后范围）是实践中的常用折中。
- **"hash mod N" 在生产中不能用**，因为扩容时几乎所有 key 要搬家；一致性哈希（+vnode）、跳跃哈希、会合哈希是工业界真正使用的算法。
- **热点是分区的永恒难题**（Zipf 分布 + Justin Bieber problem）；应对武器是 key salting、应用缓存、专属热分区、应用层 sub-sharding；Foursquare 2010 的 17 小时宕机就是分区倾斜的经典教训。
- **二级索引有两条路**：本地索引（简单但 scatter-gather 吃尾延迟）vs 全局索引（快但要跨分区写）；现代系统如 DynamoDB GSI 提供全局异步索引，本质是在一致性/延迟/复杂度之间做权衡。
- **分区和复制是正交但必须一起设计**：分区负责"写扩展+容量"，复制负责"可用+读扩展"；现代 NewSQL（Spanner、CockroachDB、TiKV）统一走"范围分区 + Raft/Paxos"，放弃了 Dynamo 时代的最终一致折中。
