# DDIA - 第 7 课：事务：正确性到底在保护什么

## 学习目标（本节结束后你能做到什么）

- 不再把"事务"理解成数据库教材里那四个字母 `ACID`，而是理解它在工程上到底在保护什么。
- 能区分"单条语句原子性""单对象原子性""多对象事务""跨系统一致性"这几层完全不同的问题。
- 理解事务真正服务的是 **应用层不变量（invariants）**，而不是"让数据库显得高级"。
- 能用具体例子解释常见并发异常：丢失更新、脏读、不可重复读、读偏斜、写偏斜、幻读。
- 理解为什么很多主流数据库的默认隔离级别并不强，以及"默认够用"和"理论最强"之间为什么总在拉扯。
- 掌握 `Read Committed`、`Snapshot Isolation`、`Serializable` 的核心差异，并理解它们各自的实现思路。
- 理解可串行化不是"性能差的教条"，而是"并发程序正确性的最直接语义"。
- 知道两阶段锁（2PL）、乐观并发控制（OCC）、可串行化快照隔离（SSI）分别在解决什么问题。
- 能区分"数据库事务边界"和"业务事务边界"，理解为什么订单、支付、库存这种流程经常超出单库事务能力。
- 掌握事务的底层物理机制（WAL、MVCC、undo log、pg_xact），理解隔离级别不是抽象规则，而是物理数据结构的表现。
- 为下一章"分布式系统里的时钟、故障与一致性"做好地基。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 DDIA 在复制、分区之后才讲事务

很多教材一开始就讲事务，因为事务看起来像数据库的"标准配置"。

但 DDIA 把事务放在复制、分区之后，是有深意的。

因为只有在你已经知道：

- 数据可以有多个副本
- 数据可以被切到多个分区
- 读写可以跨机器发生
- 延迟、故障、切换和异步传播都是常态

之后，你才会意识到：

**事务的真正问题不是"BEGIN / COMMIT 怎么写"，而是"在并发和故障都存在时，到底什么才算没写错"。**

也就是说，事务讨论的不是语法，而是 **正确性语义**。

这也是这一章标题"正确性到底在保护什么"的核心。

#### 1.1 事务概念的历史起点：System R 与 Jim Gray

"事务"不是一直就有的。它是 1970 年代 IBM Research 的 **System R 项目** 在做关系数据库原型时定义出来的工程抽象。

关键时间线：

- **1976 年**，Gray、Lorie、Putzolu、Traiger 发表 *Granularity of Locks and Degrees of Consistency in a Shared Data Base*，第一次把"一致性"量化成不同 degree（后来演化为隔离级别）。
- **1981 年**，Jim Gray 写下影响深远的 *The Transaction Concept: Virtues and Limitations*。这篇论文把事务定义成"ACID 四件套"的原型。
- **1983 年**，Härder 和 Reuter 正式提出 **ACID** 这个缩写。注意：**ACID 不是原始定义，而是后来的教学封装**。
- **1992 年**，ANSI SQL 标准把四种隔离级别（Read Uncommitted / Read Committed / Repeatable Read / Serializable）写进规范。
- **1995 年**，Berenson 等人发表 *A Critique of ANSI SQL Isolation Levels*，指出 ANSI 定义有漏洞（例如没能清晰刻画幻读和写偏斜），并提出 **Snapshot Isolation** 作为工程补丁。
- **1999 年**，Adya 博士论文 *Weak Consistency: A Generalized Theory and Optimistic Implementations for Distributed Transactions* 用有向图的方式严谨刻画了各种异常。
- **2008–2009 年**，Fekete、Cahill 等人的 *Serializable Snapshot Isolation* 让 PostgreSQL 在 2011 年的 9.1 里真的实现了 SSI——这是现代事务理论第一次大规模走进生产。

**这个时间线里藏着两件事**：

1. 事务的每一层抽象（degrees → ACID → ANSI SQL → SI → SSI）都是为了解决前一代抽象的漏洞。
2. 数据库文档里说的"Repeatable Read"和学术文献里的"Repeatable Read"经常不是同一件事——因为各家实现是在不同时间点从不同漏洞里"打补丁"打出来的。

这就是为什么工程师必须学会穿过名字看语义。

### 2. 事务首先不是数据库魔法，而是一种"把麻烦打包起来"的抽象

你可以先把事务理解成：

**数据库帮应用把一批相关操作，包装成一个更容易思考的整体。**

比如银行转账：

1. A 账户减 100
2. B 账户加 100

如果没有事务，应用就得自己担心：

- 第一步成功，第二步失败怎么办？
- 两个人同时给同一个账户转账，会不会覆盖彼此结果？
- 读到一半时，别人看到的是"只减未加"的中间态吗？

事务的价值就在于：
**它把这些并发、崩溃、中间态的问题收拢到数据库内部处理，让应用站在一个更简单的语义上思考。**

这是事务最重要的工程意义：

- 不是为了炫技术
- 不是为了追求"学院派完美"
- 而是为了让应用代码不必手写一堆脆弱的并发保护逻辑

#### 2.1 Jim Gray 的比喻：事务是"合同"

Gray 1981 年论文里有一个经典比喻：事务像一份合同。签字（COMMIT）之前，任何一方都可以撕掉它（ROLLBACK），好像什么都没发生。签字之后，双方都必须兑现承诺，即使办公室着火了（crash），合同上的字也必须能从废墟里复原（durability）。

这个比喻的价值在于：它把事务从"数据库 API"升格成了 **对世界的承诺**。后面所有难点（为什么崩溃恢复复杂？为什么分布式事务难？）都可以回到这个比喻。

### 3. ACID 到底在说什么，以及它哪里容易让人误解

一提事务，大家最容易背出的是 ACID：

- Atomicity：原子性
- Consistency：一致性
- Isolation：隔离性
- Durability：持久性

但如果你直接把这四个词背下来，通常还是会觉得事务很虚。

原因是：

- `Atomicity` 和 `Durability` 比较像数据库系统提供的实现承诺
- `Consistency` 往往被讲得最模糊
- `Isolation` 则是整章最难的部分

#### 3.1 Atomicity：不是"代码一起执行"，而是"失败时别留半截"

原子性最容易被误解成：

"事务里的语句要么一起执行，要么一条都不执行。"

这个说法方向没错，但还不够准确。

更工程化一点的表达是：

**如果事务没成功完成，系统应该像这件事从来没发生过一样。**

还是拿转账举例：

- A 扣钱成功
- B 加钱失败

如果最后数据库里真的留下"A 少了 100，B 没多 100"，这就违反了原子性。

所以原子性主要处理的是：

- 崩溃中断
- 执行报错
- 回滚恢复

它要防的是"半截写入"。

**底层怎么做到原子性？** 这不是魔法，而是靠两种物理机制：

- **WAL（Write-Ahead Log）+ Redo/Undo 日志**：PostgreSQL、MySQL InnoDB、Oracle 都是这条路。写之前先记日志，崩溃后用 redo 重放、用 undo 回滚。
- **Shadow Paging / Copy-on-Write**：LMDB、旧版 SQLite（rollback journal 模式）用这种方式——原页面不动，新写到别的地方，最后原子地替换指针。

两条路的共同点是：**把"原子提交"缩减成一次"指针交换"或"日志标记"的原子操作**。计算机硬件能保证一个字（word）的写是原子的，整个事务的原子性就是建在这个最小原子操作上。

#### 3.2 Durability：不是"绝对永不丢"，而是"提交成功后系统有责任恢复"

持久性最直观：

**事务提交成功后，即使进程崩溃、机器重启，数据库也应该能把结果恢复出来。**

注意这里是"应该能恢复"，而不是"磁盘和机房永远不会坏"。

所以持久性依赖的是：

- WAL / redo log
- checkpoint
- 崩溃恢复机制
- 副本、备份、归档

它本质上是系统对"已经承诺给外部世界的结果"负责。

**Durability 的陷阱——`fsync` 和硬件说谎**：

Durability 听起来很简单，但工程里踩过无数坑。

- MongoDB 早期默认 **不做 fsync**，commit 只意味着写进了 OS 页缓存。2013 年有过"集群重启数据丢失"的公开事件。
- Linux 内核有段时间的 bug：某些文件系统在 fsync 失败后没有正确重报错误，导致下一次 fsync 返回成功——但第一次丢掉的数据永远丢了。这就是 2018 年闻名的 **"fsync-gate"**（PostgreSQL 邮件列表爆发）。
- 普通 SSD 固件有 write cache，断电时可能丢失"已 fsync"的页，除非用带电容的企业级盘。

这告诉我们：**Durability 不是一个软件层的承诺，而是软件 + OS + 文件系统 + 硬件 + 电源 + 备份策略的总和**。所以工业界才强调 **3-2-1 备份规则**：3 份数据、2 种介质、1 份异地。

#### 3.3 Consistency：不是事务自己保证世界和平，而是"事务帮你维护约束"

一致性是 ACID 里最容易被说空的一个词。

很多解释会写成：

"事务前后数据库从一个一致状态变到另一个一致状态。"

这句话没错，但几乎没有可操作性。

真正应该这样理解：

**Consistency 说的是应用定义的数据约束，不应该因为事务执行而被破坏。**

比如这些约束：

- 账户余额不能凭空消失
- 订单总金额 = 所有明细金额之和
- 一个座位同一场电影不能卖给两个人
- 库存不能减成负数
- 用户名必须唯一

这些约束不是数据库凭空发明的，而是业务定义的。

数据库事务能做的是：

- 提供足够强的并发控制语义
- 提供唯一索引、外键、检查约束等机制
- 帮你在崩溃和并发下尽量维持这些约束

所以：

**一致性最终是应用的不变量，事务只是帮助你保护它。**

这句话特别重要。

**Kleppmann 在 DDIA 里直接说**：ACID 里的 C 其实不属于数据库的责任，它属于应用的责任。数据库只提供 A、I、D 三个机械保证，C 要靠应用写对业务逻辑。DDIA 这句话解决了无数人学事务时的"C 是什么"的困惑。

#### 3.4 Isolation：不是"互相完全看不见"，而是"并发下别产生不该有的结果"

隔离性是最容易被教成"事务互相隔离"，但如果只说到这里，几乎等于没说。

真正的重点是：

**多个事务并发执行时，结果不应该出现应用无法接受的干扰。**

这就引出整章最核心的问题：

> 并发事务到底会造成哪些坏结果？
> 数据库默认隔离级别到底挡住了哪些，没挡住哪些？

后面我们就围绕这个问题展开。

### 4. 事务真正保护的，是应用层不变量

这是本章最核心的抽象。

你可以把事务理解成：

> **数据库为应用层不变量提供的一层保护壳。**

没有这个视角，事务很容易变成一堆术语：

- 脏读
- 幻读
- MVCC
- 串行化

有了这个视角，你会知道每个异常为什么危险。

#### 4.1 什么是不变量

不变量就是：

**不管并发多复杂、系统怎么失败，业务上始终必须成立的规则。**

例如：

- 一个用户的账户余额不能是凭空错的
- 一张优惠券不能被同一用户重复使用
- 一个会议室同一时段最多被预订一次
- 一条消息不能出现"已读计数 > 总接收人数"

这些东西在单线程世界里很好维护。
但只要一进入并发环境，不变量就会开始变脆。

#### 4.2 没有并发问题时，应用代码看起来总是"对的"

例如库存扣减：

1. 读库存 = 10
2. 判断库存 > 0
3. 写库存 = 9

单线程下没问题。

但如果两个请求同时读到库存 = 10，再各自写回 9，你就只扣了一次库存，却卖出了两件商品。

这时候出问题的不是业务逻辑本身，而是：

**你默认了世界是单线程的。**

事务和隔离级别就是数据库告诉你：

"并发不是单线程世界，请你选一个你能接受的语义。"

#### 4.3 真实世界的惨痛案例：Flexcoin 被并发漏洞耗尽

2014 年 3 月，加拿大比特币交易所 **Flexcoin** 被攻击者通过并发请求耗尽了热钱包里所有比特币——约 900 BTC（当时价值约 60 万美元，按后来价格远远不止）。

攻击原理非常经典：攻击者在 **同一个账户** 同时发起多笔"内部转账"请求，每笔都是：

1. 读当前余额
2. 判断余额是否足够
3. 扣减余额，加到另一个账户

数据库事务存在，但隔离级别不够强——攻击者利用读写之间的间隙，让多笔并发事务都读到了同一个初始余额，然后各自"合法地"扣减，结果是 **负余额**，最终把钱包全部转空。

这个事故完美展示了：

- 每一条 SQL 单独看都是"对的"。
- 业务逻辑单独看也是"对的"。
- 但 **默认隔离级别配不上你保护的不变量**。

这就是 Kleppmann 反复强调的："事务不是 ACID 标签，是保护业务不变量的一层壳——如果壳不够厚，业务就会塌。"

相似案例：

- **Starbucks Mobile App (2014)**：曾有并发充值 bug 让少数用户能"免费充值"。
- **OpenTable (2017)**：并发预订让同一时段被订了超过容量的次数。
- **Uber 早期**：多次重复支付相同订单的报告，部分源自分布式事务边界设计。

这些问题的共性是：**数据库事务工作正常，但它保护的范围不够大。**

### 5. 先区分三种不同层次：单语句、单对象、多对象

很多系统面试里一上来问"有没有事务"，其实问得很不精确。

因为"事务能力"至少分三层：

1. **单条语句原子性**
2. **单对象原子性**
3. **多对象/多行/多表事务**

这三层差别非常大。

#### 5.1 单条语句原子性

比如：

```sql
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
```

一条 SQL 自己通常是原子的。

你不会看到这条语句执行到一半，某一页磁盘更新了一半就停在那里。

#### 5.2 单对象原子性

很多 NoSQL 系统提供的是"单 key 原子更新"。

例如 Redis：

- `INCR`
- `HINCRBY`
- `SETNX`

这类操作在单 key 范围内是原子的。

**DynamoDB** 的条件写（`ConditionExpression`）、**MongoDB** 早年只保证"单 document 原子"，都是这一层。

这一层能解决 **很大一部分** 工程场景：分布式锁（Redis `SETNX + EX`）、幂等键、计数器、简单扣减。但它碰到"读多个对象后再决定写"就失效了。

#### 5.3 多对象事务

真正困难的是：

- 同时改多行
- 同时改多表
- 读一批对象再决定怎么写另一批对象
- 检查某个全局约束再更新多个地方

比如：

- 转账
- 下单 + 扣库存 + 记流水
- 预约会议室时检查时段冲突
- 医生排班系统里"至少保留一个值班医生"

这些都不是单语句或单对象能轻松解决的。

DDIA 这一章真正关心的是第三层。

#### 5.4 为什么很多 NoSQL 早年"故意"只做第二层

这不是偷懒，而是工程取舍。

MongoDB 早年只提供单 doc 原子（4.0 才引入多文档事务）、DynamoDB、Cassandra 早期都刻意回避多对象事务。理由是：

- 多对象事务要求 **跨节点协调**，在分区的集群里成本极高。
- 放弃多对象事务，换来可以 **水平扩展** 和 **高可用**。
- 他们假设应用可以用"聚合根"设计（把相关数据放在同一 document/行），把多对象问题压缩成单对象问题。

这个假设很强，但在某些场景（电商、社交、日志）里确实成立。所以"要不要多对象事务"本质上是 **数据建模和扩展性取舍**，而不是"弱系统 vs 强系统"。

到 2017-2019 左右，云原生 NewSQL（Spanner、CockroachDB、TiDB）证明 **分布式多对象事务** 可以在规模上做出来，NoSQL 阵营也陆续回过头补上——整个行业经历了一轮"放弃事务 → 发现代价太大 → 把事务拿回来"的震荡。

### 6. 为什么"隔离级别"才是事务里最难、最容易被骗的部分

很多数据库宣传"支持事务"，但这句话本身信息量很低。

因为同样叫"事务"，隔离强度可能差很多。

常见误区是：

- 以为事务一开，就自动不会并发出错
- 以为 `BEGIN ... COMMIT` 之间就像独占数据库
- 以为默认配置已经很安全

但现实是：

- 大多数数据库默认不会给你最强隔离
- 很多异常在默认隔离级别下依然可能发生
- 有些数据库文档里的术语和学术定义还不完全一致

#### 6.1 主流数据库的默认隔离级别（一张冷启动必备表）

| 数据库 | 默认隔离级别 | 实际语义接近什么 | 注释 |
|---|---|---|---|
| PostgreSQL | Read Committed | Read Committed | 可以开到 `Serializable`（SSI） |
| MySQL InnoDB | Repeatable Read | 介于 RR 和 SI 之间 | 有 gap lock，阻挡部分幻读 |
| Oracle | Read Committed | Read Committed | 提供 `Serializable` 实际上是 SI |
| SQL Server | Read Committed | Read Committed（locking） | 可开启 snapshot 或 read-committed snapshot |
| DB2 | Cursor Stability（≈ RC） | RC | |
| SQLite | Serializable（单连接串行） | Serializable | 并发靠全局写锁 |
| Spanner | Strict Serializable | 最强 | 基于 TrueTime |
| CockroachDB | Serializable | Serializable（SSI 变种） | |
| MongoDB（单 doc） | — | 单 doc 原子 | 4.0+ 多 doc 快照隔离 |
| Cassandra | — | 无事务 | 仅 LWT（轻量级事务）单 key 比较 |

**这张表揭示了一个残酷事实**：没有统一的"默认安全线"。PostgreSQL 默认 RC，MySQL 默认 RR，Oracle 的"Serializable"其实是 SI——应用开发者如果不主动确认，永远在假设错误的前提下写代码。

所以学事务，关键不是记 ACID，而是学会识别：

**并发异常有哪些？当前隔离级别能挡住哪些？挡不住哪些？**

### 7. 经典并发异常：为什么"并发"会把看起来正确的逻辑写坏

下面这些异常是理解隔离级别的地基。

学术文献里它们对应的严格名字（来自 Berenson 1995 / Adya 1999）：

| 工程常用名 | 学术名 | 图论刻画（Adya） |
|---|---|---|
| Dirty Read | P1 / G1a | 读到未提交的事务状态 |
| Non-repeatable Read | P2 / G-single | 读→写依赖边产生环 |
| Phantom | P3 / G-single(谓词) | 谓词读被插入打断 |
| Lost Update | P4 / G2-item | 两个事务写同一对象的读-写-写路径 |
| Read Skew | A5A | 事务间读到不一致的对象组合 |
| Write Skew | A5B / G2 | 不同对象写但共享读约束 |

Adya 的图论方法把异常刻画为事务依赖图里的环（cycle）。这不是为了学术炫技，而是让"可串行化"有了可验证的数学定义——SSI 就是直接用这个图来决定是否回滚。

#### 7.1 脏读（Dirty Read）

事务 A 还没提交，事务 B 就读到了 A 的临时结果。

例子：

1. A 把账户余额从 100 改成 50，但还没提交
2. B 读到余额 50
3. A 回滚，余额又变回 100

那 B 刚刚看到的 50 就是脏数据。

这类问题现代主流数据库通常默认都会避免。**唯一例外**：PostgreSQL、Oracle 不支持 Read Uncommitted（实际上把它当 Read Committed 处理）；MySQL 在 Read Uncommitted 下会真的暴露脏读。

#### 7.2 不可重复读（Non-repeatable Read）

事务 A 两次读取同一行，结果不一样。

例子：

1. A 读取商品价格 = 100
2. B 更新价格为 120 并提交
3. A 再读一次，看到 120

如果 A 业务逻辑假设"事务中同一条记录应该稳定"，就会出问题。

**典型受害场景**：报表统计、备份、审计——都是读多个对象做聚合的长事务。

#### 7.3 幻读（Phantom）

事务 A 两次按条件查询，结果集合变了。

例子：

1. A 查询"今天所有未支付订单"，得到 10 条
2. B 插入一条新的未支付订单并提交
3. A 再查，变成 11 条

这次变化不是某一行字段变了，而是"结果集中多出了一行幽灵"。

**幻读的关键**：它不是针对 **具体行** 的异常，而是针对 **谓词（WHERE 条件）** 的异常。这就是为什么 InnoDB 要用 **gap lock**——把"索引间隙"也锁住，不让别的事务插入满足条件的新行。这是 MySQL RR 比学术 RR 更强的原因，也是 MySQL 更容易死锁的原因。

#### 7.4 丢失更新（Lost Update）

两个事务基于同一个旧值各自修改，后写入的把先写入的覆盖了。

例子：

1. A 读余额 100
2. B 也读余额 100
3. A 写回 90
4. B 写回 80

最后结果是 80，但本来如果两个扣款都生效，应该是 70。

这类异常非常常见，也非常危险。

**四种防御方式**（逐级升级）：

| 方式 | 机制 | 代价 |
|---|---|---|
| 原子更新 | `UPDATE SET balance = balance - 10 WHERE id = 1` | 最简单，但只能做固定运算 |
| 显式锁 | `SELECT ... FOR UPDATE` | 主动阻塞 |
| CAS（乐观锁） | `UPDATE SET v = v' WHERE v = v_old`，看影响行数 | 简单、可扩展，但需要重试 |
| 自动检测 | Snapshot Isolation + 冲突检测 | 数据库替你搞，但需要 SI 语义 |

Flexcoin 就是"原子更新"这一步没做对——它没用 `balance = balance - x`，而是先读再写。

#### 7.5 读偏斜（Read Skew）

事务读取多个彼此有关联的数据，但读到的是不同时间点的混合版本。

例子：账户转账

1. A 先读账户 X = 500
2. B 执行转账：X 减 100，Y 加 100，并提交
3. A 再读账户 Y = 600

此时 A 看到的是：

- X 的旧值
- Y 的新值

如果 A 想看"整个系统某一时刻的总余额"，这个视图就是歪的。

这种异常在做备份、报表、统计时尤其致命。

**工程案例**：早期 MySQL 的 `mysqldump` 在非事务引擎上没有快照语义，对一个"活着"的数据库做逻辑备份会产生读偏斜——备份出来的数据可能永远无法自恰恢复。这就是 InnoDB 诞生、并且必须默认支持 MVCC 快照的根本原因。

#### 7.6 写偏斜（Write Skew）

这类问题是 DDIA 特别强调的重点，因为它经常在 `Snapshot Isolation` 下漏掉。

经典例子：值班医生

规则：

- 某个时段至少要有一名医生值班

现在有两名医生 Alice 和 Bob，都在值班。

两个事务同时执行：

- 事务 A 读取：Bob 在值班，所以 Alice 可以下线
- 事务 B 读取：Alice 在值班，所以 Bob 可以下线

然后：

- A 把 Alice 改成不值班
- B 把 Bob 改成不值班

最后两人都下线了，约束被破坏。

注意这里不是同一行冲突更新，而是：

**两个事务读的是同一个全局条件，写的是不同的行。**

这就是为什么写偏斜比丢失更新更阴。

**写偏斜的通用模式**（DDIA 原话）：

1. **读**：事务先查询一组满足某条件的数据，判断是否可以继续。
2. **决策**：根据读到的结果决定要不要写。
3. **写**：写入改变了刚才查询的条件（但改的是 **另一个对象**）。

所有写偏斜都长这样。包括：

- 会议室双重预订（读"时段空闲"→ 写预订）
- 医生值班（读"至少一人在岗"→ 写下线）
- 银行透支（读"账户总额 ≥ 0"→ 写扣款）
- 游戏里抢占唯一位置（读"位置空"→ 写占位）
- 玩家称号（读"该称号无人使用"→ 写绑定）

识别出这个模式是 senior 工程师的基本功。

### 8. 不同隔离级别到底在挡什么

现在可以正式进入隔离级别。

不同数据库名字略有差异，但主线大致是：

- Read Uncommitted
- Read Committed
- Repeatable Read
- Snapshot Isolation
- Serializable

实际工程里最值得重点掌握的是：

- `Read Committed`
- `Snapshot Isolation`
- `Serializable`

#### 8.1 Read Committed（RC）

核心承诺：

- 不读到未提交的数据
- 但每次读都可能看到别人刚提交的新版本

它通常能挡住：

- 脏读

但通常挡不住：

- 不可重复读
- 幻读
- 读偏斜
- 写偏斜
- 某些丢失更新

为什么很多数据库默认会选 RC？

因为它比较便宜：

- 并发度高
- 锁冲突相对少
- 不容易把系统卡得太死

但它给应用的语义其实很弱。

**PostgreSQL 的 RC 怎么实现的？**

- 不用行锁，用 MVCC
- 每条 `SELECT` 获取一个 **新的快照**（同一事务里多次 SELECT 是不同快照）
- 写操作用行级锁（更新时如果检测到行被别的事务动过，根据隔离级别决定等待或失败）

这就是为什么 PG 的 RC "读几乎不阻塞写"——读都从快照来，写用自己的版本链。

#### 8.2 Repeatable Read（RR）

名字上看像是"事务中重复读取结果稳定"。

但现实里各家数据库对 RR 的实现并不完全一样。

例如 MySQL InnoDB 的 `Repeatable Read` 会通过 MVCC 和 gap lock 提供比学术定义更复杂的行为。

所以你在工程里不能只听名字，得看具体实现。

**InnoDB 的 RR 到底做了什么？**

- 事务开始时拿一个快照（`ReadView`），整个事务用这一个快照读
- 普通 `SELECT` 走 MVCC，不加锁
- `SELECT ... FOR UPDATE` 和 `UPDATE` 加 **next-key lock**（行锁 + 间隙锁）
- 间隙锁阻止了其他事务在范围内插入新行，**挡住了大部分幻读**

所以 MySQL RR ≠ 学术 RR，它更接近 SI，但又不完全等同。**InnoDB RR 下仍可能出现写偏斜**，这是很多"我们上了事务为什么还出 bug"的根源。

#### 8.3 Snapshot Isolation（SI）

SI 是 DDIA 里特别重要的一个概念。

它的核心承诺是：

**一个事务读到的是某个一致时间点的快照。**

这意味着：

- 事务开始后，别人提交的新数据你通常看不到
- 你看到的是"开始那一刻"或"读视图创建那一刻"的数据库快照

它通常能很好解决：

- 不可重复读
- 读偏斜
- 大部分"备份读到半截"的问题

但它仍然可能挡不住：

- 写偏斜

这就是为什么很多人觉得：

"我们都上 MVCC / 快照读了，怎么还会出并发 bug？"

因为：

**快照一致，不等于约束一定安全。**

**SI 的实现：MVCC**

MVCC (Multi-Version Concurrency Control) 是 SI 的典型实现方式。核心思想：

- 每行不是只存一个值，而是一条 **版本链**
- 每个版本带两个事务 ID：`xmin`（创建者）和 `xmax`（删除者）
- 读事务带一个 **snapshot**：一组"我开始时已提交的事务 ID"
- 读时从版本链里挑 **对当前 snapshot 可见** 的最新版本

PostgreSQL 的物理实现：
- 每个 tuple header 包含 `xmin`, `xmax`, `cmin`, `cmax`
- `pg_xact`（旧叫 `pg_clog`）记录每个 XID 的状态（进行中/提交/中止）
- 老版本由 **VACUUM** 异步回收
- 长事务会拖住 VACUUM，导致 **table bloat** 和 **XID wraparound**（32-bit XID 用完后必须 vacuum freeze，否则数据库拒绝写入）

MySQL InnoDB 物理实现不同：
- 最新版本在 B-Tree 里
- 老版本放在 **undo log**
- undo log 同时被用于回滚和 MVCC 读
- 长事务同样会拖住 undo log 清理，导致 undo 表空间膨胀

两种实现的共同痛点：**长事务是 MVCC 的天敌**。这就是为什么 OLAP 型长查询放到 OLTP 库上经常引发运维事故。

#### 8.4 Serializable（可串行化）

可串行化是最强语义。

它要求：

**并发执行出来的结果，必须等价于某种串行顺序下一个接一个执行的结果。**

这句话很重要，建议你多读两遍。

它不是说真的物理串行执行，而是说：

> 虽然大家并发跑了，但最后结果要像"某个顺序地一个个跑完"那样成立。

这就是为什么可串行化常被认为是"最好理解的正确性模型"。

因为应用程序员最容易理解串行执行。

**两种等价定义**（都来自 1970 年代数据库理论）：

- **冲突可串行化（Conflict Serializable）**：依赖图无环。冲突 = 两个事务访问同一对象，至少一个是写。这是最常用的工程定义。
- **视图可串行化（View Serializable）**：结果"看起来"像某种串行顺序。严格更大，但检测 NP-hard，所以现实系统都用冲突可串行化。

#### 8.5 Strict Serializable（严格可串行化）

这个概念 DDIA 没展开但在 Spanner、CockroachDB 文档里会反复出现。

它 = 可串行化 + **线性一致性**：不仅等价于某种串行顺序，而且这个顺序必须和 **物理时间顺序** 一致。

也就是说：

- 可串行化只要求"有某种等价顺序"
- 严格可串行化还要求"这个顺序不能违反真实世界的因果"

例如：A 转钱给 B，然后 B 再转给 C。如果只是可串行化，系统可以把这三笔事务排成"A→B→C"、"A→B, C 在 A→B 之前"等等（只要结果对应某一种顺序就行）。但 **如果真实物理时间里 A→B 是 10:00, B→C 是 10:01**，严格可串行化要求最终顺序必须把 A→B 排在 B→C 前面。

这是 Spanner 用 TrueTime 搞出来的事情。大多数分布式数据库只能做到可串行化，只有 Spanner 级别的基础设施（全局同步时钟）才能做到严格可串行化。

### 9. 为什么 Snapshot Isolation 很强，却还不够

SI 在工程里非常受欢迎，因为它比 RC 强很多，又常常比 Serializable 便宜。

它像一个很强的折中方案。

但 DDIA 特别强调：

**SI 不是串行化。**

最典型漏网之鱼就是写偏斜。

#### 9.1 写偏斜为什么 SI 挡不住

因为 SI 关注的是：

- 每个事务读到一个稳定快照
- 同一行直接写写冲突能检测出来

但它不一定能发现：

- 两个事务读了同一组条件
- 各自去改不同的行
- 结果全局约束被破坏

医生值班例子就是这样。

从数据库内部看：

- A 改的是 Alice 这一行
- B 改的是 Bob 这一行

它们没直接更新同一行，所以系统不一定把它们认成冲突。

但从业务看：

- 它们共同破坏了"至少一人值班"的约束

这就是事务系统里最容易被低估的一件事：

**数据库看到的是"读了哪些对象、写了哪些对象"；业务在乎的是"全局规则有没有被破坏"。**

#### 9.2 SI 下写偏斜的工程对策

如果你只能拿到 SI，不能拿到 Serializable，以下是常见救急手段：

1. **显式行锁（`SELECT ... FOR UPDATE`）**：在读条件的时候就把相关行锁住，让后来的事务等待。这是最常见的手法。但写偏斜涉及 **共享读 + 不同写**，有时候你锁的那行根本不是你要写的那行——这时需要"锁一个代理对象"或"物化冲突"。
2. **物化冲突（Materializing Conflicts）**：DDIA 专门讲了这个技巧——把"抽象约束"变成"具体的表行"。比如会议室预订，创建一张 `(room_id, time_slot)` 的表，每个 slot 是一行，想预订必须先 INSERT 那一行（靠唯一索引阻塞）。这样写偏斜就退化成了丢失更新，能被检测到。
3. **升级到 Serializable**：PostgreSQL 提供 SSI，MySQL 提供 2PL-based Serializable。这一条是最干净的，但要监控吞吐和回滚率。

### 10. 可串行化为什么重要：因为它最贴近应用员的直觉

如果你让应用开发自己去推：

- 在 RC 下会不会读偏斜
- 在 RR 下会不会幻读
- 在 SI 下会不会写偏斜
- 某些数据库的 RR 和 SI 边界在哪

这几乎注定会出事。

因为这要求业务开发脑子里一直维护一个复杂的并发模型。

而可串行化的好处是：

**你可以近似把数据库当成"串行执行"的。**

它极大降低了应用层思考负担。

这也是 DDIA 一个非常强的观点：

> 很多系统故障并不是因为数据库太弱，而是因为给了应用一个太复杂的并发语义，逼应用自己正确地想清楚。

### 11. 可串行化通常怎么实现

实现可串行化并没有唯一办法。

常见思路有三类。

#### 11.1 真正串行执行（Actual Serial Execution）

最简单粗暴的方式就是：

- 一个事务一个事务地执行
- 不并发

这样当然最容易正确。

但吞吐通常不高，适合特定场景，比如：

- 单线程内存数据库上的小事务
- 某些极低延迟、高度可控的工作负载

**真实例子**：

- **Redis** 的命令执行模型本质上就是单线程串行（到 6.0 才引入 I/O 多线程但命令执行仍是单线程）。所以 Redis 事务（MULTI/EXEC）是真正的 Serial Execution。
- **VoltDB**（Mike Stonebraker 的项目）在每个分区上用单线程跑存储过程——把事务变成 "在分区本地顺序执行的代码"。吞吐比传统 RDBMS 高一个数量级。
- **H-Store**、**Redis Cluster** 的 `MULTI` 也采用这个思路。

这条路的关键前提是：**单事务必须非常快**（通常是内存、预编译、没有用户输入等待）。如果事务里有网络来回或人的思考，串行执行会把整个库堵死。

#### 11.2 两阶段锁（2PL, Two-Phase Locking）

经典方法，1970 年代 System R 就在用。

核心直觉：

- 读写对象时加锁
- 锁要持有到事务结束

这样别人就不能在中间乱改。

"两阶段"指的是：**Growing Phase**（一直获取锁，不释放）和 **Shrinking Phase**（一旦开始释放锁，就不能再获取新锁）。这个约束是 2PL 保证可串行化的关键。

**2PL 的四种变体**（强度从弱到强）：

| 变体 | 什么时候释放锁 | 解决的问题 |
|---|---|---|
| Basic 2PL | Shrinking phase 里任意时刻 | 保证可串行化 |
| Conservative 2PL | 事务开始时一次性拿齐所有锁 | 避免死锁 |
| Strict 2PL（S2PL） | **写锁持有到 commit/abort** | 避免级联中止 |
| Rigorous 2PL（SS2PL） | **所有锁持有到 commit/abort** | 最强，工业界常用 |

绝大多数商业数据库用的是 **Rigorous 2PL** 或 **Strict 2PL**。

**锁粒度**（Gray 1976 那篇论文就是讲这个）：

- 表锁
- 页锁
- 行锁
- 谓词锁（逻辑的，实现成间隙锁）

粒度越小，并发越高，但维护成本越高。InnoDB 的 next-key lock（行锁 + gap lock）就是在粒度和并发之间的取舍。

**死锁检测**：2PL 会产生死锁。工业界两种思路：

- **Wait-for 图 + 周期检测**：InnoDB 每 1 秒扫一次，发现环就回滚一个事务。
- **超时**：简单粗暴，但把锁等待的用户体验降级为随机失败。
- **Wound-Wait / Wait-Die**（基于时间戳）：Spanner、CockroachDB 用这种方案，避免构建 wait-for 图。

优点：

- 语义直观
- 可以实现真正的串行化

缺点：

- 锁竞争大
- 容易阻塞
- 容易死锁
- 尾延迟可能很差
- 吞吐通常比 SSI 低一个数量级

这就是为什么现代数据库都在往 SSI 迁移。

#### 11.3 可串行化快照隔离（SSI, Serializable Snapshot Isolation）

这是现代系统很重要的思路。

SSI 是 Fekete、Liarokapis、O'Neil、O'Neil、Shasha 2005 年的论文提出，Cahill 2008 年做出原型，PostgreSQL 9.1（2011）把它搬进生产。CockroachDB、FoundationDB、YugabyteDB 都走这条路的变体。

它不是像 2PL 那样"先挡住所有危险并发"，而是：

**先让事务像 SI 一样乐观运行，再在提交时检查是否形成危险依赖图。**

可以把它理解成：

- 平时先按快照隔离高并发地跑
- 提交时再判断这个并发组合会不会破坏串行化
- 如果会，就回滚其中一个事务

**SSI 检测什么？**

SSI 跟踪 **rw-依赖**：如果事务 T1 读了 X，T2 后来写了 X，就形成 T1→T2 的 rw-边。SSI 的核心观察是：

**所有写偏斜都对应依赖图里的一个特殊环——至少包含两条连续的 rw-边。**

所以 SSI 运行时维护每个事务的入边和出边，提交时如果检测到"rw in + rw out"的危险模式就回滚。

优点：

- 并发性通常比 2PL 好（理论论文上有 10× 差距）
- 对只读事务很友好（只读事务几乎不会被回滚）
- 读不阻塞写、写不阻塞读（和 SI 一样）

缺点：

- 实现复杂
- 需要跟踪读写依赖关系
- 高冲突场景下回滚会增多（应用必须有重试逻辑）
- 对长事务不友好（依赖图维护成本高）

### 12. OCC：为什么"乐观"有时反而更适合

乐观并发控制（OCC）背后的假设是：

> 大多数事务其实不会冲突，所以没必要一上来就锁死。

OCC 由 Kung 和 Robinson 1981 年论文 *On Optimistic Methods for Concurrency Control* 提出。

典型流程（三阶段）：

1. **Read phase**：读数据、做计算，写到私有缓冲区
2. **Validation phase**：检查读到的数据是否被别人改过
3. **Write phase**：通过验证则原子写入，否则丢弃并重试

这在以下场景很常见：

- 冲突率不高
- 事务较短
- 用户更能接受少量重试，而不是长时间等待锁

但 OCC 不适合：

- 热点资源竞争极高
- 事务很长
- 重试代价很高

所以它不是银弹，而是另外一种权衡。

**OCC 和 SSI 的关系**：

SSI 本质上是 OCC 的一种实现方式，但加了 MVCC 快照作为"读视图"。经典 OCC 只检查"读到的值现在还是不是那个值"，而 SSI 还追踪依赖图来确保可串行化。

**典型工程案例**：

- **Datomic**（Rich Hickey 设计）：完全 immutable + OCC。每次事务产生新的 DB value。
- **Git**：`git push --force-with-lease` 本质上是 CAS/OCC 风格。
- **ORM 的 version 字段**：Hibernate/ActiveRecord 的 `@Version` 注解就是 OCC。
- **Elasticsearch** 的 `_version` 和 `_seq_no`：并发写冲突检测。

"乐观 vs 悲观"的选择核心是 **冲突率预测**：冲突率 < 10% 时 OCC 更快，冲突率 > 30% 时 2PL/悲观更稳。

### 13. 为什么很多系统不用最强事务

讲到这里很容易得出一个结论：

"那全部上 Serializable 不就好了？"

现实中没这么简单。

因为更强的隔离往往意味着：

- 更多锁
- 更多冲突检测
- 更多回滚
- 更高延迟
- 更复杂实现

所以工程里永远在做这个权衡：

| 追求 | 带来的代价 |
|---|---|
| 更强正确性 | 更高延迟、更低吞吐、更复杂故障模式 |
| 更高并发 | 更弱语义、更多异常交给应用层承担 |

这就是为什么有些系统会故意选弱一点的隔离：

- 点赞数、阅读量、弱一致计数：通常不用最强事务
- 核心账户、库存扣减、唯一性约束：往往需要更强保护

DDIA 一直在训练你的，就是这种判断力。

#### 13.1 真实工程里的混合策略

实际生产系统几乎永远不是"一个隔离级别走天下"，而是：

- **默认用弱隔离**（RC 或 RR），因为 99% 的请求不需要更强。
- **关键路径提升到 Serializable**：PG 的 `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE`、MySQL 的 `SERIALIZABLE`、Oracle 的 `SERIALIZABLE`（实际 SI）。
- **超高热点用 CAS + 原子更新**：`UPDATE ... WHERE v = v_old` 或 Redis `WATCH/MULTI/EXEC`。
- **跨库/跨服务改用 Saga、Outbox**：事务边界放弃在数据库层做（见下一节）。

这就是为什么系统设计师的关键技能不是"会用事务"，而是 **"能判断每个动作需要多强的事务保护"**。

### 14. 事务边界经常小于业务边界

这是很多后端工程里最关键、也最容易忽略的一点。

数据库事务通常只能覆盖：

- 一个数据库实例
- 或一个明确支持分布式事务的边界

但真实业务流程经常跨越：

- 订单库
- 库存库
- 支付系统
- MQ
- 搜索索引
- 缓存

例如"用户支付成功"这个业务动作，常常包括：

1. 写订单状态
2. 扣库存
3. 写支付流水
4. 发 MQ 通知
5. 更新搜索索引
6. 删缓存

你不可能指望一个普通数据库事务把这些全包起来。

所以：

**数据库事务解决的是局部正确性，业务一致性经常需要更高层的模式来补。**

这就是后面分布式系统里会出现：

- outbox pattern
- saga
- 幂等
- 重试
- 补偿

这些东西的原因。

#### 14.1 为什么 XA / 2PC 不是答案

历史上当然有人试过在数据库层做跨系统事务。XA（X/Open 标准）和 **两阶段提交（2PC）** 就是这个尝试——让多个数据库共同参与一个事务，所有人都说 ready 才 commit。

为什么它被大规模放弃？

- **阻塞**：2PC 的协调者崩溃时，参与者被卡在 "prepared" 状态无限等待（锁不能释放，事务不能结束）。
- **可用性糟糕**：任何一个参与者宕机都会把整个事务挂起。
- **性能灾难**：每次事务多轮 RTT，锁持有时间翻倍。
- **运维噩梦**：协调者必须有极高可用性，否则比不做事务还糟。

Pat Helland 2007 年论文 *Life Beyond Distributed Transactions* 直接提出："别再搞分布式事务了，用幂等消息和业务补偿。"这篇论文几乎定义了后面 10 年的微服务架构。

#### 14.2 Saga 模式：长事务拆成补偿链

Saga 不是 DDIA 发明，是 1987 年 Garcia-Molina 的论文 *Sagas* 里提出的。核心：

- 长业务流程拆成 N 个本地事务
- 每个本地事务有对应的 **补偿操作**（逆操作）
- 顺序执行，任何一步失败就反向执行补偿

**订单-支付-扣库存** 的 Saga 版本：

1. 创建订单（补偿：取消订单）
2. 扣减库存（补偿：恢复库存）
3. 发起支付（补偿：退款）
4. 通知发货（补偿：撤销通知）

如果第 3 步失败，反向执行第 2 和第 1 步的补偿。

**两种 Saga 风格**：

- **Choreography（编舞）**：各服务通过事件互相驱动，没有中央协调。解耦好但难以追踪。
- **Orchestration（编排）**：一个协调者按顺序调各服务。清晰但协调者是瓶颈。

#### 14.3 Outbox 模式：解决"写数据库 + 发 MQ 不一致"

"订单写完后发 MQ 通知"这件事，看起来简单，其实经典坑：

- 先写库再发 MQ：发 MQ 时如果 MQ 挂了，通知丢失。
- 先发 MQ 再写库：库写失败，通知已经发出去。

Outbox 模式解决方式：**在业务库里加一张 `outbox` 表**。事务里同时写业务表和 outbox 表（同一个本地事务，原子的）。另一个进程扫 outbox 表发 MQ。这样："写业务 + 投递消息"在本地事务里是原子的，投递行为被后续异步保证。

Outbox 和 **CDC（Change Data Capture）** 是搭档：很多现代系统直接用 Debezium 监听 WAL，把业务表的变更投递到 MQ，避免应用代码写 outbox 表。

#### 14.4 幂等键：Stripe 的经典设计

"支付失败要重试"这件事上踩过无数坑的是支付公司。Stripe 的方案已经成为业界标准：

- 每个支付请求必须带一个 **Idempotency-Key**（UUID）。
- Stripe 把 `(key, response)` 缓存 24 小时。
- 同一个 key 重试时直接返回上次的结果，不会真的重复扣款。

这本质是把"幂等性"从"业务逻辑自己写"变成"协议层保证"。所有涉及钱/外部副作用的 API 都应该这样设计。

### 15. 所以"事务"到底在保护什么

现在可以回到标题了。

事务真正保护的，是这几件事：

1. **防止半截写入**
   - 避免做一半就崩掉
2. **避免中间态暴露**
   - 别让别人读到过程中的脏状态
3. **约束并发干扰**
   - 别让多个事务把全局规则悄悄写坏
4. **降低应用思维负担**
   - 让应用不用时时刻刻自己模拟并发世界

一句话总结：

**事务不是为了让数据库拥有 ACID 标签，而是为了让业务不变量在并发和故障里尽量还能成立。**

### 16. 一个后端工程师最实用的事务判断框架

以后你遇到一个场景，不要先问"能不能开事务"，而是先问下面这些：

#### 16.1 这件事保护的是什么不变量

例如：

- 库存不能变负
- 一个邀请码只能使用一次
- 一个用户名不能重复
- 一个班次至少有一位医生值班

先把不变量说出来。

#### 16.2 这个不变量涉及几个对象

- 单行？
- 多行？
- 多表？
- 跨分区？
- 跨系统？

涉及范围越大，越难完全交给数据库单次事务解决。

#### 16.3 读写之间有没有决策逻辑

如果只是直接写一行，问题不大。

但如果是：

- 先查是否满足条件
- 再决定是否写

那就要高度警惕写偏斜和竞态条件。

#### 16.4 能不能接受重试

如果可以接受冲突回滚后重试，OCC/SSI 类方案就更可用。

如果不行，可能要更保守地加锁。

#### 16.5 这件事是不是本来就超出了数据库事务边界

如果已经跨：

- DB + MQ
- DB + 缓存
- 多服务

就别再幻想单个数据库事务能全包了，要换思路。

### 17. 深入：事务的物理代价

学事务很容易只停留在"语义层"。但工业界的事务故障，九成发生在"物理代价层"——长事务拖垮 MVCC、锁冲突拖垮延迟、2PC 协调者宕机等等。

#### 17.1 长事务的毒性

长事务（持续时间超过几十秒的事务）是 OLTP 库最常见的慢性毒素：

- **PostgreSQL**：阻止 VACUUM 清理死 tuple → table bloat → 索引肿胀 → 查询变慢 → 磁盘爆。极端情况 XID wraparound 让数据库整个拒绝写入。
- **MySQL InnoDB**：阻止 undo log 清理 → undo 表空间膨胀 → 次级索引查询走版本链越来越长 → 所谓"历史链条过长"。
- **SQL Server**：阻止 tempdb 里版本存储回收。
- **Oracle**：阻止 UNDO 段回收，容易引发 ORA-01555 "snapshot too old"。

**工程规则**：OLTP 事务必须 **短**。报表、备份、ETL 要走 **只读副本** 或专门的 OLAP 系统，绝不应该在主 OLTP 库上跑分钟级以上的查询。

#### 17.2 锁等待放大尾延迟

2PL 语义下，一个热点行的锁等待队列会直接放大 p99/p999 延迟——这是 Little's Law 的结果。

假设有 10 个并发事务都要更新同一行，每个事务持锁 10ms：

- p50 等待时间 ≈ 25ms（排队中位数）
- p99 等待时间 ≈ 95ms
- p999 甚至到秒级

这就是为什么"抽奖""秒杀""抢购"这类热点业务，**必须绕过悲观事务**：用 Redis CAS、分片扣减、异步批处理、排队器等其他手段。直接上 2PL 会把系统打穿。

#### 17.3 回滚率是 SSI 的成本信号

SSI 在"预期低冲突"时非常优雅，但如果冲突率上升，回滚率也会上升。PostgreSQL 的监控指标 `pg_stat_database.xact_rollback` 和 `serialization_failure` 就是 SSI 回滚率的直接信号。

**工程经验值**：回滚率超过 5% 时，系统就开始有明显性能损失（因为被回滚的事务做的所有工作都白费了）。超过 15% 几乎不能用。这时候要么降级到 RC + 应用层锁，要么把竞争拆成多个分片。

## 小结（10 条关键点）

- **事务首先是一层抽象**：它的核心价值不是提供一组 SQL 关键字，而是把崩溃、并发、中间态这些麻烦打包起来，让应用站在更简单的语义上工作。Jim Gray 1981 年的"合同"比喻到今天仍然是最准确的心智模型。
- **ACID 里最容易被误解的是 Consistency 和 Isolation**：一致性本质上是应用不变量是否被维护（C 其实是应用的责任，不是数据库的），隔离性本质上是并发事务是否会产生应用不可接受的干扰。
- **事务真正保护的是业务规则**：像余额守恒、库存不为负、用户名唯一、至少一名医生值班，这些都属于不变量；数据库不是凭空创造它们，而是在尽力保护它们。Flexcoin 2014 的 60 万美元损失就是"不变量保护不到位"的典型事故。
- **并发异常才是隔离级别讨论的主角**：脏读、不可重复读、幻读、丢失更新、读偏斜、写偏斜，这些异常决定了"当前事务语义到底够不够用"。Adya 1999 的图论方法把它们刻画成依赖图里的环，SSI 就是直接用这个数学来判定。
- **各家数据库的"同名隔离级别"并不是同一件事**：MySQL InnoDB 的 RR ≠ 学术 RR，Oracle 的 Serializable ≠ 学术 Serializable（它是 SI）。所以永远不能只听名字，要看文档。
- **Snapshot Isolation 很强，但不是 Serializable**：它能解决很多读取一致性问题，却依然可能在全局约束场景下发生写偏斜，这是工程中非常常见的坑。
- **Serializable 的价值在于最贴近应用直觉**：它要求并发执行结果等价于某个串行顺序，因此最容易推理正确性；实现上通常依赖串行执行（Redis、VoltDB）、2PL（传统 RDBMS）、SSI（PG 9.1+、CockroachDB）等不同路线。
- **更强事务语义从来不是免费的**：更强隔离通常意味着更多锁、更高冲突、更大延迟和更复杂实现，所以数据库默认配置往往不是"最安全"，而是"比较平衡"。生产系统几乎永远是混合策略——默认弱隔离，关键路径单独提强。
- **长事务是 MVCC 的天敌**：VACUUM 拖住、undo 表空间膨胀、XID wraparound——这些都是"语义对了但物理代价没顾到"的经典翻车。OLTP 事务必须短，报表和备份走副本或 OLAP。
- **数据库事务边界经常小于业务事务边界**：一旦流程跨 DB、缓存、MQ、搜索索引或多个服务，单库事务就不够了，后面必须引入幂等键（Stripe 模式）、Outbox + CDC、Saga（编舞/编排）等更高层机制。XA/2PC 在 Pat Helland 2007 年论文后已被大规模放弃。
