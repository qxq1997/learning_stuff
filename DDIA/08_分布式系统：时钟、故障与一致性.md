# DDIA - 第 8 课：分布式系统：时钟、故障与一致性

## 学习目标（本节结束后你能做到什么）

- 理解为什么系统一旦跨机器，很多"单机里理所当然"的假设都会失效。
- 区分 **崩溃故障、网络故障、进程暂停、时钟漂移** 这几类最常见的分布式风险，并理解它们为什么难缠。
- 理解 **部分失败（partial failure）** 为什么是分布式系统最核心的难点。
- 知道"检测节点是否挂了"不是一个可以被 100% 准确解决的问题，只能做近似判断。
- 理解同步模型、异步模型、部分同步模型分别意味着什么，以及为什么现实系统通常落在"部分同步"区域。
- 掌握物理时钟、单调时钟、逻辑时钟、Lamport 时间戳、版本向量、Hybrid Logical Clock、Spanner TrueTime 各自解决的问题与边界。
- 理解 **因果关系（causality）** 为什么重要，为什么"时间先后"不等于"因果先后"。
- 区分几类常见一致性语义：最终一致、顺序一致、线性一致、因果一致，知道它们分别在承诺什么。
- 理解为什么 **线性一致性** 很直观，但代价并不低；也理解它和事务可串行化不是一回事。
- 理解 FLP 不可能性定理为什么重要，以及它给工程实践的启示。
- 为下一章"共识：为什么有些决定必须大家同意"做好数学和工程上的地基。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么一旦跨机器，事情就突然变难了

上一章我们讨论事务时，默认了一件事：

> 至少数据库内部有一个相对统一的执行环境。

即便事务并发很复杂，它仍然是在一个数据库系统内部协调：

- 有统一的锁管理
- 有统一的日志
- 有统一的提交点
- 有统一的"谁先谁后"的判断方式

但一旦系统跨机器，你就失去了这些天然条件。

你会立刻遇到几类新问题：

- 机器 A 说"我已经写成功了"，机器 B 还没收到
- 节点没响应，到底是挂了，还是网络抖了，还是 GC 停顿了？
- 机器 A 的时间比机器 B 快了 200ms，这 200ms 会不会把"先后顺序"判断搞错？
- 两个副本都说自己是最新的，到底信谁？
- 某个请求在客户端看来超时了，但服务端实际上已经执行成功了，这算成功还是失败？

这些问题在单机里通常不会作为第一层矛盾暴露出来。

所以分布式系统真正难的地方，不是"有很多机器"，而是：

**你失去了全局视角，却还要做全局判断。**

#### 1.1 Deutsch 八大谬误：新人最容易犯的错

1994 年，Sun Microsystems 的 Peter Deutsch 写下了后来被称为 *The Eight Fallacies of Distributed Computing* 的清单，指出工程师做分布式系统时最常犯的 8 个错误假设：

1. **网络是可靠的**（The network is reliable）
2. **延迟是零**（Latency is zero）
3. **带宽是无限的**（Bandwidth is infinite）
4. **网络是安全的**（The network is secure）
5. **拓扑不会变**（Topology doesn't change）
6. **只有一个管理员**（There is one administrator）
7. **传输成本为零**（Transport cost is zero）
8. **网络是同质的**（The network is homogeneous）

James Gosling 后来又加了第 9 条："所有机器的时钟一样"。

这个清单 30 年来几乎没过时。本章剩余内容，本质上就是在把这些谬误一条一条拆开讲为什么它们错、错在哪、怎么应对。

#### 1.2 Waldo 1994：为什么"本地调用"和"远程调用"不能被当成同一件事

1994 年 Sun 的另一篇论文 *A Note on Distributed Computing*（Waldo、Wyant、Wollrath、Kendall）是分布式系统的经典。它的核心论点：

> "把远程调用做得和本地调用一样"这个诱惑是错的。

远程调用与本地调用有四个本质不同：

- **Latency**：差几个数量级
- **Memory access**：没有共享内存
- **Partial failure**：本地调用要么全成要么全失败，远程调用可能部分成功
- **Concurrency**：远程调用几乎必然引入真正的并发

这篇论文宣判了 CORBA、Java RMI、甚至早期 gRPC 风格"伪装成本地函数"的失败。它的结论是：分布式接口必须 **显式承认失败**，而不是把失败藏起来。这就是现代 API 普遍返回 `Result<T, Error>`、显式超时、显式重试策略的理论根源。

### 2. 单机世界里默认成立的五个假设，在分布式里都不再可靠

在单机或单进程系统里，我们经常无意识依赖下面这些假设：

1. 操作会很快完成
2. 如果没返回，说明失败了
3. 系统时钟大体可信
4. 先发生的事，时间戳也会更早
5. 如果某个组件"没说话"，那它大概率就是挂了

这些假设放到分布式环境里几乎都要打折。

#### 2.1 "没响应"不等于"失败"

客户端发请求给服务端，结果 2 秒没响应。

到底是哪种情况？

- 服务端真的挂了
- 服务端没挂，但网络包丢了
- 服务端已经处理完，只是返回包丢了
- 服务端没挂，但线程卡住了
- 服务端正在 Full GC
- 服务端的磁盘 I/O 卡住了
- 中间代理挂了
- 连接池打满了

你会发现：

**客户端看到的现象只有一个：超时。**

但其背后可能对应完全不同的真实原因。

这就是分布式系统里最核心的一件事：

> 你通常只能看到"现象"，而看不到全局真相。

#### 2.2 "时间先后"不等于"因果先后"

例如两台机器：

- A 的系统时钟比真实时间快 500ms
- B 的系统时钟比真实时间慢 300ms

现在：

- B 先处理了一个写请求，记录时间 `10:00:00.800`
- A 后处理了一个写请求，记录时间 `10:00:00.600`

从时间戳看，好像 A 更早。
但真实世界里，A 其实更晚。

所以一旦系统跨机器：

**用 wall clock（墙上时钟）直接推导事件先后，经常会出错。**

**一个真实案例**：Cassandra 的早期默认使用客户端时间戳做 Last Write Wins。2013 年有过一次著名事故：某个客户端机器 NTP 失配，系统时钟快了 1 分钟，它的所有写都带着"未来"时间戳，结果后续 1 分钟内所有其他客户端的合法写入都被 LWW 判定为"旧值"而被覆盖。修复只能等那 1 分钟过去——数据损失无法回滚。

这就是为什么 DDIA 反复强调：**不要把 wall clock 直接用于因果判断或冲突解决**。

#### 2.3 "节点没说话"不等于"节点死了"

这是很多故障切换事故的起点。

如果一个节点 5 秒都没心跳，管理系统就会想：

> 它是不是挂了？我要不要选一个新的主节点？

但如果那台机器只是：

- 网络临时抖了 5 秒
- 被宿主机暂停了 5 秒
- 在做 STW GC
- 线程调度被饿死了

你贸然故障切换，就可能导致脑裂。

所以：

**故障检测不是"读状态"，而是"根据超时做推断"。**

这件事后面会直接连到第 9 章的共识。

**真实的 GC 停顿数据**：

- HotSpot JVM Full GC：生产里 10-30 秒的停顿并不少见，大堆甚至到分钟级
- G1/ZGC/Shenandoah：把最坏情况从秒级压到毫秒级，但仍然存在
- CMS：remark 阶段 STW 可能数秒

Kleppmann 在 DDIA 里特别提到："JVM Full GC 会让你的进程被世界完全暂停，时间足够让 ZooKeeper 把它从集群中踢出。"这就是为什么某些生产集群会在部署时把 HotSpot 切到更保守的 GC 策略，或者直接选 Golang/Rust 这类没有 STW 的语言。

### 3. 分布式系统最麻烦的不是整体故障，而是部分失败

单机故障通常很直接：

- 机器挂了，服务整体不可用

这种故障虽然糟糕，但至少清晰。

分布式系统更麻烦的是：

**只有一部分组件坏了，另一部分组件还活着。**

这就叫 partial failure（部分失败）。

例如一个 5 节点集群里：

- 2 个节点能互相通信
- 另 3 个节点也能互相通信
- 但这两边彼此完全隔离

这时候最危险的不是"大家都挂了"，而是：

**两边都觉得自己还活着，并且都可能试图继续服务。**

这也是为什么分布式系统经常不是"全坏 / 全好"的问题，而是：

- 哪些节点彼此还能看到对方？
- 哪些节点的视图已经不一致？
- 谁还认为自己有资格继续对外承诺？

#### 3.1 真实网络有多不可靠：AWS/Azure/Google 的公开数据

Microsoft Research 2011 年论文 *Understanding Network Failures in Data Centers* 基于某大型数据中心一年的数据：

- 数据中心内部网络每月有数百次故障
- ToR 交换机是单点故障源（虽然理论上有 HA）
- 负载均衡器、VPN、核心路由器每种都会挂

Google 2015 年 SIGCOMM 论文承认其内部网络因为软件升级和配置错误偶尔会产生"单方向可达"（A 能收到 B 的包，但 B 收不到 A 的）。这是最阴的故障模式之一。

Jepsen（Kyle Kingsbury）用网络分区测试了几乎所有主流分布式数据库：

- MongoDB 早期在分区时会丢数据
- Redis Sentinel 曾经有选举漏洞
- etcd、Consul、ZooKeeper 在压力测试下也都被找到过 bug
- Cassandra 的 LWW 在分区下暴露严重冲突

每一代"号称网络分区安全"的系统，都被 Jepsen 打过脸。这是一个非常健康的提醒：**网络分区是常态，不是边缘情况**。

### 4. 分布式系统到底在和哪些故障打交道

可以先把故障模型粗分成四大类。

#### 4.1 崩溃故障（crash fault）

最简单的一类：

- 进程退出
- 机器宕机
- 服务被 kill

这类故障相对"干净"，因为节点真的停止工作了。

很多论文和算法一开始假设的就是 crash-stop（崩溃即停止，不再回来）。更弱一点的是 **crash-recovery**（崩溃后会重启，但可能丢内存状态）。

#### 4.2 网络故障

包括：

- 丢包
- 延迟抖动
- 重传
- 网络分区
- 单向可达
- **Gray failure**（灰度故障：时好时坏、概率性丢包）

网络故障比崩溃更恶心，因为节点可能还活着，只是彼此看不见。

这会直接导致：

- 超时判断变困难
- 重试语义变复杂
- 主备切换变危险

**关于 TCP 的误解**：很多人以为 "TCP 可靠"意味着 "只要写入成功就一定送达"。错。TCP 保证的是 **字节流顺序 + 重传**，但：

- TCP 连接可能被 NAT 或中间盒单方面丢弃（中间盒看不到这个流量，认为连接已 idle 超时）
- 应用层 `write()` 成功只是把数据交给了内核缓冲区，不意味着对端已收到
- 崩溃前的数据可能还在 TCP 发送缓冲区里
- SO_LINGER、TCP_KEEPALIVE 默认值通常不适合生产（keepalive 默认 2 小时）

所以 TCP 只是减少了应用要担心的事情，不是消除了它们。

#### 4.3 进程暂停 / 运行时停顿

这是工程里经常被低估的一类故障。

比如：

- JVM Full GC（10 秒级甚至更长）
- Go 的 GC（毫秒级，但在高压力下也会到秒级）
- 宿主机调度暂停（虚拟机 CPU 被其他 guest 抢走）
- 容器被 cgroup 限制（CPU throttle）
- 虚拟机 live migration（VMware/KVM 的迁移期间被暂停数秒）
- 磁盘 stall 导致线程长时间阻塞（NFS 故障、磁盘队列满）
- 系统 swap 被触发（进程被换出，读回来要几十秒）
- Linux 内核 `softirq` 风暴

从外部看，效果和"节点失联"非常像。

但它其实不是死机。

这就是为什么：

**把"超时未响应"直接等价成"节点死亡"，几乎必然会踩坑。**

**Kleppmann 讲过的一个实验**：取一台"健康"的 Linux 机器，统计它的进程暂停分布。即使是看起来正常运行的机器，每天都会出现几百次毫秒级到秒级的 **stop-the-world** 事件——这些事件有足够概率拖垮那些 "故障检测阈值设得很激进"的系统。

#### 4.4 拜占庭故障（Byzantine fault）

这是 Leslie Lamport 1982 年 *The Byzantine Generals Problem* 论文引入的概念。

**拜占庭故障** = 节点不仅失效，还可能说谎、发错误消息、恶意行为。

DDIA 里大部分讨论假设的是 crash-stop 或 crash-recovery，不是拜占庭。原因是：

- 大多数企业内部系统的节点是"可信的"，不会故意说谎
- 拜占庭容错（BFT）协议代价极高，至少需要 3f+1 个节点才能容忍 f 个拜占庭节点

但在以下场景，必须考虑拜占庭：

- 区块链（不信任参与者）
- 航空航天（硬件粒子翻转导致错误）
- 安全敏感的跨组织协议

**PBFT**（Practical BFT, Castro-Liskov 1999）、**Tendermint**、**HotStuff** 都是著名 BFT 协议。Libra/Diem 用的是 HotStuff 变种。本课重点仍是 crash fault，但值得知道另一侧世界的存在。

#### 4.5 时钟故障

包括：

- NTP 校时导致时钟跳变
- 时钟漂移
- leap second（闰秒）
- 虚拟化环境里的时钟不稳定
- TSC（时间戳计数器）在某些 CPU 架构上不可靠

时钟问题的可怕之处在于：

它不一定让系统直接宕机，
但会让你的"先后判断""过期判断""租约判断"悄悄失真。

很多工程事故并不是因为数据写不进去，而是因为 **"判断错了时间"**。

### 5. 为什么"超时"既重要又不可靠

几乎所有分布式系统都依赖 timeout：

- RPC 超时
- 心跳超时
- 锁租约超时
- 主节点租约超时

超时机制非常重要，因为如果没有超时，系统会无限等待。

但超时机制也天生不可靠，因为：

**超时只能说明"在这段时间里没收到结果"，不能说明"对方一定失败"。**

这会带来一系列经典难题。

#### 5.1 请求超时后，操作到底有没有执行成功

客户端发起扣款请求：

1. 请求到达服务端
2. 服务端已执行成功
3. 返回包丢失
4. 客户端超时

客户端此时看到的是"失败"。
但真实世界里操作其实已经成功。

如果客户端直接重试，又没有幂等保护，就可能重复扣款。

所以：

**超时之后最危险的不是"失败"，而是"不知道到底成没成功"。**

这就是为什么工程里总要反复强调：

- 幂等键
- 去重
- 请求号
- 至少一次 / 至多一次 / 恰好一次语义

**Nathan Marz 的经典观点**：分布式系统里不存在真正的"恰好一次"。任何所谓 exactly-once 语义都是在 **at-least-once + 幂等** 上模拟出来的。Kafka 0.11+ 的 "exactly-once" 也是靠幂等生产者 + 事务性消息实现的——不是真的只送一次，而是 **effectively-once**。

#### 5.2 心跳超时后，节点到底是不是该被踢出集群

同理：

- 超时太短：容易误杀健康节点
- 超时太长：故障恢复太慢

这意味着超时永远是一个权衡：

- 敏感 → 快恢复，但误判多
- 保守 → 稳一点，但故障切换慢

这就是为什么现实系统里：

**故障检测不是一个精确问题，而是一个阈值调优问题。**

**Phi Accrual Failure Detector**：Cassandra 用的故障检测器（Hayashibara 等 2004 年论文）不用固定阈值，而是根据历史心跳的分布计算一个"怀疑度 φ"。φ 越大越可能挂了。这把二元"存活/死亡"变成了连续信号，上层可以根据不同操作挑不同的 φ 阈值。

### 6. 同步模型、异步模型、部分同步模型到底在说什么

这几个词在分布式理论里非常关键。

#### 6.1 同步模型（synchronous model）

它假设：

- 消息传输有已知上界
- 进程执行速度有已知上界
- 时钟误差有已知上界

这相当于一个"理想而受控"的世界。

在这种世界里：

- 你可以根据超时非常可靠地判断谁挂了
- 很多协议能更简单地成立

但现实互联网系统通常不满足这个模型。

#### 6.2 异步模型（asynchronous model）

它假设：

- 消息可能任意延迟
- 进程可能任意慢
- 没有可靠时钟上界

这几乎是最悲观的模型。

在这个模型下，一个很残酷的结论出现了：

> 你无法仅凭"长时间没响应"来确定一个节点真的挂了。

因为它也可能只是：

- 非常慢
- 非常卡
- 网络延迟非常大

这和真实分布式系统很像，所以很多 impossibility result（不可能性结论）都建立在异步模型上。

#### 6.3 FLP 不可能性定理：1985 年的一记重锤

Fischer、Lynch、Paterson 1985 年的论文 *Impossibility of Distributed Consensus with One Faulty Process* 证明了：

**在纯异步模型下，即使只有一个节点可能崩溃，也不存在确定性共识协议能在有限时间内总是达成一致。**

这个定理的直觉是：在纯异步模型里，你无法区分"节点挂了"和"节点只是很慢"。任何尝试"等一段时间再决定"的算法都可以被对抗性调度搞得永远无法终止。

**FLP 给工程实践带来的三个重要启示**：

1. **不可能绕过，只能软化假设**：所以工程系统都走"部分同步"假设（Dwork、Lynch、Stockmeyer 1988 年论文）。
2. **引入随机性或故障检测器可以绕过**：Ben-Or 的随机共识算法、Raft/Paxos 的 leader election 都是这种做法。
3. **工程里的共识协议不是在"任何情况下都终止"，而是"网络正常时终止、网络极端时可能无限停滞"**。Raft 在网络抖动期间会反复选举无法产生结果，这不是 bug，是理论必然。

#### 6.4 部分同步模型（partially synchronous model）

现实工程更接近这个模型。

它的感觉是：

- 平时网络和机器大体正常，有某种经验上的延迟范围
- 但偶尔会出现不可预期的抖动、暂停和异常

所以大多数工程系统都是：

**假装世界大部分时间是"近似同步"的，但设计时必须承认偶尔会掉进异步深渊。**

这也是为什么：

- 超时机制有用，但不可靠
- 共识协议能在"最终恢复正常"后继续推进
- 很多系统都在赌"绝大多数时间网络是好的"

**GST（Global Stabilization Time）模型**：部分同步的严格定义是——存在某个未知时间点 GST，从那之后系统变同步。GST 之前可以完全异步。共识协议必须保证 **安全性**（从不说出错误答案）在任何时候都成立，**活性**（最终产生答案）在 GST 之后成立。这就是为什么 Raft 在 GST 之前可能无限选举——它放弃了活性以保护安全性。

### 7. 时钟：分布式系统里最容易被骗的东西之一

很多人第一次做分布式系统时，会天然觉得：

> 每台机器都有时间，那我比较时间戳不就行了？

这通常是踩坑的起点。

#### 7.1 物理时钟（wall clock）

也就是你平时看到的系统时间：

- 2026-04-17 14:30:05

它适合：

- 给人看
- 打日志
- 粗粒度过期判断

但它不适合直接用来判断事件全局顺序，因为：

- 不同机器不完全同步
- NTP 可能调整时间
- 时间可能往前跳，也可能往后拨

#### 7.2 单调时钟（monotonic clock）

单调时钟不关心真实日期，只关心"经过了多久"。

它适合：

- 测量超时
- 测量请求耗时
- 重试退避

为什么要强调它？

因为如果你用 wall clock 计算超时：

- 开始时间 10:00:00
- 结束时 NTP 把机器时间拨回 09:59:59

你就会得到一个负耗时。

所以：

**超时计算、延迟统计应该尽量基于单调时钟，而不是墙上时间。**

Linux 下的 API：
- `CLOCK_MONOTONIC`：系统启动以来的时间，不受 NTP 影响
- `CLOCK_MONOTONIC_RAW`：甚至不受 NTP slew 调整
- `CLOCK_BOOTTIME`：包括挂起时间

Go 的 `time.Since()` 和 Rust 的 `Instant` 都内部用 monotonic clock，这是现代语言的默认做法。

#### 7.3 时钟漂移（clock drift）

两台机器的晶振不可能完全一样。

即使都在走，速度也会有细微差异。

积累起来，就会变成：

- A 机器总比 B 快 50ms
- 过一段时间后差到几百毫秒甚至几秒

**典型数值**：
- 普通晶振：drift ~30 ppm（parts per million）≈ 每天漂移 2.5 秒
- 温度补偿晶振（TCXO）：~1 ppm
- 原子钟：~10^-9 级

这在以下场景特别危险：

- 租约锁
- 分布式主节点选举中的 lease
- 依赖时间戳做 LWW 冲突解决

#### 7.4 NTP 不是魔法同步器

很多系统设计里会写一句：

> "所有机器通过 NTP 同步时间。"

这句话不能等价成：

> "所有机器时间准确一致。"

因为 NTP 本身也有：

- 精度误差（局域网内通常几 ms，跨互联网可达 100ms+）
- 网络延迟影响
- 大步校时（step）——时间突然跳跃
- 平滑校时（slew）——慢慢调整速率

它能帮助系统时间大体对齐，
但不能把时间变成一个强一致真相源。

**Marzullo 算法**：NTP 的核心算法（Marzullo 1983）其实已经显式处理"多个时间源说法不一致"的问题——它计算多个区间的交集作为"最可能的真实时间区间"。但这只是减少误差，不是消除。

#### 7.5 Leap second 为什么麻烦

闰秒这事看起来像日历知识，但工程上真的很麻烦。

如果系统时间里突然多出一秒：

- 某些时间戳可能重复
- 某些依赖"时间严格递增"的逻辑会崩
- 排序和 TTL 判断会出现边界问题

**真实事故清单**：

- **2012 年 7 月 1 日**：闰秒触发 Linux 内核 bug，导致 Reddit、LinkedIn、Mozilla、Yelp 等网站的 Java 服务崩溃（Futex 相关死锁）。
- **2015 年 6 月 30 日**：Cloudflare 的 DNS 服务在闰秒瞬间计算出 "负时间差"，触发 panic。
- **2016 年 12 月 31 日**：Cloudflare 再次中招，部分 Go 服务因为 `time.Now()` 在 leap second 周围产生负 duration 而崩溃。这个事件直接推动了 Go 1.9 为 `time.Now()` 加入 monotonic clock 的内部记录。

所以很多大厂后来选择：

- **Google 的 smeared time**：把闰秒均匀摊薄到 20 小时内的每一秒里，让系统永远不会看到"多出的一秒"
- **AWS Time Sync Service**：也用 smearing
- 或者尽量避免把 wall clock 直接用于关键正确性判断

#### 7.6 Spanner TrueTime：把时钟不确定性显式化

Google Spanner 论文（OSDI 2012）提出了一个让业界震撼的设计：**接受时钟误差，但显式表达它**。

TrueTime API 返回的不是一个时间点，而是一个 **区间**：

```
TT.now() → [earliest, latest]
```

保证真实物理时间 **一定** 落在这个区间里。区间宽度 `ε` 通常是几毫秒。

Spanner 的数据中心里装有 **GPS 接收器 + 原子钟** 双源，每台机器通过 time master 定期同步，ε 被压到 1-7ms。

有了 TrueTime，Spanner 就能做一件革命性的事——**外部一致性**（相当于严格可串行化）：

- 给每个事务分配一个 commit 时间戳 `s`
- 在提交之前，**等待** 直到 `TT.now().earliest > s`（commit wait）
- 这样后续任何事务看到的开始时间都 > s
- 全局时序因此严格等于真实物理时序

这个设计用 **等待 ε** 换来了 **全球一致时序**。ε 越小，等待越短，吞吐越高。所以 Spanner 要在硬件上砸钱把 ε 压小。

**CockroachDB、YugabyteDB** 没有原子钟，改用 **Hybrid Logical Clock (HLC)** 作为替代（见下节）。ε 更大，提供的语义稍弱，但能在普通云主机上运行。

### 8. 既然物理时间不可靠，那怎么表达"先后关系"

这就要引入逻辑时钟。

#### 8.1 Lamport 时间戳

Lamport 1978 年的论文 *Time, Clocks, and the Ordering of Events in a Distributed System* 是分布式系统最重要的论文之一（引用次数超过 15,000）。

它解决的问题是：

**如果事件 A 先于事件 B 发生，并且这个"先于"是因果性的，那么时间戳也要反映这个顺序。**

它不要求还原真实时间，
而是让"有因果关系的事件"至少不要被排序反。

核心思想很简单：

- 每个进程维护一个逻辑计数器
- 本地事件发生就加一
- 发送消息时带上计数器
- 接收消息时把自己的计数器更新为 `max(本地, 对方) + 1`

这样可以保证：

- 如果 A 影响了 B，那么 `ts(A) < ts(B)`

用 Lamport 自己的符号表达：A → B（A happens-before B）⇒ ts(A) < ts(B)。

但 Lamport clock 的局限也很重要：

**它能表达"有因果关系的先后"，但不能判断"两个事件是否并发无关"。**

也就是：ts(A) < ts(B) **不一定** 意味着 A → B。可能它们完全没因果，只是计数器大小巧合。这就是向量时钟要解决的问题。

#### 8.2 版本向量 / 向量时钟（vector clock）

Fidge 1988、Mattern 1989 独立发明。

向量时钟进一步解决的问题是：

**不仅要知道谁先谁后，还要知道两个事件是不是彼此并发。**

它通常长这样：

- 节点 A: `[3,1,0]`
- 节点 B: `[2,2,0]`

规则：

- 本地事件发生时，自己的那一维加一
- 发送消息时，带上整个向量
- 接收消息时，每一维取 max，然后自己那一维加一

比较：

- `V1 < V2` 当且仅当 V1 的每一维 ≤ V2 对应维，且至少一维严格小于
- 如果两者互相都不是"逐维小于等于"，那就说明它们并发

这对多主复制、离线修改、冲突合并特别关键。

因为你终于能回答：

- 这是后续更新
- 还是两个并发分叉版本

**Dynamo 用的是"version vector"**，和 vector clock 类似但略有区别：version vector 记录每个客户端/副本的版本，只在副本数有限时成立。**Riak、Voldemort** 也用。

**缺点**：向量规模随节点数线性增长。节点数 1000 时每个对象的 metadata 就有 1000 个计数器——这在大规模系统里变得不可承受。所以工业实践里常用：

- **Dotted Version Vectors**（Riak 后来改进版）
- **version pruning**（丢掉长时间未活动节点的维度）
- **Server-side timestamps**（降级到 LWW）

#### 8.3 Hybrid Logical Clock (HLC)：把 wall clock 和 logical clock 融合

Kulkarni 等 2014 年提出的 HLC 是现代分布式数据库（CockroachDB、MongoDB、YugabyteDB）的核心时钟机制。

核心思想：

- HLC 时间戳 = `(physical_time, logical_counter)`
- 尽量贴近 NTP 时间（让人类能读懂、能和外部系统对齐）
- 但在 NTP 说谎或者同一 NTP 毫秒内多个事件时，用 logical_counter 保证严格单调递增

伪代码：

```
on_send:
  l' = max(l, pt.now())
  if l' == l: c = c + 1 else c = 0
  return (l', c)

on_receive(l_remote, c_remote):
  l' = max(l, l_remote, pt.now())
  if l' == l == l_remote: c = max(c, c_remote) + 1
  elif l' == l: c = c + 1
  elif l' == l_remote: c = c_remote + 1
  else: c = 0
```

HLC 的价值：

- 和 wall clock 贴合度好，适合做 TTL、运维分析
- 同时保证因果顺序（Lamport 性质）
- 比 vector clock 便宜得多（固定大小）

CockroachDB 内部用 HLC 给所有事务分配时间戳。Spanner 不用 HLC 是因为它有 TrueTime 更精准。

#### 8.4 因果关系（causality）才是很多系统真正关心的顺序

现实系统里，不是所有顺序都重要。

重要的是：

> 哪些事件是彼此有因果依赖的？

例如聊天系统：

- "你周末有空吗？"
- "有"

这两个消息显然有因果关系。

如果展示顺序颠倒，就很奇怪。

但两个人分别在不同群里说的话，通常没有强因果约束。

所以：

**很多系统真正想维护的不是"全局绝对顺序"，而是"因果顺序不要乱"。**

**COPS** (Lloyd et al. 2011)、**Eiger**、**Bolt-on Causal Consistency** 这些系统专门研究"只提供因果一致性，放弃更强的线性一致"。Facebook 用过这个思路做跨地域缓存同步。

### 9. 一致性到底在承诺什么

"一致性"这个词在工程里经常被说得太泛。

我们这里专门把它拆开。

#### 9.1 最终一致（eventual consistency）

承诺的是：

> 如果没有新的更新，所有副本最终会收敛到同一个值。

注意它没有承诺：

- 什么时候收敛
- 收敛前读到什么
- 是否会读到旧值

所以最终一致更像一个"长期结果承诺"，而不是"瞬时读写语义承诺"。

Werner Vogels（AWS CTO）2008 年的那篇文章 *Eventually Consistent* 是工业界普及这个概念的起点。DynamoDB、S3（早年）都走这条路。

**一致性强度不足时怎么弥补**：读写 quorum（R+W>N）、read-repair、anti-entropy（Merkle tree）——但这些都只是"概率上更一致"，不改变最终一致的本质。

#### 9.2 线性一致（linearizability）

Herlihy 和 Wing 1990 年论文 *Linearizability: A Correctness Condition for Concurrent Objects* 给出了严格定义。

它要求：

**每个操作看起来都像在某个瞬间原子生效，并且这个瞬间必须落在调用与返回之间。**

这个"瞬间"叫 **linearization point**（线性化点）。

你可以把它理解成：

- 调用完成返回之后，后面的读不能再看不到这个写
- 所有人看到的是一个与真实时间一致的单一全局顺序

为什么它直观？

因为它最符合人的常识：

> "我刚写成功，你再读就该看得到。"

**线性一致 vs 原子**：线性一致 = 原子 + 实时顺序。单机 CAS 指令是线性一致的例子。分布式锁（正确实现的）应该是线性一致的。

#### 9.3 顺序一致（sequential consistency）

Lamport 1979 年定义。

顺序一致比线性一致弱一点。

它只要求：

- 所有操作存在一个大家共同认可的顺序
- 每个进程自己的操作顺序被保留

但不要求这个顺序一定和真实时间对齐。

也就是说：

- 你可能看到一个全局统一顺序
- 但这个顺序未必尊重"谁先返回、谁后返回"的真实时序

**一个反例**：
- 10:00:00 A 完成写 x=1
- 10:00:05 B 完成写 x=2
- 某个读者看到顺序是 "x=2 然后 x=1"

这违反线性一致（真实时间 A 先），但可能满足顺序一致（只要所有读者看到同样的顺序）。

#### 9.4 因果一致（causal consistency）

因果一致关心的是：

- 有因果关系的更新，大家看到的顺序不能乱
- 无因果关系的并发更新，顺序可以因人而异

它通常比线性一致便宜（不需要全球协调），
但比最终一致更符合很多协作场景的直觉。

社交动态、聊天、评论线程这类系统，对因果一致通常很敏感。

**一致性语义强度谱**（从弱到强）：

```
Eventual  <  Read-your-writes  <  Monotonic Reads
         <  Causal  <  Sequential  <  Linearizable  <  Strict Serializable
```

其中 "Read-your-writes" 和 "Monotonic Reads" 来自 Bayou 论文（Terry 1994），属于 **session guarantees**（会话一致性），是工程里最常见的中间强度。

### 10. 线性一致和可串行化不是一回事

这是非常容易混淆的一点。

上一章我们讲过可串行化。

可以这样区分：

- **可串行化（Serializability）**：多个事务的结果，等价于某种串行顺序（事务范畴，多对象）
- **线性一致（Linearizability）**：单对象或操作的外部可见顺序，和真实时间一致

两者关注点不同：

#### 10.1 可串行化关注的是事务整体正确性

它关心：

- 并发执行会不会破坏业务约束

#### 10.2 线性一致关注的是外部观察到的实时顺序

它关心：

- 一个操作返回成功后，别人是不是立刻就必须看到

**四象限表**（DDIA 里著名的分类）：

|  | 非严格时序 | 严格实时时序 |
|---|---|---|
| **单对象** | Sequential 读写 | Linearizable 读写 |
| **多对象事务** | Serializable | Strict Serializable（= Linearizable + Serializable） |

- **Serializable**：PostgreSQL SSI、MySQL SERIALIZABLE
- **Linearizable**：etcd、ZooKeeper、CAS 寄存器
- **Strict Serializable**：Spanner、CockroachDB、FoundationDB
- **Sequential**：某些主从复制系统

所以一个系统可能：

- 事务可串行化，但单对象读写不线性一致
- 单对象线性一致，但跨对象事务不一定可串行化

这个区分后面看共识、主从复制和 quorum 系统时非常重要。

### 11. 为什么线性一致很贵

线性一致看起来很合理，但代价通常不低。

因为要满足它，系统通常必须做到：

- 对"谁是最新值"形成统一判断
- 对网络分区非常谨慎
- 在不确定时宁可拒绝服务，也不能瞎承诺

换句话说：

**线性一致本质上是在和"不确定性"硬碰硬。**

而分布式系统里，不确定性恰恰很多：

- 网络会抖
- 节点会暂停
- 时钟不准
- 包会丢

这就是为什么很多系统不会默认对所有操作都提供线性一致。

#### 11.1 Attiya-Welch 下界：线性一致的不可避免代价

Attiya 和 Welch 1994 年证明：

**在异步网络中，线性一致的读 / 写操作延迟至少是网络直径的 O(d)**。

具体说：如果系统跨两个数据中心，两个 DC 的 RTT 是 50ms，那线性一致的读写至少一半要慢 50ms——这是理论下限，不是实现问题。

这个下界直接解释了为什么：

- Spanner 写入在跨地域场景下至少 50-100ms
- etcd/ZooKeeper 在高频写场景下吞吐有限
- CockroachDB 多区域部署的写延迟明显高于单区域

**实践应对**：
- 只在 **必须** 强一致的地方用线性一致（配置、元数据、分布式锁）
- 业务主流量尽量走弱一致（本地快速响应）
- 引入 **follower reads / stale reads** 绕过 leader

### 12. CAP 为什么有用，但又不够用

CAP 的经典表述是：

在网络分区发生时，
一致性（C）和可用性（A）不能同时完美满足，
必须做取舍。

这是 Eric Brewer 2000 年 PODC 主题演讲 提出，Gilbert 和 Lynch 2002 年给出形式化证明。

这句话是对的，但它常被误用。

#### 12.1 CAP 真正强调的是"分区发生时"

不是平时。

也就是说 CAP 讨论的是：

> 当网络已经把系统切成几块时，你要继续强一致，还是继续响应请求？

所以它更像：

- 极端条件下的下限约束

而不是：

- 平时所有系统设计的全部指南

#### 12.2 CAP 的三个常见误读

Kleppmann 2015 年专门写过文章 *A Critique of the CAP Theorem* 指出 CAP 被误读：

1. **"一致性 C" 只意味着线性一致**（不是事务可串行化、也不是业务一致性）。所以 CAP 的 C 比 ACID 的 C 窄得多。
2. **可用性 A 有严格定义**（每个请求必须返回非错误响应），而不是工程里说的 99.99% 那种统计可用性。
3. **CAP 不告诉你分区外怎么做取舍**——而分区外的延迟和一致性取舍才是日常工作。

所以 CAP 的三选二说法（"CA、CP、AP 挑一个"）其实是误导：**任何分布式系统都必须在分区时放弃 C 或 A**，所以只能选 CP 或 AP，没有"CA"这个选项。

#### 12.3 实际工程里更常见的是"延迟 vs 一致性"

很多时候没发生网络分区，
系统也仍然在做权衡：

- 要不要等远端副本确认？
- 要不要同步复制？
- 这次读能不能容忍旧值？

这就是为什么 PACELC 往往比 CAP 更贴近工程（Daniel Abadi 2010 年提出）：

- **P**artition 时：**A**vailability 和 **C**onsistency 取舍
- **E**lse（没分区时）：**L**atency 和 **C**onsistency 取舍

PACELC 分类举例：
- **PA/EL**：Cassandra、Dynamo、Riak（分区时保可用，平时保低延迟）
- **PA/EC**：MongoDB（分区时保可用，平时保一致）
- **PC/EC**：Spanner、etcd、CockroachDB、HBase（任何时候都保一致）

这张分类表几乎覆盖了所有主流分布式数据库，比 CAP 的"三选二"有用得多。

### 13. 一个非常关键的结论：很多所谓"故障"，本质是知识不完整

这是分布式系统最深的一层难点之一。

你可以把很多问题理解成：

**不是系统不会处理，而是系统不知道现在到底发生了什么。**

例如：

- 节点是不是挂了？
- 请求是不是已经执行了？
- 我看到的是不是最新值？
- 对面是不是也认为自己是 leader？

这些问题的共同点都是：

> 你缺少全局真相。

所以分布式系统设计，本质上经常是在做两件事：

1. 如何在信息不完整时，安全地做决策
2. 如果暂时做不了安全决策，应该如何拒绝、等待、重试或降级

这正是下一章共识系统要解决的问题。

#### 13.1 Fencing Token：防止"失效节点还在作祟"

一个经典场景：客户端 A 拿到分布式锁，开始写数据。A 进入 GC 暂停 30 秒。锁过期。客户端 B 拿到锁，开始写。然后 A 从 GC 醒来，带着过期的锁继续写——污染数据。

解决方案是 Kleppmann 特别强调的 **fencing token**：每次发放锁时附一个严格递增的 token。资源服务器检查请求的 token，拒绝比已见过的最大值更小的 token。

这样 A 醒来时带着 `token=33` 想写，但资源服务器已经处理过 B 的 `token=34`，A 的请求被拒绝。

**Chubby、ZooKeeper 都原生支持 fencing token**。Redis 的 SETNX 分布式锁默认 **不支持**——这是 Kleppmann 和 antirez 2016 年 "Redlock" 之争的核心。Kleppmann 认为没有 fencing 的分布式锁在严格正确性上不成立。

#### 13.2 STONITH: Shoot The Other Node In The Head

有些场景 fencing token 不够（比如老系统不支持 token）。工业界还有更粗暴的方案：**STONITH**——直接通过带外通道（IPMI、远程电源管理）把疑似失效的节点强制关电。

这听起来很暴力，但在高可用集群（Pacemaker、Corosync）里是标准做法。代价是你必须有带外管理网络，而且"错杀"风险很高。

### 14. 一个后端工程师最实用的分布式判断框架

以后你看一个跨节点系统，可以先按这几个问题切：

#### 14.1 这个系统假设网络是什么样的

- 同机房？
- 跨可用区？
- 跨地域？

网络条件不同，时延和抖动完全不同。典型数值：

- 同机架内：RTT < 0.1ms
- 同数据中心：RTT 0.5-2ms
- 同地域跨 AZ：1-5ms
- 跨大陆：60-150ms

这些数量级会直接决定一致性协议的吞吐上限。

#### 14.2 这个系统如何判断故障

- 心跳？
- lease？
- timeout？
- Phi Accrual？
- 外部协调器？

它有没有把"暂停"误判成"死亡"的风险？

#### 14.3 它依赖哪种时间

- wall clock？
- monotonic clock？
- logical clock（Lamport / vector / HLC）？
- TrueTime？

它有没有把"物理时间"错误地拿来表达因果顺序？

#### 14.4 它提供的是什么一致性承诺

- 最终一致？
- 读己之写 / 单调读？
- 因果一致？
- 顺序一致？
- 线性一致？
- 事务可串行化？
- 严格可串行化？

不要只问"它一致吗"，要问"一致到什么程度"。

#### 14.5 当系统无法确定时，它选择什么

- 阻塞等待？
- 拒绝服务？
- 返回旧值？
- 允许分叉？
- 交给人工恢复？

这往往决定了系统故障时的真实行为。

#### 14.6 它对拜占庭威胁有没有假设

- 节点被信任，只会崩溃或失联？
- 或节点可能说谎、发错数据？

后者需要 BFT 协议，成本完全不同。

### 15. 这一章和下一章怎么连起来

这一章的核心结论可以压缩成一句话：

> 在分布式环境里，时间不可靠，网络不可靠，故障判断不可靠，而系统又必须继续运行。

这时自然就会产生下一个问题：

> 如果多个节点必须对"谁是 leader""某条日志是否提交""某个值到底是什么"形成统一决定，应该怎么办？

这就是第 9 章"共识"的出场原因。

第 8 章提供的是：

- 你为什么不能轻信时间
- 你为什么不能轻信超时
- 你为什么不能轻信单节点观察

第 9 章要回答的是：

- 在这些都不可靠的前提下，怎样形成尽量可靠的共同决定

#### 15.1 本章概念在下一章的去向

| 本章概念 | 第 9 章怎么用 |
|---|---|
| FLP 不可能性 | 解释为什么共识协议都需要部分同步假设 |
| 心跳 / lease | 构成 leader election 的基础 |
| fencing token | 变成 Raft 里的 term number、Paxos 里的 ballot number |
| linearizability | 成为共识协议的 **安全性** 目标 |
| Phi Accrual | 决定何时触发重新选举 |
| GC 停顿 | 决定 election timeout 的下限 |
| TrueTime / HLC | 成为分布式事务时序化的基础 |

## 小结（10 条关键点）

- **分布式系统一旦跨机器，最难的不是数量，而是失去全局视角**：你只能看到局部现象，却必须做全局决策。Waldo 1994 年早已说清："本地调用和远程调用永远不是一回事"。
- **Deutsch 八大谬误是所有新手都会犯的错**：网络可靠、延迟为零、带宽无限、拓扑稳定——这些默认假设在生产环境里都要被打脸。
- **部分失败比整体失败更麻烦**：节点可能没死，只是网络隔离、进程暂停或调度卡顿；这会让"没响应"变成一个含义极其模糊的信号。
- **FLP 1985 年不可能性定理是分布式系统的底色**：纯异步模型下不存在确定性共识。工程必须退让到部分同步假设，或引入随机性绕开。
- **超时不是故障证明，只是一个推断阈值**：它很重要，但永远有误判；所以故障检测本质上是近似判断，不是精确读状态。Phi Accrual 是把二元判断变成连续信号的改进。
- **物理时钟不能直接当作全局顺序来源**：NTP、漂移、闰秒、虚拟化暂停都会让 wall clock 失真；2012 和 2015 年的 Leap Second 灾难把这个问题刻进了每个 senior 的肌肉记忆。测耗时应优先用单调时钟，表达因果则要靠逻辑时钟。
- **Lamport → Vector → HLC → TrueTime 是时钟进化四阶段**：Lamport 表达因果顺序；Vector 还能识别并发；HLC 把物理和逻辑融合用于现代数据库；TrueTime 用硬件把不确定性压到几 ms 用于 Spanner。
- **一致性不是一个词，而是一组不同强度的语义承诺**：最终一致、读己之写、单调读、因果一致、顺序一致、线性一致、严格可串行化——它们形成一个严格的强度谱系，工程必须精确选择。
- **线性一致和可串行化不是一回事**：前者关注单对象实时可见顺序，后者关注多事务并发结果是否等价于串行执行；二者都会连到下一章的共识。组合起来是 strict serializable（Spanner 级别）。
- **CAP 不够用，PACELC 更贴近工程**：CAP 只谈分区时的极端取舍，PACELC 补上"平时也在做延迟 vs 一致性取舍"这个日常维度——PA/EL（Cassandra）、PA/EC（MongoDB）、PC/EC（Spanner）是三大类典型系统。
