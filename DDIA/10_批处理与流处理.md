# DDIA - 第 10 课：批处理与流处理

## 学习目标（本节结束后你能做到什么）

- 理解为什么 DDIA 在讲完存储、复制、分区、事务、共识之后，还必须继续讲数据处理。
- 知道**数据库、搜索引擎、缓存、数据仓库、流计算结果**本质上都可以看成“由原始数据推导出来的物化视图”。
- 能区分 **OLTP / OLAP、批处理 / 流处理、bounded / unbounded data、event time / processing time** 这些核心概念。
- 理解 MapReduce、Spark、Flink、Kafka Streams、Beam 背后真正共同的思想：**数据流（dataflow）和重放（replay）**。
- 知道批处理为什么依然重要，为什么它不是“被流处理淘汰的旧时代技术”。
- 掌握流处理里最难的几个点：**乱序、迟到事件、窗口、watermark、状态管理、checkpoint、重放、幂等 sink**。
- 理解为什么“exactly-once”通常不是一句营销口号就能成立，而是输入、状态、输出共同配合的结果。
- 理解 CDC、事件日志、事件溯源、流式物化视图之间的关系。
- 能根据业务目标判断什么时候该用批处理，什么时候该用流处理，什么时候两者应该组合。
- 能把 DDIA 前九章的内容，和后续的数据处理架构真正接起来。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么讲完“存数据”，还要讲“处理数据”

前九章我们一直在回答这些问题：

- 数据怎么表示
- 数据怎么存
- 数据怎么复制
- 数据怎么切分
- 并发下怎么保持正确
- 分布式里怎么在故障下继续协调

这些问题都很重要，但它们还只解决了：

> **原始数据如何被可靠地写进去。**

而现代系统真正每天都在做的另一件事是：

> **把原始数据加工成别的、更适合查询和消费的数据。**

例如：

- 把交易明细汇总成日报表
- 把订单表同步成搜索索引
- 把行为事件流转成实时看板
- 把用户点击日志加工成推荐特征
- 把数据库变更流同步到缓存、下游服务、消息流

所以从系统视角看：

- OLTP 数据库保存“事实源”
- 各种批处理和流处理系统负责生成“派生数据（derived data）”

DDIA 后三章真正关心的，就是这个世界。

### 2. 批处理和流处理不是对立物，而是同一条数据流上的两个视角

很多人第一次学会把它们理解成：

- 批处理：旧、慢、离线
- 流处理：新、快、实时

这个对比太粗了。

更准确的理解应该是：

- **批处理**：输入是有边界的一批数据，重点是吞吐和全量重算
- **流处理**：输入是持续到来的无界事件流，重点是低延迟和持续更新

它们本质上都在做一件事：

**把输入数据通过一串算子，变成新的派生结果。**

所以更深一层看，它们都属于 **dataflow system**。

#### 2.1 bounded vs unbounded 是更底层的区分

比“批”和“流”更基础的，是：

- **bounded data**：输入有明确结束点
- **unbounded data**：输入理论上没有结束点

例如：

- “昨天全天的订单表”是 bounded
- “今天不断产生的下单事件流”是 unbounded

一旦你用这个视角看问题，就会发现：

- 批处理是对 bounded 数据的一次性计算
- 流处理是对 unbounded 数据的持续性计算

### 3. 批处理到底在干什么

批处理不是“拿一堆文件跑个脚本”这么简单。  
它本质上是在做：

**面向完整数据集的、大吞吐量的离线重计算。**

典型场景包括：

- 每日/每小时报表
- 数据清洗与回填（backfill）
- 搜索索引重建
- 机器学习训练样本生成
- 历史数据修正和审计

#### 3.1 为什么批处理依然很重要

很多系统看起来很实时，但只要出现下面这些需求，你就会立刻需要批处理：

- “上个月某个规则写错了，能不能把历史结果重算一遍？”
- “我们要把去年所有订单重新按新口径汇总。”
- “某个流任务出了 bug，过去 7 天的数据要补。”
- “推荐特征模型要基于最近 90 天全量行为重建。”

这类问题的共同点是：

**你需要重新扫描一大批历史数据，并且希望结果是完整、可重放、可核对的。**

这正是批处理的强项。

#### 3.2 MapReduce 解决了什么

MapReduce 真正重要的，不是 `map` 和 `reduce` 这两个单词本身，  
而是它把大规模数据处理拆成了三个核心阶段：

1. **Map**：把输入记录变成中间键值对
2. **Shuffle / Sort**：按 key 重新分组
3. **Reduce**：按组聚合或进一步处理

最关键的是中间这一步：

**大规模数据处理的核心难点，经常不是算，而是重分布（shuffle）。**

例如：

- 按用户统计行为
- 按商品汇总销量
- 按地区聚合订单

你都必须先把“同一 key 的数据”收拢到一起。

这件事在单机里是 hash map，  
在集群里则变成了跨节点重分布和外部排序。

#### 3.3 MapReduce 为什么后来被嫌慢

它慢的根本原因不是“写法过时”，而是它每一阶段都很保守：

- 中间结果大量落盘
- 每一步边界都很硬
- DAG 表达能力有限

Spark、Tez、Flink Batch 后来改进的方向，本质上都是：

- 更灵活的 DAG
- 更少的中间物化
- 更高效的内存使用

但要注意：

**这些系统没有推翻批处理，只是让批处理执行得更灵活。**

### 4. 批处理真正擅长的，是“全量正确性”

如果你问：

> 批处理最核心的优势是什么？

不是“慢但是便宜”，而是：

**它天然适合对完整数据集重新算一遍，并得到一个口径统一的结果。**

这在很多场景里是不可替代的：

- 财务对账
- 审计
- 数据修正
- 指标重算
- 机器学习离线特征生成

流处理擅长“持续更新”，  
批处理擅长“全量重建”。

### 5. 流处理到底在干什么

流处理不是“消息队列多消费一点逻辑”。  
它本质上是在做：

**对持续到来的无界事件流，进行低延迟、持续性、状态化计算。**

例如：

- 实时 PV/UV 看板
- 实时风控
- 实时告警
- 实时画像更新
- CDC 驱动下游物化视图更新

#### 5.1 流处理的关键不是“快”，而是“持续”

流处理和批处理最大的差异不只是延迟，而是生命周期：

- 批任务跑完就结束
- 流任务默认是“长期存在”的

所以流处理系统必须额外处理：

- 状态怎么保存
- 出错怎么恢复
- 重启后怎么从上次继续
- 迟到事件怎么补
- 中间结果什么时候输出

这也是为什么真正成熟的流处理引擎，不只是“消费消息”，而是有完整的：

- state
- checkpoint
- replay
- window
- watermark
- sink semantics

### 6. 流处理最难的第一个点：事件时间不等于处理时间

假设有两条订单事件：

- 订单 A 在 10:00:01 真实发生，但 10:00:20 才到系统
- 订单 B 在 10:00:05 发生，10:00:06 就到了系统

如果你按处理时间算分钟窗口，就会把它们放进不同时间段。  
但从业务上，A 明明更早发生。

这就引出两个概念：

- **Event Time**：事件真实发生的业务时间
- **Processing Time**：系统真正处理到它的时间

在乱序和延迟存在时：

**event time 才更接近业务世界。**

#### 6.1 为什么乱序是默认现象

现实系统里，乱序非常常见：

- 客户端网络抖动
- 重试
- 分区间速度不同
- 上游批量发送
- 跨机房延迟不同

所以真正成熟的流处理系统从来不会假设：

> “先到的事件就一定先发生。”

### 7. 窗口（window）为什么本质上是在给无界流强行加边界

无界流没法直接“算总和”，因为它理论上永远不会结束。  
所以我们必须人为切一刀：

- 按时间切
- 按事件数切
- 按会话切

常见窗口有：

- **Tumbling Window**：固定不重叠窗口
- **Sliding Window**：滑动重叠窗口
- **Session Window**：按用户活动间隔切

窗口本质上是在说：

> “我虽然面对的是无界流，但我现在只关心其中一个可计算的局部区间。”

### 8. Watermark 在解决什么

如果事件会迟到，那窗口什么时候才能认为“差不多齐了，可以出结果”？

这就是 watermark 的意义：

**它不是保证，而是系统对“未来还会不会再来更早事件”的一个进度估计。**

例如：

- watermark = 10:05

大意是：

> 系统认为 10:05 之前的大多数事件应该都已经到了。

它非常像共识里的 timeout：

- 都是近似判断
- 都不能保证百分百正确
- 但工程上必须依赖它继续推进

#### 8.1 为什么 watermark 永远是 trade-off

- watermark 太激进：结果更快，但迟到数据更多
- watermark 太保守：结果更准，但延迟更高

所以 watermark 没有“理论最优值”，只有：

**结合业务乱序分布和结果时效要求做出的工程取舍。**

### 9. 流处理最难的第二个点：状态

很多流处理并不是“来一条算一条就完”。  
真正的业务常常要维护状态：

- 某用户最近 10 分钟点击数
- 某设备最近一次状态
- 某窗口内的聚合值
- 某 session 的累积行为

一旦有状态，你就必须回答：

- 状态存在内存还是磁盘？
- 节点挂了状态怎么恢复？
- 重放事件时如何避免重复累加？

#### 9.1 流处理状态的三种常见持久化方式

##### 9.1.1 周期性 checkpoint

把当前状态定期快照出来。  
优点是恢复直接；缺点是 checkpoint 成本高。

##### 9.1.2 Changelog / WAL

把状态变化本身也写成日志。  
恢复时重放日志即可。

##### 9.1.3 本地状态 + 远程持久化结合

例如：

- 本地 RocksDB 作为热状态
- 远程对象存储 / DFS 保存 checkpoint
- Kafka 保存输入日志和 changelog

这在 Flink、Kafka Streams 等系统里很常见。

### 10. “Exactly-once”为什么经常是一句被过度简化的话

很多宣传会说：

- 我们支持 exactly-once

但这句话如果不带上下文，几乎没有信息量。

真正应该问的是：

#### 10.1 输入端 exactly-once 还是输出端 exactly-once

- 输入能不能避免重复读取？
- 状态更新能不能避免重复应用？
- 输出到外部 sink 时能不能避免重复写？

#### 10.2 系统内部 exactly-once，不代表外部世界 exactly-once

即使流引擎内部通过 checkpoint + replay + transactional sink 做到了比较强的语义，
如果下游是：

- 一个不支持事务的 HTTP 接口
- 一个非幂等外部服务
- 一个没有去重键的数据库表

那最终结果依然可能重复。

所以更稳的说法通常是：

**effectively-once = 幂等输入/状态恢复/事务或幂等 sink 共同作用的结果。**

### 11. 数据库、搜索、缓存、流任务，其实都在维护派生数据

这部分是 DDIA 很漂亮的一个统一视角。

你可以把很多系统统一理解成：

- 原始事实先进入一个事实源
- 后续系统不断从这个事实源派生出新的视图

例如订单系统：

- 订单库保存原始交易事实
- Elasticsearch 保存搜索视图
- ClickHouse 保存分析视图
- Redis 保存热点缓存视图
- Flink 维护实时聚合视图

这些都不是“另一份真相”，而是：

**从原始事实推导出来的不同用途的物化视图。**

### 12. CDC 为什么这么重要

CDC（Change Data Capture）本质上是在说：

> 不要每次都全表扫，而是把数据库的变更流抽出来，驱动下游持续更新。

例如：

- MySQL binlog -> Kafka -> Elasticsearch
- Postgres WAL -> Debezium -> Warehouse
- DynamoDB Streams -> Lambda -> S3 / Redshift

CDC 的价值在于：

- 主库继续负责 OLTP
- 下游索引/报表/缓存通过变更流持续追上

这其实就是把数据库也纳入“流处理世界”。

### 13. 事件日志为什么是流处理世界的脊梁

Kafka、Pulsar 这类日志系统的重要性，不只是它们能传消息。  
更重要的是它们提供了：

- 有序追加
- 可重放
- 消费位置可控
- 多个下游可以各自独立消费

一旦你有一个可重放事件日志，很多事就变简单了：

- 新建一个下游服务，可以从头回放构建自己的状态
- 流任务出 bug，可以从某个 offset 重放
- 历史数据修复，可以重新跑一遍派生逻辑

这也是为什么现代数据架构越来越强调：

**log as source of truth / log as integration backbone**

### 14. Lambda 架构和 Kappa 架构到底在争什么

这两个词在面试里经常被拿出来，但很多人只会背结论。

#### 14.1 Lambda 架构

Lambda 的思路是：

- 批处理负责全量、准确结果
- 流处理负责增量、低延迟结果
- 两边最后合并

优点是：

- 批层兜底，逻辑正确性强

缺点是：

- 同一份逻辑维护两套实现，复杂度高

#### 14.2 Kappa 架构

Kappa 的思路是：

- 尽量只保留一条流处理主线
- 需要重算时，就从日志头开始重放

优点是：

- 一套主逻辑

缺点是：

- 前提是日志可长期保存、流引擎足够强、重放成本可接受

所以现实里很多系统其实是折中：

- 主链路偏 Kappa
- 某些历史回填、修正、机器学习特征仍然依赖批处理

### 15. 如何选择：批处理、流处理，还是两者结合

可以先问这 6 个问题：

1. 结果需要多快？
2. 数据是有界还是无界？
3. 是否经常需要重算历史？
4. 是否能容忍近实时而非实时？
5. 是否需要复杂状态和窗口？
6. 下游输出是否支持幂等或事务？

一般来说：

#### 15.1 更偏批处理的场景

- T+1 报表
- 审计
- 历史回填
- 全量重建

#### 15.2 更偏流处理的场景

- 实时告警
- 实时风控
- 实时看板
- 持续物化视图更新

#### 15.3 必须两者结合的场景

- 实时指标 + 次日校正
- 在线画像 + 离线重建
- 在线推荐特征 + 离线训练样本

### 16. 这一章最容易踩的 6 个误区

#### 16.1 误区一：流处理会取代批处理

不对。  
历史重算、审计、回填、离线训练仍然大量依赖批处理。

#### 16.2 误区二：批处理就是“慢”

不对。  
批处理更准确的标签是“面向完整数据集的大吞吐重计算”。

#### 16.3 误区三：消息队列加几个消费者就等于流处理

不对。  
真正的流处理要解决状态、窗口、乱序、checkpoint、恢复等一整套问题。

#### 16.4 误区四：watermark 是精确边界

不对。  
它只是近似进度估计，本质上是工程猜测。

#### 16.5 误区五：有了 exactly-once 宣传，就不用考虑幂等

不对。  
外部 sink 如果不支持幂等或事务，最终结果仍可能重复。

#### 16.6 误区六：数据库、缓存、搜索、流引擎是互相独立的岛

不对。  
它们经常是同一份事实源导出的不同视图。

## 小结（3-5 条关键点）

- 批处理和流处理不是两个割裂世界，而是对派生数据加工的两种执行形态。
- 批处理强在全量重建和口径统一，流处理强在低延迟持续更新。
- 流处理真正难的点在于乱序、事件时间、窗口、状态管理和恢复，而不是“消费消息”本身。
- CDC 和事件日志把数据库世界与流处理世界连起来，使很多系统都能被理解为“从事实源推导出物化视图”。
- “exactly-once”不是魔法词，必须把输入、状态恢复和输出 sink 一起看。
