# DynamoDB 学习路径与知识图谱

## 学习定位

很多后端工程师第一次接触 DynamoDB 时，最容易出现两种错位：

- 第一种错位是“拿着 MySQL 的脑子看 DynamoDB”。会下意识去想表关联、二级索引越多越好、先建表再慢慢补查询，结果越做越别扭。
- 第二种错位是“知道 DynamoDB 很快很弹性，但不知道它到底为什么快、为什么贵、为什么有些查询就是不该做”。

DynamoDB 真正难的地方，不在 API 本身，而在于它背后的设计哲学完全不同：

- 它不是一个让你随便查的数据库，而是一个要求你**先想清访问模式，再建模**的数据库。
- 它不是“磁盘表 + B+ 树索引”的世界，而是“分区 + KV + 预先设计好的访问路径”的世界。
- 它不是“能不能查出来”的问题，而是“这个查询在 DynamoDB 里是不是自然、是不是稳定、是不是成本可控”的问题。

这一组内容的目标，就是把 DynamoDB 讲成一套完整的工程思维，而不是几条 AWS 面试问答：

- 为什么 DynamoDB 常被拿来做高并发、低运维负担的在线存储？
- 为什么它强调 Partition Key，而不是像 MySQL 那样强调表结构和 Join？
- 为什么 Single Table Design 会被反复提到？
- 为什么 Query 和 Scan 在 DynamoDB 里几乎是两种哲学？
- 为什么 DynamoDB 能自动扩缩容，但你仍然可能被热分区打爆？
- 为什么它支持事务，但大多数系统又不会把事务当核心能力来依赖？

学完之后，你应该能做到：

- 用后端工程师能落地的语言解释 DynamoDB，而不是只会说“这是 AWS 的 NoSQL 数据库”。
- 根据访问模式设计出合理的主键、排序键和索引。
- 看懂 RCU/WCU、强一致读、最终一致读、TTL、Streams、Global Tables 这些概念在系统中的真实含义。
- 能判断 DynamoDB 适不适合某个业务，而不是把它神化成“云上万能数据库”。

## 知识图谱

1. 先建立整体认知
   - DynamoDB 是什么
   - 它和 MySQL、Redis、MongoDB 的差别
   - 它最适合什么场景
   - 它最不适合什么场景
2. 再理解建模核心
   - Access Pattern First
   - 主键设计
   - Partition Key / Sort Key
   - 单表设计的动机
3. 再理解扩展与热点
   - 分区机制
   - 热分区
   - 为什么“高基数、均匀分布”这么重要
4. 再理解读写语义
   - Put / Update / Query / Scan
   - 条件表达式
   - 乐观锁
   - 幂等设计
   - 强一致 / 最终一致
5. 再理解索引体系
   - LSI
   - GSI
   - 稀疏索引
   - 写放大和成本
6. 再理解容量与成本
   - RCU / WCU
   - Provisioned / On-Demand
   - Auto Scaling
   - 成本控制
7. 再看高级能力
   - Transactions
   - Streams
   - TTL
   - DAX
   - Global Tables
8. 最后进入工程实践
   - 分页
   - 1MB 返回限制
   - 批量操作
   - 反模式
   - 面试追问
   - 线上排障

## 为什么这样排序

很多人一上来就学：

- 什么是 GSI
- 什么是 WCU
- 什么是事务

但如果你没有先搞懂 DynamoDB 最核心的一个前提：

**它要求你先围绕访问模式建模**

那后面所有概念都会变成散点知识。

比如：

- 为什么某些查询必须 `Scan`？
- 为什么某些字段必须提前进主键或索引？
- 为什么你明明有索引，成本还是很高？
- 为什么一个热点用户或热点商户会把整张表打得很不稳定？

所以这套路径故意不是“先背功能”，而是：

1. 先建立 DynamoDB 的世界观
2. 再讲建模和主键
3. 再讲分区和热点
4. 再讲一致性、索引、容量
5. 最后再进入 Streams、事务、Global Tables 和排障

这样你学到最后，脑子里会形成一条很稳定的因果链：

```mermaid
flowchart LR
    A["访问模式"] --> B["主键与索引设计"]
    B --> C["分区分布与热点"]
    C --> D["读写性能与成本"]
    D --> E["一致性、事务、复制与工程边界"]
```

## 学习方法建议

- 每学一课都先问一句：DynamoDB 这套设计在替你省掉什么，又强迫你提前承担什么？
- 看到任何一个 API 或特性时，都不要只记“怎么用”，还要记“为什么需要它”和“它的代价是什么”。
- 学 DynamoDB 建模时，一定要先写出访问模式清单，再设计主键，不要反过来。
- 学索引时，记得始终把“写放大”和“成本增长”放在脑子里。
- 学 Streams、TTL、DAX、事务时，不要只背功能，要问自己：这个能力到底该不该依赖在核心链路上？

## 学完后你应该能回答的问题

- 为什么 DynamoDB 不是“云上的 MySQL 替代品”？
- 为什么在 DynamoDB 里，主键设计和访问模式几乎是一回事？
- 什么样的 Partition Key 容易出热点？
- Query 和 Scan 的差别到底有多大，为什么很多人把表用着用着就用崩了？
- GSI 和 LSI 各自在解决什么问题，各自的代价是什么？
- DynamoDB 为什么看起来不用运维，但建模要求反而更高？
- 什么场景适合用 DynamoDB，什么场景应该回到关系型数据库？
