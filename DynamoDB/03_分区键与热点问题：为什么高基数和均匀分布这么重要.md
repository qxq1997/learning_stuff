# DynamoDB - 第 3 课：分区键与热点问题：为什么高基数和均匀分布这么重要

## 学习目标（本节结束后你能做到什么）

- 理解 DynamoDB 的扩展能力为什么和分区密切相关。
- 明白 Partition Key 设计不当为什么会引发热分区。
- 说清楚“高基数、均匀分布”到底是在解决什么问题。
- 能看懂常见的 DynamoDB 热点场景，并知道基本缓解思路。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么分区是 DynamoDB 的核心

DynamoDB 能支撑大规模吞吐，不是因为一张逻辑表有魔法，而是因为底层会把数据拆分到多个物理分区上。

你可以把它理解成：

- 逻辑上看是一张表
- 物理上看，其实是一堆分区共同提供服务

那么这里马上就会出现一个关键问题：

**你的请求会不会均匀地打到这些分区上？**

如果能，系统就很稳定。
如果不能，某个分区就会先撑不住。

### 2. 热分区到底是什么

热分区（Hot Partition）本质上就是：

某一个或少数几个分区，承担了明显过多的流量。

这通常不是“整张表容量不够”，而是：

- 总流量也许还能扛
- 但流量太集中
- 某一个分区先爆

这和 Redis 里的 hotkey 很像，只不过这里热点不是单个 key，而是一个分区维度的问题。

### 3. 为什么 Partition Key 会直接影响热点

因为 DynamoDB 是根据 Partition Key 的值来决定数据怎么路由、怎么聚集的。

如果你的 Partition Key 很集中，比如：

- 所有热门订单都落在同一个商户 ID
- 所有日志都写进今天这一个日期键
- 所有秒杀请求都打到同一个商品键

那这些数据就会天然往同一组分区上聚。

这时候即便你的总体请求量不算特别夸张，也可能被热点打崩。

### 4. 什么叫高基数

高基数，说白了就是：

**Partition Key 的不同取值足够多。**

比如：

- 用户 ID 通常比“订单状态”基数高
- 商户 ID 通常比“省份”基数高
- 随机请求 ID 通常比“今天日期”基数高

高基数的好处是：

- 更容易把请求和数据打散
- 不容易形成大面积集中热点

但要注意，高基数不是唯一条件。

### 5. 为什么只高基数还不够，还要均匀分布

举个例子：

如果有 1000 万用户，看起来用户 ID 基数很高。
但如果 80% 的流量都打在其中 100 个超级大户身上，那一样会热点严重。

所以真正重要的是两件事：

1. 值的种类够多
2. 访问尽量分散

这也是为什么大家总说：

**Partition Key 要高基数、尽量均匀。**

### 6. 哪些字段特别容易选错

下面这些字段，做 Partition Key 时要特别小心：

- 日期（尤其是“今天”这种）
- 状态（例如 `PENDING`、`PAID`）
- 地区（例如省份、城市）
- 热门商户或热门商品 ID

这些字段往往有一个共同点：

- 取值种类有限
- 或者流量天然集中

所以它们很适合作为过滤条件、排序维度或索引的一部分，但未必适合作为主分区键。

### 7. 一个典型反例：按天写订单日志

假设有人这么设计：

- `PK = ORDER_DAY#2026-04-15`
- `SK = ORDER#123456`

看起来很直观，但问题也很明显：

- 今天所有订单都挤进一个分区键
- 只要今天交易量大，这个键就必热

这类设计在低量时还能跑，一到真实流量就会出问题。

### 8. 更合理的思路是什么

还是用订单日志为例，如果你最常见的访问是“按用户查最近订单”，那就该优先考虑：

- `PK = USER#1001`
- `SK = ORDER#时间#订单号`

如果你真的还要按天聚合，那通常需要：

- 单独的索引
- 或下游分析链路
- 或加散列前缀做分桶

而不是把所有请求压力直接压在一个日期键上。

### 9. 热点的常见缓解思路

#### 9.1 重新设计分区键

这是最根本的方法。

#### 9.2 写入分桶 / 加盐

例如把热点实体拆成：

- `MERCHANT#1001#0`
- `MERCHANT#1001#1`
- `MERCHANT#1001#2`

通过随机或规则分桶打散写入。

代价是：

- 读取时你可能要查多个桶再聚合

所以这是典型的“用读复杂度换写稳定性”。

#### 9.3 将热点访问拆成更适合的访问路径

例如某些极热点计数，本来就不适合直接让 DynamoDB 承压，可以考虑：

- 本地聚合
- Kinesis / Kafka 缓冲
- Redis 前置吸收热点

### 10. 为什么 DynamoDB 的难点常常不是“容量不够”，而是“分配不均”

这是一个很重要的理解。

很多系统看到限流、延迟升高，就会本能地以为：

“是不是表整体吞吐不够？”

但 DynamoDB 很多时候真正的问题是：

“整体还没满，但某个热点分区先爆了。”

所以你以后在看 DynamoDB 性能问题时，要养成一个习惯：

不要只看总吞吐，还要看：

- 访问模式是不是太集中
- 分区键是不是天然偏斜
- 是否存在明显热点实体

## 小结（3-5 条关键点）

- DynamoDB 的扩展能力建立在分区之上，分区分布是否健康比很多人想象中更重要。
- 热分区本质是请求和数据过于集中到少数分区，不是整张表容量一定不够。
- 高基数和均匀分布是 Partition Key 设计的两个核心标准。
- 日期、状态、地区、热点商户这类字段做 Partition Key 时要非常谨慎。
- 热点缓解通常意味着做重新建模、分桶或引入额外的分流层。
