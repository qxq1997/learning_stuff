# DynamoDB - 第 4 课：读写语义：一致性、条件写、乐观锁与幂等

## 学习目标（本节结束后你能做到什么）

- 理解 DynamoDB 的强一致读和最终一致读到底差在哪。
- 会解释条件写（Condition Expression）为什么是 DynamoDB 里特别重要的能力。
- 理解乐观锁和幂等设计在 DynamoDB 中的常见落法。
- 明白 DynamoDB 里的“事务”和“条件写”各自适合解决什么问题。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. DynamoDB 的读写不是“想当然就强一致”

很多工程师默认数据库查询都应该看到最新值，但在 DynamoDB 里，这件事不是默认的。

原因很简单：

- DynamoDB 底层是分布式存储
- 分布式系统里，一致性和延迟、吞吐、可用性之间永远存在权衡

所以 DynamoDB 给你的不是一个单一答案，而是两种读语义：

- 最终一致读（Eventually Consistent Read）
- 强一致读（Strongly Consistent Read）

### 2. 最终一致读怎么理解

最终一致读的意思不是“数据经常错”，而是：

**刚写完之后，短时间内去读，可能还读到旧值，但系统最终会收敛到新值。**

它的好处是：

- 延迟更低
- 成本更低
- 吞吐更友好

所以 DynamoDB 默认很多读取场景都更偏向最终一致。

### 3. 强一致读又意味着什么

强一致读就是：

**读请求返回时，要确保你看到的是最近一次已经成功写入的结果。**

这听起来更安心，但代价也更高。

所以强一致不是“更高级的默认选项”，而是一个更贵、更保守的选择。

工程上要想清楚：

- 这个业务是不是必须立刻看到最新值？

如果不是，就不要滥用强一致。

### 4. DynamoDB 为什么特别强调条件写

因为在高并发系统里，很多写操作并不是“直接覆盖就行”，而是：

- 只有某个状态满足时才能写
- 只有版本没变时才能写
- 只有这个 item 不存在时才能写

这类需求如果放在关系型数据库，很多人会想：

- 先查一遍
- 再更新

但在分布式高并发下，这种“先查后改”很容易有并发竞态。

所以 DynamoDB 提供了 Condition Expression，让你把条件和写操作一起提交。

这非常像是在说：

**不是我先看看能不能写，而是我提交这次写的时候，你帮我判断条件是否仍然成立。**

### 5. 一个最经典的例子：防重复创建

例如创建订单：

- 如果这个订单号已经存在，就不能再插

那你就不该先 Query 一下再 Put，而应该：

- 直接 Put
- 并带条件“这个主键不存在”

这类写法有两个好处：

1. 少一次往返
2. 避免并发下两个请求同时通过“前置判断”

### 6. 乐观锁在 DynamoDB 里怎么做

乐观锁的思路通常是：

- 每条记录有一个 version 字段
- 更新时带条件：只有 version 还是旧值时才允许更新
- 成功后把 version +1

这样就能避免并发覆盖。

这在 DynamoDB 里非常自然，因为它本来就鼓励条件写。

所以你可以这样理解：

**关系型数据库常用事务和行锁解决的部分并发控制，在 DynamoDB 里往往更多是靠条件写 + 乐观锁思路解决。**

### 7. 幂等为什么在 DynamoDB 里尤其重要

云上系统里，重试是常态。

请求超时了、网络抖了、客户端不知道到底成功没成功，很容易重试。

这时候如果你的写操作不是幂等的，就会出大问题：

- 重复扣款
- 重复下单
- 重复生成任务

所以 DynamoDB 很多设计天然会鼓励你：

- 使用业务唯一键
- 使用条件写
- 显式设计幂等 token 或幂等记录

也就是说，DynamoDB 不只是“把数据写进去”，它要求你把写入语义想清楚。

### 8. Update 不是简单 set 字段

DynamoDB 的 Update 经常会被低估。

它不只是“改几个属性”，而是很适合配合条件表达式做原子式状态推进。

例如：

- 只有订单状态是 `CREATED` 才能改成 `PAID`
- 只有库存大于 0 才能扣减

这种模式在 DynamoDB 里非常常见，因为它能避免很多“查完再改”的并发问题。

### 9. 那事务什么时候才有必要

如果一个操作只涉及单条 item，或者能靠条件写解决，通常优先用条件写。

因为：

- 更自然
- 成本更低
- 复杂度更低

只有在真正需要跨多条 item 的原子写时，才考虑事务。

所以 DynamoDB 的思路是：

**先用条件写解决绝大多数单 item 并发控制问题，事务留给更重的场景。**

### 10. 一个非常实用的判断框架

以后你面对 DynamoDB 写入逻辑时，可以先问：

1. 这是单条 item 还是多条 item？
2. 是否存在并发覆盖风险？
3. 是否可能因为重试导致重复写？
4. 能不能用条件写直接兜住？

如果能，优先条件写。

如果不能，再看事务。

## 小结（3-5 条关键点）

- DynamoDB 的读取语义分为最终一致和强一致，强一致不是默认答案，而是更贵的选择。
- 条件写是 DynamoDB 中非常核心的并发控制能力。
- 乐观锁在 DynamoDB 里通常通过 version 字段 + 条件表达式实现。
- 幂等设计在云上重试场景中非常重要，不能依赖“请求大概率只来一次”。
- 事务并不是日常第一选择，很多问题更适合先用条件写解决。
