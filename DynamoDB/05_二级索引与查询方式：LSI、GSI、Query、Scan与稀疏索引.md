# DynamoDB - 第 5 课：二级索引与查询方式：LSI、GSI、Query、Scan 与稀疏索引

## 学习目标（本节结束后你能做到什么）

- 彻底分清 `Query` 和 `Scan` 的差别，而不是把两者都当“查询”。
- 理解 LSI 和 GSI 各自在解决什么问题。
- 理解“索引不是白送的”，知道写放大和成本从哪里来。
- 能解释稀疏索引为什么在 DynamoDB 中非常实用。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. DynamoDB 里最关键的查询分水岭：`Query` vs `Scan`

在 MySQL 里，很多人看见“查询”两个字，脑子里默认都是一个概念。

但在 DynamoDB 里，`Query` 和 `Scan` 基本可以理解成两种不同人生：

#### Query

- 按主键或索引键查
- 有明确访问路径
- 高效、自然、可控

#### Scan

- 扫全表或大范围扫描
- 没有明确访问路径
- 经常意味着模型不自然、成本不稳定

所以 DynamoDB 设计时，一个特别重要的目标就是：

**尽量让核心查询都落到 Query，而不是 Scan。**

### 2. 为什么 Scan 这么不受欢迎

因为它违背了 DynamoDB 的设计哲学。

Scan 本质上是在说：

“我没提前设计好访问路径，现在想把很多数据扫出来再筛。”

这在关系型数据库里虽然也可能慢，但至少是自然能力的一部分；在 DynamoDB 里，这常常意味着：

- 读放大
- 成本高
- 延迟不稳定
- 数据量越大越难受

所以工程上看到 Scan，第一反应应该不是“能用”，而是：

**为什么我不得不用它？是不是模型该改？**

### 3. LSI 和 GSI 分别在解决什么问题

#### 3.1 LSI（Local Secondary Index）

LSI 的特点是：

- 和主表共享同一个 Partition Key
- 只是换一个 Sort Key 视角

所以它更像是：

**同一组数据，在同一分区内，换一种排序或查询方式。**

这意味着 LSI 适合的场景是：

- 同一个实体聚合内，需要多种排序视角

例如：

- 一个用户下面的订单，既想按创建时间排，也想按金额排

#### 3.2 GSI（Global Secondary Index）

GSI 更强，因为它可以重新定义另一套键。

也就是说：

- 主表按用户维度组织
- GSI 可以按商户维度、状态维度、类型维度再组织一份访问路径

所以 GSI 的本质是：

**为新的访问模式重新铺一条路。**

### 4. 为什么 GSI 很强，但不能乱加

因为 GSI 不是“免费索引”。

它背后意味着：

- 额外存储
- 额外写入同步
- 额外吞吐消耗

也就是说，主表写一条数据时，可能还要把相关索引一起更新。

这就是 DynamoDB 里的典型代价：

**你多一条访问路径，就要多付一份成本。**

所以 GSI 不是“先不用，后面慢慢补”的轻松操作，而是建模的一部分。

### 5. 稀疏索引为什么特别有用

稀疏索引（Sparse Index）不是一个单独的 AWS 功能名词，而是一种非常实用的设计思路：

**不是每条数据都进索引，只让符合某些条件的 item 进入 GSI。**

例如：

- 只有 `status = PENDING` 的订单才写入某个 GSI

这样做的好处很大：

- 索引更小
- 查询更准
- 成本更低

这比你把所有订单都丢进 GSI，再查询时慢慢过滤，要自然得多。

所以 DynamoDB 里一个很实用的建模技巧是：

**别急着想“怎么查更多”，先想“能不能只把需要查的那部分数据预先投影进索引”。**

### 6. Projection 又在干什么

索引不一定要复制整条 item。

Projection 决定的是：

- 这个索引里到底带哪些字段

如果你只需要几个字段，就不要把整条 item 全投进去。

因为投得越多：

- 索引越大
- 成本越高
- 写同步越重

这和关系型数据库里覆盖索引的思路有点像，但 DynamoDB 的成本感更直接。

### 7. 一个很典型的设计场景

假设订单主表主键是：

- `PK = USER#用户ID`
- `SK = ORDER#创建时间#订单ID`

这很好地支持了：

- 查询某用户最近订单

但现在你还需要：

- 查询某个商户所有待发货订单

这就不是原主键能自然支持的访问模式了。

这时你就可以考虑一个 GSI：

- `GSI1PK = MERCHANT#商户ID`
- `GSI1SK = STATUS#PENDING#创建时间`

这样这个查询就从“不得不 Scan”变成了自然的 Query。

### 8. 一条非常重要的设计原则

在 DynamoDB 里，查询设计的核心不是：

- “我能不能把所有情况都查出来”

而是：

- “我愿意为哪些核心查询预先修路”

因为每多一条索引，本质上都是：

- 更多成本
- 更多维护复杂度
- 更多写放大

### 9. 如何判断一个查询该不该单独做索引

可以看四个问题：

1. 这个查询是不是高频核心路径？
2. 它是否能用现有主键或索引自然完成？
3. 如果不用索引，只能 Scan 吗？
4. 这条索引带来的成本，值不值？

如果只是偶发后台查询，很多时候未必要专门修一条 GSI。

## 小结（3-5 条关键点）

- 在 DynamoDB 里，`Query` 和 `Scan` 的差别非常大，建模目标通常是尽量让核心查询走 Query。
- LSI 是“同一个分区内换排序视角”，GSI 是“为新访问模式重新铺路”。
- 索引不是免费的，GSI 会带来额外写放大、存储和吞吐成本。
- 稀疏索引是非常重要的设计技巧，可以只让需要被查询的那部分数据进入索引。
- Projection 决定索引中带哪些字段，字段投影越多，索引越重。
