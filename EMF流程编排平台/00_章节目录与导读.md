# EMF 流程编排平台 · 学习与设计手册

## 本手册的目标

本手册不是一篇介绍文,而是尝试**完整复原 EMF(内部自研流程编排工具)的设计细节**,并在此基础上系统地给出**AI 赋能的切入方案**。

写作原则:
- 每一章都要讲到**为什么这样做**、**不这样做会发生什么**、**关键代码路径 / 数据结构长什么样**。
- 避免"它是一个 XXX 工具"式的同义反复,而要给出**可对照代码落地的抽象**。
- 凡是原始输入中没有指定的细节,会**在文中显式标注为"推断"**,以便后续与真实代码校准。

## EMF 的一句话定义

> EMF 是一个**事件驱动、JSON 声明式、以"命令池"为执行原语**的 Python 流程编排 **framework**(不是中心化服务),基于 GCP(Pub/Sub / GCS / BigQuery / GKE / Secret Manager / IAM),由各业务团队自行 clone 源码、在自己的 GCP 环境部署。观测栈走 **OpenTelemetry + Grafana**(刻意避开 Cloud Monitoring)。
>
> 架构上有一处关键**非对称**:**数据面(本地命令 / DBT)是去中心化的,每 Team 一套;外部 API 命令则通过中心化的 Sidecar Service(FastAPI on GKE)统一出口**,用于共享 SaaS 凭证与做全公司级配额治理。

## 关于术语

| 术语 | 含义 | 近邻概念 |
|------|------|---------|
| Pipeline | 一份 JSON DAG 文件所定义的一次完整流水线 | Airflow DAG |
| Step / Task | Pipeline 中的一个节点 | Airflow Task |
| Command | 被 Step 调用的具体执行单元(已注册到 EMF 内部) | Airflow Operator |
| Context | 一次 Pipeline 执行期间共享的运行时上下文 | Airflow `ti.xcom` + DAG Run Conf |
| Sidecar | 拆出来的外部 API 调用微服务(FastAPI on GKE) | 独立的 Operator 进程 |
| Run | 一次 Pipeline 的具体执行实例 | Airflow DagRun |
