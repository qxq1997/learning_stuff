# EMF 工作流引擎设计与理论

## 这套笔记的定位

这份目录是 EMF 的学习和面试参考笔记,目标不是按源码逐行索引,而是把 EMF 还原成一套能讲清楚设计取舍的工作流引擎模型。

从设计上看,EMF 可以理解为:

```text
消息驱动 + DAG 调度 + 命令插件化
```

再系统一点说:

```text
EMF 是一个消息驱动的动态 DAG 工作流引擎。
```

它的控制平面负责消息、DAG、状态、调度、重试、超时和 batch/run 生命周期;执行平面负责 Databricks、Storage、Catalogue、API、文件处理等具体业务动作。

更完整的一句话:

```text
OE-RUN 或外部事件产生 Message;
Message 进入内存 DAG;
ParallelMessageConsumer 只执行依赖满足的节点;
StratusCommandExecutor 根据 command 找 module;
module 调用 command/service 做业务;
业务命令可以继续 enqueue 新消息,从而扩展 DAG.
```

这条链路是理解整个项目的主线。

## 术语统一

| 术语 | 在 EMF 里的含义 |
|---|---|
| `Message` | 最核心的任务单元,每一条可执行任务都被表示成一条消息 |
| `run_uuid` | 一次 workflow run 的 DAG 分组 key |
| `batch_id` | 一批 run 的分组 key,用于批次级治理和调度 |
| `order_id` | 业务层面的任务节点 ID,也是父子依赖匹配的 key |
| `parents` | 当前节点依赖的父任务 `order_id` 列表 |
| `command` | 要执行的命令名,由命令系统映射到具体 module |
| `DagManager` | 管理多个 run 的 DAG 上下文 |
| `DagContext` | 管理单个 `run_uuid` 对应的一张 DAG |
| `ParallelMessageConsumer` | 从 DAG 中取 ready 节点并并发执行 |
| `StratusCommandExecutor` | 根据 `message.command` 找 module 并执行 |
| `OE-RUN` | 把 workflow 定义展开成一批 `Message` 的关键命令 |
| `completion token` | 子 workflow 或动态 fan-out 回到父流程时的逻辑汇合点 |

## 这套笔记怎么用

读的时候不要急着背类名或文件位置,而是先把四个问题想清楚:

1. EMF 为什么用 `Message` 表达任务。
2. DAG 为什么能从 `order_id + parents` 推导出来。
3. 调度层和命令层为什么要拆开。
4. retry、子 workflow、动态 fan-out 为什么都能统一成“继续生成 Message”。

面试里最有价值的不是说出某个类在哪个文件,而是能解释这些设计为什么成立、解决了什么问题、带来了什么代价。
