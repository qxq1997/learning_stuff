# EMF 工作流引擎设计与理论

## 这套笔记的定位

这套笔记用于沉淀 EMF 的架构理解、生产问题、面试表达和系统设计推演。它不是源码逐行索引，而是把 EMF 还原成一套可以讲清楚设计取舍的数据编排平台。

**第 34 章《EMF 项目总结与技术选型说明》是整套文档的 source of truth。**

如果其他章节和第 34 章存在口径冲突，以第 34 章为准。其他章节只负责把第 34 章中的某个主题继续展开，例如动态 DAG、状态机、幂等、僵尸任务、Scheduler/Worker 恢复、面试 Q&A 等。

从第 34 章的基准口径看，EMF 可以概括为：

```text
消息驱动 + 元数据驱动 + DAG 编排 + Databricks 计算 + Lakehouse/Delta 数据治理
```

核心链路是：

```text
file_type
  -> LOAD_INFO
  -> OE-RUN
  -> PROCESS_TASKS
  -> command messages
  -> Databricks
  -> DATAHUB
  -> CATALOGUE.METADATA
  -> catalogue resolution
```

理解这条链路，比单独背某个类名或某个表名更重要。

## 术语统一

| 术语 | 在 EMF 里的含义 |
|---|---|
| `Message` | 最核心的任务单元，每一条可执行任务都被表示成一条消息 |
| `file_type` | EMF 的核心路由键，决定入口分支、加载配置和 catalogue resolution 第一层过滤 |
| `LOAD_INFO` | `file_type` 到加载规则、schema、目标 dataset、ingestion workflow 的配置表 |
| `OE-RUN` | 把 workflow 定义展开成一批 command messages 的关键命令 |
| `PROCESS_TASKS` | workflow 的 DAG 定义表，描述 order、command、parents、parameters |
| `DATAHUB` | 业务数据落地区，不是单张表 |
| `CATALOGUE` | 数据资产目录域，不是单张表 |
| `METADATA` | 数据实体的 key-value 描述表，用于 resolution、血缘和审计 |
| `run_uuid` | 一次 workflow run 的 DAG 分组 key |
| `batch_id` | 一批 run 的分组 key，用于批次级治理和调度 |
| `order_id` | 业务层面的任务节点 ID，也是父子依赖匹配的 key |
| `parents` | 当前节点依赖的父任务 `order_id` 列表 |
| `command` | 要执行的命令名，由命令系统映射到具体 module |
| `completion token` | 子 workflow 或动态 fan-out 回到父流程时的逻辑汇合点 |
| `lease` | 任务执行权的临时租约，用于识别 worker 死亡和防止旧 owner 覆盖新状态 |

## 当前实现与生产化演进

读后续章节时要注意一个边界：有些章节讲的是当前 EMF 的实现主线，有些章节讲的是生产化演进设计。

- 当前主线：Worker 消费消息，读取 metadata/config，展开 workflow，执行 command，调用 Databricks，并写回状态和 catalogue。
- 生产化演进：把 Scheduler、Worker pool、lease、zombie recovery、resource quota、backpressure、Jira/LLM 自动诊断等能力进一步平台化。

这两者不矛盾，但不能混成一句话。面试或评审时，可以先讲第 34 章的当前主线，再说明哪些是为了生产可靠性补上的增强能力。

## 这套笔记怎么用

读的时候先抓住四个问题：

1. EMF 为什么把业务任务表达成 `Message`。
2. `file_type`、`LOAD_INFO`、`PROCESS_TASKS` 和 `METADATA` 如何共同驱动业务行为。
3. Worker 为什么只做控制面，Databricks 为什么承担计算面。
4. retry、子 workflow、动态 fan-out、僵尸任务恢复为什么都必须回到状态、幂等和可观测性设计。

面试里最有价值的不是说出某个类在哪个文件，而是能解释这些设计为什么成立、解决了什么问题、带来了什么代价。
