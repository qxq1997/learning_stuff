# 01 一句话架构与核心链路

## 一句话架构

EMF 把每个任务表示成一个 `Message`,按 `run_uuid` 组织成一张 DAG。调度器不断找出依赖已经满足的节点,并发执行对应 `command`。命令执行过程中还可以继续生成新的消息,于是工作流可以在运行时动态扩展。

这不是单纯的消息队列消费器,也不是静态 DAG 调度器。更准确地说,它是一个:

```text
消息驱动的 DAG 调度引擎 + 命令插件化执行系统
```

如果再从系统设计角度压缩一句,可以说:

```text
EMF 是一个动态 DAG 编译器 + 调度器。
```

它把 workflow 定义、外部事件、重试消息、子 workflow 和运行时 fan-out 都统一编译成 `Message`,然后让 DAG 调度器持续推进这些消息。

## EMF 的本质

EMF 不是“按顺序跑一段脚本”,也不是“从队列里拿一条消息就执行一条消息”。

更准确的模型是:

```text
Workflow 不是一段顺序代码,
而是一批带依赖关系的 Message。
```

每条 `Message` 都像一个异步任务节点:

- `command` 说明这个节点要做什么。
- `parents` 说明这个节点依赖谁。
- `run_uuid` 说明这个节点属于哪一张运行时 DAG。
- `batch_id` 说明这张 DAG 属于哪个批次生命周期。

所以 EMF 的核心闭环不是“消费消息后结束”,而是:

```text
Message 进入 DAG
  -> 调度器找 ready 节点
  -> command 执行业务
  -> command 可能继续生成 Message
  -> 新 Message 再进入 DAG
```

这也是 EMF 和普通任务队列最大的区别:它保存并推进任务之间的依赖关系。

## 控制平面和执行平面

学习 EMF 时可以把它拆成两层:

| 层 | 关心什么 | 不关心什么 |
|---|---|---|
| 控制平面 | 消息、DAG、状态、调度、重试、超时、batch/run 生命周期 | SQL 具体怎么跑、文件具体怎么传 |
| 执行平面 | Databricks SQL、Storage、Catalogue、API call、文件上传、数据处理 | DAG 依赖如何判断、节点如何调度 |

这条边界非常重要。

调度器只需要知道:

```text
哪个 Message ready?
哪个 Message 要执行哪个 command?
执行后状态怎么更新?
```

具体 command 只需要知道:

```text
我要读哪些参数?
我要调用哪个服务?
我要不要生成后续 Message?
```

因此可以说:

```text
调度器不关心 SQL 怎么跑;
SQL command 不关心 DAG 怎么调度。
```

这种分层让 EMF 可以同时接 Databricks、Storage、Catalogue、API 和更多业务能力,而不需要把每种业务能力写进调度器。

## 核心链路

从外部消息进入系统,到具体业务命令执行,主链路可以记成:

```text
Service Bus / PubSub
  -> PubSubMessageProducer
  -> WorkerMessageAppender
  -> AppExecutionContextManager + DagManager
  -> ParallelMessageConsumer
  -> MessageHandler
  -> StratusCommandExecutor
  -> ParameterisedModuleBase
  -> 具体 Command / Service
  -> Azure Databricks / Storage / Catalogue / 新消息
```

这条链路里每一层只做一件事:

| 层 | 职责 |
|---|---|
| `PubSubMessageProducer` | 从 Service Bus / PubSub 拉取外部消息 |
| `WorkerMessageAppender` | 把外部消息转换为内部 DAG 节点并入队 |
| `DagManager` | 按 `run_uuid` 管理多张 DAG |
| `ParallelMessageConsumer` | 找 ready 节点,按并发和优先级调度 |
| `MessageHandler` | 为单条消息建立执行 scope 并交给 executor |
| `StratusCommandExecutor` | 根据 `command` 找 module,解析参数,调用执行 |
| `ParameterisedModuleBase` | 约束所有命令模块必须实现的接口 |
| 具体 command/service | 做真实业务,也可能继续 enqueue 新消息 |

## 为什么说它是 DAG,不是普通队列

普通队列只关心消息先后和消费确认。EMF 还关心消息之间的父子依赖:

```text
parent order_id COMPLETE/RETRY
          |
          v
child message becomes ready
```

也就是说,一条消息能不能被执行,不是只看它有没有被拉到本地,而是看:

1. 它属于哪一次 `run_uuid`。
2. 它依赖哪些父节点 `order_id`。
3. 父节点是否已经进入允许下游执行的状态。
4. 它自己的 `run_date` 是否已经到达。
5. 它在 batch 里的优先级如何。

## 为什么说它是动态 workflow

很多工作流系统会先把整张 DAG 固定下来,然后只执行这张静态图。EMF 不完全是这样。

`OE-RUN` 会把 workflow 定义展开成一批 `Message`。同时,某些业务命令在运行过程中也能继续产生新消息,例如 `ADB-MESSAGES-FROM-QUERY` 可以执行 query,把多条 SQL job 结果转换成新的 `ADB-SQL-EVAL` 消息再入队。

所以 EMF 的 DAG 更像运行时增长的图:

```text
初始 Message
  -> 展开 workflow messages
  -> 执行某个 command
  -> command 继续生成 messages
  -> DAG 继续扩展
```

更完整的抽象运行模型可以写成:

```text
OE-RUN
  -> 读取 workflow task / process_tasks
  -> 生成 messages
  -> messages 形成 DAG

DAG
  -> 找 ready 节点
  -> 并发执行 command
  -> 更新状态

Command
  -> 执行业务
  -> 可选:生成新 messages
  -> 新 messages 重新加入 DAG
```

## 最小心智模型

学习 EMF 时不要先把它想成 Azure、Databricks、Storage 或 Catalogue 的集合。那些都是命令背后的业务能力。

先把它想成这五个对象:

```text
Message -> DagContext -> ParallelMessageConsumer -> MessageHandler -> Command Module
```

只要这五个对象的关系通了,后面的 Azure 集成、参数替换、retry、timeout、batch cleanup 都是在这条主线上继续加治理能力。
