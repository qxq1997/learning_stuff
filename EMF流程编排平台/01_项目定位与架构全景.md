# 01 · 项目定位与架构全景

## 1.0 本章导读

本章要回答三件事:
1. **EMF 是在什么背景下长出来的**,以及为什么没有直接用 Airflow / Prefect / Dagster。
2. **EMF 的宏观架构长什么样**——每一层承担什么职责,为什么要切这一刀。
3. **一次 Pipeline 从"用户上传 JSON 文件"到"执行完毕"之间,控制流和数据流怎么走**。

后续 02~09 章都是对本章某一块的放大。10~12 章讨论 AI 介入时,落点也都锚定在本章给出的组件图上。

## 1.1 为什么自研:Airflow 不是不能用,而是"不对位"

Airflow 是"通用工作流 + 重调度器"的设计。EMF 面对的场景其实有几个很硬的约束,恰好把 Airflow 推出舒适区:

### 1.1.1 用户画像不是工程师

EMF 的编写方不全是后端工程师,很大一部分是**数据分析师、业务领域同学**。他们:
- **不一定会写 Python**,让他们维护 `dag.py` 并走代码审批 / CI/CD 流程,心智负担太重。
- 但能读写 JSON / YAML,能理解"一串步骤 + 依赖关系"。
- 需要的改动粒度通常是"改一个表名、加一个过滤条件、加一个导出目的地",**不需要重新 deploy 整个 DAG 包**。

这决定了 DAG 定义必须是**声明式、结构化、无代码执行能力**的——一个纯 JSON 文件,由平台解释执行,而不是一个 Python 模块,由调度器 import。

### 1.1.2 触发模型天然是事件驱动,不是 cron

大量数据场景是"**文件到了就开始处理**":上游系统把数据落到 GCS 某个前缀,EMF 就该开始加工。用 Airflow 做法通常是:
- 写一个 `Sensor` 轮询 GCS(浪费调度资源,时延差),或
- 用 `Eventarc → Cloud Function → Airflow REST API trigger DAG`(链路长,两套系统)。

EMF 把事件驱动作为**一等公民**——GCS 事件不仅触发执行,还直接**作为 DAG 的输入参数**(事件里的对象路径就是 pipeline 的输入)。

### 1.1.3 基础设施心智统一成 "GCP 原生件 + OTel 可观测栈"

Airflow 的生产部署不是"装个包"——你要决定 Executor(Celery / K8s / Local),Meta DB(Postgres),Web Server,Scheduler HA,DAG 文件分发机制(Git sync / shared volume / S3)……这些每一样都需要专门运维。

EMF 全面吃 GCP 原生件,再加一层**厂商无关的可观测性**:

| 能力面 | 选型 | 备注 |
|-------|------|------|
| 事件总线 | **Pub/Sub** | GCS 原生 Pub/Sub Notification,**不走 Eventarc** |
| 对象存储 | **GCS** | 既是触发源,也是 DAG 文件 / 产物落地 |
| 数仓 | **BigQuery** | 数据类命令最重要的 sink / source |
| 运行时 | **GKE** | Control Plane 与 Sidecar 都是 Deployment |
| 密钥 | **Secret Manager** | 外部 API 凭证、数据库密码 |
| 身份 / 权限 | **Service Account + IAM** | **决定性地塑造了 EMF 的部署模型**,见 §1.1.7 |
| 日志 | **Cloud Logging** | GKE 自动落地 |
| 指标 / 追踪 | **OpenTelemetry + Grafana** | 刻意**避开** Cloud Monitoring(原因见下) |

> **为什么身份走 GCP IAM,但可观测性却不走 Cloud Monitoring?**
> 这不是反复无常,而是对应两种不同的一致性需求:
> - IAM 是**全公司共享的基础设施**,EMF 借它现成的权限体系,省掉自建 RBAC。
> - 可观测性是**团队级视角**,需要跨环境(dev / staging / prod)甚至跨 GCP project 拼在同一张 Grafana 面板上看。Cloud Monitoring 的 project 边界会让这件事非常麻烦。
> 所以身份"在 GCP 里更方便"就吃 IAM,观测"脱离 GCP 更方便"就用 OTel。**选型的锚点是"一致性的边界在哪里",不是"是不是 GCP 原生"**。

运维面变窄后,团队可以把精力集中在**命令库**上——这才是 EMF 的真正业务资产。

### 1.1.4 命令是共享资产,不是一次性代码

Airflow 的 Operator 虽然也能复用,但在实际团队里,大量逻辑最后都写在 `PythonOperator` 的 callable 里,**知识散落在各个 dag.py 中**,搜索和治理都很难。

EMF 的设计把这一点反过来:**只有进入"命令池"才是合法的执行单元**。
- 一个新能力(例如"从某 SaaS 拉活动数据")先被实现成一条命令,走代码审查 / 单测 / 版本化。
- 然后所有 Pipeline 只能通过引用命令名字 + 传参来使用这个能力。
- 这样**业务上的 DAG 激增不会污染代码库**,而且每条命令都有明确 owner。

这本质上是**把 "Operator 的标准化"当成了组织边界**:研发拥有命令,业务拥有 DAG。

### 1.1.5 Airflow 的计划器模型与"按事件驱动 + 数据量瞬时波峰"不匹配

Airflow Scheduler 周期性扫描 DAG,决定调度。EMF 是"**事件一来就派发**",更接近 FaaS。真用 Airflow 去模拟,Scheduler 扫描频率和触发延迟之间会永远拉锯。

### 1.1.6 小结:EMF 是什么混合体

EMF ≈ **(Airflow 的 DAG 语义) × (FaaS 的事件触发) × (标准化命令池的治理) × (去中心化的 framework 分发)**。它不是 Airflow 的简化版,而是把**组织边界**画进了技术栈——这是自研的真正价值,不是"不用 Airflow"而是"不能用 Airflow 表达这个组织边界"。

### 1.1.7 去中心化部署:EMF 是 framework,不是 service

这是理解 EMF 的**最关键一条**,贯穿后续所有章节。

**EMF 不是中心化服务,而是一份源码,每个用户 / 团队自己下载、自己部署。** 换言之:
- 平台团队交付的是 **framework + 命令库源码**,不是一个对外的 SaaS endpoint。
- 每个使用方(一个业务团队、一个分析师小组)在**自己的 GCP project / 自己的 GKE namespace**里跑一套独立的 Control Plane。
- 每个部署有**自己的 Service Account**、自己的 Pub/Sub topic、自己的 GCS bucket、自己的 Secret Manager 条目。
- **权限不是 EMF 自己做的 RBAC,而是 GCP IAM**——谁的 ServiceAccount 能读哪张 BigQuery 表,谁就能在 DAG 里调用读那张表的命令;EMF 不做第二层权限校验。

#### 为什么要去中心化

1. **权限自然对齐到团队**。集中式服务必须代理调用,就需要"EMF 能读所有 team 的数据"这种大权限,审计和最小权限原则都很难做。去中心化之后,**EMF 能做什么 ≡ 当前部署的 ServiceAccount 能做什么**——IAM 天然是权限边界。
2. **故障不跨团队**。A 团队的 DAG 写炸了、把 BigQuery 配额打满了,不会拖累 B 团队的流水线。
3. **升级节奏由团队自己掌握**。平台团队 release 一个新版本,A 团队今天就合并,B 团队下个月再合并,互不影响。对业务侧友好。
4. **运维心智轻**。没有"EMF 线上服务",就没有"EMF on-call"。平台团队变成 **framework maintainer**,不是 SRE。
5. **"上传 DAG"就是"往自己的 bucket 放文件"**,天然避开了"谁能往中心服务推 DAG、怎么鉴权"这类麻烦问题。

#### 换来的代价(后面章节需要反复对付)

1. **命令库的 source drift**。每个团队跑的是不同 commit 的 EMF。平台团队必须在命令兼容性上非常克制——一条命令的行为要几乎永不变,要变就发新命令。这件事 §05 会展开。
2. **没有全公司统一视角**。平台团队没法从一个面板看"全司今天跑了多少 DAG、失败率是多少"。想要这个视角,就要**主动**让每个部署往共享的 OTel Collector / Grafana 推指标(§08)。
3. **公共能力下沉困难**。比如"集中式审计日志"、"跨团队 DAG 依赖图",这种功能在去中心化模型下要么做不了,要么需要额外约定。
4. **文档和默认配置要极其体面**。Framework 的失败不是"线上 P0",是"一堆团队都卡住装不起来",扩散面更大。

#### 非对称:Control Plane 去中心化,但 Sidecar 是中心化的

这是 EMF 架构里最值得品的一处**刻意的不对称**。

- **Control Plane(编排 + 本地命令 + DBT)** 留在每个团队自己的部署里——原因上面已经讲过,IAM 天然是权限边界。
- **Sidecar(外部 API 调用微服务)** 反而是**一套中心服务,跨团队共用**。

为什么这么切?因为这两类命令面对的**"一致性边界"完全不同**:

| 维度 | 本地/数据命令(去中心化) | 外部 API 命令(中心化 Sidecar) |
|------|-----|-----|
| 权限模型 | GCP IAM,每团队 SA 各管各 | 第三方 SaaS 的 API key / OAuth token,IAM **管不着** |
| 凭证管理 | Secret Manager 条目天然按团队隔离 | 往往是"整个公司的 Salesforce 账号"、"公司级 HubSpot key"——本身就是公司共享资源 |
| 速率限制 | GCP 配额按 project 计 | 第三方 vendor 按**整个公司**计,必须集中做令牌桶,否则互相挤兑 |
| SDK 升级 / API 版本 | 不涉及 | 需要专人跟进 vendor 的 deprecation,适合平台团队集中维护 |
| 故障域 | 一家团队的数据工作负载 | 一家第三方 vendor 宕机会影响所有团队——但这跟是否中心化无关,vendor 挂就是挂 |

所以这个不对称不是设计漏洞,而是**对"一致性边界在哪里"的精细判断**:IAM 能解决的就交给 IAM,IAM 解决不了的(第三方 SaaS 凭证 / 配额)就集中做。

#### Sidecar 中心化带来的新挑战(§06 展开)

1. **调用身份**:Control Plane A 调中心 Sidecar,Sidecar 怎么知道是 A 在调,并据此计入 A 的配额、用 A 的凭证(如果做了团队级隔离的话)?→ 需要 GCP ServiceAccount Token 穿透(可能是 ID Token / OIDC)。
2. **凭证隔离的细粒度**:是"公司一套凭证、所有团队共用"还是"同一个 API 每个团队各自配凭证"?选择不同,Sidecar 的复杂度差一个数量级。
3. **Sidecar 是唯一的全局共识点**:它成了"平台团队拥有的线上服务"——和整体"framework 不 service"的定位形成对比,所以它的 SLO、on-call、版本升级都要专门做好。
4. **网络路径**:每团队 Control Plane 要能访问中心 Sidecar——意味着 Sidecar 要么挂在公司 VPC,要么走 Internal Load Balancer 配合 Service Connect,不能直接公网暴露。

#### 一句话总结

> **EMF 的技术栈不是"为了跑起来",而是"为了跑在每个团队自己的边界里"。** 这一条是后续所有架构决策的前提——每次你看到某个设计让你觉得"这不就是做重复轮子吗",多半是因为它在迁就这个去中心化前提。

## 1.2 核心设计哲学(四条)

以下四条是理解 EMF 每一处决策的钥匙。后续章节里遇到"为什么这里要这么设计",基本都能归到这四条上。

### 1.2.1 声明式 over 命令式
DAG 文件只描述**意图**(做什么、谁先谁后、参数是什么),不描述**过程**(怎么循环、怎么分支怎么 retry)。循环、分支、重试、超时这些**过程性**的东西都下沉到平台。

> 推论:JSON DAG 中**不允许出现 Python 表达式**;需要"动态值"时,统一走"参数引用"语法(§02 会展开)。

### 1.2.2 事件是一等输入,不是触发方式
GCS Object Finalize 事件既是**启动信号**,也是**初始参数**。这意味着 DAG 文件里必须能够引用"触发事件"的字段(比如 `${event.bucket}`、`${event.name}`)。

### 1.2.3 命令是协议,不是函数
对外(DAG 作者),命令是一个**有 schema 的盒子**:声明输入、输出、副作用、幂等性。
对内(平台),命令实现可以是:
- 一个本地 Python 函数(大多数);
- 一个对 Sidecar 的 HTTP 调用(外部 API 类命令);
- 一个对 DBT 的子进程调用(SQL 类命令);
- 未来可以是一个对 LLM / Agent 的调用(§12 展开)。

这保证了**执行层可以替换、增长,而 DAG 作者零感知**。

### 1.2.4 可被业务安全地写坏
DAG 是用户上传的,平台必须默认"不信任"这份输入:
- Schema 校验 → 拒绝格式不对的 DAG。
- 命令白名单 + 参数 schema → 拒绝 DAG 调用不存在的命令或乱传参。
- 资源上限 → 单 Step / 单 Run 最长运行时间、最大并发、最大输出大小。
- 沙箱隔离 → 命令运行在 K8s Pod 中,带有明确的 IAM ServiceAccount。

这些都**不应该让 DAG 作者来写**,而是平台强制。

### 1.3.1 每一层的职责边界

| 层 | 核心职责 | 不负责 |
|----|----------|--------|
| 触发层 | 把 GCS 事件转成内部消息 | 不做解析、不做业务判断 |
| Event Ingestor | 去重 / 基本过滤 / 拉起一次"待装载"记录 | 不解析 DAG 内容 |
| DAG Loader & Validator | 下载 JSON、做 schema 校验、做**命令白名单校验**、做**参数引用合法性校验** | 不做拓扑 / 执行 |
| DAG Compiler | JSON → 内部图结构(节点 / 边 / 参数解析器) | 不做调度 |
| Scheduler / Executor | 拓扑排序、依赖满足时派发、状态机流转、重试 | 不实现命令本身 |
| State Store | 存 Run / Step 状态、XCom 风格的中间结果、DAG 缓存 | 不做业务数据存储 |
| Command Runner | 根据命令 kind 分派执行(本地 / DBT / Sidecar) | 不决定派发顺序 |
| Sidecar | 封装外部 API 的"怎么调、怎么重试、怎么鉴权、怎么限流" | 不关心 DAG / 命令抽象 |

### 1.3.2 为什么要把 Event Ingestor 与 Loader 分开

因为两件事的**失败域**完全不同:
- Ingestor 的失败是"消息没收到"——靠 Pub/Sub 重投递兜底。
- Loader 的失败是"JSON 解析 / schema 校验挂了"——需要**持久化记录这个错误并通知用户**,不能丢。

混在一起容易出现"Pub/Sub 收到了消息但 DAG 解析出错,然后消息被 ack 掉,用户永远得不到反馈"。

### 1.3.3 为什么 Sidecar 要独立部署,而不是合进 Control Plane

这是本项目最值得深挖的一个决策,专题放在 §06。这里先给四条原因,**前三条解释为什么独立,第四条解释为什么"独立 + 中心化"**:

1. **依赖重量**:外部 API 客户端库往往带一堆驱动(OAuth、各 SaaS SDK),放在 Control Plane 会让控制面镜像膨胀、冷启动慢。
2. **故障隔离**:外部 API 抖动 / 速率限制 / 5xx 可能拖慢或打挂 Pod;独立部署后,挂的只是 Sidecar。
3. **独立伸缩**:批量拉数据时 Sidecar 需要横向扩很多副本,而 Control Plane 不需要。
4. **第三方凭证与配额天然全公司级**:前面 §1.1.7 讲过,第三方 SaaS 的 API key / rate limit 不是按 GCP project 计的,是按公司计的——所以不管多少个 Team 部署 EMF,这一块必须**汇聚成中心服务**去做配额治理和凭证托管,否则就会出现"Team A 把公司的 Salesforce 配额吃完,Team B 被限流"。

> 换句话说:前三条告诉你"为什么要把它从 Control Plane 里拆出来",第四条告诉你"为什么拆出来之后还得是中心的一套,不能每团队各一套"。

### 1.3.4 两层并发模型:DAG 内 × DAG 间

EMF 的并发是**两层叠加**的,这是理解吞吐能力和调优点的关键。

#### 第一层:DAG 内并发(intra-DAG)

同一个 Run 里,拓扑上互不依赖的 Step **同时派发**到线程池执行:

```
            ┌──▶ [B] ──┐
  [A] ─────┤           ├──▶ [D]
            └──▶ [C] ──┘
```

A 跑完后,B 与 C 在两个线程里**并行**推进,Scheduler 不会等其中一个完成再派发另一个。

#### 第二层:DAG 间并发(inter-DAG)

Control Plane 同时服务多个 Run。每来一个 GCS Pub/Sub 事件,就拉起一个新的 Run;多个 Run **在同一个进程里并行推进**,彼此不阻塞。

两层叠在一起,理论吞吐 ≈ `(并行 Run 数) × (每 Run 的并行 Step 数)`。实际封顶点是:线程池容量、State Store 连接数、下游数据面(BigQuery slot / Sidecar QPS)的瓶颈。

#### 为什么选线程,不选 asyncio、不选多进程

| 维度 | 线程(EMF 的选型) | asyncio | 多进程 |
|------|------------------|---------|--------|
| IO-bound 命令 | ✅ IO 时释放 GIL,可真并行 | ✅ 原生非阻塞 | ✅ 但开销大 |
| CPU-bound 命令 | ⚠️ GIL 会成为瓶颈 | ❌ event loop 被堵死 | ✅ 真并行 |
| 与同步 SDK 的兼容 | ✅ `google-cloud-*`、`dbt-core` 都是同步 | ❌ 要到处 `to_thread` | ✅ |
| 进程内共享状态 | ✅ Scheduler 和 worker 天然共享 Context | ✅ | ❌ 要 IPC / 反序列化 |
| 心智负担 | 中(锁、线程安全) | 高(`async` 会全栈传染) | 中(启动开销、序列化) |

数据工作负载绝大多数是 IO-bound——**BigQuery 查询、GCS 读写、HTTP 到 Sidecar、dbt 子进程**——IO 阻塞期间 GIL 被释放,线程模型可以做到真并行。再加上 Google 的 Python SDK 基本都是同步接口,用 asyncio 就必须 everywhere-async,污染所有命令代码;多进程则因为要跨进程访问 State Store / 共享 Run 上下文,复杂度直接爆炸。

**但代价是:纯 Python 的 CPU-bound 命令会 GIL-block 整个线程池**——比如在本地命令里用 Python for-loop 遍历几 GB 数据。这类写法应该在 §05 命令开发规约里**硬性禁止**:要么改用 numpy/pandas(C 层释放 GIL),要么走 DBT 让 BigQuery 在服务端算,要么下放到 Sidecar 里用独立进程跑。

#### 线程池的拓扑:一池 vs 多池(推断,§04 会确认)

实现上有两种常见选择:

1. **全局共享池**:整个 Control Plane 一个 ThreadPoolExecutor,所有 Run 的所有 Step 都从这里取线程。
   - 优:资源利用率最高,忙 Run 能吃到空闲 Run 让出来的容量。
   - 劣:一个 Run 的慢 Step 可能饿死别的 Run;没有 Run 级公平性。
2. **每 Run 一个小池**:每个 Run 自带一个小线程池(比如 4 个 worker)。
   - 优:天然隔离,一个 Run 炸了不影响别的。
   - 劣:线程数随 Run 数膨胀,整体 CPU/内存利用率低。

折衷做法是**分层池**:一个全局线程池 + 每 Run 一个并发信号量(semaphore),既共享容量又有上限保护。这是在团队级部署(单 Pod 资源紧)下最实用的形态。

#### 两层并发强加给架构的 5 条义务

这一节非常重要,几乎每一条在 §04 / §08 都会再次出现:

1. **State Store 必须并发安全**。多线程同时更新 Run / Step 状态,需要乐观并发(版本号 / CAS)或数据库事务。Firestore 有事务 API,Postgres 的 `SELECT ... FOR UPDATE` 也行。选型时这是硬指标。
2. **Scheduler 主循环与 worker 线程必须解耦**。Scheduler 只做"哪些 Step 可以跑",把任务扔进线程池 → future 完成时回调写状态;主循环本身不能被任何 Step 的异常、超时、死锁拖住。
3. **观测上下文必须穿过线程边界**。OTel 的 `trace_id` 在 `ThreadPoolExecutor.submit` 时默认**不会**自动传递——必须在提交任务时显式用 `context.attach()` / `copy_context()` 复制 context。否则 Grafana 上看到的是断的 trace,跨 Step 串不起来。
4. **异常边界闭合在单个 Step**。一个 Step 抛未捕获异常,只应该把那个 StepRun 标 `FAILED`,**不能冒泡**到 Run 级把整条 Pipeline 搞死(除非策略显式要求 `fail_fast`)。worker 线程的顶层必须有 `try/except + log`。
5. **优雅停机是硬需求**。GKE 滚动升级时 Pod 收到 SIGTERM,Control Plane 必须:(a) 不再接受新 Run;(b) 把正在跑的 Step 要么等完成、要么标成 `INTERRUPTED`;(c) 释放 State Store 租约。否则每次升级都会丢 Run——这是团队级部署里最容易被忽视、出事又最痛的一条。

#### 水平扩展的天花板:"单 Pod 多线程多 Run" 的隐含上限

"单 Pod + 多线程 + 多 Run"是当前模型(基于用户描述推断),它有个**硬天花板**:一个 Pod 的 CPU / 内存决定了能并发多少 Run × Step。撑不住时有两条路:

- **路线 A:多 Pod Control Plane**。多个 Pod 瓜分 Run,需要做 **Run 的 ownership 分配**——Pod 领到一个 Run 后别的 Pod 别碰。实现上常用两种机制:
  - Pub/Sub 消费组:一条消息只会被一个 subscriber 消费。
  - State Store 里的租约锁(lease):Pod 拿到 `Run.owner_pod = self + ttl`,心跳续约。
  各 Pod 内部仍然是"多线程多 Run"。
- **路线 B:每 Run 起一个 Pod(Kubernetes Job 模型)**。类似 Airflow 的 KubernetesExecutor。好处是彻底隔离、按量计费、不怕互相拖累;代价是 Pod 冷启动延迟(秒级)、观测分散、State Store 压力变成"Pod 启停事件风暴"。

用户原始描述是"**多 DAG 也用多线程执行**",这意味着多 Run 在**同一个进程**里——也就是**路线 A 的单 Pod 起点**。这对小团队部署足够了(一个 Pod 开 4 核 8G 能跑很多 Run),但 §04 会把**从单 Pod 扩到多 Pod** 该做什么说清楚——因为一旦业务起来,这是不可避免的演进。

## 1.4 一次 Pipeline 的完整生命周期(端到端时序)

把一次 Pipeline 执行拆成**13 个离散步骤**。后面章节讲某一块时,都可以把读者带回这条时序上定位。

1. **作者写 DAG**:`analytics_daily.json` 描述了 6 个 Step 的依赖关系、每个 Step 用的命令名、参数引用。
2. **上传 GCS**:`gsutil cp analytics_daily.json gs://emf-pipelines/team-x/`。
3. **GCS 触发**:Object Finalize 事件经 **GCS 原生的 Pub/Sub Notification**(不经过 Eventarc)投到 Pub/Sub topic `emf-pipeline-upload`。
4. **Ingestor 消费**:Control Plane 里的 `event-ingestor` Pod 收到消息,在 State Store 中写入一条 `Run` 记录(status=`PENDING_LOAD`),并去重(按 `bucket+object+generation`)。
5. **Loader 下载**:Loader 从 GCS 拉取 JSON,走 JSON Schema 校验 + 命令白名单校验 + 参数引用静态检查。任何失败 → `Run.status = LOAD_FAILED`,把错误写回 GCS 结果前缀 + 记 Cloud Logging。
6. **编译为内部 DAG**:Compiler 把 JSON 里的 step 列表转成内部的 `Graph` 对象(节点是 `StepSpec`,边是依赖关系);同时构建 `ParamResolver`(知道每个 step 的参数从哪里来:DAG 常量 / 事件 / 前序 step 输出)。
7. **调度初始化**:Scheduler 把图写入 State Store,所有 Step 状态=`WAITING`,更新 `Run.status = RUNNING`。
8. **拓扑派发**:Scheduler 找出入度为 0 的 Step,**提交到线程池**并行执行(这是 DAG 内并发);与此同时,**本 Pod 里其他 Run 的 Scheduler 也在并行推进**(这是 DAG 间并发);每个 Step 分派前,先让 `ParamResolver` 把参数具象化——注意要把 OTel context 显式传递进线程,否则 trace 会断。
9. **执行分派**:Command Runner 按命令 `kind` 选路径:
    - `local`:进程内直接调用 Python 函数(**用当前 Team 的 ServiceAccount 访问 BigQuery / GCS 等**)。
    - `dbt`:在命名空间隔离下起子进程执行 `dbt run --models ... --vars ...`(仍然在 Team A 边界内)。
    - `sidecar`:构造 HTTP 请求发给**中心化的 Sidecar Service**,请求头带上 Team A ServiceAccount 的 **ID Token**(Sidecar 用它来识别调用方、计配额、挑凭证)。
10. **状态上报**:命令执行完回写 `StepRun.status`、`StepRun.output`(作为后续 Step 可引用的"上游产物"),Scheduler 收到事件后计算下一批可调度节点。
11. **失败处理**:若 Step 失败,按 Step 级策略重试(指数退避,次数上限);最终失败则触发下游取消 + `Run.status = FAILED`。
12. **成功收尾**:全部 Step 成功 → `Run.status = SUCCEEDED`,把汇总结果(产物 URI、执行时长、每步耗时)写回 GCS 结果前缀,便于业务侧直接消费。
13. **产物可见**:业务系统从 GCS 结果前缀拿到数据,循环闭合。

### 1.4.1 这个链路里最容易被忽视的三件事

- **第 5 步的"静态检查"**:在任何命令真正执行之前,就把**所有参数引用**(例如 `${steps.extract.output.rows}`)检查一遍——引用的 Step 是否存在、引用的字段是否在上游命令的 output schema 里。这相当于给 DAG 做"类型检查",能在上传瞬间把错误顶回去,而不是跑到一半炸。**这是 AI Authoring Copilot 的最重要抓手之一**(§10)。
- **第 8 步的"具象化"**:参数解析不是执行前一次性做完,而是**每个 Step 派发前实时做**,因为前序 Step 的输出直到它跑完才存在。这决定了 `ParamResolver` 需要能够**惰性求值**。
- **第 10 步的"输出"**:EMF 的跨 Step 传值**不走 Airflow XCom**(把值塞进 Meta DB),而更可能是"**小结果入 State Store,大结果入 GCS,Step 间传的是 GCS URI**"(推断,原始输入未指定)。这是数据编排场景和 Airflow 典型用法的关键差异。

## 1.5 EMF vs Airflow 逐维对照

| 维度 | Airflow | EMF |
|------|---------|-----|
| **部署模型** | **中心化服务**(一套 Scheduler/Webserver/DB 供全公司用) | **去中心化 framework**,每个 Team 自己 clone 源码、自己部署 |
| DAG 表达 | Python 模块 | JSON 文件 |
| DAG 分发 | Git-sync / shared volume / image | GCS 上传到**本 Team 的 bucket**(事件驱动) |
| 触发方式 | Scheduler 周期扫描 + Sensor + REST | GCS 原生 Pub/Sub Notification(一等公民) |
| 执行单元 | Operator(代码级扩展) | Command(注册 + schema,协议级扩展) |
| **执行模型** | SequentialExecutor / LocalExecutor / CeleryExecutor / KubernetesExecutor(四种选其一) | **单 Pod 多线程**:一个 ThreadPoolExecutor 服务 (多 Run × 每 Run 多 Step) 两层并发 |
| 跨任务传值 | XCom(Meta DB) | Context + 产物 URI(推断:小入 State Store / 大入 GCS) |
| 分支 / 循环 | BranchOperator / 动态 DAG | 声明式表达,或通过命令 fan-out(§05) |
| 权限 | 自建 RBAC Web UI | **GCP IAM 直接即权限**(当前 Team 的 ServiceAccount 能做什么,DAG 就能做什么) |
| 扩展门槛 | 写 Operator,打包,发版 | 写 Command(本地函数 / Sidecar endpoint) + schema 注册 |
| 外部 API 集成 | 在 Operator 里直接写 | **统一走中心化 Sidecar Service**(FastAPI on GKE) |
| 失败面 | Scheduler / Executor / DB / Worker 都是 SPOF,影响全公司 | 各 Team Control Plane 互不相干;只有中心 Sidecar 是唯一跨团队 SPOF |
| 观测栈 | 自带 UI + 接第三方 | Cloud Logging + **OpenTelemetry → Grafana** |
| 用户群 | 主要是工程师 | 工程师 + 分析师 + 业务同学 |

对照的**真正重点**不是"EMF 更好",而是:**同一类问题,Airflow 把答案留在代码里,EMF 把答案上升到配置 + 平台**。这让 EMF 有更小的业务侧学习成本,但也给平台自己带来了更多责任(schema 严苛、静态检查、命令治理)。

## 1.6 AI 赋能的切入点地图(只列锚点,展开见 §10~§12)

先把 AI 的潜在落点挂在架构图上,后面分章展开:

| 挂载点 | AI 能做什么 | 展开章节 |
|--------|-------------|----------|
| DAG 文件编写阶段(用户侧) | 自然语言→DAG 初稿、schema 纠错、参数引用补全、依赖推断 | §10 |
| Loader 静态校验阶段 | 错误消息语义化(把 JSON Schema 报错翻译成人话 + 修复建议) | §10 |

## 1.7 本章关键结论

1. **EMF 是 framework 不是 service**:每个 Team 自己 clone 源码、自己部署,权限完全走 GCP IAM。这一条是后续所有章节的隐形前提——看到任何"为什么这么设计",先回想这一条。
2. **架构是刻意的非对称**:数据面去中心化(跟着 IAM 走),外部 API 面中心化(跟着第三方 vendor 的配额 / 凭证走)。这不是偷懒,是"一致性边界在哪里"的精细判断。
3. **EMF 不是 Airflow 的阉割版,而是把"命令作为组织边界"的设计**;Operator 在 Airflow 是代码级扩展点,Command 在 EMF 是组织级治理单位。
4. **事件驱动不是一种触发方式,而是一种数据模型**——GCS Pub/Sub Notification 投来的事件字段就是 DAG 的一等输入。
5. **分层的核心不是"把事情拆细",而是"让每一层的失败域独立"**。Ingestor vs Loader、Control Plane vs Sidecar、Team A vs Team B,本质都是这个原则在不同尺度上的应用。
6. **静态检查期(Loader 阶段)是平台最大的价值洼地之一**,也是 AI 介入最容易立刻产生效果的地方。
7. **跨 Step 传值要分"小 vs 大"**:配置级的值可以走 State Store,数据级的产物应该走 GCS URI——否则会把 State Store 当 Data Lake 用,很快就崩。
8. **观测栈厂商无关(OTel + Grafana),身份栈厂商绑定(GCP IAM)**:两者看似矛盾,其实是对"谁需要跨环境统一、谁需要借现成基础设施"的分别回答。
9. **并发是两层叠加**:同一个 Run 里无依赖 Step 并行,多个 Run 之间也并行。Python 选线程而不是 asyncio / 多进程,是因为工作负载 IO-bound + 同步 SDK 生态最合线程模型;代价是纯 Python CPU-bound 命令会 GIL-block 线程池,这条必须在命令开发规约里硬禁。
