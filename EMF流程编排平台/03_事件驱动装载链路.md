# 03 · 事件驱动装载链路

## 3.0 本章导读

第 2 章定下了 DAG 文件的内容边界:文件内容只放节点图,也就是 `id / command / parents / parameters`;Pipeline 级 metadata 放在 **GCS object metadata** 上。

本章要把这件事接到运行链路里:

```text
用户 / CI 上传 DAG 文件到 GCS
        ↓
GCS Object Finalize
        ↓
GCS 原生 Pub/Sub Notification
        ↓
Event Ingestor 去重并创建 Run
        ↓
Loader 固定 generation 下载内容 + metadata
        ↓
Normalize / Validate / Compile
        ↓
创建 StepRun,交给 Scheduler
```

这里有两个关键点:

1. **不走 Eventarc**。EMF 使用 GCS 原生 Pub/Sub Notification,让每个 Team 的 Control Plane 直接消费自己 bucket 的 object finalize 事件。
2. **metadata 是对象的一部分治理契约**。Loader 不只读 JSON content,还必须读 GCS object metadata,并把它快照进 Run 记录。

这一章关注的是"装载链路",也就是从 GCS 事件到内部 DAG 编译完成。真正的拓扑调度、线程池执行和 Step 状态流转放到第 4 章。

---

## 3.1 为什么这条链路要事件驱动

EMF 的典型使用方式不是"每天凌晨调度一次某个 Python DAG",而是:

> 某个团队把一份 DAG JSON 文件放到自己的 GCS bucket,平台立即解释并执行。

这件事如果用轮询做,会有三个问题:

- 延迟不可控:轮询间隔短会浪费资源,间隔长用户体验差。
- 运维不自然:每个 Team 都要维护扫描状态、last seen marker、错误重试。
- 语义不精准:轮询看到"文件存在"和"文件刚刚完成上传"不是一回事。

GCS Object Finalize 事件刚好给了 EMF 最想要的语义:

- 文件上传完成才触发。
- 事件里带 bucket、object、generation 等定位信息。
- Pub/Sub 天然支持 at-least-once delivery 和消费者水平扩展。
- 每个 Team 的部署只消费自己 bucket 的 topic,符合去中心化 framework 模型。

所以第 1 章说"事件是一等输入",不是一个口号。事件不只是启动信号,它会变成一次 Run 的 source identity:

```text
source = bucket + object + generation + metageneration + sha256
```

没有这个 source identity,后面就没法做去重、排障、重放和审计。

---

## 3.2 为什么不用 Eventarc

GCP 里也可以用 Eventarc 接 GCS 事件,再转发给 Cloud Run / GKE / Workflows。但 EMF 当前更适合直接用 GCS 原生 Pub/Sub Notification。

| 维度 | GCS Pub/Sub Notification | Eventarc |
|------|---------------------------|----------|
| 链路长度 | GCS -> Pub/Sub -> Ingestor | GCS -> Eventarc -> 目标服务 |
| 每 Team 部署 | 简单,每个 bucket 配一个 topic/subscription | 还要维护 trigger 和目标绑定 |
| 消费模型 | Control Plane 直接拉/订阅 Pub/Sub | 更偏事件路由层 |
| 失败控制 | Pub/Sub ack / redelivery 语义清楚 | 多一层事件路由状态 |
| EMF 需要的能力 | object finalize + at-least-once 已足够 | CloudEvents 规范化暂时不是刚需 |

这不是说 Eventarc 不好,而是它解决的问题更像"跨服务事件路由",而 EMF 这里只需要一条很短的、Team 内部可控的装载链路。少一层中转,失败面也少一层。

> 推断:如果未来 EMF 要统一接收不止 GCS 的事件源,例如 BigQuery job、Scheduler、GitHub webhook、内部审批回调,那 Eventarc 或统一事件网关才值得重新评估。

---

## 3.3 Bucket、Topic、Subscription 的部署边界

每个 Team 部署一套 Control Plane,也应该有自己的 GCS bucket 和 Pub/Sub 资源。

推荐资源形态:

```text
Team A GCP Project
├── GCS bucket: team-a-emf-dags
│   └── watched prefix: prod/
├── Pub/Sub topic: emf-dag-object-finalize
├── Pub/Sub subscription: emf-control-plane-loader
└── GKE namespace: emf
    └── Deployment: event-ingestor / loader / scheduler
```

最小 IAM:

| 主体 | 权限 | 用途 |
|------|------|------|
| GCS service agent | publish 到 Pub/Sub topic | 发送 object finalize 事件 |
| EMF Control Plane ServiceAccount | `pubsub.subscriptions.consume` | 消费事件 |
| EMF Control Plane ServiceAccount | `storage.objects.get` | 读取 DAG content 和 object metadata |
| EMF Control Plane ServiceAccount | `storage.objects.create` | 写 Run 结果 / 错误报告 |

这里仍然遵守第 1 章的原则:

> 当前 Team 的 EMF 能读什么,由当前 Team 的 GCP ServiceAccount 和 IAM 决定。EMF 自己不再发明第二套权限系统。

## 3.4 GCS Object Metadata 是 DAG 治理面

第 2 章已经明确:DAG JSON content 只放节点图。那 pipeline name、owner、environment、labels 从哪里来?答案是 GCS object metadata。

推荐必填 metadata:

```text
pipeline-name: daily_export
owner: growth-analytics
environment: prod
```

推荐可选 metadata:

```text
tier: batch
description: Daily export for marketing analytics
notification-channel: data-alerts
pipeline-version: 2026.04.23
labels.domain: marketing
labels.cost-center: growth
```

### 3.4.1 metadata 不是 DAG 语法

Loader 读取 metadata,但不把 metadata 当作 DAG content 的一部分。内部结构应该分开:

```python
@dataclass(frozen=True)
class DagFileMetadata:
    bucket: str
    object: str
    generation: str
    metageneration: str
    content_sha256: str
    pipeline_name: str
    owner: str
    environment: str
    labels: dict[str, str]
```

```python
@dataclass(frozen=True)
class NodeSpec:
    id: str
    command: str
    parents: tuple[str, ...]
    parameters: dict[str, Any]
```

这样可以避免两个常见坏味道:

- 为了放 owner / env,把 DAG JSON 外面包一层越来越厚的 manifest。
- 让 `parameters` 偷偷承担治理字段,最后变成垃圾桶。

### 3.4.2 metadata 不要放 secret

GCS object metadata 会出现在对象元信息、日志、审计和 Run snapshot 里。它适合放治理信息,不适合放凭证。

不要这样:

```text
api-key: sk_live_xxx
db-password: ...
```

外部 API 凭证应该走 Secret Manager 或中心化 Sidecar 的凭证托管。metadata 只表达"谁的 DAG、哪个环境、怎么聚合、失败通知谁"。

### 3.4.3 metadata 更新不是新 Run

GCS object 的 content generation 和 metadata metageneration 是两件事:

- 内容更新会产生新的 object generation。
- metadata 单独更新会改变 metageneration。

EMF 触发 Run 的主事件应该是 **Object Finalize**,也就是新的内容 generation 完成上传。metadata 更新不应该单独触发一次 DAG 执行。

但 Loader 必须把当时读到的 metageneration 和 metadata snapshot 记进 Run,否则历史 Run 的 owner / env 会被后续 metadata 更新"改写"。

## 3.5 上传协议:最容易被低估的一层

如果 metadata 在 GCS object 上,那上传方式就变得非常重要。

### 3.5.1 metadata 必须和内容一起发布

坏流程:

```text
1. 上传 daily_export.json
2. GCS Object Finalize 触发 EMF
3. Loader 开始读 object metadata
4. CI 再 patch owner/environment metadata
```

这个流程有竞态:Loader 可能在 metadata patch 之前就读到了对象,然后因为缺 owner 失败。

推荐流程:

```text
1. CI 在同一次上传里写入 content + metadata
2. GCS Object Finalize 触发 EMF
3. Loader 读取完整 content + metadata
```

如果工具链很难做到"同次上传写 metadata",可以用两阶段发布:

```text
1. 上传到 staging/ 前缀,不触发 EMF
2. 设置好 metadata
3. copy/rewrite 到 watched prod/ 前缀,并保留 metadata
4. prod/ 下的 Object Finalize 触发 EMF
```

### 3.5.2 避免 watch 到临时文件

如果用户直接往 watched prefix 写:

```text
prod/daily_export.json.tmp
prod/daily_export.json
```

Ingestor 必须过滤:

- 只接受 `.json` 或约定后缀。
- 忽略 `_tmp/`、`staging/`、`.tmp`。
- 忽略结果目录,避免 EMF 写结果时触发自己。

推荐 bucket 布局:

```text
gs://team-a-emf/
├── dags/prod/        # 只监听这里
├── dags/staging/     # CI 发布中间态,不监听
└── results/          # Run 结果,不监听
```

### 3.5.3 object versioning 的取舍

Run source 里会保存 `bucket + object + generation`。但如果 bucket 没开 object versioning,旧 generation 在覆盖后可能无法长期读取。

推荐:

- DAG bucket 开启 object versioning,至少保留一段时间。
- Loader 在成功装载后,把原始 DAG content 快照复制到 Run 结果目录。
- Run 记录保存 content sha256,用于确认快照没有漂移。

这样即使源对象后来被覆盖或清理,历史 Run 仍然可以回答:

> 当时到底跑的是哪份 DAG?

## 3.6 Pub/Sub 消息进入 Ingestor

GCS 发到 Pub/Sub 的消息里通常会包含:

```text
bucket
object/name
generation
event type
event time
```

具体字段名以当前 GCS notification payload 格式为准。EMF 内部不要把云厂商事件对象一路传到底,而是第一时间转成自己的事件结构:

```python
@dataclass(frozen=True)
class DagObjectEvent:
    bucket: str
    object: str
    generation: str
    event_type: str
    event_time: datetime
    raw_message_id: str
    raw_attributes: dict[str, str]
```

Ingestor 的职责要保持很窄:

1. 校验事件类型是不是 object finalize。
2. 校验 bucket / prefix / suffix 是否在允许范围内。
3. 计算幂等 key。
4. 在 State Store 里创建或复用 Run。
5. ack Pub/Sub message。

它**不应该**做完整 DAG 解析。完整解析属于 Loader,因为解析失败需要形成结构化 `LOAD_FAILED` 结果,而不是让 Pub/Sub 一直重投。

### 3.6.1 幂等 key

Pub/Sub 是 at-least-once,重复投递是正常情况。Ingestor 必须用 source identity 去重。

推荐幂等 key:

```text
dag_source_key = sha256(bucket + "\n" + object + "\n" + generation)
```

为什么不把 Pub/Sub message id 当幂等 key?

- 同一个 GCS 事件可能因为重投递出现多次 delivery。
- 真正代表"这次 DAG 内容"的是 object generation,不是消息本身。

如果同一个 object path 被覆盖:

```text
daily_export.json generation=100 -> Run A
daily_export.json generation=101 -> Run B
```

这应该是两个不同 Run,因为内容 generation 不同。

### 3.6.2 ack 时机

Ingestor 的 ack 规则很关键:

```text
只有在 Run/PendingLoad 记录已经持久化后,才能 ack Pub/Sub message。
```

如果在写 State Store 前 ack,进程崩溃会直接丢 Run。

如果 Loader 解析失败,不要让 Pub/Sub 重投。正确做法是:

```text
Run.status = LOAD_FAILED
error 写入 State Store / GCS result path / 日志
Pub/Sub message 已经 ack
```

Pub/Sub 重投解决的是"消息没被平台接住",不是"用户 DAG 写错了"。

## 3.7 Run 创建:先 PENDING_LOAD,再 LOAD_FAILED/RUNNING

Ingestor 接到事件后,先创建一条轻量 Run 记录:

```json
{
  "run_id": "run_20260423_abc123",
  "source_key": "sha256(bucket/object/generation)",
  "status": "PENDING_LOAD",
  "source": {
    "bucket": "team-a-emf",
    "object": "dags/prod/daily_export.json",
    "generation": "1713840000000000"
  },
  "created_at": "2026-04-23T10:00:00Z"
}
```

这条记录的意义是:

- 去重:重复消息发现已有 source_key,直接复用 Run。
- 可观测:即使后面 Loader 失败,也有一个 Run 可以展示错误。
- 解耦:Ingestor 可以很快 ack,Loader 慢慢处理。

状态流:

```text
PENDING_LOAD
   ├── LOAD_FAILED
   └── LOADED / RUNNING
```

如果 Loader 还没有开始,Run 停在 `PENDING_LOAD`。如果 Loader 下载对象失败但判断是临时错误,可以进入 `LOAD_RETRYING`;如果是 schema 错误,进入 `LOAD_FAILED`。

## 3.8 Loader:固定 generation 读取 content + metadata

Loader 处理 `PENDING_LOAD` Run 时,必须固定 source generation 读取,不能简单读 latest:

```text
read gs://team-a-emf/dags/prod/daily_export.json#generation=1713840000000000
```

否则会出现:

```text
10:00 GCS 事件: generation=100
10:01 用户覆盖文件: generation=101
10:02 Loader 读取 latest,读到了 101
```

这会让 Run source 和实际执行内容不一致。

Loader 要做的动作:

1. 读取 object metadata。
2. 校验必填 metadata。
3. 下载 DAG content。
4. 计算 sha256。
5. 解析 JSON。
6. Normalize 成 `NodeSpec`。
7. 绑定 Command Registry。
8. 校验 parameters。
9. 编译 parents graph。
10. 创建 StepRun。
11. 把 Run 状态推进到 `RUNNING` 或 `LOADED`。

伪代码:

```python
def load_run(run: RunRecord) -> None:
    obj = gcs.get_object(
        bucket=run.source.bucket,
        name=run.source.object,
        generation=run.source.generation,
    )

    dag_metadata = read_and_validate_metadata(obj.metadata)
    content = obj.download_as_bytes()
    content_sha256 = sha256(content)

    raw_nodes = parse_json_nodes(content)
    nodes = normalize_nodes(raw_nodes)

    command_bindings = bind_commands(nodes)
    validate_parameters(nodes, command_bindings)
    graph = compile_graph(nodes)

    state_store.save_loaded_run(
        run_id=run.id,
        metadata_snapshot=dag_metadata,
        content_sha256=content_sha256,
        nodes=nodes,
        graph=graph,
    )
```

### 3.8.1 metadata 缺失怎么处理

如果缺 `owner`、`pipeline-name`、`environment`,建议直接 `LOAD_FAILED`:

```text
LOAD_FAILED:
missing required GCS object metadata key "owner".
```

原因是没有 owner 的 Run 后续无法可靠告警、聚合和审计。平台可以允许 dev 环境宽松一点,但 prod 最好强制。

### 3.8.2 JSON 错误怎么处理

JSON parse、重复 id、unknown parent、unknown command、parameters schema 错误,都属于用户可修复错误。

处理方式:

- Run 标记 `LOAD_FAILED`。
- 保存结构化错误。
- 如果有 `notification-channel`,发出可读错误。
- 不重投 Pub/Sub message。

错误对象建议结构化:

```json
{
  "code": "UNKNOWN_PARENT",
  "message": "node \"load\" declares unknown parent \"extract\"",
  "node_id": "load",
  "field": "parents",
  "retryable": false
}
```

这样第 10 章的 AI Authoring Copilot 才能基于错误对象给出修复建议,而不是解析一段自由文本。

## 3.9 State Store 里至少要有三类记录

### 3.9.1 IngestEvent / SourceRecord

用于去重和审计:

```json
{
  "source_key": "sha256(bucket/object/generation)",
  "bucket": "team-a-emf",
  "object": "dags/prod/daily_export.json",
  "generation": "1713840000000000",
  "first_seen_at": "2026-04-23T10:00:00Z",
  "run_id": "run_123"
}
```

这个表/集合要对 `source_key` 做唯一约束。并发重复消息同时进来时,只能有一个创建成功。

### 3.9.2 RunRecord

用于展示和生命周期推进:

```json
{
  "run_id": "run_123",
  "status": "PENDING_LOAD",
  "pipeline_name": null,
  "owner": null,
  "source_key": "sha256(...)",
  "source": {
    "bucket": "team-a-emf",
    "object": "dags/prod/daily_export.json",
    "generation": "1713840000000000",
    "metageneration": null,
    "sha256": null
  },
  "created_at": "2026-04-23T10:00:00Z"
}
```

Loader 成功后补上 metadata snapshot、sha256、compiled graph summary。

### 3.9.3 StepRunRecord

Loader 编译成功后创建:

```json
{
  "run_id": "run_123",
  "step_id": "export_orders",
  "command": "ExportToGcs",
  "parents": [],
  "status": "WAITING",
  "attempt": 0
}
```

第 4 章会展开 StepRun 的状态机。

## 3.10 装载链路的失败模式

### 3.10.1 Pub/Sub 重复投递

现象:

```text
同一个 GCS generation 收到两条消息。
```

处理:

- 用 `source_key = bucket/object/generation` 去重。
- State Store 对 source_key 做唯一约束。
- 重复消息直接 ack,返回已有 run_id。

### 3.10.2 metadata 竞态

现象:

```text
Object Finalize 先到,metadata 后 patch。
```

处理:

- 发布协议要求 content + metadata 同次上传。
- 或者使用 staging -> prod copy 的两阶段发布。
- prod Loader 对必填 metadata 缺失直接失败。

### 3.10.3 读 latest 导致错跑

现象:

```text
事件是 generation=100,Loader 读到 generation=101。
```

处理:

- Loader 必须按 event generation 固定读取。
- Run 记录保存 generation 和 sha256。

### 3.10.4 写结果触发自己

现象:

```text
EMF 写 results/manifest.json,同一个 bucket notification 又触发一条 DAG Run。
```

处理:

- 监听 prefix 只包含 `dags/prod/`。
- results 前缀不要被 Ingestor 接受。
- Ingestor 做 suffix 和 prefix filter。

### 3.10.5 用户重复上传同名 DAG

现象:

```text
daily_export.json generation=100 和 generation=101 都触发。
```

处理:

- 这是两个不同 Run,不是重复消息。
- UI / 查询层可以按 pipeline-name 聚合展示。
- 如果业务不希望自动执行每次覆盖,就不能把 prod prefix 作为直接执行入口,需要引入审批或发布状态。

### 3.10.6 Loader 临时失败

现象:

```text
GCS 读超时、State Store 写超时、Command Registry 临时不可用。
```

处理:

- 如果 Ingestor 已经创建 Run 并 ack,Loader 自己负责重试。
- Run 可进入 `LOAD_RETRYING`。
- 重试耗尽后才 `LOAD_FAILED`。

## 3.11 观测与审计

装载链路至少要打这些指标:

| 指标 | 含义 |
|------|------|
| `emf.ingest.events_total` | 收到的 GCS 事件数 |
| `emf.ingest.duplicates_total` | 去重命中的重复事件 |
| `emf.load.success_total` | Loader 成功次数 |
| `emf.load.failed_total` | Loader 失败次数 |
| `emf.load.duration_ms` | 从 PENDING_LOAD 到 LOADED/RUNNING 的耗时 |
| `emf.load.metadata_missing_total` | metadata 缺失错误数 |
| `emf.load.validation_failed_total` | DAG 内容校验失败数 |

低基数 attributes:

- `pipeline_name`
- `owner`
- `environment`
- `error_code`
- `command`

不要把这些作为 metric label:

- 完整 GCS object path。
- run_id。
- Pub/Sub message id。
- 完整错误文本。

这些高基数字段可以进 log 和 trace event。

审计上,Run 记录要能回答:

```text
谁的 DAG? owner metadata 是什么?
哪个文件? bucket/object/generation 是什么?
哪版 metadata? metageneration 是多少?
内容有没有漂移? sha256 是什么?
谁上传的? GCS audit log 里对应 principal 是谁?
哪版 EMF 执行的? image digest / git sha 是什么?
```

注意 owner metadata 不等于上传者身份。真正的上传者要看 GCS Audit Logs;owner 是业务归属字段。

## 3.12 AI 可以在装载链路做什么

AI 不应该参与"是否执行"这种硬判定,但可以增强错误解释和修复建议。

最有价值的点:

1. **Load error 翻译**:把 JSON schema / command schema 错误翻译成业务用户能看懂的话。
2. **修复 patch 建议**:比如 unknown parent 时建议把 `parents:["extract"]` 改成 `parents:["extract_orders"]`。
3. **metadata 发布检查**:发现缺 owner/environment 时,给出具体上传命令或 CI 配置建议。
4. **相似 DAG 检索**:从历史成功 DAG 里找相似节点结构,建议 command 和 parameters。
5. **失败聚类**:统计一个团队最近 20 次 LOAD_FAILED 是否都是同一种 metadata 或 schema 错误。

但 AI 只能基于结构化错误对象工作。Loader 如果只吐一段自由文本,AI 就只能猜;Loader 如果能产出 `code / node_id / field / expected / actual`,AI 就能给出稳定修复。

---

## 3.13 本章关键结论

1. **EMF 的装载链路是 GCS Object Finalize -> Pub/Sub -> Ingestor -> Loader**,不走 Eventarc。
2. **DAG content 和 GCS object metadata 分工明确**:content 是节点图,metadata 是 pipeline 级治理信息。
3. **上传协议必须保证 content + metadata 一起发布**,否则 Object Finalize 和 metadata patch 会产生竞态。
4. **Ingestor 只做轻量接入和去重**,不要做完整 DAG 解析;解析错误应该由 Loader 形成 `LOAD_FAILED`。
5. **幂等 key 应该基于 bucket/object/generation**,不能基于 Pub/Sub message id。
6. **Loader 必须固定 generation 读取**,不能读 latest,否则会出现事件和实际内容不一致。
7. **Run 记录必须保存 generation、metageneration、sha256、metadata snapshot 和 runtime 版本**,这是 replay 和审计的底座。
8. **Loader 错误要结构化**,这样用户、告警和后续 AI Copilot 才能真正消费。
