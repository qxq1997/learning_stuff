# 05 · 命令体系与参数注入

## 5.0 本章导读

前四章已经把 EMF 的主链路串起来了:

- DAG 文件内容是节点列表:`id / command / parents / parameters`。
- GCS object metadata 保存 pipeline 级治理信息。
- Loader 把 DAG 编译成内部 graph。
- Scheduler 根据 `parents` 找 ready step,再把 Step 派给 worker 线程。

但真正执行业务动作的是 `command`。

本章要回答:

1. `command` 为什么是稳定协议名,不是 Python 类名。
2. Command Registry 应该声明哪些信息。
3. `parameters`、平台注入的 `CommandContext`、parent outputs 怎么合成最终入参。
4. local / dbt / sidecar 三类 command 的执行边界有什么差异。
5. 重试、幂等、输出、错误语义为什么必须在 command 层声明。

一句话:

> EMF 的 DAG DSL 之所以能保持很轻,是因为复杂度被压到了 Command Registry 和 Command Runner 里。

---

## 5.1 Command 是协议,不是函数

在 DAG 里,用户写:

```json
{
  "id": "export_orders",
  "command": "ExportToGcs",
  "parents": [],
  "parameters": {
    "query": "select * from mart.orders",
    "destination": "gs://team-a-export/${run_id}/orders.jsonl"
  }
}
```

这里的 `ExportToGcs` 不应该被理解成:

```text
某个 Python 类名
某个模块路径
某个可 import 的函数名
```

它应该被理解成:

```text
一条稳定的、对 DAG 作者暴露的命令协议名
```

内部可以映射到真实实现:

```text
ExportToGcs -> emf.commands.bigquery_export.ExportToGcsCommand
```

也可以将来映射成:

```text
ExportToGcs -> sidecar endpoint /v1/commands/export-to-gcs
```

只要协议不变,DAG 不需要改。

### 5.1.1 为什么不能让 DAG 直接引用代码路径

坏例子:

```json
{
  "command": "emf.commands.bigquery_export.ExportToGcsCommand"
}
```

问题很多:

- 代码重构会破坏历史 DAG。
- 用户能看到平台内部模块结构。
- DAG 作者会开始依赖实现细节。
- Loader 很难做命令白名单治理。
- AI 生成 DAG 时会被代码路径污染。

所以 Command Registry 要做一层隔离:

```text
DAG command id  ->  CommandBinding  ->  实现类 / sidecar route / dbt adapter
```

这层隔离就是平台治理能力。

---

## 5.2 Command Registry 最少要声明什么

`ExportToGcs` 不是一个裸函数,而是一份 command spec。

推荐结构:

```json
{
  "name": "ExportToGcs",
  "display_name": "Export BigQuery result to GCS",
  "kind": "local",
  "resource_group": "bigquery",
  "implementation": "emf.commands.bigquery_export.ExportToGcsCommand",
  "input_schema": {
    "type": "object",
    "required": ["query", "destination"],
    "additionalProperties": false,
    "properties": {
      "query": {
        "type": "string"
      },
      "destination": {
        "type": "string",
        "format": "gcs_uri"
      },
      "format": {
        "type": "string",
        "enum": ["JSONL", "CSV", "PARQUET"],
        "default": "JSONL"
      }
    }
  },
  "output_schema": {
    "type": "object",
    "required": ["gcs_uri", "job_id"],
    "properties": {
      "gcs_uri": {
        "type": "string",
        "format": "gcs_uri"
      },
      "row_count": {
        "type": "integer"
      },
      "job_id": {
        "type": "string"
      }
    }
  },
  "retry_policy": {
    "max_attempts": 3,
    "backoff": "exponential",
    "retryable_errors": ["TRANSIENT", "RATE_LIMITED"]
  },
  "idempotency": {
    "mode": "required",
    "key_scope": "run_step"
  },
  "timeout": {
    "default": "30m",
    "max": "2h"
  },
  "recovery": {
    "external_status_query": true
  }
}
```

字段分组:

| 字段 | 作用 |
|------|------|
| `name` | DAG 里引用的稳定 command id |
| `kind` | 决定走 local / dbt / sidecar 哪条执行路径 |
| `resource_group` | 决定用哪个并发闸门 |
| `implementation` | 内部实现绑定 |
| `input_schema` | 校验 DAG `parameters` |
| `output_schema` | 校验 command output,支撑下游引用 |
| `retry_policy` | 控制可重试错误和退避策略 |
| `idempotency` | 告诉 Runtime 如何生成/传递幂等键 |
| `timeout` | 默认超时和最大超时 |
| `recovery` | 进程崩溃后能否查询外部状态 |

### 5.2.1 Registry 可以怎么落地

常见有三种方式。

#### 方式 A:Python decorator

```python
@command(
    name="ExportToGcs",
    kind="local",
    resource_group="bigquery",
    input_schema=EXPORT_TO_GCS_INPUT,
    output_schema=EXPORT_TO_GCS_OUTPUT,
)
class ExportToGcsCommand(Command):
    ...
```

优点:

- schema 和实现靠近。
- 本地开发体验好。
- 代码审查时能一起看。

缺点:

- AI / 文档系统要额外导出 registry。
- 去中心化部署下,不同 Team commit 的 registry 可能不同。

#### 方式 B:独立 JSON/YAML registry

```text
commands/
  ExportToGcs.json
  RunDbt.json
  CallSidecar.json
```

优点:

- 静态可读。
- 方便文档和 AI Copilot 消费。
- 容易生成 command catalog。

缺点:

- schema 和实现可能漂移。
- 需要 CI 检查实现和 registry 一致。

#### 方式 C:代码注册 + 启动时导出

启动时把所有 command spec dump 成:

```text
gs://team-a-emf-runtime/command-registry.snapshot.json
```

或写入 State Store。这样每次 Run 都能记录当时的 command registry snapshot。

这是最适合 EMF 的折中:开发仍然用 Python 注册,但运行时和 AI 可以消费稳定快照。

## 5.3 Command kind:local / dbt / sidecar

Command Registry 的 `kind` 决定执行路径。

### 5.3.1 local command

local command 在 Control Plane 进程内执行,通常是 Python 函数 / 类。

适合:

- GCS 文件操作。
- BigQuery 查询 / load / export。
- 小规模 JSON manifest 生成。
- 轻量数据校验。

特点:

- 使用当前 Team 的 GCP ServiceAccount。
- 和 Scheduler 共用同一个 Pod 资源。
- 必须设置 SDK timeout。
- 不能做长时间 CPU-bound Python 计算。

### 5.3.2 dbt command

dbt command 通常走 subprocess:

```text
dbt run --select tag:marketing --vars '{"business_date":"2026-04-23"}'
```

适合:

- SQL transformation。
- 数据 mart 构建。
- dbt test。

特点:

- 仍然在 Team 自己的 GCP/IAM 边界内。
- 需要管理 dbt project 路径、profiles、target、vars。
- 超时可以 terminate 子进程。
- stdout/stderr 要结构化采集,不要只丢到日志里。

### 5.3.3 sidecar command

sidecar command 通过 HTTP 调中心化 Sidecar Service。

适合:

- Salesforce / HubSpot / Jira 等第三方 SaaS。
- 公司级 API key / OAuth token。
- 全公司共享 rate limit。

特点:

- Control Plane 只负责构造标准请求。
- Sidecar 负责第三方凭证、限流、重试、SDK 兼容。
- 请求头需要携带调用方身份,例如 Team ServiceAccount 的 ID Token。
- Sidecar 返回要符合 command output schema。

## 5.4 Command 接口形态

local command 可以统一成一个接口:

```python
class Command(Protocol):
    spec: CommandSpec

    def run(self, parameters: dict[str, Any], context: CommandContext) -> dict[str, Any]:
        ...
```

dbt 和 sidecar 也可以适配成同样接口:

```python
class DbtCommandAdapter:
    def run(self, parameters, context):
        return run_dbt_subprocess(parameters, context)
```

```python
class SidecarCommandAdapter:
    def run(self, parameters, context):
        return call_sidecar_http(parameters, context)
```

这样 Scheduler / Worker 不需要关心底层类型:

```python
command = command_registry.get(node.command)
output = command_runner.run(command, parameters, context)
```

### 5.4.1 统一接口不等于统一实现

统一接口的意义是:

- 同一套参数解析。
- 同一套 output validation。
- 同一套错误对象。
- 同一套 retry / timeout / idempotency 处理。
- 同一套 trace span。

但实现可以完全不同:

- local 是 Python 调用。
- dbt 是 subprocess。
- sidecar 是 HTTP RPC。

这是 EMF command 体系的核心:对 DAG 作者统一,对平台实现开放。

## 5.5 参数注入:parameters + context + parent outputs

DAG 中只有:

```json
{
  "parameters": {
    "destination": "gs://team-a-export/${run_id}/orders.jsonl",
    "source_uri": "${export_orders.output.gcs_uri}"
  }
}
```

执行时真正传给 command 的内容来自三类来源:

```text
resolved_parameters = resolve(parameters_template, runtime_scope)
context = build_command_context(run, step)
```

### 5.5.1 用户参数:parameters

`parameters` 是用户显式写的入参。它应该只包含业务参数:

- query。
- destination。
- format。
- dbt selector。
- API object name。

不要把下面这些塞进 `parameters`:

- owner。
- environment。
- retry count。
- idempotency key。
- trace id。
- ServiceAccount token。
- secret 明文。

这些属于平台上下文或基础设施,不属于 DAG 作者的业务参数。

### 5.5.2 平台上下文:CommandContext

平台注入的上下文:

```python
@dataclass(frozen=True)
class CommandContext:
    run_id: str
    step_id: str
    attempt: int
    pipeline_name: str
    owner: str
    environment: str
    source_bucket: str
    source_object: str
    source_generation: str
    source_metageneration: str
    trace_id: str
    deadline: datetime
    idempotency_key: str
    service_account_email: str
```

Command 可以读 context,但 DAG 作者不能直接伪造 context。

### 5.5.3 parent outputs

上游输出来自 State Store:

```json
{
  "export_orders": {
    "gcs_uri": "gs://team-a-export/run_123/orders.jsonl",
    "row_count": 1234567,
    "job_id": "bq_job_abc"
  }
}
```

参数引用可以解析:

```json
{
  "source_uri": "${export_orders.output.gcs_uri}"
}
```

但引用前提是:

- `export_orders` 是当前节点的 parent 或 ancestor。
- `ExportToGcs.output_schema` 声明了 `gcs_uri`。
- 当前目标参数类型兼容 `gcs_uri`。

这些最好在 Loader 阶段尽量静态检查,在执行前再做一次运行时检查。

## 5.6 参数解析规则

### 5.6.1 整值引用保留类型

如果整个参数值就是一个引用:

```json
{
  "row_count": "${export_orders.output.row_count}"
}
```

而上游 output 里:

```json
{
  "row_count": 1234567
}
```

那么解析后应该是 integer:

```json
{
  "row_count": 1234567
}
```

不要变成字符串 `"1234567"`。

### 5.6.2 字符串内插值变成字符串

如果引用嵌在字符串里:

```json
{
  "message": "exported ${export_orders.output.row_count} rows"
}
```

解析后是:

```json
{
  "message": "exported 1234567 rows"
}
```

### 5.6.3 不支持任意表达式

不要支持:

```json
{
  "limit": "${export_orders.output.row_count + 1}"
}
```

也不要支持:

```json
{
  "enabled": "${env == 'prod' && row_count > 0}"
}
```

需要计算时,做成 command:

```json
{
  "id": "check_export_result",
  "command": "ValidateExportResult",
  "parents": ["export_orders"],
  "parameters": {
    "row_count": "${export_orders.output.row_count}"
  }
}
```

让 command 输出结构化结果,而不是把 DAG DSL 变成表达式语言。

### 5.6.4 缺失引用直接失败

如果执行前发现:

```json
{
  "source_uri": "${export_orders.output.gcs_uri}"
}
```

但 `export_orders.output.gcs_uri` 不存在,当前 Step 应该失败:

```text
PARAM_RESOLUTION_FAILED:
missing output field "gcs_uri" from parent "export_orders".
```

这通常意味着上游 command 的 output schema 和真实 output 不一致,或者 DAG 引用了错误字段。

## 5.7 Input Schema:把错误挡在 Loader 阶段

Command input schema 至少要支持:

- required。
- type。
- enum。
- format。
- default。
- additionalProperties。
- sensitive marker。

例如:

```json
{
  "type": "object",
  "required": ["query", "destination"],
  "additionalProperties": false,
  "properties": {
    "query": {
      "type": "string",
      "minLength": 1
    },
    "destination": {
      "type": "string",
      "format": "gcs_uri"
    },
    "format": {
      "type": "string",
      "enum": ["JSONL", "CSV", "PARQUET"],
      "default": "JSONL"
    }
  }
}
```

### 5.7.1 默认值在哪里补

默认值建议在 Loader 或参数解析阶段补齐,并进入 `parameters_snapshot`:

```json
{
  "query": "select * from mart.orders",
  "destination": "gs://team-a-export/run_123/orders.jsonl",
  "format": "JSONL"
}
```

这样排障时能看到 command 实际收到的完整参数。

不要让 command 实现偷偷自己补默认值,否则外部看不到真实执行配置。

### 5.7.2 sensitive 字段不能落日志

如果某个 command 确实有 secret ref 参数:

```json
{
  "properties": {
    "password_ref": {
      "type": "string",
      "format": "secret_ref",
      "x-emf-sensitive": true
    }
  }
}
```

那么:

- `parameters_snapshot` 只保存 secret ref,不保存 secret value。
- log / trace 不打印该字段值。
- Command Runner 在最后一刻解析 Secret Manager。
- output 中也不能返回 secret。

---

## 5.8 Output Schema:下游引用的契约

Command output 也必须有 schema。

例如 `ExportToGcs`:

```json
{
  "type": "object",
  "required": ["gcs_uri", "job_id"],
  "properties": {
    "gcs_uri": {
      "type": "string",
      "format": "gcs_uri"
    },
    "row_count": {
      "type": "integer"
    },
    "job_id": {
      "type": "string"
    }
  }
}
```

执行完成后,Command Runner 要校验 output:

```python
output = command.run(parameters, context)
validate_json_schema(command.output_schema, output)
state_store.mark_step_succeeded(step_id, output)
```

如果 output schema 不合法,Step 应该失败:

```text
OUTPUT_SCHEMA_INVALID:
command ExportToGcs did not return required field "gcs_uri".
```

### 5.8.1 output 只放小结果

output 适合放:

- GCS URI。
- BigQuery table id。
- job id。
- row count。
- checksum。
- manifest uri。

不适合放:

- 大量行数据。
- 大文件内容。
- 完整 API response。
- 超长日志。

大产物应该落 GCS / BigQuery / 外部系统,output 只保存引用。

---

## 5.9 Error Schema:别只抛 Exception

Command 错误要标准化。不要只让 worker 捕获一个 Python exception 字符串。

推荐错误结构:

```json
{
  "code": "RATE_LIMITED",
  "message": "BigQuery quota exceeded for project team-a",
  "retryable": true,
  "requires_manual_check": false,
  "details": {
    "quota": "bigquery.jobs.create"
  }
}
```

常见错误码:

| code | retryable | 含义 |
|------|-----------|------|
| `INVALID_PARAMETERS` | false | 参数不合法 |
| `AUTH_FAILED` | false | IAM / token / secret 错误 |
| `NOT_FOUND` | false/true | 取决于资源是否最终一致 |
| `RATE_LIMITED` | true | 配额或限流 |
| `TRANSIENT` | true | 临时网络/服务错误 |
| `UPSTREAM_5XX` | true | 上游服务 5xx |
| `SIDE_EFFECT_UNKNOWN` | false | 副作用状态未知 |
| `OUTPUT_SCHEMA_INVALID` | false | command 返回不符合契约 |

### 5.9.1 retryable 由 command 决定

同样是 `NOT_FOUND`,不同 command 语义不同:

- 读刚写入的 GCS object,短暂 `NOT_FOUND` 可能可重试。
- 查一个用户填写的固定路径,`NOT_FOUND` 多半不可重试。

所以 retryable 不应该只由错误码全局决定,而应该结合 command spec。

## 5.10 幂等:所有有副作用的 command 都要面对重放

有副作用的 command 包括:

- 写 GCS。
- 创建/覆盖 BigQuery 表。
- 调第三方 API 创建资源。
- 发 Slack / Email。
- 写数据库。

这些 command 必须声明幂等策略。

### 5.10.1 幂等模式

推荐几种模式:

| mode | 含义 | 例子 |
|------|------|------|
| `none` | 无副作用或纯读 | 查询状态 |
| `destination_overwrite` | 目标路径固定,重复执行覆盖同一对象 | ExportToGcs |
| `caller_provided_key` | Runtime 传 idempotency key | 创建工单 |
| `external_job_id` | 外部系统支持 job id | BigQuery job |
| `not_idempotent` | 不可自动重试 | 发送一次性通知 |

Command spec:

```json
{
  "idempotency": {
    "mode": "external_job_id",
    "key_scope": "run_step"
  }
}
```

### 5.10.2 `run_id:step_id` 通常比 attempt 更好

如果幂等 key 包含 attempt:

```text
run_123:export_orders:attempt_1
run_123:export_orders:attempt_2
```

外部系统会认为这是两次不同请求。对于 create/write 操作,这可能导致重复副作用。

所以大多数写操作应该用:

```text
run_123:export_orders
```

attempt 适合用于纯读或每次尝试都需要不同 trace 的场景,不适合作为副作用幂等边界。

### 5.10.3 BigQuery job id 例子

`ExportToGcs` 可以把 idempotency key 映射成 BigQuery job id:

```text
emf_run_123_export_orders
```

如果 worker 重试,再次提交同一个 job id:

- 如果 BigQuery 说 job 已完成,读取结果并返回 output。
- 如果 job 正在跑,等待或查询状态。
- 如果 job 失败,按错误语义处理。

这样即使 worker 在提交 job 后崩溃,恢复时也有外部状态可查。

## 5.11 Resource Group:命令层的并发治理

第 4 章提到 command-kind semaphore。这个信息应该来自 Command Registry:

```json
{
  "name": "ExportToGcs",
  "resource_group": "bigquery"
}
```

资源组可以是:

- `bigquery`
- `gcs`
- `dbt`
- `sidecar-salesforce`
- `sidecar-hubspot`
- `notification`

每组有自己的并发上限:

```json
{
  "bigquery": {"max_concurrency": 8},
  "dbt": {"max_concurrency": 2},
  "sidecar-salesforce": {"max_concurrency": 16}
}
```

为什么不只靠线程池?

因为 32 个线程可能全去打 BigQuery,把 Team project 的 job quota 打满。resource group 是保护下游的闸门,线程池只是保护本 Pod 的闸门。

---

## 5.12 Recovery:命令要告诉平台怎么恢复

进程崩溃后,某个 Step 可能停在 `RUNNING`。平台要决定:

- 可以查询外部状态并恢复吗?
- 可以安全重跑吗?
- 必须人工介入吗?

Command spec 应该声明:

```json
{
  "recovery": {
    "strategy": "query_external_status",
    "status_fields": ["job_id", "destination"],
    "safe_to_rerun": true
  }
}
```

### 5.12.1 可恢复 command

`ExportToGcs` 通常可恢复:

- 有 BigQuery job id。
- 有固定 destination。
- 可以查 destination object 是否存在。
- 可以用 checksum / row count 校验。

### 5.12.2 不可确定 command

`SendSlackMessage` 可能不可恢复:

- 消息可能已经发出。
- Slack API 超时后不一定能按幂等 key 查询。
- 重跑会重复通知。

这类 command 应该:

- 默认不自动重试。
- 或要求外部 idempotency key。
- 或把失败标记为 `SIDE_EFFECT_UNKNOWN`。

## 5.13 Command 开发规约

为了让命令池长期可治理,每条 command 都应该满足这些规约。

### 5.13.1 必须有 schema

每条 command 必须声明:

- input schema。
- output schema。
- error codes。
- retry policy。
- idempotency。
- timeout。
- resource group。

没有 schema 的 command 不应该进入生产 command pool。

### 5.13.2 禁止纯 Python CPU-bound 大计算

Control Plane 使用线程模型。纯 Python CPU-bound 任务会被 GIL 卡住,拖慢整个 Pod。

不要在 local command 里做:

```python
for row in millions_of_rows:
    heavy_python_transform(row)
```

替代方案:

- 推到 BigQuery / dbt。
- 使用释放 GIL 的库。
- 放到单独 worker / job。
- 拆成 sidecar 或批处理服务。

### 5.13.3 不要吞异常

坏例子:

```python
try:
    do_export()
except Exception:
    return {"ok": False}
```

这会让 Runtime 不知道该不该重试、怎么告警。

正确做法是抛出或返回结构化错误:

```python
raise CommandError(
    code="RATE_LIMITED",
    message="BigQuery quota exceeded",
    retryable=True,
)
```

### 5.13.4 output 必须稳定

不要今天返回:

```json
{"gcs_uri": "..."}
```

明天改成:

```json
{"uri": "..."}
```

下游 DAG 可能依赖 `${export_orders.output.gcs_uri}`。output 字段一旦发布,就要按协议维护兼容。

## 5.14 Command Registry 与去中心化部署

EMF 是 framework,每个 Team 自己部署。这带来一个现实问题:

```text
Team A 的 ExportToGcs 来自 commit X
Team B 的 ExportToGcs 来自 commit Y
```

所以 Run 必须记录:

- EMF git sha / image digest。
- command registry snapshot version。
- 每个 Step 绑定到的 command spec hash。

推荐 StepRun 增加:

```json
{
  "step_id": "export_orders",
  "command": "ExportToGcs",
  "command_spec_hash": "sha256:...",
  "command_kind": "local",
  "resource_group": "bigquery"
}
```

这样半年后排查时,能知道当时 `ExportToGcs` 的 schema 和 retry policy 到底是什么。

## 5.15 AI 赋能命令体系的抓手

AI 写 DAG 能不能靠谱,主要取决于 Command Registry 质量。

一个好的 registry 可以让 AI:

- 根据自然语言选择正确 command。
- 根据 input schema 补齐 parameters。
- 根据 output schema 引用上游字段。
- 根据 command description 避免选错相似命令。
- 根据 error schema 给出修复建议。

但 registry 里只写:

```json
{
  "name": "ExportToGcs",
  "description": "export"
}
```

AI 基本只能猜。

更好的 command description 要写清:

- 这个 command 做什么。
- 什么时候该用。
- 什么时候不该用。
- 每个参数的业务含义。
- 输出字段怎么被下游使用。
- 常见错误怎么修。

这些信息既服务 AI,也服务人类 DAG 作者。

## 5.16 本章关键结论

1. **Command 是稳定协议名,不是代码路径**。DAG 引用 `ExportToGcs`,内部通过 registry 映射到实现。
2. **DAG DSL 能保持轻量,前提是 Command Registry 足够严格**。input/output/error/retry/idempotency/timeout/resource_group 都应该声明。
3. **`parameters` 是用户业务入参,`CommandContext` 是平台注入上下文**,两者不要混在一起。
4. **参数解析要区分整值引用和字符串插值**,并且不支持任意表达式。
5. **output schema 是下游引用的契约**,不能随意改字段名或类型。
6. **错误必须结构化**,否则 Runtime 无法决定重试、告警和人工介入。
7. **所有有副作用的 command 都要声明幂等策略**,否则重试和恢复会制造重复副作用。
8. **resource_group 是保护下游系统的并发闸门**,不能只靠线程池保护本 Pod。
9. **去中心化部署下必须记录 command spec hash / registry snapshot**,否则历史 Run 很难 replay 和审计。
10. **AI Authoring Copilot 的能力上限由 command schema 和 description 决定**。
