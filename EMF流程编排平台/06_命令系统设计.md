# 06 命令系统设计

## Command Pattern + DI

EMF 的命令系统是典型的 Command Pattern 加 DI。

`Message` 里只有一个命令名:

```text
message.command
```

执行时再通过 module provider 找到具体 module,解析参数,并调用统一的执行方法。

这个设计的核心目标是:

```text
让 workflow 只依赖稳定 command 协议,
不要依赖具体 Python 类、函数名或 service 实现.
```

## 执行入口

核心流程:

1. 根据 `message.command` 找 module。
2. 读取 module 声明的参数元数据。
3. 用 `MessageParameterParser` 解析和替换参数。
4. 调用 `execute_with_parameters()`。

可以写成:

```text
message.command
  -> ModuleProvider
  -> Parameter metadata
  -> MessageParameterParser
  -> module.execute_with_parameters(...)
```

## Module 契约

所有 module 都遵循同一类契约,至少暴露这些能力:

| 方法 | 用途 |
|---|---|
| `registered_command_name()` | 内部命令名 |
| `registered_az_command_name()` | Azure 语境下的命令名 |
| `get_parameters()` | 声明参数元数据 |
| `execute_with_parameters()` | 带解析后参数执行 |

这让命令模块不是裸函数,而是“带元数据、可解析参数、可注册映射”的执行单元。

## 命令注册

命令注册列表承担索引作用:

```text
command name -> module class -> parameter contract -> execution path
```

## ModuleProvider 的双重匹配

项目处于 Azure 迁移语境中,很多内部类名仍然带有 `Gbg` / `Gcs` 这类历史命名,但对外命令会映射到 `ADB` / `ASA` 等 Azure 命令名。

因此 `ModuleProvider` 会同时匹配:

```text
internal command name
Azure command name
```

这意味着看代码时不要只按类名猜命令。应该看:

```text
registered_command_name()
registered_az_command_name()
命令注册表
module provider
```

## 命令系统的边界

命令系统解决的是“这条消息具体做什么”。

它不负责:

- 消息从哪里来。
- 节点什么时候 ready。
- batch 如何分组。
- run 如何创建。

这些分别由 producer、DagContext、WorkerMessageAppender、OE-RUN 等组件处理。

命令系统只需要拿到一条已经被调度到执行阶段的 `Message`,然后完成:

```text
command name -> module -> parameters -> execution
```

## 外部服务失败和副作用

很多 command 不只是本地计算,还会调用外部系统:

```text
Databricks / Storage / Catalogue / API / 第三方服务
```

这时 command 需要把外部服务当成不可靠依赖处理。一个节点被调度执行,不代表它一定能完成业务副作用。

典型流程应该是:

```text
Message D 被调度执行
  -> command 调用外部服务
  -> 外部服务成功
  -> D 标记 COMPLETE

Message D 被调度执行
  -> command 调用外部服务失败 / timeout
  -> D 进入 RETRY / FAILED / TIMEOUT
  -> 可重试时生成 retry Message
```

最重要的是:D 不能在外部服务失败时直接标记 `COMPLETE`。否则 DAG 会继续推进,下游节点会基于一个并没有真实完成的副作用继续执行。

外部失败可以按类型处理:

| 失败类型 | 处理方式 |
|---|---|
| 短暂网络抖动 | retry with backoff |
| 服务短暂不可用 | 生成带未来 `run_date` 的 retry message |
| 长时间不可用 | 超过最大 retry 后标记 `FAILED` 或 `TIMEOUT` |
| 服务已执行成功但响应丢失 | 用 idempotency key / correlation id 查询和去重 |
| 外部产生部分副作用 | 补偿、回滚、reconciliation 或人工介入 |

因此 command 设计里要特别关注幂等性。因为 retry 可能让同一个业务节点多次调用外部服务,所以最好给外部请求带稳定业务 key:

```text
idempotency_key = run_uuid + order_id
```

必要时也可以加 attempt 信息:

```text
external_request_id = run_uuid + order_id + attempt_id
```

这样外部服务可以识别“这是同一个业务动作的重试”,避免重复扣款、重复建资源、重复写数据或重复发布事件。

面试时可以总结:

```text
EMF 的调度层只能保证 D 到了该执行的时候被执行;
D 对外部系统的副作用一致性,要靠 command 的 retry、timeout、幂等 key 和补偿逻辑保证。
```

## 命令系统为什么适合面试讲

它体现了一个常见系统设计原则:

```text
把业务能力做成稳定协议,把实现隐藏在注册和适配层后面.
```

这带来三个收益:

1. workflow 的稳定性更好。内部重构 module 不一定影响 workflow。
2. 平台治理更集中。参数解析、命令注册、权限、审计都可以挂在命令边界上。
3. 扩展方式更清晰。新增业务能力不是改调度器,而是新增 command module。

代价也很明确:

1. command 协议一旦被大量 workflow 使用,就要保持兼容。
2. module registry 会变成平台核心资产,需要治理命名、参数和版本。
3. 参数解析层如果过于魔法,调试成本会上升。

## 和普通函数调用的区别

普通函数调用是:

```text
代码直接 import function -> call(args)
```

EMF command 调用是:

```text
Message.command
  -> registry/provider 找 module
  -> module 声明参数
  -> parser 从 message/run/template 解析参数
  -> execute_with_parameters()
```

这多出来的一层不是为了复杂,而是为了让 workflow 作者、调度器和业务实现之间解耦。
