# 08 动态工作流能力

## DAG 不是一次性静态图

EMF 支持命令运行时继续生成消息。

这意味着 EMF 的 DAG 不是只能在 run 开始时一次性构建完成,而是可以随着 command 执行继续扩展。

`OE-RUN` 是 workflow 展开器,而另一些命令可以在业务执行过程中根据运行结果继续制造新节点。

动态扩展的本质是:

```text
某个 command 在执行过程中调用 MessageAppender.enqueue_all();
这些新消息加入同一个 run_uuid 对应的 DagContext;
DagContext 重新维护 parents / children;
后续 consumer 会继续调度新 ready 节点.
```

## 例子: ADB-MESSAGES-FROM-QUERY

典型例子是:

```text
ADB-MESSAGES-FROM-QUERY
```

它做的事:

1. 执行一个 query。
2. 返回多条 SQL job。
3. 为每条结果生成新的 `ADB-SQL-EVAL` 消息。
4. 把这些新消息 enqueue 回 EMF。

## 运行时扩展示意

可以把它想成:

```text
Message A: ADB-MESSAGES-FROM-QUERY
  -> query returns job_1, job_2, job_3
  -> create Message B: ADB-SQL-EVAL(job_1)
  -> create Message C: ADB-SQL-EVAL(job_2)
  -> create Message D: ADB-SQL-EVAL(job_3)
  -> enqueue B/C/D
```

这些新消息进入 `WorkerMessageAppender` 和 `DagManager` 后,就会成为运行时 DAG 的一部分。

这不是一次性启动新 run,而是把后续消息接进当前 DAG。典型字段仍然是:

```text
run_uuid: 当前 DAG
order_id: 新节点位置
parents: 新节点依赖哪些已有节点
command: 新节点要执行什么
```

## 动态 DAG 的价值

动态 DAG 适合这些场景:

- 任务数量要运行时查询后才能确定。
- 一个上游 query 结果决定后续要跑多少 SQL job。
- catalogue resolution 后才知道实际数据集或表。
- 子 workflow 需要递归展开。

如果强行用静态 DAG 表达,就会出现大量占位节点或复杂分支。EMF 选择让 command 可以继续生成消息,用消息本身扩展图。

## 动态能力的代价

动态 DAG 也会让系统治理更重要:

| 风险 | 需要的治理 |
|---|---|
| 某个命令生成过多消息 | 限制 fan-out 数量 |
| 重复 enqueue | 幂等 key 和 audit log |
| 动态依赖难追踪 | 保留 `run_uuid`、`order_id`、`parents` |
| 失败后难恢复 | retry、timeout、batch cleanup |

这也是为什么 `Message` 必须携带足够完整的运行时上下文。

## 关键结论

EMF 的 workflow 不是一张预先固定的图,而是一张可以运行时增长的动态图。

因此学习系统时要同时关注两类入口:

```text
外部事件 / OE-RUN 生成初始消息
业务 command 运行时继续生成消息
```

前者启动 workflow,后者扩展 workflow。

## 动态扩展和 retry 的区别

动态扩展生成的是新的业务节点。retry 生成的是同一业务节点的重试消息。

| 场景 | 生成的消息含义 |
|---|---|
| 动态扩展 | 新的 `order_id`,代表新的 DAG 节点 |
| retry | 通常保留原 `order_id`,代表原节点重新尝试 |

所以排查“为什么多了一条消息”时,要先看它的 `order_id`、`parents` 和 `command`:如果 `order_id` 还是原来的,更可能是 retry;如果新 `order_id` 接到了当前 DAG,更可能是动态展开。

## 动态 DAG 的系统设计意义

动态 DAG 解决的是“任务数量和依赖关系在运行前不完全知道”的问题。

传统静态 DAG 适合:

```text
任务集合固定;
依赖关系固定;
每次运行只是参数不同.
```

EMF 的动态 DAG 适合:

```text
任务数量由 query 结果决定;
子 workflow 运行时展开;
命令结果决定后续 fan-out;
retry 和延迟消息要重新进入调度系统.
```

它的关键不是“能不能生成新任务”,而是新任务生成后仍然回到同一套 DAG 语义:

```text
run_uuid 归属同一张图;
order_id 表示节点身份;
parents 表示依赖;
command 表示执行能力;
parameters 表示执行输入.
```

这样动态扩展不会变成调度器之外的旁路逻辑。

## 动态 fan-out / fan-in

动态扩展最典型的形态是 fan-out:

```text
Q: 执行 query
  -> 查询出 T1, T2, T3, ..., T100
  -> 为每条结果生成一个 Message
```

原本 workflow 可能只是:

```text
Q -> Next
```

运行时展开后更像:

```text
Q
  -> T1 -> Next
  -> T2 -> Next
  -> T3 -> Next
  -> ...
  -> T100 -> Next
```

这里最关键的是 `Next` 的依赖要被重写或重接。否则 `Next` 可能在所有动态任务完成前就执行,也可能永远等不到正确的父节点。

所以动态 fan-out 不是“多发几条队列消息”这么简单,而是要把新消息接回当前 DAG 的依赖语义里。

## retry 也是一种动态消息

retry 可以理解成一种特殊的动态扩展。

一个节点失败后,系统通常不是在原地同步 while 重试,而是生成一条新的 retry message:

```text
原节点 M 执行失败
  -> 生成 M_retry
  -> 带未来 run_date
  -> 到时间后重新参与 ready 判断
```

这种做法的好处是:

- 每次尝试都有独立消息记录,便于审计和排查。
- retry delay 可以用 `run_date` 表达,不需要占用执行线程等待。
- retry 仍然回到同一套 DAG 调度模型,不会变成命令内部的私有循环。
- retry 次数、状态和依赖关系可以被平台统一治理。

面试时可以说:

```text
EMF 的 retry 不是原地重跑,而是把下一次尝试重新建模成 Message。
```

## timeout monitor 也是动态节点

如果 workflow 有最大运行时长,EMF 可以在启动时生成一个未来执行的 timeout monitor message:

```text
workflow start
  -> generate timeout monitor
  -> run_date = start_time + max_run_time
```

当这条 timeout message 到达执行时间时,它检查目标 run 是否已经完成:

```text
run 已完成: timeout message 什么都不做或安全结束
run 未完成: 标记 run timeout
```

这个机制的核心是:超时不是靠线程一直等待实现的,而是靠一条带未来 `run_date` 的消息重新进入调度系统实现的。

这和 retry 的设计是一致的:延迟行为也被建模成消息。

## 动态 DAG 的风险

动态能力越强,治理越重要:

| 风险 | 例子 | 治理思路 |
|---|---|---|
| fan-out 失控 | 一个 query 生成几万条消息 | 限制最大生成数量、分页、分批 |
| 依赖接错 | 新节点 parents 指向不存在的 order_id | 入队前校验、运行时检测 dangling parent |
| 重复生成 | retry 后重复 enqueue 同一批子任务 | 幂等 key、order_id 约束、audit |
| 难以观测 | 图运行中增长,排查困难 | 记录 message lineage 和生成来源 |

面试时可以把动态 DAG 讲成:

```text
它不是让命令绕过调度器,而是让命令通过生成 Message 回到调度器。
```

## 动态 DAG 的主要挑战

动态 DAG 的难点不是“能不能生成新 Message”,而是新 Message 进入系统后,图的正确性、规模、幂等和可观测性都要继续成立。

可以按下面几类问题理解。

### 1. 下游提前执行

这是最典型的问题。

原本 workflow 是:

```text
A -> D
```

如果 A 执行时动态生成了 B/C/E,但 D 仍然只依赖 A,就会变成:

```text
A -> D
A -> B
A -> C
A -> E
```

这时 A 一完成,D 就可能 ready,而 B/C/E 还没跑完。D 如果是汇总、发布、校验类节点,就会拿到不完整结果。

正确处理方式是把下游依赖重接:

```text
A -> B -> D
A -> C -> D
A -> E -> D
```

或者使用 completion token:

```text
A
  -> B
  -> C
  -> E
      -> COMPLETION_TOKEN
          -> D
```

核心原则:

```text
动态生成节点后,所有需要等待它们的下游节点都必须通过 relink 或 completion token 接回来。
```

### 2. fan-out 失控

动态 fan-out 很容易从“灵活”变成“爆炸”。

例如一个 query 原本预期返回 100 条结果,结果线上返回 100 万条。如果每条结果都生成一个 Message,会带来:

- `DagContext` 内存压力。
- ready 集合过大。
- worker 调度压力。
- audit log / 状态表膨胀。
- 外部服务被瞬间打爆。
- retry 时产生重试风暴。

治理方式:

- 限制最大 fan-out 数量。
- 分页或分批生成 message。
- 按 partition / shard / chunk 切分。
- 对 heavy command 使用独立并发限制。
- 对外部服务做 rate limit。
- 使用 backpressure,在下游拥堵时放慢生成速度。
- 大计算下推到 Databricks / Spark,worker 只编排任务。

### 3. `order_id` 冲突

动态生成的节点必须有稳定且唯一的 `order_id`。

如果不同 fan-out 分支都生成:

```text
VALIDATE
LOAD
EXPORT
```

那 `parents` 通过 `order_id` 找父节点时就可能找错,甚至覆盖已有节点语义。

更合理的命名方式是把业务 key 编进 `order_id`:

```text
VALIDATE_TABLE:orders
VALIDATE_TABLE:customers
VALIDATE_TABLE:payments
```

子 workflow 也类似,需要 namespace:

```text
PARENT_SUBFLOW:LOAD_DATA
PARENT_SUBFLOW:VALIDATE
```

### 4. dangling parent

动态节点如果写了不存在的 parent:

```text
parents = ["A"]
```

但当前 `run_uuid` 对应的 DAG 里没有 `A`,这个节点就永远不会 ready。

典型表现是:

```text
没有 ready 节点;
还有 INACTIVE 节点;
run 一直不结束。
```

治理方式:

- 入队前校验 parent 是否存在。
- 动态加边时校验 parent / child 合法性。
- 运行时检测 blocked nodes。
- 错误信息里输出 unresolved parents,方便定位。

### 5. 产生循环依赖

动态加边时也可能不小心加出环:

```text
A -> B -> C -> A
```

有环之后就不再是 DAG。调度器会卡住,因为环上的节点都在等待彼此。

处理方式:

- workflow 展开时做 topological sort。
- 动态加边时做 cycle detection。
- 如果新增边会造成环,拒绝入 DAG。
- 如果业务需要循环,展开成新的 iteration message:

```text
A_1 -> B_1 -> Check_1
              -> A_2 -> B_2 -> Check_2
```

并设置最大迭代次数、timeout 和退出条件。

### 6. retry 造成重复动态生成

假设 A 执行时已经生成了 B/C/E,但 A 后来失败并 retry。retry 后 A 可能又生成一遍 B/C/E。

问题会变成:

```text
B/C/E 被重复 enqueue;
重复执行;
重复写外部系统;
下游等待的节点集合不确定。
```

治理方式:

- 动态生成逻辑要幂等。
- 动态节点 `order_id` 要稳定。
- 入队时按 `run_uuid + order_id` 或业务 key 去重。
- audit log 记录这批子任务由谁生成。
- retry 时先判断子任务是否已经生成过。

### 7. run completion 判断变复杂

静态 DAG 里,系统一开始就知道所有节点。动态 DAG 里,运行中还会新增节点,所以不能简单判断:

```text
初始节点都完成了 => run 完成
```

更合理的判断要综合:

```text
没有 running 节点;
没有 ready 节点;
没有 future run_date 等待;
所有已知节点都进入终态;
不会再有动态生成入口。
```

复杂 workflow 里通常还需要 completion token 或 run state manager 来统一判断。

### 8. 可观测性变差

动态 DAG 会边跑边长。排查时常见问题是:

```text
这个节点是谁生成的?
为什么它的 parent 是这个?
它为什么接到了 D 前面?
为什么它重复出现?
它属于哪个 fan-out group?
```

治理方式:

- 记录 `generated_by` message。
- 记录 source command。
- 记录 fan-out group id。
- 保留 `run_uuid / order_id / parents / command / parameters` 快照。
- 记录 message lineage。
- 尽量提供运行时 DAG 可视化或审计查询。

### 9. 外部副作用更难保证

动态节点往往对应外部任务,例如 SQL job、文件处理、API 调用。

如果动态生成 100 个节点,其中 80 个成功、20 个失败,下游汇总节点必须知道如何处理:

- 是否等待全部成功。
- 是否允许部分成功。
- 是否单独 retry 失败分片。
- 是否回滚已经成功的副作用。
- 是否进入人工 reconciliation。

常见治理方式:

- 每个动态节点使用幂等 key。
- 外部写入先进入 staging。
- 下游 merge / validate 节点统一检查完整性。
- 对失败分片单独 retry,不要整批重跑。
- publish 节点只在所有必要分片完成后执行。

## 动态 DAG 的面试总结

动态 DAG 强在灵活,难在治理。

可以这样回答:

```text
动态 DAG 的挑战主要是五类:
第一是依赖正确性,新节点要接到正确位置,不能让下游提前执行。
第二是规模控制,fan-out 不能无限生成消息。
第三是幂等和重复生成,retry 时不能重复制造副作用。
第四是合法性校验,要防止 dangling parent 和 cycle。
第五是可观测性,运行时增长的图必须能追踪生成来源和依赖变化。
```

最短记忆版:

```text
动态 DAG = 灵活扩展 + 更强治理。
必须处理 relink、fan-out 限流、幂等去重、cycle detection、run completion 和 observability。
```
