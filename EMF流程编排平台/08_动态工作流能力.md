# 08 动态工作流能力

## 本章定位

第 34 章是 EMF 的 source of truth。本章只展开其中“动态 DAG 编排”这一块。

先统一口径：EMF 的基础动态能力不是“任意代码随便生成 DAG”，而是：

```text
OE-RUN 读取 PROCESS_TASKS
  -> 把每一行 task 展开成 command message
  -> 根据 order_id + parents 建立 DAG 依赖
  -> Worker 只执行依赖满足的 ready message
```

在这个基础上，EMF 可以进一步支持运行时动态扇出：某个 command 在执行过程中根据 SQL 查询结果、catalogue resolution 结果、文件内容或业务规则，继续生成一批新的 child messages。

所以本章要区分两层能力：

| 层次 | 说明 | 是否是主线 |
|---|---|---|
| `OE-RUN + PROCESS_TASKS` | workflow 定义来自表，运行时展开成 message DAG | 是，和第 34 章一致 |
| generator command / runtime fan-out | 某个节点执行后，根据运行结果继续生成子任务 | 是扩展能力 |
| retry / timeout / recovery 生成消息 | 运行治理动作也可能生成新 message | 是生产治理能力，不是动态 DAG 的核心定义 |

## DAG 为什么不是一次性静态图

在很多传统编排系统里，DAG 在运行前就基本确定：有哪些 task、依赖关系是什么、每个 task 跑什么逻辑，启动时已经写清楚。

EMF 不完全这样。EMF 的 workflow 可以在运行时根据元数据和业务输入推导出来：

1. 上游上传文件，blob metadata 中带有 `file_type`。
2. Worker 通过 `file_type` 查询 `LOAD_INFO`。
3. `LOAD_INFO` 决定 schema、dataset、表名前缀、写入方式和 `ingestion_workflow_name`。
4. Worker 构造 `OE-RUN`。
5. `OE-RUN` 通过 workflow 名称读取 `PROCESS_TASKS`。
6. `PROCESS_TASKS` 中每一行被转换成 command message。
7. 每个 message 通过 `parents` 形成依赖关系。

这已经是一种动态能力：同一套 Worker 代码不写死 workflow，而是由 metadata 和 `PROCESS_TASKS` 决定本次运行应该有哪些任务。

## 基础动态 DAG：OE-RUN 展开 PROCESS_TASKS

`PROCESS_TASKS` 是 workflow 的 DAG 定义表。它通常包含：

| 字段 | 含义 |
|---|---|
| `workflow` | workflow 名称 |
| `order_id` | task 节点 ID |
| `command` | 要执行的 EMF command |
| `parents` | 上游 task 的 `order_id` 列表 |
| `parameters` | command 参数模板 |
| `topic` | 目标消息 topic |
| `enabled` / `disabled` | 是否启用该 task |

一个简化例子：

```json
[
  {
    "workflow": "risk_ingestion",
    "order_id": "create_table",
    "command": "ADB-CREATE-TABLE-FROM-FILE-TYPE",
    "parents": [],
    "parameters": {
      "file_type": "${file_type}"
    }
  },
  {
    "workflow": "risk_ingestion",
    "order_id": "load_file",
    "command": "ADB-LOAD-TABLE-FROM-ASA-FILE",
    "parents": ["create_table"],
    "parameters": {
      "entity_uuid": "${entity_uuid}"
    }
  },
  {
    "workflow": "risk_ingestion",
    "order_id": "finalise",
    "command": "ADB-FINALISE",
    "parents": ["load_file"],
    "parameters": {
      "target_dataset": "${dataset_name}"
    }
  }
]
```

`OE-RUN` 读取后会做几件事：

1. 校验 `order_id` 是否唯一。
2. 校验 `parents` 指向的节点是否存在。
3. 用 run 参数替换 `${file_type}`、`${entity_uuid}`、`${dataset_name}` 等模板。
4. 把每一行 task 转成一条 command message。
5. 把这些 message 放入同一个 `run_uuid` / `batch_id` 下。
6. 根据 `parents` 控制 ready 条件。

这就是 EMF 最基本的动态编排。

## 运行时扇出是什么

运行时扇出是更高级的一层：workflow 里有一个 generator task，它执行后才知道要生成多少个子任务。

可以先别把它想成“动态 DAG”，而是想成一句很朴素的话：

```text
某个 task 跑完以后，才知道后面要跑几件事；
于是它把这些事生成出来，交回 EMF 调度器继续跑。
```

静态 DAG 是：

```text
A -> B -> C
```

这里 A、B、C 在运行前就已经确定。

运行时扇出更像：

```text
A -> GenerateTasks -> ? -> C
```

`?` 这一批任务在运行前不知道有几个。要等 `GenerateTasks` 跑完，查询或计算出结果后，系统才知道要生成哪些子任务。

例如：

```text
finalise
  -> generate_exports
      -> export_US
      -> export_UK
      -> export_SG
  -> notify_downstream
```

这里 `generate_exports` 在运行前并不知道到底要导出几个 region。它需要先查询 metadata、SQL 表或配置，得到一组结果，再把每一行结果映射成一个 child message。

假设它运行了这条 SQL：

```sql
select region, entity_uuid, target_path
from export_manifest
where run_uuid = '${run_uuid}'
```

返回 3 行：

```text
US, entity-us, /exports/us.csv
UK, entity-uk, /exports/uk.csv
SG, entity-sg, /exports/sg.csv
```

那它就生成 3 个 child messages：

```json
{
  "order_id": "export_US",
  "command": "EXPORT-TO-STORAGE",
  "parents": ["generate_exports"],
  "parameters": {
    "region": "US",
    "entity_uuid": "entity-us",
    "target_path": "/exports/us.csv"
  }
}
```

```json
{
  "order_id": "export_UK",
  "command": "EXPORT-TO-STORAGE",
  "parents": ["generate_exports"],
  "parameters": {
    "region": "UK",
    "entity_uuid": "entity-uk",
    "target_path": "/exports/uk.csv"
  }
}
```

```json
{
  "order_id": "export_SG",
  "command": "EXPORT-TO-STORAGE",
  "parents": ["generate_exports"],
  "parameters": {
    "region": "SG",
    "entity_uuid": "entity-sg",
    "target_path": "/exports/sg.csv"
  }
}
```

关键点是：`generate_exports` **不是自己去 export**。它只是负责“查出要 export 哪些东西，然后生成子任务”。

所以运行时扇出的核心区别是：

```python
for row in rows:
    do_export(row)          # 不推荐，绕过调度器
```

应该变成：

```python
for row in rows:
    create_child_message(row)   # 推荐，交回调度器
```

第一种方式只是一个 task 内部开 for loop 干活，EMF 看不到每个子任务的状态。第二种方式会把每个子任务变成 EMF message，所以每个 child 都能被平台管理：有状态、有重试、有日志、有依赖、有超时、有失败处理。

这个 generator command 的责任不是直接做所有业务，而是回答四个问题：

1. 要扇出哪些 child command。
2. 每个 child command 的参数是什么。
3. 这些 child command 的 `order_id` 和 `parents` 怎么生成。
4. 下游 fan-in 节点什么时候可以继续执行。

一句话总结：

```text
运行时扇出 = 一个 generator command 先算出“后面要跑哪些任务”，
再把这些任务生成成 child messages，交回 EMF 的 DAG 调度器执行。
```

## 运行时扇出的状态机主流程

运行时扇出最好按时间线理解，而不是按某个单独节点理解。

它的核心状态推进是：

```text
generator READY
  -> generator RUNNING
  -> 创建 fanout group
  -> 创建 child task instances
  -> generator SUCCESS
  -> child tasks RUNNING / SUCCESS / FAILED
  -> fanout group 聚合完成
  -> fan-in task READY
```

这里最重要的一点是：

```text
generator SUCCESS 只代表“子任务已经生成完成”；
fanout group SUCCESS 才代表“这批子任务已经按规则完成”。
```

### generator 的状态

generator task 先作为普通任务被调度：

```text
CREATED -> READY -> RUNNING
```

它在 `RUNNING` 阶段做这些事：

1. 渲染 SQL、catalogue criteria 或 manifest path。
2. 查询或读取 source，得到 child rows。
3. 校验返回字段能否填充 child message 模板。
4. 检查 `max_children`、command allowlist、dedupe key。
5. 创建 fanout group，状态为 `EXPANDING`。
6. 创建 child task instances。
7. 写 expansion snapshot / audit log。
8. 发布或 append child messages。

如果这些步骤都成功，generator 才能进入：

```text
RUNNING -> SUCCESS
```

这个 `SUCCESS` 不代表 export 已经完成，只代表 `export_US`、`export_UK`、`export_SG` 这些 child tasks 已经被可靠创建。

如果 generator 自己失败，则可能进入：

```text
RUNNING -> FAILED
RUNNING -> RETRYING
RUNNING -> TIMEOUT
RUNNING -> CANCELLED
```

### fanout group 的状态

fanout group 是“这一批动态子任务”的聚合记录。它不是业务 task，但它决定下游什么时候能继续。

推荐状态是：

```text
CREATED -> EXPANDING -> OPEN -> SUCCESS
                                |-> FAILED
                                |-> PARTIAL_SUCCESS
                                |-> CANCELLED
                                |-> EMPTY
```

含义：

| 状态 | 含义 |
|---|---|
| `CREATED` | 准备创建这一批 fan-out |
| `EXPANDING` | 正在计算 rows、创建 child tasks、写 snapshot |
| `OPEN` | child tasks 已创建，正在等待执行结果 |
| `SUCCESS` | child tasks 按 trigger rule 成功完成 |
| `FAILED` | 有 child failed，且规则不允许继续 |
| `PARTIAL_SUCCESS` | 部分成功、部分 skipped/failed，但规则允许继续 |
| `CANCELLED` | 这批 fan-out 被取消 |
| `EMPTY` | source 返回空结果，且 empty policy 允许跳过 |

fanout group 的状态应该由 child tasks 聚合出来，例如：

```text
expected_count = 3
terminal_count = count(child.status in terminal_states)
success_count = count(child.status == SUCCESS)
failed_count = count(child.status in [FAILED, TIMEOUT])
skipped_count = count(child.status == SKIPPED)
```

如果 trigger rule 是 `all_success`：

```text
terminal_count == expected_count
AND failed_count == 0
AND success_count == expected_count
=> fanout_group SUCCESS
```

如果 trigger rule 是 `all_done`：

```text
terminal_count == expected_count
=> fanout_group SUCCESS 或 PARTIAL_SUCCESS
```

如果还有 child 在 `READY`、`RUNNING`、`RETRYING`、`WAITING_EVENT`，fanout group 仍然是：

```text
OPEN
```

### child task 的状态

每个 child task 是真正的业务任务，例如：

```text
export_US
export_UK
export_SG
```

它们走普通 task 状态机：

```text
CREATED -> READY -> RUNNING -> SUCCESS
                         |-> FAILED
                         |-> RETRYING
                         |-> TIMEOUT
                         |-> CANCELLED
                         |-> SKIPPED
```

child 从 `CREATED` 变成 `READY`，至少要满足：

```text
generator 已 SUCCESS
AND child 自己的 parents 满足
AND run 没有 paused/cancelled
AND resource quota 允许
AND scheduled_at / run_date 到期
```

如果 child 可重试失败：

```text
RUNNING -> FAILED -> RETRYING -> READY -> RUNNING
```

child 没有进入终态之前，fanout group 不能关闭。

### fan-in task 的状态

fan-in task 是下游汇合节点，例如：

```text
notify_downstream
```

它不能只看：

```text
parents succeeded
```

还要看：

```text
wait_for_fanout_groups 已经满足
```

所以它的 ready 判断应该是：

```text
CREATED -> READY only if:
  all parent tasks satisfy trigger rule
  AND all required fanout groups satisfy fan-in rule
  AND run not paused/cancelled
  AND resource quota available
```

例如：

```text
generate_exports -> SUCCESS
export_US        -> SUCCESS
export_UK        -> SUCCESS
export_SG        -> RUNNING
fanout_group     -> OPEN
notify_downstream -> CREATED / not ready
```

等到：

```text
export_SG        -> SUCCESS
fanout_group     -> SUCCESS
notify_downstream -> READY
```

### 一次完整正常流转

```text
finalise
  RUNNING -> SUCCESS

generate_exports
  CREATED -> READY -> RUNNING
  RUNNING 中:
    query export_manifest
    create fanout_group(exports)
    create export_US / export_UK / export_SG
    append child messages
  RUNNING -> SUCCESS

fanout_group(exports)
  CREATED -> EXPANDING -> OPEN

export_US
  CREATED -> READY -> RUNNING -> SUCCESS

export_UK
  CREATED -> READY -> RUNNING -> SUCCESS

export_SG
  CREATED -> READY -> RUNNING -> SUCCESS

fanout_group(exports)
  OPEN -> SUCCESS

notify_downstream
  CREATED -> READY -> RUNNING -> SUCCESS
```

如果 `export_UK` 失败且不可重试：

```text
export_UK
  RUNNING -> FAILED

fanout_group(exports)
  OPEN -> FAILED

notify_downstream
  保持 CREATED / not ready
  或按策略变成 SKIPPED / FAILED
```

具体怎么处理由 fan-in trigger rule 和失败传播策略决定。

### 失败、重试和恢复

generator 最容易出事故的边界是“child tasks 生成到一半时挂了”。所以 expansion 必须可恢复：

```text
begin expansion
  create fanout_group(status = EXPANDING)
  insert child task instances with stable child_order_id
  write expansion snapshot
commit
publish child messages
mark fanout_group OPEN
mark generator SUCCESS
```

恢复时按故障点处理：

| 故障点 | 恢复方式 |
|---|---|
| query 前失败 | generator 可以 retry |
| query 后、写 child 前失败 | 用同一规则重算，依赖 stable child key 去重 |
| child 写入一半失败 | 根据 fanout group + child_order_id 幂等补齐 |
| child 已写入、消息未发布 | 从 task instance 表重新 publish message |
| generator 已 SUCCESS、child 在跑 | 不重跑 generator，只恢复 child / fanout group |

队列不是 source of truth。真正的恢复依据应该是 task instance、fanout group 和 expansion snapshot。

## 用户怎么使用 runtime fan-out

用户不应该写 Python 去直接操作调度器内部结构。更合理的使用方式是：在 `PROCESS_TASKS` 中配置一个 generator command。

示例：

```json
{
  "workflow": "risk_export",
  "order_id": "generate_exports",
  "command": "ADB-MESSAGES-FROM-QUERY",
  "parents": ["finalise"],
  "parameters": {
    "query": "select region, entity_uuid, target_path from export_manifest where run_uuid = '${run_uuid}'",
    "child_command": "EXPORT-TO-STORAGE",
    "child_order_id_template": "export_${region}",
    "child_parameters": {
      "entity_uuid": "${entity_uuid}",
      "target_path": "${target_path}",
      "region": "${region}"
    },
    "fanout_group": "exports",
    "max_children": 100,
    "empty_result_policy": "skip",
    "dedupe_key_fields": ["region", "entity_uuid"]
  }
}
```

用户需要配置的不是“怎么调度”，而是：

| 配置 | 用户在表达什么 |
|---|---|
| `query` 或 `resolver` | 从哪里算出要扇出的任务列表 |
| `child_command` | 每个子任务要执行什么 command |
| `child_order_id_template` | 子任务节点 ID 怎么生成 |
| `child_parameters` | 查询结果如何映射成 command 参数 |
| `fanout_group` | 这批动态子任务属于哪个逻辑组 |
| `max_children` | 最大允许扇出数量 |
| `empty_result_policy` | 查询为空时跳过、失败还是继续 |
| `dedupe_key_fields` | 用哪些字段防止重复生成 |

这比让用户写代码安全：平台可以校验、限流、审计和恢复。

## generator command 的输入

一个 generator command 的输入应该包含三类信息。

第一类是运行上下文：

```json
{
  "run_uuid": "run-123",
  "batch_id": "batch-20260618",
  "workflow": "risk_export",
  "parent_order_id": "generate_exports",
  "project": "risk",
  "environment": "prod"
}
```

第二类是数据选择方式，可以是 SQL、catalogue criteria、文件 manifest 或 API。

SQL 模式：

```json
{
  "mode": "sql",
  "query": "select region, entity_uuid, target_path from export_manifest where run_uuid = '${run_uuid}'"
}
```

Catalogue resolution 模式：

```json
{
  "mode": "catalogue",
  "criteria": {
    "file_type": "risk_result",
    "reporting_date": "${reporting_date}",
    "latest_only": true
  }
}
```

Manifest 文件模式：

```json
{
  "mode": "manifest",
  "manifest_path": "${manifest_path}",
  "format": "jsonl"
}
```

第三类是 child message 映射规则：

```json
{
  "child_command": "EXPORT-TO-STORAGE",
  "child_topic": "emf.commands",
  "child_order_id_template": "export_${region}",
  "child_parameters": {
    "entity_uuid": "${entity_uuid}",
    "target_path": "${target_path}",
    "region": "${region}"
  },
  "parents": ["generate_exports"],
  "fanout_group": "exports"
}
```

这样 generator command 的职责就很清楚：它拿到上下文，计算 child rows，再把每一行转换成 child message。

## SQL 模式怎么跑

SQL 模式通常分成五步。

1. 渲染查询参数。

```sql
select region, entity_uuid, target_path
from export_manifest
where run_uuid = '${run_uuid}'
```

渲染后：

```sql
select region, entity_uuid, target_path
from export_manifest
where run_uuid = 'run-123'
```

2. 提交到 Databricks 或 SQL 执行环境。

generator command 可以通过 Databricks SQL、Notebook、Job 或内部 query service 执行。关键是它要记录 query id、run id 和输入参数，方便排障。

3. 校验结果 schema。

返回结果至少要包含 child message 所需字段，例如：

```json
[
  {
    "region": "US",
    "entity_uuid": "entity-us",
    "target_path": "abfss://exports@storage.dfs.core.windows.net/us/result.csv"
  },
  {
    "region": "UK",
    "entity_uuid": "entity-uk",
    "target_path": "abfss://exports@storage.dfs.core.windows.net/uk/result.csv"
  }
]
```

如果 `child_order_id_template` 用到了 `${region}`，结果里就必须有 `region`。如果 `child_parameters` 用到了 `${target_path}`，结果里就必须有 `target_path`。

4. 把每行转换成 child message。

```json
{
  "run_uuid": "run-123",
  "batch_id": "batch-20260618",
  "order_id": "export_US",
  "parents": ["generate_exports"],
  "command": "EXPORT-TO-STORAGE",
  "parameters": {
    "entity_uuid": "entity-us",
    "target_path": "abfss://exports@storage.dfs.core.windows.net/us/result.csv",
    "region": "US"
  },
  "fanout_group": "exports"
}
```

5. 持久化 expansion 结果，再入队。

不能只把 child message 放进内存或直接发队列。正确顺序是：

```text
insert fanout_group / child task instances
commit
publish child messages
```

这样 Worker 或 Scheduler 重启后，系统仍然知道生成过哪些 child。

## 不用 SQL 时怎么扇出

动态扇出不一定必须来自 SQL。它本质上是“把某个运行时结果映射成一批 child messages”。

常见来源包括：

| 来源 | 适合场景 |
|---|---|
| SQL 查询 | 根据结果表、manifest 表、业务维表生成任务 |
| catalogue resolution | 根据 `file_type + criteria` 找到一组数据实体 |
| 文件 manifest | 上游直接给出一批要处理的文件或对象 |
| API 查询 | 从外部系统拉取要处理的对象列表 |
| command 输出 | 上一步计算产生一个 output manifest |

无论来源是什么，最后都要收敛成统一的 child rows：

```json
[
  {
    "child_key": "US",
    "command": "EXPORT-TO-STORAGE",
    "parameters": {
      "region": "US",
      "entity_uuid": "entity-us"
    }
  }
]
```

平台不要让每种来源各自发明一套调度逻辑。它们都应该走同一个 expansion 持久化和 message appender。

## 平台可以预定义哪些 generator command

可以先预定义几类，避免用户直接写低层调度代码：

| command | 作用 |
|---|---|
| `ADB-MESSAGES-FROM-QUERY` | 执行 SQL，把每一行结果转成 child message |
| `MESSAGES-FROM-CATALOGUE` | 根据 catalogue criteria 解析实体，再生成 child message |
| `MESSAGES-FROM-MANIFEST` | 读取 manifest 文件，把每一条记录转成 child message |
| `MESSAGES-FROM-API` | 调外部 API 获取列表，再生成 child message |
| `SUBWORKFLOW-FROM-ROWS` | 每一行结果触发一个子 workflow |

这些 command 的输出格式应该尽量一致，方便审计、恢复和 fan-in。

## 扇出什么 command

扇出的 command 可以有两种来源。

第一种是配置里固定：

```json
{
  "child_command": "EXPORT-TO-STORAGE"
}
```

这种最安全，适合大部分场景。

第二种是由结果行决定：

```json
{
  "child_command_field": "next_command"
}
```

这更灵活，但风险更高，必须做 allowlist 校验。例如只允许结果行中的 `next_command` 属于：

```json
["EXPORT-TO-STORAGE", "ADB-SQL-EVAL", "ADB-FINALISE"]
```

否则一次查询结果就可能触发非预期 command。

## 扇出多少个

扇出数量来自查询结果或 manifest 结果，但平台必须设置上限：

```json
{
  "max_children": 1000,
  "on_exceed": "fail"
}
```

常见策略：

| 策略 | 说明 |
|---|---|
| `fail` | 超过上限直接失败，最安全 |
| `truncate` | 只生成前 N 个，一般不建议用于生产 |
| `page` | 分页生成，适合大规模但要有背压 |
| `split_subworkflow` | 按组拆成多个子 workflow |

如果没有上限，动态 DAG 很容易变成 fan-out 爆炸：一个 task 生成几万条 child messages，把 Worker、Service Bus、Databricks 和 catalogue 全部打爆。

## 扇出在哪里

扇出不应该发生在纯内存里，也不应该只发生在消息队列里。

推荐位置是：

```text
generator command
  -> 计算 child rows
  -> 写 fanout expansion / task instance 表
  -> 写 audit log
  -> 批量 append message
```

关键原则：

- 先持久化，再发布消息。
- child `order_id` 必须稳定可重算。
- 同一个 generator 重试时不能重复生成 child。
- fanout group 要能被下游 fan-in 查询。
- 生成结果要能在排障 UI 或日志里看到。

## 下游怎么等待动态子任务

如果下游节点只依赖 generator 本身，会出现问题：

```text
generate_exports 成功
wait_all_exports 立即 ready
export_US / export_UK 还没跑完
```

这里最容易混淆的一点是：

```text
generator task 完成 != 所有 child tasks 完成
```

`generate_exports` 完成，只代表“子任务已经生成完了”。`export_US`、`export_UK`、`export_SG` 可能还没有真正执行完。

所以 fan-in 不能只依赖 generator 的成功状态，还要依赖 fanout group 的完成状态。

可以把下游节点配置成：

```json
{
  "order_id": "notify_downstream",
  "parents": ["generate_exports"],
  "wait_for_fanout_groups": ["exports"],
  "command": "NOTIFY-DOWNSTREAM"
}
```

ready 条件变成：

```text
parents succeeded
AND fanout_group(exports).all_children_terminal
AND no child failed unless trigger_rule allows
```

如果某个 child failed，下游应该按 trigger rule 决定：

| trigger rule | 行为 |
|---|---|
| `all_success` | 任一 child 失败，下游不执行 |
| `all_done` | 所有 child 终态即可执行 |
| `none_failed` | 允许 skipped，不允许 failed |
| `one_success` | 至少一个成功即可执行 |

## 动态 DAG 的判断依据

动态 DAG 不是为了炫技，而是在这些场景里有价值：

| 场景 | 为什么需要动态 |
|---|---|
| 输入数据数量运行时才知道 | 例如根据 catalogue 找到本次要处理的实体 |
| 分区或 region 数量不固定 | 每个 region 生成一个 export task |
| 上游给 manifest | manifest 中每一行变成一个 task |
| 需要按数据质量结果分支 | 失败样本、异常分组、修复任务数量运行时决定 |
| 大任务需要拆片 | 根据输入规模拆成多个并行子任务 |

不需要动态 DAG 的场景也很常见：

- workflow 步骤固定。
- 输入表固定。
- 只是参数不同。
- 只是重跑某个 task。

这些场景用 `PROCESS_TASKS` 静态定义即可，不要为了“动态”增加复杂度。

## 生产治理边界

运行时扇出必须带治理能力，否则它会把系统从“灵活”推向“不可控”。

最少需要这些约束：

| 能力 | 目的 |
|---|---|
| `max_children` | 防止 fan-out 爆炸 |
| command allowlist | 防止查询结果触发任意 command |
| stable child key | 防止 generator 重试重复生成 |
| fanout group | 支持 fan-in、排障和恢复 |
| expansion snapshot | Scheduler/Worker 重启后能恢复 |
| child task audit | 知道每个 child 是由哪次 generator 生成 |
| resource limit | 防止 child 同时打爆 Databricks |
| dry-run validation | 运行前发现 query、schema、模板错误 |

## retry 和 timeout 与动态 DAG 的关系

retry、timeout monitor、zombie recovery 也可能生成新 message，但它们不是业务动态 DAG 的主线。

可以这样区分：

| 类型 | 目的 |
|---|---|
| runtime fan-out | 根据业务数据生成新的业务子任务 |
| retry message | 为失败 task 创建新的 attempt |
| timeout monitor | 检查长时间运行任务是否超时 |
| recovery message | 修复 lease 过期、worker 死亡、外部 job 状态不一致 |

它们的共同点是都复用 message 机制；区别在于语义不同。面试时不要把 retry 简单说成“动态 DAG”，否则容易让人觉得概念混乱。

## 常见风险

**1. 下游提前执行**

只依赖 generator 成功，不等待 fanout group 完成，会导致下游提前跑。

**2. fan-out 失控**

查询结果过大或 manifest 异常，可能生成过多 child messages。

**3. `order_id` 冲突**

多个 child 生成同一个 `order_id`，DAG 会无法区分节点。

**4. dangling parent**

child 的 parent 指向不存在的节点，任务永远无法 ready。

**5. 产生循环依赖**

动态生成时如果允许任意 parent，很可能生成环。

**6. retry 重复生成**

generator 重试时如果没有 dedupe key，会重复生成同一批 child。

**7. 可观测性变差**

如果没有 expansion snapshot，排障时只能看到一堆 child task，却不知道它们从哪里来。

**8. 外部副作用更难保证**

动态 child 可能执行 export、merge、finalise 等有副作用操作，必须有幂等 key。

## 面试里可以这样讲

EMF 的动态 DAG 我会分两层讲。第一层是基础动态编排：workflow 不写死在 Python 里，而是 `OE-RUN` 根据 `PROCESS_TASKS` 把表里的 task 定义展开成 command messages，再用 `order_id + parents` 形成 DAG。第二层是运行时 fan-out：某些 generator command 可以根据 SQL 查询、catalogue resolution、manifest 或上一步输出生成新的 child messages。为了避免失控，平台会要求 child `order_id` 稳定、fanout group 持久化、max children 限制、command allowlist、幂等 key 和 fan-in completion 判断。这样动态 DAG 不是随便改图，而是可验证、可恢复、可审计的运行时扩展。

## 关键结论

动态 DAG 的重点不是“能生成新任务”，而是生成的新任务仍然要纳入 EMF 的同一套状态机、依赖管理、幂等、可观测和恢复逻辑。

如果一个动态扩展只存在内存里，或者只发了消息但没有持久化 expansion，那么它在生产上就是不可恢复的。
