# 08 · 可观测性、重试与幂等

## 8.0 本章导读

前面几章已经分别讲过:

- Loader 要记录 `LOAD_FAILED` 和 GCS source snapshot。
- Scheduler 要能回答"为什么这个 Step 还没跑"。
- Command 要返回结构化 error。
- Sidecar 要维护 idempotency ledger。
- dbt 要上传 artifacts 并解析 `run_results.json`。

本章把这些横切能力统一起来:

1. 日志、指标、trace、artifact、audit log 各自放什么。
2. OTel + Grafana 的指标口径怎么设计,避免高基数炸掉。
3. Run / Step / Command 的错误语义怎么统一。
4. 重试策略如何从错误语义、幂等能力、deadline 三者共同决定。
5. 幂等键、去重表、Sidecar ledger、外部 job id 如何配合。

一句话:

> EMF 的稳定性不是靠"失败了再跑一次",而是靠可观测、可判定、可重放、可审计。

---

## 8.1 为什么可观测性是主链路,不是附属功能

EMF 执行的是长任务、多 Step、有外部副作用的数据流程。一次 Run 失败时,用户最关心的不是一段堆栈,而是:

- 哪个 Step 失败?
- 是 DAG 写错、参数错、权限错、配额错、SQL 错、vendor 挂,还是平台中断?
- 失败前有没有产生外部副作用?
- 能不能安全重试?
- 如果不能自动重试,该找谁处理?
- 当时跑的是哪份 DAG、哪版 metadata、哪版 command、哪版 dbt project?

这些问题靠普通日志回答不了。必须从一开始就让 Run / Step / Command 具备结构化状态和证据链。

---

## 8.2 五类证据:不要混成一种日志

EMF 至少有五类证据。

| 类型 | 用途 | 示例 |
|------|------|------|
| State Store | 当前状态和可恢复边界 | RunRecord / StepRunRecord |
| Log | 人类排障细节 | stack trace、vendor response 摘要 |
| Metric | 聚合趋势和告警 | 失败率、耗时、队列长度 |
| Trace | 单次 Run/Step 的调用链 | Loader -> Scheduler -> Command -> Sidecar |
| Artifact | 大结果和调试材料 | dbt run_results、compiled SQL、manifest |
| Audit | 安全与合规事实 | 谁上传、谁调用、用了哪个 credential |

不要用一种东西承担所有职责:

- 不要把大 artifact 塞进 State Store。
- 不要靠日志恢复状态。
- 不要把 run_id 当 metric label。
- 不要把 secret / token 打进 trace。
- 不要只靠 Cloud Logging 做业务审计。

---

## 8.3 统一 Trace 模型

推荐一条 Run 一个根 trace:

```text
Run trace
  ├── ingest span
  ├── load span
  ├── schedule span
  ├── step export_orders span
  │     └── command ExportToGcs span
  ├── step dbt_run span
  │     └── dbt subprocess span
  └── step salesforce_export span
        └── sidecar HTTP span
              └── vendor API span
```

### 8.3.1 Trace attributes

推荐低基数 attributes:

```text
emf.pipeline_name
emf.owner
emf.environment
emf.command
emf.command_kind
emf.status
emf.error_code
```

高基数字段只放 event/log,不要放 metric label:

```text
emf.run_id
emf.step_id
gcs.object
gcs.generation
pubsub.message_id
vendor_request_id
idempotency_key_hash
```

严格说 `run_id` 在 trace attribute 中可以有,但不要拿它做 metrics label。OTel SDK / backend 的具体配置要确保不会把所有 trace attributes 自动变成 metric labels。

### 8.3.2 线程池里的 trace 传递

第 4 章提到,`ThreadPoolExecutor.submit` 不会自动保证 OTel context 按预期传递。提交 Step 时要显式 copy context:

```python
ctx = contextvars.copy_context()
executor.submit(lambda: ctx.run(run_step_with_guards, run_id, step_id))
```

否则 trace 会断成:

```text
Scheduler span
Command span with new trace_id
```

Grafana 里就看不到一条完整 Run。

## 8.4 结构化日志

日志必须结构化,至少包含:

```json
{
  "timestamp": "2026-04-23T10:00:00Z",
  "level": "ERROR",
  "message": "step failed",
  "run_id": "run_123",
  "step_id": "export_orders",
  "pipeline_name": "daily_export",
  "owner": "growth-analytics",
  "environment": "prod",
  "command": "ExportToGcs",
  "error_code": "RATE_LIMITED",
  "retryable": true,
  "trace_id": "abc",
  "span_id": "def"
}
```

### 8.4.1 日志里不要放什么

不要放:

- Secret value。
- OAuth token。
- ServiceAccount key。
- 完整 vendor response。
- 大量 SQL 查询结果。
- 未脱敏 PII。

SQL 文本是否可以打日志要谨慎。dbt compiled SQL 可以作为 artifact 存到受控 GCS 路径,不要无脑打进普通应用日志。

## 8.5 Metric 命名和标签口径

统一前缀:

```text
emf.*
```

核心指标:

| 指标 | 类型 | 含义 |
|------|------|------|
| `emf.run.started_total` | counter | Run 启动数 |
| `emf.run.completed_total` | counter | Run 终态数 |
| `emf.run.duration_ms` | histogram | Run 总耗时 |
| `emf.step.started_total` | counter | Step 启动数 |
| `emf.step.completed_total` | counter | Step 终态数 |
| `emf.step.duration_ms` | histogram | Step 执行耗时 |
| `emf.step.retry_total` | counter | Step 重试次数 |
| `emf.scheduler.ready_steps` | gauge | ready 队列长度 |
| `emf.scheduler.running_steps` | gauge | running Step 数 |
| `emf.command.errors_total` | counter | command 标准错误 |

推荐 labels:

```text
pipeline_name
owner
environment
command
command_kind
resource_group
status
error_code
```

谨慎使用:

```text
dbt_project
target
vendor
caller_team
```

禁止作为 metric label:

```text
run_id
step_id
gcs_object
vendor_request_id
idempotency_key
full_error_message
sql_text
```

### 8.5.1 为什么 run_id 不能当 metric label

每次 Run 一个新 id,如果作为 label:

```text
emf.step.duration_ms{run_id="run_1"}
emf.step.duration_ms{run_id="run_2"}
...
```

时间序列数量会无限增长。Grafana/Prometheus/Mimir 这类系统会被高基数拖垮。

run_id 应该用于 trace/log 查询,不是指标聚合。

## 8.6 Run / Step / Command 错误分类

统一错误分类可以避免"所有失败都叫 failed"。

### 8.6.1 Loader 错误

| code | retryable | 归属 |
|------|-----------|------|
| `MISSING_METADATA` | false | DAG 发布者 |
| `INVALID_JSON` | false | DAG 作者 |
| `UNKNOWN_COMMAND` | false | DAG 作者 / 平台命令发布 |
| `INVALID_PARAMETERS` | false | DAG 作者 |
| `UNKNOWN_PARENT` | false | DAG 作者 |
| `DEPENDENCY_CYCLE` | false | DAG 作者 |
| `GCS_READ_TRANSIENT` | true | 平台/基础设施 |

### 8.6.2 Scheduler 错误

| code | retryable | 归属 |
|------|-----------|------|
| `STATE_CONFLICT` | true | 平台内部并发 |
| `LEASE_LOST` | true/recover | 平台 |
| `STEP_INTERRUPTED` | manual/recover | 平台 |
| `QUEUE_TIMEOUT` | false/infra | 平台容量 |

### 8.6.3 Command 错误

沿用第 5 章:

| code | retryable |
|------|-----------|
| `INVALID_PARAMETERS` | false |
| `AUTH_FAILED` | false |
| `PERMISSION_DENIED` | false |
| `NOT_FOUND` | command-specific |
| `RATE_LIMITED` | true |
| `TRANSIENT` | true |
| `UPSTREAM_5XX` | true |
| `SIDE_EFFECT_UNKNOWN` | false/manual |
| `OUTPUT_SCHEMA_INVALID` | false |

### 8.6.4 dbt 错误

| code | retryable |
|------|-----------|
| `DBT_COMPILE_ERROR` | false |
| `DBT_MODEL_FAILED` | depends |
| `DBT_TEST_FAILED` | false |
| `DBT_PROFILE_ERROR` | false |
| `DBT_INFRA_TRANSIENT` | true |
| `DBT_TIMEOUT` | depends |

### 8.6.5 Sidecar 错误

| code | retryable |
|------|-----------|
| `RATE_LIMITED` | true |
| `UPSTREAM_5XX` | true |
| `AUTH_FAILED` | false |
| `IDEMPOTENCY_CONFLICT` | false |
| `SIDE_EFFECT_UNKNOWN` | false/manual |

## 8.7 重试决策:不是 error.retryable 一项决定

最终能不能重试,至少看四件事:

```text
error.retryable
command.idempotency
remaining_deadline
attempt_budget
```

伪代码:

```python
def should_retry(step, command_spec, error, now):
    if not error.retryable:
        return False
    if step.attempt >= command_spec.retry_policy.max_attempts:
        return False
    if now >= step.deadline:
        return False
    if command_spec.idempotency.mode == "not_idempotent":
        return False
    if error.code == "SIDE_EFFECT_UNKNOWN":
        return False
    return True
```

### 8.7.1 retryable 不等于应该马上重试

`RATE_LIMITED` 是 retryable,但应该按 `retry_after` 或指数退避:

```json
{
  "code": "RATE_LIMITED",
  "retryable": true,
  "retry_after_seconds": 120
}
```

Scheduler 应该进入:

```text
RETRY_WAITING until now + 120s
```

不是立即再次打爆下游。

### 8.7.2 deadline 比 max_attempts 更重要

如果 Step deadline 还剩 10 秒,而下一次退避要 60 秒,就不要排重试。直接失败或超时更诚实。

## 8.8 幂等分层

EMF 有多层幂等,不能混在一起。

| 层 | 幂等 key | 解决什么 |
|----|----------|----------|
| Ingestor | `bucket/object/generation` | Pub/Sub 重复投递 |
| Run | `source_key` | 同一份 DAG generation 不重复创建 Run |
| Step claim | `run_id/step_id/status_version` | 多线程/多 Pod 不重复执行 Step |
| Command | `run_id:step_id` | command 重试不重复副作用 |
| Sidecar | `idempotency_key + request_hash` | vendor 外部副作用去重 |
| External system | job id / request id | BigQuery/vendor 恢复查询 |

### 8.8.1 Ingestor 幂等

```text
source_key = sha256(bucket + object + generation)
```

确保 Pub/Sub 重复投递不会创建多个 Run。

### 8.8.2 Step claim 幂等

State Store 原子 claim:

```text
WAITING -> RUNNING
where version = old_version
```

确保两个 Scheduler 不会同时跑同一个 Step。

### 8.8.3 Command 幂等

推荐:

```text
idempotency_key = run_id + ":" + step_id
```

大多数写操作不要包含 attempt。

### 8.8.4 Sidecar ledger 幂等

Sidecar 保存:

```json
{
  "idempotency_key": "run_123:export_campaigns",
  "request_hash": "sha256(parameters)",
  "status": "SUCCEEDED",
  "output": {...}
}
```

相同 key + 不同 request hash 是严重错误:

```text
IDEMPOTENCY_CONFLICT
```

## 8.9 `SIDE_EFFECT_UNKNOWN` 的处理

这是最危险的一类状态。

含义:

> 请求可能已经产生外部副作用,但平台无法确认。

例子:

- Sidecar 调 vendor 创建 ticket 超时。
- dbt subprocess 被 kill,但 BigQuery job 还在跑。
- GCS 写入请求超时,但对象可能已经写成功。

处理原则:

1. 不自动重试。
2. 尝试外部状态查询。
3. 如果查询成功,恢复为 `SUCCEEDED` 或 `FAILED`。
4. 如果查询不确定,标记 `requires_manual_check`。
5. 告警要带上 idempotency key、vendor request id、job id、destination。

状态:

```text
RUNNING -> SIDE_EFFECT_UNKNOWN -> RECOVERING / NEEDS_MANUAL_CHECK
```

如果当前状态机不想增加太多状态,至少在 Step error 中保留:

```json
{
  "code": "SIDE_EFFECT_UNKNOWN",
  "requires_manual_check": true,
  "recovery_hint": {
    "vendor_request_id": "sf_req_abc",
    "idempotency_key": "run_123:create_ticket"
  }
}
```

## 8.10 Artifacts:大证据放 GCS

以下内容应该进 GCS result path:

- 原始 DAG snapshot。
- Loader validation report。
- dbt `manifest.json`、`run_results.json`、`dbt.log`、compiled SQL。
- Sidecar 大型 response 摘要或导出文件。
- Run summary manifest。
- 失败诊断包。

建议结构:

```text
gs://team-a-emf-results/{pipeline_name}/{run_id}/
  pipeline.snapshot.json
  metadata.snapshot.json
  load_report.json
  steps/
    export_orders/
      output.json
      error.json
    dbt_run/
      dbt/
        manifest.json
        run_results.json
        dbt.log
        compiled.zip
  run_summary.json
```

State Store 保存 URI:

```json
{
  "artifact_uri": "gs://team-a-emf-results/daily_export/run_123/steps/dbt_run/dbt/"
}
```

不要把 artifact 内容放 State Store。

## 8.11 Audit:owner 不是 actor

第 3 章已经强调:

- GCS object metadata 里的 `owner` 是业务归属。
- 真正上传者要看 GCS Audit Logs。

审计记录要区分:

| 字段 | 含义 |
|------|------|
| `owner` | 业务负责团队 |
| `uploader_principal` | 谁上传/修改了 DAG object |
| `caller_service_account` | 哪个 Control Plane 调用 command |
| `credential_scope` | Sidecar 用了哪类 vendor 凭证 |
| `command` | 执行了什么动作 |
| `idempotency_key_hash` | 外部副作用去重线索 |

不要把 `owner=growth-analytics` 当成安全身份。真正的安全边界来自 IAM、ServiceAccount、Sidecar policy。

## 8.12 告警策略

不同错误找不同人。

| 情况 | 通知对象 |
|------|----------|
| `LOAD_FAILED` / invalid DAG | DAG owner |
| metadata 缺失 | DAG 发布者 / CI owner |
| command 参数错误 | DAG owner |
| dbt test failed | 数据 owner |
| Sidecar vendor 429 | 平台 + 相关 owner |
| Sidecar auth failed | 平台 / credential owner |
| Scheduler lease lost / interrupted | 平台维护者 |
| 高失败率 / queue 积压 | 平台维护者 |

告警内容要包含:

- pipeline_name。
- run_id。
- failed step id。
- error code。
- retryable。
- requires_manual_check。
- artifact URI。
- trace link。

不要只发:

```text
Run failed.
```

这等于没发。

## 8.13 Grafana Dashboard 建议

至少四张面板。

### 8.13.1 Platform Overview

- Run success / failure count。
- Run duration p50/p95/p99。
- active runs。
- ready/running steps。
- queue wait p95。
- failure by error_code。

### 8.13.2 Loader Health

- ingest events。
- duplicate events。
- load success/failure。
- metadata missing。
- validation failed by code。
- load duration。

### 8.13.3 Command Health

- step duration by command。
- retry count by command。
- error by command/error_code。
- resource_group saturation。
- side_effect_unknown count。

### 8.13.4 Sidecar / Vendor Health

- requests by vendor/command。
- vendor latency。
- rate limit wait。
- 429/5xx。
- idempotency replay。
- credential auth failure。

dbt 可以单独一张:

- models succeeded/failed。
- tests failed。
- dbt duration。
- top failing project。

## 8.14 AI 运行期智能的输入

第 11 章会讲 Runtime Intelligence。这里先定输入。

AI 要做根因归因,需要结构化证据:

- RunRecord。
- StepRunRecord。
- error code。
- parameters snapshot。
- command spec hash。
- trace span summary。
- artifact URI。
- dbt run_results summary。
- Sidecar ledger summary。
- recent similar failures。

如果这些都没有,AI 只能读日志猜测。

所以第 8 章的工作不是"为了画漂亮 dashboard",也是为了让后续 AI 自愈和诊断有可靠输入。

## 8.15 常见反模式

### 8.15.1 所有异常都重试

参数错、SQL 编译错、权限错、schema 不匹配,重试一百次也不会好。无脑重试会放大成本和下游压力。

### 8.15.2 没有幂等就重试写操作

这是重复发券、重复建单、重复通知的根源。

### 8.15.3 把 run_id 放 metric label

会制造高基数灾难。

### 8.15.4 只存 stdout,不存 artifact

dbt / Sidecar / Loader 的排障证据会丢失。

### 8.15.5 owner 当 actor

owner 是业务归属,不是上传者或调用身份。审计必须看 IAM / ServiceAccount / GCS audit log。

## 8.16 本章关键结论

1. **EMF 的稳定性来自结构化状态和证据链,不是失败后盲目重跑**。
2. **State Store、Log、Metric、Trace、Artifact、Audit 各有职责**,不要混用。
3. **Trace 要跨线程池传递**,否则 Scheduler 到 Command 的调用链会断。
4. **Metric label 必须低基数**,run_id、step_id、GCS path、vendor_request_id 不应作为 label。
5. **错误码要分层统一**:Loader、Scheduler、Command、Sidecar、dbt 都要归一到结构化 error。
6. **重试决策至少看 error.retryable、idempotency、deadline、attempt_budget**。
7. **幂等是分层的**:Ingestor 去重、Step claim、Command idempotency、Sidecar ledger、外部 job id 各管一段。
8. **`SIDE_EFFECT_UNKNOWN` 不能自动重试**,必须查询外部状态或人工介入。
9. **大证据进 GCS artifact,State Store 只保存 URI 和摘要**。
10. **这些治理数据也是后续 AI Runtime Intelligence 的输入底座**。
