# 09 参数替换系统

## 参数解析入口

它处在命令执行之前:

```text
Message
  -> StratusCommandExecutor
  -> module.get_parameters()
  -> MessageParameterParser
  -> execute_with_parameters()
```

也就是说,command module 最终拿到的不是原始模板参数,而是已经解析和替换后的参数。

## 参数来源

参数主要来自五类地方:

1. message 自身属性。
2. run context 参数。
3. SQL/message template 参数。
4. 外部参数。
5. `$` 风格的参数引用。

## message 自身属性

常见的 message 属性包括:

```text
msg_id
order_id
workflow
run_uuid
batch_id
```

这类参数让命令可以在执行时知道自己处在哪个 run、哪个 batch、哪个业务节点里。

## run context 参数

run context 参数用于跨同一次 workflow run 共享上下文。

典型用途:

- run 级别配置。
- 外部触发参数。
- workflow 展开时生成的公共参数。
- 子任务继承的上下文。

## SQL / message template 参数

SQL 类命令或 message template 类命令经常需要模板替换。

例如:

```text
select * from ${source_table}
where ds = ${run_date}
```

参数替换系统负责在命令真正执行前把这些引用变成具体值。

## 外部参数

外部参数通常来自:

- `OE-RUN` 启动参数。
- run 创建参数。
- 上游系统传入的参数。
- 手工触发时输入的参数。

## 参数系统的意义

如果没有参数替换系统,每个 command 都要自己处理:

- 从 message 读字段。
- 从 run context 读参数。
- 替换模板。
- 合并外部参数。
- 校验缺失参数。

EMF 把这些收束到 `MessageParameterParser` 和 module 参数元数据里,让命令实现更专注于业务动作。

## 参数系统的设计取舍

参数系统解决的是“同一个 workflow 模板如何在不同 run 中变成不同的具体执行输入”。

如果没有统一参数系统,每个 command 都会自己写一套:

```text
读 message 字段
读 run context
读外部参数
替换 SQL 模板
处理缺省值
校验必填参数
```

这会导致参数语义分散在各个 command 里,很难治理。

统一参数系统的收益:

- command 实现更专注业务。
- 参数引用方式一致。
- workflow task 可以模板化。
- run 级参数可以统一透传给子任务。

代价:

- 参数解析规则需要非常清楚,否则会变成隐式魔法。
- 参数优先级要稳定,否则同名参数来源冲突时很难排查。
- SQL / message template 的替换要注意安全边界。

## 面试怎么讲

可以这样说:

```text
EMF 的参数系统把 message 自身字段、run context、外部输入和模板变量统一解析成 command 的最终入参。
这样 workflow 定义可以保持模板化,command 不需要自己理解各种参数来源。
```

## 阅读建议

读参数系统时,不要只看 parser 本身。建议按这条线:

```text
module 声明需要哪些参数
  -> executor 发起解析
  -> parser 从 message/run/template/外部输入取值
  -> module 拿到最终参数并执行
```

这样才能看到“参数声明、参数解析、参数消费”三者如何闭环。
