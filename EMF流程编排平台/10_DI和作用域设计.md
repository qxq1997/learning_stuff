# 10 DI 和作用域设计

## 为什么 DI 重要

EMF 的 worker 不是直接 new 一堆对象跑起来,而是通过 DI 容器注册和解析核心组件。

这让系统可以把:

- worker 生命周期组件。
- 消息运行时组件。
- command module。
- Azure / Storage / Catalogue 等 service。

统一放进可管理的依赖图里。

## 容器注册分层

DI 注册可以按三类理解:

| 注册类型 | 负责什么 |
|---|---|
| 生产环境注册 | 接入真实外部依赖,例如消息系统、存储、业务 service |
| worker 核心注册 | 注册 producer、consumer、handler、executor、provider |
| 消息运行时注册 | 注册 DAG manager、message appender、execution context |

## 核心 worker 组件注册

主要映射:

| 接口 | 实现 |
|---|---|
| `IWorker` | `Worker` |
| `IMessageHandler` | `MessageHandler` |
| `ICommandExecutor` | `StratusCommandExecutor` |
| `IModuleProvider` | `ModuleProvider` |
| `IParallelMessageConsumer` | `ParallelMessageConsumer` |
| `IMessageParameterParser` | `MessageParameterParser` |

这组注册基本覆盖了:

```text
消费 -> 调度 -> 处理 -> 找命令 -> 解析参数 -> 执行
```

## 消息运行时组件注册

主要包括:

| 接口 | 职责 |
|---|---|
| `IAppExecutionContextManager` | 管理应用执行上下文 |
| `IDagManager` | 管理运行时 DAG |
| `IMessageAppender` | 将消息追加到运行时上下文 |

这些组件支撑的是“消息进入内存 DAG 并被后续调度”。

## 每条 message 有自己的 scope

团队约定明确每个 message 在自己的 DI scope 里执行。

这点很关键。每条 message 独立 scope 可以降低这些风险:

- 上一条消息的状态泄漏到下一条。
- scoped service 被并发消息共享。
- command 执行中的上下文互相污染。
- 日志、审计、trace 上下文混乱。

## 作用域分层

可以把 DI scope 分成三层理解:

```text
Process scope
  -> worker 级组件,如 IWorker / IParallelMessageConsumer / IDagManager

Run / app execution context
  -> batch/run/DAG 相关上下文

Message scope
  -> 单条消息执行时的 handler / command / service 依赖
```

不是所有组件都应该是 singleton。尤其是携带 message 上下文、参数上下文、执行上下文的对象,更适合放到 message scope 里。

## 为什么这对工作流引擎重要

工作流引擎和普通脚本最大的不同之一是:它会长期运行、并发执行、不断处理不同 run 的消息。

如果没有清晰作用域,很容易出现:

- run A 的上下文污染 run B。
- 一个 command 的临时状态泄漏给另一个 command。
- 并发消息共享了本不该共享的 service 实例。
- trace、audit、logger 绑定错消息。

所以 DI 不是“工程洁癖”,而是并发工作流系统里的隔离机制。

## 面试怎么讲

可以这样讲:

```text
EMF 用 DI 管理 worker 级组件和 message 级组件。worker 级组件长期存在,message scope 则在单条消息执行时创建,用来隔离上下文、参数、日志和业务 service。
```

## 阅读建议

读 DI 相关代码时,按下面顺序比较清楚:

```text
先看 worker 启动时注册了哪些核心接口
再看消息进入 DAG 需要哪些运行时组件
最后看单条 message 执行时为什么要开 scope
```

这样可以从“进程怎么组装”一路看到“单条消息怎么隔离执行”。
