# 12 最重要的心智模型

## 把 EMF 记成一句话

```text
OE-RUN 或外部事件产生 Message;
Message 进入内存 DAG;
ParallelMessageConsumer 只执行依赖满足的节点;
StratusCommandExecutor 根据 command 找 module;
module 调用 command/service 做业务;
业务命令可以继续 enqueue 新消息,从而扩展 DAG.
```

这是理解整个项目的主线。

最短公式是:

```text
Message = 节点 + 命令 + 参数 + 依赖
run_uuid = 一张 DAG
batch_id = 一组 run 的生命周期容器
parents = join barrier
command = 节点上的动作
```

## 三个核心问题

看任何 EMF 机制时,都可以问这几个问题。

### 1. 这个机制在处理 Message 的哪个阶段?

可能是:

- 创建 message。
- 入队 message。
- 建 DAG 边。
- 判断 ready。
- 执行 message。
- 生成新 message。
- 更新 message 状态。

### 2. 这个机制属于调度层还是命令层?

调度层关心:

```text
run_uuid / batch_id / order_id / parents / state / priority
```

命令层关心:

```text
command / module / parameters / service / execution result
```

不要把两层混在一起读。

也可以说:

```text
调度层决定“什么时候、哪些节点可以跑”;
命令层决定“这个节点具体怎么做业务”。
```

### 3. 这个机制会不会扩展 DAG?

有些 command 只做业务副作用。有些 command 会继续 enqueue 新消息。

如果一段代码会创建或追加 `Message`,那它不仅是业务逻辑,也是运行时 DAG 的生产者。

### 4. 这个机制在维护哪种关系?

EMF 里最核心的关系有四种:

| 关系 | 字段 |
|---|---|
| 一批 run | `batch_id` |
| 一张 DAG | `run_uuid` |
| 一个业务节点 | `order_id` |
| 节点依赖 | `parents` |

很多看起来复杂的代码,最后都是在维护这四个关系之一。

## 面试里的核心表达

你可以把 EMF 讲成一个闭环:

```text
Message 描述任务
  -> DAG 解释依赖
  -> Consumer 推进状态
  -> Command 执行业务
  -> Command 还能生成 Message
```

这个闭环最重要的点是“业务执行和调度扩展没有分裂”。命令要扩展 workflow,不是去操作调度器内部结构,而是生成新的 `Message`。这让动态 DAG、子 workflow、retry 都能复用同一套机制。

## 最容易误解的点

### 误解一: EMF 的 DAG 是由 `msg_id` 连起来的

不是。依赖是:

```text
parents -> parent.order_id
```

`msg_id` 是消息实例 ID,不是 DAG 边的主要匹配 key。

### 误解二: OE-RUN 是普通业务任务

不是。`OE-RUN` 是工作流展开器。它的核心输出是一批 workflow messages。

### 误解三: command 就是一个函数名

不是。`command` 是命令协议名,通过 `ModuleProvider` 和 module 注册映射到实现。

### 误解四: EMF 只是消费 PubSub

不是。PubSub 只是入口之一。真正的执行发生在内存 DAG 和命令系统之间。

### 误解五: DAG 是静态的

不是。命令运行时可以继续 enqueue 新消息,让 DAG 动态增长。

### 误解六: parents 是顺序

不是。`parents` 不是“上一条消息”,而是 join barrier。一个节点可以有多个 parent,只有这些 parent 都满足状态条件时,当前节点才 ready。

### 误解七: batch_id 和 run_uuid 是一回事

不是。`batch_id` 是一批 run 的外层分组,`run_uuid` 是单张 DAG 的分组。每个 `run_uuid` 会对应一个 `DagContext`。

### 误解八: 子 workflow 是函数调用

不是。子 workflow 是父 workflow 中的 `OE-RUN` 节点展开出一批新的 messages。它的难点是如何把子流程入口、终点、completion token 和父流程后续节点接回同一套 DAG 语义。

### 误解九: retry 是原地重跑

不是。retry 更像生成一条带未来 `run_date` 的 retry message,让下一次尝试重新回到 ready 判断和调度系统。

## 学会后的判断标准

如果你能清楚解释下面这条概念路径,就基本抓住了 EMF:

```text
外部事件 / OE-RUN
  -> Message
  -> DagContext
  -> ready vertices
  -> Parallel consumer
  -> Message handler
  -> Command executor
  -> Module / Service
  -> 新 Message 或业务副作用
```

如果还能解释 `OE-RUN` 如何从 `process_tasks` 生成 messages,以及 `ADB-MESSAGES-FROM-QUERY` 如何运行时继续生成 messages,就已经不只是会看代码,而是理解了 EMF 的设计。
