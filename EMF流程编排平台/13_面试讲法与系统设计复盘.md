# 13 面试讲法与系统设计复盘

## 本章定位

第 34 章是 EMF 的 source of truth。本章只解决一个问题：**面试或答辩时，怎么把 EMF 讲清楚。**

不要一上来就说“我做了一个 DAG engine”。更稳的讲法是先按第 34 章建立项目定位：

```text
EMF 是一个基于 Azure 和 Databricks 的企业级数据编排平台。
它用消息驱动和元数据驱动的方式，把文件接入、workflow 展开、DAG 调度、Databricks 计算、结果 finalise、catalogue 登记和下游解析串起来。
```

然后再展开系统设计。

## 13.1 一分钟版本

如果面试官问“你做的 EMF 是什么”，可以这样讲：

```text
EMF 是一个元数据驱动的数据编排平台。
上游文件上传后，Worker 会先读取 blob metadata 里的 file_type，再用 file_type 查询 LOAD_INFO，确定 schema、目标 dataset 和 ingestion workflow。
之后系统通过 OE-RUN 读取 PROCESS_TASKS，把 workflow 定义展开成一批 command messages。
这些 messages 通过 run_uuid、order_id 和 parents 形成一张运行时 DAG。
Worker 只执行依赖满足的节点，具体计算交给 Databricks，最终结果写入 DATAHUB，并把 entity_uuid、file_type、dataset、table、run 信息登记到 CATALOGUE.METADATA。
后续任务可以再通过 file_type + criteria 从 catalogue 解析输入数据。
```

这一版要突出四点：

1. `file_type` 是入口路由。
2. `LOAD_INFO` 和 `PROCESS_TASKS` 把业务行为从代码移到配置和元数据。
3. `Message` 是运行时任务单元，`parents` 表达 DAG 依赖。
4. Worker 做控制面，Databricks 做计算面，Lakehouse/Delta 承载最终表资产。

## 13.2 三分钟版本

三分钟可以按分层讲：

```text
入口层: Azure Storage / Service Bus 触发 Worker，消息或文件 metadata 里带 file_type、run_uuid 等信息。
元数据层: Worker 根据 file_type 查询 LOAD_INFO，解析 schema、dataset、table prefix、write mode 和 ingestion workflow。
编排层: OE-RUN 根据 workflow 读取 PROCESS_TASKS，把每行 task 展开成 command message，并通过 order_id + parents 建 DAG。
调度层: DagContext / Worker 消费 ready messages，控制并发、重试、超时、取消和状态推进。
命令层: CommandExecutor 根据 command 找 module，执行 resolve、load、SQL/dbt、finalise、catalogue、export 等动作。
计算层: Databricks 承担 SQL、Spark、Notebook、dbt、Python/UDF 等实际计算。
数据层: 结果进入 DATAHUB 或目标 dataset，metadata 写入 CATALOGUE.METADATA，支持后续 resolution 和审计。
治理层: 通过幂等、lease、audit、Jira/SOP/LLM 摘要、权限探测和资源限流保证生产可运行。
```

这个讲法比“我做了一个 workflow engine”更准确，因为它把 EMF 的数据平台属性、元数据驱动和 Databricks/Lakehouse 选型都讲出来了。

## 13.3 最值得讲的设计选择

### 选择一：元数据驱动，而不是把流程写死在代码里

普通脚本可能在代码里写死：

```text
读哪个文件
用哪个 schema
落到哪张表
跑哪个 workflow
下游读哪个输入
```

EMF 把这些决策放到：

```text
blob metadata
LOAD_INFO
PROCESS_TASKS
CATALOGUE.METADATA
criteria
```

收益：

- 新增 file type 时，很多情况下新增配置和 metadata，而不是复制一套 pipeline。
- workflow 步骤由 `PROCESS_TASKS` 定义，便于版本化、校验和审计。
- 下游任务通过 catalogue resolution 找输入，减少写死表名。

代价：

- metadata 本身必须治理。
- `file_type`、schema、dataset、workflow、criteria 不一致会变成生产故障。
- 发布前需要 dry-run 和 validation。

### 选择二：用 Message 表达运行时 DAG 节点

EMF 的 task 在运行时表现为 message：

```text
run_uuid + order_id + parents + command + parameters
```

这样一批 message 就可以推导出 DAG：

```text
run_uuid 是图
order_id 是节点业务 ID
parents 是边
command 是节点动作
```

收益：

- `OE-RUN` 展开的 task、外部事件、retry、runtime fan-out 都可以统一成 message。
- 调度器只需要理解 message 和状态，不需要理解每种业务 workflow。
- 动态扩展自然，generator command 生成新 message 后仍然回到同一套调度模型。

代价：

- `order_id` 和 `parents` 质量非常关键。
- 动态 fan-out 要防重复、防爆炸、防下游提前执行。
- 排障时要能还原“实际生成了哪些 message”。

### 选择三：Worker 做控制面，Databricks 做计算面

Worker 负责：

```text
消息消费
参数解析
DAG ready 判断
状态推进
重试、取消、超时
外部 run id 绑定
日志、审计、ticket 上下文
```

Databricks 负责：

```text
Spark / SQL / Notebook / dbt / Python / UDF 计算
Delta 表写入
大规模 join / 聚合 / finalise
Lakehouse 表治理
```

收益：

- Worker 不需要自己实现分布式计算。
- Databricks job id / run id 可以作为外部执行证据。
- EMF 可以把精力放在编排、元数据、可靠性和治理上。

代价：

- Worker 内部状态和 Databricks 外部状态必须对账。
- job submit、polling、cancel、timeout、失败映射都要设计。
- 资源和成本要限流，不能盲目提交 job。

### 选择四：用 Lakehouse/Delta 承载最终数据资产

EMF 不是把文件简单搬到一张表里，而是让数据经历：

```text
Raw / Landing
  -> Staging / Bronze
  -> Refined / Silver
  -> Curated / Gold
  -> CATALOGUE.METADATA
```

Delta/Lakehouse 的价值是：

- finalise 可以成为有事务语义的表提交。
- schema enforcement 可以阻断错误数据。
- Delta version 支持审计、回滚和复盘。
- catalogue 登记的是可查询、可追踪、有版本的数据实体，而不是散落文件路径。

## 13.4 面试高频问题

### Q1：为什么不用 Airflow？

不要说“Airflow 太重”。可以这样回答：

```text
Airflow 很适合固定 DAG 的数据 pipeline 调度。
EMF 的核心是企业内部数据编排平台，它的任务不是只按 schedule 跑一个 Python DAG，而是根据 file_type、LOAD_INFO、PROCESS_TASKS 和 catalogue metadata 在运行时决定怎么加载、怎么计算、怎么 finalise、怎么解析下游输入。
```

可以补：

```text
EMF 的 Worker 已经和 Azure Storage、Service Bus、Databricks、Unity Catalog、LOAD_INFO、METADATA、PROCESS_TASKS 深度集成。
如果直接用 Airflow，仍然要自己实现这些 metadata resolution、command protocol、catalogue 登记、Databricks 状态对账和生产恢复逻辑。
```

一句话：

```text
Airflow 是通用 DAG 调度器，EMF 是围绕企业数据元数据、Databricks 和 Lakehouse 治理定制的数据编排平台。
```

### Q2：DAG 依赖怎么表达？

```text
每个 message 有 order_id 和 parents。
parents 保存父任务的 order_id。
同一个 run_uuid 下，系统根据 order_id + parents 建立 parent/child 关系。
只有所有父节点满足 trigger rule，当前节点才会 ready。
```

注意补一句：

```text
msg_id 是消息实例 ID，不适合表达业务依赖；order_id 才是 DAG 里的业务节点 ID。
```

### Q3：怎么支持动态 DAG？

先分层：

```text
第一层是基础动态编排：OE-RUN 根据 PROCESS_TASKS 把 workflow definition 展开成 message DAG。
第二层是运行时 fan-out：某个 generator command 根据 SQL、catalogue resolution、manifest 或上一步输出继续生成 child messages。
```

用户怎么用：

```text
用户在 PROCESS_TASKS 里配置 generator task。
配置里声明 source、child_command、child_order_id_template、child_parameters、fanout_group、max_children 和 dedupe key。
generator 执行后，把查询结果或 manifest 每一行转换成 child message。
child message 仍然写回同一个 run_uuid 下的 task/message 表，再由调度器按 parents 和 fanout_group completion 调度。
```

要强调：

```text
动态 DAG 不是 command 私下开线程，也不是绕过调度器。
它是把运行时结果转换成新的 message，再交回同一套 DAG 状态机。
```

详细设计见第 08 章。

### Q4：为什么 retry 要生成新 message？

```text
因为 EMF 希望 retry 也回到统一调度模型。
失败后生成新的 retry message 或新 attempt，保留 run_uuid、order_id、parents、command 和参数，同时更新 retry count 和 run_date。
这样 retry delay 不占用线程，也能被状态机、审计、幂等和并发限制治理。
```

### Q5：怎么保证并发不破坏依赖？

```text
并发只发生在 ready messages 之间。
Worker / DagContext 先判断父依赖、状态、run_date、label 和资源限制。
只有 ready 的节点才会进入执行池。
所以线程池空闲不代表可以越过 DAG 依赖执行。
```

### Q6：`batch_id` 和 `run_uuid` 为什么都需要？

```text
run_uuid 是一张具体运行时 DAG 的边界。
batch_id 是一批 run 的治理边界。
```

类比：

```text
batch_id 像文件夹；
run_uuid 像文件夹里的一张 DAG 图；
message 是图里的一个节点。
```

### Q7：Databricks 的价值是什么？

不要说“没有 Databricks 就做不了”。更准确：

```text
Databricks 的价值不是功能唯一，而是降低平台复杂度。
EMF 本身已经要负责 metadata、workflow、message、retry、状态和排障。
如果再自研 Spark 运行、Delta 表事务、Python/SQL/dbt 混合执行和数据治理，平台复杂度会很高。
Databricks 把 Spark 计算、Delta Lake、Unity Catalog、Job API、Notebook/dbt/Python/SQL 混合 workload 收敛到一个平台边界里。
```

一句话：

```text
Worker 做控制面，Databricks 做计算面，这样 EMF 可以专注元数据编排和可靠调度。
```

### Q8：生产中最大的难点是什么？

可以这样答：

```text
最大的难点不是拓扑排序，而是生产环境下状态、元数据和外部系统的一致性。
```

展开五类：

1. metadata 一致性：`LOAD_INFO`、`PROCESS_TASKS`、`METADATA` 和真实表必须一致。
2. DAG 状态：父子依赖、retry、cancel、fan-out、fan-in 都要受控。
3. 外部执行：Databricks job 可能成功、失败、排队、超时或状态未同步。
4. 幂等恢复：Worker 重启、消息重复、API 超时都不能导致重复副作用。
5. 可观测性：失败 ticket 要能串起 workflow、task、metadata、Databricks run、SOP 和 owner team。

### Q9：AI 在 EMF 里起什么作用？

AI 不应该参与核心调度判定。核心状态推进、依赖判断、幂等、lease、retry 都必须是确定性逻辑。

更合适的作用是运维增强：

```text
失败后自动汇总 workflow 信息、失败 task、错误堆栈、Databricks run 链接、metadata、最近配置变更和 SOP。
LLM 根据这些上下文生成故障摘要、可能原因、处理建议，并根据 file_type/workflow/dataset/owner metadata 路由到正确 Jira board。
```

还可以作为新 command：

```text
PII masking
日志摘要
schema drift 解释
数据质量异常说明
SOP 匹配
```

但这些 AI command 必须有输入输出契约、审计、权限和人工兜底。

## 13.5 可以画的白板图

```text
Azure Storage / Service Bus
        |
        v
  file_type / Message
        |
        v
     LOAD_INFO
        |
        v
       OE-RUN
        |
        v
   PROCESS_TASKS
        |
        v
 command messages
        |
        v
   Worker / DAG state
        |
        v
    Databricks
        |
        v
 DATAHUB / Delta tables
        |
        v
 CATALOGUE.METADATA
        |
        v
 downstream resolution
```

讲图的时候补一句：

```text
这张图的核心是 metadata 决定业务行为，Worker 负责控制面，Databricks 负责计算面，catalogue 让结果可被后续 workflow 解析和复用。
```

## 13.6 面试里要避免的说法

不要说：

```text
EMF 就是一个队列消费者。
```

应该说：

```text
它底层使用消息，但在消息之上有 metadata resolution、DAG 状态、命令协议、Databricks 执行和 catalogue 登记。
```

不要说：

```text
动态 DAG 就是运行时随便加任务。
```

应该说：

```text
动态 fan-out 必须通过 generator command 生成可审计、可去重、可恢复的 child messages。
```

不要说：

```text
Databricks 只是跑 SQL 的。
```

应该说：

```text
Databricks 是 EMF 的计算面和 Lakehouse 表治理基础，承载 Spark、SQL、dbt、Notebook、Python/UDF 和 Delta 表操作。
```

不要说：

```text
retry 就是失败后再执行一次。
```

应该说：

```text
retry 要回到状态机、幂等 key、attempt、run_date 和外部副作用对账。
```

## 13.7 最终记忆版

可以背这段：

```text
EMF 是一个元数据驱动的数据编排平台。
file_type 决定入口路由，LOAD_INFO 决定怎么加载，OE-RUN 根据 PROCESS_TASKS 展开 workflow，Message 通过 run_uuid、order_id、parents 形成运行时 DAG。
Worker 负责状态、依赖、重试、取消、超时和审计；Databricks 负责 SQL、Spark、dbt、Notebook、Python/UDF 等计算；Delta/Lakehouse 承载最终表资产；CATALOGUE.METADATA 记录实体信息，让下游可以通过 criteria 解析和复用数据。
生产难点不是拓扑排序，而是 metadata 一致性、外部 job 状态对账、幂等重试、动态 fan-out 治理、可观测性和成本控制。
```
