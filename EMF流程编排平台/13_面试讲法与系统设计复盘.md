# 13 面试讲法与系统设计复盘

## 13.1 一分钟版本

如果面试官问“你做的 EMF 是什么”,可以这样讲:

```text
EMF 是一个面向数据和平台任务的工作流执行引擎。
它没有把 workflow 固化成 Python DAG 文件,而是把每个任务抽象成 Message。
Message 里包含 run_uuid、order_id、parents、command、parameters。
系统根据 run_uuid 把消息归到同一张运行时 DAG,根据 parents 引用父节点 order_id 建图,
然后调度器只执行依赖已经满足的节点。
命令层再根据 command 找到具体 module 执行业务。
因为命令运行时还能继续 enqueue 新 Message,所以它支持动态 DAG、子 workflow、retry 和 fan-out。
```

这一分钟版本要突出三个点:

1. 任务抽象是 `Message`。
2. DAG 是由 `parents -> order_id` 推导出来的。
3. 执行能力通过命令系统插件化。

如果想再加一句系统设计味道,可以补:

```text
调度层不关心 SQL 或 Storage 具体怎么执行,命令层也不关心 DAG 怎么调度;
EMF 把控制平面和执行平面分开了。
```

## 13.2 三分钟版本

三分钟可以讲清架构分层:

```text
入口层: 外部 PubSub / Service Bus 或 OE-RUN 产生 Message。
建图层: WorkerMessageAppender 按 batch_id / run_uuid 把消息加入运行时上下文。
调度层: DagManager / DagContext 维护 vertices、parents、children 和状态。
并发层: ParallelMessageConsumer 找 ready 节点,按并发限制和 priority 派发。
执行层: MessageHandler 创建 message scope,CommandExecutor 根据 command 找 module。
业务层: module 调用 Databricks / Storage / Catalogue / Service,也可以继续生成新 Message。
治理层: retry、timeout、cancel、label filtering、order validation、audit log。
```

这个讲法比“我做了一个 workflow engine”更有分量,因为它说明了边界、状态和治理。

## 13.3 最值得讲的设计选择

### 选择一: 用 Message 表达 DAG 节点

普通做法可能是先定义一张 DAG,再把 task 放进去。EMF 反过来:

```text
Message 本身携带 order_id 和 parents,
一批 Message 就能推导出 DAG.
```

这样做的收益:

- 外部事件、workflow 展开、retry、动态 fan-out 都可以统一成“生成消息”。
- 调度器只需要理解 Message,不需要理解每种业务 workflow。
- 动态扩展自然,命令运行时生成新 Message 就能接入当前 DAG。

代价:

- `order_id` 和 `parents` 的数据质量非常关键。
- label filtering、子 workflow、retry 都要小心维护依赖关系。
- 调试时必须先还原“实际生成了哪些 Message”。

### 选择二: 用 command/module 隔离业务能力

`command` 不是函数名,而是协议名。执行时再映射到具体 module。

收益:

- workflow 作者只关心命令和参数,不关心内部代码结构。
- 命令可以统一声明参数、执行逻辑、服务依赖和副作用。
- 迁移或重构内部实现时,只要命令协议不变,workflow 不需要改。

代价:

- 命令注册、参数解析、module provider 会变成平台治理点。
- 命令协议一旦暴露给业务,就要谨慎兼容。

### 选择三: 调度和执行分层

调度层只回答:

```text
谁 ready?
谁能占用并发 slot?
谁优先?
状态怎么推进?
```

执行层只回答:

```text
这条 Message 的 command 是什么?
参数如何解析?
调用哪个 module?
module 如何产生业务副作用?
```

收益:

- 调度器不需要理解 Databricks、Storage、Catalogue。
- 命令实现不需要理解 DAG ready 逻辑。
- retry、timeout、priority 可以作为调度治理统一处理。

## 13.4 面试高频问题

### Q1: 为什么不用 Airflow?

不要只说“Airflow 太重”。可以这样回答:

```text
Airflow 更偏静态 DAG + 调度器周期扫描。
EMF 的核心是消息驱动和运行时 DAG 扩展。
很多任务来自外部事件或 workflow 展开,甚至 command 执行后还会继续生成新节点。
所以 EMF 把 Message 作为统一抽象,让外部触发、子 workflow、retry、fan-out 都变成同一种模型。
```

重点不是贬低 Airflow,而是说明模型不一样。

### Q2: DAG 依赖怎么表达?

答案:

```text
每个 Message 有 order_id 和 parents。
parents 保存的是父任务的 order_id 列表。
DagContext 根据 parents 找父节点,建立 parent/child 关系。
一个节点只有在自己是 INACTIVE、run_date 到期、所有 parent 状态满足条件时才 ready。
```

可以补一句:

```text
这里 msg_id 不是依赖主键,msg_id 是消息实例 ID;order_id 才是业务节点 ID。
```

### Q3: 怎么支持动态 DAG?

答案:

```text
命令执行过程中可以继续 enqueue 新 Message。
只要新 Message 带着同一个 run_uuid、正确的 order_id 和 parents,
它就能加入当前 DagContext,成为运行时 DAG 的一部分。
```

例子:

```text
一个 query 命令查出多条 job,为每条 job 生成一条 SQL eval message,
这些消息会作为后续节点继续被调度。
```

### Q4: retry 怎么做?

答案:

```text
retry 不是简单原地 while 重跑,而是重新生成一条 retry Message。
它保留原来的 command、order_id、run_uuid、batch_id、parents、priority,
但更新 retry 状态和 run_date。
run_date 到期后,它会重新参与 ready 判断。
```

这样讲可以体现你理解了 retry 和 DAG 状态之间的关系。

### Q5: 并发怎么保证不破坏依赖?

答案:

```text
并发只发生在 ready messages 之间。
ParallelMessageConsumer 先从 DagManager 拿 ready vertices,
再用 max_messages、Counter、Semaphore 控制并发 slot。
依赖没满足的节点不会进入 ready 集合,所以不会因为线程池有空位就越级执行。
```

### Q6: label filtering 为什么难?

答案:

```text
因为过滤一个节点不只是少执行一个任务。
如果 A -> B -> C,过滤掉 B 后,C 不能继续依赖 B,否则永远不 ready。
所以 filtering 需要 relink,把 B 的 children 接回 B 的 parents。
```

这类回答很容易拉开深度。

### Q7: 子 workflow 难点在哪里?

答案:

```text
子 workflow 本质是父 workflow 中的一个 OE-RUN task 展开出一批新的 Message。
难点不是生成这些 Message,而是把子 workflow 的入口、出口和父 workflow 后续节点重新接好。
通常需要 completion token 或类似汇合点表示子流程完成。
```

可以继续展开:

```text
如果父子 workflow 沿用同一个 run_uuid,它们就在同一张 DAG 里,父流程可以通过 parents 直接等待子流程。
这时必须给子流程 order_id 加 namespace,避免和父流程节点重名。
如果子 workflow 开新 run_uuid,它就变成另一张 DAG,父流程要等待它就需要额外的状态或事件协调。
```

### Q8: `batch_id` 和 `run_uuid` 为什么都需要?

答案:

```text
run_uuid 定义一张运行时 DAG,用于判断这个 workflow run 内哪些节点 ready、是否完成、是否 timeout。
batch_id 定义一组 run 的生命周期容器,用于 batch 级状态汇总、资源治理、cleanup 和缓存管理。
同一个 batch 里可以有多个 run_uuid,所以这两个概念不能合并。
```

类比:

```text
batch_id 像文件夹;
run_uuid 像文件夹里的一张 DAG 图;
message 是图里的一个节点。
```

### Q9: 为什么 retry 要生成新 Message?

答案:

```text
因为 EMF 希望 retry 也回到统一调度模型。
失败后生成 retry Message,带上原来的 run_uuid、batch_id、order_id、parents、command 和未来 run_date。
这样 retry delay 不占用线程,每次尝试有审计记录,也能继续受 DAG ready 判断和平台治理控制。
```

### Q10: 动态 fan-out 怎么保证后续节点不提前执行?

答案:

```text
动态 fan-out 不是只生成一堆新消息,还要把新消息接回 DAG。
如果 Q 后面原本是 Next,但 Q 运行时展开出 T1...T100,
那 Next 应该等待这些动态子任务完成,而不是只等 Q 完成。
所以动态生成时必须维护 order_id、parents 和后续 relink / completion 语义。
```

### Q11: D 依赖 A/B/C,但 A/B/C 之间还有数据依赖怎么办?

答案:

```text
D.parents = [A, B, C] 只表达 D 要等 A/B/C 都完成。
它不表达 A/B/C 之间的顺序。
如果 A/B/C 之间存在数据依赖,必须把这些依赖也显式建成 DAG 边。
```

例如业务上是:

```text
A 产出数据
B 读取 A
C 读取 B
D 汇总
```

那应该建成:

```text
A -> B -> C -> D
```

而不是:

```text
A
B
C
 \ | /
   D
```

可以补一句系统设计判断:

```text
DAG 依赖保证执行顺序,但不会自动理解数据语义。
凡是影响数据正确性的依赖,都要进入 parents;否则就是 hidden dependency,会导致并发读写、脏读或结果不确定。
```

如果 A/B/C 没有读取依赖,只是写同一个资源,那是资源冲突问题,可以通过 partition 隔离、staging + commit、merge 节点、幂等写或锁来治理。

### Q12: D 调外部服务,但服务宕机怎么办?

答案:

```text
外部服务失败时,D 不能标记 COMPLETE。
command 应该把失败转换成 RETRY、FAILED 或 TIMEOUT。
可重试场景下生成 retry Message,带未来 run_date,到时间后重新进入 ready 判断。
```

还要强调幂等:

```text
因为 retry 可能多次调用同一个外部服务,
所以 D 的外部请求应该带 idempotency key 或 correlation id,
例如 run_uuid + order_id。
这样外部服务可以识别重复请求,避免重复创建资源、重复写表或重复触发副作用。
```

更完整的回答可以按失败类型拆:

| 场景 | 处理 |
|---|---|
| 网络抖动 | retry with backoff |
| 服务短暂不可用 | retry message + 未来 `run_date` |
| 长时间不可用 | 超过最大 retry 后 `FAILED` / `TIMEOUT` |
| 成功但响应丢失 | 幂等 key 查询或去重 |
| 部分副作用 | compensation / reconciliation / 人工介入 |

总结句:

```text
DAG 保证 D 的调度顺序,但外部副作用一致性要靠 command 的幂等、retry、timeout 和补偿机制保证。
```

### Q13: 动态 DAG 会遇到哪些挑战?

答案可以按五类讲:

```text
动态 DAG 的挑战不是生成新节点本身,而是生成后图还要保持正确、可控、可恢复和可观测。
```

第一类是依赖正确性。比如原来是:

```text
A -> D
```

A 运行时生成 B/C/E 后,如果 D 仍然只依赖 A,就会提前执行。正确做法是 relink:

```text
A -> B -> D
A -> C -> D
A -> E -> D
```

或者通过 completion token 汇合:

```text
A -> B/C/E -> COMPLETION_TOKEN -> D
```

第二类是 fan-out 失控。一个 query 如果返回几十万条结果,就会生成几十万条 message,压垮 `DagContext`、worker、状态存储和外部服务。治理方式包括最大 fan-out、分页生成、partition/shard、backpressure、资源池和限流。

第三类是幂等和重复生成。A 已经生成了 B/C/E,但 A retry 后又生成一遍,就会导致重复执行和重复副作用。动态生成要用稳定 `order_id`、幂等 key、去重和 audit log 控制。

第四类是 DAG 合法性。动态节点可能引用不存在的 parent,形成 dangling parent;也可能新增边后形成 cycle。需要入队校验、cycle detection、blocked node 检测。

第五类是可观测性。动态图边跑边增长,必须记录:

```text
generated_by
source command
fan-out group
run_uuid / order_id / parents / command
message lineage
```

总结句:

```text
动态 DAG 强在灵活,难在治理。
必须处理 relink、fan-out 限流、幂等去重、cycle detection、run completion 和 observability。
```

## 13.5 可以画的白板图

面试时可以画这张:

```text
External Event / OE-RUN
        |
        v
     Message
        |
        v
  batch_id / run_uuid grouping
        |
        v
     DagContext
   parents -> children
        |
        v
   ready vertices
        |
        v
ParallelMessageConsumer
        |
        v
CommandExecutor
        |
        v
Module / Service
        |
        v
Business side effect or New Messages
```

然后补一句:

```text
这个系统的核心闭环是:命令可以产生消息,消息可以扩展 DAG,DAG 又继续驱动命令。
```

## 13.6 面试里要避免的说法

不要说:

```text
EMF 就是一个队列消费者。
```

应该说:

```text
它以消息为任务单元,但在内存里维护 DAG 依赖和状态机。
```

不要说:

```text
parents 就是上一个任务。
```

应该说:

```text
parents 是 join barrier,可以依赖多个父节点。
```

不要说:

```text
retry 就是失败后再执行一次。
```

应该说:

```text
retry 通过新的 retry Message 和 run_date 重新进入调度。
```

不要说:

```text
command 就是 Python 函数。
```

应该说:

```text
command 是协议名,module 是实现,中间有注册和参数解析层。
```

## 13.7 最终记忆版

把 EMF 记成四句话:

```text
Message 是任务,也是 DAG DSL。
run_uuid 分组出一张 DAG,parents 通过 order_id 建边。
Consumer 只执行 ready 节点,CommandExecutor 负责把 command 映射成 module。
命令可以继续生成 Message,所以 workflow 可以动态扩展。
```

再浓缩成一句:

```text
Message 是节点,parents 是边,run_uuid 是图,batch_id 是图的容器,command 是节点上的动作。
```
