# 14 高级机制与补充结论

## 14.0 先定性:EMF 到底是哪类系统

EMF 不是单纯的队列消费者,也不是固定脚本调度器。

可以把它定性为:

```text
消息驱动的动态 DAG 工作流引擎。
```

这句话里每个词都有含义:

| 关键词 | 含义 |
|---|---|
| 消息驱动 | 外部事件、OE-RUN、retry、动态 fan-out 都会以 Message 形式进入系统 |
| 动态 DAG | DAG 不是一次性固定,运行中可以继续增长和重接 |
| 工作流引擎 | 系统不只是执行任务,还维护依赖、状态、优先级、超时、取消和生命周期 |

再从分层上看:

| 平面 | 职责 |
|---|---|
| 控制平面 | Message、DAG、ready 判断、状态机、调度、retry、timeout、batch/run 生命周期 |
| 执行平面 | Databricks SQL、Storage、Catalogue、API、文件处理和具体业务副作用 |

面试时可以这样讲:

```text
EMF 的调度器不需要知道 SQL 怎么跑;
SQL command 也不需要知道 DAG 怎么调度。
它们通过 Message 和 command 协议解耦。
```

## 14.1 DAG 不是配置出来的

EMF 的 DAG 不是来自一份独立的配置文件,而是从一批 `Message` 推导出来的。

核心字段:

| 字段 | 作用 |
|---|---|
| `run_uuid` | 哪一个 workflow run,对应一张 DAG |
| `batch_id` | 哪一批 run,做批次级生命周期管理 |
| `msg_id` | 单条消息实例 ID |
| `order_id` | 业务任务节点 ID |
| `parents` | 当前任务依赖哪些父任务的 `order_id` |
| `command` | 这个节点要执行哪个命令 |
| `parameters` | 命令参数 |

最重要的一句:

```text
DAG 边不是靠 msg_id 连接,而是靠 parents 里的 order_id 去匹配父节点.
```

## 14.2 parents 是 join barrier

`parents` 不是执行顺序提示,而是 join barrier。

例如:

```text
A
|- B
|- C
   \ 
    D
```

等价 message:

```json
[
  {"order_id": "A", "parents": []},
  {"order_id": "B", "parents": ["A"]},
  {"order_id": "C", "parents": ["A"]},
  {"order_id": "D", "parents": ["B", "C"]}
]
```

D 只有在 B 和 C 都达到允许状态后才会 ready。

## 14.3 hidden dependency 和数据正确性

`parents` 表达的是调度器能看见的依赖。它能保证节点执行顺序,但不能自动理解业务数据语义。

例如:

```text
A
B
C
 \ | /
   D
```

这个 DAG 只表示 D 等 A/B/C 完成。它没有表示 A/B/C 之间有依赖。

如果真实数据关系是:

```text
A 生成数据
B 读取 A 的数据
C 读取 B 的结果
D 汇总
```

那正确 DAG 应该是:

```text
A -> B -> C -> D
```

否则 A/B/C 会被调度器视为可并发节点,产生 hidden dependency:

| hidden dependency 类型 | 可能问题 |
|---|---|
| B 读取 A 的输出,但没有写 parents | B 可能在 A 完成前执行 |
| 多个节点写同一张表或 partition | 并发覆盖、脏写、最终结果不稳定 |
| 一个节点读取另一个节点的外部副作用 | 读到旧状态或读不到资源 |
| 下游汇总节点没有等待动态 fan-out | 汇总不完整 |

处理原则:

```text
影响数据正确性的依赖,必须显式进入 DAG;
影响资源冲突的并发,必须通过隔离、锁、幂等或 commit 节点治理。
```

常见治理方式:

- 用 `parents` 显式表达读写顺序。
- 用 staging 表/path 写中间结果,最后由一个 commit 节点发布。
- 通过 partition 隔离并发写。
- 用 merge / aggregate 节点统一汇总。
- 对外部资源使用幂等 key、锁或版本检查。

## 14.4 节点什么时候 ready

一个节点 ready 需要:

1. 状态是 `INACTIVE`。
2. `run_date` 小于等于当前时间。
3. 所有父节点状态是 `COMPLETE` 或 `RETRY`。

状态流:

```text
INACTIVE -> ACTIVE -> IN_PROGRESS -> COMPLETE
                                 |-> RETRY
                                 |-> FAILED
                                 |-> TIMEOUT
                                 |-> CANCELLED
```

## 14.5 batch_id 和 run_uuid 的区别

这两个字段是 EMF 里最容易混的。

| 字段 | 作用 |
|---|---|
| `batch_id` | 一批 run 的分组,用来做批次级状态管理和调度 |
| `run_uuid` | 一次 workflow run 的分组,对应一张 DAG |

可以这样想:

```text
Batch X
  |- run_uuid A / DAG A
  |    |- message 1
  |    |- message 2
  |
  |- run_uuid B / DAG B
       |- message 4
       |- message 5
```

`DagManager` 重点按 `run_uuid` 管理 `DagContext`,而 batch context 负责更外层的批次生命周期。

## 14.6 Retry 不是原地重跑

Retry 也会走消息模型。

典型流程:

```text
message failed
  -> RetryService.retry()
  -> RetryMessageFactory.create()
  -> enqueue retry message
```

retry message 会保留原 message 的关键字段:

```text
command
order_id
run_uuid
batch_id
parents
batch_priority
```

但它会更新 retry 状态和未来的 `run_date`。因此 retry 本质上仍是“生成一条新的 Message”,只不过它代表同一个业务节点的下一次尝试。

这种设计的收益是:

- retry delay 不占用执行线程。
- 每次尝试都有独立记录,便于审计。
- retry 次数和状态可以被平台统一治理。
- 失败恢复仍然走 Message / DAG 的统一入口。

## 14.7 并发是怎么做的

真实控制点:

1. `max_messages` 控制最大并发。
2. `Counter` 记录当前占用 slot。
3. `Semaphore` 控制提交次数。
4. 每轮只拿 `available_slots` 个 ready vertex。
5. 按 `batch_priority` 从高到低排序。

关键结论:

```text
并发只发生在 ready messages 之间;
依赖不满足的节点不会因为线程池有空位而执行.
```

## 14.8 DAG 怎么动态扩展

动态扩展不是一次性生成完整 DAG,而是命令运行时继续 enqueue 新消息。

例如 `ADB-MESSAGES-FROM-QUERY` 会执行 query,然后把结果转换成多条 `ADB-SQL-EVAL` 消息:

```text
workflow message
  -> run query
  -> result row 1 -> new Message
  -> result row 2 -> new Message
  -> result row 3 -> new Message
  -> enqueue_all()
```

这就是 EMF 动态 DAG 的关键:命令既可以做业务,也可以生成后续任务。

动态 DAG 的风险在于:图运行中还会变化,所以平台必须持续维护正确性。

| 挑战 | 典型问题 | 治理方式 |
|---|---|---|
| 下游提前执行 | A 生成 B/C/E 后,D 仍然只等 A | relink 或 completion token |
| fan-out 失控 | query 返回几十万条结果,生成海量 message | 最大 fan-out、分页、分批、backpressure |
| `order_id` 冲突 | 多个动态节点重名 | 用业务 key 或 namespace 生成稳定唯一 ID |
| dangling parent | 新节点依赖不存在的父节点 | 入队校验 parent、blocked node 检测 |
| cycle | 动态加边后形成 A -> B -> A | topological sort / cycle detection |
| retry 重复生成 | retry 后重复 enqueue 子任务 | 幂等生成、去重、audit lineage |
| run completion 复杂 | 初始节点完成但后续动态节点还没完成 | completion token、run state manager |
| 可观测性下降 | 不知道节点是谁生成的 | 记录 generated_by、source command、fan-out group |

面试时可以说:

```text
动态 DAG 不是难在生成消息,而是难在生成后仍然保证依赖正确、规模可控、幂等可恢复、图合法且可观测。
```

## 14.9 子 workflow 怎么接回父流程

子 workflow 的核心是父流程里有一个 `OE-RUN` task。

流程:

```text
父 workflow 中的 OE-RUN task
  -> 找到子 workflow
  -> 生成子 workflow messages
  -> 子 workflow 内部执行
  -> completion token 汇合
  -> 父 workflow 后续节点继续执行
```

子 workflow 最难的不是生成 messages,而是把父节点、子节点和后续节点的 `parents` / `order_id` 关系接回去。

这里有三个关键点。

第一,子 workflow 不是普通函数调用:

```text
不是:父节点同步调用函数,函数内部跑完子流程后返回。
而是:父节点展开一批子 workflow messages,让它们继续进入 DAG 调度。
```

第二,子 workflow 需要处理 `order_id` 命名空间。父子流程里可能都有 `LOAD_DATA`、`VALIDATE` 这类节点名,如果直接放进同一张 DAG,`parents` 匹配会混乱。因此子节点通常要加前缀:

```text
parent_oe_run_order_id + ":" + child_order_id
```

第三,父 workflow 往往不应该直接知道子 workflow 的所有终点。更好的方式是用 completion token 做汇合点:

```text
子 workflow 多个终点
  -> completion token
  -> 父 workflow 后续节点
```

completion token 的意义是:把“子 workflow 已经完成”抽象成一个父流程可以依赖的单一节点。

子 workflow 是否沿用父流程的 `run_uuid` 也很关键:

| 方式 | 适合场景 | 注意点 |
|---|---|---|
| 同一个 `run_uuid` | 子流程是父流程的一部分,父流程要直接等待子流程 | 需要命名空间,避免 `order_id` 冲突 |
| 新 `run_uuid` | 子流程是独立 workflow run | 父流程不能天然通过 `parents` 等待它,需要额外协调 |

## 14.10 label filtering 和 relink

问题是:过滤掉一个任务后,不能让 DAG 断链。

例如:

```text
A -> B -> C
```

如果 B 被 disabled,理想结果应该是:

```text
A -> C
```

而不是:

```text
A -> B(被过滤) -> C(永远等不到 parent)
```

所以 label filtering 通常必须配合 relink 逻辑,把被过滤节点的 children 接回它的 parents。

## 14.11 顺序和依赖校验

`process_tasks` 里定义了 workflow 任务顺序,但 EMF 真正执行还是靠 DAG。

相关校验会检查 `process_tasks` 顺序问题,避免 workflow 顺序表和依赖关系互相打架。

一个重要提醒:

```text
不要把运行时调度器当成唯一的数据质量防线.
```

workflow 数据质量很重要。`order_id`、`parents`、label filtering、relink 这些字段一旦写错,后续表现可能是节点不 ready、错误跳过、重复执行或永远无法结束。

## 14.12 外部服务失败和幂等

当某个节点调用外部服务时,它面对的是不可靠边界:

```text
Message D
  -> command
  -> external service
```

如果外部服务宕机、超时或返回错误,D 不应该进入 `COMPLETE`。否则 DAG 会继续推进,下游节点会误以为 D 的业务副作用已经完成。

推荐处理模型:

```text
D 调用外部服务
  -> 成功: D COMPLETE
  -> 可恢复失败: D RETRY,生成 retry message
  -> 不可恢复失败: D FAILED
  -> 超过运行时间: D TIMEOUT
```

可重试时,最好用 retry message 表达下一次尝试:

```text
retry message
  -> 保留 run_uuid / order_id / parents / command
  -> 设置未来 run_date
  -> 到时间后重新参与 ready 判断
```

同时 command 必须考虑幂等性。因为同一个业务节点可能因为 retry 多次调用外部服务,外部请求最好携带稳定 key:

```text
idempotency_key = run_uuid + order_id
```

如果需要区分每一次尝试,可以再加 attempt:

```text
external_request_id = run_uuid + order_id + attempt_id
```

外部失败处理可以按类型拆:

| 失败类型 | 治理方式 |
|---|---|
| 网络抖动 | retry with backoff |
| 服务短暂不可用 | retry message + 未来 `run_date` |
| 服务长时间不可用 | 最大 retry 后 `FAILED` 或 `TIMEOUT` |
| 请求成功但响应丢失 | 幂等 key / correlation id 查询外部状态 |
| 已产生部分副作用 | compensation、rollback、reconciliation 或人工介入 |

核心结论:

```text
DAG 保证调度顺序;
外部副作用一致性靠 command 的幂等、retry、timeout 和补偿机制保证。
```

## 14.13 高级特性清单

| 能力 | 关键点 |
|---|---|
| 动态 DAG 扩展 | 某个 command 可继续生成新 message,加入当前 DAG |
| 动态 DAG 治理 | relink、fan-out 限流、幂等去重、cycle detection、lineage |
| 子 workflow | `OE-RUN` 可作为 workflow 内部节点,展开后接回父流程 |
| retry delay | retry message 通过未来 `run_date` 延迟执行 |
| priority 调度 | `batch_priority` 降序调度 |
| timeout / cancel | `DagManager` 支持 run 级 timeout 和 cancel |
| hidden dependency 治理 | 数据依赖必须进入 DAG,资源冲突要用隔离或幂等治理 |
| 外部副作用治理 | 外部调用失败不能直接 COMPLETE,需要 retry、timeout、幂等和补偿 |
| label filtering | `enabled` / `disabled` 过滤任务,需要 relink |
| order validation | 校验 process_tasks 顺序和依赖合理性 |
| completion token | 子 workflow 或汇合点用 token 表示完成 |

## 14.14 最后一条补充心智

EMF 的复杂度主要来自一件事:

```text
它把 workflow 定义、运行时调度、重试、过滤、子流程和动态扩展,
全部统一压到 Message + DAG 这套模型里.
```

所以排查问题时不要先问“这个 workflow 配置怎么写”,而要问:

```text
最后生成了哪些 Message?
每条 Message 的 run_uuid / order_id / parents / command 是什么?
这些 Message 进入 DagContext 后,谁会 ready,谁会等待?
```

## 14.15 面试追问时怎么展开

如果面试官继续追问,可以按“机制、收益、代价”来展开。

| 机制 | 收益 | 代价 |
|---|---|---|
| Message 推导 DAG | 入口统一,动态扩展自然 | 消息数据质量变得很关键 |
| parents join barrier | 支持多父依赖和汇合点 | dangling parent 会导致节点永远等待 |
| retry message | retry 回到统一调度模型 | 需要幂等和 retry 次数控制 |
| command/module | workflow 和实现解耦 | command 协议需要兼容治理 |
| label filtering + relink | 同一 workflow 可按 label 裁剪 | 过滤后依赖重接很容易出错 |
| dynamic fan-out | 支持运行时决定任务数量 | fan-out 失控和观测复杂度上升 |
| 外部副作用幂等 | retry 后仍能避免重复副作用 | 需要外部系统支持幂等 key 或状态查询 |

这种讲法会比背源码更像系统设计复盘。
