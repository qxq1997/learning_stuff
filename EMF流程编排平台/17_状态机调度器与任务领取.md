# 17 状态机、调度器与任务领取

## 为什么状态机是 DAG 引擎的核心

DAG 引擎看起来是在跑图,本质是在推进状态。

每个 task instance 都是一台小状态机:

```text
CREATED -> READY -> RUNNING -> SUCCESS
                         |-> FAILED
                         |-> RETRYING
                         |-> TIMEOUT
                         |-> CANCELLED
                         |-> WAITING_APPROVAL
                         |-> WAITING_EVENT
```

每个 workflow instance 也是状态机:

```text
CREATED -> RUNNING -> SUCCESS
                   |-> FAILED
                   |-> PARTIAL_SUCCESS
                   |-> CANCELLED
                   |-> PAUSED
                   |-> SUSPENDED
                   |-> TIMEOUT
```

调度器做的事,就是不断根据这些状态回答:

```text
哪些任务可以变成 READY?
哪些 READY 可以变成 RUNNING?
哪些 RUNNING 超时了?
哪些 FAILED 可以 RETRY?
哪些 workflow 已经结束?
```

所以状态不是普通字段,而是调度协议。

## 任务状态设计

一个生产级 task instance 至少需要区分:

| 状态 | 含义 |
|---|---|
| `CREATED` | 实例已创建,但还没进入可调度判断 |
| `READY` | 依赖满足,等待领取 |
| `RUNNING` | 已被 worker 领取并执行 |
| `SUCCESS` | 业务副作用已完成且可验证 |
| `FAILED` | 不可恢复失败或 retry 耗尽 |
| `RETRYING` | 失败后等待下一次尝试 |
| `SKIPPED` | 条件分支或人工跳过 |
| `CANCELLED` | 被取消 |
| `TIMEOUT` | 超过最大运行时间 |
| `WAITING_APPROVAL` | 等人工审批 |
| `WAITING_EVENT` | 等外部事件或 callback |
| `LOST` | worker 或外部执行状态丢失,需要恢复判断 |

不要只用成功/失败。生产里最难的是中间态:

```text
运行中但 worker 死了;
外部 job 已提交但没回调;
等待审批;
等待事件;
已经失败但还要延迟 retry;
被跳过但下游可能继续。
```

## 状态流转必须受控

允许的流转要显式定义。例如:

```text
CREATED -> READY
READY -> RUNNING
RUNNING -> SUCCESS
RUNNING -> FAILED
RUNNING -> TIMEOUT
RUNNING -> CANCELLED
FAILED -> RETRYING
RETRYING -> READY
WAITING_APPROVAL -> READY
WAITING_APPROVAL -> CANCELLED
WAITING_EVENT -> READY
```

不应该随便允许:

```text
SUCCESS -> RUNNING
CANCELLED -> RUNNING
FAILED -> SUCCESS
```

如果确实需要人工修复后把 `FAILED` 改成 `SUCCESS`,也应该是特殊操作:

```text
manual override
  -> 校验权限
  -> 写 operation audit
  -> 写 state transition
  -> 标记 reason
```

生产事故里,“状态乱跳”常常来自四类来源:

1. scheduler 自动推进状态。
2. worker 执行完成后更新状态。
3. timeout scanner / zombie scanner 更新状态。
4. 用户人工操作更新状态。

这些来源必须经过同一个状态机服务,不能各写各的 SQL。

## COMPLETE / SUCCESS 不能撒谎

一个节点进入 `SUCCESS`,下游就可能 ready。因此 `SUCCESS` 必须代表:

```text
业务副作用已经达到可验证完成状态。
```

常见错误是把“外部任务已提交”当成“外部任务已完成”。

例如:

```text
B 调 Databricks submit job 成功
  -> 立刻标记 SUCCESS
  -> D 开始执行
  -> 但 Databricks job 其实还在跑
```

正确方式有两种:

### 方式一:节点内部等待终态

```text
B command
  -> submit external job
  -> poll until external job terminal
  -> verify output
  -> mark SUCCESS
```

适合执行时间可控、worker 可以承受等待的任务。

### 方式二:拆成 submit / monitor

```text
B_submit
  -> B_monitor
  -> D
```

`B_submit` 只负责提交并保存 external job id。`B_monitor` 负责轮询或等待 callback,确认外部 job 成功后再让 D 继续。

适合长任务、外部异步任务、需要恢复现场的任务。

## 状态更新要用 CAS

多 scheduler、多 worker、人工操作并发时,不能直接:

```sql
UPDATE task_instance SET status = 'RUNNING' WHERE id = ?;
```

因为这可能覆盖别人的更新。

应该带上期望状态:

```sql
UPDATE task_instance
SET status = 'RUNNING',
    worker_id = ?,
    started_at = now(),
    version = version + 1
WHERE id = ?
  AND status = 'READY'
  AND version = ?;
```

只有更新成功的一方才真正拿到任务。

这叫 compare-and-set,核心是:

```text
我只在它仍然是 READY 时把它改成 RUNNING。
如果它已经被别人改了,我就失败并放弃。
```

## 任务领取机制

调度器或 worker 不能简单:

```sql
SELECT * FROM task_instance WHERE status = 'READY';
```

多实例部署时,多个 scheduler 可能拿到同一批 READY 任务。

常见做法有两种。

### 方式一:数据库锁

```sql
SELECT *
FROM task_instance
WHERE status = 'READY'
ORDER BY priority DESC, scheduled_at ASC
FOR UPDATE SKIP LOCKED
LIMIT 100;
```

优点:

- 实现直接。
- 数据库保证同一行不会被多个事务拿到。

代价:

- 依赖数据库能力。
- 高并发下 DB 压力大。
- 长事务要小心。

### 方式二:CAS 领取

先查候选,再抢占:

```sql
UPDATE task_instance
SET status = 'RUNNING',
    worker_id = ?,
    lease_until = now() + interval '5 minutes',
    attempt = attempt + 1
WHERE id = ?
  AND status = 'READY';
```

只有 affected rows = 1 的 worker 才算领取成功。

这种方式的关键是:

```text
任务领取和状态改变必须是一个原子动作。
```

## lease 和 heartbeat

如果 worker 领取任务后挂了,任务会一直 `RUNNING`。

所以 `RUNNING` 任务需要 lease:

```text
worker 领取任务时设置 lease_until = now + 5 minutes
worker 定期 heartbeat 延长 lease
如果 lease 过期,scanner 判断任务 lost / retry
```

推荐字段:

```text
worker_id
lease_until
heartbeat_at
attempt
started_at
updated_at
```

zombie scanner 可以扫描:

```sql
SELECT *
FROM task_instance
WHERE status = 'RUNNING'
  AND lease_until < now();
```

然后结合 worker heartbeat 和外部 job 状态判断:

| 情况 | 处理 |
|---|---|
| worker 死了,外部 job 没提交 | RETRYING |
| worker 死了,外部 job 还在跑 | WAITING_EVENT / RUNNING with recovered owner |
| 外部 job 成功,状态没更新 | reconcile 后 SUCCESS |
| 外部 job 失败 | FAILED / RETRYING |
| 状态无法确认 | LOST,人工或补偿扫描处理 |

这里的关键点是:lease 过期只能说明“原执行 owner 的控制权不可信”,不能直接说明“业务动作没有发生”。完整的 zombie task 发现、对账、恢复和误判防护,可以看第 32 章《僵尸任务发现与处理》。

## 调度器核心循环

简单版:

```text
scan ready tasks
claim tasks
dispatch to workers
update running / success / failed
unlock downstream tasks
```

生产版要多很多判断:

```text
while scheduler running:
  scan workflow instances not terminal
  detect timeout / paused / cancelled
  compute newly ready tasks by dependency + trigger rule
  apply resource limits
  apply tenant / workflow / task-type quotas
  claim READY tasks with CAS / lock
  dispatch to worker / queue / external executor
  record state transitions
  emit metrics and audit
```

调度器不应该一次性全表扫描所有任务。规模起来后要尽量:

- 按 status / scheduled_at / lease_until 索引扫描。
- 批量处理。
- 用事件驱动减少轮询。
- 分 shard / tenant / queue 调度。
- 冷热数据分离。

## 并发控制

生产调度器不能“有任务就跑”。

常见并发维度:

| 维度 | 目的 |
|---|---|
| 全局最大并发 | 保护平台整体 |
| 单 workflow run 并发 | 防止单个大 DAG 占满资源 |
| 单 tenant 并发 | 防止大客户影响小客户 |
| 单 task type 并发 | 区分 SQL / HTTP / metadata 任务 |
| 单外部系统并发 | 保护 Databricks、DB、第三方 API |
| 单资源池并发 | heavy / light 任务分池 |

调度判断应该类似:

```text
task is READY
and global slots available
and tenant quota available
and workflow quota available
and task type quota available
and external system quota available
```

如果资源不可用,任务保持 READY 或进入 QUEUED,并记录 blocked reason:

```text
blocked_by = tenant_quota
blocked_by = databricks_pool_full
blocked_by = workflow_concurrency_limit
```

这样排查时才能知道“为什么 ready 了但没跑”。

## 调度公平性

如果用户 A 提交 10000 个任务,用户 B 提交 10 个任务,简单 FIFO 会让 B 饿死。

常见策略:

- priority queue。
- per-tenant queue。
- weighted fair scheduling。
- resource quota。
- aging:等待越久优先级越高。
- heavy / light queue 隔离。

面试里可以说:

```text
调度器不仅要保证 DAG 依赖正确,还要保证资源公平。
否则大 DAG 会把小 DAG 饿死,大租户会把小租户挤掉。
```

## 调度器高可用

调度器挂了不能让所有流程停摆。

多实例 scheduler 要解决两个问题:

1. 同一个任务不能被重复调度。
2. scheduler 重启后能恢复现场。

设计关键:

- READY 任务领取用 CAS / lock。
- RUNNING 任务有 lease。
- scheduler 无本地关键状态,状态都落 DB。
- 状态变更有 transition log。
- 定期 scanner 修复 stuck / timeout / lease expired。
- 外部任务有 external job id,可恢复查询。

如果 scheduler 只是内存里维护图,重启后就丢现场,那生产上很危险。

更完整的故障处理要区分 worker 死、scheduler 死、两者都死、以及多 scheduler 脑裂。第 33 章《Worker 与 Scheduler 宕机恢复》专门展开这些恢复路径。

## workflow 完成判断

workflow instance 是否完成,不能只看“有没有 READY”。

需要综合:

```text
所有 task instance 是否终态?
是否还有 RUNNING?
是否还有 RETRYING 且 scheduled_at 在未来?
是否还有 WAITING_APPROVAL / WAITING_EVENT?
是否有 blocked tasks?
是否存在 dynamic generation 尚未关闭?
是否 workflow 被 cancel / pause?
```

常见终态:

| Workflow 状态 | 条件 |
|---|---|
| `SUCCESS` | 所有必要节点成功或按规则 skipped |
| `FAILED` | 有不可恢复失败且失败传播到 workflow |
| `PARTIAL_SUCCESS` | 允许部分成功的业务语义 |
| `CANCELLED` | 用户取消且运行任务已终止或标记 |
| `TIMEOUT` | workflow 超过最大运行时间 |
| `PAUSED` | 用户暂停,不再调度新任务 |

## 生产问题排查

### READY 任务为什么没跑

排查顺序:

1. 任务是否真的 `READY`。
2. workflow 是否 paused / cancelled。
3. `scheduled_at` 是否在未来。
4. 是否超过全局并发。
5. 是否超过 tenant / workflow / task type 配额。
6. 外部系统资源池是否满。
7. scheduler 是否存活。
8. 领取 CAS 是否失败。
9. 是否被其他 worker 领取后状态变成 RUNNING。

设计修复:

- blocked reason。
- 调度指标:ready count、claimed count、quota rejected count。
- 每个资源池的 running / queued 指标。
- state transition log。

### 任务一直 RUNNING

排查顺序:

1. worker heartbeat 是否还在。
2. task lease 是否过期。
3. external job id 是否存在。
4. 外部 job 当前状态是什么。
5. 是否有 callback 丢失。
6. 是否 command 卡在本地执行。
7. 是否 timeout policy 缺失。

设计修复:

- worker heartbeat。
- task lease。
- timeout scanner。
- external job polling。
- zombie detector。
- `LOST` / `TIMEOUT` 状态。

### 同一个任务被执行两次

排查顺序:

1. 是否多个 scheduler 同时领取。
2. CAS / lock 是否缺失。
3. lease 是否过短导致任务仍在执行时被重新领取。
4. MQ 是否重复投递。
5. worker 执行成功后状态更新是否失败。
6. 是否人工重跑。

设计修复:

- CAS 领取。
- 合理 lease + heartbeat 续租。
- 幂等 key。
- task attempt 表。
- operation audit。

### 状态乱跳

排查顺序:

1. 查看 state transition history。
2. 哪个组件发起状态变更。
3. from_status 是否符合状态机规则。
4. 是否人工 override。
5. 是否旧 worker 不认识新状态。

设计修复:

- central state machine service。
- transition table。
- CAS / version。
- 状态枚举兼容策略。
- manual operation audit。

## 面试总结

可以这样讲:

```text
DAG 引擎的调度器不是简单扫描 READY 任务。
生产环境下,任务状态流转必须由中心状态机控制,状态更新要用 CAS 或锁避免并发覆盖。
任务领取要是原子动作,否则多个 scheduler/worker 会重复执行同一任务。
RUNNING 任务需要 lease 和 heartbeat,否则 worker 宕机会留下僵尸任务。
调度器还要做全局、租户、workflow、任务类型、外部系统维度的并发控制和公平调度。
所以它本质上是一个可靠状态推进器,而不是一个 while loop。
```
