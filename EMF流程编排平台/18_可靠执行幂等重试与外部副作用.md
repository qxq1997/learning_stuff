# 18 可靠执行、幂等、重试与外部副作用

## 为什么可靠执行最难

DAG 引擎进入生产后,最危险的问题不是“任务失败了”,而是:

```text
任务到底有没有成功?
外部副作用到底有没有发生?
如果我重试,会不会重复发生?
```

生产里经常遇到这些不确定状态:

| 场景 | 风险 |
|---|---|
| 外部 API 调用成功,但更新 task 状态失败 | 任务被重跑,外部副作用重复 |
| task 状态更新成功,但外部 API 实际失败 | 下游基于不存在的副作用继续执行 |
| worker 执行中宕机 | 不知道任务做到哪一步 |
| 网络超时 | 对方可能成功,也可能失败 |
| MQ 重复投递 | 同一任务被消费多次 |
| callback 重复回来 | 状态被重复推进 |
| 用户重复点击运行 | 同一业务流程创建多次 |

所以 DAG 引擎不能假设任务只会执行一次。更现实的模型是:

```text
调度系统通常只能做到 at-least-once execution;
业务正确性要靠幂等、状态机、唯一约束和补偿来实现 effectively-once。
```

## exactly-once 的误区

很多人会说“我们要保证任务 exactly once 执行”。这在分布式系统里通常不现实,尤其当任务会调用外部系统时。

原因是:

```text
DB 更新、MQ 投递、worker 执行、外部 API 调用不在同一个事务里。
```

例如:

```text
调用发券系统成功
  -> worker 准备更新 SUCCESS
  -> DB 连接断了
  -> scheduler 后续发现任务没成功
  -> 再次执行任务
```

除非发券系统支持同一个事务,否则 DAG 引擎无法从外部把这两步变成真正 exactly once。

更合理的目标是:

```text
任务可以被重复调度;
外部副作用必须幂等;
状态推进必须可恢复;
排查时必须有证据链。
```

## 幂等 key 怎么设计

幂等 key 的核心问题是:外部系统如何识别“这是同一个业务动作的重复请求”。

常见选择:

```text
idempotency_key = workflow_instance_id + task_key
```

或者在 EMF 语境里:

```text
idempotency_key = run_uuid + order_id
```

这个 key 应该跨 retry 保持稳定。因为 retry 的目标是重新尝试同一个业务动作,不是创造一个新的业务动作。

如果还需要区分每一次尝试,可以另外生成:

```text
attempt_request_id = run_uuid + order_id + attempt
```

区分两类 ID:

| ID | 是否跨 retry 稳定 | 用途 |
|---|---|---|
| `idempotency_key` | 稳定 | 外部系统去重,保证业务动作只发生一次 |
| `attempt_request_id` | 每次尝试不同 | trace、日志、排查单次请求 |

不要把 attempt 放进业务幂等 key,否则每次 retry 都会被外部系统当成新业务请求。

## 有副作用任务的执行协议

一个可靠的有副作用任务,最好遵循类似流程:

```text
1. 领取任务,创建 attempt 记录
2. 生成稳定 idempotency_key
3. 调用外部系统,带 idempotency_key / correlation id
4. 记录 external_request_id、response、trace_id
5. 校验外部系统最终状态
6. CAS 标记 task SUCCESS / RETRYING / FAILED
7. 写 state transition 和 audit log
```

其中第 5 步很重要。`SUCCESS` 不应该只表示“请求发出去了”,而应该表示:

```text
外部副作用已经达到可验证完成状态。
```

例如:

| 任务 | 可验证完成条件 |
|---|---|
| 发券 | 外部发券系统能查到 request_id 对应券已发放 |
| 写表 | 目标 partition 存在且 row count / checksum 合理 |
| 提交 Databricks job | external job terminal status = success |
| 发通知 | 通知系统返回 accepted,且业务允许 accepted 即完成 |
| 创建工单 | external ticket id 已持久化并可查询 |

## 经典事故:外部成功但状态失败

以发券任务为例:

```text
task: issue_coupon
```

错误流程:

```text
1. worker 调发券系统
2. 发券成功
3. worker 更新 task_instance SUCCESS
4. DB 更新失败
5. 任务后来被重新执行
6. 用户收到两张券
```

设计修复:

```text
coupon_request_id = workflow_instance_id + task_key
```

发券系统收到同一个 `coupon_request_id` 时:

```text
第一次:创建券并返回成功
第二次:发现 request_id 已存在,直接返回同一结果
```

DAG 引擎这边还要记录:

```text
external_system = coupon
external_request_id = coupon_request_id
external_response_ref
```

后续如果状态不确定,可以通过 `coupon_request_id` 去 reconciliation。

## 状态成功但外部失败

反过来也危险:

```text
1. task 还没真正完成外部副作用
2. 系统提前标记 SUCCESS
3. 下游开始执行
```

这通常来自:

- 把 submit success 当成 job success。
- finally 里无条件 mark success。
- callback 没校验 external status。
- 人工 mark success 没审计。

设计修复:

- `SUCCESS` 前做输出校验。
- 外部长任务拆成 submit / monitor。
- 状态机禁止无条件成功。
- 人工 override 必须记录 reason 和 operator。
- 下游关键节点做输入完整性校验兜底。

## retry 分类

重试不是“失败就再跑一次”。先要分类。

| 失败类型 | 是否重试 | 例子 |
|---|---|---|
| 瞬时系统失败 | 可以 | 网络抖动、连接池耗尽、HTTP 502 |
| 下游临时不可用 | 可以 | 外部系统 500、限流 |
| 并发冲突 | 可以 | DB deadlock、乐观锁冲突 |
| 参数错误 | 不该 | SQL 语法错、必填参数缺失 |
| 权限错误 | 通常不该 | 403、secret 权限不足 |
| 业务失败 | 通常不该 | 余额不足、审批拒绝 |
| 依赖失败 | 取决于策略 | 上游 failed、输入文件缺失 |
| 超时 | 取决于是否能确认外部状态 | 外部 job 状态未知 |

这要求 command 返回结构化错误,而不是只抛一个字符串:

```text
error_type
retryable
external_status_code
error_code
message
next_action
```

## retry 策略

常见策略:

```text
max_attempts
initial_delay
backoff_multiplier
max_delay
jitter
retryable_errors
dead_letter_policy
alert_policy
```

例如:

```text
attempt 1 failed -> 10s later
attempt 2 failed -> 1m later
attempt 3 failed -> 5m later
attempt 4 failed -> FAILED + alert
```

为什么要 jitter:

```text
如果 1000 个任务同时失败,固定 1 分钟后一起重试,会再次打爆下游。
jitter 可以把重试时间打散。
```

## retry storm

重试风暴是生产常见事故。

场景:

```text
外部 API 故障
  -> 大量任务失败
  -> 大量任务立刻 retry
  -> 外部 API 更加不可用
  -> 失败更多
```

治理方式:

- 指数退避。
- jitter。
- 每个外部系统的 retry rate limit。
- circuit breaker。
- 全局 retry budget。
- 对明显不可恢复错误直接 failed。
- 下游系统恢复前进入 `SUSPENDED` 或 `WAITING_DEPENDENCY`。

## circuit breaker

当外部系统明显不可用时,继续重试没有意义。

可以维护外部系统级状态:

```text
external_system = databricks
state = HEALTHY / DEGRADED / OPEN
```

如果熔断打开:

```text
新任务不再调度到该外部系统;
已有失败任务延迟 retry;
告警给平台和业务 owner;
恢复探测成功后再逐步放量。
```

这比让每个 task 独立疯狂 retry 更稳。

## outbox pattern

如果系统需要在 DB 状态更新后发送消息到 MQ,会遇到:

```text
DB 更新成功,消息发送失败
消息发送成功,DB 更新失败
```

Outbox Pattern 的做法是:

```text
在同一个 DB 事务里:
  更新业务状态
  插入 outbox_event

异步 relay 扫描 outbox_event:
  发送 MQ
  标记 event sent
```

这样至少保证:

```text
只要 DB 事务成功,要发送的消息就不会丢。
```

MQ 消费端仍然要幂等,因为 outbox relay 也可能重复发送。

## inbox / 幂等消费

对于外部 callback 或 MQ 消费,要防重复。

可以维护 inbox 表:

```text
event_id
source
received_at
processed_at
status
```

处理前先插入:

```text
INSERT INTO inbox(event_id, source)
ON CONFLICT DO NOTHING;
```

如果插入失败,说明这个 event 已经处理过或正在处理,直接忽略或返回成功。

这对 callback 特别重要:

```text
外部 job success callback 可能发送多次;
不能让同一个 callback 多次推进状态。
```

## attempt 记录

不要只在 `task_instance` 上存一个 `attempt` 数字。生产排查常常需要每次尝试的细节。

推荐增加:

```text
task_execution_attempt
```

字段:

```text
id
task_instance_id
attempt_no
worker_id
started_at
ended_at
status
idempotency_key
attempt_request_id
external_system
external_request_id
external_trace_id
error_type
error_message
raw_error_ref
```

这样能回答:

```text
第几次尝试失败?
失败时调用了哪个外部请求?
外部 trace_id 是什么?
最后一次 retry 为什么没有继续?
```

## compensation 和 reconciliation

幂等和 retry 不能解决所有问题。

有些外部副作用无法简单回滚:

- 已经发出去的邮件。
- 已经通知的用户。
- 已经触发的第三方审批。
- 部分写入的数据。

这时需要补偿和对账。

| 机制 | 用途 |
|---|---|
| compensation task | 执行反向操作,如撤销工单、回滚标记 |
| reconciliation job | 定期比对 DAG 状态和外部系统状态 |
| manual resolution | 无法自动判断时进入人工处理 |
| audit trail | 保留完整证据链 |

重要的是:补偿也必须幂等。

## 排查清单

### 任务重复执行

排查:

1. 是否有多个 attempt。
2. 是否 lease 过期后被重新领取。
3. worker 是否在状态更新前宕机。
4. MQ / callback 是否重复投递。
5. 是否有人手动重跑。
6. 幂等 key 是否稳定。
7. 外部系统是否支持幂等。

设计修复:

- CAS 领取。
- 合理 lease + heartbeat。
- 稳定 idempotency key。
- attempt 表。
- external request log。
- operation audit。

### retry 一直失败

排查:

1. 错误是否 retryable。
2. 是否是参数/权限/业务错误。
3. retry policy 是否配置过大。
4. 下游系统是否正在故障。
5. 是否触发 retry storm。
6. 是否需要人工介入。

设计修复:

- 错误分类。
- 最大 retry 次数。
- backoff + jitter。
- circuit breaker。
- alert / DLQ。

### 外部状态不确定

排查:

1. 是否有 external_request_id。
2. 外部系统能否按 idempotency key 查询。
3. worker 日志是否有 response。
4. callback 是否丢失。
5. 状态 transition 是否缺失。

设计修复:

- 外部请求日志。
- reconciliation scanner。
- callback inbox 去重。
- submit / monitor 分离。

## 面试总结

可以这样讲:

```text
生产级 DAG 引擎通常只能保证任务至少执行一次,不能天然保证 exactly once。
因此所有有副作用的任务都要设计幂等 key,通常用 workflow_instance_id + task_key 或 run_uuid + order_id。
任务执行要记录 attempt、external_request_id 和 trace_id。
重试要区分 transient error 和 permanent error,使用指数退避、jitter、最大次数和熔断,避免 retry storm。
对于 DB 状态和 MQ 消息不一致,可以用 outbox pattern;对于 callback 重复,可以用 inbox 去重。
真正可靠的执行不是不失败,而是失败后能判断状态、避免重复副作用、可重试、可补偿、可追踪。
```
