# 20 可观测性、审计与排障体验

## 为什么可观测性是核心功能

生产上用户最常问的不是:

```text
你用了什么拓扑排序算法?
```

而是:

```text
为什么没跑?
为什么跑慢了?
为什么失败?
谁点了重跑?
这个任务用了什么参数?
上游输出是什么?
什么时候开始卡住的?
外部系统返回了什么?
下一步我该怎么办?
```

如果 DAG 引擎只能显示:

```text
Task failed.
```

那它不是生产可用平台,只是黑盒执行器。

可观测性要解决的是:

```text
用户和平台工程师能不能基于证据链定位问题,而不是靠猜。
```

## 三类证据链

生产排障至少需要三类证据。

| 证据 | 回答什么问题 |
|---|---|
| 状态证据 | 谁从什么状态变成什么状态,为什么 |
| 执行证据 | worker 做了什么,调用了什么外部系统,结果是什么 |
| 操作证据 | 哪个人做了暂停、重跑、跳过、标成功 |

缺任何一类都会很痛。

例如只有 task log,没有状态变更历史,你可能知道报错了,但不知道它为什么从 `FAILED` 又变成了 `READY`。

只有当前 status,没有 operation audit,你就不知道是谁手动跳过了节点。

## 最低限度需要哪些表

生产级 DAG 引擎建议至少有:

```text
workflow_instance
task_instance
task_execution_attempt
task_state_transition
workflow_operation_audit
external_call_log
```

如果支持动态 DAG,还需要:

```text
runtime_dependency
fanout_group
message_lineage / generated_task_log
```

如果支持 outbox / callback,还需要:

```text
outbox_event
inbox_event
```

## workflow_instance 应该能回答什么

`workflow_instance` 不只是存一个状态。

它应该能回答:

```text
这个 run 属于哪个 workflow definition version?
是谁触发的?
什么时候开始?
什么时候结束?
当前状态是什么?
失败原因是什么?
是否被暂停/取消?
当前卡在哪个阶段?
```

建议字段:

```text
id
workflow_definition_id
workflow_definition_version_id
status
trigger_type
triggered_by
input_ref
started_at
ended_at
current_phase
failure_reason
created_at
updated_at
```

## task_instance 应该能回答什么

`task_instance` 是排查的核心。

它应该能回答:

```text
这个任务属于哪个 run?
它的 task_key 是什么?
当前状态是什么?
它依赖谁?
谁依赖它?
它什么时候 ready?
什么时候被领取?
谁领取的?
尝试了几次?
下一次 retry 是什么时候?
输入输出在哪里?
```

建议字段:

```text
id
workflow_instance_id
task_key
task_type
status
trigger_rule
attempt
max_retries
scheduled_at
ready_at
started_at
ended_at
worker_id
lease_until
input_ref
output_ref
blocked_reason
error_type
error_message
created_at
updated_at
```

`blocked_reason` 很有价值。它能直接回答:

```text
为什么这个任务没跑?
```

例如:

```text
waiting_for_parent:B
tenant_quota_exceeded
external_pool_full:databricks
workflow_paused
scheduled_in_future
open_fanout_group
```

## 状态变更历史

只存当前状态是不够的。

必须记录:

```text
from_status
to_status
reason
operator
source_component
trace_id
created_at
```

示例:

| from | to | reason | source |
|---|---|---|---|
| READY | RUNNING | claimed by worker | scheduler |
| RUNNING | RETRYING | HTTP 502 retryable | worker |
| RETRYING | READY | retry delay reached | retry manager |
| FAILED | READY | manual rerun | user |

有了这张表,才能排查:

```text
任务为什么又跑了一次?
谁把失败任务改成 READY?
状态是不是被旧 worker 覆盖?
```

## attempt 日志

每次尝试都应该有 attempt 记录。

字段:

```text
task_instance_id
attempt_no
worker_id
status
started_at
ended_at
duration_ms
idempotency_key
attempt_request_id
external_system
external_request_id
external_trace_id
error_type
error_message
log_ref
```

不要只把错误写到 task 当前字段里。否则多次 retry 后,前几次失败原因会被覆盖。

## 外部调用 trace

DAG 引擎经常只是编排层,真正执行在外部系统。

因此要把外部调用串起来:

```text
task_instance_id
attempt_id
external_system
operation
request_id
idempotency_key
trace_id
status_code
response_summary
started_at
ended_at
```

这样用户问:

```text
Databricks job 到底有没有提交?
第三方 API 返回了什么?
外部系统里对应的 request id 是哪个?
```

平台能回答。

## 操作审计

人工操作必须审计。

常见操作:

- pause workflow。
- resume workflow。
- cancel workflow。
- rerun failed task。
- rerun from task。
- skip task。
- mark task success。
- modify parameter and continue。
- approve / reject manual step。

审计字段:

```text
id
workflow_instance_id
task_instance_id
operation
operator
old_status
new_status
reason
request_payload_ref
created_at
```

没有操作审计,生产事故会说不清:

```text
到底是系统自动跳过的,还是某个人手动跳过的?
为什么这个失败节点后来成功了?
是谁取消了 workflow?
```

## 指标体系

日志解决“单个问题怎么查”,指标解决“系统整体是否健康”。

关键指标:

| 指标 | 含义 |
|---|---|
| workflow started / succeeded / failed count | 流程吞吐和成功率 |
| task ready count | 等待调度数量 |
| task running count | 正在执行数量 |
| task retrying count | 重试压力 |
| task blocked count | 卡住任务数量 |
| scheduler claim success / fail count | 调度器领取效率 |
| queue delay | 从 ready 到 running 的时间 |
| execution duration | 任务执行时长 |
| retry rate | 重试比例 |
| timeout count | 超时数量 |
| zombie task count | 僵尸任务数量 |
| external error rate | 外部系统错误率 |
| quota rejected count | 因资源限制未调度数量 |

尤其要区分:

```text
排队耗时 queue_delay
执行耗时 execution_duration
端到端耗时 workflow_duration
```

否则用户说“跑慢了”,你不知道是调度慢、执行慢,还是外部系统慢。

## 告警设计

告警不要只看失败。

需要关注:

- workflow failure rate 突增。
- retry rate 突增。
- READY 积压过高。
- queue delay 超阈值。
- RUNNING 超时。
- zombie task 出现。
- scheduler 没有心跳。
- worker 心跳下降。
- external system error rate 高。
- outbox event 长时间未发送。
- callback inbox 堆积。

告警要带上下文:

```text
workflow_id
workflow_instance_id
task_key
task_instance_id
tenant
error_type
external_system
trace_id
last_transition
suggested_action
```

不要发这种告警:

```text
DAG failed.
```

更好的告警:

```text
Workflow customer_sync run 2026-05-13 failed at task LOAD_CUSTOMERS.
Error: Databricks job timeout.
External job id: xxx.
Attempt: 3/3.
Last transition: RUNNING -> TIMEOUT.
Suggested action: check Databricks cluster saturation or rerun from LOAD_CUSTOMERS.
```

## UI 排障体验

DAG 引擎非常依赖 UI。

用户至少需要看到:

- DAG 图。
- 节点状态。
- 当前 blocked reason。
- 运行历史。
- 状态流转。
- task logs。
- retry 记录。
- 输入输出引用。
- 外部 trace id。
- 上下游依赖。
- 人工操作按钮。

大 DAG UI 要特别设计:

| 问题 | 方案 |
|---|---|
| 几千个节点渲染卡顿 | 分层折叠、虚拟渲染、按状态过滤 |
| 动态节点太多 | fan-out group 聚合展示 |
| 子 workflow 展示复杂 | 折叠子图 |
| 用户只想找失败 | failed / running / blocked 快捷过滤 |
| 历史版本难比较 | 显示 definition version 和 runtime graph |

一个好的 UI 应该让用户 30 秒内回答:

```text
失败在哪?
为什么失败?
能不能重跑?
从哪里重跑?
重跑会影响哪些下游?
```

## 日志分层

不要把所有日志都塞数据库。

推荐:

| 内容 | 存储 |
|---|---|
| 状态和元数据 | DB |
| 大日志全文 | 日志系统 / object storage |
| 大输出结果 | object storage |
| trace | tracing system |
| 指标 | metrics system |

DB 里存引用:

```text
log_ref
output_ref
raw_error_ref
```

这样避免:

- task_instance 表膨胀。
- 查询变慢。
- 日志清理困难。
- 大 JSON 拖垮数据库。

## 敏感信息处理

任务参数里可能有:

- token。
- 数据库连接。
- 客户 ID。
- 账户信息。
- 文件路径。
- secret 引用。

规则:

```text
secret 不落明文;
日志默认脱敏;
UI 按权限展示;
审计记录谁查看了敏感信息;
错误消息不能泄露密钥。
```

参数快照要区分:

| 类型 | 处理 |
|---|---|
| 普通参数 | 可存 DB |
| 大参数 | 存对象存储引用 |
| 敏感参数 | 存 secret ref,不存明文 |
| 派生参数 | 记录来源和解析后摘要 |

## 标准排查问题模板

### 为什么没跑

看:

1. workflow 是否 running / paused / cancelled。
2. task 是否 ready。
3. parents 是否满足 trigger rule。
4. scheduled_at 是否在未来。
5. blocked_reason 是什么。
6. resource quota 是否满。
7. scheduler 是否领取失败。
8. task 是否有 lease。

### 为什么跑慢

拆分:

```text
workflow_duration = queue_delay + execution_duration + retry_delay + waiting_time
```

看:

1. ready 到 running 等了多久。
2. running 执行多久。
3. retry 延迟多久。
4. 是否等待审批或事件。
5. 是否外部系统慢。
6. 是否资源池满。

### 为什么失败

看:

1. error_type。
2. task attempt 日志。
3. external_call_log。
4. state_transition。
5. retry policy。
6. 上游输出。
7. 参数快照。

### 为什么重复执行

看:

1. attempt 记录。
2. state transition。
3. lease 是否过期。
4. worker 是否宕机。
5. MQ / callback 是否重复。
6. manual operation audit。
7. idempotency_key 是否稳定。

## 面试总结

可以这样讲:

```text
生产级 DAG 引擎必须把可观测性当成核心能力。
用户真正关心的是为什么没跑、为什么失败、谁操作过、用了什么参数、外部系统返回了什么。
因此系统至少要保留 workflow_instance、task_instance、task_execution_attempt、task_state_transition、operation_audit 和 external_call_log。
状态不能只存当前值,必须保留 from/to/reason/operator/source_component。
大日志和大输出不能塞 DB,要用 log_ref/output_ref 指向外部存储。
UI 要支持按失败、阻塞、运行中、重试中过滤,并展示 blocked reason、trace id、retry history 和人工操作记录。
没有这些证据链,系统一进生产就会变成黑盒。
```
