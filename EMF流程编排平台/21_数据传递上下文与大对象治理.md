# 21 数据传递、上下文与大对象治理

## 为什么数据传递是 DAG 引擎的难点

DAG 调度解决的是:

```text
哪个节点依赖哪个节点。
```

但生产工作流真正关心的是:

```text
下游节点到底拿到了什么数据。
这些数据是不是上游成功产出的。
这些数据是不是本次 run 应该使用的版本。
这些数据太大时放在哪里。
这些数据重跑时要不要复用。
这些数据如果敏感,能不能被日志和 UI 泄露。
```

所以 DAG 引擎里有两种依赖一定要分开:

| 依赖类型 | 含义 | 例子 |
|---|---|---|
| 结构依赖 | 节点执行顺序依赖 | `D` 必须等 `A/B/C` 完成 |
| 数据依赖 | 下游读取上游产物 | `D` 读取 `B.output.customer_ids` |

结构依赖不等于数据依赖。

例如:

```text
A -> D
B -> D
C -> D
```

只能说明 `D` 要等 `A/B/C` 到达某种终态,不能自动说明 `D` 拿到的数据一定完整、正确、可读。

如果 `B` 只是提交了一个外部 job,但 job 还没真正产出结果,就被标记为 `COMPLETE`,那么 `D` 虽然在结构上 ready,但在数据上还不 ready。

这就是很多生产事故的根源。

## 在 EMF 里怎么理解数据传递

EMF 的核心抽象是 `Message`。

一条 `Message` 至少承载两类信息:

```text
调度元数据:
command
run_uuid
batch_id
order_id
parents

业务参数:
parameters
parameter_object
workflow task 参数
外部参数
上下文参数
```

调度元数据回答:

```text
这条消息属于哪个 run?
它在 DAG 里是谁?
它依赖谁?
它要执行哪个 command?
```

业务参数回答:

```text
这个 command 执行时需要什么输入?
这些输入从哪里解析出来?
```

所以 `Message` 不是普通队列消息。

它更像:

```text
任务实例 + DAG 节点 + 命令调用参数 + 上下文索引
```

## 不要把所有东西都塞进 Message

一个常见错误是:

```text
上游输出什么,就直接塞进下游 message.parameters。
```

小数据这样做可以,但大数据会出问题。

例如:

```text
SQL 查询结果 500MB
文件列表 20 万条
模型输出大 JSON
数据质量报告全文
外部 API 返回的大 payload
```

如果这些都塞进消息或数据库字段,会带来:

```text
消息体过大
数据库膨胀
调度扫描变慢
序列化成本高
日志泄露风险
重试时复制大对象
UI 无法展示
```

更合理的分层是:

| 数据类型 | 存放位置 |
|---|---|
| 调度元数据 | DB / 内存 DAG |
| 小参数 | message parameters / DB JSON |
| 小输出 | task output JSON |
| 大输出 | object storage / file store / table |
| 日志全文 | log system / object storage |
| 敏感信息 | secret reference |
| 数据血缘 | lineage table / catalogue |

面试里可以说:

```text
调度系统只应该持有“引用”和“元数据”,不应该承载大业务数据本体。
```

## 输出契约比输出内容更重要

每个节点最好有清晰的输出契约。

例如一个任务不是简单说:

```text
我成功了。
```

而是要说清楚:

```text
我产出了什么。
产物在哪里。
产物对应哪个 run。
产物是否完整。
产物 schema 是什么。
产物能不能被下游消费。
```

可以抽象成:

```text
output_ref:
  type: dataset | file | table | api_result | scalar
  uri: ...
  schema_version: ...
  content_hash: ...
  row_count: ...
  partition: ...
  produced_by_task: ...
  produced_at: ...
  visibility: public | restricted | secret
```

`SUCCESS` 最好意味着:

```text
任务的副作用已经完成,并且输出契约已经满足。
```

而不是:

```text
任务已经把请求发出去了。
```

这是区分“submit 成功”和“执行成功”的关键。

## 对外部计算任务要拆成 submit + monitor

很多 EMF 类工作流会调用 Databricks、Spark、SQL Engine、Storage、Catalogue 或外部 API。

这类任务常见误区是:

```text
调用外部系统提交 job 成功 -> 任务标记 COMPLETE
```

但这通常是错的。

提交成功只说明:

```text
外部系统收到了任务。
```

不代表:

```text
外部任务执行完成。
输出已经生成。
输出可以被下游读取。
```

更稳的设计是:

```text
SUBMITTED
  -> RUNNING
  -> OUTPUT_VALIDATING
  -> COMPLETE
```

或者把它拆成两个节点:

```text
SubmitJob -> MonitorJob -> Downstream
```

其中 `MonitorJob` 负责:

```text
轮询外部 job 状态
确认外部 job terminal state
校验输出是否存在
校验 row count / manifest / marker file
写入 output_ref
```

这样下游节点判断 ready 时,看到的 `COMPLETE` 才有数据语义。

## ABC 有数据依赖时应该怎么建模

假设有四个节点:

```text
A
B
C
D depends on A/B/C
```

如果 `A/B/C` 三个节点互相没有数据依赖,只是并行产出不同输入给 `D`,可以建模为:

```text
A -> D
B -> D
C -> D
```

但如果 `B` 依赖 `A` 的输出,`C` 又依赖 `B` 的输出,那就不能只写:

```text
A -> D
B -> D
C -> D
```

否则调度器可能会把 `A/B/C` 当成并行节点。

正确方式是把真实数据依赖编码进 DAG:

```text
A -> B -> C -> D
```

如果 `D` 同时还要显式等待 `A` 和 `B` 的输出,也可以写:

```text
A -> B -> C -> D
A ---------> D
B ---------> D
```

但要注意,后两条边是否必要取决于你的语义。

如果 `C` 的输出已经包含了 `A/B` 的结果摘要,`D` 只依赖 `C` 即可。

面试里可以这样讲:

```text
DAG 的边应该表达真实的执行和数据依赖。不能为了“看起来都汇总到 D”而把有顺序关系的 A/B/C 建成并行父节点,否则 ready 判断会早于真实数据可用时间。
```

## 参数解析的优先级

一个任务的输入参数可能来自很多地方:

```text
message 自身字段
message.parameters
workflow task 模板参数
run 级参数
batch 级参数
上游 output_ref
外部配置
secret manager
默认值
```

这时必须定义稳定的优先级。

例如:

```text
message 显式参数
  > workflow task 参数
  > run 参数
  > batch 参数
  > default 参数
```

如果优先级不清楚,生产上会出现:

```text
同一个 workflow 今天跑出来和昨天不一样。
重跑时读到了新的全局参数。
用户以为覆盖了参数,实际没有生效。
下游读到了错误的分区。
```

参数解析最好记录一份快照:

```text
resolved_parameters:
  key: value
  source: message | run | batch | secret | upstream
  resolved_at: ...
```

这样排查时才能回答:

```text
这个参数到底是从哪里来的?
```

## 上下文应该是可追溯的

上下文不应该是一个随手可改的全局字典。

更好的理解是:

```text
Execution Context = 本次 run 中任务可见的、可审计的运行时事实集合。
```

它至少包括:

```text
run_uuid
batch_id
workflow definition version
trigger input
resolved parameters
upstream output refs
runtime variables
secret refs
attempt metadata
```

如果上下文可以被任务随意修改,会有几个问题:

```text
并发任务互相覆盖变量
重跑时上下文状态不一致
审计时无法还原当时输入
失败后不知道哪个任务改坏了变量
```

所以更稳的方式是:

```text
上下文读多写少。
任务输出通过 output_ref 显式发布。
下游通过依赖关系和参数解析引用上游 output_ref。
所有上下文变更要有版本和来源。
```

## 大对象治理

对于大数据量场景,推荐使用 manifest 模式。

例如上游任务处理 10000 个文件,不要把文件列表直接塞给下游。

可以输出:

```text
manifest_ref = s3://bucket/run_uuid/task_key/manifest.json
```

manifest 中记录:

```text
files:
  - uri
  - size
  - checksum
  - partition
  - status
total_count
total_bytes
schema_version
created_at
```

下游只拿 `manifest_ref`,再按需读取。

这样有几个好处:

```text
消息体稳定
DB 不会爆
可以分页读取
可以做 checksum
可以支持局部重跑
可以审计本次 run 到底处理了哪些对象
```

## 数据版本和重跑语义

重跑时最容易出现的问题是:

```text
下游到底用旧输出,还是新输出?
```

例如:

```text
A -> B -> C
```

`B` 失败后重跑。

如果 `A` 的数据源已经变化,`B` 是读取:

```text
A 第一次运行产出的 output_ref
```

还是读取:

```text
A 当前最新 output_ref
```

这必须有明确策略。

常见策略:

| 策略 | 含义 | 适用场景 |
|---|---|---|
| reuse upstream output | 复用上游已成功产物 | 修复下游逻辑,保持输入不变 |
| refresh from node | 从某个节点开始重新产出 | 上游数据或逻辑也要更新 |
| new workflow run | 创建全新 run | 希望完整重算 |
| partial rerun with snapshot | 局部重跑,但绑定原始输入快照 | 审计要求强 |

建议面试时强调:

```text
重跑不是简单把 FAILED 改成 READY,而是要定义输入快照和输出覆盖策略。
```

## 敏感数据不要进入日志

DAG 引擎里经常会有:

```text
数据库连接串
access token
客户 ID
账户信息
审批备注
业务 payload
```

这些不能直接进入:

```text
message body
resolved parameter snapshot
worker log
UI task detail
error message
audit log
```

合理设计是:

```text
参数里存 secret_ref
执行时从 secret manager 解析
日志打印前做脱敏
UI 按权限展示
审计记录谁访问过敏感参数
```

例如:

```text
db_password = secret://project/prod/db_password
```

而不是:

```text
db_password = plain_text_password
```

## 常见生产问题

### 1. 下游读取不到上游输出

常见原因:

```text
上游只提交了外部 job,还没真正完成
上游输出没有写 output_ref
output_ref 写错分区
下游参数解析没有引用上游 output_ref
对象存储写入延迟
权限不足
```

排查顺序:

```text
看上游是否真正 COMPLETE
看上游 attempt 是否有 external job terminal state
看 output_ref 是否存在
看 output_ref 是否属于同一个 run_uuid
看下游 resolved_parameters
看存储权限和对象存在性
```

设计解决:

```text
submit 和 monitor 分离
COMPLETE 前做输出校验
output_ref 加 run_uuid/task_key/attempt
resolved parameter snapshot
大对象用 manifest
```

### 2. 下游读到了旧数据

常见原因:

```text
重跑时复用了旧 output_ref
参数解析读了 latest 而不是当前 run snapshot
上游输出路径没有版本化
缓存没有失效
```

设计解决:

```text
output_ref 路径带 run_uuid 和 attempt
禁止默认读取 latest
重跑策略显式选择 reuse 或 refresh
resolved_parameters 持久化
```

### 3. Message 太大

常见表现:

```text
队列投递失败
DB 写入慢
consumer 反序列化慢
scheduler 扫描慢
UI 打不开
```

设计解决:

```text
消息只放引用
大对象进入对象存储
日志和结果分开存
manifest + pagination
```

### 4. 参数来源说不清

常见表现:

```text
用户说改了参数但没生效
同一个 workflow 多次运行结果不同
排查时不知道某个值来自哪里
```

设计解决:

```text
参数优先级规则文档化
解析结果快照化
记录每个参数 source
UI 展示 resolved parameters 和 masked raw parameters
```

## 面试总结

可以这样说:

```text
DAG 引擎里的数据传递不能只理解成把变量从上游传给下游。生产上要区分结构依赖和数据依赖,任务成功必须包含输出契约满足,尤其是调用外部计算系统时不能把 submit 成功当成业务完成。小参数可以放在 Message 或 DB,大对象应该用 output_ref、manifest、对象存储和数据血缘管理。重跑时还要明确复用旧输出还是刷新上游输出,否则会产生不可复现的结果。一个可靠设计应该记录 resolved parameter snapshot、output_ref、schema version、content hash、secret reference 和 lineage,这样下游消费、重跑和排障才有依据。
```

最关键的一句话:

```text
DAG 的边决定什么时候能跑,output contract 决定跑完以后下游能不能安全消费。
```
