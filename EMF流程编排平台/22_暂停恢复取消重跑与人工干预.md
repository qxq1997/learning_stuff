# 22 暂停、恢复、取消、重跑与人工干预

## 为什么人工干预很难

生产里的 DAG 引擎一定会被用户要求支持:

```text
暂停 workflow
恢复 workflow
取消 workflow
取消某个 task
重跑失败节点
从某个节点开始重跑
跳过某个节点
手动标记成功
修复参数后继续
人工审批后继续
```

这些功能看起来是 UI 上几个按钮,但本质上是在改变分布式状态机。

如果设计不好,会出现:

```text
用户点了取消,但外部 job 还在跑
用户点了暂停,但新任务继续被调度
用户手动标成功,下游读不到输出
用户重跑了中间节点,下游却复用了旧结果
两个管理员同时操作,状态互相覆盖
审计里看不出是谁改了状态
```

所以人工干预不能理解成:

```text
直接 update status。
```

而应该理解成:

```text
带权限、校验、审计、状态约束和副作用处理的操作命令。
```

## 人工操作也要走状态机

一个常见错误是给管理员开放:

```sql
UPDATE task_instance SET status = 'SUCCESS'
```

这会绕过:

```text
状态合法性校验
输出契约校验
下游状态重算
审计日志
并发控制
外部资源清理
```

更好的方式是把人工操作也建模成命令:

```text
PauseWorkflow
ResumeWorkflow
StopWorkflow
CancelWorkflow
TerminateWorkflow
RerunWorkflow
RetryTask
RerunFromTask
SkipTask
MarkTaskSuccess
ApproveTask
RejectTask
```

每个命令都要做:

```text
权限检查
当前状态检查
目标状态计算
副作用处理
CAS 状态更新
状态变更日志
操作审计
必要时触发重新调度
```

面试里可以说:

```text
人工操作不是状态机的例外,而是状态机的一部分。
```

## Pause 的语义

暂停看起来简单,实际要先定义清楚:

```text
暂停是停止调度新任务,还是要停止正在运行的任务?
```

通常有两种模式。

| 模式 | 含义 | 适用场景 |
|---|---|---|
| soft pause | 不再调度新 ready task,已运行任务继续 | 大多数生产场景 |
| hard pause | 尝试暂停或取消正在运行任务 | 外部任务可控且支持暂停 |

多数 DAG 引擎更适合 soft pause。

因为很多外部系统不一定支持暂停,比如:

```text
SQL 查询
Databricks job
HTTP API
文件上传
第三方审批
```

soft pause 的行为可以定义为:

```text
workflow_instance.status = PAUSED
READY 任务不再被 worker 领取
RUNNING 任务继续执行
RUNNING 任务完成后只更新状态,不继续推进下游调度
Resume 后重新扫描 ready 节点
```

关键点:

```text
PAUSED 不是 task 的终态,而是 workflow 调度门禁。
```

## Resume 的语义

恢复不是简单把:

```text
PAUSED -> RUNNING
```

它还要重新计算:

```text
哪些节点已经完成
哪些节点已经失败
哪些节点处于 ready
哪些节点因为上游失败被 blocked
哪些节点 lease 已过期
哪些动态节点已经生成
```

Resume 的推荐流程:

```text
校验 workflow 当前是 PAUSED
恢复 workflow status
扫描所有非终态 task
处理 lease 过期任务
重新计算 ready set
触发 scheduler
写 operation audit
```

如果系统支持动态 DAG,恢复时还要确认:

```text
run 是否仍允许 dynamic expansion
fanout group 是否已经 sealed
completion token 是否已经满足
```

## Cancel 的语义

取消比暂停更复杂。

取消至少分三层:

| 层级 | 含义 |
|---|---|
| workflow cancel | 用户不再需要整个 run |
| task cancel | 取消某个 task |
| external cancel | 尝试取消外部 job / API / query |

一个合理的 workflow cancel 流程:

```text
workflow_instance -> CANCELLING
停止调度新任务
READY / WAITING 任务 -> CANCELLED
RUNNING 任务发送 cancel signal
支持取消的外部 job 调用 cancel API
不支持取消的任务等待自然结束或超时
所有任务进入终态后 workflow -> CANCELLED
```

为什么需要 `CANCELLING`,而不是直接 `CANCELLED`?

因为取消是异步过程。

你发出取消命令时,外部任务可能还在运行。

如果直接标记 workflow 为 `CANCELLED`,用户会以为已经停止,但实际上外部系统还在消耗资源或产生副作用。

## 取消不是一定能成功

有些任务无法真正取消:

```text
已经发出的 HTTP 请求
正在执行的数据库事务
第三方系统内部任务
没有 cancel API 的外部 job
已经完成副作用的任务
```

所以取消语义要分清:

| 状态 | 含义 |
|---|---|
| CANCEL_REQUESTED | 平台已请求取消 |
| CANCELLING | 正在取消外部资源 |
| CANCELLED | 确认取消完成 |
| CANCEL_FAILED | 外部取消失败 |
| COMPLETED_AFTER_CANCEL | 请求取消后任务仍完成 |

简单系统可以不一定引入这么多状态,但至少要在审计和 attempt 里记录:

```text
cancel_requested_at
cancel_requested_by
external_cancel_result
final_status_after_cancel
```

## Retry、Rerun、Resume 不一样

这三个词很容易混。

| 操作 | 含义 |
|---|---|
| retry | 同一个 task 因失败自动再试一次 |
| rerun | 用户主动重新执行某个 task 或一段 DAG |
| resume | workflow 暂停后继续推进 |

`retry` 通常发生在同一个 task instance 内:

```text
attempt = attempt + 1
status FAILED -> RETRYING -> READY
```

`rerun` 更像人工修复后的重新执行:

```text
用户选择一个节点
系统重置这个节点和必要下游
生成新的 attempt 或新的 task instance revision
重新调度
```

`resume` 不应该改变已完成任务。

它只是:

```text
让调度器继续推进当前 workflow。
```

## Workflow 级操作的实现骨架

暂停、停止、重跑 workflow 时,不要把它理解成 UI 直接改几张表。

更稳的实现方式是:

```text
用户操作
  -> WorkflowOperation command
  -> 权限校验
  -> workflow 当前状态校验
  -> CAS/lock 更新 workflow 控制状态
  -> 批量调整受影响 task
  -> 写 state transition 和 audit log
  -> 唤醒 scheduler / cancel dispatcher / timeout scanner
```

也就是说,人工操作应该先落成一条可审计、可重试、可幂等的操作命令。

一个常见的数据结构可以是:

```text
workflow_operation
  operation_id
  workflow_instance_id
  operation_type       # PAUSE / RESUME / STOP / CANCEL / RERUN
  operator
  reason
  request_id           # 防重复点击
  expected_status
  payload_json         # rerun policy、参数覆盖、起始节点等
  status               # ACCEPTED / APPLIED / FAILED
  created_at
  applied_at
```

`workflow_instance` 上至少要有这些控制字段:

```text
status                 # RUNNING / PAUSED / STOPPING / STOPPED / CANCELLING / CANCELLED / SUCCESS / FAILED
schedule_paused        # 是否允许继续调度新节点
stop_requested_at
cancel_requested_at
operation_version      # CAS 防止并发人工操作互相覆盖
definition_version_id  # 重跑时决定用旧版本还是新版本
```

核心不变量是:

```text
只有 workflow 处于可调度状态时,task 才能从 READY 进入 RUNNING。
```

所以 scheduler 的领取条件不能只看 task:

```text
task.status = READY
```

还必须看 workflow:

```text
workflow.status = RUNNING
workflow.schedule_paused = false
workflow not in STOPPING/CANCELLING/CANCELLED
```

worker 真正执行前也要再查一次,因为 task 可能刚被领取,用户就点了暂停或取消。

```text
worker claimed task
  -> recheck workflow gate
  -> recheck task status and attempt
  -> execute
```

这能避免:

```text
暂停后仍然启动新任务
取消后 worker 继续执行旧任务
旧 attempt 回写覆盖新状态
```

## PauseWorkflow 怎么实现

`PauseWorkflow` 的推荐语义是 soft pause:

```text
不再调度新的 ready task,已经 RUNNING 的 task 继续执行。
```

实现步骤:

```text
1. API 接收 pause 请求,生成 operation_id/request_id。
2. 校验用户是否有 pause 权限。
3. 校验 workflow 当前是 RUNNING 或可暂停状态。
4. 用 CAS 把 workflow 标记为 PAUSED,并设置 schedule_paused = true。
5. scheduler 从领取条件里排除 PAUSED workflow。
6. 已 RUNNING task 正常完成,但完成后不继续推进下游调度。
7. 写 workflow_operation_audit 和 workflow_state_transition。
```

这里最容易忽略的是第 6 点。

如果 `B` 在暂停前已经运行:

```text
A -> B -> C
```

暂停后 `B` 可以完成,但 `C` 不应该立刻被调度。

所以 task 完成后的回调逻辑也要检查:

```text
workflow 是否仍然允许调度下游。
```

如果不检查,就会出现:

```text
用户明明点了 pause,但 workflow 还在继续往下跑。
```

## ResumeWorkflow 怎么实现

`ResumeWorkflow` 不是简单把 `PAUSED` 改回 `RUNNING`。

它要重新做一次调度视角的恢复:

```text
1. 校验 workflow 当前是 PAUSED。
2. CAS 更新 workflow.status = RUNNING, schedule_paused = false。
3. 扫描当前 run 里所有非终态 task。
4. 处理 lease 过期的 RUNNING task。
5. 重新计算 ready set。
6. 对 ready task 重新入队或让 scheduler 下轮扫描捞起。
7. 写审计日志。
```

为什么要重新计算 ready set?

因为暂停期间可能发生:

```text
RUNNING task 完成
外部 callback 返回
人工 skip / mark success
timeout scanner 标记任务超时
动态 DAG 生成了一部分节点
```

所以 resume 的本质不是“继续线程”,而是:

```text
基于当前持久化状态重新推导下一批可执行节点。
```

## StopWorkflow、CancelWorkflow、Terminate 的区别

面试或设计评审里,最好先把“停止”拆开。

| 操作 | 语义 | 是否可恢复 | 对 RUNNING task 的处理 |
|---|---|---|---|
| Pause | 暂停调度新节点 | 可恢复 | 继续跑 |
| Graceful Stop | 不再推进后续节点,等当前运行节点结束 | 通常不可恢复 | 继续跑到终态 |
| Cancel | 尝试取消整个 workflow | 通常不可恢复 | 发送 cancel signal |
| Force Terminate | 控制面直接终止追踪 | 不可恢复 | 可能无法真正停止外部副作用 |

很多系统口头说的“停止 workflow”,实际是 `CancelWorkflow`。

但如果要设计得更细,可以把它们分成:

```text
StopWorkflow       # 优雅停止: 不再调度下游,已运行任务自然结束
CancelWorkflow     # 主动取消: 尝试停止 RUNNING 外部任务
TerminateWorkflow  # 紧急终止: 控制面不再等待,只保留审计和告警
```

## StopWorkflow 怎么实现

`StopWorkflow` 更适合表达:

```text
当前已经开始的任务可以结束,但不要再启动新任务。
```

实现步骤:

```text
1. workflow RUNNING/PAUSED -> STOPPING。
2. 设置 schedule_paused = true,关闭新任务调度。
3. CREATED/READY/WAITING 的 task 标记为 STOPPED 或 CANCELLED。
4. RUNNING task 继续跑到终态。
5. RUNNING task 完成后,不再唤醒下游。
6. 所有 task 进入终态后,workflow -> STOPPED。
```

这个操作适合:

```text
用户发现参数错了,但不想强杀已经跑出去的外部 job
某批次不再需要继续往下游推进
外部系统降级,希望让已运行任务自然收尾
```

需要注意:

```text
STOPPING 是中间态,STOPPED 才是终态。
```

如果 workflow 里还有 RUNNING task,就不能立刻标成 `STOPPED`。

## CancelWorkflow 怎么实现

`CancelWorkflow` 是更强的停止。

推荐流程:

```text
1. workflow RUNNING/PAUSED -> CANCELLING。
2. 关闭新任务调度。
3. CREATED/READY/WAITING task -> CANCELLED。
4. RUNNING task -> CANCEL_REQUESTED。
5. 对有 external_job_id 的任务调用外部 cancel API。
6. cancel dispatcher / polling scanner 持续确认外部状态。
7. 外部任务确认取消后,task -> CANCELLED。
8. 不支持取消的任务等待自然结束或超时。
9. 所有 task 到终态后,workflow -> CANCELLED。
```

这里要有两个关键设计。

第一,取消是异步的。

```text
用户点 cancel != 外部任务已经停止。
```

所以 workflow 应该先进入:

```text
CANCELLING
```

UI 上也应该展示:

```text
正在取消 3 个运行中任务,其中 1 个外部 job 等待确认。
```

第二,worker 回写结果时要尊重取消状态。

比如用户 cancel 后,外部 job 还是成功返回了:

```text
task was CANCEL_REQUESTED
external job returned SUCCESS
```

这时不要无脑改成普通 `SUCCESS` 并推进下游。

更合理的是记录成:

```text
COMPLETED_AFTER_CANCEL
```

或者:

```text
SUCCESS, but workflow is CANCELLING, do not schedule downstream
```

关键是:

```text
取消后的完成结果不能继续驱动 workflow 往下走。
```

## ForceTerminate 为什么要慎用

`ForceTerminate` 只适合极端场景:

```text
调度器状态严重不一致
外部系统长期无响应
用户确认不再等待任何结果
运维需要释放控制面资源
```

它通常只能保证:

```text
平台不再调度、不再等待、不再推进这个 workflow。
```

但不能保证:

```text
已经发出去的外部副作用被撤销。
```

所以 force terminate 必须要求:

```text
高权限
强制 reason
二次确认
记录未确认 external job 列表
触发告警或后续 reconcile
```

面试里可以说:

```text
Terminate 是控制面的终止,不等于数据面副作用被回滚。
```

## RerunWorkflow 怎么实现

workflow 级重跑有两种主流实现。

| 方案 | 做法 | 优点 | 代价 |
|---|---|---|---|
| 新建 run | 创建新的 workflow_instance 和 run_uuid | 审计清楚,不会污染旧 run | 占更多存储,UI 要展示 run lineage |
| 原地重跑 | 在原 workflow_instance 内重置部分 task | 适合失败修复和局部恢复 | 状态复杂,容易污染旧结果 |

生产上更推荐:

```text
整条 workflow 重跑 -> 新建 run。
局部节点重跑 -> 原 run 内 rerun selected task/downstream。
```

### 整条 workflow 重跑

整条重跑最好创建一个新的 workflow instance:

```text
1. 读取原 workflow_instance。
2. 决定使用哪个 definition_version。
3. 决定使用原 input snapshot,还是用户覆盖后的 input。
4. 创建新的 workflow_instance,生成新的 run_uuid。
5. 记录 rerun_of_workflow_instance_id。
6. 根据不可变 DAG 定义重新生成 task_instance。
7. 所有 output_ref 使用新 run_uuid/attempt namespace。
8. 启动 scheduler。
9. 在审计里记录谁因为什么重跑。
```

这里最关键的设计选择是:

```text
重跑用旧版本还是新版本?
```

建议默认使用原 workflow 绑定的:

```text
definition_version_id
```

这样重跑是为了复现或修复当时那次运行,不会被后来修改过的 DAG 定义影响。

如果用户明确选择用新版本重跑,也要在审计里记录:

```text
rerun_from_definition_version = v3
rerun_to_definition_version = v5
```

### 原地局部重跑

局部重跑适合:

```text
某个任务失败
修复参数后继续
某个外部系统短暂故障
只想重跑某个子图
```

实现步骤:

```text
1. 暂停或锁定当前 workflow,避免 scheduler 同时推进。
2. 选定重跑起点 task。
3. 计算受影响子图,通常是 selected task + downstream。
4. 校验受影响节点没有不可取消的 RUNNING task。
5. 将受影响节点旧结果标记为 INVALIDATED。
6. 增加 task attempt 或 task revision。
7. 为新 attempt 生成新的 output_ref namespace。
8. 重新解析参数,生成 resolved parameter snapshot。
9. selected task -> READY/CREATED。
10. 下游等待依赖重新满足。
11. 恢复 workflow 调度。
```

局部重跑最容易踩的坑是:

```text
只重跑了 B,但 C/D 继续使用旧输出。
```

所以一旦重跑上游节点,下游至少要进入:

```text
INVALIDATED
```

或者重新计算是否需要重跑。

## Rerun 的输出和 attempt 设计

重跑不应该覆盖旧执行记录。

推荐用:

```text
task_instance_id
attempt
output_ref
state_transition
```

共同表达一次执行。

例如:

```text
B attempt 1 -> output_ref=/run_1/B/attempt_1/output.json -> FAILED
B attempt 2 -> output_ref=/run_1/B/attempt_2/output.json -> SUCCESS
```

这样下游解析参数时必须明确使用:

```text
当前有效 attempt 的 output_ref。
```

不要使用:

```text
/run_1/B/latest
```

除非 `latest` 是一个受事务保护的指针,并且有版本号和审计。

面试里可以强调:

```text
重跑的本质不是把旧状态抹掉,而是追加一条新的执行历史,然后把当前有效指针切到新 attempt。
```

## 从失败节点重跑

假设:

```text
A -> B -> C -> D
```

`C` 失败了。用户修复参数后想从 `C` 重跑。

系统必须回答:

```text
B 的输出是否复用?
C 的旧输出是否清理?
D 的旧状态是否重置?
如果 D 之前已经跑过,是否需要重跑?
这次重跑算同一个 workflow run 还是新 run?
审计怎么记录?
```

推荐先定义几种 rerun policy。

| 策略 | 含义 |
|---|---|
| rerun failed only | 只重跑失败节点 |
| rerun downstream | 重跑选中节点及所有下游 |
| rerun selected subgraph | 只重跑用户选择的子图 |
| rerun with fresh upstream | 从某个上游节点开始刷新 |
| create new run | 创建全新 workflow instance |

对大多数系统来说,最常用的是:

```text
rerun selected task + downstream
```

因为下游可能已经消费了旧输出。

如果上游输出不变,可以复用:

```text
A/B output_ref
```

但 `C/D` 应该生成新的:

```text
attempt
output_ref
state transition
```

## 下游状态如何处理

还是这个图:

```text
A -> B -> C -> D
```

如果重跑 `B`,那么 `C/D` 的旧结果通常不再可信。

可以将下游重置为:

```text
INVALIDATED
```

或者:

```text
READY / CREATED
```

更清晰的方式是先进入:

```text
INVALIDATED
```

表示:

```text
这个节点之前的结果因为上游重跑而失效。
```

然后调度器重新计算后再变成:

```text
READY
```

这样审计更清楚。

如果系统状态枚举不想太复杂,至少要在审计里记录:

```text
downstream reset caused by rerun of B
```

## 手动跳过节点

手动跳过适合:

```text
非关键通知失败
某个可选分支不需要执行
外部系统已手动处理
人工确认可以忽略某个失败
```

但它有风险:

```text
下游需要这个节点的输出
跳过后 join 语义改变
审计不清楚为什么没执行
```

所以 skip 操作要校验:

```text
这个节点是否允许 skip
下游 trigger rule 是否接受 SKIPPED
是否需要替代 output_ref
是否要求填写 reason
```

例如:

```text
SKIPPED with reason = external process completed manually
```

比单纯 `SKIPPED` 更可追溯。

## 手动标记成功

手动标记成功是高风险操作。

因为 `SUCCESS` 往往不只是状态,还意味着:

```text
下游可以消费输出。
```

如果用户把一个没有输出的节点标成功,下游可能立刻失败。

所以 `MarkTaskSuccess` 应该要求:

```text
权限更高
必须填写 reason
必须确认 output contract
可选地提供 manual output_ref
记录 operator 和审批人
触发下游前做校验
```

可以设计成:

```text
MANUAL_SUCCESS
```

而不是普通 `SUCCESS`。

这样后续排查时能区分:

```text
系统执行成功
人工确认成功
```

如果不想增加状态,也至少要在 state transition 里记录:

```text
source = manual
operator = ...
reason = ...
```

## B 没执行完却被标记完成怎么办

这是一个非常典型的生产问题。

场景:

```text
A -> B -> D
```

`B` 实际还没执行完,却被标记 `COMPLETE`,导致 `D` 开始执行。

常见原因:

```text
B 调用外部 job 时只提交成功就标完成
worker 在异步任务未结束时提前返回成功
callback 语义错,收到 started 就当 succeeded
状态更新代码没有校验 output contract
人工误标成功
并发竞态导致旧 attempt 覆盖新 attempt 状态
```

排查顺序:

```text
看 B 的 state transition,谁把它改成 COMPLETE
看 B 的 attempt,外部 job 是否 terminal success
看 B 的 output_ref 是否生成且校验通过
看 B 是否有多个 attempt 互相覆盖
看 D 的 resolved parameters 是否引用 B 的输出
看是否有人工 MarkTaskSuccess 审计
```

设计解决:

```text
区分 SUBMITTED/RUNNING/OUTPUT_VALIDATING/COMPLETE
外部 job terminal success 后才能 COMPLETE
COMPLETE 前必须写 output_ref 或 completion marker
状态更新带 attempt_id/version CAS
旧 attempt 不能覆盖新 attempt
人工标成功必须提供输出或豁免说明
D ready 前校验父节点 output contract
```

最关键的是:

```text
complete 不是函数返回了,而是下游依赖的事实已经成立。
```

## 人工审批节点

人工审批节点不是普通任务。

它的状态通常是:

```text
WAITING_APPROVAL
APPROVED
REJECTED
TIMEOUT
CANCELLED
```

审批节点要记录:

```text
审批人
审批时间
审批意见
审批前参数
审批结果
是否超时自动拒绝/通过
```

审批完成后,再根据结果推进:

```text
APPROVED -> 下游 ready
REJECTED -> 失败或走 reject 分支
TIMEOUT -> 失败、重试或走兜底分支
```

不要把审批简单实现成:

```text
人工点按钮后把任务改成 SUCCESS
```

因为审批本身有业务语义。

## 并发人工操作

生产里可能两个用户同时操作同一个 workflow:

```text
用户 A 点 cancel
用户 B 点 rerun
```

或者:

```text
scheduler 正在把 task 改成 RUNNING
管理员正在 skip task
```

必须靠 CAS 或 version 防止覆盖:

```text
UPDATE task_instance
SET status = ?
WHERE id = ?
  AND status = expected_status
  AND version = expected_version
```

如果更新失败,说明状态已经变了,操作方要重新读取当前状态再决定。

这比加全局锁更常见,也更容易扩展。

## 操作审计

所有人工操作至少记录:

```text
operation
operator
target_type
target_id
old_status
new_status
reason
request_id
created_at
source_ip / user_agent
```

对于高风险操作,还可以记录:

```text
approval_id
second_operator
parameter_diff
affected_downstream_tasks
external_cancel_result
```

面试里可以强调:

```text
人工干预一定要审计,否则生产事故后无法区分是系统调度问题还是人为操作问题。
```

## 常见生产问题

### 1. 点了暂停但任务还在跑

先判断 pause 语义。

如果是 soft pause,这是正常的:

```text
已运行任务继续,新任务不再调度。
```

如果用户期望停止外部任务,需要:

```text
hard pause / cancel
```

设计上 UI 要展示:

```text
paused, running tasks will continue
```

而不是让用户误解。

### 2. 点了取消但外部 job 还在跑

排查:

```text
外部系统是否支持 cancel API
cancel API 是否调用成功
worker 是否还活着
external job id 是否记录
cancel 后是否继续 polling
```

设计解决:

```text
记录 external_job_id
workflow 先进入 CANCELLING
外部取消异步确认
失败时告警
不支持取消的任务标记 cancel_requested
```

### 3. 重跑后结果不一致

排查:

```text
重跑时是否复用了上游 output_ref
参数是否重新解析
外部数据源是否变化
下游旧结果是否清理
definition version 是否一致
```

设计解决:

```text
rerun policy 显式化
resolved parameter snapshot
output_ref 带 run/attempt
下游 invalidation
```

### 4. 手动标成功后下游失败

排查:

```text
手动成功是否提供 output_ref
下游需要哪些上游输出
join trigger rule 是否允许该状态
是否是 MANUAL_SUCCESS 被当作 SUCCESS
```

设计解决:

```text
MarkTaskSuccess 前做 output contract 校验
要求 reason 和 output override
下游 ready 前校验父节点产物
```

## 面试总结

可以这样说:

```text
生产级 DAG 引擎必须支持人工干预,但人工干预不能绕过状态机。暂停、恢复、停止、取消、重跑、跳过、手动标成功都应该是受控操作命令,包含权限校验、状态校验、CAS 更新、状态变更日志和操作审计。暂停一般只停止新任务调度,停止通常是不再推进下游并等待已运行任务收尾,取消则是异步过程,需要 CANCELLING 和外部 cancel 确认。重跑也不是简单把 FAILED 改回 READY,而是要定义新建 run 还是原地重跑、使用旧版本还是新版本、复用上游输出还是刷新输入、是否 invalid 下游、attempt 如何记录。对于手动标成功,还要校验 output contract,否则下游会在结构上 ready,但数据上不 ready。
```

最关键的一句话:

```text
人工按钮不是状态机的后门,而是状态机里权限更高、审计更强的一类事件。
```
