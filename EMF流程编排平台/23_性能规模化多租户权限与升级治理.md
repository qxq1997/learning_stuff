# 23 性能规模化、多租户权限与升级治理

## 为什么规模化是另一类难题

小规模 DAG 引擎看起来很好做:

```text
扫描 ready task
拿出来执行
更新状态
继续扫描
```

但生产规模上来后,问题会完全变样。

你会遇到:

```text
每天几万个 workflow run
每天几十万到几百万 task instance
单个 DAG 几千个节点
动态 fan-out 生成海量子任务
大量 RUNNING 任务需要轮询外部状态
大量日志和状态变更写入
多个租户争抢资源
一个大客户把全局队列堵死
升级时还有大量 run 正在执行
```

此时 DAG 引擎不再只是调度器,而是:

```text
资源调度系统 + 多租户平台 + 高写入状态系统 + 可演进协议系统。
```

## 性能瓶颈通常在哪里

生产级 DAG 引擎常见瓶颈:

| 瓶颈 | 表现 |
|---|---|
| DB 状态表 | task_instance 太大,扫描 ready 变慢 |
| 调度扫描 | scheduler 高频全表扫 |
| 队列堆积 | ready 任务很多,worker 消费跟不上 |
| 外部轮询 | 大量 job polling 打爆外部系统 |
| 日志写入 | task log / state transition 写入量大 |
| 大 DAG 渲染 | UI 打不开几千节点图 |
| 动态 fan-out | 一次生成过多任务 |
| 热点 workflow | 单个 run 产生大量状态更新 |

所以优化要先问:

```text
瓶颈在调度、执行、存储、外部系统、还是 UI?
```

不要一上来就盲目加 worker。

如果瓶颈在数据库扫描,加 worker 只会让状态竞争更严重。

## task_instance 表会成为核心压力点

`task_instance` 是 DAG 引擎最热的表之一。

它会被频繁用于:

```text
查 READY 任务
更新 RUNNING/SUCCESS/FAILED
检查上游依赖
计算 workflow 是否完成
查询运行历史
展示 UI
排查失败原因
```

如果表设计粗糙,生产上会很快变慢。

常用索引:

```text
(status, scheduled_at)
(status, lease_until)
(workflow_instance_id, status)
(workflow_instance_id, task_key)
(worker_id, status)
(tenant_id, status, priority)
(created_at)
```

如果支持多租户,建议把 `tenant_id` 或 `workspace_id` 作为很多查询的前缀条件。

否则一个租户的数据会拖慢全局扫描。

## 避免 scheduler 全表扫描

最初版本可能这样做:

```sql
SELECT * FROM task_instance WHERE status = 'READY';
```

数据少时没问题。

数据多时会变成:

```text
每次调度都扫一大片数据
大量 scheduler 互相竞争
READY 任务被重复检查
数据库 CPU 飙高
```

更好的方向:

```text
只扫描 scheduled_at <= now 的任务
按租户/队列/优先级分页
使用 lease 或 SKIP LOCKED
事件驱动地把 ready task 推入队列
用增量游标而不是全量扫描
把历史任务归档到冷表
```

例如:

```text
READY + scheduled_at + priority + tenant_id
```

共同决定调度顺序。

## 事件驱动和扫描的取舍

纯扫描模型:

```text
scheduler 定时扫描 DB,发现 ready task。
```

优点:

```text
简单
容易恢复
不依赖消息不丢
```

缺点:

```text
延迟取决于扫描周期
数据量大时 DB 压力大
```

事件驱动模型:

```text
上游完成后,发送事件触发下游 ready 计算。
```

优点:

```text
延迟低
减少无效扫描
```

缺点:

```text
事件可能丢
DB 和 MQ 一致性更复杂
需要补偿扫描
```

生产里通常是混合模式:

```text
事件驱动负责实时推进。
周期扫描负责补偿和纠偏。
```

这和 outbox pattern 经常一起使用。

## 批量调度

大规模系统里不要一个 task 一次 DB 往返。

可以做:

```text
批量领取 ready tasks
批量更新状态
批量写 transition
批量发送 queue messages
批量查询父节点状态
```

但批量也有风险:

```text
一次领取太多导致租约过期
大事务锁太久
单个 worker 占用过多任务
失败回滚影响大
```

所以批量大小要有限制:

```text
batch_size = min(global_limit, tenant_limit, worker_capacity, queue_capacity)
```

并且最好按队列或租户分片。

## 大 DAG 的调度优化

对于一个几千节点的 DAG,每次节点完成后都重新遍历全图会很慢。

更高效的做法:

```text
维护每个节点 remaining_dependencies_count
父节点成功后只更新直接 children
child 的 remaining count 归零才进入 ready
```

类似:

```text
A -> B
A -> C
B -> D
C -> D
```

当 `A` 成功:

```text
B.remaining -= 1
C.remaining -= 1
```

当 `B/C` 都成功:

```text
D.remaining == 0
```

这比每次遍历 D 的所有父节点更高效。

但要注意 trigger rule。

如果不是 `all_success`,`remaining_dependencies_count` 就不能只记录成功父节点,还要记录:

```text
total parents
terminal parents
success parents
failed parents
skipped parents
cancelled parents
```

所以生产设计通常要有:

```text
dependency_summary
```

而不是只靠简单入度。

## 动态 fan-out 的限流

动态 DAG 最容易造成任务爆炸。

例如:

```text
一个 query 返回 100 万行
每一行生成一个 task
```

如果不限制,会导致:

```text
task_instance 表暴涨
队列爆炸
worker 被淹没
下游系统被打爆
UI 无法展示
```

要设计 fan-out 保护:

```text
max_children_per_task
max_tasks_per_run
max_dynamic_depth
max_fanout_groups_per_run
tenant fanout quota
external system rate limit
```

当超过限制时,不要默默生成。

应该进入:

```text
FAILED / BLOCKED / WAITING_APPROVAL
```

并给出明确原因:

```text
fanout limit exceeded: generated 1,000,000, max allowed 10,000
```

更好的方式是把任务分块:

```text
100 万行 -> 1000 个 chunk task -> 每个 chunk 处理 1000 行
```

而不是每一行一个 task。

## 外部系统限流

EMF 这类平台通常会触发:

```text
Databricks
Storage
Catalogue
SQL engine
API call
文件上传下载
```

这些外部系统都有容量边界。

所以调度器要支持资源池:

```text
databricks_pool: max 20
sql_pool: max 50
storage_pool: max 100
external_api_x: max 10
```

每个 task definition 或 command 可以声明:

```text
resource_type
resource_weight
rate_limit_key
```

调度时不仅要看 DAG ready,还要看:

```text
资源池是否有额度。
```

这点非常适合面试展开:

```text
ready 不等于立即执行,ready 只是依赖满足;真正执行还要通过资源调度。
```

## 多租户和公平性

如果多个团队共享一个 DAG 平台,必须防止:

```text
一个租户提交大批任务,拖垮所有人。
```

常见策略:

```text
租户级并发限制
租户级队列
租户级优先级
租户级 quota
按租户 round-robin 调度
按权重公平调度
资源池隔离
```

例如:

```text
tenant A max_running_tasks = 100
tenant B max_running_tasks = 20
tenant A databricks_concurrency = 10
tenant B databricks_concurrency = 5
```

公平调度不是平均分配。

它要考虑:

```text
租户权重
业务优先级
任务类型
SLA
资源池容量
历史占用
```

如果不做,生产上会出现:

```text
大客户跑批把小客户交互式任务堵住
低优先级 backfill 抢走线上同步资源
测试环境任务占用生产资源池
```

## 优先级不是简单数字越大越先跑

优先级要防止饥饿。

如果永远先跑高优先级任务,低优先级任务可能永远跑不到。

可以设计:

```text
priority + waiting_time aging
```

例如:

```text
effective_priority = base_priority + waiting_minutes * aging_factor
```

这样低优先级任务等得足够久也会被执行。

也可以按队列隔离:

```text
critical queue
normal queue
backfill queue
test queue
```

每个队列有自己的并发和权重。

## 权限控制

DAG 平台要控制的不只是“谁能登录”。

还要控制:

```text
谁能创建 workflow
谁能修改 workflow definition
谁能运行 workflow
谁能取消 workflow
谁能重跑失败节点
谁能手动标成功
谁能查看日志
谁能查看参数
谁能访问 secret
谁能审批
谁能提升优先级
谁能修改资源池
```

权限粒度至少包括:

```text
tenant / workspace / project
workflow
workflow run
task
secret
operation
```

简单 RBAC:

| 角色 | 能力 |
|---|---|
| Viewer | 查看状态和非敏感日志 |
| Operator | 重跑、取消、暂停、恢复 |
| Developer | 创建和修改 workflow |
| Approver | 审批人工节点 |
| Admin | 资源池、权限、强制操作 |

高风险操作可以要求双人审批:

```text
manual mark success
skip critical task
cancel production workflow
view sensitive parameters
```

## Secret 和日志脱敏

生产 DAG 经常携带:

```text
数据库密码
API token
客户数据
账户信息
审批内容
```

权限治理要覆盖:

```text
参数存储
参数解析
日志打印
UI 展示
审计导出
错误消息
```

设计原则:

```text
Secret 存引用,不存明文。
执行时按权限解析。
日志默认脱敏。
UI 需要权限才可查看。
查看敏感数据也要审计。
```

## 部署升级为什么难

DAG 引擎升级时,生产环境里可能还有:

```text
RUNNING workflow
RUNNING task
WAITING_APPROVAL task
RETRYING task
PAUSED workflow
旧版本 worker
新版本 scheduler
旧 definition version
新 definition version
```

如果升级不兼容,会出现:

```text
旧 worker 不认识新状态
新 scheduler 调度了旧 worker 不支持的 command
schema migration 锁表
任务协议字段缺失
状态枚举变更导致解析失败
运行中的 workflow 行为变化
```

所以要做版本兼容。

## 协议和状态枚举要兼容

假设新增一个状态:

```text
WAITING_EVENT
```

旧 worker 不认识它。

如果旧 worker 看到这个状态直接报错,可能影响整个系统。

更稳的方式:

```text
未知状态默认不领取
scheduler 按 worker capability 分发任务
状态枚举新增要灰度
旧 worker 继续处理旧状态
新状态只给新 worker
```

command/module 也类似。

可以有:

```text
worker_capabilities:
  supported_commands
  supported_state_version
  supported_schema_version
```

调度时匹配:

```text
task.command in worker.supported_commands
task.protocol_version <= worker.supported_protocol_version
```

## 数据库 migration 要谨慎

DAG 引擎的核心表通常很大:

```text
task_instance
task_state_transition
task_execution_attempt
workflow_instance
```

升级时如果做:

```sql
ALTER TABLE task_instance ADD COLUMN ... NOT NULL DEFAULT ...
```

可能锁表很久。

更稳的 migration 模式:

```text
先加 nullable 字段
代码兼容新旧字段
后台 backfill
验证完成后再加约束
最后清理旧字段
```

也就是 expand / migrate / contract:

```text
Expand: schema 扩展,兼容旧代码
Migrate: 数据回填
Contract: 移除旧字段或加严格约束
```

## 运行中 workflow 的版本绑定

升级不应该改变正在运行的 workflow definition。

一个 workflow instance 应该绑定:

```text
workflow_definition_version_id
```

这样即使用户修改定义:

```text
A -> B -> C
```

变成:

```text
A -> B -> D -> C
```

正在跑的旧 instance 仍然按旧版本执行。

如果用户想用新定义重跑,应该创建:

```text
new workflow instance with new definition version
```

而不是悄悄修改旧 run 的 DAG。

## 常见生产问题

### 1. READY 很多但执行很慢

排查:

```text
worker 是否不足
资源池是否满
租户配额是否满
队列是否堆积
scheduler claim 是否慢
DB 查询是否慢
外部系统是否限流
```

设计解决:

```text
区分 DAG ready 和 resource ready
引入资源池指标
ready queue 分租户/队列
worker autoscaling
DB 索引和分页扫描
```

### 2. 单个大 workflow 拖垮系统

排查:

```text
这个 run 生成了多少 task
fanout group 是否超过阈值
这个 run 占用多少 worker
是否大量轮询外部系统
是否大量写日志
```

设计解决:

```text
max_tasks_per_run
fanout chunking
run 级并发限制
租户级 quota
大 DAG UI 聚合展示
```

### 3. scheduler DB 压力大

排查:

```text
是否全表扫描 READY
索引是否命中
历史数据是否归档
扫描周期是否太频繁
是否多个 scheduler 重复扫描
```

设计解决:

```text
按 status/scheduled_at/tenant 建索引
分页 claim
SKIP LOCKED / CAS
冷热表分离
事件驱动 + 补偿扫描
```

### 4. 某个租户把其他人堵住

排查:

```text
按 tenant 看 running/ready/retry 数量
按 tenant 看资源池占用
按 tenant 看 queue delay
```

设计解决:

```text
租户队列
租户并发上限
公平调度
优先级 aging
资源池隔离
```

### 5. 升级后旧任务失败

排查:

```text
失败任务绑定哪个 definition version
worker 版本是多少
command 协议是否兼容
状态枚举是否新增
schema 字段是否缺失
```

设计解决:

```text
worker capability
definition version binding
schema expand/migrate/contract
灰度发布
新旧 worker 并行
```

## 面试总结

可以这样说:

```text
DAG 引擎规模化后,瓶颈通常不在拓扑排序,而在状态表扫描、任务领取、外部系统限流、日志写入、多租户公平性和升级兼容。设计上要避免 scheduler 全表扫描,用 status/scheduled_at/tenant 等索引、lease/CAS、分页 claim、事件驱动加补偿扫描。对于大 DAG 和动态 fan-out,要用 run 级并发、fanout limit、chunking、资源池和队列隔离防止任务爆炸。多租户场景要做 quota、公平调度、权限和 secret 脱敏。系统升级时,workflow instance 必须绑定不可变 definition version,worker 协议和状态枚举要兼容,数据库 migration 采用 expand/migrate/contract,避免运行中的任务被新版本破坏。
```

最关键的一句话:

```text
ready 只是依赖满足,能不能执行还取决于资源、配额、优先级、租约和版本兼容。
```
