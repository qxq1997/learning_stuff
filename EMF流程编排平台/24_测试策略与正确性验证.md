# 24 测试策略与正确性验证

## 为什么 DAG 引擎测试很难

DAG 引擎不是普通业务 CRUD。

它的复杂度来自:

```text
图结构
状态机
并发调度
重试
超时
动态节点生成
外部系统不确定性
人工干预
版本变更
```

所以它不能只测:

```text
输入一个 DAG,最后跑成功。
```

真正要证明的是:

```text
在失败、重复、乱序、宕机、并发、重跑、暂停、取消的情况下,系统仍然不违反核心不变量。
```

面试里可以说:

```text
DAG 引擎测试的重点不是覆盖 happy path,而是验证状态推进和依赖调度在不可靠环境下仍然满足不变量。
```

## 先定义系统不变量

测试前要先明确哪些事情永远不能发生。

常见不变量:

| 不变量 | 含义 |
|---|---|
| no early execution | 父节点未满足 trigger rule,子节点不能执行 |
| single ownership | 同一 task 同一 attempt 不能被两个 worker 同时执行 |
| terminal immutable | 终态不能被普通调度逻辑改回非终态 |
| attempt isolation | 旧 attempt 不能覆盖新 attempt 状态 |
| idempotent dynamic creation | 动态节点重复生成请求不能生成重复 child |
| output before complete | 标记 complete 前必须满足 output contract |
| audit completeness | 人工操作和状态变更必须有审计 |
| version binding | 运行中的 instance 不能被新 definition version 影响 |
| bounded fan-out | 动态扩展不能超过 run/tenant 限制 |
| recoverable running | worker 宕机后 RUNNING 任务不能永久卡死 |

测试应该围绕这些不变量设计。

如果一个测试只是断言:

```text
workflow status == SUCCESS
```

那还不够。

更重要的是断言过程中没有发生:

```text
子节点提前执行
重复执行外部副作用
旧 attempt 覆盖新 attempt
状态跳转不合法
```

## 测试分层

可以把测试分成六层。

| 层级 | 测什么 |
|---|---|
| 纯逻辑测试 | DAG 校验、拓扑排序、trigger rule、状态机 |
| 状态存储测试 | CAS、lease、唯一约束、transition 记录 |
| 调度集成测试 | scheduler + DB + queue + worker |
| 故障注入测试 | worker crash、DB 更新失败、callback 重复 |
| 模拟/回放测试 | 用事件序列模拟复杂 workflow |
| 生产验证 | canary、shadow、dry run、指标告警 |

越底层越确定,越上层越接近真实事故。

## DAG 建模测试

DAG 合法性测试包括:

```text
正常 DAG
单节点 DAG
多父节点 join
多分支 DAG
子 workflow
动态 fan-out/fan-in
缺失依赖
重复节点
自依赖
环依赖
不可达节点
孤立节点
非法 trigger rule
不存在的 command
```

例如:

```text
A -> B -> C -> A
```

必须在提交定义或动态扩展时拦截。

测试目标不是只验证会报错,还要验证错误信息足够可排查:

```text
cycle detected: A -> B -> C -> A
```

比:

```text
invalid DAG
```

有用得多。

## trigger rule 测试

trigger rule 是故障高发点。

例如:

```text
A -> D
B -> D
C -> D
```

要测:

| 父节点状态 | `all_success` | `all_done` | `none_failed` | `one_success` |
|---|---|---|---|---|
| A/B/C all success | ready | ready | ready | ready |
| A success, B failed, C success | blocked | ready | blocked | ready |
| A success, B skipped, C success | depends | ready | ready | ready |
| A running, B success, C success | not ready | not ready | not ready | ready or not,看语义 |

关键是:

```text
父节点未到终态时,join 节点不能被过早判断。
```

哪怕 `one_success` 已经满足,也要看系统语义是否允许提前执行。

有些系统允许:

```text
one_success 一满足就跑。
```

有些系统要求:

```text
所有父节点终态后再判断。
```

这个语义必须被测试固定下来。

## 状态机测试

状态机要测试合法流转和非法流转。

合法例子:

```text
CREATED -> READY
READY -> RUNNING
RUNNING -> COMPLETE
RUNNING -> FAILED
FAILED -> RETRYING -> READY
RUNNING -> TIMEOUT
RUNNING -> CANCEL_REQUESTED -> CANCELLED
WAITING_APPROVAL -> READY
```

非法例子:

```text
COMPLETE -> RUNNING
CANCELLED -> READY
FAILED -> COMPLETE
RUNNING -> READY
TIMEOUT -> COMPLETE
```

如果支持人工操作,要单独测试:

```text
FAILED -> MANUAL_SUCCESS
FAILED -> SKIPPED
PAUSED -> RUNNING
CANCELLING -> CANCELLED
```

并且检查:

```text
是否要求 reason
是否写 operation audit
是否触发下游重算
```

## 任务领取测试

任务领取是并发事故核心。

要测试多个 scheduler / worker 同时抢同一个 `READY` task。

期望结果:

```text
只有一个 worker claim 成功。
其他 worker 看到更新失败或拿不到锁。
```

可以模拟:

```text
10 个 worker 同时 claim 同一个 task
100 个 worker 同时 claim 100 个 task
worker claim 后立刻宕机
lease 过期后重新 claim
旧 worker lease 过期后又回来提交结果
```

关键断言:

```text
同一个 task attempt 只被一个 owner 持有。
旧 owner 不能在 lease 失效后覆盖新 owner。
```

这通常要求:

```text
task_id
attempt
worker_id
lease_token
version
```

一起参与 CAS。

## 并发调度测试

要测试多个 scheduler 同时运行。

场景:

```text
两个 scheduler 同时扫描 READY
两个 scheduler 同时发现同一个 child ready
一个 scheduler 正在推进下游,另一个 scheduler 在处理 retry
人工 skip 与 scheduler ready 计算同时发生
```

核心断言:

```text
不会重复入队
不会重复生成 task
不会让 blocked 节点变 ready
不会让 terminal 节点重新运行
```

可以通过唯一约束保护:

```text
unique(workflow_instance_id, task_key)
unique(outbox_event_id)
unique(fanout_group_id, child_key)
```

测试应该验证这些约束真的能挡住并发重复。

## 重试和幂等测试

可靠执行一定要测这些经典事故:

### 1. 外部调用成功,状态更新失败

流程:

```text
worker 调外部系统成功
DB 更新 COMPLETE 失败
worker 重启
任务被重试
```

期望:

```text
外部系统通过 idempotency_key 去重
平台能通过 external_call_log 或 reconciliation 确认结果
不会产生重复副作用
```

### 2. 状态更新成功,消息发送失败

流程:

```text
父节点 COMPLETE
下游 ready 事件发送失败
```

期望:

```text
outbox 或补偿扫描能重新发送
下游最终会被调度
```

### 3. callback 重复回来

流程:

```text
外部系统发送两次 success callback
```

期望:

```text
inbox 去重
第二次 callback 不改变终态
状态变更日志不重复或标记 duplicate
```

### 4. callback 乱序

流程:

```text
success callback 先到
running callback 后到
```

期望:

```text
terminal 状态不会被旧的 running callback 覆盖
```

## 动态 DAG 测试

动态 DAG 要重点测:

```text
重复生成 child
生成时出现环
生成超过 fanout limit
生成 child 后 run completion 是否延后
父节点 retry 后是否重复生成 child
动态子节点失败后 completion token 是否正确处理
子 workflow 多层嵌套
```

例如:

```text
RunSpawnService 第一次生成 A/B/C
worker crash
同一个生成命令被 retry
RunSpawnService 再次生成 A/B/C
```

期望:

```text
不会出现两个 A、两个 B、两个 C。
```

实现上靠:

```text
stable child_key
unique(run_uuid, order_id)
fanout_group_id
idempotency_key
```

测试要验证重复提交不会破坏图。

## 数据和输出契约测试

要测:

```text
父节点没有 output_ref 时不能标 COMPLETE
output_ref 指向不存在对象时不能标 COMPLETE
output schema 不匹配时不能推进下游
下游 resolved_parameters 是否引用正确 output_ref
重跑时是否复用或刷新上游输出
secret 是否不会出现在日志
大对象是否只传引用
```

尤其是你前面问的场景:

```text
B 还没真正完成,却被标记 COMPLETE,D 提前执行。
```

测试可以模拟:

```text
B submit external job success
B external job still running
系统尝试 mark COMPLETE
```

期望:

```text
mark COMPLETE 被拒绝,因为 output contract 未满足。
```

## 人工操作测试

人工操作要测:

```text
暂停后 READY 不再被领取
暂停后 RUNNING 是否继续
恢复后 ready set 是否重算
取消后 READY/WAITING 是否 CANCELLED
取消后外部 job 是否收到 cancel request
重跑上游后下游是否 invalidated
手动 skip 是否满足 trigger rule
手动 mark success 是否要求 output_ref 或 reason
所有人工操作是否有 audit
```

还要测并发:

```text
用户 A cancel workflow
用户 B 同时 rerun task
```

期望:

```text
只有一个操作成功。
失败的一方看到状态冲突,需要刷新后重试。
```

## 故障注入测试

生产问题往往来自中途失败。

建议系统性注入:

```text
DB update failure
MQ publish failure
worker crash after external success
worker crash before state update
scheduler crash during ready calculation
network timeout
external API 500
external API timeout but actually succeeded
callback duplicate
callback lost
object storage eventual consistency
clock skew
```

每个故障都要问:

```text
系统最后能否收敛?
有没有重复副作用?
有没有任务永久卡住?
有没有证据链可排查?
```

## 模拟器和事件回放

复杂 DAG 很难靠手写测试覆盖。

可以做一个轻量模拟器:

```text
输入:
  DAG definition
  task duration
  failure pattern
  callback order
  retry policy
  concurrency limit

输出:
  state transition sequence
  final workflow status
  invariant violations
```

例如模拟:

```text
A success
B failed
C skipped
D trigger_rule = none_failed
```

检查 `D` 是否应该 ready。

事件回放也很有价值。

线上出问题后,把这些事件保存下来:

```text
task claimed
external submitted
callback received
state changed
retry scheduled
manual operation
```

在测试环境回放,可以复现复杂竞态。

## Property-based testing

如果想更高级,可以用 property-based testing。

随机生成:

```text
DAG 结构
父子依赖
节点状态
trigger rule
失败模式
重试次数
并发操作
```

然后验证不变量:

```text
没有环
没有提前执行
终态不会被普通事件覆盖
所有 ready 节点的父依赖都满足
所有生成节点都有唯一 order_id
```

这种测试很适合发现边界条件。

面试里可以轻描淡写地提一句:

```text
对于状态机和 DAG ready 判断,我会考虑用模型测试或 property-based testing 来验证不变量,因为手写 case 很难覆盖所有组合。
```

这会显得你真的理解系统复杂度。

## 生产发布前验证

上线新调度逻辑时,可以用:

```text
dry run
shadow scheduler
canary tenant
read-only simulation
```

### dry run

只计算:

```text
哪些 task 会 ready
哪些 task 会 blocked
哪些 task 会 skipped
```

但不真正执行。

### shadow scheduler

新 scheduler 不写状态,只在旁边计算决策。

对比:

```text
old scheduler decision
new scheduler decision
```

如果差异异常,先不上线。

### canary tenant

只让少量 workflow 或租户走新逻辑。

观察:

```text
失败率
retry rate
queue delay
重复 claim
zombie task
状态冲突
```

## 观测自身也要测

可观测性不是写完就行。

要测试:

```text
每次状态变更是否有 transition
每次人工操作是否有 audit
每次外部调用是否有 trace_id
失败时 error_message 是否可读
敏感参数是否被脱敏
UI 能不能定位失败节点
大 DAG 是否能聚合展示
```

否则真实事故时才发现:

```text
日志里没有 task_id。
状态变更没有 reason。
外部 job id 没保存。
```

那就晚了。

## 最小测试矩阵

一个可面试的最小测试矩阵:

| 能力 | 必测场景 |
|---|---|
| DAG 校验 | 环、缺失依赖、重复节点、动态新增环 |
| 状态机 | 合法/非法流转、终态不可逆 |
| 调度 | 父依赖未满足不执行、trigger rule |
| 并发领取 | 多 worker 抢同一 task |
| lease | worker 挂掉后任务可恢复 |
| 重试 | 可重试/不可重试错误 |
| 幂等 | 外部成功但状态失败 |
| 动态 DAG | 重复生成、fanout limit、completion token |
| 人工操作 | pause/resume/cancel/rerun/skip |
| 可观测性 | transition、attempt、audit、trace |

## 面试总结

可以这样说:

```text
DAG 引擎的测试不能只覆盖一个 workflow 最终成功,而要围绕系统不变量设计。核心不变量包括父依赖满足前不能执行、同一任务不能被重复领取、终态不能被旧事件覆盖、动态节点不能重复生成、任务 complete 前必须满足输出契约、worker 宕机后 RUNNING 任务可恢复。测试上我会分层做 DAG 校验、状态机、CAS/lease、scheduler 集成、故障注入、事件回放和 canary。尤其要模拟外部系统成功但状态更新失败、callback 重复或乱序、scheduler/worker 宕机、人工操作并发这些生产事故,验证系统最终能收敛且不会产生重复副作用。
```

最关键的一句话:

```text
DAG 引擎测试的核心不是证明它能跑通,而是证明它在混乱事件下仍然不违反不变量。
```
