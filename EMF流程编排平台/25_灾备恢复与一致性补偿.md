# 25 灾备恢复与一致性补偿

## 为什么一致性补偿很重要

DAG 引擎通常由多个组件组成:

```text
API / Workflow Service
Database
Scheduler
Message Queue
Worker
External Systems
Object Storage
Log System
```

这些组件之间不会天然一致。

生产里一定会出现:

```text
DB 更新成功,MQ 消息发送失败
MQ 消息发送成功,DB 更新失败
worker 执行成功,状态更新失败
外部 job 成功,callback 丢失
callback 成功到达,平台处理失败
对象存储已写入,output_ref 没写入 DB
任务已取消,外部 job 仍在跑
```

所以生产级设计不能假设:

```text
每一步都会成功。
```

而要假设:

```text
任意两步之间都可能失败。
```

系统目标不是永远不出错,而是:

```text
出错后能发现、能补偿、能收敛、能解释。
```

## 一致性的三条主线

可以把一致性问题拆成三条。

| 主线 | 典型问题 |
|---|---|
| 状态一致性 | task 状态是否反映真实执行进度 |
| 消息一致性 | DB 状态变化是否对应 MQ 事件 |
| 产物一致性 | output_ref 是否对应真实可消费产物 |

对应的补偿机制:

| 主线 | 补偿 |
|---|---|
| 状态一致性 | lease scanner、zombie detector、external polling |
| 消息一致性 | outbox / inbox、补偿扫描、幂等消费 |
| 产物一致性 | output validator、manifest checker、reconcile job |

## DB 和 MQ 不一致

最经典的问题:

```text
父节点 COMPLETE 了,但下游 ready 消息没发出去。
```

如果系统只依赖 MQ 事件,下游会永远不跑。

反过来:

```text
下游 ready 消息发出去了,但 DB 里 task 还不是 READY。
```

worker 消费后可能发现状态不对。

### Outbox Pattern

更稳的方式是:

```text
业务状态更新和 outbox_event 写入同一个 DB transaction。
```

例如父节点完成时:

```text
BEGIN
  update task_instance set status = COMPLETE
  insert task_state_transition
  insert outbox_event(type = TASK_COMPLETED, task_id = ...)
COMMIT
```

然后由 outbox publisher 异步发送:

```text
读取未发送 outbox_event
发送 MQ
标记 sent
失败则重试
```

这样至少保证:

```text
只要 DB 状态变了,一定有一条可恢复的事件记录。
```

### Inbox Pattern

消费者侧也要去重。

因为 outbox publisher 可能重复发送,或者 MQ 本身至少一次投递。

可以记录:

```text
inbox_event:
  event_id
  consumer_name
  processed_at
```

如果同一个 event_id 已处理,直接忽略。

这让消息消费变成幂等。

## Worker 宕机恢复

场景:

```text
task status = RUNNING
worker 已经挂了
```

如果没有恢复机制,任务会永远卡在 `RUNNING`。

所以需要:

```text
worker heartbeat
task lease
lease scanner
zombie detector
```

推荐语义:

```text
worker claim task 时拿到 lease_token
task.lease_until = now + lease_duration
worker 执行中定期 heartbeat / renew lease
如果 lease 过期且 worker heartbeat 失效,任务进入 LOST / RETRYING / FAILED
```

注意:

```text
lease 过期不等于外部副作用没有发生。
```

所以恢复前要结合任务类型判断:

```text
纯计算任务可以 retry
外部副作用任务要先 reconcile
长时间外部 job 要先查询 external status
```

也就是说,worker 宕机恢复不是简单把 `RUNNING` 改回 `READY`。它要先判断任务是否真的失去 owner,再判断外部副作用是否已经发生,最后才能决定补成功、补失败、重试、取消或进入人工处理。更完整的设计见第 32 章《僵尸任务发现与处理》。

## 外部 job 状态不一致

很多任务不是 worker 本地执行,而是提交到外部系统:

```text
Databricks Job
Spark Job
SQL Query
Kubernetes Job
HTTP Workflow
文件处理任务
```

平台状态和外部状态可能不一致:

| 平台状态 | 外部状态 | 说明 |
|---|---|---|
| RUNNING | SUCCEEDED | callback 丢失或处理失败 |
| RUNNING | FAILED | callback 丢失 |
| COMPLETE | FAILED | 平台提前标完成或外部最终失败 |
| FAILED | SUCCEEDED | 网络超时但外部实际成功 |
| CANCELLED | RUNNING | cancel 没生效 |

所以外部任务必须记录:

```text
external_system
external_job_id
external_request_id
external_status
last_polled_at
external_trace_id
```

并通过 polling / reconciliation 修正。

## Reconciliation Job

reconciliation 是生产系统的救命机制。

它定期扫描异常状态并尝试收敛。

典型扫描:

```text
RUNNING 且 lease 过期
RUNNING 且 external job terminal
SUBMITTED 长时间没有 external_job_id
COMPLETE 但 output_ref 不存在
FAILED 但 external job succeeded
CANCELLING 但外部 job 仍 running
outbox_event 未发送
callback inbox 处理失败
fanout_group expected_count 和 child 数量不一致
```

reconcile 不应该盲目改状态。

它要:

```text
查询证据
判断当前状态
做 CAS 更新
写 transition reason = reconciliation
必要时告警或进入人工处理
```

例如:

```text
平台 task = RUNNING
external job = SUCCEEDED
output_ref exists
```

可以修正为:

```text
RUNNING -> COMPLETE
reason = external succeeded, callback missed
```

如果:

```text
平台 task = RUNNING
external job = UNKNOWN
lease expired
```

不要直接 retry。

更稳的是:

```text
RUNNING -> UNKNOWN_EXTERNAL_STATE
或进入 manual intervention
```

如果没有这个状态,至少要把任务放入人工排查队列。

## 灾备恢复场景

### 1. Scheduler 重启

Scheduler 重启后应该能恢复:

```text
READY 任务继续被领取
RUNNING 任务由 lease/heartbeat 判断
RETRYING 任务按 scheduled_at 重试
PAUSED workflow 不被调度
CANCELLING workflow 继续取消流程
```

所以 scheduler 本身不要保存唯一关键状态。

关键状态应该在:

```text
DB
queue
outbox
lease
transition log
```

### 2. Worker 全挂

Worker 全挂后:

```text
RUNNING 任务 lease 逐渐过期
READY 任务堆积
外部 job 可能仍在跑
```

恢复时要做:

```text
先恢复 worker capacity
再由 lease scanner 处理过期任务
外部任务先 reconcile,不要盲目重跑
观察 retry storm
限制恢复速度
```

如果所有过期任务同时 retry,可能打爆下游。

所以恢复要有:

```text
retry backoff
throttled recovery
per-tenant recovery limit
```

Scheduler 重启和 worker 全挂只是两个最常见场景。单 worker crash、scheduler leader crash、多 scheduler 并发接管、内嵌式 EMF 进程死亡等情况,可以看第 33 章《Worker 与 Scheduler 宕机恢复》。

### 3. DB 短暂不可用

DB 不可用时:

```text
scheduler 无法 claim task
worker 无法更新状态
outbox 无法写
callback 无法落库
```

设计原则:

```text
不能确认状态更新成功时,不要假装成功。
```

Worker 可以:

```text
停止领取新任务
保留本地执行结果短时间重试上报
外部任务用 external request id 以后 reconcile
```

但不能无限依赖本地缓存。

DB 恢复后:

```text
reconcile external job
处理 outbox backlog
扫描 lease expired task
检查 callback backlog
```

### 4. MQ 不可用

MQ 不可用时,如果有 outbox:

```text
DB 状态仍可更新
outbox_event 会堆积
publisher 等 MQ 恢复后重发
```

但要监控:

```text
outbox backlog
event age
publish failure rate
```

如果没有 outbox,状态和消息可能永久不一致。

### 5. Object Storage 不可用

如果任务需要写 output_ref:

```text
对象存储不可用
```

那即使业务计算完成,也不能标 `COMPLETE`。

应该:

```text
进入 RETRYING / OUTPUT_COMMIT_FAILED
保留临时产物
重试写 manifest/output
```

否则下游会看到 `COMPLETE`,却读不到数据。

## 备份和恢复

至少要考虑:

```text
workflow definition version 备份
workflow instance / task instance 备份
state transition / audit 备份
outbox / inbox 备份
external job mapping 备份
output manifest 备份
```

恢复时最重要的问题是:

```text
恢复到某个时间点后,外部系统状态已经继续变化怎么办?
```

例如 DB 恢复到 10:00,但外部 Databricks job 在 10:05 已经成功。

这时不能只按 DB 里的旧状态判断。

需要通过:

```text
external_job_id
idempotency_key
output_ref
manifest
trace_id
```

重新对账。

所以外部引用信息非常关键。

## RPO 和 RTO

面试里可以适当提两个概念。

```text
RPO: 最多能接受丢多少数据。
RTO: 多久能恢复服务。
```

DAG 引擎里:

```text
RPO 影响状态变更、审计、outbox 是否必须强持久化。
RTO 影响 scheduler/worker 是否能快速横向恢复。
```

如果是金融或生产编排平台,通常:

```text
状态变更和审计不能丢。
任务可以重试,但操作证据不能丢。
```

## 补偿不是万能的

补偿有边界。

有些副作用不能完全回滚:

```text
邮件已发送
优惠券已发放
外部审批已创建
文件已被下游消费
数据库写入已被其他系统读取
```

所以设计顺序是:

```text
能幂等就幂等
不能幂等就先查询再执行
不能查询就记录 request_id
不能回滚就做人工补偿
```

不要在面试里轻易说:

```text
失败了就回滚。
```

分布式外部副作用通常不能简单回滚。

更成熟的表达是:

```text
通过幂等键、状态查询、补偿任务和人工介入让系统最终收敛。
```

## 常见生产问题

### 1. 父节点完成了,下游没跑

排查:

```text
父节点状态是否 COMPLETE
state transition 是否写入
outbox_event 是否生成
outbox_event 是否发送
MQ 是否投递
下游 task 是否变 READY
下游是否被 pause/quota/resource limit block
```

设计解决:

```text
outbox pattern
ready 补偿扫描
blocked_reason
调度指标和 event age 告警
```

### 2. 任务一直 RUNNING

排查:

```text
worker heartbeat
task lease_until
external job status
last state transition
last log timestamp
timeout policy
```

设计解决:

```text
lease scanner
external polling
zombie detector
timeout transition
manual intervention queue
```

### 3. 外部成功但平台失败

排查:

```text
external_request_id
external_job_id
idempotency_key
external trace
output_ref 是否存在
平台 attempt error
callback 是否丢失
```

设计解决:

```text
external_call_log
reconciliation
output validator
idempotent completion
```

### 4. 平台成功但外部失败

排查:

```text
是否 submit 成功就标 COMPLETE
是否误处理 callback
是否 output contract 缺失
是否人工标成功
```

设计解决:

```text
SUBMITTED/RUNNING/OUTPUT_VALIDATING/COMPLETE 分离
terminal external status 校验
completion marker
manual success 审计和输出校验
```

### 5. 灾后恢复重复执行

排查:

```text
恢复点之后哪些任务已执行
外部系统有哪些 request_id
平台是否丢失 attempt 记录
是否有幂等 key
```

设计解决:

```text
workflow_instance_id + task_key 幂等
external request log
reconcile before retry
throttled recovery
```

## 面试总结

可以这样说:

```text
DAG 引擎进生产后,一致性问题主要发生在 DB、MQ、worker 和外部系统之间。不能假设状态更新、消息投递、外部执行和 callback 会同时成功。设计上我会用 outbox 保证 DB 状态变化和事件发布可恢复,用 inbox 保证消息幂等消费,用 lease 和 heartbeat 处理 worker 宕机,用 external_job_id 和 idempotency_key 做外部系统对账,用 reconciliation job 定期扫描 RUNNING 超时、callback 丢失、output_ref 缺失、outbox 堆积等异常。灾备恢复时不能盲目重跑,要先根据外部状态和输出产物做 reconcile,再决定 retry、complete、fail 或人工介入。
```

最关键的一句话:

```text
生产级 DAG 引擎不是保证永不失败,而是保证失败后有证据、有补偿、有收敛路径。
```
