# 27 工作流引擎设计纵览与技术取舍

## 这章的定位

前面的笔记已经把 EMF 拆成了很多局部:

```text
Message
DAG
Worker
Command
OE-RUN
动态 DAG
状态机
幂等
重试
可观测性
多租户
灾备恢复
```

这一章做的是反向收束:

```text
从顶到底看,一个生产级工作流引擎到底应该怎么设计。
```

你可以把它当成面试里的系统设计总纲。

核心问题不是:

```text
怎么把 A、B、C、D 按顺序跑起来。
```

而是:

```text
如何在分布式不可靠环境中,可靠推进一组有依赖、有状态、有副作用、有人工干预的任务。
```

## 一句话总览

EMF 这类系统可以概括成:

```text
消息驱动的动态 DAG 工作流引擎。
```

如果从系统设计角度再展开一点:

```text
它用 Message 表达任务实例,
用 order_id + parents 推导 DAG,
用状态机控制节点生命周期,
用 scheduler/consumer 找出 ready 节点,
用 command/module 执行业务动作,
用 retry/timeout/lease/idempotency 保证可靠推进,
用 audit/log/metrics 支撑生产排障,
用动态 message 生成支持运行时扩展 workflow。
```

这句话里已经包含了最重要的设计分层。

## 顶层目标

工作流引擎的顶层目标不是“执行任务”,而是:

```text
把复杂业务流程拆成可调度、可恢复、可观察、可治理的任务网络。
```

它要解决的问题包括:

```text
任务之间有依赖
任务可能失败
任务可能需要重试
任务可能调用外部系统
任务可能有副作用
任务可能动态生成更多任务
任务可能需要人工介入
任务可能运行很久
任务可能跨多个资源池和租户
```

所以这个系统不是单纯的 executor。

它更像:

```text
分布式状态机 + DAG 调度器 + 可靠任务系统 + 运维平台。
```

## 第一层: 业务入口和工作流定义

最上层要回答:

```text
workflow 从哪里来?
```

常见入口:

```text
用户/API 触发
定时任务触发
外部事件触发
OE-RUN 展开
Catalogue resolution
上游 command 动态生成
```

EMF 里很重要的一点是:

```text
workflow 不一定是一份固定的代码 DAG。
```

它可能来自:

```text
配置
表数据
外部参数
process_tasks
catalogue resolution
运行时 query 结果
```

这意味着它天然更接近:

```text
配置/消息驱动的动态 DAG。
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 静态 DAG 定义 | 可验证、可视化、易审计 | 灵活性弱,运行前必须知道所有节点 |
| 动态 DAG 生成 | 灵活,适合数据驱动和子 workflow | 图治理复杂,需要幂等、限流、lineage |
| workflow-as-code | 表达力强,适合复杂逻辑 | 版本、确定性、沙箱和权限更复杂 |
| config/message-driven | 适合内部平台和业务命令 | DSL 设计、校验、排障要求高 |

EMF 的取舍更偏:

```text
用 Message 和 command 承载动态业务编排,牺牲一部分静态可分析性,换取运行时扩展能力。
```

## 第二层: 核心领域模型

工作流引擎要先把概念分清楚。

最重要的四组模型:

```text
Workflow Definition
Workflow Instance
Task Definition
Task Instance
```

在 EMF 语境里,最核心的是 `Message`:

```text
Message = 任务实例 + DAG 节点 + 命令调用 + 参数上下文
```

它至少包含:

```text
run_uuid: 属于哪次 workflow run
batch_id: 属于哪个 batch
order_id: 这个节点是谁
parents: 依赖哪些父节点
command: 执行哪个命令
parameters: 执行参数
state: 当前执行状态
```

这里的关键设计是:

```text
DAG 不是由 msg_id 建出来的,而是由 order_id 和 parents 这组业务节点关系推导出来的。
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 用 Message 表达任务 | 统一入口,天然适合队列和事件驱动 | Message 会变重,需要控制边界 |
| 用 order_id/parents 表达 DAG | 业务节点关系清晰,适合动态生成 | 必须保证 order_id 稳定唯一 |
| definition/instance 分离 | 支持版本和审计 | 模型更复杂 |
| 只存当前状态 | 简单 | 事故无法追溯 |
| 保存 transition history | 可审计可排查 | 写入量更大 |

成熟设计一定要接受:

```text
状态历史和审计是生产成本,不是可选装饰。
```

## 第三层: 控制平面和执行平面

工作流引擎要拆成两类职责。

### 控制平面

控制平面负责:

```text
接收 workflow
生成 task/message
维护 DAG
判断 ready
推进状态机
重试和超时
取消和恢复
记录审计和日志
管理资源池和并发
```

### 执行平面

执行平面负责:

```text
执行具体 command
调用 Databricks / Storage / Catalogue / API / SQL
提交外部 job
轮询外部 job
写 output_ref
处理业务副作用
```

EMF 的设计里:

```text
ParallelMessageConsumer / DagManager / DagContext 更偏控制平面。
StratusCommandExecutor / Module / Service 更偏执行平面。
```

### 为什么要拆开

如果不拆,调度逻辑会和业务命令耦合:

```text
调度器知道每个业务模块怎么执行
业务模块直接操作 DAG 内部状态
重试、限流、日志、审计散落在业务代码里
```

拆开后:

```text
调度层只关心节点是否 ready、状态怎么流转。
命令层只关心当前 message 要做什么业务动作。
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 控制/执行分离 | 职责清楚,易扩展,易治理 | 接口契约要设计好 |
| 调度器直接执行业务 | 简单低延迟 | 耦合重,难扩展 |
| command plugin 化 | 新业务易接入 | module 契约、参数校验、权限要治理 |
| service 调用外部系统 | 复用内部能力 | 外部状态不确定,需要 monitor/reconcile |

## 第四层: DAG 调度模型

DAG 调度的核心问题:

```text
什么时候一个节点可以执行?
```

最基础条件:

```text
当前节点未执行
所有父节点满足依赖条件
run_date 未到未来
workflow 未暂停/取消
资源池有额度
租户没超 quota
```

在 EMF 简化理解里:

```text
节点 ready = 当前状态可执行 + 父节点都满足 + 调度约束满足。
```

生产里要注意:

```text
ready 不等于立即执行。
```

因为还要看:

```text
worker capacity
resource pool
priority
tenant quota
rate limit
lease
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 内存 DAG | 低延迟,实现直观 | 重启恢复和多实例一致性复杂 |
| DB 持久 DAG | 可恢复、可审计 | 读写成本更高 |
| 扫描式调度 | 简单,容易补偿 | 延迟和 DB 压力较大 |
| 事件驱动调度 | 延迟低,减少扫描 | DB/MQ 一致性更复杂 |
| 混合调度 | 实时性和恢复性平衡 | 系统复杂度上升 |

成熟系统通常是:

```text
事件驱动推进 + 周期扫描补偿。
```

## 第五层: 状态机

DAG 引擎真正的核心是状态机。

一个节点不能只有:

```text
success / failed
```

至少要区分:

```text
INACTIVE
ACTIVE
IN_PROGRESS
COMPLETE
RETRY
FAILED
TIMEOUT
CANCELLED
SKIPPED
WAITING_APPROVAL
```

更生产级的设计还会区分:

```text
SUBMITTED
RUNNING_EXTERNAL
OUTPUT_VALIDATING
CANCELLING
LOST
BLOCKED
```

状态机的关键不是枚举多,而是:

```text
每个状态能去哪,谁有权限改,改的时候要记录什么。
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 状态少 | 简单,学习成本低 | 语义混乱,排障困难 |
| 状态细 | 语义清晰,方便恢复 | 状态流转复杂 |
| 允许人工直接改状态 | 操作方便 | 容易绕过约束 |
| 人工操作走命令和审计 | 安全可追踪 | 实现更重 |

面试里可以这样说:

```text
状态机设计宁可一开始稍微细一点,也不要把 submit success、external success、output ready 全部混成 COMPLETE。
```

## 第六层: Worker 和 Command 执行模型

Worker 的职责不是“什么都做”。

它更像:

```text
从调度层领取 ready message,
打开 message scope,
解析 command,
解析参数,
找到 module,
执行 command/service,
记录结果,
必要时 enqueue 新 message。
```

在 EMF 里可以理解成:

```text
MessageHandler
  -> StratusCommandExecutor
  -> ModuleProvider
  -> ParameterParser
  -> ParameterisedModuleBase
  -> command/service
```

### 执行方式选择

| 模式 | 优点 | 代价 |
|---|---|---|
| worker 内置执行 | 简单、低延迟 | 隔离弱,重任务会拖垮 worker |
| worker 提交外部 job | 隔离好,适合 Databricks/Spark | 状态回传和 monitor 复杂 |
| 每任务容器化 | 隔离最强,资源可控 | 启动慢,成本高,平台复杂 |

EMF 更偏:

```text
worker 作为 command 调度和内部 service 调用入口。
```

对于重计算任务,更稳的是:

```text
worker submit + monitor,不要把重计算都塞进 worker 进程。
```

## 第七层: 动态 DAG

动态 DAG 的设计动机:

```text
运行前不知道任务数量。
运行前不知道完整依赖。
某个 command 的结果会决定后续任务。
子 workflow 需要展开成一批 message。
```

典型场景:

```text
query 返回 N 条结果,为每条结果生成任务
catalogue resolution 后生成实际处理节点
OE-RUN 展开 workflow tasks
子 workflow 根据父任务参数继续生成
```

动态 DAG 的关键不是:

```text
能不能生成新节点。
```

而是:

```text
生成后图仍然合法、幂等、可限流、可观测、可完成。
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 静态 DAG | 易校验、易展示 | 灵活性弱 |
| 动态 DAG | 灵活,适合数据驱动 | 环检测、fan-out、run completion 复杂 |
| 每条数据一个 task | 粒度细,可重试 | task 数爆炸 |
| chunk task | 规模可控 | 局部重试粒度变粗 |
| 动态修改原 definition | 看似简单 | 审计和版本混乱 |
| runtime graph snapshot | 可追溯 | 存储和 UI 更复杂 |

成熟表达:

```text
动态 DAG 不是修改 workflow definition,而是在某次 workflow instance 里追加 runtime vertices 和 runtime dependencies。
```

## 第八层: 可靠执行

工作流引擎最难的是:

```text
任务状态和外部副作用之间不可能天然原子。
```

例如:

```text
外部 API 调用成功,DB 状态更新失败。
DB 状态更新成功,MQ 消息发送失败。
worker 超时,外部 job 其实成功。
callback 重复回来。
```

所以现实目标不是 exactly once。

更现实的是:

```text
at-least-once execution + idempotency + reconciliation = effectively-once business effect
```

### 可靠执行工具箱

```text
幂等 key
CAS 状态更新
lease
heartbeat
outbox
inbox
retry policy
external_call_log
reconciliation job
output contract
audit trail
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 简单 retry | 易实现 | 容易重复副作用 |
| 幂等 key | 防重复执行 | 需要外部系统配合 |
| outbox/inbox | 消息一致性可恢复 | 多表、多后台任务 |
| lease | worker 挂了可恢复 | lease 过期和外部副作用要协调 |
| 强事务 | 语义简单 | 跨外部系统通常做不到 |
| 最终一致 | 可落地 | 需要 reconcile 和观测 |

面试里要避免说:

```text
我们保证 exactly once。
```

更稳的是:

```text
平台层通常只能保证至少一次调度,业务层通过幂等键、唯一约束和对账实现效果上的 exactly once。
```

## 第九层: 数据和上下文

任务之间不只是有顺序依赖,还有数据依赖。

要区分:

```text
结构依赖: D 等 A/B/C 完成。
数据依赖: D 消费 A/B/C 的具体输出。
```

这两个不能混。

一个节点 `COMPLETE` 应该意味着:

```text
下游依赖的事实已经成立。
```

包括:

```text
外部 job terminal success
output_ref 已写入
manifest 可读
schema 校验通过
必要的 marker file 存在
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 参数直接放 message | 简单 | 大对象和敏感信息风险高 |
| output_ref | 轻量,适合大数据 | 需要产物管理 |
| manifest | 可审计、可分页、可校验 | 多一层产物协议 |
| 上下文全局可变 | 使用方便 | 并发和重跑不确定 |
| 上下文快照化 | 可复现、可审计 | 存储和解析更复杂 |

核心原则:

```text
调度系统保存元数据和引用,不要承载大业务数据本体。
```

## 第十层: 可观测性和运维

生产用户最关心:

```text
为什么没跑?
为什么失败?
为什么跑慢?
谁点了重跑?
上游输出是什么?
外部系统返回了什么?
```

所以可观测性不是附属功能。

至少要有:

```text
workflow_instance
task_instance
task_execution_attempt
task_state_transition
workflow_operation_audit
external_call_log
message_lineage
metrics
alerts
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 只存当前状态 | 简单 | 无法排查 |
| 状态变更历史 | 证据链完整 | 写入量大 |
| 日志集中存储 | 易搜索 | 成本和权限要治理 |
| UI 展示全 DAG | 直观 | 大 DAG 会卡 |
| UI 聚合 fan-out | 可扩展 | 细节需要 drill down |

成熟系统要接受:

```text
没有证据链,就没有生产可用性。
```

## 第十一层: 规模化和多租户

生产规模上来后,最常见的问题不是 DAG 算法,而是:

```text
DB 热表
scheduler 扫描
worker capacity
资源池限流
租户公平性
动态 fan-out 爆炸
外部系统被打爆
日志写入量
UI 大图渲染
```

所以执行前要判断:

```text
依赖 ready
资源 ready
租户 quota ready
优先级 ready
worker ready
版本兼容 ready
```

### 取舍

| 设计 | 优点 | 代价 |
|---|---|---|
| 全局队列 | 简单 | 大租户会堵住小租户 |
| 租户队列 | 公平 | 调度复杂 |
| 高优先级优先 | SLA 好 | 低优先级饥饿 |
| priority aging | 防饥饿 | 排序复杂 |
| 每个节点独立 task | 粒度细 | 规模爆炸 |
| chunking | 规模可控 | 局部失败粒度变粗 |

一句话:

```text
ready 是逻辑条件,resource quota 是物理条件。
```

## 第十二层: 技术选型

如果从零设计,要决定很多底层技术。

### 状态存储

| 选项 | 优点 | 代价 |
|---|---|---|
| 关系型数据库 | 事务、查询、约束强 | 高并发扫描和写入要优化 |
| NoSQL | 扩展性强 | 事务和复杂查询弱 |
| 纯内存 | 快 | 不可恢复 |

生产级工作流引擎通常需要:

```text
强状态一致性 + 查询能力 + 审计能力。
```

所以核心状态更适合关系型数据库。

### 消息系统

| 选项 | 优点 | 代价 |
|---|---|---|
| Service Bus / PubSub / MQ | 解耦、削峰、异步 | 至少一次投递,要幂等 |
| DB polling | 简单可恢复 | 延迟和 DB 压力 |
| 事件流 Kafka | 高吞吐、可回放 | 运维和消费语义复杂 |

EMF 的消息驱动设计适合:

```text
外部事件进入 + worker 异步消费 + 动态继续 enqueue。
```

但必须配套:

```text
幂等消费
outbox/inbox
重复消息处理
```

### 调度器

| 选项 | 优点 | 代价 |
|---|---|---|
| 单 scheduler | 简单 | 单点 |
| 多 scheduler + DB lock/CAS | HA | 并发控制复杂 |
| 分片 scheduler | 扩展性好 | 分片迁移和均衡复杂 |

生产上最好支持:

```text
多 scheduler + CAS/lease + 补偿扫描。
```

### Worker 执行

| 选项 | 优点 | 代价 |
|---|---|---|
| 进程内执行 | 快、简单 | 隔离弱 |
| 外部 job | 隔离好 | monitor 复杂 |
| 容器执行 | 隔离强 | 成本和延迟高 |

EMF 这类内部平台通常可以混合:

```text
轻量 command 进程内执行。
重任务提交外部 executor。
用户自定义任务容器化。
```

## 和成熟系统的选型 trade-off

如果比较 Airflow、Temporal、Argo:

```text
Airflow 偏数据 DAG 批调度。
Temporal 偏 durable workflow / 长事务。
Argo 偏 Kubernetes 容器工作流。
EMF 偏消息驱动内部命令编排和动态 DAG。
```

所以选型要看:

```text
任务是不是周期性数据 pipeline?
流程是不是长事务?
任务是不是容器?
是否需要运行时动态生成任务?
是否需要和内部 command/service 深度集成?
是否需要 durable replay?
是否主要依赖 Kubernetes?
```

EMF 类自研系统的优势:

```text
贴合内部业务命令体系。
可以把 Service Bus / PubSub 事件自然变成任务。
可以运行时根据业务结果扩展 DAG。
可以深度集成内部 Databricks/Storage/Catalogue/API。
```

代价:

```text
成熟工作流平台已有的可靠性、UI、审计、资源治理、版本兼容、生态能力需要自己建设。
```

## 自研 vs 采用成熟平台

### 适合自研的情况

```text
内部已有强 command/service 体系
workflow 深度绑定内部平台
任务需要运行时动态生成
成熟平台难以表达业务语义
需要定制权限、审计、参数和资源治理
团队有能力长期维护平台
```

### 不适合自研的情况

```text
只是标准数据 pipeline
只是容器批处理
只是长事务编排
团队没有平台维护能力
可观测性、HA、幂等、权限都没资源建设
```

面试里可以这样说:

```text
自研工作流引擎的理由不能是“我们也能写一个 DAG runner”,而应该是成熟平台的核心抽象和内部业务模型不匹配,同时团队愿意承担可靠性和运维治理成本。
```

## 端到端执行链路

可以把一次 workflow run 的完整链路讲成:

```text
1. 用户/API/OE-RUN/外部事件触发 workflow。
2. 系统解析 workflow definition 或外部参数。
3. 生成一批 Message,每条 Message 有 run_uuid、order_id、parents、command。
4. Message 被加入 DagManager 对应的 DagContext。
5. DagContext 根据父子依赖和状态判断 ready 节点。
6. ParallelMessageConsumer 在并发、优先级、资源池限制下领取 ready 节点。
7. MessageHandler 打开 message scope。
8. CommandExecutor 根据 command 找 module。
9. ParameterParser 解析 message/run/batch/upstream 参数。
10. module 调用 command/service 做实际业务。
11. 业务执行成功后写 output_ref、状态、attempt、transition。
12. command 可以继续 enqueue 新 Message,动态扩展 DAG。
13. 下游节点依赖满足后继续执行。
14. 直到所有节点进入终态,workflow run 完成。
```

生产级版本还要在每一步加:

```text
幂等
CAS
lease
retry
timeout
outbox
audit
metrics
reconcile
```

## 面试中的讲法

### 一分钟版本

```text
我会把这个系统理解成消息驱动的动态 DAG 工作流引擎。它用 Message 表达每个任务节点,通过 run_uuid 把一次 workflow run 分组,通过 order_id 和 parents 推导 DAG,调度器只执行依赖满足的 ready 节点。执行层通过 command 找到对应 module 和 service 做业务动作,业务动作还能继续 enqueue 新 Message,所以支持动态 DAG 和子 workflow。生产级难点不在拓扑排序,而在状态机、幂等、重试、超时、外部副作用、人工干预、可观测性和资源治理。
```

### 三分钟版本

```text
从架构上我会拆成控制平面和执行平面。控制平面负责 workflow/message/DAG/状态机/调度/重试/超时/审计,执行平面负责 command/module/service 和外部系统调用。核心数据模型是 Message,它既是任务实例,也是 DAG 节点,包含 run_uuid、batch_id、order_id、parents、command 和 parameters。DagContext 根据父节点状态和 trigger rule 判断 ready,ParallelMessageConsumer 在并发和资源约束下执行 ready 节点。CommandExecutor 根据 command 找 module,解析参数并调用业务服务。为了生产可靠性,任务领取要用 CAS/lease,外部副作用要有幂等 key,状态变更和事件发布要用 outbox/inbox,worker 宕机要靠 heartbeat 和 reconcile 恢复。动态 DAG 通过运行时生成新 Message 实现,但要控制 fan-out、环依赖、重复生成和 completion token。
```

### 十分钟版本的结构

可以按这个顺序讲:

```text
1. 需求: 依赖调度、动态任务、外部系统编排、可恢复可审计。
2. 领域模型: definition/instance、Message、run_uuid、order_id、parents。
3. 架构分层: API/OE-RUN、DagManager、Consumer、CommandExecutor、Module/Service。
4. 调度: ready 判断、trigger rule、状态机、并发和资源池。
5. 可靠性: CAS、lease、heartbeat、retry、timeout、idempotency。
6. 动态 DAG: fan-out/fan-in、子 workflow、completion token、runtime snapshot。
7. 数据: output_ref、manifest、context snapshot、secret。
8. 可观测性: attempt、transition、audit、metrics、trace。
9. 规模化: 多 scheduler、多租户、公平调度、DB 索引、限流。
10. 选型取舍: 与 Airflow、Temporal、Argo 的边界。
```

## 最高频追问的回答方向

### Q1: 这个系统本质是什么?

```text
不是简单 DAG runner,而是分布式状态机驱动的工作流平台。
```

### Q2: 为什么用 Message?

```text
因为 Message 同时承载任务实例、命令、参数和 DAG 节点关系,适合 Service Bus/PubSub 这种消息驱动模型,也适合运行时继续生成新任务。
```

### Q3: 为什么调度和命令执行要拆开?

```text
调度层只关心依赖、状态和资源;命令层只关心业务动作。这样新 command 可以插件化接入,调度治理也不会散落到业务代码里。
```

### Q4: 动态 DAG 最大风险是什么?

```text
不是生成节点,而是生成后还能保证图合法、幂等、不爆炸、可追踪、run completion 正确。
```

### Q5: 怎么保证任务不重复执行?

```text
平台层很难保证 exactly once,所以通常用至少一次调度 + 幂等 key + CAS + 外部 request_id + reconciliation,实现业务效果上的 exactly once。
```

### Q6: 如果 worker 挂了怎么办?

```text
任务领取时写 lease 和 worker_id,worker heartbeat 续租。lease 过期后 scanner 判断是否 LOST/RETRYING/FAILED。外部任务还要先查询 external job 状态,不能盲目重跑。
```

### Q7: 如果工作流很大怎么办?

```text
ready 判断要增量化,fan-out 要限流和 chunking,调度要分页 claim,资源要按租户和类型限流,UI 要聚合展示,历史数据要归档。
```

## 最终心智模型

可以把整个系统压缩成五句话:

```text
1. Message 是任务和 DAG 节点的统一抽象。
2. run_uuid 划定一张运行时 DAG,order_id + parents 定义边。
3. DagContext 管状态和依赖,Consumer 只执行 ready 节点。
4. CommandExecutor 把调度和业务解耦,业务执行可以继续生成 Message。
5. 生产级能力来自状态机、幂等、lease、outbox、reconcile、审计和资源治理。
```

最关键的一句话:

```text
工作流引擎的设计不是把任务排个序,而是在不可靠的分布式环境里,持续、可恢复、可审计地推进一组有依赖关系的业务状态机。
```
