# 28 高频追问 Q&A

## 这章怎么用

这一章不是从头讲设计,而是模拟面试里的追问。

每个问题都按这个结构回答:

```text
核心回答
展开要点
一句话收束
```

目标是让你在面试里做到:

```text
先给结论,再讲原理,最后落到 trade-off。
```

## Q1: 为什么不用简单队列?

### 核心回答

简单队列只能表达:

```text
有一批任务需要执行。
```

但工作流引擎要表达:

```text
任务之间有依赖、有状态、有重试、有失败传播、有动态扩展、有人工干预。
```

所以简单队列只能解决“异步执行”,解决不了“依赖编排”。

### 展开要点

简单队列模型通常是:

```text
producer -> queue -> consumer
```

它擅长:

```text
削峰
异步
解耦
水平扩展消费
```

但 DAG 工作流需要回答:

```text
B 能不能跑,取决于 A 有没有完成。
D 能不能跑,取决于 A/B/C 的 trigger rule。
父节点失败后,下游是跳过、失败、还是继续。
某个节点 retry 后,下游旧结果要不要失效。
workflow 暂停时,ready 任务能不能继续领取。
一个 command 动态生成新节点后,run 什么时候算完成。
```

这些不是普通队列能直接表达的。

普通队列也不负责:

```text
DAG 合法性校验
任务状态机
父子依赖判断
运行历史
人工操作审计
外部副作用幂等
```

所以 EMF 这类系统虽然底层可能用 Service Bus / PubSub,但在队列之上必须有:

```text
Message 数据模型
DagContext
状态机
ready 判断
重试和超时治理
审计与可观测性
```

### 一句话收束

```text
队列解决“任务怎么异步分发”,工作流引擎解决“任务在什么依赖、状态和治理规则下推进”。
```

## Q2: 为什么需要 DAG,不能在代码里 if/else 顺序调用?

### 核心回答

代码顺序调用适合简单流程,但生产工作流需要:

```text
可视化
可重试
可跳过
可恢复
可审计
可并行
可动态扩展
```

DAG 把流程从代码调用链变成显式的依赖图,调度器才能独立管理状态和恢复。

### 展开要点

如果写成:

```text
run A
run B
run C
run D
```

问题是:

```text
B 失败后怎么只重跑 B/D?
C 已经成功要不要复用?
系统重启后从哪里恢复?
哪个节点卡住了?
谁手动跳过了哪个节点?
A 和 C 能不能并行?
```

DAG 模型让这些问题变成:

```text
每个节点有状态。
每条边有依赖语义。
调度器可以只推进 ready 节点。
失败可以局部处理。
运行历史可以记录。
```

### 一句话收束

```text
DAG 的价值不是让任务“能按顺序跑”,而是让流程变成可调度、可恢复、可观察的状态图。
```

## Q3: 为什么需要版本化?

### 核心回答

因为 workflow definition 会变,但已经运行的 workflow instance 必须可复现、可审计。

如果没有版本化,用户修改 DAG 后,历史 run 和运行中的 run 会被新定义污染。

### 展开要点

假设今天的 DAG 是:

```text
A -> B -> C
```

明天改成:

```text
A -> B -> D -> C
```

必须回答:

```text
昨天还没跑完的 instance 按旧 DAG 还是新 DAG?
昨天失败的 C 重跑时用旧定义还是新定义?
审计时怎么知道当时跑的是哪版?
用户说结果不一致,怎么复现?
```

正确做法:

```text
workflow_definition 是逻辑对象。
workflow_definition_version 是不可变快照。
workflow_instance 绑定 definition_version_id。
运行时动态生成的节点记录在 runtime graph snapshot,不反向修改 definition。
```

### 一句话收束

```text
版本化的目的不是管理文件,而是保证每次 workflow run 都绑定一个可复现、可审计的流程快照。
```

## Q4: 怎么保证 exactly-once?

### 核心回答

严格意义上的 exactly-once 很难跨 DB、MQ、worker 和外部系统保证。

更现实的设计是:

```text
at-least-once 调度 + 幂等 key + CAS + 唯一约束 + 外部对账 = 业务效果上的 exactly-once。
```

### 展开要点

典型问题:

```text
worker 调外部系统成功,但更新 DB 状态失败。
worker 重启后任务被重新执行。
```

如果任务是:

```text
发券
扣款
写库存
创建审批单
```

就可能产生重复副作用。

解决方式:

```text
每个任务生成稳定 idempotency_key。
外部请求携带 request_id。
外部系统用唯一约束或幂等接口去重。
平台记录 external_call_log。
状态更新用 CAS 防止旧 attempt 覆盖新 attempt。
callback 用 inbox 去重。
异常时 reconcile 外部状态。
```

幂等 key 通常可以是:

```text
workflow_instance_id + task_key
```

在 EMF 语境下可以理解为:

```text
run_uuid + order_id
```

如果同一个 task 多次 attempt,要区分:

```text
business_idempotency_key: 防重复副作用,不随 attempt 变化
attempt_request_id: 排查和 trace,可以随 attempt 变化
```

### 一句话收束

```text
平台层很难承诺物理 exactly-once,但可以通过幂等和对账实现业务效果上的 exactly-once。
```

## Q5: 为什么需要状态机,不能只存 success/failed?

### 核心回答

因为生产里的任务不只有成功和失败,还会有:

```text
等待执行
正在执行
提交外部 job
等待外部回调
重试中
超时
取消中
跳过
等待审批
```

如果状态太粗,系统不知道下一步该做什么,排障也没有证据。

### 展开要点

至少要区分:

```text
CREATED
READY
RUNNING
COMPLETE
FAILED
RETRYING
TIMEOUT
CANCELLED
SKIPPED
WAITING_APPROVAL
```

对于外部任务,最好进一步区分:

```text
SUBMITTED
RUNNING_EXTERNAL
OUTPUT_VALIDATING
```

否则会出现:

```text
外部 job 只是 submit 成功,平台就标 COMPLETE。
下游提前开始执行。
```

状态机还要限制非法流转:

```text
COMPLETE 不能被普通事件改回 RUNNING。
旧 attempt 不能覆盖新 attempt。
CANCELLED 不能被 scheduler 再次领取。
```

### 一句话收束

```text
状态机不是为了多几个枚举,而是为了让系统知道每个任务现在处于什么语义阶段、下一步允许做什么。
```

## Q6: 为什么需要 `run_uuid` 和 `batch_id` 两个概念?

### 核心回答

`run_uuid` 管一张具体运行时 DAG,`batch_id` 管一批 run 的分组和治理。

二者解决的问题不同。

### 展开要点

`run_uuid` 用来回答:

```text
这条 message 属于哪一次 workflow run?
这张 DAG 里有哪些节点?
这个节点依赖哪些父节点?
这个 run 是否完成?
```

`batch_id` 用来回答:

```text
这一批 workflow run 属于哪个批次?
这个 batch 是否整体完成?
这个 batch 的优先级是什么?
是否要批量取消、重跑、清理?
```

如果只有 `batch_id`,同一批里多张 DAG 会混在一起。

如果只有 `run_uuid`,批次级治理会困难。

### 一句话收束

```text
run_uuid 是单次 DAG 的边界,batch_id 是批次治理的边界。
```

## Q7: 为什么 DAG 用 `order_id + parents`,而不是用 `msg_id`?

### 核心回答

`msg_id` 是消息实例 ID,通常每次生成都不同；`order_id` 是业务任务节点 ID,可以稳定表达依赖关系。

DAG 依赖应该基于稳定业务节点,而不是瞬时消息实例。

### 展开要点

如果用 `msg_id` 建依赖:

```text
retry 生成新 message 时 msg_id 变了。
动态生成时父子关系很难稳定复用。
审计时很难看出业务节点关系。
```

用 `order_id + parents`:

```text
order_id 表示当前业务节点是谁。
parents 表示依赖哪些父业务节点。
run_uuid 表示在哪张 DAG 里。
```

所以同一个节点 retry 时可以有新 message 或新 attempt,但业务节点身份仍然稳定。

### 一句话收束

```text
msg_id 适合标识消息实例,order_id 适合标识 DAG 里的业务节点。
```

## Q8: 为什么调度层和命令层要拆开?

### 核心回答

调度层负责:

```text
谁可以跑。
```

命令层负责:

```text
跑什么业务。
```

拆开后,调度治理和业务逻辑不会互相污染。

### 展开要点

调度层关心:

```text
DAG 依赖
状态机
并发
优先级
资源池
重试
超时
取消
审计
```

命令层关心:

```text
command 映射哪个 module
参数怎么解析
调用哪个 service
产出什么 output_ref
是否继续 enqueue 新 message
```

如果不拆:

```text
每个业务模块都要自己处理重试、日志、状态和依赖。
调度器会知道太多业务细节。
新命令接入成本高。
```

### 一句话收束

```text
调度层处理通用流程语义,命令层处理具体业务语义。
```

## Q9: 为什么需要动态 DAG?

### 核心回答

因为很多 workflow 在运行前不知道完整任务集合。

任务数量和依赖关系可能来自:

```text
query 结果
catalogue resolution
配置展开
子 workflow
外部事件
上游任务输出
```

### 展开要点

典型场景:

```text
先查出 100 个 dataset,再为每个 dataset 生成处理任务。
先做 catalogue resolution,再生成真正需要执行的 Storage/Databricks/API 任务。
父 workflow 调用子 workflow,子 workflow 展开出一组 message。
```

动态 DAG 的设计重点:

```text
新增节点要有稳定 order_id。
新增边要做环检测。
fan-out 要限流。
重复生成要幂等。
run completion 要等动态节点完成。
要记录 generated_by 和 lineage。
```

### 一句话收束

```text
动态 DAG 解决的是运行前任务图不完全确定的问题,代价是图治理和可观测性更复杂。
```

## Q10: 动态 fan-out 怎么保证后续节点不提前执行?

### 核心回答

要引入 fan-out group 或 completion token,让 join 节点等动态生成的子任务全部进入终态后再判断 ready。

### 展开要点

错误做法:

```text
父节点生成 A/B/C 后立刻标 COMPLETE,下游 D 只看父节点完成就执行。
```

正确做法:

```text
父节点创建 fanout_group。
记录 expected_count / created_count。
动态 child 绑定 fanout_group。
completion token 依赖所有 child terminal。
join 节点依赖 completion token。
```

这样:

```text
父 command 完成生成动作 != 子任务全部完成。
```

### 一句话收束

```text
动态 fan-out 的 join 不能只等生成者完成,而要等生成出来的那批任务完成。
```

## Q11: 如果 A/B/C 之间有数据依赖,但 D 依赖 A/B/C,怎么处理?

### 核心回答

要把真实数据依赖建进 DAG。

如果 B 依赖 A,C 依赖 B,就不能把 A/B/C 当成纯并行父节点。

### 展开要点

错误建模:

```text
A -> D
B -> D
C -> D
```

如果真实依赖是:

```text
A -> B -> C
```

正确建模应是:

```text
A -> B -> C -> D
```

如果 D 还要直接消费 A/B 的输出,可以再加:

```text
A -> D
B -> D
```

但边必须表达真实执行和数据依赖。

否则调度器可能提前执行 B/C/D。

### 一句话收束

```text
DAG 的边不能只画业务汇总关系,必须表达真实的执行依赖和数据依赖。
```

## Q12: D 节点调用外部服务,服务宕机怎么办?

### 核心回答

要区分错误类型,对可恢复错误重试,对不可恢复错误失败或人工介入,并用幂等和熔断保护外部系统。

### 展开要点

处理流程:

```text
设置 timeout。
捕获错误类型。
网络抖动、5xx、限流可以 retry。
4xx、参数错误、权限错误通常不 retry。
重试使用指数退避和 jitter。
达到最大次数后进入 FAILED 或 WAITING_MANUAL。
记录 external_call_log 和 trace_id。
如果外部系统部分成功,要 reconcile。
```

还要避免 retry storm:

```text
下游服务宕机时,大量任务同时重试会把它打得更惨。
```

所以需要:

```text
circuit breaker
rate limit
resource pool
backoff
告警
```

### 一句话收束

```text
外部服务宕机不是简单重试,而是错误分类、退避、幂等、熔断和人工兜底的组合。
```

## Q13: B 还没执行完却被标记完成,导致 D 提前执行,怎么办?

### 核心回答

这是“状态语义不准确”的问题。

`COMPLETE` 必须代表下游依赖的事实已经成立,不能只代表请求提交成功或函数返回。

### 展开要点

常见原因:

```text
submit external job 成功就标 COMPLETE。
异步任务未完成但 worker 提前返回。
callback 把 RUNNING 当成 SUCCESS。
人工误标成功。
旧 attempt 覆盖新 attempt。
```

设计解决:

```text
拆分 SUBMITTED / RUNNING / OUTPUT_VALIDATING / COMPLETE。
外部 job terminal success 后才能 COMPLETE。
COMPLETE 前校验 output_ref / marker / manifest。
状态更新带 attempt_id 和 version CAS。
旧 attempt 不能覆盖新 attempt。
manual success 要提供 reason 和 output contract。
D ready 前校验父节点 output contract。
```

### 一句话收束

```text
complete 不是“我把活发出去了”,而是“下游可以安全依赖的结果已经成立”。
```

## Q14: 为什么需要 outbox/inbox?

### 核心回答

因为 DB 状态更新和 MQ 消息发送不是原子操作。

outbox 保证状态变化对应的事件不会丢,inbox 保证重复事件不会被重复处理。

### 展开要点

典型问题:

```text
父节点 COMPLETE 已写 DB,但通知下游的 MQ 消息发送失败。
```

如果没有 outbox,下游可能永远不跑。

outbox 做法:

```text
同一个 DB transaction 里:
  更新 task 状态
  写 transition
  写 outbox_event

后台 publisher:
  读取未发送事件
  发送 MQ
  标记 sent
```

inbox 用来处理:

```text
MQ 重复投递
publisher 重复发送
callback 重复到达
```

### 一句话收束

```text
outbox 解决事件不丢,inbox 解决事件重复。
```

## Q15: 多个 worker 怎么避免抢同一个任务?

### 核心回答

任务领取必须是原子状态转换,通常靠 DB 锁、CAS 或 lease。

不能先 select 再 update。

### 展开要点

错误方式:

```text
SELECT READY tasks
worker A 和 worker B 都看到同一个 task
两个 worker 都执行
```

正确方式:

```text
UPDATE task_instance
SET status = RUNNING,
    worker_id = ?,
    lease_until = ?
WHERE id = ?
  AND status = READY
  AND version = ?
```

只有 update 成功的 worker 拿到任务。

还可以用:

```text
SELECT FOR UPDATE SKIP LOCKED
```

领取后用 lease 防止 worker 挂死。

### 一句话收束

```text
任务领取不是读操作,而是一个带条件的状态转移。
```

## Q16: 为什么需要可观测性,日志不够吗?

### 核心回答

日志只记录“执行时说了什么”,但排障还需要状态证据、操作证据和外部调用证据。

### 展开要点

生产排障要回答:

```text
为什么没跑?
为什么失败?
谁点了重跑?
从哪个状态变成哪个状态?
外部 job id 是什么?
用了什么参数?
上游输出是什么?
```

所以要有:

```text
task_state_transition
task_execution_attempt
workflow_operation_audit
external_call_log
resolved_parameters
output_ref
metrics
trace_id
```

只有日志会有问题:

```text
日志丢失。
日志没有状态变更。
日志没有人工操作。
日志不适合做聚合指标。
```

### 一句话收束

```text
日志是证据链的一部分,不是完整的可观测性。
```

## Q17: DAG 有环怎么办?

### 核心回答

DAG 有环就不是 DAG,必须在定义提交和动态节点生成时拦截。

### 展开要点

环依赖例子:

```text
A -> B -> C -> A
```

后果:

```text
没有节点能 ready。
run 永远卡住。
调度器可能不断扫描但无法推进。
```

检测方式:

```text
拓扑排序
DFS visiting/visited
入度表
动态新增边时做局部 cycle detection
```

动态 DAG 更要注意:

```text
运行时新增边也可能引入环。
不能只在 workflow 提交时检查。
```

### 一句话收束

```text
环依赖不是运行时失败,而是 workflow 定义或动态扩展阶段就必须拒绝的建模错误。
```

## Q18: 工作流很大、数据量很大,怎么优化?

### 核心回答

要从任务规模、调度扫描、资源池、数据传递和 UI 展示一起优化。

### 展开要点

优化方向:

```text
fan-out limit
chunking / partition
run 级并发限制
租户 quota
资源池限流
ready 判断增量化
task_instance 索引
事件驱动 + 补偿扫描
大对象用 output_ref / manifest
UI 聚合 fanout group
历史数据归档
```

不要简单说:

```text
加 worker。
```

如果瓶颈在 DB 扫描或外部系统限流,加 worker 反而会恶化。

### 一句话收束

```text
大规模优化不是单点扩容,而是控制任务粒度、调度压力、资源配额和数据搬运成本。
```

## Q19: 为什么需要人工操作审计?

### 核心回答

因为生产 workflow 一定会有人手动暂停、取消、重跑、跳过、标成功。

这些操作会改变状态机,必须可追溯。

### 展开要点

审计至少记录:

```text
operator
operation
target
old_status
new_status
reason
timestamp
request_id
affected_tasks
```

否则事故后无法回答:

```text
这个任务为什么被跳过?
谁把它标成功了?
为什么下游突然开始跑?
取消有没有发到外部系统?
```

### 一句话收束

```text
人工操作不是状态机的后门,而是必须被建模和审计的一类事件。
```

## Q20: 怎么回答“这个系统最大的难点是什么”?

### 核心回答

最大难点不是拓扑排序,而是生产环境下的可靠状态推进。

### 展开要点

可以按五类讲:

```text
1. 状态机: 状态流转必须受控,不能乱跳。
2. 幂等: 任务可能重复执行,外部副作用要防重复。
3. 一致性: DB、MQ、worker、外部系统会不一致,要 outbox/reconcile。
4. 动态 DAG: 运行时生成节点要防环、防重复、防爆炸、可观测。
5. 运维治理: 日志、审计、资源池、多租户、权限、升级兼容。
```

### 一句话收束

```text
DAG 引擎的难点不是把图跑起来,而是让这张图在失败、并发和人工干预下仍然可控、可恢复、可审计。
```

## Q21: 怎么把 EMF 和 Airflow / Temporal / Argo 区分开?

### 核心回答

它们都是工作流系统,但核心范式不同。

### 展开要点

```text
Airflow 偏数据 pipeline DAG 调度。
Temporal 偏 durable workflow / 长事务。
Argo 偏 Kubernetes 容器工作流。
EMF 偏消息驱动的内部命令 DAG 编排。
```

EMF 的特点:

```text
任务是 Message。
DAG 由 run_uuid/order_id/parents 推导。
执行由 command -> module -> service 完成。
业务 command 可以继续 enqueue 新 Message。
适合内部平台服务和动态任务编排。
```

### 一句话收束

```text
选型不是比较谁更强,而是看任务抽象、状态恢复模型、执行环境和业务集成方式是否匹配。
```

## 最后可以背的总回答

如果被连续追问,可以用这段收束:

```text
我理解工作流引擎的核心不是简单队列或拓扑排序,而是可靠推进有依赖的任务状态机。简单队列只能异步分发任务,但不能表达 DAG 依赖、状态机、失败传播、动态扩展和人工干预。生产设计里要把 definition 和 instance 分离,每次 run 绑定不可变版本;任务用 Message 表达,通过 run_uuid、order_id、parents 建 DAG;调度器只执行 ready 节点,命令层负责具体业务。可靠性上不能承诺物理 exactly-once,而是用 at-least-once 调度、幂等 key、CAS、lease、outbox/inbox 和 reconciliation 实现业务效果上的 exactly-once。真正的难点在动态 DAG、外部副作用、状态一致性、可观测性和资源治理。
```
