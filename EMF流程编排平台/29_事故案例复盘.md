# 29 事故案例复盘

## 这章的定位

这一章把工作流引擎的典型生产事故整理成复盘模板。

每个事故按这个结构:

```text
事故现象
直接影响
常见根因
排查路径
设计修复
面试表达
```

面试里讲事故案例很有价值,因为它能证明你不是只懂概念,也知道系统进生产后会怎么坏。

## 复盘总原则

工作流引擎事故通常不是单点 bug,而是这些东西叠加:

```text
状态语义不清
幂等缺失
并发竞争
外部系统不确定
动态 DAG 治理不足
可观测性缺口
人工操作绕过状态机
```

复盘时不要只说:

```text
某个代码写错了。
```

而要追问:

```text
为什么系统允许它错?
为什么没有提前拦截?
为什么没有指标告警?
为什么没有补偿收敛?
为什么排查证据不够?
```

## 事故 1: 任务重复执行

### 事故现象

同一个业务任务被执行了两次。

例如:

```text
同一个用户收到两张优惠券。
同一个外部审批单创建了两次。
同一份数据被重复写入。
同一个 Databricks job 被提交两次。
同一个通知发了两遍。
```

在 EMF 语境里可能表现为:

```text
同一个 run_uuid + order_id 对应的业务动作执行了多次。
```

### 直接影响

```text
重复副作用
数据重复
外部系统污染
用户体验问题
财务或合规风险
下游结果不可解释
```

### 常见根因

#### 1. 外部执行成功,状态更新失败

流程:

```text
worker 调外部系统成功
worker 更新 task COMPLETE 失败
任务 lease 过期或 worker 重启
系统重新调度同一任务
外部动作再次执行
```

#### 2. MQ 重复投递

```text
同一条 message 被 consumer 收到两次。
```

如果 consumer 没有幂等处理,会重复执行。

#### 3. 多 worker 抢同一任务

```text
worker A 和 worker B 同时看到 READY。
两者都执行。
```

通常是因为:

```text
先 SELECT 再 UPDATE。
没有 CAS。
没有锁。
没有 lease token。
```

#### 4. retry 生成新 message,但没有稳定幂等 key

如果 retry 只是生成一条新 message,但业务侧无法识别它和旧 attempt 是同一业务任务,就会重复执行。

#### 5. 动态 DAG 重复生成 child

父任务 retry 后再次生成同一批子任务:

```text
A 第一次生成 B/C/D。
A retry 后又生成 B/C/D。
```

如果没有唯一约束,就会出现重复节点。

### 排查路径

先定位:

```text
重复的是 message,task attempt,还是外部副作用?
```

检查:

```text
同一个 run_uuid + order_id 是否有多个 task/message。
同一个 task 是否有多个 attempt。
多个 attempt 的 worker_id 和 lease_token。
state_transition 是否有重复 READY/RUNNING。
external_call_log 是否有多个 request_id。
外部系统是否收到相同 idempotency_key。
是否发生 worker crash / lease expired。
outbox/inbox 是否重复处理。
```

关键证据:

```text
task_instance
task_execution_attempt
task_state_transition
external_call_log
message_lineage
operation_audit
```

### 设计修复

#### 1. 任务领取用 CAS / lease

```text
只有 status = READY 且 version 匹配的任务才能被领取。
领取成功后写 worker_id、lease_until、lease_token。
```

#### 2. 业务幂等 key

稳定幂等 key:

```text
workflow_instance_id + task_key
```

或 EMF 语境:

```text
run_uuid + order_id
```

这个 key 不应该随 attempt 改变。

#### 3. 外部系统幂等

所有有副作用的外部调用携带:

```text
idempotency_key
correlation_id
```

外部系统用唯一约束去重。

#### 4. 动态节点唯一约束

```text
unique(run_uuid, order_id)
unique(fanout_group_id, child_key)
```

#### 5. callback / message inbox 去重

重复事件不能重复推进状态。

### 面试表达

```text
任务重复执行通常不是队列重复这么简单,而是 at-least-once 调度和外部副作用之间没有幂等边界。我的设计会把 task identity 和 attempt identity 分开,用 run_uuid + order_id 做业务幂等 key,attempt_id 做执行追踪;任务领取用 CAS 和 lease,外部调用携带 idempotency_key,callback 用 inbox 去重,动态生成子任务用唯一约束保证重复生成不会产生重复 child。这样即使 worker crash 或 MQ 重复投递,也最多重复调度,不会重复产生业务副作用。
```

## 事故 2: 节点提前完成,下游提前执行

### 事故现象

一个节点实际还没完成,却被标记为 `COMPLETE`,导致下游节点开始执行。

典型例子:

```text
B 提交外部 job 成功后立刻 COMPLETE。
D 依赖 B,于是 D 开始读取 B 的输出。
但 B 的外部 job 还在跑,output_ref 不存在。
```

### 直接影响

```text
下游读取不到数据
下游读取旧数据
下游用不完整数据计算
workflow 最终结果错误
问题隐蔽,不一定立刻失败
```

### 常见根因

#### 1. 把 submit success 当成 task success

```text
提交 Databricks/Spark/SQL job 成功 != job 执行成功。
```

#### 2. 异步任务提前返回

worker 开始异步操作后没有等待 terminal state。

#### 3. callback 语义错误

收到:

```text
job started
```

却当作:

```text
job succeeded
```

#### 4. output contract 缺失

系统只看状态,不检查:

```text
output_ref
manifest
marker file
row count
schema
```

#### 5. 人工误标成功

管理员手动把失败或运行中的节点标成成功,但没有提供产物。

### 排查路径

检查:

```text
B 是谁标记 COMPLETE 的?
state_transition 的 reason/source 是什么?
B 的 attempt 是否有 external_job_id?
external job 最终状态是什么?
B 的 output_ref 是否存在?
D 的 resolved_parameters 引用了什么?
D 开始执行时间是否早于 B 外部完成时间?
是否有 manual mark success 审计?
```

关键时间线:

```text
B submitted_at
B external_started_at
B external_finished_at
B marked_complete_at
D started_at
```

如果:

```text
B marked_complete_at < B external_finished_at
```

基本就是提前完成。

### 设计修复

#### 1. 拆分状态

不要只有:

```text
RUNNING -> COMPLETE
```

而是:

```text
RUNNING -> SUBMITTED -> RUNNING_EXTERNAL -> OUTPUT_VALIDATING -> COMPLETE
```

#### 2. submit + monitor 分离

```text
SubmitJob -> MonitorJob -> Downstream
```

`MonitorJob` 负责确认外部任务终态和输出产物。

#### 3. COMPLETE 前做 output contract 校验

必须满足:

```text
external terminal success
output_ref exists
manifest valid
schema valid
marker exists
```

#### 4. 下游 ready 前校验父节点输出

对于有数据依赖的边,不仅检查父状态,还检查父 output contract。

#### 5. 人工标成功加限制

要求:

```text
reason
operator
manual output_ref 或豁免说明
审批
audit
```

### 面试表达

```text
提前完成的根因通常是状态语义太粗,把“请求已提交”误当成“业务已完成”。我会把外部任务状态拆成 submitted、running external、output validating、complete,并要求 complete 前必须满足 output contract。对于下游 ready 判断,如果是数据依赖,不仅要看父节点状态,还要看 output_ref/manifest/marker 是否可用。这样可以避免 D 在 B 只提交成功但结果未产出时提前执行。
```

## 事故 3: 任务卡死在 RUNNING

### 事故现象

workflow 长时间不结束,某些 task 一直处于:

```text
RUNNING
IN_PROGRESS
SUBMITTED
CANCELLING
WAITING
```

用户看到:

```text
流程一直转圈。
没有失败。
没有成功。
也没有明显日志。
```

### 直接影响

```text
workflow 无法完成
下游永远不触发
资源被占用
用户无法判断是否要重跑
积累大量 zombie task
```

### 常见根因

#### 1. worker 宕机

worker 执行中挂掉,没有更新状态。

#### 2. 外部 callback 丢失

外部 job 已经成功或失败,但平台没收到 callback。

#### 3. 外部 job 卡死

Databricks/Spark/SQL job 自己卡住。

#### 4. lease / heartbeat 缺失

平台不知道 worker 是否还活着。

#### 5. timeout 没配置

任务没有最大运行时间。

#### 6. cancel 不完整

workflow 已经取消,但外部 job 没取消成功,平台停在 CANCELLING。

### 排查路径

检查:

```text
task started_at 到现在多久?
updated_at 是否长时间不变?
worker heartbeat 是否正常?
lease_until 是否过期?
external_job_id 是否存在?
外部 job 当前状态是什么?
是否有 timeout 配置?
是否有 callback 记录?
是否有 cancel_requested?
```

关键判断:

```text
worker alive?
external job alive?
platform state stale?
```

### 设计修复

#### 1. worker heartbeat

worker 周期性上报:

```text
worker_id
last_heartbeat_at
capacity
current_tasks
```

#### 2. task lease

task 领取时写:

```text
lease_until
lease_token
```

执行中续租。

#### 3. zombie scanner

定期扫描:

```text
RUNNING 且 lease 过期
RUNNING 且 worker heartbeat 失效
SUBMITTED 长时间没有 external status
```

#### 4. external polling

对于外部 job,不能只依赖 callback。

需要定期查询:

```text
external_job_id -> external_status
```

#### 5. timeout policy

每类任务有:

```text
start timeout
run timeout
callback timeout
cancel timeout
```

#### 6. 恢复策略

根据任务类型决定:

```text
retry
mark failed
mark lost
reconcile external success
manual intervention
```

### 面试表达

```text
任务卡死本质是平台失去了对执行状态的判断。解决方式是让 worker 和 task 都有 heartbeat/lease,让外部任务有 external_job_id 和 polling,让每类任务有 timeout policy,再通过 zombie scanner 定期扫描 RUNNING 但无心跳、lease 过期或外部终态已到达的任务。恢复时不能盲目重跑,要先查询外部状态,如果外部已成功就补 complete,如果外部失败就补 failed,如果未知再进入 retry 或人工处理。
```

如果要把这个事故设计成平台能力,可以把它抽象成第 32 章里的 zombie detector:先发现候选任务,再做 worker、lease、外部 job、callback、output_ref 多方对账,最后用 CAS 推进到可解释的新状态。

## 事故 4: fan-out 爆炸

### 事故现象

一个任务动态生成了远超预期的子任务。

例如:

```text
一个 query 返回 100 万行。
系统为每行生成一个 message。
task_instance 表暴涨。
queue 堆积。
worker 被打满。
外部系统被大量请求打爆。
UI 无法打开 DAG。
```

### 直接影响

```text
全局调度变慢
数据库压力飙升
队列堆积
外部服务限流或宕机
其他租户被影响
workflow 无法排查
```

### 常见根因

#### 1. 没有 fan-out 上限

动态生成任务没有:

```text
max_children_per_task
max_tasks_per_run
max_dynamic_depth
```

#### 2. 粒度过细

把每一行、每一个小对象都变成一个 task。

#### 3. 没有分页生成

一次性把所有子任务写入 DB 和队列。

#### 4. 没有资源池限流

所有子任务同时抢同一个外部系统。

#### 5. UI 没有聚合展示

几万个节点直接渲染。

### 排查路径

检查:

```text
哪个 task 生成了大量 child?
generated_by 是哪个 command?
fanout_group expected_count 是多少?
task_instance 增长速度是多少?
queue backlog 是多少?
外部系统错误率是否上升?
是否影响其他 tenant?
```

关键指标:

```text
tasks_created_per_minute
tasks_per_run
fanout_group_size
queue_depth
scheduler_db_latency
external_rate_limit_errors
tenant_resource_usage
```

### 设计修复

#### 1. fan-out limit

设置:

```text
max_children_per_task
max_tasks_per_run
max_fanout_group_size
max_dynamic_depth
tenant_task_quota
```

超过限制:

```text
BLOCKED / FAILED / WAITING_APPROVAL
```

不要静默生成。

#### 2. chunking

把:

```text
100 万行 -> 100 万 task
```

改成:

```text
100 万行 -> 1000 个 chunk task,每个处理 1000 行
```

#### 3. 分页生成

不要一次性生成全部任务。

```text
按 page / partition 逐批生成。
```

#### 4. backpressure

当队列、DB、外部系统压力过高时,暂停继续 fan-out。

#### 5. 资源池隔离

不同 command / tenant / external system 有不同并发限制。

#### 6. UI 聚合

大 fan-out 在 UI 上展示为:

```text
fanout group: 10000 tasks
success: 9000
running: 500
failed: 20
pending: 480
```

需要时再 drill down。

### 面试表达

```text
fan-out 爆炸是动态 DAG 最典型的生产风险。解决不能只靠扩 worker,因为瓶颈可能在 DB、队列、外部系统和 UI。设计上要限制 max tasks per run、max children per task、dynamic depth,并采用 chunking 和分页生成,同时用资源池、租户 quota 和 backpressure 控制执行速度。UI 上要聚合 fanout group,不能直接渲染所有节点。
```

## 事故 5: DAG 环依赖

### 事故现象

workflow 提交后无法推进,或者动态生成节点后 run 卡住。

依赖结构类似:

```text
A -> B -> C -> A
```

或动态生成后变成:

```text
A -> B
B -> C
C -> A
```

### 直接影响

```text
没有 ready 节点
workflow 永远等待
scheduler 不断扫描但无法推进
用户不知道为什么没跑
```

### 常见根因

#### 1. 提交时没有 DAG 校验

workflow definition 没有做 cycle detection。

#### 2. 动态新增边没有校验

初始 DAG 合法,但运行时新增节点或 relink 引入环。

#### 3. 子 workflow relink 错误

子 workflow 的 completion token 接回父 workflow 时,边接反了。

#### 4. order_id 复用错误

两个节点用了相同 `order_id`,导致父子关系错乱。

#### 5. parents 配置错误

用户或配置把当前节点写进自己的祖先链。

### 排查路径

检查:

```text
run_uuid 对应 runtime graph。
所有 vertices 的 order_id。
每个节点的 parents。
是否有 duplicate order_id。
动态生成记录 generated_by。
最后一次 add edge / relink 操作。
```

用图算法找:

```text
cycle path
```

错误信息最好能输出:

```text
cycle detected: A -> B -> C -> A
```

而不是:

```text
workflow invalid
```

### 设计修复

#### 1. 提交前校验

对 workflow definition 做:

```text
拓扑排序
DFS cycle detection
重复节点检查
缺失依赖检查
不可达节点检查
```

#### 2. 动态扩展时校验

每次新增 vertex / edge 时:

```text
检查 order_id 唯一。
检查 parent 是否存在。
检查新增边是否引入 cycle。
```

#### 3. runtime graph snapshot

动态 DAG 不要只存在内存中。

要记录:

```text
runtime vertices
runtime edges
generated_by
created_at
```

这样才能排查谁引入了环。

#### 4. 子 workflow relink 规则固定

子 workflow 接回父 workflow 时,明确:

```text
父 -> 子入口
子 completion token -> 父后续节点
```

不要让业务模块随意改内部边。

### 面试表达

```text
环依赖不是运行时普通失败,而是 DAG 建模错误。系统应该在 workflow definition 提交时用拓扑排序或 DFS 拦截,同时在动态 DAG 运行时每次新增 vertex/edge 也做局部 cycle detection。对动态 DAG 来说,初始图合法不代表运行时一直合法,所以必须记录 runtime graph snapshot 和 generated_by,一旦检测到环要给出具体 cycle path,否则用户无法修复。
```

## 事故 6: 下游没跑,但用户以为依赖满足

### 事故现象

用户看到上游都执行过了,但下游节点没有跑。

例如:

```text
A success
B skipped
C success
D 没有执行
```

用户问:

```text
为什么 D 不跑?
```

### 常见根因

```text
D 的 trigger rule 是 all_success,不接受 skipped。
某个父节点其实不是 terminal。
父节点 failed 但 UI 没突出显示。
workflow 被 pause。
资源池或 quota 满了。
D 的 run_date 在未来。
动态 fanout completion token 未满足。
```

### 排查路径

检查:

```text
D 的 parents 是谁。
每个 parent 的状态。
D 的 trigger rule。
D 的 blocked_reason。
workflow 是否 paused/cancelling。
资源池是否可用。
fanout_group 是否 sealed/complete。
```

### 设计修复

```text
给 blocked 节点记录 blocked_reason。
UI 展示每个父节点为什么阻塞。
trigger rule 显式化。
SKIPPED 传播语义文档化。
资源阻塞和依赖阻塞分开显示。
```

### 面试表达

```text
下游不跑不一定是调度 bug,可能是 trigger rule、父节点状态、workflow pause、资源配额或动态 fanout token 未满足。生产设计里要给每个 blocked 节点记录 blocked_reason,并在 UI 上把依赖阻塞、资源阻塞和人工暂停区分开,否则用户只能猜。
```

## 事故 7: 重跑后结果不一致

### 事故现象

用户重跑失败节点后,结果和第一次不一致。

例如:

```text
A -> B -> C
```

`B` 失败后重跑,但 `A` 的数据源已经变了,导致 `B/C` 结果变化。

### 常见根因

```text
重跑策略不清楚。
resolved parameters 没有快照。
上游 output_ref 使用 latest。
外部数据源不是固定版本。
下游旧结果没有 invalidated。
workflow definition version 改变。
```

### 排查路径

检查:

```text
rerun policy 是什么。
是否复用 A 的 output_ref。
B 的 resolved_parameters 前后是否一致。
definition_version_id 是否一致。
output_ref 路径是否带 run_uuid/attempt。
下游 C 是否被 reset。
```

### 设计修复

```text
rerun policy 显式化。
resolved_parameters 快照化。
output_ref 带 run_uuid/task_key/attempt。
禁止默认读取 latest。
重跑上游时 invalid 下游。
新定义重跑创建新 workflow instance。
```

### 面试表达

```text
重跑不是把 FAILED 改成 READY。必须定义输入快照和输出覆盖策略: 是复用上游 output_ref,还是从某个节点刷新;下游旧结果是否失效;definition version 是否保持不变。没有这些语义,重跑结果不可复现。
```

## 事故复盘时的统一模板

如果面试官让你现场复盘一个事故,可以按这套讲:

```text
1. 现象: 用户看到什么,哪个 workflow/task 异常。
2. 影响: 重复副作用、数据错误、卡死、资源打爆、审计风险。
3. 时间线: 状态变更、attempt、外部调用、callback、人工操作。
4. 根因: 状态语义、幂等、并发、外部系统、动态 DAG、观测缺口。
5. 短期止血: 暂停调度、限流、人工修复、reconcile、回滚配置。
6. 长期修复: 状态机、CAS/lease、幂等 key、outbox/inbox、fanout limit、审计和告警。
7. 防复发: 测试、故障注入、指标、runbook、UI blocked_reason。
```

## 最后可以背的事故总结

```text
DAG 工作流引擎的典型事故主要有五类: 第一是重复执行,通常来自 at-least-once 消费和外部副作用缺少幂等;第二是提前完成,本质是把 submit success 当成业务 complete;第三是任务卡死,需要 heartbeat、lease、timeout 和 external polling;第四是 fan-out 爆炸,要用限流、chunking、backpressure 和资源池治理;第五是环依赖,必须在 definition 和 runtime expansion 阶段都做 cycle detection。复盘时不能只看单个报错,而要还原状态变更、attempt、外部调用、消息投递和人工操作的完整时间线。
```

最关键的一句话:

```text
工作流事故的根因往往不是“某个任务失败”,而是状态、依赖、消息、外部副作用之间没有形成可恢复、可审计的闭环。
```
