# 30 工作流隔离级别与设计

## 为什么要专门讲隔离

工作流引擎里,隔离不是一个单点问题。

它要回答:

```text
一个 workflow run 会不会影响另一个 run?
一个大 workflow 会不会拖垮整个平台?
一个任务的上下文会不会污染另一个任务?
一个租户能不能看到另一个租户的数据和日志?
一个 workflow 的失败、取消、重跑会不会波及其他 workflow?
一个动态 fan-out 会不会把所有 worker 和外部系统打满?
```

所以“各个工作流之间的隔离”不能只说:

```text
用 run_uuid 区分。
```

`run_uuid` 只是第一层逻辑隔离。

生产级工作流平台至少要考虑:

```text
图隔离
状态隔离
上下文隔离
数据隔离
执行隔离
资源隔离
权限隔离
可观测性隔离
故障隔离
版本隔离
```

## 总体原则

可以先记住一句话:

```text
workflow run 之间默认不共享状态、不共享 DAG、不共享上下文、不共享输出命名空间;需要跨 run 协作时必须通过显式事件、消息、output_ref 或外部状态完成。
```

这句话很重要。

因为生产事故常常来自隐式共享:

```text
多个 run 写同一个 output path。
多个 run 读同一个 latest 参数。
多个 workflow 共用一个全局上下文字典。
多个 tenant 共用同一组外部 API quota。
子 workflow 复用了父 workflow 的 order_id。
```

隔离设计的目标是:

```text
默认隔离,显式共享。
```

## 隔离级别总览

可以按强度从弱到强分成几级。

| 级别 | 隔离方式 | 适用场景 | 代价 |
|---|---|---|---|
| L0 无隔离 | 所有 workflow 共用状态、队列、资源 | 原型或脚本 | 生产不可用 |
| L1 run 逻辑隔离 | `run_uuid` / workflow_instance 分开 | 单租户内部多个 run | 资源仍可能互相抢 |
| L2 batch / project 隔离 | `batch_id`、project/workspace 命名空间 | 批次治理、团队隔离 | 需要权限和配额模型 |
| L3 资源隔离 | per-run/per-tenant quota、queue、resource pool | 多租户生产平台 | 调度复杂 |
| L4 执行隔离 | worker pool / process / container 隔离 | 重任务、不可信任务 | 成本和延迟上升 |
| L5 硬隔离 | 独立账号、网络、数据库、集群 | 高安全/强合规场景 | 成本最高,运维复杂 |

多数内部平台不会一上来做到 L5。

更常见的是:

```text
L1 run 逻辑隔离 + L3 资源隔离 + 局部 L4 执行隔离。
```

也就是:

```text
每个 run 是独立 DAG。
资源和队列按 tenant/workflow/task type 限制。
高风险或重计算任务用独立 worker pool 或外部 executor。
```

## 1. DAG 图隔离

EMF 里最核心的图隔离是:

```text
一个 run_uuid 对应一张 DagContext。
```

也就是说:

```text
同一个 run_uuid 下的 Message 进入同一张运行时 DAG。
不同 run_uuid 默认是不同 DAG。
```

即使两个 run 在同一个 `batch_id` 下:

```text
batch_id = BATCH_001
  |- run_uuid = RUN_A
  |- run_uuid = RUN_B
```

`RUN_A` 和 `RUN_B` 的节点也不应该天然互相依赖。

如果 `RUN_B` 要等待 `RUN_A`,必须通过显式机制:

```text
事件
消息
completion token
外部状态
父 batch 状态
```

而不是偷偷在两个 `DagContext` 之间读状态。

### 设计规则

```text
DagManager 按 run_uuid 管 DagContext。
DagContext 内部维护 vertices / parents / children / state。
order_id 只要求在同一个 run_uuid 内唯一。
跨 run 依赖必须显式建模。
```

### 常见坑

```text
只用 order_id 查父节点,没有带 run_uuid。
不同 run 里都有 order_id = A,结果依赖匹配错。
子 workflow 沿用父 run_uuid,但 order_id 没命名空间,发生冲突。
```

所以父节点查找应该至少是:

```text
(run_uuid, order_id)
```

而不是:

```text
order_id
```

## 2. 子 workflow 的隔离选择

子 workflow 是隔离设计里最容易混的点。

有两种模式。

| 模式 | 含义 | 优点 | 代价 |
|---|---|---|---|
| 同一个 `run_uuid` | 子 workflow 是父 DAG 的一部分 | 父子依赖可直接用 `parents` 表达 | DAG 变大,order_id 要命名空间 |
| 新 `run_uuid` | 子 workflow 是独立 run | 独立追踪、取消、重跑、权限更清楚 | 父流程等待子流程需要额外事件或 token |

### 同一个 run_uuid

适合:

```text
子 workflow 是父流程的一段展开逻辑。
父流程必须直接等待子节点。
希望整个流程作为一张 DAG 展示。
```

需要注意:

```text
子节点 order_id 要加 namespace。
子 workflow fan-out 要受父 run 限制。
父流程 completion 要等子 workflow completion token。
```

例如:

```text
parent_order_id = LOAD
child_order_id = LOAD/SUB_VALIDATE
child_order_id = LOAD/SUB_EXPORT
```

### 新 run_uuid

适合:

```text
子 workflow 生命周期独立。
子 workflow 可以被单独取消、重跑、审计。
子 workflow 属于另一个团队或权限域。
子 workflow 很大,不希望撑爆父 DAG。
```

需要额外机制:

```text
父 workflow 创建 child run。
父节点进入 WAITING_CHILD_RUN。
child run 完成后发送 completion event。
父 workflow 收到 event 后继续。
```

### 面试表达

```text
子 workflow 是否沿用父 run_uuid 是一个隔离取舍。同 run_uuid 依赖表达简单,但 DAG 更大且要处理 order_id 命名空间;新 run_uuid 隔离更强,但父流程等待子流程需要额外 completion event 或 token。
```

## 3. 状态隔离

状态隔离要求:

```text
一个 run 的状态变化不能误改另一个 run。
一个 task 的 attempt 不能覆盖另一个 attempt。
一个人工操作只能影响目标 workflow 或目标子图。
```

核心状态表设计应该带上:

```text
tenant_id
workflow_instance_id
run_uuid
task_key / order_id
attempt
version
```

更新状态时使用 CAS:

```text
UPDATE task_instance
SET status = ?
WHERE workflow_instance_id = ?
  AND task_key = ?
  AND status = expected_status
  AND version = expected_version;
```

### 为什么只靠 task_id 不够

`task_id` 当然可以是主键。

但很多查询和调度逻辑天然会按:

```text
run_uuid
status
order_id
tenant_id
```

过滤。

如果状态更新或父节点匹配没有带 run/workflow 边界,就容易跨 run 污染。

### 终态隔离

终态节点也要被保护:

```text
COMPLETE
FAILED
CANCELLED
SKIPPED
```

旧事件、旧 attempt、重复 callback 不能把另一个 run 或当前 run 的终态覆盖掉。

## 4. 上下文隔离

EMF 里有一个重要机制:

```text
每条 Message 在自己的 DI scope 里执行。
```

这就是执行上下文隔离。

每条 message 应该有自己的:

```text
参数上下文
日志上下文
trace context
service scope
resolved parameters
attempt metadata
```

不能让多个 message 共享可变上下文对象。

否则会出现:

```text
A 的参数污染 B。
B 的 trace_id 出现在 C 的日志里。
一个 message 修改了 scoped service,影响另一个 message。
并发任务互相覆盖 run context。
```

### 设计规则

```text
Process scope 放无状态或线程安全组件。
Worker scope 放 worker 生命周期组件。
Message scope 放当前 message 的上下文和参数。
```

可以理解成:

```text
进程级共享能力。
worker 级共享基础设施。
message 级隔离业务上下文。
```

## 5. 数据隔离

数据隔离要解决:

```text
不同 workflow run 的输入、输出、临时产物、日志和参数不能互相覆盖。
```

### output_ref 命名空间

输出路径最好包含:

```text
tenant_id
workflow_id
workflow_instance_id / run_uuid
task_key / order_id
attempt
```

例如:

```text
s3://bucket/{tenant_id}/{workflow_id}/{run_uuid}/{task_key}/{attempt}/manifest.json
```

不要用:

```text
s3://bucket/latest/output.json
```

除非这个 `latest` 是一个受控发布指针,并且有版本和审计。

### 参数快照

不同 run 之间不能共享可变参数。

应该在 run 启动或 task 执行时保存:

```text
resolved_parameters
source
resolved_at
```

这样重跑时能知道:

```text
这次 task 到底用了哪个参数版本。
```

### 数据依赖显式化

如果两个 workflow run 需要共享数据,不要隐式读写同一个路径。

更好的方式是:

```text
上游发布 output_ref。
下游通过参数显式引用 output_ref。
或者通过 catalogue / dataset version 读取。
```

## 6. 队列隔离

如果所有 workflow 共用一个全局队列,会出现:

```text
一个大 workflow 生成 10 万个 task。
其他 workflow 的小任务排在后面一直等。
```

所以生产上要考虑:

```text
tenant queue
priority queue
resource queue
heavy/light queue
backfill queue
interactive queue
```

### 常见设计

| 队列类型 | 解决什么问题 |
|---|---|
| tenant queue | 防止大租户挤压小租户 |
| priority queue | 支持 SLA 和紧急任务 |
| resource queue | 不同外部系统分开限流 |
| heavy/light queue | 重任务不阻塞轻任务 |
| backfill queue | 历史补数不影响实时任务 |

调度时可以做:

```text
先按 tenant / queue 公平选择。
再按 priority / scheduled_at 排序。
最后检查资源池 quota。
```

## 7. 资源隔离

资源隔离是防止一个 workflow 抢光平台能力。

要限制:

```text
global max running tasks
tenant max running tasks
workflow max running tasks
batch max running tasks
task type max running tasks
external system max concurrency
fanout group max active children
```

例如:

```text
tenant A max_running = 100
workflow X max_running = 20
databricks_pool max_running = 10
storage_pool max_running = 50
```

### ready 不等于 executable

节点 ready 只表示:

```text
依赖满足。
```

真正可执行还要满足:

```text
workflow 未暂停
tenant quota 有额度
workflow quota 有额度
resource pool 有额度
worker 有 capacity
外部系统没有熔断
```

所以可以区分:

```text
dependency_ready
resource_ready
dispatchable
```

这样 UI 和排障会清楚很多。

## 8. 执行隔离

执行隔离回答:

```text
一个任务执行时会不会拖垮 worker 或影响其他任务?
```

常见级别:

| 执行隔离 | 优点 | 代价 |
|---|---|---|
| 同进程线程/协程 | 快、简单 | 隔离弱 |
| 独立 worker process | 进程级隔离 | 资源利用率下降 |
| 独立 worker pool | 按任务类型隔离 | 调度和部署复杂 |
| 外部 job | 重任务隔离好 | 需要 monitor/reconcile |
| 容器/Pod | 隔离最强 | 启动慢、成本高 |

### EMF 类平台的合理组合

```text
轻量内部 command: worker 进程内执行。
重计算任务: submit 外部 job,worker monitor。
不可信或资源重任务: 独立 worker pool / container。
高风险外部系统任务: 单独 resource pool 和 rate limit。
```

### 隔离 trade-off

```text
隔离越强,安全性和稳定性越好。
隔离越强,成本、延迟和运维复杂度越高。
```

所以不需要所有任务都容器化。

应该按任务风险分层。

## 9. 权限隔离

权限隔离回答:

```text
谁能看、谁能跑、谁能改、谁能操作。
```

至少要按这些维度控制:

```text
tenant / workspace / project
workflow definition
workflow instance
task instance
logs
parameters
secrets
manual operations
resource pools
```

常见权限:

```text
view workflow
run workflow
cancel workflow
rerun task
skip task
mark success
view logs
view sensitive parameters
edit definition
approve manual task
admin resource pool
```

高风险操作:

```text
manual success
skip critical task
cancel production workflow
view secret
change priority
```

可以要求更高权限或二次审批。

## 10. Secret 隔离

不同 workflow 和 tenant 不应该共享 secret 权限。

参数里应该存:

```text
secret_ref
```

而不是明文。

执行时根据:

```text
tenant_id
workflow_id
task identity
operator
environment
```

解析 secret。

日志和 UI 默认脱敏。

查看敏感参数也要审计。

## 11. 可观测性隔离

可观测性也要隔离。

不同 workflow 的:

```text
logs
metrics
trace
audit
state transition
external_call_log
```

都要带:

```text
tenant_id
workflow_id
workflow_instance_id / run_uuid
task_key / order_id
attempt
```

否则会出现:

```text
日志搜索串了 run。
不同 tenant 的 trace 混在一起。
用户看到不属于自己的 workflow。
审计无法按项目导出。
```

UI 查询也要按权限过滤。

## 12. 故障隔离

故障隔离回答:

```text
一个 workflow 失败、卡死、爆量、取消,会不会影响其他 workflow?
```

常见手段:

```text
run-level concurrency limit
tenant quota
resource pool
fanout limit
circuit breaker
retry backoff
bulkhead isolation
per-tenant queue
worker pool isolation
```

### Bulkhead 思想

Bulkhead 可以理解成船舱隔板。

一个舱进水,不要让整艘船沉。

工作流平台里对应:

```text
一个 tenant 的任务爆量,不能拖垮其他 tenant。
一个 external API 宕机,不能让所有 task retry storm。
一个 workflow fan-out 爆炸,不能打满全局 worker。
一个重任务卡死,不能占住轻任务 worker。
```

## 13. 版本隔离

版本隔离回答:

```text
workflow 定义变更会不会影响正在运行的 workflow?
worker 升级会不会影响旧任务?
command 协议变更会不会影响历史 run?
```

设计规则:

```text
workflow_instance 绑定 workflow_definition_version_id。
definition version 不可变。
运行时动态 DAG 记录 runtime snapshot。
worker 声明 supported_commands / protocol_version。
scheduler 按 capability 分发任务。
```

这样旧 run 仍按旧定义跑。

新定义只影响新 run。

## 14. 数据库和索引隔离

隔离不仅是逻辑概念,还影响表设计和查询性能。

核心表建议带:

```text
tenant_id
workflow_id
workflow_instance_id
run_uuid
batch_id
task_key
status
created_at
```

常见索引:

```text
(tenant_id, status, scheduled_at)
(tenant_id, workflow_instance_id, status)
(run_uuid, order_id)
(batch_id, run_uuid)
(worker_id, status)
```

这样 scheduler 和 UI 查询都能自然落在隔离边界内。

如果查询经常漏掉 `tenant_id` 或 `run_uuid`,说明隔离边界在代码里不够显式。

## 15. 跨 workflow 协作怎么设计

隔离不等于不能协作。

关键是:

```text
协作必须显式。
```

跨 workflow 协作可以有几种方式:

| 方式 | 适用场景 |
|---|---|
| completion event | A 完成后通知 B |
| output_ref | B 显式消费 A 的产物 |
| dataset version / catalogue | 多 workflow 通过数据版本协作 |
| parent-child run relation | 父 workflow 等子 workflow |
| batch-level barrier | 一批 run 全完成后执行汇总 |
| external coordination table | 多 run 共享一个受控状态 |

不推荐:

```text
直接读另一个 run 的内存 DagContext。
直接复用另一个 run 的 mutable context。
直接写同一个 output path。
```

## 16. 不同隔离策略的取舍

| 目标 | 强隔离收益 | 强隔离代价 |
|---|---|---|
| 稳定性 | 降低 blast radius | 资源利用率下降 |
| 安全性 | 防止越权访问 | 权限模型复杂 |
| 可观测性 | 排查边界清楚 | 采点字段更多 |
| 性能 | 热点可分片 | 查询和调度更复杂 |
| 成本 | 可按租户计量 | 多队列/多 worker 运维成本 |
| 灵活性 | 边界清晰 | 跨 workflow 协作需要显式协议 |

面试里不要说:

```text
所有东西都强隔离。
```

更成熟的表达是:

```text
根据任务风险和租户边界分层隔离。轻量可信任务用逻辑隔离和资源配额;重计算或不可信任务用独立 worker pool 或容器;高安全场景再考虑账号、网络、存储层硬隔离。
```

## 常见生产问题

### 1. 一个 workflow 拖慢所有 workflow

常见原因:

```text
全局队列。
没有 run/tenant 并发限制。
fan-out 没有限制。
外部系统资源池共用。
```

设计解决:

```text
per-workflow max_running。
per-tenant quota。
resource pool。
fanout limit。
heavy/light queue。
```

### 2. 两个 run 互相串数据

常见原因:

```text
output path 没带 run_uuid。
参数读取 latest。
上下文对象全局共享。
日志 trace 没带 run_id。
```

设计解决:

```text
output_ref namespace。
resolved parameter snapshot。
message scope。
trace/log 带 run_uuid/task_key/attempt。
```

### 3. 子 workflow 节点冲突

常见原因:

```text
父子 workflow 共用 run_uuid,但子节点 order_id 没加 namespace。
```

设计解决:

```text
child order_id prefix。
fanout_group_id。
completion token。
或者子 workflow 独立 run_uuid。
```

### 4. 一个租户看到另一个租户日志

常见原因:

```text
日志查询没有 tenant filter。
trace_id 不带 tenant。
UI 权限只按登录态,没按 workspace 过滤。
```

设计解决:

```text
所有观测数据带 tenant_id。
查询层强制 tenant scope。
敏感日志脱敏。
查看敏感数据审计。
```

### 5. 外部 API 宕机导致全平台 retry storm

常见原因:

```text
所有 workflow 共用外部资源。
没有 circuit breaker。
retry 没有 backoff。
```

设计解决:

```text
external system resource pool。
circuit breaker。
指数退避 + jitter。
按 tenant/workflow 限流。
失败隔离和告警。
```

## 面试总结

可以这样说:

```text
工作流之间的隔离至少有几层。第一是 DAG 逻辑隔离,每个 run_uuid 对应一张 DagContext,不同 run 默认不共享依赖和状态;第二是状态和上下文隔离,状态更新必须带 workflow_instance/run/task/attempt 边界,每条 message 在独立 scope 里执行;第三是数据隔离,output_ref、manifest、日志和参数快照都要带 run_uuid/task_key/attempt,避免串数据;第四是资源隔离,通过 tenant/workflow/task-type/external-system quota、队列和 resource pool 控制并发;第五是执行隔离,轻任务可以进程内执行,重任务或不可信任务用独立 worker pool、外部 job 或容器;第六是权限和可观测性隔离,日志、审计、secret、手动操作都要按 tenant/workspace/workflow 控制。跨 workflow 协作不是禁止,但必须通过显式事件、output_ref、dataset version 或 completion token,不能隐式共享内部状态。
```
