# 32 僵尸任务发现与处理

## 为什么要单独讲僵尸任务

僵尸任务不是普通失败。

普通失败是:

```text
任务执行结束了,并明确返回失败。
```

僵尸任务是:

```text
平台以为任务还在执行或等待外部结果,
但真实执行 owner 已经失效、失联、卡死,
或者外部状态已经变化而平台没有收敛。
```

它最危险的地方是“不确定”:

```text
worker 可能死了,也可能只是心跳延迟。
外部 job 可能没提交,也可能已经成功。
callback 可能丢了,也可能还在路上。
output_ref 可能没写入 DB,也可能文件已经落在 GCS。
```

所以僵尸任务处理的本质不是:

```text
发现超时 -> 直接重跑
```

而是:

```text
发现可疑任务 -> 收集证据 -> 对账真实状态 -> 带 fencing 地恢复状态
```

这是一套状态一致性和副作用一致性的恢复机制。

## 定义

可以把 zombie task 定义为:

```text
task instance 处于非终态,
但当前 owner、外部执行状态或等待条件已经不再可信,
导致平台无法仅靠正常回调或 worker 更新继续推进。
```

常见非终态包括:

```text
RUNNING
SUBMITTED
RUNNING_EXTERNAL
WAITING_EVENT
OUTPUT_VALIDATING
CANCELLING
```

注意,下面这些不一定是 zombie:

| 情况 | 是否 zombie | 说明 |
|---|---|---|
| 长时间运行但 heartbeat 正常 | 不一定 | 可能只是慢任务,要看 run timeout |
| lease 过期但 worker heartbeat 仍正常 | 可疑 | 可能是续租线程异常或 DB 更新失败 |
| 外部 job 还在跑且能查到进度 | 不一定 | 平台可恢复 owner 或继续等待 |
| callback 没到但 polling 能查到终态 | 是可恢复异常 | 应该 reconcile,不该一直等 |
| 外部副作用状态未知 | 高危 zombie | 不能盲目重试 |

## 僵尸任务的类型

### 1. Worker owner 丢失型

典型状态:

```text
task.status = RUNNING
task.lease_until < now
worker.last_heartbeat_at 很久没更新
```

含义:

```text
原 worker 不再可信,平台不知道它是否还会更新任务。
```

这类任务要先判断是否有外部副作用。

### 2. 外部异步任务失联型

典型状态:

```text
task.status = SUBMITTED / RUNNING_EXTERNAL / WAITING_EVENT
external_job_id exists
last_polled_at 很久没更新
callback 没收到
```

含义:

```text
平台已经把任务交给外部系统,但外部最终状态没有同步回来。
```

这类任务不能按 worker 死亡处理,而应该查询外部系统。

### 3. 提交结果不确定型

最危险的场景:

```text
worker 调 external submit 成功
worker 写 DB external_job_id 之前崩溃
```

平台看到:

```text
task.status = RUNNING
external_job_id is null
lease expired
```

但真实世界里外部 job 可能已经创建。

如果没有 `idempotency_key` 或 `external_request_id`,这类任务只能进入人工或特殊对账流程,不能直接重跑。

### 4. Callback 丢失型

典型状态:

```text
external job 已经 SUCCEEDED / FAILED
platform task 仍然 WAITING_EVENT / RUNNING_EXTERNAL
```

含义:

```text
外部系统已经给出结果,但平台没有消费到 callback,
或者 callback 到了但处理失败。
```

解决方式是 polling 和 inbox reconciliation。

### 5. Output 卡住型

典型状态:

```text
external job = SUCCEEDED
task.status = OUTPUT_VALIDATING / RUNNING
output_ref missing or invalid
```

含义:

```text
计算可能成功了,但产物契约没有满足。
```

这种任务不能直接 `SUCCESS`,要检查 GCS 文件、manifest、marker、schema、row count 等输出证据。

### 6. 取消卡住型

典型状态:

```text
task.status = CANCELLING
cancel_requested_at 很久以前
external job 仍然 RUNNING
```

含义:

```text
平台已经发起取消,但外部系统没有确认取消。
```

这类任务需要继续 cancel、kill 外部 job、标记取消失败,或升级人工处理。

## 检测信号

### 1. Lease 过期

最基本的扫描:

```sql
SELECT *
FROM task_instance
WHERE status = 'RUNNING'
  AND lease_until < now();
```

生产里通常要加 grace period:

```sql
SELECT *
FROM task_instance
WHERE status = 'RUNNING'
  AND lease_until < now() - interval '60 seconds';
```

原因是网络抖动、DB 主从延迟、worker GC pause 都可能让 lease 短暂过期。

### 2. Worker heartbeat 失效

只看 task lease 不够,还要看 worker 是否活着:

```sql
SELECT t.*
FROM task_instance t
LEFT JOIN worker_heartbeat w
  ON t.worker_id = w.worker_id
WHERE t.status = 'RUNNING'
  AND t.lease_until < now() - interval '60 seconds'
  AND (w.last_heartbeat_at IS NULL
       OR w.last_heartbeat_at < now() - interval '120 seconds');
```

如果 worker 还活着,但 task lease 过期,更像是续租线程或局部任务卡住,不能马上当作 worker 死亡。

### 3. 状态长时间不更新

有些任务虽然 lease 没过期,但状态长期没有任何进展:

```text
updated_at stale
last_polled_at stale
heartbeat_at stale
```

例如:

```sql
SELECT *
FROM task_instance
WHERE status IN ('RUNNING_EXTERNAL', 'WAITING_EVENT', 'OUTPUT_VALIDATING')
  AND updated_at < now() - interval '30 minutes';
```

这类信号通常是 zombie candidate,需要结合 command 的期望运行时间。

### 4. Callback 超时

对于异步外部任务:

```text
submit 成功
external_job_id exists
等待 callback
超过 callback_timeout
```

应该触发 polling reconciliation,而不是无限等待 callback。

### 5. 外部终态和平台状态不一致

通过外部 polling 查到:

```text
external_status = SUCCEEDED / FAILED / CANCELLED
platform status 仍然 RUNNING_EXTERNAL / WAITING_EVENT
```

这基本就是可恢复 zombie。

### 6. Cancel 超时

取消状态也会僵住:

```sql
SELECT *
FROM task_instance
WHERE status = 'CANCELLING'
  AND cancel_requested_at < now() - interval '15 minutes';
```

后续要看外部系统是否支持 cancel、cancel 是否已经确认、是否需要 kill 或人工介入。

### 7. Output validation 超时

外部 job 成功后,output contract 还没满足:

```text
external_status = SUCCEEDED
output_ref missing
OUTPUT_VALIDATING 超过阈值
```

这可能是:

```text
产物写入延迟
manifest 生成失败
DB 写 output_ref 失败
下游引用路径错误
外部 job 提前成功但数据没落地
```

## 扫描器设计

一个生产级 zombie detector 不建议直接“扫到就改状态”。

更稳的流水线是:

```text
scan candidates
claim recovery ownership
collect evidence
classify root signal
reconcile external state
decide recovery action
CAS update task state
write transition and audit
emit metrics
create or update Jira if needed
```

可以拆成几个组件:

| 组件 | 职责 |
|---|---|
| Candidate Scanner | 按索引扫描可疑非终态任务 |
| Recovery Claimer | 用 CAS 抢占恢复处理权 |
| Evidence Collector | 读取 worker、外部 job、callback、output、DAG metadata |
| Classifier | 判断 zombie 类型和风险等级 |
| Reconciler | 查询外部系统和产物状态 |
| Recovery Executor | 推进到 SUCCESS、FAILED、RETRYING、LOST、CANCELLED 等 |
| Escalation Manager | 告警、Jira、人工处理 |

候选扫描 SQL 可以是:

```sql
SELECT *
FROM task_instance
WHERE status IN (
  'RUNNING',
  'SUBMITTED',
  'RUNNING_EXTERNAL',
  'WAITING_EVENT',
  'OUTPUT_VALIDATING',
  'CANCELLING'
)
AND (
  lease_until < now() - interval '60 seconds'
  OR timeout_at < now()
  OR updated_at < now() - interval '30 minutes'
  OR last_polled_at < now() - interval '30 minutes'
)
ORDER BY updated_at ASC
LIMIT 500;
```

配套索引:

```text
(status, lease_until)
(status, timeout_at)
(status, updated_at)
(status, last_polled_at)
(external_system, external_job_id)
(run_uuid, status)
```

不要全表扫历史任务。终态任务应该冷热分离或按时间分区。

## 恢复前先抢占恢复权

zombie detector 自己也可能多实例部署。

因此它处理候选任务前也要 CAS:

```sql
UPDATE task_instance
SET recovery_owner = :detector_id,
    recovery_started_at = now(),
    version = version + 1
WHERE id = :task_id
  AND status IN ('RUNNING', 'SUBMITTED', 'RUNNING_EXTERNAL', 'WAITING_EVENT', 'OUTPUT_VALIDATING', 'CANCELLING')
  AND version = :version
  AND (recovery_owner IS NULL
       OR recovery_started_at < now() - interval '10 minutes');
```

抢不到就跳过。

是否需要持久化 `RECOVERING` 状态,取决于系统复杂度:

| 方案 | 优点 | 代价 |
|---|---|---|
| 不新增状态,只写 recovery_owner | 状态机简单 | UI 上不容易看出正在恢复 |
| 新增 `RECOVERING` | 语义清楚,便于防并发 | 状态流转更复杂 |

如果 zombie recovery 逻辑重、外部查询慢,建议加 `RECOVERING` 或至少加 `recovery_owner`。

## 恢复决策矩阵

核心原则:

```text
先判断 owner 是否失效,
再判断外部副作用是否发生,
最后决定状态怎么收敛。
```

| 平台状态 | Worker / lease | 外部状态 | 建议处理 |
|---|---|---|---|
| RUNNING | worker alive, lease active | unknown | 不是 zombie,跳过 |
| RUNNING | worker dead, lease expired | no external_job_id, no side effect | RETRYING / READY |
| RUNNING | worker dead, lease expired | external running | RUNNING_EXTERNAL / WAITING_EVENT,恢复 polling owner |
| RUNNING | worker dead, lease expired | external succeeded, output valid | SUCCESS |
| RUNNING | worker dead, lease expired | external succeeded, output missing | OUTPUT_VALIDATING / NEEDS_MANUAL_CHECK |
| RUNNING | worker dead, lease expired | external failed retryable | RETRYING |
| RUNNING | worker dead, lease expired | external failed non-retryable | FAILED |
| RUNNING | worker dead, lease expired | side effect unknown | LOST / NEEDS_MANUAL_CHECK |
| WAITING_EVENT | no callback | external succeeded | SUCCESS 或 OUTPUT_VALIDATING |
| WAITING_EVENT | no callback | external failed | FAILED / RETRYING |
| CANCELLING | cancel timeout | external cancelled | CANCELLED |
| CANCELLING | cancel timeout | external still running | retry cancel / escalate |
| OUTPUT_VALIDATING | stale | output exists and valid | SUCCESS |
| OUTPUT_VALIDATING | stale | output invalid | FAILED / NEEDS_MANUAL_CHECK |

这里的 `NEEDS_MANUAL_CHECK` 可以是 task 状态,也可以是 incident 状态。关键是不能把未知副作用的任务自动重跑。

## 推荐状态流转

不一定要把所有 zombie 类型都做成 task status,否则状态机会膨胀得很快。

推荐把主状态保持少而明确:

```text
RUNNING
RUNNING_EXTERNAL
WAITING_EVENT
OUTPUT_VALIDATING
CANCELLING
RECOVERING
SUCCESS
FAILED
RETRYING
CANCELLED
LOST
```

其中 `RECOVERING` 只表示:

```text
平台正在接管一个不可信的非终态任务,并准备做证据对账。
```

它不是最终结论。

典型流转:

```text
RUNNING -> RECOVERING -> SUCCESS
RUNNING -> RECOVERING -> FAILED
RUNNING -> RECOVERING -> RETRYING
RUNNING -> RECOVERING -> LOST
RUNNING_EXTERNAL -> RECOVERING -> WAITING_EVENT
WAITING_EVENT -> RECOVERING -> SUCCESS
OUTPUT_VALIDATING -> RECOVERING -> SUCCESS
CANCELLING -> RECOVERING -> CANCELLED
```

不建议:

```text
RUNNING -> READY
```

因为这会丢掉“为什么从运行中回到待调度”的证据。

更好的方式是:

```text
RUNNING -> RECOVERING -> RETRYING -> READY
```

这样 transition log 能说明:

```text
任务不是自然失败,而是 zombie detector 根据证据决定重试。
```

## Fencing: 防止旧 worker 覆盖新状态

僵尸任务最容易出现的竞态是:

```text
task lease 过期
zombie detector 判定可重试
新 attempt 开始执行
旧 worker 又醒过来,写 SUCCESS
```

如果没有 fencing,旧 worker 会覆盖新 attempt。

所以 worker 更新任务时必须带:

```text
task_id
attempt
lease_token
expected_status
```

例如:

```sql
UPDATE task_instance
SET status = 'SUCCESS',
    finished_at = now(),
    output_ref = :output_ref,
    version = version + 1
WHERE id = :task_id
  AND status = 'RUNNING'
  AND attempt = :attempt
  AND lease_token = :lease_token;
```

如果 affected rows = 0,说明这个 worker 的执行权已经失效。

此时不能强行写成功,应该把结果写到 stale event 或 attempt log:

```text
attempt_result:
  task_id
  attempt
  lease_token
  result
  rejected_reason = stale_lease
```

callback 也一样。

外部 callback 至少要携带:

```text
external_job_id
external_request_id
attempt
correlation_id
```

平台处理 callback 时要确认它对应的是当前有效 attempt,否则只能作为补充证据,不能直接推进主状态。

## 外部副作用对账

处理 zombie 时最容易犯的错误是:

```text
worker 死了 -> 任务没执行 -> 可以重跑
```

真实情况是:

```text
worker 可能死在外部调用之后。
```

所以每个 command 最好声明自己的执行类型:

```text
execution_mode:
  local
  external_sync
  external_async
  side_effect_write

idempotency_class:
  none
  weak
  strong
  queryable

supports_status_query:
  true / false

supports_cancel:
  true / false

zombie_recovery_policy:
  retry
  reconcile
  fail
  manual
```

几个例子:

| command 类型 | 恢复策略 |
|---|---|
| 纯本地计算,无外部写入 | 可以 retry |
| 写 GCS 文件,路径由 run_uuid/order_id 决定 | 先检查 output_ref 和 GCS object,再决定 success 或 retry |
| 提交 Databricks job | 查询 job id 或 request id,再 reconcile |
| 调第三方 API 创建资源 | 必须有 idempotency_key,否则未知时 manual |
| PII 打码 command | 检查输入/输出 manifest、处理批次、脱敏结果校验,不能重复覆盖不明产物 |

对于外部调用,推荐有一张 ledger:

```text
external_call_ledger:
  idempotency_key
  request_hash
  command
  task_id
  attempt
  external_system
  external_job_id
  side_effect_status
  created_at
  updated_at
```

这样即使 worker 在写 task 状态前崩溃,平台也可以通过 ledger 或 idempotency_key 找回外部执行记录。

## Timeout 和 zombie 的区别

timeout 是业务语义:

```text
这个任务允许最多运行多久。
```

zombie 是控制语义:

```text
平台是否还拥有可信的执行 owner 和状态更新路径。
```

两者不要混用。

| 情况 | 判断 |
|---|---|
| 任务运行 2 小时,heartbeat 正常,未超过 max_runtime | 正常长任务 |
| 任务运行 2 小时,heartbeat 正常,超过 max_runtime | timeout |
| 任务运行 5 分钟,lease 过期,worker heartbeat 丢失 | zombie candidate |
| 外部 job 已成功,平台仍 WAITING_EVENT | zombie / reconcile |

timeout scanner 可以触发失败或取消,zombie detector 应该触发对账和恢复。

## 避免误判

误判 zombie 会引发重复执行,所以需要防护。

常见误判来源:

```text
lease_duration 设置太短
worker 负载过高,心跳线程被业务线程阻塞
DB 或网络抖动
JVM/运行时长时间 GC pause
节点时钟漂移
外部系统 callback 延迟
scanner 和 worker 使用不同时间源
```

防护手段:

| 风险 | 防护 |
|---|---|
| 短暂网络抖动 | grace period,多次观测 |
| worker 忙导致心跳延迟 | 心跳线程和业务线程隔离 |
| 旧 worker 迟到更新 | lease_token + attempt fencing |
| 时钟漂移 | NTP,使用 DB server time,记录 clock skew |
| 外部 callback 延迟 | callback timeout 后先 polling,不直接失败 |
| scanner 重复处理 | recovery_owner + CAS |

经验规则:

```text
lease_duration >= 3 * heartbeat_interval + network_skew_budget
```

例如 heartbeat 每 30 秒一次,lease 可以是 2 到 5 分钟,再加 scanner grace period。

## 和人工处理、Jira、SOP 的关系

大多数 zombie 可以自动恢复,但下面几类应该升级:

```text
side effect unknown
external status API unavailable
same command repeated zombie
same worker pool 大量 zombie
cancel 多次失败
output exists but contract invalid
recovery action 失败
```

升级时不要只发一句“任务卡住了”。

Jira ticket 应该包含:

```text
workflow / run_uuid / batch_id
task order_id / command / attempt
current status
zombie type
worker_id / worker heartbeat / lease_until
external_system / external_job_id / external_status
idempotency_key / side_effect_status
output_ref / manifest validation result
last state transitions
detector evidence
recommended action
owner_team / jira_project / component
SOP links
```

如果 EMF 接入 LLM,这里是一个很自然的使用点:

```text
zombie detector 收集结构化证据
LLM 根据 SOP、历史事故、command owner metadata 生成摘要
自动判断应该发到哪个 team 的 Jira board
给出可执行的排查步骤和推荐恢复动作
```

但 LLM 不应该直接改任务状态。

它适合做:

```text
归纳证据
匹配 SOP
生成 ticket
推荐 owner
解释当前进度
提示风险
```

真正的状态变更仍然要由规则化的 recovery executor 通过 CAS 和状态机完成。

## 可观测性指标

建议至少有这些指标:

```text
emf.zombie.candidate_count
emf.zombie.detected_total
emf.zombie.recovered_total{action}
emf.zombie.escalated_total{reason}
emf.zombie.false_positive_total
emf.task.lease_expired_total
emf.task.oldest_zombie_age_seconds
emf.worker.heartbeat_stale_count
emf.external.reconcile_total{system,result}
emf.recovery.cas_conflict_total
```

关键告警:

```text
oldest zombie age > threshold
zombie candidate count sudden spike
side_effect_unknown > 0
same command zombie rate above baseline
same worker_pool heartbeat stale spike
recovery failure rate too high
```

UI 上最好能看到:

```text
为什么被判定为 zombie
用了哪些证据
外部系统查到了什么
为什么选择当前恢复动作
是否已经创建 Jira
下一次 scanner 什么时候处理
```

排障体验的核心是可解释,否则自动恢复会让人更不敢相信系统。

## 测试用例

僵尸任务必须靠故障注入测试。

至少覆盖:

```text
worker claim 后立刻崩溃
worker 提交外部 job 前崩溃
worker 提交外部 job 成功后、写 external_job_id 前崩溃
worker 写 external_job_id 后崩溃
external job 成功,callback 丢失
external job 失败,callback 丢失
external status API 临时不可用
旧 worker 在 lease 过期后写 SUCCESS
新 attempt 开始后旧 callback 到达
多个 zombie detector 同时处理同一任务
CANCELLING 状态外部 job 取消失败
OUTPUT_VALIDATING 状态 output_ref 缺失
GCS object 存在但 manifest 不合法
DB 时钟和 worker 时钟有偏差
```

每个测试都要验证:

```text
不会重复产生业务副作用
不会被旧 attempt 覆盖
最终状态可解释
transition log 完整
指标和告警正确
需要人工时能生成足够证据
```

## 一个完整例子

假设任务:

```text
order_id = export_customer_snapshot
command = ExportToGcs
status = RUNNING
worker_id = worker-17
lease_until = 10:00
external_job_id = dbx-123
output_ref = null
```

10:05 zombie detector 扫到 lease 过期。

它不应该立刻 retry,而是:

```text
1. 查 worker-17 heartbeat,发现 09:58 后没有心跳。
2. 查 Databricks job dbx-123,发现 job = SUCCEEDED。
3. 查 GCS output path,发现文件存在。
4. 校验 manifest,发现 row count 和 schema 合法。
5. 用 CAS 把 task 从 RUNNING 更新为 SUCCESS。
6. 写 transition reason = zombie_reconcile_external_success。
7. 发出 TASK_COMPLETED 事件,让下游继续。
```

如果第 3 步发现 GCS 文件不存在,就不能标成功。

它应该进入:

```text
OUTPUT_VALIDATING
```

或:

```text
NEEDS_MANUAL_CHECK
```

并创建 ticket,说明:

```text
外部 job 成功,但产物缺失,疑似 output commit 失败。
```

## 面试表达

```text
僵尸任务不是简单的超时任务,而是平台失去了对执行 owner 或外部状态的可信判断。我的设计会让 RUNNING 任务带 lease_token、attempt 和 heartbeat,worker 更新状态必须通过 CAS 和 fencing;zombie detector 定期扫描 lease 过期、heartbeat 丢失、callback 超时、external terminal 但平台未收敛等信号。恢复时先查询 worker、外部 job、callback inbox、output_ref 和幂等 ledger,再决定补 success、补 failed、retry、cancel、lost 或人工处理。对于外部副作用未知的任务绝不盲目重跑,否则会把 zombie 恢复变成重复执行事故。
```

## 一句话总结

```text
僵尸任务处理的核心不是“把 RUNNING 改成重试”,
而是“用证据重新建立平台状态和真实世界之间的一致性”。
```
