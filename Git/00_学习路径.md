# Git 学习路径与知识图谱

## 学习定位

Git 表面上是一组命令，实际是一个维护快照图与引用指针的本地数据库。很多误操作不是因为命令记少了，而是没有先判断：

- 当前变更在工作区、暂存区，还是已经变成 commit？
- 这个 commit 是否已经发布给别人依赖？
- 目标是撤销内容、改写历史，还是只把某个修复搬到另一条分支？
- 出问题后应该读差异、读提交图，还是二分定位引入故障的提交？

这一组内容以“状态和安全边界”为主线。参考图片里的四个区域会用可编辑的图示重新表达，不在仓库中保存图片文件。

## 知识图谱

1. Git 对象模型与四个工作区域
   - Working Tree、Index、本地仓库、远程仓库
   - blob、tree、commit、branch、tag、`HEAD`
   - `add` / `commit` / `fetch` / `push` / `restore`
   - `git diff` 三种最常用对比面
   - detached `HEAD` 与远程跟踪分支
2. 撤销、安全恢复与敏感信息清理
   - `restore`：撤工作区或撤暂存
   - `reset --soft` / `--mixed` / `--hard`
   - `revert`：共享历史上的安全回退
   - `commit --amend`、`reflog`、`force-with-lease`
   - `.gitignore` 与已跟踪文件
   - 密钥泄露：先轮换，再清历史
3. 分支整合与日常协作
   - branch / `switch` / `merge`
   - fast-forward 与 merge commit
   - `rebase` 的复制提交本质及边界
   - 冲突处理、`cherry-pick`、`stash`
   - fork、clone、Pull Request
4. 工作流、发布标记与定位问题
   - GitHub Flow、Trunk-Based、版本维护分支
   - 为什么不机械搬运一整套 Git Flow
   - `log`、`tag`、`blame`、`bisect`
   - 分支清理与可追溯发布

## 学习方法建议

- 每执行一个会改变状态的命令，先画出 `HEAD`、Index、Working Tree 三者是否会变化。
- 看到“撤销”先问提交是否已经被共享；未共享可以改写，已共享优先新增反向提交。
- `merge` 和 `rebase` 先按提交图理解，不要先按“历史好不好看”选命令。
- 遇到线上回归，不要靠猜：用 `log` 找范围、用 `bisect` 找第一条坏提交、用测试脚本验证。
- 任何密钥一旦进入可访问远端，都按已泄露处理；Git 历史清理不能替代密钥轮换。

## 学完后你应该能回答的问题

- 工作区、暂存区、本地仓库和远程仓库分别保存了什么？
- `git add` 暂存的是“文件名”还是“当时文件内容的快照”？
- `HEAD`、分支名和远程跟踪分支三者之间是什么关系？
- `git diff`、`git diff --staged` 与 `git diff HEAD` 各在比较什么？
- 为什么撤一个已经推送到共享分支的提交通常应该使用 `git revert`？
- `reset --soft`、`--mixed`、`--hard` 分别移动哪些状态？
- `merge` 和 `rebase` 对提交图和 commit id 的影响有什么不同？
- 为什么不能随意 rebase 已经让别人基于其开发的提交？
- 密钥被提交后，为什么删除文件或 revert 仍然不够？
- `git bisect` 为什么能高效定位一段历史中引入故障的提交？
