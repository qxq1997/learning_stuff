# JVM - 第 2 课：运行时数据区、对象创建与对象访问

## 学习目标（本节结束后你能做到什么）

- 说清 JVM 运行时数据区分别在做什么。
- 理解堆、虚拟机栈、方法区在日常开发里最常见的意义。
- 理解对象从 `new` 出来，到真正放进内存，中间经历了什么。
- 理解“对象在堆里，引用在栈上”这种常见说法为什么只对了一半。
- 为后续理解垃圾回收、对象存活和 GC Roots 建立基础。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 JVM 要有运行时数据区

程序运行时，需要放很多不同性质的数据：

- 当前线程执行到哪一行
- 方法调用链长什么样
- 局部变量和参数放哪
- 对象实例放哪
- 类元信息放哪

如果这些数据全都混在一起，管理会非常混乱。  
所以 JVM 把运行时空间按职责拆成了多个区域。

你可以把它想成一个公司：

- 有办公桌，放当前手头正在处理的工作
- 有仓库，放大量共享物品
- 有档案室，放制度、模板和文档

JVM 的运行时数据区也是类似思想：不同数据放在不同区域里。

### 2. 程序计数器：记录线程下一步该执行哪

程序计数器可以理解成：

**当前线程执行位置的小书签。**

因为 Java 线程是可以被切换的，一个线程被暂停后，下次恢复执行时必须知道自己该从哪继续。

所以每个线程都需要一份独立的程序计数器。  
这就是为什么它是线程私有的。

### 3. 虚拟机栈：方法调用的工作现场

虚拟机栈里装的是一个个栈帧。  
每次方法调用，都会创建一个新的栈帧压进去；方法返回时，这个栈帧再弹出去。

栈帧里通常包括：

- 局部变量表
- 操作数栈
- 动态链接信息
- 方法返回地址

所以如果你调用链很深，或者无限递归，就可能把栈打爆，这就是常说的栈溢出。

很多初学者会把“栈”理解成“所有临时变量都放栈里”。  
更准确的说法是：

- 局部变量和对象引用通常在栈帧里
- 对象实例本身通常在堆里

### 4. 堆：对象实例主要待的地方

堆是 JVM 里最重要也通常最大的一块区域。

大多数对象都会分配在堆上。  
你可以把堆想成“运行中对象的大仓库”。

垃圾回收器最关注的，也是堆里的对象，因为：

- 创建对象大多发生在堆
- 回收对象也主要在堆

后面我们讲的新生代、老年代、Region、Page，本质上都是围绕堆展开的。

### 5. 方法区：类的元信息放哪

方法区主要存放类相关信息，比如：

- 类结构信息
- 运行时常量池
- 方法元数据
- 静态变量等

它不是存普通对象实例的地方，而更像“类的档案室”。

所以对象和类一定要分开理解：

- 对象实例更多在堆里
- 类本身的描述信息更多在方法区这类地方

### 6. 一个对象是怎么创建出来的

当你写下：

```java
User user = new User();
```

表面上只是一个 `new`，但 JVM 背后做了不少事。

大致流程可以理解成：

1. 先确认 `User` 这个类有没有被加载、解析、初始化
2. 在堆里为对象分配内存
3. 把分配到的内存初始化成默认值
4. 设置对象头信息
5. 执行构造方法，把对象真正初始化成你想要的状态

也就是说，“分配空间”和“对象逻辑初始化”不是一回事。

### 7. 对象怎么被访问

当你在代码里写 `user.name` 时，并不是说变量 `user` 本身就装着整个对象。

更常见的理解应该是：

- 栈帧里的局部变量表里有一个引用
- 这个引用指向堆里的对象
- 再通过对象内部布局找到对应字段

所以“对象在堆里，引用在栈上”是个有帮助但不完全严格的入门说法。  
它帮助你快速区分：

- 变量位置
- 对象实体位置

但真实实现会更细，比如对象头、字段布局、压缩指针等。

### 8. 为什么理解这些区域对 GC 很重要

后面学 GC 时，你会经常听到这些词：

- GC Roots
- 从栈上引用的对象
- 堆上对象回收
- 方法区类元信息

如果你现在不知道对象和引用分别在哪，这些概念会特别抽象。

比如说“从 GC Roots 出发做可达性分析”，其中一个重要来源就是：

- 虚拟机栈里还在使用的引用

所以运行时数据区不是孤立知识，而是 GC 的前置地图。

## 小结

- JVM 运行时数据区是为了把不同职责的数据分开管理。
- 程序计数器和虚拟机栈偏线程私有，堆和方法区更偏共享运行时空间。
- 大多数对象实例在堆上，方法调用现场主要在栈上。
- 一个对象从 `new` 到真正可用，中间经历了类加载检查、内存分配、默认初始化、对象头设置和构造方法执行等步骤。
- 理解对象、引用、堆、栈的关系，是后续学习 GC 的基础。
