# JVM - 第 3 课：类加载机制，双亲委派、类加载器与类初始化

## 学习目标（本节结束后你能做到什么）

- 理解类为什么不是“程序启动时一次性全进内存”。
- 说清类加载的大致生命周期。
- 理解双亲委派到底在保护什么。
- 分清启动类加载器、扩展类加载器、应用类加载器的大致职责。
- 对“类被加载”和“类被初始化”建立清晰区别。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 类为什么需要“加载”

Java 程序写完后，类是以 `.class` 字节码文件形式存在的。  
JVM 不可能在程序刚启动时，就把所有类一次性全部处理完，因为：

- 类可能很多
- 有些类这次运行根本不会用到
- 有些类还可能在运行时动态生成

所以 JVM 的思路是：

**用到时再把类装进运行时世界。**

这就是类加载。

### 2. 类加载的大致生命周期

细分起来很复杂，但对学习最重要的是这几个阶段：

- 加载
- 验证
- 准备
- 解析
- 初始化

你可以先粗理解为：

1. 把字节码读进来
2. 检查它合不合法
3. 给类变量分配初始空间
4. 把符号引用解析成直接引用
5. 执行类初始化逻辑

其中最容易混的是“加载”和“初始化”。

### 3. 为什么“加载了类”不等于“初始化了类”

例如一个类被读取进 JVM，并不意味着它的静态代码块已经执行了。

初始化通常更接近：

- 真正执行类级别初始化逻辑
- 给静态变量赋开发者写下的初值
- 执行静态代码块

所以：

- 加载更像“把类引进来”
- 初始化更像“让类真正开始工作”

这也是面试里为什么常问“什么时候会触发类初始化”。

### 4. 类加载器到底是什么

类加载器本质上就是：

**负责把类字节码变成 JVM 运行时类对象的组件。**

你可以把它理解成图书管理员：

- 某本书在哪
- 该不该让它进馆
- 进来后归档在哪

不同类加载器会负责不同来源的类。

常见分工大致是：

- 启动类加载器：负责最核心的 Java 基础类库
- 平台/扩展类加载器：负责平台扩展类库
- 应用类加载器：负责应用自身 classpath 下的类

### 5. 双亲委派到底在做什么

双亲委派不是说“有父子继承关系”，而是说：

- 一个类加载器收到加载请求时
- 先不急着自己加载
- 先往上问父加载器能不能处理
- 父处理不了，自己再处理

这个机制最重要的价值是：

**避免核心类被随便篡改或重复加载。**

举个直觉例子：

假设你自己也写了一个 `java.lang.String`。  
如果没有委派机制，你的应用类加载器可能就真把它加载了，整个 Java 基础世界都会乱套。

双亲委派让加载 `String` 这种核心类的权力优先交给更上层、更可信的加载器。

### 6. 双亲委派除了安全，还有一致性价值

除了防篡改，它还有一个很重要的效果：

- 同一个类尽量由同一个层级加载器加载

否则会出现一个很经典的问题：

- 类全限定名一样
- 但加载器不同
- JVM 会把它们当成两个不同类

这也是很多框架、容器和热部署问题里，类加载器成为重点的原因。

### 7. 为什么类加载器会成为工程问题

在普通开发里，你可能感觉不到类加载器的存在。  
但到了这些场景，它就非常重要：

- Tomcat 多应用隔离
- SPI
- JDBC 驱动加载
- 热部署
- 插件化系统

因为这时候问题已经不是“类能不能加载”，而是：

- 谁来加载
- 从哪加载
- 加载后和别的类是不是兼容

### 8. 类加载和后续 JVM 学习的关系

理解类加载机制之后，后面的很多点会顺很多：

- 为什么对象创建前必须先确保类已加载
- 为什么方法区里要存类元信息
- 为什么反射、动态代理、SPI 都离不开类加载器

也就是说，类加载机制其实是 JVM 运行时入口之一。

## 小结

- 类不会在程序启动时一次性全部进入 JVM，而是按需加载。
- 类加载和类初始化不是一回事，初始化更接近“执行类级逻辑”。
- 类加载器负责把字节码变成 JVM 中可用的类对象。
- 双亲委派的核心价值是保护核心类加载安全，并维持类加载的一致性。
- 类加载器问题在普通开发中不显眼，但在容器、插件化和框架机制里非常关键。
