# JVM - 第 5 课：垃圾回收器全景，Serial、CMS、G1、ZGC 到底怎么选

## 学习目标（本节结束后你能做到什么）

- 建立各类垃圾回收器的整体地图，而不是只知道几个名字。
- 理解吞吐量、停顿时间、内存占用这几个维度分别代表什么。
- 分清“追求高吞吐”和“追求低停顿”不是一个目标。
- 理解 CMS、G1、ZGC 各自是在什么历史背景下出现的。
- 为后面深入 G1 和 ZGC 建立比较框架。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 JVM 会有这么多垃圾回收器

如果垃圾回收只有一个永远最优的方案，那 JVM 根本不需要搞这么多回收器。

问题就在于，不同系统的目标不一样：

- 有的系统更在乎吞吐量
- 有的系统更在乎停顿时间
- 有的系统内存非常大
- 有的系统更能接受偶尔长停顿

所以垃圾回收器本质上是在做取舍，而不是在追求一把万能锤子。

### 2. 看 GC 先看三个维度

#### 吞吐量

吞吐量关注的是：

**应用真正干活的时间占总时间的比例。**

如果一个系统允许偶尔停几百毫秒，但整体处理能力很强，那它可能更偏爱高吞吐回收器。

#### 停顿时间

停顿时间关注的是：

**一次 GC 最长会把业务线程停多久。**

对接口延迟敏感的系统，往往更看重这个指标。

#### 资源开销

包括：

- 额外内存占用
- CPU 开销
- 后台并发线程开销

低停顿通常不是白来的，往往要用更多结构、更复杂并发、更高 CPU 或内存成本换取。

### 3. 早期回收器：简单但停顿明显

像 Serial、Parallel 这类回收器，你可以先粗理解成：

- 设计更直接
- 可能会有更明显的 Stop-The-World

它们未必“落后”，只是目标更偏吞吐或简单性。

例如一些批处理、离线任务，对几百毫秒甚至更长停顿并不敏感，但非常在意整体吞吐，这类场景就未必追求最极致的低停顿。

### 4. CMS 为什么会出现

CMS 出现的核心背景是：

**大家开始越来越受不了老年代 GC 时的大停顿。**

所以 CMS 试图把老年代回收里的很多工作搬到并发阶段做，减少一次性 STW。

它的历史意义非常大，因为它代表了 JVM GC 从“只会停下来狠狠干”往“边跑边收”转变。

但 CMS 也有明显问题：

- 容易产生内存碎片
- 需要更多空间担保
- 并发阶段会和业务线程抢资源

所以它虽然重要，但不是最终答案。

### 5. G1 为什么会出现

G1 的目标不是简单替代 CMS 的所有细节，而是想进一步解决几个问题：

- 不只收老年代，而是能统一看整个堆
- 减少内存碎片
- 提供更可预测的停顿
- 在大堆场景下表现更稳

G1 最重要的设计变化是：

- 不再坚持传统整块新生代 / 老年代布局
- 而是把堆切成很多 Region

然后根据回收价值去挑 Region 回收。

所以 G1 的关键词是：

- Region
- 可预测停顿
- Young GC
- Mixed GC

### 6. ZGC 为什么会出现

即使 G1 已经很强了，随着堆越来越大、业务对延迟越来越敏感，大家还是会继续追问：

**能不能把停顿再压低，甚至压到个位数毫秒级？**

这就是 ZGC 诞生的背景。

ZGC 的目标很鲜明：

- 极低停顿
- 尽可能把回收过程并发化

为了做到这一点，ZGC 引入了非常不一样的技术路线，比如：

- 染色指针
- 读屏障
- 对象迁移时的“自愈”

所以它和 G1 的差异不只是参数不同，而是设计哲学都更激进。

### 7. 怎么理解 CMS、G1、ZGC 的历史演进

如果用一句话概括演进方向，大致是：

- 早期方案：实现简单，但停顿更明显
- CMS：开始认真压低老年代停顿
- G1：更系统化地管理整个堆，追求可预测停顿
- ZGC：进一步把停顿压到极低，几乎全程并发

也就是说，GC 的演进不是“旧的完全错，新的完全对”，而是：

**系统对延迟越来越敏感，于是回收器不断往更复杂的并发化方向发展。**

### 8. 到底怎么选

没有脱离场景的标准答案，但可以先有一个大方向：

- 吞吐优先、停顿要求没那么苛刻：可以更偏传统吞吐型选择
- 大堆、服务端、希望停顿可控：G1 很常见也很稳妥
- 对停顿极度敏感，愿意接受更复杂设计和一定吞吐代价：可以重点关注 ZGC

所以真正成熟的问法不是：

- 哪个 GC 最牛

而是：

- 我的系统最怕什么
- 我想拿什么换什么

## 小结

- JVM 有很多垃圾回收器，不是因为设计混乱，而是因为不同系统目标不同。
- 看 GC 至少要看吞吐量、停顿时间和资源开销三条线。
- CMS 是低停顿方向的重要里程碑，但有碎片和空间担保等问题。
- G1 重点在 Region 化和可预测停顿，ZGC 重点在极低停顿和更彻底的并发化。
- 选 GC 不是选“最先进”的，而是选最适合你业务目标的。
