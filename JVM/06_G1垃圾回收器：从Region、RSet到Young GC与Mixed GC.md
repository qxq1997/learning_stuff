# JVM - 第 6 课：G1 垃圾回收器，从 Region、RSet 到 Young GC 与 Mixed GC

## 学习目标（本节结束后你能做到什么）

- 不再把 G1 理解成“一个比较新的 GC 名字”，而是真正理解它想解决什么问题。
- 理解 Region、Card Table、RSet 各自在解决什么问题。
- 说清 G1 的 Young GC 和 Mixed GC 分别是什么。
- 理解三色标记、SATB 和 G1 并发标记之间的关系。
- 能用更自然的语言把 G1 的整体流程讲给别人听。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先说 G1 到底想解决什么问题

G1 的名字叫 `Garbage First`，但你不要一上来就被这个名字绕进去。

G1 真正想解决的问题，主要有三个：

- **堆越来越大时，传统回收器停顿太难看**
- **CMS 虽然能降低停顿，但会有碎片和一些老问题**
- **希望 GC 不只是“尽量快”，而是“停顿时间更可预测”**

所以理解 G1 的正确入口不是“它有哪些阶段”，而是：

**它想在大堆场景下，把回收做得更稳、更可控。**

### 2. 为什么 G1 不再坚持传统整块新生代 / 老年代布局

传统分代思路里，我们常把堆理解成：

- 一整块年轻代
- 一整块老年代

这样思考当然直观，但在大堆场景下会遇到一个问题：

- 如果某一代很大，每次回收就很容易牵一大片

G1 的做法是把整个堆切成很多大小相同的 **Region**。  
每个 Region 在某一时刻可以扮演不同角色：

- Eden
- Survivor
- Old
- Humongous

注意，这里不是说一个 Region 永久属于某一代，而是它的角色可以变化。

这一步特别关键，因为它让 G1 拥有了更灵活的回收单位。

你可以把传统堆想成“大平层”，而 G1 更像把房子切成很多独立房间。  
以后回收时，不一定非得一整层全动，而可以优先挑“最值得清理的房间”。

### 3. Humongous Region 为什么存在

大对象是 GC 世界里的一个麻烦角色。

如果你把一个很大的对象也像普通对象一样来回复制，复制成本会很高；  
但如果把它粗暴扔进老年代，又可能让一些本来短命的大对象过早占据老年代空间。

于是 G1 引入了 Humongous Region 来专门处理大对象。

你可以先这样记：

- 普通对象走普通 Region 分配逻辑
- 特别大的对象走 Humongous 路线

这不是为了让概念更多，而是为了避免大对象把普通回收路径搞得太难受。

### 4. Card Table 和 RSet 到底在解决什么问题

这是 G1 里最容易让人一上来就看晕的地方。

如果只记定义会很痛苦，所以先抓住问题本身：

**当我要回收一部分 Region 时，怎么知道外部还有谁在引用它里面的对象？**

假设现在你想只回收年轻代那部分 Region。  
理论上，为了确认哪些年轻代对象还活着，你需要从 GC Roots 往下找引用链。

但问题来了：  
如果年轻代对象被老年代对象引用了，那你是不是要把整个老年代都扫一遍？

这显然太贵。

于是 G1 引入了记忆集思想：

- `Card Table`：把 Region 再切成更小的卡片单位，方便记录引用变化
- `RSet`（Remembered Set）：记录“别的 Region 对我这个 Region 的引用入口”

你可以把 RSet 粗理解成：

**某个 Region 的‘外部引用备忘录’。**

这样当我要回收 Region3 时，不需要重新把全堆扫一遍，而是可以先看：

- 谁在外面引用我
- 引用大概落在我哪几张卡上

这大幅减少了扫描范围。

### 5. 为什么更新 RSet 不直接在每次赋值时立刻完成

因为引用赋值操作很频繁。

如果每次对象引用一变化，就立刻去更新 RSet，线程之间会出现明显竞争，性能会很差。  
所以 G1 的思路是：

- 先把“哪张卡脏了”记录下来
- 放到 Dirty Card Queue 里
- 后续再由后台线程慢慢整理并更新到 RSet

这样做的本质是：

**把高频写路径上的直接维护成本，延后到更可控的异步整理流程。**

这也是为什么你会在 G1 里看到：

- Dirty Card Queue
- Refinement 线程

它们不是为了复杂而复杂，而是在给 RSet 的维护降成本。

### 6. G1 的 Young GC 怎么理解

先抓一句最重要的话：

**G1 的 Young GC 本质上仍然是在回收年轻代，只不过年轻代不再是一整块连续内存，而是一组 Eden / Survivor 角色的 Region。**

Young GC 大致可以这样理解：

1. STW
2. 扫描 GC Roots
3. 利用 RSet 补上老年代到年轻代的引用信息
4. 找出年轻代里还活着的对象
5. 把它们复制到 Survivor 或晋升到 Old
6. 清空回收掉的 Eden Region

这里最关键的一步是：

**它不用为了找跨代引用而把整个老年代全扫一遍。**

这正是 RSet 的核心价值。

### 7. 为什么 G1 还需要三色标记和 SATB

Young GC 相对还比较直观，真正复杂的是老年代相关的并发标记。  
这就回到了上一课讲的三色标记问题：

- 并发标记时，业务线程还在改引用
- 一改，就可能漏标

G1 采用的是 `SATB`（Snapshot At The Beginning）思路。

它的直觉可以这么理解：

**既然并发过程中引用关系一直在变，那我干脆把“本轮标记看到的世界”尽量固定成“开始那一刻的快照”。**

这样做会牺牲一点精度，带来所谓浮动垃圾，但能换来并发标记的正确性和可实现性。

所以学习 G1 时，一定要明白：

- SATB 不是为了炫技
- 它是在解决并发标记漏标问题

### 8. G1 的 Mixed GC 到底是什么

这是原文里最容易让人看着看着就迷路的地方。

很多人一听 Mixed GC，就以为它是“整个堆彻底大扫除”。  
其实不是。

Mixed GC 的意思更接近：

- 先通过全局并发标记，搞清楚哪些 Old Region 垃圾多、回收值高
- 然后在后续回收时，不只回收年轻代
- 还**顺手挑一部分最值得回收的 Old Region 一起收**

所以它叫 Mixed，不是因为“所有区混在一起乱收”，而是因为：

**年轻代 Region + 部分老年代 Region 混合进入本轮回收集合。**

### 9. G1 的全局并发标记怎么理解

可以按这几个阶段来抓主线：

#### 初始标记

会有一次短暂停顿，先把从 GC Roots 直接能碰到的对象标出来。

#### 根分区扫描

进一步处理和根分区相关的引用关系，尤其是和 Survivor 有关的部分。

#### 并发标记

真正花时间的大头，在这里和业务线程并发进行，对整个堆做可达性分析。

#### 最终标记

因为并发阶段仍可能有引用变化，所以需要一次短暂停顿来把尾巴收干净。

#### 清理 / 选择回收集

这一步不是马上大搬家，而是先评估：

- 哪些 Region 垃圾多
- 哪些 Region 值得优先收

然后形成 Collection Set。

所以这条线的本质是：

**先全局看清垃圾分布，再局部挑最值钱的 Region 去回收。**

### 10. 为什么 G1 叫 Garbage First

现在你就能更自然理解这个名字了。

它不是说“永远先收垃圾最多的单个对象”，而是说：

- 在 Region 级别评估回收收益
- 优先选择回收价值高的 Region

也就是说，G1 追求的是：

**有限停顿预算下，优先收最划算的那部分堆空间。**

### 11. 怎么评价 G1

G1 的优点很鲜明：

- 面向大堆更稳
- 能减少碎片
- 停顿时间相对更可预测
- 统一了年轻代和部分老年代回收策略

但它也不是没有代价：

- 结构更复杂
- 需要维护额外元数据，比如 RSet
- 内存开销比简单回收器更高

所以 G1 的价值不在于“绝对最简单”，而在于：

**它是在复杂度可接受的前提下，把停顿控制和大堆回收做得更均衡。**

## 小结

- G1 的核心目标是面向大堆场景，提供更可预测的停顿时间。
- Region 是 G1 最重要的结构基础，它让堆从“大块划分”变成“很多可灵活选择的小块”。
- Card Table 和 RSet 的核心任务，是避免为了处理跨 Region / 跨代引用而扫描整个大堆。
- Young GC 仍然是年轻代回收，只是年轻代由多块 Region 扮演；Mixed GC 则是在回收年轻代时顺带回收部分高价值老年代 Region。
- SATB 不是额外负担，而是 G1 为了解决并发标记漏标问题付出的关键设计。
