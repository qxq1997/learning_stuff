# JVM - 第 7 课：ZGC，从染色指针、读屏障到几乎全程并发回收

## 学习目标（本节结束后你能做到什么）

- 理解 ZGC 到底想解决什么问题，而不是只会背“低延迟”三个字。
- 理解 Page、染色指针、读屏障、转发表、自愈这些关键词各自在做什么。
- 理解 ZGC 为什么能把停顿时间压得很低。
- 理解 ZGC 和 G1 的差异不只是流程不同，而是设计思路不同。
- 能把原本很绕的 ZGC 流程，用更容易理解的语言讲出来。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先说 ZGC 到底想解决什么问题

如果说 G1 的关键词是“可预测停顿”，那 ZGC 的关键词就是：

**极低停顿。**

它想解决的是这样一类场景：

- 堆很大
- 服务很忙
- 但业务又特别怕 Stop-The-World

例如有些在线服务，延迟抖一下就可能直接影响用户体验。  
这时候即使 G1 已经不错了，团队仍然会追问：

- 能不能把停顿再压低？
- 能不能把更多工作放到并发阶段？

ZGC 就是在这个方向上走得非常激进的回收器。

### 2. 为什么 ZGC 和 G1 给人的感觉差别这么大

G1 的直觉更像：

- 把堆切成 Region
- 做好记录
- 挑值得回收的 Region
- 尽量控制停顿

ZGC 的直觉则更像：

- 允许对象搬家
- 允许引用在访问时被修正
- 尽量别为了“统一修所有指针”而停住世界

也就是说，G1 重点在“选哪些地方收、怎么稳稳地收”，  
ZGC 重点在“对象搬了也没关系，我想办法让访问路径自己修过来”。

这就是为什么 ZGC 会引出一整套看起来很陌生的词：

- 染色指针
- 读屏障
- 转发表
- 自愈

### 3. ZGC 的 Page 是什么

ZGC 和 G1 一样，也不是把整个堆当成一整块大内存来管理。  
它也会把堆切成很多块，但官方更常叫它 **Page**。

你可以先粗理解成：

- G1 的基本回收和管理单位叫 Region
- ZGC 的基本管理单位叫 Page

ZGC 会根据对象大小把 Page 做不同分类：

- 小对象用小 Page
- 中对象用中 Page
- 特别大的对象用大 Page

这一点本身不是 ZGC 最难的部分，它只是说明：

**ZGC 也在做分块管理，只是它真正厉害的不在“切块”，而在“并发搬迁和引用修正”。**

### 4. 染色指针到底是什么

这是 ZGC 里最容易把人吓到的词。

先不要急着背位图细节，先抓住思想：

正常来说，一个引用就是一个地址。  
ZGC 的做法是：

**不只把它当地址，还顺手在高位里塞一点 GC 阶段相关的信息。**

这就叫染色指针（Colored Pointer）。

你可以把它理解成：

- 普通指针：只有“住址”
- 染色指针：住址上再贴一个“当前状态标签”

这个标签能帮助 JVM 判断：

- 这个引用现在属于哪个视图阶段
- 它是不是已经被标过
- 它是不是需要重映射

也就是说，染色指针不是为了炫技，而是为了让 GC 状态信息跟着引用一起走。

### 5. 读屏障在做什么

读屏障也非常容易被讲晦涩。

把它说简单一点：

**当程序读取一个对象引用时，JVM 先插一小段检查逻辑。**

这个检查逻辑会看看：

- 当前引用状态对不对
- 如果对象已经搬家了，我是不是该把引用修正到新地址

你可以把它想成快递改地址系统：

- 旧地址还能收到信
- 但门卫会告诉你“这个人搬去新地址了”
- 同时把通讯录也帮你改掉

这就是后面会提到的“自愈”。

### 6. 什么叫自愈（Self-Healing）

这是理解 ZGC 的关键直觉。

传统思路里，如果对象搬家了，大家很自然会想：

- 那不得先把所有指向它的旧引用都找出来
- 全部改对之后，程序才能继续安全访问？

这件事很贵，也很容易带来更长停顿。

ZGC 的思路更聪明：

- 对象先搬走
- 老地址和新地址的对应关系先记下来
- 以后谁再通过旧引用访问它，就在读屏障里顺手修正

这就像：

- 你朋友搬家了
- 不需要先全城广播所有人手里的旧通讯录
- 谁哪天真的联系他了，再即时告诉他新地址并改掉记录

所以“自愈”的本质是：

**引用不要求一次性全修完，而是在访问路径上渐进修复。**

这正是 ZGC 能把停顿压低的关键原因之一。

### 7. Forwarding Table 为什么存在

要做到“对象搬家后访问时修正”，JVM 必须知道：

- 旧对象地址对应的新对象地址是什么

这就需要一个映射关系表，通常就叫转发表（Forwarding Table）。

它的作用非常直接：

- 旧地址 -> 新地址

当读屏障发现你拿的是旧引用时，就去查这张表，把访问转去新对象，并把引用修正。

所以 Forwarding Table 可以理解成：

**对象搬迁期间的‘最新地址登记簿’。**

### 8. ZGC 的整体流程怎么抓主线

原文如果直接从各种术语切进去，会让人很晕。  
更容易理解的办法是先抓住大主线：

1. 先标记哪些对象还活着
2. 再选出哪些 Page 值得回收 / 需要迁移
3. 把活对象搬到新地方
4. 利用读屏障和转发表，让旧引用在访问时逐步自愈

如果把这条线记住，后面那些阶段名就不会散。

### 9. 初始标记和并发标记在干嘛

和很多现代 GC 一样，ZGC 也需要先知道谁活着。

#### 初始标记

会有一次非常短的 STW，从 GC Roots 出发，先把最直接那批对象确定下来。

#### 并发标记

真正大量的标记工作在这里和业务线程并发完成。

这一步里，染色指针开始真正发挥作用：  
JVM 可以根据指针上的状态信息，识别哪些对象已经纳入当前标记视图。

所以这条线的重点是：

- 不是单纯“标过没标过”
- 而是“如何在并发场景下识别当前标记视图”

### 10. 为什么 ZGC 还需要重新标记

因为并发标记时，业务线程仍在运行，引用关系可能继续变化。

所以 ZGC 也需要一次很短的收尾阶段，把并发阶段没来得及处理干净的那部分工作补齐。

这和 G1、CMS 一类思路并不矛盾，只是 ZGC 会尽量把这次停顿做得更短。

### 11. 迁移阶段到底在做什么

标记完成后，ZGC 会识别哪些 Page 要进入重定位集合。

然后它会：

- 把存活对象复制到新的 Page
- 在转发表里登记新旧地址映射

这一步最重要的思想不是“搬对象”本身，而是：

**对象搬完后，不要求马上把所有引用一次性全修好。**

这件事后面由读屏障和自愈慢慢完成。

### 12. 并发重映射为什么没你想象中那么死板

原文里这部分常常越看越绕。  
你可以先抓一句话：

**ZGC 并不执着于在本轮 GC 里一次性修完所有引用。**

因为只要有：

- 读屏障
- 转发表
- 染色指针状态识别

那旧引用即使暂时还在，也不是立刻危险。  
谁访问，谁修正。

这也是 ZGC 与传统“必须先统一修完再放行业务”的思路最大的不同。

### 13. 为什么 ZGC 停顿能这么低

现在我们可以把答案真正说完整了。

ZGC 停顿低，不是因为“它更快”这么简单，而是因为它做了这些设计：

- 大量工作并发完成
- 用染色指针携带状态信息
- 用读屏障在访问时介入
- 用转发表保存搬迁关系
- 用自愈避免一次性全量修指针

也就是说，它不是把所有工作都做得更快，而是把大量原本必须停世界才能做的事，改造成了运行时渐进完成。

### 14. 怎么评价 ZGC

ZGC 的优势非常鲜明：

- 停顿极低
- 大堆场景下延迟表现很有吸引力

但它不是没有代价：

- 设计复杂
- 需要更多运行时机制配合
- 某些版本下吞吐未必总赢 G1

所以 ZGC 的价值，不在于“无脑碾压所有 GC”，而在于：

**如果你的系统极度怕停顿，它提供了一条非常强力的低延迟路线。**

## 小结

- ZGC 的核心目标是极低停顿，而不是单纯追求高吞吐。
- 染色指针本质上是在引用里顺带携带 GC 状态信息。
- 读屏障的价值是在读取引用时介入检查，并在必要时完成引用修正。
- 自愈的本质是：对象搬家后不要求立刻全量修引用，而是在后续访问中渐进修复。
- ZGC 停顿能很低，关键不只是“并发”，更是它把“引用修正”从集中式工作改造成了分散式、访问时修复。
