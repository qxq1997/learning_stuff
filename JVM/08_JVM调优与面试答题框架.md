# JVM - 第 8 课：JVM 调优与面试答题框架

## 学习目标（本节结束后你能做到什么）

- 建立 JVM 调优的基本思路，而不是把调参理解成“背几个参数”。
- 说清线上 JVM 问题排查时应该先看什么。
- 理解“选 G1 还是 ZGC”应该基于什么判断。
- 建立 JVM 面试中的稳定答题框架。
- 把前面各课知识真正串成一张图，而不是散在脑子里。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. JVM 调优最容易犯的错误

很多人一说 JVM 调优，第一反应就是：

- 改 `-Xmx`
- 改 `-Xms`
- 切 GC
- 背几个参数

这其实很危险，因为 JVM 调优不是“调参竞赛”，而是问题定位。

更好的顺序应该是：

1. 先确认症状
2. 再确认瓶颈在哪
3. 再决定要不要调 JVM
4. 最后才是具体调什么

因为线上很多问题看上去像 JVM，实际上可能是：

- 代码泄漏
- 慢 SQL
- 锁竞争
- I/O 卡顿
- 下游依赖抖动

JVM 调优从来不是脱离业务和系统上下文的孤立动作。

### 2. 排查 JVM 问题先看什么

可以先建立一个很实用的顺序：

#### 先看症状

- 是吞吐低？
- 是接口抖动？
- 是 RT 尖刺？
- 是内存越来越高？
- 是 Full GC 太频繁？

#### 再看数据

- 堆使用量
- GC 次数和暂停时间
- 年轻代回收频率
- 老年代增长速度
- 线程数
- CPU 使用率

#### 最后再看原因

- 对象分配过快？
- 晋升过快？
- 大对象太多？
- 缓存没上限？
- 某批任务造成短时间对象风暴？

这比一上来就猜“是不是 G1 不好”稳得多。

### 3. 什么情况下更偏向 G1

如果你的系统是典型服务端应用，特点大致是：

- 堆不小
- 需要较稳定的停顿表现
- 不想上特别激进的新路线

那 G1 往往是一个很自然、很稳妥的选择。

它的优点是：

- 生态成熟
- 默认支持度高
- 可预测停顿思路清晰

所以很多时候，G1 不是“最酷”的选择，但经常是“最稳”的选择。

### 4. 什么情况下值得认真考虑 ZGC

如果你的系统很怕延迟抖动，尤其是：

- 大堆
- 高并发服务
- 对暂停特别敏感

那 ZGC 就值得认真评估。

但评估不等于无脑切换。你还得看：

- JDK 版本
- 现网表现
- 吞吐目标
- 团队对新 GC 的熟悉度

所以面试里如果被问“G1 和 ZGC 怎么选”，成熟答案一定不是一句“ZGC 更新，所以更好”，而是：

- 先讲目标
- 再讲取舍

### 5. JVM 面试怎么答才稳

JVM 面试最怕两种情况：

- 只会背定义
- 一开口就细节爆炸，把自己绕进去

更稳的答题框架通常是：

#### 第一步：先讲它在解决什么问题

例如问 G1：

- 它主要想解决大堆下停顿可预测问题

问 ZGC：

- 它主要想把停顿压到极低

#### 第二步：再讲关键设计

例如：

- G1：Region、RSet、SATB、Young GC / Mixed GC
- ZGC：染色指针、读屏障、转发表、自愈

#### 第三步：最后讲代价和边界

这一步最能体现成熟度。

例如：

- G1 结构复杂、额外元数据多
- ZGC 停顿低，但设计更复杂，未必总是吞吐最优

### 6. 最有价值的 JVM 学习方法

到这里你会发现，JVM 知识真正难的不是定义，而是关系。

你应该把前面几课串起来：

- 类加载：类怎么进入 JVM
- 运行时数据区：对象和类待在哪里
- 可达性分析：怎么判断对象是否还能活
- 分代思想：为什么对象回收要分层做
- G1 / ZGC：如何在不同目标下实现回收

如果能把这张图讲清楚，你的 JVM 就不是零碎八股，而是系统理解。

## 小结

- JVM 调优的前提是先定位问题，不是先背参数。
- G1 更像成熟稳妥的默认路线，ZGC 更像低停顿目标下的激进路线。
- JVM 面试回答最稳的方式，是“问题 -> 设计 -> 代价和边界”。
- 真正高质量的 JVM 学习，不是记住很多术语，而是能把类加载、内存、对象、GC 和回收器设计串起来。
- 到这一课为止，你应该开始形成自己的 JVM 叙事框架，而不再只是重复书上的原句。
