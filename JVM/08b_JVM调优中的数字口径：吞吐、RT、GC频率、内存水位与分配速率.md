# JVM - 第 8b 课：JVM 调优中的数字口径：吞吐、RT、GC 频率、内存水位与分配速率

## 学习目标（本节结束后你能做到什么）

- 不再用“GC 有点频繁”“RT 有点抖”这种模糊说法，而是能给出一组默认数字口径。
- 知道什么叫吞吐低、什么叫接口抖动、什么叫内存水位危险。
- 能区分“对象分配太快”“对象晋升太快”“活对象本来就很多”这三种完全不同的问题。
- 看 `jstat`、GC 日志或监控面板时，知道哪些数字进入了关注区、危险区。
- 建立一套适合在线 Java 服务的默认判断标准，再按业务 SLA 做微调。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先说明一个前提：下面这些数字是默认工程基线，不是宇宙真理

很多同学最想听的是：

- 多久一次 GC 算频繁
- 多大的停顿算大
- 内存用到多少算危险

这些问题确实需要数字，但必须先说清楚一个前提：

**所有阈值都和业务类型有关。**

同样是 `80ms` 的停顿：

- 对一个离线任务，可能几乎没感觉
- 对一个 TP99 目标只有 `50ms` 的在线接口，就已经很危险

所以这一课给你的不是“永远正确的唯一标准”，而是一套适合大多数在线 Java 服务的默认基线。你可以把它理解成：

- 先有一把默认尺子
- 再根据你们服务的 SLA 去修正

这套尺子主要适用于：

- RPC 服务
- HTTP API 服务
- 网关
- 中台服务
- 绝大多数延迟敏感的后端服务

如果是批处理、离线计算、大数据任务，这些数字要放宽很多。

### 2. 先把观察窗口定下来，不然数字会失真

很多人看 GC 指标会混乱，一个很大原因是：

- 有的数字应该按“单次事件”看
- 有的应该按“1 分钟窗口”看
- 有的应该按“10 分钟趋势”看

建议先固定三种观察口径：

#### 单次事件口径

适合看：

- 单次 Young GC 停顿
- 单次 Mixed GC / CMS Remark 停顿
- 单次 Full GC 停顿
- 某一次 RT 尖刺

#### 1 分钟窗口口径

适合看：

- GC 总暂停时间占比
- Young GC 频率
- RT 抖动
- 短时间对象分配速率

#### 5 到 15 分钟趋势口径

适合看：

- 老年代基线是否持续抬高
- MetaSpace 是否持续上涨
- Full GC 是否周期性出现
- 堆回落是否越来越差

你以后做判断时，最好先问一句：

**我现在看的这个指标，应该按单次、分钟，还是趋势来看？**

### 3. 什么叫“吞吐低”：先看 GC 暂停时间占比

理论上的吞吐量定义很严格，但线上最实用的第一把尺子其实是：

**GC 暂停时间占比。**

公式可以先简单记成：

```text
GC 暂停时间占比 = 观察窗口内所有 STW 暂停总时间 / 观察窗口总时长
```

例如：

- 1 分钟内 GC 总共停了 `600ms`
- 那么暂停时间占比就是 `600 / 60000 = 1%`

对在线接口服务，可以先用下面这张表做默认判断：

| 1 分钟窗口内 GC 暂停占比 | 判断 |
| --- | --- |
| `< 1%` | 很健康 |
| `1% ~ 3%` | 基本可接受 |
| `3% ~ 5%` | 明显偏高，值得关注 |
| `5% ~ 10%` | 吞吐开始明显受影响 |
| `> 10%` | 基本进入故障态或强风险区 |

换成更直观的数：

- 1 分钟内 GC 总暂停 `< 600ms`：通常还算健康
- 1 分钟内 GC 总暂停 `600ms ~ 1800ms`：可以接受但需要观察
- 1 分钟内 GC 总暂停 `1800ms ~ 3000ms`：已经偏高
- 1 分钟内 GC 总暂停 `> 3000ms`：大概率会拖吞吐、拖 RT

这里要注意：

- 这只是第一层判断
- 不代表 `1%` 就一定没问题

因为有些系统虽然暂停占比不高，但单次停顿很长，用户照样会感知到。

### 4. 什么叫“RT 抖动”或“接口尖刺”

很多人把 RT 抖动理解得太玄了。  
线上最实用的定义其实很简单：

**尾延迟明显高于基线，并且和 GC 停顿时间窗口对齐。**

你可以先用两把尺子：

#### 尺子一：看相对基线涨了多少

假设某个接口平时：

- `p50 = 8ms`
- `p99 = 35ms`

如果一段时间变成：

- `p50 = 9ms`
- `p99 = 80ms`

那这就不是“整体变慢”，而是典型的尾延迟抖动。

对在线服务，可以先这么判断：

| RT 指标变化 | 判断 |
| --- | --- |
| `p99` 比基线涨幅 `< 20%` | 正常波动 |
| `20% ~ 50%` | 轻微抖动 |
| `50% ~ 100%` | 明显抖动 |
| `> 100%` | 尖刺或故障级抖动 |

#### 尺子二：看 STW 停顿和接口目标的关系

这是更关键的一条：

| 单次 STW 停顿 vs 接口目标 | 判断 |
| --- | --- |
| `< 接口 TP99 的 1/4` | 通常不明显 |
| `约等于接口 TP99 的 1/2` | 已可能被用户感知 |
| `>= 接口 TP99` | 很可能直接打穿尾延迟 |
| `> 接口 TP999 或 > 300ms` | 非常危险 |

举个例子：

- 你的接口 TP99 目标是 `80ms`
- 如果某次 Young GC 停了 `8ms`

通常影响不大。

- 如果某次停了 `40ms`

就已经进入“会让尾延迟难看”的区间。

- 如果某次停了 `120ms`

那这次 GC 几乎肯定会在监控上表现成一次明显尖刺。

所以你以后别再只问：

- GC 停顿大不大

而应该问：

- 这个停顿相对于我的接口 SLA 到底大不大

### 5. 什么叫“GC 太频繁”：分 Young、Old/Mixed、Full 三层看

#### 5.1 Young GC 多久一次算频繁

Young GC 不能简单地说“发生了就不好”，因为它本来就应该频繁一点。  
真正有意义的是看：

- 它多久来一次
- 单次停多久
- 一分钟加总停了多久

对在线服务，可以先用下面这个默认口径：

| Young GC 频率 | 等价口径 | 判断 |
| --- | --- | --- |
| `<= 6 次/分钟` | `>= 10 秒/次` | 通常健康 |
| `6 ~ 30 次/分钟` | `2 ~ 10 秒/次` | 需要关注 |
| `30 ~ 60 次/分钟` | `1 ~ 2 秒/次` | 明显频繁 |
| `> 60 次/分钟` | `< 1 秒/次` | 对象分配风暴或新生代过小 |

同时还要结合单次停顿：

| 单次 Young GC 停顿 | 判断 |
| --- | --- |
| `< 10ms` | 很健康 |
| `10ms ~ 30ms` | 常见可接受 |
| `30ms ~ 50ms` | 偏高 |
| `> 50ms` | 对低延迟服务开始危险 |
| `> 100ms` | 已非常不正常 |

一个非常实用的判断方法是：

- 如果 Young GC 虽然很多，但单次只有几毫秒，而且分钟级暂停占比不高

那通常还没有到事故级。

- 如果 Young GC 已经到了 `每 1 到 2 秒一次`，而且单次还有 `30ms` 以上

那大概率已经会影响接口表现了。

#### 5.2 CMS Old GC / G1 Mixed GC 多久一次算频繁

Old 相关回收的容忍度应当更低，因为它们更重，也更容易带来长停顿。

可以先按下面这个口径看：

| CMS Old GC / G1 Mixed GC 频率 | 判断 |
| --- | --- |
| `>= 30 分钟/次` | 通常可接受 |
| `5 ~ 30 分钟/次` | 需要关注 |
| `1 ~ 5 分钟/次` | 明显频繁 |
| `< 1 分钟/次` | 高风险 |

单次停顿可以先这样看：

| 单次 Old / Mixed / Remark 停顿 | 判断 |
| --- | --- |
| `< 100ms` | 较健康 |
| `100ms ~ 300ms` | 已开始有业务感知风险 |
| `300ms ~ 1000ms` | 高风险 |
| `> 1000ms` | 事故级信号 |

这里有个常见误区：

- 很多人只盯单次停顿

但 Old GC 更常见的问题其实是：

- 单次不一定特别长
- 但是来得太勤
- 最后把吞吐一点点吃掉

#### 5.3 Full GC 多久一次算危险

对绝大多数在线服务来说，`Full GC` 的默认目标就是：

**尽量不发生。**

你可以先这么记：

| Full GC 情况 | 判断 |
| --- | --- |
| `0 次` | 最理想 |
| `偶发 1 次，但能解释原因` | 需要排查，不算直接事故 |
| `>= 1 次/小时` | 明显有问题 |
| `>= 1 次/10 分钟` | 高危 |
| `连续发生` | 故障态 |

单次 Full GC 停顿也可以直接这样记：

| 单次 Full GC 停顿 | 判断 |
| --- | --- |
| `< 300ms` | 少见但还能忍 |
| `300ms ~ 1000ms` | 明显危险 |
| `1s ~ 3s` | 对在线服务很伤 |
| `> 3s` | 很容易造成雪崩 |

### 6. 什么叫“内存高”：看 GC 后的水位，而不是只看 GC 前峰值

很多同学看内存时只盯一个数字：

- 堆用了 `80%`

这不够，因为 GC 前 `80%` 和 GC 后 `80%` 完全不是一回事。

更有价值的是：

**看一次回收之后，还剩多少活对象。**

#### 6.1 Old 区 / 堆在 GC 后的水位

你可以先用下面这个口径：

| GC 后 Old 区或老年代相关水位 | 判断 |
| --- | --- |
| `< 60%` | 比较从容 |
| `60% ~ 75%` | 偏高但可控 |
| `75% ~ 85%` | 风险上升 |
| `> 85%` | 高风险 |
| `> 90%` | 非常危险，容易触发退化或 Full GC |

这个判断比“堆峰值多少”更重要，因为它更接近真实活对象规模。

#### 6.2 MetaSpace 水位

MetaSpace 也不要只看峰值，最好看 GC 后是否还能回落：

| GC 后 MetaSpace 使用率 | 判断 |
| --- | --- |
| `< 70%` | 比较安全 |
| `70% ~ 85%` | 需要关注 |
| `85% ~ 90%` | 明显偏高 |
| `> 90%` | 高风险，可能存在持续类加载 |

#### 6.3 什么样像内存泄漏

如果你看到的是下面这种走势，就要开始怀疑泄漏或活对象持续增长：

- 每次 GC 后，堆的最低点都比上一次更高
- 连续 `3` 次以上 Old / Mixed / Full GC 后，最低点仍然持续抬升
- 单次 Old / Mixed / Full GC 的回收比例已经低于 `10% ~ 15%`

这通常说明：

- 不是“瞬时峰值高”
- 而是“活对象底盘越来越大”

### 7. 什么叫“对象分配太快”：先看 Eden 被填满得有多快

对象分配速率是 JVM 调优里最容易被忽略的一条线。  
很多线上问题不是 GC 算法差，而是：

**对象创建速度实在太快，年轻代根本顶不住。**

最实用的近似判断方式不是先算复杂公式，而是先看：

**Eden 或 Young 区多久被填满一次。**

你可以直接用 Young GC 间隔近似感知：

| Eden / Young 被填满时间 | 判断 |
| --- | --- |
| `>= 10 秒` | 通常比较平稳 |
| `3 ~ 10 秒` | 分配压力偏高 |
| `1 ~ 3 秒` | 分配很快 |
| `< 1 秒` | 很可能是对象风暴 |

举个例子：

- Young 区约 `1GB`
- Young GC 间隔约 `2 秒`

那你可以粗理解为这个实例在以接近 `500MB/s` 的速度制造短命对象。

这时候你就别先急着怪 GC，而要先想：

- 最近是不是有一批请求进来了
- 是不是反序列化对象暴增
- 是不是某个批任务在疯狂构造临时对象
- 是不是日志、JSON、DTO 转换过于密集

### 8. 什么叫“晋升太快”：看老年代增长速度，不要只看 Young GC 次数

有时候 Young GC 很多，但问题不在 Young 本身，而在于：

**Young 回收完以后，很多对象都活着进了 Old。**

这时候你要看的是晋升速度。

一个很实用的经验判断是：

- 如果老年代在每次 Young GC 后都明显上涨
- 并且 `5 到 10 分钟` 内几乎只涨不回
- 或者晋升速率已经接近对象分配速率的 `20% ~ 30%` 以上

那就说明很多对象已经不是“朝生夕死”的了。

这类问题常见于：

- 新生代太小
- Survivor 太小
- 动态年龄导致过早晋升
- 缓存或连接对象存活时间比想象长

如果进一步看到：

- Young GC 很频繁
- Old 水位还在持续上涨
- CMS / Mixed GC 也越来越勤

那就基本可以判断：  
不是单纯“新生代不够”，而是**对象生命周期结构和分代配置不匹配**。

### 9. 一张适合在线服务的默认速查表

下面这张表你可以当成第一版告警口径：

| 指标 | 健康区 | 关注区 | 危险区 |
| --- | --- | --- | --- |
| 1 分钟 GC 暂停占比 | `< 1%` | `1% ~ 5%` | `> 5%` |
| Young GC 频率 | `<= 6 次/分钟` | `6 ~ 30 次/分钟` | `> 30 次/分钟` |
| 单次 Young GC 停顿 | `< 10ms` | `10ms ~ 50ms` | `> 50ms` |
| CMS Old / Mixed 频率 | `>= 30 分钟/次` | `5 ~ 30 分钟/次` | `< 5 分钟/次` |
| 单次 Old / Mixed / Remark 停顿 | `< 100ms` | `100ms ~ 300ms` | `> 300ms` |
| Full GC 频率 | `0` | `偶发但可解释` | `>= 1 次/小时` |
| GC 后 Old 水位 | `< 60%` | `60% ~ 85%` | `> 85%` |
| GC 后 MetaSpace 水位 | `< 70%` | `70% ~ 85%` | `> 85%` |
| Eden 填满时间 | `>= 10 秒` | `3 ~ 10 秒` | `< 3 秒` |
| RT 的 `p99` 相对基线涨幅 | `< 20%` | `20% ~ 50%` | `> 50%` |

这张表最适合干什么？

- 先做一轮快速分诊

它不适合干什么？

- 直接替代根因分析

因为同样是 `Young GC 30 次/分钟`，可能是：

- 流量真的上来了
- 新生代太小
- 对象创建太碎
- 某个接口疯狂分配短命对象

数字只能告诉你“哪里不对劲”，不能自动告诉你“为什么不对劲”。

### 10. 给你一个最实用的判断顺序

以后你线上看 JVM，可以直接按下面这套顺序来：

1. 先看 1 分钟 GC 暂停占比有没有超过 `3%`
2. 再看单次停顿是否已经接近接口 TP99
3. 再看 Young GC 是不是已经到了 `每 1 到 2 秒一次`
4. 再看 Old / Mixed / CMS 是不是已经进入 `5 分钟内反复触发`
5. 再看 GC 后 Old 水位是不是长期高于 `75%`
6. 再看 Eden 填满时间是不是已经短到 `3 秒以内`
7. 最后判断是：
   - 对象分配过快
   - 晋升过快
   - 活对象本来就太多
   - 还是确实存在泄漏

这套顺序的好处是：

- 不会一上来就扎进参数细节
- 先把“症状级别”定清楚
- 再决定是看 GC、看堆，还是回到业务代码

### 11. 最后记住一句最重要的话

JVM 调优里很多词都很容易被说得很虚：

- GC 太频繁
- 停顿有点大
- 内存有点高
- RT 有点抖

真正能让你开始有工程判断力的，不是再多背几个参数，而是把这些词改写成：

- 1 分钟 GC 暂停占比是多少
- Young GC 到底多久一次
- 单次停顿和 TP99 的关系是什么
- GC 后 Old 水位到底是 `60%` 还是 `90%`
- Eden 到底是 `10 秒` 填满还是 `1 秒` 填满

一旦这些数字有了，你对 JVM 的判断就会开始从“感觉”变成“证据”。

## 小结（3-5 条关键点）

- 线上判断 JVM 问题，先把模糊词换成数字：暂停占比、单次停顿、GC 频率、GC 后水位、Young 填满时间。
- 吞吐低最先看 1 分钟窗口内 GC 暂停占比，超过 `3%` 就值得关注，超过 `5%` 往往已经不健康。
- RT 抖动不能脱离 SLA 看，单次 STW 一旦接近甚至超过接口 TP99，就很容易在监控上形成尖刺。
- GC 是否频繁要分 Young、Old/Mixed、Full 三层看，尤其要警惕 `Young GC < 2 秒/次`、`Old/Mixed < 5 分钟/次`、`Full GC >= 1 次/小时`。
- 内存是否危险要看 GC 后水位而不是 GC 前峰值，GC 后 Old 长期 `> 85%`、MetaSpace 长期 `> 85%` 都属于高风险区。
