# JVM - 第 10 课：堆内存相关 JVM 参数：Xms、Xmx、Xmn、NewRatio、SurvivorRatio 与 HeapDump

## 学习目标（本节结束后你能做到什么）

- 理解堆内存相关参数分别在控制什么，而不是只会机械背参数名。
- 说清 `-Xms`、`-Xmx` 为什么经常建议在生产环境设置成相同值。
- 说清 `-Xmn`、`-XX:NewRatio`、`-XX:SurvivorRatio` 分别在影响新生代的哪一部分。
- 知道如何配置 OOM 自动保留现场，例如自动生成 Heap Dump。
- 建立“参数是为问题服务，而不是为了炫技”的调参意识。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 学 JVM 参数最容易走偏的地方

很多人一学 JVM 参数，就会陷入两个误区：

- 误区一：觉得参数越多，说明自己越懂
- 误区二：看到线上问题，第一反应就是改参数

但参数本质上是“控制杆”，不是“答案本身”。

你要先知道自己想解决什么问题：

- 是堆太小导致 OOM？
- 是堆会动态扩缩导致抖动？
- 是新生代太小导致对象频繁晋升？
- 是 Survivor 太小导致存活对象挤进老年代？
- 还是你其实根本没参数问题，而是代码里有内存泄漏？

所以这节课最重要的不是背参数，而是建立一个判断：

**每个参数到底在控制哪块区域，它会带来什么副作用。**

### 2. `-Xms` 和 `-Xmx`：控制整个堆的下限和上限

#### `-Xms`

`-Xms` 表示 JVM 初始堆大小。

例如：

```bash
-Xms512m
```

表示 JVM 启动时，初始就申请 512MB 堆。

#### `-Xmx`

`-Xmx` 表示 JVM 最大堆大小。

例如：

```bash
-Xmx1g
```

表示整个堆最多可以长到 1GB。

#### 为什么很多生产环境建议 `-Xms` 和 `-Xmx` 设成一样

这是个非常经典的问题。

如果两者不一样，JVM 运行过程中可能会根据负载：

- 扩堆
- 缩堆

这个过程会带来额外开销，也可能导致性能抖动。  
尤其在服务端应用里，你通常更想要的是：

- 内存使用稳定
- GC 行为稳定
- 响应时间更可预期

所以很多稳定型服务都会直接设成：

```bash
-Xms4g -Xmx4g
```

这样做的核心目的不是“显得专业”，而是减少运行时动态调堆带来的不确定性。

当然，这个建议的前提是你得有足够机器内存。  
如果机器总共就 8GB，你却给 JVM 固定 6GB，再加上元空间、线程栈、直接内存、操作系统缓存，最后可能把整台机器压得很难受。

所以参数永远要结合机器资源一起看。

### 3. `-Xmn`：最直接控制新生代大小

`-Xmn` 是最直接控制新生代大小的方式。

例如：

```bash
-Xmn1g
```

表示给新生代直接分配 1GB。

这时候有一个直觉公式：

- 整个堆 = `-Xmx`
- 新生代 = `-Xmn`
- 老年代 = 整个堆 - 新生代

所以如果你配置：

```bash
-Xmx4g -Xmn1g
```

那么可以粗理解为：

- 新生代 1GB
- 老年代大约 3GB

什么时候你会认真关注 `-Xmn`？

- 你的业务短生命周期对象很多
- 请求一来会创建大量临时对象
- Minor GC 特别频繁
- 老年代晋升速度过快

这时候合理加大新生代，有时会明显改善 GC 行为。

但也不要机械地认为“新生代越大越好”。  
因为新生代变大后：

- Minor GC 次数可能下降
- 但单次 Young GC 扫描和复制的对象范围也可能变大

所以它本质上仍然是在做平衡，而不是一路加大。

### 4. `-XX:NewRatio`：另一种控制新生代的方法

`-XX:NewRatio` 也是用来控制新生代大小的，但它不是直接给具体值，而是给老年代和新生代的比例。

例如：

```bash
-XX:NewRatio=2
```

通常表示：

- 老年代 : 新生代 = 2 : 1

也就是说：

- 新生代约占整个堆的 1/3
- 老年代约占整个堆的 2/3

如果改成：

```bash
-XX:NewRatio=3
```

那么可以粗理解为：

- 老年代 : 新生代 = 3 : 1
- 新生代约占整个堆的 1/4

这就有个很实用的经验：

**通常在 `-Xmn` 和 `-XX:NewRatio` 之间，选择一种思路即可。**

因为：

- `-Xmn` 是直接定大小
- `-XX:NewRatio` 是间接定比例

如果你对业务对象生命周期比较有把握，往往用 `-Xmn` 更直接。  
如果你只是做大致比例控制，`NewRatio` 会更方便。

### 5. `-XX:SurvivorRatio`：控制 Eden 和 Survivor 的比例

新生代不是一整块平地，它内部通常还分成：

- Eden 区
- From Survivor
- To Survivor

`-XX:SurvivorRatio` 用来控制 Eden 和单个 Survivor 的比例。

例如：

```bash
-XX:SurvivorRatio=8
```

通常表示：

- Eden : From : To = 8 : 1 : 1

也就是说：

- Eden 占新生代的 8/10
- 两个 Survivor 各占 1/10

这个参数为什么重要？

因为对象不是一创建就直接进老年代。  
很多对象会先在新生代里经历几轮存活，如果 Survivor 太小，就可能放不下这些“还没死，但也没老到该晋升”的对象。

一旦放不下，就可能出现：

- 对象提前晋升到老年代
- 老年代压力过大
- Full GC 更早到来

所以 `SurvivorRatio` 调整的是：

**对象在新生代内部“缓冲”和“过渡”的空间结构。**

### 6. OOM 现场保留参数：比“事后猜原因”重要得多

线上最怕的事情之一是：

- 服务 OOM 了
- 但是没有现场
- 你只能靠日志猜

这时候最有价值的参数其实不是多高级的 GC 参数，而是下面两个：

#### `-XX:+HeapDumpOnOutOfMemoryError`

表示一旦发生 `OutOfMemoryError`，自动生成 Heap Dump 文件。

```bash
-XX:+HeapDumpOnOutOfMemoryError
```

#### `-XX:HeapDumpPath`

表示指定 Heap Dump 文件的保存路径。

```bash
-XX:HeapDumpPath=/logs/heapdump.hprof
```

这两个参数的重要性非常高，因为它们决定了：

- 你有没有内存事故现场
- 你能不能把问题交给 MAT、VisualVM 这类工具做深挖

一个很实用的生产环境配置习惯是：

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/app-oom.hprof
```

但这里有个现实问题：  
Heap Dump 文件可能非常大。

所以配置路径时一定要考虑：

- 磁盘空间够不够
- 目录是否有写权限
- 出问题时能不能及时把文件拿走分析

否则参数虽然配了，但 OOM 真来时写不下来，还是等于没有现场。

### 7. 一个实用的堆参数思考框架

面对堆参数，不要先问“网上推荐多少”，而要先问这几个问题：

#### 第一步：机器有多少内存

这一步是上限约束。  
JVM 不是机器上唯一吃内存的东西，还要留给：

- 元空间
- 线程栈
- 直接内存
- 操作系统页缓存
- 其他进程

#### 第二步：业务对象生命周期是什么样

如果短命对象特别多：

- 重点关注新生代和 Young GC

如果大量对象会长期存活：

- 更要关注老年代增长速度和泄漏风险

#### 第三步：服务最怕什么

- 怕 OOM：先保证现场可保留
- 怕抖动：先保证堆大小稳定
- 怕 Full GC：再考虑新生代和晋升节奏

### 8. 一个保守但很有用的经验

如果你是做后端服务，而不是专门研究 GC 调参，很多时候最有价值的第一步不是改很多参数，而是先把下面几件事做好：

```bash
-Xms4g
-Xmx4g
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/app-oom.hprof
```

这组参数的价值在于：

- 堆大小稳定
- 出事能留现场

在没有明确症状之前，不要过早去细抠 `-Xmn`、`NewRatio`、`SurvivorRatio`。  
因为很多线上问题最终并不是“参数没配好”，而是：

- 代码有泄漏
- 缓存没上限
- 大对象太多
- 批任务打爆堆

参数是放大镜，不是魔法。

## 小结（3-5 条关键点）

- `-Xms` 和 `-Xmx` 控制整个堆的初始值和最大值，生产环境常把它们设成相同值以减少抖动。
- `-Xmn` 直接控制新生代大小，`-XX:NewRatio` 通过比例间接控制新生代，通常选择一种方式即可。
- `-XX:SurvivorRatio` 控制 Eden 和 Survivor 的比例，会影响对象在新生代中的存活和晋升节奏。
- `-XX:+HeapDumpOnOutOfMemoryError` 和 `-XX:HeapDumpPath` 是非常关键的事故现场保留参数。
- 调 JVM 参数之前，必须先结合机器资源、对象生命周期和业务目标来判断，而不是盲目照搬配置。
