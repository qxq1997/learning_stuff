# JVM - 第 13 课：CMS 常见问题（上）：空间震荡、显式 GC 与 MetaSpace OOM

## 学习目标（本节结束后你能做到什么）

- 理解为什么 `-Xms` / `-Xmx` 不一致会引发空间震荡，以及这类问题为什么常被忽视。
- 说清 `System.gc()` 在 CMS 场景下为什么危险，以及为什么“简单禁掉”也不一定总是好主意。
- 理解 MetaSpace OOM 背后的真正根因通常不是“空间小”，而是“类和类加载器回不掉”。
- 能针对这三类问题分别说出：现象、原因、定位方法、处理策略。
- 建立“先判断机制，再动参数”的 CMS 诊断习惯。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 场景一：动态扩容引起的空间震荡

#### 1.1 现象长什么样

这类问题最常见的表现是：

- 服务刚启动时 GC 很多
- GC Cause 常见为 `Allocation Failure`
- 明明堆总剩余空间还很多，却还是在频繁做 GC
- GC 前后堆各分区大小还在变化

也就是说，问题不在“对象回收不掉”，而在：

**JVM 一边跑，一边还在做堆空间伸缩。**

#### 1.2 为什么会这样

如果你把：

```bash
-Xms
-Xmx
```

配置成不同的值，那么 JVM 启动时只会先按 `-Xms` 初始化。  
后面如果发现空间不够，就要：

- 触发一次 GC
- 看是否需要扩容
- 再向操作系统申请更多 committed 空间

反过来，如果空间长期空闲太多，JVM 也可能尝试缩容。

所以堆会出现一种“呼吸式变化”：

- 负载一上来，扩容
- 空闲一变大，缩容

这个过程本身就会带来额外 GC 和不确定性，这就是所谓的空间震荡。

#### 1.3 怎么定位

可以重点观察：

- GC 日志中每次回收前后，堆分区容量是否变化
- Old、Young、MetaSpace 的 committed 大小是否在波动
- 是否在启动期或流量初期特别明显

如果你发现：

- 使用率还不高
- 但是 committed / capacity 一直在变化

那大概率就是这类问题。

#### 1.4 怎么处理

最常见的做法非常直接：

- 尽量把成对出现的大小参数设成固定值

例如：

```bash
-Xms4g
-Xmx4g
```

类似地，MetaSpace 也有对应思路：

- `-XX:MetaspaceSize`
- `-XX:MaxMetaspaceSize`

这里的核心目标不是把内存“配得更大”，而是：

**让 JVM 的空间模型更稳定。**

#### 1.5 这类问题的本质

空间震荡看起来很初级，但生产里出现频率真的不低。  
因为它往往不是代码 bug，而是参数规范没统一。

这类问题特别值得优先修，因为：

- 改动成本低
- 收益稳定
- 能减少很多“无意义的 GC”

### 2. 场景二：显式 GC 的去与留

#### 2.1 现象长什么样

如果你发现：

- Old 区没有明显到阈值
- MetaSpace 也没有特别紧张
- 但日志里仍然出现了 GC
- `GC Cause` 明确写着 `System.gc()`

那就说明这不是 JVM 正常自适应触发，而是显式 GC。

#### 2.2 为什么显式 GC 在 CMS 下危险

在 CMS 场景里，`System.gc()` 最危险的地方在于：

**它可能触发一次非常重的 STW Full GC。**

而 CMS 的普通优势本来就是：

- 尽量并发
- 尽量压低停顿

如果显式 GC 把你直接拉回前台压缩式 Full GC，那停顿时间就可能非常难看。

所以如果代码里或某些库里频繁调用 `System.gc()`，对在线服务会非常不友好。

#### 2.3 那是不是直接禁掉就好了

很多文章会建议直接加：

```bash
-XX:+DisableExplicitGC
```

这当然能让显式 GC 失效，但问题是：

有些场景确实依赖显式 GC 来帮助触发某些清理逻辑，尤其是和 `DirectByteBuffer`、`Cleaner`、堆外内存释放相关的场景。

也就是说：

- 保留它，有 STW 风险
- 彻底禁掉它，又可能让某些堆外内存释放不及时

所以这个问题不能一句“关掉就完了”。

#### 2.4 更稳妥的工程处理方式

在 CMS 场景里，更常见的折中做法是：

```bash
-XX:+ExplicitGCInvokesConcurrent
```

或者再配合类卸载相关选项。

它的核心思路是：

- 不把显式 GC 当成前台重型 Full GC
- 尽量让它走并发收集路线

这样通常可以兼顾：

- 避免超长 STW
- 又不完全失去显式 GC 的清理能力

#### 2.5 处理显式 GC 的正确姿势

真正成熟的做法一般有两层：

1. 参数层面把显式 GC 的停顿伤害降到最低。
2. 代码规范层面尽量避免业务代码随意调用 `System.gc()`。

也就是说，参数是在兜底，规范才是治本。

### 3. 场景三：MetaSpace 区 OOM

#### 3.1 现象长什么样

这类问题的典型表现是：

- MetaSpace 使用量持续上涨
- 每次 GC 后也不怎么回落
- 即便把 `MaxMetaSpaceSize` 调大，也只是延缓，不是根治

很多人第一反应是：

- MetaSpace 配小了

但这通常不是根因，最多只是“报警阈值太小”。

#### 3.2 真正的根因是什么

MetaSpace 放的是：

- 类元数据
- 方法元数据
- 常量池相关结构

而这些元数据什么时候能回收，不是看“这个类还用不用”，而是更关键地看：

**这个类对应的类加载器是不是还活着。**

只要类加载器不死，这批类元数据就通常回不掉。

所以 MetaSpace OOM 最常见的深层原因不是普通对象泄漏，而是：

- 动态类加载太多
- 自定义类加载器未释放
- 动态代理 / 字节码增强不停生成新类
- 某些框架的 class map、ASM serializer、Groovy / CGLIB / Javassist 相关结构堆积

#### 3.3 为什么“调大 MetaSpace”治标不治本

如果根因是类加载器和动态类一直在增长，那么你把空间调大，只会得到：

- 更晚出事

不会得到：

- 问题消失

所以 MetaSpace 问题一定要把排查重心放在：

- 哪些类在不断增加
- 是哪个类加载器在不断 load 新类

#### 3.4 怎么定位

这类问题的定位手段通常比 Heap 问题更直接：

- 用 `jcmd <pid> GC.class_stats`
- 多次抓 Histogram 比较类数量变化
- 必要时打开 `TraceClassLoading` / `TraceClassUnLoading`

如果你连续看几次类直方图，发现某个包下的类数量一直涨，就很有机会锁定问题来源。

常见高发方向包括：

- Orika
- ASM 序列化
- CGLIB 动态代理
- Groovy 动态编译
- OSGi / 自定义类加载器体系

#### 3.5 怎么处理

处理 MetaSpace OOM 的核心不是“调参”，而是：

1. 找到不断增长的类来源
2. 找到背后的类加载器或动态生成逻辑
3. 修代码或限制生成策略

监控层面也应该加上：

- MetaSpace 使用率趋势
- 类加载数量趋势
- 类卸载数量趋势

因为这类问题通常不是瞬间炸，而是慢慢涨。

## 小结（3-5 条关键点）

- 空间震荡的本质是堆或元空间在运行时频繁伸缩，常见根因是成对参数没有固定。
- `System.gc()` 在 CMS 下可能触发非常重的 STW Full GC，但简单禁掉也可能带来堆外内存释放不及时的问题。
- 显式 GC 更稳妥的处理思路通常是尽量改为并发触发，而不是一刀切禁掉。
- MetaSpace OOM 的根因通常不是“空间不够大”，而是类和类加载器没有被及时释放。
- 面对这三类问题，都应该先理解触发机制，再决定是调参数、改框架配置还是修业务代码。
