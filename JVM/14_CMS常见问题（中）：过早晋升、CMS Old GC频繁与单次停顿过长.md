# JVM - 第 14 课：CMS 常见问题（中）：过早晋升、CMS Old GC 频繁与单次停顿过长

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 场景四：过早晋升

#### 1.1 什么叫过早晋升

分代回收的核心假设是：

- 大多数对象朝生夕死
- 少数对象活得久，才值得晋升到 Old 区

所谓“过早晋升”，就是：

**本来还应该在 Young 区多待一会、顺便被回收掉的对象，太早被送进了 Old 区。**

它本身不一定立刻炸，但它会导致：

- Young GC 更频繁
- Old 区涨得更快
- Full GC / CMS GC 更容易提前到来

所以它很像慢性病，不一定当场出大事，但很容易把系统往坏方向推。

#### 1.2 常见现象

这类问题常见信号有：

- 分配速率很接近晋升速率
- 对象晋升年龄偏小
- GC 日志里出现 `new threshold 1(max 6)` 这种信息
- Full GC 之后 Old 区掉下去很多，说明里面其实混了大量短命对象

如果一个 Old 区在 GC 前用了 80%，GC 后掉到 10% 或 20%，那就说明：

- 很多对象并不是真正“老”
- 它们只是太早被推进去了

#### 1.3 为什么会发生

最常见的根因有两个：

#### Young / Eden 太小

Young 区太小意味着：

- Eden 很快装满
- 对象来不及在年轻代完成自然死亡
- Survivor 也更容易顶不住

于是原本应该在 Young 区被收走的对象，只能更早进入 Old。

#### 分配速率突然过大

如果某一波流量、批任务或大查询把对象创建速率瞬间拉高，那么即便参数平时没问题，也可能在短时间内触发晋升压力。

#### 动态年龄计算

JVM 并不是死板地只看 `MaxTenuringThreshold`。  
如果 Survivor 装不下，JVM 会根据对象年龄分布动态调整晋升阈值。

这就是为什么你会在日志里看到：

```text
Desired survivor size ... new threshold 1(max 6)
```

它的潜台词是：

- Survivor 压力太大
- JVM 不得不把阈值压低
- 于是对象经历更少次数 GC 就进 Old

#### 1.4 怎么处理

这类问题最有效的思路往往不是先怀疑 CMS 本身，而是先怀疑分代比例是否合理。

可以先做这些动作：

- 重新评估 Young / Old 比例
- 看真实活跃对象到底有多少
- 确认 Old 区是不是配得明显过大而 Young 区被压缩得太小

一个常见经验是：

- 如果业务是典型 IO / RPC 型，大量对象都很短命
- 那 Young 区往往值得给得更大一点

但也不要极端地认为 Young 越大越好，因为：

- Young 大了，单次 Young GC 处理对象的体量也会变大

这始终是一个平衡问题。

### 2. 场景五：CMS Old GC 频繁

#### 2.1 现象长什么样

这类问题的特点通常是：

- CMS Old GC 很频繁
- 但单次未必特别慢
- 最大停顿还能勉强接受
- 只是 GC 总体太频繁，把吞吐拖下来了

换句话说，这类问题更像“慢性失血”，不是“一次暴毙”。

#### 2.2 为什么会频繁触发

CMS 后台线程会不停判断是否该开启下一轮并发回收。  
常见触发方向包括：

- Old 区占用达到阈值
- 元数据分配压力上来
- 预计下一次 Young GC 可能失败
- 显式 GC 请求
- 到了某个触发时间间隔

在工程上，最常见的仍然是：

- Old 区使用率太早逼近触发阈值
- 里面塞了很多本不该那么快进来的对象
- 或者本来就有一批中等寿命对象长期堆着

#### 2.3 这类问题怎么查

这类问题特别适合用“前后对比”的方式看：

- CMS GC 前 dump 一份
- CMS GC 后再 dump 一份
- 对比 Histogram、Leak Suspects、Top Component

重点不是只看“谁大”，而是看：

- 哪一类对象在持续累积
- 哪些集合、缓存、连接池一直持有引用
- 哪些对象理论上该过期了，但实际上没有掉下去

如果你只看一次快照，很多问题会模糊。  
但前后对比后，问题对象往往会非常明显。

#### 2.4 怎么处理

这类问题的处理，很多时候根本不是“换 GC”，而是：

- 找到那批中等寿命对象
- 判断它们是否符合预期
- 决定是代码问题、缓存问题，还是参数问题

一些高发源头包括：

- 带过期时间但上限不合理的缓存
- 长连接对象
- 某些框架内部滑动窗口或统计结构
- 业务上的批量加载数据未及时释放

所以 CMS Old GC 频繁，常常是“对象活太久”带来的，而不是纯收集器缺陷。

### 3. 场景六：单次 CMS 停顿过长

#### 3.1 现象长什么样

这类问题通常不是高频，而是低频但很致命：

- 平时看起来还好
- 偶尔来一次 STW 超长停顿
- 可能从几百毫秒一下跳到数秒

这种问题特别危险，因为它容易引发：

- RT 雪崩
- 线程积压
- 上游重试放大

#### 3.2 CMS 里最该怀疑哪两个阶段

在 CMS 的正常并发周期里，最值得重点关注的 STW 阶段通常是：

- `Initial Mark`
- `Final Remark`

其中真正更容易出大问题的，往往是 `Final Remark`。

为什么？

因为这里除了重扫对象图，还经常要处理：

- 各类 `Reference`
- 类卸载
- `SymbolTable`
- `StringTable`
- `CodeCache` 等元数据相关结构

所以它不只是“再标一次”，而是做了很多额外清理工作。

#### 3.3 哪些方向最容易拖长停顿

实际排查里，常见高发点是：

#### Reference 处理

比如：

- `FinalReference`
- `SoftReference`
- `WeakReference`

如果这部分对象链特别复杂，或者数量很大，Remark 阶段就会明显被拖慢。

#### 类卸载与元数据清理

如果系统存在：

- 动态类加载
- 大量 symbol / string table 处理
- 元数据变动频繁

那么 Remark 阶段也可能在类卸载上花很多时间。

#### 3.4 怎么定位

这类问题的第一步，不是先猜代码，而是先打开更细的日志：

- 观察 `Final Remark` 各子阶段耗时
- 看 `Reference` 处理是不是异常高
- 看 `Class Unloading`、`Scrub Symbol Table` 是否异常高

一旦方向确认了，再去：

- dump 快照
- 看 `Finalizer` 相关对象
- 看引用链
- 看类加载和元数据走势

#### 3.5 怎么处理

如果主要卡在 `Reference`，可以优先考虑：

- 找到具体对象来源
- 优化对象生命周期
- 必要时启用并行 Reference 处理

如果主要卡在类卸载和符号表清理，而你的系统又几乎没有动态类加载需求，可以再谨慎评估是否要减少类卸载相关开销。

但这里一定要强调：

**先确认系统确实不依赖动态类释放，再考虑收紧这类处理。**

否则你只是把问题藏起来，不是解决问题。

## 小结（3-5 条关键点）

- 过早晋升的本质是短命对象太早进入 Old 区，它常常是很多更严重 CMS 问题的前奏。
- CMS Old GC 频繁，很多时候不是收集器坏了，而是对象生命周期和分代结构不匹配。
- 单次 CMS 停顿很长时，通常要优先怀疑 `Final Remark` 阶段，而不是只看整体 CMS 时间。
- `Reference` 处理、类卸载、符号表清理是拖长 CMS Remark 停顿的几个高频方向。
- 处理这几类问题时，思路应该先从对象生命周期和引用链出发，再决定是否动参数。
