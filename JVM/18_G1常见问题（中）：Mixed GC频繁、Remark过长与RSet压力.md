# JVM - 第 18 课：G1 常见问题（中）：Mixed GC 频繁、Remark 过长与 RSet 压力

## 学习目标（本节结束后你能做到什么）

- 理解 G1 的 Mixed GC 为什么会频繁触发，以及为什么有时“收了很多次但收益不高”。
- 说清 `Pause Remark` 变长时，为什么要重点怀疑引用处理、SATB 收尾和类卸载。
- 理解 RSet / Dirty Card / Refinement 压力为什么会把 CPU 和停顿一起拖坏。

- 建立“先看回收收益，再看收尾停顿，再看维护开销”的诊断顺序。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 场景四：Mixed GC 频繁

#### 1.1 现象长什么样

这类问题的典型表现是：

- Young GC 之后很快进入 Mixed GC 阶段
- Mixed GC 来得很勤
- 但每轮 Mixed 后 Old 水位下降并不明显
- 整体停顿次数变多，吞吐开始下降

它很像 CMS 里的“Old GC 频繁但单次不一定特别长”。

#### 1.2 为什么会这样

Mixed GC 频繁，常见说明下面几件事里至少中了一条：

#### Old 区里可回收收益并不高

也就是说：

- Region 看上去该收
- 但真正收完后，腾出来的空间并不多

#### 中等寿命对象很多

这类对象最麻烦：

- 比短命对象活得久
- 比真正老对象又没那么稳定

于是它们会不断把 Old 顶高，又让 Mixed 回收效果不够漂亮。

#### 并发标记启动偏晚

如果标记启动太晚，Old 区会在比较高的水位才开始准备回收，后续就更容易进入：

- Mixed 一轮接一轮
- 但系统始终喘不过气

#### 1.3 怎么定位

可以重点看：

- Mixed GC 的连续次数
- 每次 Mixed 之后 Old 水位下降多少
- GC 后水位是不是依然长期高于 `75% ~ 85%`

如果你看到：

- 每轮都在收
- 但收完下降不明显

那就说明问题不在“G1 没收”，而在：

- 这批对象本身就活得久
- 或者回收集合的收益已经不高

#### 1.4 怎么处理

这类问题处理的重点通常不在“继续压榨 Mixed GC”，而在：

- 找到那些持续把 Old 顶高的对象
- 判断它们是不是符合预期
- 看是否有缓存、批量数据、连接对象长期滞留

如果确认不是代码问题，再考虑：

- 并发标记启动时机
- 堆大小是否偏紧

### 2. 场景五：Pause Remark 过长

#### 2.1 现象长什么样

这类问题通常不是高频，但很容易在监控上形成明显尖刺：

- 平时 Young / Mixed 看起来还可以
- 偶尔某次 `Pause Remark` 特别长
- 接口 RT 突然冒出明显尖峰

#### 2.2 为什么 Remark 会长

和 CMS 类似，G1 的 Remark 也不是简单“补几个标记”。

它通常还会处理：

- SATB 队列收尾
- 各类 `Reference`
- 类卸载相关工作
- 一些元数据收尾

所以它变长时，重点方向通常是：

- 引用处理多了
- 并发期间积压的 SATB 工作太多
- 类卸载和元数据清理偏重

#### 2.3 怎么定位

第一步先看日志细节：

- Remark 子阶段耗时
- `Reference` 处理耗时
- 类卸载相关耗时

如果日志已经明确显示：

- `Reference` 异常高

那就去看：

- `FinalReference`
- 相关对象链
- 是不是有大量生命周期异常的引用对象

如果更像类卸载方向，就去看：

- 动态类加载
- 元数据是否持续波动

#### 2.4 怎么处理

一般处理思路是：

- 找到对象或类的真实来源
- 先处理代码和生命周期问题

如果短期难以定位，再考虑：

- 并行 Reference 处理
- 评估类卸载相关开销是否确实必要

但原则还是一样：

**先定位对象来源和元数据来源，再决定要不要调参数。**

### 3. 场景六：RSet / Refinement 压力大

#### 3.1 现象长什么样

这类问题是 G1 很有代表性的“结构性成本”问题。

常见表现包括：

- CPU 明显偏高
- GC 停顿不一定特别夸张，但整体系统变重
- 日志里与 `Update RS`、`Scan RS`、Dirty Card 相关阶段耗时偏高
- 某些业务写引用特别频繁时更明显

#### 3.2 为什么会这样

G1 为了避免回收某批 Region 时全堆扫描，必须维护：

- 脏卡片
- RSet

问题在于，这些结构不是白来的。

如果应用本身有很多：

- 跨 Region 引用更新
- 大量容器对象写入
- 高频对象图改动

那 G1 在后台整理这些脏卡片时就会持续消耗 CPU。

如果后台处理跟不上，后续暂停阶段也会被拖累。

#### 3.3 怎么定位

可以重点观察：

- GC 日志里 `Update RS`、`Scan RS` 的时间
- CPU 使用率是否和对象引用更新高峰对齐
- 某些写多读少、对象图复杂的业务路径是否正好在异常窗口

如果发现：

- 不是对象特别多
- 而是引用维护成本特别高

那就要从对象结构和写入模式去想问题。

#### 3.4 怎么处理

常见方向包括：

- 优化对象图结构，减少过度复杂的跨对象引用
- 避免某些场景下超大 Map / List 被频繁写入
- 关注脏卡片处理线程和系统 CPU 资源是否匹配

这类问题特别说明一件事：

**G1 的低停顿不是免费的，它用 RSet 等额外结构换来了更小的扫描范围。**

## 小结

- Mixed GC 频繁时，不要只看“收得勤不勤”，更要看“收完之后 Old 水位到底降了多少”。
- Remark 过长往往不是单纯标记慢，而是 `Reference`、SATB 收尾、类卸载等附加工作太重。
- RSet / Refinement 压力是 G1 很典型的结构性成本问题，它可能同时拖高 CPU 和暂停。
- 这组问题的根因很多时候在对象生命周期和对象图结构，而不在 G1 算法“天生不行”。
- 真正排查顺序通常是：先看回收收益，再看 Remark，再看 RSet / Dirty Card 压力。
