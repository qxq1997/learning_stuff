## 1. GC 回收的是“不可达”，不是“业务上不用”

GC 的判断标准不是“这个订单上下文已经处理完了”，而是：

**从 GC Roots 出发，还能不能沿着引用找到这个对象。**

例如一次 HTTP 请求结束后，用户信息对象本应过期；但如果它还被线程池工作线程的 `ThreadLocalMap` 持有，它就仍然可达。对 GC 来说，它完全是一个合法存活对象。

```mermaid
flowchart LR
    R["GC Root: 工作线程"] --> T["Thread"]
    T --> M["ThreadLocalMap"]
    M --> E["Entry"]
    E --> V["已结束请求的大对象"]
```

这就是内存泄漏在托管语言中的准确含义：

- 对象不再有业务价值；
- 但仍存在不应存在的强引用链；
- 因而空间不能按预期回收；
- 累积后可能表现为老年代不断抬高、频繁 GC，最终 OOM。

两个分析指标尤其重要：

| 指标 | 含义 | 排障价值 |
| --- | --- | --- |
| Shallow Heap | 对象自身占用的空间 | 能看单个对象大小，但容易低估集合、容器的影响 |
| Retained Heap | 如果这个对象消失，能一并释放的对象总空间 | 适合寻找真正“扣住”大量内存的支配者 |

因此分析泄漏时，不应只问“哪个类的实例多”，还要问：

**是哪条从 GC Root 出发的路径，把这批对象长期保留下来？**

## 2. 四种引用强度解决的不是同一个问题

`java.lang.ref` 提供了不同强度的引用，让代码可以表达“这个对象还要不要阻止 GC 回收”的意图。

| 引用类型 | 典型写法 | 回收语义 | 适合场景 | 常见误区 |
| --- | --- | --- | --- | --- |
| 强引用 | `User u = new User()` | 只要强引用链可达，就不能回收 | 正常对象所有权、业务状态 | 以为调用 GC 就能清掉仍被持有的对象 |
| 软引用 | `new SoftReference<>(v)` | 内存压力较大时才可能清除 | 允许丢失、可重新生成的数据的有限辅助缓存 | 把它当成可预测容量、可控淘汰策略的业务缓存 |
| 弱引用 | `new WeakReference<>(v)` | 仅剩弱引用时，下一轮相关 GC 可清除 | 不应延长 key/元数据生命周期的关联关系 | 以为 value 也会随着弱 key 自动安全释放 |
| 虚引用 | `new PhantomReference<>(v, queue)` | 对象进入回收处理阶段后用于收到通知，不能通过 `get()` 取回对象 | 配合 `ReferenceQueue` 做清理跟踪，或由 `Cleaner` 封装清理动作 | 以为它可以复活对象，或自动替你释放外部资源 |

### 2.1 强引用：最容易造成“GC 怎么都收不掉”

普通字段、集合元素、静态变量、线程局部变量，本质上都可能构成长引用链。只要链路仍在，`System.gc()` 也不能把对象当垃圾回收。

这也是为什么“加大堆”不能修复泄漏：它只会延后强引用链填满内存的时间。

### 2.2 软引用：不是现代缓存治理的替代品

软引用表达的是“内存实在紧张时允许回收”，但回收时机受 JVM 和内存压力影响。业务缓存通常还需要：

- 明确容量上限；
- 淘汰策略；
- 过期策略；
- 命中率与装载成本监控。

因此，软引用不能替代 Caffeine、容量受控 Map 或远端缓存的治理能力。

### 2.3 弱引用：不延长被引用对象生命周期

弱引用很适合保存“附加关系”：对象一旦没有正常所有者，这份附加关系不应把它继续留在堆中。

但要特别留意容器结构。`ThreadLocalMap` 的 key 是弱引用，并不意味着 value 也是弱引用；弱 key 被清理后，value 仍可能通过 `Entry` 保持强可达。

### 2.4 虚引用与 ReferenceQueue：用于通知，不用于读取

虚引用必须和 `ReferenceQueue` 一起理解：

1. 创建对象和对应的虚引用。
2. 当对象达到相应回收阶段，引用会进入注册的队列。
3. 清理线程消费队列，释放与该对象相关的外部资源或记录生命周期事件。

关键边界是：

- `phantomReference.get()` 永远不能让你拿回目标对象；
- 队列只是通知机制，文件句柄、直接内存或其他 native 资源仍要由清理逻辑可靠释放；
- 明确的 `close()` / `try-with-resources` 仍优于等待 GC 驱动清理。

## 3. 长生命周期对象如何形成泄漏保留链

泄漏定位的关键不是背“高危对象名单”，而是识别两端：

- 根的一端：谁活得特别久，例如静态字段、线程池线程、类加载器、注册中心。
- 数据的一端：谁本来应该很快释放，例如请求上下文、批次结果、大字节数组。

中间那条不该存在的强引用路径，就是修复目标。

### 3.1 静态集合和无上限缓存

最直观的路径是：

```text
GC Root -> Class -> static Map/List -> value -> 大对象图
```

前面的 `11_线上故障实战` 已经用静态集合示例讲过 Heap Dump 定位。本节只补修复判断：

- 这是必需缓存，还是忘记移除的历史数据？
- 缓存是否有最大容量和过期策略？
- key 的维度是否爆炸，例如把请求 ID 当成永久 key？
- 被缓存的 value 是否意外引用了更大的对象图？

如果缓存确实需要存在，修复常常不是“换成弱引用”，而是给业务所有权加上边界：容量、TTL、主动失效和指标报警。

### 3.2 类加载器和元空间泄漏

类的元数据能否卸载，与定义它的类加载器能否不可达高度相关。热部署、插件系统、动态代理较多的服务中，旧类加载器可能被下面的对象留住：

- 线程上下文类加载器；
- 注册后没有反注册的驱动、监听器或回调；
- 静态集合中的插件实例；
- 框架创建但未释放的线程。

典型路径可以是：

```text
GC Root -> Thread -> contextClassLoader -> 旧 WebAppClassLoader -> 它加载的类与静态对象
```

此时单纯扩大 `MaxMetaspaceSize` 只是延迟故障。详细的元空间诊断已在 `13_CMS常见问题（上）：空间震荡、显式GC与MetaSpace OOM.md` 展开，这里记住判断原则即可：

**如果类加载数量和旧 ClassLoader 实例持续增长，先查保留链，再考虑调大空间。**

### 3.3 未关闭资源：不一定首先表现为 Java 堆 OOM

数据库连接、输入输出流、Socket、直接缓冲区等资源，可能同时占用：

- 堆中的包装对象；
- native 内存；
- 文件描述符或操作系统连接额度。

因此资源没有关闭时，现象可能是：

- 堆并不特别高，却报直接内存不足；
- 连接池耗尽；
- `too many open files`；
- 进程常驻内存持续上涨。

优先修复方式仍是明确生命周期管理：

```java
try (InputStream in = openStream()) {
    consume(in);
}
```

不要把释放外部资源寄托在对象何时被 GC 发现。

### 3.4 ThreadLocal：弱 key 为什么仍会保留 value

在线程池中，工作线程通常与线程池一样长寿。`ThreadLocalMap.Entry` 的关键关系是：

```text
GC Root -> Worker Thread -> ThreadLocalMap -> Entry -> value
                                      Entry --weak--> ThreadLocal key
```

如果业务代码不再持有 `ThreadLocal` 对象本身：

- key 因为是弱引用，可以在 GC 后变成 `null`；
- 但 `Entry` 对 value 仍是强引用；
- 只要长寿命工作线程还活着，value 就可能继续被挂住；
- Map 在后续部分操作中可能顺带清理过期 entry，但这不是请求生命周期的可靠清理协议。

更常见的业务写法是把 `ThreadLocal` 声明成 `static final`。此时 key 通常不会变成 `null`，但请求结束后未清理的 value 仍会随线程池工作线程存活，既占内存又可能被下一次任务误读。弱 key 只能缓解一类 key 生命周期问题，不能代替业务在边界处结束 value 的生命周期。

所以 `ThreadLocal` 的标准使用姿势不是期待 GC 善后，而是：

```java
private static final ThreadLocal<RequestContext> CONTEXT = new ThreadLocal<>();

void handle(RequestContext context) {
    CONTEXT.set(context);
    try {
        invokeBusinessLogic();
    } finally {
        CONTEXT.remove();
    }
}
```

这里 `remove()` 同时解决两个风险：

- 防止请求对象长时间被线程保留，形成内存滞留；
- 防止下一次复用同一个线程的请求读到上一请求上下文。

## 4. 用证据定位：从症状到保留链

| 现象 | 第一判断 | 首要证据 | 常见保留根 | 修复方向 |
| --- | --- | --- | --- | --- |
| Full GC 后堆使用量仍阶梯式上升 | 堆内对象被长期可达 | Heap Dump、MAT Dominator Tree、Path to GC Roots | 静态集合、缓存、线程本地值 | 移除引用、限界缓存、补 `remove()` |
| 元空间持续上涨且加载类数量增长 | 类加载器无法卸载 | 类加载统计、Heap Dump 中 ClassLoader 引用路径 | 线程、注册器、动态代理缓存 | 关闭旧容器、反注册、清理线程 |
| 堆正常但进程 RES / Direct Memory 升高 | 堆外或 native 资源问题 | Native Memory Tracking、直接缓冲区/资源统计 | native buffer、JNI、未关闭资源 | 释放资源、限制堆外配额 |
| 线程数持续上涨甚至 native thread OOM | 线程生命周期失控 | 线程快照、OS 线程数 | 未关闭线程池、无限建线程 | 池化、背压、关闭生命周期 |

### 4.1 现场收集的最短路径

服务仍可响应但怀疑堆泄漏时，可以按下面路径取证：

```bash
jcmd <pid> GC.class_histogram
jcmd <pid> GC.heap_dump /path/to/heap.hprof
```

分析 `.hprof` 时优先查看：

1. Leak Suspects 是否指出集中持有者。
2. Dominator Tree 中 retained heap 最大的对象。
3. 对可疑对象执行 Path to GC Roots，找到最短强引用链。
4. 区分“业务必需缓存”与“生命周期结束却没有移除”的对象。

怀疑元空间或 native 内存时，可进一步查看：

```bash
jcmd <pid> VM.classloader_stats
jcmd <pid> VM.native_memory summary
```

其中 Native Memory Tracking 需要进程启动时已开启相应能力，否则无法事后凭空得到完整分类现场。

生产环境还应尽量预置 OOM 快照参数：

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/heapdump.hprof
```

### 4.2 结论必须写到“是谁持有谁”

低质量结论是：

> 堆太大了，建议加内存或调 GC。

高质量结论应该长这样：

> 线程池工作线程通过 `ThreadLocalMap` 的 stale entry 持有 1.8 GB 请求缓冲区，路径为 `Thread -> threadLocals -> Entry.value -> byte[]`；请求出口未调用 `remove()`。修复为 `try/finally` 清理上下文并补充回归压测。

GC 排障真正有用的输出，不是一个模糊现象，而是一条可修复的引用路径。

## 5. 高频面试题答法

### 5.1 Java 有 GC，为什么仍会内存泄漏？

GC 只能回收不可达对象。业务已经不用的对象，如果仍被静态集合、长寿命线程、类加载器或监听器沿着强引用链持有，它在 JVM 看来仍然存活，因此会造成内存泄漏甚至 OOM。

### 5.2 ThreadLocal 的 key 是弱引用，为什么还可能泄漏？

`ThreadLocalMap.Entry` 对 key 使用弱引用，key 可以被 GC 清掉；但 value 仍由 entry 强引用，而 entry 又挂在长期存活的线程上。即便 key 作为 `static final` 一直存在，未清理的 value 也会随线程存活。如果线程池线程长期不退出且业务不调用 `remove()`，value 就可能持续滞留或串请求。标准做法是在 `finally` 中调用 `remove()`。

### 5.3 软引用能不能直接当缓存方案？

它只能表达“内存紧张时允许回收”，不能提供业务所需的精确容量、TTL、淘汰策略和命中率管理。生产缓存仍应显式设计边界，软引用最多作为特定场景下的辅助手段。

### 5.4 看到堆不断上涨，如何确认是不是泄漏？

先观察 Full GC 后基线是否持续抬升，再保留 Heap Dump，用 MAT 找 retained heap 大户并追到 GC Roots；只有找到不合理的持有路径，才可把增长判断为泄漏并提出针对性修复。

## 小结

- GC 判断的是可达性，因此“业务上没用了”不等于“可以被回收”。
- 强、软、弱、虚四种引用表达不同的生命周期意图，不能互相替代。
- 真正的泄漏修复依赖保留链：静态集合、类加载器、资源包装对象和线程池线程都是常见的长寿命根。
- `ThreadLocal` 的弱 key 不会替你管理 value 生命周期，在线程池中必须主动 `remove()`。
- 线上结论应落到 Heap Dump 中的 retained heap 和 Path to GC Roots，而不是只停留在“GC 频繁”或“内存大”的表象。
