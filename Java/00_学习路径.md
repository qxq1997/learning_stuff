# Java 学习路径与知识图谱

## 学习定位

Java 对后端工程师来说，不只是“一门会写业务代码的语言”。

它同时是一整套工程体系：

- 语言本身的语法、面向对象、泛型和集合
- 函数式风格里的 `lambda`、方法引用、`Optional`、`Stream`
- JVM 的运行时模型、类加载、内存分区和 GC
- 并发编程里的线程、锁、线程池、AQS、异步编排
- Spring 生态里的事务、AOP、IOC、Web、数据访问
- 线上问题里的内存溢出、线程阻塞、GC 抖动、吞吐和延迟

很多人学 Java 时也容易走两个极端：

- 只会写业务，能把功能做出来，但一遇到 JVM、并发、线上排障就发虚
- 只会背八股，知道很多名词，但一到真实代码、线程池、异步流程、OOM 定位就落不了地

这条学习路径的目标，是把“语言基础”“JVM 原理”“JUC 并发”“工程实践”“面试表达”连成一张图。学完之后，你不只是会写 Java，而是能解释 Java 程序为什么这样运行、为什么会这样出问题、为什么要这样设计。

## 为什么这样排序

这条路线不是按“语法手册”来排，而是按后端工程师最容易真正用上的顺序来排。

现有章节最早从虚拟机栈、`ThreadLocal`、异步任务模型、AQS、虚拟线程切入，是因为它们刚好位于 Java 学习里最容易“写业务时会碰到、面试时会被问到、线上时会踩坑到”的交叉点。补入基础章节与 JUC 素材后，首次系统学习可以先读第 12 课建立平台、类型与对象语义，再用第 7 课的 JMM/锁规则作为并发地基，继续读线程隔离、异步任务、同步器、线程池和虚拟线程。

你一旦把这几块串起来，很多东西会清楚很多：

- 为什么一个方法调用会压栈
- 为什么 Java 字节码可迁移而 JVM 与 native 依赖仍然受平台约束
- 为什么 Java 始终是值传递，为什么包装类型和对象相等性会制造业务坑
- 为什么加载类不等于初始化类，反射与注解如何连接到框架运行时
- 为什么网络 I/O 的并发模型与 JNI 边界会影响吞吐、部署和故障诊断
- 为什么递归太深会 `StackOverflowError`
- 为什么 `volatile` 能安全发布停止标记，却不能保证 `count++` 正确
- 为什么 monitor 竞争表现为 `BLOCKED`，而 `wait()` 通知后仍要重新竞争锁
- 为什么 `ThreadLocal` 在线程池里会串请求
- 为什么 `CompletableFuture` 比老 `Future` 更适合编排异步流程
- 为什么 `ReentrantLock`、`Semaphore`、`CountDownLatch` 看起来不一样，底层却都和 AQS 有关
- 为什么线程池通常应显式配置有界队列、拒绝策略和监控
- 为什么虚拟线程会让“高并发”和“阻塞式直观代码”重新变得更能兼得

后面再继续补 GC 和 Spring，就能把 Java 从“会写”推进到“会解释、会诊断、会设计”。

这次补进来的 `lambda`、`Optional`、`map`、`flatMap`，虽然表面上更像“语法和 API”，但它们其实和后面的并发、异步编排也能连起来：

- `lambda` 是 `Runnable`、`Callable`、`CompletableFuture` 这些 API 更自然的基础
- `Optional`、`map`、`flatMap` 是理解链式转换和结果编排的基础心智模型
- 你越早把这些抽象看顺，后面看 `Stream`、异步链路、函数式接口就越不容易乱

## 学习方法建议

- 每学一个知识点，都先回答：它在解决什么现实问题。
- 不要孤立背定义，要把它放回真实代码里理解，比如请求线程、线程池、数据库连接、异步任务。
- 遇到并发和 JVM 题，不只记术语，要明确“谁在管状态、谁在排队、谁在等待、谁在唤醒”。
- 学 `ThreadLocal`、AQS、`CompletableFuture` 时，尽量同时问自己“好处是什么”“坑在哪里”“线上会怎么出问题”。
- 对高频面试题，最好准备三层答案：
  - 一句话定义
  - 原理链路
  - 工程场景或常见坑

## 学完后你应该能回答的问题

- Java 虚拟机栈和本地方法栈分别服务于什么，为什么规范上要区分它们。
- 为什么 `lambda` 只是行为的简写，不等于异步或并行。
- `Optional` 为什么更适合作为返回值，而不是到处包字段和参数。
- `map` 和 `flatMap` 的本质区别是什么，为什么二者都能在 `Optional` 和 `Stream` 里出现。
- 为什么 `ThreadLocal` 不是做线程共享，而是做线程隔离。
- `Runnable`、`Callable`、`Future`、`CompletableFuture` 分别处在哪一层抽象上。
- 为什么 `Future` 不擅长复杂异步编排，而 `CompletableFuture` 更适合。
- AQS 到底抽象了什么，为什么很多并发同步器都能复用它。
- JMM 的 `happens-before` 如何把 `volatile`、monitor 和线程启动/结束规则连起来。
- `ThreadPoolExecutor` 为什么先入队再尝试扩到最大线程数，生产环境如何设计背压。
- 虚拟线程为什么适合高吞吐 I/O 型应用，它和线程池、`CompletableFuture`、Reactor 的关系是什么。
- 为什么大多数顺序表场景优先选 `ArrayList`，为什么 `HashMap` 的 key 必须满足稳定的 `equals/hashCode` 契约。
- `HashMap` 的桶、扩容与树化如何协作，`ConcurrentHashMap` 又如何让读取与冲突写入在并发下安全推进。
- `JDK`、Java 运行时与 `JVM` 的职责分别是什么，为什么不能机械套用旧版目录图。
- 受检异常与非受检异常的真实差别是什么，跨 JVM 传递对象时为什么要优先设计协议边界。
- BIO、NIO、虚拟线程与 Netty 各自解决哪类并发成本，JNI 又引入哪些风险。
- 线程池、锁、异步流程、JVM 内存问题在真实后端项目里会怎样联动。

## 推进规则

- 如果目标是后端工程与面试并重，建议按顺序学。
- 如果你第一次系统整理 Java 基础，先看第 12、8、11、9、13、14 课，再进入并发主线。
- 如果你当前最想补的是并发，优先按第 7、2、3、4、6、5 课的顺序看：先立内存语义，再读上下文、异步、同步器、线程池和虚拟线程。
- 如果你当前最想补的是 Java 语言层的函数式写法，优先看第 11 课，再回头看第 3 课里的异步编排。
- 如果你当前最想补的是 JVM，优先反复看第 12、1、9、15 课。
- 后续每课尽量沉淀成“原理 + 例子 + 误区 + 面试表达 + 自测题”的结构。
