# Java - 第 1 课：虚拟机栈、本地方法栈与栈帧基础

## 学习目标（本节结束后你能做到什么）

- 说清楚“栈”在 JVM 运行时语境里到底指什么。
- 理解虚拟机栈和本地方法栈分别服务于什么对象。
- 知道栈帧里通常会保存哪些内容，以及它和堆的区别。
- 理解为什么递归太深会触发 `StackOverflowError`。
- 能把程序计数器、虚拟机栈、本地方法栈、堆、方法区放在一张图里区分开。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先说“栈”在这里到底是什么

这里的“栈”，不是 `Stack` 集合，也不是算法题里的数据结构定义本身，而是线程执行方法时的一块工作区。

每当一个方法被调用时，JVM 会为这次调用创建一个对应的栈帧。方法执行完，栈帧就出栈。所以上层方法调用下层方法时，看起来就像一层一层往上叠抽屉：

- 先进入的方法先压栈
- 后结束的方法后出栈
- 谁最后调用进去，谁最先返回出来

这就是“栈”这个名字的直观来源。

你可以把它想成线程办公时的一摞临时文件夹：

- 每调用一个方法，就抽一层新文件夹出来
- 里面放这次方法运行需要的局部变量、参数、中间结果
- 方法执行完，这层文件夹收走

### 2. 栈帧里通常有什么

栈帧是方法运行时真正的工作单元。一个栈帧里通常会有这些东西：

- 局部变量表
- 操作数栈
- 方法参数
- 返回地址
- 一些用于支持方法调用和字节码执行的附加信息

这里最容易混的是“局部变量表”和“对象本身”。

比如这句代码：

```java
User user = new User();
```

更常见的理解是：

- `user` 这个引用变量在栈帧里
- `new User()` 创建出来的对象实体通常在堆里

所以不要把“栈里有对象”粗暴地理解成“对象实体都在栈里”。在大多数常规讨论里，更准确的说法是：

- 栈里更多放引用、局部变量、参数、执行过程信息
- 堆里更多放对象实体

### 3. 虚拟机栈是什么

虚拟机栈是 JVM 规范里定义的、专门给 Java 方法执行使用的栈。

比如你写了一个方法：

```java
public int add(int a, int b) {
    int c = a + b;
    return c;
}
```

当线程执行 `add()` 时，JVM 会在该线程的虚拟机栈中压入一个新的栈帧。这个栈帧会保存：

- 参数 `a`、`b`
- 局部变量 `c`
- 操作数栈中的中间计算值
- 方法返回所需的信息

所以虚拟机栈本质上可以理解为：

**Java 方法调用时，每个线程自己的方法调用记录区。**

### 4. 虚拟机栈的几个关键特点

#### 4.1 线程私有

每个线程都有自己独立的虚拟机栈。

- 线程 A 调方法，压的是线程 A 自己的栈
- 线程 B 调方法，压的是线程 B 自己的栈

线程之间不会把方法调用记录混在一起。

#### 4.2 生命周期和线程一致

线程创建时，它对应的虚拟机栈也会建立；线程结束时，对应的虚拟机栈也会销毁。

这和堆不一样。堆通常是进程级共享区域，而虚拟机栈更像每个线程自己随身带的工作台。

#### 4.3 典型异常和它有关

虚拟机栈最典型会牵扯到两类异常：

- `StackOverflowError`
- `OutOfMemoryError`

但两者成因不同，不能混着背。

### 5. 本地方法栈是什么

本地方法栈是给本地方法，也就是 `native` 方法使用的。

所谓本地方法，指的不是用 Java 字节码直接实现的逻辑，而是通过 JNI 等机制调用的 C、C++ 或其他本地语言实现的方法。

比如：

```java
public native void someNativeMethod();
```

这个方法的方法体可能根本不在 Java 代码里，而是在本地动态库中实现。

当 JVM 调用这种 native 方法时，就会涉及本地方法栈。你可以把它理解成：

**JVM 在进入本地代码执行环境时使用的调用工作区。**

### 6. 虚拟机栈和本地方法栈最本质的区别

#### 6.1 服务对象不同

这是第一核心区别：

- 虚拟机栈服务于 Java 方法
- 本地方法栈服务于 native 方法

这句话已经足够作为一轮面试里的第一句回答。

#### 6.2 栈帧内容的语义来源不同

虚拟机栈中的栈帧，主要对应 JVM 规范下 Java 方法调用时需要的信息。

本地方法栈中的内容，则更偏底层本地语言运行时所需的调用信息，具体实现会更依赖 JVM 产品本身和底层平台。

也就是说：

- 虚拟机栈更“Java 规范化”
- 本地方法栈更“贴近本地实现和操作系统”

#### 6.3 规范严格程度不同

JVM 规范对虚拟机栈的语义描述更明确。

而本地方法栈虽然在概念上被单独划分出来，但很多实现细节留给不同的 JVM 去决定。所以在实际实现里，像 HotSpot 这类 JVM 并不一定非常机械地把两者彻底物理拆开。

这个点在面试里说出来会比较像真正理解过，而不是只死记“它们有两个名字”。

### 7. 为什么要区分这两个栈

因为 JVM 不只是运行纯 Java 代码。

Java 程序在很多底层场景中都可能绕到 native 层，例如：

- 调操作系统能力
- 调 C/C++ 库
- 调 JVM 自己底层的一些实现
- 调某些性能敏感的本地组件

于是 JVM 要同时面对两套执行环境：

- Java 世界：按 Java 字节码规则执行
- Native 世界：按本地方法规则执行

所以在概念上，需要区分两条调用车道：

- 一条给 Java 方法
- 一条给 native 方法

### 8. 结合一个简单例子看调用过程

```java
public class Demo {
    public static void main(String[] args) {
        foo();
    }

    public static void foo() {
        bar();
    }

    public static void bar() {
        System.out.println("hello");
    }
}
```

这里你自己写的：

- `main()`
- `foo()`
- `bar()`

这些 Java 方法的调用，主要都会反映在虚拟机栈里。执行时大致会形成这样的压栈顺序：

- `main` 栈帧
- `foo` 栈帧
- `bar` 栈帧

然后：

- `bar()` 执行完出栈
- 回到 `foo()`
- `foo()` 执行完出栈
- 回到 `main()`

而像 `System.out.println()` 再更底层的某些实现，可能会进一步涉及 native 层逻辑，这时就可能和本地方法栈有关系。

这也是为什么你可以这样粗略理解：

- 自己写的普通 Java 方法，主要走虚拟机栈
- JVM 进入本地实现层时，可能会涉及本地方法栈

### 9. `StackOverflowError` 为什么和栈有关

这是一道非常高频的基础题。

比如下面的代码：

```java
public void f() {
    f();
}
```

每调用一次 `f()`，就会压入一个新的栈帧。如果递归一直不结束，线程的栈会不断变深，最终超过允许的深度限制，就可能抛出：

```java
java.lang.StackOverflowError
```

所以更准确地说，`StackOverflowError` 常见于：

- 栈深度无限增长
- 方法一直压栈但不返回

你平时在 Java 业务代码里最常见的，就是递归写错导致虚拟机栈溢出。

理论上 native 层如果调用层级极深，也可能引发类似问题，但日常后端开发里最常见的还是 Java 方法递归过深。

### 10. 栈也会 OOM 吗

会，但别和 `StackOverflowError` 混为一谈。

如果某种 JVM 实现支持栈动态扩展，而扩展时又申请不到足够内存，就可能抛出 `OutOfMemoryError`。

所以两者的区别要记住：

- `StackOverflowError`：更偏“栈深度太深”
- `OutOfMemoryError`：更偏“申请内存失败”

在实际线上里，大家更常见的 OOM 还是堆内存不够，而不是栈内存不够。但从规范上说，栈相关区域也可能触发 OOM。

### 11. 它和程序计数器、堆、方法区怎么区分

这块最好顺手一起记，不然你会把 JVM 运行时数据区混在一起。

#### 11.1 程序计数器

它更像“当前线程下一条要执行到哪”的位置指针。

#### 11.2 虚拟机栈

Java 方法调用时的工作空间。

#### 11.3 本地方法栈

native 方法调用时的工作空间。

#### 11.4 堆

对象实例主要分配在这里。

#### 11.5 方法区

主要存类信息、常量、静态变量、JIT 编译后的代码缓存等。

你可以把这五个区域看成一张简单分工图：

- 程序计数器：记录执行位置
- 栈：记录方法执行现场
- 堆：放对象
- 方法区：放类级元数据

### 12. 面试里怎么回答更像样

可以这样组织：

Java 虚拟机栈是线程私有的，主要服务于 Java 方法执行。每次方法调用时，都会创建一个栈帧，里面保存局部变量表、操作数栈、方法返回地址等信息。  
本地方法栈同样是线程私有的，但它服务于 native 方法，也就是 JVM 调用的本地代码。  
两者核心区别在于服务对象不同：一个对应 Java 方法，一个对应本地方法。  
从规范上它们是分开的，但在具体 JVM 实现里不一定严格物理分离。  
虚拟机栈过深时，常见会导致 `StackOverflowError`。

这个版本已经足够应对大多数基础面试场景。

## 小结

- 这里的“栈”是线程执行方法时的工作区，不是普通集合类型。
- 虚拟机栈服务于 Java 方法，本地方法栈服务于 native 方法。
- 栈帧里通常保存局部变量、操作数栈、返回地址等执行现场信息。
- 栈里常放引用和局部变量，对象实体通常在堆里。
- 递归太深最常见会导致虚拟机栈相关的 `StackOverflowError`。
