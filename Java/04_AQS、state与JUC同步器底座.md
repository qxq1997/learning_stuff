# Java - 第 4 课：AQS、state 与 JUC 同步器底座

## 学习目标（本节结束后你能做到什么）

- 用大白话解释 AQS 到底在解决什么问题。
- 知道 AQS 的两个核心组成：`state` 和等待队列。
- 理解独占模式和共享模式的区别。
- 说清楚 AQS 和 `ReentrantLock`、`Semaphore`、`CountDownLatch` 的关系。
- 明白为什么 AQS 被称为很多并发同步器的“通用底座”。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. AQS 到底是什么

AQS 全名是 `AbstractQueuedSynchronizer`，翻成大白话就是：

**抽象的队列同步器。**

这个名字虽然看起来很唬人，但它解决的问题其实很朴素：

很多并发工具都需要完成这些事情：

- 看看资源现在有没有被占用
- 多个线程来抢资源时怎么保证安全
- 抢不到的线程放到哪里等
- 什么时候唤醒下一个线程
- 怎么支持独占模式和共享模式

如果每一种锁、每一种同步器都从零开始写一遍，会非常痛苦。

于是 AQS 的思路就是：

**把排队、阻塞、唤醒、状态管理这些通用脏活累活统一封装起来。**

至于“你这个同步器到底什么叫获取成功，什么叫释放成功”，交给子类去定义。

所以它叫 `Abstract`，因为它不是一个能直接拿来用的锁，而是一个框架底座。

### 2. AQS 最核心的两个东西

如果你只记两个点，就记下面这两个：

#### 2.1 一个 `state`

```java
private volatile int state;
```

这个字段表示同步状态。

但这个状态具体代表什么，要看子类在实现什么同步器。

例如：

- 在 `ReentrantLock` 里，`state` 可以表示锁被重入了多少次
- 在 `Semaphore` 里，`state` 可以表示还剩多少许可证
- 在 `CountDownLatch` 里，`state` 可以表示还剩多少计数

所以你能看出来：

**AQS 不关心这个数字业务上到底是什么意思，它只是提供一个通用、可原子操作的同步状态位。**

#### 2.2 一个 FIFO 等待队列

如果线程没抢到资源怎么办？

AQS 的思路不是让线程一直死循环空转，而是：

- 把线程包装成一个节点
- 加入等待队列
- 挂起等待
- 合适的时候再唤醒

你可以把它理解成银行排队：

- 窗口空着，直接办理
- 窗口被占，后面的人排队
- 前一个人办完，叫下一个

这就是 AQS 名字里 `Queued` 的意义。

### 3. AQS “抽象”在哪里

AQS 把通用流程写好了，但并不替你决定“获取资源成功的业务规则”。

所以它把一些关键判断交给子类实现，比如：

- `tryAcquire(int arg)`
- `tryRelease(int arg)`
- `tryAcquireShared(int arg)`
- `tryReleaseShared(int arg)`
- `isHeldExclusively()`

你可以这样理解分工：

#### AQS 负责

- 线程竞争时的排队
- 队列维护
- CAS 改状态
- 线程阻塞和唤醒

#### 子类负责

- 什么情况下算拿到资源
- 什么情况下算释放资源
- 是独占语义还是共享语义

这其实就是很典型的模板方法模式。

### 4. 独占模式和共享模式

这是理解 AQS 的另一个关键维度。

#### 4.1 独占模式

同一时刻只能有一个线程拿到资源。

典型例子：

- `ReentrantLock`

如果线程 A 拿到了锁，线程 B 只能排队。

#### 4.2 共享模式

同一时刻可以有多个线程一起拿到资源。

典型例子：

- `Semaphore`
- `CountDownLatch`
- `ReentrantReadWriteLock` 中的读锁

例如 `Semaphore(3)` 表示最多允许 3 个线程同时通过。前 3 个线程拿到许可证继续走，后面的线程再排队。

所以 AQS 并不只是“锁框架”，它其实是很多同步器的统一底盘。

### 5. 以独占锁为例看一遍运行流程

我们先拿最容易理解的“锁”场景来讲。

#### 5.1 第一阶段：先尝试直接获取

线程不会一上来就排队，而是先试一次快速获取。

例如：

- 如果 `state == 0`
- 通过 CAS 把它改成 1
- 成功了就表示拿到锁

这叫快速路径。因为如果本来没人占用资源，就没必要排队。

#### 5.2 第二阶段：获取失败就入队

如果资源已经被别人占了，当前线程就会：

- 被包装成一个 `Node`
- 放到等待队列尾部

#### 5.3 第三阶段：阻塞等待

线程入队后，不会一直疯狂抢，而是会在合适的时候：

- 判断自己是否轮到尝试获取
- 不该自己出手时就先 `park()`

AQS 底层通常通过 `LockSupport.park()` 来挂起线程，这比死循环自旋省 CPU 得多。

#### 5.4 第四阶段：前驱释放后被唤醒

当前面的线程释放资源后，AQS 会去唤醒后继节点。

被唤醒的线程再去尝试获取资源：

- 如果成功，就继续执行
- 并把自己变成新的头结点

这就是“获取失败入队、释放后唤醒后继”的基本链路。

### 6. 为什么一定要搞队列

并发竞争里最怕两种极端：

#### 6.1 所有线程一直自旋

这样 CPU 会被打爆。

#### 6.2 完全没有秩序，大家乱抢

这样会导致竞争混乱、调度混乱，还可能造成不公平和性能抖动。

所以队列的价值就在于：

**让竞争失败的线程有秩序地等待，并在合适的时候被唤醒。**

### 7. 队列里的 `Node` 是什么

AQS 队列里存的不是线程裸对象，而是节点 `Node`。

一个 `Node` 大致会携带这些信息：

- 当前线程是谁
- 前驱节点是谁
- 后继节点是谁
- 当前节点处于独占模式还是共享模式
- 当前等待状态是什么

你不需要背源码字段，但要有一个整体印象：

**AQS 的等待队列本质上是一个双向链表，链表节点里包着线程。**

### 8. 为什么 `state` 要用 `volatile + CAS`

因为多个线程会同时竞争修改 `state`。

如果没有原子性保证，就可能出现灾难，例如两个线程都以为自己拿到了锁。

所以 AQS 的核心套路是：

- `volatile` 保证可见性
- CAS 保证原子更新

比如多个线程同时看到 `state == 0`，但最终只能有一个线程 CAS 成功把它改成 1。

这就是为什么可以说：

**AQS 的竞争核心，本质上是围绕 `state` 做的 CAS。**

### 9. 为什么 AQS 这么通用

因为很多同步器最终都能抽象成同一种模式：

1. 有一个共享状态
2. 获取失败的线程要排队等待

这刚好就是 AQS 最擅长封装的部分：

- 管状态
- 管队列
- 管阻塞
- 管唤醒

所以它才能同时支撑：

- `ReentrantLock`
- `Semaphore`
- `CountDownLatch`
- `ReentrantReadWriteLock`

### 10. 拿 `ReentrantLock` 感受一下 AQS 怎么被使用

#### 10.1 加锁时

`lock()` 最终会触发 AQS 的获取流程。

在子类实现的 `tryAcquire()` 里，大致逻辑会是：

- 如果 `state == 0`，尝试 CAS 抢锁
- 如果当前线程本来就持有锁，则允许重入，`state++`
- 否则获取失败，进入 AQS 队列等待

#### 10.2 释放时

`unlock()` 会走 `tryRelease()`：

- `state--`
- 如果减到 0，表示锁彻底释放
- 然后 AQS 去唤醒后继线程

所以你可以看到，AQS 和具体锁之间的分层非常清楚：

- “什么叫拿到锁” 是子类定
- “拿不到时怎么排队” 是 AQS 管

### 11. 再看 `CountDownLatch`，更能感受到抽象能力

`CountDownLatch` 不是锁，但照样可以用 AQS。

它的逻辑是：

- 初始 `state = count`
- 每次 `countDown()` 就把 `state--`
- 当 `state == 0` 时，唤醒等待线程
- `await()` 时如果 `state != 0`，线程就进入等待

你会发现：

- 它和锁的语义完全不同
- 但底层仍然能复用 AQS 的状态管理和队列等待机制

这正说明 AQS 抽象得很巧。

### 12. AQS 和 `synchronized` 有什么关系

这也是一道高频对比题。

#### 12.1 `synchronized`

它是 JVM 层面的内置锁机制。

#### 12.2 AQS

它是 JDK 层面用 Java 代码实现的一套同步器框架。

所以：

- `synchronized` 不是基于 AQS 实现的
- `ReentrantLock` 这类显式锁通常是基于 AQS 实现的

你可以把它理解成：

- `synchronized` 更像系统自带门锁
- AQS 更像你自己搭一套门禁系统的底盘

### 13. 公平和非公平怎么理解

虽然 AQS 有 FIFO 等待队列，但这不代表所有实现都绝对公平。

比如非公平锁里可能会出现：

- 队列里已经有人排着
- 新来的线程却因为时机刚好，直接 CAS 抢到了锁

这叫插队。

为什么允许这样做？

因为完全公平通常吞吐量更低，很多场景下适度不公平反而性能更好。

所以你在面试里最好这样说：

- AQS 支持公平和非公平策略
- 具体实现会在吞吐与公平之间做权衡

### 14. AQS 和 `Condition` 的关系

`Condition` 可以理解成：

在锁内部再提供一套等待/通知机制。

比如 `ReentrantLock.newCondition()` 背后就和 AQS 高度相关。线程可以：

- 先拿到锁
- 调 `await()` 进入条件等待
- 另一个线程 `signal()` 后，再让它回到同步队列竞争锁

这一块比较进阶，你当前阶段先知道：

**AQS 不只管同步队列，也常和条件队列协同工作。**

### 15. AQS 的设计哲学，用四个词记住

可以把 AQS 压成四个关键词：

- 模板方法模式
- CAS
- FIFO 等待队列
- `park/unpark`

拆开看就是：

- 模板方法模式：框架写好通用流程，子类补充获取/释放规则
- CAS：安全地更新 `state`
- FIFO 队列：让获取失败的线程有序等待
- `park/unpark`：让线程挂起和唤醒

这四个词，基本就是 AQS 的灵魂。

### 16. 面试里怎么回答更像样

你可以这样答：

AQS 是 `AbstractQueuedSynchronizer`，是 Java 并发包中很多同步器的基础框架，比如 `ReentrantLock`、`Semaphore`、`CountDownLatch` 等都基于它实现。  
它的核心思想是用一个 `volatile` 的 `state` 表示同步状态，用一个 FIFO 双向队列维护获取同步状态失败的线程。  
子类只需要实现 `tryAcquire`、`tryRelease`、`tryAcquireShared`、`tryReleaseShared` 等方法，定义如何获取和释放资源；而线程排队、阻塞、唤醒等通用逻辑由 AQS 统一处理。  
AQS 支持独占模式和共享模式。它底层大量使用 CAS 保证 `state` 更新的原子性，并通过 `LockSupport` 的 `park/unpark` 实现线程挂起和唤醒。

## 小结

- AQS 是很多 JUC 同步器的通用底座，不是一个直接拿来用的锁。
- 它的两个核心组成是 `state` 和等待队列。
- 子类定义“资源怎么拿、怎么放”，AQS 负责“线程怎么排、怎么等、怎么醒”。
- AQS 支持独占模式和共享模式，所以既能做锁，也能做信号量和倒计时器。
- `synchronized` 是 JVM 内置锁，`ReentrantLock` 这类显式同步器通常基于 AQS。
