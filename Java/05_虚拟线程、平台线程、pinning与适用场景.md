# Java - 第 5 课：虚拟线程、平台线程、pinning 与适用场景

## 学习目标（本节结束后你能做到什么）

- 用一句话解释什么是 Java 虚拟线程。
- 说清虚拟线程和平台线程最本质的区别。
- 理解虚拟线程为什么适合高并发 I/O 型应用，而不是 CPU 密集计算神器。
- 知道虚拟线程和线程池、`Future`、`CompletableFuture` 分别是什么关系。
- 对 `ThreadLocal`、`pinning`、阻塞式写法回归这些工程问题有基本认知。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先用一句话记住什么是虚拟线程

Java 虚拟线程可以先粗暴理解成：

**JDK 提供的一种非常轻量的线程。写法上仍然是 `Thread`，但底层不再长期一对一绑定某个固定的操作系统线程。**

这句话里最关键的是“写法还是线程”“底层不再长期绑死 OS 线程”。

它不是一种全新的语法，也不是一种和 `Thread` 平行的陌生接口。它仍然是 `java.lang.Thread` 的实例，只不过背后的实现模型升级了。

### 2. 它和普通线程最大的区别是什么

现在可以把 Java 线程先分成两类：

#### 2.1 平台线程

平台线程可以理解成传统 Java 线程。它本质上更接近对操作系统线程的一层包装。

通常你可以粗略理解为：

- 一个 Java 平台线程
- 长期对应一个 OS 线程
- 它在运行、等待、阻塞时，底下那个 OS 线程也基本跟着它走

#### 2.2 虚拟线程

虚拟线程也是 `Thread`，但它不长期绑定某一个固定的 OS 线程。

当虚拟线程执行到某些阻塞点时，JDK 可以把这个虚拟线程挂起，等它将来该继续执行时再恢复；而底下那个平台线程在这段时间可以去服务别的虚拟线程。

所以你可以用一个非常直观的类比：

- 平台线程像“一个人长期占一张固定工位”
- 虚拟线程像“很多人共享一批工位，谁需要 CPU 谁上来干一会儿，等 I/O 时先让出工位”

这就是虚拟线程最核心的变化。

### 3. 它真正解决的是什么问题

过去 Java 服务端一直有一个老矛盾：

- 线程编程模型很直观，代码写起来像同步逻辑，理解和调试都比较自然
- 但传统平台线程很贵，线程一多就会带来内存开销、调度开销、上下文切换开销

于是工程上大家才会被迫搞很多复杂手段：

- 线程池
- 回调
- Reactor
- 链式异步编排
- 各种“别阻塞线程”的设计

虚拟线程想解决的痛点其实特别朴素：

**线程这个编程模型很好，但传统线程太贵，那就把线程做便宜。**

所以虚拟线程最大的工程价值不是“单个任务执行得更快”，而是：

**让你可以继续用很直观的“一请求一线程”“同步阻塞式逻辑”来写高并发 I/O 应用，而不用过早为了省线程成本把代码扭成很复杂的异步形状。**

### 4. 为什么说它“轻”

很多人听到“轻量线程”会下意识觉得是不是做了什么魔法。

其实它轻的本质并不神秘，核心在于：

- 虚拟线程对象本身更便宜
- 它不长期占着某个固定 OS 线程
- 当它阻塞等待时，底下的平台线程可以被拿去服务别的虚拟线程
- JDK 运行时帮你做了挂起、恢复和调度复用

也就是说，虚拟线程轻，不是因为“阻塞消失了”，而是因为：

**阻塞等待不再等价于长期浪费一个宝贵的平台线程。**

### 5. 它最适合什么场景

虚拟线程最适合的是这种任务：

- 并发量很大
- 每个请求逻辑本身不算特别重
- 但中间有很多等待

例如：

- 调数据库
- 调 HTTP / RPC
- 网络读写
- 文件 I/O

这类任务的特点不是“CPU 忙到爆”，而是“有很多时间在等外部资源返回”。过去如果你给每个请求配一个平台线程，会很快觉得线程数量太贵；但虚拟线程正好擅长解决“高并发等待很多”的问题。

所以它特别适合：

- 高吞吐 Web 服务
- 网关
- RPC 调用链比较多的后端服务
- 大量外部 I/O 等待的业务逻辑

### 6. 它不是什么

这块一定要讲清，不然很容易神化。

#### 6.1 它不是 CPU 密集计算神器

如果你的任务是：

- 大量加密
- 图像处理
- 复杂数学计算
- 纯 CPU 大循环

那瓶颈在 CPU 本身，而不是“线程太贵”。

虚拟线程并不会神奇地把 CPU 变多，也不会让纯计算自动并行得更快。对于 CPU 密集型问题，真正决定性能的还是：

- 核心数
- 调度
- 算法本身
- 任务拆分方式

所以不要把虚拟线程理解成“只要换上就能无脑提速”。

#### 6.2 它也不是异步框架的简单废除令

虚拟线程最大的价值是让很多以前“为了不阻塞平台线程”才不得不写得很绕的代码，现在可以重新考虑回到更直观的同步风格。

但这不等于：

- `CompletableFuture` 没用了
- Reactor 没用了
- 所有异步编排都没价值了

它降低的是并发代码的复杂度，不是所有性能和架构问题都能靠它一招解决。

### 7. 最简单的写法长什么样

在现代 Java 中，虚拟线程最直观的写法就是：

```java
Thread.startVirtualThread(() -> {
    System.out.println("hello from virtual thread");
});
```

你会发现这件事很重要：

- 你仍然是在启动线程
- 代码风格仍然非常自然
- 并没有被迫改成很奇怪的回调模型

另一种常见写法是配合 executor：

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> future = executor.submit(() -> "done");
    System.out.println(future.get());
}
```

这个模式背后的思想很清楚：

**一个任务一个虚拟线程。**

### 8. 它和线程池是什么关系

这个点特别容易误解。

在传统平台线程时代，线程池非常重要，因为平台线程昂贵。我们通常需要：

- 限制线程数
- 复用线程
- 避免线程创建和销毁成本过高

所以会强调：

- 线程池大小怎么配
- 任务怎么排队
- 拒绝策略怎么选

但到了虚拟线程世界，思路发生了很大变化。

因为虚拟线程本身已经足够便宜，所以很多时候你不再需要为了节省线程而强行做“少量任务线程复用”。更常见的思路会变成：

- 一个任务
- 一个虚拟线程

也就是 `newVirtualThreadPerTaskExecutor()` 这种模式。

这并不是说线程池不重要了，而是说“池化线程来节省线程”这件事的重要性下降了。以后更重要的限制往往不是线程本身，而是：

- 数据库连接池
- 下游服务承载能力
- 外部接口限流
- CPU 和内存

### 9. 它和 `ThreadLocal` 是什么关系

很多人会问：用了虚拟线程以后，`ThreadLocal` 还能不能用？

先记结论：

**能用。**

从工程角度看，这很重要，因为很多老代码和框架都大量依赖 `ThreadLocal`。

但你也要有边界感：

- 当虚拟线程数量非常多时，`ThreadLocal` 的内存开销和管理复杂度会更敏感
- 以前线程数少时不明显的问题，在线程规模上来后会变得更值得关注

所以现实建议通常是：

- 老代码很多可以继续用 `ThreadLocal`
- 但如果以后大量使用虚拟线程，就要对 `ThreadLocal` 的生命周期、上下文规模、继承和清理更敏感

也正因为这个方向，Java 后续才会继续推动像 `Scoped Values` 这类更适合大规模线程上下文传播的机制。

### 10. 一个重要坑：pinning

虚拟线程虽然轻，但不是任何阻塞场景都能完美“卸载”到底层平台线程。

有一种需要先建立印象的现象叫：

**pinning**

你可以先粗略理解成：

某些情况下，虚拟线程会“钉”在一个平台线程上，没法优雅地把底下的 OS 线程让出来。

一旦发生这种情况，虚拟线程“很多线程共享少量平台线程”的优势就会被削弱。

你当前阶段不需要一口气把所有触发条件背成源码级知识，但至少要建立下面这个判断：

- 普通阻塞 I/O：虚拟线程通常很合适
- 某些受限场景会 pin 住：这时别把虚拟线程想成永远免费

这也是为什么虚拟线程虽然很强，但仍然需要性能测试和工程观察，而不是盲目信仰。

### 11. 它和 `Future` / `CompletableFuture` 是替代关系吗

不是替代关系，更像是不同层次的问题。

你可以这样区分：

#### 11.1 虚拟线程

解决的是：

**任务跑在什么线程模型上。**

#### 11.2 `CompletableFuture`

解决的是：

**异步任务怎么编排、怎么组合、怎么做回调和异常恢复。**

很多过去之所以被迫写成 `CompletableFuture` 链，并不是因为业务逻辑天然适合那种编排，而是因为大家不想阻塞昂贵的平台线程。

有了虚拟线程之后，一部分代码就可以重新回到更直观的风格：

- 一个虚拟线程处理一个请求
- 中间同步调用数据库 / RPC
- 代码按自然顺序往下写

这就是虚拟线程非常大的工程价值：

**把高并发和代码可读性重新往一起拉。**

但如果你的场景本身就适合：

- 多任务并行汇总
- 抢答式返回
- 明确的异步流程编排

那 `CompletableFuture` 仍然有它的价值。

### 12. 它和 Reactor 怎么看

Reactor 这种响应式模型，核心价值通常在于：

- 非阻塞 I/O
- 事件驱动
- 对数据流和异步链路做统一建模

虚拟线程则更像是另一条思路：

- 我不强迫你把逻辑写成响应式流
- 我让阻塞式线程模型本身变便宜

所以很多以前为了省线程才引入的复杂 Reactor 链路，现在可以重新评估：

- 是不是真的必须用响应式
- 还是虚拟线程下直接用同步写法会更简单

但这依然不是谁“彻底淘汰”谁，而是看：

- 团队技术栈
- 下游模型
- 观测和调试能力
- 业务复杂度

### 13. 一个特别通俗的类比

你可以把平台线程和虚拟线程想成办公位管理：

#### 平台线程

一个员工长期占一个固定工位。

哪怕他现在只是打电话等别人回消息，这个工位也一直被他占着。

#### 虚拟线程

很多员工共享少量工位。

谁真正要动手处理 CPU 上的事，谁去占一个工位干一会儿；谁现在只是等待数据库、网络、RPC 返回，就先让出工位，等轮到自己再回来继续。

所以虚拟线程并不是让员工能力突然变强，而是让工位利用率更高。

### 14. 面试里怎么回答更像样

可以这样组织：

Java 虚拟线程是 JDK 提供的一种轻量级线程实现。它仍然是 `java.lang.Thread`，但和传统平台线程不同，虚拟线程不会长期一对一绑定某个固定的 OS 线程。  
它的核心价值在于，当线程在 I/O 场景中发生阻塞等待时，底层平台线程可以被释放出来复用，从而用更少的平台线程支撑非常高的并发量。  
因此虚拟线程特别适合高并发、I/O 密集型应用，比如 Web 服务、数据库调用、RPC 调用链等。  
它不是 CPU 密集计算的性能神器，也不是简单替代 `CompletableFuture` 或 Reactor，而是让很多原本为了省线程而写得很复杂的异步代码，可以重新考虑回到更直观的阻塞式线程模型。  
使用时还需要注意 `ThreadLocal` 成本、外部资源限制以及 `pinning` 等问题。

## 小结

- 虚拟线程仍然是 `Thread`，但不再长期一对一绑定固定 OS 线程。
- 它的核心价值不是让单个任务更快，而是让阻塞式线程编程模型也能支撑超高并发 I/O。
- 它特别适合数据库、RPC、HTTP、文件 I/O 等等待很多的场景。
- 它不是 CPU 密集型加速神器，也不是所有异步框架的简单替代品。
- 工程上要关注线程模型之外的真实瓶颈，比如连接池、限流、内存、`ThreadLocal` 和 `pinning`。
