# Kafka 学习路径与知识图谱

## 学习定位

很多人学 Kafka 时，容易停留在“会用”和“会背概念”的层面：

- 知道怎么 `kafka-console-producer` / `kafka-console-consumer`
- 知道 Topic 是分 Partition 的，Partition 有副本
- 知道有 Consumer Group，知道 Offset 要提交
- 知道 Kafka 很快，常听人说“顺序写 + 零拷贝”
- 知道有 ISR、有 acks、有 Rebalance

但一被深挖就会发懵：

- Kafka 为什么这么快，仅仅是“顺序写”和“零拷贝”吗？PageCache 在里面到底扮演什么角色
- 一条消息从 Producer 发出到 Consumer 提交 Offset，端到端到底经过了多少道生死关
- ISR、HW、LEO、Leader Epoch 这一组概念到底在解决什么问题，为什么必须是这一组
- `acks=all` 真的就不会丢消息吗，`min.insync.replicas` 为什么要配
- Kafka 的 Exactly-Once 到底是“真的 EOS”还是“营销话术”，事务消息底层是怎么做的
- Rebalance 为什么是 Kafka 生产环境最大的疼点之一，为什么有 Eager / Cooperative / Sticky 三代协议
- 为什么 Kafka 要从 ZooKeeper 切到 KRaft，背后真正的痛点是什么
- 消息丢失、重复、乱序、积压这四件事在生产上是怎么互相纠缠在一起的

这条学习路径的目标，不是把你训练成“Kafka 命令手册”或者“Kafka 八股文背诵机”，而是把 Kafka 放回真实分布式系统里，**从“消息队列”提升成“分布式日志 / 数据总线”来理解**。

学完这一套，你应该能做到：

- 说清楚 Kafka 到底是什么，它和 RocketMQ、RabbitMQ、Pulsar 的边界分别在哪
- 把 Kafka 的存储模型、副本模型、消费模型、事务模型四张图同时画出来，并且能讲清楚它们怎么互相支撑
- 解释 Kafka 为什么快，而不是只会说“顺序写 + 零拷贝”
- 把消息丢失、重复、乱序、积压这四类生产事故的根因和解法讲到链路级别
- 看懂 ISR、HW、LEO、Leader Epoch、Producer Idempotent、Transaction Coordinator 这些机制各自堵的是哪个洞
- 面对线上 Kafka 故障（Controller 抖动、ISR 收缩、Rebalance 风暴、消费 lag 暴涨），有一套排障的链路化思路，而不是只会“重启大法”

## 知识图谱（从零到专家）

整张图分成 **6 层**，从外到里、从概念到工程：

### 第 1 层：整体认知

- Kafka 是什么、不是什么
- 为什么会出现 Kafka（LinkedIn 的日志总线问题）
- Kafka 的“分布式日志（distributed commit log）”心智模型
- Kafka 在系统架构里的典型位置：业务解耦、削峰填谷、数据总线、CDC、流计算入口、日志中枢

### 第 2 层：核心抽象

- Topic / Partition / Replica / Segment / Offset
- Broker / Controller / Coordinator
- Producer / Consumer / Consumer Group
- 为什么是“分区 + 副本 + 顺序日志”这一组抽象，而不是“队列 + 副本”
- 一条消息端到端的完整链路（Produce → Broker → Replica → PageCache → 磁盘 → Fetch → Consumer → Commit Offset）

### 第 3 层：存储与高性能

- 顺序写、append-only log
- Segment、`.log` / `.index` / `.timeindex`
- PageCache、`sendfile`、零拷贝
- Producer 端的 Batch、Compression、Linger
- 为什么 Kafka 写得快、读得也快，瓶颈通常在哪
- mmap、刷盘策略、`flush.messages` / `flush.ms` 为什么默认不动

### 第 4 层：副本与一致性

- Leader / Follower 复制模型
- ISR、HW、LEO、`min.insync.replicas`
- `acks=0 / 1 / all` 的真实语义
- unclean leader election 为什么默认关
- Leader Epoch 解决了什么老问题（截断不一致）
- Controller 与元数据：ZooKeeper 时代 vs KRaft 时代
- 为什么要去 ZK：Controller 写 ZK 的瓶颈、元数据传播延迟、运维复杂度

### 第 5 层：生产者 / 消费者深挖

- 生产者：分区策略、Batch、`linger.ms`、`max.in.flight.requests.per.connection`、retries 与乱序
- 幂等 Producer：PID + Sequence Number、broker 端去重的范围与边界
- 事务 Producer：`__transaction_state`、Transaction Coordinator、两阶段提交、消费-生产闭环 EOS
- 消费者：Consumer Group、Coordinator、Heartbeat、Session Timeout、`max.poll.interval.ms`
- Rebalance 协议三代：Eager → Cooperative → Sticky / Incremental Cooperative
- Offset 提交：自动提交、手动提交、`__consumer_offsets`、`read_committed` 与 LSO（Last Stable Offset）
- 消费语义：at-most-once / at-least-once / exactly-once，分别落地在哪一层

### 第 6 层：工程实战与排障

- 消息丢失：链路上的每一段都可能丢，怎么逐段定位
- 消息重复：为什么 at-least-once 是默认，业务幂等怎么做
- 消息乱序：分区内有序 + 跨分区无序，retries / inflight 还会破坏分区内有序
- 消息积压：是消费慢还是分区少，怎么扩容、怎么 catch-up
- Rebalance 风暴：长任务、GC、网络、`max.poll.interval.ms` 配置错位
- ISR 抖动、Controller 切换、磁盘打满、网络分区
- 性能调优的五个层级：硬件 → OS → broker → producer → consumer
- 选型与边界：Kafka vs RocketMQ vs Pulsar vs RabbitMQ

## 为什么这样排序

这条路线刻意不是“先讲 Producer 怎么用、再讲 Consumer 怎么用”，而是先把 Kafka 放回它要解决的真实问题里：

- **前 3 课**建立“Kafka 是什么、抽象是什么、一条消息怎么走完全链路”的心智模型。
  Kafka 的所有其他设计——副本、ISR、事务、Rebalance——本质都是在保护这条链路上的某一段。心智模型不立起来，后面学的全是散点。
- **第 4 到 6 课**进入服务端三大支柱：存储、副本、元数据。
  这三块是 Kafka 真正“扛得住、不会丢、能选主”的底盘，也是面试和排障最喜欢挖的地方。
- **第 7 到 10 课**进入客户端的两端：Producer 和 Consumer。
  特别是把幂等、事务、Rebalance 单独拎成专题——因为这三块既是 Kafka 最有设计美感的部分，也是生产事故最高发的部分。
- **第 11 到 13 课**是经典“四大件”：丢失、重复、顺序、积压。
  这四件事在面试和线上几乎 100% 会被问到、被踩到，单独拆开讲。
- **第 14 到 16 课**是工程纵深：网络线程模型、线上排障、性能调优。
  到这一层，能不能讲到“Acceptor / Processor / Purgatory 之间是怎么交接的”、“ISR 抖动到底是 follower 拉慢了还是 leader fetch 处理不过来”，就是“面过”和“真的做过”的分水岭。
- **第 17、18 课**收口：把 Kafka 放回更大的系统里，谈数据总线、CDC、流计算入口，再做一次横向选型。

## 学习方法建议

- **每篇先问一个问题**：这一篇到底在保护链路上的哪一段、堵的是哪个洞。
  比如 ISR 是在堵“follower 跟不上但还假装自己同步好了”的洞，Leader Epoch 是在堵“老 Leader 苏醒后日志截断不一致”的洞，事务 Producer 是在堵“consume-process-produce 中间崩溃”的洞。
- **学概念时要追到链路**：看到“acks=all”不要只记“最强”，要画出 Producer → Leader → ISR follower → 落盘 → ack 返回这条链路，问自己每一步如果崩了会怎样。
- **学客户端时要看服务端怎么配合**：Producer 的 `enable.idempotence` 不是客户端单方面的事，broker 端有 PID 状态机；事务 Producer 必须配合 `__transaction_state` 和 Transaction Coordinator。
- **学 Rebalance 时一定要把版本讲清楚**：Eager / Cooperative / Sticky 三代是不同协议，混在一起讲必然糊。
- **学排障时要建立“层级假设”**：一个 Kafka 抖动，先怀疑是 broker（磁盘 / GC / ISR），还是 Coordinator（rebalance / heartbeat），还是网络（partition / 慢节点），不要见到 lag 就动手扩 Consumer。

## 学完后你应该能回答的问题

- 为什么 Kafka 写得快、读得也快，瓶颈到底在哪——能讲到 PageCache、sendfile、Batch、Compression、网络 RTT 这一组协同
- `acks=all` + `min.insync.replicas=2` + `replication.factor=3` 是不是就稳了，什么情况下还会丢
- Leader Epoch 是补的什么坑，为什么没有它光靠 HW 不够
- KRaft 解决了 ZooKeeper 时代的什么具体问题，不是“ZK 慢”这种含糊回答
- Producer 端 retries > 0 + max.in.flight.requests > 1 在没开幂等的情况下，为什么会破坏分区内有序
- Kafka 的 Exactly-Once 到底覆盖到哪——Producer→Broker、Consumer→业务、跨系统三段哪段是 EOS、哪段不是
- Cooperative Rebalance 比 Eager 强在哪，为什么不能完全替代 Eager
- 消费 lag 暴涨先看哪几个指标，怎么判断是“消费者慢”还是“分区不够”还是“GC / 网络”
- ISR 频繁缩进 / 扩出，第一时间该怀疑什么，怎么定位
- 为什么很多公司从 Kafka 迁去 Pulsar 又有人迁回来，两边的真实痛点分别在哪
