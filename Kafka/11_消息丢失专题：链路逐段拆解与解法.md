# 消息丢失专题：链路逐段拆解与解法

## 这一篇要回答什么

“Kafka 会不会丢消息？”这个问题不能用一句“配 `acks=all` 就不会丢”回答。那太粗了，也容易误导。

真正成熟的回答要先区分两个概念：

1. **发送失败**：Producer 没拿到成功确认，业务知道这条消息没可靠写入，可以重试或补偿。
2. **已确认后丢失**：Producer 已经收到成功 ack，业务以为消息可靠存在，后来 Kafka 里找不到了。

这两件事完全不同。系统可以接受“写不进去就报错”，但关键业务很难接受“明明成功了，后来没了”。

这一篇沿端到端链路拆：

```text
业务事务 / Producer 本地
  -> Producer 网络发送
  -> Broker Leader 写入
  -> Follower 复制 / ISR / ack
  -> Leader 切换 / 日志截断
  -> Consumer 拉取 / 业务处理 / offset commit
  -> 外部系统副作用
```

每一段都问三个问题：

- 哪里会丢？
- 怎么配置和编码？
- 怎么验证真的没丢？

## 先立边界：Kafka 的“不丢”到底承诺什么

Kafka 不是靠每条消息强 fsync 保证不丢。它的默认设计是：

> **写入走 PageCache，可靠性靠多副本和 ISR。**

所以 Kafka 的强可靠配置通常是：

```properties
replication.factor=3
min.insync.replicas=2
unclean.leader.election.enable=false
```

Producer 侧：

```properties
acks=all
enable.idempotence=true
retries=Integer.MAX_VALUE
```

Consumer 侧：

```properties
enable.auto.commit=false
```

这组配置的意思不是“宇宙绝对不丢”，而是：

- Producer 成功 ack 前，至少有满足 ISR 条件的副本写入。
- ISR 不足时拒绝写，而不是假装成功。
- Leader 挂了只从 ISR 里选新 Leader，不选落后副本。
- Producer 重试不会因为 ack 丢失而重复 append。
- Consumer 处理成功后再提交 offset，避免“处理没完成但进度已提交”。

边界也必须承认：

- 如果多个 ISR 副本所在机器同时掉电，PageCache 都没刷盘，仍可能丢。
- 如果业务写 DB 成功但 Kafka 发送失败，Kafka 管不了这个双写缝隙。
- 如果消费者先提交 offset 再处理业务，Kafka 会认为你已经处理完。
- 如果日志已经超过 retention 被删，之后再想消费不是“丢消息事故”，而是保留策略过期。

## 丢失点一：业务本地事务和发 Kafka 之间

最常见的“消息丢了”，其实发生在 Kafka 之前。

典型场景：

```text
T1: 订单服务写 MySQL，订单状态变成 PAID
T2: 准备发送 order_paid 事件到 Kafka
T3: 服务宕机 / 线程池满 / 发送失败
T4: 下游永远收不到 order_paid
```

Kafka 没收到这条消息，它当然不会帮你保住。这里的根因是：

> **业务数据库和 Kafka 是两个资源，普通代码无法把它们做成一个原子提交。**

### 错误姿势：数据库成功后直接 send

```java
orderRepository.markPaid(orderId);
producer.send(new ProducerRecord<>("order-events", orderId, event));
```

风险：

- DB 成功，send 失败。
- send 只是进了 Producer buffer，进程崩了。
- callback 失败没处理。
- 重试时应用已经不知道这条业务事件该不该补。

### 正确姿势：Outbox

关键业务更稳的方式是 Outbox：

```text
同一个本地事务：
  1. 更新订单状态
  2. 写 outbox_event 表

后台 relay：
  1. 扫描未发送事件
  2. 发送 Kafka
  3. 发送成功后标记 sent
```

这样至少能保证：

- 只要业务状态变了，待发送事件也一定存在。
- Kafka 临时不可用时，事件可以稍后补发。
- 发送重复时，下游靠业务幂等处理。

Outbox 的代价是多一张表和 relay 任务，但对关键业务，这比“发消息失败记日志”靠谱太多。

### 什么时候可以不用 Outbox

不是所有消息都需要 Outbox。比如：

- 日志。
- 埋点。
- 监控指标。
- 可丢弃通知。

这些场景可以接受少量丢失，直接异步发送更简单。

关键在于：**业务要明确分级**。不要把支付、库存、结算事件和点击日志放在同一个可靠性等级里。

## 丢失点二：Producer 本地 buffer

Kafka Producer 是异步批量发送器。`send()` 默认只是把消息放进本地 Accumulator，然后由 Sender 线程发出去。

风险场景：

```text
T1: producer.send(record) 返回
T2: 消息还在 Accumulator，没发到 broker
T3: JVM 被 kill -9 / 容器直接停止
T4: 本地 buffer 消失
```

这就是 Producer 端最隐蔽的丢消息。

### 怎么避免

关键链路要看发送结果：

```java
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        // 告警、补偿、Outbox 保留未发送状态
    }
});
```

服务关闭时要：

```java
producer.flush();
producer.close();
```

容器环境还要注意：

- termination grace period 要足够。
- 不要在 shutdown hook 里做无限等待。
- 发送线程池关闭顺序要在 Producer close 之前。
- `kill -9` 没法优雅，关键业务仍要靠 Outbox 补。

## 丢失点三：Producer 盲发或忽略 callback

如果代码只是：

```java
producer.send(record);
```

然后什么都不管，那么很多失败都会变成静默丢失：

- Topic 不存在且禁止自动创建。
- 权限失败。
- 消息太大。
- 序列化失败。
- metadata 获取失败。
- 超过 `delivery.timeout.ms`。
- ISR 不足导致 `NotEnoughReplicasException`。

这些并不是 Kafka “偷偷丢了”，而是应用没有处理发送失败。

### 关键业务的发送底线

至少做到：

- 使用 callback 或 `Future.get()` 感知最终结果。
- 记录业务幂等键、topic、key、异常类型。
- 可重试错误进入重试 / Outbox。
- 不可重试错误进入告警和人工处理。
- 不要只打印日志然后丢掉上下文。

### 发送失败和丢消息的区别

如果 Producer 收到明确异常：

```text
NotEnoughReplicasException
TimeoutException
RecordTooLargeException
AuthorizationException
```

这不应该叫“Kafka 丢消息”。这叫**写入失败**。业务要把它作为失败路径处理。

真正麻烦的是：Producer 收到了成功，后来消息不见了。后面几段才讨论这个。

## 丢失点四：acks 配置太弱

`acks` 决定 broker 到什么程度才给 Producer 成功响应。

| 配置 | 含义 | 丢失风险 |
|---|---|---|
| `acks=0` | Producer 发出后不等响应 | broker 没收到也不知道 |
| `acks=1` | Leader 写入后返回 | Leader 崩且 follower 没复制到时丢 |
| `acks=all` | ISR 内副本同步后返回 | ISR 配置正确时最稳 |

### `acks=0`

适合：

- 指标。
- debug 日志。
- 明确允许丢的埋点。

不适合：

- 订单事件。
- 支付事件。
- 库存扣减。
- 审计事件。

因为它连 broker 是否收到都不知道。

### `acks=1`

看起来比 `acks=0` 稳，但仍有经典丢失：

```text
T1: Leader 写入 PageCache
T2: Leader 给 Producer 返回 ack
T3: follower 还没复制
T4: Leader 机器宕机
T5: follower 被选为新 Leader
T6: 刚才 ack 的消息不存在
```

这就是“已确认后丢失”。

### `acks=all`

`acks=all` 等 ISR 内所有副本同步后再 ack。但它必须配合 `min.insync.replicas`。

如果 ISR 只剩 Leader 一个，`acks=all` 等“ISR 内全部副本”，实际上就只等 Leader，自然退化成 `acks=1`。

所以单独说 `acks=all` 没用，必须说三件套。

## 丢失点五：`min.insync.replicas` 没配

可靠写入的典型组合：

```properties
replication.factor=3
min.insync.replicas=2
acks=all
```

意思是：

- 每个 Partition 3 副本。
- ISR 至少 2 个才允许写。
- Producer 等 ISR 内同步后才算成功。

如果没配 `min.insync.replicas=2`：

```text
RF=3
acks=all
ISR={Leader}
Producer 写入成功
Leader 挂
消息丢
```

正确行为应该是：

```text
ISR={Leader}
min.insync.replicas=2
Producer 写入被拒绝
业务感知失败，稍后重试
```

这是可靠系统的一个重要取舍：

> **宁可短暂不可写，也不要假装写成功后再丢。**

## 丢失点六：unclean leader election

Leader 挂了以后，Controller 要选新 Leader。

如果允许从非 ISR 副本里选 Leader：

```properties
unclean.leader.election.enable=true
```

那么 Kafka 可以在所有 ISR 都不可用时，从落后副本里选一个新 Leader，让分区尽快恢复可用。

代价是：

```text
旧 Leader 有 offset 100..200
落后副本只有 0..150
落后副本被选为新 Leader
151..200 丢失
```

这也是“可用性 vs 不丢”的开关。

关键业务建议：

```properties
unclean.leader.election.enable=false
```

这样如果没有干净的 ISR 副本可选，分区宁可不可用，也不选落后副本覆盖已确认数据。

## 丢失点七：PageCache 和整机掉电

Kafka 默认不每条消息 fsync。Leader 和 follower 写入通常先到 PageCache，由 OS 后台刷盘。

这带来一个边界：

```text
acks=all
Leader + follower 都写入 PageCache
Producer 收到成功
两台机器同时掉电
PageCache 都没刷盘
消息可能丢
```

这不是 bug，而是 Kafka 的性能 / 可靠性取舍。

为什么还能接受？

- 正常单机 broker 进程崩溃，OS 还在，PageCache 不丢。
- 单台机器掉电，另一个 ISR 副本通常还在。
- 同时多台 ISR 机器掉电的概率比每条 fsync 带来的吞吐损失更可接受。

如果你要更强的物理落盘保证，可以考虑：

```properties
log.flush.interval.messages=1
```

但这会让吞吐明显下降，通常不推荐。Kafka 的主线哲学是：**用多副本抵消不 fsync 的风险**。

## 丢失点八：ISR 抖动和错误运维

ISR 频繁缩进 / 扩出时，可靠性正在下降。

风险场景：

```text
follower 因磁盘慢被踢出 ISR
ISR 从 {1,2,3} 变成 {1}
如果 min.insync.replicas=1，仍能写
Leader 再挂，最新消息没有副本
```

所以不能只看 Producer 是否成功，还要看 ISR 健康。

要监控：

- UnderReplicatedPartitions。
- OfflinePartitionsCount。
- ISR shrink / expand 频率。
- follower fetch lag。
- broker 磁盘 I/O。
- broker GC。
- 网络延迟。

常见治理：

- 修慢 broker，别让它长期拖副本。
- Leader 分布均衡。
- 避免高峰期做大规模副本迁移。
- 保证 broker 磁盘和网络余量。
- `min.insync.replicas` 不要为了可用性偷偷降到 1。

## 丢失点九：消费者自动提交 offset

消费端最常见的“丢消息”不是 Kafka broker 丢，而是 offset 提交早了。

错误场景：

```text
poll 返回 offset 100..199
Consumer 自动提交 offset=200
业务线程还在处理 150
进程崩溃
重启后从 200 读
150..199 中未完成的消息不会再投递
```

解决：

```properties
enable.auto.commit=false
```

然后：

```text
poll
处理业务成功
commit offset
```

这变成 at-least-once：不丢，但可能重复。重复靠业务幂等兜底。

## 丢失点十：批量消费中跳过失败 offset

Kafka offset 是进度条，不是每条消息的 ACK。

如果你 poll 到：

```text
100, 101, 102, 103
```

其中 101 失败，但你提交 104：

```text
committed offset = 104
```

那 101 就被跳过了。Kafka 会认为 100..103 都处理完成。

正确选择有三个：

### 1. 提交到失败前

100 成功，101 失败：

```text
commit 101
```

下次从 101 继续。

### 2. 整批不提交

让 100..103 重放。前提是业务幂等。

### 3. 明确跳过坏消息

把 101 写入 DLQ：

```text
topic=order-events
partition=3
offset=101
key=order-1
error=schema mismatch
raw payload=...
```

然后提交 102 或继续处理。

这必须有业务确认和审计。没有 DLQ 直接跳 offset，本质就是人为丢消息。

## 丢失点十一：多线程消费提交最大 offset

多线程消费很容易犯这个错：

```text
100 成功
101 处理中
102 成功
103 成功

错误：提交 104
```

如果 101 最后失败，重启也不会再读 101。

正确规则：

> 只能提交“从上次 committed offset 开始，连续成功处理的下一位”。

上面的例子只能提交 101，不能提交 104。

如果业务要多线程消费，必须维护每个 Partition 的完成区间，而不是只维护最大完成 offset。

## 丢失点十二：错误使用 offset reset

`auto.offset.reset=latest` 不会让已有 Group 每次都从最新读。它只在没有 committed offset 或 committed offset 已过期时生效。

但风险在这里：

```text
Group 的 offset 因 retention 过期
auto.offset.reset=latest
Consumer 重启
直接从最新位置开始
旧数据被跳过
```

关键业务建议：

```properties
auto.offset.reset=none
```

没有 offset 或 offset 无效时直接报错，由人工决定：

- 从 earliest 重放。
- 从某个时间点 seek。
- 从最新位置跳过。
- 走补偿任务。

不要让客户端悄悄 latest。

## 丢失点十三：retention 到期不是事故，但会造成“读不到”

Kafka 消息不会因为被消费就删除，而是按保留策略删除：

```properties
retention.ms
retention.bytes
```

如果 Consumer lag 太久，旧 Segment 被删：

```text
Consumer Group committed offset=100
Topic 最早 offset 已经变成 10000
Consumer 想从 100 读
OffsetOutOfRange
```

这不是 Kafka “中途丢了”，而是保留期设计不足。

治理：

- 根据最长故障恢复时间设置 retention。
- 对关键审计事件设置更长保留或落对象存储。
- 监控 Consumer lag 和最早 offset 差距。
- 对新下游回放需求提前评估历史保留。

## 丢失点十四：Kafka 到外部系统的缝隙

消费者常见动作：

```text
consume Kafka
write MySQL
commit offset
```

有两个方向的风险。

### MySQL 成功，offset 没提交

```text
write MySQL 成功
commit offset 前崩溃
重启后重复消费
```

这不是丢，是重复。靠幂等解决。

### offset 提交了，MySQL 没成功

```text
先 commit offset
write MySQL 失败
重启后不再消费
```

这就是丢业务结果。

正确顺序通常是：

```text
write MySQL 成功
commit offset
```

然后接受重复，用 MySQL 唯一约束、状态机、去重表兜住。

如果你要更完整的模式，可以做 Inbox：

- 消费 Kafka 后先把消息写入本地 inbox 表。
- inbox 表和业务更新放同一个本地事务。
- offset 提交失败后重复消费时，inbox 唯一键识别已处理。

## `acks=all` 还会丢吗

面试很喜欢追这个。

答案是：**会，但要分情况**。

### 情况一：只配了 `acks=all`，没配 `min.insync.replicas`

ISR 只剩 Leader 时退化成 `acks=1`，Leader 挂会丢。

### 情况二：开了 unclean leader election

落后副本被选主，已 ack 但未复制到它的数据会丢。

### 情况三：多个 ISR 副本同时掉电

消息在多个副本 PageCache 中，但都没刷盘，同时掉电可能丢。

### 情况四：业务没处理发送失败

Producer 因 ISR 不足拿到异常，但应用忽略了 callback。业务上看就是“丢”。

### 情况五：消费端先提交 offset

Broker 没丢，Consumer 自己跳过了。

所以更成熟的回答是：

> `acks=all` 是必要条件，不是充分条件。它必须配合 `replication.factor>=3`、`min.insync.replicas>=2`、关闭 unclean leader election、Producer 处理 callback、消费端手动提交和业务幂等，才能形成端到端可靠链路。

## 可靠性配置清单

### Broker / Topic

```properties
replication.factor=3
min.insync.replicas=2
unclean.leader.election.enable=false
```

谨慎调整：

```properties
log.flush.interval.messages
log.flush.interval.ms
retention.ms
retention.bytes
```

### Producer

```properties
acks=all
enable.idempotence=true
retries=Integer.MAX_VALUE
delivery.timeout.ms=120000
request.timeout.ms=30000
max.in.flight.requests.per.connection=5
```

代码要求：

- callback 必须处理。
- 退出前 flush / close。
- 关键业务用 Outbox。
- 不可重试错误告警。

### Consumer

```properties
enable.auto.commit=false
auto.offset.reset=none
isolation.level=read_committed
```

代码要求：

- 处理成功后再 commit。
- 批量失败只提交到失败前。
- 多线程只提交连续成功 offset。
- 坏消息进 DLQ，不静默跳过。
- Rebalance revoke 时提交被撤销 Partition 的 offset。

## 怎么验证“不丢”

配置写了不等于真的可靠。要验证。

### 1. Producer 侧对账

每条关键消息有业务唯一 ID：

```text
event_id = order_id + event_type + version
```

Producer 记录：

- 应产生事件数。
- Kafka callback 成功数。
- callback 失败数。
- Outbox 未发送数。

### 2. Broker 侧观测

监控：

- UnderReplicatedPartitions。
- OfflinePartitionsCount。
- ISR shrink / expand。
- produce error rate。
- `NotEnoughReplicasException`。
- broker 磁盘 / 网络 / GC。

### 3. Consumer 侧对账

Consumer 记录：

- 拉取数。
- 处理成功数。
- 处理失败数。
- DLQ 数。
- committed offset。
- 消费结果落库数。

### 4. 端到端审计

关键链路要能回答：

```text
某个 order_id 的 order_paid 事件：
业务表里有没有？
outbox 里有没有？
Kafka topic 里有没有？
Consumer 有没有处理？
下游表有没有结果？
DLQ 里有没有？
```

没有这条审计链路，“不丢”只能靠信仰。

### 5. 故障演练

至少演练：

- Producer 发送后立刻 kill。
- broker Leader kill。
- follower 慢磁盘 / 踢出 ISR。
- ISR 不足时 Producer 是否失败。
- Consumer 处理成功后、commit 前 kill。
- Consumer 自动提交关闭是否生效。
- DLQ 是否能回放。
- retention 到期时是否有告警。

## 常见事故定位

### 事故一：业务说下游没收到

先问：

1. 上游业务状态是否真的成功？
2. Outbox / 发送日志有没有这条 event_id？
3. Producer callback 成功还是失败？
4. Kafka topic 里能否按 key / offset 找到？
5. Consumer Group committed offset 是否越过它？
6. Consumer 日志是否处理失败或进 DLQ？
7. 下游是否幂等忽略或写库失败？

不要直接说 Kafka 丢了。多数情况下是链路某一段没有审计。

### 事故二：Producer 成功但 Topic 查不到

可能原因：

- 查错 Topic / 环境 / key。
- 消息已过 retention。
- compact Topic 旧 key 被压缩。
- unclean leader election 造成截断。
- 多副本同时掉电极端丢失。
- Producer 实际 callback 失败但应用误以为成功。

### 事故三：Consumer 明明 poll 到，却没有业务结果

可能原因：

- 自动提交提前提交。
- 业务线程池处理失败。
- 批量中间 offset 失败但提交了后面 offset。
- 多线程提交最大 offset。
- 写外部系统失败但 offset 已提交。

这类通常是消费端丢，不是 broker 丢。

## 面试怎么回答

如果被问“Kafka 怎么保证消息不丢，什么情况下还会丢”，可以这样答：

> 我会按链路拆。Producer 侧，`send()` 是异步的，消息先进 Accumulator，所以关键消息必须处理 callback，服务退出要 flush / close；如果业务数据库和 Kafka 双写，还要用 Outbox 保证业务状态变更和待发送事件在同一个本地事务里。Producer 配置上用 `acks=all`、`enable.idempotence=true`、足够 retries。
>
> Broker 侧，Kafka 默认写 PageCache，不靠每条 fsync，而是靠副本。可靠配置是 `replication.factor=3`、`min.insync.replicas=2`、Producer `acks=all`，并关闭 unclean leader election。这样 ISR 不足时会拒写，而不是假装成功。`acks=all` 不是充分条件，如果没配 min ISR、允许 unclean 选主、多个 ISR 同时掉电，仍可能丢已确认消息。
>
> Consumer 侧，最常见的丢是 offset 提交早了。关键链路关闭自动提交，业务处理和外部系统落库成功后再手动 commit。批量消费只能提交到连续成功的位置，多线程不能提交最大完成 offset，坏消息要进 DLQ，不能静默跳过。端到端还要靠业务幂等、Outbox / Inbox 和审计对账，因为 Kafka EOS 不覆盖 MySQL、Redis、HTTP 这类外部副作用。

这个回答的重点是：**不把“不丢”当成一个配置，而是当成一条链路。**

## 这一篇要带走的结论

- 发送失败不等于丢消息，已确认后找不到才是更严重的丢失。
- Producer 端最容易丢在本地 buffer、忽略 callback、业务 DB 和 Kafka 双写缝隙。
- `acks=all` 必须配合 `replication.factor`、`min.insync.replicas` 和关闭 unclean leader election。
- Kafka 默认不每条 fsync，可靠性主线是 PageCache + 多副本 + ISR。
- 消费端最常见的丢是先提交 offset 后处理业务，或批量 / 多线程提交越过失败消息。
- retention 到期导致读不到，是容量和保留策略问题，不是 broker 突然丢。
- 端到端可靠必须有业务幂等、Outbox / Inbox、DLQ、审计和故障演练。
