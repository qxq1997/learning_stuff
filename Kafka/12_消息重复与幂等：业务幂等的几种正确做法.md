# 消息重复与幂等：业务幂等的几种正确做法

## 这一篇要回答什么

上一章讲“不丢消息”，最后一定会走到一个现实结论：

> 你越想不丢，就越要接受重复。

Kafka 的默认工程姿势通常是 at-least-once：

```text
先处理业务
再 commit offset
```

这能避免“处理没完成但 offset 已提交”的丢失，但会带来另一个问题：

```text
业务处理成功
commit offset 前崩溃
重启后从旧 offset 重读
业务又处理一遍
```

所以这一篇专门回答：

1. Kafka 里重复消息到底从哪来？
2. Producer 幂等和业务幂等有什么区别？
3. 为什么不要指望 Kafka 自动帮你做业务去重？
4. 幂等键应该怎么选？
5. 唯一约束、去重表、状态机、版本号、业务流水、Inbox 分别适合什么场景？
6. Redis 去重为什么常常不够？
7. 面试里怎么把“至少一次 + 幂等”讲成熟？

先给结论：

> Kafka 的基础设施可以减少 Producer 重试造成的重复，但不能消除业务语义上的重复。真正的幂等要落在业务副作用发生的地方：数据库唯一约束、状态机条件更新、去重表、本地事务、业务流水、Inbox / Outbox 和审计补偿。

## 重复从哪里来

Kafka 链路里至少有 8 个重复来源。

### 1. Producer ack 丢失后重试

```text
T1: Producer 发 batch A
T2: broker append 成功
T3: ack 在网络中丢失
T4: Producer 重试 batch A
```

没开幂等 Producer 时，broker 可能 append 两次。

开启 `enable.idempotence=true` 后，Kafka 可以用 PID + Sequence Number 识别同一 Producer 会话内的重复重试。

但它只能挡“同一 Producer 会话内”的重试重复，挡不住业务重新构造一条语义相同的新消息。

### 2. 应用重启后重新发送同一业务事件

```text
T1: 订单事件发送到 Kafka 成功
T2: 应用没来得及记录发送成功，宕机
T3: 重启后从 Outbox / 本地任务重新发送
```

新 Producer 可能拿到新 PID，Kafka 会把它当成新消息。业务上这是重复事件。

这类重复只能靠 event_id、业务幂等键、下游幂等。

### 3. Outbox relay 重扫

Outbox 常见模式：

```text
业务事务写 outbox_event
relay 发送 Kafka
发送成功后标记 sent
```

如果 relay 在“发送 Kafka 成功”和“标记 sent”之间崩溃，重启后会再发一次。这是 Outbox 为了不丢消息主动接受的重复。

### 4. Consumer 处理成功但 commit 前崩溃

```text
poll offset=100
写 MySQL 成功
commit offset 前崩溃
重启后从 offset=100 再读
```

这是消费端最常见的重复。

### 5. Rebalance 前进度没提交

```text
C1 处理 p0 到 1000
只提交到 900
Rebalance 后 p0 给 C2
C2 从 900 开始读
```

900..1000 会重放。

### 6. `commitAsync` 失败或乱序

```text
提交 300
提交 400
400 成功
300 后成功
最终 committed offset 回退到 300
```

重启后 300..400 重放。

### 7. 手工 reset offset / seek

为了修复、回放、补数，人工把 offset 调回去，重复消费是预期行为。

### 8. 上游本身重复产生事件

例如：

- HTTP 请求重试。
- 定时任务重复扫描。
- CDC 重放。
- 人工补偿重复触发。

Kafka 只能传递事件，不能判断两个 value 是否代表同一个业务动作。

## 幂等 Producer 不是业务幂等

08 已经讲过幂等 Producer。这里再把边界钉牢。

幂等 Producer 管的是：

```text
同一个 Producer 会话内
同一个 PID
同一个 Partition
同一个 Sequence
因网络 / ack / leader 切换导致的重试
```

它不管：

- 应用重启后重新发的业务事件。
- Outbox relay 重扫。
- Consumer 重复消费。
- 写 MySQL / Redis / HTTP 的重复副作用。
- 两条不同消息 value 一样。
- 同一个业务事件被上游生成两次。

所以不要把：

```properties
enable.idempotence=true
```

理解成“业务幂等已经完成”。它只是 Kafka Producer 到 broker 这一小段的幂等。

业务幂等的目标是：

> **同一个业务动作执行多次，只产生一次有效副作用。**

比如：

- 同一笔支付只入账一次。
- 同一张订单只从 CREATED 变 PAID 一次。
- 同一个库存扣减请求只扣一次。
- 同一条搜索索引更新重复执行结果一致。

## 幂等键怎么选

幂等的第一步不是选 Redis 还是 MySQL，而是选 **幂等键**。

一个好的幂等键要满足：

- 能唯一表示一次业务动作。
- 上游重试、Kafka 重投、Outbox 重扫时保持不变。
- 下游能拿到并持久化。
- 能和 Consumer Group / 业务场景区分开。

常见幂等键：

| 场景 | 幂等键 |
|---|---|
| 支付成功 | `payment_id` / `transaction_id` |
| 订单状态变更 | `order_id + event_type + version` |
| 库存扣减 | `deduct_request_id` |
| 积分变更 | `points_txn_id` |
| 退款 | `refund_id` |
| 发券 | `coupon_grant_id` |
| 搜索索引更新 | `entity_id + version` |
| 通用事件 | `event_id` |

不要用这些当幂等键：

- Kafka offset。offset 只在某个 Topic Partition 内有意义，重发到另一个 Topic / Partition 就变了。
- 发送时间。重试时会变。
- 随机 UUID 如果每次重试都重新生成，也没用。
- message value hash。字段顺序、无关字段、序列化格式变化都会影响。

最好在事件里显式带：

```json
{
  "event_id": "order-1001-paid-v3",
  "event_type": "ORDER_PAID",
  "aggregate_id": "order-1001",
  "version": 3,
  "occurred_at": "2026-06-10T00:00:00Z"
}
```

## 方案一：唯一约束

最简单、最可靠的幂等方案是数据库唯一约束。

适合创建类、流水类业务：

- 支付记录。
- 退款记录。
- 发券记录。
- 积分流水。
- 消费处理日志。

例子：支付成功事件。

```sql
create table payment_record (
  id bigint primary key,
  payment_id varchar(64) not null,
  order_id varchar(64) not null,
  amount decimal(18,2) not null,
  created_at timestamp not null,
  unique key uk_payment_id (payment_id)
);
```

消费者处理：

```sql
insert into payment_record(payment_id, order_id, amount, created_at)
values (?, ?, ?, ?);
```

重复消息再次执行时，唯一键冲突。应用捕获重复键异常，认为“这条业务动作已经生效”，然后可以安全提交 offset。

优点：

- 靠数据库强约束，可靠。
- 和业务数据天然绑定。
- 不怕进程崩溃。

缺点：

- 只适合有天然唯一业务键的动作。
- 对纯更新类操作还需要状态机或版本号。

## 方案二：去重表 / Inbox

如果业务表本身没有合适唯一约束，可以建去重表。

```sql
create table mq_inbox (
  id bigint primary key,
  consumer_group varchar(128) not null,
  event_id varchar(128) not null,
  topic varchar(128) not null,
  partition_id int not null,
  offset_value bigint not null,
  status varchar(32) not null,
  created_at timestamp not null,
  updated_at timestamp not null,
  unique key uk_group_event (consumer_group, event_id)
);
```

处理流程：

```text
开启本地事务
  1. insert mq_inbox(group, event_id, status=PROCESSING)
  2. 执行业务更新
  3. update mq_inbox set status=SUCCESS
提交本地事务
commit Kafka offset
```

如果重复消费：

- 插入 `mq_inbox` 唯一键冲突。
- 查到 status=SUCCESS，直接跳过并提交 offset。
- 如果 status=PROCESSING 且超时，要根据业务结果做恢复判断。

关键点：

> **去重记录和业务更新必须在同一个本地事务里。**

否则会出现：

```text
去重表写成功
业务更新失败
重复消息再来时被当成已处理
业务结果丢了
```

Inbox 是去重表的加强版：它不仅记录“处理过”，还保存原始消息、处理状态、异常和重放入口，适合关键链路审计。

## 方案三：状态机条件更新

订单、工单、支付、物流这类业务通常有状态机。幂等可以靠状态流转约束实现。

比如订单状态：

```text
CREATED -> PAID -> SHIPPED -> FINISHED
```

处理 `ORDER_PAID`：

```sql
update orders
set status = 'PAID',
    paid_at = ?,
    version = version + 1
where order_id = ?
  and status = 'CREATED';
```

重复消息再来：

```text
status 已经是 PAID
update affected rows = 0
可以安全忽略
```

乱序消息也能挡一部分：

```text
ORDER_SHIPPED 已经把状态推进到 SHIPPED
迟到的 ORDER_PAID 再来，where status='CREATED' 不满足
不会把状态倒回 PAID
```

优点：

- 不需要额外去重表。
- 同时防重复和防乱序覆盖。
- 很贴合业务语义。

缺点：

- 状态机要设计清楚。
- 对非状态类副作用不够。
- “更新 0 行”要能区分已处理、乱序、非法状态。

## 方案四：版本号 / 乐观锁

适合实体更新类事件：

- 用户资料变更。
- 商品信息同步。
- 搜索索引更新。
- 账户状态同步。

事件里带版本号：

```json
{
  "user_id": "u1",
  "version": 15,
  "name": "Alice"
}
```

下游更新：

```sql
update user_profile
set name = ?,
    version = ?
where user_id = ?
  and version < ?;
```

这样：

- 重复 version=15 不会再次更新。
- 旧 version=14 晚到不会覆盖 version=15。
- 新 version=16 可以推进。

如果是严格连续版本，还可以要求：

```sql
where version = ?
```

即只允许从 14 更新到 15。缺点是中间缺事件会卡住，需要补偿或重放。

## 方案五：业务流水

支付、转账、积分、库存扣减这类业务，不建议只维护一个余额字段。

错误直觉：

```sql
update account set balance = balance + 100 where account_id = ?
```

重复消息会重复加钱。

更稳的做法是流水：

```sql
create table account_ledger (
  id bigint primary key,
  txn_id varchar(64) not null,
  account_id varchar(64) not null,
  amount decimal(18,2) not null,
  direction varchar(16) not null,
  created_at timestamp not null,
  unique key uk_txn_id (txn_id)
);
```

本地事务里：

```text
1. insert ledger(txn_id, amount)
2. update account balance
```

重复消息：

- `txn_id` 唯一键冲突。
- 不再更新余额。

流水的好处不只是幂等：

- 可审计。
- 可对账。
- 可回放重算。
- 可定位重复 / 漏处理。

金融类业务基本都应该有流水，而不是只改最终值。

## 方案六：幂等 Upsert

有些业务天然是“最终状态覆盖”，重复执行没问题。

例如搜索索引：

```text
把 product_id=1001 的索引更新为 version=20 的内容
```

如果重复执行同一版本 Upsert，结果一致。

但仍要注意乱序：

```text
version=20 先到
version=19 后到
```

如果直接覆盖，旧数据会覆盖新数据。所以 Upsert 最好也带版本判断：

```text
只在 incoming_version >= current_version 时更新
```

或者下游索引文档里保存 version。

## Redis 去重为什么常常不够

很多人第一反应是：

```text
SETNX event_id 1 EX 7d
成功就处理
失败就跳过
```

这个方案不是不能用，但有明显边界。

### 问题一：Redis 成功，DB 失败

```text
SETNX 成功
写 DB 失败
消息重试
SETNX 失败，被认为已处理
业务结果丢了
```

这就是“去重标记”和“业务副作用”不在同一个事务里的缝。

### 问题二：TTL 到期

重复消息如果在 TTL 后到达，Redis 去重失效。

Kafka retention、Outbox 重扫、人工补偿、历史回放都可能跨过 TTL。

### 问题三：Redis 数据丢失或淘汰

如果 Redis 内存淘汰、主从切换、持久化策略不强，去重记录本身可能丢。

### 什么时候 Redis 可以用

- 非关键业务。
- 短时间窗口去重。
- 下游业务本身还有数据库幂等兜底。
- 用来减轻数据库压力，而不是作为唯一防线。

更稳的姿势是：

```text
Redis 做快速过滤
DB 唯一约束 / 去重表做最终防线
```

## 幂等和事务要放在一起看

消费者处理外部系统通常是：

```text
consume Kafka
write DB
commit offset
```

Kafka 没法把 DB 写入和 offset commit 做成一个跨系统原子事务。

所以常见策略是：

```text
write DB 成功
commit offset
```

这样崩在 commit 前会重复消费，但 DB 幂等能兜住。

如果你反过来：

```text
commit offset
write DB
```

崩在中间就是丢业务结果。

这就是为什么消费端幂等不是锦上添花，而是 at-least-once 的组成部分。

## Outbox / Inbox 组合

关键链路常见组合：

```text
上游服务：
  业务表 + outbox_event 在一个事务里提交
  relay 发送 Kafka，可能重复

下游服务：
  inbox / 去重表 + 业务表在一个事务里提交
  处理成功后 commit Kafka offset
```

这套模式的语义是：

- 上游不丢事件。
- Kafka 发送允许重复。
- 下游重复消费可识别。
- 外部副作用和去重记录在同一个本地事务里。
- 端到端达到“最终一致 + 可审计 + 可补偿”。

它不是全局强一致，但工程上非常稳。

## 不同场景怎么选

| 场景 | 推荐幂等方案 |
|---|---|
| 支付成功入账 | `payment_id` 唯一约束 + 账户流水 |
| 订单状态变更 | `order_id + version` + 状态机条件更新 |
| 库存扣减 | `deduct_request_id` 唯一约束 + 库存流水 / 预扣记录 |
| 发券 | `coupon_grant_id` 唯一约束 |
| 积分变更 | `points_txn_id` 唯一约束 + 积分流水 |
| 搜索索引 | entity id + version 的幂等 upsert |
| 审计事件 | event_id 唯一约束 + 长期存储 |
| 通用消费框架 | Inbox / mq_consume_log + 本地事务 |
| 非关键日志 | 可接受重复或短 TTL Redis 去重 |

## 幂等设计的几个反模式

### 1. 只靠 Kafka offset 去重

Kafka offset 可以定位一条物理消息，但不能代表业务动作。

同一业务事件重发后，offset 会变；跨 Topic、DLQ、临时 Topic 也会变。

### 2. 先写去重表，再异步写业务

去重和业务不在一个事务里，崩溃后容易出现“标记已处理，但业务没生效”。

### 3. 用随机 message_id，但重试时重新生成

每次重试一个新 ID，等于没有幂等键。

### 4. 无条件覆盖状态

```sql
update orders set status = ?
where order_id = ?
```

重复和乱序都可能把新状态覆盖成旧状态。要加状态机条件或版本号。

### 5. 吞掉重复异常但没确认业务结果

唯一键冲突不一定永远代表“业务已完成”。如果之前在半途中留下了记录，要能区分 PROCESSING、SUCCESS、FAILED。

## 怎么验证幂等

不要只写一段去重代码就相信它。要做故障演练。

### 演练一：处理成功后、commit 前 kill

预期：

- 消息重放。
- 业务表没有重复副作用。
- offset 最终能推进。

### 演练二：Outbox 发送成功后、标记 sent 前 kill

预期：

- relay 重发。
- 下游幂等识别。
- 不产生重复订单状态 / 重复入账。

### 演练三：同一消息并发到达

预期：

- 唯一约束或状态机只有一个成功。
- 其他请求安全识别为重复。

### 演练四：旧版本消息晚到

预期：

- 版本号 / 状态机阻止旧消息覆盖新状态。

### 演练五：Redis 去重记录丢失

预期：

- DB 最终防线仍然挡住重复。

## 面试怎么回答

如果被问“Kafka 为什么会重复消费，怎么做幂等”，可以这样答：

> Kafka 生产环境通常按 at-least-once 设计，先处理业务再提交 offset，所以处理成功但 commit 前崩溃、Rebalance 前没提交、`commitAsync` 失败、手工 reset offset、Outbox 重扫、Producer ack 丢失重试，都可能导致重复。幂等 Producer 只能解决同一 Producer 会话内写 Kafka 的重试重复，解决不了业务重复和消费端重复。
>
> 业务幂等要靠业务键和外部系统约束。创建类动作优先用唯一约束，比如 payment_id、refund_id、coupon_grant_id；通用消费可以用 Inbox / 去重表，并且去重记录和业务更新放在同一个本地事务里；订单、工单这类状态流转用状态机条件更新；实体同步用版本号或乐观锁；账户、积分、库存这类资金库存场景用业务流水和唯一 txn_id。Redis SETNX 可以做短窗口过滤，但不能作为关键业务唯一防线，因为它和 DB 不在同一个事务里，还有 TTL 和淘汰问题。
>
> 所以成熟的设计不是追求 Kafka 层面绝对 exactly-once，而是接受至少一次投递，用幂等键、本地事务、唯一约束、状态机、Outbox / Inbox、DLQ 和审计，把重复副作用压成一次有效结果。

这段回答的重点是：**重复不是异常路径，而是可靠消息系统的正常成本。**

## 这一篇要带走的结论

- 不丢和重复是一对 trade-off，at-least-once 必然要求业务幂等。
- 幂等 Producer 只管 Producer 写 Kafka 的重试重复，不管业务重复。
- 幂等键要稳定表达业务动作，不能用易变字段或 Kafka offset 替代。
- 唯一约束、去重表、状态机、版本号、业务流水是最常见的五类硬防线。
- 去重记录必须尽量和业务副作用放在同一个本地事务里。
- Redis 去重适合短窗口优化，不适合作为关键业务唯一防线。
- Outbox + Inbox 是关键事件链路里非常稳的最终一致组合。
