# 消息顺序与积压：分区内有序的边界与积压治理

## 这一篇要回答什么

Kafka 线上最常见的两个问题，经常是绑在一起出现的：

- 业务要求同一个订单、用户、商品的消息按顺序处理。
- 消费端为了保证顺序，不敢随便并发，结果流量一上来就 lag 暴涨。

所以这一篇不只回答“Kafka 怎么保证顺序”，还要重点回答一个生产问题：

> Kafka 消息积压可能是什么原因，怎么定位，怎么解决？

一句话先立住：**Kafka 积压的直接原因永远是生产速度大于消费速度，但真正根因可能在生产流量、分区设计、消费者处理、下游依赖、Rebalance、Broker 资源、坏消息反复重试等不同层。**

## 顺序的前提：同 key 进同一个 Partition

Kafka 只保证 **Partition 内有序**，不保证 Topic 全局有序。

如果业务要求“同一个订单的状态变更必须有序”，Producer 必须用稳定的业务 key，比如 `orderId`：

```text
order-1: created -> paid -> shipped
          │          │       │
          └──────────┴───────┴──► hash(orderId) -> same partition
```

同一个 Consumer Group 内，一个 Partition 同一时刻只会分配给一个 Consumer，因此这个 Consumer 按 offset 顺序 `poll()`，天然能拿到分区内顺序。

但这也带来一个硬约束：

> **同一个 Consumer Group 的最大消费并发，受 Partition 数限制。**

10 个 Partition 最多同时让 10 个 Consumer 直接消费；你启动 50 个 Consumer，多出来 40 个拿不到 Partition，只是在空转。

## 顺序为什么还是会乱

分区内有序只是基础，不代表业务一定有序。常见乱序点有这些：

1. Producer 没传业务 key，或者 key 选错了。
   比如订单状态消息用状态字段当 key，`created`、`paid`、`shipped` 被打到不同 Partition。

2. Producer 开了重试，但没开幂等。
   旧请求失败后重试，新请求先成功，最终写入顺序可能被打乱。Kafka 的解法是开启 `enable.idempotence=true`，用 PID + Sequence Number 保住单分区内顺序。

3. Consumer 对同一个 Partition 的消息做了无序并发。
   `poll()` 是有序的，但业务线程池并发处理后，完成顺序可能变成乱序。需要按 key 分发到固定 worker、单 key 串行处理，或者用状态机兜底。

4. 失败消息被单独重试。
   offset=100 失败，101、102 已经处理完，100 稍后从重试 Topic 回来，如果业务没有版本号 / 状态机约束，就可能覆盖新状态。

5. 下游写入没有防乱序条件。
   订单状态更新如果只是 `update orders set status = ? where order_id = ?`，旧消息回来就能覆盖新状态。更稳的是带状态机或版本判断。

## 什么叫 Kafka 消息积压

Kafka 里常说的“积压”，一般指 **consumer lag**：

```text
lag = partition log end offset - consumer group committed offset
```

更口语一点：

- Broker 里这个 Partition 已经写到了 offset=100000。
- 某个 Consumer Group 只提交到了 offset=70000。
- 这个 Group 在这个 Partition 上 lag=30000。

注意，lag 是 **按 Consumer Group 看的**。同一个 Topic 可以有多个 Group：风控 Group 没积压，不代表数仓 Group 没积压。

排查时要看两层：

1. **总 lag**：这个 Group 总共落后多少条。
2. **分区 lag**：是不是少数几个 Partition 特别高。

如果总 lag 高但每个 Partition 都差不多，通常是整体消费能力不足；如果只有一两个 Partition lag 高，通常是 key 倾斜、热点分区或单分区坏消息卡住。

## 积压原因一：生产流量突增

最直接的原因是上游流量突然变大：

- 大促、秒杀、热点事件。
- 日志采集峰值。
- CDC 或批处理任务一次性补历史数据。
- 上游服务重试风暴，把同一批事件重复写入 Kafka。

这种积压不一定是故障，Kafka 本来就有削峰能力。关键看三个问题：

- lag 是否还在持续变大。
- 按当前消费速率，多久能追平。
- 这个 Topic 的业务是否允许排队这么久。

解决思路：

- 如果是可预期峰值，提前扩 Consumer、扩下游资源，并确认 Partition 数足够。
- 如果是异常流量，先限流 Producer 或关闭非关键写入，避免把 Topic、Broker、下游一起拖垮。
- 如果是历史补数，给补数任务限速，最好和实时 Topic 隔离，避免挤占线上消费。

## 积压原因二：Partition 数不够

Kafka 消费扩容不是“消费者越多越快”，而是：

```text
有效消费并发 <= Partition 数
```

如果 Topic 只有 4 个 Partition，你启动 20 个 Consumer，也只有 4 个能拿到分区。剩下的 Consumer 没有分配，lag 不会因为它们存在而下降。

解决思路：

- 短期：把 Consumer 扩到 Partition 数附近即可，超过太多没意义。
- 中期：增加 Topic Partition 数，提升后续写入和消费并发。
- 长期：按容量规划 Partition 数，给未来峰值留余量。

但有一个 Kafka 特有的坑：

> **增加 Partition 只能提升后续消息的并发，不能自动把历史积压重新打散。**

已经堆在旧 Partition 里的消息，仍然只能由旧 Partition 对应的 Consumer 顺序读出来。要处理超大历史积压，通常要用临时 Topic 或流处理任务重新分发。

## 积压原因三：Key 倾斜或热点 Partition

即使 Partition 数很多，也可能只有少数 Partition 积压。典型原因是 key 分布不均：

- 大客户 / 大商家流量远大于普通客户。
- 某个热门商品、热门直播间、热门活动 ID 成为热点 key。
- key 只有少数几个枚举值，比如 `status`、`region`、`eventType`。

现象通常是：

- Topic 总 Partition 很多。
- 大部分 Partition lag 很低。
- 一两个 Partition lag 持续暴涨。
- 对应 Consumer CPU 很高，其他 Consumer 很闲。

解决思路：

- 重新设计 partition key，不要用低基数字段。
- 对超热点 key 做拆分，比如 `merchantId + bucketNo`，牺牲这个大 key 的全局顺序，换吞吐。
- 如果必须保证单 key 严格顺序，只能承认它的并发上限是 1，重点优化单条处理耗时。
- 对热点业务单独拆 Topic，避免拖累普通流量。

这里的 trade-off 要讲清楚：**顺序和并发天然冲突。一个 key 越严格有序，它越难水平扩。**

## 积压原因四：消费者处理变慢

最常见的根因其实不在 Kafka，而在消费者业务代码：

- 单条消息逻辑太重。
- 每条消息都同步 RPC 调多个下游。
- MySQL 慢查询、锁等待、死锁。
- Redis / ES / 外部 API 超时。
- 线程池、连接池打满。
- 批量能力没用起来，每条消息单独写库。
- 消费者频繁 Full GC、OOM、重启。

解决思路按优先级来：

1. 先看消费者日志和下游指标，确认是不是异常、超时、慢 SQL。
2. 把逐条写改成批量写，把逐条 RPC 改成批量或异步并发。
3. 给下游调用设置超时、限流和熔断，不要让一个慢依赖拖死整个 poll 循环。
4. 调整线程池和连接池，但要以数据库 / 下游承载能力为上限。
5. 对重业务拆分：Consumer 只做轻量校验和分发，重计算交给异步 worker 或专门计算链路。

Kafka 参数也可以配合，但不要拿参数掩盖业务慢：

- `max.poll.records`：控制每次 `poll()` 返回多少条，太大可能导致处理超时，太小吞吐上不去。
- `fetch.min.bytes` / `fetch.max.wait.ms`：控制批量拉取效率。
- `max.partition.fetch.bytes`：单 Partition 一次 fetch 的上限，消息很大时要关注。

## 积压原因五：坏消息卡住或反复重试

有些 lag 暴涨不是整体慢，而是某条坏消息把消费卡住：

- 反序列化失败。
- schema 不兼容。
- 脏数据导致业务校验永远失败。
- 下游唯一约束冲突但代码没有正确识别幂等。
- 消费代码捕获异常后不提交 offset，下一轮继续读同一批，形成死循环。

现象是：

- Consumer 日志反复出现同一条消息或同一个 offset。
- lag 不怎么下降。
- CPU 可能不高，但错误日志很多。

解决思路：

- 解析失败、不可恢复的业务失败，不要无限重试，要打到 DLQ Topic，比如 `xxx.DLQ`。
- DLQ 消息要带原始 topic、partition、offset、key、异常原因，方便补偿。
- 可恢复失败要设置重试次数和退避，不要在主消费线程里死等。
- 消费成功和失败的 offset 提交策略要明确：跳过坏消息必须有审计和补偿入口。

Kafka 原生没有 RabbitMQ 那种统一的死信语义，DLQ 通常由消费者自己写入。

## 积压原因六：Rebalance 风暴

Consumer Group 频繁 Rebalance 时，消费会反复停顿。Eager Rebalance 下甚至会让整个 Group 短暂停止消费。

常见触发原因：

- Consumer 处理一批消息时间太长，超过 `max.poll.interval.ms`。
- JVM GC 停顿太久，心跳断了。
- 网络抖动导致 `session.timeout.ms` 过期。
- Consumer 实例频繁发布、重启、扩缩容。
- 订阅 Topic 或 Partition 数变化。

解决思路：

- 缩短单次 `poll()` 后的处理时间，必要时降低 `max.poll.records`。
- 重任务从 poll 线程剥离，但要控制 offset 提交和并发顺序。
- 合理调大 `max.poll.interval.ms`，让它覆盖最坏处理时间。
- 使用 Sticky / Cooperative 分配策略，降低 Rebalance 影响面。
- 发布时滚动、限速，避免整个 Group 同时重启。

判断这类问题时，不要只看 lag，要同时看 Consumer 日志里的 JoinGroup / SyncGroup、Rebalance 次数、心跳超时和 GC 日志。

## 积压原因七：Broker 或集群资源瓶颈

有时消费者没问题，是 Broker 端变慢：

- Broker 磁盘 I/O 打满。
- 网络带宽打满。
- PageCache 命中率下降，读历史数据触发大量磁盘读。
- Leader 分布不均，少数 Broker 扛了太多 Partition Leader。
- 副本同步慢，ISR 抖动。
- Controller 切换、分区重分配、扩缩容迁移占用资源。
- 配了 quota，Producer 或 Consumer 被限速。

解决思路：

- 看 broker 的磁盘、网络、请求队列、fetch 延迟、produce 延迟。
- 做 leader rebalance，让热点 Partition Leader 分散。
- 暂停或限速副本重分配、历史回放、批量补数这类重任务。
- 扩 Broker 或换更好的磁盘 / 网络。
- 对不同业务做 Topic / 集群隔离，避免低优先级大流量影响核心业务。

Kafka 很多“消费慢”表面上在 Consumer，实际上是 broker fetch 慢、网络慢、磁盘读慢。

## 排查顺序：不要一上来就加机器

线上处理 lag 暴涨，可以按这个顺序问：

1. 哪个 Topic、哪个 Consumer Group 积压？
2. 是所有 Partition lag 都高，还是少数 Partition 高？
3. 生产速率是否突然上涨？
4. 消费速率是否下降？
5. Consumer 是否有异常、GC、重启、Rebalance？
6. 下游 DB / RPC / Redis / ES 是否变慢？
7. 是否卡在同一个 offset 或同一种异常？
8. Broker 是否磁盘、网络、请求队列异常？
9. Partition 数是否限制了消费并发？
10. 业务是否允许跳过、降级、丢弃或延后处理？

这套顺序的价值是先分清楚：**是生产太快、消费太慢、并发上限太低、单分区热点，还是基础设施慢。**

## 应急处理：先止血，再追平

积压已经影响业务时，先做止血：

- 非核心 Producer 限流或暂停。
- 消费端扩容到 Partition 数上限。
- 临时关闭非必要的重逻辑，只保留关键落库或关键状态更新。
- 下游服务限流，避免 Consumer 追数据时把数据库打崩。
- 坏消息转 DLQ，主链路继续向前。
- 对可丢弃数据，在业务确认后 reset offset 到 latest。

如果历史积压特别大，可以建一个临时高分区 Topic：

```text
old topic (少分区, 大积压)
        │
        ▼
搬运 Consumer：只做轻量读取和重新分发
        │
        ▼
temp topic (更多分区)
        │
        ▼
更多 Consumer 并行处理
```

这个方案适合“历史积压已经被少数旧 Partition 锁住”的场景。清完之后再回到正常链路，或者迁移到新 Topic。

## 长期治理：把积压变成可预期问题

真正成熟的 Kafka 使用方式，不是等 lag 爆了再处理，而是提前治理：

- 设计 Topic 时按峰值吞吐和目标 catch-up 时间估算 Partition 数。
- partition key 选高基数、分布均匀的业务字段。
- 核心链路和日志 / 埋点 / 补数链路隔离。
- 消费者默认做幂等，避免重试带来重复副作用。
- 失败消息有 DLQ、重放工具和审计记录。
- 给 consumer lag、records-lag-max、消费速率、处理耗时、Rebalance 次数、错误率设置告警。
- 对下游调用设置超时、重试上限、熔断和限流。
- 定期压测 Producer、Broker、Consumer、下游整体链路，而不是只压 Kafka。

## 面试怎么回答

面试里被问“Kafka 消息积压可能是什么原因，怎么解决”，不要只说“加消费者”。一个更成熟的回答可以这样组织：

> 我会先看 consumer lag 是整体上涨还是某几个 Partition 上涨。整体上涨说明生产速率超过消费能力，要进一步看是流量突增还是消费者 / 下游变慢；少数 Partition 上涨通常是 key 倾斜、热点分区或坏消息卡住。然后看 Consumer 日志、Rebalance、GC、下游 DB/RPC 指标和 Broker 磁盘网络指标。解决上，短期可以限流 Producer、扩 Consumer 到 Partition 数上限、批量处理、跳过坏消息到 DLQ、必要时 reset offset；中期可以增加 Partition、优化 partition key、优化消费者批量和异步能力；如果历史积压被旧 Partition 锁住，可以用临时高分区 Topic 重新分发。长期要做 lag 告警、容量规划、DLQ、幂等和压测。

这个回答的关键是：你不是把积压当成一个单点问题，而是把它拆成 **生产、分区、消费、下游、Rebalance、Broker** 六层。

## 这一篇要带走的结论

- Kafka 只保证 Partition 内有序，顺序依赖业务 key、Producer 幂等、消费模型和业务状态机共同保证。
- 消息积压的直接原因是生产速度大于消费速度，但根因要分层定位。
- Consumer 数不能突破 Partition 数带来的并发上限，多加空转消费者没有用。
- 增加 Partition 主要改善后续吞吐，不能自动重分布历史积压。
- 少数 Partition lag 高，优先怀疑 key 倾斜、热点 key、坏消息卡住。
- Rebalance 风暴、下游慢、Broker 资源瓶颈都会表现成 lag 暴涨。
- 治理积压要先止血，再追平，最后做容量规划、告警、DLQ 和压测。
