# Kubernetes - 第 7 课：配置与存储：ConfigMap、Secret、PV、PVC与StatefulSet

## 学习目标（本节结束后你能做到什么）

- 理解为什么配置、密钥和镜像应该解耦。
- 掌握 ConfigMap、Secret 的常见使用方式和边界。
- 理解 Volume、PV、PVC、StorageClass 之间的关系。
- 能说清 Kubernetes 管理有状态服务时能做什么、不能做什么。

## 内容讲解（核心概念，用类比、例子、图示说清楚。不要太提纲化，加强每一节深度，力求深度。）

一个服务要运行，除了代码和镜像，还需要配置、密钥和数据。比如订单服务需要数据库地址、Redis 地址、线程池参数、功能开关；还需要数据库密码、JWT 私钥、第三方 API token；如果是数据库或消息队列，还需要持久化数据目录。Kubernetes 把这些问题拆成不同对象，避免所有东西都塞进镜像。

ConfigMap 用于保存非敏感配置。它可以以环境变量形式注入容器，也可以以文件形式挂载到容器里。比如不同环境的 `application.yaml`、日志级别、开关参数，都可以放在 ConfigMap 中。这样同一个镜像可以部署到测试、预发、生产，只需挂不同配置。镜像表达“程序是什么”，ConfigMap 表达“这个环境下如何运行”。

Secret 用于保存敏感信息，比如密码、token、证书。它和 ConfigMap 使用方式类似，也可以通过环境变量或文件挂载注入。但要注意，Kubernetes 默认 Secret 只是 base64 编码，不等于天然强加密。真正生产环境需要配合 etcd 加密、RBAC 权限控制、外部密钥管理系统、审计和最小权限原则。不要因为名字叫 Secret 就以为可以随意存放高敏感凭证。

配置注入有两种常见方式：环境变量和文件挂载。环境变量简单，适合少量配置；文件挂载适合配置文件、证书和复杂结构。一个细节是：通过环境变量注入的配置通常不会自动更新到已运行进程；通过挂载文件的 ConfigMap/Secret 可能会在一定延迟后更新文件内容，但应用是否能热加载取决于应用本身。很多生产系统仍然通过重启 Pod 来确保配置生效。

接着看存储。容器本地文件系统是临时的，Pod 删除后数据可能消失。Volume 是 Kubernetes 给 Pod 挂载存储的抽象。最简单的 `emptyDir` 随 Pod 生命周期存在，适合临时缓存、同 Pod 内容器共享文件，但不适合保存业务数据。要保存跨 Pod 生命周期的数据，就需要持久卷体系。

PV 是 PersistentVolume，表示集群里一块可用的持久化存储资源。PVC 是 PersistentVolumeClaim，表示应用对存储的申请。StorageClass 则描述动态创建存储的类型，比如使用哪种云盘、性能等级、回收策略。它们之间的关系像这样：

```text
应用 Pod
  -> 使用 PVC：我需要 20Gi，可读写一次挂载
  -> 绑定 PV：集群提供的一块具体存储
  -> StorageClass：决定如何动态创建这块存储
```

为什么要分 PV 和 PVC？因为应用开发者通常不应该关心底层存储细节，比如具体云盘 ID、NFS 路径、存储后端。应用只声明“我需要什么样的存储”，平台侧提供“可以满足这种诉求的存储”。这和 Kubernetes 声明式思想一致：消费者表达需求，控制器和基础设施完成绑定。

访问模式也很重要。ReadWriteOnce 表示卷可以被一个节点读写挂载，很多云盘都是这种模式。ReadOnlyMany 表示多个节点只读挂载。ReadWriteMany 表示多个节点读写挂载，通常需要共享文件系统支持。后端服务如果多个副本同时写同一个不支持并发语义的卷，很容易造成数据损坏，所以不能只看“能不能挂上”，还要理解存储语义。

StatefulSet 经常和 PVC 一起出现。它可以通过 volumeClaimTemplates 为每个 Pod 创建独立 PVC，比如 `data-mysql-0`、`data-mysql-1`。当 `mysql-0` 重建时，它仍然使用自己的 PVC，从而保持数据身份。这就是 StatefulSet 提供稳定身份和稳定存储关系的原因。

但必须强调：Kubernetes 的存储抽象不等于自动解决数据库一致性。比如 MySQL 主从复制、数据备份、故障切换、脑裂处理，这些不是 PV/PVC 自动完成的。Kubernetes 可以帮你把有状态进程拉起来、挂上盘、保持身份，但数据系统本身的高可用逻辑仍然需要数据库机制、运维流程或 Operator。

配置与存储的另一个生产重点是变更风险。ConfigMap 改错可能导致大面积配置异常，Secret 泄漏可能造成安全事故，PVC 删除可能造成数据丢失。对这些对象要有权限控制、审计、备份和发布流程。尤其是生产环境，不建议手工随意 edit 关键 Secret 或 PVC。

从后端视角看，配置、密钥、存储三者代表不同生命周期：镜像版本跟随代码发布，配置跟随环境和业务策略变化，密钥跟随安全轮换，数据跟随业务持续存在。Kubernetes 把这些生命周期拆开，是为了让系统更可控，而不是为了增加对象数量。

## 小结（3-5 条关键点）

- ConfigMap 保存非敏感配置，Secret 保存敏感信息，但 Secret 仍需要 RBAC、加密和密钥管理配合。
- 配置可以通过环境变量或文件挂载注入，是否热更新取决于注入方式和应用能力。
- emptyDir 适合临时数据，持久数据应该使用 PV/PVC/StorageClass 体系。
- PVC 表达应用存储诉求，PV 表达具体存储资源，StorageClass 支持动态供给。
- StatefulSet 能提供稳定身份和稳定存储关系，但不自动解决数据库高可用和一致性。
