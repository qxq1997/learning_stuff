# LangGraph 学习路径与知识图谱

## 学习定位

LangGraph 很容易被两类人学偏:

- **只看 API 文档的人**:学会了 `StateGraph`、`add_node`、`add_edge`、`compile`,做了一个 ReAct demo,以为自己掌握了 LangGraph。等到要做长任务、人工接管、状态分支、并发节点写同一个字段、子图嵌套时,完全不知道为什么 `state` 不是覆盖而是合并、为什么中断后能从上次的位置恢复、为什么 `Send` 不是普通的函数调用。
- **只看博客名词的人**:听过 Pregel、superstep、channel、reducer、checkpointer、interrupt,但说不清这些抽象具体落到哪段代码、为什么 LangGraph 要这样设计而不是直接做成 DAG 执行器。

这条路线的目标是把两类人都接住:**既懂用法,又懂为什么是这样设计的,也能在真实业务里写得稳**。

你有后端工程经验和一个公司内部 Airflow-like 编排平台(EMF)的背景。这是巨大优势,因为 LangGraph 本质是**带状态、带循环、带中断、带持久化的有向图执行引擎**——和你做编排平台时关心的问题(节点抽象、状态传递、并发合并、失败恢复、人工介入、子流程嵌套)高度同构。学 LangGraph 的过程,也是给 EMF 的下一步演进找参照系的过程。

**核心心智模型(贯穿全程,反复回来)**:

> LangGraph 不是 DAG 执行器,而是一个**支持循环的、状态在 channel 上做 reduce 的、可中断可恢复的 Pregel 风格图运行时**。
> 节点是纯函数(输入 state、输出 state 的偏差),边决定下一步谁跑,channel 决定状态怎么合并,checkpointer 决定能不能从中间继续,interrupt 决定人能不能介入。

学完这条路线后,你应该具备这些能力:

- 能用 Pregel/superstep/channel/reducer 这套抽象解释 LangGraph 的执行模型,而不是只会背 API。
- 能判断一个业务场景适合用 LangGraph、还是用普通的 ReAct loop、还是用 EMF 这种工作流编排平台。
- 能设计 State Schema、选对 reducer、合理拆分子图、规划中断点。
- 能用 Checkpointer 做长任务恢复、时间旅行(time travel)和分支(branching)。
- 能在 Multi-Agent 场景里选对协作模式(Supervisor / Swarm / Hierarchical / 自定义 routing)。
- 能在生产环境里处理状态膨胀、checkpoint 体积、错误恢复、可观测性、版本演进。
- 能把 LangGraph 的设计思想反过来评估和改进 EMF——哪些可以借鉴,哪些是过度设计。

## 知识图谱(从基础到深水区)

### Part 1 — 心智模型与基础抽象(回答"LangGraph 是什么")

1. **LangGraph 是什么,它解决了普通 LLM 调用解决不了的什么问题**
   - 从一次 prompt → ReAct loop → 多步可控流程的演进
   - LangGraph 和 LangChain Agent / AutoGen / CrewAI 的定位差异
   - 为什么不是 DAG,为什么不是普通的 Python 函数链
2. **StateGraph 与 State Schema:状态在 Graph 里怎么流动**
   - State 是 TypedDict / Pydantic / dataclass
   - 节点的输入 = 完整 state,输出 = state 的偏差
   - "偏差"如何被合并:`Annotated[T, reducer]`
3. **Node 与 Edge:函数式节点、条件边、动态路由**
   - `add_node`、`add_edge`、`add_conditional_edges`
   - `START` / `END` / `Send` / `Command` 的语义差别
   - 同步节点、异步节点、流式节点的写法差异
4. **MessagesState 与 add_messages reducer:为什么消息列表要单独处理**
   - 消息合并的幂等性、ID 去重、append vs replace
   - 为什么 LLM 应用几乎都会用到这个特殊 reducer

### Part 2 — 底层原理(回答"为什么是这样设计的")

5. **Pregel 模型与 superstep:LangGraph 为什么不是普通的 DAG 执行器**
   - Google Pregel 论文背景:BSP(Bulk Synchronous Parallel)模型
   - 一次 superstep = 一批节点同步执行 + channel 合并 + 决定下一批
   - 为什么这种模型天然支持循环、并发、确定性
6. **Channel 与 Reducer:状态合并、并发写入冲突怎么解**
   - LastValue / Topic / BinaryOperatorAggregate / EphemeralValue 等内置 channel
   - 自定义 reducer 的写法和坑(可结合性、幂等性、顺序敏感性)
   - 并发节点写同一个字段时 LangGraph 怎么决定最终值
7. **Checkpointer 与持久化:任务恢复、时间旅行、分支**
   - Checkpoint 的数据结构:state + channel versions + tasks + writes
   - InMemory / Sqlite / Postgres checkpointer 的取舍
   - `thread_id` 是会话维度,`checkpoint_id` 是时间点维度
   - 时间旅行(rewind)和分支(fork)在 checkpoint 层怎么实现
8. **Interrupt 与 Human-in-the-loop:中断协议、回填、approve/edit/reject**
   - `interrupt()` 函数 vs static `interrupt_before/after` 配置的差别
   - 中断时 graph 状态是什么样的,resume 时怎么继续
   - approve / edit / reject / abort 四种人工介入语义
   - 中断和 checkpoint 的关系:没有 checkpointer 就没有 interrupt
9. **Subgraph 与多 Graph 嵌套:命名空间、State 隔离、共享 channel**
   - 子图作为一个节点 vs 子图作为独立 graph 在主 graph 中编排
   - 共享 state schema vs 各自 state schema 的取舍
   - Subgraph 的 checkpoint 怎么和主 graph 协同

### Part 3 — 实战范式(回答"怎么用,典型场景怎么写")

10. **ReAct Agent 在 LangGraph 里怎么实现**
    - `create_react_agent` 预制函数 vs 手写 graph
    - tool node、condition edge、loop 的拼装
    - 何时该用预制,何时该手写
11. **Plan-Execute / Reflection / Reflexion 等高级范式**
    - Plan-Execute 在 LangGraph 里:planner 节点 + executor 子图 + replanner 循环
    - Reflection 模式:generator/critic 双节点循环
    - 何时这些范式才真正比 ReAct 强
12. **Multi-Agent:Supervisor、Swarm、Hierarchical**
    - Supervisor 模式:一个 router 节点决定调用哪个 agent
    - Swarm 模式:agent 之间用 handoff(Command + goto)互相转交
    - Hierarchical:多层 Supervisor 嵌套
    - 真实业务里多 Agent 的复杂度陷阱
13. **流式输出:values / updates / messages / events / debug 五种 stream 模式**
    - 五种模式各自暴露什么粒度,前端怎么消费
    - LLM token-level 流式 vs 节点级事件流的差别
    - `astream_events` 的 v2 事件协议

### Part 4 — 工程深水区(回答"上线要注意什么")

14. **状态膨胀、长任务、Checkpoint 体积治理**
    - State 里塞 LLM 全部历史的代价
    - 消息历史压缩、摘要、滑动窗口在 graph 里的位置
    - Checkpoint 行数膨胀、清理策略
15. **错误处理、重试、超时:LangGraph 的边界**
    - 节点级 retry policy
    - LangGraph 不擅长做的事:复杂 SLA、跨进程编排、跨天长任务
    - 何时该把 LangGraph 套在 Temporal/Celery/EMF 之上
16. **LangGraph Platform / Studio / Cloud:部署形态与可观测性**
    - LangGraph Server 的 task queue、scheduler、persistent thread
    - Studio 的 graph 可视化、time travel、断点调试
    - LangSmith trace 在 graph 上的呈现
17. **LangGraph vs 其他框架的工程对照**
    - vs AutoGen:actor model + topic 路由 vs graph + channel
    - vs CrewAI:配置驱动 vs 代码驱动
    - vs Temporal:workflow engine 的成熟工程能力 vs LangGraph 的轻量与 LLM 友好
    - vs Airflow / EMF:DAG 离线编排 vs 在线 stateful agent
18. **LangGraph 的不足、争议与替代方案**
    - State schema 偏静态、热更新不友好
    - Multi-tenant、SaaS 化时的工程负担
    - 何时根本不该用 LangGraph

### Part 5 — 综合实战与反哺 EMF

19. **从零实现:一个可控的研发助手 Agent**
    - 需求拆解 → state 设计 → 节点拆分 → 中断点设计 → checkpoint 策略
    - 把第 10–13 课的范式真实拼起来
20. **把 LangGraph 思想搬回 EMF:哪些值得借鉴,哪些是过度设计**
    - Channel + Reducer 模型 vs EMF 现有状态传递
    - Checkpointer 风格的"任务时间旅行"在 EMF 里怎么落地
    - Interrupt + Resume 在 EMF 人工节点上的对照
    - 不要把 EMF 改成 LangGraph,而是借鉴抽象去解决具体痛点

## 推荐推进顺序

1. **先 Part 1(第 1–4 课)**:建立"State + Node + Reducer"的最小心智模型,不碰 Pregel 这些字眼。
2. **再 Part 2(第 5–9 课)**:把 LangGraph 当成一个引擎拆开看,这是和别人拉开差距的关键。
3. **然后 Part 3(第 10–13 课)**:把各种 Agent 范式落到 graph 上,理解什么时候该自己写、什么时候用预制。
4. **进入 Part 4(第 14–18 课)**:补齐生产环境会踩的坑、框架边界与替代方案。
5. **最后 Part 5(第 19–20 课)**:动手做综合实战,并把所学反哺 EMF。

## 评估标准(沿用根 readme 的掌握学习风格)

- **完全掌握**:能用自己的话解释设计动机,能在真实业务里判断该不该用、怎么用 → 进入下一课
- **基本掌握**:概念清楚,但边界判断略弱 → 简短纠正 + 推进
- **部分理解**:把 channel/reducer/checkpoint/interrupt 混淆 → 生成补充课 `XXb_补充.md`
- **尚未理解**:还停留在"LangGraph 就是一个 graph 库" → 回到第 1 课重建认知
