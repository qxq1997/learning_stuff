# Linux 学习路径与知识图谱

## 学习定位

Linux 对后端工程师来说，不只是“会敲几个命令”这么简单。

它更像是你每天都站在上面的那块地基：

- 你的服务跑在 Linux 上
- 你的进程、线程、文件、网络连接都受 Linux 管
- 你的线上排障、性能分析、稳定性治理，最后几乎都会回到 Linux

很多人学 Linux 时容易走两个极端：

- 只背命令，知道 `ls`、`grep`、`ps`、`top` 怎么用，但说不清这些现象背后的系统原理
- 只背原理，知道虚拟内存、epoll、页缓存这些词，但一到真实机器上不会定位问题

这条学习路径的目标，是把“命令使用”“系统原理”“线上排障”“面试表达”串成一条线。学完之后，你不仅能用 Linux，更能解释它为什么这样工作。

## 知识图谱（从零到专家）

1. 先建立 Linux 的整体认知
   - Linux 到底在管什么
   - 用户态与内核态
   - 进程、文件、网络、内存这几条主线如何串起来
2. 文件系统与日常操作
   - 目录结构
   - 文件权限
   - 常用命令
   - 管道、重定向、环境变量
3. 进程、线程与调度
   - `fork`、`exec`
   - 进程和线程的区别
   - 调度、上下文切换
   - 信号
4. 内存系统
   - 虚拟内存
   - 页、页表
   - 栈、堆
   - 页缓存
   - `mmap`
   - OOM
5. I/O 与文件描述符
   - `fd` 是什么
   - 阻塞与非阻塞
   - 同步 I/O 与异步 I/O
   - `select`、`poll`、`epoll`
6. 网络与 socket
   - TCP 连接建立与断开
   - `listen`、`accept`
   - 收发缓冲区
   - 长连接与高并发
7. 线上排障工具
   - `ps`、`top`、`htop`
   - `ss`、`netstat`
   - `lsof`
   - `vmstat`、`iostat`
   - `strace`、`perf`
8. 服务管理与工程落地
   - `systemd`
   - 日志
   - 守护进程
   - 发布、重启、故障恢复
9. 性能优化与面试表达
   - 零拷贝
   - 页缓存命中
   - 高并发网络模型
   - 常见 Linux 面试题的答题框架

## 为什么这样排序

这条路线不是按“命令手册”排的，而是按后端工程师最容易真正用上的顺序排的。

之所以先从 `select`、`epoll`、I/O 多路复用开始，是因为它是后端和系统面试里的高频交叉点。你一旦把这个点吃透，后面再学 socket、Nginx、Redis、网关、高并发模型时，很多内容都会突然串起来。

然后再补文件系统、进程线程、内存、网络和排障工具，是为了把“会回答面试题”升级成“真的能在机器上定位问题”。最后再回到性能优化和工程落地，把原理和实践闭环。

## 学习方法建议

- 每学一个概念，都先问自己：它在解决什么现实问题。
- 不要把 Linux 学成碎命令，要把命令背后的资源模型串起来：CPU、内存、磁盘、网络到底谁是瓶颈。
- 学原理时尽量站在“用户态程序如何调用内核”的角度理解，而不是只背定义。
- 每学完一课，最好都配一轮命令验证，比如看进程、看 socket、看文件描述符、看权限。
- 对高频面试题，不只记标准答案，还要能讲“为什么”“适用场景”“边界条件”和“常见误区”。

## 学完后你应该能回答的问题

- 为什么高并发服务器不会采用“一个连接一个线程”作为主流方案。
- 文件描述符到底是什么，socket 为什么也能被当作 `fd`。
- 为什么 `epoll` 更适合大量连接但活跃连接不多的场景。
- 进程和线程分别适合解决什么问题，调度开销来自哪里。
- 为什么机器内存没打满，服务仍然可能因为 OOM 或页缓存行为出问题。
- 当线上出现 CPU 飙高、连接堆积、磁盘打满、请求超时，你应该先看哪些 Linux 指标和工具。

## 推进规则

- 如果是从零开始，建议按顺序学，不要跳。
- 如果你的目标偏后端面试，可以先重点学第 1、3、4、5、8 课。
- 如果你的目标偏线上排障，可以重点学第 2、4、5、6、7、8 课。
- 后续每一课都尽量沉淀成“原理 + 命令验证 + 面试表达 + 易错点”的结构。
