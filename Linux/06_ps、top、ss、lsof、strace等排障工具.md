# Linux - 第 6 课：运行态观察与线上排障工具

## 学习目标

本节不是命令列表，而是一套把线上现象收敛为证据的排障方法。学完后你应该能：

- 用 `ps`、`top` 区分进程、线程、CPU 热点和阻塞任务。
- 解释 `load average` 为什么高，并判断它更像 CPU 拥塞还是 I/O 等待。
- 用 `free`、`vmstat`、`sar -B` 判断内存是否真的成为瓶颈，而不是看到 `free` 小就误判。
- 用 `ss`、`lsof` 定位监听端口、连接状态和连接积压问题。
- 面对“CPU 100%”“请求变慢”“服务端口不通”“机器抖动”时，按证据链继续深挖。

> 本课聚焦观察与诊断。进程调度、虚拟内存和 TCP 原理会在对应原理章节进一步展开。

## 1. 先建立排障方法：现象不是结论

线上看到的通常只是症状：

- 请求响应变慢
- CPU 打满
- 内存看起来快用完
- 端口连不上
- 负载升高但 CPU 并不高

一个稳定的诊断顺序是：

1. 确认影响范围和发生时间：单实例还是全局，持续还是突发。
2. 查看系统层总体现象：CPU、负载、内存、I/O、网络连接。
3. 找到负责该现象的进程或线程。
4. 进入进程内部或业务日志，定位代码、请求或依赖。
5. 修改前保留证据，修改后用同一组指标验证恢复。

常见入口如下：

| 现象 | 第一眼看什么 | 下一层证据 | 常用命令 |
| --- | --- | --- | --- |
| CPU 高 | CPU 各项占比、热点 PID | 热点线程与调用栈 | `top`、`pidstat`、`top -H`、`perf` |
| 负载高 | 负载趋势、CPU、等待任务 | 运行队列或 I/O 等待 | `uptime`、`vmstat`、`mpstat` |
| 内存紧张 | `available`、swap 活动 | 回收、OOM、进程 RSS | `free`、`vmstat`、`sar -B`、`ps` |
| 端口不通 | 是否监听、连接状态 | 进程归属、队列与防火墙 | `ss`、`lsof`、`nc` |
| 单服务卡顿 | 进程树、线程状态 | 系统调用或语言栈 | `ps`、`strace`、`jstack`、`perf` |

关键心法是：**命令输出告诉你证据，不会自动告诉你根因。** 比如负载高可能是 CPU 忙，也可能是磁盘卡住；swap 已有使用量不等于此刻仍在换页，必须看动态指标。

## 2. `ps`：回答“谁在运行”

`ps` 提供一个时刻的进程快照，适合检索、筛选和记录证据；`top` 则适合连续观察变化。

### 2.1 常用查询方式

```bash
# BSD 风格：适合快速浏览全部进程及资源占用
ps aux

# POSIX 风格：适合看父子关系、启动命令
ps -ef

# 精确选择输出列，排障记录更清晰
ps -eo pid,ppid,user,stat,pcpu,pmem,rss,vsz,lstart,cmd --sort=-pcpu

# 找内存常驻集最大的进程
ps -eo pid,user,pmem,rss,vsz,comm --sort=-rss
```

`ps aux` 中的 `a`、`u`、`x` 是 BSD 选项组合；`ps -ef` 使用带短横线的 Unix/POSIX 选项。两种风格可以使用，但不要把 `ps -aux` 当成最清晰的标准写法。

### 2.2 必须读懂的字段

| 字段 | 含义 | 排障时怎么理解 |
| --- | --- | --- |
| `PID` | 进程 ID | 继续追踪、发信号、附加工具的入口 |
| `PPID` | 父进程 ID | 判断拉起关系、worker 是否属于某个 master |
| `%CPU` | 进程采样期 CPU 占比 | 找热点候选，需结合线程与持续时间 |
| `%MEM` | RSS 占物理内存比例 | 找内存大户的入口 |
| `VSZ` / `VIRT` | 虚拟地址空间规模 | 包含映射和保留空间，不能直接当真实内存消耗 |
| `RSS` / `RES` | 当前驻留物理内存 | 比 VSZ 更接近实际内存压力，但共享页可能被重复统计 |
| `TTY` | 控制终端 | `?` 常见于守护进程，并不说明异常 |
| `STAT` | 进程状态及附加标志 | 区分运行、睡眠、不可中断、僵尸等状态 |
| `TIME` | 累计消耗的 CPU 时间 | 判断进程是否长期消耗 CPU |
| `COMMAND` / `CMD` | 启动命令 | 识别服务及参数，注意可能被截断 |

### 2.3 `STAT` 状态比 CPU 数字更能解释“卡住”

常见主状态：

| 状态 | 含义 | 诊断提示 |
| --- | --- | --- |
| `R` | runnable/running，正在运行或等待 CPU | 大量持续 `R` 且 CPU 高，可能是 CPU 竞争 |
| `S` | 可中断睡眠 | 服务等待事件时很常见，不必恐慌 |
| `D` | 不可中断睡眠，常见于等待 I/O | 大量或长期 `D` 会推高负载，重点看磁盘、网络存储或内核路径 |
| `T` | stopped/traced | 进程被暂停或被调试 |
| `Z` | zombie，子进程已退出但父进程未回收 | 重点追查父进程的 `wait` 处理 |

常见附加标志中，`s` 表示会话首进程，`l` 表示多线程，`+` 表示前台进程组。

### 2.4 看父子进程：master/worker 不是多个无关服务

像 Nginx、数据库或进程管理器，经常采用 master/worker 模式。可以查看树形关系：

```bash
ps -ef --forest
pstree -ap
pgrep -a nginx
```

如果一个 master 下挂了多个 worker，不能只杀某个 worker 后便认定故障消失：master 可能重新拉起它，真正要查的是 worker 为什么异常或配置是否导致资源竞争。

### 2.5 停进程先发送可清理的信号

```bash
# 知道 PID：默认发送 SIGTERM，让程序有机会清理资源
kill <pid>

# 按完整进程名发送 SIGTERM
pkill -x <process_name>

# 仅在进程不能响应且已保留证据时强制结束
kill -KILL <pid>
```

`SIGKILL` 不能被进程捕获：事务、临时文件、请求收尾和日志刷新都没有机会完成，因此不应作为第一步。

## 3. Linux 线程：一个 PID 下可能藏着真正热点

进程是资源容器，线程才是被调度执行的实体。一个 Java、MySQL 或 Nginx worker 进程的总 CPU 很高时，通常还要继续找到其中的热点线程。

### 3.1 查看线程

```bash
# 指定进程内线程；SPID/LWP 对应线程 ID
ps -T -p <pid>

# 更完整的线程列
ps -eLf

# 持续观察某个进程各线程的 CPU
top -H -p <pid>
```

排障时需要区分：

- `PID`/`TGID`：线程组，也就是通常所说的进程 ID。
- `TID`/`LWP`/`SPID`：具体线程的内核任务 ID。
- 某个线程消耗 CPU 并不等于整个进程所有线程都忙。

### 3.2 从热线程走到代码位置

```bash
# 原生程序：持续采样热点函数
perf top -p <pid>

# 观察某个线程卡在哪些系统调用上，谨慎短时使用
strace -fp <pid>

# Java 程序：先把十进制 tid 转为 jstack 中使用的十六进制 nid
printf '%x\n' <tid>
jstack <pid>
```

对 JVM 应用，应在 `jstack` 输出中搜索 `nid=0x...` 对应的线程栈。对原生服务，`perf` 更适合 CPU 热点；`strace` 适合确认是否反复等待、重试或阻塞在系统调用上，但附加到繁忙进程会带来额外开销。

## 4. `top` 与 `uptime`：负载高不等于 CPU 高

### 4.1 `top` 顶部区域在回答什么

`top` 的顶部通常包含：

- 当前时间、运行时长、在线用户数。
- `load average`：最近 1、5、15 分钟平均负载。
- `Tasks`：运行、睡眠、停止、僵尸任务数。
- CPU 时间分布。
- 物理内存与 swap 概况。

常用交互动作：

```text
1        展开每个逻辑 CPU 的统计
P        按 CPU 使用率排序
M        按内存使用率排序
H        切换线程显示
```

### 4.2 平均负载的精确定义

Linux 的平均负载近似反映一段时间内：

- 正在运行或等待 CPU 的任务数，即 `R` 状态任务。
- 处于不可中断睡眠的任务数，即常见因 I/O 等待形成的 `D` 状态任务。

因此：

- CPU 密集任务会同时推高 CPU 使用率和负载。
- I/O 阻塞可能使负载很高，但 CPU 仍然空闲。
- 不能仅凭一个 load 数字判定 CPU 不够。

三个数字的读法：

| 现象 | 解释 |
| --- | --- |
| 1 分钟值明显高于 15 分钟值 | 压力正在上升 |
| 1 分钟值明显低于 15 分钟值 | 压力正在回落 |
| 三者接近 | 压力相对稳定，仍需判断是否处于高位 |

负载应和可使用的逻辑 CPU 数量一起看。例如 8 个 CPU 上稳定负载 `8` 和单 CPU 上负载 `8` 的含义完全不同。所谓“超过核数多少倍一定有问题”只能作为告警经验值，最终要结合延迟、队列和等待原因。

```bash
uptime
nproc
top
```

### 4.3 CPU 字段如何定位方向

| `top` 字段 | 含义 | 可能的下一步 |
| --- | --- | --- |
| `us` | 用户态执行 | 查业务计算热点、序列化、循环、JIT 后栈 |
| `sy` | 内核态执行 | 查系统调用、网络包、锁、上下文切换 |
| `id` | 空闲 | 高负载且 `id` 高时，怀疑不可中断等待而非计算 |
| `wa` | CPU 等待 I/O 的时间占比 | 查磁盘、块设备、远程盘、存储延迟 |
| `hi` / `si` | 硬中断 / 软中断 | 查网络包量、驱动、网卡队列 |
| `st` | 虚拟化环境被宿主机偷走的 CPU 时间 | 云主机邻居干扰或宿主资源争用 |

负载异常时可以用采样工具交叉验证：

```bash
mpstat -P ALL 1
pidstat -u 1
vmstat 1
```

`vmstat` 中 `r` 反映等待运行的任务，`b` 反映不可中断睡眠任务；第一行往往是自启动以来的累计平均值，诊断当前问题时应重点看随后按间隔刷新的行。

## 5. 内存诊断：`free` 很小不等于机器没内存

Linux 会主动利用闲置内存做缓存，以减少磁盘读取。因此“空闲内存小”本身通常不是故障。

### 5.1 先看 `available` 与动态压力

```bash
free -h
top
ps -eo pid,user,pmem,rss,vsz,cmd --sort=-rss
```

`free` 的关键字段：

| 字段 | 含义 | 正确读法 |
| --- | --- | --- |
| `total` | 总物理内存 | 机器容量基准 |
| `used` | 当前已使用 | 不能脱离缓存和 available 单独判断 |
| `free` | 完全未使用 | 低是正常现象的可能性很大 |
| `buff/cache` | 内核缓冲和文件页缓存等 | 部分通常可回收以服务新分配 |
| `available` | 在不发生明显 swap 的前提下可提供给应用的估算值 | 判断即时余量的首要入口 |
| `Swap used` | 已位于 swap 的页面量 | 需结合当前换入换出，历史用过不等于现在抖动 |

比较可靠的“内存正在成为瓶颈”证据组合是：

- `available` 长期低，且业务延迟或失败同时发生。
- `vmstat` 中 `si`、`so` 持续明显非零，说明发生换入换出。
- 内存回收或直接回收活跃，系统延迟出现尖刺。
- 内核日志出现 OOM kill。
- 少数进程 RSS 或 cgroup 内存持续逼近约束。

### 5.2 `vmstat`：看系统是否在为内存挣扎

```bash
vmstat 1
```

重要列：

| 列 | 含义 | 异常线索 |
| --- | --- | --- |
| `free` | 空闲内存 | 只用于配合其他字段观察 |
| `buff` / `cache` | 缓冲和缓存 | 缓存可回收，单独变大不必紧张 |
| `si` / `so` | 每秒从 swap 换入 / 换出 | 持续活跃并伴随延迟，说明压力真实 |
| `bi` / `bo` | 块设备读 / 写 | 和 `wa` 一起判断存储瓶颈 |
| `r` / `b` | 等待运行 / 阻塞任务 | 用于解释负载从哪里来 |

### 5.3 `sar -B`：看内存回收是否制造抖动

```bash
sar -B 1
```

其中：

| 指标 | 含义 | 诊断价值 |
| --- | --- | --- |
| `pgscank/s` | 后台回收线程 `kswapd` 每秒扫描页数 | 表示后台正在回收 |
| `pgscand/s` | 分配路径触发直接回收时每秒扫描页数 | 高时应用线程可能被迫等待回收，容易造成延迟抖动 |
| `pgsteal/s` | 每秒成功回收的页数 | 配合扫描量看回收效率 |
| `%vmeff` | 页回收效率指标 | 不能孤立使用，仍需结合延迟和换页 |

如果请求延迟尖刺与较高的 `pgscand/s` 同时出现，直接内存回收是重要嫌疑；如果 `si`/`so` 也持续升高，则还应考虑物理内存或内存限制确实不足。

### 5.4 查 OOM 证据

```bash
dmesg -T | grep -i -E 'out of memory|oom|killed process'
journalctl -k --since '1 hour ago' | grep -i -E 'out of memory|oom|killed process'
```

发生 OOM 时，要继续确认：

- 是主机全局 OOM，还是容器/cgroup 的内存限制触发。
- 被杀进程是否是最大消费者，还是 `oom_score_adj` 影响了选择。
- 是否存在内存泄漏、缓存上限不合理、批量任务峰值或堆配置过大。

## 6. 网络连接与端口：优先使用 `ss`

旧环境中经常看到 `netstat -napt`，现代 Linux 上通常应优先使用 `ss`：它读取内核 socket 信息更直接，输出快且过滤能力更好。

### 6.1 看监听端口归属

```bash
# TCP 监听端口与所属进程；查看他人进程可能需要足够权限
ss -lntp

# 精确查询 80 端口，不要用 grep 80 误匹配 8080 等值
ss -lntp 'sport = :80'

# 另一种定位打开端口的方式
lsof -nP -iTCP:80 -sTCP:LISTEN
```

诊断“服务启动了但访问不到”时，至少区分三件事：

- 根本没有监听：服务未启动、启动失败或监听了别的端口。
- 只监听 `127.0.0.1`：本机可访问，外部不能直接连接。
- 已监听 `0.0.0.0`/指定网卡地址：继续查防火墙、安全组、路由或上游代理。

### 6.2 看连接状态和队列

```bash
ss -antp
ss -s
ss -ant state established
ss -ant state time-wait
```

常见状态与含义：

| 状态 | 含义 | 排障提示 |
| --- | --- | --- |
| `LISTEN` | 服务正在接受新连接 | 确认监听地址、端口与进程 |
| `ESTABLISHED` | TCP 已连接 | 数量突增时查流量、慢请求、连接池 |
| `SYN-SENT` / `SYN-RECV` | 建连中 | 堆积时查对端可达性、攻击、队列和丢包 |
| `TIME-WAIT` | 主动关闭方等待旧报文过期 | 大量短连接时常见，应结合端口资源和架构判断 |
| `CLOSE-WAIT` | 对端已关闭，本端应用尚未关闭 socket | 堆积通常提示应用未正确释放连接 |

`ss` 输出里的 `Recv-Q` 与 `Send-Q` 也很重要：已连接 socket 上队列持续增长，可能提示应用读写不及时、对端变慢或网络拥塞。

### 6.3 测试可达性不要混同于服务正确

```bash
nc -vz <host> <port>
curl -v http://<host>:<port>/health
```

`nc` 成功只能说明 TCP 建连成立，不证明 HTTP 路由、鉴权、数据库依赖或业务逻辑正确；有协议级健康接口时，应进一步用对应客户端验证。

## 7. 高频故障：从症状到下一步

### 7.1 CPU 跑满

```bash
top
pidstat -u 1
top -H -p <pid>
```

行动链路：

1. 先在 `top` 判断高的是 `us`、`sy`、`wa` 还是 `st`。
2. 用 `pidstat` 或 `top` 锁定持续占用的 PID，而不是只看一次峰值。
3. 对多线程进程找到高 CPU 的 TID。
4. 对 Java 使用 `jstack` 对应 `nid`，对原生程序使用 `perf top` 或短时 `strace`。
5. 结合发布、流量、重试、GC、慢查询等事件找触发条件。

如果 `wa` 很高，问题并不是“业务 CPU 计算太重”，应转向 I/O 与阻塞任务分析。

### 7.2 负载高但 CPU 不高

```bash
uptime
vmstat 1
ps -eo pid,stat,wchan:24,cmd | awk '$2 ~ /D/'
```

重点看：

- `vmstat` 的 `b` 是否高。
- 是否有大量 `D` 状态进程。
- `top` 中 `wa` 是否变高。
- 磁盘、网络存储、挂载点或内核路径是否等待过久。

这类场景最容易被误判为“加 CPU 就能解决”，而实际瓶颈可能在磁盘或远程存储。

### 7.3 内存紧张或周期性卡顿

```bash
free -h
vmstat 1
sar -B 1
ps -eo pid,user,rss,pmem,cmd --sort=-rss
dmesg -T | grep -i -E 'oom|out of memory|killed process'
```

重点是把三类证据区分开：

- 容量不足：`available` 低，消费者 RSS 明显，可能 OOM。
- 换页抖动：`si`/`so` 活跃，I/O 与延迟一起变差。
- 直接回收抖动：`pgscand/s` 高，即使还没 OOM，应用也可能卡顿。

### 7.4 端口不通或连接异常

```bash
ss -lntp 'sport = :<port>'
ss -antp
lsof -nP -iTCP:<port>
nc -vz <host> <port>
```

排查顺序：

1. 本机是否有正确进程监听正确地址和端口。
2. 本机访问是否成功。
3. 远端 TCP 是否能建立。
4. 连接状态是否堆积在 `SYN-RECV`、`CLOSE-WAIT` 等异常区域。
5. 再转向安全组、防火墙、反向代理和业务健康检查。

## 8. 一套可复用的线上观察清单

拿到一台“服务变慢”的机器时，可先做低侵入性的观察：

```bash
# 1. 主机整体趋势
uptime
free -h
vmstat 1

# 2. 任务与资源大户
top
ps -eo pid,ppid,user,stat,pcpu,pmem,rss,cmd --sort=-pcpu

# 3. 服务监听与连接态
ss -lntp
ss -s

# 4. 明确 PID 后，再进入进程内部
top -H -p <pid>
lsof -p <pid>
```

使用时注意两点：

- 连续采样比单点截图更可信。一次瞬时高峰未必是故障，持续和趋势才决定处置优先级。
- 在生产环境附加 `strace`、采集栈或运行高成本性能分析前，要评估开销并缩小目标范围。

## 9. 面试表达：如何回答“Linux 服务器变慢怎么查”

可以按下面的逻辑回答：

> 我会先用 `uptime`、`top`、`free`、`vmstat` 做低侵入性的系统快照，区分是 CPU、I/O、内存还是网络方向。负载高不直接等价于 CPU 高，因为 Linux 负载也包含不可中断等待任务；我会结合 `top` 的 CPU 分布、`vmstat` 的 `r`/`b` 和换页指标判断。若是 CPU 问题，用 `top`/`pidstat` 找 PID，再用 `top -H` 找线程并进入栈或 `perf`；若是内存问题，重点看 `available`、`si`/`so`、回收和 OOM 证据；若是连接问题，优先用 `ss` 看监听地址、连接状态和队列，再定位进程与链路。最后会把诊断结论与发布、流量或依赖事件对应起来，并用同一组指标验证修复效果。

这比只报一串命令更有说服力，因为它表达了：

- 指标的含义
- 工具之间如何串联
- 容易误判的边界
- 从系统证据回到根因和验证的完整过程

## 本课小结

- `ps` 是快照，`top` 是连续观察；进程高 CPU 后往往还要追线程。
- `load average` 包含运行中和不可中断等待任务，不能等同于 CPU 使用率。
- 内存诊断优先看 `available` 与动态换页/回收，而不是仅看 `free`。
- 网络排障优先使用 `ss`，并将监听、连通、连接状态和业务健康分层验证。
- 真正的线上排障不是背命令，而是形成“现象 -> 指标 -> 进程/线程 -> 根因 -> 验证”的证据链。
