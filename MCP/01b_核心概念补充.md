> 本课回答第 1 课遗留的核心疑问，逐一拆解底层概念。

## 一、JSON-RPC 2.0 是什么？为什么选它？

### 先理解 RPC

RPC（Remote Procedure Call，远程过程调用）你肯定熟——就是"调用远程机器上的函数，像调本地函数一样"。你用过的 gRPC、Dubbo、Thrift 都是 RPC 框架。

**JSON-RPC** 是 RPC 的一种极简实现，消息体就是纯 JSON。没有 IDL 文件、没有代码生成、没有二进制序列化。

一个 JSON-RPC 2.0 的请求长这样：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "list_pull_requests",
    "arguments": { "repo": "my-project" }
  }
}
```

对应的响应：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      { "type": "text", "text": "找到 3 个 PR..." }
    ]
  }
}
```

就这么简单。`method` 是你要调的"函数"，`params` 是参数，`id` 用来匹配请求和响应（因为可以并发发多个请求）。

JSON-RPC 还定义了一种**单向通知**（Notification），就是没有 `id` 字段的消息，表示"我通知你一下，不需要你回复"：

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
```

### 为什么 MCP 选 JSON-RPC 而不是别的？

这里做个横向对比，你一看就明白：

| 方案 | 优点 | 为什么 MCP 没选 |
| --- | --- | --- |
| **REST/HTTP** | 你最熟，生态好 | REST 是资源导向的（CRUD），而 MCP 需要的是"调用动作"（RPC 语义）。而且 REST 天然是**单向**的（客户端请求 → 服务端响应），不支持服务端主动推送 |
| **gRPC** | 性能高，有 stream | 依赖 Protobuf 编译、HTTP/2，太重了。MCP 要能跑在 stdio 上（后面解释），gRPC 做不到 |
| **GraphQL** | 查询灵活 | 查询语言太复杂，MCP 的场景不需要这种灵活性 |
| **自定义协议** | 完全可控 | 没有生态，大家要从头学 |
| **JSON-RPC 2.0** ✅ | 极简、语言无关、支持双向、支持通知、已有成熟规范 | —— |

总结一下 MCP 选 JSON-RPC 的核心原因：

1. **极简**——整个规范只有一页纸，任何语言都能在几十行代码内实现解析
2. **传输无关**——JSON-RPC 只定义消息格式，不绑定传输层。同样的消息可以走 stdio、HTTP、WebSocket，MCP 正好需要这种灵活性
3. **天然支持双向**——请求和通知都是 JSON 对象，谁发都行，不像 HTTP 那样角色固定
4. **有 `id` 机制**——支持并发请求，通过 `id` 匹配请求和响应

## 二、stdio 是什么？

### 基础概念

stdio 就是 **Standard I/O（标准输入输出）**，是操作系统层面最基础的进程间通信方式。每个进程启动时自动拥有三个流：

- **stdin**（标准输入，fd=0）：进程从这里读数据
- **stdout**（标准输出，fd=1）：进程往这里写数据
- **stderr**（标准错误，fd=2）：进程往这里写错误/日志

你天天在用：

```bash
# 你在终端里打的每个字 → stdin
# 程序输出的每一行 → stdout
echo "hello" | grep "h"
#    ↑stdout      ↑stdin
# echo 的 stdout 通过管道（pipe）接到了 grep 的 stdin
```

### MCP 怎么用 stdio？

当 MCP Host（比如 Claude Desktop）要连接一个本地的 MCP Server 时：

```
Host 启动 Server 进程（比如 python my_server.py）
    │
    │  Host 往 Server 的 stdin 写 JSON-RPC 消息
    ▼
 ┌──────────────────┐
 │  Server 进程      │
 │  stdin  ← 读请求  │
 │  stdout → 写响应  │
 │  stderr → 写日志  │
 └──────────────────┘
    │
    │  Server 往自己的 stdout 写 JSON-RPC 响应
    ▼
  Host 从 Server 的 stdout 读取响应
```

具体来说，每条消息就是一行 JSON，用换行符 `\n` 分隔：

```
Host → Server stdin:  {"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}\n
Server → Host stdout: {"jsonrpc":"2.0","id":1,"result":{"capabilities":{...}}}\n
```

### 为什么需要 stdio？

**因为简单且安全**。对于本地运行的 MCP Server（和 Host 在同一台机器上）：

- 不需要开端口、不需要网络配置、不需要 TLS
- 操作系统级别的进程隔离，天然安全
- 任何编程语言都能读写 stdin/stdout，零依赖
- Host 启动 Server 进程，结束时直接杀进程，生命周期管理极简

**你现在用的 Claude Code 连接的 MCP Server，大部分就是走 stdio 的。**

## 三、Streamable HTTP 是什么？

stdio 只能用于本地（Host 和 Server 在同一台机器上）。如果 Server 部署在远程服务器上呢？这就需要 Streamable HTTP。

### 基本机制

Streamable HTTP 是 MCP 协议定义的远程传输方式，它基于标准 HTTP，但增加了**服务端流式推送**的能力：

```
MCP Client                          MCP Server（远程）
    │                                     │
    │  POST /mcp (JSON-RPC 请求)          │
    │ ──────────────────────────────────→  │
    │                                     │
    │  响应方式有两种：                      │
    │                                     │
    │  方式1: 普通 JSON 响应               │
    │  ←──── Content-Type: application/json│
    │                                     │
    │  方式2: SSE 流式响应                  │
    │  ←──── Content-Type: text/event-stream
    │  ←──── data: {"jsonrpc":"2.0",...}   │
    │  ←──── data: {"jsonrpc":"2.0",...}   │
    │  ←──── ... (持续推送)                 │
    │                                     │
    │  GET /mcp (建立 SSE 监听通道)         │
    │ ──────────────────────────────────→  │
    │  ←──── Server 主动推送通知            │
```

SSE（Server-Sent Events）是 HTTP 的一个标准特性——服务端可以在一个 HTTP 响应里持续不断地发送数据，客户端保持连接持续接收。你可能在做实时通知、日志流时用过。

### 为什么不直接用 WebSocket？

| | WebSocket | Streamable HTTP |
|---|---|---|
| 基础设施兼容性 | 很多企业防火墙、代理不支持 WS | 标准 HTTP，到处都能通过 |
| 认证 | 需要自己处理 | 复用 HTTP 的 Authorization header |
| 负载均衡 | 有状态连接，LB 配置复杂 | 无状态请求 + 可选有状态流 |
| 灵活性 | 全双工但必须维持长连接 | 简单请求走普通 HTTP，需要流时才用 SSE |

简单说：Streamable HTTP 更"企业友好"，更容易部署到生产环境。

## 四、双向通信怎么做到的？

这是一个很好的问题。传统 HTTP 是单向的——客户端发请求，服务端返响应。MCP 怎么做到双向？

### stdio 模式下的双向通信

最直观，因为 stdin 和 stdout 是两个独立的流：

```
Host → stdin  → Server    （Host 给 Server 发消息）
Host ← stdout ← Server    （Server 给 Host 发消息）
```

两边都可以随时往自己的输出流写消息，不需要等对方先说话。**本质上就是两根管道，各走各的。**

### Streamable HTTP 模式下的双向通信

稍微复杂一点，通过两种 HTTP 方法组合实现：

```
方向1：Client → Server
  Client 发 POST /mcp，Server 返回响应
  （这是标准的 HTTP 请求-响应，没什么特别的）

方向2：Server → Client（主动推送）
  Client 先发 GET /mcp 建立一个 SSE 长连接
  Server 随时通过这个 SSE 连接推送消息给 Client
```

**举个实际场景**：

假设 MCP Server 管理着一个数据库，数据库的表结构变了（有人加了一列），Server 需要主动通知 Client "我的 Resources 列表变了，你要不要重新拉一下"：

```
1. Client 启动时发 GET /mcp，建立 SSE 监听通道
2. （过了一会儿，数据库表结构变了）
3. Server 通过 SSE 通道推送：
   data: {"jsonrpc":"2.0","method":"notifications/resources/list_changed"}
4. Client 收到通知，决定重新调 POST 拉取最新的 Resource 列表
```

所以 MCP 的"双向"并不是像 WebSocket 那样一个连接两边都能发，而是**通过 POST（Client→Server）+ SSE（Server→Client）两条通道组合实现的**。

## 五、Resources 到底是什么？以 Excel 为例

### Resources 的准确理解

Resources 是 MCP Server **暴露给 AI 应用的数据**，通过 URI（统一资源标识符）来标识。

关键点：**Resources 是"应用层面"的概念，不是"LLM 层面"的。** 也就是说，是 Host 应用（或用户）决定要加载哪些 Resource 放进上下文，而不是 LLM 自己决定去读哪个 Resource。

### 用 Excel 的例子说清楚

假设你写了一个 MCP Server 来对接 Excel 文件。这个 Server 可以暴露这样的 Resources：

```json
[
  {
    "uri": "excel:///财务报表.xlsx/Sheet1",
    "name": "财务报表 - Sheet1",
    "description": "2024年Q4财务数据",
    "mimeType": "application/json"
  },
  {
    "uri": "excel:///财务报表.xlsx/Sheet2",
    "name": "财务报表 - Sheet2",
    "description": "2024年Q4预算对比",
    "mimeType": "application/json"
  }
]
```

当 Client 请求读取 `excel:///财务报表.xlsx/Sheet1` 时，Server 会：
1. 打开 Excel 文件
2. 读取 Sheet1 的数据
3. 把数据格式化成 JSON 或文本返回

```json
{
  "contents": [
    {
      "uri": "excel:///财务报表.xlsx/Sheet1",
      "mimeType": "application/json",
      "text": "[{\"月份\":\"10月\",\"收入\":150000,\"支出\":120000}, ...]"
    }
  ]
}
```

**所以你理解得基本对——Resource 就是数据本身（或数据的引用位置），让 LLM 能"看到"这些数据。**

### Resources vs Tools：很多人搞混的地方

| | Resources | Tools |
|---|---|---|
| **语义** | "看"——获取数据 | "做"——执行操作 |
| **类比** | GET /api/users/123 | POST /api/users/123/deactivate |
| **谁决定用** | 应用/用户选择加载 | LLM 自主决定调用 |
| **副作用** | 无（只读） | 可能有（写数据库、发消息、创建文件……） |
| **Excel 例子** | 读取某个 Sheet 的数据 | 往某个 Sheet 写入一行新数据 |

还是用 Excel 的例子：
- **Resource**：`excel:///报表.xlsx/Sheet1` → 读取 Sheet1 的内容给 LLM 看
- **Tool**：`add_row(file="报表.xlsx", sheet="Sheet1", data={...})` → 往 Sheet1 里插入一行

## 六、Prompts 的具体例子

Prompts 是 MCP Server 提供的**预制交互模板**。最好的理解方式是看一个完整例子。

### 例子：代码审查 Prompt

假设你写了一个 MCP Server 对接 GitHub，它除了提供 Tools（创建 PR、合并 PR）和 Resources（代码文件内容），还提供了一个 Prompt 模板：

```json
{
  "name": "review-code",
  "description": "按照团队规范对代码进行全面审查",
  "arguments": [
    {
      "name": "pr_number",
      "description": "要审查的 PR 编号",
      "required": true
    },
    {
      "name": "focus",
      "description": "审查重点：security / performance / readability",
      "required": false
    }
  ]
}
```

当用户在 AI 应用里选择使用这个 Prompt（比如在 Claude Desktop 里输入 `/review-code`），Server 会返回一组预制的消息：

```json
{
  "messages": [
    {
      "role": "user",
      "content": {
        "type": "text",
        "text": "请审查 PR #142 的代码变更。\n\n审查维度：\n1. 安全性：是否有注入、越权、敏感信息泄露\n2. 性能：是否有 N+1 查询、不必要的内存分配\n3. 可读性：命名是否清晰、逻辑是否易懂\n4. 测试覆盖：关键路径是否有测试\n\n请对每个维度给出 ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改 的评级。"
      }
    },
    {
      "role": "user",
      "content": {
        "type": "resource",
        "resource": {
          "uri": "github://pulls/142/diff",
          "text": "（这里是 PR 的 diff 内容，Server 自动嵌入）"
        }
      }
    }
  ]
}
```

**看到了吗？Prompt 做的事情是：**

1. 把用户的简单输入（"审查 PR #142"）展开成一个**结构化的、高质量的 prompt**
2. 自动**附带相关的上下文数据**（PR 的 diff）
3. 确保 LLM 按照**团队规范的审查维度**来工作

**如果没有 Prompt 模板**，用户每次想做代码审查都得自己写一大段话告诉 LLM 要从哪些维度看、输出什么格式。有了 Prompt 模板，一句话搞定，且保证一致性。

### 再给一个更简单的例子

一个 SQL 助手 MCP Server 的 Prompt：

```json
{
  "name": "explain-query",
  "description": "解释 SQL 查询的执行计划",
  "arguments": [
    { "name": "sql", "description": "要解释的 SQL 语句", "required": true }
  ]
}
```

用户选择这个 Prompt 并填入 SQL 后，Server 返回：

```json
{
  "messages": [
    {
      "role": "user",
      "content": {
        "type": "text",
        "text": "请分析以下 SQL 的执行计划，指出潜在的性能问题，并给出优化建议：\n\nSELECT * FROM orders o JOIN users u ON o.user_id = u.id WHERE o.created_at > '2024-01-01'\n\n数据库引擎：PostgreSQL 15\n表统计信息：orders 表约 500 万行，users 表约 50 万行"
      }
    }
  ]
}
```

Server 不仅把 SQL 嵌入了，还自动补充了数据库引擎版本和表统计信息（因为 Server 连着数据库，能查到）。

## 七、Function Calling 是什么？

### 本质

Function Calling（函数调用）是 **LLM 自身的一个能力**——模型能在对话过程中决定"我现在需要调用一个外部函数来获取信息或执行操作"，并生成结构化的调用参数。

### 没有 Function Calling 之前

早期的 LLM 只会生成文本。如果你想让它查天气，你得：

```
你：今天北京天气怎么样？
LLM：我无法访问实时天气数据...（或者瞎编一个）
```

有些人搞了 hack——让 LLM 输出特定格式的文本，然后用正则去解析：

```
你：如果你需要查天气，请输出 ACTION: get_weather(city="北京")
LLM：ACTION: get_weather(city="北京")
你的代码：（用正则解析出来，调 API，把结果塞回去）
```

这种方式很脆弱——LLM 的输出格式不稳定，解析经常出错。

### 有了 Function Calling 之后

2023 年，OpenAI 首先在 GPT 中引入了 Function Calling。你在调 API 时，可以声明一组"可用函数"：

```json
// 你调 LLM API 时传入的 tools 参数
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "查询指定城市的天气",
        "parameters": {
          "type": "object",
          "properties": {
            "city": { "type": "string" }
          }
        }
      }
    }
  ]
}
```

LLM 在推理时，如果认为需要调用某个函数，会返回一个**结构化的 tool_use 响应**（而不是普通文本）：

```json
// LLM 的响应——不是文本，是结构化的函数调用请求
{
  "tool_calls": [
    {
      "id": "call_abc123",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\": \"北京\"}"
      }
    }
  ]
}
```

然后你的代码执行这个函数，把结果传回 LLM，LLM 基于结果生成最终回答。

### Function Calling 和 MCP 的关系

**Function Calling 是"大脑的决策"，MCP 是"手脚的连接"。**

```
用户："帮我看看 GitHub 上的 PR"

┌─────────────────────────────────────────┐
│           LLM（大脑）                     │
│                                          │
│  "我需要调 list_pull_requests 工具"       │
│   ↑ 这是 Function Calling 的能力         │
│   （模型知道什么时候该调什么函数）          │
└──────────────┬───────────────────────────┘
               │ 返回 tool_use
               ▼
┌─────────────────────────────────────────┐
│           Host 应用                      │
│   "好，LLM 要调 list_pull_requests"      │
│   "这个工具在 GitHub MCP Server 上"       │
│    → 通过 MCP 协议发送 JSON-RPC 请求      │
│   ↑ 这是 MCP 的职责                      │
│   （应用知道怎么连到工具并调用它）          │
└──────────────┬───────────────────────────┘
               │ MCP (JSON-RPC over stdio)
               ▼
┌─────────────────────────────────────────┐
│         MCP Server (GitHub)              │
│   执行 list_pull_requests                │
│   调用 GitHub API，返回结果               │
└─────────────────────────────────────────┘
```

**没有 Function Calling**：LLM 不知道自己能调函数，只会生成文本
**没有 MCP**：LLM 知道要调函数，但每个应用都要自己写一套连接外部工具的代码
**两者结合**：LLM 决策 + MCP 连接 = 完整的 AI Agent 能力

## 小结

1. **JSON-RPC 2.0**：极简的 RPC 协议，消息就是 JSON，传输无关，天然支持双向通信和并发——正好适合 MCP "轻量 + 灵活"的需求
2. **stdio**：进程间通过 stdin/stdout 传递 JSON 消息，用于本地 MCP 通信，简单安全无需网络配置
3. **Streamable HTTP**：POST 发请求 + SSE 收推送，用于远程 MCP 通信，企业友好
4. **双向通信**：stdio 靠两根管道各走各的；HTTP 靠 POST + SSE 两条通道组合
5. **Resources**：Server 暴露的只读数据（通过 URI 标识），应用/用户决定加载哪些——你 Excel 的理解基本正确
6. **Prompts**：预制的高质量交互模板，把简单输入展开成结构化 prompt 并自动附带上下文
7. **Function Calling**：LLM 自身"决定调什么函数"的能力，MCP 负责"实际连到函数并调用"——二者互补
