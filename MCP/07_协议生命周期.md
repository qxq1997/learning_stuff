# MCP - 第 7 课：MCP 协议生命周期——初始化、能力协商与会话管理

1. 能画出 MCP 连接从建立到关闭的完整状态机

## 一、协议生命周期全景

一个 MCP 连接从建立到关闭，经历三个阶段。用状态机来表示：

```
┌──────────────┐
│              │
│  未连接       │ ← 初始状态
│              │
└──────┬───────┘
       │ Client 发起连接（启动进程 / HTTP 连接）
       ▼
┌──────────────┐     initialize 请求
│              │ ──────────────────────→ Server
│  初始化中     │
│ Initializing │ ←────────────────────── Server
│              │     initialize 响应
└──────┬───────┘
       │ Client 发送 initialized 通知
       ▼
┌──────────────┐
│              │
│  运行中       │ ← 正常工作状态
│  Running     │   可以调用 tools/list, tools/call,
│              │   resources/read, prompts/get 等
└──────┬───────┘
       │ 任一方发起关闭 / 连接断开
       ▼
┌──────────────┐
│              │
│  已关闭       │
│  Closed      │
│              │
└──────────────┘
```

**关键规则：**

- **初始化之前不能做任何其他操作**——Client 不能在 initialize 完成之前调 `tools/list`，Server 也不能在收到 `initialized` 之前发通知
- **初始化只做一次**——一个连接只握手一次，不能重新 initialize
- **任一方都可以发起关闭**——Client 可以关闭连接，Server 也可以

## 二、初始化阶段：三步握手

### 2.1 完整的握手流程

```
Client                                          Server
  │                                                │
  │  ════════ 第一步：initialize 请求 ════════      │
  │                                                │
  │  {                                             │
  │    "jsonrpc": "2.0",                           │
  │    "id": 1,                                    │
  │    "method": "initialize",                     │
  │    "params": {                                 │
  │      "protocolVersion": "2025-03-26",          │
  │      "capabilities": {                         │
  │        "roots": {                              │
  │          "listChanged": true                   │
  │        },                                      │
  │        "sampling": {}                          │
  │      },                                        │
  │      "clientInfo": {                           │
  │        "name": "my-ai-app",                    │
  │        "version": "2.1.0"                      │
  │      }                                         │
  │    }                                           │
  │  }                                             │
  │ ─────────────────────────────────────────────→ │
  │                                                │
  │  ════════ 第二步：initialize 响应 ════════      │
  │                                                │
  │  {                                             │
  │    "jsonrpc": "2.0",                           │
  │    "id": 1,                                    │
  │    "result": {                                 │
  │      "protocolVersion": "2025-03-26",          │
  │      "capabilities": {                         │
  │        "tools": { "listChanged": true },       │
  │        "resources": {                          │
  │          "subscribe": true,                    │
  │          "listChanged": true                   │
  │        },                                      │
  │        "prompts": { "listChanged": true },     │
  │        "logging": {}                           │
  │      },                                        │
  │      "serverInfo": {                           │
  │        "name": "github-mcp-server",            │
  │        "version": "0.5.2"                      │
  │      }                                         │
  │    }                                           │
  │  }                                             │
  │ ←───────────────────────────────────────────── │
  │                                                │
  │  ════════ 第三步：initialized 通知 ════════     │
  │                                                │
  │  {                                             │
  │    "jsonrpc": "2.0",                           │
  │    "method": "notifications/initialized"       │
  │  }                                             │
  │ ─────────────────────────────────────────────→ │
  │                                                │
  │  ══════ 握手完成，进入 Running 状态 ══════      │
  │                                                │
```

**为什么是三步而不是两步？**

和 TCP 三次握手类似：

```
TCP 三次握手：                      MCP 三步初始化：
SYN      →                         initialize 请求  →
         ← SYN+ACK                                  ← initialize 响应
ACK      →                         initialized 通知 →

第三步的作用：
TCP: 告诉 Server "我收到你的确认了"    MCP: 告诉 Server "我处理完你的
     双方都确认对方在线                      能力声明了，你可以开始工作了"
```

如果没有第三步，Server 不知道 Client 是否已经处理完它的响应。比如 Client 可能需要根据 Server 的能力声明做一些内部初始化（比如注册通知监听器），这些都要在 `initialized` 通知之前完成。

### 2.2 版本协商

`protocolVersion` 是关键字段——Client 和 Server 可能支持不同版本的 MCP 协议。

```
协商规则：

Client 发送：protocolVersion = "2025-03-26"（Client 支持的最新版本）

Server 决定：
  ├── Server 也支持 "2025-03-26"
  │   → 响应 protocolVersion = "2025-03-26" ✅ 达成一致
  │
  ├── Server 不支持 "2025-03-26"，但支持更早的 "2024-11-05"
  │   → 响应 protocolVersion = "2024-11-05" ✅ 降级到双方都支持的版本
  │
  └── Server 不支持任何 Client 能接受的版本
      → 返回 JSON-RPC error ❌ 连接失败
```

**Client 收到响应后，也要检查**：如果 Server 返回的版本比 Client 能支持的最低版本还低，Client 应该断开连接。

这和 TLS 的版本协商一模一样——Client 说"我支持 TLS 1.3"，Server 说"我只支持 TLS 1.2"，双方降级到 1.2；如果 Server 只支持 SSL 3.0（太老了），Client 拒绝连接。

### 2.3 能力声明详解

初始化时双方交换的 `capabilities` 是整个协商的核心。我们把所有可能的能力整理到一张表：

**Client 可以声明的能力：**

```json
{
  "capabilities": {
    "roots": {                    // "我可以告诉你我的工作目录"
      "listChanged": true         //   "目录变了我会通知你"
    },
    "sampling": {}                // "你可以请求我帮你调 LLM"
  }
}
```

| Client 能力 | 含义 | Server 可以做什么 |
| --- | --- | --- |
| `roots` | Client 可以提供工作目录列表 | Server 可以调 `roots/list` 获取 Client 的工作目录 |
| `roots.listChanged` | Client 的工作目录变了会通知 | Server 可以监听 `notifications/roots/list_changed` |
| `sampling` | Client 支持 Sampling 请求 | Server 可以请求 Client 帮忙调 LLM（第 10 课详讲） |

**Server 可以声明的能力：**

```json
{
  "capabilities": {
    "tools": {                    // "我有 Tools 可用"
      "listChanged": true         //   "Tool 列表可能动态变化"
    },
    "resources": {                // "我有 Resources 可用"
      "subscribe": true,          //   "你可以订阅 Resource 变更"
      "listChanged": true         //   "Resource 列表可能动态变化"
    },
    "prompts": {                  // "我有 Prompts 可用"
      "listChanged": true         //   "Prompt 列表可能动态变化"
    },
    "logging": {}                 // "我支持日志级别控制"
  }
}
```

| Server 能力 | 含义 | Client 可以做什么 |
| --- | --- | --- |
| `tools` | 有 Tools | Client 可以调 `tools/list` 和 `tools/call` |
| `tools.listChanged` | Tool 列表会动态变 | Client 应监听 `notifications/tools/list_changed` |
| `resources` | 有 Resources | Client 可以调 `resources/list` 和 `resources/read` |
| `resources.subscribe` | 支持订阅 | Client 可以调 `resources/subscribe` |
| `resources.listChanged` | Resource 列表会动态变 | Client 应监听 `notifications/resources/list_changed` |
| `prompts` | 有 Prompts | Client 可以调 `prompts/list` 和 `prompts/get` |
| `prompts.listChanged` | Prompt 列表会动态变 | Client 应监听 `notifications/prompts/list_changed` |
| `logging` | 支持日志控制 | Client 可以调 `logging/setLevel` 设置日志级别 |

**协商的核心逻辑图：**

```
Client 声明的能力                Server 声明的能力
┌─────────────────┐            ┌─────────────────┐
│ roots ✓          │            │ tools ✓          │
│ sampling ✓       │            │ resources ✓      │
└────────┬────────┘            │ prompts ✓        │
         │                     │ logging ✓        │
         │                     └────────┬────────┘
         │                              │
         ▼                              ▼
    ┌─────────────────────────────────────┐
    │        双方根据对方的能力决定行为      │
    │                                     │
    │  Client 看到 Server 有 tools        │
    │  → Client 会调 tools/list           │
    │                                     │
    │  Client 看到 Server 没有 prompts    │
    │  → Client 不会调 prompts/list       │
    │                                     │
    │  Server 看到 Client 有 sampling     │
    │  → Server 可能会请求 Sampling       │
    │                                     │
    │  Server 看到 Client 没有 roots      │
    │  → Server 不会调 roots/list         │
    └─────────────────────────────────────┘
```

## 三、运行阶段：请求、通知与心跳

### 3.1 请求与通知的区别

进入 Running 状态后，双方通过两种消息类型通信：

```
请求（Request）：                     通知（Notification）：
有 id，期待响应                       没有 id，不期待响应

Client → Server:                     Client → Server:
{                                    {
  "jsonrpc": "2.0",                    "jsonrpc": "2.0",
  "id": 5,              ← 有 id       "method": "notifications/..."
  "method": "tools/call",             }
  "params": { ... }                   ← 没有 id
}                                     ← 不需要回复
Server → Client:
{
  "jsonrpc": "2.0",
  "id": 5,              ← 匹配的 id
  "result": { ... }
}
```

**MCP 中所有的方法分类：**

```
Client → Server 的请求（需要响应）：
├── initialize
├── tools/list
├── tools/call
├── resources/list
├── resources/read
├── resources/subscribe
├── resources/unsubscribe
├── resources/templates/list
├── prompts/list
├── prompts/get
├── logging/setLevel
├── completion/complete
└── ping

Client → Server 的通知（不需要响应）：
├── notifications/initialized
├── notifications/cancelled         ← 取消一个正在执行的请求
├── notifications/roots/list_changed
└── notifications/progress          ← 报告进度（虽然通常是 Server 发的）

Server → Client 的请求（需要响应）：
├── sampling/createMessage          ← 请求 Client 帮忙调 LLM
├── roots/list                      ← 请求 Client 的工作目录列表
└── ping

Server → Client 的通知（不需要响应）：
├── notifications/tools/list_changed
├── notifications/resources/list_changed
├── notifications/resources/updated
├── notifications/prompts/list_changed
├── notifications/progress
├── notifications/message           ← Server 发日志消息
└── notifications/cancelled
```

### 3.2 请求取消（Cancellation）

如果 Client 发了一个请求，但不想等了（比如用户取消了操作），可以发一个取消通知：

```
Client                                    Server
  │                                          │
  │  ── tools/call (id=7) ────────────→     │
  │      "run_query: SELECT ... 复杂查询"    │  Server 开始执行...
  │                                          │
  │  （用户等了 10 秒，不耐烦了，点击取消）    │
  │                                          │
  │  ── notifications/cancelled ──────→      │
  │  { "params": { "requestId": 7 } }       │
  │                                          │  Server 收到取消通知
  │                                          │  尝试中止查询（尽力而为）
  │                                          │
  │       ←── error 响应 ──────────────      │
  │  { "id": 7, "error": {                  │
  │      "code": -32800,                     │
  │      "message": "Request cancelled"      │
  │  }}                                      │
```

**注意**：取消是"尽力而为"的——Server 收到取消通知时，操作可能已经执行完了。这和你在后端做的"优雅取消"（比如 Go 的 `context.Cancel()`）是同一个思路。

### 3.3 进度报告（Progress）

对于长时间运行的操作，Server 可以发送进度通知：

```
Client                                    Server
  │                                          │
  │  ── tools/call (id=8) ────────────→     │
  │  {                                       │
  │    "params": {                           │
  │      "name": "import_csv",               │
  │      "arguments": { "file": "big.csv" }, │
  │      "_meta": {                          │
  │        "progressToken": "prog-001"       │  ← Client 提供 progressToken
  │      }                                   │     表示"我想收到进度更新"
  │    }                                     │
  │  }                                       │
  │                                          │
  │       ←── progress 通知 ───────────      │
  │  { "method": "notifications/progress",   │
  │    "params": {                           │
  │      "progressToken": "prog-001",        │
  │      "progress": 2500,                   │
  │      "total": 10000,                     │   ← 25% 完成
  │      "message": "已导入 2500/10000 行"    │
  │  }}                                      │
  │                                          │
  │       ←── progress 通知 ───────────      │
  │  { "params": {                           │
  │      "progressToken": "prog-001",        │
  │      "progress": 7500,                   │   ← 75% 完成
  │      "total": 10000                      │
  │  }}                                      │
  │                                          │
  │       ←── 最终响应 ────────────────      │
  │  { "id": 8, "result": {                  │   ← 100% 完成
  │    "content": [{ "type": "text",         │
  │      "text": "成功导入 10000 行" }]       │
  │  }}                                      │
```

### 3.4 Ping/Pong 心跳

MCP 支持双向的 Ping/Pong 机制来检测连接是否存活：

```
任一方都可以发 ping：

Client → Server:
  { "jsonrpc": "2.0", "id": 99, "method": "ping" }

Server → Client:
  { "jsonrpc": "2.0", "id": 99, "result": {} }

如果 ping 超时没有收到 pong（result），
发送方应该认为连接已断开，进行重连或清理。
```

**用途：**
- **检测死连接**：特别是 stdio 模式下，Server 进程可能已经崩溃但管道还没断
- **保持连接活跃**：HTTP 模式下，防止 SSE 长连接被中间代理（Nginx、ALB）因超时断开

## 四、关闭阶段

### 4.1 正常关闭

```
方式 1：Client 主动关闭

  Client 发送 close 请求（或直接关闭传输层连接）
  → Server 清理资源
  → 连接关闭

方式 2：Server 主动关闭

  Server 发送 close 请求（或直接关闭传输层连接）
  → Client 清理资源
  → 连接关闭
```

### 4.2 异常关闭

```
stdio 模式：
  ├── Server 进程崩溃 → Client 读到 EOF → 检测到连接断开
  ├── Host 进程崩溃 → Server 读到 EOF → Server 应退出
  └── Server 挂起（死循环/死锁） → Client ping 超时 → 强制 kill 进程

HTTP 模式：
  ├── Server 不可达 → HTTP 连接失败 → Client 重试或放弃
  ├── SSE 连接断开 → Client 重新 GET 建立 SSE → 带 Last-Event-ID 续传
  └── Session 过期 → Server 返回 404 → Client 重新 initialize
```

### 4.3 HTTP 模式的断线重连

```
正常状态：
Client ── GET /mcp (SSE) ──→ Server
         ←── 持续推送通知 ──

网络抖动，SSE 断开：
Client ── 检测到连接断开 ──

重连：
Client ── GET /mcp ──────────→ Server
          Mcp-Session-Id: sess-abc123
          Last-Event-ID: evt-042        ← 告诉 Server 上次收到的最后一条消息

Server 检查：
  ├── Session 还在 → 从 evt-042 之后继续推送 ✅
  └── Session 过期 → 返回 404 → Client 需要重新 initialize ❌
```

## 六、常见问题排查

作为一个后端工程师，你早晚会遇到 MCP 连接问题。以下是常见问题和排查思路：

| 现象 | 可能的原因 | 排查方法 |
| --- | --- | --- |
| Server 连不上 | 进程启动失败（依赖缺失、路径错误） | 手动运行 Server 命令看报错 |
| 初始化卡住 | Server 的 stdout 有非 JSON 输出（第三课的坑2） | 检查 Server 的 print/log 输出目标 |
| 初始化失败 | 版本不兼容 | 检查双方的 protocolVersion |
| Tool 调用超时 | Server 执行太慢 / 死锁 | 给 Server 加 stderr 日志，检查外部 API 响应时间 |
| 收不到通知 | Server 没声明 listChanged / Client 没监听 | 检查初始化时的 capabilities |
| SSE 频繁断开 | 中间代理超时（Nginx proxy_read_timeout） | 调大超时 / 加 ping 保活 |
| Session 丢失 | Server 重启 / Session 过期 | Client 需要处理 404 并重新 initialize |

## 小结

1. **生命周期三阶段**：初始化（三步握手）→ 运行（请求/通知/心跳）→ 关闭（正常或异常）
2. **三步握手**：initialize 请求 → initialize 响应 → initialized 通知。和 TCP 三次握手同理，确保双方都准备好
3. **版本协商**：Client 提出最新版本，Server 可以降级到双方都支持的版本
4. **能力协商**：双方声明各自支持的特性，后续行为以协商结果为准——不调没声明的接口
5. **运行阶段的消息类型**：请求（有 id，要响应）和通知（无 id，不响应）。支持取消、进度报告、心跳
6. **关闭与重连**：stdio 靠进程生命周期管理，HTTP 靠 Session + Last-Event-ID 实现断线续传
