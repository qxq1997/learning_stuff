# MCP - 第 11b 课：MCP 生态补充 Q&A

## Q1：为什么需要下载 Server？Server 不是服务提供方维护的吗？

### 两种部署模式

MCP Server 有两种部署模式，只有本地模式需要下载：

```
模式 1：本地 Server（stdio）—— 需要下载到本机
─────────────────────────────────────────────

你的电脑
┌─────────────────────────────────────────────┐
│                                              │
│  Host (Claude Desktop)                       │
│    │                                         │
│    │ stdio（进程间管道）                      │
│    ▼                                         │
│  Server 进程（跑在你电脑上）                   │
│    │                                         │
│    │ HTTPS                                   │
│    ▼                                         │
│  ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄              │
└──│───────────────────────────────────────────┘
   │ 出站请求
   ▼
  GitHub API / 数据库 / 其他外部服务

Server 下载到你本机 → Host 启动它作为子进程 → 通过 stdin/stdout 通信
Server 本身只是一个"翻译层"，把 MCP 协议翻译成具体的 API 调用


模式 2：远程 Server（Streamable HTTP）—— 不需要下载
─────────────────────────────────────────────

你的电脑                          云端
┌──────────────────┐            ┌──────────────────┐
│                   │            │                   │
│  Host             │   HTTPS   │  Server           │
│  (Claude Desktop) │ ────────→ │  (服务提供方部署)   │
│                   │            │    │               │
└──────────────────┘            │    │ 内部调用       │
                                │    ▼               │
                                │  数据库/API/...    │
                                └──────────────────┘

你什么都不用装，只需要在 Host 配置一个 URL
```

### 为什么目前大多数 Server 是本地模式？

```
1. MCP 诞生才一年多（2024年11月发布）
   远程 Server 的基础设施（认证、托管、计费）还在建设中

2. 本地 Server 更简单
   ├── 不需要部署服务器
   ├── 不需要认证/鉴权（进程隔离天然安全）
   ├── 不需要付费（除了外部 API 的费用）
   └── npx 一行命令就能跑

3. 敏感数据更安全
   ├── API Key 留在本地，不经过第三方
   └── 数据不出你的电脑
```

### 未来趋势：远程 Server 将成为主流

```
现在（2024-2025）：                    未来（2026+）：

大部分是本地 Server                    越来越多远程 Server
npm install → 本地运行                 服务商直接提供 MCP 端点

配置示例（本地）：                      配置示例（远程）：
{                                     {
  "github": {                           "github": {
    "command": "npx",                     "url": "https://mcp.github.com/v1",
    "args": ["@mcp/server-github"],       "headers": {
    "env": {                                "Authorization": "Bearer ghp_xxx"
      "GITHUB_TOKEN": "ghp_xxx"            }
    }                                     }
  }                                     }
}                                     }

你本地装 Server + 配 Token              GitHub 官方提供 MCP 端点
Server 用你的 Token 调 GitHub API       你只需要配 URL 和认证信息
```

### 什么时候需要/不需要下载 Server

| 场景 | 需要下载 Server 吗 | 例子 |
| --- | --- | --- |
| 服务商提供了远程 MCP 端点 | ❌ 不需要，配 URL 即可 | 未来的 `mcp.github.com` |
| 服务商没提供 MCP 端点（目前大多数情况） | ✅ 需要下载，本地运行 | 现在的 `npx @mcp/server-github` |
| 自己的内部系统 | ✅ 需要自己开发 Server | 你的内部 API、数据库 |
| 操作本地资源（文件系统、本地 Git） | ✅ 必须本地 Server | 文件读写、本地数据库 |

### 结论

长远来看，就像 REST API 一样，服务提供方会直接暴露 MCP 端点，用户只需要配置连接信息。现在的"本地安装 Server"只是生态早期的过渡形态。但操作本地资源（文件系统、本地 Git）的场景下，本地 Server 会一直存在。
