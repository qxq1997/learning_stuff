# 01 Machine Coding 是什么，凭什么单独拉一轮

## 一句话定义

Machine Coding（也叫 LLD coding round、低代码量系统实现轮）：**给你一个小而真实的需求，限时 60～120 分钟，现场写出一套能编译、能跑、结构清晰、容易扩展的面向对象代码，并能当场演示它工作。**

注意每个词都有分量：

- **小而真实**：不是 LeetCode 的脑筋急转弯，是「设计一个停车场 / 分账 App / 电梯」这种你工作里真会遇到的小系统
- **限时现场**：和算法题一样要在压力下写，但产出的不是一个函数，是一组类
- **能编译能跑**：这是底线（table stakes）。一份设计精妙但跑不起来的代码，在这一轮基本判负
- **结构清晰、容易扩展**：这是真正的得分点。面试官在问的潜台词是「这段代码交给团队，半年后有人能轻松改吗」
- **能演示**：你得自己写一个 `main`/`Driver`，跑出可观察的输出，证明它真的工作

## 凭什么单独拉一轮：三类编码面试的分工

很多人把 machine coding 和算法题、系统设计混为一谈，一上来就答错频道。先把三者钉死：

| 维度 | DSA 算法轮 | **Machine Coding（LLD）** | System Design（HLD） |
| --- | --- | --- | --- |
| 时长 | 30–45 min | **60–120 min** | 45–60 min |
| 产出物 | 一个函数 / 一段算法 | **一组能跑的类（一个小系统）** | 框图 + 口头权衡，通常不写代码 |
| 核心考点 | 数据结构、复杂度、边界 | **OOD、SOLID、可扩展、clean code** | 容量估算、组件选型、一致性/可用性权衡 |
| 规模 | 单机、单函数 | **单机、内存态、一个进程内** | 分布式、多服务、跨机房 |
| 要跑起来吗 | 通常要过测试用例 | **必须能编译能跑能演示** | 不写代码，不需要跑 |
| 典型题 | 两数之和、LRU、拓扑排序 | **停车场、Splitwise、电梯、限流器** | 设计 Twitter、短链、秒杀 |
| 答错频道的样子 | —— | 把停车场答成「要不要分库分表、QPS 多少」 | 把短链答成现场写哈希函数实现 |

关键结论：**machine coding 卡在 DSA 和 HLD 中间的空档。** DSA 测不出你能不能写出别人愿意维护的代码；HLD 测不出你能不能把设计真正落地成代码。Machine coding 专测一件事——**给你一个小需求，你能不能现场写出工程上干净、可扩展的实现。** 这恰恰是日常工作里占比最大、却最难在前两轮被考察到的能力，所以产品型大厂要单拎一轮。

> 它和你笔记里「设计模式」「系统设计」两个专题是互补的：设计模式给你工具，系统设计给你架构视野，machine coding 是把这两者**压进 90 分钟、落成能跑的代码**的实战考场。

## 面试官的评分表，拆成一张可执行清单

machine coding 不是「写完就行」，面试官手里有一张 rubric。把它翻译成你写代码时**主动去命中**的检查项：

| # | 评分项 | 潜台词 | 你要做的具体动作 |
| --- | --- | --- | --- |
| 1 | 能编译能跑（底线） | 「这是不是个能工作的东西」 | 留够时间真的跑一遍，别到最后还在编译报错 |
| 2 | 需求覆盖与边界 | 「核心场景都实现了吗，想到异常了吗」 | 先实现 MVP 主流程，再补 2～3 个关键边界（满了、找不到、重复） |
| 3 | 职责单一 / 模块化 | 「有没有 God class，逻辑该在的地方在吗」 | 实体、策略、门面、存储分开；别把所有逻辑塞 `main` 或一个 Manager |
| 4 | 可扩展（OCP） | 「加个新类型要不要改一堆旧代码」 | 把会变的维度（车型/计费/拆分方式）抽成接口，新增 = 加新类 |
| 5 | 模式恰当、不过度 | 「该用模式的地方用了，没乱秀」 | 策略/工厂/状态用在刀刃上；不为一个 if 包五个接口 |
| 6 | 命名与可读性 | 「不看注释能读懂吗」 | 名词做类名、动词做方法名、enum 替代魔法值、方法短小 |
| 7 | 输入校验与异常 | 「乱输入会不会崩，异常清不清晰」 | 自定义异常（`ParkingFullException`）而不是返回 `null`/魔数 |
| 8 | 并发（若题目涉及） | 「多线程会不会出问题」 | 共享状态加锁或用并发容器；不涉及就别强行加，但要能说出加在哪 |
| 9 | 可演示 | 「你能证明它跑通吗」 | 一个 driver：happy path + 几个边界，打印可观察输出 |

**记忆口诀：能跑 > 覆盖 > 干净 > 可扩展 > 优化。** 顺序很重要——一个跑得通、结构清楚的最小系统，永远赢过一个抽象漂亮但跑不起来的半成品。

## 高频致命失分点（这一轮挂掉的人，多半栽在这里）

1. **跑不起来**：最后 10 分钟还在编译报错。底线没守住，前面写得再漂亮也白搭
2. **逻辑全塞 `main`**：`main` 里几百行 if-else，没有任何类设计。这一轮直接判「不会面向对象」
3. **God class**：一个 `ParkingLot` 类既造车、又算钱、又管车位、又打印。职责单一这栏直接 0 分
4. **过度设计**：停车场题硬上抽象工厂 + 装饰器 + 观察者 + 反射配置，时间烧光，主流程没跑通。machine coding 最忌「为模式而模式」
5. **没有 demo**：写完一堆类，没有 driver，无法证明它工作。等于交了一份「看起来对」的代码
6. **纠结算法细节**：在「车位分配用什么数据结构最优」上纠结 20 分钟,把可扩展和 demo 的时间烧掉。这一轮不考算法最优
7. **全程不出声**：闷头写 40 分钟。面试官看不到你的思路,也没机会帮你纠偏。machine coding 也是沟通题

> 把这 7 条贴在显示器边上。每次练习结束,对照它做复盘——你失分几乎一定落在这 7 条里。

## 90 分钟时间预算（按这个切,别凭感觉）

以最常见的 90 分钟为例:

| 阶段           | 时间     | 做什么                                        | 红线                    |
| ------------ | ------ | ------------------------------------------ | --------------------- |
| 需求澄清         | 5 min  | 问清范围,写下 in/out scope 和假设,定 MVP             | 别超过 8 分钟,问太多显得抓不住重点   |
| 建模与接口        | 10 min | 列实体、enum、找变化点、画轻量类图、定门面方法签名                | 画在纸上/注释里即可,别画精美 UML   |
| 范围确认         | 5 min  | 跟面试官对一遍:「我先实现 A/B/C,D 留作扩展,可以吗」            | 拿到「可以」再动手,避免方向跑偏      |
| 编码           | 50 min | 按 enum → model → 接口 → 策略 → 门面 → driver 顺序写 | 先写主流程跑通,再补边界,别一开始追求完美 |
| demo + 自测    | 10 min | 跑 driver,补 2～3 个边界用例,修跑出来的 bug             | 一定要留出这段,跑通是底线         |
| buffer + 讲扩展 | 10 min | 修剩余问题,主动讲「加新车型只动哪里」                        | 用扩展性收尾,把 OCP 这栏的分拿满   |

核心纪律一句话:**先让主流程能跑通,再让它好扩展,最后才谈优化。** 任何时候发现时间不够,砍的永远是「优化」和「锦上添花的边界」,绝不砍「能跑」和「主流程覆盖」。
