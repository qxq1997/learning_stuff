# 02 通用解题框架：90 分钟从需求到可运行代码

这一章给你一套**可复用的七步法**。machine coding 的题千变万化（停车场、分账、电梯、限流器……），但解题路径几乎不变。把这条路径练成肌肉记忆，你拿到任何新题都不会慌——只是往同一个模板里填不同的实体和策略。

贯穿全章,我用一个迷你例子演示每一步:**图书馆借还书的一个切片**(只做「会员借书 / 还书 / 超期罚款」)。它足够小,能完整展示七步,又不抢第 3 章停车场实战的戏。

---

## 第 1 步:需求澄清与圈定范围(5 min)

拿到题先别写代码,先把模糊需求收敛成一个**明确的 MVP**。machine coding 的题目永远是故意说得很泛的,面试官就是要看你会不会问、会不会砍。

要问 / 要写下来三样东西:

1. **in scope(本轮要做)**:核心实体和核心动作。例:会员能借书、能还书、超期要算罚款
2. **out of scope(明确不做)**:把发散的需求挡在外面。例:不做预约排队、不做多副本库存、不做支付对接、不做持久化(纯内存)
3. **assumptions(假设)**:把没人回答的细节用合理假设钉死。例:一个会员最多借 5 本;罚款 = 超期天数 × 1 元;书用 ISBN 唯一标识

> 动作要点:把这三样**写在代码顶部的注释里或纸上**,然后花 1 句话跟面试官对齐:「我先做借/还/罚款三条主线,预约和支付留作扩展,可以吗?」拿到「可以」再动手。
>
> 这一步直接对应评分表的「需求覆盖与边界」,也避免了最惨的失分——**方向跑偏写了半天不是面试官想要的**。

迷你例子定稿:

```
In scope : addBook / registerMember / borrow / returnBook / 超期罚款
Out      : 预约、库存多副本、支付、持久化
Assume   : 一人最多借 5 本;罚款 1 元/天;超期 = 还书日 - 应还日
```

---

## 第 2 步:名词 → 实体 → 类(建模,约 4 min)

把需求里的**名词**圈出来,它们就是你的候选类。再区分两种:

- **实体(Entity)**:有身份、有生命周期、状态会变。例:`Member`(谁借了几本)、`Book`(借出/在馆)、`BorrowRecord`(借阅记录)
- **值对象 / 枚举(Value / Enum)**:用来描述、本身没身份。把所有「有限取值」做成 `enum`,替代魔法字符串。例:`BookStatus { AVAILABLE, BORROWED }`

迷你例子的实体:

```java
enum BookStatus { AVAILABLE, BORROWED }

class Book {
    String isbn;
    String title;
    BookStatus status = BookStatus.AVAILABLE;
}

class Member {
    String id;
    String name;
    List<BorrowRecord> activeRecords = new ArrayList<>();
}

class BorrowRecord {
    Book book;
    Member member;
    LocalDate borrowDate;
    LocalDate dueDate;
}
```

> 建模心法:**名词做类名,有限取值做 enum。** 看到「状态/类型/等级」这种词,几乎一定是 enum——这是消灭魔法值、提高可读性最便宜的动作。

---

## 第 3 步:动词 → 行为 → 方法归属(约 4 min)

把需求里的**动词**圈出来:借、还、算罚款、登记。每个动词是一个方法,关键问题是——**它该放在哪个类上?**

这一步决定你会写出贫血模型还是充血模型,也是 God class 的高发区。两个原则:

- **行为跟着数据走**:操作哪个对象的状态,方法就尽量放在那个对象上。`book.markBorrowed()` 比在 Manager 里 `book.setStatus(BORROWED)` 更内聚
- **跨实体的协调逻辑,放门面/Service**:`borrow(memberId, isbn)` 涉及查会员、查书、改状态、建记录——这种编排放在 `LibraryService`(门面),而不是塞进 `Book` 或 `Member`

反面教材(God class,直接失分):

```java
// ❌ 所有逻辑塞一个 Manager,Book/Member 沦为纯数据袋
class LibraryManager {
    void borrow(...) { /* 查会员、查书、改状态、算上限、建记录,全在这 */ }
    double calculateFine(...) { ... }
    void addBook(...) { ... }
    // ... 越长越像一个上帝
}
```

正确的职责切分:

```java
// 实体管自己的状态
class Book    { void markBorrowed(){ status = BORROWED; } void markReturned(){ status = AVAILABLE; } }
class Member  { boolean canBorrowMore(){ return activeRecords.size() < MAX; } }
// 门面只做"编排",把动作派发给实体和策略
class LibraryService { BorrowRecord borrow(...){...} double returnBook(...){...} }
```

## 第 4 步:找变化点 → 抽接口(OCP 的核心动作,约 3 min)

这是 machine coding 拉开差距的一步,也是评分表「可扩展」那一栏的来源。问自己一句话:

> **这个需求里,哪些维度将来最可能变?**

把「会变的维度」抽成接口,让新增 = 加一个新实现类,而不是改老代码(开闭原则 OCP)。识别变化点的信号词:**「按……不同」「支持多种……」「将来可能加……」。**

machine coding 里最高频的四类变化点,几乎每道题都能套:

| 变化点信号 | 抽成什么接口 | 典型题 |
| --- | --- | --- |
| 「按不同规则**计算/收费**」 | `FeeStrategy` / `PricingStrategy` | 停车场计费、打车计价 |
| 「按不同规则**分配/选择**」 | `AllocationStrategy` | 停车场选车位、电梯调度 |
| 「按不同方式**拆分/处理**」 | `SplitStrategy` | Splitwise 均摊/精确/百分比 |
| 「**生命周期状态流转**」 | `State` 接口或状态机 | 订单、电梯、售货机 |

迷你例子里的变化点:**罚款规则**(现在是 1 元/天,将来可能 VIP 免罚、超过 30 天封顶)。抽成策略:

```java
interface FineStrategy {
    double calculate(long overdueDays);
}
class PerDayFineStrategy implements FineStrategy {     // 默认:1 元/天
    public double calculate(long days){ return Math.max(0, days) * 1.0; }
}
// 将来加 VIP 免罚 = 新增一个 VipFineStrategy,不动任何老代码 ← 这就是 OCP
```

> 纪律提醒:**只抽真正会变的维度。** 把「书名怎么存」也抽成接口,就是过度设计(评分表会扣「模式过度」)。判断标准——你能说出至少两个未来变体,才值得抽接口。

## 第 5 步:设计对外 API / 门面(约 3 min)

现在定下 driver(`main`)会调用的那组方法——也就是你系统的**门面(Facade)**。把方法签名写清楚:入参、返回值、抛什么异常。

```java
class LibraryService {
    // 返回借阅记录,书不存在 / 借满了 → 抛自定义异常,而不是返回 null
    BorrowRecord borrow(String memberId, String isbn);
    // 返回本次产生的罚款(0 表示没超期)
    double returnBook(String memberId, String isbn, LocalDate returnDate);
    void addBook(Book book);
    void registerMember(Member member);
}
```

两个高频细节,直接对应评分表:

- **错误用自定义异常,不要返回 `null`/`-1`/魔数**:`throw new BookNotAvailableException(isbn)` 比 `return null` 清晰得多,也好演示。建一个 `LibraryException` 基类,下面挂几个具体异常
- **返回类型要表达语义**:`borrow` 返回 `BorrowRecord`(凭证),`returnBook` 返回 `double`(罚款)。别让 driver 去猜

> 门面是你和面试官对齐设计的最好抓手:把这几行方法签名一摆,「这就是这个系统对外能做的事」,设计是否清晰一目了然。

## 第 6 步:按依赖顺序写可运行代码(编码主体,约 50 min)

不要随意从哪个类开始写。按**依赖从底到上**的顺序写,保证每写完一层都能编译:

```
enum  →  model(实体)  →  接口(策略/状态)  →  策略实现  →  门面/Service  →  内存存储  →  Driver(main)
```

为什么这个顺序:底层(enum、实体)不依赖别人,先写;策略接口和实现是门面的零件,次之;门面把零件装配起来,最后写 driver 驱动。**每写完一层就编译一次**,别攒到最后一起爆。

编码期的三个默认选择(不用纠结,直接用):

**① 内存存储用 `Map`,别上数据库。** machine coding 全程内存态。

```java
Map<String, Book>   booksByIsbn   = new HashMap<>();
Map<String, Member> membersById   = new HashMap<>();
```

**② 线程安全:题目不点名并发就用普通容器,但要能说出「并发版怎么改」。** 一旦题目强调多线程(秒杀抢座、限流器、停车场多入口),默认方案:

- 读多写少、键值存取 → `ConcurrentHashMap`
- 需要「检查 + 修改」原子(如「还有空位就占一个」)→ 对共享资源 `synchronized` 或 `ReentrantLock`,把「检查 + 占用」包在一个临界区里
- 计数器 → `AtomicInteger` / `AtomicLong`

> 面试表达:就算这道题不要求并发,主动说一句「单线程版本我用 `HashMap`;如果要支持多入口并发停车,我会把车位分配用 `synchronized` 包住『找空位 + 占用』这一步,避免两辆车抢到同一个位」——这句话直接拿下「并发」那一栏。

**③ 先写 happy path,跑通了再补边界。** 先让「正常借一本书 → 还书 → 算罚款」这条主线编译通过、跑出正确结果,再回头加「借满了」「书不存在」「重复还」这些边界。**绝不要一开始就追求面面俱到**——那是 60 分钟还没跑通主流程的根源。

## 第 7 步:写 Driver 演示 + 自测边界(约 10 min)

最后写一个 `main`,把系统**演**给面试官看。结构固定:**一段 happy path + 2～3 个边界**,每步打印可观察输出。

```java
public class Driver {
    public static void main(String[] args) {
        LibraryService lib = new LibraryService();
        lib.addBook(new Book("isbn-1", "DDIA"));
        lib.registerMember(new Member("m1", "Alice"));

        // —— happy path ——
        BorrowRecord r = lib.borrow("m1", "isbn-1");
        System.out.println("Alice 借到: " + r.book.title + ",应还 " + r.dueDate);
        double fine = lib.returnBook("m1", "isbn-1", r.dueDate.plusDays(3)); // 超 3 天
        System.out.println("还书罚款: " + fine);   // 期望 3.0

        // —— 边界 1:借不存在的书 ——
        try { lib.borrow("m1", "not-exist"); }
        catch (BookNotAvailableException e) { System.out.println("预期异常: " + e.getMessage()); }

        // —— 边界 2:同一本书重复还 ——
        // ... 类似地 try/catch 并打印
    }
}
```

> driver 不只是「测试」,它是你的**演示脚本**。面试官看着这段输出,就信了你的系统真的工作——这一栏(可演示)拿满。跑出来的 bug 当场修,这正是 demo 阶段的价值。

## SOLID 在 machine coding 里的落地动作(不是背定义)

七步法本身就把 SOLID 织进去了。把它翻译成你写代码时的**具体动作**:

| 原则 | 一句话 | 在七步法里的落地动作 |
| --- | --- | --- |
| **S** 单一职责 | 一个类只干一件事 | 第 3 步:实体管状态、门面做编排、策略做计算,别让一个类全包 |
| **O** 开闭 | 对扩展开放、对修改关闭 | 第 4 步:把会变的维度抽接口,新增 = 加新实现类 |
| **L** 里氏替换 | 子类能无缝替换父类 | 策略 / 子类型实现统一接口,门面只依赖接口,换实现不出错 |
| **I** 接口隔离 | 接口要小而专 | `FeeStrategy` 只有 `calculate` 一个方法,别塞一堆不相关的 |
| **D** 依赖倒置 | 依赖抽象不依赖实现 | 门面持有 `FineStrategy` 接口字段,具体策略从外面注入(构造函数传入) |

记住:**OCP 是 machine coding 的灵魂。** 评分表的「可扩展」和扩展性追问,考的全是 OCP。第 4 步抽接口 + 第 5 步门面依赖接口 + 构造注入,就是 OCP 的标准落地三连。

## 七步法速查卡(贴显示器)

```
1. 圈范围    in/out scope + 假设,跟面试官对齐
2. 名词→类   实体 + enum(有限取值)
3. 动词→法   行为跟数据走,编排放门面,警惕 God class
4. 找变化点  会变的维度抽接口(费/分配/拆分/状态)← OCP 灵魂
5. 定门面    方法签名:入参/返回/自定义异常
6. 写代码    enum→model→接口→策略→门面→存储→driver,先 happy path
7. 写driver  happy path + 2~3 边界,打印可观察输出,当场跑
口诀:能跑 > 覆盖 > 干净 > 可扩展 > 优化
```
