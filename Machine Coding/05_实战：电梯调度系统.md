# 05 实战：电梯调度系统

电梯是 machine coding 里「状态机 + 调度」类题的代表。它的考点和前两题都不同：停车场考状态+并发，Splitwise 考建模+策略，电梯考的是——**怎么把「电梯怎么移动」这件随时间演进、有方向、有顺序的逻辑，建模得既正确又干净**。而且这道题有一个极容易被带偏的岔路：「状态机」是不是就等于「上 State 模式」？这一章会专门在这个岔路上较真。

> 读法同前：每步拆 **🤔 自问 / 🔀 岔路 / 💬 说出口**，代码是想清楚后的产物。

---

## 第 1 步：需求澄清 —— 先把「电梯到底在响应什么」想清楚

**🤔 自问**：「设计一个电梯系统」能发散到天上（多电梯群控、高峰模式、能耗优化、消防联动……）。剥到骨头，**电梯存在是为了响应什么**？

推导：响应两种**请求**——

- **外召（hall call）**：某层的人按了「上 / 下」，他想去某个方向
- **内选（car call）**：电梯里的人按了「目标楼层」

系统要做的就是：**收请求 → 决定电梯往哪走 → 一层层移动、到站开门**。MVP 锁定：`收外召 / 收内选 → 调度 → 推进(step) → 到站服务`。

**🔀 岔路：问哪几个结构性问题？**

| 要问的 | 改变什么结构 |
| --- | --- |
| 一部电梯还是多部？ | 决定要不要「派哪部电梯」这层调度（多电梯才有派单问题） |
| 调度策略要多智能？（FCFS？最近优先？LOOK/SCAN？） | 决定电梯「下一步往哪走」是不是要抽成策略 |
| 要不要建模开关门、维修、满载这些状态？ | **决定状态用 enum 还是 State 模式**——这题的命门 |
| 要实时（多线程后台跑）还是离散步进（step 模拟）？ | 决定主循环形态 |

**🔀 岔路（很重要的简化决策）：实时多线程，还是离散步进？** 真实电梯是后台线程不停跑的。但在 90 分钟里，**起一堆线程 + sleep 来「模拟时间」是给自己挖坑**——既难演示又难调试。我选**离散步进**：`step()` 推进一个时间片，driver 循环调用直到全空闲。这样逻辑确定、可断言、可演示。**主动说出这个简化**，并说明真实系统会换成后台调度线程。

**💬 说出口**：「核心是响应外召和内选两类请求，决定电梯往哪走并逐层服务。我做多部电梯，所以有『派哪部』和『单部怎么走』两层决策；调度我实现 LOOK（扫描）算法。为了可演示，我用离散 `step()` 步进来模拟时间，而不是起后台线程——真实系统这里会是个调度线程。开关门、维修这些状态我先用方向枚举表达，够不够要不要升级成状态模式，我待会儿专门权衡一下。可以吗？」

```
In scope :
  - 外召 requestElevator(floor, dir) / 内选 pressFloor(elevatorId, floor)
  - 多部电梯：派单策略（哪部去）+ 单部调度策略（LOOK：顺路服务，到顶/底再反向）
  - step() 离散步进推进，到站开门
Out of scope :
  - 多线程实时调度、能耗/高峰优化、消防联动、超重保护、门的物理时序
Assume :
  - 楼层 [minFloor, maxFloor]，越界请求报错
  - LOOK：同方向先把顺路的请求服务完，没有了再反向；不做电梯群最优
扩展（被问到再做）：
  - 方向感知派单（同向顺路的电梯优先）；维修/停运状态 → 升级 State 模式
```

## 第 2 步：名词 → 类 —— 顺带定下「待停靠楼层」用什么结构

**🤔 自问**：圈名词——**电梯、楼层、方向、请求、调度器**。建什么？

- **enum**：`Direction { UP, DOWN, IDLE }`——电梯的运行方向，有限取值
- **实体**：`Elevator`（每部电梯）、`ElevatorSystem`（门面/调度器）
- **请求**：外召/内选最终都归结为「让某部电梯**多一个要停的楼层**」。所以我**不为 Request 单独建复杂类层级**——一个楼层 int 足够；外召的方向只在「派单」时用一下。（克制：别建用不到的类）

**🔀 岔路：一部电梯「还要停哪些楼层」用什么数据结构？** 这是这题的隐藏命门。候选：

| 结构 | 问题 |
| --- | --- |
| `List<Integer>`（按请求顺序） | 那就只能 FCFS——电梯在 1 楼，先按 9 再按 2，会先冲到 9 再回到 2，反人类。否 |
| `Queue` | 同上，丢了「顺路」语义 |
| **`TreeSet<Integer>`（排序集合）** | 天然有序，能 O(log n) 找到「当前层上方最近的目标」`higher()` 和「下方最近的」`lower()`——这正是 LOOK 要的。✅ |

选 `TreeSet`。**为什么这个选择是 LOOK 算法的地基**：LOOK 的本质就是「沿当前方向，依次服务路过的目标，到头再掉头」，而「沿方向找下一个目标」= 在排序集合里 `higher`/`lower`。数据结构一选对，调度逻辑就顺了。

**💬 说出口**：「每部电梯维护一个**排序的待停靠楼层集合** `TreeSet`，而不是按按键顺序的队列——因为我要的是『顺路服务』（LOOK），不是先来先到（FCFS）。排序集合能让我 O(log n) 找到当前层往上/往下最近的目标。」

## 第 3 步：动词 → 方法归属 —— 移动归电梯自己，派单归调度器

**🤔 自问**：动词有 **加目标层、推进一步移动、到站开门、派单、收请求**。各归谁？

- **加目标层 / 推进 / 开门**：都是改 `Elevator` **自己**的状态（当前层、方向、待停集合）→ 放 `Elevator`：`addStop / step / openDoors`
- **派单（哪部电梯去）**：跨多部电梯的决策 → 不属于任何单部电梯，属于调度器 → 第 4 步抽策略
- **收外召/内选的编排**：`ElevatorSystem` 门面

**🔀 岔路（God class 悬崖，这题尤其容易踩）**：很多人把所有逻辑塞进一个 `ElevatorController`——既派单、又替每部电梯算下一步、又维护每部电梯的位置。这就是上帝类。**正确切分**：

- **「单部电梯怎么移动」是电梯自己的事** → `Elevator.step()` 自己管自己的方向和位置
- 调度器只做两件事：**把外召派给某部电梯**、**驱动所有电梯各走一步**

**💬 说出口**：「我让每部电梯自己负责『我该往哪走、到站开门』，调度器只负责『把外召派给哪部』和『驱动大家各走一步』。这样调度器不会变成替所有电梯做主的上帝类。」

## 第 4 步：找变化点 → 抽策略 —— 这题有**两个**真变化点

**🤔 自问**：哪些维度会变、且能说出 ≥2 个变体？

1. **单部电梯的调度算法**：FCFS / LOOK / SCAN——教科书级的多变体 ✅ → 抽 `SchedulingStrategy`
2. **多部电梯的派单**：最近优先 / 负载最轻 / 同向顺路优先——也是多变体 ✅ → 抽 `DispatchStrategy`

> 注意：这次我**理直气壮地抽了两个策略**。这和「别过度设计」不矛盾——判据始终是「能说出 ≥2 个真实变体 + 面试官很可能让我换」。电梯调度算法和派单算法都铁定满足，不抽反而是失分。**抽不抽的纪律是一致的，不是『能少抽就少抽』，而是『按判据抽』。**

**变化点 1：单部调度（LOOK）**——给当前层、当前方向、待停集合，决定下一步往哪个方向：

```java
public enum Direction { UP, DOWN, IDLE }

public interface SchedulingStrategy {
    Direction nextDirection(int currentFloor, Direction current, NavigableSet<Integer> stops);
}

public class LookSchedulingStrategy implements SchedulingStrategy {
    @Override
    public Direction nextDirection(int currentFloor, Direction current, NavigableSet<Integer> stops) {
        if (stops.isEmpty()) return Direction.IDLE;
        switch (current) {
            case UP:                                   // 上行：上方还有目标就继续上，否则掉头
                return stops.higher(currentFloor) != null ? Direction.UP : Direction.DOWN;
            case DOWN:                                 // 下行：下方还有目标就继续下，否则掉头
                return stops.lower(currentFloor) != null ? Direction.DOWN : Direction.UP;
            default:                                   // 空闲启动：朝最近的目标方向出发
                Integer above = stops.higher(currentFloor);
                Integer below = stops.lower(currentFloor);
                if (above == null) return Direction.DOWN;
                if (below == null) return Direction.UP;
                return (above - currentFloor) <= (currentFloor - below) ? Direction.UP : Direction.DOWN;
        }
    }
}
```

**变化点 2：多电梯派单（最近优先）**：

```java
public interface DispatchStrategy {
    Elevator selectElevator(List<Elevator> elevators, int floor, Direction direction);
}

public class NearestCarStrategy implements DispatchStrategy {
    @Override
    public Elevator selectElevator(List<Elevator> elevators, int floor, Direction direction) {
        return elevators.stream()
                .min(Comparator.comparingInt(e -> Math.abs(e.getCurrentFloor() - floor)))
                .orElseThrow(() -> new IllegalStateException("没有可用电梯"));
    }
}
```

> 派单这里没用上 `direction` 参数——但我**留着它**，因为「同向顺路优先」是最常见的升级追问：上行的电梯路过你这层且你也要上行，该优先派它。接口先留好口子，实现 MVP 用最近优先。

## 第 5 步：门面 API

**🤔 自问**：外部（楼宇、driver）怎么用这个系统？

```java
public class ElevatorSystem {
    void requestElevator(int floor, Direction direction); // 外召：某层按了上/下
    void pressFloor(int elevatorId, int destFloor);       // 内选：电梯内按目标层
    void step();                                          // 推进所有电梯一个时间片
    boolean allIdle();                                    // 是否全部空闲（driver 用来判停）
}
```

**🔀 岔路：外召和内选为什么是两个方法？** 它们语义不同：外召**还没确定哪部电梯**，要先派单；内选是**已经在某部电梯里**了，直接给那部加目标层。合并成一个方法反而要传一堆可空参数判分支。分开更清晰。

## 第 6 步：实现电梯 —— 这里就是「状态机」，而且我要在这儿做关键取舍

**🤔 自问**：`Elevator.step()` 一个时间片该做什么？无非三种情形：**已在目标层 → 开门服务**；**没目标 → 转空闲**；**有目标但不在 → 按策略移动一层**。

这其实就是一个**状态机的转移逻辑**。代码掉下来：

```java
public class Elevator {
    private final int id;
    private final int minFloor, maxFloor;
    private int currentFloor;
    private Direction direction = Direction.IDLE;
    private final NavigableSet<Integer> stops = new TreeSet<>();   // 待停靠楼层（排序）
    private final SchedulingStrategy strategy;

    public Elevator(int id, int minFloor, int maxFloor, SchedulingStrategy strategy) {
        this.id = id; this.minFloor = minFloor; this.maxFloor = maxFloor;
        this.currentFloor = minFloor; this.strategy = strategy;
    }

    public void addStop(int floor) {
        if (floor < minFloor || floor > maxFloor)
            throw new IllegalArgumentException("楼层越界: " + floor);
        stops.add(floor);
    }

    /** 推进一个时间片：到站→开门；无目标→空闲；否则按 LOOK 移动一层 */
    public void step() {
        if (stops.contains(currentFloor)) { arrive(); return; }       // 已在目标层（含刚被加的当前层）
        if (stops.isEmpty()) { direction = Direction.IDLE; return; }
        direction = strategy.nextDirection(currentFloor, direction, stops); // 状态转移：决定方向
        currentFloor += (direction == Direction.UP ? 1 : -1);          // 移动一层
        if (stops.contains(currentFloor)) arrive();                    // 抵达目标层
    }

    private void arrive() {
        System.out.printf("  电梯#%d 到达 %d 层，开门%n", id, currentFloor);
        stops.remove(currentFloor);
        if (stops.isEmpty()) direction = Direction.IDLE;
    }

    public boolean isIdle()        { return stops.isEmpty(); }
    public int getCurrentFloor()   { return currentFloor; }
    public int getId()             { return id; }
    public Direction getDirection(){ return direction; }
}
```

### 🔀 岔路（这一章的灵魂）：方向状态用 `enum`，还是上 State 模式？

「这是状态机题，是不是该上 State 模式？」——很多人条件反射地把 `Direction` 拆成 `UpState / DownState / IdleState` 三个类。**停下来，用判据想清楚**：

State 模式的价值是：**每个状态下行为显著不同，且状态间有需要被守护的非法转移**。拿这把尺子量电梯：

- 「下一步往哪走」的差异，**已经被我抽进 `SchedulingStrategy` 了**——这才是真正会变的逻辑。方向本身只是个数据标记
- UP/DOWN/IDLE 之间的转移很简单（到头掉头、没目标转空闲），没有复杂的「非法转移要拦截」

**结论：方向用 `enum` 足够，上 State 模式是过度设计**（评分表会扣「模式过度」）。

那什么时候**才该**升级成 State 模式？当电梯有**多个行为分叉的运营模式**、且转移要被守护时——比如加入「**维修中 / 停运 / 满载 / 开门中**」：

- 维修中：拒绝一切外召，`step()` 不动
- 开门中：不能移动（非法转移要拦）
- 满载：外召跳过、只服务内选

这时每个状态的 `step()`、`requestStop()` 行为真的不同，`enum + switch` 会在多处膨胀，State 模式才划算：

```java
// —— 仅当出现多个行为分叉的运营模式时，才升级成这个 ——
interface ElevatorMode {
    void onRequest(Elevator e, int floor); // 正常：受理；维修：拒绝
    void step(Elevator e);                 // 正常：移动；开门中：先关门；维修：不动
}
class OperationalMode implements ElevatorMode { /* 正常受理 + LOOK 移动 */ }
class MaintenanceMode  implements ElevatorMode { /* 拒绝请求；step 不动 */ }
class DoorsOpenMode    implements ElevatorMode { /* 先关门才能移动 */ }
```

**💬 说出口**（这段话本身就是高分回答）：「方向我用枚举，没上 State 模式——因为真正会变的『往哪走』我抽成了调度策略，方向只是数据，三态转移也简单。如果需求加入维修、开关门、满载这些**行为分叉的运营模式**，且要拦非法转移（比如开门时禁止移动），我才会把它升级成 State 模式，每个模式一个类。**『状态机』不等于『必须用 State 模式』**，这是我刻意控制的抽象度。」

> 这就是为什么这道题被排成「状态机」章却**没有**满屏 State 类——它教的恰恰是**怎么判断该不该用 State 模式**，这比无脑套用值钱得多。

## 第 7 步：实现门面 + Driver

门面把派单和步进编排起来：

```java
public class ElevatorSystem {
    private final List<Elevator> elevators;
    private final DispatchStrategy dispatchStrategy;

    public ElevatorSystem(List<Elevator> elevators, DispatchStrategy dispatchStrategy) {
        this.elevators = elevators; this.dispatchStrategy = dispatchStrategy;
    }

    public void requestElevator(int floor, Direction direction) {           // 外召
        Elevator e = dispatchStrategy.selectElevator(elevators, floor, direction);
        System.out.printf("外召 %d 层(%s) → 派电梯#%d (当前在 %d 层)%n",
                floor, direction, e.getId(), e.getCurrentFloor());
        e.addStop(floor);
    }

    public void pressFloor(int elevatorId, int destFloor) {                 // 内选
        elevators.stream().filter(x -> x.getId() == elevatorId).findFirst()
                .orElseThrow(() -> new IllegalArgumentException("无此电梯: " + elevatorId))
                .addStop(destFloor);
    }

    public void step() { elevators.forEach(Elevator::step); }
    public boolean allIdle() { return elevators.stream().allMatch(Elevator::isIdle); }
}
```

**🤔 自问（driver 要证明什么）**：①LOOK **顺路服务**——先按 5 再按 3，从 0 上行应「先停 3 再停 5」，而不是按按键顺序先冲 5；②多电梯**就近派单**。

```java
public class Driver {
    public static void main(String[] args) {
        SchedulingStrategy look = new LookSchedulingStrategy();

        // ① LOOK 演示：单部电梯，先按 5 再按 3，看它顺路先停 3
        Elevator e1 = new Elevator(1, 0, 9, look);
        ElevatorSystem one = new ElevatorSystem(List.of(e1), new NearestCarStrategy());
        one.requestElevator(5, Direction.UP);
        one.requestElevator(3, Direction.UP);          // 后按，但在去 5 的路上
        System.out.println("=== LOOK：期望先到 3 层再到 5 层（不是按按键顺序）===");
        runUntilIdle(one);

        // ② 多电梯就近派单：a 在 0 层、b 在 8 层，7 层外召应派 b
        Elevator a = new Elevator(1, 0, 9, look);
        Elevator b = new Elevator(2, 0, 9, look);
        b.addStop(8); while (!b.isIdle()) b.step();     // 先把 b 开到 8 层
        ElevatorSystem multi = new ElevatorSystem(List.of(a, b), new NearestCarStrategy());
        System.out.println("=== 就近派单：a 在 0 层、b 在 8 层，7 层外召应派 b ===");
        multi.requestElevator(7, Direction.DOWN);       // 期望：派电梯#2
        runUntilIdle(multi);

        // ③ 边界：越界楼层
        try { one.pressFloor(1, 99); }
        catch (IllegalArgumentException ex) { System.out.println("预期异常: " + ex.getMessage()); }
    }

    static void runUntilIdle(ElevatorSystem s) {
        int tick = 0;
        while (!s.allIdle() && tick++ < 200) s.step();
    }
}
```

**💬 说出口**：「我跑两条：单部电梯先按 5 再按 3，演示 LOOK 顺路先停 3——这证明它不是 FCFS；再用两部电梯演示就近派单。最后跑一个越界楼层边界。」

跑通它，你交出的是：**正确的 LOOK 调度、可换的调度/派单策略、刻意克制的状态建模、可演示**——评分九栏到手，而且「模式恰当不过度」这栏因为你**讲清了为什么不上 State 模式**而拿满。

## 常见追问与应答

- **「为什么不用 State 模式？这不是状态机吗？」** → 见第 6 步：真正会变的「往哪走」抽成了策略，方向只是数据、三态转移简单；只有出现维修/开门/满载这类**行为分叉的运营模式**时才升级 State 模式。这是刻意控制抽象度
- **「改成 FCFS / SCAN 调度？」** → 新增一个 `SchedulingStrategy` 实现注入电梯，电梯一行不改（OCP）
- **「派单要『同向顺路优先』？」** → `DispatchStrategy` 已留 `direction` 参数，新增一个实现：优先选「方向相同且会路过该层」的电梯，距离做次级排序
- **「真实电梯不是步进的，怎么做实时？」** → 把 `step()` 放进每部电梯的后台线程循环（带 sleep 模拟楼层耗时），外召/内选通过线程安全队列投递；`stops` 换并发结构或加锁。MVP 用步进是为了可演示和确定性
- **「同一个外召会不会被派给多部电梯？」** → 当前一次外召只派一部；要防重复可在派单后标记该 (楼层,方向) 已被某电梯接管
