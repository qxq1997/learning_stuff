# Machine Coding 学习进度

## 学习主题

Machine Coding（机器编码轮 / 现场低代码量系统实现 / LLD coding round）

## 学习者画像

- 背景：后端开发工程师，5 年经验，主语言 Java，正在做面试复盘
- 目标：拿到任意 machine coding 题，能在 60～120 分钟内用固定方法论写出能跑、干净、可扩展、可演示的面向对象代码，并扛住扩展性追问
- 学习方式：先建立方法论（定位 + 七步法），再用经典原题反复练同一套路径，每道实战都把代码敲进 IDE 跑一遍

## 当前判断

- 优势假设：已有「设计模式」「系统设计」「算法题」三个专题打底，OO 与架构视野不弱，缺的是「压进 90 分钟落成可跑代码」的实战肌肉
- 风险假设：容易把 machine coding 答成系统设计（谈容量/分库）或算法题（纠结最优数据结构）；也可能为秀模式而过度设计
- 当前阶段：第 1 阶段，已铺好方法论 + 第一道实战，下一步是多刷经典原题把七步法练成肌肉记忆

## 重点关注的理解难点

- 容易把 machine coding 和系统设计混为一谈，上来谈 QPS、分库分表、容量估算，答错频道
- 容易把它和算法题混淆，在「车位分配用什么数据结构最优」上纠结，烧光可扩展和 demo 的时间
- 容易写出 God class（一个 Manager 全包）或把逻辑全塞 `main`，丢「职责单一」
- 容易走另一个极端——为秀模式而过度设计（停车场硬上抽象工厂+装饰器+观察者），主流程跑不通
- 容易忘记「能跑能演示」是底线，写完一堆类却没有 driver，无法证明系统工作
- 容易识别不出「变化点」，要么什么都不抽接口（不可扩展），要么什么都抽（过度设计）
- 容易全程不出声，闷头写，丢掉沟通分也错失被纠偏的机会

## 评估标准（用于自适应推进）

- 完全掌握：能独立用七步法推一道新题并写出可跑代码，能讲清每步对应的评分项 -> 进入下一题
- 基本掌握：路径正确，代码能跑，个别地方职责切分或变化点识别有小偏差 -> 简短纠正后进入下一题
- 部分理解：能背七步但落不成代码，或抽接口/职责切分明显混乱 -> 生成补充课（如 `03b_补充.md`）或重做同类题
- 尚未理解：分不清和系统设计/算法题的区别，或写不出能跑的最小系统 -> 回到 01/02 重建方法论
