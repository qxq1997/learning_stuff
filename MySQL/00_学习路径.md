# MySQL 学习路径与知识图谱

## 学习定位

后端工程师学 MySQL，最容易卡住的不是“不会写增删改查”，而是：

- 平时会用 MySQL，但一问到 `redo log`、`binlog`、`undo log`、两阶段提交、MVCC，就感觉脑子里只有零散名词。
- 看了很多文章，知道有 Buffer Pool、页、刷盘、checkpoint、Read View，可一旦放到同一条更新链路里，就串不起来。
- 面试或者线上排障时，只知道“binlog 用于主从、redo log 用于恢复”，但说不清它们为什么必须同时存在，为什么还要多一个 `undo log`。
- 遇到慢查询时，只会说“加索引试试”，但不知道怎么从慢日志、执行计划、锁等待和真实数据量一步步定位原因。

这一组内容的目标，是把 MySQL 从表设计、查询执行到 InnoDB 持久化与并发控制的主线真正讲顺：

- 表结构和 JOIN 语义怎样影响查询路径与一致性维护？
- 一次 `update` 到底改了哪里？
- 为什么修改数据不是直接刷数据页，而是先写日志？
- `redo log` 和 `binlog` 分别保什么，为什么不能互相替代？
- 数据页刷到一半损坏时，为什么还需要 Doublewrite？
- 两阶段提交到底在解决哪种不一致问题？
- `undo log` 为什么不仅用于回滚，还和 MVCC 紧密相关？
- 线上发现慢查询后，应该按什么顺序拿现场、看计划、定位根因、验证优化？

学完之后，你应该能把“MySQL 的查询与存储机制”看成一条完整的数据流，也能把表设计、恢复判断和慢查询排查从“凭感觉”变成有证据、有闭环的工程动作。

## 为什么这样排序

很多人会直接去背：

- `redo log` 保证崩溃恢复
- `binlog` 用于主从复制
- `undo log` 用于回滚

这些结论本身没错，但如果没有“整体链路”意识，很容易出现三个问题：

1. 只会背定义，不会解释为什么。
2. 说不清三者是在事务生命周期的哪个阶段发挥作用。
3. 一遇到“事务提交了一半宕机怎么办”这种题，就只能硬背“两阶段提交”。

所以这里故意先从总览开始，把 MySQL 的日志体系分层看清楚，再单独拆 `redo`、`binlog`，最后再回到它们之间最关键的协作问题，也就是两阶段提交。等这条链路稳定了，再去讲 `undo log` 和 MVCC，你就会更容易理解为什么“可见性”和“回滚”其实是同一批历史版本在不同场景下的两种用途。

最后再补慢查询排查，是为了把前面的日志、事务、MVCC 知识落到线上场景里。真实排障时，慢 SQL 可能是扫描太多、索引不合适，也可能是锁等待、长事务、统计信息过旧、主从延迟或业务查询模型不合理。只有把“SQL 写法”和“MySQL 内部状态”一起看，判断才不会飘。

第 07 课（一条 SELECT 的执行流程）放在最后，但它其实是整组内容的“架构骨架”：把 Server 层 / 存储引擎层、连接器、解析器、优化器、执行器讲清楚后，前面 redo / binlog / undo / 两阶段提交 / MVCC 才有明确的“挂载位置”。如果你是第一次系统学 MySQL 内部机制，也可以先读 07 建立全局地图，再回头按 01→06 啃日志与事务主线，两种顺序都行。

第 11 课补的是「把 B+ 树还原成页」的视角。第 08 课用页结构推导容量和 2000W，第 09/10 课讲联合索引与失效机理，第 11 课则专门讲一个页内怎么找记录、页之间怎么连接、聚簇索引和二级索引的叶子页到底存什么。读完它，再回头看回表、覆盖索引、最左匹配和范围扫描，会更有画面。

第 12 课进入锁与死锁专题。它不是孤立地背锁名词，而是从订单幂等场景出发，把 RR 下的 `select ... for update`、next-key lock、gap lock、insert intention lock、隐式锁、唯一索引冲突和线上排查命令串起来。读它时要带着第 06 课的排障 SOP 和第 09/10/11 课的索引结构意识：锁到底加在哪个索引上、锁住的是记录还是间隙，往往比“这条 SQL 看起来简单”更重要。

第 13 课把第 12 课里用到的锁规则系统化。第 12 课偏真实事故复盘：为什么“先查不存在，再插入”会死锁；第 13 课偏通用推导：唯一索引、非唯一索引、等值查询、范围查询、没走索引分别怎么加锁。读这一课时要先画索引有序序列，再看 SQL 扫描路径，最后用 `data_locks` 验证 `LOCK_MODE` 和 `LOCK_DATA`。

第 14 课专门回答一个高频面试追问：next-key lock 是不是只防插入，能不能防删除导致的结果集变化？这一课把“幻读”从死背定义拉回谓词结果集稳定性：insert 会让结果集变多，delete 会让结果集变少，update 会让行进入或离开范围。读完它，你应该能把 MVCC、当前读、记录锁、间隙锁、执行计划和锁范围串成一套完整解释。

第 15 课把锁专题收成一张全景图。前面几课主要在行级锁里深挖，第 15 课补上全局锁、显式表锁、MDL、意向锁、AUTO-INC 锁这些更高层的并发控制。线上遇到“整张表卡住”时，真正原因未必是行锁，也可能是等待 MDL、表锁、FTWRL 或自增锁；这一课就是为了先帮你判断阻塞发生在哪个层级。

第 16 课把锁知识落到 `performance_schema.data_locks` 的读法上。前面几课知道了 gap lock、next-key lock、insert intention lock 是什么，但线上排查更重要的是把 `LOCK_MODE`、`LOCK_STATUS`、`LOCK_DATA` 读成等待关系。这一课用一道死锁面试题演练：两个事务都持有同一段 gap lock，为什么一插入就互相等待。

第 17 课回到一个更高频的生产事故：`update/delete` 没走索引。它把第 13/14 课的“当前读加锁规则”和第 06 课的“慢查询排查闭环”接起来：一条 SQL 最后只影响 1 行，不代表只锁了 1 行；真正要看的是执行计划扫描了哪些索引记录。读完这一课，你应该能把“update 没加索引会锁全表”这句口号，拆成全表扫描、next-key lock、执行计划、安全更新和变更规范。

第 18 课再回到日志主线，专门把一条 `update` 从执行到提交再到崩溃恢复完整串起来。前面第 02/03/04/05 课分别讲了 redo、binlog、两阶段提交和 undo，第 18 课则把这些点放到同一条时间线上：先定位记录和加锁，再写 undo、改 Buffer Pool、写 redo、生成 binlog，最后通过两阶段提交和组提交完成一致性与性能之间的平衡。读完它，再回头看日志体系，应该不再是几个孤立名词，而是一条能排障、能面试、能做参数取舍的更新链路。

第 19 课把视角继续下沉到“页里的一行”。第 11 课讲了 B+ 树节点本质是页，第 18 课讲了更新时会修改 Buffer Pool 中的数据页，第 19 课则回答：这个页里的每条记录内部到底怎么摆。理解 COMPACT 行格式后，`NULL` 为什么有位图、`varchar(n)` 为什么还要记录实际长度、65535 字节限制为什么要扣额外开销、大字段为什么会发生行溢出，这些问题会自然连成一条线。它也能反过来解释为什么大字段会影响 Buffer Pool 命中率、范围扫描和回表成本。

第 20 课回到事务隔离这条主线，把第 05 课的 MVCC 直觉升级成完整可见性推导。第 05 课重点回答“undo log 为什么能串出历史版本”，第 20 课则重点回答“某个历史版本为什么对当前事务可见或不可见”。读这一课时要按事务时序画 Read View：RC 为什么每条语句能看到新提交版本，RR 为什么同一个事务内普通 `select` 能稳定复用快照，以及为什么当前读仍然必须靠 next-key lock。

第 21 课专门收拢一个高频追问：可重复读到底有没有彻底解决幻读？它不是重复第 14 课和第 20 课，而是把二者拼起来：第 14 课讲当前读靠 next-key lock 稳定谓词范围，第 20 课讲快照读靠 Read View 稳定历史视图，第 21 课则讲“混用快照读和当前读”为什么还会看到结果集变化。读完后，你应该能把“RR 解决幻读”这句口号改成更严谨的工程回答。

第 22 课补上表设计和查询语义这一层。前面的索引、锁与行格式都会受表模型影响：一份事实在多少处重复、连接保留哪些行、过滤发生在聚合前还是聚合后，最终都会落到 InnoDB 的页、索引和事务代价上。这一课也明确区分交易库的规范化设计与分析模型的事实/维度组织，避免把“拆表”或“反范式”背成脱离场景的规则。

第 23 课补上日志主线里一个容易漏掉的物理恢复环节。第 02 与第 18 课说明了 redo 如何保护已提交修改，但 redo 默认要面对完整可解析的数据页；若后台刷页途中发生 torn page，必须先靠 Doublewrite 找回完整页，再继续 redo 重放。读完这一课，事务持久性与页完整性这两条防线才能真正拼齐。

## 学习方法建议

- 学每一种日志时，都先问一句：它记录的是“物理修改”、还是“逻辑操作”、还是“历史版本”？
- 把“事务执行过程中发生了什么”和“宕机重启后如何恢复”分开理解，再拼起来。
- 不要只记参数值，比如 `innodb_flush_log_at_trx_commit=1`，要知道它到底控制的是“写到 page cache”还是“真正 fsync 落盘”。
- 看两阶段提交时，务必代入“主库已经恢复，但从库和备库还要靠 `binlog` 同步”这个真实场景，否则你会低估它的重要性。
- 学 MVCC 时，不要一上来就死磕源码，先把“每一行数据其实可能挂着一串旧版本”这个画面建立起来。
- 学慢查询时，不要只看 SQL 文本，要同时看真实参数、慢日志聚合、执行计划、扫描行数、锁等待和调用频次。
- 学 B+ 树时，不要只看抽象节点图，要把节点还原成 InnoDB 数据页：页内有槽，页间有链表，叶子页和非叶子页存的内容不同。
- 学死锁时，不要只背“四个必要条件”，要还原等待图：每个事务持有什么锁、等什么锁、锁在哪个索引和哪个范围上。
- 学行级锁规则时，不要只背“next-key 会退化”，要先画出索引顺序；如果是二级索引，一定画成 `(二级索引值, 主键值)`。
- 学幻读时，不要只记“防 insert”，要按结果集变化来想：多行、少行、行进出范围，分别由 gap lock 和 record lock 的哪部分挡住。
- 学锁全景时，先判断锁的层级：全局、表级、行级。不要一看到阻塞就默认是行锁，MDL 和表锁经常才是线上大面积阻塞的源头。
- 学 `data_locks` 时，要按事务分组，看 `LOCK_MODE` 判断锁类型，看 `LOCK_STATUS` 判断已持有还是等待，再用 `LOCK_DATA` 推导索引范围。
- 学 `update/delete` 安全变更时，不要只问“有没有 where”，要问“执行计划是否走预期索引、预计扫描多少行、事务会持锁多久”。
- 学一条 `update` 的完整链路时，要按时间线画出来：执行计划怎么找行、InnoDB 何时写 undo/redo、Server 何时写 binlog、崩溃恢复时靠什么判断提交还是回滚。
- 学行格式时，要把 SQL 层字段定义翻译成页内字节布局：哪些列需要长度列表、哪些列进入 NULL 位图、隐藏字段在哪里、大字段是否可能走溢出页。
- 学事务隔离时，不要只背“RC 每次生成 Read View、RR 复用 Read View”，要拿一条记录的版本链逐个套可见性规则，直到你能解释每一次读为什么读到当前版本或旧版本。
- 学 RR 幻读时，要先标注每条 SQL 是快照读还是当前读；只要读类型切换了，就不能继续用同一个 Read View 口径解释所有现象。
- 学关系建模时，不要把范式当成“拆得越多越好”，要先找函数依赖和更新异常，再决定哪些历史事实或读模型值得受控冗余。
- 学 Doublewrite 时，要先区分“日志可重放”和“页面可重放”：数据页物理撕裂后，需要先恢复完整页面底座，redo 才有意义。
