# MySQL - 第 5 课：undo log与MVCC：回滚、历史版本、Read View与purge

## 学习目标（本节结束后你能做到什么）

- 理解 `undo log` 为什么不仅用于回滚，还支撑了 MVCC。
- 能说清一行记录为什么会有“当前版本”和“历史版本链”。
- 理解 `DB_TRX_ID`、`DB_ROLL_PTR`、Read View 的基本作用。
- 知道长事务为什么会让 `undo log` 清不掉，并拖慢系统。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

前面我们已经学了：

- `redo log`：保证提交后的修改别丢
- `binlog`：保证逻辑变更能复制、能归档

那 `undo log` 呢？

很多文章一句话写成“回滚日志”，这没错，但只说到一半。

更完整的理解应该是：

**`undo log` 一方面保存“怎么撤销这次修改”，另一方面也保存了“这行数据的旧版本线索”，所以 MVCC 也要依赖它。**

### 1. 什么叫 undo

先看一个最简单的事务：

```sql
begin;
update account set balance = 900 where id = 1;
rollback;
```

如果没有 `undo log`，数据库根本不知道应该把这行恢复成什么。

所以在更新前，InnoDB 会先把旧版本相关信息记下来。  
这样一旦回滚，就能把数据恢复到修改之前的状态。

这里要注意一个常见误区：

很多入门文章会说“`undo log` 记录的是反向 SQL”。这可以帮助初学者理解，但你不要机械地把它理解成数据库真的把整条 `INSERT`、`DELETE`、`UPDATE` 文本原样存进去。

更准确的说法是：

**`undo log` 记录的是足以把当前版本退回旧版本的逻辑信息。**

### 2. 为什么 `undo log` 还和 MVCC 有关

假设有两个事务：

- 事务 A：把余额从 100 改成 80，但还没提交
- 事务 B：此时来读这行数据

事务 B 到底应该看到 100 还是 80？

在 MVCC 里，很多场景下它会看到一个“对自己可见的旧版本”。  
而这个旧版本并不是凭空出现的，它正是顺着 `undo log` 找出来的。

所以你可以把 `undo log` 看成：

- rollback 时的“倒带素材”
- 快照读时的“历史版本素材”

### 3. 一行记录为什么能挂着一串旧版本

InnoDB 的一行数据，除了你自己定义的列，还会隐含一些内部字段。最常见你需要知道的是：

- `DB_TRX_ID`：最后一次修改这行的事务 ID
- `DB_ROLL_PTR`：指向这行对应的 `undo log` 记录

于是，一行数据就能和旧版本串起来，形成版本链。

```mermaid
flowchart LR
    A["当前版本<br/>balance=80<br/>DB_TRX_ID=200"] --> B["undo 记录 1<br/>旧值=100<br/>上一个版本指针"]
    B --> C["undo 记录 2<br/>更早旧值=120"]
```

当一个事务做快照读时，如果发现当前版本对自己不可见，就会沿着这条链往回找，直到找到一个对自己可见的版本。

### 4. Read View 是什么

Read View 可以理解成：

**事务在做快照读时，对“哪些事务算已提交、哪些事务还不能看”的一张可见性名单。**

它不是数据本身，而是一套判断规则。

当事务读到一行记录时，会先看这行的 `DB_TRX_ID`：

- 如果这个修改事务对当前事务来说是可见的，就直接读当前版本
- 如果不可见，就顺着 `DB_ROLL_PTR` 找 `undo log` 中的旧版本

这就是为什么我们说：

- MVCC 依赖隐藏字段
- MVCC 依赖 Read View
- MVCC 依赖 `undo log`

三者缺一不可。

### 5. 快照读和当前读

这是 MVCC 里必须分清的两个概念。

#### 5.1 快照读

比如普通的：

```sql
select * from account where id = 1;
```

在很多隔离级别下，它读到的未必是“这一刻物理上最新的值”，而是“对当前事务可见的版本”。

#### 5.2 当前读

比如：

```sql
select * from account where id = 1 for update;
```

或者直接 `update` / `delete`。

这类操作关注的是当前最新版本，并且会和锁、并发控制一起工作。

所以 MVCC 主要解决的是：

- **快照读如何既不加太重的锁，又能读到一致的数据视图**

### 6. Read View 什么时候生成

这个问题和隔离级别关系很大，特别容易在面试里被问到。

在最常见的两个隔离级别里，你可以先这样记：

- **读已提交（RC）**：每次执行一致性读时，都可能重新生成一个 Read View
- **可重复读（RR）**：事务里的第一次一致性读创建 Read View，后续一致性读尽量沿用它

这也是为什么在 RR 下，同一个事务里你多次普通 `select`，更容易看到“前后一致”的快照结果。

### 7. `undo log` 为什么不会在提交后立刻全部删除

这里又是一个特别容易误解的点。

如果 `undo log` 只用于 rollback，那事务提交以后确实就可以立刻删掉了。

但问题是：

- 别的事务可能还在做快照读
- 它们可能还需要沿着版本链回头看旧版本

所以很多 `undo log` 在事务提交后不会马上删，而是先进入历史链表，等后台 purge 线程确认再也没人需要这些旧版本了，才真正清理。

### 8. rollback segment 和 history list 可以怎么理解

如果你继续往底层看，会遇到几个术语：

- rollback segment
- undo segment
- history list

初学时不必一头扎进源码，但最好建立一个朴素画面：

- 事务做修改时，需要有地方存它的 `undo log`
- 这些 `undo log` 不是随便扔在一处，而是按一定结构组织管理
- 已提交但暂时还不能删的 `undo log`，会被挂到 history list 上，等 purge 线程之后再清

所以你可以粗略理解成：

- rollback segment：管理一批 `undo` 记录的更大容器
- undo segment：某次事务实际占用的撤销记录片段
- history list：那些“已经提交，但还不能清”的历史版本待办清单

这个结构存在的意义，不是为了增加概念，而是为了让 InnoDB 能同时管理：

- 大量并发事务的回滚需求
- 快照读对旧版本的访问需求
- 后台 purge 的延迟清理需求

### 9. 什么是 purge，为什么长事务会拖垮系统

purge 线程的工作，可以简单理解成：

**回收那些已经没人会再访问的旧版本。**

但如果系统里存在长事务，就会出现一个问题：

- 长事务创建了一个很早的 Read View
- 为了保证它还能看到当时该看到的旧版本
- 很多本该清理掉的 `undo log` 不能删

于是会发生：

- 历史版本越来越多
- purge 跟不上
- 版本链变长
- 查询判断可见性更重
- 存储与系统压力上升

这也是为什么线上系统里经常强调：

- **不要开着事务不提交**
- **不要让报表、脚本、后台任务长时间持有事务**

### 10. 一个更完整的理解：undo 不是“配角”

很多人学日志时，会把注意力都放在：

- `redo log`
- `binlog`

因为它们更像“提交成功、宕机恢复、主从复制”这种戏剧性场景里的主角。

但在真实系统中，`undo log` 同样非常重要，因为：

1. 没有它，rollback 没法做
2. 没有它，MVCC 没法看历史版本
3. 没有 purge，系统会被长事务拖出大量历史包袱

而且还有一件事要记住：

**`undo log` 自己也需要持久性保护，所以它相关的修改信息同样会受到 `redo log` 保护。**

也就是说，`redo`、`undo` 并不是互斥关系，而是：

- `undo` 负责“怎么回退 / 怎么找旧版本”
- `redo` 负责“这些日志和页修改别在崩溃后丢了”

## 小结（3-5 条关键点）

- `undo log` 不只是“回滚日志”，它还为 MVCC 提供了历史版本链。
- InnoDB 通过类似 `DB_TRX_ID`、`DB_ROLL_PTR` 这样的隐藏信息，把当前行和 `undo log` 串成版本链。
- 快照读会先看当前版本是否可见，不可见再沿着 `undo log` 找旧版本；RC 和 RR 主要区别之一就在 Read View 生成时机。
- 事务提交后，很多 `undo log` 不会立刻删除，而是等待 purge 线程在安全时机回收。
- 长事务会拖住旧版本清理，导致 `undo` 膨胀、history list 变长、版本链变长、系统性能变差。
