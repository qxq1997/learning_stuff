# MySQL - 第 9 课：索引体系全解——四维分类、最左匹配、联合索引范围查询与失效机理

> 第 7 课讲了「执行器/优化器怎么用索引、回表与覆盖索引、ICP」，第 8 课讲了「B+ 树的页结构、扇出与层高」。这一课把索引本身**成体系**地讲清楚：怎么分类、联合索引为什么有最左匹配、范围查询时哪些字段还能用上索引（Q1–Q4 用 `key_len` 实锤）、区分度怎么影响字段顺序、以及索引失效背后的统一机理。B+ 树结构、`EXPLAIN type/Extra`、回表/覆盖/ICP 这些已在第 6、7、8 课讲透，本课只做交叉引用、不重复展开。

## 学习目标（本节结束后你能做到什么）

- 能从「数据结构 / 物理存储 / 字段特性 / 字段个数」四个角度给所有索引分类，并说清每类的实现差异。
- 能用「联合索引的 B+ 树排序规则」解释最左匹配原则，而不是只背口诀。
- 能判断一条带范围查询的 SQL，联合索引里**哪些字段真正参与了扫描区间**，并用 `key_len` 验证。
- 能讲清「`>` 停止匹配，但 `>=`/`BETWEEN`/`like 前缀` 不停止」的原因，以及 ICP 在这里救了什么。
- 能用区分度（选择性）决定联合索引字段顺序，并权衡「等值优先 / 区分度优先 / 排序列后置」的冲突。
- 能用统一机理（破坏 key 的有序性）解释绝大多数索引失效场景，而不是背一张清单。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

索引的本质是「**以空间换时间的有序目录**」——像书的目录，让存储引擎不用一页页翻。一句贯穿全课的原理先放这里：

> **能用上索引的前提，是索引里的 key 是有序的；任何破坏「有序可二分定位」的操作，都会让索引用不上。** 后面所有结论——最左匹配、范围查询停止匹配、函数/类型转换失效——都是这一句话的推论。

### 1. 索引的四维分类（先把名词理顺）

面试常让你「把聚簇索引、主键索引、二级索引、唯一索引、Hash 索引……分个类」。一上来就背名词会乱，正确做法是**按四个互不冲突的维度切**：

| 维度 | 分类 | 关键区别 |
| --- | --- | --- |
| 按数据结构 | B+Tree / Hash / Full-Text | InnoDB 默认且最常用 B+Tree；Hash 仅等值 O(1) 不支持范围；Full-Text 做文本检索 |
| 按物理存储 | 聚簇索引（主键索引） / 二级索引（辅助/非聚簇） | 聚簇索引叶子存**整行**；二级索引叶子存**主键值**→ 非覆盖要回表 |
| 按字段特性 | 主键 / 唯一 / 普通 / 前缀 | 约束强弱不同；前缀索引只对字符串前 N 字符建索引 |
| 按字段个数 | 单列 / 联合（复合） | 联合索引有最左匹配原则 |

注意这是**四个正交维度**，同一个索引会同时属于多个分类。比如「主键索引」= 数据结构上是 B+Tree + 物理存储上是聚簇索引 + 字段特性上是主键 + 字段个数上是单列。把这点想通，名词就不再打架。

InnoDB 选聚簇索引键的规则（面试高频）：① 有主键用主键；② 没主键选第一个**非 NULL 的唯一列**；③ 都没有，InnoDB 自动生成一个 6 字节隐藏自增 `rowid` 当聚簇索引键。聚簇索引以外的都是二级索引。

> B+Tree 为什么打败 B 树 / 二叉树 / Hash，第 7、8 课已从「叶子才存数据 → 单节点扇出大 → 3~4 层撑千万行 → 点查 3~4 次 IO」「叶子双向链表 → 范围查询高效」「Hash 不支持范围」讲过，这里不重复。本课重点放在**联合索引**，这是分类里最容易考、也最容易错的部分。

聚簇索引 vs 二级索引、回表 vs 覆盖索引的机制，第 7 课 §5/§6 已讲透（覆盖索引 `Using index`、回表、ICP `Using index condition`）。一句话回顾衔接：**二级索引叶子存的是主键值，要不要回表取决于「查的列是否都在这个二级索引里」**——这正是后面讲覆盖索引优化和联合索引设计的基础。

### 2. 最左匹配原则：不是口诀，是 B+ 树的排序规则

联合索引 `(a, b, c)` 的 B+ 树，**key 是「先按 a 排序，a 相同再按 b 排序，b 相同再按 c 排序」**（多列字典序）。画出来：

- `a` 的值：`1, 2, 2, 3, 4, 5, 6, 7, 8` —— **全局有序**。
- `b` 的值：`12, 7, 8, 2, 3, 8, 10, 5, 2` —— **全局无序**。
- 但在 `a` 相同的局部里，`b` 有序（例如 `a=2` 时 `b` 是 `7, 8`）。

由此直接推出最左匹配：

- 能用上索引（能在 B+ 树上二分定位）：`where a=1`、`where a=1 and b=2`、`where a=1 and b=2 and c=3` —— 因为每一步定位时，下一列在「上一列固定」的局部里是有序的。
- 用不上：`where b=2`、`where c=3`、`where b=2 and c=3` —— 因为脱离 `a`，`b`/`c` 全局无序，没法二分。

两个常被追问的点：

1. **`where` 里字段顺序不重要**：`where b=2 and a=1` 和 `where a=1 and b=2` 等价，优化器会重排条件（呼应第 7 课优化器）。最左匹配指的是「**索引定义的列顺序**」，不是「SQL 里写的顺序」。
2. **不连续就断**：`(a,b,c)` 上 `where a=1 and c=3`，只有 `a` 能用索引定位，`c` 因为跳过了 `b` 而无法用于定位（但 `c=3` 可能被 ICP 在引擎层过滤，见 §3 末）。

### 3. 联合索引范围查询：到底哪几个字段「用到了索引」（Q1–Q4 实锤）

这是本文最硬的部分，也是面试最容易翻车的地方。核心结论先给：

> **最左匹配会一路向右，直到遇到范围查询就停止**——范围查询那个字段**本身**能用索引，但它**后面**的字段不能再用于形成扫描区间。**关键例外**：`>=`、`<=`、`BETWEEN`、`like '前缀%'` 这类「带等值边界」的范围，**不会让后面字段停止匹配**。

为什么？因为这些「带等值边界」的范围，可以拆出一段「该列取边界值」的子区间，在那个子区间里下一列又变回有序，于是下一列还能用来缩小扫描范围。下面用联合索引 `(a, b)`（a、b 均 `int NOT NULL`）逐个看，并用 `key_len` 验证。

**`key_len` 怎么算**（验证「优化器用了几列形成扫描边界」的硬指标）：

- 定长字段按类型字节数：`int` = 4。
- 字段允许 `NULL`：在类型字节数上 **+1**（多 1 字节标记是否为 NULL）。
- 变长字段（`varchar` 等）：实际最大字节数 + **2**（无论行格式实际用 1 还是 2 字节，`key_len` 一律按 2 算——因为 `key_len` 在 Server 层算，不去问 InnoDB 行格式，它只为告诉你「用了哪几列」，不为精确算空间）。
  - 例：`varchar(30)` + `utf8mb4`（1 字符最多 4 字节）+ `NOT NULL` → `30×4 + 2 = 122`。

| 查询 | 条件 | 谁形成扫描区间 | `key_len`(a,b 均 int NOT NULL) | 结论 |
| --- | --- | --- | --- | --- |
| Q1 | `a > 1 and b = 2` | 只有 `a` | 4 | `b` 不进扫描边界（但走 ICP） |
| Q2 | `a >= 1 and b = 2` | `a` 和 `b` | 8 | `>=` 不停止匹配 |
| Q3 | `a BETWEEN 2 AND 8 and b = 2` | `a` 和 `b` | 8 | MySQL `BETWEEN` 含两端 ≈ `>= and <=` |
| Q4 | `name like 'j%' and age = 22` | `name` 和 `age` | 126（122+4） | `like` 前缀不停止匹配 |

逐个讲透：

- **Q1 `a > 1 and b = 2`**：二级索引先按 `a` 排，`a > 1` 的记录连续相邻，定位到 `a > 1` 第一条后沿叶子链表扫。但在「`a > 1`」这一大段里 `b` 是无序的（如 `a=5,b=8` / `a=6,b=10` / `a=7,b=5`），没法用 `b=2` 缩小范围。所以扫描区间是 `(1, +∞)`，边界只由 `a > 1` 决定，`key_len=4`，**只有 `a` 用于扫描定位**。
  > 但**别误以为 `b=2` 没用**：它会被**索引下推 ICP**（第 7 课 §6.3）在存储引擎层用来过滤——`Extra` 会出现 `Using index condition`。所以「`b` 没用到索引」的精确含义是「`b` 没参与形成扫描区间边界」，而不是「`b` 这个条件被丢到 Server 层才判断」。这个区分是 Q1 最有价值、也最少被讲清的点。

- **Q2 `a >= 1 and b = 2`**：把它**拆成两段** `a = 1 and b = 2` ∪ `a > 1 and b = 2`。在 `a = 1` 这段里 `b` 是有序的，于是能用 `b = 2` 直接定位到「`a=1 and b=2` 的第一条」开始扫，而不用从「第一个 `a=1`」开始扫。所以 `a`、`b` 都参与了扫描边界，`key_len=8`。`<=`、`BETWEEN`（Q3，MySQL 含两端等价于 `>= and <=`）同理。

- **Q4 `name like 'j%' and age = 22`**：`like 'j%'` 是前缀范围，扫描区间 `['j','k')`（`'j'` 是闭端）。在「`name = 'j'`」这段里 `age` 有序，能用 `age=22` 缩小范围，所以 `name`、`age` 都用上，`key_len = 122 + 4 = 126`。

一句话记忆模型（来自读者问答，很好用）：**`a >= 2` ≈ `a = 2`（这段 b 有序，b 能用）∪ `a > 2`（这段 b 无序，b 不能用）；`a > 2` 则只有后半段，所以 b 用不上。** 是否「有一段等值」决定了后面字段能否继续吃索引。

### 4. 索引区分度（选择性）：决定联合索引字段顺序

```
区分度 = count(distinct column) / count(*)
```

越接近 1 越好（如 UUID、订单号）；越接近 0 越差（如「性别」只有男女）。

为什么区分度低不该建索引 / 不该放联合索引前面：

- 区分度低 → 一个值命中表里很大比例的行。
- 优化器有个经验阈值：**当某个值估计命中超过约 30% 的行，优化器会放弃这个索引直接全表扫**（因为「走二级索引 + 大量回表的随机 IO」比「顺序全表扫」还慢，呼应第 8 课「回表随机 IO 才是大表真正瓶颈」）。这就是「区分度低还不如不建索引」的机理。

**联合索引字段顺序是个权衡题**（这是工程重点，原文只说了「区分度大的放前面」，但实战常有冲突）：

1. **等值列优先于范围列**：范围列一旦出现，它后面的列就吃不到扫描边界（§3）。所以把等值过滤的列放前、范围列尽量靠后。
2. **区分度高的优先**：能更早把候选集缩小，回表更少。
3. **要利用排序的列要放在过滤列之后、且顺序方向匹配**（见 §5）。

这三条经常打架，例如「区分度最高的列恰好是范围查询列」。优先级建议：**先保证等值列在前（让更多列吃到索引）→ 再按区分度排等值列 → 最后考虑能否顺带覆盖 order by**。脱离具体高频 SQL 谈字段顺序没意义——索引是为「稳定的高频查询模式」设计的（呼应第 6 课）。

### 5. 用联合索引顺带把排序做了（消灭 filesort）

题目：`where status = ? order by create_time`，怎么优化？

- 只给 `status` 单列索引：`status` 过滤完，还要对结果按 `create_time` 排序 → `Extra: Using filesort`（第 6 课讲过 filesort 很贵）。
- 给 `(status, create_time)` 联合索引：`status` 是等值，定位到这段后，**这段记录天然就是按 `create_time` 有序的**（B+ 树排序规则，§2），直接顺序读即可，**无需 filesort**。

成立条件（深一层，原文没展开）：① 排序列前面的列必须是**等值**（否则那段不是按排序列有序的，正是 §3 的逻辑）；② 排序方向要能被索引满足（MySQL 8.0 起支持 descending index，可建 `(status, create_time DESC)` 适配 `order by create_time desc`，8.0 之前混合升降序无法用索引消除 filesort）。`group by` 同理——有序输入能避免 `Using temporary`。

### 6. 索引失效：一个机理解释一堆场景

别背清单，记住 §0 那句话「**用上索引的前提是 key 有序、能二分定位；破坏这个前提就失效**」，然后逐条推：

| 失效场景 | 为什么（统一机理） |
| --- | --- |
| `like '%xx'` / `like '%xx%'` | 前缀不确定，没法在按前缀有序的 B+ 树上定位起点；`like 'xx%'` 前缀确定，**不失效**（这就是 Q4 能走索引的原因） |
| 对索引列做函数/计算/表达式，如 `where year(t)=2024`、`where a+1=5` | 索引存的是 `t`/`a` 的有序值，不是 `year(t)`/`a+1` 的；引擎没法用原始有序性定位。改写成 `where t between ... and ...` 即可 |
| 隐式类型转换，如 `varchar` 列 `where phone = 138...`（传了数字） | MySQL 把列做隐式转换 ≈ 对列套了函数，同上失效。**字段类型和传参类型务必一致** |
| 联合索引不满足最左匹配 | §2：脱离左前缀，目标列全局无序，无法二分 |
| `OR` 一侧是索引列、另一侧非索引列 | 非索引列那部分必须全表扫，整体退化为全表扫（可用 `union` 或给另一列也建索引化解） |
| 索引列大量 `NULL` / 区分度极低 | §4：命中比例过高时优化器主动弃用索引走全表扫（这是优化器「正确的选择」，不是 bug） |

排查唯一可靠的办法是看执行计划（第 6 课）：`key` 为 `NULL` 即没用索引；`type` 从差到好 `ALL < index < range < ref < eq_ref < const`，要让高频 SQL 至少到 `range` 及以上；`Extra` 关注 `Using filesort`/`Using temporary`（坏）、`Using index`（覆盖索引，好）、`Using index condition`（ICP，好）。`type`/`Extra` 的完整含义第 6、7 课已展开，这里不重复。

### 7. 索引设计与优化清单（带「为什么」）

- **前缀索引**：对长字符串列只取前 N 字符建索引，减小索引项 → 一页放更多索引项 → 树更矮（呼应第 8 课「索引项越小扇出 x 越大」）。代价：**不能用于 `order by`、不能做覆盖索引**（只有前缀，拿不到完整列值）。选 N 的方法：让 `count(distinct left(col,N))/count(*)` 逼近完整列区分度即可，别盲取。
- **覆盖索引**：把高频查询要的列都放进联合索引，命中后不回表（第 7 课）。例如只查「名称、价格」，建 `(商品ID, 名称, 价格)`，省掉大量回表随机 IO。
- **主键尽量自增 bigint**：自增 = 追加写，页写满开新页，避免**页分裂**和碎片（第 8 课已详述页分裂）；且主键越小 → 所有二级索引叶子（存的就是主键值）越小 → 二级索引整体更省空间、扇出更大。无序主键（UUID/雪花乱序）会触发页分裂。需要分布式唯一又想有序，可用「时间有序」的 ID 方案折中。
- **索引列尽量 `NOT NULL`**：① `NULL` 让优化器统计/比较/索引选择更复杂（`count` 跳过 `NULL`、`NULL` 比较语义特殊）；② InnoDB 行格式里只要有可空列，至少要 1 字节的「NULL 值列表」存占用信息（行格式细节见第 5 课 MVCC/隐藏列那篇），白占空间。
- **不需要索引的场景**：`where/group by/order by` 用不到的列；区分度极低的列（性别）；数据量很小的表（全表扫更快）；频繁更新的列（如用户余额——每次改都要维护 B+ 树有序性，写放大，呼应第 8 课写放大）。

索引的代价别忘：占空间、拖慢增删改（每次写都要动态维护 B+ 树有序）、索引越多优化器选择成本越高。所以成熟说法不是「给它加个索引」，而是「**为某个稳定高频的查询模式，设计一条收益大于成本的访问路径**」（与第 6 课收尾一致）。

## 小结（3-5 条关键点）

- 索引按「数据结构 / 物理存储 / 字段特性 / 字段个数」四个正交维度分类；一个索引同时属于多类，理顺这点名词就不打架。
- 最左匹配不是口诀，是联合索引 B+ 树「多列字典序」排序的必然结果：脱离左前缀，目标列全局无序就无法二分定位；`where` 里写的顺序无关，索引定义的列顺序才算。
- 范围查询会让最左匹配在该列**停止**（后面列不进扫描边界），但 `>=`/`<=`/`BETWEEN`/`like '前缀%'` 因为含等值边界段而**不停止**；用 `key_len` 可实锤用了几列；停在边界外的列仍可被 ICP 在引擎层过滤。
- 联合索引字段顺序是权衡：等值列优先于范围列、区分度高的靠前、需排序的列按方向放在过滤列之后；区分度过低优化器会主动弃用索引走全表扫。
- 索引失效几乎都能用一句话解释：破坏了「key 有序、可二分定位」——左模糊、函数/计算/隐式转换包列、不满足最左匹配、OR 接非索引列；排查一律以执行计划为准（`type` 至少到 `range`，关注 `key/Extra`）。

## 问题（检测用户对当前章节内容是否了解）

1. 把「聚簇索引、唯一索引、联合索引、Hash 索引、前缀索引」按四个维度各归一次类，并说明为什么同一个主键索引能同时属于多类。
2. 用联合索引 B+ 树的排序规则解释：为什么 `(a,b,c)` 上 `where b=2 and c=3` 用不上索引，而 `where a=1 and c=3` 里只有 `a` 能定位、`c` 不能？
