# MySQL - 第 10 课：索引失效六场景深挖——从范围优化器到 handler 边界

> 第 9 课 §6 给了一句直觉口诀「破坏了 key 的有序可二分定位」。**这一课不复述那句直觉，而是挖到它底下的真实机制：MySQL 的「范围优化器（range optimizer）」如何把 `WHERE` 拆成索引上的区间，以及在哪一步、因为什么数据结构原因，某个谓词被判定为「不能形成区间」。** 把这一层吃透，六大失效场景就不再是六条要背的规则，而是同一个机制在不同输入下的同一种结果。B+ 树/页结构、`EXPLAIN`、ICP、最左匹配的基础在第 6/7/8/9 课已讲，本课只交叉引用、不复述，全部篇幅用于挖机制、解法和边界。实验基于 MySQL 8.0.26。

## 学习目标（本节结束后你能做到什么）

- 能讲清 MySQL 范围优化器把 `WHERE` 转成 `SEL_TREE`/区间的过程，并指出「索引列被包裹」时它具体卡在哪一步。
- 能写出 `LIKE 'abc%'` 被范围优化器改写成的**精确区间**，并解释 collation 在其中的角色。
- 能说清隐式类型转换不仅是性能问题，还是**正确性 bug**（会多匹配行），并解释字符集 coercibility 如何决定 join 时哪一侧索引失效。
- 能讲清函数索引的内部实现（隐藏虚拟生成列）、确定性约束和「表达式必须按 item tree 匹配」的精确含义。
- 能讲清 ICP 在 `handler::idx_cond_push` 这条引擎边界上到底做了什么，以及 index_merge 三种算法（union/sort-union/intersection）为什么对 rowid 有序性有不同要求。

## 内容讲解

### 0. 先把脊柱立起来：范围优化器是怎么把 WHERE 变成索引区间的

几乎所有「索引失效」，失效的不是「索引」，而是**范围访问（range access）这条路没被生成出来**。要理解失效，得先看 MySQL 怎么生成它。

MySQL 优化器里有一个独立子系统叫**范围优化器**（源码在 `sql/range_optimizer/`，8.0 重构前在 `opt_range.cc`）。它的核心任务：把 `WHERE` 条件树，转换成每个候选索引上的一组**区间（range）**。核心数据结构是 `SEL_ARG` / `SEL_TREE`——一棵描述「在某个 key part 上，取值落在哪些区间」的树。

它能为一个谓词建出区间的**充要条件**是：该谓词能化简成

```
<index_key_part>  CMP  <constant_or_constant_expr>
```

其中 `CMP ∈ { =, <=>, <, <=, >, >=, BETWEEN, IN, IS NULL, IS NOT NULL, LIKE '常量前缀%' }`，谓词之间用 `AND`/`OR` 组合。**左边必须是「裸的索引列」这个 `Item_field` 节点本身**——范围优化器在匹配时，是拿条件左侧的 `Item` 和索引 key part 对应的 `Field` 做**身份比对**。

关键就在这句话：

> 一旦索引列被函数、算术、`CAST` 包了一层，条件左侧就不再是那个 `Item_field` 节点，而是一个 `Item_func_*` 节点。范围优化器拿它跟 key part 比，**身份不匹配 → 这个 key part 上建不出 `SEL_ARG` → 没有区间 → 没有 range / ref 访问路径 → 退回全表扫**。

这才是「索引失效」的真实机制。「破坏有序性」是它在 B+ 树上的**后果直觉**；「范围优化器匹配不到裸 key part」是**代码层的因**。下面六个场景，全都是这个因的不同表现。这也顺带解释了一个第 9 课留的问题：为什么优化器不替你把 `id+1=10` 移项成 `id=9`——因为范围优化器的设计边界就是「只识别裸列 CMP 常量」，它做常量折叠（对右侧常量求值），但**不做对左侧表达式的代数反解**（见 §3）。

记住这条脊柱，继续往下。

### 0.5 前置补充：InnoDB vs MyISAM 存储结构（决定「回表」存不存在）

| | InnoDB | MyISAM |
| --- | --- | --- |
| 聚簇索引 | 有，主键索引叶子**存整行** | 无聚簇索引概念 |
| 二级索引叶子 | 存**主键值** → 非覆盖要回表查聚簇索引 | 存**行物理地址/行号** |
| 「回表」 | 存在（二级索引→聚簇索引，两棵 B+ 树） | 不存在这种两棵树回表，叶子即行地址，一次寻址取行 |

这层差异决定了后面几个机制的语境：**ICP（§5）和回表优化只在 InnoDB「二级索引→聚簇索引」语境下才有意义**；MyISAM 任何索引叶子都直达行地址，没有「先拿主键再查一棵树」这一步。InnoDB 主键查询数据就在叶子里，省一次间接寻址——这也是第 8 课「主键越小、二级索引越省、点查越快」的底层原因。

### 1. 模糊匹配：`LIKE` 的区间是怎么算出来的，collation 怎么搅局

**现象**：`name LIKE '%林'` → `ALL`；`name LIKE '林%'` → `range`,`key=index_name`。

**机制（精确到区间构造）**：范围优化器处理 `Item_func_like` 时，对「**常量前缀 + `%`**」这种形态，调用 collation 的 `like_range()` 算出一对边界字符串：取前缀，构造该前缀下的**最小串**和**最大串**，得到一个闭/半开区间。直觉上 `name LIKE '林%'` ≈

```
name >= '林'  AND  name < '林\xFF...'   （上界 = 前缀 + 该 collation 下的最大排序权重字符填充）
```

这是一个**有界区间** → 能建 `SEL_ARG` → `range` 扫描：定位到区间左端叶子，沿叶子双向链表顺扫到右端。

`LIKE '%林'`：没有常量前缀，`like_range()` 无法确定左端从哪开始（结果可能是「陈林/张林/周林」散落在按 `name` 排序的整棵树各处）→ 建不出有界区间 → 全表扫。`'%林%'` 同理。

**深挖：collation 不是旁观者，它直接决定这个优化能不能成立**：

- `like_range` 的上界靠「该 collation 的最大排序权重字符」填充。**前缀范围优化要求 collation 满足「前缀序与排序序一致」**。绝大多数常用 collation（`utf8mb4_0900_ai_ci`、`utf8mb4_bin` 等）满足。
- 但比较语义跟着 collation 走：`utf8mb4_bin` 区分大小写、`utf8mb4_general_ci` 不区分。`name LIKE 'J%'` 在 ci collation 下区间会同时覆盖 `J` 和 `j` 的排序权重段——区间边界因此变化。这解释了一个实战困惑：同一条前缀 `LIKE` 在不同 collation 的库上 `rows` 估算和命中行差很多。
- 反例边界：`LIKE 'a%c'` 中间还有 `%`，范围优化器**只用 `'a'` 这段常量前缀建区间**，`'%c'` 成为区间内的**残余谓词**——若 `name` 在所用索引里，残余谓词可由 ICP 在引擎层过滤（§5），否则回表后在 Server 层判断。所以 `LIKE 'a%c'` 不是「失效」，是「只用了前缀那段索引 + 残余过滤」，和第 9 课 §3 Q1 是同构现象。

**解法/边界**：

- 中后缀检索（`%xx%`）的正路是 `FULLTEXT` + `MATCH...AGAINST`，中文需 `ngram` 解析器（按 N 元切词建倒排，本质换了一种索引数据结构，不再依赖前缀有序）；或外置搜索引擎（呼应第 6 课）。
- 「按后缀查」的工程技巧：加一列存 `REVERSE(col)` 并建索引，把后缀查询变前缀查询——代价是多一列 + 写时维护，属于「以空间和写放大换可索引性」。

### 2. 函数包列：范围优化器卡在哪一步，以及函数索引的内部实现

**现象**：`WHERE LENGTH(name)=6` → `ALL`。

**机制**：`WHERE` 树里这个条件左侧是 `Item_func_length(Item_field(name))`，是个 `Item_func` 节点。范围优化器遍历到它，拿它和 `index_name` 的 key part（`Field name`）比身份——不等。于是 `name` 这个 key part 上**没有任何 `SEL_ARG` 被创建**，range 路径不生成；ref 路径也要求 `field = const` 形态，同样不成立 → 只剩全表扫，逐行算 `LENGTH` 再判。这就是 §0 那条脊柱的最直接体现。

**深挖：函数索引（8.0.13+）到底是什么**：

```sql
ALTER TABLE t_user ADD INDEX idx_name_len ((LENGTH(name)));
```

它**不是**「对函数结果建的特殊索引」，而是 MySQL 内部偷偷建了一个**隐藏的虚拟生成列**（`GENERATED ALWAYS AS (LENGTH(name)) VIRTUAL`），再在这个隐藏列上建一个**普通 B+ 树二级索引**。于是「按 `LENGTH(name)` 有序的结构」真实存在了，范围优化器能在这个隐藏列的 key part 上建区间。代价也跟着来：虚拟列在**读取/索引维护时计算**，写入要维护这棵新树（索引通用代价，第 9 课 §7）。

精确边界（容易踩，且原文没讲到点子上）：

- **必须确定性**：表达式不能含 `NOW()`、`RAND()`、用户变量、子查询、非确定存储函数——否则 `ALTER` 直接报错（生成列要求确定）。
- **匹配的是 item tree，不是文本**：优化器把查询里的表达式解析成 item 树，和索引定义的表达式 item 树做结构比对。`LENGTH(name)` 命中；`length(name)`（大小写）命中（标识符/函数名归一）；但 `LENGTH(name)+0`、`CHAR_LENGTH(name)`、`LENGTH(CONCAT(name,''))` **都不命中**——树结构不同。所以函数索引很「脆」，查询写法必须和索引定义严格同构。
- 典型正用场景是 JSON：`((CAST(doc->>'$.uid' AS UNSIGNED)))`；以及多值索引 `((CAST(doc->'$.tags' AS CHAR(20) ARRAY)))`（8.0.17+），配合 `MEMBER OF` / `JSON_CONTAINS` / `JSON_OVERLAPS` 走索引。
- 但 90% 情况更该做的是**改写谓词让裸列回到一侧**（§3），而不是堆函数索引。

### 3. 表达式计算：常量折叠 vs 代数反解，以及 SARGable 的代码学定义

**现象**：`WHERE id+1=10` → `ALL`；`WHERE id=10-1` → 走索引。

**机制**：`id+1=10` 左侧是 `Item_func_plus(Item_field(id), 1)` → 同 §2，key part 身份不匹配，建不出区间。`id=10-1` 左侧是裸 `Item_field(id)`，右侧 `10-1` 是常量表达式，范围优化器在建区间前会对**右侧做常量折叠**（求值成 `9`）→ 区间 `[9,9]` → 走索引。

**深挖：为什么 MySQL 做常量折叠却坚决不做 `id+1=10 → id=9` 的移项**——这不是偷懒，是有意的设计边界：

- 常量折叠是对**右侧子树**求值，输入输出类型确定、无副作用、永远安全。
- 代数反解是对**含索引列的左侧**做符号变换。要对所有形态正确：`id*2=10`（整数除可能不整除）、`id-1=10`、`-id=10`、`id&3=1`（位运算不可逆）、`f(id)=x`（多数函数无逆）、含多列、溢出与精度（`id+1` 在 `id=BIGINT_MAX` 溢出，反解 `id=10-1` 与原式语义不等价）。要安全覆盖这些，优化器代码复杂度和出错风险都不可接受。MySQL 的选择是把「写成可索引形态」的责任交给开发者，并通过这条规则把契约讲明白。
- 这正是 **SARGable** 的精确定义（很多文章只会喊这个词）：一个谓词 SARGable ⟺ **它能被范围优化器化简成「裸 index key part `CMP` 常量」从而生成 `SEL_ARG`**。§1–§4 的失效，本质都是写出了 non-SARGable 谓词——左侧不是裸 key part。这比「破坏有序性」精确得多，也能直接指导改写：把一切作用在索引列上的运算/函数/转换挪到常量侧或消除掉。

### 4. 隐式类型转换：不只是慢，是会多匹配行的正确性 bug

**现象**：`phone` 是 `varchar` 二级索引，`WHERE phone = 13800138000`（数字）→ `ALL`；`id` 是 `int`，`WHERE id = '1'`（字符串）→ 走索引。

**先定转换规则**，经典判定法：

```sql
SELECT '10' > 9;   -- = 1
```

结果 `1` 说明：字符串与数字比较时，MySQL 按 SQL 标准把**两侧都按浮点数（REAL/DOUBLE）**比较，即字符串被转成数字。

**机制（落点决定失效）**：

- `phone（varchar） = 13800138000`：为按数字比较，等价于 `CAST(phone AS DOUBLE) = 13800138000`——`CAST` 套在**索引列**上，退化成 §2 的「函数包列」，范围优化器在 `phone` key part 上建不出区间 → 全表扫。
- `id（int） = '1'`：等价于 `id = CAST('1' AS DOUBLE)`——`CAST` 套在**常量**上，`id` 仍是裸 key part，区间 `[1,1]` → 走索引。
- 一句话：**字符串↔数字时一律「字符串转数字」；看 `CAST` 被迫落在列上还是常量上**。由此推出方向性结论：字符串列传数字会失效，数字列传字符串不会。

**深挖一：这同时是正确性 bug，不止性能**（原文完全没提，但这是更值钱的点）。字符串转 DOUBLE 是**宽松且有损**的：`'13800138000'`、`'13800138000abc'`、`' 13800138000'`、`'1.38e10'`、`'+13800138000'`、`'13800138000.0'` 转成 DOUBLE 后**都等于 `1.38e10`**，于是 `WHERE phone = 13800138000` 会**额外匹配到这些本不该等的行**——结果集是错的，不只是慢。而且超过 15~16 位有效数字时 DOUBLE 精度不足，长数字串比较会出现「看起来相等其实不等 / 看起来不等其实相等」。**所以「参数类型对齐列类型」不是性能优化建议，是正确性要求。** 排查可用 `SHOW WARNINGS`：隐式转换通常会留下警告，`EXPLAIN` 的 `filtered`/`rows` 异常偏大也是信号。

**深挖二：join 时的字符集隐式转换——更隐蔽的同类坑**。两表关联列 `a.uid utf8mb3` vs `b.uid utf8mb4`，比较前 MySQL 必须统一字符集。统一方向由 **collation coercibility（强制性等级）** 决定：显式 `COLLATE`=0、列=2、常量=4、`NULL`=6，**数值越小越「不可强制」、越被保留**；两列都是 level 2 时，按字符集「可无损转换」的方向走——`utf8mb3` 是 `utf8mb4` 子集，于是 `utf8mb3` 那一侧被转成 `utf8mb4`（`CONVERT(a.uid USING utf8mb4)`）。结果：**被转换的那一侧 = 索引列被函数包裹 = §2，该侧 join 用不上索引**，`type` 退化为 `ALL`/`index`，大表 join 直接拖垮。`EXPLAIN` 不报错、不直观，极易漏诊。根治：建表期就统一 `CHARSET`+`COLLATION`，尤其所有用于 join/外键的列；存量问题用 `ALTER TABLE ... CONVERT TO CHARACTER SET` 对齐，而不是在 SQL 里到处加 `CONVERT`（那只是把失效从一侧挪到另一侧）。

### 5. 联合索引非最左 & 索引截断：ICP 在 handler 边界做了什么，skip scan 的精确触发条件

最左匹配为什么失效（联合索引多列字典序、脱离左前缀全局无序）第 9 课 §2 已用 B+ 树排序讲透，这里只深挖两个机制点。

**索引截断 `(a,b,c)` 上 `WHERE a=1 AND c=3`：ICP 到底在哪条边界、做了什么**：

- 范围优化器只能在 `a` 上建区间（`a=1`）；`c` 跳过了 `b`，无法进入扫描区间边界，`key_len` 只算到 `a`。
- 5.5：`a` 定位 → 拿主键回表取整行 → **Server 层**判断 `c=3`。每条都回表，浪费。
- 5.6+ ICP：优化器发现 `c` 这一列**就在 `(a,b,c)` 索引里**，于是通过 `handler::idx_cond_push()` 把谓词 `c=3` **推给存储引擎**。InnoDB 在扫 `a=1` 这段二级索引叶子时，**对每条索引记录先就地用索引里的 `c` 值判断 `c=3`，不满足直接跳过、不回表**；满足才用主键回聚簇索引取整行。`Extra=Using index condition` 即此。
- 深挖边界：ICP **只对二级索引**（聚簇索引叶子已是整行，无回表可省）；推下去的谓词**只能引用该索引覆盖的列**（`c` 在索引里才能下推，引用索引外的列无法下推）；它省的是**回表的随机 IO**，不是索引扫描本身。这与第 9 课 §3 Q1（`a>1 and b=2` 中 `b` 走 ICP）完全同构——「没进 `key_len`」≠「没用」。

**index skip scan（8.0.13+）的精确触发条件**（原文没提，重要例外但别记反）：它属于 loose index scan 家族。对 `(a,b)` 上 `WHERE b=2`（缺最左列），优化器估 `a` 的不同值个数（用 `records_per_key`/统计信息）；当 `a` 的**基数很低**（distinct 值很少）时，它把查询拆成「对 `a` 每个不同值 `v` 做一次 `a=v AND b=2` 的范围扫，再拼起来」，相当于替你补齐缺失的最左前缀。`Extra=Using index for skip scan`，受 `optimizer_switch='skip_scan=on'` 控制。**触发的关键是「被跳过的前导列基数低」**——基数高时拆分次数太多反而比全表扫还贵，优化器会放弃。结论：skip scan 是「非最左」在低基数前导列下的有限兜底，索引设计仍按最左匹配来。

### 6. OR 接非索引列：index_merge 三种算法为什么对 rowid 有序性要求不同

**现象**：`id` 主键、`age` 无索引，`WHERE id=1 OR age=20` → `ALL`；给 `age` 建索引 → `type=index_merge`。

**机制**：`OR` 要「满足其一」。`age` 无索引 → 判断 `age=20` 必须看每一行 → 这一支天然全表扫 → 整条退化全表扫，`id` 的区间也没意义（范围优化器对 `OR` 建 `SEL_TREE` 时，一支无法成树，`OR` 合并结果即「全集」）。

**深挖：两侧都建索引后，index_merge 的三种内部算法，差别在 rowid（主键）有序性**：

- **union**（`OR` 用）：分别用 `id`、`age` 两个索引各扫一遍，得到两批主键，用一个 `Unique`（去重结构，内存放不下落盘）**去重并集**后再回表。
- **sort-union**：当各分支产出的主键**不是按主键有序**时，无法流式归并，先把主键全收集、**排序**，再归并去重。比 union 多一次排序成本，是 union 的退化形态。
- **intersection**（`AND` 多索引各自选择性都不高时）：要求每个分支都是 **ROR（rowid-ordered retrieval，按主键有序输出）**，这样能像归并排序一样**流式取交集**，无需缓存全部主键。InnoDB 二级索引末尾隐含追加主键，天然能提供主键有序输出，这是 intersection 可行的结构基础。

**深挖：为什么实战常建议用 `UNION` 改写而不是依赖 index_merge**：index_merge 的代价模型**历史上以低估/高估著称**——去重/排序/多次回表的成本难估准，优化器时而该用不用、时而误用导致更慢。`UNION`/`UNION ALL` 改写把每个分支固定成一条干净的 `ref`/`range`，**执行计划确定、可预测、可单独验证**：

```sql
SELECT ... FROM t WHERE id=1
UNION
SELECT ... FROM t WHERE age=20;   -- 注意 UNION 去重语义与 OR 一致；确认无重复可用 UNION ALL 更快
```

可用 `optimizer_switch` 的 `index_merge / index_merge_union / index_merge_sort_union / index_merge_intersection` 分别开关来诊断优化器是否误判。边界澄清：`status=1 OR status=2` 这种**同列等值 OR**会被归一成 `IN (1,2)`、走单索引区间，**不属于失效**——别把所有 `OR` 一棍子打死。

### 7. 收束：一条机制，六种表象

不要背六条，记这条因果链：

> `WHERE` → 范围优化器逐谓词尝试建 `SEL_ARG` → **只有「裸 index key part `CMP` 常量」能建出区间** → 否则该索引路径不生成 → 全表扫。

六场景全是它的表象（含解法/边界）：

| 场景 | 机制定位（都归到「左侧非裸 key part」或「建不出有界区间」） | 解法 | 验证 |
| --- | --- | --- | --- |
| 左/中模糊 `%xx` | `like_range` 无常量前缀 → 建不出有界区间 | 前缀化 / FULLTEXT+ngram / 反转列 | `type ALL→range` |
| 函数包列 `f(col)` | 左侧 `Item_func`，与 key part 身份不匹配 | 改写谓词 / 函数索引（item tree 须严格同构） | `key` 非 NULL |
| 表达式 `col±1=x` | 同上；优化器只折叠右侧常量，不反解左侧 | 把运算移到常量侧（写成 SARGable） | `type/key` |
| 隐式转换（串列+数字） | 等价 `CAST(col)`，落到列上 = 函数包列；**且多匹配行** | 参数类型对齐列；join 列统一 charset/collation | `SHOW WARNINGS` 看 cast |
| 非最左/截断 | 缺左前缀，目标列在 B+ 树全局无序，建不出区间 | 调列序；低基数前导列可被 skip scan 兜底 | `key_len` 看用了几列 |
| `OR` 接非索引列 | 一支无索引→`SEL_TREE` 退化全集 | 两侧建索引（index_merge）或改 `UNION` | `type=index_merge` |

工程总纲一句话：**让索引列在谓词里裸露在比较的一侧、不被任何函数/算术/类型/字符集转换包裹，且联合索引从最左列连续使用**——这就是 SARGable，是「能吃索引的 SQL」的充要写法。排查永远以执行计划为准（`type` 至少 `range`，看 `key/key_len/Extra/filtered` 与 `SHOW WARNINGS`）。

### 8. 实战案例：四个真实线上事故复盘

机制讲完，看它在生产里怎么咬人。每个案例按「现象 → 误判 → 定位 → 根因（对应上文哪节）→ 修复 → 事后」复盘。

#### 案例一：测试飞快、全量上线后 DB CPU 打满——join 字符集不一致

- **现象**：新订单查询服务关联老 `user(uid)` 与新 `order_ext(uid)`，两列都有索引。测试环境（几千行）2ms，灰度全量后接口 P99 从 20ms 飙到 3s，DB CPU 100%。
- **误判**：查了两表 `uid` 都有索引 → 怀疑「数据量大了就是慢」，差点直接上分库分表。
- **定位**：抓真实参数 `EXPLAIN`，驱动表 `ref`、被驱动表 `type=ALL`；`SHOW CREATE TABLE` 发现 `user` 是 `utf8mb3`、`order_ext` 是 `utf8mb4`；`SHOW WARNINGS` 有隐式 `convert` 痕迹。
- **根因**：§4 深挖二。join 比较前统一字符集，coercibility 相同时按「可无损转换」方向把 `utf8mb3` 侧 `CONVERT(... USING utf8mb4)` 包起来——该侧 `uid` 索引失效，每次关联在被驱动表全表扫。小数据扫不出代价，全量线性放大。
- **修复**：短期把新表对齐老表 `ALTER TABLE order_ext MODIFY uid VARCHAR(...) CHARACTER SET utf8mb3`（影响面小、风险低）；长期统一 `utf8mb4`。**挑战点**：大表改字符集是 rebuild 级重 DDL（锁、binlog 暴涨、主从延迟），生产必须走 `gh-ost`/`pt-online-schema-change` 在线变更，并预估磁盘和复制延迟，不能直接 `ALTER`。
- **事后**：「join/外键列必须同 `charset`+`collation`」写进建表规范，并加 CI 检查（扫 `information_schema.columns` 比对关联列）。

> 这个案例最毒的地方：**它在测试环境永远复现不了**（小表全表扫也快），是「索引失效」里最典型的「线上才爆」类型。

#### 案例二：客服查单偶发「查到别人订单」——隐式转换是数据安全事故

- **现象**：`WHERE phone = ?`，`phone varchar`。某次重构 DAO 把参数按 `long` 传，SQL 变成 `phone = 13800138000`（无引号）。慢是其次，关键是偶发**越权**投诉：A 的客服查到了 B 的订单。
- **误判**：当成偶发数据脏 / 缓存串号，查了一周缓存。
- **定位**：`SHOW WARNINGS` 出现 `Truncated incorrect DOUBLE value`；`EXPLAIN` `type=ALL`、`rows` 异常大、`filtered` 偏低。
- **根因**：§4 深挖一。`varchar`→`DOUBLE` 有损宽松，脏数据 `'13800138000-1'`、带空格 `' 13800138000'`、超 16 位号码精度丢失，**都 cast 成同一个 double 与常量相等** → `=` 命中多行，把别人订单带出来。这不是性能问题，是**正确性 + 越权安全事故**。
- **修复**：参数严格按列类型传字符串；加单测断言绑定参数类型；CI 静态扫描禁止「数字字面量比较字符串列」。
- **事后**：作为团队「隐式转换 = 数据安全问题，不只是慢」的反面教材。**这是把 §4 那段理论变成血淋淋后果的案例——务必让人记住「列类型对齐」是正确性要求不是优化建议。**

#### 案例三：建了函数索引还是全表扫——函数索引没生效

- **现象**：报表 `WHERE DATE(create_time) = '2024-05-01'` 慢。DBA 建 `ADD INDEX idx_d ((DATE(create_time)))`，上线后 `EXPLAIN` 仍 `ALL`。
- **误判**：以为 MySQL 版本不支持 / 索引没建成功。
- **定位**：`SHOW INDEX` 确认索引在；但应用层 ORM 实际生成的是 `CAST(create_time AS DATE) = '2024-05-01'`，item tree 与索引定义 `DATE(create_time)` **不同构** → §2 深挖「按 item tree 匹配」，不命中。
- **根因**：双重。① 函数索引表达式与查询写法不同构；② **更根本：这里压根不该用函数索引**。`create_time` 本身就有索引，正解是把谓词改成 SARGable 范围（§3）：`create_time >= '2024-05-01 00:00:00' AND create_time < '2024-05-02 00:00:00'`，走原索引、还能 `range` 扫，比函数索引（全表/全索引扫隐藏列）更优。
- **修复**：优先改写成裸列范围谓词；确需函数索引时，用 `EXPLAIN` 实测**真实 ORM 出参的 SQL**是否命中，别假设。
- **事后**：函数索引是最后手段；「能把函数从列上挪走、改成裸列范围」永远优先。

#### 案例四：同一条 SQL 有时 2ms 有时 800ms——index_merge 计划抖动

- **现象**：`WHERE user_id = ? OR order_no = ?`（两列都有索引），多数 2ms，特定参数 800ms，监控偶发尖刺，难复现。呼应第 6 课「同 SQL 不同参数计划不同」。
- **误判**：归因到锁等待 / 网络抖动，查了 `innodb_trx`。
- **定位**：抓慢样本真实参数 `EXPLAIN`：部分参数走 `index_merge union`，部分因选择性/统计误估走 `ALL` 或 union 去重落盘。
- **根因**：§6。index_merge 代价模型不稳，随参数选择性与统计信息 plan flip。
- **修复**：改写成 `UNION`（去重语义与 `OR` 对齐；确认无重叠可 `UNION ALL` 更快），每分支固定 `ref`，计划稳定可预测、可单独验证。
- **事后**：高频「不同列 `OR`」默认用 `UNION` 改写，不赌 index_merge。

### 9. 实战挑战：八道推演题（先自己判，再看解析）

下面每题都卡在「简单规则会答错、必须用机制推」的点上。建议先写下你的答案和 `key_len`/`type`/`Extra` 预测，再看解析。

**C1.** `(a,b)` 索引，`WHERE a IN (1,2,3) AND b > 5`，`b` 用到索引吗？
> **解析**：用。范围优化器把 `IN` 展开成 `a=1 ∪ a=2 ∪ a=3` 三个**等值点**，每个等值点内部 `b` 有序，`b>5` 在每个点下都是有界子区间 → `a`、`b` 都进 `key_len`。这与 `a>1 AND b>5`（纯开区间、无等值点，`b` 不能用，第 9 课 §3 Q1）**本质不同**：`IN` ≈ 多个等值，不是范围。边界：`IN` 列表过大超 `range_optimizer_max_mem_size` 时优化器会放弃逐点展开而退化。

**C2.** `(a,b)` 索引，`WHERE a = 1 OR a = 2 AND b = 3` 会失效吗？
> **解析**：不失效。运算符优先级陷阱：`AND` 紧于 `OR`，等价 `a=1 OR (a=2 AND b=3)`。范围优化器建 `SEL_TREE` 并集：`{a=1 整段} ∪ {a=2 且 b=3 的点}`，两支都从最左列 `a` 出发，合并成 `range`。常见误读是「见 `OR` 就失效」——`OR` 失效的前提是**某一支无可用索引**（案例四/§6），不是 `OR` 本身。教训：先括号化再套机制。

**C3.** InnoDB，`(name)` 二级索引，`SELECT id FROM t WHERE LENGTH(name)=6`，`EXPLAIN` 的 `type`/`Extra`？
> **解析**：`range` 仍建不出（§2，左侧函数包列）。但要的列只有 `id`（二级索引叶子隐含主键 `id`），过滤只依赖 `name`（在索引里）→ 走 **`type=index`（全索引扫描）+ `Extra: Using where; Using index`**：扫整棵 `(name)` 索引树不回表，逐条算 `LENGTH` 过滤。它仍是 O(n)，但比 `type=ALL` 扫聚簇索引整行轻（二级索引页小、IO 少，第 8 课 `y` 更大）。**核心认知：「走了索引(`type=index`)」≠「用索引定位(`range`/`ref`)」**，第 6 课 type 体系最大陷阱，面试高频。

**C4.** `(a)` 二级索引，`WHERE a IS NULL` 走索引吗？`WHERE a != 1` 呢？
> **解析**：破谣言。InnoDB 索引**会存 `NULL`，且 `NULL` 排最前、有序** → `a IS NULL` 可走 `ref`（`key_len` 含 1 字节 NULL 标记）。`a != 1`/`a <> 1`/`a IS NOT NULL` 可转成 `a<1 OR a>1` 形态的范围，优化器**按代价决定**：命中比例高（>~30%，第 9 课 §4）就弃用索引走全表扫。所以「`NULL`/`!=` 一定不走索引」是错的——是**选择性 + 代价**决定，看 `EXPLAIN`。

**C5.** `create_time` 是 `datetime` 且有索引，`WHERE create_time = '2024-05-01'` 与 `WHERE create_time = 20240501`，哪个失效？
> **解析**：前者不失效——字符串字面量被转成 `datetime`，转换落**常量侧**，列裸露，走索引（且是正确写法）。后者失效——`datetime` 与数字比较，`datetime` 转数字，等价 `CAST(create_time)` 落**列侧**（§4），且 `20240501` 当数字比较语义本身也错。方向规律同 §4：转换落常量侧安全、落列侧失效。教训：日期用字符串字面量/参数化，别传裸数字。

**C6.** `name varchar(200)`，前缀索引 `idx_name(name(10))`，`WHERE name='x' ORDER BY name` 能避免 `filesort` 吗？
> **解析**：不能。前缀索引只保证前 10 字符有序，「前 10 相同、第 11+ 不同」的记录间完整 `name` 顺序不确定 → 仍需 `filesort`（第 9 课 §7：前缀索引不能用于 `order by`、不能覆盖）。延伸：等值 `name='x'` 用前缀定位后还要**回表精确比对完整 `name`**（前缀索引有假阳性，必须 recheck）。教训：前缀索引省空间，代价是丧失排序与覆盖能力。

**C7.** `(status, create_time)`，5000 万行，`WHERE create_time > ?`，`status` 只有 0/1。会 skip scan 吗？业务新增 `status=2` 占 90% 数据后呢？
> **解析**：初始 `status` 基数=2 极低 → 优化器可能 `Using index for skip scan`（对 `status∈{0,1}` 各做一次 `status=v AND create_time>?` 合并）。新增 `status=2` 占 90% 后：基数仍低但分布极度倾斜，`create_time>?` 命中比例可能很高，代价重估后**可能放弃 skip scan 退回全表扫**——表现为「上线一个新状态后这条查询毫无改动却突然变慢」。教训：skip scan 依赖前导列基数+分布，统计变化会让计划翻车（呼应第 6 课偶发慢/优化器误选）；要稳定就别指望 skip scan，调整索引列序或让 `create_time` 能独立走索引。

**C8.** 多值/函数索引 `idx ((CAST(JSON_EXTRACT(doc,'$.uid') AS UNSIGNED)))`，`WHERE doc->>'$.uid' = 100` 命中吗？`WHERE CAST(doc->>'$.uid' AS UNSIGNED)=100` 呢？
> **解析**：都不命中。`doc->>'$.uid'` = `JSON_UNQUOTE(JSON_EXTRACT(...))`，item tree 与索引定义 `CAST(JSON_EXTRACT(...) AS UNSIGNED)` 不同构；第二个虽有 `CAST`，但内层多了 `JSON_UNQUOTE`，仍不同构 → 不命中。**必须逐节点同构**（写成 `CAST(JSON_EXTRACT(doc,'$.uid') AS UNSIGNED)=100`）才命中。这是 §2「函数索引脆、按 item tree 匹配」的实证。教训：函数索引上线必须 `EXPLAIN` 验证**真实查询写法**命中，绝不能假设。

> 八题的共同点：用「简单口诀」（见 `OR` 失效、`!=` 失效、`NULL` 不走索引、建了索引就生效、走了索引就快）都会答错；用 §0 那条脊柱（能否为裸 key part 建出区间 / 优化器按代价与 item tree 决策）才能答对。这就是「机制 > 规则」的价值。

## 小结（3-5 条关键点）

- 索引失效的真实机制是「范围优化器无法为该谓词在索引 key part 上建出 `SEL_ARG` 区间」；它只认「裸 index key part `CMP` 常量」，左侧被任何东西包裹就匹配不到 key part——这是「破坏有序性」直觉底下的代码学原因，也是 SARGable 的精确定义。
- `LIKE '前缀%'` 被 `like_range` 按 collation 算成有界区间才走 range；无常量前缀建不出区间；collation 直接影响区间边界与命中行数，`'a%c'` 只用 `'a'` 段、`'%c'` 是残余谓词（同 ICP 同构）。
- 隐式类型转换不仅让索引列等价被 `CAST` 包裹而失效，更会因字符串→DOUBLE 的有损宽松转换**多匹配行**（正确性 bug）；join 时字符集 coercibility 决定哪一侧被转换、哪一侧索引失效，是最隐蔽的高发线上事故。
- 函数索引本质是「隐藏虚拟生成列 + 普通索引」，要求表达式确定性，且查询表达式必须与索引定义 item tree 严格同构才命中；ICP 经 `handler::idx_cond_push` 只对二级索引、只下推索引内列，省的是回表随机 IO；skip scan 仅在被跳过的前导列低基数时兜底。
- index_merge 的 union/sort-union/intersection 差别在对主键（rowid）有序性的要求；其代价模型不稳，`UNION`/`UNION ALL` 改写更可预测；同列等值 `OR` 会转 `IN` 不算失效。
