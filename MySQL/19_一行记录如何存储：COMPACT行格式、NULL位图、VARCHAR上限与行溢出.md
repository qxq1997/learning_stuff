# MySQL - 第 19 课：一行记录如何存储：COMPACT 行格式、NULL 位图、VARCHAR 上限与行溢出

> 前面我们从数据页角度看过 B+ 树，也知道 InnoDB 默认按 16KB 的页读写数据。但页里真正存放的是一条条行记录。那一行记录内部到底怎么摆？`NULL` 值占不占空间？`varchar(n)` 的 `n` 到底按字符算还是字节算？为什么一个 `varchar(65535)` 建表会失败？行太大时 InnoDB 又怎么处理？这一课就把这些问题放到一条记录的存储结构里讲清楚。

## 学习目标

- 能从表空间、段、区、页、行的层级理解一条记录到底存在哪里。
- 能说清 InnoDB 常见行格式，尤其是 COMPACT 行格式的组成。
- 能解释 `NULL` 值为什么不直接放在真实数据区，而是用 NULL 位图表示。
- 能解释 `varchar(n)` 的实际字节长度为什么需要单独记录。
- 能推导单个 `varchar(n)` 和多个变长字段下的最大行长度限制。
- 能说明 COMPACT、DYNAMIC 在行溢出处理上的区别。
- 能把行格式知识落到表设计建议上：字段是否 `NOT NULL`、大字段怎么放、为什么大行会影响 Buffer Pool 和索引效率。

## 先从一条记录存在哪里说起

MySQL 不是自己直接管理所有数据文件，真正的存储行为由存储引擎决定。我们日常最常用的是 InnoDB，所以这一课默认讨论 InnoDB。

先建立一张从大到小的地图：

```text
表空间 tablespace
  -> 段 segment
    -> 区 extent
      -> 页 page
        -> 行 row
```

也就是说，一条记录不是直接“躺在表文件里”这么简单，而是位于某个表空间文件里的某个数据页中。

### 数据文件在哪

在早期和 MySQL 5.7 及以前的资料里，你经常会看到：

```text
/var/lib/mysql/数据库名/
  db.opt
  表名.frm
  表名.ibd
```

大致含义是：

- `db.opt`：数据库默认字符集和校验规则等信息。
- `.frm`：表结构元数据。
- `.ibd`：InnoDB 独立表空间文件，存放表的数据和索引。

不过这里要加一个版本提醒：

**MySQL 8.0 已经移除了 `.frm` 文件，表结构元数据改由事务型数据字典管理。**

但对我们理解本课核心并不冲突。你只需要抓住：

```text
InnoDB 的表数据和索引，通常存放在表空间中；
开启 innodb_file_per_table 时，一张表通常有自己的 .ibd 独立表空间文件。
```

`innodb_file_per_table` 从 MySQL 5.6.6 开始默认开启，所以很多场景下，一张 InnoDB 表对应一个独立的 `.ibd` 文件。

## 表空间、段、区、页、行

### 1. 行：逻辑上的一条记录

表里的每条数据，逻辑上就是一行。

例如：

```sql
create table t_user (
  id int primary key,
  name varchar(20),
  phone varchar(20),
  age int
) engine = InnoDB;
```

一行可能是：

```text
id=1, name='a', phone='123', age=18
```

但这只是 SQL 层看到的“逻辑行”。在 InnoDB 页里，它会被拆成：

- 记录额外信息；
- 隐藏字段；
- 用户定义字段值；
- 某些大字段的溢出页指针。

### 2. 页：InnoDB 读写磁盘的基本单位

InnoDB 不会查一行就只从磁盘读一行。

它按页读写。默认页大小是 16KB。

这意味着：

- 查一行时，如果目标页不在 Buffer Pool，会把整个 16KB 页读入内存。
- 改一行时，先改 Buffer Pool 中的页，形成脏页。
- 刷盘时，也通常以页为单位写回磁盘。

页的类型有很多：

- 数据页；
- 索引页；
- undo 页；
- 溢出页；
- 系统页等。

我们这一课重点看数据页里的用户记录。

### 3. 区：连续的页，减少范围扫描随机 I/O

一个区（extent）默认是 1MB。

如果页大小是 16KB，那么一个区就是：

```text
1MB / 16KB = 64 个连续页
```

为什么要有区？

因为 B+ 树叶子页之间虽然通过双向链表逻辑连续，但如果物理位置太分散，范围扫描会变成大量随机 I/O。

InnoDB 在表变大后，会更多按区分配空间，让一批页物理上更接近，从而提升范围扫描效率。

### 4. 段：按用途组织区

段（segment）由多个区组成。

常见段包括：

- 数据段：存放 B+ 树叶子节点。
- 索引段：存放 B+ 树非叶子节点。
- 回滚段：存放 undo 相关数据。

你可以这样记：

```text
页负责真正装记录；
区负责连续空间；
段负责按用途组织区；
表空间负责承载整张表或多个对象的存储。
```

## InnoDB 有哪些行格式

行格式（row format）就是一条记录在页内的存储结构。

InnoDB 常见行格式有：

- `REDUNDANT`
- `COMPACT`
- `DYNAMIC`
- `COMPRESSED`

它们的历史关系可以简化理解：

- `REDUNDANT` 很老，空间利用率差，现在基本不作为学习重点。
- `COMPACT` 是更紧凑的格式，MySQL 5.0 之后引入。
- `DYNAMIC` 和 `COMPRESSED` 基于 COMPACT 思路改进，尤其在大字段溢出处理上不同。
- MySQL 5.7/8.0 常见默认行格式是 `DYNAMIC`。

这一课重点讲 `COMPACT`，因为把它理解了，`DYNAMIC` 也很容易理解。

## COMPACT 行格式总览

一条 COMPACT 记录可以分成两大部分：

```text
记录的额外信息
  -> 变长字段长度列表
  -> NULL 值列表
  -> 记录头信息

记录的真实数据
  -> 隐藏字段
  -> 用户定义列的值
  -> 溢出页指针，必要时存在
```

用文字画一下：

```text
| 变长字段长度列表 | NULL 值列表 | 记录头信息 | row_id | trx_id | roll_ptr | 列1值 | 列2值 | ... |
```

其中：

- 前半部分是 InnoDB 为了管理记录额外附加的信息。
- 后半部分是这条记录真正的数据。

接下来拆开看。

## 变长字段长度列表：MySQL 怎么知道 varchar 实际占了多少字节

`varchar(n)` 是变长字段。

例如：

```sql
name varchar(20)
```

如果存的是：

```text
'a'
```

实际只需要 1 个字符对应的字节数；如果存的是：

```text
'xiaolin'
```

实际字节数又变了。

所以 InnoDB 必须在记录中保存：

```text
这个变长字段这一次实际占用了多少字节。
```

这个信息就放在“变长字段长度列表”里。

### 一个例子

假设表结构是：

```sql
create table t_user (
  id int primary key,
  name varchar(20),
  phone varchar(20),
  age int
) charset = ascii
  row_format = compact;
```

这里 `name` 和 `phone` 是变长字段，`id` 和 `age` 不是。

假设第一行是：

```text
id=1, name='a', phone='123', age=18
```

在 `ascii` 字符集下，一个字符占 1 字节：

- `name='a'`：实际占 1 字节，十六进制 `0x01`。
- `phone='123'`：实际占 3 字节，十六进制 `0x03`。

那么变长字段长度列表里保存的是：

```text
03 01
```

你会发现它不是 `01 03`，而是逆序。

### 为什么逆序存放

COMPACT 记录里，记录头和真实数据的起始位置被设计成非常重要的定位点。

变长字段长度列表逆序存放，有两个好处：

1. 解析记录时，可以从记录头附近向左读取额外信息，向右读取真实数据。
2. 靠前列的真实数据和对应长度信息更容易离得近，有利于局部性和 CPU Cache 命中。

不用把 CPU Cache 这点背成玄学，只需要记住：

**行格式不是随便摆的，它在尽量让读取记录时少跳来跳去。**

### NULL 的变长字段，不会记录长度

如果某个变长字段是 `NULL`，它的值不会出现在真实数据区，因此变长字段长度列表也不需要记录它的长度。

例如：

```text
id=3, name='ccc', phone=NULL, age=NULL
```

`phone` 是 `NULL`，所以它不会在长度列表里保存 `phone` 的长度。

这也引出下一个问题：那 MySQL 怎么知道哪个字段是 `NULL`？

靠 NULL 值列表。

## NULL 值列表：NULL 不是直接存在真实数据区

在 COMPACT 行格式中，允许为 `NULL` 的列，会用一个位图来标记是否为 `NULL`。

它的规则是：

- 每个允许为 `NULL` 的列，对应 1 个 bit。
- bit 为 `1`，表示该列值是 `NULL`。
- bit 为 `0`，表示该列值不是 `NULL`。
- 位图按列顺序逆序排列。
- 位图必须按整数字节存储，不足 1 字节时高位补 0。

还是用这张表：

```sql
create table t_user (
  id int primary key,
  name varchar(20),
  phone varchar(20),
  age int
) charset = ascii
  row_format = compact;
```

假设 `name`、`phone`、`age` 都允许为 `NULL`，它们对应的 NULL 位图会按逆序表达。

### 第一行：没有 NULL

```text
id=1, name='a', phone='123', age=18
```

所有可空列都不是 `NULL`，NULL 位图为：

```text
00000000
```

十六进制就是：

```text
0x00
```

### 第二行：age 为 NULL

```text
id=2, name='bb', phone='1234', age=NULL
```

`age` 是 `NULL`，对应 bit 为 1。

最终 NULL 位图可以表示为：

```text
0x04
```

### 第三行：phone 和 age 为 NULL

```text
id=3, name='ccc', phone=NULL, age=NULL
```

`phone` 和 `age` 都是 `NULL`，对应 bit 都为 1。

最终 NULL 位图可以表示为：

```text
0x06
```

关键点是：

**NULL 值本身不会存放在真实数据区，它只通过 NULL 位图标记。**

### 每条记录都有 NULL 值列表吗

不是。

如果表中所有列都定义为 `NOT NULL`，那么这条记录就不需要 NULL 值列表。

这也是为什么很多表设计建议里会说：

```sql
字段能明确非空，就尽量定义为 NOT NULL。
```

不过也别机械理解。`NOT NULL` 的主要价值不只是省一点存储空间，还包括：

- 表达业务约束；
- 简化 SQL 三值逻辑；
- 减少索引和统计信息里的不确定性。

空间收益只是其中一部分。

### NULL 位图固定 1 字节吗

不是。

NULL 值列表按 bit 表示，但最终按整数字节存储。

如果有 1 到 8 个可空列，需要 1 字节。

如果有 9 到 16 个可空列，需要 2 字节。

公式是：

```text
NULL 位图字节数 = ceil(允许 NULL 的列数 / 8)
```

如果没有允许 `NULL` 的列，则没有这部分。

## 记录头信息：删除标记、记录类型和下一条记录

COMPACT 行格式里的记录头信息占 5 字节。

它包含不少内部字段，最常用来理解的有：

| 字段 | 含义 |
| --- | --- |
| `delete_mask` | 是否被标记删除 |
| `min_rec_mask` | B+ 树每层非叶子节点里最小记录相关标记 |
| `n_owned` | 当前页目录分组中，这条记录拥有的记录数 |
| `heap_no` | 记录在页堆中的编号 |
| `record_type` | 记录类型 |
| `next_record` | 下一条记录的位置 |

### delete_mask：delete 通常先标记删除

当你执行：

```sql
delete from t_user where id = 1;
```

InnoDB 往往不是立刻把这条记录从页里物理移除，而是先把记录头里的 `delete_mask` 标记为 1。

后续等没有事务再需要这条记录的历史版本时，purge 线程再真正清理。

这和前面讲 MVCC、undo、purge 是连在一起的。

### next_record：页内记录是链表

页内记录之间通过 `next_record` 组织成单向链表。

这也是第 11 课讲页目录时的重要背景：

- 页内记录按索引键有序。
- 记录之间靠单向链表串起来。
- 页目录通过槽做二分定位。
- 定位到槽后，再在小分组内顺着链表扫几条。

### record_type：普通记录还是虚拟记录

`record_type` 常见值可以这样理解：

- `0`：普通用户记录。
- `1`：B+ 树非叶子节点目录项记录。
- `2`：页内最小虚拟记录 `infimum`。
- `3`：页内最大虚拟记录 `supremum`。

所以页里不只有用户数据，还会有一些用于维护页结构的虚拟记录。

## 记录真实数据：用户列之外还有隐藏字段

COMPACT 记录的真实数据区，不只包含你定义的列。

对聚簇索引记录来说，还可能包含 InnoDB 的隐藏字段：

| 隐藏字段 | 字节数 | 是否一定存在 | 作用 |
| --- | --- | --- | --- |
| `row_id` | 6 字节 | 不一定 | 表没有合适主键时，InnoDB 自动生成聚簇索引键 |
| `trx_id` | 6 字节 | 通常存在于聚簇索引记录 | 最后修改这条记录的事务 ID |
| `roll_ptr` | 7 字节 | 通常存在于聚簇索引记录 | 指向 undo 版本链中的上一版本 |

### row_id 什么时候出现

InnoDB 的数据必须按聚簇索引组织。

选择顺序大致是：

1. 如果表有主键，用主键作为聚簇索引。
2. 如果没有主键，选择第一个非空唯一索引。
3. 如果还没有，InnoDB 自动生成隐藏 `row_id`。

只有第三种情况才需要隐藏 `row_id`。

所以这也是为什么强烈建议每张 InnoDB 表都显式定义主键：

- 主键是聚簇索引的组织依据；
- 二级索引叶子节点会存主键值；
- 隐藏 `row_id` 不利于业务理解和问题排查。

### trx_id 和 roll_ptr 连接到 MVCC

`trx_id` 和 `roll_ptr` 是 MVCC 的地基。

当事务更新一行时：

- 当前记录的 `trx_id` 变成当前事务 ID。
- `roll_ptr` 指向 undo log 中的旧版本。

于是多次更新后形成版本链：

```text
当前版本 -> undo 中的旧版本 -> 更旧版本 -> ...
```

快照读时，InnoDB 根据 Read View 判断当前版本是否可见。如果不可见，就沿着 `roll_ptr` 找旧版本。

所以一行记录的物理结构，和 MVCC 并不是两件事。

## varchar(n) 的 n 最大是多少

这是面试里非常喜欢问的问题。

先抓三个前提。

### 前提一：n 是字符数，不是字节数

```sql
varchar(100)
```

表示最多 100 个字符，不是最多 100 字节。

如果字符集是 `ascii`，1 个字符最多 1 字节。

如果是历史 `utf8`/`utf8mb3`，1 个字符最多 3 字节。

如果是现在更常用的 `utf8mb4`，1 个字符最多 4 字节。

所以同样是 `varchar(100)`：

```text
ascii     最多约 100 字节
utf8mb3   最多约 300 字节
utf8mb4   最多约 400 字节
```

### 前提二：MySQL 行大小有 65535 字节限制

MySQL 规定：除了 `TEXT`、`BLOB` 这类大对象类型外，一行中所有列的最大字节数加起来不能超过 65535 字节。

注意几个细节：

- 这是行内列定义层面的限制，不是说一个 16KB 数据页真能完整塞下 65KB 行内数据。
- 这个限制包含一部分行存储开销，比如变长字段长度列表、NULL 位图。
- `TEXT`、`BLOB` 的大内容不按普通列完整计入这个 65535，但它们在行内仍会有指针或部分前缀等开销。

所以 `varchar(n)` 最大值不能只看 65535，还要扣掉额外开销。

### 前提三：变长字段长度本身也占空间

`varchar` 是变长字段，所以要在“变长字段长度列表”中记录它实际占用的字节数。

粗略估算时：

- 如果该变长字段最大字节数不超过 255，长度信息通常 1 字节。
- 如果该变长字段最大字节数超过 255，长度信息通常按 2 字节估算。

内部编码还有一些细节，但用于理解建表上限时，这个估算足够抓住主线。

## 单个 varchar 字段的最大值

假设只有一个字段：

```sql
create table test (
  name varchar(n) null
) engine = InnoDB
  default charset = ascii
  row_format = compact;
```

`ascii` 下 1 字符 = 1 字节。

如果 `name` 允许为 `NULL`，需要：

- 真实数据最多 `n` 字节。
- 变长字段长度列表：2 字节，因为最大长度超过 255。
- NULL 位图：1 字节。

所以：

```text
n + 2 + 1 <= 65535
n <= 65532
```

因此：

```sql
varchar(65532) null
```

在这个极端条件下可以成立，而：

```sql
varchar(65533) null
```

会因为行大小超过限制而失败。

如果改成 `NOT NULL`：

```sql
create table test (
  name varchar(n) not null
) engine = InnoDB
  default charset = ascii
  row_format = compact;
```

就不需要 NULL 位图。

这时：

```text
n + 2 <= 65535
n <= 65533
```

所以 `NOT NULL` 时可以比 nullable 多出 1 字节空间。

### 换成 utf8mb3 和 utf8mb4 怎么算

如果是 `utf8mb3`，1 个字符最多 3 字节：

```text
floor(65532 / 3) = 21844
```

如果是 `utf8mb4`，1 个字符最多 4 字节：

```text
floor(65532 / 4) = 16383
```

注意，真实建表时还会受到字段数量、是否可空、其他列长度、字符集、行格式、索引限制等影响，不要把这个数字当作所有场景通用答案。

更好的回答方式是：

```text
varchar(n) 的 n 是字符数；
最大 n 要用行最大 65535 字节，扣掉 NULL 位图和变长字段长度列表，再除以字符集最大字节数。
```

## 多字段时怎么算

多字段时就不能只算某一个 `varchar` 了。

总原则是：

```text
所有列最大字节数
+ 变长字段长度列表开销
+ NULL 位图开销
<= 65535
```

例如：

```sql
create table test4 (
  id   varchar(255) not null,
  name varchar(65277) not null
) engine = InnoDB
  default charset = ascii
  row_format = compact;
```

估算：

```text
id   真实数据最大 255 字节，长度开销 1 字节
name 真实数据最大 65277 字节，长度开销 2 字节
没有 NULL 位图

255 + 1 + 65277 + 2 = 65535
```

刚好卡住上限。

如果把 `name` 改成 `varchar(65278)`：

```text
255 + 1 + 65278 + 2 = 65536
```

就超过限制。

这类计算的核心不是让你线上设计 6 万字节的 `varchar`，而是帮你理解：

**MySQL 的行大小限制是整行限制，不是单列限制。**

## 行溢出：一页放不下一行怎么办

现在出现一个看似矛盾的问题：

- MySQL 行大小限制可以接近 65535 字节。
- InnoDB 默认页大小只有 16KB。

那 16KB 的页怎么装得下一条 60KB 的记录？

答案是：装不下时会发生行溢出，长字段的数据放到溢出页。

### 溢出页是什么

溢出页就是专门存放大字段剩余内容的页。

当一条记录中的某个长字段太大时，InnoDB 不会把全部内容都塞在数据页里，而是在记录中留下一个指针，指向存放剩余数据的溢出页。

行内记录大致变成：

```text
| 记录其他信息 | 部分大字段数据或仅指针 | 20 字节溢出页指针 |
```

这个指针通常是 20 字节，用来定位溢出页中的真实内容。

### COMPACT 怎么处理行溢出

COMPACT 行格式在行溢出时，通常会在记录的真实数据区保留该长字段的一部分前缀数据，再用 20 字节指针指向溢出页。

可以简化理解为：

```text
COMPACT：
行内保留一部分数据 + 20 字节指针 -> 溢出页剩余数据
```

在很多资料中，会提到 COMPACT 对长字段可能保留 768 字节前缀。

这带来的问题是：即使大字段主体放到了溢出页，数据页里仍然会被大字段前缀占掉不少空间。

### DYNAMIC 怎么处理行溢出

DYNAMIC 行格式更激进。

它倾向于把长字段完整放到溢出页，行内只保留 20 字节指针。

可以简化理解为：

```text
DYNAMIC：
行内只保留 20 字节指针 -> 溢出页完整数据
```

这也是 MySQL 5.7/8.0 常用 `DYNAMIC` 的原因之一：它对大字段更友好，能让聚簇索引页中保留更多核心行数据。

### COMPRESSED 呢

COMPRESSED 和 DYNAMIC 的大字段溢出思路相近，同时引入页压缩。

压缩能节省磁盘空间，但会带来 CPU 开销，也会让问题分析更复杂。

实际业务里，如果没有明确压缩收益和测试数据，不建议只因为“看起来省空间”就随意开启。

## 大字段为什么容易拖慢查询

行格式不是只影响存储上限，也影响性能。

一个数据页默认 16KB。如果每行很小，一个页可以放很多行。

如果每行很大，一个页只能放很少行。

这会带来几个后果：

### 1. Buffer Pool 命中率下降

Buffer Pool 缓存的是页，不是行。

如果每页只能放少量记录，同样大小的 Buffer Pool 能缓存的行数就变少。

结果是：

```text
更多查询需要读磁盘页；
更多更新会产生更多脏页；
整体 I/O 压力更高。
```

### 2. B+ 树叶子页更稀疏

聚簇索引叶子页存整行。

行越大，叶子页能放的行越少，页数量就越多。

这会影响：

- 范围扫描页数；
- Buffer Pool 缓存效率；
- 页分裂和页合并成本；
- 回表读取的 I/O 成本。

### 3. select * 更容易把冷字段拖进来

如果表里有 `TEXT`、`BLOB`、大 `varchar` 之类字段，`select *` 很容易把并不需要的大字段也带出来。

即使 DYNAMIC 行格式把大内容放到了溢出页，访问这些字段时仍可能带来额外页读取。

所以线上建议：

```text
热查询只查需要的列；
大字段、低频字段考虑拆到扩展表；
不要让大字段污染高频查询路径。
```

## 表设计建议

### 1. 能 NOT NULL 就明确 NOT NULL

前提是业务语义真的不允许为空。

`NOT NULL` 的好处有：

- 约束更清晰；
- 减少 NULL 位图开销；
- 避免 SQL 三值逻辑带来的判断坑；
- 对索引统计和查询条件更友好。

不要为了省 1 bit 乱改语义，但也不要把所有字段都默认设计成可空。

### 2. varchar 长度按真实业务边界给

不要无脑：

```sql
varchar(255)
```

更不要为了“以后可能用得上”直接给超大长度。

字段长度会影响：

- 行最大长度校验；
- 变长字段长度列表开销；
- 临时表、排序、内存估算；
- 索引最大长度和存储成本。

### 3. 关注字符集最大字节数

`varchar(100)` 在 `ascii`、`utf8mb3`、`utf8mb4` 下不是同一个最大字节成本。

现在更推荐 `utf8mb4`，因为它能完整支持 emoji 和更多 Unicode 字符。

但你要知道：

```text
utf8mb4 下 varchar(n) 的最大字节数可能是 4n。
```

设计字段长度、索引长度时都要按最大字节数估算。

### 4. 大字段拆表

如果某些字段：

- 很大；
- 很少查询；
- 不参与过滤和排序；
- 只在详情页或导出场景使用；

可以考虑拆到扩展表：

```text
t_order
  -> 高频字段：id、user_id、status、amount、created_at

t_order_ext
  -> 低频大字段：remark、snapshot_json、extra_payload
```

这样主表聚簇索引页更紧凑，Buffer Pool 能缓存更多热数据。

### 5. 不要把行格式当成唯一优化点

行格式能解释很多现象，但真正线上优化仍要综合看：

- 查询是否走索引；
- 是否 `select *`；
- 是否频繁回表；
- Buffer Pool 命中率；
- 大字段是否被高频访问；
- 事务是否持锁过久；
- 是否有页分裂、写放大等问题。

行格式是底层地基，不是万能按钮。

## 高频面试题

### 1. MySQL 的 NULL 值会占用空间吗

会占用一点空间，但不是把 `NULL` 当成真实值放在数据区。

在 COMPACT 行格式中，允许为 `NULL` 的列会通过 NULL 值列表中的 bit 标记：

```text
1 表示该列为 NULL
0 表示该列不为 NULL
```

如果表里没有允许为 `NULL` 的列，就不需要 NULL 值列表。

### 2. MySQL 怎么知道 varchar 实际占了多少字节

通过变长字段长度列表。

`varchar`、`TEXT`、`BLOB` 等变长字段的实际字节长度，会记录在额外信息区。读取真实数据区时，InnoDB 就知道该读多少字节。

### 3. varchar(n) 的 n 是字符还是字节

是字符数，不是字节数。

真实最大字节数取决于字符集：

```text
ascii    最多 1 字节/字符
utf8mb3  最多 3 字节/字符
utf8mb4  最多 4 字节/字符
```

### 4. varchar(n) 最大能取多少

没有一个脱离上下文的固定答案。

要看：

- 字符集；
- 是否允许 `NULL`；
- 表里是否还有其他列；
- 变长字段长度列表开销；
- NULL 位图开销；
- 行格式和存储引擎限制。

在“只有一个 nullable 的 `varchar` 字段、`ascii` 字符集、COMPACT 行格式”的极端例子里：

```text
n = 65535 - 2 - 1 = 65532
```

如果是 `utf8mb4`，还要再除以 4。

### 5. 为什么行最大 65535 字节，但页只有 16KB

因为大字段可以通过溢出页存储。

数据页里不一定完整存放全部大字段内容，可能只保留部分前缀或 20 字节指针，真实大内容放在溢出页中。

### 6. COMPACT 和 DYNAMIC 的行溢出有什么区别

简化理解：

- COMPACT：行内保留一部分长字段数据，再用 20 字节指针指向溢出页。
- DYNAMIC：长字段通常完整放到溢出页，行内只保留 20 字节指针。

所以 DYNAMIC 对大字段更友好，也是现代 MySQL 常见默认行格式。

## 小结

- InnoDB 的存储层级可以按“表空间 -> 段 -> 区 -> 页 -> 行”理解，行记录最终存放在数据页中。
- COMPACT 行格式由记录额外信息和真实数据组成，额外信息包括变长字段长度列表、NULL 值列表、记录头信息。
- `NULL` 值不会直接存放在真实数据区，而是通过 NULL 位图标记。
- `varchar(n)` 的 `n` 是字符数，不是字节数；真实最大字节数取决于字符集。
- 一行最大 65535 字节的限制包含变长字段长度列表和 NULL 位图等开销，而且是整行限制，不是单列限制。
- 行溢出时，大字段会放到溢出页；COMPACT 会保留部分前缀，DYNAMIC 通常只在行内保留 20 字节指针。
- 大行会降低页密度、影响 Buffer Pool 命中率、增加回表和范围扫描成本，表设计时应谨慎处理大字段和可空字段。
