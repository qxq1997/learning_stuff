# MySQL - 第 21 课：可重复读是否彻底解决幻读：快照读、当前读与特殊幻读场景

## 学习目标（本节结束后你能做到什么）

- 能准确说出幻读的定义，而不是只把幻读等同于 `insert`。
- 能区分快照读避免幻读的方式和当前读避免幻读的方式。
- 能解释为什么 InnoDB 的可重复读隔离级别是“很大程度避免幻读”，而不是“所有场景绝对没有幻读”。
- 能推导两类特殊幻读场景：事务内先快照读再更新、事务内快照读和当前读混用。
- 能在面试和线上设计中给出更稳的回答：何时需要先执行锁定读，何时只依赖 MVCC 不够。

## 1. 先把结论说清楚

InnoDB 默认隔离级别是可重复读（Repeatable Read，RR）。

很多人会把它总结成一句话：

**MySQL RR 解决了幻读。**

这句话不算完全错，但不够严谨。更准确的说法是：

**InnoDB 在 RR 下，对普通快照读主要通过 MVCC 避免幻读；对当前读主要通过 next-key lock 避免幻读。但如果事务内混用快照读、当前读，或者事务自己修改了原本不可见的新行，仍然可能观察到类似幻读的现象。**

所以面试时不要只背“RR 解决幻读”，而要补一句：

**RR 是按读类型分别处理幻读：快照读靠 Read View，当前读靠 next-key lock；特殊混用场景下仍可能出现结果集变化。**

这句话才经得住追问。

## 2. 什么是幻读

MySQL 文档对幻读的描述大意是：

在一个事务中，同一个查询在不同时间返回了不同的行集合。如果第二次查询返回了第一次没有返回的行，这条新出现的行就是幻影行。

可以把幻读理解成：

**同一个事务内，用同一个谓词条件查询，前后结果集的成员或数量发生变化。**

比如：

```sql
-- 事务 A 第一次查询
select * from t_stu where id > 2;
-- 返回 id = 3、4

-- 事务 B 插入 id = 5 并提交

-- 事务 A 第二次查询
select * from t_stu where id > 2;
-- 返回 id = 3、4、5
```

第二次多了一条 `id = 5`，这就是最典型的幻读。

但要注意，幻读不是只有“多出一行”：

- 其他事务插入满足条件的新行，会让结果集变多。
- 其他事务删除满足条件的旧行，会让结果集变少。
- 其他事务更新某行的索引列或过滤列，会让某行进入或离开结果集。

第 14 课已经从“谓词结果集稳定性”的角度讲过：幻读关注的是结果集，而不是单行值是否变化。

## 3. 先分清：快照读和当前读

判断 RR 下会不会出现幻读，第一步不是背锁名词，而是先问：

**这条 SQL 是快照读，还是当前读？**

### 3.1 快照读

普通 `select` 是快照读：

```sql
select * from t_stu where id > 2;
```

快照读不主动加行锁，而是通过 MVCC 读取对当前事务可见的历史版本。

在 RR 下，一个事务内第一次一致性读会创建 Read View，后续一致性读复用这个 Read View。因此，在同一个事务中，即使别的事务后来插入、提交了新记录，普通 `select` 也看不到这些新记录。

### 3.2 当前读

下面这些操作属于当前读：

```sql
select * from t_stu where id > 2 for update;
select * from t_stu where id > 2 for share;
update t_stu set score = 100 where id > 2;
delete from t_stu where id > 2;
insert into t_stu values (...);
```

当前读读取的是最新版本的数据，并且为了保证后续修改安全，会加锁。

在 RR 下，当前读通常通过 record lock、gap lock、next-key lock 来控制并发。对范围查询来说，关键就是 next-key lock：

**next-key lock = 记录锁 + 间隙锁。**

它既锁住已经存在的记录，也锁住记录之间的间隙，从而阻止其他事务在范围内插入新的幻影记录。

## 4. 快照读如何避免幻读

假设表中有这些记录：

```text
id | name | score
---+------+------
1  | 小林 | 50
2  | 小明 | 60
3  | 小红 | 70
4  | 小蓝 | 80
```

在 RR 下，事务 A 执行：

```sql
begin;

select * from t_stu where id > 2;
-- 返回 id = 3、4
```

第一次普通 `select` 创建 Read View。

这时事务 B 插入一条记录并提交：

```sql
begin;
insert into t_stu(id, name, score) values(5, '小飞', 100);
commit;
```

事务 A 再执行同一条普通查询：

```sql
select * from t_stu where id > 2;
-- 仍然返回 id = 3、4
```

为什么看不到 `id = 5`？

因为事务 A 的 Read View 是第一次查询时创建的。`id = 5` 是在 Read View 之后由事务 B 创建并提交的版本，对事务 A 的这个 Read View 不可见。

所以：

**RR 下，如果事务内一直使用普通 select 这种快照读，MVCC 可以让前后查询看到同一个一致性快照，从而避免幻读。**

## 5. 当前读如何避免幻读

当前读不能只读历史快照，因为它通常要服务后续修改。

比如：

```sql
select * from t_stu where id > 2 for update;
```

这条语句的语义不是“看看历史上有哪些数据”，而是：

**读取当前最新的满足条件的数据，并锁住它们以及相关间隙，避免别人并发修改或插入破坏我接下来的操作。**

仍然用 `id > 2` 的例子。

事务 A 执行：

```sql
begin;
select * from t_stu where id > 2 for update;
```

如果 `id` 是主键索引，已有记录是 `1、2、3、4`，那么这类范围当前读会沿主键索引扫描，并对扫描范围加 next-key lock。简化理解，它会保护 `id > 2` 这个查询范围，尤其会锁住最后一条记录之后到正无穷的间隙。

这时事务 B 想插入：

```sql
begin;
insert into t_stu(id, name, score) values(5, '小飞', 100);
```

事务 B 会被阻塞。

原因是插入 `id = 5` 的位置落在事务 A 已经锁住的范围内。事务 B 插入前会尝试获取插入意向锁，但它和事务 A 持有的 gap / next-key lock 冲突，只能等待事务 A 提交。

所以：

**RR 下，如果事务一开始就用 `select ... for update` 这种当前读锁住谓词范围，InnoDB 可以通过 next-key lock 阻止其他事务插入幻影行。**

## 6. 反过来想：如果当前读不加锁会怎样

为了理解 next-key lock 的必要性，可以做一个反事实实验。

假设 `select ... for update` 是当前读，但它不加锁。

事务 A：

```sql
begin;
select name from t_stu where id > 2 for update;
-- 返回：小红、小蓝
```

事务 B：

```sql
begin;
insert into t_stu values(5, '小飞', 100);
commit;
```

事务 A 再查一次：

```sql
select name from t_stu where id > 2 for update;
-- 返回：小红、小蓝、小飞
```

因为当前读读取最新已提交版本，如果第一次当前读没有加锁，事务 B 的插入就能成功，事务 A 第二次当前读就会看到新行。

这就是幻读。

所以 next-key lock 的存在不是多余的，它解决的就是：

**当前读不能靠旧快照稳定结果集，只能靠锁稳定谓词范围。**

## 7. RR 是否完全解决幻读

现在可以回答核心问题了：

**InnoDB RR 很大程度避免了幻读，但不能简单说完全解决了所有形式的幻读。**

它的基本盘是：

- 快照读一直用快照读：Read View 稳定，通常不会看到新插入的幻影行。
- 当前读一开始就锁住范围：next-key lock 阻止其他事务插入、删除、更新导致结果集变化。

但有两类特殊场景，会让结果集在同一个事务内发生变化。

## 8. 特殊场景一：先快照读看不到，后续自己 update 后又看到了

这是最容易把人绕住的场景。

初始表中没有 `id = 5`。

事务 A：

```sql
begin;

select * from t_stu where id = 5;
-- 空结果
```

第一次普通 `select` 创建了 Read View。此时 `id = 5` 不存在。

事务 B：

```sql
begin;
insert into t_stu(id, name, score) values(5, '小美', 18);
commit;
```

事务 B 提交后，事务 A 继续：

```sql
update t_stu set name = '小林coding' where id = 5;
```

问题来了：事务 A 明明通过快照读看不到 `id = 5`，为什么还能更新它？

因为 `update` 是当前读。

当前读会读取最新已提交版本。事务 B 已经提交了 `id = 5`，所以事务 A 的 `update` 可以找到这条记录，然后把它更新掉。

更新之后，这条记录的最新版本的 `trx_id` 会变成事务 A 自己的事务 id。

接着事务 A 再普通查询：

```sql
select * from t_stu where id = 5;
-- 可以查到 id = 5
```

为什么此时普通快照读又能看到它？

因为 MVCC 的可见性规则里有一个非常重要的特例：

**当前事务自己修改过的版本，对当前事务可见。**

也就是说，这条记录原本是事务 B 插入的，事务 A 的旧 Read View 看不到；但事务 A 后来对它执行了 `update`，产生了一个由事务 A 自己创建的新版本。自己创建的版本当然应该对自己可见。

于是事务 A 前后两次 `select * from t_stu where id = 5` 的结果从空变成有一行。

这就是一个典型的“RR 下仍可能出现类似幻读”的特殊场景。

本质不是 Read View 失效了，而是：

**事务 A 混用了快照读和当前写，且当前写把原本不可见的行变成了自己可见的版本。**

## 9. 特殊场景二：先快照读，后当前读

第二类场景是快照读和当前读混用。

事务 A：

```sql
begin;

select * from t_stu where id > 100;
-- 返回 3 条
```

事务 B：

```sql
begin;
insert into t_stu(id, name, score) values(200, '小飞', 100);
commit;
```

事务 A 后续执行：

```sql
select * from t_stu where id > 100 for update;
-- 返回 4 条
```

从业务谓词 `id > 100` 的角度看，事务 A 前后看到的结果集变了。

为什么？

因为第一次是快照读，看的是 Read View 中可见的历史版本；第二次是当前读，看的是最新已提交版本，并加锁。

这两个读的语义本来就不一样。

严格说，普通 `select` 和 `select ... for update` 不是完全相同的 SQL；但面试里经常会用它追问“RR 是否绝对没有幻读”。你要回答清楚：

**如果事务内先用快照读建立旧视图，再改用当前读读取最新数据，就可能看到前面快照读看不到的新行。**

这不是“MVCC 坏了”，而是你切换了读取语义。

## 10. 为什么一开始执行当前读就能避免这类问题

如果业务要求在一个事务中保证某个范围不会被其他事务插入新行，就不要先普通查询、后面才锁。

应该在事务开始后尽早执行锁定读：

```sql
begin;

select * from t_stu where id > 100 for update;

-- 后续基于这个范围做业务判断和修改

commit;
```

这样 InnoDB 会在当前读时锁住相关记录和间隙。其他事务如果想插入 `id > 100` 范围内的新记录，就会被阻塞。

但是这里有三个工程细节必须强调：

1. 锁定读要尽量早执行。等别人已经插入并提交后再加锁，就已经晚了。
2. 查询条件要有合适索引。没有索引时，InnoDB 可能全表扫描并锁住大量范围，业务影响很大。
3. 锁只保护当前执行计划扫描到的索引范围。不同索引、不同谓词、不同执行计划，锁范围都可能不同。

换句话说：

**不是写了 `for update` 就万事大吉，还要看它什么时候执行、走哪个索引、锁住哪个范围。**

## 11. 快照读和当前读混用时，怎么判断读到什么

遇到这类题，可以按下面四步推导。

第一步，看 SQL 类型：

- 普通 `select`：快照读。
- `select ... for update` / `for share`：当前读。
- `update` / `delete`：当前读。

第二步，看 Read View：

- RC：每条普通 `select` 都创建新的 Read View。
- RR：事务内第一次普通 `select` 创建 Read View，后续普通 `select` 复用。

第三步，看当前读是否加锁：

- 范围当前读通常加 next-key lock。
- 等值命中唯一索引可能退化为记录锁。
- 等值未命中唯一索引可能退化为间隙锁。
- 非唯一索引要按 `(二级索引值, 主键值)` 判断范围。

第四步，看事务自己是否修改过该版本：

- 自己插入、更新过的版本，对自己可见。
- 这可能让原本快照读不可见的行，经过本事务 `update` 后变得可见。

用这四步，基本能推导出大多数 RR 幻读追问题。

## 12. 一个更稳的面试回答

如果面试官问：

**MySQL 可重复读隔离级别彻底解决幻读了吗？**

可以这样回答：

InnoDB 的 RR 默认隔离级别很大程度避免了幻读，但说“彻底解决所有幻读场景”不严谨。

对普通 `select` 这种快照读，RR 下事务第一次一致性读会创建 Read View，后续一致性读复用同一个 Read View，因此其他事务后续插入并提交的新行对当前事务不可见，所以快照读场景可以避免幻读。

对 `select ... for update`、`update`、`delete` 这种当前读，InnoDB 读取最新版本，并通过 next-key lock 锁住记录和间隙，阻止其他事务在锁范围内插入、删除或修改导致结果集变化，所以当前读如果一开始就锁住范围，也能避免幻读。

但如果事务内先普通快照读，后续又执行当前读，当前读会读最新数据，就可能看到快照读看不到的新行；或者事务自己更新了其他事务插入的新行，这条记录变成当前事务自己的版本，后续普通 select 也能看到它。因此 RR 不是一句“完全解决幻读”就能概括的，要结合快照读、当前读和 SQL 执行顺序分析。

## 13. 生产实践：什么时候必须主动加锁

如果业务只是展示列表、报表查询，通常普通快照读就够了。RR 下同一事务内看到一致快照，不会被后续提交的新行打断。

但如果业务是“先查有没有，再决定插入/更新/扣减/占用资源”，就不能只靠普通 `select`。

典型场景：

- 下单前检查库存区间。
- 创建订单前检查某个业务号是否存在。
- 抢券、抢座、抢任务。
- 按范围选择待处理任务并标记处理中。
- 检查某个用户当天是否已有记录，没有就插入。

这些场景要考虑：

- 用唯一索引兜住业务不变量，比如订单号、幂等号、用户日期唯一键。
- 必要时用 `select ... for update` 锁住即将判断和修改的范围。
- 尽量让锁定读走精确索引，避免全表扫描。
- 把事务做短，不要拿着 next-key lock 做远程调用或复杂计算。
- 对死锁和锁等待做好重试，尤其是高并发写场景。

一句话：

**MVCC 适合提高读并发，业务互斥和范围占用仍然要靠锁、唯一约束和正确的事务边界。**

## 14. 小结（3-5 条关键点）

- 幻读的本质是同一事务内，同一谓词条件前后查到的结果集成员或数量发生变化，不只是 `insert` 多一行。
- RR 下普通快照读通过复用 Read View 避免幻读；当前读通过 next-key lock 锁住记录和间隙避免幻读。
- 如果事务内一直使用快照读，通常能看到稳定一致的历史快照；如果一开始就使用锁定当前读，也能阻止别人插入幻影记录。
- RR 下仍可能出现特殊幻读：先快照读看不到，后续自己 `update` 了其他事务插入的新行；或者先快照读，后当前读，读语义切换导致结果集变化。
- 生产中如果要保证某个判断范围在事务内不被别人插入新行，应该尽早执行锁定读，并确保查询走合适索引。
