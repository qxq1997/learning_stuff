# Nginx - 第 2 课：为什么 Nginx 这么快：master-worker、事件驱动、epoll 与高并发

## 学习目标（本节结束后你能做到什么）

- 说清楚 Nginx 为什么能用少量进程扛住大量并发连接。
- 理解 master-worker 模型各自负责什么，为什么 Nginx 更偏好多进程而不是多线程。
- 理解事件驱动、非阻塞 I/O、I/O 多路复用和 epoll 的关系。
- 能回答 select、poll、epoll 的关键差异，以及惊群、accept_mutex 这些面试追问。
- 建立“连接很多不等于 CPU 很忙”的高并发理解方式。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先看传统模型为什么会吃力

假设有 1 万个用户同时连到你的服务，其中绝大多数连接在大部分时间其实都处于“空闲但未断开”的状态，比如：

- 正在等待服务器返回数据
- 打开了页面但暂时没继续点击
- 保持长连接等待下一次请求

如果你用“一个连接一个线程”或者“一个请求一个进程”的思路，那么 1 万个连接就意味着可能要维护 1 万个执行单元。问题是：

- 每个线程有内存开销
- 上下文切换有 CPU 开销
- 大量线程其实多数时间都在等 I/O，不是在做有效计算

这时系统的瓶颈不是“业务逻辑太难”，而是你为了等待 I/O 付出了太多调度成本。

### 2. Nginx 的核心思路：不要给每个连接发一个线程

Nginx 的设计哲学可以概括成一句话：

**用少量 worker 进程，配合事件驱动和 I/O 多路复用，管理海量连接。**

它不认为“连接多”就必须“线程多”。因为很多连接只是存在，并没有持续产生可读可写事件。

更形象一点，你可以把 worker 想成餐厅里的一个服务员：

- 阻塞模型：服务员给 A 桌下完单后，就站在厨房门口傻等 A 桌的菜出锅，期间完全没法管 B、C、D 桌。
- 事件驱动模型：服务员把很多桌的需求都登记好，谁的菜好了，厨房就通知他去端。他只处理“已经就绪”的事情。

Nginx 采用的就是第二种思路。

### 3. master-worker 模型

Nginx 启动后，通常有两类进程：

- master 进程
- worker 进程

#### 3.1 master 进程负责什么

master 像一个总调度：

- 读取和校验配置
- 启动、关闭、拉起 worker
- 接收外部信号，比如 reload、stop、quit
- 管理平滑重载和平滑升级

它几乎不直接处理用户请求。

#### 3.2 worker 进程负责什么

真正处理网络连接和 HTTP 请求的是 worker。

每个 worker 都会：

- 监听事件
- accept 新连接
- 读取请求
- 解析 HTTP 报文
- 匹配配置
- 访问磁盘或代理后端
- 发送响应

多个 worker 进程通常是平等的、相互独立的。

#### 3.3 为什么偏好多进程而不是多线程

原因主要有三个：

- 进程间地址空间隔离，一个 worker 崩了不容易把其他 worker 一起拖死，稳定性更好。
- 事件驱动模型下，不需要靠大量线程来换取并发，多进程已经够用。
- 编程模型更简单，很多共享状态问题不需要复杂加锁。

所以 Nginx 的高并发不是“疯狂开线程”，而是“少量进程管理大量事件”。

### 4. 事件驱动和 epoll 到底是什么关系

这里最容易混淆几个词：

- 非阻塞 I/O
- I/O 多路复用
- 事件驱动
- epoll

它们的关系可以这样理解：

#### 4.1 非阻塞 I/O

当你去读一个连接，如果当前没有数据，阻塞 I/O 会一直卡住；非阻塞 I/O 会立刻返回，告诉你“现在没数据”。

但如果你不断轮询 1 万个连接去问“有数据吗”，这又很浪费 CPU。

#### 4.2 I/O 多路复用

于是内核提供了更好的办法：你把很多文件描述符都交给内核，告诉它“等这些 fd 里谁就绪了再通知我”。

这就叫 I/O 多路复用。

#### 4.3 epoll

在 Linux 上，Nginx 主要依赖 epoll 来做 I/O 多路复用。

worker 会把自己关心的连接 fd 注册到 epoll 中，然后调用 `epoll_wait()`。当某些连接：

- 可读
- 可写
- 发生错误

内核会把这些“就绪事件”返回给 worker。worker 只处理这些真正有事发生的连接。

#### 4.4 事件驱动

所以所谓事件驱动，就是：

**程序不是傻等某一个连接，而是等“事件”发生，再根据事件类型做相应处理。**

这让一个 worker 在一个线程中也能同时管理很多连接。

### 5. select、poll、epoll 的差别

这是面试高频题。

#### select 的问题

- 需要把 fd 集合从用户态拷贝到内核态
- 默认 fd 数量有限制
- 返回后你还要自己遍历一遍，找哪些 fd 就绪了

#### poll 的改进和限制

- 没有 select 那么死的 fd 上限
- 但本质上仍然需要遍历整个集合

#### epoll 的优势

- fd 注册一次后就交给内核维护
- 返回的是“已经就绪”的列表，不需要每次全量扫描
- 更适合高并发、海量连接场景

严格来说，epoll 也不是魔法，它只是把“找谁就绪”这件事从应用层低效轮询，变成了内核更高效地维护事件就绪队列。

### 6. 一个请求在 worker 里如何流转

把一条请求生命周期串起来，你会更容易理解 worker 的角色：

1. 客户端建立 TCP 连接
2. 某个 worker 通过事件机制感知到新连接
3. worker `accept` 连接
4. 客户端发送请求报文，worker 收到可读事件
5. worker 读取并解析 HTTP 请求
6. worker 根据 `server`、`location` 等配置决定怎么处理
7. 如果是静态文件，就读文件并返回
8. 如果是代理请求，就转发到后端再取回响应
9. 记录日志，关闭连接或保持 keepalive

你会发现，这里真正消耗资源的是“就绪后的处理”，而不是每个连接都固定占着一个线程。

### 7. 惊群问题和 accept_mutex

多个 worker 进程都可能监听同一个端口。当一个新连接到来时，如果所有 worker 都同时被唤醒去抢这个连接，但最后只有一个抢到，其他都白白醒一次，这就叫惊群。

这类无效唤醒会造成额外开销。

Nginx 的经典做法之一是 `accept_mutex`，本质上是让同一时刻只有一个 worker 更积极地去 `accept` 新连接，从而减少无效争抢。

更现代的内核和网络能力里，还会结合：

- `EPOLLEXCLUSIVE`
- `SO_REUSEPORT`

从更底层缓解同类问题。

你不用把这些细节背成论文，但要知道：高并发服务器不仅要会处理“有事的连接”，还要尽量减少“没必要的唤醒”。

### 8. worker 数量应该怎么设

经典建议是：`worker_processes` 设为 CPU 核心数，或者直接 `auto`。

背后的直觉是：

- 每个 worker 是一个事件循环
- 过少会让 CPU 利用不足
- 过多会导致上下文切换和竞争增加

但这不是死规则。真实场景还要看：

- 是静态文件为主，还是代理为主
- 磁盘 I/O 是否明显
- 是否开启了线程池处理某些阻塞任务
- 宿主机资源是否还被别的进程共享

所以面试里比较好的回答不是“永远等于核心数”，而是“通常从核心数起步，再结合压测和场景微调”。

### 9. 一个重要边界：Nginx 也不是完全不会阻塞

很多人会把 Nginx 理解成“完全异步、永不阻塞”，这不准确。

如果涉及：

- 磁盘文件读取
- 某些第三方模块的阻塞逻辑
- DNS 解析等外部依赖

worker 仍然可能受到影响。

所以后面你会看到 Nginx 还会引入：

- `sendfile`
- AIO
- thread pool

去减少这类阻塞对事件循环的伤害。

## 小结

- Nginx 快，不是因为它“什么都没有做”，而是因为它用更高效的方式管理等待 I/O 的成本。
- master 负责管理，worker 负责真正处理请求。
- worker 依赖 epoll 这类 I/O 多路复用机制，只处理已经就绪的连接事件。
- epoll 比 select/poll 更适合海量连接场景，因为它避免了每次全量扫描 fd 集合。
- 惊群、`accept_mutex`、worker 数量等问题，体现的是 Nginx 在高并发下对调度细节的优化。
