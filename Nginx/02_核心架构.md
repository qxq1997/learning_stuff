# Nginx - 第 2 课：核心架构

## 学习目标（本节结束后你能做到什么）

- 说清 Nginx 为什么快：事件驱动 + 异步非阻塞 + 多进程模型
- 理解 master-worker 进程模型的设计原因和工作方式
- 解释 epoll 的原理以及它和 select/poll 的本质区别
- 描述一个请求在 Nginx 中的完整生命周期
- 知道 Nginx 在哪些场景下不是完全异步的，以及如何用 thread pool 解决

## 内容讲解

### 1. 传统服务器的瓶颈：一请求一线程

在理解 Nginx 为什么快之前，先看看传统服务器（如 Apache prefork 模式）怎么工作的：

```
传统模型（Apache prefork）：

客户端1 --> [线程/进程1] --> 处理请求（可能阻塞在 I/O）
客户端2 --> [线程/进程2] --> 处理请求（可能阻塞在 I/O）
客户端3 --> [线程/进程3] --> 处理请求（可能阻塞在 I/O）
...
客户端N --> 没线程了！排队或拒绝
```

问题在哪：
- 每来一个请求就要分配一个线程/进程，内存开销大（每个线程约 1-8 MB 栈空间）
- 大部分时间线程在等 I/O（磁盘读写、网络传输），CPU 没干活
- 线程数有上限，几千个连接就扛不住了
- 线程切换有开销（上下文切换 context switch）

所以传统模型的核心问题是：**用大量线程去"等"少量 I/O，资源浪费严重。**

### 2. Nginx 为什么快：三驾马车

Nginx 用了完全不同的思路来解决高并发问题：

```
+---------------------------+
|     事件驱动架构           |
|  (Event-Driven)           |
+---------------------------+
|     异步非阻塞 I/O         |
|  (Non-blocking I/O)       |
+---------------------------+
|     多进程模型             |
|  (Master-Worker)          |
+---------------------------+
```

**事件驱动：** 不是给每个连接分配一个线程，而是用一个事件循环（event loop）监听所有连接上的事件（有数据来了、可以写了、连接断了），哪个就绪就处理哪个。

**异步非阻塞：** 发起 I/O 操作后不等结果，继续处理其他事件。等 I/O 完成了内核会通知，再回来处理。

**多进程：** 一个 master 进程管理多个 worker 进程，每个 worker 独立运行事件循环，充分利用多核 CPU。

### 3. Master-Worker 进程模型

```
                    +------------------+
                    |   Master 进程     |
                    |  (管理者，不处理   |
                    |   具体请求)       |
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
     +--------+----+  +-----+-------+  +---+---------+
     | Worker 进程1 |  | Worker 进程2 |  | Worker 进程3 |
     | (事件循环)   |  | (事件循环)    |  | (事件循环)   |
     | 处理请求     |  | 处理请求     |  | 处理请求     |
     +-------------+  +-------------+  +-------------+
```

**Master 进程的职责：**
- 读取和验证配置文件
- 创建、绑定、关闭 socket
- 启动、管理、监控 worker 进程
- 处理信号（reload、stop、quit）
- 不处理任何网络请求

**Worker 进程的职责：**
- 接收客户端连接
- 处理请求（解析 HTTP、读文件、代理转发）
- 返回响应
- 每个 worker 运行一个独立的事件循环

**Worker 数量的设置原则：**

```nginx
worker_processes auto;    # 推荐设为 auto，自动等于 CPU 核心数
```

为什么是 CPU 核心数？因为每个 worker 是事件驱动的，一个 worker 就能处理数万连接。多个 worker 只是为了利用多核，而不是为了"一连接一进程"。设多了反而增加进程切换开销。

### 4. 为什么用多进程不用多线程

这是面试常问的问题。Nginx 选择多进程而非多线程，核心原因是：

**进程隔离性：**
- 一个 worker 进程崩溃了，不会影响其他 worker，master 会自动重启新的 worker
- 线程共享地址空间，一个线程出问题可能整个进程都挂

**不需要加锁：**
- 多线程共享内存，访问共享数据需要加锁，锁竞争会降低性能
- 多进程各自独立，不需要锁，代码更简单，性能更可预测

**稳定性优先：**
- Nginx 作为反向代理是基础设施层面的组件，稳定性比极致性能更重要
- 进程隔离带来的稳定性优势大于线程模型带来的微小性能优势

```
多线程模型的风险：
  线程1 崩溃 --> 整个进程可能崩溃 --> 所有连接断开

多进程模型的优势：
  Worker1 崩溃 --> Master 检测到 --> 启动新 Worker --> 其他 Worker 不受影响
```

### 5. 事件驱动 + epoll：餐厅服务员比喻

理解事件驱动模型，用餐厅比喻最直观：

**传统模型（一请求一线程）= 每桌一个服务员：**
- 客人点菜后，服务员站在厨房门口等这桌的菜做好
- 菜没好之前，这个服务员什么都不干，就在那等
- 100 桌客人需要 100 个服务员

**Nginx 模型（事件驱动）= 一个服务员服务所有桌：**
- 服务员不会站在厨房门口等
- 服务员在大厅巡视，哪桌需要点菜就去哪桌
- 厨房菜做好了会"叫号"（事件通知），服务员再去端菜
- 一个服务员就能服务几十桌

epoll 就是那个"叫号系统"。它是 Linux 内核提供的 I/O 事件通知机制：

```
Worker 进程的事件循环：

while (true) {
    // 1. 调用 epoll_wait，阻塞等待事件（但不是等某一个连接）
    events = epoll_wait(epfd, ...);

    // 2. 遍历所有就绪的事件
    for (event in events) {
        if (event 是新连接) {
            accept 新连接，加入 epoll 监听列表;
        }
        if (event 是数据可读) {
            读取数据，解析 HTTP 请求;
        }
        if (event 是可写) {
            发送响应数据;
        }
    }
}
```

关键：worker 不是在等某一个连接的 I/O，而是在等"任意连接的任意事件"。只要有任何一个连接就绪，worker 就能立刻处理，不浪费时间。

### 6. 一个请求的完整生命周期（7 步）

```
                        Nginx Worker 进程
                        +-----------------------------------------+
客户端 ---TCP连接--->    | 1. accept() 接受连接                     |
                        | 2. 读取请求数据（非阻塞）                  |
                        | 3. 解析 HTTP 请求（请求行、头部、body）     |
                        | 4. 匹配 location 规则                    |
                        | 5. 执行对应的处理：                       |
                        |    - 静态文件：读磁盘返回                  |
                        |    - 反向代理：转发到 upstream             |
                        |    - rewrite/redirect 等                 |
                        | 6. 构建 HTTP 响应并发送                   |
                        | 7. 记录 access log，关闭或保持连接         |
                        +-----------------------------------------+
```

**每一步的要点：**

1. **accept 连接：** 多个 worker 通过 accept_mutex（或 EPOLLEXCLUSIVE）竞争新连接，避免惊群效应
2. **读取数据：** 非阻塞读取，数据没到就先处理其他连接
3. **解析请求：** Nginx 用状态机解析 HTTP 报文，高效且健壮
4. **匹配 location：** 按照精确匹配 > 前缀匹配 > 正则匹配的优先级（第 3 课详讲）
5. **执行处理：** 这一步可能涉及磁盘 I/O（读静态文件）或网络 I/O（代理转发）
6. **发送响应：** 利用 sendfile 零拷贝技术（第 5 课详讲）
7. **日志和连接管理：** 写 access log，根据 keep-alive 决定是否保持连接

### 7. epoll vs select/poll（面试高频）

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| 最大连接数 | **1024**（FD_SETSIZE 限制） | 无硬限制（但性能下降） | **无上限**（受系统内存限制） |
| 实现方式 | 每次调用把所有 fd 集合**拷贝**到内核 | 同 select，用链表代替数组 | fd 注册到内核，**不需要每次拷贝** |
| 返回方式 | 返回后需要**遍历所有 fd** 找就绪的 | 同 select | **只返回就绪的 fd 列表** |
| 时间复杂度 | O(n)，n 是总 fd 数 | O(n) | **O(1)**（就绪列表直接返回） |
| 触发模式 | 只有水平触发（LT） | 只有水平触发（LT） | 支持水平触发和**边缘触发（ET）** |

**为什么 select 慢：**

```
select 的工作方式：

用户态                    内核态
  |                         |
  |-- 拷贝 1000 个 fd -->    |
  |                    检查每个 fd 是否就绪
  |                    （遍历 1000 个）
  |<- 返回，标记就绪的 fd ---|
  |                         |
  遍历 1000 个 fd           |
  找到就绪的那几个           |
```

每次调用 select 都要拷贝所有 fd，内核要遍历所有 fd，返回后用户态还要遍历一遍。当 fd 数量很大时，绝大多数 fd 其实没有事件，白白浪费 CPU。

**epoll 的改进：**

```
epoll 的工作方式：

1. epoll_create：创建 epoll 实例
2. epoll_ctl：注册 fd（只需注册一次，不需每次拷贝）
3. epoll_wait：阻塞等待

用户态                    内核态
  |                         |
  |  （fd 已注册，不需要     |
  |    重复拷贝）            |
  |-- epoll_wait ---------->|
  |                    某 fd 就绪时
  |                    加入就绪列表
  |<-- 只返回就绪列表 ------|
  |                         |
  直接处理就绪的 fd         |
  不需要遍历所有 fd         |
```

**水平触发（LT）vs 边缘触发（ET）：**
- LT：只要 fd 上有数据可读，每次 epoll_wait 都会返回它（安全但可能有冗余通知）
- ET：只在 fd 状态变化时通知一次（高效但必须一次读完，否则可能丢事件）

Nginx 默认使用 ET 模式，性能更高但编程复杂度也更高。

### 8. Nginx 不是完全异步：磁盘 I/O 的问题

很多人以为 Nginx 一切都是异步非阻塞的，其实不是。

**网络 I/O：** 完全异步非阻塞，epoll 支持得很好。

**磁盘 I/O：** Linux 上 epoll 对磁盘文件的支持不好。读磁盘文件时（比如读取静态文件返回给客户端），如果文件不在 page cache 中，read() 调用会阻塞整个 worker 进程，期间这个 worker 无法处理其他连接。

```
问题场景：

Worker 在处理连接 A 的请求，需要读一个大文件
  |
  +--> read(大文件) --> 磁盘 I/O 阻塞 --> Worker 整个卡住
  |                                        连接 B、C、D 全部等待
  +--> 直到磁盘读完才恢复
```

**解决方案：Thread Pool（线程池）**

从 Nginx 1.7.11 开始引入了线程池机制。把可能阻塞的磁盘 I/O 操作丢给线程池中的线程去执行，worker 的事件循环不受影响。

```nginx
# 在配置中启用线程池
aio threads;
# 或自定义线程池
thread_pool my_pool threads=32 max_queue=65536;

location /bigfiles/ {
    aio threads=my_pool;
    directio 512;    # 大文件直接 I/O，跳过 page cache
}
```

```
Thread Pool 解决方案：

Worker 事件循环                  线程池
  |                              |
  +--> 需要读大文件               |
  +--> 提交任务到线程池 --------> [线程] 执行磁盘读取
  +--> 继续处理其他连接           |
  |    （不阻塞）                 |
  |                              |
  |<-- 读取完成，回调通知 --------|
  +--> 处理连接 A 的响应          |
```

## 小结（3-5 条关键点）

- Nginx 快的核心原因：事件驱动 + 异步非阻塞 + 多进程，用少量 worker 进程处理数万并发连接，不像传统服务器那样一请求一线程。
- Master-Worker 模型：master 管理配置和 worker 生命周期，worker 处理实际请求。worker 数等于 CPU 核心数。选多进程而非多线程是为了进程隔离和避免加锁。
- epoll 是 Nginx 高性能的关键：无 fd 数量限制、不需要每次拷贝所有 fd、只返回就绪列表、O(1) 时间复杂度。select 的 1024 限制和 O(n) 遍历是面试常考的对比点。
- 一个请求的生命周期：accept -> 读数据 -> 解析 HTTP -> 匹配 location -> 处理 -> 响应 -> 记日志。
- Nginx 磁盘 I/O 可能阻塞 worker，解决方案是 thread pool。所以 Nginx 不是"完全异步"的。
