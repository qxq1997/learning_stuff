# Nginx - 第 3 课：配置体系

## 学习目标（本节结束后你能做到什么）

- 理解 Nginx 配置文件的层级结构和指令继承机制
- 掌握虚拟主机的工作原理和 default_server 的作用
- 熟练掌握 location 匹配规则的优先级（面试必考）
- 理解 rewrite 规则中 permanent/last/break 的区别
- 区分常用变量如 $uri 和 $request_uri

## 内容讲解

### 1. 配置文件层级结构

Nginx 配置文件是一个树形结构，从外到内有严格的层级关系：

```
main（全局块）
├── events { }                    # 事件模型配置
├── http {                        # HTTP 服务配置
│   ├── server {                  # 虚拟主机（站点）
│   │   ├── location / { }       # 路由匹配规则
│   │   ├── location /api/ { }
│   │   └── location ~ \.php$ { }
│   ├── server {                  # 另一个虚拟主机
│   │   └── location / { }
│   └── upstream backend { }      # 后端服务器组
```

**每个层级能写什么：**

| 层级 | 关键指令 | 作用 |
|------|----------|------|
| main | worker_processes, error_log, pid | 全局进程参数 |
| events | worker_connections, use epoll | 连接和事件模型 |
| http | include, log_format, gzip, keepalive_timeout | HTTP 公共配置 |
| server | listen, server_name, root, ssl | 虚拟主机配置 |
| location | proxy_pass, root, alias, rewrite | 具体路径的处理方式 |

**指令的继承和覆盖规则：**

很多指令可以在多个层级中出现。规则是：**内层覆盖外层，就近原则。**

```nginx
http {
    gzip on;                      # http 层设置 gzip
    root /var/www/default;        # http 层默认 root

    server {
        server_name a.com;
        root /var/www/a;          # server 层覆盖 http 层的 root
                                  # gzip 继承自 http 层，仍然 on

        location /api/ {
            root /var/www/api;    # location 层覆盖 server 层的 root
                                  # gzip 继续继承，仍然 on
        }

        location /static/ {
            gzip off;             # 这个 location 关闭 gzip
                                  # root 继承 server 层的 /var/www/a
        }
    }
}
```

**重要注意：** 不是所有指令都支持继承。proxy_pass 只能写在 location 里，不能写在 server 层让所有 location 继承。具体哪些指令支持哪些层级，要查官方文档。

### 2. 虚拟主机：一台机器跑多个网站

虚拟主机让一台 Nginx 同时服务多个域名的网站。核心机制是通过 HTTP 请求头中的 `Host` 字段来区分。

```nginx
# 站点 A
server {
    listen 80;
    server_name www.a.com;
    root /var/www/a;
}

# 站点 B
server {
    listen 80;
    server_name www.b.com;
    root /var/www/b;
}

# 站点 C（API 服务）
server {
    listen 80;
    server_name api.c.com;
    location / {
        proxy_pass http://127.0.0.1:8080;
    }
}
```

**匹配流程：**

```
客户端请求：GET / HTTP/1.1
            Host: www.b.com

Nginx 处理流程：
1. 先看 listen 的端口 --> 80 端口有 3 个 server 块
2. 再看 Host 头 --> 匹配 server_name
3. www.b.com 匹配到站点 B --> 用站点 B 的配置处理
```

**server_name 的匹配优先级：**

| 优先级 | 类型 | 示例 |
|--------|------|------|
| 1（最高） | 精确匹配 | `server_name www.example.com;` |
| 2 | 前缀通配符 | `server_name *.example.com;` |
| 3 | 后缀通配符 | `server_name www.example.*;` |
| 4（最低） | 正则匹配 | `server_name ~^www\.(.+)\.com$;` |

**default_server 的作用：**

当请求的 Host 不匹配任何 server_name 时，Nginx 需要一个兜底。如果没有显式指定 default_server，Nginx 会用配置中第一个 server 块作为默认。

```nginx
# 显式指定默认虚拟主机
server {
    listen 80 default_server;     # 注意：default_server 写在 listen 后面
    server_name _;                # _ 是一个无效域名，表示匹配不到其他任何 server
    return 444;                   # 返回 444（Nginx 特有，直接关闭连接）
}
```

这种配置常用于安全防护：不认识的域名请求直接拒绝，避免暴露默认站点。

### 3. Location 匹配规则（必考）

location 是 Nginx 配置中最核心也最容易出错的部分。面试几乎必考。

**四种匹配修饰符：**

| 修饰符 | 含义 | 示例 | 优先级 |
|--------|------|------|--------|
| `=` | 精确匹配 | `location = /login { }` | 最高 |
| `^~` | 前缀匹配，匹配后**停止正则搜索** | `location ^~ /static/ { }` | 第二 |
| `~` | 正则匹配（区分大小写） | `location ~ \.php$ { }` | 第三 |
| `~*` | 正则匹配（不区分大小写） | `location ~* \.(jpg|png)$ { }` | 第三 |
| 无修饰符 | 普通前缀匹配 | `location /api/ { }` | 第四 |

**完整匹配流程（5 步）：**

```
请求 URI 进来后，Nginx 的匹配过程：

第 1 步：检查所有 "=" 精确匹配
         如果找到完全一致的 --> 立即使用，停止搜索

第 2 步：收集所有"前缀匹配"（包括 ^~ 和普通前缀）
         找到最长匹配的那个前缀

第 3 步：如果最长匹配的前缀用了 "^~" 修饰
         --> 立即使用，停止搜索（不再检查正则）

第 4 步：按配置文件中的顺序，逐个检查正则 location（~ 和 ~*）
         如果找到第一个匹配的正则 --> 使用它（正则是按顺序，先匹配先赢）

第 5 步：如果没有正则匹配成功
         --> 使用第 2 步找到的最长前缀匹配
```

用一句话总结优先级：**精确 > 前缀停止正则 > 正则（按顺序）> 普通前缀最长匹配**

流程图：

```
请求 URI
  |
  v
有精确匹配(=)? --是--> 使用该 location，结束
  |
  否
  v
找到最长前缀匹配
  |
  v
最长前缀是 ^~? --是--> 使用该 location，结束
  |
  否
  v
按顺序检查正则(~/~*)
  |
  v
有正则匹配? --是--> 使用第一个匹配的正则，结束
  |
  否
  v
使用最长前缀匹配，结束
```

### 4. Location 匹配练习题

**配置：**

```nginx
server {
    location = / {
        return 200 "精确匹配 /";              # A
    }
    location / {
        return 200 "前缀匹配 /";              # B
    }
    location /api/ {
        return 200 "前缀匹配 /api/";          # C
    }
    location ^~ /api/internal/ {
        return 200 "前缀停止正则 /api/internal/";  # D
    }
    location ~ ^/api/.*\.json$ {
        return 200 "正则匹配 .json";           # E
    }
    location ~* \.(jpg|png|gif)$ {
        return 200 "正则匹配图片";              # F
    }
}
```

**题目和答案：**

| 请求 URI | 匹配结果 | 理由 |
|----------|----------|------|
| `/` | A | 精确匹配 `= /`，最高优先级 |
| `/index.html` | B | 只有 `/` 前缀匹配，没有正则匹配 |
| `/api/users` | C | 前缀匹配 `/api/`，没有匹配的正则 |
| `/api/data.json` | **E** | 前缀匹配 `/api/`，但正则 `^/api/.*\.json$` 也匹配，正则优先于普通前缀 |
| `/api/internal/config` | **D** | 前缀匹配 `/api/internal/` 且有 `^~`，停止检查正则 |
| `/api/internal/data.json` | **D** | 虽然正则 E 也能匹配，但 `^~` 阻止了正则检查 |
| `/images/logo.png` | **F** | 前缀匹配 `/`，正则匹配图片后缀，正则优先 |
| `/api/avatar.jpg` | **F** | 前缀匹配 `/api/`，但正则 F（图片）先于 E（.json）被检查到并匹配 |

**关键结论：**
- `^~` 的作用是"前缀匹配成功后，阻止后续的正则检查"
- 正则匹配是**按配置文件中的顺序**，不是按最长匹配
- 普通前缀匹配是按最长匹配，和顺序无关

### 5. Rewrite 规则

rewrite 用于重写 URI，常用于 URL 跳转和路径改写。

```nginx
rewrite regex replacement [flag];
```

**三个关键 flag 的区别：**

| flag | 行为 | 使用场景 |
|------|------|----------|
| `permanent` | 返回 **301** 永久重定向给客户端 | SEO 友好的永久跳转 |
| `redirect` | 返回 **302** 临时重定向给客户端 | 临时跳转 |
| `last` | 重写 URI 后，**重新发起 location 匹配**（内部跳转） | 需要用新 URI 匹配其他 location |
| `break` | 重写 URI 后，**不再重新匹配 location**，在当前 location 继续执行 | 不需要再匹配其他 location |

**last vs break 的核心区别（面试常考）：**

```nginx
# 示例配置
server {
    location /old/ {
        rewrite ^/old/(.*)$ /new/$1 last;    # 重写后重新匹配 location
    }

    location /new/ {
        root /var/www;
    }
}
```

```
用 last：
  请求 /old/page.html
  --> rewrite 成 /new/page.html
  --> 重新走 location 匹配流程
  --> 匹配到 location /new/
  --> 在 /var/www/new/page.html 找文件

用 break：
  请求 /old/page.html
  --> rewrite 成 /new/page.html
  --> 不重新匹配，仍在当前 location /old/ 继续
  --> 用当前 location 的 root 去找 /new/page.html
```

**permanent 和 redirect 的区别：**

```
permanent (301)：
  浏览器 --> 请求 /old --> Nginx 返回 301 Location: /new
  浏览器记住这个跳转，下次直接请求 /new，不再请求 /old

redirect (302)：
  浏览器 --> 请求 /old --> Nginx 返回 302 Location: /new
  浏览器不记住，下次还是请求 /old，每次都要跳转
```

**常用 rewrite 示例：**

```nginx
# HTTP 跳转 HTTPS
server {
    listen 80;
    server_name www.example.com;
    rewrite ^(.*)$ https://$host$1 permanent;
}

# 旧路径跳转到新路径
location /blog/ {
    rewrite ^/blog/(.*)$ /articles/$1 permanent;
}

# 去掉 URL 中的 index.html
rewrite ^(.*)/index\.html$ $1/ permanent;

# if 条件 + rewrite
if ($request_uri ~ "^/api/v1/") {
    rewrite ^/api/v1/(.*)$ /api/v2/$1 last;
}
```

**注意：** Nginx 社区有句名言"if is evil"。在 location 中使用 if 可能导致意想不到的行为，因为 if 在 Nginx 中不是真正的条件判断，而是创建了一个嵌套的 location。能用 try_files 或 map 替代的，尽量不用 if。

### 6. 变量机制

Nginx 内置了大量变量，用 `$` 开头引用。

**常用变量：**

| 变量 | 含义 | 示例值 |
|------|------|--------|
| `$host` | 请求的 Host（不含端口） | www.example.com |
| `$remote_addr` | 客户端 IP | 192.168.1.100 |
| `$request_uri` | 完整的原始 URI（含参数，不会被 rewrite 改变） | /api/users?page=1 |
| `$uri` | 当前的 URI（不含参数，会被 rewrite 改变） | /api/users |
| `$args` | 查询参数部分 | page=1&size=10 |
| `$scheme` | 协议（http 或 https） | https |
| `$server_name` | 匹配到的 server_name | www.example.com |
| `$request_method` | 请求方法 | GET |
| `$http_xxx` | 任意请求头（xxx 用小写+下划线） | $http_user_agent |

**$uri vs $request_uri 的区别（面试常考）：**

```
请求：GET /old/page?id=123 HTTP/1.1

rewrite 前：
  $request_uri = /old/page?id=123    （原始 URI，带参数）
  $uri         = /old/page           （不带参数）

rewrite ^/old/(.*)$ /new/$1 last;

rewrite 后：
  $request_uri = /old/page?id=123    （不变！永远是原始值）
  $uri         = /new/page           （被 rewrite 改了）
```

简单记忆：`$request_uri` 是客户端发来的原始值，永远不变；`$uri` 是 Nginx 内部处理后的当前值，会被 rewrite 修改。

**实际应用示例：**

```nginx
# 日志中记录原始请求和最终处理的 URI
log_format detailed '$remote_addr - $request_uri -> $uri';

# 用 map 根据变量设置值
map $request_uri $is_api {
    ~^/api/   1;
    default   0;
}

# 根据条件设置头
server {
    location / {
        add_header X-Debug-URI $uri;
        add_header X-Original-URI $request_uri;
        proxy_pass http://backend;
    }
}
```

## 小结（3-5 条关键点）

- 配置层级 main -> events/http -> server -> location，指令从外层继承到内层，内层可以覆盖外层。
- 虚拟主机靠 Host 请求头区分不同 server 块。default_server 是没有匹配到任何 server_name 时的兜底。
- location 匹配优先级：`=` 精确 > `^~` 前缀停止正则 > `~`/`~*` 正则（按配置顺序）> 普通前缀最长匹配。这是面试必考内容。
- rewrite 的 last 会重新匹配 location，break 不会。permanent 是 301 永久重定向，redirect 是 302 临时重定向。
- `$request_uri` 是原始请求 URI 永远不变，`$uri` 是当前处理后的 URI 会被 rewrite 修改。
