# Nginx - 第 4 课：反向代理、负载均衡与 proxy_pass 的关键细节

## 学习目标（本节结束后你能做到什么）

- 理解反向代理到底在代理什么，为什么后端服务通常不直接暴露给公网。
- 掌握 `proxy_pass` 的基本行为，尤其是“带 `/` 和不带 `/`”的经典坑。
- 说清楚 `Host`、`X-Real-IP`、`X-Forwarded-For`、`X-Forwarded-Proto` 等头的作用。
- 理解 `upstream`、常见负载均衡策略、会话保持和健康检查的取舍。
- 面对 502、503、504 这类网关侧错误时，知道先从哪里排。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 反向代理的本质：代替后端接待所有外部请求

当 Nginx 做反向代理时，客户端看到的是 Nginx，真正处理业务的是后端。

```text
浏览器 -> Nginx -> Java / Go / Node.js / Python 服务
```

这样做的价值非常大：

- 后端服务不必直接暴露公网
- 后端可以自由更换端口、语言、实例数量
- 统一入口可以集中做 HTTPS、鉴权、日志、限流等基础治理
- 多台后端可以挂在同一个域名后面

所以反向代理不是“多绕一跳”，而是把入口治理抽到了一层专门的基础设施里。

### 2. `proxy_pass` 是怎么工作的

最基础写法：

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:8080;
}
```

意思是：匹配到 `/api/` 的请求，交给 `127.0.0.1:8080` 处理。

但真正容易出问题的地方，是 URI 转发规则。

### 3. `proxy_pass` 带 `/` 和不带 `/` 的区别

这是 Nginx 面试里的经典必考坑。

假设客户端请求：

```text
/api/users
```

#### 情况一：不带 `/`

```nginx
location /api/ {
    proxy_pass http://backend;
}
```

这时 Nginx 会把原始 URI 原样转过去，后端收到的是：

```text
/api/users
```

#### 情况二：带 `/`

```nginx
location /api/ {
    proxy_pass http://backend/;
}
```

这时 Nginx 会把匹配到的前缀 `/api/` 替换成 `/`，后端收到的是：

```text
/users
```

如果你不理解这个细节，就很容易出现：

- 本地接口是 `/users`
- 线上转发后变成 `/api/users`
- 后端路由死活对不上

一句话记忆：

- `proxy_pass` 后面没有路径时，更偏向“原样转发 URI”
- `proxy_pass` 后面带路径时，更偏向“替换匹配前缀”

### 4. 为什么要补这些代理头

真实请求经过 Nginx 后，后端看到的“连接对端”通常已经是 Nginx，而不再是原始客户端。所以需要显式把一些信息传下去。

最常见配置：

```nginx
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
```

#### `Host`

告诉后端客户端原本访问的是哪个域名。很多多租户、回调生成、绝对地址拼装都依赖这个头。

#### `X-Real-IP`

给后端一个“直接客户端 IP”的值。单层代理下通常够用。

#### `X-Forwarded-For`

它是一个代理链。每经过一层代理，就在后面追加一段 IP。

所以如果系统中有多层代理，真正排查来源时更应该看这条链，而不是只看 `X-Real-IP`。

#### `X-Forwarded-Proto`

告诉后端原始协议是 `http` 还是 `https`。很多应用需要依靠它判断是否生成 HTTPS 链接、是否设置安全 Cookie。

### 5. upstream：一组后端，而不是一台后端

当后端从一台变多台时，通常会写成：

```nginx
upstream backend {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}
```

然后：

```nginx
location / {
    proxy_pass http://backend;
}
```

`upstream` 本质上是一组可选后端实例。

### 6. 常见负载均衡策略和适用场景

#### 轮询

默认策略，按顺序依次分配。

适合：

- 后端实例性能相近
- 请求耗时差异不大

#### 加权轮询

性能更强的机器分更多请求。

适合：

- 混部环境
- 新老机器性能差异明显

#### `ip_hash`

同一客户端 IP 尽量打到同一台后端。

适合：

- 还没把会话外置
- 临时需要做粘性会话

缺点也很典型：

- NAT 场景下大量用户共享出口 IP，容易倾斜
- 扩缩容后映射波动大
- 本质上是在为“会话没外置”兜底，不是最优长期方案

#### `least_conn`

把请求给当前活跃连接数最少的实例。

适合：

- 请求处理时长波动大
- 长连接较多

### 7. 会话一致性别只会背 `ip_hash`

面试里如果问“怎么解决 session 粘性问题”，很多人第一反应就是 `ip_hash`。这当然能答，但只能算基础分。

更成熟的答案应该是：

- 短期可以用 `ip_hash` 做会话粘性
- 更长期、更稳妥的方案是把 session 放到 Redis 或数据库等共享存储中
- 这样后端无状态，负载均衡策略就更自由

这体现的是架构意识：不要把入口层策略当成业务状态设计的永久替代品。

### 8. 健康检查：开源版更偏被动

Nginx 开源版对 upstream 的健康感知主要偏被动。

比如：

```nginx
server 10.0.0.1:8080 max_fails=3 fail_timeout=30s;
```

这表示：

- 在 `fail_timeout` 时间窗口内，如果失败达到 `max_fails`
- 这台机器会被暂时标记为不可用

问题在于：它通常要等到真实用户请求失败后，才知道实例有问题。

所以更准确的说法是：**这是失败驱动的感知，不是提前主动探测。**

如果你希望定时健康检查接口、提前摘流，往往需要：

- 商业版能力
- 第三方模块
- 或交给云负载均衡、Kubernetes、Service Mesh 做更强的健康治理

### 9. 超时配置是排障必修课

常见的几个超时：

```nginx
proxy_connect_timeout 5s;
proxy_send_timeout 60s;
proxy_read_timeout 60s;
```

含义分别是：

- `proxy_connect_timeout`：和后端建立连接最多等多久
- `proxy_send_timeout`：向后端发送请求数据最多等多久
- `proxy_read_timeout`：发完请求后，等后端响应最多等多久

理解它们非常重要，因为不同阶段超时，说明问题位置不同。

### 10. 502、503、504 怎么区分

这是 Nginx 排障与面试中的高频题。

#### 502 Bad Gateway

Nginx 作为网关，去找后端了，但后端返回了异常结果，或者压根没法正常握手。

典型场景：

- 后端进程挂了
- 端口没监听
- 后端返回了非法响应
- upstream 连接被重置

#### 503 Service Unavailable

服务当前不可用，可能是：

- 没有可用 upstream
- 限流策略主动拒绝
- 维护模式

#### 504 Gateway Timeout

Nginx 找到后端了，也成功发过去了，但等后端响应等太久，超过 `proxy_read_timeout`。

很多人会把 502 和 504 混在一起。更好的记忆方式是：

- 502 更像“连上了但对方不正常”
- 504 更像“对方太慢，等超时了”

### 11. 一个实战排障顺序

当线上大量 502/504 出现时，不要只盯着 Nginx 配置。更好的排查链路是：

1. 看 Nginx error log，先确认是 connect 阶段失败还是 read 阶段超时
2. 确认后端进程是否存活、端口是否监听
3. 确认后端机器 CPU、内存、线程池、连接池是否打满
4. 看 upstream 是否大面积被标记失败
5. 检查最近是否发版、是否改了 `proxy_pass` 路径、是否有证书/域名/网络侧变化

能把这条排查思路说出来，已经比单纯背状态码强很多。

## 小结

- 反向代理的本质，是由 Nginx 统一接待客户端，再把请求分发给后端。
- `proxy_pass` 带 `/` 和不带 `/` 决定了 URI 是否替换前缀，是经典高频坑。
- 代理头不是“装饰品”，它们决定了后端能否看到原始域名、客户端 IP 和协议。
- `upstream` 是一组后端实例，常见策略包括轮询、加权、`ip_hash`、`least_conn`。
- 502、503、504 代表的是不同阶段的异常，排障时必须结合超时配置和错误日志定位。
