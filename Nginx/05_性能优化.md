# Nginx - 第 5 课：性能优化

## 学习目标（本节结束后你能做到什么）

- 理解 sendfile 零拷贝的原理，以及它比普通文件传输快在哪里
- 掌握 tcp_nopush 和 tcp_nodelay 的区别及同时开启的策略
- 配置 keepalive 连接复用（客户端到 Nginx 和 Nginx 到后端两段）
- 合理配置 gzip 压缩，知道什么该压缩什么不该压缩
- 掌握 worker 参数调优、proxy_cache 缓存和操作系统层面优化

## 内容讲解

### 1. sendfile 与零拷贝

**普通文件传输流程（不用 sendfile）：**

当 Nginx 需要把磁盘上的文件发送给客户端时，传统方式需要经过 4 次数据拷贝和 4 次上下文切换：

```
                    用户态                     内核态

                                        1. 磁盘 --> 内核缓冲区
                                           (DMA 拷贝)
                    2. 内核缓冲区 --> 用户态缓冲区
                       (CPU 拷贝)
                    3. 用户态缓冲区 --> Socket 缓冲区
                       (CPU 拷贝)
                                        4. Socket 缓冲区 --> 网卡
                                           (DMA 拷贝)

总共：4 次拷贝，4 次用户态/内核态切换
其中 2 次 CPU 拷贝是浪费的——数据经过用户态只是"路过"
```

**sendfile 零拷贝流程：**

```
                    用户态                     内核态

                    （不经过用户态！）     1. 磁盘 --> 内核缓冲区
                                           (DMA 拷贝)
                                        2. 内核缓冲区 --> 网卡
                                           (DMA 拷贝/或内核内拷贝)

总共：2 次拷贝，2 次上下文切换
数据不经过用户态，直接从磁盘到网卡
```

配置方式：

```nginx
http {
    sendfile on;    # 开启 sendfile，默认 off
}
```

**为什么叫"零拷贝"：** 不是真的零次拷贝，而是零次 CPU 拷贝。数据不经过用户态内存，省掉了两次 CPU 参与的拷贝操作。这对于传输大量静态文件（图片、CSS、JS）的场景，性能提升非常明显。

### 2. tcp_nopush vs tcp_nodelay

这两个选项看起来矛盾（一个要攒包，一个要立刻发），但实际上可以同时开启。

**tcp_nopush（攒包发大文件）：**

```nginx
tcp_nopush on;    # 需要配合 sendfile 一起用
```

作用：等数据攒够一个完整的 TCP 包再发，避免发送大量小包。适合传输大文件时减少网络开销。

原理：对应 Linux 的 `TCP_CORK` 选项，像软木塞一样"堵住"TCP 连接，等数据积累够了再一起发。

**tcp_nodelay（立刻发送）：**

```nginx
tcp_nodelay on;    # 默认就是 on
```

作用：禁用 Nagle 算法，数据不攒包，立刻发送。适合交互式场景，减少延迟。

Nagle 算法本意是合并小包减少网络拥塞，但会引入延迟（等凑够数据或等前一个包的 ACK）。

**两者同时开启的策略：**

```nginx
http {
    sendfile on;
    tcp_nopush on;     # 传输文件时攒包
    tcp_nodelay on;    # keepalive 连接上立刻发
}
```

Nginx 的行为：
1. 发送文件的主体数据时，用 `tcp_nopush` 攒满整包再发（高吞吐）
2. 发送文件的最后一个包时，关闭 `tcp_nopush`，切换到 `tcp_nodelay` 立刻发出（低延迟）
3. 后续的 keepalive 连接上，用 `tcp_nodelay` 快速响应

简单记忆：**大块数据攒着发（高吞吐），最后一包和交互数据立刻发（低延迟）。** 两者并不矛盾。

### 3. Keepalive 连接复用

TCP 建立连接需要三次握手，断开需要四次挥手，开销不小。keepalive 让连接建立一次后复用多次。

**客户端到 Nginx 的 keepalive：**

```nginx
http {
    keepalive_timeout 65;      # 连接空闲多久后关闭（秒）
    keepalive_requests 100;    # 一个连接最多处理多少个请求后关闭
}
```

这个比较简单，HTTP/1.1 默认就支持 keep-alive。

**Nginx 到后端的 keepalive（重点！）：**

默认情况下，Nginx 每次向后端转发请求都会新建一个 TCP 连接，用完就关。在高并发场景下，大量的建连和断连会消耗后端资源（TIME_WAIT 堆积）。

```
不用 keepalive：
  请求1：Nginx --建连--> 后端 --断连-->
  请求2：Nginx --建连--> 后端 --断连-->
  请求3：Nginx --建连--> 后端 --断连-->
  （大量短连接，TIME_WAIT 堆积）

用 keepalive：
  请求1：Nginx --建连--> 后端（保持连接）
  请求2：Nginx ---------> 后端（复用连接）
  请求3：Nginx ---------> 后端（复用连接）
  （少量长连接，高效复用）
```

**配置方法（必须三步齐全）：**

```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;

    keepalive 32;    # 每个 worker 到 upstream 保持的空闲连接数
}

server {
    location / {
        proxy_pass http://backend;

        # 这两行是必须的！否则 keepalive 不生效
        proxy_http_version 1.1;                  # 必须用 HTTP/1.1
        proxy_set_header Connection "";           # 清除 Connection 头
    }
}
```

**为什么必须设这两行：**
- `proxy_http_version 1.1`：HTTP/1.0 默认是短连接，必须升到 1.1 才支持长连接
- `proxy_set_header Connection ""`：客户端发来的请求可能带 `Connection: close`，如果不清掉，Nginx 转发给后端时也会带上，导致后端关闭连接

这是面试高频考点：很多人配了 `keepalive` 但忘了这两行，导致连接复用不生效。

### 4. Gzip 压缩

Gzip 压缩可以大幅减小响应体大小（文本类通常压缩到原来的 20-30%），减少网络传输时间。

```nginx
http {
    gzip on;                         # 开启 gzip
    gzip_min_length 1024;            # 小于 1KB 的不压缩（压缩收益小）
    gzip_comp_level 4;               # 压缩级别 1-9（推荐 4-6）
    gzip_types                       # 需要压缩的 MIME 类型
        text/plain
        text/css
        text/javascript
        application/json
        application/javascript
        application/xml
        application/xml+rss;
    gzip_vary on;                    # 添加 Vary: Accept-Encoding 头
    gzip_proxied any;                # 对代理请求也压缩
    gzip_buffers 16 8k;              # 压缩缓冲区
}
```

**压缩级别的性价比：**

```
级别    压缩率      CPU 开销     推荐程度
 1      较低        很低         不推荐（压缩效果差）
 2-3    中等        低           一般场景可用
 4-6    较高        中等         最佳性价比，推荐
 7-9    很高        很高         不推荐（CPU 开销大，收益递减）
```

级别从 4 到 9，压缩率只提升几个百分点，但 CPU 开销翻倍。所以一般设 4 到 6 即可。

**什么不应该开启 gzip：**

| 类型 | 原因 |
|------|------|
| 图片（jpg、png、gif、webp） | 已经是压缩格式，再 gzip 几乎不减小，白浪费 CPU |
| 视频（mp4、webm） | 同上，已经压缩过 |
| 压缩文件（zip、gz、br） | 已经压缩过 |
| 很小的文件（< 1KB） | 压缩后加上 gzip 头可能反而变大 |

### 5. Worker 参数调优

```nginx
# 工作进程数，设为 auto 等于 CPU 核心数
worker_processes auto;

events {
    # 每个 worker 的最大连接数
    worker_connections 10240;

    # 使用 epoll（Linux 下最高效）
    use epoll;

    # 允许一个 worker 同时接受多个新连接
    multi_accept on;
}
```

**worker_connections 的实际含义：**

一个 Nginx 能处理的最大并发连接数 = `worker_processes` x `worker_connections`

但注意：**如果是反向代理场景，实际能处理的客户端连接数要除以 2。** 因为每个客户端连接，Nginx 还需要一个连接去连后端。

```
反向代理场景：

客户端连接 --> Nginx --> 后端连接
 (占 1 个)              (再占 1 个)

所以每个客户端请求实际占 2 个连接

最大客户端并发数 = worker_processes x worker_connections / 2

假设 4 核 CPU，worker_connections = 10240：
最大客户端并发 = 4 x 10240 / 2 = 20480
```

如果是纯静态文件服务（不需要连后端），就不用除以 2。

### 6. Proxy Cache 缓存

Nginx 可以缓存后端的响应，相同请求直接从 Nginx 缓存返回，不再打到后端。

```nginx
# 定义缓存区域（写在 http 块中）
proxy_cache_path /var/cache/nginx
    levels=1:2                     # 目录层级
    keys_zone=my_cache:10m         # 缓存名称和索引内存大小
    max_size=1g                    # 缓存最大磁盘空间
    inactive=60m                   # 60 分钟没被访问就清除
    use_temp_path=off;             # 不用临时路径，直接写缓存目录

server {
    location /api/ {
        proxy_pass http://backend;

        # 启用缓存
        proxy_cache my_cache;
        proxy_cache_valid 200 10m;         # 200 响应缓存 10 分钟
        proxy_cache_valid 404 1m;          # 404 响应缓存 1 分钟
        proxy_cache_key "$scheme$host$request_uri";  # 缓存 key
        proxy_cache_use_stale error timeout updating;  # 后端出错时用旧缓存

        # 添加头信息方便调试
        add_header X-Cache-Status $upstream_cache_status;
    }
}
```

`$upstream_cache_status` 的值：
- `HIT`：命中缓存
- `MISS`：未命中，请求了后端
- `EXPIRED`：缓存过期，请求了后端
- `STALE`：使用了过期的缓存（因为后端不可用）
- `BYPASS`：被 proxy_cache_bypass 跳过了缓存

**缓存穿透问题：**

当大量请求访问不存在的资源时，每次都 MISS 缓存打到后端，这就是缓存穿透。

解决方案：

```nginx
# 方案 1：缓存 404 响应
proxy_cache_valid 404 5m;    # 即使是 404 也缓存 5 分钟

# 方案 2：proxy_cache_lock 防止缓存击穿
proxy_cache_lock on;                  # 相同 key 的请求排队
proxy_cache_lock_timeout 5s;          # 排队超时时间
# 第一个请求去后端取数据，其他相同请求等待第一个的结果

# 方案 3：proxy_cache_use_stale updating
proxy_cache_use_stale updating;       # 缓存过期更新期间，其他请求用旧缓存
```

### 7. 操作系统层面优化

Nginx 的性能不仅取决于自身配置，还受操作系统参数限制。

**worker_rlimit_nofile -- 文件描述符限制：**

```nginx
worker_rlimit_nofile 65535;    # 每个 worker 能打开的最大文件描述符数
```

每个连接占一个 fd，每个打开的文件占一个 fd。如果 fd 不够，新连接会被拒绝（报错 "too many open files"）。

同时要调整系统层面的限制：

```bash
# /etc/security/limits.conf
nginx soft nofile 65535
nginx hard nofile 65535

# 或临时设置
ulimit -n 65535
```

**somaxconn -- TCP 连接队列长度：**

```bash
# /etc/sysctl.conf
net.core.somaxconn = 65535           # listen() 的 backlog 上限
net.ipv4.tcp_max_syn_backlog = 65535 # SYN 队列上限
```

当 Nginx 来不及 accept 新连接时，新连接会在队列中排队。队列满了，新连接会被丢弃。高并发场景下必须调大。

**tcp_tw_reuse -- TIME_WAIT 复用：**

```bash
# /etc/sysctl.conf
net.ipv4.tcp_tw_reuse = 1     # 允许复用 TIME_WAIT 状态的连接
net.ipv4.tcp_fin_timeout = 15 # FIN_WAIT2 超时时间（默认 60 太长）
```

反向代理场景下，Nginx 频繁和后端建连断连，会产生大量 TIME_WAIT。开启 tcp_tw_reuse 可以快速复用这些连接。

**完整的系统优化配置汇总：**

```bash
# /etc/sysctl.conf

# TCP 连接队列
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535

# TIME_WAIT 优化
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_fin_timeout = 15

# 文件描述符
fs.file-max = 1000000

# TCP 内存和缓冲区
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# 应用配置
sysctl -p    # 使配置生效
```

## 小结（3-5 条关键点）

- sendfile 零拷贝让数据不经过用户态，从磁盘直接到网卡，减少 2 次 CPU 拷贝。传输静态文件必开。
- tcp_nopush 攒包发大文件提高吞吐，tcp_nodelay 立刻发减少延迟。两者同时开启不矛盾：Nginx 在传输主体时攒包，最后一包和交互场景立刻发。
- Nginx 到后端的 keepalive 必须同时设三个东西：upstream 里的 `keepalive N`、`proxy_http_version 1.1`、`proxy_set_header Connection ""`。缺一不可。
- gzip 压缩级别推荐 4-6（性价比最高），图片和视频不要开 gzip（已经压缩过）。
- 反向代理场景下实际客户端并发数 = worker_processes x worker_connections / 2（每个客户端请求占 2 个连接）。
