# Nginx - 第 6 课：HTTPS、安全与线上排障

## 学习目标（本节结束后你能做到什么）

- 理解 HTTPS 不只是“把 80 改成 443”，而是在 HTTP 之下增加了一层 TLS 安全能力。
- 说清楚 TLS 握手、证书链、TLS 1.2 与 TLS 1.3 的核心区别。
- 掌握 Nginx 中最常见的 HTTPS 配置项和优化点。
- 理解限流、限连、防盗链、安全响应头这些常见安全配置在解决什么问题。
- 面对证书报错、握手失败、混合内容、502/504 等线上问题时，形成一套排障顺序。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. HTTPS 到底多做了什么

HTTP 本身是明文的。也就是说，客户端和服务器之间如果没有额外保护，中间链路上理论上可以看到请求内容。

HTTPS 的本质是：

**HTTP + TLS**

TLS 帮你解决三件事：

- 机密性：别人就算截获数据，也看不懂内容
- 完整性：中途被篡改更容易被发现
- 身份认证：客户端能更有把握地确认自己连的真是目标服务器

所以 HTTPS 不是一个“新协议替代 HTTP”，而是在 HTTP 和 TCP 之间加了一层安全壳。

### 2. TLS 握手用一句话怎么解释

可以先记住一个最核心的直觉：

**握手阶段是为了安全地商量出后续通信要用的对称密钥。**

因为：

- 非对称加密安全但慢
- 对称加密快，适合真正的大量数据传输

所以 TLS 的设计思路不是“全程都用慢速非对称加密”，而是先安全协商，再高效传输。

### 3. TLS 1.2 的简化握手流程

用面试能讲清楚的粒度，可以这么说：

1. 客户端发 `Client Hello`
   - 告诉服务端自己支持哪些协议版本、加密套件
   - 带上一个随机数
2. 服务端回 `Server Hello`
   - 选定协议和加密套件
   - 返回证书
   - 再带一个随机数
3. 客户端校验证书
   - 看证书链是否可信、域名是否匹配、是否过期
4. 双方通过密钥交换过程，得到后续会话密钥
5. 之后真正的 HTTP 数据都用对称加密传输

面试时不要陷入过细的密码学细节，重点讲清楚“为什么先握手、为什么最后用对称密钥”就很好。

### 4. TLS 1.3 为什么更好

面试追问里经常会问 TLS 1.3 的改进。

你可以抓这几个点：

- 握手轮次更少，延迟更低
- 去掉了一些旧的不安全算法
- 更强调前向保密

所以一句话概括就是：**更快，也更安全。**

### 5. 证书链为什么经常出问题

很多人知道要配证书，却不知道为什么“有些浏览器能开，有些浏览器报错”。

原因常常在证书链。

客户端真正信任的通常是根 CA，但你的服务器证书很多时候不是根 CA 直接签的，而是：

```text
根证书 -> 中间证书 -> 服务器证书
```

Nginx 上一般要配的是：

- 服务器证书
- 中间证书

拼成完整链，也就是常说的 `fullchain.pem`。

如果漏了中间证书，部分客户端会补链成功，部分不会，于是你就会看到“同一个站点有人能访问，有人证书不可信”的怪现象。

### 6. Nginx 最常见的 HTTPS 配置

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;
}
```

配合一个 80 端口跳转：

```nginx
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}
```

这套配置体现了几个关键点：

- 443 监听开启 SSL
- 提供证书和私钥
- 限制协议版本，避免旧协议
- 启用会话复用降低握手开销
- HTTP 统一跳转到 HTTPS

### 7. HTTPS 优化：别只会“能访问”

线上不只是“通了就行”，还要尽量减少握手开销。

#### 会话复用

`ssl_session_cache`、`ssl_session_timeout` 可以降低重复连接的握手成本。

#### OCSP Stapling

客户端理论上还可能去 CA 查询证书吊销状态，这会增加额外延迟。OCSP Stapling 的思路是让服务器提前拿好这份状态，握手时一起带给客户端。

#### HTTP/2

HTTP/2 的价值不只是在语法层，而是在一个连接上支持多路复用、减少队头阻塞、优化头部传输。Nginx 中常见方式是：

```nginx
listen 443 ssl http2;
```

### 8. 基础安全能力：别把 Nginx 只当“转发器”

Nginx 经常是最靠前的一层，所以很适合承担部分基础安全策略。

#### 限流

```nginx
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;
```

它更像“单位时间内允许通过多少请求”。

#### 限连接

```nginx
limit_conn_zone $binary_remote_addr zone=conn_limit:10m;
```

它更像“一个来源最多能占多少并发连接”。

#### 防盗链

基于 `Referer` 做简单来源校验，常用于图片、下载资源。

#### 安全响应头

常见包括：

- `X-Frame-Options`
- `X-Content-Type-Options`
- `Strict-Transport-Security`
- `Content-Security-Policy`

你不需要一开始就把所有头都背全，但要知道：安全不仅是“接口做鉴权”，入口层也要承担一部分通用防护。

### 9. HSTS 和混合内容

两个很实际的线上问题：

#### HSTS

它的作用是告诉浏览器：以后这个域名请强制走 HTTPS。

优点是增强安全，缺点是如果你一旦配错证书或临时回退，会让问题更难救。所以要在确认 HTTPS 稳定后再谨慎开启。

#### 混合内容

页面本身是 HTTPS，但内部引用了 HTTP 图片、脚本、接口，这会导致浏览器告警甚至拦截。

所以如果“证书明明没问题，页面还是报不安全”，要想到混合内容。

### 10. 一套 HTTPS 故障排查顺序

当用户反馈 HTTPS 访问异常时，建议按这个顺序排：

1. 证书是否过期
2. 域名是否和证书匹配
3. 证书链是否完整
4. Nginx 是否正确加载了最新证书
5. 是否有 TLS 版本或加密套件兼容问题
6. 页面里是否存在混合内容
7. 是否有上游代理、CDN、云负载均衡做了额外终止，导致你看到的并不是最终链路

### 11. 把安全和可用性一起看

有些人把“安全”和“性能”分开学，容易学碎。

其实在 Nginx 里它们经常是一起出现的：

- 证书和 TLS 配置影响握手延迟
- 限流既是安全控制，也是稳定性控制
- 代理头既影响业务正确性，也影响审计追踪

所以更成熟的视角是：Nginx 入口层同时承担了安全、性能和治理三类职责。

## 小结

- HTTPS 本质上是在 HTTP 之下增加 TLS，解决机密性、完整性和身份认证问题。
- TLS 握手的核心，是安全地协商后续对称密钥。
- 证书链不完整是最常见的 HTTPS 线上故障之一。
- Nginx 里不只要会配证书，还要理解会话复用、HTTP/2、OCSP Stapling、HSTS 等优化和风险点。
- 限流、限连、防盗链和安全响应头说明 Nginx 不只是流量转发器，也是入口安全层。
