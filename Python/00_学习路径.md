# Python 学习路径与知识图谱

## 学习定位

Python 对后端工程师来说，不只是“写脚本方便”的语言。

它同时覆盖了很多非常实用的场景：

- 自动化脚本
- 后端服务与 API
- 数据处理
- 爬虫与批处理
- AI / LLM 工具链开发
- 运维和研发效率工具

很多人学 Python 时也容易走两个极端：

- 只会写脚本，遇到并发、包管理、项目结构、异常处理、异步模型就容易乱
- 只会背语法，知道装饰器、生成器、GIL、协程这些词，但落不到真实项目里

这条学习路径的目标，是把“语言基础”“工程写法”“并发模型”“Web 与脚本实践”连成一条线。学完之后，你不仅会写 Python，还能知道在什么场景下该用线程、协程、多进程，怎么把脚本写成可靠的工程代码。

## 为什么这样排序

如果按“从零语法”来排，Python 并发通常不会这么早出现。

但你这次已经先给了 Python 多线程这部分内容，而且它又是实际工作中非常常见的主题，所以这轮学习路径会先从一个很实战的切口切入：多线程最佳实践。

这样做的好处是，你会先把一个高价值问题吃透：

- Python 为什么明明有线程，却不总适合 CPU 密集型任务
- 什么场景该用 `ThreadPoolExecutor`
- 什么场景该切到多进程或 `asyncio`
- 线程安全、异常处理、停止机制这些工程问题该怎么处理

后面再去补协程、多进程、工程写法，就能把“能跑”升级成“会设计并发方案”。

## 学习方法建议

- 先问清楚任务类型：I/O 密集还是 CPU 密集。
- 不要只记语法，要带着工程问题去学，比如失败重试、线程安全、资源回收、日志、超时、退出。
- 学并发时，不只看“怎么开线程”，更要看“怎么控并发、怎么收结果、怎么处理异常、怎么优雅退出”。
- 每学完一课，尽量写一个可运行的小模板，而不是只看概念定义。
- 多把 Python 放在真实场景里理解，例如爬虫、批处理、并发调接口、异步 Web 服务。

## 学完后你应该能回答的问题

- 为什么 CPython 里的多线程更适合 I/O 密集型，而不是纯 CPU 密集型。
- `threading.Thread` 和 `ThreadPoolExecutor` 在工程实践里怎么选。
- 线程、协程、多进程分别适合哪类问题。
- 为什么共享状态越多，多线程程序越难维护。
- 如何设计一个有异常处理、有限并发、可优雅退出的 Python 并发程序。

## 推进规则

- 如果目标是实战和工程落地，建议按顺序学。
- 如果你近期主要在写批处理、脚本工具、接口聚合，优先反复看第 1、2、3、5 课。
- 如果你近期主要在写高并发 I/O 服务，优先反复看第 1、2、4、6 课。
- 后续每课尽量沉淀成“场景判断 + 原理解释 + 推荐写法 + 常见误区 + 自测题”的结构。
