# RAG学习路径

## 学习目标

这个专题不是为了让你记住一串 AI 名词，而是为了把你训练到这样一种程度：

- 你能用后端工程师能落地的方式解释什么是 RAG，而不是只会说“向量检索 + 大模型”。
- 你能把一个 RAG 系统拆成`数据准备 -> 索引构建 -> 检索召回 -> 上下文组装 -> 生成回答 -> 评测优化`几个阶段。
- 你能说清楚 chunk、embedding、metadata、topK、rerank、token 预算这些参数为什么会直接影响效果和成本。
- 你能设计一个最小可用的 RAG 服务，知道什么时候用 PostgreSQL + pgvector 就够了，什么时候要考虑专职向量库或搜索引擎。
- 你能区分 RAG、微调、传统搜索、规则查询各自解决的问题，不会把所有“知识问答”都粗暴丢给大模型。

## 重要说明

很多人一提到 RAG，就会下意识把它理解成一句话：把文档切块、做向量、相似度检索、拼进 Prompt。  
这句话不算错，但工程上远远不够。真正决定效果的，经常不是“有没有接上向量库”，而是下面这些细节有没有处理对：

- 文档有没有清洗干净，页眉页脚和噪声有没有去掉
- chunk 是按 token 硬切，还是按标题、段落、表格、代码结构切
- metadata 是否足够支撑过滤、权限控制和结果解释
- 检索是纯向量、BM25，还是混合检索
- 召回结果有没有 rerank、去重、合并相邻片段
- Prompt 里有没有明确要求“基于资料回答，不知道就说不知道”
- 效果不好时，你有没有评测方法去定位问题是在解析、切块、检索还是生成

所以，这个专题会按“真正能落地”的顺序来学，而不是只讲一些概念名词。

## 知识图谱

建议把 RAG 拆成 6 个层次来学：

1. 问题定义层
   - 大模型为什么会幻觉
   - 为什么私有知识、最新知识、长文档知识不能完全靠预训练记忆
   - RAG 到底在补什么短板

2. 数据准备层
   - PDF、Word、网页、表格怎么解析
   - 清洗规则怎么定
   - chunk 为什么不能随便切
   - metadata 为什么经常比向量本身还重要

3. 检索召回层
   - embedding 是什么
   - 向量检索和关键词检索各自擅长什么
   - topK、过滤条件、混合检索、rerank 分别在解决什么问题

4. 生成组装层
   - 怎么把检索结果变成 Prompt
   - token 预算怎么分配
   - 如何要求模型引用来源、避免胡编
   - 什么情况下应该拒答或降级

5. 工程落地层
   - PostgreSQL + pgvector 怎么建模
   - 文档更新、重建索引、版本管理怎么做
   - 多租户、权限过滤、缓存、监控、失败重试怎么接进去

6. 评测优化层
   - 怎么判断问题出在召回还是生成
   - 如何做离线样本集和人工验收
   - RAG 常见坑：切坏、召回偏、上下文污染、成本失控、索引漂移

## 课程安排

1. `01_RAG是什么、它解决什么问题.md`
   - 先把问题定义清楚
   - 重点讲大模型的知识边界、RAG 的基本链路、RAG 和微调/搜索的区别

2. `02_Chunk分片：RAG里最容易被低估、却最影响效果的一层.md`
   - 讲清文档解析、清洗、切块、metadata、评估和生产问题
   - 重点讲 chunk 为什么既是检索单元，也是整条 RAG 链路的质量杠杆

3. `02b_经典RAG之后：Data Proxy、Knowledge Unit与Progressive Search.md`
   - 讨论经典 chunk RAG 的边界以及知识库的下一代形态
   - 重点讲 Data Proxy、结构化知识单元、渐进式搜索和树/图拓扑的工程权衡

4. `03_Embedding与向量检索.md`
   - 讲清 embedding、本体向量空间、相似度和索引
   - 重点讲 query/document embedding 一致性和模型选择

5. `04_召回、过滤、Rerank与混合检索.md`
   - 讲清检索链路怎么收口
   - 重点讲 topK、BM25、向量召回、metadata filter、rerank 的配合

6. `05_上下文组装、Prompt与Token预算.md`
   - 讲清检索结果为什么不能原样全塞给模型
   - 重点讲上下文打包、引用来源、拒答策略和成本控制

7. `06_RAG服务的数据建模与工程实现.md`
   - 讲清一个后端服务怎么把 RAG 真正跑起来
   - 重点讲入库表结构、异步索引、缓存、权限、多租户和更新机制

8. `07_评测方法与常见问题排查.md`
   - 讲清效果不好时怎么定位问题
   - 重点讲离线评测、人工抽检、指标设计和常见反模式

9. `08_面试表达与系统设计回答.md`
   - 把前面的知识压缩成能讲出来的系统设计表达
   - 重点讲面试中如何解释 trade-off，而不是堆技术名词

## 推荐学习方式

- 每一课都带着一个问题学：这一层是在解决什么具体问题，而不是只是多了一个组件。
- 每次看到一个新名词，都追问三件事：它放在链路哪一步、它改善的指标是什么、它带来的代价是什么。
- 先建立完整链路，再深挖某个点。不要一上来就钻进向量库实现细节，结果反而忘了 RAG 的目标是“让回答更可信、更可控”。
- 学习时尽量把自己代入一个真实业务，比如企业内部知识库、客服问答、制度检索、代码文档问答，这样很多 trade-off 会更具体。

## 学完后你应该能回答的问题

- 为什么大模型明明很强，企业知识问答还是常常需要 RAG？
- 为什么“接上向量库”不等于 RAG 效果就会好？
- chunk、topK、rerank、token 预算之间是什么关系？
- 为什么很多场景里 RAG 和传统关键词检索应该一起用，而不是二选一？
- 一个最小可用的 RAG 服务，后端应该有哪些核心表、任务和接口？
