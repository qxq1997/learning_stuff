# RAG - 07b：元数据过滤：按用户、时间、来源、权限过滤不是附属功能，而是检索边界

## 学习目标（本节结束后你能做到什么）

1. 你能解释为什么 metadata filtering 不是“结果页上的筛选器”，而是 RAG 检索问题本身的一部分。
2. 你能区分`pre-filter`、`post-filter`、`filter-aware ANN`三类方案，并知道它们各自的延迟、召回和安全代价。
3. 你能从用户、时间、来源、权限四类约束出发，设计一个真正可上线的过滤模型。
4. 你能讲清 2024-2026 向量数据库在过滤能力上的关键演进：namespaces、payload partition、ACORN、tiered multitenancy。
5. 面试里谈多租户 RAG、ACL、时间窗口检索时，你能给出结构清楚且不犯安全错误的答案。

---

## 1. 先把问题摆正：过滤不是“找到了以后再筛掉”，而是“哪些文档有资格参与检索”

很多团队第一次做 RAG，会天然把过滤理解成 UI 层的事：

- 用户点了“只看本周”
- 用户点了“只看产品文档”
- 用户点了“只看自己部门”

于是他们在系统里做成：

1. 先向量检索 topK
2. 再把不满足条件的结果过滤掉

这个思路在搜索引擎 demo 里还能勉强凑合，但在生产 RAG 里会同时带来两个问题：

1. `安全问题`
   - 如果权限过滤不是硬约束，而是召回后的附加步骤，你就默认允许“不该看的文档”先参与了检索排序。
   - 某些系统里甚至会把未授权文档的摘要、标题、embedding 相似性间接暴露出来。

2. `召回问题`
   - 如果 topK 里大部分高分结果都被过滤掉，你最后真正能传给模型的上下文可能只剩 1-2 个，甚至为空。
   - 这不是 rerank 能救的，这是召回空间一开始就设错了。

所以 metadata filtering 的正确数学表达是：

```text
在满足约束 F(d)=true 的文档子集里，寻找与查询 q 最相关的 d
```

而不是：

```text
先全局找最相关的 d，再把不合法的删掉
```

这两句话看起来只差一点点，工程后果却完全不同。

---

## 2. 原理：这是一个“受约束的近邻搜索”问题

向量检索原本做的是：

```text
argmax_d sim(q, d)
```

加上 metadata filtering 之后，问题变成：

```text
argmax_d sim(q, d), s.t. F_user(d) ∧ F_time(d) ∧ F_source(d) ∧ F_acl(d)
```

这里最关键的变化是：

`可行域变小了。`

一旦可行域变小，ANN 索引的很多默认假设就不再成立：

- 原本图上最近的点，可能全都不满足 filter；
- 原本 HNSW 只需要往“离 query 最近”的区域走，现在还要考虑“这些区域里有没有合法点”；
- 如果 filter 和向量分布低相关，遍历会突然变慢。

所以 metadata filtering 的难点从来不只是“支持 where 条件”，而是：

`在受约束子空间里还能不能高效找到足够好的近邻。`

---

## 3. 三种典型实现方式：它们的差别非常大

### 3.1 Post-filter：先查，再删

这是最容易实现的方案：

1. 全局 ANN 取 topK
2. 在应用层过滤用户/时间/来源/权限
3. 不够再补查，或者直接返回较少结果

优点：

- 实现最简单
- 不依赖底层索引能力

缺点：

- 安全边界脆弱
- 当过滤条件严格时，容易`过滤完几乎没剩`
- 需要更大的 K 才能补足，延迟和成本上升

这个方案只适合：

- 非敏感数据
- 过滤条件很弱
- 或者你只是做原型验证

在企业 RAG 里，如果涉及权限，不建议把它当默认方案。

### 3.2 Pre-filter：先缩子集，再检索

这类方案会先基于 metadata 建 allow-list，再在这个子集上做向量检索。

优点：

- 权限边界更清晰
- 过滤后的 topK 更可信

缺点：

- 当 allow-list 很大时，性能压力仍然明显
- 如果底层只会“过滤后全扫”，规模一大就会慢

### 3.3 Filter-aware ANN：让索引结构理解过滤

这是 2024-2026 里最值得关注的方向。

核心思想不是“先过滤再搜”，也不是“先搜再过滤”，而是：

`搜索过程本身就知道哪些点是合法的。`

以 Weaviate 文档里的 ACORN 为例，它针对 HNSW 过滤搜索做了专门优化：

- 不满足过滤条件的对象直接不参与距离计算
- 通过 multi-hop 更快到达满足过滤条件的局部区域
- 额外随机播种满足条件的 entrypoints，加速收敛到 filtered zone

Weaviate 官方说明里特别强调：  
当 filter 与 query vector 的相关性很低、而且 filter 很严格时，ACORN 会明显更快。

这句话背后的工程含义非常大：

`过滤并不是“额外多做一步”，它会改变近邻图的可达性。`

## 4. 四类最常见的过滤约束：它们的工程性质并不一样

### 4.1 用户过滤：解决“我只能搜到自己的数据”

典型字段：

- `user_id`
- `tenant_id`
- `org_id`
- `workspace_id`

这一类过滤的本质是`隔离`，不是排序偏好。  
所以它通常应该是硬约束。

Pinecone 2026 文档把多租户实现明确分成两类：

1. `namespace per tenant`
   - 一个 tenant 一个 namespace
   - 强隔离、语义清晰

2. `metadata filtering`
   - 通过 metadata 表达 tenant
   - 更灵活，但依赖过滤执行质量

Pinecone 文档里的建议非常直接：  
在 serverless 架构下，用 namespace 做租户隔离是最自然的做法。

Qdrant 则提供了另一种非常有代表性的思路：

- 单 collection
- 用 payload 里的 `group_id` 分区
- 把该字段标记为 `is_tenant=true`

这样存储层会把同租户向量尽量共置，查询时顺序读更友好。

这说明到 2026 年，多租户过滤已经不是“业务层拼几个 if”，而是`存储布局问题`。

### 4.2 时间过滤：解决“我要最近的，不是最像的”

典型字段：

- `created_at`
- `updated_at`
- `effective_from`
- `version`

时间过滤常见于：

- 政策制度
- 知识库版本
- 新闻资讯
- 工单和客服记录

这里最容易犯的错是把时间当 rerank feature，而不是检索约束。

举例：

- 用户问“现在报销上限是多少”
- 你召回了三年前的制度和上周的新制度
- 然后把它们一起塞给模型

模型未必能稳定自己判断哪份是当前有效版本。  
更安全的做法通常是：

- 先用`有效时间窗口`过滤
- 再在窗口内做相关性排序

当然，时间也不总是硬过滤。  
比如“过去一年客户投诉的主要原因是什么”，它既需要时间窗口，又可能需要保留一定历史跨度。

所以时间约束更像：

- 有时是 hard filter
- 有时是 recency prior
- 有时需要 hard + soft 结合

### 4.3 来源过滤：解决“我只信哪一类知识源”

典型字段：

- `source_type = handbook / ticket / wiki / code / web`
- `source_system = notion / jira / confluence / crm`
- `doc_level = official / draft / external`

来源过滤的价值在于：

- 控制答案可信度
- 控制语料风格
- 减少跨源污染

举例：

- 问制度条款时，应优先检索官方手册，而不是 Slack 讨论记录；
- 问“用户上周都报了什么 bug”时，工单系统比知识库更重要。

来源约束很多时候不一定是绝对过滤，也可能是：

- source-aware routing
- 或 source prior in rerank

这一点和权限过滤不同。  
权限通常必须是硬的；来源很多时候可以是软的。

### 4.4 权限过滤：解决“看不见的文档必须从召回空间里消失”

这是最不能做错的一类。

典型字段：

- `acl_users`
- `acl_groups`
- `classification_level`
- `is_confidential`

这里一定要建立一个硬直觉：

`ACL 不是 relevance feature，而是 feasibility constraint。`

也就是说：

- 不能让模型自己根据提示词判断能不能看；
- 不能先检索再把不该看的文档摘要传给模型；
- 不能指望 reranker“学会”权限逻辑。

权限逻辑必须在检索前或检索过程中被执行。

## 5. 2024-2026 的关键变化：过滤开始进入“索引结构”和“租户布局”层

### 5.1 Pinecone：namespace 是强隔离语义

Pinecone 文档明确写的是：

- 使用 namespaces 安全隔离 tenant 数据
- 一个 namespace 对应一个 tenant
- 所有 upsert / query 都显式指向 namespace

这说明它把多租户问题主要建模为`物理或逻辑隔离单元`问题。

适合场景：

- SaaS 型单租户隔离
- 权限边界清楚
- tenant 间基本不需要联查

### 5.2 Qdrant：payload partition + tenant-aware storage + tiered multitenancy

Qdrant 这两年的方向更偏“共享集合下的高效隔离”。

它的 2026 文档里有两点很值得面试时讲出来：

1. `is_tenant=true`
   - 会把同 tenant 数据组织得更接近
   - 更适合顺序读和过滤查询

2. `tiered multitenancy`
   - 小租户共享 shard
   - 大租户可提升到 dedicated shard
   - 随租户增长进行 tenant promotion

这背后的工程思想非常成熟：

`不是所有租户都值得一人一库，但也不是所有租户都该永远挤在一起。`

### 5.3 Weaviate：过滤不只是 where，而是 HNSW 的搜索策略

Weaviate 在文档里把过滤策略直接分成：

- `sweeping`
- `acorn`

而且明确说 ACORN 对“低相关、高限制”的过滤尤其有效。  
这说明过滤性能不再只是 metadata index 的问题，而是 ANN 遍历算法本身的问题。

这会影响你对系统瓶颈的判断：

- 如果过滤慢，不一定是 SQL where 慢；
- 也可能是 HNSW 在错误区域里白跑了很多步。

## 6. 系统设计上该怎么选：namespace、collection、payload、ACL 表，到底怎么组合

### 6.1 一个实用的决策框架

可以按这四个问题判断：

1. `权限边界是否绝对硬隔离？`
   - 如果是，优先 namespace / collection / dedicated shard

2. `tenant 数量是否非常多？`
   - 如果很多，不要轻易一 tenant 一 collection

3. `tenant 规模是否高度不均匀？`
   - 如果是，tiered multitenancy 很有价值

4. `是否经常跨 tenant / 跨来源联合检索？`
   - 如果经常联查，过度物理拆分会带来编排复杂度

### 6.2 一个常见的生产折中

对大多数企业知识库，我更推荐这样的组合：

- `硬权限边界`
  - tenant / workspace / org 作为 pre-filter 主约束

- `时间约束`
  - 有效版本 hard filter
  - 近似新鲜度作为 rerank feature

- `来源约束`
  - 默认 soft prior
  - 某些 query 类型升级为 hard filter

- `ACL`
  - 永远 hard constraint

如果你把这四类约束全都塞到一个简单 where 里，系统也许能跑，但很快会出现两个问题：

- 召回过窄
- explainability 很差

更成熟的做法是区分：

- `必须满足`
- `最好满足`

## 7. 最容易踩的 7 个坑

### 7.1 把 ACL 放到生成后

这是高危错误。  
模型一旦拿到未授权上下文，问题就已经发生了。

### 7.2 每个 tenant 一个 collection，最后几千个 collection 爆炸

很多库在 collection 数量极大时都会有明显管理和资源开销。  
Qdrant 文档就明确不建议在一个集群里创建成百上千 collection。

### 7.3 时间过滤和版本语义没设计清楚

`updated_at` 不等于 `effective_from`。  
文档最近被编辑，不代表它是当前生效版本。

### 7.4 来源过滤做成硬编码 if/else

今天只有 wiki 和 jira，明天就会变成：

- wiki
- jira
- crm
- email
- policy db
- code search

如果一开始不抽象成 metadata schema 和 routing schema，后面会很难演进。

### 7.5 过滤后候选太少，却不回补

如果 filter 很严格，topK 需要动态放大，或者做多级回补。  
否则 reranker 和生成器拿到的上下文根本不够。

### 7.6 用 metadata 代替真正的授权系统

metadata 只是权限的表达载体，不是权限系统本身。  
真正的用户组、组织树、继承关系、临时授权，往往要先由 auth 服务算出 allow-list，再投射到检索层。

### 7.7 只评测无过滤场景

很多团队离线评测都在公开 benchmark 上做，结果线上一开 ACL、多租户、时间窗口，性能立刻掉一截。  
这是因为你测的是“全局检索”，不是“受约束检索”。

## 8. 一个足够靠谱的实现骨架

```python
def constrained_retrieve(query, user_ctx):
    filters = {
        "tenant_id": {"$eq": user_ctx.tenant_id},
        "effective_from": {"$lte": user_ctx.now},
        "effective_to": {"$gte": user_ctx.now},
        "classification": {"$in": user_ctx.allowed_levels},
    }

    return index.search(
        query=query,
        top_k=80,
        filter=filters,
    )
```

如果系统更复杂，再往上叠：

- allow-list bitset
- tenant-aware shard routing
- source-aware branch retrieval
- filter-aware ANN strategy

正确的优化顺序通常是：

1. 先把约束表达清楚
2. 再决定它们是 hard 还是 soft
3. 再看底层索引如何高效执行

不要一上来就在向量库里试各种奇技淫巧，却没先定义清楚：

`什么是合法候选，什么只是更优候选。`

## 9. 面试里怎么回答，才不像只会讲“加个 where 条件”

如果面试官问：

`多租户 RAG 里你怎么做权限隔离？`

一个成熟回答应该覆盖四层：

1. `鉴权层`
   - 用户属于哪些 tenant、workspace、group

2. `过滤表达层`
   - 把权限结果投影成 namespace、tenant_id、acl filter 或 allow-list

3. `索引执行层`
   - pre-filter / namespace / tenant-aware shard / filter-aware ANN

4. `生成保护层`
   - 未授权内容绝不进入 prompt，也不出现在 citation 里

如果面试官再问：

`时间过滤应该放检索前还是 rerank？`

你可以答：

> 如果时间是合法性约束，比如“当前有效制度”，应该在检索前作为 hard filter；如果时间只是偏好，比如“优先最近案例”，可以把 recency 作为 rerank feature。核心是先区分时间到底是 feasibility 还是 preference。

---

## 小结

1. 元数据过滤不是界面筛选，而是`受约束检索`问题。
2. ACL、tenant、有效版本通常应被视为 hard constraints，而不是排序特征。
3. post-filter 简单但不稳；生产里更常见的是 pre-filter 或 filter-aware ANN。
4. 2024-2026 的最新变化是：过滤能力开始进入存储布局、ANN 遍历策略和多租户分片层。
5. 最成熟的系统不会把所有约束都揉成一个 where，而会区分`必须满足`和`最好满足`。
