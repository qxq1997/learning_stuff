# RAG - 08a：多路召回：把一个问题改写成多个检索子问题

## 学习目标（本节结束后你能做到什么）

1. 你能区分`多路召回`、`query expansion`、`query decomposition`、`多查询改写`之间的关系，而不是把它们都叫“改写一下 query”。
2. 你能解释为什么单条 query 往往不足以表达完整检索意图，以及多路召回到底在补哪一种缺口。
3. 你能设计`多子问题生成 -> 分支召回 -> 融合 -> 重排`链路，并知道分支数量、配额和去重怎么控制。
4. 你能讲出 2024-2026 这一方向的关键进展，包括 Query2doc、CSQE、MuGI 以及框架化的 sub-question engine。
5. 面试里被问到“为什么一条 query 不够”，你能从检索理论和工程经验两边都讲透。

---

## 1. 先把问题摆正：很多问题天然不是一个检索目标，而是多个检索目标叠在一起

用户问的表面形式，经常是“一句话”，但这不等于它只对应“一次检索”。

举几个典型例子：

### 1.1 比较类问题

`Redis 和 Memcached 在持久化和淘汰策略上有什么区别？`

这句话里至少包含两层拆分：

- Redis 的持久化
- Memcached 的持久化
- Redis 的淘汰策略
- Memcached 的淘汰策略

如果你只拿整句去检索，往往会出现两种结果：

- 命中“Redis vs Memcached”的综述文，但细节不够
- 或只命中其中一个维度，导致答案偏科

### 1.2 多跳类问题

`员工试用期离职后，门禁权限多久停用，设备回收流程由谁发起？`

它同时在问：

- 门禁权限停用时点
- 设备回收责任人

这不是一个单点事实，而是两个业务子问题。

### 1.3 含糊类问题

`最近我们的成本为什么涨了？`

这里面没写清：

- 成本指云资源、营销费用、客服成本，还是物流成本
- 最近是近一周、近一月，还是近一季度

如果你只把这句话原样 embedding，再去找相似 chunk，召回目标其实是模糊的。

所以多路召回真正解决的是：

`原始 query 对检索目标的表达，往往既不充分，也不解耦。`

---

## 2. 原理：把单个问题展开成“多个覆盖不同失配模式的检索探针”

你可以把单路检索想成：

- 发射一个探针
- 看它在语料里命中什么

多路召回则是：

- 发射多个探针
- 每个探针从不同角度切问题
- 最后把命中的候选并起来

这背后常见有三种模式。

### 2.1 Paraphrase Expansion：同义改写

目标是补足`表达差异`：

- 原 query 和文档说法不同
- 需要用几种近义方式去覆盖

例如：

- `停用权限`
- `冻结账号`
- `回收访问资格`

这类多路召回不是把问题拆开，而是把说法展开。

### 2.2 Aspect Decomposition：按维度拆子问题

目标是补足`多方面约束`：

- 比较
- 对比
- 多指标
- 多跳

例如：

- `Redis 持久化机制`
- `Memcached 持久化机制`
- `Redis 淘汰策略`
- `Memcached 淘汰策略`

这类更像检索版的 task decomposition。

### 2.3 Answer-oriented Expansion：朝“答案长什么样”去扩展

这类方法不只是改写 query，而是让模型推测：

- 如果答案是对的，它大概会包含哪些术语、实体和表述

HyDE 就属于这一脉，后面一课会单独展开。

所以多路召回可以看成一个总类：

`通过多种 query 表述或子问题，把原始检索空间展开。`

## 3. 为什么单条 query 常常不够

### 3.1 查询太短，信息熵太低

检索器最怕的不是 query 太长，而是 query 太短却承载了复杂意图。

比如：

`报销上限`

这四个字可能对应：

- 差旅报销上限
- 招待报销上限
- 实习生报销上限
- 2025 新制度上限

单 query 的表达容量就这么大。  
它不可能把每个隐含约束都带出来。

### 3.2 文档索引是碎的，而问题往往是组合的

知识库里的 chunk 通常按局部主题切分。  
但用户问题往往横跨多个 chunk、多个文档、多个来源。

单 query 去打一个碎片化索引，天然容易发生：

- 某一方面召回很好
- 但其他方面没进去

这也是为什么 multi-hop RAG 里，多路召回几乎是默认配置。

### 3.3 召回器很难同时兼顾“广覆盖”和“精约束”

你把 query 写得很短，召回范围更大，但噪声也会多。  
你把 query 写得很细，又可能把相关结果卡掉。

多路召回其实是在做一种折中：

- 有的分支偏泛化
- 有的分支偏约束
- 最后通过 union + rerank 来平衡

## 4. 2024-2026 的关键进展：从“多生成几条 query”走向“有原则的分支设计”

### 4.1 Query2doc：让扩写结果更像文档，而不是更像 query

微软在 EMNLP 2023 的 Query2doc 里提出的关键想法是：

`不是只生成几个近义问题，而是让 LLM 生成一个伪文档，再把它用于查询扩展。`

他们报告在 BM25 上最高有 3%-15% 的效果提升。  
这个结果很重要，因为它点出了一个本质：

检索系统更容易匹配“长得像文档的东西”，而不是过短的问题。

### 4.2 CSQE：让扩写受语料库本身约束

EMNLP 2024 Findings 的 CSQE（Corpus-Steered Query Expansion）进一步指出：

- 直接让 LLM 自由扩写，容易漂移
- 生成出来的内容可能看似合理，但并不贴近语料库真实表达

所以更好的做法是：

- 让扩写过程受到 corpus 的 steering
- 减少与库内语言分布脱节的“空想扩写”

这对生产系统很关键，因为很多失败不是“没扩写”，而是“扩写得太像模型脑补，不像你的文档”。

### 4.3 MuGI：多路生成不是越多越好，关键在于样本质量和整合方式

EMNLP 2024 Findings 的 MuGI 研究强调了两件事：

1. 多次生成可以带来多样性
2. 但收益取决于如何采样、如何融合、如何避免重复

这直接击中了生产痛点。  
很多团队做 multi-query 的第一版，会简单生成 5 条问题，然后全部检索。  
结果往往是：

- 5 条里 3 条高度相似
- latency 翻倍
- recall 没涨多少

MuGI 给出的提醒是：

`多路召回的核心不是“多”，而是“多样而互补”。`

### 4.4 2026：MQ4CS 把“多方面 query”从经验技巧推进成明确研究对象

EACL 2026 的 MQ4CS（Generating Multi-Aspect Queries for Conversational Search）进一步把这件事讲得更清楚：

- 很多复杂会话查询天然包含多个 aspect
- 单次 query rewrite 无法完整表达复杂信息需求
- 用 LLM 把一个 utterance 拆成多个 aspect-specific queries，能比单次改写拿到更好的检索效果

这篇工作的意义在于，它让我们对多路召回的理解更精确了：

`多路召回不只是“多生成几条说法”，而是显式覆盖信息需求中的多个方面。`

这和工程里的经验完全契合。  
很多复杂问题不是缺一个更聪明的 query，而是缺一组能覆盖不同方面的 query。

### 4.5 2025-2026：框架层把 sub-question engine 做成了标配能力

到 2026 年，LlamaIndex 已经把 `SubQuestionQueryEngine` 作为标准组件提供出来。  
它背后的思路非常直白：

- 先把复杂问题拆成子问题
- 再把子问题路由到合适的数据源或工具
- 最后汇总

这说明多路召回已经从“论文技巧”进入了工程中台层。  
它不再只出现在学术 multi-hop benchmark 上，而是开始成为复杂问答系统的正常设计。

## 5. 多路召回到底怎么做，才不是无脑并发

### 5.1 分支数不是越多越好

一个很实用的经验是：

- 轻量同义改写：2-4 条
- 比较/多跳拆解：3-6 条
- 超过 6-8 条以后，边际收益通常快速下降

因为分支一多，副作用会立刻出现：

- 成本上升
- 时延变差
- 候选去重难度上升
- rerank 压力变大

### 5.2 每个分支需要有“职责”

生产里最好不要让模型自由散射式生成 5 条 query。  
更稳的方式是明确告诉它：

- 1 条保留原始 query
- 1 条做近义改写
- 1 条做术语展开
- 2 条做子问题拆分

也就是让分支具备互补性，而不是随机多样性。

### 5.3 Union 之后必须做两件事：去重和配额

多路召回很容易出现两个问题：

1. 同一父文档下的多个 chunk 被重复命中
2. 某一个分支因为写得太泛，把候选池挤满

所以一般至少要做：

- parent-level dedupe
- per-branch quota
- per-parent quota

否则多路召回会从“覆盖更多面”变成“重复更多次”。

## 6. 一个可落地的生产骨架

```python
def multi_recall(query: str):
    subqueries = planner.generate_subqueries(
        query,
        max_branches=4,
        strategy=[
            "original",
            "paraphrase",
            "term_expansion",
            "decomposition",
        ],
    )

    all_hits = []
    for sq in subqueries:
        hits = hybrid_retriever.search(sq, top_k=30)
        all_hits.extend(tag_branch(hits, sq))

    candidates = dedupe_and_cap(
        all_hits,
        per_branch_limit=20,
        per_parent_limit=3,
        total_limit=80,
    )
    return reranker.rerank(query, candidates, top_k=12)
```

这里一个很重要的点是：

- 最终 rerank 还是回到`原始用户问题`

因为真正要回答的是用户最初的问题，而不是某个中间子问题。

## 7. 哪些问题特别适合多路召回

尤其值得打开多路召回的场景包括：

- 比较类：A 和 B 的区别
- 多条件类：同时涉及多个维度或约束
- 多跳类：答案跨多个来源或多个事实
- 对话续问类：当前问题省略了前文关键主语
- 企业术语混乱类：同一概念在不同系统里叫法不同

不太适合默认打开的场景：

- 精确 ID 查询
- 错误码查询
- 单字段定位
- 很短但很明确的 exact lookup

因为这些问题，BM25 / metadata filter / exact route 通常更有效。

## 8. 多路召回最容易踩的 8 个坑

### 8.1 生成很多相似分支

这会把系统变成高成本版单路召回。

### 8.2 不保留原始 query

原始 query 本身往往包含用户真实表达。  
很多“优化版分支”反而会过度抽象。

### 8.3 子问题写得太完整，反而限制召回

分支不是答案。  
它应该是好的检索探针，而不是把所有细节都钉死。

### 8.4 用子问题结果直接生成答案，不回到原问题整合

这样容易答成几段零碎小答案，缺失整体结构。

### 8.5 对所有 query 都默认开多路

多路召回是有成本的，不该当成全局默认开关。

### 8.6 没有 branch-level 评测

你需要知道：

- 哪类 query 受益最大
- 哪种分支贡献最大
- 哪些分支只是增加延迟

### 8.7 不做多样性约束

这会导致多个分支只是换个说法重复命中同样的内容。

### 8.8 忘了和 metadata / routing 联动

多路召回不是替代路由和过滤。  
复杂系统里，通常是：

- 先路由到合适知识源
- 再在该源内做多路召回

## 9. 面试里怎么讲，才像真的做过复杂检索

如果面试官问：

`为什么一条 query 不够？`

你可以答：

> 因为用户问题经常把多个检索目标压缩在一句话里，尤其是比较、多跳和对话续问场景。单条 query 要么太泛、召回噪声大，要么太细、漏掉相关文档。我们会把原问题展开成多个互补探针，比如原始表达、同义改写、术语展开和子问题拆分，再做 union + rerank。

如果面试官再问：

`多路召回和 HyDE 有什么区别？`

你可以答：

> 多路召回是总框架，重点是通过多个检索分支提高覆盖率；HyDE 是其中一种具体 query transform，它通过先生成“假想答案文档”来改造 dense retrieval 的输入。前者强调多分支，后者强调让 query 更像文档。

---

## 小结

1. 多路召回的本质，是把一个不足以表达完整检索意图的 query 展开成多个互补探针。
2. 它常见有三类：同义扩写、维度拆解、答案导向扩写。
3. 2024-2026 的关键变化，是从“随便多生成几条 query”走向“受语料约束、强调多样互补、框架原生支持”的设计。
4. 多路召回真正有效的前提，是分支有职责、union 后做去重配额、最终回到原问题 rerank。
5. 它特别适合比较、多跳、续问和术语混乱场景，不适合所有 query 默认开启。
