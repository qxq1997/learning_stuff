# RAG - 专题 S3：生产 RAG 踩坑实录 — 电力行业两年落地复盘

> 这一篇是一段一线从业者的复盘整理，原作者在电力行业（调度规程、继电保护、配网运维）做了两年多的知识库与 RAG 落地。本文保留作者原始叙述风格，并在每个坑后补一段`大纲对照`，把它和这套课程里对应的章节串起来，让你在背完概念之后，知道这些概念在生产里长什么样、会以什么方式翻车。
>
> 阅读建议：先一口气读完作者的故事，再回头看每段的"大纲对照"和"工程教训"。

---

## 学习目标

读完这一篇，你应该能：

1. 把"BM25 + 向量 + Rerank"这条标准链路在`真实业务`里的失败模式记到肌肉里，不会再用 demo 思维讲生产 RAG。
2. 理解为什么`领域路由 / 子智能体`经常比`调 TopK 和权重`更有效，并能在系统设计面里讲清楚它的代价。
3. 理解为什么`用户怎么问`比`检索怎么调`更影响系统效果，并能把它对应到第 8 课系列的查询理解。
4. 理解为什么`规程冲突`、`版本变更`、`运维盲区`是 RAG 的隐形杀手，并把它对应到 S1（metadata）、第 15 课（工程落地）、第 14 课（评测）。
5. 能在回答`RAG 还有没有价值`这种问题时，说出"它从主角变成了底层能力"这种带判断的回应，而不是站队。

## 0. 故事的舞台：电力行业 ToB 知识库

先建立一下情境。这个项目要服务的对象不是 C 端用户，而是电网内部的`值班员、调度员、运维人员`。他们手里的资料长这样：

- 调度规程（厚得像砖，每年小改一次大改一次）
- 继电保护手册（公式、定值、动作逻辑）
- 配网运维规范（场景化操作流程）
- 设备说明书（多到几十类几百份）

用户的问题往往不是"是什么"，而是"现在怎么办"——值班员盯着告警屏幕，发现一台主变跳闸，他需要的是`几秒内能给我可执行的处置建议`。

知道了这个背景，下面的故事才有重量。

## 1. 故事原文（作者口吻）

BM25 + 向量检索 + Rerank，我们也用了。

搭起来不难，开源框架一堆，半天跑通 demo。

然后上线第一个礼拜就被骂了。

说说踩的坑。

### 坑一：知识库越大，召回越差

这个最反直觉。

用户把调度规程、继电保护手册、配网运维规范、设备说明书……全扔一个知识库里。你问一个继电保护的问题，检索结果里混进来三条配网的、两条调度的。关键词都命中了，但答非所问。

一开始我们硬调，TopK 从 10 调到 20，调到 30，最后调到 40。40 条候选里才勉强能捞到真正需要的那条。但 TopK=40 意味着每条 query 过 40 条文档再喂大模型，延迟上去了 token 也上去了，而且大模型拿到 40 条上下文反而更容易被带偏。

后来换了思路——**子智能体 + 意图识别**。先判断用户问题属于哪个领域，路由到对应的子智能体，每个子智能体只搜自己的知识库。调度的问题去调度知识库搜，保护的问题去保护知识库搜，互不干扰。

改完之后 TopK 调回 10 就够了。准确率反而比 TopK=40 的时候高。

折腾了那么久调参调权重，最后发现换个组织方式就解决了。**一锅炖不如分灶做**。

> **大纲对照**：这个坑是第 8c 课`Query Routing`的真实落地形态。把它链接起来你会发现：
> - 这种现象不是"知识库越大越差"那么玄学，本质是`领域间向量空间相互干扰`+ `BM25 在跨领域专有名词上失去 IDF 区分度`。
> - "TopK 从 10 调到 40" 是典型的`想用召回数量补领域噪声`的反模式。第 7c 课讲过：reranker 的 precision 来自候选集质量，如果候选集本身有结构性偏差，rerank 救不了。
> - "改成子智能体" 在工程上等价于`显式 corpus partitioning + query routing`。它把一个`一对多`的检索问题降维成了多个`一对一`检索问题。
> - 代价：`router 错路由`变成新的故障点，所以第 8c 课特别强调要给 router 配 fallback。
>
> **一句话教训**：当你发现 TopK 只能靠加大才能命中，先别调参，先问"这堆文档是不是根本就不该在一个 corpus 里"。

### 坑二：混合检索权重怎么调都不对

BM25 权重高了，"差动保护动作"只能搜到精确包含这五个字的内容。向量权重高了，"主变差动保护"给你搜出来"线路差动保护"——它觉得都是差动保护嘛，差不多。

reranker 呢，那种"看起来相关、关键词都命中、但业务上完全不搭"的东西它分不出来。因为 reranker 不懂业务啊。通用 reranker 怎么知道"主变"和"线路"是两个完全不同的东西？它不知道，它就觉得都是"差动保护"嘛。

后来我们加了一层**业务规则过滤**——用领域分类做前置筛选，先把明显不属同一专业的结果干掉，再交给 reranker 排。不是完美方案，但比纯靠权重调参强多了。

> **大纲对照**：这个坑串起了三章——第 7a（混合检索）、第 7b（元数据过滤）、第 7c（rerank）。
> - 通用 reranker 的`领域无知`是 2024-2026 年最被低估的问题。`bge-reranker-v2-m3` 在 MTEB 上看着指标漂亮，落地一遇到电力 / 法律 / 医疗这种重术语领域就翻车，因为预训练语料里"主变"和"线路"在 attention 里没有被强制区分过。
> - 解法不是"换更强的 reranker"，而是`业务规则前置过滤 + 领域 metadata 强约束`。这正是 S1（metadata 设计）反复强调的：**ACL 和领域不能放在排序特征里，必须放在过滤条件里**。
> - 真正的生产链路通常是：`领域 hard filter → BM25+向量混合 → reranker → LLM`。不是 reranker 不行，是它不该被放在没过滤好的候选集前面。
>
> **一句话教训**：通用 reranker 是一个`领域中性`的精排器，业务区分要靠过滤而不是排序。

### 坑三：用户根本不会提问

demo 测试的时候我们精心准备了测试问题："变压器油温过高应如何处置？""断路器拒动的原因有哪些？"

上线后值班员问的是：

- "昨天那个跳闸的怎么处理"
- "3 号主变那个告警"
- "上次那个操作票"

就"那个"。没有设备名没有告警类型没有时间没有上下文。

我们光"理解用户到底在问什么"这一层就迭代了四五个版本。比调检索参数重要十倍。

最后的方案是 **query 改写 + 多轮对话上下文传递 + 意图识别三件套**。用户说"昨天那个跳闸的"，得先从对话历史里捞出来他昨天问的是哪条线路、什么类型的跳闸，再拼成完整 query 去检索。这一层做好了，后面的检索和生成才有着力点。

> **大纲对照**：这是第 8 系列（08a 多路召回 / 08b HyDE / 08c Routing / 08d Step-back）整章的最佳证据。
> - 在工业值班这种`实战场景`下，用户 query 的特征是：极度省略、严重依赖时空上下文、习惯用代词。这和 ChatGPT 上的玩具 query 完全不同。
> - "昨天那个跳闸的"——这种 query 直接喂检索器是 0% 命中率的，因为它本身没有任何检索锚点。必须先经过`coreference resolution（指代消解）+ session-aware rewrite`补齐成"3 号主变 2026-04-19 18:23 的差动保护动作跳闸"，才能进检索。
> - 第 13 课（上下文工程）强调过：在多轮对话场景，`上下文 = 系统 prompt + 检索结果 + 对话历史 + 当前 query`，而当前 query 经常是上下文里信息密度最低的一段。
> - 这个坑也解释了为什么`Agentic RAG`在 ToB 实战里那么受欢迎：让 LLM 自己根据对话历史决定要查什么、要不要再问一次澄清，比静态 pipeline 强得多。这正是第 10 课的核心。
>
> **一句话教训**：你以为你在做`检索系统`，其实你 60% 的工程量在做`query 理解系统`。

### 坑四：规程之间会打架

规程 A 说"跳闸后 15 分钟内汇报调度"，规程 B 说"跳闸后立即处置并汇报"。两条都检索出来了，都相关，都正确。但到底听哪个？

靠技术解决不了，只能靠业务规则。哪个规程更新、适用范围是啥、优先级怎么排，要一条一条梳理。梳理完了还得维护。规程半年更新一次，知识库也得跟着更新。谁来做？不知道，反正程序员不想做。

（我到现在也不确定我们的优先级排序就是最优的，但总比两条打架强。）

我们最后建了一个**规程优先级表**——哪个文件更新优先听哪个、哪些是通用规则哪些是特殊情况，一条一条标好。标的过程比写代码还痛苦，要拉着业务专家一起过。但标完之后这种矛盾好处理多了，至少不会同时把两条矛盾答案喂给大模型。

> **大纲对照**：这个坑直接对应 S1（metadata 与知识结构建模）和第 15 课（工程落地）里讲的`版本元数据 + lifecycle 字段`。
> - 规程冲突的本质是 RAG 系统假设了`所有 chunk 是平等证据`。但真实知识里证据有等级：`国标 > 行标 > 集团规程 > 公司规程 > 临时通知`，并且`新版 > 旧版`，`专项 > 通用`。
> - 解法不是 prompt 里写"如果矛盾就听新的"，而是把这些等级编码成 metadata 字段：`document.authority_level`、`document.version`、`document.effective_date`、`document.scope`，在`检索结果合并阶段`就按权威性裁决。
> - 第 13 课讲的`引用与证据约束`在这种场景下尤其重要——给最终答案附上`依据：《XX 规程 2026 版》第 X 条`，让用户能验证。
> - 但作者说的最现实的一点是："谁来维护？" 这就是第 15 课讲的`知识库的 ops 问题`：metadata 标注是`人 + 工具`混合工程，不是纯算法问题。优先级表必须有 owner、有更新流程、有变更审计。
>
> **一句话教训**：在多源知识库里，`证据等级`必须显式建模到 metadata，不能指望 LLM 在生成时自己判断。

### 坑五：评估体系从零开始建

RAG 效果好不好你怎么衡量？我们之前跑了 benchmark，85% 准确率，看着挺美。

然后拉了两个老值班员，每人花一周，对着 300 个真实问题逐条标"对不对、够不够、有没有遗漏"。

标完发现真实问题集上只有 62%。

那 23% 的差距全在用户提问不规范、多文档冲突、上下文缺失这些地方。你说气不气。

后来我们把这 300 条标注数据当成了**基准测试集**，每次迭代跑一遍对比。说实话比什么 RAGAs 框架好用，因为它是你自己的真实数据。

> **大纲对照**：这是第 14 课（评测体系）的核心论点的真实证词。
> - 公开 benchmark（NQ、HotpotQA、MS MARCO）上的 85% 不能外推到你的领域，因为这些 benchmark 上的 query 分布、答案模式、文档语料都和你的业务不重合。
> - 第 14 课讲过：生产 RAG 必须有`golden set`——你自己业务专家标注的 200-500 条真实样本，覆盖`高频意图、长尾意图、易错意图`三类。每次模型/索引/prompt 改动都跑一遍。
> - "比 RAGAs 好用"这句话要稍微辨证地看：RAGAs 适合做`无标注情况下的近似评估`和`大规模回归扫描`；golden set 适合做`高保真的小样本基线`。两者互补，不是替代。
> - 23% 的 gap 拆开看是什么？典型构成：query 不规范造成的检索失败（10-15%）、多文档冲突造成的回答冲突（5-8%）、上下文缺失造成的答非所问（3-5%）。每一类都对应这套大纲里某一章的工程问题。
>
> **一句话教训**：没有自家业务的 golden set，benchmark 再漂亮都是自欺欺人。

### 坑六：运维才是地狱

知识库不是做一次就完了。新规程发布了得加进去，旧规程废止了得删掉。删了之后之前引用这条规程的其他回答会不会受影响？不知道。

设备改名了（对，设备会改名），知识库里旧的名称要不要改？改了之后检索还能搜到吗？

最烦的是你不知道用户遇到了什么问题。他搜了一个东西，不满意，关掉了。你连日志都看不到这个"不满意"。你说这玩意儿怎么进化？连用户不满意你都看不到。

后来我们加了两个东西：一个是**检索日志全量记录**（用户搜了啥、点了哪条、停留多久），一个是**每个回答下面加了个"有用/没用"的反馈按钮**。数据量不大，但至少能看到"哪些问题答得不好"了。比盲调强。

> **大纲对照**：这是第 15 课（工程落地）和第 14 课（评测在线观测）的合体。
> - "新规程发布 → 旧规程废止" 是 S1 metadata schema 里 `lifecycle: active | deprecated | replaced_by` 字段存在的理由。光删除不够，要保留`tombstone`并标记被谁替代，这样老对话里的引用还能解释。
> - "设备改名" 是知识图谱里的 `entity alias` 问题。第 9 课（GraphRAG）里讲的实体对齐就是为这个场景设计的。生产里通常做法是维护一张 `entity_aliases` 表，检索时把 query 里的实体名扩展成所有别名再去召回。
> - "用户不满意你看不到" 是第 14 课在线评测最关键的一句话：**离线评测能告诉你 baseline，在线评测告诉你 drift**。最低成本的在线信号就是作者说的两件事——
>   - **隐式反馈**：检索日志（点击、停留、复制、拒绝重试）
>   - **显式反馈**：thumbs up/down
> - 进阶一点的会接入 LangSmith / Langfuse / Arize Phoenix 做 trace + session 分析。
>
> **一句话教训**：知识库系统是一个`长期演进的活物`，没有 lifecycle 管理 + 反馈通路的 RAG，上线第二个月就会进入腐烂模式。

## 2. 作者的最终判断：RAG 没白做，只是从主角变成了底层

> 最近看到国网内部的说法，RAG 已经不是关键场景了，主推的是智能体。说实话做了这么久 RAG 第一反应是有点白瞎了。
>
> 但想想又不是。RAG 本身不是终点，它是智能体的基础技术。没有知识库打底，智能体连事实都搞不清楚，还干什么活。我们后来做的子智能体路由、query 理解、评估体系，全都在智能体架构里用上了。
>
> 所以 RAG 没白做，只是它从"主角"变成了"底层能力"。

这一段判断，是这套课程第 1 课`RAG 重心迁移`和 S2`RAG 已死`争议的最好脚注。

把它和我们的大纲串起来：

- 第 1 课讲过：RAG 不是单一组件，而是把外部知识接到模型的范式。范式没死。
- S2 讲过：Claude Code / Codex 的 agentic search 路线没有抛弃 RAG，只是把`静态 pipeline 检索`换成了`LLM 主动调用工具检索`。
- 第 10 课讲过：Agentic RAG 不是替代 RAG，而是在 RAG 之上加了`决策层`。
- 这位作者的观察：他们做 RAG 时打磨出来的`子智能体路由 / query 理解 / 评估体系`三件套，**直接迁移到了智能体架构**。

也就是说——

> **RAG 这两年最有价值的资产，不是"我们建了一个向量库"，而是"我们沉淀了一套面向 Agent 的可探索知识空间"。** 后者是任何 Agent 时代都需要的基建。

这句话的反面就是：如果你做 RAG 时只是搭了个`langchain demo + chroma + gpt-3.5`，那你确实白做了，因为没有沉淀任何 metadata、评测、路由、领域结构。

## 3. 作者给"正在做 RAG"的人的建议

> demo 能跑通和真能用之间，差的是对业务的理解，不是技术。但最后能跑多好，取决于你花了多少时间去搞明白用户到底怎么问问题、文档之间会不会打架、行业里那些"常识"到底是什么。没有开源框架能帮你解决这些。
>
> 如果你正在做 RAG，别急着调参。先找真实的终端用户，收集 100 个他们真实会问的问题，用这 100 个问题测你的系统。比跑 1000 个 benchmark 发现的都多。

把这段翻译成可执行清单：

| 行动 | 大纲对应 | 预期产出 |
| --- | --- | --- |
| 收集 100 条真实用户 query | 第 14 课、第 8 系列 | golden set 雏形 + query 形态分析 |
| 标注每条 query 的领域、意图、答案位置 | S1、第 8c | 领域分类器训练数据 + 路由表 |
| 跑 baseline（BM25、向量、混合） | 第 7a-7c | 看到"领域错配"和"权重打架"的真实样子 |
| 拉 2 个领域专家做 7 天人工评测 | 第 14 课 | 真实准确率 vs benchmark 准确率的 gap 分析 |
| 加 thumbs up/down + 检索日志 | 第 15 课 | 上线后第一周就能看到 drift |
| 把规程冲突标进 metadata（authority、version、effective_date） | S1 | 多源冲突自动裁决能力 |

如果你打算做电力、医疗、法律、金融这种重术语 ToB 知识库，把这张表直接当 day-1 路线图就行。

## 4. 把这一篇的工程教训压成 6 句话

1. **领域错配靠路由不靠 TopK**：当一个 corpus 装不下多个领域时，不要试图用更大的召回数量补，应该把它拆成多个 corpus + 一个 router。
2. **业务区分靠过滤不靠排序**：通用 reranker 不懂你的领域语义，必须靠 metadata hard filter 在它前面把噪声拦掉。
3. **query 理解的工程量是检索的两倍**：值班员、医生、律师不会按 prompt 模板问问题，多轮指代消解 + 意图分类 + 改写是必备前置层。
4. **多源冲突要进 metadata**：authority_level、version、effective_date、scope 必须显式建模，不能指望 LLM 在生成时判断。
5. **没有 golden set 就没有迭代**：benchmark 上的 85% 在你领域可能就是 60%，差距全在 query 分布。每个项目必须有自己的 200-500 条标注集。
6. **RAG 不是项目，是产品的活体器官**：lifecycle、反馈通路、运维流程、owner 制度，缺一个就会在第二个月开始腐烂。
