# RPC 学习路径与知识图谱

## 学习定位

RPC 不是一个单独的协议名词，而是一整套“让远程调用看起来像本地方法调用”的设计思想和工程体系。

很多人学 RPC 会卡在两个地方：

- 只会背一句 Remote Procedure Call，却不知道它到底比普通 HTTP 请求多解决了什么。
- 只会从某个具体框架入手，比如 Dubbo 或 gRPC，但脑中没有抽象骨架，所以换个框架就又像重新学一遍。

这条路线的目标，是先建立 RPC 的通用骨架，再去理解不同框架的实现差异。

学完之后，你应该逐步具备这些能力：

- 你能解释 RPC 为什么存在，它和 HTTP/REST 是什么关系。
- 你能从一次调用的视角，把代理、序列化、协议、网络传输、服务端调度、返回结果串成一条线。
- 你能理解动态代理、异步调用、Future、requestId 为什么是 RPC 框架的基础零件。
- 你能说清服务发现、注册中心、客户端负载均衡、重试、熔断、幂等是如何配合工作的。
- 你能对比 gRPC、Dubbo、Thrift 的设计差异，并结合 K8s、Service Mesh 讨论演进方向。

## 知识图谱（从基础到进阶）

1. 先理解 RPC 解决什么问题
   - 远程服务之间为什么不能像本地方法一样直接调用
   - HTTP、REST、RPC 的关系
   - 对外 API 与对内服务调用的边界
2. 搞懂一次 RPC 调用的完整链路
   - 客户端 stub / proxy
   - 序列化与反序列化
   - 协议帧设计
   - 网络传输与 requestId
   - 服务端调度与返回
3. 理解“像本地调用”的魔法从哪来
   - 静态代理
   - 动态代理
   - 同步、异步、Future、回调
4. 理解协议和序列化
   - JSON、Protobuf、Hessian、Kryo
   - 长度前缀、粘包拆包
   - 向前兼容与字段演化
5. 理解服务发现
   - 注册中心
   - 心跳与健康检查
   - CAP 和为什么 P 是前提
   - K8s Service / DNS 模式
6. 理解稳定性治理
   - 客户端负载均衡
   - 重试、退避、超时
   - 熔断、降级、隔离
   - 幂等与去重
7. 理解框架与生态
   - gRPC、Dubbo、Thrift
   - K8s、Service Mesh、OpenTelemetry
8. 理解高级工程问题
   - 链路追踪
   - 优雅上下线
   - 金丝雀、路由、灰度
   - 设计一个最小 RPC 框架

## 为什么这样排序

这条路线故意先不从具体框架 API 开始，而是先回答两个更关键的问题：

1. 为什么会需要 RPC
2. 一次远程调用背后到底发生了什么

因为如果你没有先理解这两点，那么：

- 动态代理会变成“一个神秘黑盒”
- Protobuf 会变成“一个更快的 JSON”
- gRPC、Dubbo、Thrift 会变成“只是不同厂商的工具名字”

只有先建立抽象骨架，后面学任何框架，你都会知道它是在这条链路的哪个位置做了取舍。

## 学习方法建议

- 每一课都先把抽象概念放回真实业务场景，比如订单服务调用用户服务、库存服务、优惠券服务。
- 不要把 RPC 看成单点技术，而要始终和服务发现、负载均衡、超时重试、幂等这些稳定性问题一起理解。
- 学到框架时，不要直接比较“谁快”，先看协议、生态、治理能力和团队上下文。
- 学 CAP、注册中心、K8s 服务发现时，要始终带着“如果网络不稳定会怎样”这个问题去理解。

## 学完后你应该能回答的问题

- 为什么微服务之间常常不用裸 HTTP，而要用 RPC 框架。
- 一次 RPC 调用里，客户端到底是怎么把一个方法变成网络请求的。
- 动态代理和 Future 在 RPC 里分别扮演什么角色。
- Protobuf 为什么比 JSON 更适合高性能服务调用。
- 为什么注册中心常常更偏 AP，为什么 CAP 里的 P 不是可选项。
- gRPC、Dubbo、Thrift 各适合什么团队和场景。
- 如果让你设计一个最小 RPC 框架，你会包含哪些核心部件。

## 推进规则

- 从零开始建议按顺序学习，不跳课。
- 如果你已经会用某个 RPC 框架，建议至少补完第 2、3、5、6、8 课，把底层链路和稳定性治理补齐。
- 如果目标是面试，重点反复看第 2、3、5、6、7、8 课。
