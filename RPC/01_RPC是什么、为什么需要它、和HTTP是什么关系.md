# RPC - 第 1 课：RPC 是什么、为什么需要它、和 HTTP 是什么关系

## 学习目标（本节结束后你能做到什么）

- 用自己的话解释 RPC 的核心思想，而不是只背出英文全称。
- 理解“远程调用”和“本地调用”的本质差异。
- 分清 RPC 和 HTTP 不是互斥关系，而是不同层次的问题。
- 理解为什么内部微服务调用常偏向 RPC，而对外开放接口常偏向 HTTP/REST。
- 建立一张后续所有 RPC 学习都会反复用到的整体地图。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先从一个最朴素的问题出发

假设你有两个服务：

- 订单服务
- 用户服务

它们部署在不同机器上。订单服务现在想获取用户信息。

如果这两个类在同一个 JVM、同一个进程里，写法很简单：

```java
User user = userService.getUserById(123);
```

但现在它们不在同一进程里，中间隔着网络、机器、端口、协议和可能的故障。你不能真的像本地函数调用一样直接跳过去执行。

这就是 RPC 想解决的问题。

### 2. RPC 到底是什么

RPC 全称是 Remote Procedure Call，远程过程调用。

它的核心理想不是“让网络消失”，而是：

**把远程调用的复杂性尽量封装起来，让调用方以接近本地方法的方式使用远程服务。**

也就是说，表面上你写的是：

```java
userService.getUserById(123)
```

但背后其实发生了很多事：

- 代理拦截这个方法调用
- 把方法名和参数打包
- 通过网络发给远端服务
- 远端执行后把结果返回
- 本地再把结果还原成对象交给你

RPC 的本质，就是把这整条链路包装得像普通方法一样顺手。

### 3. 为什么不直接手写 HTTP 请求

当然可以手写 HTTP。比如订单服务自己拼一个 URL：

```text
http://user-service/api/users/123
```

然后自己：

- 构造请求
- 发 HTTP
- 处理状态码
- 解析 JSON
- 处理异常和重试

这在简单系统里完全可行。

但当系统服务越来越多时，问题就来了：

- 每次调用都要手工处理网络细节，开发体验差
- 文本协议和大量头部字段带来额外开销
- 接口契约弱，变更不容易在编译期发现
- 服务发现、负载均衡、重试、熔断都得自己拼装

所以 RPC 的价值不只是“更快”，而是把服务间调用变成一套标准化工程能力。

### 4. RPC 和 HTTP 不是对立面

这是一个很容易被问到也很容易答偏的问题。

很多人会说“HTTP 是协议，RPC 是框架”，这句话方向没错，但还不够完整。

更准确的理解是：

- HTTP 是一种应用层传输协议
- RPC 是一种调用抽象和通信模式

RPC 完全可以跑在 HTTP 之上。最典型例子就是 gRPC，它底层就是 HTTP/2。

所以“RPC vs HTTP”真正要比较的，通常不是“谁把谁替代”，而是：

- 你是想做通用、可读、面向外部的接口
- 还是想做高效、强契约、面向内部的服务调用

### 5. 为什么内部服务更偏 RPC

内部微服务之间的调用通常更看重这些点：

- 性能和传输效率
- 强类型约束
- 自动生成客户端代码
- 内建服务发现与治理能力

而 RPC 框架往往正好擅长这些：

- 可以用 Protobuf 这类二进制序列化，体积更小
- 可以通过 IDL 或接口约束接口变更
- 可以把重试、熔断、负载均衡、超时治理标准化

所以很多公司会形成一种很常见的边界：

- 对外：HTTP/REST 或 HTTP + JSON
- 对内：RPC

这不是铁律，但非常常见。

### 6. 为什么对外接口常偏 HTTP/REST

因为对外开放时，系统往往更看重：

- 通用性
- 可调试性
- 浏览器、移动端、第三方系统兼容性
- 文档和生态

而 HTTP/REST + JSON 在这些方面天然更友好：

- 抓包容易读
- 调试工具多
- 不要求调用方引入特定 SDK
- 跨语言、跨平台门槛更低

所以从工程角度看，不是 RPC 一定“更高级”，而是不同场景下的优化目标不同。

### 7. 一张最小认知图

你可以先把 RPC 理解成下面这条链：

```mermaid
flowchart LR
    A["调用方代码"] --> B["代理对象"]
    B --> C["序列化 / 协议编码"]
    C --> D["网络传输"]
    D --> E["服务端解码 / 调度"]
    E --> F["真实业务实现"]
    F --> G["返回结果"]
```

后面学的所有内容，本质上都在补这张图：

- 动态代理是在解释 B
- 序列化是在解释 C
- 粘包拆包和 requestId 是在解释 C 到 D
- 注册中心和服务发现是在解释 B 之前“到底该调谁”
- 负载均衡、重试、熔断是在解释 D 周围“调用失败怎么办”

### 8. RPC 的代价是什么

学任何技术都不能只看优点。

RPC 的代价也很明显：

- 隐藏了网络边界，容易让人误以为远程调用和本地调用一样便宜
- 框架会引入额外的协议、SDK、治理系统和学习成本
- 如果调用链太深，问题定位和稳定性治理会变复杂

所以 RPC 的正确姿势不是“哪里都上”，而是在内部服务协作足够复杂时，用它换标准化和效率。

### 9. 一个常见误区：本地调用语法，不等于本地调用成本

这是 RPC 学习里最重要的心智模型之一。

哪怕你写的是：

```java
userService.getUserById(123)
```

也不能忘了它背后可能意味着：

- 一次网络往返
- 一次序列化和反序列化
- 一次超时风险
- 一次重试风险
- 一次调用链放大的可能性

所以成熟的工程师不会因为“调用长得像本地方法”就忘掉这是远程调用。

## 小结

- RPC 的核心目标，是让远程服务调用尽量像本地方法调用一样自然。
- RPC 和 HTTP 不对立，RPC 可以运行在 HTTP 之上。
- 内部服务调用更偏 RPC，是因为更看重性能、契约和治理能力；对外接口更偏 HTTP/REST，是因为更看重通用性和可读性。
- 学 RPC 不能只记框架名字，必须先建立“代理、协议、传输、发现、治理”这条完整链路。
- 本地调用语法只是表象，网络边界和失败成本始终真实存在。
