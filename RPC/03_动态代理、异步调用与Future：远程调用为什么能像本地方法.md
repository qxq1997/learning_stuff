# RPC - 第 3 课：动态代理、异步调用与 Future：远程调用为什么能像本地方法

## 学习目标（本节结束后你能做到什么）

- 理解代理模式在 RPC 里的作用，分清静态代理和动态代理。
- 理解为什么远程调用能写成“像本地方法一样”的语法。
- 理解同步调用、异步调用、回调、Future 分别是什么。
- 说清楚 requestId 和 `pendingRequests` 这类结构为什么在异步 RPC 中必不可少。
- 能把动态代理和 Future 用通俗语言讲给刚接触 RPC 的人听。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先看“代理”这个词到底是什么意思

现实生活里，代理最容易理解的例子是中介。

你找中介租房，表面上你在和房子相关的人打交道，但很多事情不是房东本人亲自和你对接，而是中介代替房东完成：

- 展示房源
- 收集信息
- 帮你联系房东
- 转达结果

代码里的代理也是类似含义：

你调用的是 `userService`，但它不一定是真正实现业务逻辑的那个对象，也可能是一个“代你和远程服务打交道”的中间人。

### 2. 静态代理：手写一个中间人

最朴素的代理方式，是你自己手写一个代理类：

```java
class UserServiceProxy implements UserService {
    private final UserService target;

    public User getUserById(int id) {
        System.out.println("before");
        User result = target.getUserById(id);
        System.out.println("after");
        return result;
    }
}
```

这种写法的问题是：

- 每个方法都得手写转发
- 每个接口都得写一遍
- 方法一多，重复劳动爆炸

所以静态代理能帮助你理解“代理是什么”，但不适合大规模 RPC 场景。

### 3. 动态代理：运行时自动生成中间人

动态代理的核心思想是：

**代理类不用你一个个手写，而是程序运行时按统一规则帮你生成。**

你只需要告诉框架：“当有人调用这个接口的方法时，请统一走这段拦截逻辑。”

在 Java 世界里，最常见的思路是：

- JDK 动态代理：基于接口
- CGLIB：基于继承
- Dubbo 这类框架还可能使用 Javassist 等字节码方案

无论底层实现方式怎么变，本质都一样：**拦截调用，接管调用。**

### 4. RPC 里的动态代理到底接管了什么

以这行代码为例：

```java
userService.getUserById(123)
```

如果 `userService` 是动态代理对象，那么它不会真的在本地执行 `getUserById` 的业务逻辑，而是会拦截到：

- 方法名：`getUserById`
- 参数：`123`
- 参数类型：`int`

然后做这些事：

1. 封装请求对象
2. 序列化
3. 发网络请求
4. 等响应回来
5. 反序列化
6. 返回结果

所以动态代理真正“神奇”的地方，不是魔法，而是把这串机械流程藏在统一拦截器里了。

### 5. 为什么动态代理特别适合 RPC

因为 RPC 框架面对的服务接口通常很多，而“调用远程服务”的通用套路又非常一致：

- 取方法签名
- 打包请求
- 发出去
- 等结果

如果每个方法都手写一遍，不现实；如果统一由动态代理完成，就可以：

- 代码量大幅减少
- 所有远程调用走统一治理逻辑
- 更容易加超时、重试、链路追踪、熔断等横切能力

所以动态代理是 RPC 框架“像本地调用”的第一块基石。

### 6. 同步调用是什么

同步调用很好理解：

你发起请求后，就停在这里等结果。

```java
User user = userService.getUserById(123);
```

在结果没回来之前，这个线程通常不能继续往下做依赖该结果的逻辑。

同步调用的优点是：

- 编程模型简单
- 代码直观

缺点是：

- 线程会等待
- 如果调用链上有多个远程依赖，整体延迟容易累加

### 7. 异步调用是什么

异步调用的核心不是“结果不要了”，而是：

**先把请求发出去，调用线程先去做别的事，结果以后再取。**

最经典的生活类比是点外卖：

- 同步：你下单后站在门口一直等
- 异步：你下单后先去做别的，等骑手到了再取

在 RPC 里，异步特别有价值的场景是并行调用多个下游：

- 查用户
- 查库存
- 查优惠券

如果都同步串行，总耗时可能是三者相加；如果异步并发，总耗时更接近最慢那个。

### 8. Future 是什么

Future 可以理解成一个“未来会装进结果的盒子”。

你发出请求时，真正的结果还没回来，但框架可以先给你一个 Future：

- 现在盒子可能是空的
- 以后结果到了，会被放进去
- 你需要时可以从盒子里取

比如：

```java
CompletableFuture<User> future = new CompletableFuture<>();
```

这时候 `future` 还没有真正的用户数据，但它代表“将来会有一个 User 结果”。

### 9. 为什么异步调用必须把请求和响应对应起来

如果同一条连接上同时飞了很多请求：

- 请求 A：查用户
- 请求 B：查库存
- 请求 C：查优惠券

服务端返回时，完全可能是：

- 先回 B
- 再回 C
- 最后回 A

问题来了：客户端怎么知道这个响应是给谁的？

答案就是 requestId。

### 10. `pendingRequests` 这类结构在做什么

一个非常典型的简化模型是：

```java
ConcurrentHashMap<Long, CompletableFuture<?>> pendingRequests;
```

它的含义是：

- key：请求编号 requestId
- value：对应这个请求的 Future

发送请求时：

1. 生成一个 requestId
2. 创建一个 Future
3. 把它放进 `pendingRequests`
4. 把 requestId 和请求一起发出去

收到响应时：

1. 从响应里拿到 requestId
2. 去 `pendingRequests` 找对应 Future
3. 把结果塞进去
4. 从 map 中删除，表示这个请求已经完成

所以这张 map 本质上就是“订单号到收件人”的映射表。

### 11. 为什么 `future.get()` 会阻塞

因为 Future 只是一个承诺，不代表结果已经就绪。

如果你在结果还没到的时候调用：

```java
User user = future.get();
```

当前线程就会等待，直到：

- 有人调用 `future.complete(result)`
- 或者超时 / 失败

这也说明一个很重要的事实：

**异步调用并不等于永不阻塞。**

只是你可以把“什么时候阻塞、阻塞在哪个线程上”控制得更灵活。

### 12. 动态代理和 Future 是怎么配合的

你可以把它们的分工理解成这样：

- 动态代理：负责把“方法调用”拦截下来，发起远程请求
- Future：负责代表“这个远程请求未来的结果”

如果是同步调用，代理内部可能发出请求后直接等待结果再返回。

如果是异步调用，代理内部发出请求后可以立刻返回一个 Future，后续由网络线程在响应回来时把结果完成进去。

### 13. 一个最容易踩的理解误区

很多人学到动态代理后，会误以为 RPC 已经“像本地调用了，所以和本地调用差不多”。

这是错的。

动态代理只是把调用形式伪装得像本地方法，但它并没有抹掉这些现实：

- 网络会抖动
- 下游可能超时
- 响应会乱序
- 调用链会放大故障

所以成熟工程师会同时记住两层事实：

- 表面上它像本地方法
- 本质上它是远程调用

## 小结

- 代理的本质，是由一个中间对象代替你完成真正调用。
- 动态代理让 RPC 不必为每个接口手写代理类，而是统一接管方法调用。
- 异步调用的意义，是先发请求、后取结果，适合并行调用多个下游。
- Future 是一个“未来结果容器”，requestId 和 `pendingRequests` 用来把响应交回正确的 Future。
- 动态代理让远程调用“看起来像本地”，但并没有消除网络边界和失败成本。
