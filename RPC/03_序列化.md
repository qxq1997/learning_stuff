# RPC - 第 3 课：序列化

## 学习目标（本节结束后你能做到什么）

- 用类比解释序列化和反序列化的本质
- 对比五种主流序列化方案的优缺点，说清选型依据
- 解释 Protobuf 为什么体积最小、解析最快
- 理解 Protobuf 字段编号的规则和前后向兼容机制
- 面试中能流畅回答序列化相关问题

## 内容讲解

### 1. 序列化是什么

**序列化（Serialization）**就是把内存中的对象变成字节流，以便通过网络传输或写入磁盘。**反序列化（Deserialization）**就是反过来，把字节流变回对象。

用搬家打包来理解：你在北京的东西（内存中的对象），要搬到上海的新家（另一台机器）。你不能直接把整个房间搬走，而是要把东西**打包成箱子**（序列化成字节流），通过物流送过去（网络传输），到了上海再**拆箱还原**（反序列化成对象）。

```
北京的家（JVM A）                                上海的新家（JVM B）
┌──────────────┐                               ┌──────────────┐
│ UserInfo 对象 │                               │ UserInfo 对象 │
│  name = "张三" │   打包      物流      拆箱     │  name = "张三" │
│  age = 28     │ ──────> [字节流] ──────> ──>  │  age = 28     │
│  level = VIP  │  序列化    网络传输   反序列化  │  level = VIP  │
└──────────────┘                               └──────────────┘
```

序列化的好坏直接影响 RPC 的三个核心指标：
- **体积**：序列化后的字节流越小，网络传输越快
- **速度**：序列化/反序列化速度越快，调用延迟越低
- **兼容性**：能不能跨语言、能不能前后版本兼容

### 2. 五种主流序列化方案

#### 2.1 JSON

JSON 是最常见的序列化格式，人人都认识。

```json
{
  "userId": 12345,
  "name": "张三",
  "age": 28,
  "email": "zhangsan@example.com",
  "level": "VIP"
}
```

**优点**：
- 人类可读，调试方便，浏览器直接看
- 几乎所有语言都支持，天然跨语言
- 生态丰富，工具链完善

**缺点**：
- **体积大**：字段名重复传输（每条消息都带 `"userId"`、`"name"` 这些字符串）
- **解析慢**：文本格式需要逐字符解析，数字还要从字符串转成整数
- **类型信息弱**：不区分 int/long/float，精度可能丢失

适用场景：对外 API、配置文件、调试场景。**不适合高性能 RPC**。

#### 2.2 Protobuf（Protocol Buffers）

Google 开源的二进制序列化方案，gRPC 的默认序列化方式。

首先要写 `.proto` 文件定义数据结构：

```protobuf
// user.proto
syntax = "proto3";

message UserInfo {
    int64 user_id = 1;     // 1 是字段编号，不是值
    string name = 2;
    int32 age = 3;
    string email = 4;
    string level = 5;
}
```

编译后自动生成 Java 类：

```java
// 序列化
UserInfo user = UserInfo.newBuilder()
    .setUserId(12345)
    .setName("张三")
    .setAge(28)
    .setEmail("zhangsan@example.com")
    .setLevel("VIP")
    .build();

byte[] bytes = user.toByteArray();  // 序列化成字节数组，约 45 字节

// 反序列化
UserInfo parsed = UserInfo.parseFrom(bytes);
```

**优点**：
- **体积最小**：不传字段名，用编号代替；数字用 Varint 变长编码
- **解析最快**：二进制格式，直接按字节偏移读取，不需要逐字符解析
- **强类型 IDL**：`.proto` 文件定义接口，编译期检查，改接口就报错
- **跨语言**：支持 Java、Go、Python、C++ 等十几种语言

**缺点**：
- 二进制不可读，调试不方便
- 需要额外的 `.proto` 文件和编译步骤
- 学习曲线比 JSON 稍高

#### 2.3 Hessian

Dubbo 框架的默认序列化方式（Dubbo 2.x）。

```java
// Hessian 序列化
ByteArrayOutputStream bos = new ByteArrayOutputStream();
HessianOutput out = new HessianOutput(bos);
out.writeObject(userInfo);
byte[] bytes = bos.toByteArray();

// Hessian 反序列化
ByteArrayInputStream bis = new ByteArrayInputStream(bytes);
HessianInput in = new HessianInput(bis);
UserInfo user = (UserInfo) in.readObject();
```

**优点**：
- 不需要 IDL 文件，直接序列化 Java 对象
- 体积比 JSON 小，速度比 JSON 快
- 跨语言支持（Java、Python、C++ 等）

**缺点**：
- 性能不如 Protobuf 和 Kryo
- 社区活跃度一般

#### 2.4 Java 原生 Serializable

Java 内置的序列化方式，实现 `Serializable` 接口即可。

```java
public class UserInfo implements Serializable {
    private static final long serialVersionUID = 1L;
    private Long userId;
    private String name;
    private int age;
}

// 序列化
ObjectOutputStream oos = new ObjectOutputStream(outputStream);
oos.writeObject(userInfo);

// 反序列化
ObjectInputStream ois = new ObjectInputStream(inputStream);
UserInfo user = (UserInfo) ois.readObject();
```

**缺点非常多，生产不推荐**：
- **体积最大**：携带完整的类名、字段描述符等元信息
- **速度最慢**：需要大量反射和安全检查
- **安全漏洞**：反序列化攻击是 Java 安全的重灾区（Apache Commons Collections 漏洞等）
- **不跨语言**：纯 Java 专用
- **版本兼容差**：改了类结构容易反序列化失败

唯一的优点是用起来简单，但代价太大。**面试里提到 Java 序列化一定要说它的问题**。

#### 2.5 Kryo

Java 生态中性能最好的序列化框架之一。

```java
Kryo kryo = new Kryo();
kryo.register(UserInfo.class);  // 需要注册类

// 序列化
Output output = new Output(new ByteArrayOutputStream());
kryo.writeObject(output, userInfo);
byte[] bytes = output.toBytes();

// 反序列化
Input input = new Input(new ByteArrayInputStream(bytes));
UserInfo user = kryo.readObject(input, UserInfo.class);
```

**优点**：
- 速度非常快，体积非常小（Java 生态最优之一）
- 使用简单，不需要 IDL

**缺点**：
- **不跨语言**：纯 Java 专用
- **线程不安全**：Kryo 实例不能多线程共享，需要用 ThreadLocal 或对象池
- 需要手动注册类（不注册也行但性能会差）

### 3. 五种方案对比表

| 维度 | JSON | Protobuf | Hessian | Java 原生 | Kryo |
| --- | --- | --- | --- | --- | --- |
| **体积** | 大 | 最小 | 中 | 最大 | 小 |
| **速度** | 慢 | 最快 | 中 | 最慢 | 快 |
| **跨语言** | 支持 | 支持 | 部分支持 | 不支持 | 不支持 |
| **可读性** | 人类可读 | 不可读 | 不可读 | 不可读 | 不可读 |
| **IDL** | 不需要 | 需要 .proto | 不需要 | 不需要 | 不需要 |
| **安全性** | 好 | 好 | 一般 | 差 | 一般 |
| **典型使用** | REST API | gRPC | Dubbo 2.x | 不推荐 | 本地缓存 |

### 5. Protobuf 为什么这么小（面试加分项）

这是深入考察点，理解了这个你会比大多数候选人有优势。

Protobuf 小的原因有四个：

#### 5.1 不传字段名，用编号代替

JSON 每条消息都要传完整的字段名字符串：

```json
{"userId": 12345, "name": "张三", "age": 28}
//  ^^^^^^          ^^^^          ^^^
//  6字节           4字节         3字节  —— 光字段名就 13 字节
```

Protobuf 用 1-2 个字节的编号代替字段名：

```
字段编号1 + 值12345 | 字段编号2 + 值"张三" | 字段编号3 + 值28
// 编号只占 1 字节（编号 < 16 时），比字段名省得多
```

#### 5.2 Varint 变长编码

普通的 int32 固定占 4 字节，不管值是 1 还是 2147483647。

Protobuf 用 Varint 编码：小数字用更少的字节。

```
数值 1     →  1 字节  (0x01)
数值 127   →  1 字节  (0x7F)
数值 128   →  2 字节  (0x80 0x01)
数值 300   →  2 字节  (0xAC 0x02)
数值 12345 →  2 字节  (0xB9 0x60)
```

原理：每个字节的最高位（MSB）表示后面是否还有更多字节。1 表示还有，0 表示结束。

```
数值 300 的 Varint 编码过程：
300 = 100101100 (二进制)

拆成 7 位一组（从低位开始）：
  低 7 位: 0101100
  高 2 位: 10

编码为：
  第一个字节: 1_0101100 = 0xAC  (MSB=1，后面还有)
  第二个字节: 0_0000010 = 0x02  (MSB=0，结束)

结果：[0xAC, 0x02]，只用了 2 字节
```

实际业务中，大多数 ID 和计数值都不大，Varint 编码能省很多空间。

#### 5.3 TLV 结构，无分隔符

JSON 需要大括号 `{}`、逗号 `,`、冒号 `:` 等分隔符，这些都占字节。

Protobuf 用 TLV（Tag-Length-Value）结构，不需要任何分隔符：

```
┌──────────┬──────────┬──────────┐
│ Tag      │ Length   │ Value    │
│ (编号+类型) │ (值长度)  │ (实际值)  │
└──────────┴──────────┴──────────┘

Tag = (字段编号 << 3) | wire_type

wire_type:
  0 = Varint (int32, int64, bool)    → 不需要 Length 字段
  1 = 64-bit (double, fixed64)       → 固定 8 字节，不需要 Length
  2 = Length-delimited (string, bytes, 嵌套 message) → 需要 Length
  5 = 32-bit (float, fixed32)        → 固定 4 字节，不需要 Length
```

对于数字类型（wire_type=0），连 Length 字段都不需要，因为 Varint 自带终止标志。

#### 5.4 默认值不传输

在 proto3 中，如果字段值等于默认值（数字是 0，字符串是 ""，布尔是 false），这个字段就不传输，直接省掉。

```protobuf
message UserInfo {
    int64 user_id = 1;   // 默认值 0
    string name = 2;     // 默认值 ""
    int32 age = 3;       // 默认值 0
    bool vip = 4;        // 默认值 false
}
```

如果 `vip = false`，序列化时这个字段直接不出现在字节流里。反序列化时发现没有编号 4 的字段，就自动填默认值 false。

### 6. 字段编号的规则和兼容性

#### 字段编号不能改

```protobuf
// 版本 1
message UserInfo {
    int64 user_id = 1;
    string name = 2;
}

// 版本 2 —— 错误做法！
message UserInfo {
    int64 user_id = 1;
    string nickname = 2;  // 把 name 改成了 nickname，但编号不变
    // 旧版本反序列化时会把 nickname 的值当成 name，语义错误！
}
```

Protobuf 是靠编号来识别字段的，不是靠字段名。所以：
- **编号一旦分配就不能改**
- **字段名可以改**（只是代码层面的命名，不影响序列化）
- **已删除的字段编号不能复用**

#### reserved 关键字

当你要删除一个字段时，用 `reserved` 把编号和名字保留，防止以后被误用：

```protobuf
message UserInfo {
    reserved 2;            // 编号 2 已废弃，不能再用
    reserved "name";       // 字段名 "name" 已废弃

    int64 user_id = 1;
    string nickname = 3;   // 新字段用新编号
    int32 age = 4;
}
```

如果有人不小心用了编号 2，protoc 编译器会直接报错。

#### 前后向兼容机制

这是 Protobuf 在生产环境最重要的特性之一：

**前向兼容**（旧代码读新数据）：
- 旧版本遇到不认识的字段编号 → 直接跳过，不会报错
- 所以你可以安全地给 message 加新字段，旧版本的服务不受影响

**后向兼容**（新代码读旧数据）：
- 新版本发现某个字段在数据里不存在 → 填默认值
- 所以旧数据可以被新版本的服务正常读取

```
场景：用户服务升级，加了 phone 字段（编号 5）

新版本 UserInfo:
  user_id = 1
  name = 2
  age = 3
  email = 4
  phone = 5        ← 新增

旧版本订单服务收到新版本的数据：
  → 读到编号 1~4 正常解析
  → 读到编号 5，不认识，跳过
  → 正常工作，不报错 ✓

新版本用户服务收到旧版本的数据：
  → 读到编号 1~4 正常解析
  → 没有编号 5，填默认值 ""
  → 正常工作，不报错 ✓
```

这就意味着你可以在不停机的情况下，逐步升级各个服务。先升级提供方，再升级消费方，或者反过来，都不会出问题。

## 小结（3-5 条关键点）

- 序列化就是把对象变成字节流以便网络传输，选型看三个维度：体积、速度、跨语言。
- Protobuf 是综合最优的方案：体积最小（编号替代字段名 + Varint + 默认值不传）、速度最快（二进制直接解析）、跨语言好。
- Java 原生 Serializable 在生产上不推荐：体积大、速度慢、有反序列化安全漏洞。
- Protobuf 的字段编号一旦分配不能改、不能复用，删除字段要用 reserved 保留。
- Protobuf 天然支持前后向兼容：旧代码跳过不认识的新字段，新代码给缺失字段填默认值，可以做到无停机滚动升级。
