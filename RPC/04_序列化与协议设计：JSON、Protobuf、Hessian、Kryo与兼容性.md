# RPC - 第 4 课：序列化与协议设计：JSON、Protobuf、Hessian、Kryo 与兼容性

## 学习目标（本节结束后你能做到什么）

- 理解序列化在 RPC 里到底解决什么问题。
- 能从体积、速度、跨语言、易用性四个维度比较 JSON、Protobuf、Hessian、Kryo。
- 理解为什么 RPC 协议通常需要魔数、版本号、消息长度、序列化类型等字段。
- 理解 TCP 粘包拆包和长度前缀法。
- 理解 Protobuf 字段编号、向前兼容和协议演化的基本原则。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 序列化到底在干什么

本地对象在内存里，并不是网络可直接发送的形式。

比如一个 Java 对象：

```java
User user = new User(123, "张三", 25);
```

它在内存里可能包含：

- 对象头
- 字段值
- 指针引用
- JVM 私有布局

这些东西不能直接跨进程、跨机器传递。

所以序列化的作用，就是把对象转换成一段连续字节；反序列化则是把字节重新还原成目标语言中的数据结构。

你可以把它理解成搬家打包：

- 序列化：把家具拆好、打包、编号
- 反序列化：到了新家再按编号组装回来

### 2. 为什么 RPC 特别在乎序列化

因为 RPC 是高频服务间调用，序列化几乎发生在每一次请求和响应中。

它直接影响：

- 消息体大小
- 网络传输时间
- CPU 编解码开销
- 跨语言协作能力

所以序列化不是一个“可有可无的小细节”，而是 RPC 性能和工程体验的重要基础。

### 3. JSON：最容易理解，但不是最适合高频 RPC

JSON 的优点非常明显：

- 人能直接看懂
- 调试方便
- 各语言生态都成熟

比如：

```json
{"id":123,"name":"张三","age":25}
```

但它的缺点也很明显：

- 字段名重复出现，体积较大
- 文本解析成本比二进制高
- 类型表达相对松散

所以 JSON 非常适合对外 API、调试、低频调用，但在高性能内部 RPC 场景里往往不是首选。

### 4. Protobuf 为什么常被认为是 RPC 首选

Protobuf 是 Google 的 Protocol Buffers。

它通常先用 `.proto` 定义结构：

```protobuf
message User {
  int32 id = 1;
  string name = 2;
  int32 age = 3;
}
```

然后生成多语言代码，再用二进制进行编解码。

它的核心优势是：

- 体积小
- 解析快
- 跨语言好
- 契约强

它为什么小？

- 不传字段名，而是传字段编号
- 数字用变长编码
- 默认值字段可以不传
- 结构字符冗余少

这就是为什么 gRPC 默认选择 Protobuf。

### 5. Hessian、Kryo、Java 原生序列化分别怎么看

#### Hessian

在 Java 生态尤其是 Dubbo 历史里很常见。

优点：

- 使用方便
- 对 Java 对象支持自然

缺点：

- 跨语言能力不如 Protobuf
- 性能和协议演化能力通常不如现代主流二进制方案

#### Kryo

偏纯 Java 高性能序列化。

优点：

- 速度快
- 体积较小

缺点：

- 跨语言弱
- 线程安全等工程细节要小心

#### Java 原生序列化

最不推荐。

原因主要是：

- 体积大
- 速度慢
- 历史上安全问题很多
- 跨语言几乎没有可谈性

面试里如果被问到，通常直接说生产级 RPC 不建议使用 Java 原生序列化。

### 6. 一张实用对比图

你可以先用这四个维度快速比较：

| 方案 | 体积 | 速度 | 跨语言 | 可读性 |
| --- | --- | --- | --- | --- |
| JSON | 大 | 较慢 | 强 | 强 |
| Protobuf | 小 | 快 | 强 | 弱 |
| Hessian | 中等 | 中等 | 一般 | 弱 |
| Kryo | 较小 | 快 | 弱 | 弱 |

如果面试官问“怎么选”，比较稳妥的回答是：

- 对外接口重可读性和通用性，JSON 很合适
- 内部高性能、多语言 RPC，Protobuf 很常见
- 纯 Java 历史系统可能见到 Hessian、Kryo，但要看生态和治理成本

### 7. 协议设计为什么不等于序列化格式

这是一个容易混淆的点。

序列化格式关心的是“body 如何表示一个对象”。

协议设计关心的是“整条消息怎么组织，让接收方知道怎么解析”。

比如你完全可能设计一个协议帧：

```text
| magic | version | codec | compress | requestId | bodyLength | body |
```

其中：

- `body` 用 Protobuf
- 整个外层帧是你自定义

所以协议层和序列化层是相关但不同的两个层次。

### 8. 为什么要有魔数、版本号、序列化类型

#### 魔数（magic）

用于快速识别“这是不是我这个协议的消息”，相当于身份证。

#### 版本号

方便协议升级和兼容判断。

#### 序列化类型

告诉对方 body 用的是哪种编码方式，比如 JSON、Protobuf、Hessian。

#### 请求类型 / 响应类型

帮助区分是普通请求、心跳、异常响应还是其他控制消息。

#### bodyLength

用于解决消息边界问题，是最关键字段之一。

### 9. TCP 粘包拆包为什么一定要处理

TCP 不关心“消息”，只关心“字节流”。

所以你发两条请求：

- 请求 1：100 字节
- 请求 2：200 字节

服务端可能一次收到 300 字节，也可能只收到前 50 字节。

这就带来两个问题：

- 粘包：多条消息黏在一起
- 拆包：一条消息被分成多次收到

RPC 要解决它，最常见方式是长度前缀法：

1. 先读固定长度头部
2. 从头部拿到 bodyLength
3. 再按长度读完 body

只要这步做好，消息边界就清晰了。

### 10. Protobuf 字段编号为什么不能乱改

这是面试很爱追问的点。

因为序列化后的二进制里，真正依赖的是字段编号，而不是字段名。

如果你原来定义：

```protobuf
string name = 2;
```

后来把 `name` 改成：

```protobuf
string nickname = 2;
```

这还不算最糟。但如果你把字段 2 改成别的含义，旧版本客户端就会把新数据解析错位。

所以 Protobuf 的基本规则是：

- 字段编号一旦发布，就尽量不要改
- 删除字段时最好 `reserved`
- 新增字段用新的编号

这就是协议演化意识。

### 11. 协议演化为什么是工程问题，不只是语法问题

很多人以为兼容性只是“语法能不能解析”，其实更重要的是系统升级过程中的版本共存。

真实环境里很常见：

- 服务 A 已升级
- 服务 B 还没升级
- 某些老客户端还在跑旧版本

所以一个成熟协议必须允许一段时间内新老版本共存。

这也是为什么二进制协议设计里，“版本号”“字段保留”“向前向后兼容”都特别重要。

## 小结

- 序列化解决的是“对象如何变成可传输字节”的问题。
- JSON 易读但偏大，Protobuf 紧凑高效且跨语言强，是现代 RPC 的主流选择。
- 协议层和序列化层不是一回事，协议头要解决识别、边界、类型和版本问题。
- TCP 是字节流，没有消息边界，所以长度前缀法非常关键。
- Protobuf 字段编号一旦发布就要谨慎对待，协议演化能力是生产系统必须考虑的问题。
