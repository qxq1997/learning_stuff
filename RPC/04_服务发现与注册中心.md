# RPC - 第 4 课：服务发现与注册中心

## 学习目标（本节结束后你能做到什么）

- 解释为什么需要注册中心，以及它在 RPC 架构中的角色
- 说清服务注册、服务发现、变更通知三步流程
- 对比 ZooKeeper、Eureka、Nacos、Consul、etcd 的特点和选型依据
- 理解 CAP 理论，解释为什么 P 必须有，以及注册中心该选 CP 还是 AP
- 解释 K8s 如何做服务发现，以及它和传统注册中心的区别
- 串联前四课，说出完整的 RPC 调用流程

## 内容讲解

### 1. 地址写死配置文件的问题

在第 1 课和第 2 课里，我们默认调用方知道服务端的地址。但现实中这个地址从哪来？

最简单的做法：写在配置文件里。

```properties
# application.properties
user-service.address=192.168.1.100:8080
order-service.address=192.168.1.101:8080
```

这在服务少、机器少的时候能用，但规模稍微大一点就崩了：

- **扩容问题**：用户服务从 1 台扩到 5 台，你要改所有调用方的配置文件并重启
- **缩容/故障**：某台机器挂了，如果不及时改配置，请求还会打到挂掉的机器上
- **发布问题**：新版本上线时地址可能变化，手动改配置容易出错
- **环境管理**：测试环境、预发环境、生产环境地址都不一样，配置管理混乱

你需要一个**动态感知服务地址变化**的机制。这就是注册中心。

### 2. 注册中心的角色：通讯录比喻

注册中心就像手机通讯录。你不需要记住每个人的电话号码（IP 地址），只要知道他的名字（服务名），去通讯录里一查就能找到。

而且这个通讯录是**实时更新**的——某人换了号码，通讯录自动同步；某人注销了号码，通讯录自动删除。

### 3. 三步流程：注册 → 发现 → 通知

```
                    ┌─────────────────────┐
                    │      注册中心         │
                    │                     │
         ①注册      │  user-service:      │     ②发现
     ───────────>   │    192.168.1.100    │  <───────────
     服务启动时      │    192.168.1.101    │   拉取地址列表
     上报地址        │    192.168.1.102    │
                    │                     │
                    │  order-service:     │     ③通知
                    │    192.168.1.200    │  ───────────>
                    │                     │   地址变更时
                    └─────────────────────┘   推送给消费方

┌─────────────────┐                     ┌─────────────────┐
│  用户服务实例 1    │                     │    订单服务       │
│  192.168.1.100  │                     │  （消费方/调用方）  │
├─────────────────┤                     │                   │
│  用户服务实例 2    │                     │  本地缓存：       │
│  192.168.1.101  │                     │  user-service ->  │
├─────────────────┤                     │   [100, 101, 102] │
│  用户服务实例 3    │                     └─────────────────┘
│  192.168.1.102  │
└─────────────────┘
```

**第一步：服务注册（Register）**

服务提供方（如用户服务）启动时，把自己的地址（IP:Port）注册到注册中心。

```java
// 伪代码：服务启动时注册
registryClient.register("user-service", "192.168.1.100:8080");
```

**第二步：服务发现（Discover）**

服务消费方（如订单服务）启动时，去注册中心拉取目标服务的地址列表。

```java
// 伪代码：消费方拉取地址列表
List<String> addresses = registryClient.discover("user-service");
// ["192.168.1.100:8080", "192.168.1.101:8080", "192.168.1.102:8080"]
```

**第三步：变更通知（Watch/Subscribe）**

消费方订阅目标服务的变更事件。当提供方有实例上线或下线时，注册中心主动推送变更给所有消费方。

```java
// 伪代码：订阅变更通知
registryClient.subscribe("user-service", (newAddresses) -> {
    // 收到通知后更新本地缓存的地址列表
    localCache.update("user-service", newAddresses);
});
```

### 4. 健康检查：怎么知道服务还活着

服务注册上去后，怎么知道它还在正常运行？万一进程挂了但没来得及注销呢？

#### 方式一：心跳检测（Heartbeat）

服务提供方定期向注册中心发送心跳（比如每 5 秒一次）。注册中心如果连续 N 次没收到心跳（比如 15 秒），就认为服务不可用，从注册表中移除。

```
服务实例 ──心跳──> 注册中心
         每5秒

如果 15 秒没收到心跳 → 判定下线 → 通知消费方
```

**优点**：实现简单，客户端主动上报。
**缺点**：不够准确。进程还在但已经无法处理请求（比如线程池满了、数据库连接断了），心跳可能还在正常发，注册中心以为它是健康的，但请求打过去就超时。

#### 方式二：临时节点 + Session（ZooKeeper 方式）

ZooKeeper 用**临时节点**（Ephemeral Node）实现健康检查。服务注册时创建一个临时节点，这个节点和客户端的 Session 绑定。如果客户端断开连接（进程挂了），Session 超时后临时节点自动删除。

```
服务实例 ──创建临时节点──> ZooKeeper
         /services/user-service/192.168.1.100

进程挂了 → TCP 连接断开 → Session 超时 → 临时节点自动删除
       → ZooKeeper 通知所有 Watcher → 消费方更新地址列表
```

**优点**：不需要额外的心跳逻辑，节点生命周期和连接绑定。
**缺点**：Session 超时时间通常较长（默认 30 秒），感知不够及时。网络抖动可能导致误删。

#### 服务端探测

注册中心主动调用服务的健康检查接口（如 `/health`），检查服务是否真正可用。

```
注册中心 ──GET /health──> 服务实例
         每 10 秒

返回 200 → 健康
返回 503 或超时 → 不健康 → 移除
```

**优点**：最可靠，能检测到"进程在但不可用"的情况。
**缺点**：成本高，注册中心需要主动发请求给每个实例。当实例数量很大时，注册中心的压力不小。

### 5. 主流注册中心对比

#### ZooKeeper

- **一致性模型**：CP（强一致性）
- **共识协议**：ZAB（ZooKeeper Atomic Broadcast）
- **工作方式**：Leader + Follower 模式，所有写请求由 Leader 处理
- **核心问题**：选举期间整个集群不可用（通常几秒到几十秒）。对于注册中心这种读多写少的场景，为了强一致性牺牲可用性有些过度
- **适用场景**：分布式锁、配置管理、早期 Dubbo 注册中心

#### Eureka

- **一致性模型**：AP（最终一致性）
- **架构**：对等节点（Peer-to-Peer），没有 Leader
- **自我保护机制**：如果短时间内大量服务心跳失败（可能是网络问题而不是服务真挂了），Eureka 不会立即移除这些服务，而是进入保护模式，保留注册信息
- **核心问题**：已停止维护（Eureka 2.x 闭源），Spring Cloud 官方推荐迁移到其他方案
- **适用场景**：Spring Cloud 早期生态

#### Nacos

- **一致性模型**：AP/CP 可切换。临时实例用 AP（Distro 协议），持久实例用 CP（Raft 协议）
- **附加能力**：集成了配置中心（类似 Apollo），一套系统解决两个问题
- **生态**：阿里出品，Dubbo 3.x 和 Spring Cloud Alibaba 默认注册中心
- **适用场景**：国内 Java 微服务，特别是阿里系技术栈

#### Consul（北美主流）

- **一致性模型**：CP（Raft 协议）
- **特色**：内置健康检查（支持 HTTP、TCP、Script 多种方式）、K/V 存储、多数据中心支持
- **跨语言**：提供 HTTP API，不依赖特定语言 SDK
- **适用场景**：北美公司、Go 语言生态、多语言微服务

#### etcd（K8s 底层）

- **一致性模型**：CP（Raft 协议）
- **定位**：分布式 K/V 存储，Kubernetes 的底层数据存储
- **特色**：Watch 机制高效，API 简洁
- **适用场景**：K8s 生态、Go 语言服务发现、需要强一致 K/V 的场景

#### 对比表

| 维度 | ZooKeeper | Eureka | Nacos | Consul | etcd |
| --- | --- | --- | --- | --- | --- |
| **一致性** | CP | AP | AP/CP | CP | CP |
| **协议** | ZAB | 对等复制 | Distro/Raft | Raft | Raft |
| **健康检查** | Session | 心跳 | 心跳+探测 | HTTP/TCP/Script | 需自建 |
| **配置管理** | 可以 | 不支持 | 内置 | K/V 存储 | K/V 存储 |
| **跨语言** | 一般 | Java 为主 | Java 为主 | 好（HTTP API） | 好（gRPC API） |
| **社区** | Apache | 停维 | 阿里 | HashiCorp | CNCF |
| **主要市场** | 国内老项目 | 遗留系统 | 国内新项目 | 北美 | K8s 生态 |

### 6. 注册中心应该选 CP 还是 AP？

这个问题面试经常问，答案是：**大多数场景下 AP 更合适**。

原因：注册中心的核心诉求是"消费方能拿到可用的服务地址"。

- **如果选 CP**：在 Leader 选举期间，注册中心不可用。消费方无法注册也无法发现服务。但如果消费方有本地缓存（见下一节），短暂的不可用其实问题不大。
- **如果选 AP**：注册中心始终可用，但可能短暂返回不完全一致的地址列表（比如某个实例刚下线但还没同步到所有节点）。这种不一致影响不大——最多某次请求打到了一个刚下线的节点，调用失败后重试到其他节点就行了。

> 注册中心提供的是地址列表，不是金融交易数据。地址列表短暂不一致的代价远小于注册中心不可用的代价。

### 7. 客户端缓存：注册中心挂了也能继续调用

成熟的 RPC 框架都会在消费方本地缓存服务地址列表。这样即使注册中心全部挂掉，消费方仍然可以用缓存的地址继续调用。

```
正常情况：
  消费方 ←── 从注册中心拉取 ←── 注册中心
  消费方 ──> 用地址列表调用 ──> 服务提供方

注册中心挂了：
  消费方 ←── 从本地缓存读取    ✓ 还能用
  消费方 ──> 用缓存的地址调用 ──> 服务提供方   ✓ 正常工作

注意：
  新服务上线 → 注册中心挂了 → 消费方感知不到新实例  ✗
  旧服务下线 → 注册中心挂了 → 消费方还往旧地址打  ✗（但会快速失败+重试）
```

Dubbo 还会把地址列表持久化到本地文件。即使消费方也重启了，也能从本地文件恢复地址列表。

### 8. K8s 做服务发现详解

在云原生时代，Kubernetes 本身就内置了服务发现能力，很多团队不再需要独立的注册中心。

#### 核心概念

```
┌─────────────────────────────────────────────────────┐
│                   Kubernetes 集群                     │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │
│  │ Service: user-service                        │    │
│  │ ClusterIP: 10.96.0.100                       │    │
│  │ Port: 8080                                   │    │
│  │ Selector: app=user-service                   │    │
│  └──────────────┬──────────────────────────────┘    │
│                 │ label selector 自动匹配            │
│    ┌────────────┼────────────────────┐               │
│    ▼            ▼                    ▼               │
│  ┌──────┐   ┌──────┐           ┌──────┐            │
│  │ Pod  │   │ Pod  │           │ Pod  │            │
│  │ 实例1 │   │ 实例2 │           │ 实例3 │            │
│  └──────┘   └──────┘           └──────┘            │
│  Endpoints: [10.244.1.5, 10.244.2.8, 10.244.3.12]  │
└─────────────────────────────────────────────────────┘
```

- **Pod**：最小部署单元，包含一个或多个容器，每个 Pod 有独立 IP
- **Service**：逻辑上的服务抽象，通过 label selector 自动关联一组 Pod
- **Endpoints**：Service 对应的 Pod IP 列表，K8s 自动维护
- **EndpointSlice**：新版 K8s 更常用的端点对象，作用和 Endpoints 类似，但更适合大规模集群

#### 自动注册

在 K8s 中不需要手动注册。你只要定义 Service 的 label selector，K8s 会自动找到匹配的 Pod 并维护 Endpoints 列表。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service    # 自动匹配所有 label 包含 app=user-service 的 Pod
  ports:
    - port: 8080
```

Pod 创建、销毁时，K8s 自动更新 EndpointSlice/Endpoints。这就是自动注册和注销。

#### 从 Pod 启动到“注册完成”的真实链路

很多人说“K8s 会自动注册服务”，但这里的“注册”并不是应用自己去调用一个 `register()` 接口，而是 **K8s 控制面根据声明式配置自动维护服务实例列表**。

一条更准确的链路是：

1. 你创建 `Deployment`，里面的 Pod 带有 `app=user-service` 这样的标签
2. Pod 被调度到某个 Node 上启动，并拿到自己的 Pod IP
3. `Service` 的 selector 发现这个 Pod 标签匹配自己
4. EndpointSlice Controller 继续判断这个 Pod 是否 `Ready`
5. 如果 Pod 已就绪，就把 `Pod IP:Port` 写入对应的 `EndpointSlice/Endpoints`
6. CoreDNS 和 kube-proxy 监听这些变化，更新 DNS 结果和转发规则
7. 后续如果 Pod 删除、重建或 readiness 失败，端点列表会自动更新

```text
Deployment 创建 Pod
        ↓
Pod 获得 IP + labels
        ↓
Service selector 匹配 Pod
        ↓
EndpointSlice Controller 检查 readiness
        ↓
把 Pod IP:Port 加入 EndpointSlice
        ↓
CoreDNS / kube-proxy 感知更新
        ↓
调用方通过服务名访问到最新可用实例
```

所以从面试表述上，你可以把它概括成：

**K8s 里的服务注册不是“实例主动上报地址”，而是“平台通过标签选择器和就绪状态，自动把 Pod 加入或移出可用实例列表”。**

#### DNS 发现（CoreDNS）

K8s 集群内置 CoreDNS。每个 Service 会自动获得一个 DNS 域名：

```
user-service.default.svc.cluster.local
─────────── ─────── ─── ─────────────
  服务名      命名空间 固定  集群域名
```

其他服务只需要用服务名就能访问：

```java
// 在 K8s 中，直接用服务名访问，DNS 会解析到 ClusterIP
String url = "http://user-service:8080/api/user/123";
```

#### 调用方到底“发现”了什么

这里有一个很容易被忽略的点：

在 K8s 里，调用方不一定总是像传统注册中心那样直接拿到“所有实例 IP 列表”。

常见有两种模式：

**模式一：普通 Service**

- DNS 解析结果是 `ClusterIP`
- 调用方只看到一个稳定虚拟地址
- 真实 Pod 选择由 kube-proxy / IPVS 完成

这更像：

- 调用方只知道一个统一入口
- 平台负责把流量分发到后端实例

**模式二：Headless Service**

- Service 配置 `clusterIP: None`
- DNS 不再返回一个虚拟 IP，而是直接返回多个 Pod IP
- 调用方可以自己做客户端负载均衡

这更接近传统 RPC 里的“从注册中心拿实例列表，再由客户端选一个节点调用”。

#### Headless Service：更像传统 RPC 的服务发现

如果你的 RPC 框架希望自己掌握实例列表，而不是只打一个 `ClusterIP`，那么 Headless Service 很有价值。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service-headless
spec:
  clusterIP: None
  selector:
    app: user-service
  ports:
    - port: 8080
```

此时 DNS 查询结果更接近：

```text
user-service-headless.default.svc.cluster.local
-> 10.244.1.5
-> 10.244.2.8
-> 10.244.3.12
```

这样调用方就可以：

- 自己感知实例列表
- 自己做轮询、权重、最少连接等策略
- 因为没有 `ClusterIP`，很多时候会直接连到具体 Pod
- 更方便配合 gRPC、StatefulSet 或自定义 RPC 客户端

所以你可以把两种方式记成：

- 普通 Service：发现的是稳定入口
- Headless Service：发现的是真实实例列表

#### kube-proxy 流量转发

当请求发到 Service 的 ClusterIP 时，kube-proxy 负责把请求转发到实际的 Pod。

两种实现方式：
- **iptables 模式**：用 iptables 规则做随机负载均衡。简单但规则多时性能下降。
- **IPVS 模式**：用 Linux 内核的 IPVS 模块。性能更好，支持更多负载均衡算法（轮询、加权、最少连接等）。大规模集群推荐 IPVS。

这一步很关键，因为它说明：

- 在普通 Service 模式下，DNS 只负责把名字解析成 `ClusterIP`
- 真正把请求送到某个 Pod，是 kube-proxy / IPVS 在做
- 所以 K8s 把“发现”和“转发”两件事一起打包成了平台能力

#### 健康检查探针

K8s 提供两种探针来判断 Pod 是否健康：

```yaml
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: user-service
      image: user-service:1.0
      livenessProbe:          # 存活探针
        httpGet:
          path: /healthz
          port: 8080
        initialDelaySeconds: 15
        periodSeconds: 10
      readinessProbe:         # 就绪探针
        httpGet:
          path: /ready
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 5
```

- **livenessProbe（存活探针）**：检测容器是否还活着。如果失败，K8s 会重启容器。用于发现死锁、假死等情况。
- **readinessProbe（就绪探针）**：检测容器是否可以接收流量。如果失败，K8s 把 Pod 从 EndpointSlice/Endpoints 中移除，不再给它分发流量。用于启动预热、依赖检查等。

区别：liveness 失败重启容器，readiness 失败只是不给流量。

#### readiness 为什么是 K8s 服务发现里最关键的一环

如果只靠“Pod 进程还活着”来做注册，问题会很大。

因为很多时候服务虽然没死，但并不适合接流量，比如：

- 应用刚启动，还在预热缓存
- 数据库连接池还没建立完成
- 依赖下游全部超时
- 服务正在优雅下线

这时如果 Pod 仍然出现在可用地址列表中，请求就会打到一个“活着但不可用”的实例上。

所以从服务发现角度看，`readinessProbe` 本质上是在回答：

**这个实例此刻要不要出现在可用实例列表里。**

这也是为什么很多人会说：K8s 的服务发现不只是“有 Pod 就注册”，而是“只有 Ready 的 Pod 才参与流量接收”。

#### K8s vs 传统注册中心对比

| 维度 | 传统注册中心（Nacos/Consul） | K8s Service |
| --- | --- | --- |
| **注册方式** | 服务主动注册 | 自动通过 label selector |
| **发现方式** | SDK 拉取 + Watch | DNS + ClusterIP / Headless DNS |
| **健康检查** | 心跳/探测 | liveness + readiness 探针，readiness 直接影响端点列表 |
| **负载均衡** | 客户端侧，策略丰富 | kube-proxy（四层），策略有限 |
| **灵活性** | 高，各种路由规则 | 中，需 Service Mesh 补充 |
| **运维成本** | 需要独立部署维护 | 内置，无额外成本 |
| **跨集群** | 支持（多数据中心） | 需额外方案（Federation/Mesh） |

#### K8s 服务发现的局限

K8s Service 做的是**四层负载均衡**（TCP/UDP 层），它不理解应用层协议（HTTP 头部、gRPC 方法名等）。如果你需要七层的能力（按 HTTP 路径路由、按请求头灰度、流量镜像等），就需要 Service Mesh（如 Istio/Linkerd）来补充。

另外还要注意：

- 普通 Service 默认更像“稳定入口 + 平台转发”，不等于传统 RPC 那种完整实例列表发现
- 如果你需要客户端强感知实例列表，通常要结合 Headless Service
- 如果你需要更复杂的路由、熔断、灰度和观测能力，K8s 原生 Service 不够，还要再叠 Mesh 或网关能力

### 9. CAP 理论

CAP 理论是注册中心选型的理论基础，也是面试高频考点。

#### C/A/P 含义

- **C（Consistency，一致性）**：所有节点在同一时刻看到的数据完全一致。你在节点 A 写入的数据，立刻能在节点 B 读到。
- **A（Availability，可用性）**：每个请求都能收到响应（不是超时或错误），虽然不保证是最新的数据。
- **P（Partition Tolerance，分区容忍性）**：当网络分区发生时（节点之间通信中断），系统仍然能继续工作。

#### P 为什么必须有

很多人把 CAP 理解成"三选二"，以为可以选 CA 放弃 P。这是错的。

**网络分区是客观现实，不是一个你可以选择要或不要的选项。**

用一个比喻来理解：你们公司在北京和上海各有一个办公室，两个办公室之间靠专线通信。有一天专线断了（网络分区），北京和上海的同事暂时联系不上了。

这时你有两个选择：

- **选 C（一致性）**：两边都停止工作，等专线恢复后再同步数据，确保数据一致。代价是在专线恢复之前，两边都不可用。
- **选 A（可用性）**：两边各自继续工作，各自接单、各自处理。代价是专线恢复后两边的数据可能不一致（比如北京给客户 A 打了折，上海没打折）。

你不能选"不要 P"，因为专线断不断不是你能控制的。你只能选在专线断了的时候，是要 C 还是要 A。

#### 分区时 C 和 A 的矛盾

```
正常情况（无分区）：
  节点A ←同步→ 节点B    C 和 A 都满足 ✓

网络分区发生：
  节点A    ✗    节点B    连接断开

  选 C：
    节点A 收到写请求 → 无法同步到 B → 拒绝写入（返回错误）
    → 一致性保证了 ✓，但可用性没了 ✗

  选 A：
    节点A 收到写请求 → 无法同步到 B → 先写本地，之后再同步
    → 可用性保证了 ✓，但两边数据可能不一致 ✗
```

#### 更准确的表述

> 网络正常时，C 和 A 都可以满足。网络分区发生时，必须在 C 和 A 之间二选一。

所以 CAP 不是"三选二"，更准确地说是：**P 一定存在，在 P 发生时要在 C 和 A 之间做取舍**。

### 10. 完整服务调用流程（串联前四课）

到这里我们可以把前四课的内容串成一个完整的流程了：

```
┌──────────────────────────────────────────────────────────────┐
│                        完整 RPC 调用流程                       │
│                                                              │
│  ① 服务提供方启动，注册到注册中心                                │
│     user-service → 注册中心: "我在 192.168.1.100:8080"         │
│                                                              │
│  ② 服务消费方启动，从注册中心拉取地址列表 + 订阅变更              │
│     order-service ← 注册中心: [100, 101, 102]                 │
│                                                              │
│  ③ 消费方发起调用                                              │
│     业务代码: userService.getUserInfo(userId)                  │
│                                                              │
│  ④ 动态代理拦截调用                                            │
│     封装：服务名 + 方法名 + 参数 → RpcRequest                  │
│                                                              │
│  ⑤ 负载均衡选择一个地址（下一课详讲）                            │
│     从 [100, 101, 102] 中选一个，比如 101                      │
│                                                              │
│  ⑥ 序列化                                                    │
│     RpcRequest → Protobuf/Hessian → 字节流                   │
│                                                              │
│  ⑦ 网络传输                                                   │
│     协议帧（魔数+长度+消息体）通过 TCP 发到 101                  │
│                                                              │
│  ⑧ 服务端反序列化 + 调度 + 反射调用                              │
│     字节流 → RpcRequest → 查注册表 → 反射调用 → 拿到结果        │
│                                                              │
│  ⑨ 结果序列化 + 返回                                           │
│     结果 → 字节流 → TCP → 消费方反序列化 → 返回给业务代码        │
│                                                              │
│  ⑩ 如果调用失败                                                │
│     容错策略处理：重试其他节点 / 快速失败 / 降级（第 5 课详讲）    │
└──────────────────────────────────────────────────────────────┘
```

## 小结（3-5 条关键点）

- 注册中心解决的是服务地址的动态管理问题，核心三步流程：服务注册 → 服务发现 → 变更通知。
- 健康检查有三种方式：心跳（简单但不够准）、临时节点（ZooKeeper 方式，自动清理但感知慢）、服务端探测（最准但成本高）。
- 注册中心选型大多数场景下 AP 优于 CP，因为地址列表短暂不一致的代价远小于注册中心不可用的代价。客户端缓存可以进一步降低注册中心故障的影响。
- K8s 通过 Service、EndpointSlice、CoreDNS、kube-proxy 内置了服务发现能力，本质上是平台自动维护实例列表并提供稳定访问入口。
- 普通 Service 发现的是 ClusterIP，Headless Service 发现的是真实 Pod 列表；后者更接近传统 RPC 注册中心模式。
- K8s 原生 Service 只做四层负载均衡，需要 Service Mesh 补充七层路由、灰度和流量治理能力。
- CAP 中 P 是客观现实不是选项，更准确的理解是：网络正常时 CA 都满足，分区发生时在 C 和 A 之间二选一。
