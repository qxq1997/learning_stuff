## 内容讲解

### 1. 客户端 vs 服务端负载均衡

负载均衡的目标是把请求分摊到多个服务实例上，避免单个实例过载。根据负载均衡逻辑放在哪里，分为两种模式：

```
方式一：服务端负载均衡（以 Nginx 为例）

  客户端 ──────> Nginx（负载均衡器） ──────> 服务实例 1
                      │                      服务实例 2
                      └─────────────────────> 服务实例 3

  特点：所有流量经过中间节点
  优点：集中控制，客户端不用关心后端有几台机器
  缺点：中间节点是瓶颈和单点（虽然可以主备，但多了一跳延迟）


方式二：客户端负载均衡（Dubbo / gRPC 的做法）

                    ┌──────────────> 服务实例 1
  客户端（内置LB）───┼──────────────> 服务实例 2
                    └──────────────> 服务实例 3

  特点：客户端从注册中心拿到实例列表，自己决定调谁
  优点：无中间节点，性能好，不存在中间层瓶颈
  缺点：策略分散在每个客户端，升级需要所有客户端配合
```

**选型结论：** 微服务内部调用用客户端负载均衡（Dubbo/gRPC 内置），对外暴露入口（面向浏览器/APP）用服务端负载均衡（Nginx/网关）。

### 2. 负载均衡策略

#### 2.1 五种常见策略

```
┌──────────────┬──────────────────────────────────────────────────┐
│    策略       │              说明                                │
├──────────────┼──────────────────────────────────────────────────┤
│  随机         │ 随机选一个实例。简单，大样本下近似均匀              │
│  轮询         │ 按顺序依次分配。绝对均匀，但不考虑机器性能差异      │
│  加权轮询/随机 │ 性能好的机器权重大，分配更多请求                   │
│  最少活跃数    │ 选当前正在处理请求最少的实例。自适应，但需维护计数   │
│  一致性哈希    │ 相同参数的请求总是打到同一台机器。适合有本地缓存的场景│
└──────────────┴──────────────────────────────────────────────────┘
```

- **Dubbo 默认**：加权随机（RandomLoadBalance）
- **gRPC 默认**：pick_first（选第一个可用的，适合单连接场景）

#### 2.2 一致性哈希深入（高频考点）

**普通哈希的问题：**

假设有 3 台服务器，用 `hash(请求) % 3` 来分配。当新增一台变成 4 台时，`hash(请求) % 4` 的结果几乎完全不同——大部分请求的映射都变了。如果服务器上有本地缓存，这意味着缓存几乎全部失效。

**一致性哈希的解决思路：**

把哈希值空间组织成一个环（0 到 2^32-1），服务器和请求都映射到这个环上：

```
                        0
                        │
                   ┌────┴────┐
                  /           \
                /               \
              /     Node A        \
            /      (hash=1000)     \
           │                        │
           │                        │
  2^32-1 ──│                        │── 2^32 * 1/4
           │                        │
           │        Node C          │
            \     (hash=50000)     /
              \                  / Node B
                \              / (hash=30000)
                  \          /
                   └────┬───┘
                        │
                     2^32 / 2

  请求映射规则：hash(请求key) 落在环上某个位置，
  顺时针找到的第一个节点就是目标服务器。

  增减节点时，只影响相邻节点之间的请求映射，
  不影响其他所有节点。
```

**顺时针查找过程：**
1. 把每个服务器节点 hash 到环上一个位置
2. 请求的 key 也 hash 到环上一个位置
3. 从这个位置顺时针走，遇到的第一个服务器节点就是目标

**增减节点的影响：** 只有新节点和它逆时针方向上一个节点之间的请求会重新映射，其他请求不受影响。比起普通哈希"几乎全部失效"，一致性哈希只需迁移 1/N 的数据。

**虚拟节点解决数据倾斜：**

如果节点少（比如 3 个），它们在环上的分布可能不均匀，导致某个节点承担了大部分请求（数据倾斜）。

解决方案：每个物理节点映射多个虚拟节点到环上。比如 Node A 映射 NodeA-1、NodeA-2、...、NodeA-150，每个虚拟节点 hash 到环上不同位置。虚拟节点越多，分布越均匀。

```
没有虚拟节点：3 个点分布不均
──A─────────────────────B──────C──
  （A 承担了 70% 的请求，严重不均）

有虚拟节点（每个物理节点 150 个虚拟节点）：
──A2─B1─C3─A5─B2─C1─A1─B3─C2─A3─B5─...─
  （450 个点均匀分布，每个物理节点约承担 33%）
```

### 3. 容错策略（必考）

RPC 调用失败是常态（网络抖动、服务重启、超时等），容错策略决定失败后怎么办。

#### 3.1 六种容错策略

**Failover 失败切换（默认）：**
调用失败后自动切换到另一台实例重试，默认重试 2 次（共调用 3 次）。适合**读操作**——读是幂等的，多读几次没关系。

```java
// Dubbo 配置示例
@DubboReference(cluster = "failover", retries = 2)
private UserService userService;
```

**Failfast 快速失败：**
调用失败立刻报错，不重试。适合**写操作**——比如扣款，重试可能导致扣了两次。

```java
@DubboReference(cluster = "failfast")
private PaymentService paymentService;
```

**Failsafe 安全失败：**
调用失败直接忽略，不报错也不重试。适合**非关键操作**——比如写审计日志，失败了不影响主流程。

**Failback 失败恢复：**
调用失败后记录下来，后台定时重试。适合**对实时性要求不高但最终必须成功**的操作——比如消息通知。

**Forking 并行调用：**
同时调用多个实例，任意一个成功就返回。最可靠但最浪费资源，适合**极端高可用场景**——比如实时行情查询。

**Broadcast 广播调用：**
调用所有实例，任一失败则失败。适合**通知所有节点更新缓存**等场景。

```
┌────────────┬───────────────────────┬───────────────────────┐
│   策略      │     行为              │     适用场景           │
├────────────┼───────────────────────┼───────────────────────┤
│ Failover   │ 失败切换重试（2次）    │ 读操作                │
│ Failfast   │ 失败立刻报错          │ 写操作（防重复）       │
│ Failsafe   │ 失败忽略              │ 非关键操作（日志）     │
│ Failback   │ 失败后台重试          │ 最终一致（通知）       │
│ Forking    │ 并行调用取最快        │ 极高可用（行情）       │
│ Broadcast  │ 广播所有节点          │ 缓存更新通知          │
└────────────┴───────────────────────┴───────────────────────┘
```

### 4. 重试的坑

#### 4.1 重试风暴（Retry Storm）

```
         A ──────> B ──────> C
       retries=3  retries=3  retries=3

如果 C 故障：
  B 调 C 失败，重试 3 次 → B 对 C 发了 3 次请求
  A 调 B 失败（因为 B 也超时了），重试 3 次
  每次 A 重试，B 又对 C 重试 3 次

总请求数 = 3 × 3 = 9 次（指数爆炸）

如果链路更长：A → B → C → D
总请求数 = 3 × 3 × 3 = 27 次
```

**防止重试风暴的方法：**
- 只在最外层重试，内层用 Failfast
- 限制重试总次数而不是每层重试次数
- 加退避策略（第一次重试等 100ms，第二次等 200ms）

#### 4.2 非幂等操作不能重试

幂等：同一个操作执行一次和执行多次，效果一样。

```
幂等操作（可重试）：
  - 查询用户信息：查 10 次结果一样
  - 设置余额为 100：设 10 次还是 100

非幂等操作（不可重试）：
  - 扣款 50 元：重试一次变成扣了 100 元
  - 创建订单：重试一次变成创建了两个订单
```

#### 4.3 幂等接口设计：唯一请求 ID + Redis 去重

```java
// 客户端生成唯一请求 ID
String requestId = UUID.randomUUID().toString();

// 服务端去重逻辑
public Result deductBalance(String requestId, BigDecimal amount) {
    // 用 Redis SETNX 实现去重
    Boolean isFirst = redis.opsForValue()
        .setIfAbsent("deduct:" + requestId, "1", 24, TimeUnit.HOURS);

    if (!isFirst) {
        // 已经处理过，直接返回上次结果
        return getStoredResult(requestId);
    }

    // 第一次处理，执行扣款
    Result result = doDeduct(amount);
    storeResult(requestId, result);
    return result;
}
```

### 5. 熔断 Circuit Breaker（防雪崩）

当某个下游服务频繁超时或报错时，继续调用只会浪费资源、拖慢自己。熔断器的作用是**主动断开**对故障服务的调用，快速失败，保护调用方。

**三种状态转换：**

```
                  失败率 < 阈值
              ┌─────────────────┐
              │                 │
              ▼                 │
         ┌─────────┐     ┌─────────┐
         │  关闭    │────>│  打开    │
         │ CLOSED  │     │  OPEN   │
         └─────────┘     └────┬────┘
              ▲               │
              │          超时窗口到
              │               │
              │          ┌────▼────┐
              └──────────│  半开    │
              探测成功    │HALF-OPEN│
                         └────┬────┘
                              │
                         探测失败
                              │
                         ┌────▼────┐
                         │  打开    │
                         │  OPEN   │
                         └─────────┘
```

**三种状态详解：**

- **关闭（CLOSED）**：正常状态，请求正常发给下游。同时统计最近 N 次调用的失败率
- **打开（OPEN）**：失败率超过阈值（比如 50%），熔断器打开。所有请求**直接拒绝**，不再调下游。设置一个超时窗口（比如 30 秒）
- **半开（HALF-OPEN）**：超时窗口结束后，放一个探测请求去试下游。如果成功，转回关闭状态；如果失败，回到打开状态继续等

**熔断 vs 限流的区别：**

```
┌──────────┬─────────────────────────────────────────┐
│          │                说明                      │
├──────────┼─────────────────────────────────────────┤
│  限流     │ 控制"进来"的速度，保护自己不被压垮        │
│          │ 我承受不了这么多请求，多余的直接拒绝       │
├──────────┼─────────────────────────────────────────┤
│  熔断     │ 控制"出去"的方向，保护调用方不被拖死      │
│          │ 下游有问题了，我不再调它，快速失败         │
└──────────┴─────────────────────────────────────────┘
```

一句话：**限流是"别来了"，熔断是"我不去了"。**

### 6. 降级

降级是熔断后的**兜底方案**。下游服务挂了，不能给用户返回一个错误页面，而是返回一个"还能接受"的备选结果。

**例子：推荐服务挂了**

```java
@HystrixCommand(
    fallbackMethod = "getDefaultRecommendations"
)
public List<Product> getRecommendations(Long userId) {
    // 正常调用推荐服务
    return recommendService.getPersonalized(userId);
}

// 降级方法：返回热门商品列表
public List<Product> getDefaultRecommendations(Long userId) {
    // 推荐服务挂了，返回预先缓存的热门商品
    return hotProductCache.getTop20();
}
```

降级策略的层次：
1. **返回缓存数据**：推荐服务挂了，返回上一次成功的推荐结果
2. **返回默认值**：推荐服务挂了，返回热门商品列表
3. **功能裁剪**：直接隐藏推荐模块，只展示搜索和分类

核心思想是：**有损服务优于无服务**。宁可给用户看一个不那么精准的推荐，也不能让整个页面打不开。

## 小结（3-5 条关键点）

- 微服务内部用客户端负载均衡（无中间节点，性能好），对外入口用服务端负载均衡（Nginx/网关集中控制）。
- 一致性哈希用环形空间+顺时针查找解决增减节点时大量缓存失效的问题，虚拟节点解决节点少时数据倾斜的问题。面试必考。
- 容错策略核心区分：Failover 适合读（可重试），Failfast 适合写（不可重试），Failsafe 适合非关键操作。非幂等操作绝不能重试，用唯一请求 ID + Redis 去重保证幂等。
- 重试风暴是分布式系统中容易被忽视的陷阱：链路每层都重试，总请求数指数爆炸。解决办法：只在最外层重试，内层 Failfast。
- 熔断保护调用方（控制出去方向），限流保护自己（控制进来速度）。熔断后用降级返回兜底结果，有损服务优于无服务。
