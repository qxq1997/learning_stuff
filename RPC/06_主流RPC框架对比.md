# RPC - 第 6 课：主流 RPC 框架对比

## 学习目标（本节结束后你能做到什么）

- 掌握 gRPC、Dubbo、Thrift 三大框架的核心特点和架构差异
- 说清 gRPC 的四种通信模式以及为什么选择 HTTP/2
- 画出 Dubbo 的 Provider-Registry-Consumer 架构图
- 从七个维度对比三大框架，给出清晰的选型建议
- 回答"为什么 gRPC 用 HTTP/2"这个高频追问

## 内容讲解

### 1. gRPC（Google）

gRPC 是 Google 开源的高性能 RPC 框架，核心特点是 **HTTP/2 传输 + Protobuf 序列化**。

**核心架构：**

```
┌──────────────────┐            ┌──────────────────┐
│    gRPC Client   │            │    gRPC Server   │
│                  │            │                  │
│  ┌────────────┐  │   HTTP/2   │  ┌────────────┐  │
│  │ Proto Stub │──│───────────>│──│ Service Impl│  │
│  └────────────┘  │  Protobuf  │  └────────────┘  │
│                  │  二进制帧   │                  │
└──────────────────┘            └──────────────────┘
```

开发流程：先写 `.proto` 文件定义服务接口和消息格式，然后用 `protoc` 编译器生成各语言的客户端 Stub 和服务端骨架代码。

```protobuf
// user.proto
syntax = "proto3";

service UserService {
    rpc GetUser (GetUserRequest) returns (UserResponse);           // Unary
    rpc ListUsers (ListUsersRequest) returns (stream UserResponse); // Server Streaming
    rpc UploadUsers (stream UserRequest) returns (UploadResult);    // Client Streaming
    rpc Chat (stream Message) returns (stream Message);            // Bidirectional
}

message GetUserRequest {
    int64 user_id = 1;
}

message UserResponse {
    int64 user_id = 1;
    string name = 2;
    string email = 3;
}
```

**四种通信模式：**

```
┌────────────────────┬─────────────────────────────────────────────────┐
│      模式           │              说明                               │
├────────────────────┼─────────────────────────────────────────────────┤
│ Unary（一元）       │ 客户端发一个请求，服务端返回一个响应               │
│                    │ 最常见，类似普通 HTTP 请求                        │
├────────────────────┼─────────────────────────────────────────────────┤
│ Server Streaming   │ 客户端发一个请求，服务端返回一个流（多条消息）      │
│ （服务端流）        │ 场景：分页查询、实时推送股票行情                   │
├────────────────────┼─────────────────────────────────────────────────┤
│ Client Streaming   │ 客户端发一个流（多条消息），服务端返回一个响应      │
│ （客户端流）        │ 场景：批量上传日志、文件分片上传                   │
├────────────────────┼─────────────────────────────────────────────────┤
│ Bidirectional      │ 双方同时发送流，互不等待                          │
│ （双向流）          │ 场景：聊天、实时协作编辑                          │
└────────────────────┴─────────────────────────────────────────────────┘
```

**gRPC 优缺点：**

优点：
- **跨语言能力强**：官方支持 10+ 语言（Go、Java、Python、C++、Rust 等），.proto 生成各语言代码
- **流式通信**：四种模式覆盖各种场景，HTTP RESTful 做不到双向流
- **性能优秀**：HTTP/2 多路复用 + Protobuf 二进制序列化，比 JSON over HTTP/1.1 快很多
- **强类型**：.proto 定义了严格的接口契约，编译期就能发现类型错误

缺点：
- **需要 .proto 文件**：新增接口需要更新 .proto 并重新编译，多一步流程
- **浏览器不能直接调**：浏览器不支持 HTTP/2 的原始帧格式，需要 gRPC-Web 或 gRPC-Gateway 做转换
- **调试不方便**：二进制协议，不像 JSON 那样用 curl 就能测试，需要 grpcurl 等工具

### 2. Dubbo（阿里）

Dubbo 不只是一个 RPC 通信框架，而是一个**完整的微服务治理方案**。它内置了服务注册发现、负载均衡、容错、链路追踪、流量管理等能力。

**Provider-Registry-Consumer 架构：**

```
                    ┌─────────────────────┐
                    │     Registry        │
                    │    (注册中心)        │
                    │   ZK / Nacos        │
                    └──┬──────────┬───────┘
                       │          │
              ①注册    │          │ ②订阅
              (启动时)  │          │ (启动时)
                       │          │
                       ▼          ▼
┌─────────────────┐              ┌─────────────────┐
│    Provider      │   ③通知      │    Consumer      │
│   (服务提供方)    │ <─ ─ ─ ─ ─ │   (服务消费方)    │
│                  │ (地址变更时)  │                  │
│  实现接口逻辑     │              │  持有接口代理     │
│                  │ ④调用(直连)   │                  │
│                  │<────────────│                  │
└────────┬─────────┘              └────────┬────────┘
         │                                 │
         │          ⑤监控                   │
         └──────────┬──────────────────────┘
                    ▼
            ┌───────────────┐
            │    Monitor    │
            │  (监控中心)    │
            └───────────────┘
```

**五步流程：**
1. **注册（Register）**：Provider 启动时把地址注册到 Registry
2. **订阅（Subscribe）**：Consumer 启动时从 Registry 订阅 Provider 的地址列表
3. **通知（Notify）**：Provider 地址变化时，Registry 推送给 Consumer
4. **调用（Invoke）**：Consumer 直接调用 Provider（不经过 Registry，客户端负载均衡）
5. **监控（Monitor）**：调用数据异步上报到 Monitor

**默认协议栈：** Dubbo 协议（自定义二进制协议） + Hessian2 序列化。Dubbo 3.0 也支持 Triple 协议（兼容 gRPC）。

**Dubbo 优缺点：**

优点：
- **功能全面**：服务注册发现、负载均衡、容错、限流、链路追踪、灰度发布全都内置
- **Spring 集成好**：用注解就能暴露和引用服务，Java 开发者学习成本低
- **生态成熟**：国内大厂广泛使用，阿里、美团、京东等大规模验证

缺点：
- **主要是 Java**：虽然有 Go 版本（Dubbo-Go），但生态和成熟度远不如 Java 版
- **框架较重**：引入了很多依赖，对于简单的 RPC 需求有点杀鸡用牛刀
- **国际化不足**：主要在国内市场，北美用得少

### 3. Thrift（Facebook）

Thrift 由 Facebook 开发，后捐给 Apache。核心设计思想是**可插拔**——传输层、协议层、序列化方式都可以自由组合。

**IDL 定义：**

```thrift
// user.thrift
namespace java com.example.user

struct User {
    1: i64 userId,
    2: string name,
    3: string email,
}

service UserService {
    User getUser(1: i64 userId),
    list<User> listUsers(),
}
```

**可插拔架构：**

```
┌───────────────────────────────────────┐
│            应用代码                    │
├───────────────────────────────────────┤
│         Processor（调度层）            │
├───────────────────────────────────────┤
│  Protocol（协议层）                    │
│  TBinaryProtocol / TCompactProtocol  │
│  / TJSONProtocol                     │
├───────────────────────────────────────┤
│  Transport（传输层）                   │
│  TSocket / TFramedTransport          │
│  / THttpTransport                    │
└───────────────────────────────────────┘
```

**Thrift 优缺点：**

优点：
- **高度灵活**：传输和序列化可自由组合，适应不同场景
- **跨语言广**：支持 20+ 语言，比 gRPC 还多
- **性能好**：TCompactProtocol 序列化效率很高

缺点：
- **没有流式通信**：只支持一元请求-响应模式
- **社区活跃度下降**：新项目很少选 Thrift，逐渐被 gRPC 取代
- **缺少服务治理**：只是通信框架，负载均衡、熔断等需要自己实现或集成其他组件

### 5. 选型标准答案

面试时被问"你们会怎么选 RPC 框架"，可以这样回答：

```
场景一：北美公司 / 多语言微服务 → gRPC
  理由：跨语言强、HTTP/2 标准化、Google 背书、全球生态

场景二：国内公司 / 纯 Java 微服务 → Dubbo
  理由：服务治理全套内置、Spring 集成好、国内大厂验证

场景三：存量 Thrift 系统 → 继续用 Thrift 维护
  理由：迁移成本高、Thrift 本身性能没问题
  新服务可以逐步切 gRPC

场景四：需要流式通信（实时推送、双向流）→ gRPC
  理由：唯一原生支持四种流式模式的
```

### 6. gRPC 为什么选 HTTP/2（高频追问）

面试官问这个问题，是想看你对 HTTP/2 的理解。回答四个点：

**第一：多路复用（Multiplexing）**

HTTP/1.1 一个连接同一时间只能处理一个请求（队头阻塞）。HTTP/2 用帧（Frame）和流（Stream）机制，一个 TCP 连接上可以同时发多个请求和响应，互不干扰。

```
HTTP/1.1：连接 1 处理请求 A → 完成后才能处理请求 B（排队）

HTTP/2：
  连接 1 ─┬─ Stream 1 (请求 A) ──────────>
           ├─ Stream 2 (请求 B) ──────────>
           └─ Stream 3 (请求 C) ──────────>
           同时并行，不用排队
```

这对 RPC 非常重要：微服务间调用频率极高，HTTP/2 多路复用可以用更少的连接支撑更高的并发。

**第二：头部压缩（HPACK）**

HTTP/1.1 每个请求都要发完整的请求头（Cookie、User-Agent 等），反复传输相同的内容很浪费。HTTP/2 用 HPACK 算法压缩头部，相同的头部只传索引号，大幅减少传输量。

**第三：生态兼容**

HTTP/2 可以穿越大部分网络基础设施（负载均衡器、CDN、防火墙）。如果 gRPC 用自定义 TCP 协议，运维和部署会麻烦很多。

**第四：标准化**

HTTP/2 是 IETF 标准（RFC 7540），不是某个公司的私有协议。用标准协议意味着更好的工具支持、更广泛的社区、更长的生命周期。

## 小结（3-5 条关键点）

- gRPC = HTTP/2 + Protobuf，强项是跨语言和流式通信；Dubbo 是完整的微服务治理方案，强项是功能全面和 Spring 集成；Thrift 灵活可插拔但社区在下降。
- gRPC 四种通信模式：Unary、Server Streaming、Client Streaming、Bidirectional Streaming。这是 gRPC 相比 Dubbo 和 Thrift 的独特优势。
- 选型标准答案：北美/多语言选 gRPC，国内/纯 Java 选 Dubbo，存量 Thrift 维护为主。
- gRPC 选 HTTP/2 的四个原因：多路复用（高并发少连接）、头部压缩（省带宽）、生态兼容（穿越基础设施）、标准化（长期可靠）。
- Dubbo 的 Provider-Registry-Consumer 架构中，调用是客户端直连 Provider 的（不经过 Registry），Registry 只负责地址发现和通知。
