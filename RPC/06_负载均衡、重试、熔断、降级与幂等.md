# RPC - 第 6 课：负载均衡、重试、熔断、降级与幂等

## 学习目标（本节结束后你能做到什么）

- 理解客户端负载均衡和服务端负载均衡的差异。
- 掌握随机、轮询、加权、最少活跃数、一致性哈希等常见策略的适用场景。
- 理解重试不是“多试几次”这么简单，而是和幂等、超时、退避、调用链放大强相关。
- 理解熔断、降级、隔离各自解决什么问题。
- 建立 RPC 稳定性治理的整体视角。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 客户端负载均衡和 Nginx 式负载均衡有什么不同

Nginx 那种做法更像服务端负载均衡：

- 所有流量先到一个中间节点
- 由中间节点决定转发给哪台后端

而 RPC 框架中更常见的是客户端负载均衡：

- 调用方本地已经缓存了一组可用实例地址
- 每次调用前，客户端自己选一台
- 然后直接发给那台

优点是：

- 少一跳
- 性能更好
- 中间层更少

缺点是：

- 负载均衡逻辑分散在每个客户端进程里
- 策略变更要依赖客户端升级或控制面推送

### 2. 最常见的负载均衡策略

#### 随机

实现简单，请求量足够大时总体会比较均匀。

适合：

- 大多数通用场景

#### 轮询

按顺序一个一个发。

适合：

- 实例性能接近
- 请求耗时波动不大

#### 加权随机 / 加权轮询

给更强的机器更高权重。

适合：

- 异构机器
- 新旧版本混布

#### 最少活跃数

优先选当前正在处理请求最少的那台。

适合：

- 请求耗时差异明显
- 不希望慢实例持续堆积请求

#### 一致性哈希

相同 key 的请求尽量打到同一台。

适合：

- 有本地缓存
- 某些用户或对象需要尽量保持请求落到同一实例

### 3. 一致性哈希为什么重要

普通哈希常见做法是：

```text
hash(key) % 节点数
```

问题是节点数一变，几乎所有映射都会变化。

一致性哈希的思想是把节点和请求都映射到一个环上，新增或删除节点时，只影响局部区间，而不是全盘洗牌。

它的价值在于：

- 扩缩容时迁移范围更小
- 本地缓存命中率更稳

常见改进是虚拟节点，用来解决数据倾斜。

### 4. 重试是最容易被低估的复杂度来源

很多人第一反应是：调用失败了，那就再试一次。

听起来很合理，但问题在于：

- 你确定第一次一定没成功吗
- 如果下游已经很慢，再试会不会把它压垮
- 多层调用链叠加后，会不会形成重试风暴

所以重试从来不是“默认正确”，而是一个带风险的策略。

### 5. 哪些请求适合重试，哪些不适合

一个非常重要的分界线是幂等性。

#### 读请求

比如查询用户信息、查询库存，通常幂等性较强，更适合有限重试。

#### 写请求

比如扣款、发券、下单，如果没有幂等设计，重试可能导致重复执行。

所以更成熟的回答是：

- 可安全重放的幂等读请求，更适合重试
- 非幂等写请求，谨慎重试，必要时用请求唯一 ID 做服务端去重

### 6. 幂等到底是什么

幂等可以简单理解成：

**同一个请求执行一次和执行多次，最终效果一致。**

比如“查询用户”天然幂等；“把账户余额减 100”天然不幂等。

工程里常见做法是给每次业务请求带一个唯一请求号：

- 服务端先查这个请求号是否处理过
- 处理过就直接返回之前结果
- 没处理过才真正执行业务

这就是为什么在支付、下单、发券等场景里，请求唯一 ID 非常关键。

### 7. 重试风暴为什么危险

假设：

- A 调 B
- B 调 C

如果 C 变慢了，B 可能重试 3 次；A 看到 B 慢，也重试 3 次。那 A 的一次原始请求，可能放大成对 C 的 9 次调用。

这就是经典重试风暴。

它会导致：

- 下游本来只是慢，现在彻底被打爆
- 上游线程池、连接池跟着耗尽
- 故障沿调用链扩散

所以重试必须和：

- 总超时预算
- 重试次数上限
- 指数退避
- 失败分类

一起设计，而不是简单 `retry = 3`。

### 8. 熔断在保护谁

熔断的核心场景是：

- 下游已经明显不健康
- 继续请求它没有意义
- 再打只会把上游也拖死

这时熔断器会从“正常放行”切到“直接失败”。

你可以把它想成电路里的保险丝。

典型状态有：

- Closed：正常通过
- Open：直接拒绝，短时间不再调用下游
- Half-open：过一段时间放少量请求试探下游是否恢复

所以熔断本质上是在保护调用方和整个系统，不是在惩罚下游。

### 9. 降级是什么

熔断打开后，不一定只能报错。

有些业务可以返回兜底结果，比如：

- 推荐服务挂了，返回默认热门商品
- 评论服务挂了，页面只显示“评论暂不可用”

这就叫降级。

一句话区分：

- 熔断：决定“还要不要继续调”
- 降级：决定“调不了时返回什么”

### 10. 隔离为什么也重要

如果一个下游服务特别慢，它不只会导致超时，还可能占满：

- 调用线程池
- 连接池
- 队列

于是本来和它无关的其他请求也受影响。

所以除了熔断，很多系统还会做资源隔离，比如：

- 不同下游服务使用不同线程池
- 不同业务使用不同连接池

这类设计的目的是把故障限制在更小范围内，避免互相拖死。

### 11. 一个更成熟的稳定性治理视角

你可以把这些能力串成一条链：

- 服务发现告诉你“有哪些实例可选”
- 负载均衡决定“本次挑哪台”
- 超时决定“等多久算失败”
- 重试决定“失败后是否换一台再试”
- 熔断决定“下游明显不健康时是否直接拒绝”
- 降级决定“拒绝后返回什么兜底结果”
- 幂等决定“重试时如何避免重复副作用”

它们不是彼此独立的小开关，而是一整套稳定性协同机制。

## 小结

- RPC 更常用客户端负载均衡，由调用方本地选择实例。
- 负载均衡策略没有银弹，要看实例差异、请求时长和缓存需求。
- 重试必须结合幂等、超时预算和退避策略，否则容易引发重试风暴。
- 熔断是在保护调用方，降级是在调不了时提供兜底结果。
- 稳定性治理不是某一个组件，而是服务发现、负载均衡、超时、重试、熔断、幂等的组合设计。
