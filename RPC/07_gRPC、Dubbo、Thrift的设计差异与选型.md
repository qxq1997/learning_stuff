# RPC - 第 7 课：gRPC、Dubbo、Thrift 的设计差异与选型

## 学习目标（本节结束后你能做到什么）

- 说清 gRPC、Dubbo、Thrift 各自的核心设计特点。
- 理解“框架选型”不能只看性能，还要看语言生态、部署平台和治理需求。
- 理解 gRPC 为什么选择 HTTP/2，Dubbo 为什么在国内 Java 场景长期流行。
- 能给出面试中常见的选型回答，而不是只说“我们用过某某框架”。
- 建立从“协议、生态、治理、团队背景”综合评估 RPC 框架的意识。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么要比较框架，而不是背框架

学到这里，核心骨架你已经有了：

- 代理
- 序列化
- 协议
- 传输
- 服务发现
- 负载均衡
- 稳定性治理

具体框架之间真正不同的地方，主要体现在：

- 底层协议怎么选
- 是否跨语言
- 是否自带治理能力
- 是否贴合当前平台生态

所以比较框架，本质上是在比较设计取舍。

### 2. gRPC：现代云原生世界的主流选择之一

gRPC 是 Google 推出的 RPC 框架。

它最核心的几个特点是：

- 基于 HTTP/2
- 默认使用 Protobuf
- 强调多语言
- 原生支持流式通信

#### 为什么 HTTP/2 对 gRPC 很关键

很多人第一反应会觉得“自定义 TCP 协议不是更快吗”，但 Google 选择 HTTP/2 的价值不只在传输效率。

HTTP/2 带来的好处包括：

- 多路复用
- 头部压缩
- 生态兼容性好
- 更容易穿透代理、网关、基础设施

在云原生和北美生态里，这一点很重要，因为：

- 现有负载均衡、监控、网关体系更容易兼容 HTTP
- 与 Envoy、Istio、xDS 等生态耦合更自然

#### gRPC 的四种通信模式

- Unary：一问一答
- Server Streaming：服务端流
- Client Streaming：客户端流
- Bidirectional Streaming：双向流

这让 gRPC 不只适合普通请求响应，也适合更复杂的长连接和流式交互。

### 3. Dubbo：Java 生态里的“通信 + 治理”一体化方案

Dubbo 在国内 Java 微服务里长期非常有代表性。

它的特点不是只做“消息传输”，而是更强调一整套服务治理能力：

- 服务注册与发现
- 负载均衡
- 容错策略
- 路由规则
- 流量管控
- 监控统计

也就是说，Dubbo 更像“RPC 通信框架 + 服务治理框架”的结合体。

这也是它在纯 Java、业务中台、内部服务调用场景里长期受欢迎的重要原因：开箱即用能力很强。

### 4. Thrift：更早一代的跨语言 RPC 框架

Thrift 来自 Facebook，后进入 Apache。

它也支持 IDL 定义和多语言代码生成，历史上在很多公司都出现过。

它的优势在于：

- 跨语言
- 协议和传输层可插拔

但从近年来的主流趋势看：

- 在现代云原生生态里，gRPC 更活跃
- 在国内 Java 治理场景里，Dubbo 更常见

所以 Thrift 在新项目里的存在感相对下降，更多出现在存量系统中。

### 5. 三者怎么快速对比

| 维度 | gRPC | Dubbo | Thrift |
| --- | --- | --- | --- |
| 传输 | HTTP/2 | 自定义协议为主，也可扩展 | 多种可选 |
| 序列化 | Protobuf | Hessian / Protobuf / Triple 等 | Thrift Binary / Compact 等 |
| 跨语言 | 强 | 相对弱，近年增强但核心仍偏 Java | 强 |
| 流式通信 | 强 | 相对有限 | 一般 |
| 服务治理 | 需要结合生态组件 | 很强，框架内建较多 | 较弱 |
| 典型场景 | 云原生、多语言 | 国内 Java 微服务 | 历史跨语言系统 |

### 6. 为什么很多北美公司偏 gRPC

这个问题和用户对北美生态的追问可以一起理解。

北美很多公司的技术背景通常是：

- 多语言团队
- 全面容器化 / K8s
- 强依赖云原生基础设施
- 观测和治理更多交给平台层

在这种背景下，gRPC 的优势就非常自然：

- 多语言好
- 协议标准化
- 和 HTTP/2、Envoy、Istio 体系兼容
- 和 OpenTelemetry 等观测体系衔接自然

### 7. 为什么很多国内 Java 团队偏 Dubbo

因为国内很多成熟业务系统长期是：

- 纯 Java 或 Java 主导
- 业务中台和内部调用非常多
- 希望框架本身自带比较丰富的治理能力

Dubbo 在这种背景下就非常合适：

- 调用模型贴近 Java 接口
- 治理能力开箱即用
- 社区和实践案例多

所以“选型”从来不是抽象世界里的优劣，而是看团队和系统实际长什么样。

### 8. 面试里怎么回答“为什么选 gRPC / Dubbo”

一个比较成熟的回答方式是：

“框架选型不能只看极限性能，还要看语言生态、部署平台和治理边界。如果团队是多语言、跑在 K8s 和云原生基础设施上，gRPC 往往更合适，因为它基于 HTTP/2、Protobuf，和网关、Service Mesh、可观测性生态兼容很好。如果团队主要是 Java、内部服务治理诉求很强，希望注册发现、路由、容错、流量控制这些能力尽量开箱即用，Dubbo 会更顺手。Thrift 更多见于历史系统或特定跨语言存量场景。” 

这类回答体现的是判断力，而不是工具崇拜。

### 9. 别只从“快不快”比较 RPC 框架

如果面试时你只说：

- “gRPC 比 HTTP 快”
- “Dubbo 性能好”

这种回答通常不够有层次。

更完整的比较维度应该包括：

- 语言和团队结构
- 基础设施生态
- 服务治理由框架承担还是由平台承担
- 协议和调试体验
- 是否需要流式能力
- 是否已有大量存量系统

这才是真实工程里的选型方式。

## 小结

- gRPC、Dubbo、Thrift 都是 RPC 框架，但它们的设计重心不同。
- gRPC 强在标准化、多语言、HTTP/2 与云原生生态；Dubbo 强在 Java 生态和内建治理能力；Thrift 更多见于存量跨语言系统。
- 选型不能只比较速度，要同时看协议、生态、治理边界和团队上下文。
- 北美云原生场景下 gRPC 很常见，国内 Java 微服务场景下 Dubbo 很常见。
- 面试里更重要的不是背结论，而是讲清楚为什么某个框架更适合某类场景。
