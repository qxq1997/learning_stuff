## 内容讲解

### 1. 链路追踪

微服务架构下，一个用户请求可能经过十几个服务。某个环节慢了或报错了，你怎么知道是哪个服务的问题？这就需要链路追踪。

#### 1.1 核心概念

```
┌────────────┬────────────────────────────────────────────────────┐
│   概念      │              说明                                  │
├────────────┼────────────────────────────────────────────────────┤
│ Trace      │ 一次完整的请求链路，从入口到所有下游服务              │
│ TraceID    │ 全局唯一 ID，整条链路共享同一个 TraceID             │
│ Span       │ 链路中的一个操作段（比如一次 RPC 调用、一次 DB 查询） │
│ SpanID     │ 当前 Span 的唯一 ID                                │
│ ParentSpanID│ 父 Span 的 ID，用来构建调用树                      │
└────────────┴────────────────────────────────────────────────────┘
```

#### 1.2 调用链示例

假设用户下单，经过 API Gateway → OrderService → UserService → InventoryService：

```
TraceID: abc-123（整条链路共享）

API Gateway          OrderService          UserService       InventoryService
    │                     │                     │                   │
    │  Span1              │                     │                   │
    │  SpanID: s1         │                     │                   │
    │  Parent: null       │                     │                   │
    │  耗时: 350ms        │                     │                   │
    │────────────────────>│                     │                   │
    │                     │  Span2              │                   │
    │                     │  SpanID: s2         │                   │
    │                     │  Parent: s1         │                   │
    │                     │  耗时: 120ms        │                   │
    │                     │────────────────────>│                   │
    │                     │                     │                   │
    │                     │<────────────────────│                   │
    │                     │                     │                   │
    │                     │  Span3                                  │
    │                     │  SpanID: s3                              │
    │                     │  Parent: s1                              │
    │                     │  耗时: 200ms                             │
    │                     │────────────────────────────────────────>│
    │                     │                                         │
    │                     │<────────────────────────────────────────│
    │<────────────────────│                     │                   │

从这个调用链可以直观看到：
- 总耗时 350ms，其中 OrderService 自身处理 30ms
- UserService 调用耗时 120ms
- InventoryService 调用耗时 200ms（瓶颈在这里）
```

#### 1.3 TraceID 如何传递

TraceID 通过请求头或 Metadata 在服务间传递：

```java
// gRPC：通过 Metadata 传递
Metadata headers = new Metadata();
headers.put(Metadata.Key.of("x-trace-id", Metadata.ASCII_STRING_MARSHALLER), traceId);
headers.put(Metadata.Key.of("x-span-id", Metadata.ASCII_STRING_MARSHALLER), spanId);

// HTTP：通过请求头传递
httpRequest.setHeader("X-Trace-Id", traceId);
httpRequest.setHeader("X-Span-Id", spanId);

// Dubbo：通过 RpcContext 隐式传参
RpcContext.getContext().setAttachment("traceId", traceId);
```

**主流方案：** Jaeger（Uber 开源，CNCF 项目）、Zipkin（Twitter 开源）、OpenTelemetry（统一标准，推荐新项目使用）。

### 2. 优雅上下线

#### 2.1 优雅下线流程

直接 kill 进程会导致正在处理的请求被中断，客户端收到连接重置错误。正确的做法是"优雅下线"：

```
下线流程（4 步）：

  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │  注册中心      │    │   消费方       │    │   提供方       │
  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
         │                   │                   │
         │  ① 注销注册        │                   │
         │<──────────────────────────────────────│
         │                   │                   │ Provider 通知
         │  ② 通知消费方      │                   │ Registry 注销
         │──────────────────>│                   │
         │                   │  消费方从地址列表    │
         │                   │  移除该 Provider   │
         │                   │                   │
         │                   │  ③ 等待在途请求完成  │
         │                   │                   │ 设置一个超时
         │                   │  (已发出的请求      │ 比如 30 秒
         │                   │   等它们返回)       │
         │                   │                   │
         │                   │                   │ ④ 关闭连接和进程
         │                   │                   │ shutdown
         │                   │                   X
```

**关键细节：**
- 第 ① 步先注销，让新请求不再打过来
- 第 ③ 步等待在途请求，给一个超时时间（比如 30 秒），超时后强制关闭
- Spring Boot 中通过 `server.shutdown=graceful` 和 `spring.lifecycle.timeout-per-shutdown-phase=30s` 配置

#### 2.2 通知延迟解决方案

注册中心通知消费方有延迟（几秒甚至十几秒），在这段时间里消费方可能还会发请求给正在下线的 Provider。

解决方案：
- **主动通知**：Provider 下线前直接给所有连接的 Consumer 发一个"我要下线了"的信号
- **接口返回特殊状态码**：Provider 收到新请求时返回一个"服务不可用"的状态，Consumer 收到后自动切换其他实例
- **连接级别关闭**：Provider 发送 TCP RST 或 HTTP/2 GOAWAY 帧，客户端感知到后不再发新请求

#### 2.3 上线预热（Warmup）

新实例刚启动时，JIT 编译还没完成、缓存还是冷的、连接池还没建好。如果立刻接收和老实例一样多的流量，容易被打挂。

**预热策略：权重渐增**

```
                权重
                 │
            100% │                    ────────────────
                 │                  /
                 │                /
                 │              /
             50% │            /
                 │          /
                 │        /
                 │      /
             10% │    /
                 │  /
                 │/
                 └──────────────────────────────────── 时间
                 0        5min       10min

新实例启动后：
  0-5min：权重 10%，只接收少量请求（JIT 预热、缓存加载）
  5-10min：权重逐渐增加到 100%
  10min 后：和其他实例一样的权重
```

Dubbo 配置：`<dubbo:service warmup="600000" />` （600 秒 = 10 分钟预热期）

### 3. 服务路由

#### 3.1 标签路由（灰度发布）

标签路由的核心思想：给服务实例和请求打标签，请求只能路由到标签匹配的实例。

```
场景：商品服务 v2 要做灰度发布

  用户请求 ──> 网关
                │
                ├─ 灰度用户（tag=gray）──> 商品服务 v2（3 台，tag=gray）
                │                          新版本，只接灰度流量
                │
                └─ 普通用户（tag=normal）──> 商品服务 v1（10 台，tag=normal）
                                            老版本，接正常流量
```

灰度流量可以按用户 ID 尾号、地域、白名单等规则筛选。

#### 3.2 金丝雀发布 vs 蓝绿发布

```
┌────────────┬───────────────────────────────────────────────────┐
│            │              说明                                  │
├────────────┼───────────────────────────────────────────────────┤
│ 金丝雀发布  │ 新版本只部署少量实例（如 5%），逐渐扩大比例           │
│ (Canary)   │ 如果出问题，影响范围小，回滚快                       │
│            │ 5% → 10% → 30% → 50% → 100%                     │
│            │ 需要两个版本同时运行一段时间                         │
├────────────┼───────────────────────────────────────────────────┤
│ 蓝绿发布    │ 两套完全独立的环境：蓝（当前）和绿（新版）            │
│ (Blue-Green)│ 切换时把流量从蓝环境一次性切到绿环境                 │
│            │ 回滚就是切回蓝环境，速度极快                         │
│            │ 缺点：需要双倍资源                                  │
└────────────┴───────────────────────────────────────────────────┘
```

**面试回答：** 金丝雀发布更省资源（不需要双倍环境），但灰度过程中两个版本并存可能有兼容性问题。蓝绿发布切换干净利落，但成本高。实际中金丝雀更常用。

### 4. 服务限流

#### 4.1 单机限流 vs 分布式限流

```
单机限流：每台实例各自维护计数器
  实例 A：100 QPS 限制
  实例 B：100 QPS 限制
  问题：如果有 10 台实例，总 QPS 是 1000。但某台实例可能承受更多

分布式限流：用 Redis 维护全局计数器
  所有实例共享一个 Redis 计数器
  总限制 800 QPS（不管几台实例，全局控制）
  优点：精准控制总量
  缺点：每次请求都要访问 Redis，多一次网络开销
```

```java
// Redis 分布式限流（滑动窗口）
public boolean isAllowed(String key, int maxRequests, int windowSeconds) {
    long now = System.currentTimeMillis();
    String redisKey = "ratelimit:" + key;

    // 用 Redis Sorted Set 实现滑动窗口
    Pipeline pipeline = redis.pipelined();
    pipeline.zremrangeByScore(redisKey, 0, now - windowSeconds * 1000);  // 移除窗口外的
    pipeline.zadd(redisKey, now, now + ":" + UUID.randomUUID());         // 添加当前请求
    pipeline.zcard(redisKey);                                           // 统计窗口内数量
    pipeline.expire(redisKey, windowSeconds);
    List<Object> results = pipeline.syncAndReturnAll();

    long count = (Long) results.get(2);
    return count <= maxRequests;
}
```

#### 4.2 令牌桶 vs 漏桶

```
漏桶算法（Leaky Bucket）：

     请求涌入（速率不固定）
         │ │ │ │ │ │
         ▼ ▼ ▼ ▼ ▼ ▼
    ┌───────────────────┐
    │     桶（缓冲区）    │  ← 满了就丢弃
    │  ┌─────────────┐  │
    │  │  排队请求     │  │
    │  └─────────────┘  │
    └────────┬──────────┘
             │
             ▼
     匀速流出（固定速率）

  特点：强制匀速输出，不管输入多快，输出速率固定
  适合：削峰填谷，保护下游系统
  缺点：突发流量也被平滑掉了，延迟高


令牌桶算法（Token Bucket）：

    以固定速率往桶里放令牌
         ┌─────┐
         │Token│
         │Token│ ← 桶满了令牌丢弃
         │Token│
         └──┬──┘

    请求到达时取一个令牌：
    - 桶里有令牌 → 取走，请求通过
    - 桶里没令牌 → 请求被拒绝或等待

  特点：允许一定程度的突发（桶里攒够了令牌，突发来了可以一次用掉）
  适合：允许突发但有上限的场景，延迟低
  优点：突发流量来了不用排队，桶里有令牌就能立刻处理
```

```
┌──────────┬──────────────────────┬──────────────────────┐
│          │      漏桶            │      令牌桶            │
├──────────┼──────────────────────┼──────────────────────┤
│ 输出速率  │ 严格匀速             │ 允许突发               │
│ 突发处理  │ 排队等待，延迟高       │ 有令牌就通过，延迟低    │
│ 适用场景  │ 削峰、保护下游         │ API限流、用户请求限制   │
│ Nginx    │ limit_req（漏桶）     │ —                     │
│ Guava    │ —                    │ RateLimiter（令牌桶）   │
└──────────┴──────────────────────┴──────────────────────┘
```

**一句话总结：** 漏桶强制匀速适合削峰，令牌桶允许突发延迟低适合 API 限流。

### 5. 序列化安全

#### 5.1 Java 反序列化攻击原理

Java 原生序列化（`ObjectInputStream.readObject()`）在反序列化时会自动调用对象的 `readObject()` 方法。攻击者可以构造一个恶意的序列化数据，里面包含一个重写了 `readObject()` 的对象，反序列化时就会执行恶意代码。

```java
// 攻击原理简化版
public class MaliciousObject implements Serializable {
    private void readObject(ObjectInputStream in) throws Exception {
        in.defaultReadObject();
        // 反序列化时自动执行！
        Runtime.getRuntime().exec("rm -rf /");  // 恶意命令
    }
}

// 服务端代码（有漏洞的）
ObjectInputStream ois = new ObjectInputStream(inputStream);
Object obj = ois.readObject();  // 触发 MaliciousObject.readObject()
// 恶意代码已经执行了！
```

实际攻击中，利用的是 Apache Commons Collections、Spring 等常用库中的 gadget chain（利用链），不需要服务端代码中有 `MaliciousObject`。

#### 5.2 防护措施

```
┌────────────────────────────────────────────────────────────┐
│ 1. 不要用 Java 原生序列化                                    │
│    用 Protobuf、JSON、Hessian2 等安全的序列化方式              │
│    gRPC 用 Protobuf，天然安全                                │
│                                                            │
│ 2. 白名单机制                                               │
│    如果必须用 Java 序列化，配置允许反序列化的类白名单           │
│    JEP 290 提供了 ObjectInputFilter 机制                     │
│                                                            │
│ 3. 升级依赖                                                 │
│    Apache Commons Collections 3.x 有漏洞，升级到 4.x         │
│    定期检查 CVE 漏洞公告，及时升级有问题的依赖                  │
│                                                            │
│ 4. 网络隔离                                                 │
│    RPC 端口不要暴露到公网，内网通信也要做鉴权                   │
└────────────────────────────────────────────────────────────┘
```

**面试回答要点：**

> 一个 RPC 框架的核心模块包括：
>
> **客户端：** 业务代码通过动态代理调接口（第 2 课），代理内部先做服务路由（第 7 课灰度标签路由），再做负载均衡（第 5 课一致性哈希/加权随机），然后做容错（第 5 课 Failover/Failfast），接着序列化请求数据（第 3 课 Protobuf），最后通过网络发送（第 2 课 Netty/HTTP2）。
>
> **服务端：** 收到请求后先限流（第 7 课令牌桶），通过了再反序列化（第 3 课），然后线程调度（第 2 课线程池），最后执行业务逻辑。
>
> **基础设施：** 注册中心（第 4 课 Nacos/ZK）负责服务发现，监控层（第 7 课链路追踪）负责可观测性。

### 7. RPC 知识体系全貌总结

```
RPC 七课知识图谱：

第 1 课：什么是 RPC
  └─ RPC 本质：让远程调用像本地调用一样简单
  └─ vs HTTP：内部高频用 RPC，对外用 HTTP

第 2 课：调用完整流程
  └─ 客户端：动态代理 → 序列化 → 网络发送
  └─ 服务端：网络接收 → 反序列化 → 线程调度 → 执行
  └─ 异步：CompletableFuture

第 3 课：序列化
  └─ Protobuf（性能最优）、JSON（可读性最优）、Hessian（Java 通用）
  └─ Protobuf Varint 编码原理

第 4 课：服务发现与注册中心
  └─ 注册 → 发现 → 通知三步流程
  └─ ZK(CP) vs Eureka(AP) vs Nacos(AP+CP)
  └─ K8s DNS 服务发现

第 5 课：负载均衡与容错
  └─ 一致性哈希 + 虚拟节点
  └─ Failover/Failfast/Failsafe 容错策略
  └─ 熔断 Circuit Breaker 三状态

第 6 课：主流框架对比
  └─ gRPC(HTTP/2+PB) vs Dubbo(全套治理) vs Thrift(可插拔)
  └─ 选型：北美 gRPC，国内 Dubbo

第 7 课：高级话题（本课）
  └─ 链路追踪：TraceID/SpanID 全链路可观测
  └─ 优雅上下线：注销→通知→等待→关闭
  └─ 限流：令牌桶(突发) vs 漏桶(匀速)
  └─ 序列化安全：不用 Java 原生序列化
```

## 小结（3-5 条关键点）

- 链路追踪靠 TraceID（全局唯一，整条链路共享）和 SpanID/ParentSpanID（构建调用树）实现。TraceID 通过请求头或 Metadata 在服务间传递。新项目推荐 OpenTelemetry。
- 优雅下线四步：注销注册 → 通知消费方 → 等待在途请求完成 → 关闭进程。上线用预热（Warmup），权重从低到高渐增，避免冷启动被打挂。
- 令牌桶允许突发延迟低（适合 API 限流），漏桶强制匀速适合削峰（Nginx limit_req 就是漏桶）。分布式限流用 Redis 全局计数器。
- Java 原生序列化有反序列化攻击风险（readObject 触发恶意代码），RPC 框架应该用 Protobuf/JSON/Hessian 等安全的序列化方式。
- 设计 RPC 框架的回答要串联全部七课：动态代理 → 路由 → 负载均衡 → 容错 → 序列化 → 传输（客户端），限流 → 反序列化 → 调度 → 执行（服务端），加上注册中心和监控层。
