# RPC - 第 8 课：高级话题：链路追踪、优雅上下线、服务路由与设计 RPC 框架

## 学习目标（本节结束后你能做到什么）

- 理解链路追踪为什么在微服务里几乎是必需品。
- 说清优雅上下线、预热、灰度路由、金丝雀发布在服务治理里的作用。
- 理解限流、隔离、观测、安全这些高级能力如何和 RPC 框架结合。
- 能从模块划分角度描述“设计一个最小 RPC 框架”。
- 形成一套更接近中高级后端面试表达的 RPC 全景视角。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 链路追踪为什么是微服务时代的基础设施

单体应用里，一个请求通常在同一个进程里跑完。可一旦进入微服务，一个用户请求可能经过：

- API 网关
- 订单服务
- 用户服务
- 库存服务
- 营销服务
- 支付服务

这时如果用户反馈“接口慢了”，你靠单点日志很难判断瓶颈在哪一跳。

链路追踪就是为了解决这个问题。

它通常通过：

- TraceId：标识一次完整请求链路
- SpanId：标识链路中的一个处理片段
- ParentSpanId：把整条链串成树

让你看到请求在每一跳的耗时和状态。

### 2. TraceId 为什么必须跨服务传递

如果每个服务都自己生成一个新 ID，那链路就断了。

所以入口层通常会生成一个 TraceId，然后通过：

- HTTP Header
- gRPC Metadata
- Dubbo Attachment

一路往下游透传。

这就是为什么链路追踪既是监控问题，也是协议扩展问题。

### 3. OpenTelemetry 为什么值得知道

现代系统里，链路追踪、日志和指标越来越希望走统一标准。OpenTelemetry 就是在扮演这个角色。

面试时如果你能提到：

- 用统一标准采集 trace、metrics、logs
- 通过自动注入或中间件把 Trace 上下文透传

通常会比只会说 Zipkin、Jaeger 更显得跟得上当前生态。

### 4. 优雅下线为什么重要

如果一个实例要下线，而你直接把进程杀掉，会发生什么？

- 正在处理的请求可能中断
- 上游调用方可能还以为它活着
- 短时间内会产生很多失败请求

优雅下线的目标就是：

1. 先从服务发现里摘掉，不再接新流量
2. 让已经在途的请求处理完
3. 再关闭进程

这套过程对于 RPC 服务尤其重要，因为它往往承接大量内部依赖，一旦粗暴退出，影响会沿调用链放大。

### 5. 优雅上线和预热为什么也要重视

新实例刚启动时，往往并不在最佳状态：

- JIT 还没热起来
- 本地缓存还没加载完
- 数据库连接池刚建好
- 依赖服务连接还在初始化

如果一上线就立刻给它全量流量，很容易出现：

- 新实例响应慢
- 被误判为不健康
- 形成来回摘流和抖动

所以成熟系统通常会：

- 先 readiness 不通过，不接流量
- 等准备好后再加入服务列表
- 或在一段预热期内逐渐提升权重

### 6. 服务路由和灰度流量

负载均衡回答的是“在一组实例里怎么均匀选”，服务路由回答的是“某类请求应该只去哪些实例”。

常见路由场景包括：

- 灰度用户只打到新版本
- 上海流量优先去上海机房
- 特定租户走专属实例

这类能力在 RPC 框架、网关、Service Mesh 中都可能出现，本质上都属于流量治理。

### 7. 金丝雀发布和蓝绿发布怎么区分

#### 金丝雀发布

先把少量流量切到新版本，观察无误后逐步放大。

优点：

- 风险小
- 回滚影响范围可控

缺点：

- 过程更长
- 流量管理更复杂

#### 蓝绿发布

准备一整套新环境，切换时整批流量从旧环境切到新环境。

优点：

- 切换和回滚都很快

缺点：

- 成本更高
- 对双环境一致性要求高

### 8. 限流和隔离在 RPC 高级治理里的位置

前面我们讲过熔断和重试，这里再把稳定性治理再补完整一点。

RPC 高级治理里，常见还会有：

- 限流：控制进入服务的请求速度
- 隔离：把不同下游或不同业务的资源池分开
- Bulkhead：避免某一个依赖拖垮整个系统

你可以把它们理解成：

- 限流：控制“进来多少”
- 熔断：控制“还要不要继续往外调”
- 隔离：控制“谁坏了别影响全局”

### 9. 安全问题别只想到鉴权

RPC 安全里还有一个很重要但常被忽略的话题：反序列化安全。

尤其在 Java 历史系统里，原生序列化类漏洞很多。

更稳妥的实践通常是：

- 不使用 Java 原生序列化
- 对可反序列化类型做白名单限制
- 尽量使用 Protobuf 等结构化二进制协议

这说明 RPC 的问题不只在“能不能通”，还在“安不安全”。

### 10. 设计一个最小 RPC 框架，应该有哪些模块

如果面试官让你设计一个 RPC 框架，你可以按模块拆：

#### 客户端

- 代理层：把接口方法调用拦截下来
- 服务发现层：拿到可用实例列表
- 路由与负载均衡层：决定这次调哪台
- 容错层：超时、重试、熔断、降级
- 协议与编解码层：序列化、反序列化、协议帧处理
- 传输层：连接管理、发送接收、requestId 关联

#### 服务端

- 传输层：接收连接和消息
- 解码层：解析协议帧与请求体
- 调度层：根据服务名和方法名找到真实实现
- 执行层：调用业务逻辑
- 编码层：把结果包装为响应

#### 控制与治理

- 注册中心 / 配置中心
- 健康检查
- 指标与链路追踪
- 灰度路由与发布控制

这就是一个很完整的答案框架。

### 11. 一段适合面试的总结性表达

如果面试官让你整体讲 RPC，你可以这样收束：

“RPC 的底层骨架其实不复杂，核心是代理、协议、传输和调度。但真正把 RPC 做成生产级体系，难点不在于把请求发出去，而在于服务发现、负载均衡、超时重试、幂等、熔断、链路追踪、优雅上下线这些治理能力。框架之间差异也主要体现在协议选择、跨语言能力、治理边界和生态整合上。真正理解 RPC，不是会用某个框架 API，而是知道一次调用在整个分布式系统里会引入哪些成本、失败模式和治理需求。” 

这类收束能体现出你不是在背零散知识点，而是在讲一个完整体系。

## 小结

- 链路追踪让你能看见一次请求在多服务间的完整路径和耗时分布。
- 优雅上下线和预热决定了服务发布时会不会把故障带给调用链。
- 灰度路由、金丝雀和蓝绿发布，体现的是流量治理能力而不只是部署动作。
- RPC 的高级问题往往集中在稳定性、可观测性和安全，而不只是“通信快不快”。
- 设计 RPC 框架时，最好按客户端、服务端、治理控制三层来组织思路。
