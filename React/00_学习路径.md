# React 学习路径与知识图谱

## 学习定位

这组 React 教程的目标，不是让你背一堆前端 API，也不是一上来就让你记 `useEffect`、`useMemo`、路由、状态管理这些名词。

你有后端开发经验，所以我们会把 React 当成一种“前端应用的工程模型”来学：

- 页面不是一堆零散 DOM 操作，而是一棵由状态驱动的 UI 树。
- 组件不是简单的 HTML 片段，而是前端侧的模块边界。
- State 不是普通变量，而是驱动渲染和交互的业务事实。
- Effect 不是“生命周期函数换皮”，而是 React 与外部系统同步的出口。
- 性能优化不是到处加缓存，而是先理解渲染、更新、提交和浏览器绘制的成本。

学完这条路线后，你应该能从零写出一个稳定的 React 业务页面，并且能解释它为什么这样组织，而不是只会复制组件库示例。

## 为什么这样排序

很多 React 学习路线容易从 JSX 或 Hook 开始，但这样会有一个问题：你先记住了写法，却没有建立“React 为什么要这样设计”的直觉。

比如你会很快遇到这些疑问：

- 为什么改普通变量页面不会变，必须用 State？
- 为什么函数组件会反复执行？
- 为什么 `useEffect` 依赖数组一写错就死循环？
- 为什么列表 key 不能用随机数？
- 为什么有时候子组件明明 props 没变，也被重新执行了？
- 为什么全局状态管理不是越早引入越好？

这些问题的根都在第一层心智模型里：React 是一个用状态描述界面的系统。它的核心不是“操作 DOM”，而是“根据当前状态重新计算 UI，然后把必要变化提交给浏览器”。

所以第一阶段先建立核心模型：React 解决什么问题、组件是什么、状态怎么驱动界面、渲染到底发生了什么。

第二阶段再进入高频业务能力：Effect、通信、列表、表单、异步请求、路由。

第三阶段才看工程判断：状态管理、性能优化、后端协作、TypeScript、测试和 Next.js。这样学下来，你后面遇到新框架或新 API，也能知道它们到底在解决哪一层问题。

## 学习方法建议

- 每学一个 API，都先问：它解决的是状态、渲染、通信、副作用、性能，还是工程组织问题？
- 不要只背“怎么写”，要能说清“为什么普通变量不行”“为什么依赖数组要这样写”“为什么 key 影响复用”。
- 遇到 React 的报错或奇怪行为，优先回到“状态变化 -> 重新渲染 -> 提交 DOM”这条链路排查。
- 写示例时不要只写计数器，要逐步贴近真实后端业务：列表页、搜索框、分页、详情页、编辑表单、权限按钮、错误提示。
- 学到状态管理和性能优化时，不急着上工具，先判断问题是不是真的存在。

## 学完后你应该能回答的问题

- React 和直接操作 DOM 的核心差异是什么？
- 为什么说 React 的核心模型是 `UI = f(state)`？
- Props、State、Effect 分别承担什么职责？
- 函数组件重新执行时，React 到底重新做了什么？
- 为什么列表渲染必须有稳定 key？
- `useEffect` 适合做什么，不适合做什么？
- 什么时候应该状态提升，什么时候应该用 Context 或外部状态库？
- 请求数据时，如何处理 loading、error、竞态和缓存？
- React 页面如何和后端 API 做稳定协作？
- 性能问题出现时，应该先看渲染边界、数据结构，还是先加 `memo`？
