# Redis 学习路径与知识图谱

## 学习定位

很多人学 Redis 时，容易停留在“会背命令”的层面：

- 知道 `SET`、`GET`、`HSET`、`ZADD` 怎么用
- 知道 Redis 很快，常拿来做缓存
- 知道它支持主从、哨兵、集群

但一到更深一点的问题就会发懵：

- Redis 为什么快，单线程为什么反而高性能
- 为什么有序集合要用跳表
- RDB 和 AOF 的真实取舍是什么
- 缓存击穿、穿透、雪崩到底分别是什么
- Redis Cluster 为什么要用 16384 个哈希槽
- 为什么有些场景适合 Redis，有些场景不适合

这条学习路径的目标，不是把你训练成“Redis 命令手册”，而是把 Redis 放回真实后端系统里，从“缓存工具”提升成“内存数据基础设施”来理解。

学完这一套，你应该能做到：

- 说清楚 Redis 到底是什么，它为什么几乎出现在所有后端系统里
- 理解 Redis 五大数据类型与底层实现之间的关系
- 解释 Redis 为什么快，而不是只会说“因为在内存里”
- 理解持久化、复制、哨兵、集群各自在解决什么问题
- 面对缓存设计题、Redis 面试题、线上 Redis 故障时，不只会背结论，而能讲清楚原理、边界和取舍

## 知识图谱（从零到专家）

1. 先建立整体认知
   - Redis 是什么
   - Redis 不是什么
   - 它在系统架构里的典型位置
   - 什么场景适合 Redis，什么场景不适合
2. 再掌握核心数据模型
   - String、Hash、List、Set、ZSet
   - Bitmap、HyperLogLog、Geo、Stream
   - 命令层能力和底层结构之间的关系
3. 再理解高性能本质
   - 内存访问
   - 事件循环
   - 单线程执行命令
   - I/O 多路复用
   - 为什么 Redis 不是“不会并发”，而是“换了一种并发模型”
4. 再补齐数据可靠性
   - RDB
   - AOF
   - 混合持久化
   - 重写、恢复、fork 和 Copy-On-Write
5. 再进入缓存设计
   - 过期删除
   - 内存淘汰
   - 缓存穿透、击穿、雪崩
   - 双写一致性、延迟双删、旁路缓存
6. 再进入高可用
   - 主从复制
   - PSYNC
   - Sentinel
   - 故障转移
7. 再进入扩展能力
   - Redis Cluster
   - 哈希槽
   - MOVED / ASK
   - 重分片与扩缩容
8. 最后进入工程实践
   - 事务
   - Lua
   - 分布式锁
   - Stream
   - big key、hot key、慢查询、线上治理
9. 面试追问与专题展开
   - `String` vs `Hash`
   - SDS
   - 购物车与对象缓存设计
   - Set / ZSet 的真实使用边界
   - Bitmap / HyperLogLog / Bloom Filter
   - 事务、Pipeline、Lua 的区别
   - bigkey / hotkey / 慢查询 / 阻塞排查
   - Sentinel / Cluster 高频追问
   - Redis 选型边界与替代方案

## 文章结构优化建议

Redis 这一组文章原来更偏“概念解释 + 面试追问”，适合打基础，但如果要服务真实项目和生产排障，结构需要更稳定。后续每篇建议统一成这个节奏：

1. 先讲这个主题解决什么现实问题。
2. 再讲核心原理和关键概念。
3. 再讲典型业务场景怎么落地。
4. 再讲线上可能怎么出问题。
5. 最后用小结和问题做复盘。

对应到文章里的固定模块就是：

- `学习目标`：告诉你学完能判断什么、解释什么。
- `内容讲解`：先把概念和原理讲透。
- `实战落地`：把知识放进缓存、排行榜、锁、限流、秒杀、Session 等真实场景。
- `生产问题处理`：给出排障入口、止血动作、指标和复盘方向。
- `小结 / 问题`：用于复习和面试表达。

这样做的好处是：每篇都不只停留在“Redis 是什么”，而是能回答“我在项目里怎么用、上线后怎么监控、出事后怎么救、复盘时怎么改”。

## 为什么这样排序

这条路线故意不是“先背高级特性”，而是先把 Redis 放回真实系统里。

因为如果你没有先理解：

- Redis 为什么存在
- 它到底在系统里解决什么问题
- 它的核心数据模型和高性能本质是什么

那么后面的哨兵、集群、分布式锁、跳表、Lua 都会变成散乱的知识点。

正确顺序应该是：

1. 先建立“Redis 是什么”的整体地图
2. 再理解“Redis 存什么、怎么存”
3. 再理解“Redis 为什么快”
4. 然后进入“如何不丢数据、如何做缓存、如何高可用、如何横向扩展”
5. 最后再进入“真实工程里怎么用、怎么踩坑、怎么优化”

而跳表这件事之所以单独放一个专题，是因为它非常典型：

- 它不是 Redis 所有数据结构都会用到的东西
- 但它又是理解 ZSet 和 Redis 内部设计取舍的一个关键窗口

你一旦把跳表真正吃透，对 Redis 的理解会从“会用命令”上升到“能看懂为什么这样设计”。

这次新补的专题，定位也很明确：

- 前 1 到 8 课是“主线”，帮你先建立 Redis 的系统地图。
- 后面的 `09` 到 `18` 以及 `06b`、`07b` 更像“面试延伸 + 线上实战专题”。

也就是说，如果你前面已经学过 Redis 基础，那完全可以把这些新增专题当成第二轮深挖材料：

- 某个问题被面试官追问了，再回来看对应专题
- 某个线上事故真的遇到了，再看对应排障专题
- 某类业务模型要落地，再看数据结构专题

## 学习方法建议

- 每学一课，都先回答一个问题：这一课到底在解决什么现实问题。
- 学到 `实战落地` 时，强迫自己补一个项目例子：key 怎么设计、TTL 怎么设、失败后怎么补偿。
- 学到 `生产问题处理` 时，把内容整理成排障清单：先看什么指标，怎么止血，事后怎么治理。
- 学命令时，不只记“怎么用”，还要记“底层结构是什么”“复杂度是什么”“适合什么场景”。
- 学高可用和集群时，不要只看名词，要把请求链路、故障切换链路、数据同步链路画出来。
- 学缓存设计时，要时刻把数据库、应用、Redis 三者放在一起看，而不是孤立地看 Redis。
- 学跳表、哈希表、SDS、listpack 这些内部结构时，要问自己：如果不用这个设计，会慢在哪里、复杂在哪里、贵在哪里。

## 学完后你应该能回答的问题

- Redis 为什么快，快在什么地方，慢又会慢在哪里。
- Redis 五大数据类型分别适合什么业务场景，底层结构为什么不是“一种结构打天下”。
- RDB 和 AOF 分别在性能、数据安全、恢复速度上怎么取舍。
- 为什么缓存系统不是“把数据库数据塞进去”这么简单，而是一整套一致性和故障设计。
- Sentinel 在解决什么问题，Redis Cluster 又在解决什么问题，它们的边界分别是什么。
- 为什么 ZSet 要同时维护字典和跳表，而不是只用一种结构。
- 为什么很多公司 Redis 基础用得不差，但一到 `Pipeline`、`Lua`、`bigkey`、`hotkey`、`Bloom Filter`、`Stream` 这些话题就会答得发虚。
- 为什么面试里喜欢追问 “Redis 事务为什么鸡肋”“为什么不用 Redis 当主库”“为什么 Cluster 用 16384 个槽”。

## 推进规则

- 如果你是第一次系统学 Redis，建议严格按顺序推进。
- 如果你已经会日常命令，但原理不扎实，可以从第 2 课开始补。
- 如果你的重点是面试，优先反复看第 2、3、4、5、6、7 课和跳表专题。
- 如果你的重点是 Redis 面试深挖，建议把新增的 `09` 到 `18`、`06b`、`07b` 当成第二轮强化题库来学。
- 如果你被追问到底层结构，优先补 `19`；如果你被追问缓存一致性、布隆过滤器、秒杀库存，优先补 `20`。
