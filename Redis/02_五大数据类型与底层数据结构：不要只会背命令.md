# Redis - 第 2 课：五大数据类型与底层数据结构，不要只会背命令

## 本篇定位

主线入门层。重点解决“Redis 存什么、为什么这样存”的问题，是后续对象缓存、排行榜、特殊结构和底层追问的基础。

## 学习目标（本节结束后你能做到什么）

- 说清 Redis 五大数据类型分别适合什么业务场景。
- 理解“数据类型”和“底层数据结构”不是一回事。
- 能把 String、Hash、List、Set、ZSet 放回真实后端需求中理解。
- 对 SDS、哈希表、quicklist、intset、listpack、跳表建立整体印象。
- 避免“看见需求就无脑上 String 或 JSON 大对象”的常见误区。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先分清两个层次：数据类型 vs 底层结构

很多初学者一学 Redis 就会混淆这两个概念。

比如：

- 看到 `String`，以为底层就是简单 C 字符串
- 看到 `ZSet`，以为底层永远就是跳表
- 看到 `List`，以为底层就是普通双向链表

这都不够准确。

更正确的理解是：

- **数据类型**是给用户暴露出来的抽象能力
- **底层数据结构**是 Redis 为了实现这种能力而选择的内部实现

也就是说，用户看到的是“我要一个有序集合”，Redis 内部思考的是“用什么结构来实现插入、删除、查分数、按范围遍历这些操作更划算”。

### 2. String：最简单，但千万别低估它

String 是 Redis 最基础也最常用的数据类型。

它不只是“存一段文本”，还常被用来做：

- 缓存一个 JSON 字符串
- 计数器，自增自减
- 分布式锁的值
- 位图底层载体

Redis 内部并不是用传统 C 字符串来直接存它，而是用了 `SDS`，也就是简单动态字符串。SDS 的价值在于：

- 记录长度，取长度是 O(1)
- 可以预分配空间，减少频繁扩容
- 二进制安全，不依赖 `\0` 结尾

这说明 Redis 的“快”不是只有内存层面，连最基础的字符串实现都做了工程优化。

### 3. Hash：特别适合存结构化小对象

如果你要存一个用户对象：

- id
- name
- age
- city

你当然可以把整个对象序列化成 JSON 存成一个 String，但这样会有一个问题：你每次改一个字段，都得整段取出、反序列化、修改、再整段写回去。

Hash 更适合这种“一个 key 下挂很多 field”的场景。

例如：

```text
key = user:1001
field = name -> xinqi
field = city -> shanghai
field = age -> 30
```

这样做的好处是：

- 读写单个字段更方便
- 结构语义更清晰
- 对小对象很友好

Redis 在小 Hash 场景下通常会使用更紧凑的编码，比如 listpack，等字段数和数据量变大后再切换到哈希表。

这体现了一个很重要的设计思想：

**Redis 不是固定死一种底层结构，而是会根据数据规模在空间和时间之间做权衡。**

### 4. List：不是“消息队列万能解”

List 提供的是有序序列能力，支持从左边或右边 push / pop。

它很适合：

- 简单任务队列
- 时间顺序消息缓冲
- 最新消息列表

但也要注意，它并不是现代消息队列的完整替代品。你不能因为 `LPUSH`、`RPOP` 能用，就把所有 MQ 场景都扔给 List。

Redis 的 List 在较新版本里通常基于 `quicklist` 实现。你可以把 quicklist 理解成：

- 逻辑上像链表
- 每个节点里装的不是单个元素，而是一段连续紧凑存储的数据块

这样做比“纯链表一个节点一个元素”更节省内存，也更有缓存友好性。

### 5. Set：去重和集合运算的利器

Set 最大的特点是：

- 无序
- 元素唯一

特别适合：

- 标签集合
- 去重
- 交并差集运算

例如：

- 某用户点赞过哪些文章
- 两个用户共同关注了哪些人
- 某活动有哪些参与用户

对于整数较少的集合，Redis 可能会用 `intset` 这种紧凑结构；数据变复杂、规模变大后再切到哈希表。

这再次说明，Redis 很在意“小数据场景下的空间效率”。

### 6. ZSet：Redis 里最有“味道”的数据类型之一

ZSet 是有序集合，每个元素带一个分数 `score`。它特别适合：

- 排行榜
- 延迟任务
- 按时间排序的数据
- 按权重或分值排序的业务集合

例如游戏积分榜：

- userA -> 1200
- userB -> 980
- userC -> 3000

你既要能按成员查分值，也要能按分值范围查成员，还要能按排名取 TopN。

这就是为什么 ZSet 的内部实现通常不是单一结构，而是：

- 一个字典，方便按 member 找 score
- 一个跳表，方便按 score 排序和范围遍历

这里的组合特别经典：

- 字典解决“按成员找分数快”
- 跳表解决“按排序维度访问快”

如果只用字典，排序访问会很痛苦；如果只用跳表，按成员直接查找又不够高效。

### 7. 为什么 Redis 不用“一种结构打天下”

这个问题非常重要。

很多工程系统如果偷懒，会做成：

- 所有东西都存成字符串
- 所有逻辑都靠应用层自己拼

Redis 没这么干。它为不同场景设计了不同的数据类型和编码方式，因为不同需求的最优结构根本不一样。

比如：

- 纯字符串读写，SDS 很合适
- 小对象字段存储，Hash 更合适
- 去重和集合运算，Set 更合适
- 排序和范围查询，ZSet 更合适

这背后的核心思路是：

**让用户面对的是稳定抽象，让 Redis 内部根据数据规模和访问模式做最合适的底层选择。**

### 8. 工程上最常见的类型误用

#### 误用一：什么都用 String

最常见的偷懒方式就是把所有对象都塞成 JSON String。短期方便，长期会出现：

- 局部字段更新不方便
- 大对象读写成本高
- 容易形成 big key

#### 误用二：把 List 当万金油消息队列

List 可以做轻量队列，但没有天然复杂消费组语义。更复杂场景更适合 Stream 或专业 MQ。

#### 误用三：ZSet 用得多，却不知道它为什么能做排行榜

如果只会 `ZADD`、`ZRANGE`，却不知道内部为什么能同时支持查成员和查范围，那你对 Redis 的理解仍停留在 API 层。

## 实战落地：按访问模式选类型

实际建模时，可以按“读写动作”反推数据类型，而不是先背命令：

- 只需要整体读写的小值，用 `String`，例如验证码、配置片段、计数器。
- 需要局部字段读写的小对象，用 `Hash`，例如购物车、用户轻量扩展信息。
- 需要先进先出或简单阻塞消费，用 `List`，但要确认可靠性要求不高。
- 需要去重、交并差，用 `Set`，例如已报名用户、抽奖候选池。
- 需要排序、范围查询、TopN，用 `ZSet`，例如排行榜、延时任务、按时间窗口统计。

一个很常见的生产经验是：对象缓存不要一上来就全量 JSON。JSON String 开发快，但字段级更新、局部读取、bigkey 控制都更难。Hash 更适合字段规模稳定、单个对象不大的场景；如果字段特别多、单对象可能无限膨胀，就要拆 key 或回到数据库建模。

## 生产问题处理：类型选错后的治理

类型误用通常不是当天爆炸，而是数据量上来后慢慢变成事故：

- `String` JSON 变成 bigkey：先用 `MEMORY USAGE`、`--bigkeys` 或离线 RDB 分析定位，再拆成 Hash、多个 key 或只缓存热点字段。
- `Hash` 字段无限增长：按业务维度拆分，例如 `cart:{userId}:{shopId}`，避免单个 hash 过大。
- `List` 被当可靠队列：检查是否有消费确认、重试和死信；没有的话迁移到 `Stream` 或专业 MQ。
- `Set` / `ZSet` 成员过多：限制窗口、按日期或分桶拆 key，避免单次集合运算拖慢主线程。

治理时不要只改 Redis 结构，还要补 key 命名规范、TTL 策略、容量预估和删除脚本，否则同类问题会在新 key 上重复出现。

## 小结

- Redis 暴露给用户的是数据类型，内部真正支撑它们的是底层数据结构和不同编码实现。
- String、Hash、List、Set、ZSet 各有最适合的业务问题，不要一把梭全用 String。
- Redis 会根据数据规模选择更紧凑或更高效的底层结构，这体现了它对性能和内存的精细权衡。
- ZSet 很典型，它常通过“字典 + 跳表”同时满足按成员查找和按分数排序访问两类需求。
- 真正会用 Redis，不是背会命令，而是会为业务问题选合适的数据类型。
