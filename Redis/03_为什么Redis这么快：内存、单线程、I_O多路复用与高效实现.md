# Redis - 第 3 课：为什么 Redis 这么快，内存、单线程、I/O 多路复用与高效实现

## 本篇定位

主线入门层。重点建立 Redis 的性能模型，帮助你理解它为什么快，以及为什么 bigkey、慢命令、重脚本会把它拖慢。

## 学习目标（本节结束后你能做到什么）

- 不再用“因为在内存里”这种一句话解释 Redis 的性能。
- 理解 Redis 的高性能来自哪些层面，而不是只来自其中一个点。
- 说清楚“单线程”到底指什么，为什么它不等于低吞吐。
- 建立 Redis 事件循环和 I/O 多路复用的基本心智模型。
- 理解 Redis 快的同时，瓶颈通常会出现在什么地方。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. Redis 快，首先当然是因为内存

这句结论本身没错，但太粗了。

磁盘访问和内存访问不是一个量级。尤其是随机 I/O，磁盘会慢得多。Redis 把主要工作集放在内存里，天然就绕开了磁盘寻址和机械/SSD 访问成本。

但如果你只说“因为在内存里”，面试官完全可以追问：

- 那 Memcached 也在内存里，为什么还要学 Redis 内部设计？
- 既然都在内存里，为什么大 key、慢命令、阻塞命令还是能把 Redis 拖慢？

这就说明，“在内存里”只是 Redis 快的第一层，不是全部。

### 2. Redis 的核心命令执行模型是单线程

这里的“单线程”，说的是：

**大部分命令执行逻辑由一个主线程串行处理。**

这听上去像缺点，但它反而带来几个特别大的工程优势：

- 不需要为核心数据结构加大量锁
- 不需要处理复杂的多线程并发修改一致性问题
- 代码路径更短，CPU cache 行为更稳定
- 避免多线程争用带来的上下文切换和锁竞争开销

也就是说，Redis 不是不会并发，而是把问题换了一种方式解决：

- 网络上可以同时有很多连接
- 事件循环可以同时关注很多 socket
- 但真正修改内存数据结构时，尽量在一个主线程里顺序执行

这样做特别适合 Redis 的典型工作负载：

- 命令粒度小
- 单次执行快
- 大量操作是简单读写

### 3. 单线程为什么不会很慢

很多人下意识会想：

- 一个线程处理所有命令，那不就排队了吗？

是的，确实排队。但关键在于 Redis 的单条命令通常非常快，而且省掉了多线程争用的额外成本。

想象两个模型：

#### 模型一：多线程并发改共享哈希表

- 线程多
- 需要锁
- 需要同步
- 可能有 cache line 争用
- 可能有上下文切换

#### 模型二：单线程顺序改共享哈希表

- 没有锁竞争
- 不需要复杂同步
- 路径更短
- 单次操作更可预期

对于“短平快”的命令来说，第二种方式反而可能更高效。

所以 Redis 的高性能并不和“单线程”矛盾，反而在很多场景下是因此受益。

### 4. Redis 怎么同时处理很多连接

这时候就轮到 I/O 多路复用出场了。

Redis 并不是说：

- 一个线程
- 只能盯着一个连接

它用的是事件循环 + I/O 多路复用。

你可以把它想成一个服务员，不是每桌轮流死等，而是：

- 先登记所有桌子
- 谁准备好了、谁有请求了、谁可以写数据了，就通知服务员
- 服务员只处理“就绪事件”

在 Linux 下，这通常靠 `epoll` 这种机制实现。Redis 会把很多客户端连接注册给内核，等内核告诉它：

- 某连接可读了
- 某连接可写了

Redis 再去处理对应事件。

所以 Redis 的真正模型更接近：

```mermaid
flowchart LR
    A["很多客户端连接"] --> B["I/O 多路复用"]
    B --> C["Redis 事件循环"]
    C --> D["主线程执行命令"]
```

这就是为什么 Redis 可以在单线程执行命令的同时，还处理大量并发连接。

### 5. Redis 快，还因为它的内部实现很抠细节

Redis 的性能不是只靠大方向，很多细节也很讲究。

例如：

- 字符串用 SDS，而不是裸 C 字符串
- 小对象用更紧凑的编码
- List 用 quicklist
- ZSet 组合字典和跳表
- 渐进式 rehash 避免一次性重哈希卡死

这说明 Redis 的性能来自两个层次：

- 大的执行模型选得好
- 小的数据结构实现也做得很细

### 6. Redis 6 之后不是完全“只有一个线程”

这也是面试爱考的点。

很多资料会说 Redis 是单线程，但更准确的说法应该是：

- 命令执行核心仍主要由主线程负责
- 但持久化、异步删除、集群通信、某些后台任务并不都在主线程
- Redis 6 开始也引入了可选 I/O 线程，用来分担网络读写准备工作

所以“Redis 是单线程”更像一句工程上的简化口号，它强调的是：

**Redis 避免用多线程并发执行大量命令逻辑，而不是整个进程里永远只有一个线程。**

### 7. Redis 快，但也有清晰边界

Redis 很快，不代表永远快。它的瓶颈通常会出现在这些地方：

- big key 导致单次操作很重
- `KEYS`、大范围 `ZRANGE`、大批量删除等命令阻塞主线程
- 网络带宽打满
- AOF 重写、RDB fork 带来额外内存和 CPU 压力
- 热点 key 过于集中
- Lua 脚本太长

因为命令执行主线程很珍贵，所以一旦有“单次很重”的操作，Redis 就特别容易显出阻塞感。

这也解释了 Redis 设计的另一面：

**它擅长很多小而快的操作，不擅长少量很重的大操作。**

## 实战落地：用延迟预算约束 Redis 使用方式

生产系统里，不要只说“Redis 很快”，而要给它一个延迟预算。例如核心接口希望 Redis 单次调用在 1ms 级别，偶发抖动也不能把接口整体 P99 拉爆。基于这个预算，代码层要做几件事：

- 避免在线路径执行 `KEYS`、大范围 `ZRANGE`、`SMEMBERS`、`HGETALL` 这类不可控命令。
- 对批量读取优先用 Pipeline 或批量接口，但要限制单批大小。
- 对大对象删除用 `UNLINK` 或异步清理，避免 `DEL` 阻塞主线程。
- 对 Lua 脚本设置复杂度红线，脚本里不要循环扫描大量 key。
- 客户端连接池要有超时、最大连接数和熔断策略，防止 Redis 抖动拖死应用线程池。

Redis 的高性能不是让业务随便用，而是要求业务把每个命令都控制在“短、平、快”的形态里。

## 生产问题处理：Redis 变慢的定位顺序

线上接口突然变慢，可以按这个顺序查：

1. 先看 `SLOWLOG GET`，确认有没有慢命令或重 Lua。
2. 再看 `INFO commandstats`，确认高频命令是否异常，比如 `hgetall`、`zrange` 突然暴涨。
3. 再看 `INFO clients` 和客户端监控，确认是否连接池耗尽、等待队列堆积。
4. 再看 CPU、网卡、内存、swap、磁盘 fsync，判断是不是资源层瓶颈。
5. 最后回到业务发布记录，看是否新代码引入大 key、大批量读取或错误循环。

如果 `SLOWLOG` 干净但接口慢，问题常常在网络、客户端连接池、序列化、回源数据库，而不是 Redis 命令本身。这个区分很重要，否则会把时间花在错的地方。

## 小结

- Redis 快，首先因为内存，但不只是因为内存。
- Redis 核心命令执行采用单线程模型，主要收益是省掉锁竞争、同步和上下文切换开销。
- Redis 通过事件循环和 I/O 多路复用，在单线程执行命令的同时处理大量并发连接。
- Redis 的高性能还来自对底层数据结构和编码实现的精细优化。
- Redis 很适合处理大量短平快操作，但 big key、慢命令和重脚本仍然会拖慢它。
