# Redis - 第 4 课：持久化，RDB、AOF、混合持久化与数据恢复

## 本篇定位

主线入门层兼生产排障层。它负责回答 Redis 数据如何恢复、最多可能丢多少、持久化抖动和恢复演练怎么处理。

## 学习目标（本节结束后你能做到什么）

- 理解 Redis 为什么明明是内存数据库，还必须讨论持久化。
- 说清 RDB、AOF、混合持久化分别在解决什么问题。
- 理解快照、日志追加、AOF 重写、fork 和 Copy-On-Write 的基本逻辑。
- 能从性能、数据安全、恢复速度三个维度比较 RDB 和 AOF。
- 面对“Redis 重启后数据怎么恢复”“最多会丢多少数据”这类问题时给出有层次的回答。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么 Redis 还需要持久化

Redis 的主要数据在内存里，这带来了高性能，但也天然带来一个问题：

**内存不是长期可靠存储。**

如果机器重启、进程崩溃、容器被杀，内存内容就没了。

于是 Redis 必须回答两个问题：

- 我要不要把内存状态保存下来
- 我要以什么代价保存，才能在性能和数据安全之间平衡

这就是 Redis 持久化存在的根本原因。

### 2. RDB：某个时刻的整库快照

RDB 的思路很直观：

- 在某个时间点
- 把当前内存中的数据整体生成一个快照文件

你可以把它理解成给 Redis 做“拍照”。

好处是：

- 文件相对紧凑
- 恢复速度通常比较快
- 对日常读写的侵入相对可控

缺点也很明显：

- 两次快照之间新写入的数据如果还没来得及落盘，故障时会丢

所以 RDB 更像是在回答：

**我能不能用比较低的运行时成本，保存一个周期性的全量状态。**

### 3. AOF：把写命令按顺序记下来

AOF 的思路不是“拍照”，而是“记账”。

每次有写命令进来，Redis 都把这条写操作按协议格式追加到 AOF 文件里。将来重启时，只要把这些命令按顺序重放，就能把内存状态恢复出来。

这有点像：

- RDB 是存“某一时刻账户余额”
- AOF 是存“每笔交易流水”

AOF 的优点是：

- 数据丢失窗口通常更小
- 语义直观，本质是重放写历史

缺点是：

- 文件可能持续变大
- 恢复时要重放命令，速度可能慢于直接加载快照

### 4. AOF 的三种刷盘策略

AOF 不是每条写命令一来就必须立刻刷盘，它有不同的 `fsync` 策略，常见包括：

- 每次写都刷
- 每秒刷一次
- 交给操作系统自己决定

这背后本质是数据安全和性能的权衡：

- 刷得越勤，越安全，但写放大和延迟成本更高
- 刷得越松，性能更好，但崩溃时可能丢的数据更多

工程里很常见的选择是“每秒刷一次”，因为它在性能和可接受的数据丢失窗口之间通常比较平衡。

### 5. AOF 重写为什么需要

如果 AOF 只是不断追加写命令，文件会越来越大。

例如：

- 一个 key 被改了 1000 次
- 最终你恢复时其实只关心最后结果

于是 Redis 提供 AOF 重写机制。它不是把老日志简单压缩，而是根据当前内存状态重新生成一份“更短但能恢复到同样状态”的新 AOF。

所以 AOF 重写的本质是：

**保留最终状态恢复所需的最小写操作集合，而不是保留全部历史细节。**

### 6. fork 和 Copy-On-Write 为什么重要

无论是做 RDB 快照，还是某些 AOF 重写流程，Redis 都很依赖 `fork`。

大致逻辑是：

- 主进程继续处理请求
- 子进程基于当时内存视图去做快照或重写

之所以能这么做，是因为操作系统支持 Copy-On-Write：

- fork 后父子进程先共享同一片物理页
- 谁修改某页，才真正复制那一页

这让 Redis 能在不停机的前提下做持久化，但代价也不是没有：

- fork 本身有开销
- 内存写入较多时，COW 会放大内存占用

所以线上 Redis 机器内存规划时，不能只看业务数据量，还要给持久化过程中的额外成本留空间。

### 7. 混合持久化在解决什么问题

RDB 和 AOF 各有优缺点：

- RDB 恢复快，但丢数据窗口相对更大
- AOF 更安全，但恢复慢、文件更大

于是 Redis 后来引入混合持久化，思路是：

- 前面用 RDB 快照快速描述大部分状态
- 后面再跟上增量 AOF 日志

这样既想要：

- 像 RDB 一样恢复快
- 又想保留像 AOF 一样更小的数据丢失窗口

它的本质不是发明全新类型，而是：

**试图把“快照恢复快”和“增量日志更安全”拼在一起。**

### 8. Redis 重启时怎么恢复

恢复顺序通常取决于你启用了哪些持久化方式。

大方向上可以理解为：

- 如果有更完整、更新的数据文件，就优先用它
- Redis 读取持久化文件，把内存状态重建起来

如果只有 RDB，就直接加载快照。  
如果有 AOF，就按 AOF 内容恢复。  
如果是混合方式，就先加载 RDB 部分，再回放后面的增量操作。

所以持久化不仅影响“会不会丢数据”，也直接影响“重启恢复多快”。

### 9. 工程上怎么选

这不是一个标准答案问题，而是一个取舍问题。

如果你更在意：

- 恢复速度
- 文件紧凑
- 运行时压力相对低

那 RDB 会更讨喜。

如果你更在意：

- 数据丢失窗口尽量小
- 更强的可恢复性

那 AOF 通常更合适。

如果你两边都想兼顾，混合持久化往往是一个折中方案。

但别忘了，Redis 的“可靠性”从来不只靠持久化，还和：

- 主从复制
- 哨兵
- 集群
- 运维备份

一起构成整体方案。

## 实战落地：持久化策略要和数据等级绑定

生产里不要笼统问“RDB 好还是 AOF 好”，要先给 Redis 里的数据分等级：

- 纯缓存数据：允许丢失，重点是重启后预热和保护数据库。
- 可重建状态：允许短时间丢失，但要能从 DB、MQ、日志恢复。
- 半事实数据：例如库存预扣、延时任务、消费进度，丢失会造成业务补偿成本。

纯缓存场景可以更关注恢复速度和运行时性能；半事实数据就要考虑 AOF、备份、主从、对账和补偿。尤其是秒杀预扣、延时任务这类状态，如果只开 RDB，故障窗口内的数据丢失可能直接变成库存不准或任务漏执行。

一个比较稳的做法是：缓存 Redis 明确允许重建；基础设施 Redis 单独集群部署，开启更严格持久化和备份；不要把两类负载混在同一个实例里互相影响。

## 生产问题处理：持久化相关事故怎么查

持久化问题常见症状包括重启后数据少了、AOF 文件过大、fork 时延迟抖动、磁盘写满、复制突然变慢。处理时可以按下面检查：

- 查 `INFO persistence`：看 `rdb_last_bgsave_status`、`aof_last_bgrewrite_status`、`aof_delayed_fsync`。
- 查系统资源：fork 期间内存是否足够，磁盘 I/O 是否打满，是否发生 swap。
- 查配置：`appendfsync`、`save`、`aof-use-rdb-preamble` 是否符合数据等级。
- 查日志：是否出现 `MISCONF`、AOF rewrite 失败、RDB 保存失败。
- 做恢复演练：定期拿备份在隔离环境启动，确认文件不是“看起来存在，实际不可恢复”。

事故现场不要急着删除 AOF 或 RDB 文件。先复制现场文件、记录实例配置和日志，再决定是用 `redis-check-aof` 修复、从备份恢复，还是让业务重新回源预热。

## 小结

- Redis 需要持久化，是因为内存天然不具备长期可靠性。
- RDB 是某一时刻的全量快照，恢复快，但快照间隔内的数据可能丢失。
- AOF 是写命令追加日志，数据更安全，但文件可能更大、恢复更慢。
- AOF 重写的目标不是保留全部历史，而是用更短日志恢复相同最终状态。
- fork 和 Copy-On-Write 是 Redis 持久化的重要基础，也是线上内存规划必须考虑的因素。
