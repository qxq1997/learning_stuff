# Redis - 第 5 课：缓存设计，过期删除、内存淘汰、缓存穿透击穿雪崩与一致性

## 学习目标（本节结束后你能做到什么）

- 理解 Redis 在缓存系统里真正承担的角色，而不是只会“查不到就写缓存”。
- 说清过期删除和内存淘汰分别在解决什么问题。
- 分清缓存穿透、击穿、雪崩三者的区别。
- 对缓存一致性形成工程化理解，而不是只背几个套路名词。
- 面对缓存故障题时，能从数据流、过期流、回源流三个角度分析。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 缓存不是“多放一份数据”这么简单

很多人一提缓存，第一反应是：

- 查 Redis
- 没有就查数据库
- 再把结果写回 Redis

这个流程没错，但真正难的是：

- 什么时候过期
- 内存满了怎么办
- 热点 key 同时失效怎么办
- 缓存和数据库不一致怎么办
- 不存在的数据一直被打怎么办

也就是说，缓存系统真正难的地方不是“怎么命中”，而是“失效和故障时怎么不出事”。

### 2. 过期删除在解决什么问题

Redis 支持给 key 设置过期时间。  
但设置过期时间，不等于时间一到，Redis 就立刻把那个 key 删除。

因为如果它为了每个 key 都安排一个精确闹钟，维护成本会很高。

所以 Redis 常见的过期删除思路是：

- **惰性删除**：访问到这个 key 时，发现它已经过期了，再删
- **定期删除**：后台定期抽样扫描一批带过期时间的 key，发现过期就删

这两个策略组合起来的好处是：

- 不必为所有 key 维护高昂的精确调度
- 又不至于让大量过期键无限堆着不管

所以过期删除解决的是：

**如何在删除成本和内存回收及时性之间平衡。**

### 3. 内存淘汰在解决什么问题

过期删除处理的是“已经设置了 TTL 的 key”。  
内存淘汰处理的是另一件事：

**当 Redis 总内存到上限时，新数据还要不要写、旧数据该淘汰谁。**

这是一种容量保护机制。

常见策略包括：

- 不淘汰，直接报错
- 只从设置了过期时间的 key 里淘汰
- 按最近最少使用、最不经常使用等近似策略淘汰

所以：

- 过期删除关注的是“key 自己该不该活着”
- 内存淘汰关注的是“内存装不下了系统该怎么办”

这两个概念常被混淆，但它们不是一回事。

### 4. 缓存穿透是什么

缓存穿透指的是：

- 用户请求的数据本来就不存在
- Redis 没有
- 数据库也没有
- 每次请求都打到数据库

比如有人不断请求一个根本不存在的商品 ID。  
由于这个 ID 永远查不到，系统每次都会回源数据库。

常见解决思路有：

- 缓存空值
- 布隆过滤器
- 参数合法性校验

穿透的关键不是“缓存没命中”，而是：

**请求的目标本来就不存在，缓存层无法天然拦住。**

### 5. 缓存击穿是什么

缓存击穿指的是：

- 一个非常热点的 key
- 在某个瞬间失效了
- 大量并发同时回源数据库

这里的重点是：

- 数据是存在的
- 而且特别热点
- 只是在失效那一刻，回源流量被瞬间放大

常见解决思路有：

- 热点 key 永不过期，由后台异步刷新
- 使用互斥锁 / singleflight，限制同一时刻只有一个请求回源
- 提前续期或逻辑过期

### 6. 缓存雪崩是什么

雪崩比击穿范围更大。

它通常指：

- 大量缓存同时失效
- 或 Redis 整体不可用
- 导致大量请求集中打到下游数据库或服务

常见诱因有：

- 批量 key 设置了相同过期时间
- Redis 实例故障
- 网络抖动
- 发布导致热点预热缺失

常见应对包括：

- 过期时间加随机值
- 多级缓存
- 限流与降级
- 高可用 Redis
- 关键数据预热

所以三者可以这样记：

- 穿透：查的是不存在的数据
- 击穿：一个热点 key 瞬时失效
- 雪崩：大量 key 或整个缓存层同时出问题

### 7. 缓存一致性为什么难

这也是后端面试的高频点。

数据库和缓存各存一份，就天然可能不一致。  
例如：

1. 先更新数据库
2. 还没来得及删缓存
3. 别的请求读到了旧缓存

或者：

1. 先删缓存
2. 数据库更新失败
3. 缓存没了，数据库还是旧值

这说明一致性不是一句“更新时顺手改一下缓存”能解决的。

### 8. 最常见的缓存模式：旁路缓存

旁路缓存，也叫 Cache Aside，大致流程是：

- 读：先查缓存，未命中再查数据库并回填
- 写：先更新数据库，再删除缓存

为什么通常不是“更新数据库后顺便更新缓存”？

因为删除缓存往往更稳：

- 避免复杂序列化逻辑散落在写路径
- 避免并发下把旧值重新写回

当然，删除缓存也不是绝对一致，只是工程上通常更好控制。

### 9. 缓存设计的真正核心

缓存设计不是背几个故障名词，而是要理解三条流：

- 数据流：数据从数据库如何进入缓存
- 失效流：缓存什么时候过期、什么时候删除
- 回源流：缓存失效时，请求怎么回到下游

只要这三条流里任何一条设计得粗糙，系统就可能出现：

- 热点被打穿
- 数据不一致
- 内存耗尽
- 大量慢请求

## 小结

- 过期删除解决的是过期 key 何时回收，内存淘汰解决的是总内存不够时谁该被赶走。
- 缓存穿透、击穿、雪崩分别对应“不存在数据”“热点单 key”“大面积失效或整体故障”。
- Redis 做缓存时，真正难的不是命中，而是失效、回源和一致性。
- 旁路缓存是最常见模式，但它也不是绝对一致，只是工程上比较稳妥。
- 设计缓存时，要同时看数据流、失效流和回源流，而不是只盯着某一个命中率指标。
