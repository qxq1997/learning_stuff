# Redis - 第 7 课：分片扩展，Redis Cluster、哈希槽、重分片与扩缩容

## 本篇定位

主线入门层兼生产排障层。重点回答 Redis 如何突破单机容量和写入上限，以及 Cluster 带来的多 key、迁移和路由复杂度。

## 学习目标（本节结束后你能做到什么）

- 理解 Redis Cluster 为什么出现，它和 Sentinel 的问题域有什么不同。
- 说清哈希槽是什么，为什么 Redis Cluster 不直接用“节点数取模”。
- 理解 MOVED、ASK、重分片、扩缩容这些动作的基本含义。
- 知道 Cluster 在能力上解决了什么，又引入了哪些限制。
- 建立“高可用”和“水平扩展”是两类问题的清晰认知。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. Sentinel 解决不了什么问题

第 6 课里我们讲过，Sentinel 很适合解决：

- 主节点挂了怎么办
- 从节点谁接班

但它不解决一个更大的问题：

**单机内存和单主写能力总有上限。**

如果你的数据已经大到一台 Redis 放不下，或者单主压力已经成为瓶颈，那么“再多从节点”也没法解决。

这时问题从“高可用”变成了“横向扩展”，Redis Cluster 就是为这个问题而来的。

### 2. Redis Cluster 在做什么

Redis Cluster 的核心思想很直接：

- 把整个 key 空间拆成很多小片
- 这些小片分布到不同主节点上
- 每个主节点再挂自己的从节点做高可用

所以你可以把它理解成：

- Sentinel 时代是一套主从
- Cluster 时代是很多套主从拼成一个逻辑集群

### 3. 为什么要用哈希槽

Redis Cluster 最经典的设计就是 16384 个哈希槽。

每个 key 并不是直接“对节点数取模”决定去哪，而是：

1. 先对 key 算哈希
2. 再映射到 0 到 16383 之间的某个槽
3. 某个节点负责若干槽

也就是说，真正分布的是槽，不是 key 直接对节点取模。

为什么这样更好？

因为如果直接按节点数取模，一旦节点数变化，大量 key 的映射都会变。  
而有了槽之后：

- 槽是稳定中间层
- 你只需要把槽从一个节点迁到另一个节点
- 不需要重算整个世界

这本质上是在降低扩缩容时的重映射成本。

### 4. 客户端怎么知道 key 在哪台机器

客户端访问集群时，会根据 key 算出槽，再找到负责该槽的节点。

如果客户端访问错了节点，Redis 会返回重定向信息。

常见有两个词：

- `MOVED`
- `ASK`

你可以先粗理解为：

- `MOVED`：这个槽已经正式搬家了，以后你应该去新节点
- `ASK`：这个槽正在迁移过程中，这次先去目标节点问一下

这两个返回值的存在，正说明 Cluster 的世界不是静态的，它要支持在线迁移。

### 5. 重分片到底在做什么

扩容时，你要加机器。  
加完机器后，新的节点不能空着，所以要从旧节点上搬一部分槽给它。

缩容时，反过来，你要把待下线节点上的槽都搬走。

这就是重分片的本质：

**把槽的归属关系重新分配。**

注意，迁移的基本单位是槽，而槽下面才挂着很多 key。  
所以 Cluster 扩缩容不是“全量搬家”，而是“按槽为单位分步搬迁”。

### 6. Cluster 带来的限制是什么

Redis Cluster 不是把一切都变简单了，它在获得横向扩展能力的同时，也带来了更多约束。

最典型的是多 key 操作。

如果两个 key 不在同一个槽里，那么很多需要它们一起参与的操作就做不了，或者做起来很麻烦。

这就是为什么 Redis Cluster 里经常会看到 hash tag 设计，比如：

- `user:{1001}:profile`
- `user:{1001}:orders`

这样带同一标签的 key 会被映射到相同槽中，方便做相关操作。

所以学 Cluster 时一定要明白：

**分布式带来的不是白赚的扩展，而是扩展与约束同时到来。**

### 7. Cluster 的高可用怎么理解

每个主节点通常还会有自己的从节点。

也就是说，Cluster 不是只有分片，没有高可用；它是：

- 横向分片
- 每片内部再做主从容灾

因此 Cluster 其实同时在解决两件事：

- 数据分布到多台主节点，突破单机上限
- 某个主节点故障时，由对应从节点接管

这也是为什么它比 Sentinel 的复杂度高很多。

### 8. 什么时候该用 Cluster

如果你的主要诉求是：

- 一套 Redis 数据量已经大到单机装不下
- 单主写压力过高
- 想要更自然的横向扩展

那 Cluster 很合适。

如果你的主要诉求只是：

- 数据不算太大
- 主要担心单点故障

那 Sentinel + 主从往往更轻、更简单。

所以不要把 Cluster 理解成“更高级就一定更好”，而是：

- 它是为分片扩展付出额外复杂度换来的方案

## 实战落地：Cluster 设计要提前约束 key 模型

Redis Cluster 不是把单机 Redis 直接换成多机就完事。上线前要先审视 key 模型：

- 是否存在大量多 key 操作，例如 `MGET`、`SUNION`、事务、Lua。
- 哪些 key 必须落同一个槽，需要设计 hash tag，例如 `{user:123}:profile`、`{user:123}:cart`。
- 是否有超大 key 或热点 key，即使用 Cluster 也不会自动消失。
- 客户端是否支持 Cluster 拓扑刷新、`MOVED`、`ASK`、重试和超时控制。
- 运维上是否具备 reshard、扩容、缩容、故障恢复的脚本和演练。

Cluster 解决的是容量和吞吐横向扩展，但不能替业务重新建模。多 key 强依赖的业务如果没有 hash tag 或拆分方案，上了 Cluster 后反而会频繁遇到 `CROSSSLOT`。

## 生产问题处理：扩缩容期间重点看什么

Cluster 扩缩容或迁移槽时，要重点关注：

- `MOVED` / `ASK` 是否暴涨，客户端是否正确刷新路由。
- 迁移槽所在节点 CPU、网络、内存是否抖动。
- 是否存在大 key 导致槽迁移很慢。
- 业务错误中是否出现 `CROSSSLOT`、`TRYAGAIN`、连接超时。
- 迁移后槽位是否完整覆盖，节点主从关系是否正常。

如果迁移期间业务大量超时，优先降低迁移速率、暂停非核心批处理、限制大 key 写入，再继续迁移。Cluster 扩缩容虽然支持在线进行，但“在线”不等于“无成本”。

## 小结

- Sentinel 解决的是高可用，Cluster 解决的是横向分片扩展，并同时兼顾每片的高可用。
- Redis Cluster 用 16384 个哈希槽作为稳定中间层，降低扩缩容时的大规模重映射问题。
- MOVED 和 ASK 本质上反映了节点重定向和迁移中的不同状态。
- 重分片的核心是迁移槽，而不是简单“整库搬家”。
- Cluster 带来了扩展能力，也带来了多 key 操作和访问模型上的约束。
