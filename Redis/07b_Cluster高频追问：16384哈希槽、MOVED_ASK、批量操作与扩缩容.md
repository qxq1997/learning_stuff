# Redis - Cluster 高频追问：16384 哈希槽、MOVED / ASK、批量操作与扩缩容

## 本篇定位

生产排障层和面试追问层。它专门处理 Cluster 深水区问题：槽位、重定向、批量操作、Lua 限制和扩缩容告警。

## 学习目标

- 不只知道 Cluster“能分片”，还知道它到底怎么分。
- 能解释为什么是 16384 个槽，而不是别的数字。
- 分清 `MOVED` 和 `ASK`。
- 理解 Redis Cluster 下批量操作、Lua、事务为什么会出现限制。

## 内容讲解

### 1. Cluster 到底解决什么问题

Sentinel 解决高可用，但一个 master 再怎么切，也还是单点容量上限。

Cluster 解决的是：

- 数据分片
- 横向扩容
- 集群级高可用

所以 Cluster 的关键词不是“自动切主”，而是：

**分片。**

### 2. Redis Cluster 是怎么分片的

Redis Cluster 没有用一致性哈希环，而是用了哈希槽。

总共有 `16384` 个 slot。

每个 key 会先算：

```text
CRC16(key) % 16384
```

算出它属于哪个槽，再由槽映射到具体节点。

所以本质上不是：

- key -> 节点

而是：

- key -> slot -> 节点

### 3. 为什么是 16384 个槽

这题很高频。

你不必背源码八股，但要抓住两个核心点：

1. 槽太少，分布不够细，迁移不够灵活
2. 槽太多，元数据传播和管理成本更高

16384 是 Redis 在：

- 分片颗粒度
- 管理成本
- 协议包大小

之间做的一个工程折中。

不是神奇数字，而是平衡后的结果。

### 4. `MOVED` 和 `ASK` 有什么区别

这是 Cluster 的经典追问。

#### `MOVED`

表示：

**这个 slot 的归属已经稳定变了。**

客户端应该更新自己本地的 slot -> node 映射。

#### `ASK`

表示：

**这个 slot 正在迁移过程中，你这次先去另一个节点问一下，但映射关系还没彻底稳定。**

所以：

- `MOVED`：永久性重定向
- `ASK`：迁移过程中的临时重定向

### 5. 为什么 Cluster 下批量操作会变复杂

像 `MGET`、`MSET`、事务、Lua 这些能力，在单机 Redis 里很自然。

但到了 Cluster，会立刻遇到一个问题：

**这些 key 可能根本不在同一个节点。**

这时你就会碰到限制：

- 不能保证一次命令跨多个节点还能保持原子性
- 客户端可能得拆请求

这也是为什么 Cluster 下经常会强调：

**多 key 操作最好让 key 落在同一个 hash slot。**

### 6. 哈希标签是干什么的

Redis Cluster 提供了哈希标签，也就是 `{}`。

例如：

- `order:{1001}:info`
- `order:{1001}:status`

虽然 key 不一样，但只要大括号里的内容一样，算槽时只看这部分。

这样多个相关 key 就能稳定落在同一个 slot。

这对什么很重要？

- 多 key 操作
- Lua
- 事务
- 一组业务对象的局部聚合

### 7. 扩容缩容时可以不停机吗

可以，Cluster 设计上就是支持在线重分片的。

但“可以在线”不等于“没有影响”。

因为迁移期间会有：

- slot 迁移
- 重定向
- 大 key 迁移带来的抖动

所以工程上要注意：

- 尽量避开高峰
- 控制迁移节奏
- 提前治理 bigkey

### 8. 节点之间怎么通信

Cluster 节点之间会通过 cluster bus 互相交换状态信息，比如：

- 槽位归属
- 节点健康状态
- 故障检测

所以 Cluster 不是“客户端自己拼出来的一堆 Redis”，而是：

**节点之间自己也在形成一个集群视图。**

### 9. Cluster 的边界

Cluster 非常适合：

- 大数据量
- 横向扩展
- 分布式缓存

但也会带来一些限制：

- 多 key 操作受限
- 事务能力受限
- Lua 受 slot 约束
- 运维复杂度更高

所以不是所有场景都该直接上 Cluster。

如果你只是：

- 数据量不大
- 更关注高可用而不是扩容

Sentinel 可能就够了。

## 实战落地：把 Cluster 限制前置到开发规范

Cluster 的很多坑不是运维阶段才出现，而是开发阶段没有规范：

- 多 key 命令默认要求同槽，跨槽批量要在客户端拆分并合并结果。
- Lua 脚本里访问的 key 必须显式声明，并且尽量落在同一个 slot。
- hash tag 只能包住真正需要同槽的部分，不能滥用到所有 key 上，否则会制造热点槽。
- 批量操作要控制每个 slot 的单批大小，避免某个节点被瞬间打满。
- 业务代码要能识别重定向和临时失败，不能把 `MOVED` 当普通错误抛给用户。

这类规则最好沉到 Redis 客户端封装层，而不是靠每个业务同学临场记忆。

## 生产问题处理：Cluster 常见告警怎么判断

如果线上出现 Cluster 相关错误，可以先这样分流：

- `MOVED` 很多：通常是客户端路由缓存没及时刷新，或拓扑频繁变化。
- `ASK` 很多：通常处在槽迁移过程中，观察是否符合扩缩容窗口。
- `CROSSSLOT`：业务用了跨槽多 key 命令，要改 hash tag 或拆请求。
- `TRYAGAIN`：迁移或故障恢复期间临时不可用，客户端应短暂重试。
- 某个 slot 特别热：hash tag 可能过度集中，或者业务天然热点没有拆散。

排障时不要只看单个节点 QPS，要看 slot 分布和 key 访问倾斜。Cluster 最怕“节点很多，但热点都在一个槽里”。

## 小结

- Redis Cluster 通过 16384 个哈希槽来完成分片。
- key 的目标节点不是直接算出来的，而是先映射到 slot，再由 slot 映射到节点。
- `MOVED` 是稳定重定向，`ASK` 是迁移过程中的临时重定向。
- Cluster 支持在线扩容缩容，但迁移期间并非完全无感。
- Cluster 的代价是多 key 操作、事务、Lua 的使用边界明显增强。
