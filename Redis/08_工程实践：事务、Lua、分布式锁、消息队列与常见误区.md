# Redis - 第 8 课：工程实践，事务、Lua、分布式锁、消息队列与常见误区

## 本篇定位

主线入门层。它把 Redis 从“缓存工具”推进到“工程基础设施”，用于建立锁、限流、Lua、轻量队列等能力的边界感。

## 学习目标（本节结束后你能做到什么）

- 理解 Redis 事务、Lua、分布式锁、消息队列这些常见用法各自在解决什么问题。
- 分清 Redis 的“原子性”“事务性”和数据库事务不是同一个概念。
- 知道为什么 Lua 在 Redis 工程实践里非常重要。
- 对 Redis 分布式锁的适用边界有清醒认知。
- 能识别 big key、hot key、慢命令、滥用 `KEYS` 等典型线上误区。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. Redis 事务在解决什么问题

Redis 提供了 `MULTI`、`EXEC`、`WATCH` 这些能力，但它的事务和数据库事务不是一回事。

Redis 事务更像是在说：

- 把一批命令放进队列
- 到 `EXEC` 时按顺序执行

它解决的是“一组操作按顺序执行，不被其他命令插入”的问题，而不是关系型数据库那种复杂隔离级别、多语句回滚语义。

所以 Redis 事务的理解重点应该是：

- 它比散着发命令更有组织
- 但它不是拿来替代数据库强事务的

### 2. WATCH 在做什么

`WATCH` 本质上是乐观锁思路。

它的逻辑是：

- 先监视某个 key
- 如果事务提交前这个 key 被别人改了
- 那本次事务就放弃执行

这个设计很适合：

- 并发条件判断
- 更新前先校验版本或状态

但它也意味着你得在应用层处理失败重试。

### 3. Lua 为什么这么重要

Redis 虽然命令丰富，但很多业务逻辑不是单条命令能表达的。

例如：

- 先判断库存是否足够，再扣减，再记录用户
- 先判断锁值是否匹配，再删除锁

如果你把这些步骤拆成多次网络往返，就会出现竞态条件。  
Lua 脚本的价值就在于：

- 把多步逻辑放进 Redis 里一次执行
- 避免多次网络往返
- 保持脚本执行期间的原子性

所以 Lua 在 Redis 世界里的角色，和“把一小段关键逻辑下沉到服务端执行”很像。

### 4. 分布式锁为什么常用 Redis

分布式系统里，如果多个实例要竞争某个资源，就需要锁。  
Redis 因为有：

- 高性能
- 原子命令
- 过期时间

所以常被拿来做分布式锁。

最经典的思路是：

- `SET lock_key value NX PX 30000`

含义大致是：

- 只有 key 不存在时才能设置成功
- 同时给锁一个过期时间，避免死锁

释放锁时又不能直接 `DEL`，而要先校验 value 是否还是自己写进去的那个，再删除，这通常会通过 Lua 保证原子性。

### 5. Redis 锁的边界

Redis 分布式锁很常用，但不能神化。

它特别适合：

- 控制某些短时资源互斥
- 减少重复执行
- 做轻量协调

但它不适合被理解成“分布式世界里的万能强一致锁”。

因为一旦涉及：

- 网络分区
- 锁续期
- 业务执行时间超出 TTL
- 主从切换导致状态不一致

事情就会复杂很多。

所以面试里如果被问到 Redis 锁，成熟答案通常不是“会写命令”，而是能主动讲它的适用边界和风险。

### 6. Redis 能不能做消息队列

答案是：能做一部分，但不能把所有 MQ 需求都往上套。

#### 用 List

优点：

- 简单
- 适合轻量生产消费

缺点：

- 能力有限
- 缺少成熟消费组语义

#### 用 Pub/Sub

优点：

- 实时推送感强

缺点：

- 更像广播
- 掉线期间消息通常不会为你保留

#### 用 Stream

Stream 是 Redis 相对更像“消息系统”的能力，支持：

- 消息追加
- 消费组
- 待确认消息管理

所以如果你只是做一个轻量队列，Redis 可以胜任；如果你要的是大规模、高可靠、复杂路由消息系统，Kafka 这类专业 MQ 通常更合适。

### 7. 线上最常见的 Redis 误区

#### 误区一：滥用 big key

比如：

- 一个 key 下面挂几十万元素
- 一个 String 存超大 JSON

结果就是：

- 单次操作很重
- 删除和序列化成本高
- 容易阻塞主线程

#### 误区二：线上直接 `KEYS *`

这类命令会扫描大量数据，极易阻塞 Redis。线上应该更谨慎，通常更推荐 `SCAN` 一类渐进式遍历思路。

#### 误区三：把 Redis 当低成本数据库

内存昂贵、冷热分层差、复杂查询弱，这些边界如果不看清，项目后期会很痛苦。

#### 误区四：只关注命中率，不关注回源放大

真正线上危险的常常不是“命中率低一点”，而是：

- 热点失效
- 回源风暴
- 下游打挂

这和第 5 课的缓存设计是强关联的。

### 8. Redis 工程实践的核心思维

Redis 用得好不好，核心不是“会多少命令”，而是三点：

- 能不能选对抽象
- 能不能看清边界
- 能不能在性能和一致性之间做现实取舍

所以一个成熟后端工程师看 Redis，应该同时从：

- 数据结构
- 访问模式
- 故障模式
- 运维模式

四个角度去理解它。

## 实战落地：把 Redis 能力包装成业务原语

工程里不要让业务代码到处裸写 Redis 命令，尤其是锁、限流、消息和 Lua。更稳的做法是封装成明确的业务原语：

- `LockService`：统一设置唯一 value、TTL、续期、释放校验和失败降级。
- `RateLimiter`：统一 Lua 脚本、窗口语义、限流 key 和监控埋点。
- `CacheClient`：统一 TTL 抖动、空值缓存、回源保护、删除重试。
- `RedisQueue`：明确是否支持确认、重试、死信、积压告警。

这样做的价值不是“多一层封装”，而是把 Redis 的边界和生产经验固化下来。否则同一个项目里可能出现五种锁写法、三种限流语义，出事故时很难判断哪一种出了问题。

## 生产问题处理：基础设施能力要有兜底路径

Redis 被当基础设施用时，故障影响面会比普通缓存大。上线前要为每种能力准备兜底：

- 分布式锁失败：业务是否有数据库唯一约束或幂等表兜底。
- 限流失败：网关或应用本地是否有粗粒度限流兜底。
- Lua 执行异常：是否能快速回滚脚本 SHA 或降级到简单逻辑。
- 轻量队列积压：是否有消费积压告警、重试上限和人工补偿入口。
- 缓存删除失败：是否有异步重试和定期校验。

Redis 工程实践的成熟度，最终看的是“Redis 不稳定时，业务还能不能保持可控退化”。

## 小结

- Redis 事务更像命令队列和乐观控制，不是数据库那套完整强事务语义。
- Lua 的价值在于把多步逻辑下沉到 Redis 内部一次原子执行，减少竞态和网络往返。
- Redis 分布式锁很实用，但不是万能强一致锁，必须清楚它的适用边界。
- Redis 可以承接轻量消息场景，但不能想当然替代专业 MQ。
- 真正的 Redis 工程能力，体现在识别 big key、慢命令、误用模式和系统边界。
