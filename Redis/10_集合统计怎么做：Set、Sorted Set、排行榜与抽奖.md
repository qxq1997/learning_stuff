# Redis - 第 10 课：集合统计怎么做：Set、Sorted Set、排行榜与抽奖

## 本篇定位

场景实战层。重点把 Set 和 ZSet 放进去重、抽奖、排行榜、延时任务等高频业务模型里。

## 学习目标

- 真正理解 `Set` 和 `Sorted Set` 的区别，而不是只背“一个无序、一个有序”。
- 能把去重、共同好友、抽奖、排行榜这些题，落成 Redis 数据结构设计。
- 理解为什么 `ZSet` 要同时维护字典和跳表。
- 明白哪些排行榜 Redis 很擅长，哪些复杂统计其实不该硬做。

## 内容讲解

### 1. `Set`：重点不是“集合”，而是“唯一性”

Redis 的 `Set` 本质上是：

- 元素无序
- 元素唯一
- 支持交集、并集、差集

所以它最适合解决的问题，不是“我要一个列表”，而是：

- 不能重复
- 要做集合关系运算
- 要随机抽取

例如点赞用户列表：

```text
SADD post:1001:likes user:1 user:2 user:3
SISMEMBER post:1001:likes user:2
SCARD post:1001:likes
```

它天然就比 `List` 更适合“点赞去重”这种场景。

### 2. `Set` 最常见的三个业务模型

#### 2.1 去重

- 文章点赞用户
- 今日签到用户
- 某活动已报名用户

#### 2.2 集合关系

- 共同好友：交集
- 共同关注：交集
- 推荐关注：差集

例如：

```text
SINTER user:1:follows user:2:follows
SDIFF user:1:follows user:2:follows
```

#### 2.3 随机抽样

抽奖系统经常就能直接落在 `Set` 上。

```text
SADD lottery:20260412 u1 u2 u3 u4
SPOP lottery:20260412 2
```

`SPOP` 会随机弹出元素，很适合“中奖后不能再中”的场景。

如果只是随机看看，不想移除，可以用 `SRANDMEMBER`。

### 3. `Set` 的边界在哪里

不要看到“唯一 + 快”就什么都放 `Set`。

它不擅长：

- 需要排序
- 需要范围查询
- 需要按分数取 TopN
- 需要保留插入顺序

所以：

- 点赞去重适合 `Set`
- 点赞排行榜不适合 `Set`

### 4. `Sorted Set`：重点不是“集合”，而是“分数排序”

`Sorted Set`，也就是 `ZSet`，相比 `Set` 多了一个 `score`。

所以它解决的是：

**元素唯一，但每个元素还有一个可排序的分数。**

例如排行榜：

```text
ZADD rank:game 11071 user1 9907 user2 9793 user3
ZREVRANGE rank:game 0 9 WITHSCORES
ZREVRANK rank:game user2
ZSCORE rank:game user2
```

一旦你要回答下面这些问题，基本就该想到 `ZSet`：

- Top 10 是谁
- 某人当前第几名
- 某人的分数是多少
- 某个分数区间内有哪些人

### 5. 排行榜为什么这么适合 `ZSet`

因为排行榜场景本质上就有两个核心诉求：

1. 通过 member 快速找到这个人
2. 按 score 排序后快速取范围

Redis 的 `ZSet` 底层正是围绕这两个诉求做的：

- 一份字典：用于 member -> score 的快速查找
- 一份跳表：用于按 score 有序排列和范围查询

这也是经典面试题：

**为什么 ZSet 不是只用跳表，或者只用哈希表？**

答案很简单：

- 只用哈希表，按分数排序会很痛苦
- 只用跳表，按 member 找元素又会变慢

两份结构配合，分别承担“按人查”和“按分数查”。

### 6. 为什么跳表特别适合 Redis

这里不展开跳表全部细节，但要抓住 Redis 为什么偏爱它：

- 实现比平衡树简单
- 插入 / 删除 / 查找都能做到 `O(log n)`
- 范围查询非常自然
- 在内存数据库里，它足够快、足够简单、足够实用

所以 Redis 选跳表，本质上不是“它最学术最强”，而是：

**它在实现复杂度和性能之间，给了 Redis 很好的平衡。**

### 7. 典型排行榜设计怎么讲

这是面试里特别高频的问题。

比如直播打赏榜：

```text
key = room:1001:rank
member = userId
score = giftScore
```

用户送礼时：

```text
ZINCRBY room:1001:rank 100 user:888
```

取前 10 名：

```text
ZREVRANGE room:1001:rank 0 9 WITHSCORES
```

取某个用户名次：

```text
ZREVRANK room:1001:rank user:888
```

如果要做“周榜”“月榜”，通常不是把所有周期都混在一个 key 里，而是分 key：

- `rank:room:1001:day:20260412`
- `rank:room:1001:week:2026w15`

### 8. 排行榜的真实工程问题

别把排行榜理解成“一个 ZSet 就结束了”。

真实系统里常见追问有：

- 数据是否永久累积
- 什么时候清榜
- 只保留前 N 名还是保留全量
- 排名变化是否要推送
- 同分怎么处理

Redis 可以很好地做“在线排序层”，但它不一定是“最终统计层”。

很多系统会这样分工：

- Redis：实时排行榜
- MySQL / OLAP：离线统计与最终归档

如果面试官继续追问“亿级用户、好友榜、分段榜、全局精确排名和近似排名怎么做”，这已经从 Redis 命令题升级成系统设计题。可以继续看 `系统设计/44_排行榜系统真题模拟.md`，重点讲清单个超大 `ZSet` 的 big key 风险、Redis Cluster 不能拆 key 内部元素、分段榜、分片排行榜、TopN 合并和长尾近似排名。

### 9. `Set` 和 `ZSet` 怎么选

你可以这样理解：

| 场景 | 选型 |
| --- | --- |
| 只关心是否存在、是否重复 | `Set` |
| 要做交并差 | `Set` |
| 要随机抽取 | `Set` |
| 要排序、排名、TopN | `ZSet` |
| 要根据分数做范围查询 | `ZSet` |

最常见误区是：

“我有一组用户，就都放 Set。”

不对。你要先问：

- 这组用户是否要排序？
- 是否要维护分值？
- 是否要查询名次？

如果答案是要，那就是 `ZSet` 的领域。

## 实战落地：排行榜要先定义业务口径

排行榜不是简单 `ZINCRBY` 加 `ZRANGE`。上线前要先问清楚：

- 排行榜周期是实时、小时榜、日榜、周榜，还是活动总榜。
- 分数是否允许重复，重复时按什么二级规则排序。
- 是否需要展示“我的排名”，这会影响是否高频调用 `ZRANK`。
- 是否需要分赛区、分租户、分活动，key 要按维度拆。
- 榜单结束后是否要固化结果，避免后续写入改变历史排名。

抽奖也是一样，`Set` 可以做候选人去重和随机抽样，但如果涉及奖品库存、中奖记录、黑名单、风控规则，Redis 只能承接高频状态，最终中奖事实要落 DB，并且要有唯一约束防止重复中奖。

## 生产问题处理：集合类 key 的风险

`Set` / `ZSet` 常见生产问题是成员过多、范围查询过大、热点榜单打满单节点：

- 成员过多：按时间、活动、区域拆 key，设置历史归档策略。
- 大范围查询：禁止一次取全榜，统一分页，并限制最大页深。
- 热点榜单：本地缓存 TopN、读写分离、拆分维度，必要时异步聚合。
- 分数异常：保留写入流水，榜单分数要能重算。
- 删除卡顿：大集合删除用 `UNLINK` 或分批 `ZSCAN` 清理。

排障时不要只看榜单 key 的大小，还要看访问模式。一个不大的 ZSet 如果每秒被大量 `ZRANGE 0 1000` 打，也会变成热点问题。

## 小结

- `Set` 的核心价值是去重和集合运算，不是“存一批元素”这么简单。
- `Sorted Set` 的核心价值是“唯一元素 + 可排序分数”。
- 抽奖、共同好友、点赞去重这类题，优先想 `Set`。
- 排行榜、热度榜、积分榜、TopN 这类题，优先想 `ZSet`。
- `ZSet` 底层同时维护字典和跳表，是为了兼顾按 member 查找和按 score 排序。
