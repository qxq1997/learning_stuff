# Redis - 第 13 课：事务、Pipeline 与 Lua：为什么 Redis 事务很鸡肋

## 学习目标

- 说清楚 Redis 事务到底提供了什么，不提供什么。
- 分清 `MULTI/EXEC`、`Pipeline`、`Lua` 分别在解决什么问题。
- 理解为什么很多项目里几乎不用 Redis 事务，而更偏向 Pipeline 或 Lua。
- 面试时能回答“Redis 事务为什么鸡肋”“为什么不支持回滚”。

## 内容讲解

### 1. Redis 事务的本质：打包顺序执行，不是数据库事务

Redis 事务通常由下面几个命令组成：

- `MULTI`
- `EXEC`
- `DISCARD`
- `WATCH`

最直观的理解是：

**Redis 事务不是让你获得 MySQL 那种 ACID 事务，而是让你把一批命令排队，最后按顺序执行。**

例如：

```text
MULTI
SET a 1
INCR b
EXEC
```

在 `EXEC` 之前，这些命令只是入队，不会真正执行。

### 2. Redis 事务能保证什么

它最核心能保证的是：

- 入队后的命令会按顺序执行
- 执行期间不会被其他客户端命令插进中间

这意味着它有一种“串行隔离感”。

但注意，这不等于关系型数据库那种完整事务语义。

### 3. Redis 事务不能保证什么

#### 3.1 不支持回滚

这是最经典的追问。

如果事务里的某条命令运行时报错，前面已经成功执行的命令不会回滚。

例如：

```text
MULTI
SET x 1
LPOP x
SET y 2
EXEC
```

如果 `x` 实际上不是 list，那么 `LPOP x` 会报类型错误，但：

- `SET x 1` 已经生效
- `SET y 2` 仍然会继续执行

所以 Redis 事务不满足我们熟悉的“要么都成功，要么都失败”。

#### 3.2 不能自动解决命令间依赖

Redis 事务只是排队执行，不会把前一条命令的运行结果自动喂给后一条命令。

所以如果你的后续命令依赖前一条命令的结果，Redis 事务并不优雅。

### 4. Redis 为什么不支持回滚

这个问题经常会被问。

站在 Redis 的设计立场，核心逻辑是：

1. Redis 更强调简单和性能
2. 大多数运行错误本质上是程序错误，不是 Redis 想帮你兜底的那种业务异常
3. 一旦支持回滚，就要记录前镜像 / 后镜像，复杂度和成本都会显著上升

所以 Redis 的态度是：

**我保证队列式串行执行，但不帮你做数据库那种完整回滚语义。**

### 5. `WATCH` 在干什么

`WATCH` 可以理解成一种很轻量的乐观锁。

你先 watch 某个 key：

```text
WATCH stock:1001
MULTI
DECR stock:1001
EXEC
```

如果在 `EXEC` 前，这个 key 被别的客户端改了，整个事务就不会执行成功。

所以 `WATCH` 不是锁住 key，不是阻塞别人，而是：

**先监视，提交时再检查有没有被改过。**

这和 CAS 的味道很像。

### 6. 为什么说 Redis 事务很鸡肋

因为它卡在一个尴尬位置：

- 比单条命令强一点
- 但远没有数据库事务强

实际开发里你常常会发现：

- 如果只是想减少网络往返，用 `Pipeline` 更合适
- 如果想把多条命令作为一个原子执行单元，Lua 更合适
- 如果想做真正复杂事务，Redis 本来就不是最适合的地方

所以大家常说 Redis 事务鸡肋，不是它完全没用，而是：

**它刚好不能很好地解决大家最想让“事务”解决的那些问题。**

### 7. Pipeline 解决的是“网络成本”，不是事务语义

Pipeline 和事务经常被混淆。

Pipeline 的核心价值是：

**一次发多条命令，减少 RTT。**

也就是说，它关注的是网络效率，而不是事务一致性。

例如：

- 不用一条命令一次往返
- 可以一批命令一起发给 Redis
- Redis 再顺序处理并返回一批结果

所以 Pipeline 非常适合：

- 批量写
- 批量读
- 多条命令彼此独立

但它不保证原子性，也不适合有强依赖关系的命令链。

### 8. Lua 解决的是“把多条命令当一条命令执行”

Lua 是 Redis 里非常重要的增强能力。

它的关键价值有两个：

1. 多条命令只做一次网络往返
2. 脚本执行期间不会被其他命令插入

所以 Lua 往往比 Redis 事务更像大家想象中的“原子块”。

例如分布式锁释放，经常就是：

- 先判断 value 是否还是自己
- 再删除 key

这两个动作如果拆成两条普通命令，中间会有竞态。

Lua 正好能把它们包成一个整体。

但别过度神化 Lua：

- Lua 出错后，前面已经执行的写操作也不会自动回滚
- 在 Cluster 下也要考虑所有 key 是否在同一 hash slot

### 9. 三者怎么选

可以直接记成一张表：

| 工具 | 主要解决什么 | 适合什么 |
| --- | --- | --- |
| Redis 事务 | 一批命令串行排队执行 | 简单的队列式提交 |
| Pipeline | 减少网络往返 | 批量独立命令 |
| Lua | 把多条命令当一个逻辑块执行 | 有条件判断、有依赖、有并发竞态 |

所以一个很实用的判断方式是：

- 你只是嫌 RTT 高：用 `Pipeline`
- 你需要条件判断和原子逻辑：用 `Lua`
- 你只是想把几条命令打包顺序执行：才考虑 Redis 事务

## 小结

- Redis 事务不是关系型数据库事务，本质是命令排队后顺序执行。
- Redis 不支持回滚，因此事务并不满足严格原子性。
- `WATCH` 是乐观锁思路，不是悲观锁。
- `Pipeline` 解决的是网络开销，不是事务一致性。
- `Lua` 往往比 Redis 事务更适合承载“原子逻辑块”。
- 所谓 Redis 事务鸡肋，是因为它刚好不能很好满足很多人对“事务”的期待。
