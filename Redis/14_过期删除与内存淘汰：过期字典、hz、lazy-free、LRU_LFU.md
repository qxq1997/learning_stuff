# Redis - 第 14 课：过期删除与内存淘汰：过期字典、hz、lazy-free、LRU/LFU

## 本篇定位

原理深挖层兼生产排障层。它专门解释 key 过期、内存淘汰、lazy-free 和 TTL 抖动这些缓存稳定性问题。

## 学习目标

- 分清“key 过期了”和“Redis 内存满了被淘汰”是两回事。
- 理解 Redis 过期字典、惰性删除、定期删除各自在做什么。
- 理解 `hz`、`dynamic-hz`、`lazy-free` 对性能的影响。
- 能根据业务场景说出常见淘汰策略的适用边界。

## 内容讲解

### 1. 很多人第一次就混掉的两个概念

Redis 面试里一个很常见的误区是：

- 过期删除
- 内存淘汰

被混成一件事。

实际上它们是两个完全不同的问题。

#### 过期删除

针对的是：

**这个 key 到期了，它理论上已经不该再用了。**

#### 内存淘汰

针对的是：

**Redis 内存已经达到 `maxmemory` 了，必须腾地方。**

也就是说：

- 过期是业务时间语义
- 淘汰是资源压力语义

### 2. Redis 怎么知道一个 key 什么时候过期

Redis 不是把过期时间直接硬编码在 key 本体里，而是维护了一张专门的“过期字典”。

可以粗略理解成：

- 主字典：保存 key -> value
- 过期字典：保存 key -> 过期时间戳

所以一个 key 是否设置了过期时间，本质上是看它在不在 `expires` 这张表里。

这也是为什么 Redis 检查过期 key 很快：查字典本身就是 `O(1)` 级别。

### 3. 为什么 key 到期了，不一定立刻消失

这是 Redis 使用者最容易困惑的地方之一。

你给 key 设了 60 秒 TTL，不代表 60 秒一到 Redis 就会立刻同步删掉它。

因为 Redis 并没有给每个 key 都配一个精确闹钟。

那样做成本太高：

- key 太多时，定时器本身就很重
- 大量 key 同时到期时会把 CPU 打爆

所以 Redis 采取的是组合策略。

### 4. Redis 的过期删除策略：惰性删除 + 定期删除

#### 4.1 惰性删除

当你访问一个 key 时，Redis 顺手检查：

- 还没过期：正常返回
- 已经过期：删掉并返回空

优点：

- 对 CPU 很友好

缺点：

- 如果这个 key 一直没人访问，它可能会在内存里躺很久

#### 4.2 定期删除

Redis 还会周期性地抽样检查部分设置了过期时间的 key。

发现过期了就删掉。

它不是全量扫，而是随机抽样，因为全量扫太贵了。

所以 Redis 的思路是：

**通过“访问时顺手删”和“后台定期抽查删”两种办法，平衡 CPU 和内存。**

### 5. `hz` 在这里扮演什么角色

`hz` 可以理解成 Redis 内部定期任务执行频率的一个节拍器。

默认通常是 `10`，也就是每秒钟大约做 10 次这种周期性检查。

它影响的不只是过期 key 清理，还包括一些其他后台定期任务。

如果你把 `hz` 调高：

- 过期 key 会被更积极地清理
- 但 CPU 消耗也会上升

所以它不是“越大越好”，而是一个权衡。

官方通常也不建议把它调得特别离谱。

### 6. 为什么会出现“大量 key 同时过期卡顿”

这是线上很真实的问题。

如果你有一批 key 都设成同样的 TTL，比如都在整点过期，就可能在某个时刻集中触发删除压力。

后果包括：

- 主线程花更多时间清理过期 key
- 正常命令响应变慢
- 极端时造成抖动

所以工程上一个很重要的经验是：

**给缓存过期时间加随机值，尽量打散过期时间。**

### 7. `lazy-free` 是在解决什么问题

删除一个 key，不一定慢。

慢的是：

**释放这个 key 背后那一大坨内存。**

尤其是 bigkey。

所以 Redis 后来引入 `lazy-free`，核心思路是：

- 主线程先把 key 从主结构摘掉
- 真正耗时的内存释放交给后台线程慢慢做

这样就能减少主线程被大对象删除卡住的概率。

这个思路非常重要，因为它体现了 Redis 一贯的风格：

**主线程尽量只做必须实时完成的事，重活脏活往后台挪。**

### 8. 内存淘汰又是另一套机制

当 Redis 设置了 `maxmemory`，并且内存真的打满时，就会触发淘汰策略。

这时 Redis 面临的问题不是“这个 key 该不该过期”，而是：

**我必须删点东西，不然新数据进不来。**

常见策略可以先分两大类理解：

#### 8.1 `volatile-*`

只从设置了过期时间的 key 中挑牺牲品。

#### 8.2 `allkeys-*`

所有 key 都可能被淘汰。

然后再按规则细分：

- `lru`：最近最少使用
- `lfu`：最近最不常用
- `random`：随机
- `ttl`：优先淘汰快过期的

### 9. LRU 和 LFU 到底怎么理解

#### LRU

谁最近最久没被访问，谁更可能被淘汰。

适合：

- 热点变化比较快
- 最近访问行为更能代表价值

#### LFU

谁访问频率长期偏低，谁更可能被淘汰。

适合：

- 热点比较稳定
- 你更关心“长期热度”而不是“最近一下”

一个非常通俗的理解是：

- LRU 更看“最近”
- LFU 更看“长期”

### 10. 实际项目里怎么选

如果 Redis 主要就是做缓存，通常更常见的是：

- `allkeys-lru`
- `allkeys-lfu`

原因很简单：

- 真正热点未必都设置了 TTL
- 只在过期键里选牺牲品有时不够灵活

但如果你要求“只有可过期缓存可以被踢掉，核心常驻 key 不能碰”，那就会更倾向 `volatile-*`。

### 11. 一条非常重要的工程结论

过期删除和淘汰策略都不是为了“绝对完美”，而是为了：

**在内存、CPU、响应时间三者之间做平衡。**

所以你在 Redis 上看到的很多“为什么不直接……”问题，答案经常都是：

“因为那样太贵。”

## 实战落地：TTL 和淘汰策略要按数据类型分层

生产缓存不要所有 key 都一个 TTL。更合理的做法是按业务分层：

- 热点核心数据：较长 TTL，加主动刷新或逻辑过期。
- 普通详情数据：中等 TTL，加随机抖动。
- 空值缓存：短 TTL，避免把不存在数据长期固化。
- 临时状态：明确短 TTL，并由业务兜底过期后的行为。
- 不允许被淘汰的数据：最好不要和普通缓存混同实例。

淘汰策略也一样。如果实例里全是可淘汰缓存，`allkeys-lru` 或 `allkeys-lfu` 更自然；如果只有带 TTL 的缓存可以被淘汰，才考虑 `volatile-*`。不要把锁、限流、Session 和普通缓存随便混在一个 `allkeys-*` 实例里，否则内存压力下可能淘汰掉关键状态。

## 生产问题处理：内存和过期抖动怎么救

常见症状和处理动作：

- 内存逼近 `maxmemory`：先看大 key、key 数量、TTL 覆盖率，再决定扩容或清理。
- evicted keys 增长：说明已经触发淘汰，要确认被淘汰的数据是否业务可接受。
- 大量 key 同时过期：给 TTL 加随机抖动，热点 key 改逻辑过期或后台刷新。
- 删除大 key 卡顿：用 `UNLINK`、`lazyfree-lazy-expire`、分批扫描删除。
- CPU 被过期扫描打高：检查是否大量短 TTL key，必要时调整数据模型和写入频率。

事故时不要只临时加内存。加内存能止血，但如果 key 模型和 TTL 策略不改，下一次流量峰值还是会把实例推到同一个边界。

## 小结

- 过期删除和内存淘汰不是一回事。
- Redis 通过过期字典记录 key 的过期时间。
- 过期 key 的处理采用惰性删除 + 定期删除组合策略。
- `hz` 会影响定期任务执行频率，调高会更积极清理，但也更耗 CPU。
- `lazy-free` 通过后台线程异步释放大对象内存，减少主线程阻塞。
- 当内存达到 `maxmemory` 时，才会触发内存淘汰策略。
- `LRU` 更看最近访问，`LFU` 更看长期访问频率。
