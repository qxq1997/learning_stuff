# Redis - 第 16 课：缓存故障专题：穿透、击穿、雪崩、旁路缓存与一致性

## 本篇定位

场景实战层兼生产排障层。它是第 5 课缓存总览的专项展开，重点处理缓存事故、回源保护和一致性补偿。

## 学习目标

- 真正分清缓存穿透、击穿、雪崩，而不是三者混着背。
- 理解 Cache Aside 为什么常用，以及它为什么天然带来一致性问题。
- 能把空值缓存、布隆过滤器、互斥锁、预热、随机 TTL、重试补偿讲成一套完整方案。
- 面试时不只会说“加 Redis”，还能说清缓存失败时系统怎么活下来。

## 内容讲解

### 1. 先别急着背方案，先分清故障类型

这三类问题经常被一起问，但根因完全不同。

#### 缓存穿透

请求的 key：

- Redis 里没有
- 数据库里也没有

所以请求永远穿过缓存打到数据库。

#### 缓存击穿

请求的 key：

- 数据库里有
- Redis 里暂时没有，通常是热点 key 过期了

于是大量请求在某个瞬间一起打到数据库。

#### 缓存雪崩

不是某一个 key 的问题，而是：

- 一大批 key 同时失效
- 或者 Redis 整体不可用

大量请求一起掉到数据库。

一句话：

- 穿透：查不存在的数据
- 击穿：热点单 key 失效
- 雪崩：大面积缓存失效或缓存服务宕机

### 2. 缓存穿透为什么危险

因为它意味着缓存这一层根本没挡住。

如果有人故意构造大量非法 key：

- Redis 没有
- DB 也没有

那数据库就会变成第一线承压者。

最基础的思路其实不是 Redis，而是：

**先做参数校验。**

很多非法请求，压根不该进缓存逻辑。

### 3. 穿透的典型解法

#### 3.1 缓存空值

数据库查不到，也往 Redis 写一个空结果，并设置较短 TTL。

优点：

- 简单
- 对重复无效请求有效

缺点：

- 对“每次都换新非法 key”的攻击效果有限

#### 3.2 布隆过滤器

这是更经典的方案。

在进入 Redis / DB 前，先判断这个 key 是否可能存在。

不存在就直接拦掉。

优点：

- 节省大量无意义 DB 压力

缺点：

- 有误判
- 维护有成本

#### 3.3 接口限流 / 黑名单

如果请求本身就不正常，不要只靠缓存层兜底。

### 4. 击穿为什么强调“热点 key”

因为如果不是热点 key，过期就过期了，影响通常有限。

真正危险的是：

- 一个访问量极高的 key
- 恰好失效
- 同一时刻有大量请求过来

那数据库就会在瞬间被放大流量冲击。

### 5. 击穿的典型解法

#### 5.1 预热

对于秒杀商品、首页热点、活动配置这种你提前就知道会热的数据，最好提前灌入缓存。

这是最务实的办法。

#### 5.2 互斥锁

缓存 miss 后，不让所有线程都去查库，而是：

- 只有一个线程回源 DB
- 其他线程等结果或快速失败

这个方案很常见，但实现时要注意：

- 锁超时时间
- 回源失败怎么办
- 是否会造成锁热点

#### 5.3 热点 key 较长 TTL

不是说永不过期一定好，而是：

**真正热点且变化不频繁的数据，不要让它轻易在流量高峰时过期。**

### 6. 雪崩为什么比击穿更可怕

击穿通常是单热点。

雪崩是大面积一起崩。

例如：

- 一批缓存都设成同样过期时间
- Redis 集群整体不可用
- 某次大促前预热失败

这时数据库面对的就不是一个热点 key，而是整个缓存层失效后的洪峰。

### 7. 雪崩的典型解法

#### 7.1 随机 TTL

最实用也最便宜。

不要让大量 key 在同一时刻一起过期。

#### 7.2 多级缓存

例如：

- 本地缓存 + Redis

这样 Redis 出故障时，本地缓存还能兜一层热点。

#### 7.3 Redis 高可用

- 主从
- Sentinel
- Cluster

别把“缓存雪崩”只理解成 TTL 问题，服务整体不可用同样会雪崩。

#### 7.4 限流、降级、熔断

数据库不是无限抗打的。

缓存层失守时，应用层必须准备好保护数据库。

### 8. Cache Aside 为什么最常用

Cache Aside（旁路缓存）是最常见的模式。

读：

1. 查缓存
2. miss 再查 DB
3. 回填缓存

写：

1. 先更新 DB
2. 再删除缓存

读流程很自然，写流程也简单，所以它最常见。

### 9. 为什么不是“先删缓存，再改数据库”

因为如果你先删缓存，再改数据库，中间如果有并发读：

1. 缓存没了
2. 读请求去查 DB
3. 读到旧值又写回缓存
4. 你后面才把 DB 改成新值

这样旧数据就被重新写回缓存了。

所以更常见、更稳妥的是：

**先改 DB，再删缓存。**

### 10. 但“先改 DB，再删缓存”也不是绝对一致

如果更新 DB 成功了，删缓存失败了，就会出现：

- DB 是新值
- 缓存还是旧值

这就是缓存一致性的经典问题。

常见补救方式：

#### 10.1 TTL 缩短

能缓解，但治标不治本。

#### 10.2 删除缓存重试

更常见，也更靠谱。

通常做法是：

- 失败后投递消息
- 后台异步重试删缓存

### 11. 这类题最重要的不是“标准答案”，而是取舍

缓存一致性题没有免费午餐。

你要同时兼顾：

- 性能
- 可用性
- 复杂度
- 一致性

所以真正高级的回答，不是说“我有绝对完美方案”，而是：

**我知道哪里会不一致，我知道这类不一致能不能接受，以及如何把风险压到业务可接受范围。**

## 实战落地：缓存故障要按预案演练

缓存故障不是只靠代码里写几个 if 就能解决，最好把预案演练出来：

- 穿透演练：构造不存在 ID 的高并发请求，看空值缓存、布隆过滤器、限流是否生效。
- 击穿演练：让热点 key 失效，看互斥重建、逻辑过期、热点预热是否能控制回源。
- 雪崩演练：批量 key 同时过期或 Redis 故障，看限流、降级、多级缓存是否接住流量。
- 一致性演练：模拟删缓存失败，看重试队列、CDC、定期校验是否能补回来。

演练时要观察的不只是 Redis 指标，还要看数据库 QPS、接口 P99、错误率、降级命中量和用户可见影响。

## 生产问题处理：一致性事故先分级

缓存不一致发生后，先按业务影响分级：

- 展示类旧数据：通常可以删缓存、缩短 TTL、等待回源修复。
- 权限、价格、库存类数据：需要立即失效缓存，必要时临时绕过缓存读主库。
- 写路径错误导致旧值反复回填：要先暂停错误写入或回滚代码，再清理缓存。
- 删除补偿堆积：先扩容消费者或限速写入，再处理积压消息。

不要笼统说“缓存和数据库不一致了”。要明确是哪类 key、错了多久、影响哪些用户、是否可自动修复。这个分级决定你是可以异步修，还是必须立即止血。

## 小结

- 穿透、击穿、雪崩三者根因不同，别混着背。
- 穿透重点在于非法 key / 不存在数据；击穿重点在于热点 key 失效；雪崩重点在于大面积缓存失效或缓存整体不可用。
- Cache Aside 是最常用的缓存模式，写路径通常采用“先更新 DB，再删除缓存”。
- 缓存一致性不是是否会发生问题，而是如何在复杂度和一致性之间做业务可接受的取舍。
