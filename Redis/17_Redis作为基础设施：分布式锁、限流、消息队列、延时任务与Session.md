# Redis - 第 17 课：Redis 作为基础设施：分布式锁、限流、消息队列、延时任务与 Session

## 学习目标

- 理解 Redis 不只是缓存，还常被拿来当“基础设施能力提供者”。
- 能说清 Redis 做分布式锁、限流、消息队列、延时任务、Session 共享的基本思路与边界。
- 明白“Redis 能做”和“Redis 适合做”是两回事。

## 内容讲解

### 1. 为什么大家总爱拿 Redis 做基础设施

因为 Redis 同时满足几个条件：

- 快
- 命令天然原子
- 分布式共享状态方便
- 部署成本相对低

所以很多“本来不是缓存”的能力，也会顺手放到 Redis 上。

这很常见，但也很容易被滥用。

### 2. 分布式锁：Redis 最经典的基础设施玩法之一

最基础的加锁思路就是：

```text
SET lock:order:1001 uuid NX PX 30000
```

关键点：

- `NX`：只有不存在才设置成功
- `PX`：设置过期时间，防止死锁
- value 不能乱写，要写唯一标识，便于解锁时校验“是不是自己上的锁”

解锁不能直接 `DEL`，而应该：

- 先比对 value
- 再删除

这个动作通常用 Lua 封成一个原子逻辑。

#### Redis 锁的边界

- 它不是数据库行锁
- 不适合长事务
- 续期、重入、公平锁等复杂能力建议直接交给 Redisson

### 3. 限流：Redis 很适合做共享计数

限流的本质是：

- 在分布式多实例下共享访问次数

Redis 非常适合做这个。

常见方案：

- 固定窗口
- 滑动窗口
- 令牌桶
- 漏桶

最常见实现方式：

- Redis + Lua
- 或 Redisson 的 `RRateLimiter`

这里 Redis 的价值在于：

**跨实例共享计数 + 命令执行足够快。**

### 4. 消息队列：Redis 能做，但别神化

Redis 做消息队列，大概有三种层级：

#### 4.1 `List`

最简单：

- `LPUSH`
- `BRPOP`

适合：

- 很轻量
- 对可靠性要求不高

缺点：

- 消费确认能力弱
- 消息重投能力弱

#### 4.2 发布订阅 `Pub/Sub`

更像广播，不适合做可靠消息队列。

因为消费者不在线时，消息可能就过去了。

#### 4.3 `Stream`

这是 Redis 对消息流最认真的一次补强：

- 消费组
- ACK
- Pending List

但边界仍然存在：

- 中小规模业务很好用
- 真正高吞吐、强可靠、长链路治理，还是 Kafka / RabbitMQ 更合适

### 5. 延时任务：Redis 也很常被拿来做

典型场景：

- 订单 30 分钟未支付自动取消
- 优惠券到期提醒
- 红包超时退回

常见方案：

#### 5.1 过期事件监听

不太推荐，因为：

- 时效不稳定
- 多实例消费容易复杂
- 丢消息风险更难控

#### 5.2 `ZSet`

把执行时间作为 score。

后台线程轮询：

- 找 score 小于当前时间的任务
- 拉出来执行

这是最经典的 Redis 延迟任务方案。

#### 5.3 Redisson 延时队列

工程里更常见，也更省心。

### 6. Session 共享

在多实例 Web 应用里，Redis 还很适合做分布式 Session 存储。

常见方式：

- key：`session:{sessionId}`
- value：Session 数据
- 配合 TTL

优点：

- 多台应用都能读
- 天然支持过期

风险：

- Redis 故障会影响登录态
- Session 设计过重会放大 Redis 压力

### 7. Redis 做基础设施时最重要的判断

Redis 能做很多，不代表都该交给 Redis。

判断时你至少要问：

1. 这个能力是不是轻量状态共享问题？
2. 对可靠性要求是不是极高？
3. 失败后补偿成本大不大？
4. 是否有更成熟的专业中间件？

例如：

- 分布式锁：Redis 很常见，也通常合适
- 限流：Redis 很合适
- 轻量延时任务：Redis 可以
- 核心交易 MQ：Redis 通常不该硬扛

### 8. 面试里怎么体现“边界感”

真正加分的不是说“Redis 什么都能做”，而是：

**我知道 Redis 的优势在于快、共享、轻量、基础能力丰富；也知道它不该替代专业消息系统、数据库和复杂调度系统。**

## 小结

- Redis 常被用作分布式锁、限流、轻量消息队列、延时任务、Session 共享等基础设施能力。
- Redis 之所以适合这些场景，是因为它快、共享、原子操作方便。
- 但 Redis 不是万能中间件，尤其在消息可靠性、长事务、复杂调度上存在明显边界。
- 工程里最重要的是判断“Redis 能做”和“Redis 该不该做”的差别。
