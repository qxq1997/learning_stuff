# Redis - 第 18 课：选型与边界：为什么用 Redis、为什么不用 Redis 当主库、Redis vs Memcached

## 本篇定位

选型边界层。它放在后半段收尾，用来回答为什么用 Redis、为什么不用 Redis、什么时候该拒绝 Redis。

## 学习目标

- 不再把“Redis 很流行”误当成选型理由。
- 能说清 Redis 为什么适合做分布式缓存，却不适合直接充当主数据库。
- 能比较 Redis、Memcached，以及本地缓存、分布式缓存之间的区别。
- 面试时能把技术选型说成“有边界、有理由”的答案。

## 内容讲解

### 1. 为什么大家爱问“为什么选 Redis”

因为这个问题能一下子暴露两件事：

1. 你是不是只会跟风
2. 你有没有真正理解 Redis 的边界

所以别把它当成送分题。

### 2. Redis 为什么适合做分布式缓存

核心原因通常有四个：

#### 2.1 快

Redis 主要基于内存，读写延迟低。

#### 2.2 支持丰富数据结构

不是只能缓存一个字符串，还能顺手承接很多基础设施场景。

#### 2.3 可以独立部署

相比 JVM 本地缓存，它不受单机内存限制，也更适合多实例共享。

#### 2.4 生态成熟

客户端、云服务、监控、治理方案都比较完善。

### 3. 为什么不建议直接拿 Redis 当主库

这也是非常高频的问题。

原因主要有：

#### 3.1 内存成本高

Redis 的核心价值来自内存，但内存本身就贵。

拿它当主库存放海量冷数据，成本通常不划算。

#### 3.2 查询能力有限

Redis 擅长：

- key-value 访问
- 简单数据结构操作

它不擅长：

- 多表关联
- 复杂事务
- 条件过滤
- 复杂聚合

#### 3.3 持久化不是数据库事务系统那套强保证

Redis 有持久化，但它不是为了取代关系型数据库的完整事务和恢复体系而设计的。

#### 3.4 数据模型更偏“在线热点层”

Redis 更像“快而轻”的共享数据层，不像 MySQL 那样适合作为业务事实数据中心。

所以不是 Redis 不够强，而是它的设计目标压根就不是“当通用主库”。

### 4. 本地缓存和 Redis 缓存怎么选

这也很常见。

#### 本地缓存

优点：

- 超快
- 没有网络开销

缺点：

- 多实例数据不一致
- 容量受单机限制

#### Redis 分布式缓存

优点：

- 多实例共享
- 容量大
- 管理集中

缺点：

- 有网络开销
- 引入额外基础设施

所以很多系统实际采用的是：

**本地缓存 + Redis 二级缓存。**

### 5. Redis vs Memcached

这题现在虽然不像早年那么高频，但仍然值得会。

共同点：

- 都基于内存
- 都常做缓存
- 性能都高

区别重点：

#### 5.1 数据结构

- Memcached：基本就是简单 KV
- Redis：丰富数据结构

#### 5.2 持久化

- Memcached：基本不考虑持久化
- Redis：支持 RDB、AOF

#### 5.3 集群能力

- Memcached：更依赖客户端分片
- Redis：原生支持 Cluster、Sentinel 生态成熟

#### 5.4 特性

- Redis 还有 Lua、事务、发布订阅、Stream、限流、锁等玩法

所以现在大多数业务里，只要不是极其简单的纯缓存场景，基本都会偏向 Redis。

### 6. 那 Redis 的替代品呢

你可以知道一些名字：

- Dragonfly
- KeyDB
- Tendis

但在大部分场景下，选型建议仍然是：

**优先 Redis。**

因为选型看得不只是理论性能，还包括：

- 生态
- 稳定性
- 资料丰富度
- 团队认知成本

### 7. 技术选型最怕哪种回答

最怕的是：

“因为大家都用 Redis，所以我也用 Redis。”

更好的回答应该是：

“我们选 Redis，是因为我们的场景需要一个高性能、共享式、支持多种数据结构的缓存 / 基础设施层；而且它在高频读写、简单原子操作、热点数据承接上非常合适。我们没有拿它当主库，因为主数据仍然需要数据库那套更强的事务、查询和存储能力。” 

### 8. 一个很实用的选型框架

以后你可以按这几个问题判断：

1. 这是热点数据还是冷数据？
2. 需要共享状态吗？
3. 需要复杂查询吗？
4. 能接受多大程度的数据丢失风险？
5. 更看重极致性能还是强事务能力？

如果答案偏向：

- 热点
- 高并发
- 简单读写
- 共享状态

那 Redis 很可能合适。

如果答案偏向：

- 冷数据
- 复杂查询
- 强事务
- 强一致业务事实数据

那数据库才是主角。

## 实战落地：选型要写出“为什么不用”

成熟的技术选型不只写“为什么用 Redis”，还要写“为什么不用其他方案”：

- 为什么不用本地缓存：是否需要多实例共享、统一失效、跨节点一致视图。
- 为什么不用数据库硬扛：数据库是否会被热点读压垮，查询是否太频繁。
- 为什么不用 MQ：当前需求是不是状态查询而不是事件传递。
- 为什么不用专业搜索或分析引擎：是否只需要简单 key-value，而不是复杂检索。
- 为什么不用 Redis 当主库：是否缺少强事务、复杂查询、低成本冷存储和审计能力。

把这些写进 ADR 或设计文档，后续容量增长、故障复盘、成本优化时，团队才知道当初的边界在哪里。

## 生产问题处理：Redis 被滥用后的迁移路线

如果一个系统已经把 Redis 用成“事实主库”，迁移不要一刀切：

1. 先盘点 key：哪些是缓存，哪些是业务事实，哪些可以重建。
2. 给事实数据补数据库表或事件日志，建立写入双轨。
3. 新写入先落主存储，再写 Redis；历史数据分批回填。
4. 读路径逐步改为 DB / Redis 双读校验，再切到新主路径。
5. 保留 Redis 作为缓存或加速层，去掉它的唯一事实来源身份。

这个过程的核心不是“替换 Redis”，而是把 Redis 放回更适合的位置：高性能热点层，而不是承担所有长期事实和复杂查询。

## 小结

- 选 Redis 不是因为它流行，而是因为它适合承接热点、高并发、共享状态这类问题。
- Redis 很适合做分布式缓存和轻量基础设施，但不适合直接取代数据库做通用主库。
- 本地缓存和 Redis 各有边界，很多系统会采用多级缓存。
- Redis 相比 Memcached 的优势主要在于数据结构、持久化、高可用和工程扩展能力。
