# Redis 专题：什么是跳表，怎么用跳表，有什么好处

## 学习目标（本节结束后你能做到什么）

- 理解跳表到底是什么，而不是只会背“它是多层链表”。
- 理解跳表在查找、插入、删除时的直观工作方式。
- 说清为什么跳表能把查找复杂度从普通链表的 O(N) 改善到期望 O(logN)。
- 理解 Redis 为什么在 ZSet 里选择跳表，而不是平衡树。
- 把“跳表怎么用”放回 Redis 的有序集合、排行榜、范围查询等真实场景中。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先别急着看定义，先看它在解决什么问题

假设你有一个按分数排好序的链表：

```text
10 -> 20 -> 30 -> 40 -> 50 -> 60 -> 70
```

现在你要找 `60`。  
如果这是普通链表，你只能从头开始，一个个往后走：

- 看 10，不是
- 看 20，不是
- 看 30，不是
- ...
- 最后走到 60

这就是普通链表的问题：

- 插入删除局部方便
- 但查找太慢

如果元素很多，查找成本会越来越接近 O(N)。

那有没有一种结构：

- 保留链表插入删除比较顺手的特点
- 同时让查找时不要老老实实一个个爬

跳表就是为这个问题设计出来的。

### 2. 什么是跳表

跳表可以理解成：

**在有序链表的基础上，再额外搭几层“快捷通道”。**

最底层是一条完整链表，包含所有元素。  
再往上，每一层只保留一部分元素，作为“索引层”。

例如：

```text
Level 3:      20 -------- 60
Level 2: 10 --20 --40 ----60 --70
Level 1: 10 20 30 40 50 60 70
```

当你查找 `60` 时，就不必从最底层一步步走，而是：

- 先从高层快速跳
- 发现再跳就会超过目标时，下降一层
- 再继续向前跳
- 逐层逼近目标

这就是“跳表”这个名字的由来：  
它不是老老实实走，而是在多层索引之间“跳着找”。

### 3. 查找是怎么做的

假设我们找 `50`。

过程大致是：

1. 从最高层起步
2. 当前节点往右看，如果下一个值还小于等于目标，就往右走
3. 如果再往右就会超过目标，就往下一层降
4. 继续这个过程，直到最底层

可以把它想成爬楼找教室：

- 先在高层看大方向
- 确认大概区域后再下楼缩小范围
- 最后在底层精准找到门牌号

所以跳表查找本质上不是暴力全扫，而是：

- 高层负责快速缩小范围
- 低层负责精确定位

### 4. 插入为什么不难

插入一个新元素，例如插入 `55`，大致流程是：

1. 先像查找一样，找到它在各层应该插入的位置
2. 在最底层把它插进去，保证有序
3. 然后用一定概率决定它要不要出现在上一层、再上一层

这个“随机向上晋升”的设计很关键。  
不是每个节点都上很多层，而是：

- 大部分节点层数低
- 少部分节点层数高

这样整体结构就能在概率意义上保持比较好的平衡，而不需要像平衡树那样做复杂旋转。

### 5. 删除怎么做

删除本质上也不复杂：

1. 先找到目标节点在各层的位置
2. 在它出现的每一层都把指针摘掉

因为跳表的每层本质上还是链表连接，所以删除逻辑比平衡树的重平衡通常更容易理解。

### 6. 为什么跳表能做到 O(logN)

关键在于“每往上一层，节点数量会显著减少”。

如果你把每层都看成对下一层的抽样，那么：

- 底层有全部节点
- 上一层大约只有一半
- 再上一层大约再减半

这意味着你在高层时能用很少的步数快速跨过大量节点。  
于是整体查找、插入、删除都能做到期望 O(logN)。

注意这里通常说的是“期望 O(logN)”，因为跳表的层级构造常依赖随机化，不是像红黑树那样用严格的结构约束来保证最坏情况。

### 7. 怎么用跳表

这里要分两个层面来理解。

#### 7.1 从数据结构角度怎么用

如果你自己的业务需要一个结构，同时满足：

- 元素有序
- 支持快速查找
- 支持范围遍历
- 支持动态插入删除

那跳表就是一个很自然的候选方案。

它特别适合：

- 排行榜
- 延迟任务
- 时间窗口内有序事件
- 按分值或时间范围查询的数据集

#### 7.2 从 Redis 角度怎么用

在 Redis 里，你不会手动创建一个“跳表对象”，而是通过 `ZSet` 间接使用它。

典型命令有：

- `ZADD`
- `ZRANGE`
- `ZREVRANGE`
- `ZRANK`
- `ZRANGEBYSCORE`

例如做排行榜：

```text
ZADD rank 1200 alice
ZADD rank 980 bob
ZADD rank 3000 carol
```

这时候 Redis 内部要同时支持：

- 查某个成员的分数
- 查 TopN
- 按分数区间查成员
- 查排名

这也是为什么 Redis 的 ZSet 常采用：

- 字典：member -> score
- 跳表：按 score 有序组织节点

### 8. Redis 为什么用跳表，不直接用平衡树

这是跳表专题里最有价值的问题。

理论上，平衡树也能做有序集合，为什么 Redis 没选它作为核心有序索引结构？

主要原因有几个：

#### 8.1 实现简单

跳表的插入、删除、维护逻辑相对直观，不像红黑树那样要处理旋转、染色、复杂平衡维护。

对于工程系统来说，“足够快 + 更简单 + 更容易维护”往往比“理论上最优雅”更重要。

#### 8.2 范围遍历很自然

跳表底层本来就是按顺序连起来的，所以从某个节点开始往后扫一个范围，非常自然。

而 Redis 的 ZSet 一个特别重要的需求就是：

- 按 score 范围查
- 按排名区间查

跳表在这类顺序遍历场景里很顺手。

#### 8.3 性能足够好

跳表查找、插入、删除在期望上都是 O(logN)，对 Redis 这种以内存操作为主、追求工程实用性的系统来说，已经很好用了。

#### 8.4 更适合和字典组合

Redis 的 ZSet 不是只用跳表，而是“字典 + 跳表”组合。  
字典解决按成员直接查，跳表解决按分数排序和范围访问。  
这个组合非常贴合有序集合的实际需求。

### 9. 跳表有什么好处

把前面的内容压缩一下，跳表的好处主要是：

- **查找快**：相比普通链表，能大幅减少线性扫描
- **插入删除也快**：期望 O(logN)，并且局部改指针即可
- **范围查询友好**：天然适合顺序遍历
- **实现比平衡树简单**：工程复杂度更低
- **很适合内存系统中的有序索引结构**

但也要看到它不是没有代价：

- 需要额外多层指针，空间会比纯链表更高
- 复杂度通常说的是期望值，不是最坏严格保证

所以真正成熟的理解不是“跳表完美”，而是：

**在 Redis 这种强调内存访问、有序范围操作、实现可维护性的场景里，跳表是一个非常合适的工程折中。**

## 小结

- 跳表是在有序链表上增加多层索引，让查找不必线性扫描到底。
- 它通过“高层快速缩范围，低层精确定位”的方式，把查找、插入、删除做到期望 O(logN)。
- 在 Redis 里，你通常不是直接手写跳表，而是通过 ZSet 间接使用跳表能力。
- Redis 选择跳表，是因为它在实现复杂度、范围遍历和性能之间取得了很好的工程平衡。
- 理解跳表，不只是为了会一个数据结构，更是为了看懂 Redis 为什么这样设计有序集合。
