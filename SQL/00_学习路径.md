# SQL 学习路径与知识图谱

## 学习定位

后端工程师学 SQL，最怕两种状态：

- 第一种是“只会写能跑的 SQL”，功能做出来了，但一上生产就慢，不知道为什么慢，也不知道该怎么改。
- 第二种是“背了很多优化口诀”，比如“不要 `SELECT *`”“要走索引”“`EXISTS` 比 `IN` 快”，但一到真实业务场景就不会判断，甚至会把口诀用反。

这组内容的目标，不是让你背 30 条 SQL 优化建议，而是让你建立一套更稳的判断框架：

- SQL 为什么会慢？
- 索引到底帮你省掉了什么工作？
- 哪些写法最容易让索引失效？
- 排序、分组、分页、子查询为什么经常成为慢 SQL 的源头？
- 线上拿到一条慢 SQL 后，应该先看什么，再看什么，最后怎么改？

学完之后，你应该能把“SQL 优化”从玄学变成工程判断，而不是看到慢查询就只会说“加个索引试试”。

## 知识图谱

1. SQL 为什么会慢
   - 扫描行太多
   - 回表太多
   - 排序 / 分组 / 去重太重
   - 临时表太大
   - 锁等待、网络传输、事务过长
2. 索引与存储结构
   - B+ 树
   - 聚簇索引与二级索引
   - 回表
   - 覆盖索引
   - 联合索引与最左匹配
3. 常见慢 SQL 写法
   - `SELECT *`
   - 对索引列做函数或表达式
   - 隐式类型转换
   - `LIKE '%xx'`
   - 深分页
   - 不必要的 `OR`、`NOT IN`
4. 排序、分组、聚合与子查询
   - `ORDER BY`
   - `GROUP BY`
   - `DISTINCT`
   - `HAVING`
   - `UNION ALL`
   - `EXISTS`
5. 线上排查慢 SQL
   - 慢日志
   - `EXPLAIN`
   - `EXPLAIN ANALYZE`
   - `rows`、`filtered`
   - `Using temporary`
   - `Using filesort`
6. 真实工程优化
   - 批量写入
   - 合理分页
   - 缩小结果集
   - 冷热数据归档
   - 缓存与搜索系统分流
   - 不要迷信 `force index`
7. 顺手又实用的 SQL 技巧
   - `CASE WHEN`
   - `COALESCE`
   - 自连接
   - 批量更新
   - 条件聚合

## 为什么这样排序

很多人一上来就学“联合索引最左匹配原则”，但其实还没搞清一个更前面的事：

- 数据库为了执行一条 SQL，到底做了哪些活？

如果你没有这个整体感，后面就很容易把优化技巧当成死规定。比如：

- 为什么 `SELECT *` 常常更慢？
- 为什么 `GROUP BY` 容易引发临时表和排序？
- 为什么深分页在数据量大时会越来越慢？
- 为什么有索引也不一定快？

所以这条学习路径故意先从“性能直觉”开始，让你先知道数据库到底在忙什么；再去看索引为什么能减少工作；再去拆各种高频慢 SQL 写法；最后再回到线上排查和工程治理。

这样你以后看到一条慢 SQL，脑子里不会只有“要不要加索引”这一种动作，而是会自然去问：

- 是扫描太多行了？
- 是回表太多了？
- 是排序太重了？
- 是临时表太大了？
- 是分页方式有问题？
- 还是这类需求本来就不该全靠数据库硬扛？

## 学习方法建议

- 每看一条优化建议，都先问一句：它到底帮我减少了哪一种成本？
- 不要只看 SQL 文本，要同时看数据量、索引、执行计划和业务场景。
- 不要迷信单条经验，`IN`、`EXISTS`、`JOIN`、`DISTINCT`、`HAVING` 这些都要结合执行计划判断。
- 养成“先缩小数据集，再做排序 / 分组 / 聚合”的思维习惯。
- 线上优化时，坚持“小步快跑、一次只改一个变量”，避免把问题越改越乱。

## 学完后你应该能回答的问题

- 一条 SQL 慢，最常见的几种根因分别是什么？
- 回表到底是什么，为什么覆盖索引通常更快？
- 联合索引的最左匹配到底是“写 SQL 时的列顺序”，还是“查询条件是否约束了前导列”？
- 为什么 `LIKE '%abc'`、函数包列、隐式类型转换都容易让索引失效？
- 为什么深分页越翻越慢，什么是更适合线上接口的分页方式？
- `WHERE` 和 `HAVING` 分别应该放什么条件？
- 看到 `Using filesort` 和 `Using temporary` 时应该怎么想？
- 哪些问题该靠 SQL 改写解决，哪些问题该靠表设计、缓存、归档或者搜索系统分流解决？
- 那些流传很广的“SQL 优化 30 条建议”，哪些是真经验，哪些要结合场景看？
