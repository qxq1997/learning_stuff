# SQL - 第 1 课：SQL 为什么会慢：先建立性能直觉

## 学习目标（本节结束后你能做到什么）

- 不再把“SQL 慢”简单理解成“没走索引”。
- 能从执行成本的角度，解释一条 SQL 为什么会慢。
- 能区分“扫描太多”“排序太重”“返回太多”“锁等待”“网络太慢”这些不同类型的慢。
- 看到一条慢 SQL 时，先问对问题，而不是上来就盲目加索引。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

很多后端同学刚开始学 SQL 优化时，最容易形成一个误解：

- SQL 慢 = 索引没建好

这句话不能说完全错，但它只说到了问题的一部分。  
更完整的说法应该是：

**SQL 慢，本质上是数据库为了得到结果，做了太多“没必要的工作”。**

你可以把数据库想象成一个仓库管理员。  
你问他：“把昨天所有已支付订单里，金额最高的前 20 条给我。”

他可能要做这些事：

1. 先去仓库里找出“昨天”的订单
2. 再筛出“已支付”的订单
3. 再读取每一行里你需要展示的列
4. 再按金额排序
5. 再取前 20 条
6. 再把结果传回给应用程序

如果这中间哪一步工作量特别大，SQL 就会慢。

所以你以后看 SQL 性能，脑子里先别急着想“语法”，先想“成本”。

### 1. 一条 SQL 最常见的 6 类成本

#### 1.1 扫描行太多

这是最经典的慢查询来源。

比如表里有 1000 万行，你本来只想找 20 行结果，但数据库为了找到这 20 行，先扫了 300 万行，那大概率就慢。

常见原因：

- 没有合适索引
- 索引没用上
- 条件选择性太差
- 深分页需要先跳过很多行

#### 1.2 读取列太多

很多人忽略这一点。  
如果你用 `SELECT *`，数据库可能把一整行几十个字段都读出来，哪怕你页面上只展示 3 个字段。

这会带来两个额外成本：

- 读盘更多
- 网络传输更多

如果本来索引里就有你要的字段，却因为 `SELECT *` 被迫回表读整行，性能会进一步变差。

#### 1.3 排序 / 分组 / 去重太重

这类操作特别容易被低估。

凡是涉及这些关键字，你都要下意识警觉：

- `ORDER BY`
- `GROUP BY`
- `DISTINCT`
- `UNION`
- 聚合函数

因为它们经常意味着数据库要把很多行拿出来重新整理。  
如果数据量一大，内存装不下，还可能写临时文件，性能会明显掉下来。

#### 1.4 临时表和中间结果太大

有些 SQL 虽然看起来不复杂，但会生成很大的中间结果，比如：

- 子查询先算出一张临时结果
- 再在外层继续过滤
- 或者先连接出很多重复行
- 再 `DISTINCT`

这种写法的问题在于：

**你先把很多本来不该生成的数据生成了，再想办法把它们丢掉。**

这就像你只想吃 10 个苹果，却先搬了 1000 个苹果上桌，再从里头挑 10 个出来。

#### 1.5 锁等待或事务拖住了

有些 SQL “慢”，其实不是它自己执行慢，而是它在等。

比如：

- 别的事务还没提交
- 你要更新的那几行正被别人锁住
- 一个大事务持有锁太久

这时候你看 SQL 文本本身可能并不复杂，但执行时间却很长，因为时间消耗在“等待”上。

#### 1.6 结果集太大，网络传输慢

这个在后台导出、报表接口、批处理任务里特别常见。

有时候数据库计算很快，但你一次查回几十万行，把应用内存打爆，或者把网络拖慢，接口整体还是会慢。

所以优化 SQL，不只是让数据库少做事，也包括：

- 少查一些行
- 少查一些列
- 少回一些数据

### 2. 用一张图理解 SQL 的执行成本

```mermaid
flowchart LR
    A["收到 SQL"] --> B["筛选数据"]
    B --> C["读取需要的列"]
    C --> D["连接 / 排序 / 分组 / 去重"]
    D --> E["生成最终结果集"]
    E --> F["返回给应用"]
```

你以后判断一条 SQL 慢不慢，就沿着这条链路去拆：

- 是 B 太重，筛得太慢？
- 是 C 太重，读了太多列？
- 是 D 太重，排序 / 分组 / 去重太猛？
- 是 F 太重，回了太多数据？

### 3. 先问这 4 个问题，别急着改 SQL

线上拿到一条慢 SQL，可以先问自己 4 个问题：

#### 3.1 它本来应该返回多少行？

如果一个列表页正常只展示 20 条，但你背后其实查了 10 万条再截断，那方向已经错了。

#### 3.2 它到底扫描了多少行？

这一步非常关键。  
很多慢 SQL 的本质都是：

- 返回 20 行
- 但扫描了 20 万行

这种“扫描量远大于返回量”的情况，通常最有优化空间。

#### 3.3 它有没有排序、分组、去重？

如果有，要特别留意：

- 排序字段有没有索引支持
- 分组前能不能先过滤
- 去重是不是本来就不该发生

#### 3.4 它慢在数据库，还是慢在数据库之外？

比如：

- 数据库执行只用了 30ms
- 但应用层对象映射用了 200ms
- 或者接口把 5MB 结果集序列化成 JSON 又花了很久

这时候你如果只盯着 SQL 改，可能方向就偏了。

### 4. 一个非常常见的误区：有索引就一定快

不是这样的。

索引只是“可能让数据库少做事”的工具，不是性能保证书。

下面这些情况，即使有索引，也可能慢：

- 条件命中行数太多，索引选择性差
- 虽然走了索引，但回表次数太多
- 排序字段和过滤字段没协同好
- 深分页先跳过了大量数据
- 结果集本来就很大

所以以后不要只问：

- “走没走索引？”

更要问：

- “走这个索引，真的比别的方式更省活吗？”

### 5. 后端开发最实用的一个判断标准

如果你现在还没把执行计划研究很深，可以先记住一个非常实用的朴素判断：

**SQL 优化的核心，就是尽量让数据库更早地排除无关数据，更少地读取无关列，更晚地做重操作。**

翻译成工程动作就是：

- 越早过滤越好
- 越少回表越好
- 越少排序越好
- 越少临时表越好
- 越少返回数据越好

这套思路，后面会贯穿所有索引和 SQL 写法优化。

## 小结（3-5 条关键点）

- SQL 慢，不只是“没走索引”，而是数据库做了太多没必要的工作。
- 最常见的成本包括：扫描太多行、读取太多列、排序分组太重、临时表过大、锁等待、结果集过大。
- 看慢 SQL 时，先拆执行成本链路，不要上来就盲目加索引。
- 一个很实用的优化原则是：越早过滤、越少回表、越少排序、越少返回。
- 索引很重要，但它只是减少工作量的工具，不是万能答案。
