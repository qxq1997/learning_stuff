# SQL - 第 2 课：索引与执行计划：B+树、回表、覆盖索引、联合索引

## 学习目标（本节结束后你能做到什么）

- 能用简单的话讲清索引到底在帮数据库省什么工作。
- 理解 InnoDB 里主键索引、二级索引、回表、覆盖索引分别是什么。
- 理解联合索引为什么会有“最左匹配”，以及它真正限制的是什么。
- 知道什么样的查询更容易用好索引，什么样的写法容易让索引价值打折。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 为什么索引能加速查询

如果没有索引，数据库找数据很像你在一大叠纸里一页一页翻：

- 找到符合条件的记录
- 再继续往后找

这就叫全表扫描。

而有索引之后，更像一本书后面的目录：

- 先根据目录定位大概范围
- 再直接跳到对应位置

所以索引真正帮你省掉的是：

- 不必要的扫描

它不是把数据“算得更快”，而是让数据库少看很多不相关的数据。

### 2. B+ 树，你只需要先理解这件事

MySQL 里最常见的索引底层是 B+ 树。  
你先不用把它想得特别数学化，只要先抓住一个直觉：

**B+ 树是一种有序的数据目录结构，特别擅长做范围查找和有序访问。**

为什么数据库喜欢它？

- 查单点快
- 查范围也快
- 天然有序，适合排序、范围过滤
- 磁盘 I/O 次数可控

所以很多优化建议最后都能回到一句话：

- 你的查询方式，能不能顺着 B+ 树高效地找？

### 3. InnoDB 里最重要的两个概念

#### 3.1 聚簇索引

InnoDB 的主键索引，一般就是聚簇索引。  
你可以理解成：

- 表的数据本身，就是按主键这棵树组织的

这意味着，如果你通过主键查一行，通常很直接，因为叶子节点上就是整行数据。

#### 3.2 二级索引

除了主键索引之外，普通索引、联合索引，很多都是二级索引。

你可以先把它理解成：

- 二级索引叶子节点上通常不是完整数据
- 而是“索引列值 + 主键值”

所以通过二级索引找到记录后，如果你要的字段不在索引里，数据库还要再根据主键回到主键索引里把整行取出来。

这一步，就叫回表。

### 4. 回表到底是什么

假设你有一张订单表：

- 主键：`id`
- 联合索引：`(status, create_time)`

现在你执行：

```sql
SELECT * 
FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 20;
```

数据库可能先走 `(status, create_time)` 这个索引找到符合条件的主键 id，  
但因为你用了 `SELECT *`，索引里没有整行所有字段，于是它还得：

- 再拿着这些主键 id 回到主键索引里把整行捞出来

这就是回表。

如果命中了 20 行，回表 20 次还好；  
如果命中了 20 万行，回表 20 万次，代价就很高了。

### 5. 覆盖索引为什么通常更快

如果你查询的字段，刚好都在索引里，数据库就不用回表了。

比如：

```sql
SELECT status, create_time
FROM orders
WHERE status = 1
ORDER BY create_time DESC
LIMIT 20;
```

如果联合索引正好是 `(status, create_time)`，那数据库可能只扫索引就把结果拿全。

这就是覆盖索引。

覆盖索引快的原因很朴素：

- 少了一轮回表
- 读的数据更少
- 通常 I/O 更少

所以“尽量别 `SELECT *`”背后最重要的一层含义，其实就是：

- 给覆盖索引创造机会

### 6. 联合索引和最左匹配，到底在说什么

假设有联合索引：

```text
(a, b, c)
```

很多人背过一句口诀：

- 联合索引要满足最左匹配

但最容易误解的是，以为它说的是：

- SQL 里条件书写顺序必须和索引列顺序一样

这其实不准确。

更准确的理解应该是：

**索引能不能高效使用，关键看查询条件是否约束到了前导列，而不是你在 SQL 里把条件写在前还是写在后。**

比如这两条，本质上都能使用 `(a, b, c)`：

```sql
WHERE a = 1 AND b = 2
WHERE b = 2 AND a = 1
```

因为优化器会做等价改写，它不看你写的先后顺序，它看的是：

- 有没有 `a`
- 有了 `a` 之后，有没有继续用到 `b`
- 再往后，有没有继续用到 `c`

真正会出问题的是这种：

```sql
WHERE b = 2 AND c = 3
```

因为前导列 `a` 没被约束，数据库很难顺着 `(a, b, c)` 这棵树高效定位。

### 7. 范围查询为什么经常“截断”联合索引

还是 `(a, b, c)` 这个索引。

如果你的条件是：

```sql
WHERE a = 1 AND b > 10 AND c = 5
```

通常到 `b > 10` 这种范围条件这里，后面的 `c` 就不一定还能像前面那样继续高效利用。

你可以把它想象成：

- `a = 1` 是精准定位到一个大抽屉
- `b > 10` 是在这个抽屉里选一大片范围
- 进入“大片范围”之后，再想继续靠 `c = 5` 精准缩小，就没前面那么顺了

所以联合索引设计时，要特别注意：

- 等值匹配列
- 范围列
- 排序列

它们的顺序是很有讲究的。

### 8. 索引不只是服务 WHERE，也服务排序和分组

如果你的查询既要过滤，又要排序，索引顺序经常决定性能高低。

比如：

```sql
SELECT id, create_time
FROM orders
WHERE user_id = 1001
ORDER BY create_time DESC
LIMIT 20;
```

如果索引是：

```text
(user_id, create_time)
```

那数据库通常更容易做到：

- 先按 `user_id` 快速定位
- 再顺着索引有序地拿出 `create_time`

这就比“先找到很多行，再单独排序”更省活。

### 9. 什么时候不要乱加索引

索引不是越多越好。

因为索引也有成本：

- 占空间
- 插入更慢
- 更新更慢
- 删除更慢
- 优化器选择变复杂

下面这些情况就不适合盲目加索引：

- 字段离散度很低，比如只有男女、启用禁用两三个值
- 表写入非常频繁，但查询收益不大
- 同类索引重复建设
- 没有真实查询场景支撑

所以索引设计不是“我能加就加”，而是：

- 这条查询值不值得我长期为它维护一棵树

### 10. 一个后端工程师最该掌握的索引直觉

以后你设计索引时，先从这 4 个问题入手：

1. 我最常用什么条件过滤？
2. 这些条件的区分度高不高？
3. 查询后还要不要排序？
4. 我返回的字段能不能尽量被索引覆盖？

如果这 4 个问题想清楚了，你的索引设计通常就不会太差。

## 小结（3-5 条关键点）

- 索引的核心价值，是减少不必要的扫描。
- InnoDB 里二级索引很多时候只存索引列和主键，所以查询额外字段会产生回表。
- 覆盖索引通常更快，因为它能避免回表。
- 联合索引的最左匹配，关键看有没有约束前导列，不是 SQL 里条件书写顺序。
- 索引还能服务排序和分组，但索引不是越多越好，写入成本也要考虑。
