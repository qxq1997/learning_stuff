# SQL - 第 3 课：最常见的慢 SQL 写法：函数、类型转换、LIKE、分页与 SELECT 星

## 学习目标（本节结束后你能做到什么）

- 能识别几类最常见的“看起来能跑，但很容易变慢”的 SQL 写法。
- 理解为什么函数包列、隐式类型转换、前导 `%`、深分页、`SELECT *` 经常会拖慢查询。
- 知道 `IN`、`EXISTS`、`OR` 这些写法为什么不能只靠一句口诀判断。
- 能把“写法优化”落到真实接口和后台任务场景中。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

这一课我们不讲太抽象的原理，只讲后端最容易在线上遇到的几类坑。

很多慢 SQL 不是因为你不会写 SQL，而是因为：

- 你写出了“数据库不太容易高效执行”的样子

### 1. `SELECT *` 为什么经常是坏习惯

先说结论：

**`SELECT *` 不一定一定慢，但它非常容易在数据量变大后变慢。**

原因主要有三层：

#### 1.1 会多读很多你根本不用的列

比如接口页面只展示：

- 订单号
- 金额
- 状态

但你却把整行三四十个字段都拿回来了。

#### 1.2 容易破坏覆盖索引

本来索引里就已经有你要的几个字段，数据库可以只扫索引。  
但你一旦 `SELECT *`，就往往得回表。

#### 1.3 会加重网络和应用层开销

数据库查出来只是第一步。  
你还要：

- 网络传输
- JDBC 反序列化
- ORM 映射对象
- 最后转成 JSON

所以只取必要列，是非常朴素但很有效的优化。

### 2. 不要在索引列上做函数、表达式或计算

下面这种写法很常见：

```sql
WHERE DATE(create_time) = '2026-04-10'
```

或者：

```sql
WHERE amount * 1.1 > 100
```

或者：

```sql
WHERE SUBSTRING(phone, 1, 3) = '138'
```

这类写法的问题是：

- 你不是拿“原始索引值”去查
- 而是先把每一行算一遍，再比较

数据库就很难顺着索引直接定位。

更推荐的写法通常是把计算挪到常量一侧，或者改成范围：

```sql
WHERE create_time >= '2026-04-10 00:00:00'
  AND create_time <  '2026-04-11 00:00:00'
```

你以后可以记一句更稳的话：

**索引列最好保持“原样参与比较”，别把它包进函数里。**

### 3. 隐式类型转换也会悄悄害你

假设 `user_id` 是字符串列，但你这样写：

```sql
WHERE user_id = 1001
```

数据库会发生类型转换。  
不同数据库、不同场景下优化器处理方式不完全一样，但很常见的问题是：

- 额外增加比较成本
- 甚至导致索引利用变差

所以更稳妥的做法是：

- 字符串就按字符串传
- 数字就按数字传
- 参数类型和列类型尽量保持一致

这件事在 Java 里尤其重要，因为 ORM、JDBC、手写 SQL 参数绑定很容易把类型传歪。

### 4. `LIKE` 只有前缀匹配更容易用索引

这三种写法要分清：

```sql
LIKE 'abc%'
LIKE '%abc'
LIKE '%abc%'
```

通常只有第一种更容易顺着索引查，因为它保留了“从前往后”的有序性。

而后两种为什么慢？

因为你让数据库回答的问题变成了：

- “结尾是不是 abc？”
- “中间有没有 abc？”

这种查法很难沿着 B+ 树目录直接定位，往往需要扫很多数据。

所以如果你需求是全文检索、模糊搜索、包含匹配，很多时候就不该只靠普通 B+ 树索引硬扛，而要考虑：

- 倒排索引
- 搜索引擎
- 专门的全文索引能力

### 5. 深分页为什么越翻越慢

这类 SQL 你一定见过：

```sql
SELECT id, title
FROM article
ORDER BY id
LIMIT 100000, 20;
```

它的问题不是“只取 20 条”，而是：

- 为了拿到第 100001 条开始的 20 行，数据库前面那 100000 行通常也得先处理过去

你可以把它理解成：

- 不是直接跳过去
- 而是先数、先走、先丢，再给你最后几条

所以深分页在大表上非常容易变慢。

更常见的改法有两种：

#### 5.1 使用基于游标 / 主键的翻页

比如上一页最后一条 id 是 `100000`，下一页直接：

```sql
WHERE id > 100000
ORDER BY id
LIMIT 20
```

这比 `LIMIT offset, size` 更适合连续翻页接口。

#### 5.2 先用覆盖索引拿主键，再回表

这就是很多文章说的“延迟关联”或“延迟查询”。

先从窄索引里找出这一页 id，再用 id 回表拿完整数据，通常比直接在大宽表上深分页更省。

### 6. `IN`、`EXISTS` 不是非黑即白

很多文章喜欢一句话总结：

- `EXISTS` 比 `IN` 快

这句话不够严谨。

更稳的理解是：

- 小的常量列表，`IN` 往往完全没问题
- 关联存在性判断，`EXISTS` 往往语义更贴切
- 最终快不快，要结合执行计划看

比如这两类问题就适合不同写法：

#### 6.1 一小串固定值过滤

```sql
WHERE status IN (1, 2, 3)
```

这很常见，也很合理，不必为了“优化口诀”强行改掉。

#### 6.2 判断另一张表里是否存在匹配记录

```sql
WHERE EXISTS (
  SELECT 1
  FROM pay_record p
  WHERE p.order_id = o.id
)
```

这种“只关心有没有”而不关心具体有多少条时，`EXISTS` 往往表达更自然。

重点不是背哪个永远更快，而是搞清：

- 你的业务问题到底是在“值集合过滤”，还是在“存在性判断”

### 7. `OR` 不是不能用，而是要小心

很多人背口诀说：

- `OR` 会导致索引失效

这也不够准确。

真实情况通常是：

- 某些 `OR` 条件优化器还能处理
- 某些会让执行计划变差
- 尤其是不同字段上的 `OR`，更容易让扫描量变大

有时候把它改成两段查询再 `UNION ALL`，反而更稳。

例如：

```sql
WHERE mobile = ? OR email = ?
```

如果两边各有索引，且结果允许保留重复或你能自己处理重复，拆成两段有时更容易优化。

但请注意，这不是绝对规则，最终还是要看 `EXPLAIN`。

### 8. 否定条件经常不友好

这类条件要特别谨慎：

- `!=`
- `<>`
- `NOT IN`
- `NOT LIKE`

它们的共同问题是：

- 数据库更难快速定位“我要什么”
- 往往只能从“不是这个”的巨大集合里慢慢排除

所以你以后看到否定条件，第一反应应该是：

- 能不能改写成明确的正向范围？

### 9. 只要需要一条，就记得 `LIMIT 1`

很多业务逻辑其实只是判断：

- 有没有这条数据
- 有没有重复
- 有没有配置

但 SQL 却没写 `LIMIT 1`。

如果数据库找到第一条满足条件的记录就能停，那通常比继续扫更多行更省。

这是一条非常简单但很实用的习惯。

## 小结（3-5 条关键点）

- `SELECT *` 会增加读取、回表、网络传输和对象映射成本，尽量只查必要列。
- 对索引列做函数、表达式和隐式类型转换，常常会让索引利用变差。
- `LIKE` 里前缀匹配比前导 `%` 和中间 `%` 更容易利用索引。
- 深分页常常是线上慢 SQL 高发区，更适合改成游标式分页或延迟关联分页。
- `IN`、`EXISTS`、`OR` 没有绝对口诀，最终要结合业务语义和执行计划判断。
