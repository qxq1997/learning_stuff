# SQL - 第 4 课：排序、分组、去重与子查询：ORDER BY、GROUP BY、DISTINCT、HAVING 与 EXISTS

## 学习目标（本节结束后你能做到什么）

- 理解为什么排序、分组、去重常常是慢 SQL 的高发区。
- 能判断 `WHERE` 和 `HAVING` 应该分别放什么条件。
- 能理解 `DISTINCT`、`UNION`、子查询为什么常常会制造额外成本。
- 能把“先过滤再聚合”“能不去重就不去重”“能不用临时表就尽量不用”的思路落到 SQL 改写里。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

前面两课我们已经知道：

- SQL 慢，很多时候是数据库做了太多无谓工作

而排序、分组、去重、子查询，就是最容易让“额外工作量”暴涨的几类操作。

### 1. 为什么 `ORDER BY` 常常很贵

如果结果本来就按索引有序，那数据库只要顺着索引往后读就行。

但如果不能直接利用索引顺序，它就可能要：

- 先把大量数据取出来
- 再在内存里排序
- 内存不够时再写临时文件

这就是为什么看到排序时，你要特别关注：

- 排序字段有没有和过滤条件一起形成合适索引
- 排序前能不能先把数据集压小

一个很实用的思维是：

**越晚排序越贵，越早过滤越值钱。**

### 2. `GROUP BY` 不只是“分组”，还是一种重操作

很多人写统计 SQL 时，只关注能不能写对结果，却忽略了分组过程本身很可能很重。

比如：

```sql
SELECT user_id, COUNT(*)
FROM orders
GROUP BY user_id;
```

如果数据量很大，数据库需要：

- 找出每个 `user_id`
- 把同一个 `user_id` 的记录归到一起
- 再做计数

如果前面一点过滤都没有，那它就要处理整张表。

所以统计类 SQL 最重要的一个优化原则就是：

- 能先过滤就先过滤

### 3. `WHERE` 和 `HAVING` 到底怎么分

一句最实用的话：

- `WHERE` 是聚合前过滤原始数据
- `HAVING` 是聚合后过滤结果

这意味着，如果条件本来就能在聚合前过滤，就应该尽量放 `WHERE`。

例如：

```sql
SELECT sale_date, SUM(amount)
FROM sales
WHERE sale_date = '2026-04-10'
GROUP BY sale_date;
```

通常会比下面这种更省：

```sql
SELECT sale_date, SUM(amount)
FROM sales
GROUP BY sale_date
HAVING sale_date = '2026-04-10';
```

为什么？

因为前者先缩小了原始数据集，再分组。  
后者更像是：

- 先分组一大堆
- 再把大部分结果丢掉

当然，如果你筛选的是聚合结果本身，比如：

- 总金额大于 1000
- 订单数大于 10

那就应该用 `HAVING`。

### 4. `DISTINCT` 的代价，很多人低估了

`DISTINCT` 的语义很简单：

- 去重

但去重的代价通常不低。  
数据库要判断哪些行相同、哪些不同，常常伴随排序或哈希过程。

所以你看到 `DISTINCT` 时，先问两个问题：

1. 重复数据为什么会出现？
2. 能不能从源头避免重复，而不是最后再去重？

很多时候，`DISTINCT` 其实是在帮你掩盖：

- 错误的连接方式
- 不必要的一对多展开

### 5. `UNION ALL` 为什么经常比 `UNION` 更值得优先考虑

`UNION` 默认要去重。  
`UNION ALL` 只是把两段结果拼起来，不负责去重。

所以如果业务允许保留重复行，或者你本来就知道两边结果不会重复，优先考虑：

```sql
UNION ALL
```

别小看这一个 `ALL`，它经常意味着：

- 少一轮去重
- 少一轮排序 / 哈希
- 少很多额外开销

### 6. 子查询和中间表，问题不在“子查询”三个字

很多人会简单地说：

- 子查询慢

这也不够严谨。

真正的问题不是“写了子查询”，而是：

- 子查询是不是制造了大中间结果
- 优化器能不能把它合并掉
- 它是不是让索引利用变差了

比如这种写法：

```sql
SELECT *
FROM (
  SELECT user_id, MAX(amount) AS max_amount
  FROM orders
  GROUP BY user_id
) t
WHERE max_amount > 1000;
```

逻辑没错，但它先生成了一张中间结果表 `t`。  
如果你本来只是过滤聚合结果，更自然的写法是：

```sql
SELECT user_id, MAX(amount) AS max_amount
FROM orders
GROUP BY user_id
HAVING MAX(amount) > 1000;
```

这通常更直接，也更容易让数据库少走弯路。

### 7. `EXISTS` 很适合“只判断有没有”

有一类查询特别常见：

- 查出“有支付记录的订单”
- 查出“有登录行为的用户”
- 查出“有销售记录的商品”

这类问题的本质不是拿另一张表的数据，而是判断：

- 另一张表里有没有匹配行

这时 `EXISTS` 的语义非常贴切。

例如：

```sql
SELECT i.item_no
FROM items i
WHERE EXISTS (
  SELECT 1
  FROM sales_history sh
  WHERE sh.item_no = i.item_no
);
```

相比“先连接再 `DISTINCT`”，这种写法经常更自然，也常常更少去重成本。

### 8. 极值查询为什么常常值得单独优化

`MAX`、`MIN` 看起来只是聚合函数，但如果它们作用在合适索引列上，数据库有机会非常快地得到结果。

比如：

```sql
SELECT MAX(id) FROM orders;
```

如果 `id` 就是主键或有序索引列，数据库不一定需要把整张表扫描一遍。

这提醒我们一件事：

- 索引不只是“查等值和范围”
- 也可能帮排序、极值、分组省掉大量工作

### 9. 一条非常实用的总原则

以后你看到排序、分组、去重、子查询时，脑子里直接跑这三个问题：

1. 这个操作能不能在更小的数据集上做？
2. 这个去重 / 子查询是不是本来就不该出现？
3. 有没有更贴近业务语义的写法，比如 `EXISTS`、`UNION ALL`、`HAVING`？

只要你习惯了这么问，很多 SQL 会在写出来之前就已经避免掉慢的形态。

## 小结（3-5 条关键点）

- `ORDER BY`、`GROUP BY`、`DISTINCT`、`UNION` 都可能让数据库做额外的重操作。
- `WHERE` 用来过滤原始数据，`HAVING` 用来过滤聚合结果，能前置的条件尽量前置。
- `DISTINCT` 经常是在为前面产生的重复结果“擦屁股”，很多时候更应该从源头避免重复。
- 如果业务允许不去重，`UNION ALL` 往往比 `UNION` 更省。
- 判断存在性时，`EXISTS` 往往比“连接后再去重”更贴近问题本质。
