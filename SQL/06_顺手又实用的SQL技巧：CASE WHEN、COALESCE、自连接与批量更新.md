# SQL - 第 6 课：顺手又实用的 SQL 技巧：CASE WHEN、COALESCE、自连接与批量更新

## 学习目标（本节结束后你能做到什么）

- 理解 `CASE WHEN` 在统计和批量更新里的常见用途。
- 理解 `COALESCE` 是什么，以及它适合用在哪里、不适合用在哪里。
- 知道自连接能解决哪些问题，也知道哪些场景在 MySQL 8 里有更现代的写法。
- 能把“技巧”和“性能优化”分开看，不会把所有写法技巧都误当性能手段。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

前面几课主要讲的是性能。  
这一课补一些你工作里经常会顺手用到的 SQL 技巧。

这些技巧的价值主要有两个：

- 让 SQL 表达更直接
- 减少多次往返、多条 SQL、代码层额外处理

注意一件事：

**技巧不一定天然更快，但它们经常能让 SQL 更简洁，也更不容易写错。**

### 1. `CASE WHEN`：把“条件分支”写进 SQL

你可以把 `CASE WHEN` 理解成 SQL 里的 `if / else if / else`。

#### 1.1 条件统计特别好用

比如你要按不同条件做计数：

```sql
SELECT
  SUM(CASE WHEN status = 1 THEN 1 ELSE 0 END) AS paid_cnt,
  SUM(CASE WHEN status = 2 THEN 1 ELSE 0 END) AS refund_cnt
FROM orders;
```

这类写法的好处是：

- 一次扫描
- 同时产出多个统计指标

否则很多人会写成多条 SQL，或者查出来再在 Java 里遍历统计，既啰嗦又浪费。

#### 1.2 条件更新时非常稳

比如工资调整：

```sql
UPDATE salaries
SET salary = CASE
  WHEN salary >= 10000 THEN salary * 0.9
  WHEN salary < 10000 THEN salary * 1.2
  ELSE salary
END;
```

这种写法的核心价值是：

- 一条 SQL 内部把分支逻辑说清楚
- 避免你分两条 `UPDATE` 执行时互相影响

这也是为什么 `CASE WHEN` 常被用于：

- 条件更新
- 条件汇总
- 标签映射

### 2. `COALESCE`：从多个值里找第一个非空值

`COALESCE(a, b, c)` 的意思很简单：

- 从左到右，返回第一个不是 `NULL` 的值

最常见场景是展示默认值：

```sql
SELECT COALESCE(city, 'N/A')
FROM customers;
```

它很适合出现在：

- 查询结果展示
- 导出报表
- 默认字段兜底

但要特别注意一个性能坑：

如果你把它用在 `WHERE` 条件里包住索引列，比如：

```sql
WHERE COALESCE(city, 'N/A') = '北京'
```

那它就回到了前面讲过的问题：

- 对列做函数，索引利用可能变差

所以你可以记成一句话：

- `COALESCE` 很适合出现在 `SELECT` 里做展示
- 放进 `WHERE` 里要谨慎

### 3. `HAVING` 不只是跟 `GROUP BY` 一起用

大家最常见的是：

- 分组后用 `HAVING` 过滤聚合结果

但它本质上是“对聚合结果再筛一层”的语义。

比如：

```sql
SELECT sale_date, MAX(quantity)
FROM sales
GROUP BY sale_date
HAVING MAX(quantity) >= 10;
```

这类写法比“先套一层子查询，再外层 `WHERE`”更自然。

你可以把它理解成：

- 如果你要筛的是聚合结果本身，就优先考虑 `HAVING`

### 4. 自连接到底能干什么

自连接就是：

- 一张表和它自己连

很多人第一次学时觉得它很绕，但其实非常实用。

#### 4.1 去重删除

例如按业务字段去重，只保留 id 最大的一条：

```sql
DELETE p1
FROM products p1
JOIN products p2
  ON p1.name = p2.name
 AND p1.price = p2.price
 AND p1.id < p2.id;
```

这种写法的思路是：

- 给重复组里“不是最新的那几条”打上目标
- 再删除它们

#### 4.2 排名

早期 MySQL 没窗口函数时，自连接经常用来做排名。  
不过如果你现在用的是 MySQL 8，很多排名问题更推荐：

- `ROW_NUMBER()`
- `RANK()`
- `DENSE_RANK()`

所以这里你要形成一个工程判断：

- 自连接是有用的
- 但不是所有老技巧都还是今天的最优写法

### 5. 批量更新，比循环发 SQL 更像线上写法

很多业务代码会这样干：

- Java 里循环 1000 次
- 每次发一条 `UPDATE`

这样的问题是：

- 网络往返多
- 事务开销多
- 日志刷盘多

更稳的思路通常是：

#### 5.1 一条条件更新

如果更新规则一致，用一条 `UPDATE ... WHERE ...`。

#### 5.2 用 `CASE WHEN` 做多值批量更新

比如：

```sql
UPDATE user_score
SET score = CASE id
  WHEN 1 THEN 80
  WHEN 2 THEN 90
  WHEN 3 THEN 100
  ELSE score
END
WHERE id IN (1, 2, 3);
```

它的好处是：

- 一次发给数据库
- 一次执行完成
- 很适合后台批量修复任务

### 6. 技巧和性能，怎么区分看

你以后碰到 SQL 技巧，最好先问：

- 它是在提升表达能力？
- 还是在提升性能？
- 还是两者都有一点？

例如：

- `CASE WHEN`：主要提升表达能力，也可能减少多次扫描
- `COALESCE`：主要提升展示层可读性
- 自连接：主要是解决某类查询表达问题
- `HAVING`：既是语义工具，也可能避免不必要的中间表

只有你把这层区分想清楚，才不会把所有“写法技巧”都误当成“性能优化神招”。

### 7. 最后给你几个很实用的小建议

- 复杂统计能在一条 SQL 用条件聚合做完时，别拆成很多条再在 Java 里拼。
- 展示层默认值，优先在 `SELECT` 里处理，别把展示逻辑带进过滤逻辑。
- 批量修复数据时，优先想“按集合更新”，不要上来就写循环逐条改。
- 老文章里很多自连接技巧今天依然有价值，但如果你在 MySQL 8，先想想窗口函数是不是更直接。
- SQL 技巧的第一价值通常是“更不容易写错”，第二价值才是“可能更快”。

## 小结（3-5 条关键点）

- `CASE WHEN` 很适合条件统计和单条 SQL 内的分支更新。
- `COALESCE` 适合在查询结果里做空值兜底，但放在过滤条件里要小心索引利用问题。
- 自连接可以做去重、排名等问题，但在 MySQL 8 里很多场景可以优先考虑窗口函数。
- 批量更新通常比逐条更新更适合真实线上任务。
- SQL 技巧和性能优化不能混为一谈，很多技巧先解决的是“表达和正确性”。
