# SQL - 第 7 课：书写高质量 SQL 的 30 条建议：带边界的实战清单

## 学习目标（本节结束后你能做到什么）

- 能把常见的 SQL 优化建议变成一份真正可落地的日常检查清单。
- 知道哪些建议是“高概率正确”，哪些建议必须结合场景判断。
- 不再机械背“口诀”，而是能用更稳的工程语言解释为什么要这么写。
- 在评审 SQL、排查慢查询、设计接口查询时，有一套顺手可用的判断框架。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

网上关于 SQL 优化的文章很多，但它们常常有一个共同问题：

- 结论说得很猛
- 边界讲得不够

比如：

- “`EXISTS` 一定比 `IN` 快”
- “`OR` 一定会让索引失效”
- “索引不要超过 5 个”
- “一定要用 `varchar` 不要用 `char`”

这些话不能说完全错，但如果你把它们当成铁律，就很容易在真实项目里用反。

这节课就不再把前面 01 到 06 课已经展开讲过的原理再重复一遍了。  
它更适合作为一份“上线前 / Code Review 前 / 排查慢 SQL 前”的压缩清单来用。

如果你发现某条建议自己还讲不清为什么成立，直接回看前面的课：

- 查询成本与慢 SQL 直觉：第 01 课
- 索引、回表、覆盖索引、联合索引：第 02 课
- 函数、类型转换、`LIKE`、分页、`SELECT *`：第 03 课
- 排序、分组、去重、子查询、`HAVING`、`EXISTS`：第 04 课
- `EXPLAIN`、慢日志、`Using temporary/filesort`：第 05 课
- `CASE WHEN`、`COALESCE`、自连接、批量更新：第 06 课

### 这篇怎么用

你可以把下面 30 条建议理解成三类：

- `高频默认动作`：大多数接口和任务都值得优先这么做
- `有条件成立`：通常更好，但必须结合语义和执行计划判断
- `工程习惯`：不是单条 SQL 的“技巧”，但长期收益很高

---

### 一、先减少无谓数据

1. 优先写具体字段，别习惯性 `SELECT *`。这是最稳的默认动作。
2. 明确只要一条时，加 `LIMIT 1`，让数据库更早停。
3. 过滤条件尽量下推到 SQL，不要把“查回来再筛”交给 Java。
4. 控制结果集大小，列表、导出、任务都要有边界。
5. 能缩小时间范围、状态范围、租户范围，就不要查大宽区间。

这组建议的详细原因，回看第 01 课和第 03 课。

---

### 二、让索引更容易发挥作用

6. 不要在索引列上套函数，比如 `DATE(create_time)`。
7. 不要在索引列上做表达式运算，比如 `age - 1 = 10`。
8. 参数类型尽量和列类型一致，少给隐式类型转换制造机会。
9. `LIKE` 优先前缀匹配，前导 `%` 和中间 `%` 要格外谨慎。
10. 联合索引按“等值过滤 -> 范围过滤 -> 排序需求”设计，别只背最左匹配口诀。
11. 能形成覆盖索引时，尽量别把查询写成必须回表。
12. `WHERE` 和 `ORDER BY` 经常要一起看，不要只顾过滤不顾排序。
13. 低选择性字段不要单独迷信索引，更常见的用法是放进联合索引。
14. 别重复建索引，尤其是联合索引已经覆盖前导列时。
15. 索引数量要克制，但别背成“一个表不能超过 5 个索引”。

这组建议的原理和边界，回看第 02 课和第 03 课。

---

### 三、减少排序、分组、去重和中间结果

16. 原始数据就能过滤的条件，优先放 `WHERE`，别拖到 `HAVING`。
17. `DISTINCT` 慎用，先问清重复数据为什么会出现。
18. 已知不需要去重时，优先 `UNION ALL`。
19. 子查询不是原罪，但要警惕它制造大中间结果。
20. 多指标统计优先考虑条件聚合，少拆多条 SQL。

这组建议主要在第 04 课已经展开过，07 只保留结论。

---

### 四、分页、批量操作和大数据量处理

21. 深分页要警惕，接口优先考虑游标分页或延迟关联。
22. 批量插入通常优于逐条插入，但批量大小也别无限放大。
23. 大批量更新 / 删除尽量分批做，别一次改太多行。
24. 超大查询不一定继续抠 SQL，可能应该用缓存、归档、搜索或预计算。

这些更偏工程实战，详细展开见第 03 课和第 05 课。

---

### 五、连接、`IN/EXISTS`、`OR` 这些争议点

25. 语义允许时优先 `INNER JOIN`，但不要为了性能把结果语义写错。
26. `LEFT JOIN` 时，能前置过滤就尽量前置，尤其左表很大时。
27. `OR` 要慎用，但不要背成“用了就一定失效”。
28. `IN` 和 `EXISTS` 没有绝对赢家，先看业务语义，再看执行计划。
29. OLTP 接口不要无脑堆很多表连接，连表太多往往说明建模或职责有问题。

这组是最容易被经验文章误导的部分，建议结合第 03、04、05 课一起看。

---

### 六、表结构与日常习惯

30. 重要 SQL 上线前先跑 `EXPLAIN`，这是最值得长期坚持的一条习惯。

除了这 30 条，再额外记住 5 条常被讲得太绝对、但其实要看场景的建议：

- 真正只存数值语义的字段，优先用数值类型；但手机号、身份证号、订单编码这类“数字字符串”不要硬转数值。
- `varchar` 通常更通用，但固定长度字段也不是绝对不能用 `char`。
- `NULL` 设计要克制，但别为了“怕索引失效”就一刀切禁用。
- 大事务比单条慢 SQL 更容易放大事故，别只盯查询，不看事务边界。
- 写 SQL 时优先“可维护 + 可解释”，再谈微观优化。

## 小结（3-5 条关键点）

- 大多数 SQL 建议都不是铁律，而是“高概率正确的经验”。
- 真正靠谱的 SQL 编写方式，是先减少数据量，再让索引好发挥，再避免重排序和大中间结果。
- `OR`、`IN/EXISTS`、`varchar/char`、索引数量这些话题都要结合场景看，别背成绝对规则。
- 养成 `EXPLAIN`、控制结果集、控制分页深度、分批写入和删除的习惯，长期收益非常高。
- 高质量 SQL 的核心不是炫技巧，而是让数据库少做无谓工作，让人也更容易维护。
