# SQL - 第 7 课：书写高质量 SQL 的 30 条建议：带边界的实战清单

## 学习目标（本节结束后你能做到什么）

- 能把常见的 SQL 优化建议变成一份真正可落地的日常检查清单。
- 知道哪些建议是“高概率正确”，哪些建议必须结合场景判断。
- 不再机械背“口诀”，而是能用更稳的工程语言解释为什么要这么写。
- 在评审 SQL、排查慢查询、设计接口查询时，有一套顺手可用的判断框架。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

网上关于 SQL 优化的文章很多，但它们常常有一个共同问题：

- 结论说得很猛
- 边界讲得不够

比如：

- “`EXISTS` 一定比 `IN` 快”
- “`OR` 一定会让索引失效”
- “索引不要超过 5 个”
- “一定要用 `varchar` 不要用 `char`”

这些话不能说完全错，但如果你把它们当成铁律，就很容易在真实项目里用反。

所以这节课我会把这些建议重新整理成：

- 什么情况下值得优先这么做
- 为什么这么做通常更好
- 它的边界在哪里

你可以把这一课当成一份“后端写 SQL 的日常 checklist”。

---

### 一、先减少无谓数据

#### 1. 优先查具体字段，不要习惯性 `SELECT *`

更推荐：

```sql
SELECT id, name
FROM employee;
```

而不是：

```sql
SELECT *
FROM employee;
```

原因很直接：

- 少读无关列
- 少传无关数据
- 更容易命中覆盖索引

边界：

- 排查问题、临时验证时，`SELECT *` 不是原罪
- 但在线上接口、后台任务、热点查询里，最好只拿必须字段

#### 2. 如果只需要一条，就明确写 `LIMIT 1`

比如：

```sql
SELECT id, name
FROM employee
WHERE name = 'jay'
LIMIT 1;
```

它的价值是让数据库更早停下来。

边界：

- 如果条件本身是唯一索引，`LIMIT 1` 对性能帮助通常不大，但写上也能表达语义

#### 3. 只查你真正要用的数据，别把过滤放到 Java 里做

坏味道通常长这样：

- 先把一堆 id 全查出来
- 再在代码里 `contains`
- 或者全量查回后再筛

更稳的思路是：

- 让数据库在查询阶段就把无关数据过滤掉

数据库本来就是干这件事的。

#### 4. 控制返回结果集大小

如果一次返回几十万行，就算 SQL 本身不算慢，接口整体也会慢。

优先考虑：

- 分页
- 限制最大页数
- 限制导出时间范围
- 分批处理

#### 5. 能提前缩小时间范围、状态范围，就别查大宽区间

尤其是：

- 最近一年
- 全部状态
- 全部订单
- 全站搜索

这类需求特别容易把一个本来不大的接口，写成重查询。

### 二、让索引更容易发挥作用

#### 6. 不要在索引列上套函数

比如：

```sql
WHERE DATE(create_time) = '2026-04-10'
```

通常不如：

```sql
WHERE create_time >= '2026-04-10 00:00:00'
  AND create_time <  '2026-04-11 00:00:00'
```

原因是前者把列变形了，数据库更难顺着索引直接找。

#### 7. 不要在索引列上做表达式运算

比如：

```sql
WHERE age - 1 = 10
```

不如直接写成：

```sql
WHERE age = 11
```

原则很简单：

- 尽量让索引列原样参与比较

#### 8. 注意参数类型和列类型要一致

如果字段是字符串，就尽量传字符串；  
如果字段是数字，就尽量传数字。

原因不是“引号本身有魔法”，而是：

- 隐式类型转换可能增加成本
- 还可能让索引利用变差

#### 9. `LIKE` 优先使用前缀匹配

```sql
LIKE 'abc%'
```

通常比：

```sql
LIKE '%abc'
LIKE '%abc%'
```

更容易用到索引。

如果你业务上经常是“包含匹配”，很多时候应该考虑：

- 全文索引
- 搜索引擎

而不是让 MySQL 的普通索引硬扛。

#### 10. 联合索引要按查询模式设计，不要只会背最左匹配

你真正该思考的是：

- 哪些列是等值过滤
- 哪些列是范围过滤
- 哪些列要排序

联合索引顺序经常决定查询成败。

#### 11. 能形成覆盖索引时，尽量让查询只扫索引

覆盖索引不是非要刻意炫技，而是很朴素地省掉回表。

典型场景：

- 列表页
- 任务拉取待处理数据
- 按状态 + 时间查一页记录

#### 12. 排序字段和过滤字段要一起考虑索引

很多慢 SQL 不是慢在过滤，而是慢在：

- 先查出很多行
- 再额外排序

如果查询模式固定，排序又高频，就要把 `WHERE` 和 `ORDER BY` 一起看。

#### 13. 低选择性字段不要单独迷信索引

像这些字段经常区分度很低：

- 性别
- 是否删除
- 状态只有 2~3 种

单独给它们建索引，很多时候收益很一般。  
它们更常见的价值是作为联合索引的一部分。

#### 14. 索引不要重复建设

比如你已经有：

```text
(user_id, age)
```

那单独再建一个：

```text
(user_id)
```

很多时候就是冗余。

#### 15. 索引数量要克制，但没有“5 个以内”这种绝对铁律

有些文章喜欢说：

- 一个表索引别超过 5 个

这只能当经验提醒，不能当硬规则。

真正要看的是：

- 这张表的查询模式有多少种
- 写入频率高不高
- 这些索引是不是都在被真实使用

比起“数字”，更重要的是：

- 不要保留无用索引
- 不要堆重复索引

### 三、减少排序、分组、去重和中间结果

#### 16. `WHERE` 能过滤的条件，不要拖到 `HAVING`

如果条件本来作用在原始数据上，就尽量前置。

因为：

- 先过滤，再分组 / 聚合
- 通常比先分组一大堆再过滤更省

#### 17. `DISTINCT` 慎用，先问问为什么会有重复

很多时候 `DISTINCT` 只是在替前面的连接方式收拾残局。

所以先问：

- 是不是 join 把结果放大了？
- 能不能从源头避免重复？

#### 18. 已知不需要去重时，优先 `UNION ALL`

`UNION` 默认要去重，常常意味着额外成本。  
如果你知道两边结果不会重复，或者允许重复，优先：

```sql
UNION ALL
```

#### 19. 子查询不是原罪，但要警惕它制造大中间表

真正的问题不是“用了子查询”，而是：

- 它是不是先算出一张很大的临时结果
- 然后外层又过滤掉大半

如果只是过滤聚合结果，很多时候 `HAVING` 更自然。

#### 20. 大量统计时，优先考虑条件聚合

比如：

- 已支付数量
- 已退款数量
- 待发货数量

这类指标，很多时候一条 `CASE WHEN + SUM/COUNT` 就能完成，没必要拆很多条 SQL。

### 四、分页、批量操作和大数据量处理

#### 21. 深分页要警惕，优先考虑游标分页

`LIMIT 100000, 20` 这类写法在大表上经常越来越慢。

更适合线上接口的思路通常是：

- 基于主键 / 时间戳的游标分页
- 或先用窄索引找 id，再回表

#### 22. 批量插入比逐条插入更适合任务场景

这个建议非常实用，而且几乎总是成立。

原因包括：

- 网络往返更少
- 事务开销更少
- 日志刷盘更少

边界：

- 也不要一条 SQL 塞几万行，通常分批更稳

#### 23. 大批量更新 / 删除要分批做

一次删 10 万、100 万行，风险很大：

- 锁时间长
- 日志量大
- 容易影响线上

更稳的做法是：

- 按主键范围分批
- 每批几百到几千
- 观察执行时间和锁影响

#### 24. 超大查询不一定该继续抠 SQL，可能该改业务方案

例如：

- 全文模糊检索
- 超长时间范围导出
- 多维复杂筛选
- 热点榜单反复刷新

这时候可能更该考虑：

- 搜索引擎
- 缓存
- 归档表
- 预计算

### 五、连接、IN / EXISTS、OR 这些常见争议点

#### 25. 语义允许时，优先用 `INNER JOIN`

因为 `INNER JOIN` 的语义更直接：

- 只要匹配行

如果必须保留左表全部数据，那当然就该用 `LEFT JOIN`。  
不要为了“性能”把语义写错。

真正实用的建议是：

- 在不改变结果语义的前提下，尽量让连接前的数据集更小

#### 26. `LEFT JOIN` 时，能前置过滤就尽量前置

尤其是左表很大时，先在左侧缩小结果，再去连接右表，经常更划算。

但也要注意：

- 不是所有子查询包一层都会更快
- 最终还是要看执行计划

#### 27. `OR` 要慎用，但不是“一定失效”

`OR` 的真实问题是：

- 它有时会让优化器很难选出高效路径
- 特别是跨不同字段时，容易把扫描量拉大

这时可以考虑：

- 拆成两条 SQL
- 或改成 `UNION ALL`

但请记住：这是“常见优化方向”，不是死规则。

#### 28. `IN` 和 `EXISTS` 没有绝对赢家

更稳的理解是：

- 小常量列表过滤，用 `IN` 很正常
- 判断“是否存在关联记录”，`EXISTS` 经常更贴切
- 最终快不快，得看数据量、索引和执行计划

如果你只记一句话，就记这句：

**先看业务语义，再看执行计划，别背谁永远更快。**

#### 29. 表连接不要无脑堆太多，尤其是 OLTP 接口

一条查询连很多表，不一定一定错，但它经常意味着：

- 模型太耦合
- 查询链路太长
- 优化难度陡增

线上接口如果要连 6、7 张甚至更多表，就值得停下来问一句：

- 这是查询问题，还是建模 / 架构问题？

### 六、表结构、类型和日常习惯

#### 30. 养成 `EXPLAIN` 的习惯，重要 SQL 上线前先看计划

这是最值得你长期坚持的一条。

因为很多慢 SQL 在上线前其实就能看出来，例如：

- 没走索引
- 扫描行数太大
- `Using temporary`
- `Using filesort`
- 误选索引

对后端工程师来说，这比背更多“优化口诀”都更值钱。

### 额外补充：这 5 条也很实用

上面的 30 条已经够你日常用了，再额外补 5 条更偏工程实践的建议：

#### A. 真正只存数字的字段，优先用数值类型

像 id、状态码、数量、金额分这类字段，用数值类型通常更自然，也更省。

但如果它本质上是“看起来像数字的字符串”，比如：

- 手机号
- 身份证号
- 订单编码

那就别为了“省空间”硬改成数字类型。

#### B. `varchar` 通常比 `char` 更通用，但 `char` 不是不能用

`varchar` 更适合大部分变长文本。  
但如果字段长度固定，比如某些固定长度状态码、哈希片段，`char` 也不是完全不能用。

所以不是：

- “永远别用 `char`”

而是：

- “默认优先 `varchar`，固定长度场景再考虑 `char`”

#### C. `NULL` 设计要克制，但别为了怕索引就一刀切禁用

用默认值代替 `NULL` 有时能让语义和查询更简单。  
但也别为了“怕不走索引”就把所有 `NULL` 都抹掉。

关键是：

- 这个字段业务上到底有没有“未知 / 缺失”这个状态？

#### D. 大事务比单条慢 SQL 更容易放大事故

很多时候数据库抖动不是因为某条查询慢，而是：

- 一个事务改了太多行
- 锁持有太久
- 影响了一大片请求

#### E. 写 SQL 时优先“可维护 + 可解释”，再谈微观优化

线上真正可持续的 SQL，不只是“跑得快”，还要满足：

- 下一位同事看得懂
- 出问题时你能解释它为什么这么写
- 以后数据量放大后还有优化空间

## 小结（3-5 条关键点）

- 大多数 SQL 建议都不是铁律，而是“高概率正确的经验”。
- 真正靠谱的 SQL 编写方式，是先减少数据量，再让索引好发挥，再避免重排序和大中间结果。
- `OR`、`IN/EXISTS`、`varchar/char`、索引数量这些话题都要结合场景看，别背成绝对规则。
- 养成 `EXPLAIN`、控制结果集、控制分页深度、分批写入和删除的习惯，长期收益非常高。
- 高质量 SQL 的核心不是炫技巧，而是让数据库少做无谓工作，让人也更容易维护。
