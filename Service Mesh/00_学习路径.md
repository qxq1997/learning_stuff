# Service Mesh 学习路径与知识图谱

## 学习定位

Service Mesh 不是“又一个 RPC 框架”，也不是“给 Kubernetes 加一个更复杂的网络插件”。

它更准确的定位是：

**把服务治理能力从业务代码和语言 SDK 中抽出来，下沉到独立的网络基础设施层。**

很多工程师第一次接触 Service Mesh 时会有两个困惑：

- 明明 Dubbo、gRPC、Spring Cloud 已经能做服务发现、负载均衡、熔断、链路追踪，为什么还需要 Mesh？
- Sidecar、数据面、控制面、Istio、Envoy 这些词之间到底是什么关系？

这条学习路径的目标，就是把这些问题一步一步拆开。

学完之后，你应该能逐步具备这些能力：

- 解释 Service Mesh 到底在解决什么问题，它和 SDK 治理模式的边界在哪里。
- 理解 Sidecar、数据面、控制面的基本架构。
- 理解 Mesh 为什么在多语言、云原生、平台化治理场景中越来越重要。
- 理解流量管理、安全、可观测性为什么会被统一收拢到 Mesh 层。
- 面试时能把 Mesh 和 RPC、K8s、网关、Service Discovery 的关系讲清楚。

## 知识图谱（从基础到进阶）

1. 为什么需要 Service Mesh
   - SDK 模式的痛点
   - 多语言、升级、治理一致性问题
2. 核心架构
   - Sidecar
   - 数据面
   - 控制面
   - xDS / Envoy / Istio 的角色
3. Service Mesh 能做什么
   - 服务发现与负载均衡
   - 超时、重试、熔断
   - 金丝雀、镜像流量、故障注入
4. 可观测性与安全
   - 指标、日志、Trace
   - mTLS
   - 身份与策略
5. 落地取舍
   - 性能开销
   - 运维复杂度
   - 什么场景适合，什么场景不适合

## 建议学习顺序

1. `00_学习路径.md`
2. `01_为什么需要Service Mesh：从SDK治理到Sidecar.md`
3. `02_核心架构：数据面、控制面、Sidecar与Istio.md`

后续建议逐步扩展：

4. 流量治理：路由、金丝雀、故障注入
5. 可观测性：指标、日志、链路追踪
6. 安全：mTLS、身份认证、授权策略
7. 落地考量：性能开销、适用场景、替代关系

## 为什么这样排序

Service Mesh 最容易学偏的地方，是一上来就被 Istio CRD、YAML、Envoy 配置淹没。

但如果你没先理解：

- 它到底在解决什么问题
- 为什么 SDK 治理模式会在大规模服务里变得吃力
- Sidecar 模式为什么是关键转折点

那后面的控制面、数据面、VirtualService、DestinationRule 都会变成机械配置。

所以顺序必须是：

1. 先理解痛点
2. 再理解架构
3. 再理解它提供的治理能力
4. 最后再谈落地取舍

## 学完后你应该能回答的问题

- 为什么已有 RPC 框架和注册中心之后，还会出现 Service Mesh。
- Sidecar 为什么能做到对业务零侵入，它的代价又是什么。
- 数据面和控制面分别负责什么，为什么要分开。
- Istio、Envoy、Kubernetes、gRPC 这些组件在 Mesh 里分别站在哪一层。
- 什么团队适合上 Mesh，什么团队会被 Mesh 的复杂度反噬。
