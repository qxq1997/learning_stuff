# Service Mesh - 第 1 课：为什么需要 Service Mesh：从 SDK 治理到 Sidecar

## 学习目标（本节结束后你能做到什么）

- 理解 Service Mesh 的诞生背景，而不是把它当成一个凭空出现的新名词。
- 理解把服务治理能力做在 SDK 里的优点和局限。
- 说清楚语言绑定、升级困难、业务侵入为什么会在大规模微服务中变成痛点。
- 理解 Sidecar 模式为什么能成为 Mesh 的核心设计。
- 建立 Service Mesh 和 RPC、K8s、网关的关系坐标。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. 先从你已经熟悉的 RPC 体系出发

你已经知道，微服务之间调用不仅仅是“发一个请求”这么简单。

真实调用里往往需要一整套能力：

- 服务发现
- 负载均衡
- 超时
- 重试
- 熔断
- 降级
- 链路追踪
- 指标采集
- 安全认证

在传统模式里，这些能力通常以 SDK 的方式嵌进业务进程里。比如：

- Java 里用 Dubbo、Spring Cloud
- Go 里接一套自己的客户端库
- Python 里再做一套

这种方式当然能跑，而且在很多团队里长期很好用。

但当服务规模、团队规模、语言种类继续增长时，问题就会越来越明显。

### 2. SDK 模式的第一个痛点：语言绑定

假设你最开始全是 Java 服务，用一套 Java SDK 统一治理完全没问题。

可后来团队里出现了：

- Go 写的推荐服务
- Python 写的算法服务
- Rust 写的高性能组件

这时候麻烦来了：

- 每种语言都要有一套能力对齐的 SDK
- 每套 SDK 都要维护服务发现、负载均衡、熔断、重试、trace 透传
- 一旦某个策略升级，所有语言都得跟上

于是原本“治理能力下沉到 SDK”这件事，开始变成“治理能力被复制到很多语言实现里”。

这会导致：

- 能力不一致
- 行为不一致
- 升级节奏不一致

所以语言多样化，是推动 Service Mesh 出现的重要背景之一。

### 3. SDK 模式的第二个痛点：升级困难

假设你的治理 SDK 里修了一个很关键的 bug，比如：

- 重试策略有问题
- trace 上下文透传丢失
- 熔断阈值逻辑有缺陷

如果治理能力在 SDK 里，意味着你要：

- 升级每个业务服务依赖
- 回归测试
- 重新发版
- 等所有服务逐步更新

如果你有 200 个服务，这不是一个“发个版本”的问题，而是一次大规模协调工程。

最终很可能出现：

- 有些服务升级了
- 有些服务没升级
- 同一个公司里同时跑着好几个 SDK 版本

这时排查问题会非常痛苦，因为同样的调用逻辑，不同服务可能行为不一样。

### 4. SDK 模式的第三个痛点：业务侵入

理论上，业务开发者应该更关注：

- 订单怎么创建
- 库存怎么扣减
- 支付状态怎么流转

但当治理能力在 SDK 中时，业务代码往往不得不一起关心：

- 这个接口超时时间设多少
- 是否要重试
- 熔断规则是什么
- Trace 上下文怎么透传
- TLS / mTLS 怎么处理

于是业务代码和治理逻辑越来越缠在一起。

从软件工程角度看，这其实是一种横切关注点污染业务代码的问题。

### 5. Service Mesh 的核心思想：把治理逻辑从业务进程里拆出来

既然问题出在“治理能力嵌在业务进程里”，那一个自然想法就是：

**把这些能力从业务代码中剥离出来，放到一个独立的进程里。**

这个独立进程就是 Sidecar。

它和业务服务部署在一起，通常同 Pod、同主机、同网络命名空间附近工作，专门负责处理这个服务所有进出流量。

于是系统就变成了：

- 业务进程专心做业务
- Sidecar 专心做治理

### 6. Sidecar 为什么叫边车

这个名字非常形象。

摩托车边上挂一个边车，本体和边车一起行动，但职责不同：

- 摩托车负责主要行驶
- 边车承载额外功能

Service Mesh 里的 Sidecar 也是类似关系：

- 业务服务仍然是“主角”
- Sidecar 附着在旁边，接管通信和治理

这就让：

- 业务逻辑
- 通信治理逻辑

在进程边界上实现了分离。

### 7. Mesh 到底“Mesh”在什么地方

有了 Sidecar 之后，每个服务旁边都会有一个代理进程：

```text
服务 A <-> Sidecar A
服务 B <-> Sidecar B
服务 C <-> Sidecar C
```

服务之间的流量不再是业务进程直接互调，而更像：

```text
业务 A -> Sidecar A -> Sidecar B -> 业务 B
```

大量 Sidecar 之间形成一层专门处理服务间通信的网络，这就是“Mesh”。

所以 Service Mesh 不是单个代理，而是：

**由很多 Sidecar 组成的一层服务通信网络。**

### 8. 它和 RPC 框架、网关、K8s 分别是什么关系

这个关系一定要讲清楚。

#### 和 RPC 框架的关系

RPC 框架把治理能力更多做在 SDK 中；Service Mesh 把治理能力下沉到代理层。

两者不是完全替代关系，而是不同治理边界。

#### 和 API 网关的关系

网关更偏“南北流量”，也就是外部请求进入系统。

Service Mesh 更偏“东西流量”，也就是服务与服务之间的内部调用。

#### 和 Kubernetes 的关系

K8s 解决的是编排、调度、基础服务发现。

Service Mesh 在 K8s 之上进一步提供：

- 更细粒度的流量治理
- 可观测性
- 安全策略

所以很多 Mesh 方案都天然和 K8s 结合得很紧。

### 9. Service Mesh 真正解决的问题是什么

把前面总结起来，它主要在解决这些系统性痛点：

- 多语言服务治理能力不一致
- SDK 升级成本高
- 业务代码被基础治理逻辑污染
- 流量治理和安全策略难以统一下发
- 可观测性采集不统一

所以更准确地说，Service Mesh 不是单纯“加一个代理”，而是在推动：

**服务通信能力的平台化、统一化、基础设施化。**

### 10. 它的代价也必须提前看到

任何技术设计都有代价，Mesh 也不例外。

它会引入：

- 额外代理进程
- 更多资源消耗
- 更多网络跳转
- 更复杂的运维和调试路径

所以 Mesh 适合的是：

- 服务规模较大
- 多语言明显
- 平台治理诉求强
- 团队有能力维护这套基础设施

如果团队还很小、服务很少、语言单一，Mesh 很可能是复杂度过度前置。

## 小结

- Service Mesh 的背景，不是“RPC 不行了”，而是 SDK 治理模式在大规模、多语言场景下暴露出了边界。
- SDK 模式的主要痛点包括语言绑定、升级困难和业务侵入。
- Sidecar 的核心意义，是把通信治理能力从业务进程剥离出来。
- Service Mesh 是由大量 Sidecar 组成的一层服务通信网络，主要处理东西向流量。
- Mesh 很强，但也有资源和运维代价，不是所有团队都应该一上来就上。
