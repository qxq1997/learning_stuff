# Spring 面试专题学习路径

这组笔记把 Spring、Spring MVC、Spring Boot、Spring Cloud 和常见生态集成放在同一个专题下，但重点不是罗列 API，而是把面试中最常被追问的链路讲清楚：对象如何交给容器管理，代理如何把横切逻辑织入业务，Bean 为什么会有生命周期和三级缓存，事务为什么看起来加了注解却不生效，请求如何从 `DispatcherServlet` 走到 Controller，Boot 又是怎样把自动配置装配进来的。

## 阅读顺序

1. `01_Spring整体认知_核心模块_IOC_DI_AOP与企业分层.md`
   先建立 Spring 的全局地图：核心容器、数据访问、Web、AOP、测试、消息等模块分别解决什么问题。

2. `02_IoC容器_BeanFactory_ApplicationContext_依赖注入.md`
   理解 IoC 和 DI 的本质，以及 `BeanDefinition`、`BeanFactory`、`ApplicationContext` 的关系。

3. `03_AOP与动态代理_切点通知_JDK代理_CGLIB与失效边界.md`
   掌握 AOP 术语、JDK 动态代理、CGLIB 代理和常见失效边界。

4. `04_Bean生命周期_作用域_后置处理器与扩展点.md`
   记住 Spring 创建 Bean 的主线，以及每个扩展点适合做什么。

5. `05_循环依赖与三级缓存_早期引用_ObjectFactory_AOP代理.md`
   深入单例循环依赖为什么能解决、构造器循环依赖为什么不能解决、三级缓存为什么不是简单的 Map 叠加。

6. `06_Spring事务_传播机制_隔离级别_回滚规则与失效场景.md`
   按事务代理、传播行为、隔离级别、回滚规则、失效场景来组织答案。

7. `07_SpringMVC_DispatcherServlet_执行链_视图解析与REST.md`
   复盘一次 HTTP 请求进入 Spring MVC 后的完整处理链。

8. `08_SpringBoot_启动流程_自动配置_条件装配与Starter.md`
   理清 Boot 的启动过程、自动配置来源、条件装配和自定义 Starter。

9. `09_Web请求扩展点_Filter_Interceptor_AOP与ControllerAdvice.md`
   对比 Filter、Interceptor、AOP、`ControllerAdvice` 的执行位置和适用场景。

10. `10_SpringCloud与微服务_注册配置网关熔断链路追踪.md`
    建立微服务组件图谱，知道注册中心、配置中心、网关、负载均衡、熔断限流、链路追踪各自解决什么问题。

11. `11_Spring与MyBatis集成_JDBC模板_Mapper代理_参数绑定.md`
    把数据访问层从 Spring 视角串起来：JDBC、事务、Mapper 代理、参数绑定和常见集成问题。

## 面试复习节奏

第一轮先读每篇的“核心结论”和“主流程”，目标是能用自己的话讲出链路。第二轮重点看“追问”和“失效场景”，把边界条件补齐。第三轮用白纸画出这些链路：IoC 创建 Bean、AOP 代理创建、三级缓存、事务代理、MVC 请求流、Boot 自动配置。能画出来，基本就能讲清楚。

## 高频主线

- Spring 最核心的价值：把对象创建、依赖装配、生命周期、配置解析、扩展点和横切能力收拢到容器里，让业务代码更关注业务本身。
- IoC 是控制权反转，DI 是把依赖注入对象的手段；IoC 是思想，DI 是实现方式之一。
- AOP 依赖代理，代理只能拦住“从代理对象发出的调用”。自调用、非 Spring Bean 调用、部分不可代理方法，是很多注解失效的根源。
- Bean 生命周期是一条主线加很多钩子：实例化、属性填充、Aware、初始化前后处理器、初始化方法、销毁方法。
- 三级缓存不是为了“多缓存一点”，核心是让早期引用可以被延迟加工，尤其能和 AOP 代理协调。
- 声明式事务本质是 AOP 增强，失效排查时先问：是不是走了代理？异常是否被抛出？事务管理器是否匹配？传播行为是否符合预期？
- Spring MVC 的核心是 `DispatcherServlet` 前端控制器，配合 HandlerMapping、HandlerAdapter、HandlerExceptionResolver、ViewResolver 或 HttpMessageConverter 完成请求分发。
- Spring Boot 的核心不是替代 Spring，而是用自动配置、Starter 和约定优于配置降低工程装配成本。
- Spring Cloud 不是单个框架，而是一组分布式系统模式的实现集合：服务发现、配置、调用、网关、熔断限流、链路追踪、分布式事务等。
