# 项目原则与约束

本文档定义 TalentHub 的基础工程约束。后续架构设计、代码实现、接口风格和异常处理策略都必须遵守这里的原则。

## 1. 项目定位

TalentHub 是一个用于企业培训、题库管理、考试评估和学习分析的 AI 驱动系统。

当前项目以个人私有仓库形式开发，部署目标以本地环境为主，不依赖公有云基础设施。

## 2. 技术选择原则

- 本地优先：默认支持开发机或本地服务器部署
- 开源优先：数据库、中间件、向量检索、任务队列优先选择开源方案
- 简单优先：先用模块化单体完成业务闭环，再决定是否拆分服务
- 可演化优先：设计必须允许后续逐步接入 Agent、模型路由、评测体系和训练能力

## 3. 失败策略

本项目采用明确的 `fail fast` 策略。

### 3.1 允许的行为

- 输入不合法时直接拒绝并返回明确错误
- 配置缺失时启动失败
- 数据状态不一致时中断流程
- 外部依赖不可用时返回显式失败，而不是悄悄换成其他路径

### 3.2 明确禁止的行为

- 不做“猜测用户本意”的兜底逻辑
- 不做多层 fallback 链式补偿
- 不做难以追踪的静默降级
- 不用补丁式代码把设计缺陷临时糊住
- 不在领域规则不清晰时擅自放宽约束

### 3.3 实施要求

- 配置必须显式声明并校验
- 领域对象创建时必须校验不变量
- 状态迁移必须受限，非法迁移直接失败
- 异常必须可观察、可定位、可复现

## 4. 文档策略

设计文档是本项目的一等产物，不是代码完成后的补充说明。

### 4.1 文档要求

- 所有核心设计以中文沉淀在仓库中
- 文档要清晰、可读、可维护，面向未来接手项目的人
- 架构、领域模型、数据库、接口和部署方案都要有对应文档
- 代码变更涉及设计调整时，必须同步更新文档

### 4.2 文档目标

- 帮助当前开发者理解系统边界和规则
- 帮助后续维护者快速进入上下文
- 帮助对照代码验证设计是否落地

## 5. 架构原则

- 使用 DDD 进行领域建模
- 使用模块化单体承载限界上下文
- 每个领域模块按 `domain`、`application`、`infrastructure`、`interfaces` 分层
- 通过领域事件和应用服务协调跨模块行为

## 6. AI 能力接入原则

- AI 能力首先服务于明确业务流程，而不是反过来主导流程
- 题目生成、判卷、反馈、知识检索都属于业务链路中的可替换能力
- Agent、ReAct、MCP 属于第二阶段增强能力，不在第一阶段作为系统主干
- Prompt、RAG、评分规则和结构化输出优先于盲目引入训练或多 Agent 编排

## 7. 近期默认技术路线

- 前端：React + TypeScript
- 后端：FastAPI
- 数据库：PostgreSQL
- 向量检索：Qdrant
- 缓存与任务：Redis + Celery 或 Dramatiq
- 文件存储：本地文件系统或 MinIO

## 8. 当前阶段工作顺序

1. 固定项目原则和架构边界
2. 确认 DDD 领域模型和上下文划分
3. 设计数据库模型和持久化映射
4. 搭建后端骨架
5. 搭建前端骨架
6. 打通最小业务闭环
