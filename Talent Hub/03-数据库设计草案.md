# 数据库设计草案

本文档给出 TalentHub 第一版数据库设计草案。该设计服务于 DDD 建模，不以页面结构为中心，而以领域对象和聚合边界为中心。

## 1. 数据库选型

推荐：

- `PostgreSQL`
- `Qdrant`

原因：

- 结构化数据和向量数据分层更清晰
- 适合本地部署
- 开源、成熟、生态稳定
- PostgreSQL 继续承载业务真相，Qdrant 承载检索

## 2. 建模原则

- 聚合根对应核心主表
- 历史版本必须保留，尤其是文档和题目
- 值对象和扩展属性优先考虑 `jsonb`
- 跨上下文引用通过 ID，避免数据库层强耦合
- 不为了“灵活”而增加含糊字段

## 4. 关键设计说明

### 4.1 为什么要有 `question_versions`

题目被修改后，历史考试仍然必须引用原版本题目，否则会导致历史作答和标准答案错位。

### 4.2 为什么 `exam_paper_questions` 必须保存 `question_version_id`

试卷是一次发布快照。试卷一旦发布，就必须绑定具体题目版本，而不是只引用题目 ID。

### 4.3 为什么 `exam_paper_tags` 不能省略

考试页面现在支持“按标签筛题后组卷”。如果创建试卷时不保存这一层标签上下文，后面只看试卷内容时就不知道：

- 这套试卷原本想覆盖哪些主题
- 当时是围绕哪些标签筛出来的
- 后续再按标签维护试卷时该从哪里接着做

因此 `exam_paper_tags` 的定位不是冗余字段，而是试卷编纂行为的显式持久化结果。

### 4.3 为什么 `knowledge_document_versions` 需要独立出来

企业文档会更新。如果只有一张文档表，就无法追踪“某道题目究竟基于哪一版文档生成”。

### 4.4 为什么答案先用 `jsonb`

不同题型的答案结构差异很大，例如：

- 单选题是单值
- 多选题是数组
- 简答题是文本
- 案例题可能是结构化分段文本

第一版使用 `jsonb` 可以保持模型清晰，同时避免为每种题型拆过多表。

### 4.5 为什么标签独立成 `question_tags`

标签本质上是“一题多值”的业务属性。

如果把它塞进 `questions.tags_json`：

- 约束不清晰
- 去重困难
- 按标签筛题和按标签组卷不自然
- 后续索引也不够直接

因此第一版明确使用独立表承载题目标签。
