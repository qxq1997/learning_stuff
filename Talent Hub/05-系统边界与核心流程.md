# 系统边界与核心流程

本文档回答两个问题：

1. TalentHub 到底负责什么
2. TalentHub 明确不负责什么

只有边界清晰，后面的领域建模、数据库设计和接口设计才不会发散。

## 1. 系统边界

### 1.1 系统负责的内容

TalentHub 负责以下业务能力：

- 企业内部培训知识的导入、切片和检索
- 基于知识上下文生成培训题目
- 题目版本化管理和题库维护
- 基于题库进行组卷与考试分配
- 用户作答、提交和评分
- 基于 AI 生成逐题反馈与总结建议
- 学习路径、进度和成长趋势管理
- 团队与平台层面的指标统计和报表

### 1.2 系统不负责的内容

第一阶段明确不做：

- 通用聊天产品
- 企业 HR 主数据系统
- 企业统一身份平台
- 复杂审批流引擎
- 多租户 SaaS 能力
- 多 Agent 自主协作平台
- 模型训练平台

如果未来需要接入这些能力，应通过清晰边界集成，而不是把 TalentHub 自身做成“大一统平台”。

## 2. 外部依赖边界

TalentHub 当前依赖但不拥有的能力包括：

- 大模型推理服务
- embedding 模型服务
- 本地对象存储
- PostgreSQL
- Redis

系统对这些能力的要求是“依赖明确、失败即显式报错”，不做偷偷切换。

## 3. 核心业务流程

## 3.1 文档入库流程

1. 管理员上传文档
2. 系统创建逻辑文档和文档版本
3. 系统解析文本并切片
4. 系统生成 embedding 并写入 `knowledge_chunks`
5. 文档状态进入 `indexed`

任何一步失败，流程直接终止，文档状态进入 `failed`，等待人工处理。

## 3.2 题目生成流程

1. 用户选择知识点、题型、难度和数量
2. 系统检索相关知识片段
3. 系统构造 prompt 并请求模型生成结构化题目
4. 系统校验输出结构
5. 系统创建题目聚合和题目版本
6. 系统记录来源 chunk 和知识点映射

结构化输出不合法时直接失败，不做“猜测模型本意”的修正。

## 3.3 组卷流程

1. 管理员选择题目版本或规则
2. 系统创建试卷
3. 系统将题目快照写入试卷题目表
4. 管理员发布试卷

发布后试卷视为快照，不允许无痕修改。

## 3.4 考试流程

1. 管理员向用户分配试卷
2. 用户开始考试
3. 系统创建一次 `exam_attempt`
4. 用户作答并提交
5. 系统锁定作答
6. 客观题立即判分
7. 主观题进入 AI 评估
8. 汇总总分与逐题反馈

当前第一版已经做成的实际链路是：

- 显式创建考试分配
- 开始一次考试尝试
- 提交整套答案
- 同步完成 AI 判卷
- 返回总评和逐题反馈

## 3.5 学习闭环流程

1. 考试完成后生成反馈
2. 系统识别知识薄弱项
3. 系统将结果写入学习路径或能力画像
4. 用户查看个人趋势
5. 管理者查看团队趋势和报表

## 4. 核心角色

- 员工：学习、参加考试、查看反馈
- 题库管理员：维护知识库、题目、试卷
- 培训管理员：安排培训路径和考试
- 管理者：查看团队和平台指标

## 5. 第一阶段的主链路

第一阶段只追求一个闭环：

`文档入库 -> RAG 检索 -> 题目生成 -> 题库存储 -> 组卷 -> 考试 -> AI 判卷 -> 反馈`

这个闭环通了，项目才算真正站住。
