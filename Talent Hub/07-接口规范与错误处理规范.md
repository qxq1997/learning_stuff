# 接口规范与错误处理规范

本文档定义 TalentHub 的 HTTP API 风格、错误响应格式和失败处理原则。

## 1. API 总体规范

- 路径前缀：`/api/v1`
- 风格：REST 风格优先
- 数据格式：`application/json`
- 时间格式：ISO 8601，统一使用带时区时间
- 主键格式：UUID

当前开发阶段补充约定：

- 公开接口只保留真正的匿名入口
- 其余接口默认要求 `Authorization: Bearer <token>`
- 后端会基于 token 解析当前用户、角色和部门

## 2. 路径设计规则

### 2.1 使用名词，不使用动词

推荐：

- `GET /api/v1/exams/papers/{exam_paper_id}`
- `POST /api/v1/question-bank/questions`

谨慎使用动作型路径，只有确实表达状态迁移时才允许：

- `POST /api/v1/exams/attempts/{attempt_id}/submit`

### 2.2 资源命名统一

- 用户：`users`
- 文档：`knowledge/documents`
- 题目：`questions`
- 试卷：`exams/papers`
- 作答：`exams/attempts`

## 3. 成功响应约定

默认返回业务数据，不强制包一层 `data` 外壳。

示例：

```json
{
  "id": "7b2b2ec0-5e84-4ac4-821c-7b9cf46d6baf",
  "title": "新员工安全考试",
  "status": "published"
}
```

## 4. 错误响应约定

统一返回：

```json
{
  "error": {
    "code": "domain_invariant_violation",
    "message": "exam attempt has already been submitted",
    "details": {}
  }
}
```

字段定义：

- `code`：稳定错误码，供前端和日志检索使用
- `message`：给开发者和调用方看的简明错误信息
- `details`：结构化上下文，不放无意义文本堆栈

## 5. 错误码建议

- `request_validation_failed`
- `unauthorized`
- `forbidden`
- `resource_not_found`
- `domain_error`
- `domain_invariant_violation`
- `conflict`
- `internal_error`

## 6. HTTP 状态码规则

- `200`：成功查询或成功执行
- `201`：资源创建成功
- `204`：删除或无响应体成功
- `400`：通用业务请求错误
- `401`：未认证
- `403`：已认证但无权限
- `404`：资源不存在
- `409`：状态冲突或领域不变量冲突
- `422`：请求结构校验失败
- `500`：未处理异常

当前实现需要注意：

- 缺少 Bearer token 时，会直接返回认证失败
- token 非法、过期或与当前用户不匹配时，也会直接失败
- 普通成员调用造题 / 造卷管理接口时，会按领域不变量错误返回 `409`

这符合当前“正式登录已接入、业务权限仍由领域层明确控制”的设计。

## 7. 失败处理原则

### 7.1 fail fast

- 参数错误直接返回 `422`
- 资源状态不允许直接返回 `409`
- 依赖缺失或配置错误直接启动失败
- AI 输出不合格直接判定为失败，不做隐式修正

### 7.2 不允许的错误处理方式

- 不吞异常
- 不静默返回空结果当作成功
- 不做多层 fallback
- 不把业务错误伪装成成功响应

## 8. 幂等相关接口约定

对以下接口，后续实现中必须支持 `Idempotency-Key`：

- 文档导入
- AI 题目生成
- 考试提交
- AI 判卷触发

如果同一个 key 的请求体不一致，必须直接返回冲突错误。
