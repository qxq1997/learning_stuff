# 状态流转、幂等与一致性

本文档定义 TalentHub 关键业务对象的状态机、幂等要求和一致性边界。

## 1. 状态流转原则

- 状态必须有限且可枚举
- 状态迁移必须显式定义
- 非法状态迁移直接失败
- 不允许通过补丁逻辑自动修正非法状态

## 2. 文档状态

状态：

- `uploaded`
- `parsed`
- `indexed`
- `archived`
- `failed`

允许迁移：

- `uploaded -> parsed`
- `parsed -> indexed`
- `uploaded -> failed`
- `parsed -> failed`
- `indexed -> archived`

不允许迁移：

- `failed -> indexed`
- `archived -> parsed`

## 3. 题目状态

状态：

- `draft`
- `review`
- `published`
- `archived`

允许迁移：

- `draft -> review`
- `review -> published`
- `published -> archived`

补充规则：

- 编辑任何题目都会新建 `question_versions`
- 当前管理员编辑题目后，新版本会直接生效
- `review` 状态仍保留在底层枚举中，主要用于兼容已有数据，不再作为日常人工审核流的一部分

## 4. 试卷状态

状态：

- `draft`
- `review`
- `published`
- `archived`

允许迁移：

- `draft -> review`
- `review -> published`
- `published -> archived`

补充规则：

- 当前管理员创建试卷后直接进入可用状态
- `review` 状态仍保留在底层枚举中，主要用于兼容已有数据，不再作为日常人工审核流的一部分
- `published` 后不允许静默改题
- `archived` 后不可再分配

## 5. 考试尝试状态

状态：

- `in_progress`
- `submitted`
- `graded`
- `abandoned`

允许迁移：

- `in_progress -> submitted`
- `submitted -> graded`
- `in_progress -> abandoned`

规则：

- `submitted` 必须有 `submitted_at`
- `graded` 必须有最终得分和通过标记
- 已提交后禁止再次提交

当前第一版实现采用“同步判卷提交”：

- 提交成功后最终持久化状态直接落在 `graded`
- 如果 AI 判卷失败，则整次提交失败，不会写入半套答案和反馈

这个实现偏向强一致，后面如果要做异步判卷，再把 `submitted` 作为更稳定的中间持久化状态单独展开。

## 5.1 考试可见性规则

考试除了状态外，还有单独的可见性：

- `public`
- `private`

规则：

- `public` 已发布考试，所有已知账号都可以查看和使用
- `private` 已发布考试，只允许本部门访问
- 跨部门访问私有考试时直接失败，不做静默降级

## 6. 学习路径状态

路径状态：

- `draft`
- `published`
- `archived`

用户路径进度状态：

- `not_started`
- `in_progress`
- `completed`
- `blocked`

## 7. 幂等策略

## 7.1 需要幂等的场景

- 文档重复导入
- AI 题目生成重试
- 考试提交重试
- AI 判卷任务触发重试

## 7.2 幂等实现原则

- 调用方必须提供 `Idempotency-Key`
- 服务端必须保存请求摘要和处理结果
- 相同 key 且相同请求体可以返回同一结果
- 相同 key 但不同请求体必须直接返回 `409`

## 7.3 计划中的持久化支持

我们会通过 `idempotency_records` 表实现幂等记录。

## 8. 一致性边界

### 8.1 强一致

以下操作必须在单事务内完成：

- 创建题目及题目版本
- 创建试卷及试卷题目快照
- 提交考试并锁定作答
- 保存评分结果和逐题反馈

### 8.2 最终一致

以下行为可以通过事件异步完成：

- 指标聚合
- 报表快照
- 学习路径推荐
- 通知类任务

## 9. 一致性优先级

优先保障顺序如下：

1. 核心业务正确性
2. 状态可追踪
3. 历史快照稳定
4. 指标最终同步

如果两者冲突，绝不为了“看起来更智能”而牺牲业务正确性。
