# 状态流转、幂等与一致性

本文档定义 TalentHub 关键业务对象的状态机、幂等要求和一致性边界。

## 1. 状态流转原则

- 状态必须有限且可枚举
- 状态迁移必须显式定义
- 非法状态迁移直接失败
- 不允许通过补丁逻辑自动修正非法状态

## 2. 文档状态

状态：

- `uploaded`
- `parsed`
- `indexed`
- `archived`
- `failed`

允许迁移：

- `uploaded -> parsed`
- `parsed -> indexed`
- `uploaded -> failed`
- `parsed -> failed`
- `indexed -> archived`

不允许迁移：

- `failed -> indexed`
- `archived -> parsed`

## 3. 题目状态

状态：

- `draft`
- `review`
- `published`
- `archived`

允许迁移：

- `draft -> review`
- `review -> published`
- `published -> archived`

补充规则：

- 编辑任何题目都会新建 `question_versions`
- 当前管理员编辑题目后，新版本会直接生效
- `review` 状态仍保留在底层枚举中，主要用于兼容已有数据，不再作为日常人工审核流的一部分

## 4. 试卷状态

状态：

- `draft`
- `review`
- `published`
- `archived`

允许迁移：

- `draft -> review`
- `review -> published`
- `published -> archived`

补充规则：

- 当前管理员创建试卷后直接进入可用状态
- `review` 状态仍保留在底层枚举中，主要用于兼容已有数据，不再作为日常人工审核流的一部分
- `published` 后不允许静默改题
- `archived` 后不可再分配

## 5. 考试尝试状态

状态：

- `in_progress`
- `submitted`
- `graded`
- `abandoned`

允许迁移：

- `in_progress -> submitted`
- `submitted -> graded`
- `in_progress -> abandoned`

规则：

- `submitted` 必须有 `submitted_at`
- `graded` 必须有最终得分和通过标记
- 已提交后禁止再次提交

当前第一版实现采用“同步判卷提交”：

- 提交成功后最终持久化状态直接落在 `graded`
- 如果 AI 判卷失败，则整次提交失败，不会写入半套答案和反馈

这个实现偏向强一致，后面如果要做异步判卷，再把 `submitted` 作为更稳定的中间持久化状态单独展开。

当前这部分约束已经进入 [backend/sql/001_init_schema.sql](/Users/xinqi/WebstormProjects/TalentHub/backend/sql/001_init_schema.sql)。

## 5.1 考试可见性规则

考试除了状态外，还有单独的可见性：

- `public`
- `private`

规则：

- `public` 已发布考试，所有已知账号都可以查看和使用
- `private` 已发布考试，只允许本部门访问
- 跨部门访问私有考试时直接失败，不做静默降级

## 6. 学习路径状态

路径状态：

- `draft`
- `published`
- `archived`

用户路径进度状态：

- `not_started`
- `in_progress`
- `completed`
- `blocked`

## 7. 幂等策略

## 7.1 需要幂等的场景

- 文档重复导入
- AI 题目生成重试
- 考试提交重试
- AI 判卷任务触发重试

## 7.2 幂等实现原则

- 调用方必须提供 `Idempotency-Key`
- 服务端必须保存请求摘要和处理结果
- 相同 key 且相同请求体可以返回同一结果
- 相同 key 但不同请求体必须直接返回 `409`

## 7.3 计划中的持久化支持

我们会通过 `idempotency_records` 表实现幂等记录。

当前更准确的边界是：TalentHub 已经设计好了正式 API 幂等的数据库结构和目标行为，但还没有把 `Idempotency-Key` 的读取、请求摘要计算、幂等记录查询/写入和响应重放完整接到所有创建类接口里。

也就是说，现在有的是幂等基础设施雏形，不是“所有创建类接口都已经支持 `Idempotency-Key` 重放”。

确认依据：

- 幂等表已经在 `backend/sql/001_init_schema.sql` 中定义，核心约束是 `uq_idempotency_records_scope_key`
- ORM 模型已经在 `backend/app/modules/analytics/infrastructure/models.py` 中定义
- 幂等状态枚举已经在 `backend/app/shared/infrastructure/db/enums.py` 中定义
- 本文档已经定义目标行为：相同 key 且相同请求体重放结果，相同 key 但不同请求体返回 `409`
- 当前代码全局搜索只命中表、模型和枚举定义，没有命中业务链路中读取 `Idempotency-Key`、计算 `request_hash`、查询/写入 `idempotency_records`、命中后重放 `response_json` 的实际调用逻辑

因此当前状态可以拆成三层：

- 数据库层：已准备
- 文档层：已设计
- 接口落地层：还没全面接完

## 7.4 `idempotency_records` 表设计

核心字段：

- `scope`
- `idempotency_key`
- `request_hash`
- `status`
- `resource_type`
- `resource_id`
- `response_json`
- `expires_at`

字段语义：

- `scope`：幂等作用域，用来区分不同接口或不同业务动作。比如未来可以是 `question_preview:web_page`、`exam_attempt_submission`、`knowledge_import`。同一个 key 不一定全局唯一，但在同一个业务动作里必须唯一。
- `idempotency_key`：客户端传入的幂等键，一般应该是 UUID 或高熵随机串。它和 `scope` 组成唯一约束，同一个 `scope` 下同一个 `idempotency_key` 只能有一条记录。
- `request_hash`：请求摘要，用来防止“同一个 key 被拿去干两件不同的事”。只有相同 key 且相同 hash 才算重试；相同 key 但不同 hash 应该直接冲突。
- `status`：处理状态，当前枚举包括 `processing`、`succeeded`、`failed`。
- `resource_type` / `resource_id`：记录这次幂等请求最终创建或影响了哪个业务资源，例如 `exam_paper` 加具体资源 ID。
- `response_json`：记录第一次成功处理后的响应体，便于后续相同请求直接重放结果。
- `expires_at`：记录过期时间，幂等记录不应该永久保留，后续可以按过期时间清理，避免表无限增长。

`status` 的语义：

- `processing`：第一个请求已经进入系统，但业务还没执行完。
- `succeeded`：请求已经成功处理完，可以直接重放结果。
- `failed`：请求处理失败。后续是否允许相同 key 重试，需要按具体接口语义决定。

## 7.5 正式落地后的标准流程

如果这套机制完整接到 API，预期流程是：

1. 客户端发起写请求，例如创建知识文档、创建试卷、生成题目预览、提交考试，并带上 `Idempotency-Key: <uuid>`。
2. 服务端计算请求摘要，把请求方法、业务 `scope`、规范化后的请求体、必要时的用户身份一起纳入 hash，得到 `request_hash`。
3. 服务端尝试插入一条 `processing` 记录。如果插入成功，说明这是第一次请求，当前请求获得执行权。
4. 如果唯一约束冲突，则读取旧记录并分流处理。
5. 真正执行业务逻辑，例如创建考试、写数据库、提交 background job、保存结果。
6. 业务成功时更新幂等记录为 `succeeded`，写入 `resource_type`、`resource_id`、`response_json`；业务失败时更新为 `failed`。

唯一约束冲突后的分流规则：

- 相同 key、相同 hash、状态是 `succeeded`：直接返回 `response_json`，不重复执行业务。
- 相同 key、相同 hash、状态是 `processing`：说明另一个相同请求还在处理中，通常返回 `409` 或 `202`。
- 相同 key、不同 hash：说明同一个 key 被复用于不同请求体，直接返回 `409`。

这个流程解决的是协议级 API 幂等，而不是单纯依靠数据库唯一约束挡重复写入。

## 7.6 当前已经具备的防重复能力

虽然正式 `Idempotency-Key` 流程还没全面接上，但系统里已经有一些业务层去重和防重复保护。它们不等于标准幂等，不过能挡住一部分重复操作。

### 7.6.1 数据库唯一约束

很多核心表都有唯一约束和状态约束，可以防止明显重复或非法状态写入。`idempotency_records` 自身也已经通过 `uq_idempotency_records_scope_key` 约束同一作用域下的重复 key。

### 7.6.2 状态机约束

考试、题目、文档、后台任务等流程都依赖状态边界保护，例如状态不对就不允许重复执行、已提交就不能再次提交、已分配就不能随意修改、已归档或已删除就不再进入正常业务流。

这类能力更接近业务幂等或状态幂等，不等同于协议级 API 幂等。

### 7.6.3 后台任务状态保护

异步任务已经有 `pending`、`running`、`succeeded`、`failed` 等状态。它解决的是重操作不要堵住请求链路，以及任务执行过程可追踪。

但异步任务不等于幂等。两者经常一起出现，不过标准幂等还需要 `Idempotency-Key -> request_hash -> 查表/写表 -> 响应重放` 这条完整链路。

## 7.7 最适合优先补齐幂等的接口

下一步生产化时，优先接这些写接口：

- 考试提交：网络重试常见，副作用重，最值得先做幂等。
- 知识文档导入：尤其网页导入和手工导入，容易重复点击或重复提交。
- 题目预览生成 / 后台任务创建：同一次请求重复触发很常见。
- 试卷创建 / 分配：典型 create 型 POST，也需要避免重复副作用。

## 7.8 面试表述

如果被问到“TalentHub 幂等怎么实现的”，最诚实、也最有边界感的回答是：

> TalentHub 目前把幂等的数据库基础设施和设计约束先搭好了，核心是 `idempotency_records` 表。它按 `scope + idempotency_key` 建唯一约束，同时记录 `request_hash`、处理状态、关联资源和可重放响应。这样同一个 key 的重试请求可以做到三种分流：相同请求直接重放结果、处理中请求返回冲突、同 key 不同请求体直接拒绝。
>
> 不过当前仓库里，这套机制还主要停留在 schema 和设计层，尚未全面接到具体 API 上。现在系统更多依赖业务状态机、唯一约束和后台任务状态来避免重复副作用。下一步如果生产化，会把 `Idempotency-Key` 正式接到创建类和提交类接口上。

一句话总结：TalentHub 现在的幂等是“表和设计已经有了，但接口级正式幂等还没全面落地”。

## 8. 一致性边界

### 8.1 强一致

以下操作必须在单事务内完成：

- 创建题目及题目版本
- 创建试卷及试卷题目快照
- 提交考试并锁定作答
- 保存评分结果和逐题反馈

### 8.2 最终一致

以下行为可以通过事件异步完成：

- 指标聚合
- 报表快照
- 学习路径推荐
- 通知类任务

## 9. 一致性优先级

优先保障顺序如下：

1. 核心业务正确性
2. 状态可追踪
3. 历史快照稳定
4. 指标最终同步

如果两者冲突，绝不为了“看起来更智能”而牺牲业务正确性。
