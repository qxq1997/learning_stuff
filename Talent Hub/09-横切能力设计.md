# 横切能力设计

本文档定义 TalentHub 的横切能力，包括可观测性、配置管理、缓存策略和权限模型。

## 1. 可观测性

## 1.1 日志

要求：

- 统一输出到标准输出
- 结构化日志优先
- 关键字段包括：时间、级别、模块、请求 ID、操作者、资源 ID、错误码

当前已落地：

- FastAPI 统一 JSON 日志输出
- 请求级日志中间件
- `X-Request-ID` 响应头
- 关键业务事件日志
- 外部依赖日志
- 错误日志与 API 错误码对齐

当前统一日志字段：

- `timestamp`
- `level`
- `logger`
- `event`
- `request_id`
- `actor_email`
- `method`
- `path`
- `status_code`
- `latency_ms`
- `error_code`

业务日志会按场景补充：

- `resource_id`
- `question_id`
- `exam_paper_id`
- `assignment_id`
- `attempt_id`
- `provider`
- `model`
- `tokens_in`
- `tokens_out`
- `source_kind`
- `source_url`

关键日志场景：

- 文档导入
- 题目生成
- 题目保存与更新
- 试卷创建与更新
- 考试提交
- AI 判卷
- 权限拒绝
- 外部依赖失败

当前请求级日志事件：

当前外部依赖事件示例：

## 1.2 指标

第一阶段建议关注：

- 文档导入次数
- 检索命中次数
- 题目生成成功率
- 考试提交次数
- AI 判卷耗时
- 模型 token 消耗
- 平台活跃用户数

## 1.3 审计

以下操作必须可审计：

- 文档导入与删除
- 题目发布与归档
- 试卷发布
- 考试分配
- 权限变更

## 2. 配置管理

原则：

- 所有关键配置必须显式声明
- 缺失关键配置时服务启动失败
- 配置不允许隐式回退到未知默认值

- `DATABASE_URL`
- `REDIS_URL`
- `LOG_LEVEL`
- `LLM_BASE_URL`
- `LLM_MODEL_NAME`
- `EMBEDDING_MODEL_NAME`
- `VECTOR_DIMENSION`
- `QUESTION_BANK_TAG_CACHE_TTL_SECONDS`
- `WEB_PAGE_FETCH_CACHE_TTL_SECONDS`

## 3. 缓存策略

当前缓存统一使用 Redis，但边界非常明确：

- 只缓存“可重算”的读结果
- 不把缓存当事实来源
- 写入成功后再做缓存失效
- 缓存异常不允许破坏主业务写入

当前已经落地的缓存点：

- 题库标签列表缓存
- 网页抓取结果缓存

当前明确不缓存：

- 考试提交与判卷写入
- 权限校验
- 用户学习进度

更完整说明见：

- [缓存策略与 Redis 设计](./25-缓存策略与Redis设计.md)

## 4. 权限模型

第一阶段采用 RBAC。

推荐角色：

- `employee`
- `question_admin`
- `training_admin`
- `manager`
- `system_admin`

### 4.1 授权原则

- 默认拒绝
- 显式授权
- 权限检查在应用层完成
- 重要读接口也必须做范围过滤

### 4.2 数据范围示例

- 员工只能查看自己的考试记录和学习路径
- 管理者只能看自己负责组织范围内的数据
- 题库管理员可以维护题目和试卷，但不能修改组织权限

## 5. 外部模型调用治理

要求：

- 所有模型调用都要记录到 `llm_jobs`
- 明确记录模型名、耗时、token 消耗、输入引用
- 模型返回结构不合法时直接失败

## 6. 文件与对象存储

第一阶段优先本地文件系统，必要时切换 MinIO。

规则：

- 存储路径必须显式记录
- 文档处理基于版本，不直接覆盖原文件
- 文件找不到时直接失败，不猜测替代路径
