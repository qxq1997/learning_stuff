# 知识库首条联调链路

## 目标

这一条链路的目标是让 TalentHub 从“只有基础设施和数据模型”进入“前后端真正联动”的阶段。

第一批落地范围最初只做只读查询：

- 后端提供知识库文档列表接口
- 前端首页展示真实知识库文档列表
- 接口与页面都直接暴露异常，不做静默 fallback

## 为什么先做文档列表

在当前阶段，知识库模块是后续 RAG、题目生成和题目来源追踪的入口。

比起先做一个没有后端支撑的上传按钮，先把文档列表查询做通更合理，因为它可以立刻验证三件事：

- 模块分层是否真的可用
- 前端与后端的联调链路是否畅通
- 数据库持久化结构能否被稳定读取

## 后端分层

本次实现按 DDD 的最小闭环拆成 4 层：

- `domain`
  - `KnowledgeDocumentSummary`
  - `KnowledgeDocumentSummaryPage`
  - `KnowledgeDocumentQueryRepository`
- `application`
  - `ListKnowledgeDocumentsQuery`
  - `ListKnowledgeDocumentsUseCase`
- `infrastructure`
  - `SqlAlchemyKnowledgeDocumentQueryRepository`
- `interfaces`
  - `GET /api/v1/knowledge/documents`

依赖方向保持单向：

- `interfaces -> application -> domain`
- `infrastructure -> domain`

## 查询返回内容

当前文档列表接口返回这些字段：

- 文档 ID
- 标题
- 来源类型
- 当前状态
- 版本数量
- 最新版本号
- 最新文件名
- 最新版本时间
- 文档创建时间

接口同时支持：

- `limit`
- `offset`

前端工作台已经按这两个参数做分页查询，不是一次性把所有文档拉回本地再做假分页。

这里故意没有塞入“过多推断字段”，例如解析进度百分比、切片命中率、向量状态摘要，因为这些能力目前还没有稳定业务语义。项目遵循 `fail fast`，所以先只返回确定无歧义的事实数据。

## 前端呈现方式

前端首页新增“知识库文档工作台”区域：

- 页面启动时直接查询后端接口
- 如果接口失败，页面直接显示错误
- 如果接口为空，页面显示明确空状态
- 如果有数据，则展示真实表格

这意味着首页已经不再是纯静态展示页，而是第一个真实的业务控制台入口。

## 后续自然扩展位

在这条链路基础上，下一批开发可以继续往下接：

1. 文档上传接口
2. 文档版本创建
3. 文档解析任务
4. chunk 切分与向量化
5. 文档检索预览
6. 题目生成时的来源追踪

## 当前约束

- 暂不实现前端上传组件
- 暂不实现后台任务编排
- 暂不实现对象存储写入
- 暂不实现文档详情页

这些都不是被忽略，而是有意按顺序推进，避免在系统骨架尚未稳定前引入太多半成品逻辑。
