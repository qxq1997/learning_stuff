# 知识库手动导入、网页导入与开发种子

## 目标

在文档列表查询已经打通之后，这一步继续补齐知识库的第一条写链路：

- 前端可以提交文本文档
- 前端也可以提交网页 URL
- 后端可以把原始内容落到本地存储目录
- 数据库可以写入 `knowledge_documents` 和 `knowledge_document_versions`
- 列表页可以立即看到新文档

## 为什么先做手动导入

当前项目仍处于从 0 到 1 的阶段。

如果现在直接上完整文件上传、对象存储、PDF/Word 解析和后台任务，复杂度会上升得太快，很难保持清晰边界。

因此第一阶段先支持：

- 文档标题
- `.txt` 文件名
- 文本正文
- 单页网页 URL 导入

然后把这一条最小闭环先做稳。

本次新增的核心文件包括：

- [application/commands.py](/Users/xinqi/WebstormProjects/TalentHub/backend/app/modules/knowledge/application/commands.py)
- [application/ports.py](/Users/xinqi/WebstormProjects/TalentHub/backend/app/modules/knowledge/application/ports.py)

- [interfaces/router.py](/Users/xinqi/WebstormProjects/TalentHub/backend/app/modules/knowledge/interfaces/router.py)

新增接口：

- `POST /api/v1/knowledge/documents/manual-imports`
- `POST /api/v1/knowledge/documents/url-imports`

当前规则非常明确：

- `title` 不能为空
- `file_name` 不能为空
- `file_name` 不能包含目录层级
- 当前只接受 `.txt`
- `content` 不能为空
- `url-imports` 只接受 `http/https`
- `url-imports` 只抓当前页，不自动扩展到站内子页
- 如果网页正文抽取失败或太短，接口直接失败

不满足就直接失败，不做自动纠错，不偷偷替你猜一个文件名或来源。

## 本地存储策略

当前原始文本文件会写到：

- `storage_data/knowledge/documents/{document_id}/v1/{file_name}`

这个目录通过 Docker Compose 挂载到 backend 容器里，因此容器重建后数据仍然保留在宿主机。

## 为什么需要开发种子

数据库里 `knowledge_documents.uploaded_by` 是必填外键。

而当前项目还没有正式登录系统，所以这一步需要一个显式的本地开发管理员作为操作人。

因此新增了：

- [backend/sql/002_seed_dev_identity.sql](/Users/xinqi/WebstormProjects/TalentHub/backend/sql/002_seed_dev_identity.sql)
- `make db-seed-dev`

这个种子会创建：

- 开发部门 `TalentHub Dev`
- 角色 `system_admin`
- 开发管理员 `admin@talenthub.local`

当前手动导入接口会固定使用这个开发管理员邮箱。如果数据库里没有这个用户，接口会直接失败，并提示先执行开发种子。

## 前端实现

前端知识库页现在提供两个入口：

- 手动文本导入
- 网页 URL 导入
- 成功或失败提示
- 导入完成后自动刷新文档列表，并展示 `source_url`

这使得知识库已经具备两种长期材料沉淀方式，而不是只能手工粘贴文本。
