# 题目预览确认入库

## 目标

这一阶段把“AI 题目生成预览”继续推进成“人工确认后入题库”的可执行链路。

说明：

- 文件名和部分历史接口里还保留了 `draft` 命名
- 但当前实际业务语义已经改成“保存后直接进入题库并立即生效”

- 先生成结构化题目预览
- 由管理员确认
- 再正式写入 `questions` 与 `question_versions`

这样做的目的很明确：

- 不让模型直接静默改库
- 保留人工确认节点
- 让题库沉淀从第一天开始就是版本化的

## 当前实现

本次新增的核心接口是：

- `POST /api/v1/question-bank/drafts/from-previews`

输入直接复用当前预览结果结构：

- `provider`
- `model`
- `source_document_id`
- `items`

每个题目项包含：

- `type`
- `stem`
- `options`
- `standard_answer`
- `explanation`
- `difficulty`
- `source_excerpt`

后端不会因为这是“来自前端的预览结果”就默认相信它，而是会再次做明确校验。

## fail fast 校验规则

保存题目时，后端会直接失败于以下情况：

- `provider` 或 `model` 为空
- `items` 为空
- 来源知识文档不存在
- 来源知识文档状态不是 `parsed` 或 `indexed`
- 开发管理员不存在
- 题型不在支持范围内
- 难度不在 `1..5`
- `stem` / `explanation` / `source_excerpt` 为空
- `options` 或 `standard_answer` 里有空字符串
- `options` 或 `standard_answer` 存在重复项
- 同一次提交里有重复题干
- 单选题没有且仅有一个标准答案
- 多选题答案不在选项里
- 判断题选项不是严格的 `["正确", "错误"]`
- 简答/论述/案例题仍然携带了选项

也就是说，保存逻辑不会偷偷替你修数据。

## 数据库存储方式

### 1. 题目主体

保存时会创建：

- 一条 `questions`
- 一条 `question_versions`

当前约定：

- `questions.status = published`
- `questions.source_type = ai_generated`
- `question_versions.version_no = 1`

### 2. 生成元数据

为了不把“来源摘录”“模型信息”硬塞到解析文本或说明字段里，这次新增了：

- `question_versions.generation_metadata_json`

它当前承载这些信息：

- `provider`
- `model`
- `source_kind`
- `source_document_id`
- `source_document_title`
- `source_document_version_no`
- `source_file_name`
- `source_url`
- `source_excerpt`

这比把这些信息塞进 `explanation` 或 `rubric_json` 更清晰，也为后面接入更多 Provider 留出了空间。

## 前端交互

前端首页当前已经支持：

1. 选择知识文档并生成题目预览
2. 点击“保存到题库”
3. 查看返回的 `question_id` 与 `question_version_id`

这意味着首页现在已经不只是“看预览”，而是能真正把题目送进题库。

## 当前边界

这一版故意还没有做这些事情：

- 预览会话持久化
- 题库列表页
- 题目编辑页

原因不是这些不重要，而是先把“可生成、可确认、可入库”这条最短业务链路做稳。
