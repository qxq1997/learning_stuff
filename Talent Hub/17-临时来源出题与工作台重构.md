# 临时来源出题与工作台重构

## 这次为什么要重构

之前前端虽然已经能联调知识库、生成题目、保存题目，但存在两个明显问题：

- 所有能力都堆在首页，操作路径不清晰
- 题目生成过度依赖“先入知识库”，不适合临时网页和一次性文本

这次调整的目标就是：

- 把前端改成真正的多页面工作台
- 让“出题”从知识库流程里解耦出来
- 默认支持临时来源，不强制把全文持久化到本地

## 新的前端结构

当前前端已经拆成三个主页面：

- `总览`
- `出题工作台`
- `知识库`

其中：

- `总览`
  - 只展示系统状态和入口导航
- `出题工作台`
  - 变成主入口
  - 支持三种来源：
    - 网页 URL
    - 粘贴文本
    - 知识库文档
- `知识库`
  - 只承载明确要长期复用的内容

这意味着首页不再承担所有职责，页面边界更清晰了。

另外，题目数量现在不再设置固定硬上限。

当前原则是：

- 只校验数量必须大于等于 `1`
- 不再在前后端写死 `5` 或 `20` 这样的业务上限
- 真正的边界由模型耗时、返回体积和成本决定

## 新的出题来源模型

当前系统把题目生成来源分成三类：

### 1. `knowledge_document`

适合：

- 公司制度
- 培训手册
- 需要反复复用的内部资料

这类内容会进入知识库并持久化。

### 2. `inline_text`

适合：

- 临时文章摘要
- 复制来的培训内容
- 一次性整理的段落

这类内容只用于本次出题，不会把全文落到数据库或本地文档库。

### 3. `web_page`

适合：

- 某篇网页文章
- 某个公开资料页面
- 临时需要抓取正文的内容

系统会抓取网页正文并在内存中用于出题，不默认持久化全文。

如果网页中检测到了真正的同域子页链接，系统不会自动递归抓取，而是会先要求用户确认范围。

## 后端新增能力

### 1. 新增临时来源预览接口

新增了两个题目预览接口：

- `POST /api/v1/question-bank/generation-previews/from-inline-text`
- `POST /api/v1/question-bank/generation-previews/from-web-page`

原有接口仍保留：

- `POST /api/v1/question-bank/generation-previews`

它继续用于知识库文档出题。

### 2. 网页抓取策略

网页抓取当前使用：

- `Playwright`
- `BeautifulSoup`

处理原则：

- 先用无头 Chromium 渲染页面
- 等待 `article` / `main` / `body` 中出现足够长度的正文
- 再提取渲染后的 HTML
- 最后去掉 `script/style/noscript/nav/footer/form/svg`
- 优先抽取 `article`
- 其次 `main`
- 再其次 `body`

如果页面没有标题、渲染超时，或渲染后的正文太短，会直接失败，而不是猜测性兜底。

## 为什么默认不持久化全文

这是这次重构里非常重要的一条设计判断。

如果把所有网页和临时文本都落库，会马上出现这些问题：

- 本地磁盘快速膨胀
- 知识库变成杂项缓存
- 后续管理和检索边界变模糊

所以现在的约定是：

- 临时来源用于本次题目生成
- 真正需要复用时，再由你主动把内容导入知识库

## 题目保存时保留什么

虽然临时来源不保存全文，但保存到题库时仍然会记录足够的来源元信息：

- `source_kind`
- `source_title`
- `source_url`
- `source_excerpt`
- `provider`
- `model`

这些信息保存在：

- `question_versions.generation_metadata_json`

因此题目不会变成“无来源的孤儿数据”。

对于超过单段上限的长网页或长文本，当前还会额外记录：

- `source_segment_index`
- `source_segment_count`

这样你后面可以明确知道一题来自长文的哪一段。

## 超长内容的确认规则

这次还补了一条明确的交互约束：

- 如果来源内容没有超过单次上限，系统直接生成题目预览
- 如果来源内容超过单次上限，后端不会偷偷截断，也不会自动开始分页生成
- 前端会先弹窗告诉用户：
  - 当前总字数
  - 单页上限
  - 需要分成几页
  - 这次按当前题目数量会实际使用前几页
- 只有用户点击确认，系统才会继续分段生成

这条规则同时适用于：

- `web_page`
- `inline_text`
- `knowledge_document`

## 当前已经验证的结果

这次重构后，已经实际验证通过：

- 粘贴文本生成题目预览
- 网页 URL 抓取正文并生成题目预览
- 网页 URL 检测同域子页并弹出范围确认
- 临时文本生成结果直接保存到题库

也就是说，系统现在已经具备一条新的主业务链路：

网页 URL / 粘贴文本 / 知识库文档  
-> AI 题目预览  
-> 人工确认  
-> 题库存储并立即可用

## 下一步

这条链路稳定后，最自然的下一步是：

1. 做题库列表页
2. 支持编辑已保存的题目
3. 把网页正文抽取做得更稳
4. 再接 RAG 检索式出题
