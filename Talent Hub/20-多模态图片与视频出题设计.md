# 多模态图片与视频出题设计

## 目标

在现有来源基础上，继续扩展两类新来源：

- 图片
- 视频

这两类能力都应该接入同一条“出题工作台 -> 预览 -> 确认 -> 入库”主链路，而不是单独长出一套旁路系统。

## 规划中的新来源类型

- `image_asset`
- `video_transcript`

含义：

- `image_asset`
  - 基于图片内容出题
- `video_transcript`
  - 先把视频转成文字，再基于转写文本出题

## 图片出题

### 第一版目标

先支持：

1. 单张图片上传
2. 网页中关键图片的识别

### 推荐处理链路

1. 获取图片
2. 产出 OCR 文本或图片描述
3. 把识别结果当成题目生成来源
4. 走现有题目生成链路

### 为什么先这样做

因为这可以复用现有文本出题链路，不需要一开始就把 provider 改造成非常复杂的原生多模态消息协议。

## 视频出题

### 第一版目标

先支持：

1. 视频文件输入
2. 音轨抽取
3. 语音转写
4. 基于转写文本出题

### 推荐处理链路

1. 上传视频或提供本地视频路径
2. 使用 `ffmpeg` 抽取音频
3. 使用语音识别模型转写
4. 把转写文本作为 `video_transcript` 来源
5. 进入现有文本题目生成链路

## 本地优先技术建议

在你这个项目里，更适合的本地优先路线是：

- 图片：`OCR/图片描述 -> 文本出题`
- 视频：`ffmpeg + whisper/faster-whisper -> 文本出题`

这样更符合：

- 本地优先
- 开源优先
- 结构清晰
- fail-fast

## 与当前架构的关系

图片和视频都应该只是新的来源适配器，而不是重新发明一套题目系统。

统一入口仍然应该是：

1. 来源获取
2. 来源规范化
3. 长内容分段
4. 题目生成预览
5. 人工确认
6. 草稿题库存储

## fail-fast 约束

图片和视频接入后仍然遵守当前项目基线：

- 图片识别失败直接报错
- 视频转写失败直接报错
- 转写为空直接报错
- 内容过短直接报错
- 内容过长则走明确分段逻辑

不做自动猜测、自动补文、自动忽略失败片段。

## 推荐开发顺序

建议顺序：

1. 先把长文分段链路做稳
2. 再做图片上传 + OCR/描述
3. 再做视频转写为文本
4. 最后再考虑真正的原生多模态直接出题
