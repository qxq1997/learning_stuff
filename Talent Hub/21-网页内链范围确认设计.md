# 网页内链范围确认设计

## 背景

网页 URL 出题在很多场景下不会只面对一个孤立页面。

常见情况包括：

- 文档首页包含多个章节链接
- 文章页包含“上一章 / 下一章”
- 目录页包含多个同域子页面

如果系统自动递归抓取所有链接，会马上出现这些问题：

- 抓取范围失控
- token 成本失控
- 题目来源边界不清
- 很容易把导航页、重复页、无关页一起抓进来

因此 TalentHub 对网页内链的策略不是“自动爬整站”，而是“先检测，再让用户确认范围”。

## 当前规则

### 1. 默认只抓当前页

用户第一次输入网页 URL 时，系统只保证抓当前页正文。

### 2. 只有检测到真正的同域子页时，才弹范围确认

系统会检测正文区域中的链接，并做这些过滤：

- 只保留同域链接
- 去掉 `mailto:`、`javascript:`、`tel:` 等非网页链接
- 去掉当前页内部锚点
- 去掉同一页面身份下的重复链接

这里特别强调：

- 像目录锚点、章节内跳转这类“同页定位”，不会再被误判成子页

### 3. 范围选择必须显式确认

当前前端提供三种范围：

- `仅当前页`
- `当前页 + 选中的子页`
- `当前页 + 全部检测到的子页`

只有用户点击确认后，后端才会继续抓取并生成题目。

### 4. 子页范围有硬上限

为了避免目录页一次抓太多内容，当前只展示前 `10` 个可选同域子页。

如果检测到更多链接：

- 前端会明确提示“只展示前 10 个”
- 多出来的链接不会偷偷纳入本次出题范围

这个限制由配置项控制：

- `WEB_PAGE_MAX_NESTED_LINK_CANDIDATES`

## 两层确认的顺序

网页 URL 出题现在可能会经历两层确认：

1. 网页范围确认
2. 长内容分页确认

顺序是固定的：

1. 先确认是否只用当前页，还是纳入子页
2. 确认后的正文如果仍然超过单次处理上限
3. 再弹“需要分几页处理”的确认框

这样用户能明确知道：

- 先决定“抓哪些页”
- 再决定“这些页怎么分页生成”

## 后端协议

当检测到可选子页而用户尚未确认范围时，后端返回：

- HTTP `409`
- `error.code = confirmation_required`
- `error.details.reason = web_page_scope_required`

并附带这些信息：

- `current_title`
- `current_url`
- `nested_link_candidates`
- `truncated_candidate_count`

用户确认范围后，请求会附带：

- `web_page_scope`
- `selected_child_urls`
- `confirm_web_page_scope = true`

## 当前实现边界

当前版本只支持：

- 同域子页选择
- 单层范围扩展

当前不支持：

- 自动递归到孙页面
- 跨域扩展
- 自动猜测哪个链接才是“最佳章节范围”

这符合项目的 `fail fast` 原则：

- 能明确确认的就要求确认
- 不能明确判断的就不猜
