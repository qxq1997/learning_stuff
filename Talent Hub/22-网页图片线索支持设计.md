# 网页图片线索支持设计

## 背景

之前网页 URL 出题虽然已经能抓正文，但网页里的图片信息会被完全忽略。

这会带来两个明显问题：

- 图片中的关键说明、图注、流程图标题无法参与出题
- 某些文档页的文字较少，真正重要的信息反而写在配图说明里

因此这次先补第一阶段能力：

- 不直接做复杂的 OCR 或完整视觉推理
- 先把网页中可解释的图片线索提取出来，并带进题目生成

## 当前实现范围

当前只对网页来源生效：

- `web_page`

当前会提取这些图片线索：

- 图片 URL
- `alt`
- `title`
- `figcaption`
- 如果都没有，再尝试用图片文件名做一个保底标签

这些内容会被整理成一段结构化文本，追加到网页正文后面：

- `[网页图片线索]`

然后和正文一起进入题目生成流程。

## 为什么先不用“直接看图”

这是一个刻意的阶段性选择。

如果一开始就让模型直接读取所有网页图片，会立刻引入这些复杂度：

- 图片 URL 是否对外可访问
- 图片数量过多导致请求成本暴涨
- 视觉输入和文本分页如何对齐
- OCR、截图、图表理解是否要单独建链路

所以当前先做更清晰的一步：

- 先消费网页中已经存在的图片语义线索
- 不对图片内容做猜测性识别

这符合项目当前的 `fail fast` 原则。

主要行为：

1. 渲染网页并抽出正文
2. 在正文区域里查找 `img`
3. 提取图片线索
4. 最多保留固定数量的图片
5. 把图片线索结构化追加到正文
6. 题目生成 prompt 明确告知模型：
   - 可以使用这些图片线索
   - 不能脑补图片里未提供的细节

## 配置项

新增配置：

- `WEB_PAGE_MAX_IMAGE_CANDIDATES`

含义：

- 单次网页抓取最多提取多少张图片的线索

## 当前边界

当前支持的是：

- 配图语义线索参与出题

当前还不支持的是：

- 直接把网页图片作为视觉输入发给模型
- OCR 识别图片中的文字
- 图表、流程图、截图的真正视觉理解
- 图片分段与正文分段的精细对齐

## 下一阶段

如果后面继续往多模态走，下一步更自然的是：

1. 对网页图片建立独立的 `image_asset` 来源模型
2. 支持把公开图片 URL 直接作为视觉输入传给模型
3. 对本地图片或复杂图表增加 OCR / 图像理解链路
