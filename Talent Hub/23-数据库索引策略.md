# 数据库索引策略

本文档说明 TalentHub 在 PostgreSQL 中应该如何设计和演进索引。

目标不是“索引越多越好”，而是让索引严格服务于真实查询路径，并保持：

- 查询快
- 写入成本可控
- 结构清晰
- 可通过 Alembic 显式演进

## 1. 设计原则

TalentHub 的索引设计遵守这几条规则：

1. 先看查询路径，再决定是否建索引。
2. 默认优先使用 `btree`，只有明确需要时才上 `hnsw`、`gin`、表达式索引。
3. 复合索引的列顺序按“等值过滤 -> 范围过滤 -> 排序”来排。
4. 唯一约束和主键不等于“所有查询都被优化”，热点查询仍然可能需要额外索引。
5. 索引变更必须同时落到 ORM、空库 SQL 和 Alembic migration。

## 2. 当前真实查询路径

### 2.1 Knowledge

- 按文档查最新版本
- 按版本查 chunk
- 按 embedding 做向量召回
- 按 checksum 排重

### 2.2 Question Bank

- 按状态、题型、来源分页列题
- 按标签筛题
- 按关键字搜题
- 按 `question_id` 取最新版本
- 编辑题目时按题目读取其全部标签

### 2.3 Exam

- 按标签挑题生成试卷
- 按试卷查题目
- 按用户查分配和考试尝试
- 按 attempt 查答案

### 2.4 Analytics

- 按用户和时间查行为事件
- 按事件类型聚合
- 按 scope 查指标快照
- 按状态查 LLM 任务

## 4. 为什么题库这几类索引已经提前落地

当前 Question Bank 的热点查询已经比较明确，所以这几类索引不再停留在“以后再说”，而是已经正式落地：

- 标签筛选和标签聚合
- 题目关键字搜索
- 按题目读取最新版本
- 按来源类型筛题

### 4.1 标签索引

`question_tags` 现在承担三类查询：

- 按标签筛题
- 标签聚合统计
- 未来按标签组卷

因此当前优先落的是：

```sql
CREATE INDEX ix_question_tags_tag_name_question_id
ON question_tags (tag_name, question_id);
```

这个索引能同时覆盖：

- `WHERE tag_name = ?`
- 先按标签找到题，再继续 join `questions`

而 `PRIMARY KEY (question_id, tag_name)` 则负责：

- 同题下标签去重
- 按题目读取其全部标签

如果用户输入的是模糊标签关键字，例如“入职”“培训”“缓存”，还会走：

```sql
CREATE INDEX ix_question_tags_tag_name_trgm
ON question_tags
USING gin (tag_name gin_trgm_ops);
```

这个 trigram 索引服务于：

- 标签搜索框
- 标签聚合接口的关键字过滤

### 4.2 题干全文近似搜索索引

题库页现在已经支持关键字搜索，所以 `question_versions.stem` 已经接入：

```sql
CREATE INDEX ix_question_versions_stem_trgm
ON question_versions
USING gin (stem gin_trgm_ops);
```

这个索引的前提是：

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

它主要服务于：

- 题库页按题干关键字搜索
- 后续题目去重和相似题排查

### 4.3 最新版本读取索引

按题目读取最新版本已经是当前题库列表和详情的真实路径，因此现在正式使用：

```sql
CREATE INDEX ix_question_versions_question_id_version_created_desc
ON question_versions (question_id, version_no DESC, created_at DESC);
```

这个索引的目标是：

- 先按 `question_id` 定位
- 再快速拿到最新版本
- 避免每次都在该题的全部版本上做额外排序

### 4.4 来源类型表达式索引

当前题目生成元数据已经写进 `generation_metadata_json`，并且题库页已经支持按来源类型筛选，所以现在也正式落了表达式索引：

```sql
CREATE INDEX ix_question_versions_source_kind
ON question_versions ((generation_metadata_json->>'source_kind'));
```

这个索引服务于：

- `knowledge_document`
- `inline_text`
- `web_page`
- 后续可能新增的 `image_asset`、`video_transcript`

## 5. 什么时候再加新索引

### 5.1 Exam 题库候选筛选继续变热时

考试模块现在已经正式支持“按标签筛题 -> 显式勾选 -> 创建试卷”，因此和试卷自身相关的两个索引已经落地：

- `ix_exam_papers_status_created_at`
- `ix_exam_paper_tags_tag_name_exam_paper_id`

如果后面“候选题筛选”继续变热，再考虑在 `question_tags`、`questions` 和题目状态上补更贴近组卷查询的复合索引。例如：

```sql
CREATE INDEX ix_questions_status_created_at
ON questions (status, created_at DESC);
```

### 5.2 Analytics 聚合变热时

如果后面管理端要频繁看题库趋势、标签命中率、来源占比，再考虑围绕 `activity_events`、`llm_jobs` 做更细的时间窗口索引，例如：

```sql
CREATE INDEX ix_activity_events_event_type_occurred_at
ON activity_events (event_type, occurred_at DESC);
```

## 6. 明确不建议现在就做的事

当前阶段不建议：

- 给所有 `jsonb` 列一股脑上 `GIN`
- 给低基数字段堆很多单列索引
- 在没有稳定查询模式前，提前做复杂全文检索体系
- 为了“保险”给每个外键都重复建多个相似复合索引

TalentHub 的索引设计必须跟着真实功能走，而不是跟着焦虑走。
