# 题库标签、搜索、编辑与分页

本文档说明 TalentHub 题库模块第二阶段的实现目标和当前做法。

这一阶段解决的问题是：

- 每个题目可以绑定标签
- 题库列表支持关键字搜索
- 题库列表支持服务端分页
- 已入库题目可以继续编辑
- 题目生成预览支持分页查看
- 题型收敛为：单选题、多选题、代码题
- 代码题编辑先使用 CodeMirror

## 1. 为什么要有题目标签

标签不是“装饰信息”，而是后续组卷和筛题的核心索引键。

在 TalentHub 里，标签至少服务于这些场景：

- 题库搜索
- 题库筛选
- 未来按标签组卷
- 未来按标签分析知识覆盖情况

所以标签不是临时前端字段，而是正式落库到：

- `tags`
- `question_tags`

其中：

- `tags` 是标签主表
- `question_tags` 是题目与标签的绑定表

## 2. 当前后端接口

### 2.1 列表查询

`GET /api/v1/question-bank/questions`

支持参数：

- `limit`
- `offset`
- `status`
- `question_type`
- `source_kind`
- `keyword`
- `tag_names`

返回的是“每道题当前最新版本”的摘要，不会把全部历史版本直接摊开。

补充约束：

- 这些接口当前只对 `department_admin` 和 `system_admin` 开放
- 普通成员调用时会直接失败
- 后端通过 Bearer token 识别当前账号

### 2.2 标签聚合

`GET /api/v1/question-bank/tags`

当前用于：

- 给前端筛选器提供标签选项
- 给出每个标签当前关联的题目数量

### 2.3 题目详情

`GET /api/v1/question-bank/questions/{question_id}`

返回：

- 题干
- 选项
- 标准答案
- 解析
- 难度
- 标签
- 当前状态
- 来源元信息

### 2.4 题目编辑

`PATCH /api/v1/question-bank/questions/{question_id}`

编辑逻辑遵守：

- 不覆盖旧版本
- 每次编辑都新建 `question_versions`
- 编辑后题目直接保持 `published`
- 标签集合会被明确替换

### 2.5 直接生效规则

当前规则已经改成：

- 有权限的管理员保存题目后直接生效
- 不再要求先送审再发布
- 题目编辑仍然保留版本化，不会覆盖旧版本

这保证了题库编辑仍然符合“版本化”设计，而不是直接改旧数据。

## 3. 当前前端页面行为

### 3.1 出题工作台

出题工作台现在支持在生成题目前直接输入标签：

- 网页 URL 出题
- 粘贴文本出题
- 知识库文档出题

生成成功后，这批题目的标签会随着“保存到题库”一起写入数据库。

如果这些标签之前不存在，系统会先把它们登记到 `tags` 主表，再建立 `question_tags` 绑定关系。

### 3.2 题目预览分页

预览结果不再一次性把全部题目堆在一个长页面里。

当前做法：

- 预览结果前端分页
- 可切换每页条数
- 显示当前区间和总题数

这对“大批量出题”场景更友好。

### 3.3 题库分页

题库列表改成服务端分页：

- 前端只拿当前页
- 后端返回 `total / limit / offset`
- 前端分页器驱动后续查询

这样不会因为题目数量增长把整库一次性拉到浏览器。

### 3.4 题库搜索

当前支持：

- 关键字搜索
- 状态筛选
- 题型筛选
- 来源筛选
- 标签筛选

关键字搜索当前覆盖：

- 题干
- 来源标题
- Provider
- Model
- 标签名

标签查询来源现在是 `tags` 主表，所以即使某个标签当前还没有关联题目，也可以在下拉里看到它，只是 `question_count = 0`。

### 3.5 代码题编辑

当前官方题型已经收敛为：

- `single_choice`
- `multiple_choice`
- `coding`

其中代码题在题库编辑弹窗中使用 CodeMirror：

- 不再显示选项编辑区域
- 标准答案改为代码编辑器
- 当前先按“单语言、单参考解”方式建模

## 4. 编辑题目时的设计选择

### 4.1 为什么编辑时新建版本

因为题目是会被试卷引用的。

如果直接改老版本：

- 历史试卷会失真
- 历史考试答案可能和题目不再对应

所以当前规则很明确：

- 编辑题目 = 生成新版本
- 历史版本保留

### 4.2 为什么前端编辑仍然允许失败

题目类型不同，规则也不同：

- 单选题必须一个标准答案
- 多选题答案必须都在选项里
- 代码题不应该带选项
- 代码题必须至少有一个参考解

这些核心规则全部由后端兜底，并且：

- 非法输入直接失败
- 不做静默修正
- 不偷偷帮用户改题

这符合 TalentHub 的 `fail fast` 原则。

## 5. 与按标签组卷的关系

这一阶段还没有把“按标签直接生成考试”完整做完，但数据库和题库接口已经为它打好了基础：

- 题目已有标签集合
- 标签可聚合查询
- 题库可按标签过滤

因此下一步做按标签组卷时，可以直接复用这套模型，而不需要重做标签体系。

## 6. 当前边界

这一轮已经完成：

- 标签落库
- 标签聚合接口
- 列表搜索
- 服务端分页
- 题目详情
- 题目编辑
- 预览分页
- 代码题编辑器

这一轮还没有完成：

- 按标签直接生成试卷
- 多版本历史对比页
- 全文检索或向量化题库搜索
- 代码题运行沙箱和测试用例执行

这些会在下一阶段继续演进，而不是在这一轮里硬塞进去。
