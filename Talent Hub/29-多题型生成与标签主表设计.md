# 多题型生成与标签主表设计

本文档说明 TalentHub 新一轮关于“多题型同时生成”和“标签主表”的实现。

## 1. 为什么要支持多题型同时生成

之前出题工作台一次只能选一个题型，这会带来两个问题：

- 一次想生成“单选 + 多选 + 代码题”时，必须重复操作多次
- 同一批题目的标签、来源和难度配置需要反复填写

现在改成：

- 题型支持多选
- 总题数仍然只填一个
- 系统会按照勾选顺序尽量平均分配到各题型

分配规则是显式的，不做猜测：

- `总题数 / 题型数` 先取整
- 余数按勾选顺序分给前几个题型

例如：

- 勾选 `单选题 + 代码题`，总题数 `3`
  - 分配结果：`单选题 2`，`代码题 1`
- 勾选 `单选题 + 多选题 + 代码题`，总题数 `8`
  - 分配结果：`单选题 3`，`多选题 3`，`代码题 2`

前端会在提交前直接展示这条分配结果。

## 2. 为什么增加标签主表

之前系统只有：

- `question_tags`
- `exam_paper_tags`

这能表达“某道题/某套试卷绑定了什么标签”，但不能表达：

- 系统当前有哪些标签
- 新标签是否已经被正式登记
- 没有题目但已经被考试或工作台使用过的标签

所以现在增加正式主表：

- `tags`

表职责很单一：

- 存标签本身
- 存规范化后的唯一键
- 为标签搜索和下拉提供稳定来源

当前结构：

- `id`
- `name`
- `normalized_name`
- `created_at`
- `updated_at`

其中：

- `normalized_name` 唯一
- `name` 建了 trigram 索引，方便模糊搜索

## 3. 标签什么时候写入数据库

当前规则是：

- 在题目真正保存到题库时，标签写入 `tags`
- 在题目编辑并保存时，标签写入 `tags`
- 在考试创建时，`selection_tags` 也会写入 `tags`

也就是说：

- 只在“真实持久化”时写入
- 不会因为一次临时预览就产生脏标签

这符合 TalentHub 当前的 `fail fast` 和“不要猜测用户意图”的原则。

## 4. 旧数据怎么迁移

这一轮 Alembic 迁移会做三件事：

1. 新建 `tags` 主表
2. 从 `question_tags` 和 `exam_paper_tags` 回填历史标签
3. 把题目和试卷的默认状态统一到 `published`

其中回填规则是：

- 去空白
- 以规范化名称去重
- 已存在则跳过，不报错

## 5. 标签查询现在怎么工作

`GET /api/v1/question-bank/tags`

现在不再只盯着 `question_tags`，而是：

- 以 `tags` 主表为基准
- 左连接题目标签
- 返回每个标签当前关联的题目数量

这意味着：

- 没有题目但已经登记过的标签，也能出现在下拉里
- `question_count` 可能是 `0`

这正是我们需要的行为，因为考试标签和未来更多业务标签也会复用这套体系。

## 6. 当前边界

这一轮已经完成：

- 多题型同时生成
- 显式分配规则
- 标签主表
- 历史标签回填
- 新标签随题目/考试保存自动入库

这一轮还没有做：

- 单独的“标签管理后台”
- 标签别名/合并
- 标签层级
- 标签颜色、描述等展示元数据

这些能力以后再扩展，不在这一轮里提前做复杂化设计。
