# 知识库网页 URL 导入设计

本文档说明 TalentHub 为什么给知识库增加“网页 URL 导入”，以及当前实现边界。

## 1. 目标

知识库不是只接收手工粘贴文本。

很多长期材料本来就存在网页上，例如：

- 团队知识库页面
- 内部 Wiki
- 在线培训手册
- 公开课程文档

因此知识库现在新增：

- `POST /api/v1/knowledge/documents/url-imports`

它的目标很单一：

- 输入一个网页 URL
- 渲染网页并抽取正文
- 把正文作为长期知识文档写入知识库

## 2. 和“出题工作台网页 URL”有什么区别

这两条能力看起来都叫“网页 URL”，但定位不同：

- 出题工作台网页 URL
  - 默认是临时来源
  - 主要用于立即出题
  - 不要求先进入知识库

- 知识库网页 URL 导入
  - 是长期沉淀
  - 导入成功后会真正写数据库和本地存储
  - 以后可以继续被知识库、RAG、题目生成复用

所以它们不是同一个入口。

## 3. 当前行为

当前知识库 URL 导入规则非常明确：

- 只接受 `http` / `https`
- 只抓当前页
- 不自动追踪站内子页
- 使用浏览器渲染后再抽正文
- 如果网页内容太短、不是 HTML、渲染超时，直接失败

这里故意不做：

- 自动整站抓取
- 自动猜测章节范围
- 自动忽略正文异常

这符合 TalentHub 的 `fail fast` 原则。

## 4. 为什么复用浏览器渲染抓取

很多现代网页正文不是服务端直接返回的，而是前端 JS 渲染出来的。

如果只用普通 HTTP 抓取：

- 能拿到页面壳
- 拿不到真正正文

因此知识库 URL 导入复用了现有的浏览器渲染抓取能力：

- 先渲染网页
- 再抽取正文
- 再写入知识库

这样像 Docsify、React、Vue 文档站也能被导入。

## 5. 数据落库方式

本次新增了两个持久化信息：

- `knowledge_documents.source_type = web_import`
- `knowledge_documents.source_url`

正文仍然落在：

- `knowledge_document_versions.parsed_text`

同时会把原始文本快照写到本地：

- `storage_data/knowledge/documents/{document_id}/v1/{generated_file_name}.txt`

这样知识库里既能看出“这是网页导入的”，也能知道原始 URL。

## 6. 文件名策略

网页导入不是上传文件，所以没有天然文件名。

当前策略是根据：

- 域名
- 路径

生成一个稳定的 `.txt` 文件名，例如：

- `datawhalechina-github-io-hello-agents.txt`

这里不让用户手动填文件名，是为了避免多余输入，也避免知识库里出现和 URL 完全无关的随机文件名。

## 7. 当前边界

这一轮已经完成：

- 知识库页新增网页 URL 导入入口
- 后端新增 `url-imports` 接口
- 新增 `web_import` 来源类型
- 知识库文档列表展示 `source_url`
- 动态网页可被导入

这一轮还没有做：

- 一次导入多个 URL
- URL 导入时站内子页范围确认
- 导入后自动 chunk / embedding
- 网页正文清洗规则的可视化预览

这些能力后面可以继续增强，但这一轮先把“单页 URL 进入知识库”做稳。
