# 知识库预览、编辑与永久删除设计

本文档说明 TalentHub 当前知识库页面已经支持的管理动作：

- 新增
- 预览
- 编辑
- 删除

其中“删除”当前已经改成：

- 永久删除

不是归档。

## 1. 当前接口

知识库当前主要接口有：

- `GET /api/v1/knowledge/documents`
- `GET /api/v1/knowledge/documents/{document_id}`
- `POST /api/v1/knowledge/documents/manual-imports`
- `POST /api/v1/knowledge/documents/url-imports`
- `PATCH /api/v1/knowledge/documents/{document_id}`
- `DELETE /api/v1/knowledge/documents/{document_id}`

权限规则已经收口为：

- 知识库所有接口都要求登录
- `GET` 列表和详情允许已登录用户访问
- `POST / PATCH / DELETE` 只允许 `department_admin` 和 `system_admin`
- 文档写入时会记录真实操作者邮箱，不再使用开发期默认邮箱

## 2. 预览为什么单独做详情接口

知识库列表不应该把全文直接带回来。

否则会有两个问题：

- 列表体积会随着正文变大而迅速膨胀
- 前端分页也会变得没有意义

所以现在的做法是：

- 列表只返回摘要
- 预览时单独调用详情接口

详情接口会返回：

- 标题
- 来源类型
- 来源 URL
- 当前状态
- 最新版本号
- 最新文件名
- 最新正文内容

## 3. 编辑为什么要新建版本

知识库文档和题目一样，也有版本概念。

编辑规则不是：

- 直接覆盖旧正文

而是：

- 保留原文档 `document_id`
- 新增一条 `knowledge_document_versions`
- `version_no + 1`

这样做的原因是：

- 保留历史快照
- 避免后续 RAG、题目来源、审阅过程失真

当前编辑时允许修改：

- 标题
- 文档正文

当前不做：

- 手动改文件名
- 网页文档重新抓取原 URL

这些以后如果需要，可以再补“重新导入”能力。

## 4. 删除为什么改成永久删除

当前删除按钮的真实含义是：

- 直接删除 `knowledge_documents`
- 级联删除所有 `knowledge_document_versions`
- 级联删除所有 `knowledge_chunks`
- 同步删除 `Qdrant` 里的向量点
- 同步删除本地 `storage_data/knowledge/documents/{document_id}` 目录
- 删除这份文档关联的 `question_sources` 引用

这样做的原因是：

- 你已经明确要求知识库不要走归档语义
- 本地知识库存储需要随删除一起释放空间
- 当前项目还没有进入正式生产运行，不需要为“恢复文档”保留复杂分支

当前保留的唯一历史性副作用是：

- 如果题目曾引用过这份知识文档的 chunk，删除文档时会把对应 `question_sources` 一起清掉
- 题目本身仍然保留，但不会再显示这份文档作为来源引用

## 5. 前端行为

知识库页面现在增加了操作列：

- `预览`
- `编辑`
- `删除`

具体交互是：

- 预览：弹窗查看最新正文
- 编辑：弹窗修改标题和正文，保存后生成新版本
- 删除：确认后永久删除
- 批量删除：支持表格多选后一次性永久删除多份知识文档

## 6. 当前边界

这一轮已经完成：

- 知识库新增
- 知识库 URL 导入
- 知识库正文预览
- 知识库版本化编辑
- 知识库单条永久删除
- 知识库多选批量永久删除

这一轮还没有做：

- 版本历史对比页
- 文档正文 diff
- 基于文档版本的 chunk 重建

这些能力后面可以继续增强，但当前管理闭环已经成立。
