# 结构化日志与请求观测设计

## 目标

把当前项目的日志从“能看到报错”提升到“能快速定位问题”。

本次落地重点不是引入复杂平台，而是先把日志基础打稳：

1. 统一 JSON 日志
2. 请求级日志中间件
3. 关键业务事件日志
4. 外部依赖调用日志
5. 错误日志与接口错误码对齐

## 1. JSON 日志

当前后端统一输出单行 JSON 到标准输出。

这样做的好处是：

- Docker 日志更容易检索
- 后续接 Loki / Grafana 更顺
- 不需要再靠肉眼阅读松散字符串

## 2. 请求级日志

每个请求都会产生日志：

- `request_started`
- `request_finished`

当前会记录：

- `request_id`
- `actor_email`
- `method`
- `path`
- `status_code`
- `latency_ms`
- `error_code`

并且响应头会返回：

- `X-Request-ID`

这样前端、接口报错和 Docker 日志就能通过同一个请求 ID 串起来。

## 3. 业务事件日志

为了避免日志只有技术信息、没有业务语义，关键成功动作会单独记业务事件。

当前已接入：

- 知识库导入
- 知识库编辑
- 知识库归档
- 题目预览生成
- 题目批量入题库
- 题目编辑
- 试卷创建
- 试卷编辑
- 考试分配
- 开始作答
- 提交并 AI 判卷

## 4. 外部依赖日志

当前重点覆盖两类外部依赖：

### 4.1 网页抓取

会记录：

- 抓取开始
- 抓取成功
- 抓取失败
- 缓存命中
- 缓存读写失败

关键字段包括：

- `url`
- `timeout_ms`
- `cache_hit`
- `content_chars`
- `image_count`
- `nested_link_count`
- `latency_ms`

### 4.2 LLM 调用

会记录：

- 出题开始 / 成功 / 失败
- 判卷开始 / 成功 / 失败

关键字段包括：

- `provider`
- `model`
- `question_count`
- `question_type`
- `attempt_id`
- `tokens_in`
- `tokens_out`
- `latency_ms`

## 5. 错误码对齐

当前错误处理不只是返回统一 JSON 错误结构，也会在日志里写入同一个 `error_code`。

例如：

- `domain_invariant_violation`
- `domain_not_found`
- `request_validation_failed`
- `external_service_error`
- `internal_error`

这样查日志时，不需要再自己猜这条异常最终回给前端的是什么错误。

## 6. 当前边界

本次没有直接接入完整日志平台，只先把日志源头规范化。

也就是说：

- 现在适合先用 `docker logs` 看
- 后续如果要更舒服地检索，再接 `Loki + Grafana`

## 7. 后续建议

当前最自然的下一步是：

1. 给关键业务事件补 `department_id`、`user_id`
2. 给 AI 调用日志补 prompt 模板版本
3. 接 `Loki + Grafana`
4. 再把指标和日志串起来
