# 登录注册与身份权限认证设计

本文档说明 TalentHub 当前的登录、注册、身份识别与权限认证实现。

## 1. 为什么要从邮箱切换改成正式登录

项目早期为了快速联调，前端通过手工输入邮箱，把邮箱写进请求头来模拟当前操作者。

这个做法在联调阶段够用，但问题也很明显：

- 身份完全靠前端伪装，不是真正的登录态
- 权限链路和真实系统不一致
- 日志里缺少可靠的用户上下文
- 后续做注册、个人中心、会话管理时会越来越别扭

所以当前已经切换为正式 Bearer token 登录。

## 2. 当前认证链路

### 2.1 注册

- 前端进入“注册”页
- 读取部门列表
- 用户填写工号、姓名、邮箱、密码、部门和岗位
- 后端创建用户
- 默认授予 `employee`
- 立即返回 access token

### 2.2 登录

- 用户输入邮箱和密码
- 后端校验账户状态和密码哈希
- 成功后返回 access token
- 前端保存 token 和当前用户信息

### 2.3 当前用户

- 前端启动时如果本地存在 token
- 会主动调用 `/api/v1/identity/me`
- 后端解析 Bearer token
- 返回当前用户、部门和角色

如果 token 无效，前端会清空本地登录态并回到登录页。

## 3. 认证实现方式

### 3.1 密码哈希

当前使用标准库 `hashlib.pbkdf2_hmac`：

- 算法：`PBKDF2-SHA256`
- 迭代次数：`600000`
- 每条密码单独随机盐

格式：

- `pbkdf2_sha256$iterations$salt$digest`

这样做的原因是：

- 不引入额外密码依赖
- 本地开发足够清晰
- 便于 fail fast

### 3.2 Access Token

当前使用自定义 HMAC 签名 token，而不是额外引入 JWT 库：

- 负载中包含 `sub / email / iat / exp / ver`
- 使用 `AUTH_TOKEN_SECRET` 做签名
- 前端统一通过 `Authorization: Bearer <token>` 发送

这样做的目标是：

- 本地依赖更少
- 行为更可控
- 先把认证链路跑通

## 4. 当前公开接口

以下接口不需要登录：

- `GET /api/v1/identity/departments`
- `POST /api/v1/identity/register`
- `POST /api/v1/identity/login`
- `GET /api/v1/healthz`

其余业务接口默认需要 Bearer token。

补充说明：

- 知识库写接口现在也已经纳入登录态
- 不再允许匿名导入、匿名编辑或匿名归档知识文档

## 5. 当前角色模型

系统内置三类角色：

- `system_admin`
  - 跨部门可见，可管理全部题目与考试
- `department_admin`
  - 本部门管理员，可在本部门造题、造卷
- `employee`
  - 普通成员，只能参加自己有权限访问的考试

注册默认只会创建：

- `employee`

高权限角色不允许用户自行注册获取。

## 6. 和权限模型的衔接

当前后端会在请求阶段完成这几件事：

1. 解析 Bearer token
2. 校验签名和过期时间
3. 根据邮箱读取用户
4. 校验 token 的 `sub` 和数据库中的用户 id 一致
5. 读取角色列表
6. 把当前用户写入请求上下文和日志上下文

这样业务接口就可以直接依赖：

- 当前用户
- 当前部门
- 当前角色

而不需要自己重复查用户。

知识库、题库、考试这些写操作现在都基于当前登录用户记录真实操作者，而不是继续依赖开发期默认邮箱。

## 7. 当前前端行为

前端现在已经改成：

- 未登录先显示认证页
- 登录后才进入工作台
- 右上角展示当前用户、部门、岗位和角色
- 支持退出登录

不再保留“手工切邮箱”的开发期模式。

## 8. 开发环境默认账号

执行：

- `make db-seed-dev`

后，系统会写入这些账号：

- `admin@talenthub.local`
- `dept-admin@talenthub.local`
- `member@talenthub.local`
- `other-admin@talenthub.local`
- `other-member@talenthub.local`

默认密码统一为：

- `talenthub123`

## 9. 关键约束

当前认证实现明确遵守这些约束：

- 没有 token 就直接失败
- token 非法就直接失败
- token 过期就直接失败
- 用户不存在或已停用就直接失败
- 不保留旧的 `X-Actor-Email` 兼容逻辑

也就是说，这条链路故意不做“双模式长期兼容”，避免系统同时存在两套身份模型。

## 10. 之后可以继续演进的方向

后续可以继续增强，但当前还没有做：

- refresh token
- 忘记密码 / 重置密码
- 邮箱验证码
- 会话吊销
- 细粒度权限声明
- 单点登录（SSO）

这些能力可以以后再加，但不影响现在的本地开发和权限建模主链路。
