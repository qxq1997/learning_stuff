# TalentHub 整体架构与设计原则

## 1. 系统使命

TalentHub 的目标不是提供一个泛用聊天入口，而是把企业培训拆成可控、可追踪、可复用的业务链路：

- 获取知识来源
- 生成题目
- 沉淀题库
- 编纂考试
- 完成作答
- 进行 AI 判卷与反馈
- 沉淀指标与趋势

AI 是关键能力放大器，但不是系统边界本身。系统边界始终由业务对象和状态流转定义。

## 2. 架构风格

项目采用 **DDD + 模块化单体**。

核心原因：

- 当前仍处于从 0 到 1 阶段，模块化单体的认知成本更低
- 领域边界可以先稳定下来，避免过早拆服务
- 统一事务边界对题目保存、考试提交、判卷回写更友好
- 后续若知识处理或考试流量单独膨胀，再拆服务也更自然

目录层次按四层组织：

- `domain`
  - 实体、值对象、读模型、领域规则
- `application`
  - 用例、命令、查询、端口接口
- `infrastructure`
  - ORM、仓储实现、文件存储、网页抓取、Redis、外部 AI
- `interfaces`
  - API schema、router、presenter、policy

## 3. 核心设计原则

### 3.1 fail fast

系统不靠猜测补全用户意图，不靠隐式 fallback 掩盖错误。

具体体现：

- 鉴权头缺失直接失败
- 跨部门越权直接失败
- 网页正文过短直接失败
- 长文本超限先要求显式确认
- 结构化输出不符合预期直接失败

### 3.2 接口层保持薄

router 只做四件事：

- 解析请求
- 获取 actor 与依赖
- 调用用例
- 记录语义日志并返回 presenter 输出

映射逻辑、权限规则、provider 构造、缓存辅助不继续堆在 router 内部。

### 3.3 缓存不是事实来源

事实来源只有数据库和明确持久化文件。缓存只承担：

- 标签列表加速
- 网页抓取短期复用

因此：

- 写路径先保证数据库成功
- 缓存失效失败要被日志显式记录
- 但不让缓存反向控制领域正确性

### 3.4 AI 通过端口接入

应用层依赖抽象端口，基础设施层提供具体 provider 实现。这样可以：

- 先用 OpenAI / DeepSeek 兼容接口快速落地
- 后续接入本地模型或多 provider 路由
- 避免在业务用例里散落 SDK 细节

## 4. 当前限界上下文

### 4.1 Identity

解决“谁是谁、属于哪个部门、拥有什么权限”的问题。

### 4.2 Knowledge

解决“培训内容从哪里来、如何导入、如何查看、如何维护”的问题。

### 4.3 Question Bank

解决“题目如何生成、保存、搜索、编辑、标注”的问题。

### 4.4 Exam

解决“如何从题库组卷、分配、作答、提交与查看结果”的问题。

### 4.5 Shared / Cross-cutting

解决配置、日志、鉴权、缓存、错误、数据库会话等跨模块共性问题。

## 5. 当前代码层面的关键改进

本轮重构后，接口层开始形成清晰分工：

- `presenters.py`
  - 负责读模型到 API response 的映射
- `policies.py`
  - 负责接口层的访问控制规则
- `shared/ai/factories.py`
  - 负责 AI provider 的统一构造
- `interfaces/cache.py`
  - 负责缓存 key 和失效逻辑

这一步解决了过去 router 同时承担“请求入口 + DTO 映射 + 权限规则 + provider 工厂 + 缓存策略”的问题。

## 6. 下一阶段演进方向

- 把更多接口层细节继续收敛为依赖装配模块
- 为知识库、题库、考试补更清晰的 query/service 分层
- 把前端也按页面与领域组件继续拆薄
- 将当前 smoke test 沉淀为稳定回归基线
