# 身份权限与组织域设计

## 1. 要解决的问题

TalentHub 不是公开练习站，而是企业培训系统。系统必须先回答三个问题：

- 当前是谁
- 这个人属于哪个部门
- 这个人能做什么，能看到什么

如果这三个问题不稳定，题库、考试、报表都会失去边界。

## 2. 领域对象

核心对象包括：

- `User`
- `Department`
- `Role`
- `ActorContext`

其中：

- `User` 和 `Department` 是业务身份事实
- `Role` 是权限语义
- `ActorContext` 是接口层在一次请求中的最小有效身份载体

## 3. 权限模型

当前权限分成三层：

- `system_admin`
  - 跨部门可见，可跨部门管理
- `department_admin`
  - 仅管理本部门的题目、考试与知识库
- 普通成员
  - 可登录、查看自己有权访问的考试、参与作答

当前规则：

- 只有 `department_admin` / `system_admin` 可以造题、编辑题、造考试、编辑考试、写知识库
- 私有考试只允许所属部门访问
- 公共考试对登录用户开放
- 普通成员不能进入作者工作台执行写操作

## 4. 鉴权流程

系统使用 Bearer token：

1. 用户注册或登录
2. 后端签发 access token
3. 后续请求通过 `Authorization: Bearer <token>` 传递
4. 服务端校验 token 后查询当前活跃用户
5. 组装 `ActorContext`

重要约束：

- token subject 必须与数据库当前用户一致
- 用户必须是 `ACTIVE`
- 鉴权失败立即中止，不做匿名降级

## 5. 为什么不用把权限写死在前端

前端只承担交互优化，不承担授权事实。真正的边界必须在后端。

否则会出现：

- 前端隐藏了按钮，但接口仍可直接调用
- 跨部门访问靠 UI 约束而不是服务端约束
- 审计日志无法反映真实访问控制

## 6. 当前设计亮点

### 6.1 ActorContext 统一身份入口

当前请求一旦通过鉴权，就会生成统一的 `ActorContext`。接口层不再到处传邮箱字符串，也不再通过临时 header 模拟身份。

### 6.2 权限拒绝日志语义清晰

像 `authoring_permission_denied`、`exam_paper_access_denied`、`question_access_denied` 这些事件会明确记录：

- 谁被拒绝
- 因为什么被拒绝
- 涉及哪个资源

这比只看 `403/409` 更适合审计和排错。

### 6.3 权限与领域边界一致

权限不是按页面控制，而是按领域动作控制：

- 造题
- 造考试
- 查看私有考试
- 查看考试尝试

## 7. 已解决的问题

- 移除了开发期通过请求头硬切邮箱的做法
- 修复了知识库写接口可匿名调用的问题
- 统一了“部门管理员可写，普通成员只读/作答”的行为边界

## 8. 未来改进点

- 增加 refresh token 和会话失效策略
- 增加用户管理页与角色管理页
- 细化到“题库管理员 / 考试管理员 / 只读审计员”等更细角色
- 对登录失败、注册失败加入更细粒度的安全审计
