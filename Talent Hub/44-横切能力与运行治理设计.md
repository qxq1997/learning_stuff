# 横切能力与运行治理设计

## 1. 为什么横切能力要单独设计

很多项目在早期会先写业务，等功能越来越多后再补日志、缓存、错误处理、配置管理。那样很容易形成隐性耦合，后期维护成本高。

TalentHub 把这些能力收在 `shared` 层，目标是：

- 业务模块尽量不重复造轮子
- 日志、错误、配置表现一致
- 基础设施替换时影响面可控

## 2. 配置管理

当前配置集中在 `Settings` 中，主要覆盖：

- 鉴权 token
- 数据库连接
- Redis
- 日志输出
- AI provider
- 网页抓取
- 缓存 TTL

设计原则：

- 所有环境依赖都显式配置
- 密钥不进代码仓库
- provider 切换通过配置驱动，而不是业务代码分支到处散落

## 3. 错误处理

错误分层：

- `DomainInvariantError`
  - 业务规则不满足
- `DomainNotFoundError`
  - 资源不存在
- `ExternalServiceError`
  - 外部依赖失败

接口层统一把这些错误转成结构化响应，保证前端和日志能对齐。

## 4. 语义日志

日志设计目标不是“把所有系统噪音都打出来”，而是让日志成为业务观察面。

当前重点记录：

- 请求开始 / 结束
- 鉴权与权限拒绝
- 知识库导入、编辑、归档、预览
- 题目生成、保存、编辑、查看、标签列表
- 考试创建、编辑、查看、分配、作答、判卷
- 网页抓取开始 / 成功 / 失败
- AI 出题 / 判卷开始 / 成功 / 失败

这解决了过去只能看“接口 200/500”的问题。

## 5. 缓存策略

缓存只覆盖低风险、高复用读路径：

- 题库标签列表
- 网页抓取结果

策略明确：

- 缓存不是事实来源
- 失效失败要记录日志
- 写路径不应该让缓存成为强依赖

## 6. 可观测性

当前本地栈支持：

- 后端结构化 JSON 日志
- Promtail 采集
- Loki 存储
- Grafana 查询与 dashboard

这样即使在本地开发，也能按：

- `event`
- `request_id`
- `actor_email`
- `error_code`

去追踪系统行为。

## 7. 当前设计亮点

### 7.1 外部日志噪音被压低

像 Uvicorn、httpx、OpenAI SDK 的底层 INFO 已被降噪，Grafana 中更容易看到 TalentHub 自己的业务语义日志。

### 7.2 provider 构造集中化

AI provider 已集中到工厂模块，避免接口层直接拼装 SDK 参数。

### 7.3 接口层开始职责分离

presenter、policy、factory、cache helper 的拆分，降低了 router 的认知负担。

## 8. 未来改进点

- 为慢接口和慢查询增加独立事件
- 引入更明确的 metrics 指标体系
- 对关键任务增加 trace 级关联
- 将 smoke test 接入 CI
