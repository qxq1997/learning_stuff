# 出题链路语义日志与错误可读性设计

## 目标

TalentHub 的日志不应该主要回答“哪个 API 被调用了”，而应该主要回答下面这些业务问题：

- 现在是不是开始出题了
- 这次出题用的是什么来源
- 题型和数量是怎么分配的
- 是单次生成还是分页生成
- 当前已经走到哪一步了
- 调用的模型是什么，耗时多少
- 失败到底失败在哪一步，错误原因是什么

这份设计文档专门约束“出题链路”的日志表达方式，让它在 `docker logs`、Loki、Grafana 里都更接近业务流程图，而不是一堆底层请求噪声。

## 设计原则

### 1. 成功请求日志降噪

成功的 `request_started` / `request_finished` 不再占据 `INFO` 级别，而是降到 `DEBUG`。

这样做的目标不是取消请求观测，而是把 `INFO` 层让给真正的业务动作。当前保留策略是：

- 成功请求：`DEBUG`
- 失败请求：`WARNING`
- 业务步骤：`INFO`
- 业务失败：`ERROR`

换句话说，现在默认看到的日志应该是“业务步骤流”，不是“路由打点流”。

### 2. 用业务阶段表达流程

出题链路的日志必须能串成一条完整叙事。当前统一使用：

- `workflow=question_generation`

并按阶段输出事件，例如：

- `question_generation_source_prepared`
- `question_generation_source_resolved`
- `question_generation_flow_started`
- `question_generation_single_pass_selected`
- `question_generation_segmentation_planned`
- `question_generation_model_dispatch`
- `llm_question_generation_started`
- `llm_question_generation_response_received`
- `llm_question_generation_output_validated`
- `question_generation_flow_succeeded`

这样查日志时，不需要把 API 路由名翻译回业务动作。

### 3. 每条关键日志都要有可扫读摘要

纯字段式 JSON 很适合机器，但不适合人眼快速扫描。因此关键业务日志统一补充：

- `summary`

例如：

- `prepared common topic source 'Python 基础'`
- `dispatching single_choice generation for 'Python 基础' with 1 question(s)`
- `openai:gpt-5-mini generation succeeded for single_choice in 24263 ms`

这样即使在原始 `docker logs` 里，人也能快速看懂每一行在讲什么。

## 出题链路的日志分层

## 1. 来源准备层

这一层回答“题目是从哪里来的”。

当前支持：

- 知识库文档
- 粘贴文本
- 网页 URL
- 通用基础题主题

对应日志：

- `question_generation_source_resolved`
- `question_generation_source_prepared`
- `question_generation_web_fetch_requested`
- `question_generation_web_fetch_completed`

关键字段包括：

- `source_kind`
- `source_title`
- `source_document_id`
- `source_url`
- `source_char_count`
- `source_status`

## 2. 生成规划层

这一层回答“这次准备怎么生成”。

对应日志：

- `question_generation_flow_started`
- `question_generation_single_pass_selected`
- `question_generation_segmentation_planned`
- `question_generation_segmentation_confirmation_required`
- `question_generation_web_scope_confirmation_required`
- `question_generation_web_scope_resolved`
- `question_generation_web_child_pages_selected`
- `question_generation_web_scope_assembled`

关键字段包括：

- `question_types`
- `question_type_allocations`
- `requested_question_count`
- `source_max_chars`
- `segment_count`
- `used_segment_count`
- `web_page_scope`

这一步的价值是把“为什么需要确认分页”“为什么需要确认网页范围”直接解释清楚，而不是只返回前端弹窗。

## 3. 模型调度层

这一层回答“到底发给模型的是什么任务”。

对应日志：

- `question_generation_model_dispatch`
- `question_generation_model_dispatch_succeeded`
- `question_generation_model_dispatch_failed`

关键字段包括：

- `question_type`
- `question_count`
- `difficulty`
- `segment_index`
- `segment_count`

这一步特别重要，因为它把“总任务”和“单次模型调用”拆开了。一次总生成可能拆成多次模型请求，调试时必须看得到这个边界。

## 4. LLM 执行层

这一层回答“模型调用是否成功，成功后返回了什么”。

对应日志：

- `llm_question_generation_started`
- `llm_question_generation_response_received`
- `llm_question_generation_output_validated`
- `llm_question_generation_output_invalid`
- `llm_question_generation_succeeded`
- `llm_question_generation_failed`

关键字段包括：

- `provider`
- `model`
- `prompt_char_count`
- `question_type`
- `question_count`
- `returned_question_count`
- `tokens_in`
- `tokens_out`
- `latency_ms`

这一步主要解决两个问题：

- 区分“模型没返回”和“模型返回了但结构不合法”
- 快速判断慢在哪里，是调用时间长，还是前后业务处理长

## 5. 结果落点层

这一层回答“整个生成流程最终有没有完成”。

对应日志：

- `question_generation_flow_succeeded`
- `question_preview_generated`
- `question_draft_save_started`
- `question_draft_save_validated`
- `question_draft_save_succeeded`

这样最终能把“预览成功”和“正式入题库成功”明确区分开。

## 错误可读性设计

## 1. 请求校验错误必须指出位置

过去的错误信息是统一的：

- `request validation failed`

这对前端和日志都不够友好。现在会返回更清晰的 message，例如：

- `validation failed at body.question_types: List should have at least 1 item after validation, not 0`

日志里也会同时带：

- `message`
- `first_error_location`
- `first_error_message`
- `first_error_type`

这样一眼就知道到底是哪个字段不符合约束。

## 2. 业务错误必须直接暴露原因

例如：

- 网页正文过短
- 网页需要先确认范围
- 内容超长需要分页生成
- 模型返回的题型不符合请求
- 模型返回的推荐分值不是正数

这些错误都不应该被“包装成泛化失败”，而应该明确写进 message。当前异常处理要求：

- `DomainInvariantError` 直接返回业务约束原因
- `ExternalServiceError` 直接返回外部依赖失败原因
- 日志里同步保留 `message`

## 3. 失败必须落在具体步骤上

如果失败发生在模型调度阶段，就应该看到：

- `question_generation_model_dispatch_failed`

如果失败发生在模型输出校验阶段，就应该看到：

- `llm_question_generation_output_invalid`

而不是只看到最外层一个“出题失败”。

## 实际阅读方式

如果想查一次出题流程，最推荐按 `request_id` 看，并优先筛：

- `workflow=question_generation`

观察顺序通常是：

1. `question_generation_source_*`
2. `question_generation_flow_started`
3. `question_generation_*_selected / *_planned`
4. `question_generation_model_dispatch`
5. `llm_question_generation_*`
6. `question_generation_flow_succeeded`
7. `question_preview_generated`

如果中断，就看最后一条业务日志停在哪一步。

## 当前收益

这套设计已经解决了几个实际问题：

- 个人考试中心那类请求校验问题，不再只显示笼统报错
- 出题日志不再被成功 API 请求刷屏
- 可以用业务步骤直接判断卡在“来源准备”“分页确认”“模型调用”“输出校验”的哪一层
- 在 Grafana 和 `docker logs` 里都能直接扫读 `summary`

## 后续改进点

当前还有几个自然的下一步：

1. 给出题日志补 `department_id` 和 `actor_user_id`
2. 给不同出题入口补统一的 `entry_point`
3. 把 `summary` 再压缩得更适合 Grafana 列表视图
4. 给考试判卷链路补同等粒度的逐步业务日志
5. 在 Grafana 里按 `workflow=question_generation` 做专门 dashboard
