# 全链路业务日志与错误表达设计

## 目标

TalentHub 的日志和错误信息，不应该只服务于“程序员排查 API”，而应该服务于下面这些更接近业务的问题：

- 谁刚刚登录了
- 谁看了哪些考试、题目、指标
- 哪份知识文档被导入、更新、归档了
- 哪次出题走到了哪一步
- 哪套试卷什么时候创建、分配、开始、提交、判卷
- 某次失败到底是权限不够、参数不合法、业务状态不允许，还是外部依赖出错

因此这套设计的目标是：

1. 让所有主要操作都有明确业务日志
2. 让成功日志优先体现业务语义，而不是 HTTP 噪声
3. 让错误响应和错误日志都直接说明失败原因
4. 让后来者能按统一规则继续补日志，而不是每个模块各写各的

## 一、日志设计总原则

### 1. 业务日志优先

默认 `INFO` 级别应该主要看到：

- 登录
- 查看
- 创建
- 更新
- 导入
- 归档
- 分配
- 开始考试
- 提交考试
- AI 出题
- AI 判卷
- 指标查看
- 权限拒绝

而不是：

- 每个成功请求的 `request_started`
- 每个成功请求的 `request_finished`

因此当前策略是：

- 成功请求日志：`DEBUG`
- 失败请求日志：`WARNING`
- 业务成功日志：`INFO`
- 业务失败日志：`WARNING / ERROR`

### 2. 同时服务机器和人

每条结构化日志都保留机器友好字段，例如：

- `event`
- `request_id`
- `resource_id`
- `user_id`
- `department_id`
- `status_code`
- `error_code`

同时关键日志还必须带：

- `summary`

`summary` 的要求很简单：人扫一眼就能知道这一行发生了什么。

### 3. 失败信息直接说原因

错误表达坚持 `fail fast`，不做模糊描述，也不做过度包装。

例如：

- `Authorization header is required`
- `only department admin can perform this action`
- `validation failed at body.question_types: List should have at least 1 item after validation, not 0`
- `source content exceeds the single-pass limit and must be generated in multiple pages`

也就是说，错误应该回答“为什么不行”，而不是只回答“失败了”。

## 二、当前覆盖到的业务模块

## 1. 身份与权限

当前已经有明确业务日志：

- `identity_departments_listed`
- `identity_user_registered`
- `identity_user_logged_in`
- `identity_current_user_viewed`
- `authoring_permission_denied`

它们解决的问题是：

- 能看清谁登录了、谁注册了
- 能看清权限是在哪一层被拒绝的
- 能把身份动作和后续业务动作串到同一个 `request_id`

## 2. 知识库

当前已经有明确业务日志：

- `knowledge_documents_listed`
- `knowledge_document_viewed`
- `knowledge_manual_import_started`
- `knowledge_operator_resolved`
- `knowledge_manual_import_succeeded`
- `knowledge_web_import_started`
- `knowledge_web_import_fetched`
- `knowledge_web_import_succeeded`
- `knowledge_update_started`
- `knowledge_update_succeeded`
- `knowledge_archive_started`
- `knowledge_archive_succeeded`

它们解决的问题是：

- 可以区分“列表查看”和“真实导入动作”
- 可以区分“网页抓到了”和“文档真的入库了”
- 可以追踪文档更新和归档

## 3. 题库与出题

当前已经有明确业务日志：

- `question_drafts_listed`
- `question_tags_listed`
- `question_viewed`
- `question_updated`
- `question_access_denied`
- `question_generation_*`
- `llm_question_generation_*`
- `question_drafts_saved`

它们解决的问题是：

- 列表、详情、更新不再只是路由调用
- 出题流程能够按步骤定位卡点
- 跨部门访问题目时可以明确记录拒绝原因

## 三、错误表达规则

## 1. API 错误结构统一

当前所有错误统一返回：

```json
{
  "error": {
    "code": "...",
    "message": "...",
    "details": {}
  }
}
```

其中：

- `code` 负责稳定分类
- `message` 负责直接解释原因
- `details` 负责附加结构化上下文

## 2. 校验错误必须说清字段位置

以前只有：

- `request validation failed`

现在要求至少明确到：

- `body.question_types`
- `query.limit`
- `path.exam_paper_id`

并在日志中保留：

- `first_error_location`
- `first_error_message`
- `first_error_type`

## 3. 业务状态错误必须说清当前不允许什么

例如：

- `exam attempt is not in progress`
- `knowledge document is already archived`
- `assignment is expired`
- `cross-department private exam access is forbidden`

不使用这种模糊说法：

- `operation failed`
- `bad request`
- `state invalid`

## 四、当前仍然存在的边界

虽然日志可读性已经明显改善，但还有几个边界仍然存在：

1. 不是所有 repository 内部步骤都打日志  
说明：目前更偏向在“用例层”和“接口层”打语义日志，避免底层过度刷屏。

2. 某些只读查询目前仍然只有一条完成日志  
说明：如果后面某类查询明显复杂，再拆“开始/完成”两段。

3. AI 判卷链路的输出校验日志还可以继续细化  
说明：目前已经有开始、响应收到、成功，但还可以继续补“输出校验失败”的细分原因。

## 五、后续改进方向

下一阶段最值得继续做的是：

1. 给所有业务日志统一补 `department_id` 和 `actor_user_id`
2. 给知识库网页导入补更细的抓取阶段日志
3. 给题目保存、考试编辑、指标查询补更稳定的 `resource_type`
4. 在 Grafana 中按模块预置面板：
   - 身份权限
   - 知识库
   - 出题
   - 考试
   - 指标
5. 给所有 AI 调用日志补 prompt 模板版本或策略版本

## 六、结论

TalentHub 的日志设计，不再是“框架日志 + 少量异常日志”的模式，而是：

- 请求日志退后
- 业务日志居中
- 错误信息直说原因
- 关键步骤按流程展开

这套规则的直接收益是：

- 更容易在 `docker logs` 里人工排查
- 更容易在 Loki / Grafana 里按业务动作搜索
- 更容易让后来者理解系统到底发生了什么
