# 题目与考试删除设计

## 设计目标

TalentHub 现在把题目和考试的删除统一定义为**永久删除**，不再保留“归档删除”这条业务语义。

这样做是为了满足两个直接需求：

1. 管理员希望删掉的内容真正消失，而不是继续以隐藏状态留在系统里。
2. 前端不再需要理解“已归档但仍存在”的双重语义，交互和权限判断可以更直接。

## 当前策略

### 题目删除

接口：

- `DELETE /api/v1/question-bank/questions/{question_id}`
- `POST /api/v1/question-bank/questions/bulk-delete`

行为：

- 题目会被物理删除
- 题目版本、题目标签、题目来源引用会随题目一起删除
- 如果题目已经被某些试卷使用，相关试卷也会被一并永久删除
- 相关试卷的分配、作答、判卷结果也会一起删除

原因：

题目和试卷之间存在真实外键依赖。既然产品要求“题目想删就删”，系统就不能再靠阻止删除来保护历史，而必须把整条依赖链一起删干净。

### 考试删除

接口：

- `DELETE /api/v1/exams/papers/{exam_paper_id}`
- `POST /api/v1/exams/papers/bulk-delete`

行为：

- 试卷会被物理删除
- 试卷标签、试卷题目快照会一起删除
- 相关考试分配会一起删除
- 相关作答、判卷结果、AI 判卷任务记录会一起删除

原因：

考试不是一个孤立对象。它和分配、作答、判卷是一整张业务图。永久删除必须作用于整张图，而不是只删根对象。

## 批量删除

题库页和考试页现在都支持多选。

批量删除遵循与单条删除完全相同的语义：

- 没有“批量归档”
- 没有“只隐藏”
- 没有“删根对象，保留依赖”

系统会先校验权限，再执行整批永久删除；如果请求里的任意对象不存在或无权删除，请求会直接失败，不会做部分成功。

## 权限边界

- `system_admin` 可以跨部门删除题目和考试
- `department_admin` 只能删除本部门拥有的题目和考试
- 普通成员没有题库和考试作者权限，因此也没有删除权限

## 一致性策略

为了保持数据库一致性，删除顺序是显式定义的。

### 删除题目时

1. 先找到引用这些题目的试卷
2. 先删除相关试卷图
3. 再删除题目本体及其版本、标签、来源引用

### 删除考试时

1. 先删除相关 AI 判卷任务引用
2. 再删除作答
3. 再删除考试分配
4. 再删除试卷题目与标签
5. 最后删除试卷本体

这条顺序不是为了“温柔兜底”，而是为了明确依赖方向，确保每一步都符合数据库约束。

## 错误表达

删除链路现在只保留与永久删除语义一致的错误信息。

典型示例：

- `question not found`
- `one or more questions were not found`
- `exam paper not found`
- `one or more exam papers were not found`

系统不再返回：

- `question is already archived`
- `exam paper is already archived`
- `question selected by active exam papers cannot be archived`
- `exam paper with assignments or attempts cannot be archived`

## 日志设计

删除链路会输出明确的业务日志，重点描述删除动作和连带影响。

题目删除：

- `question_delete_requested`
- `question_delete_succeeded`
- `question_deleted`
- `question_bulk_deleted`

考试删除：

- `exam_paper_delete_requested`
- `exam_paper_delete_succeeded`
- `exam_paper_deleted`
- `exam_paper_bulk_deleted`

日志会带出这些字段：

- 删除对象数量
- 连带删除的试卷数量
- 连带删除的分配数量
- 连带删除的作答数量

这样在 Grafana / Loki 里能直接看出一次删除究竟影响了多大范围。

## 前端交互

题库页和考试页现在都提供：

- 单条永久删除
- 多选批量永久删除

确认弹窗会明确提示：

- 删除无法恢复
- 题目删除可能连带删掉关联试卷
- 考试删除会连带删掉分配、作答和判卷结果

这样用户看到的动作和系统真实行为是一致的。

## 历史归档清理

系统已经补了维护脚本，用来把旧版本遗留的归档题目和归档考试一次性转成永久删除：

```bash
docker exec -i talenthub-backend python - < backend/scripts/purge_archived_questions_and_exams.py
```

这个脚本适合两种场景：

- 从“归档语义”迁移到“永久删除语义”
- 本地环境里清理历史遗留数据

## 未来可演进点

当前删除能力优先满足“删掉就彻底消失”的产品预期。后续如果需要再增强，可以考虑：

1. 删除影响预估
   在确认前先展示“将删除多少题、多少试卷、多少作答”

2. 删除审计页
   保留独立的审计日志视图，而不是保留业务对象本身

3. 异步大批量删除
   当删除范围特别大时，可以切到后台任务执行并回报进度
