# 知识库 RAG 闭环设计

## 1. 目标

TalentHub 里的 RAG 不是一个独立聊天机器人，而是一条围绕企业培训的业务闭环：

1. 知识库文档入库
2. 切片、索引、检索相关片段
3. 基于检索片段生成题目
4. 用户作答
5. AI 判卷
6. 再按题目引用回查知识片段，生成学习建议

核心目标是把“内部知识”真正带入培训流程，而不是只把文档存起来。

## 2. 为什么这样设计

如果直接把整篇文档塞给模型，会有三个典型问题：

- 上下文太长，成本和延迟都偏高
- 出题焦点不清，容易绕开真正关键规则
- 判卷和学习建议缺少可追溯依据

因此这一版采用了更清晰的业务做法：

- 文档入库时立即切片并建立向量索引
- 出题时先检索，再把检索结果作为生成上下文
- 保存题目时，把题目和知识片段的引用关系落库
- 判卷时沿着题目引用关系再回查知识片段，给出推荐回看资料

## 3. 领域边界

### Knowledge 模块负责什么

- 管理知识文档和文档版本
- 维护 `knowledge_chunks`
- 提供检索能力

### Question Bank 模块负责什么

- 决定什么时候需要走知识库检索
- 把检索片段转成出题上下文
- 在保存题目时写入 `question_sources`

### Exam 模块负责什么

- 用户作答和提交
- 调用 AI 判卷
- 在反馈阶段回查 `question_sources`，补出学习依据

这三个模块各自只做自己最该负责的事，没有把 RAG 逻辑堆进一个“大服务”里。

## 4. 当前落地方案

### 4.1 文档入库即索引

当前不再停留在 `parsed` 状态。

知识文档创建或更新时，会同步完成：

- 文本规范化
- 父子 chunk 切片
- 本地轻量 embedding 计算
- 写入 `knowledge_chunks`
- 文档状态推进到 `indexed`

这样后续出题不需要再猜“这个文档有没有准备好”。

### 4.2 出题先检索

当来源是 `knowledge_document` 时，出题流程不再直接把全文传给模型，而是先构造检索问题：

- 文档标题
- 目标题型
- 难度
- 标签

然后召回最相关的知识片段，并把这些片段拼成 RAG 上下文，再交给 LLM 出题。

### 4.3 题目保存时记录依据

生成预览和保存草稿之间，不再丢失检索结果。

保存时会把本次用于生成的知识片段一起带回后端，并写进：

- `question_versions.generation_metadata_json`
- `question_sources`

这样一来，题目与知识依据的关系就是结构化、可追溯的。

### 4.4 判卷后补学习建议

考试提交后，系统会按题目版本回查 `question_sources`，再把这些知识片段作为学习依据：

- 一方面带进 AI 判卷 prompt，帮助模型给出更贴近知识库的建议
- 另一方面直接作为“推荐回看资料”返回给前端

所以当前考试反馈不再只有：

- 对错
- 分数
- 一段泛泛建议

而是会额外告诉用户：

- 应该回看哪一段知识
- 这段知识来自哪份文档

## 5. 关键数据结构

### `knowledge_chunks`

存储：

- 文档版本
- chunk 序号
- chunk 内容
- embedding
- token 数
- chunk 元数据

### `question_sources`

存储：

- 题目版本
- 关联 chunk
- relevance score

它的意义不是“统计用途”，而是把题目依据正式建模。

## 6. 日志设计

RAG 这条链路已经补了明确的业务日志，关键事件包括：

- `knowledge_document_index_built`
- `question_generation_retrieval_started`
- `question_generation_retrieval_completed`
- `exam_attempt_learning_references_requested`
- `exam_attempt_learning_references_loaded`

这些日志的目的不是记录“接口被调了”，而是告诉我们：

- 文档有没有真正建索引
- 出题有没有先检索
- 检索到了多少片段
- 判卷反馈有没有成功拿到学习依据

## 7. 这版解决了什么问题

- 知识库终于从“存文档”升级到“参与业务闭环”
- 内部培训题不再只能靠整篇文档强灌模型
- 题目依据可追溯
- 判卷后的学习建议不再悬空

## 8. 当前边界

这版是“先闭环、再增强”，所以仍然有边界：

- embedding provider 已经抽象完成，并已将 `BGE-M3` 设为当前默认文本 embedding provider
- 父子 chunk 结构已经落地，检索后端当前固定为 `Qdrant` dense 路径；最终目标里的 sparse / multi-vector / reranker 仍未接入
- 历史已索引文档不会被自动重切成父子结构，避免破坏既有题目来源引用
- 检索范围当前以文档级为主，还没有细化到部门级知识域
- 学习建议当前按题目引用关系回查，不是独立多跳 agent

这些都是有意识的收敛，不是遗漏。

## 9. 下一步演进

最值得继续做的方向是：

1. 检索评测与 embedding 升级
2. 知识域范围控制
3. 反馈里的引用高亮和跳转
4. 基于错题标签和知识片段的学习路径推荐

下一阶段的重点不是把 RAG 做得更“花”，而是把：

- 检索质量
- 可解释性
- 学习效果

继续做深。
