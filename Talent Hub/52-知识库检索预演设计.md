# 知识库检索预演设计

## 目标

知识库进入 RAG 闭环后，真正决定出题质量的不是“文档有没有导入”，而是“检索阶段到底命中了什么片段”。  
如果这一步不可见，用户只能在题目质量下降后被动猜测：

- 是知识库正文不对
- 是切片太碎或太长
- 是检索问题写得不好
- 还是模型本身生成偏了

因此 TalentHub 增加“知识库检索预演”能力，让用户在生成题目前，直接看到当前文档面对某个问题时会命中哪些 chunk。

## 解决的问题

### 1. 让 RAG 从黑盒变成可观察流程

原先用户只能看到：

- 文档已经入库
- 最终题目已经生成

中间最关键的“检索”步骤是隐形的。  
检索预演把这一步显式暴露出来，让用户可以先验证：

- 当前问题是否能命中正确段落
- 命中的片段是否足够具体
- 当前 chunk 切分是否合理

### 2. 把问题尽量前移到知识库阶段

如果题目生成质量不好，再回头排查会同时牵涉：

- 文档内容
- 检索查询
- prompt
- 模型输出

而检索预演可以把问题压缩到更早的阶段：

- 若检索错了，先修文档和检索
- 若检索对了，再看生成逻辑

### 3. 支持教学和调试

TalentHub 本身也是一个学习型项目。  
检索预演不仅服务业务，也服务学习：

- 方便观察 chunk 化和向量检索是否符合直觉
- 方便比较不同问题对召回结果的影响
- 方便后续替换 embedding provider 后做效果对比

## 设计原则

### 1. 预演只读，不隐式触发生成

检索预演只做一件事：

- 输入问题
- 返回命中的片段

它不会顺手生成题目，也不会写数据库中的题目实体。  
这样边界最清晰，行为最可预测。

### 2. 只针对单一知识文档

当前预演范围限制为“单文档内的当前版本”。  
原因是：

- 用户心智最清晰
- 便于调试当前文档的质量
- 不会把跨文档召回、权限边界、结果融合一次性混进来

后续如果需要跨文档检索，再单独扩展。

### 3. 复用真实 RAG 检索逻辑

检索预演不是一套“展示专用逻辑”，而是直接复用知识库检索基础设施：

- 同样的 query embedding 生成
- 同样的最新版本 chunk 召回
- 同样的 relevance score 排序

这样预演看到的结果和真正出题时用到的结果才有可比性。

## 后端设计

### 查询模型

新增 `PreviewKnowledgeRetrievalUseCase`，输入包括：

- `document_id`
- `query_text`
- `limit`

执行步骤：

1. 校验知识文档存在
2. 生成 query embedding
3. 在当前文档最新版本的 chunk 上执行相似度检索
4. 返回结构化检索结果

### 返回结构

返回结构包含：

- 文档 id
- 文档标题
- 原始 query text
- 本次返回上限
- 实际命中的 chunk 列表

每个 chunk 都带：

- chunk id
- chunk index
- content
- token count
- relevance score
- source url

### 业务日志

新增两条明确语义日志：

- `knowledge_retrieval_preview_requested`
- `knowledge_retrieval_preview_completed`

这样 Grafana 和本地日志里可以直接按 `workflow=knowledge_retrieval_preview` 过滤。

## 前端设计

### 入口位置

检索预演被放进知识库文档的“预览弹窗”中，而不是单独做一页。

原因：

- 用户已经先选中了某份知识文档
- 同一弹窗里可以先看正文，再切到检索预演
- 让“文档内容”和“检索结果”形成直接对照

### 页面结构

预览弹窗分成两个页签：

- `文档正文`
- `RAG 检索预演`

检索预演页签提供：

- 检索问题输入框
- 返回片段数
- 命中结果列表

每个结果展示：

- chunk 编号
- 相关度
- token 数
- 片段正文

## 当前边界

当前实现明确保留了几个边界：

- 只支持单文档预演
- 不做跨文档召回
- 不做高亮命中词
- 不做检索结果人工标注
- 仍使用当前本地轻量 embedding 实现

这些边界是刻意保留的，目的是先把“可见、可调、可验证”的最小闭环做稳。

## 未来演进

最值得继续做的方向有三个：

### 1. 跨文档检索预演

允许用户直接问一个问题，看整个知识库会命中哪些文档和片段。

### 2. 检索质量评测

对常见问题集维护“期望命中文档/片段”，形成简单评测，支持 embedding 升级前后对比。

### 3. 检索结果到出题结果的链路追踪

把“哪些 chunk 被检索到”与“最终哪些题引用了这些 chunk”直接关联展示，形成更完整的 RAG 可解释链路。
