# 数据库约束、取舍与潜在升级点

## 目标

前两篇文档解释了“有哪些表、它们怎么连”。本文档进一步解释：

- 为什么某些关系用强外键，有些不用
- 为什么文档、题目、试卷要版本化或冻结
- 当前 schema 的优势和代价分别是什么
- 后面如果继续演进，数据库层最值得升级什么

## 一、当前 schema 的核心原则

### 1. 核心业务链路强约束

TalentHub 在以下地方尽量使用强外键和唯一约束：

- 用户必须属于部门
- 文档版本必须属于文档
- chunk 必须属于文档版本
- 题目版本必须属于题目
- 试卷选题时，题目版本必须属于对应题目
- 作答时，assignment、试卷、用户必须互相匹配
- 一次作答中的一道题只能有一条答案记录

这种做法的好处是：

- 数据不容易脏
- 很多一致性问题在数据库层直接失败
- 后端逻辑可以更清晰

### 2. 横切能力和多态能力弱约束

以下表或字段没有使用强外键：

- `learning_path_items.ref_id`
- `activity_events.entity_type + entity_id`
- `metric_snapshots.scope_type + scope_id`
- `llm_jobs.input_ref_type + input_ref_id`
- `question_tags.tag_name`
- `exam_paper_tags.tag_name`

原因不是偷懒，而是这些地方天然跨上下文：

- 学习路径要能指向不同资源
- 事件表要能记录多种实体
- 指标快照要能覆盖多种 scope
- 标签需要保持写入简单和查询直接

代价是：

- 完整一致性更多依赖应用层
- 重构和迁移时要更小心

## 二、为什么文档和题目要版本化

### 1. 文档版本化

`knowledge_documents` 和 `knowledge_document_versions` 分拆后，解决了两个问题：

- 同一逻辑文档可以反复更新
- 历史题目来源仍能追溯到当时的文档内容

如果只有一张文档表，直接覆盖全文，会导致：

- 旧题出处失真
- 检索评测失真
- 考后反馈引用失真

### 2. 题目版本化

`questions` 和 `question_versions` 分拆后，解决了：

- 题目可持续编辑
- 老试卷能冻结老版本
- 题目生成元信息、来源元信息有明确落点

如果只有一张题目表，会出现：

- 编辑题目时覆盖旧内容
- 老试卷展示的题和当时不一致

## 三、为什么试卷要冻结题目版本

`exam_paper_questions` 同时保存：

- `question_id`
- `question_version_id`

并使用复合外键保证二者匹配。

这样设计解决了一个很关键的问题：

- 题库是活的
- 考试历史必须是静态快照

所以 TalentHub 的设计不是“考试永远看题库最新内容”，而是“考试看组卷时冻结的版本”。

这也是在线考试系统非常典型的一条建模原则。

## 四、为什么 `question_sources` 重要

`question_sources` 是数据库里最有系统价值的一张桥表之一。

它把：

- 题目版本
- 知识 chunk

连在了一起。

它解决的问题不是“能不能出题”，而是：

- 题目能不能解释来源
- 判卷后能不能给出回看资料
- 知识库检索和题目生成能不能形成闭环
- 后续能不能做 RAG 评测

没有这张表，TalentHub 就更像“黑箱生成题目”；有了它，系统才真正具备知识闭环。

## 五、为什么标签没有强外键到 `tags`

现在：

- `tags` 是标签主表
- `question_tags` 和 `exam_paper_tags` 存的是 `tag_name`

而不是：

- `tag_id`

这么做的优点：

- 写入简单
- 查询直接
- 搜索和筛选路径短
- 前端输入新标签时不需要复杂先建后绑流程

代价：

- 标签一致性依赖应用层同步
- 理论上可能出现主表和挂载表不一致

这是一种偏工程效率的取舍。

## 六、为什么学习路径用多态引用

`learning_path_items.ref_id` 没有强外键，是因为学习路径本质上是“编排器”，它可能引用：

- 题目
- 试卷
- 文档
- 知识点
- 未来的课程、视频、练习等资源

如果这里强绑定某一张表，学习路径的可扩展性会被锁死。

所以它是一个典型的“数据库少一点强约束，换更高扩展性”的位置。

## 七、当前 schema 的几个亮点

### 1. 版本链与考试快照链是分开的

- 文档版本链服务知识演进
- 题目版本链服务题库演进
- 试卷冻结链服务考试历史稳定

这三条链没有混在一起，这是结构清晰的地方。

### 2. RAG 关系落在数据库，而不是只在代码里临时算

`question_sources` 让“知识 -> 题目 -> 反馈”成为可追踪数据关系，而不是一段临时内存逻辑。

### 3. 强约束和弱约束边界比较明确

核心业务链路强约束，多态和横切能力弱约束，这种边界是有意识设计出来的。

## 八、当前 schema 的潜在弱点

### 1. 标签主表和挂载表不是强一致

这是当前最明显的“应用层一致性”点之一。

### 2. 多态引用不受数据库保护

比如：

- `learning_path_items.ref_id`
- `llm_jobs.input_ref_id`

数据库本身无法确认它们一定指向真实存在的目标。

### 3. “最新版本”目前没有显式指针

现在要拿最新文档版本、最新题目版本，通常要通过：

- 最大 `version_no`
- 或按创建时间排序

这很清晰，但查询成本会略高于显式维护 `current_version_id`。

### 4. 状态枚举比当前运行语义更宽

比如 schema 里还有：

- `archived`
- `review`

但应用当前很多地方已经偏向：

- 直接发布
- 直接永久删除

这意味着数据库枚举和当前业务语义之间还有一些历史兼容痕迹。

## 九、未来最值得升级的点

### 1. 标签关系标准化

如果后续标签体系更复杂，可以逐步演进到：

- `question_tags.tag_id`
- `exam_paper_tags.tag_id`

而不是只存 `tag_name`

### 2. 当前版本指针

可以考虑在：

- `knowledge_documents`
- `questions`

上显式维护：

- `current_version_id`

以换取更直接的查询语义。

### 3. 学习路径引用规范化

如果学习路径对象类型越来越多，可以引入：

- 更正式的 item type registry
- 或专门的学习资源表

### 4. 指标与事件模型再标准化

如果后续分析能力继续增强，可以把：

- `activity_events`
- `metric_snapshots`
- `llm_jobs`

做成更明确的事件模型和指标体系，而不只是通用记录表。
