# 学习闭环与专项练习设计

## 目标

TalentHub 不希望停在“知识库 -> 出题 -> 考试 -> 判卷”这一步。真正的培训系统应该把考试结果继续转成下一轮学习动作，因此需要一个独立的学习闭环入口，把以下能力收敛起来：

- 基于已判卷结果识别薄弱标签
- 汇总最近得分不足的题目和逐题反馈
- 给出可以直接回看的知识依据
- 基于薄弱标签生成一套新的专项练习

这也是为什么系统同时保留“我的考试”和“我的学习”两个成员入口：

- `我的考试` 负责考试生命周期：开始、继续作答、查看判卷结果
- `我的学习` 负责考试后的学习动作：看弱点、回看资料、再练一轮

## 解决的问题

在没有学习闭环前，成员虽然能在考试详情里看到逐题反馈和知识引用，但仍然要自己回答几个问题：

- 我最近到底薄弱在哪些领域
- 应该优先回看哪些资料
- 我下一轮练什么最划算

这会导致系统停留在“考试工具”层，而不是“培训平台”层。

第一版学习闭环先聚焦最短可用路径：

1. 读取当前用户已判卷考试
2. 汇总标签维度的得分率，得出强项和弱项
3. 拉取最近得分不足题目的反馈和知识引用
4. 允许用户按弱项标签一键生成专项练习
5. 生成出的专项练习仍然进入现有考试中心，不另造一套作答系统

## 业务边界

当前版本刻意不一次性做完整学习路径编排，而是分两层：

- 第一层：轻量学习闭环
  - 基于现有考试数据直接生成学习视图和专项练习
- 第二层：正式学习路径
  - 以后再把 `learning_paths / user_learning_paths / user_learning_progress` 真正产品化

这样做的原因是：

- 第一层能立刻复用现有题库、考试、判卷和知识引用
- 第二层需要更复杂的内容编排、解锁规则和进度跟踪
- 先把“考后该学什么”做明白，比先造一个大而全的学习路径壳更有价值

## 核心数据来源

### 1. 薄弱标签

薄弱标签来自已判卷考试的逐题得分率：

- 数据表：`exam_attempts / exam_answers / exam_paper_questions / question_tags`
- 计算方式：`final_score / question_score`
- 聚合粒度：`用户 + 标签`

输出结果：

- `strongest_tags`
- `weakest_tags`
- `recommended_tag_names`

这里没有引入新的打分模型，而是复用现有判卷结果，保证口径统一。

### 2. 最近待提升题目

最近待提升题目来自：

- 已判卷
- 有最终分
- 得分低于该题满分

同时拼接：

- `answer_feedback`
  - `feedback / weaknesses / suggestion`
- `question_sources`
  - 找到题目版本绑定的知识 chunk

这样成员在“我的学习”页看到的不是抽象标签，而是：

- 哪道题最近没做好
- 系统怎么评价
- 应该回看哪段资料

### 3. 专项练习候选题

专项练习候选题来自题库的最新已发布题目：

- 先按标签命中筛一批题
- 优先挑选当前用户较少做过的题
- 再按标签命中数和新鲜度排序

这一步仍然复用现有题库，不额外生成新题，避免把学习闭环做成“再跑一次 AI 出题”。

## 接口设计

### `GET /api/v1/learning/me/overview`

作用：

- 返回当前用户学习总览

包含：

- 已判卷考试数
- 强项标签
- 薄弱标签
- 推荐标签
- 最近待提升题目
- 每题绑定的知识引用

这条接口是读模型接口，不直接改写业务数据。

### `POST /api/v1/learning/me/practice-exams`

作用：

- 按当前用户选中的标签，自动生成一套私有专项练习并分配给自己

输入：

- `tag_names`
- `question_count`
- `title` 可选

输出：

- 新建的 `exam_paper_id`
- 自己的 `assignment_id`
- 题目数量
- 使用的标签
- 时间窗

这里的练习本质上还是考试域里的私有试卷，只是创建入口放在学习域。

## 代码结构

第一版学习闭环落在独立的 `learning` 模块中，而不是继续堆在 `exam` 或 `analytics` 模块里：

- `application/queries.py`
  - 负责学习总览读取
- `application/use_cases.py`
  - 负责生成专项练习
- `infrastructure/repositories.py`
  - 负责从考试、题库、反馈表里拼学习视图
- `interfaces/router.py`
  - 提供学习总览和专项练习接口

其中有意保留了一层应用编排：

- 学习总览复用 `exam repository` 的知识引用查询
- 专项练习复用 `exam create paper + assignment` 逻辑

这样既没有把逻辑堆进 router，也避免复制现有考试能力。

## 前端交互

成员侧现在分成两个入口：

- `/my-exams`
- `/my-learning`

`我的学习` 页面包含三块：

1. 学习画像
  - 强项标签
  - 薄弱标签

2. 专项练习生成
  - 选择重点标签
  - 选择题目数量
  - 一键生成私有练习

3. 最近待提升题目
  - 逐题反馈
  - 建议
  - 推荐回看资料

## 为什么不直接在考试详情页里继续堆

原因很简单：

- 考试页应该聚焦“完成一场考试”
- 学习页应该聚焦“接下来该怎么补”

如果把这些动作都继续塞进考试详情页，会带来三个问题：

- 页面职责混乱
- 用户心智不清晰
- 学习动作很难沉淀成独立产品能力

把“我的学习”单独拆出来，本质上是在明确业务边界。

## 当前版本的亮点

- 没有另造一套学习系统，而是复用已有考试与题库能力
- 学习建议不是空泛文案，而是绑定真实知识引用
- 专项练习不是作者功能，而是成员可自助使用的个人练习能力
- 保持了“考试入口”和“学习入口”的页面职责分离

## 当前边界

这版仍然是第一阶段，明确保留了几个边界：

- 还没有正式启用 `learning_paths` 做路径编排
- 还没有错题本聚合页
- 还没有“完成一轮专项练习后自动验证提升”的趋势对比
- 还没有支持成员手动从全量标签库自由选标签

## 后续演进

下一步最自然的升级方向有三条：

1. 错题本
   - 把高频错题、反复出错题单独沉淀

2. 学习路径
   - 把专项练习、推荐资料、考试和知识点编成连续学习路径

3. 提升验证
   - 对比用户在某个标签上的前后得分趋势，真正回答“是否补上来了”

这三步完成后，TalentHub 才算从“考试系统”真正升级成“学习效果改进系统”。
