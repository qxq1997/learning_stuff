# 异步任务与后台 Worker 设计

## 目标

TalentHub 在接入网页抓取、embedding、rerank、AI 出题和 AI 判卷后，最明显的生产风险已经不再是功能缺失，而是这些重操作仍然挂在请求链路上。

如果继续把这些步骤同步放在 HTTP 请求里，会直接带来：

- 请求超时
- 外部依赖抖动被放大到用户体验
- 应用线程被长时间占用
- 出题和判卷失败难以和“接口不可用”区分
- 后续很难继续做重试、限流、优先级和审计

这份文档描述 TalentHub 当前为什么要引入后台 Worker，以及现在的异步任务系统解决了什么问题，还有哪些边界与升级点。

## 设计原则

- 请求链路尽量只做校验、入队、查询状态和返回结果
- 重操作由独立 Worker 执行
- 任务状态必须可查询、可审计、可失败定位
- 失败信息保持业务可读，而不是只暴露基础设施异常
- 在不引入过重消息中间件的前提下，优先选择最小可用实现

## 当前覆盖范围

当前异步任务系统已经覆盖：

- 知识库 RAG 出题预览
- 网页抓取后出题预览
- 通用基础题预览
- 跨知识库检索出题预览
- 考试提交后的 AI 判卷

当前仍然保持同步的部分：

- 知识文档创建与编辑后的在线增量索引
- 题目保存
- 试卷创建与编辑

也就是说，这一轮先优先处理了**最重、最容易超时、最依赖外部模型与外部网页**的链路。

## 核心结构

### 后台任务表

当前任务表是 `background_jobs`。

它的核心字段包括：

- `id`
- `job_type`
- `requested_by_user_id`
- `status`
- `input_json`
- `result_json`
- `error_json`
- `attempt_count`
- `created_at`
- `started_at`
- `finished_at`
- `updated_at`

其中最关键的是：

- `job_type`
  - 用于区分任务执行器，例如出题预览、考试判卷
- `status`
  - `pending / running / succeeded / failed`
- `input_json`
  - Worker 执行任务时的输入
- `result_json`
  - 成功结果，供前端轮询读取
- `error_json`
  - 失败结果，供前端按统一错误结构处理

这样设计的目的，是把“HTTP 请求是否成功返回”与“后台任务是否真正执行完成”解耦。

### Worker 模型

当前 Worker 是一个独立进程，由 Docker Compose 单独拉起：

- API 服务负责入队和状态查询
- Worker 服务负责轮询待处理任务并执行

当前实现采用数据库作为轻量任务队列，并使用 `FOR UPDATE SKIP LOCKED` 做任务争抢，避免同一条任务被多个 Worker 重复处理。

这是一个**数据库任务表 + 独立 Worker**的最小可用方案。它的优点是：

- 依赖少
- 结构简单
- 便于理解和调试
- 与当前模块化单体架构契合

## 当前任务执行流

### 出题预览

1. 前端提交出题请求
2. API 做参数校验和权限校验
3. API 写入 `background_jobs`
4. API 立刻返回 `202 Accepted`
5. 前端轮询任务状态
6. Worker 取到任务并执行：
   - 准备来源
   - 检索知识片段
   - rerank
   - 调用 LLM 生成题目
   - 校验结果
7. Worker 写回 `result_json` 或 `error_json`
8. 前端拿到最终结果并继续后续保存

### AI 判卷

1. 前端发起“提交作答”
2. API 只做鉴权、状态校验和入队
3. 前端轮询任务状态
4. Worker 读取作答、题目版本和考试上下文
5. Worker 调用 AI 判卷
6. Worker 写入 `grading_results` 和 `answer_feedback`
7. Worker 把最终 attempt 详情写入 `result_json`
8. 前端轮询完成后展示结果

## 为什么当前先不用独立 MQ

TalentHub 当前没有直接上 Kafka、RabbitMQ、SQS 或 Celery，不是因为这些方案不好，而是因为当前阶段更重视：

- 可理解性
- 可维护性
- 本地部署成本
- 架构简洁度

当前项目最需要先解决的是：

- 请求链路太重
- AI 调用太慢
- 失败难排查

数据库任务表 + Worker 轮询，已经能明显解决请求超时和链路耦合问题，同时不会把系统复杂度一下抬得太高。

这个选择解决了：

- 请求链路过重的问题
- 没有后台 Worker 的问题
- 任务不可观测的问题

它没有解决的是：

- 高吞吐大规模任务调度
- 任务优先级与延迟队列
- 多阶段工作流编排
- 大规模分布式消费

所以它是**生产化第一步**，不是最终形态。

## 失败处理策略

当前任务系统不会直接把 traceback 原样吐给前端，而是统一序列化成业务错误。

典型错误码包括：

- `confirmation_required`
- `domain_invariant_violation`
- `domain_not_found`
- `external_service_error`
- `request_validation_failed`
- `internal_error`

这样做的意义是：

- 前端可以复用现有错误处理逻辑
- 用户看到的是业务错误而不是 Python 异常
- 日志、轮询结果和前端提示可以对齐

## 为什么前端仍然像“同步调用”

当前前端没有额外做一整套任务管理状态机，而是采用了**透明轮询**：

- API 层负责创建后台任务
- 自动轮询 `/background-jobs/{id}`
- 成功时返回最终业务结果
- 失败时抛出统一 `ApiError`

这样做的好处是：

- 页面层侵入小
- 能快速把系统变稳
- 保持现有交互连续性

也就是说，当前阶段是**后端先异步化，前端先低侵入接入**。

## 当前收益

引入 Worker 后，TalentHub 在这些方面已经更稳：

- 出题和判卷不再长时间占住请求线程
- reranker、网页抓取、模型调用失败时更容易隔离和降级
- 后台任务可审计、可轮询、可排查
- 后续可以继续接重试、限流、告警和任务面板

这一步本质上是把 TalentHub 从“开发期同步原型”推进到“具备后台作业能力的业务系统”。

## 当前边界

当前实现仍然有几个明确边界：

- 任务仍由数据库轮询，不是专门消息队列
- 还没有任务重试策略
- 还没有优先级、延迟执行和取消机制
- 还没有后台任务管理页面
- 知识文档索引尚未全部迁到后台任务

这意味着：

- 当前已经足够支撑小规模正式部署和内测
- 但还不是大规模任务平台

## 后续升级点

### 1. 把知识文档索引迁入后台任务

当前文档创建/编辑后的 embedding 和写索引仍是在线增量索引。后续应改成：

- 文档先写数据库
- 入队“索引重建任务”
- Worker 完成 chunk、embedding、Qdrant 写入

这样网页导入和长文档导入会更稳。

### 2. 增加任务重试与退避

最适合对以下步骤引入：

- 外部 LLM 调用
- reranker 调用
- 网页抓取

### 3. 增加任务面板

管理员可以看到：

- 正在执行的任务
- 最近失败任务
- 失败原因
- 平均耗时

### 4. 增加任务优先级

例如：

- 用户提交考试后的判卷优先级高
- 大批量知识索引优先级低

### 5. 进一步切换到正式 MQ

如果后续任务量明显增长，再考虑：

- Redis queue
- RabbitMQ
- Celery
- 更完整的工作流引擎

## 总结

TalentHub 当前的后台任务系统解决的不是“代码更花哨”问题，而是：

- 把重操作从请求链路中拆出来
- 把同步等待变成后台执行
- 把失败从基础设施异常提升成可理解的业务错误
- 为检索、出题、判卷这些 AI 重操作提供统一 Worker 承载层

这一步让 TalentHub 更接近一个真正可上线、可扩展的 AI 业务系统。
