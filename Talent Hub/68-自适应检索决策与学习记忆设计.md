# 自适应检索决策与学习记忆设计

## 背景

TalentHub 已经具备知识库、RAG 出题、AI 判卷、学习资料推荐和专项练习生成能力，但如果所有链路都固定采用同一种检索策略，系统会出现两个问题：

- 对简单请求过度检索，增加延迟和噪音
- 对复杂学习场景检索不足，无法给出真正有帮助的资料和下一步练习建议

因此，这一轮设计不再把系统做成“无差别固定流程”，而是引入一套**受约束的决策引擎**：

- 规则先兜底
- LLM 后续只补歧义
- 当前第一版先在应用层落地结构化 decision plan

## 设计目标

- 让知识库出题能够显式说明“为什么这样检索”
- 让学习中心能够显式说明“为什么这样推荐资料”
- 把“系统自行判断”变成可审计、可解释、可回放的结构化结果
- 为后续接入更强的 adaptive RAG 和 learning memory 奠定稳定接口

## 核心设计

### 1. 检索决策对象

知识库出题新增 `QuestionGenerationRetrievalPlan`，描述本次 RAG 检索的结构化计划：

- `should_retrieve`
- `scope`
- `strategy`
- `query_texts`
- `source_type_filters`
- `confidence`
- `reason`

当前第一版先用于 `knowledge_library` 来源，核心目标不是做自由 agent，而是让 TalentHub 明确回答：

- 这次是否必须查知识库
- 查整个知识库还是先收缩到网页导入文档
- 用单轮检索还是多轮补检索

### 2. 学习推荐决策对象

学习中心新增 `LearningRecommendationPlan`，描述本次学习建议的整体策略：

- `strategy`
- `recommended_tag_names`
- `expanded_tag_names`
- `confidence`
- `reason`

同时，逐题反馈新增：

- `learning_reference_strategy`
- `learning_reference_limit`
- `learning_reference_reason`

这让系统不仅返回推荐资料，还返回“为什么对这一题扩大或收缩资料范围”。

## 当前第一版规则

### 知识库出题

当前规则优先，不使用自由模型规划。

#### 单轮 / 多轮检索

满足任一条件时，走 `multi_pass`：

- 选择的标签数量大于等于 2
- 用户问题能明显切分为多个子主题

否则走 `single_pass`。

#### 检索范围

若 query 中明显出现网页/站点信号，例如：

- `网页`
- `页面`
- `官网`
- `网站`
- `链接`
- `docs`

则先收缩到 `web_import_only`，并通过 `source_type_filters=("web_import",)` 限制召回结果。

否则默认走 `knowledge_library_all`。

### 学习资料推荐

学习中心当前围绕“弱项标签”和“近期错题重复度”来决策。

#### 整体 recommendation plan

- 若最近错题在同一批弱项标签上重复出现，则 `strategy=expanded`
- 若弱项标签已经较清晰但没有明显重复，则 `strategy=focused`
- 若有效样本不足，则 `strategy=baseline`

#### 逐题资料范围

- 命中重复弱项标签，或题目得分率低于 `40%`
  - `expanded`
  - 扩大资料条数
- 命中当前弱项标签
  - `focused`
  - 使用默认条数
- 否则
  - `baseline`
  - 给轻量资料

## 为什么这样做

### 1. 不把复杂性直接推给 LLM

TalentHub 当前的目标不是构建一个自由 agent，而是构建一个稳定、可解释的培训系统。固定结构化 decision plan 有几个好处：

- 前端可以直接展示“系统为何这样做”
- 日志可以记录策略选择而不是只记录结果
- 后续评测时能对比不同策略的效果
- 当某个策略表现不好时，可以回放具体输入和决策结果

### 2. 先解决最有价值的判断点

当前第一版只覆盖两个最关键场景：

- 知识库 RAG 出题
- 学习中心资料推荐

这两处最直接影响：

- 题目是否够贴知识依据
- 用户考后是否真的知道下一步怎么学

## 与 learning memory 的关系

这次实现还没有把完整的 `learning memory / mastery memory / practice memory` 全量落库，但接口形态已经开始向那条路线收敛。

当前已使用的记忆信号：

- `weakest_tags`
- `recent_incorrect_answers`
- 同一弱项标签在最近错题中的重复出现次数
- 当前题目的得分率

这意味着后续只要继续引入：

- 推荐资料历史
- 专项练习历史
- 某标签近几次练习后的趋势变化

就能把当前 decision plan 继续升级，而不需要推翻接口。

## 日志与可观测性

本次设计的重点之一是：**让系统判断被显式记录**。

当前知识库出题会新增检索计划日志：

- `question_generation_retrieval_plan_selected`
- `question_generation_retrieval_pass_started`
- `question_generation_retrieval_pass_completed`

日志中会带：

- strategy
- scope
- source_type_filters
- confidence
- query count

这使后续调参不再依赖猜测。

## 当前边界

第一版仍然是**规则优先的轻量决策系统**，还没有做到：

- LLM planner 参与结构化决策
- 根据用户历史推荐结果自动调整策略
- 按部门、标签、来源类型做更细粒度的知识库权限过滤
- 多轮检索后的自动 query rewrite
- 学习建议的自动扩大召回范围评测闭环

## 潜在升级点

### 1. DecisionContext

把分散在应用层的输入信号统一收口成标准上下文对象，例如：

- 用户角色
- 部门范围
- query 类型
- rerank 得分分布
- 历史学习趋势

### 2. Policy Engine

把当前散在 use case 里的规则提升成独立 policy engine，支持：

- 策略版本化
- A/B 对比
- 回归评测

### 3. LLM Planner

在规则边界内，让模型输出结构化 plan，例如：

- 是否需要补检索
- 是否扩大召回范围
- 多轮检索如何拆 query

### 4. Learning Memory

逐步补齐三类长期状态：

- `learning memory`
- `mastery memory`
- `practice memory`

让 TalentHub 从“会出题的考试系统”真正升级成“会持续辅导的学习系统”。

## 总结

这次设计解决的不是“系统能不能继续多做一步”，而是“系统做判断时能不能保持可控、可解释和可持续优化”。

TalentHub 当前第一版的答案是：

- 不做自由 agent
- 先做结构化 decision plan
- 先在最有价值的两条链路落地
- 为后续更强的 adaptive RAG 和 learning memory 打基础
