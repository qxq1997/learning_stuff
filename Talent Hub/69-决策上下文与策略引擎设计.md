# 决策上下文与策略引擎设计

## 背景

在 `68-自适应检索决策与学习记忆设计` 中，TalentHub 已经引入了第一版结构化 decision plan，用于解释：

- 为什么知识库出题这样检索
- 为什么学习中心这样推荐资料

但第一版仍然偏“规则函数集合”，还没有把“决策输入”和“决策规则”本身抽象成稳定结构。随着学习闭环、专项练习和更复杂的 adaptive RAG 继续演进，这会带来两个问题：

- 决策逻辑难以扩展和复用
- 后续接入更多学习信号时，规则容易散落在 use case 中

因此，这一轮把系统进一步收敛成：

- `DecisionContext`
- `PolicyEngine`

并首次把 `practice memory` 作为正式信号接入学习推荐。

## 设计目标

- 把“输入信号”和“决策逻辑”解耦
- 让 question / learning 两条链路都具备清晰的 policy 边界
- 把最近专项练习历史纳入学习推荐，避免只看错题标签
- 为后续引入 LLM planner 和更强 learning memory 保留接口

## 核心概念

### 1. DecisionContext

DecisionContext 负责描述“系统这次凭什么做判断”。

当前已落地两个 context：

- `KnowledgeLibraryRetrievalDecisionContext`
- `LearningRecommendationDecisionContext`

它们不是结果，而是输入信号容器。

### 2. PolicyEngine

PolicyEngine 负责在固定上下文下输出结构化决策结果。

当前已落地两个 engine：

- `KnowledgeLibraryRetrievalPolicyEngine`
- `LearningRecommendationPolicyEngine`

这样 TalentHub 的判断逻辑不再分散在 use case 里，而是显式变成可替换策略。

## 知识库出题链路

### DecisionContext

当前知识库出题的 context 包含：

- 用户 query
- 题型
- 难度
- 标签
- query fragments
- source type filters
- base retrieval query

这些信号的作用是：

- 判断 query 是否明显多主题
- 判断是否需要优先收缩到网页导入文档
- 统一生成后续检索问题

### PolicyEngine

当前 `KnowledgeLibraryRetrievalPolicyEngine` 输出：

- `single_pass` 或 `multi_pass`
- `knowledge_library_all` 或 `web_import_only`
- 去重后的 retrieval query 列表
- confidence
- reason

这一层的意义在于：

- rule 先落在 engine
- use case 只消费 plan，不再自己决定策略

## 学习推荐链路

### DecisionContext

当前学习推荐的 context 包含：

- weakest tags
- recent incorrect answers
- practice memory

其中 `practice memory` 是这次新增的关键输入信号。

### practice memory 是什么

当前不是完整的长期学习记忆，而是从已有考试数据中抽出来的“专项练习历史”：

- 某个标签被专项练习覆盖了多少次
- 这些专项练习里有多少次已经完成判卷
- 对应的平均得分率大致如何
- 最近一次分配时间

这部分不需要新建复杂表结构，先复用：

- `exam_assignments`
- `exam_attempts`
- `exam_paper_tags`
- `exam_paper_questions.section_name`

来提取“专项练习”相关信号。

### LearningRecommendationPolicyEngine

当前规则从只看：

- 弱项标签
- 最近错题重复度

升级为同时看：

- 弱项标签重复出错
- 以及“这个弱项是否已经进入专项练习，但提升仍不稳定”

这让系统可以区分两种不同情况：

1. 还没真正练过
   - 先聚焦推荐

2. 已经练过多次但仍不稳定
   - 扩大资料范围
   - 补更多上下文
   - 换学习切入角度

## 为什么 practice memory 很重要

如果只看错题，系统容易陷入一个问题：

- 同一个弱项标签刚练过
- 系统下一轮还是继续推完全一样的资料和题

这样用户会感觉：

- 系统只会重复
- 不会根据“我已经练过这个了”来调整

接入 practice memory 后，TalentHub 开始具备最基础的“训练记忆”：

- 我知道你在哪些标签上已经练过
- 我知道这些练习是否完成过
- 我知道这些练习后的分数是否有改善

这让系统开始更像“教练”，而不是“题目生成器”。

## 关于规则与阈值的现实定位

这一节专门讲清楚一件容易被名字误导的事：当前 `PolicyEngine` 到底是什么，以及里面那些阈值是怎么定出来的。

### 当前 PolicyEngine 的真实形态

老实说，当前 `LearningRecommendationPolicyEngine` 内部就是一串 if-else。它没有：

- 规则 DSL
- 动态规则加载
- 多策略并行调度
- 自动调参

把它叫做 “engine” 容易让人误以为有很重的运行时。当前价值其实只在两件具体的事：

- 它是一个纯函数：输入 `DecisionContext`，输出 `Plan`，不查数据库、不发网络请求，因此可以脱离整套系统单独测试
- 它的每个分支都带 `reason`，系统能回答“我为什么这样推荐”，可解释性是真功能，不是装饰

把这两点说清楚，比强调“引擎”二字更诚实，也更接近 TalentHub 的实际工程现状。

### 当前阈值与背后的业务语义

当前学习推荐链路中出现的阈值集中列在下面，每一个都有可被追问的业务理由，而不是魔法数字：

| 阈值 | 含义 | 业务理由 |
|---|---|---|
| 最近错题 `>= 2` 次 | 算“反复出错” | 1 次可能是手滑；2 次是“试过两次仍不会”的最小可信样本 |
| `assignment_count >= 2` | 练习“给过机会” | 同理，1 次不足以说明练过仍没提升 |
| `avg_score_rate < 75` | 练习“没稳定提升” | 60 及格 + 15 buffer，代表“还没稳定掌握” |
| 单题 `score_rate < 40` | 触发扩召回 | 远低于及格，说明题目对该用户偏陌生，需要补更多上下文 |
| confidence `0.80 ~ 0.95` | 仅作为分支间相对排序 | 当前没有统计含义，属于伪精确，后续考虑降级为 high/medium/low 三档或直接移除 |

这些阈值都还停留在“拍脑袋 + 业务语义”的阶段，没有经过数据校准。这是新项目的合理起点，但必须知道下一步该往哪里走。

### 阈值方法论的四个层级

任何带阈值的系统，规则的成熟度大致会沿着下面四级演进：

1. **拍脑袋 + 业务语义**
   - 每个数字都有一句话能说清来源
   - 不允许出现没有理由的魔法数字
   - 新项目的合理起点

2. **真实数据分布校准**
   - 跑离线分析，看错题次数、分数率在用户群里的分布
   - 把阈值放在有意义的分位点（如 P75）上，而不是凭直觉

3. **评测集反推最优参数**
   - 标注一批“理想推荐”作为 ground truth
   - 对阈值做网格搜索 / 贝叶斯优化，让数据选最优组合
   - 这条路径可以复用 TalentHub 已有的 Ragas + ranx 评测基础设施

4. **A/B 上线验证**
   - 不同阈值分流给不同用户
   - 看真实指标（推荐点击率、相同标签后续答题正确率、留存）决定胜负
   - 最终的真相只有“用户有没有真的学进去”

### TalentHub 当前所处阶段与下一步

- 当前明确处于第 1 级
- 下一步的演进方向是第 2 级：用真实答题数据校准 `>=2`、`<75` 这类阈值是否仍然合理
- 第 3 级的入口已经具备：检索链路已有 Ragas + ranx，思路可以平移到推荐策略
- 第 4 级（A/B）依赖更完整的指标管道和分流能力，属于更后期的工作

把当前阶段写清楚，比假装“规则经过严密论证”更经得起追问，也更符合这份文档的整体口径。

## 当前实现边界

这一轮仍然是第二阶段，不是最终形态。

还没有做：

- 完整的 `learning memory` 持久化事件表
- 推荐资料历史落库
- recommendation outcome tracking
- LLM planner 接管策略选择
- policy 版本管理和 A/B 对比

当前的定位很明确：

- 先把 decision context 和 policy engine 结构稳定下来
- 先把 practice memory 信号接进学习推荐
- 不急着引入新的持久化复杂度

## 潜在升级点

### 1. Learning Memory Event Store

后续可以把这些事件正式持久化：

- 推荐过哪些资料
- 推荐原因
- 是否阅读
- 阅读后是否提升
- 练习后是否提升

### 2. Decision Evaluator

把 decision result 和后续表现关联起来，形成：

- 当前策略效果
- 哪种策略对哪类用户更有效

### 3. LLM Planner

在 `PolicyEngine` 之后，再逐步加入 LLM 输出的结构化 planner，但仍要受 policy 边界约束。

### 4. More Explicit Practice Modes

当前专项练习通过 `section_name='专项练习'` 来识别。后续可以进一步显式建模：

- practice exam type
- learning intent
- targeted remediation metadata

## 总结

第二阶段真正解决的问题不是“规则更多了”，而是：

- 决策逻辑终于有了独立结构
- 练习历史终于开始参与学习推荐
- TalentHub 的学习闭环从“看错题”升级到了“看错题 + 看练习历史”

这一步为后面的 adaptive RAG、LLM planner 和长期学习记忆打下了更稳的基础。
