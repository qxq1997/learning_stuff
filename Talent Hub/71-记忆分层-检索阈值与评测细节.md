# 记忆分层、检索阈值与评测细节

## 目标

前面的设计文档已经解释了 TalentHub 为什么要做自适应检索、策略引擎和 learning memory，但面向真正的系统设计讨论，还需要进一步回答这些更细的问题：

- memory 是不是每个用户都有
- 用户多了之后 memory 会不会失控
- memory 要不要分短期、中期、长期
- 检索里的 `k` 怎么定
- 什么叫“召回率”
- 怎么判断当前检索结果不好，需要扩大范围
- top-k 分数该怎么看
- 当前系统里哪些阈值已经落地，哪些只是后续建议

本篇文档专门回答这些问题。

## 一、TalentHub 的 memory 到底是什么

TalentHub 里的 memory 不是聊天机器人那种“记住你上次说了什么”，而是围绕学习闭环的结构化状态。

当前最重要的三类 memory 是：

1. `mastery memory`
2. `practice memory`
3. `learning memory`

### 1. mastery memory

它回答：

- 这个用户在哪些标签上强
- 哪些标签弱
- 当前整体得分率大致怎样

当前真实实现来自：

- 已判卷考试
- 题目标签
- 每题得分率聚合

也就是说，mastery memory 是**按用户隔离**的。不同用户的 mastery 完全独立。

### 2. practice memory

它回答：

- 这个用户在哪些标签上做过专项练习
- 做了多少次
- 练习后的平均得分率如何
- 最近一次练习是什么时候

当前真实实现同样是**按用户隔离**的，来源于：

- `exam_assignments`
- `exam_attempts`
- `exam_paper_tags`
- `section_name='专项练习'`

### 3. learning memory

它回答：

- 系统给这个用户推荐过哪些资料
- 针对哪道题推荐过
- 最近一次是什么时候
- 推荐过多少次
- 当时用的是什么策略

当前真实实现使用：

- `learning_reference_memories`

它也是**按用户隔离**的。

## 二、是不是每个用户都有 memory

是，但要分清楚“有没有”和“有多少”。

### 当前实现

只要用户有这些行为，就会开始产生对应 memory：

- 参加过考试 -> mastery memory
- 做过专项练习 -> practice memory
- 打开我的学习并触发资料推荐 -> learning memory

如果用户还没有任何学习行为：

- memory 不是空对象，而是“弱信号状态”
- 系统会退回 baseline 策略

这也是为什么 TalentHub 不会一上来假装自己“懂这个用户”，而是明确告诉你：

- 当前有效样本少
- 先给轻量建议

## 三、用户多了之后 memory 会不会过大

会增长，但**当前设计是可控的**，因为 TalentHub 存的不是全文对话，而是结构化学习事件。

### 当前 learning memory 的增长方式

`learning_reference_memories` 的唯一键是：

- `user_id`
- `question_version_id`
- `document_id`
- `source`

这意味着它不是“每推荐一次就新增一行”，而是：

- 同一个用户
- 针对同一道题
- 推荐同一份文档
- 只会更新已有记录的 `delivery_count` 和 `last_recommended_at`

所以增长速度远小于聊天记忆或日志流水。

### 什么情况下会变大

如果系统未来做到：

- 大规模用户
- 高频重复推荐
- 更多学习入口
- 更细粒度的资料项

那这张表仍然会持续增长。

但它的增长是**可预期且可分层治理**的。

## 四、memory 要不要分层

要，而且这是非常重要的设计点。

### 建议的三层模型

#### 1. 短期 memory

关注最近几次行为，强调“刚发生了什么”。

适合存：

- 最近推荐的资料
- 最近错题
- 最近专项练习
- 最近一次检索和推荐决策

用途：

- 避免重复推送
- 判断是否需要扩大范围
- 给用户即时反馈

当前 `learning_reference_memories` 实际上已经承担了短期 memory 的一部分职责。

#### 2. 中期 memory

关注过去几周或一两个月，强调“这个用户最近是否在改善”。

适合存：

- 标签级平均得分率
- 标签级专项练习效果
- 最近几轮练习后的变化趋势

用途：

- 决定哪些标签应继续练
- 决定某个策略是否还有效

当前 `mastery memory + practice memory` 就属于中期 memory 的雏形。

#### 3. 长期 memory

关注更长周期，强调“长期画像和历史沉淀”。

适合存：

- 标签长期掌握度
- 学习路径完成情况
- 历史推荐效果
- 关键能力曲线

用途：

- 个人成长趋势
- 管理者画像
- 岗位能力图谱

当前 TalentHub **还没有真正完整的长期 memory 层**，这是后续演进点。

### 当前项目的结论

现在最合理的做法不是一开始就建一个超复杂 memory 系统，而是：

- 先把短期和中期做扎实
- 再逐步汇总成长期画像

## 五、检索里的 `k` 是怎么定义的

在 TalentHub 当前实现里，至少有三个不同层级的 `k`。

### 1. candidate k

也就是向量检索的候选召回数。

当前配置：

- `knowledge_retrieval_candidate_limit = 18`

它表示：

- 先从 Qdrant 里粗召回 18 个候选 chunk

### 2. final chunk k

也就是最终真正返回给出题或预演的 chunk 数。

当前配置：

- `knowledge_retrieval_chunk_limit = 6`

它表示：

- 最终只把 6 个 chunk 进入下游

### 3. learning reference k

也就是学习中心给每道题推荐几条资料。

当前配置：

- `exam_feedback_learning_reference_limit = 3`

策略上会做动态调整：

- baseline：通常降到 `2`
- focused：保持 `3`
- expanded：提升到 `5`

## 六、为什么当前默认是 18 -> 6

这不是绝对真理，而是当前工程折中。

### 为什么不是直接 top 6

如果直接从向量库只取 top 6：

- reranker 的空间太小
- 多轮检索和混合召回的收益会变弱
- 一旦前几条被噪声占掉，就没有补救空间

### 为什么不是 100 -> 6

太大也不划算：

- reranker 成本更高
- 延迟更高
- 对 TalentHub 当前文档规模来说收益不明显

所以 `18 -> 6` 是一个中等偏保守的取值：

- 足够给 reranker 留空间
- 又不至于把延迟和成本拉得太高

## 七、怎么定义“召回率”

严格来说，召回率不是线上请求里实时凭感觉定义的，而是通过评测集计算。

在 TalentHub 当前评测里，召回相关指标主要包括：

- `Recall@5`
- `MRR@5`
- `NDCG@5`
- `IDBasedContextRecall`
- `IDBasedContextPrecision`

这些都来自：

- `ranx`
- `Ragas`

脚本在：

- [backend/scripts/eval_retrieval_baseline.py](/Users/xinqi/WebstormProjects/TalentHub/backend/scripts/eval_retrieval_baseline.py)

### 召回率的直观理解

- `Recall@5`
  - 参考答案应该命中的 chunk，有多少在前 5 条里被召回了

- `MRR@5`
  - 正确结果排得靠不靠前

- `NDCG@5`
  - 排序质量整体如何

- `IDBasedContextPrecision`
  - 返回的结果里有多少是真正应该返回的

所以：

- 召回率回答“找没找回来”
- precision 回答“找回来的是不是够干净”

## 八、怎么判断当前检索结果不好，需要扩大范围

这里要区分**当前真实实现**和**建议中的下一阶段实现**。

### 当前真实实现

当前还没有上线“按 top-k 分数自动扩召”的在线策略。  
目前是**规则驱动的扩召**：

#### 出题链路

当前会触发 `multi_pass` 的条件是：

- 标签数 `>= 2`
- 或 query fragment 数 `>= 2`

当前会收缩到 `web_import_only` 的条件是：

- query 含明显网页关键词

#### 学习链路

当前会走 `expanded` 的条件是：

- 最近错题在同一弱项标签上重复出现 `>= 2`
- 或 practice memory 判断该弱项“练过但仍不稳”
- 或某道题得分率 `< 40%`

也就是说，当前系统还没有用“分数分布阈值”动态扩召，而是先用**稳定、可解释的启发式规则**。

### 下一阶段建议

如果后续要做更强自适应 RAG，最自然的增强是基于这些在线信号：

1. top1 与 topk 的分差
2. topk 分数是否过于接近
3. top1 本身是否过低
4. rerank 前后排序是否大幅变化
5. 多个子主题是否都没有稳定命中

例如可以使用类似规则：

- 如果 `top1 < 0.35`
  - 认为召回信号偏弱，考虑扩范围
- 如果 `top1 - top5 < 0.05`
  - 认为结果发散，说明缺少清晰主命中
- 如果 reranker 后前 3 条全部来自不同主题
  - 说明当前 query 过宽，考虑拆 query 多轮检索

这些阈值**目前还没有落成线上规则**，但这是 TalentHub 后续最值得做的增强。

## 九、怎么判断 top-k 分数

这里有一个很重要的工程判断：

**top-k 分数不是绝对概率，不能跨 query 机械比较。**

原因：

- 向量相似度分数受 embedding 模型和向量库实现影响
- reranker relevance score 也不一定跨 query 可校准

所以当前更可靠的用法是：

- 在同一个 query 内看相对分布
- 不要跨不同 query 直接比较“0.62 比 0.55 好多少”

### 当前系统怎么用分数

当前主要用法仍然比较保守：

- 记录 top 向量分数
- 记录 top rerank 分数
- 在日志中暴露出来
- 供后续分析和评测使用

当前还没有基于它们做强在线决策门限。

### 为什么当前先不这样做

因为如果没有足够大的评测集和线上经验，直接把分数阈值写死，会非常容易：

- 过度扩召
- 不必要的多轮检索
- 成本上升
- 检索噪音变大

所以当前 TalentHub 的策略是：

- 先记录
- 先评测
- 再逐步把阈值引入线上决策

## 十、当前系统里哪些阈值已经真实落地

### 出题决策

- `len(tags) >= 2` -> `multi_pass`
- `len(query_fragments) >= 2` -> `multi_pass`
- 命中网页关键词 -> `source_type_filters=("web_import",)`

### 学习推荐

- 重复弱项标签出现次数 `>= 2`
- practice stalled:
  - `assignment_count >= 2`
  - 且 `avg_score_rate is None or < 75`
- 逐题低分扩召：
  - `score_rate < 40`

### 默认 k

- `candidate_limit = 18`
- `chunk_limit = 6`
- `learning_reference_limit = 3`
- expanded 学习资料 = `default_limit + 2`
- baseline 学习资料 = `max(1, default_limit - 1)`

这些都是当前代码里真实存在的阈值，不是未来规划。

## 十一、为什么当前不直接做长期 memory

因为长期 memory 最容易过度设计。

如果一开始就做：

- 全量事件存储
- 多层聚合
- 时间衰减
- 长周期画像

会带来两个问题：

1. 复杂度过高
2. 很难验证哪些信息真有用

所以 TalentHub 当前的选择是：

- 先把短期和中期做扎实
- 等这些信号真的被策略用起来，再汇总成长期层

这是一个典型的“问题驱动，而不是预设全能模型”的取舍。

## 十二、这一块在面试时怎么讲

建议这样讲：

“我没有把 TalentHub 的 memory 做成聊天上下文，而是把它拆成 mastery、practice、learning 三类结构化记忆；同时我把检索里的 k 和扩召条件做成明确可解释的工程参数。当前已经落地的是启发式阈值和离线评测，分数分布驱动的动态扩召我刻意留到下一阶段，因为那需要更大的评测集做支撑。” 

## 总结

这一层设计最核心的价值不是参数本身，而是把这些问题从‘凭感觉调’变成了：

- 哪些阈值已经真实落地
- 哪些只是后续建议
- 为什么当前先这样选
- 后续如何用评测把这些阈值继续收敛

这也是 TalentHub 从“能跑”向“可解释、可调优、可面试表达”继续推进的一步。
