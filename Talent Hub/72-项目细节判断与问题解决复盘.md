# 项目细节判断与问题解决复盘

## 目标

TalentHub 不是一开始就长成现在这样。它是在不断暴露问题、重做边界、替换方案的过程中逐步收敛出来的。

这篇文档不再强调“做了什么功能”，而是专门回答：

- 实现过程中遇到了哪些具体问题
- 为什么当时的方案不够好
- 后来怎么改
- 这些改动解决了什么

这类内容对系统设计复盘和面试都非常重要。

## 一、知识获取：静态抓取不够，动态页面抓不到正文

### 问题

最开始如果只用普通 HTML 抓取，很多 JS 渲染页面只能拿到壳 HTML，正文非常短，几乎无法出题。

### 为什么会这样

`requests/httpx + BeautifulSoup` 只能看到服务器返回的初始 HTML，不会执行页面里的 JS。

### 解决方案

后来接入了 `Playwright`：

- 动态网页优先走浏览器渲染
- 如果浏览器依赖不可用，再回退到静态抓取

### 解决了什么

- React/Vue/docsify 这类页面终于能拿到真正正文
- 网页导入的成功率明显提升
- 同时保留了 fallback，不至于因为浏览器依赖问题整条链路直接报废

### 剩余边界

- 动态页面第一次渲染更慢
- 浏览器运行时依赖仍然是生产部署中的一个注意点

## 二、BGE-M3 本地推理太重，冷启动和依赖都拖系统

### 问题

`BGE-M3` 能力强，但本地运行时明显更重：

- 首次下载模型
- Hugging Face 缓存
- 容器镜像体积
- 推理时延

### 为什么这是问题

TalentHub 当前不是专门做搜索引擎，而是一个培训与考试系统。默认路径如果过重，会直接拖慢：

- 文档索引
- 检索预演
- 出题速度

### 解决方案

后来默认 embedding 路线切回：

- `OpenAI text-embedding-3-large`

同时保留：

- `BGE-M3` 可切换能力

### 解决了什么

- 默认链路更轻、更稳
- 保留了后续深入 hybrid 的空间
- 避免把所有用户都绑在本地重模型上

### 核心取舍

这不是说 `BGE-M3` 不好，而是：

- 它更适合作为“深入检索路线”的能力储备
- 不一定适合作为当前默认主路径

## 三、pgvector 和业务数据库绑得太紧，不适合继续演进 hybrid

### 问题

最开始把向量和业务数据都放在 PostgreSQL 里，虽然简单，但一旦明确要走：

- dense
- rerank
- hybrid
- 后续可能的 sparse / multi-vector

就会越来越别扭。

### 解决方案

后来把检索层独立到 `Qdrant`：

- PostgreSQL 继续做业务真相来源
- Qdrant 负责检索索引
- 通过 `chunk_id` 把两边关联起来

### 解决了什么

- 检索系统和业务系统解耦
- 后续接 reranker、调整候选规模更自然
- 数据模型更清晰：结构在 PostgreSQL，索引在 Qdrant

### 新问题

引入了新的同步复杂度：

- 文档新增/更新/删除时要同步更新 Qdrant
- 需要回填脚本和全量重建脚本

## 四、重操作都在请求链路里，接口极易卡死

### 问题

网页抓取、embedding、rerank、AI 出题、AI 判卷，本质上都是重操作。

如果直接塞进 HTTP 请求里，会导致：

- 请求超时
- 用户界面一直卡
- 外部服务抖动时整体体验很差

### 解决方案

后来新增：

- `background_jobs`
- 独立 worker
- 前端轮询

### 解决了什么

- API 只负责入队
- 真正重任务下沉到 worker
- 用户不必一直堵在一次请求里等结果

### 剩余边界

- 当前默认 worker 仍偏轻量
- 还没有做到真正高吞吐的分布式任务平台

## 五、日志一开始太像系统噪音，看不懂业务发生了什么

### 问题

最初只有框架日志和异常堆栈，很难回答：

- 谁触发了什么
- 为什么系统这么判断
- 哪一层慢
- 哪个环节出错

### 解决方案

后来把日志收敛成结构化业务日志：

- 出题流程日志
- 检索与 rerank 日志
- 学习建议日志
- 权限拒绝日志
- 异步任务日志

并且把成功请求级日志整体降到 `DEBUG`，减少刷屏。

### 解决了什么

- 看日志时终于能顺着业务事件理解系统
- 结合 Grafana/Loki 更容易定位问题
- 面试里也更容易讲“可观测性不是口号”

## 六、学习建议容易重复推相同资料

### 问题

如果每次都按相关性取 top-k，用户会反复看到同一批知识文档。

### 为什么这是问题

这会让系统看起来像：

- 只会重复
- 不知道用户已经看过什么

### 解决方案

新增：

- `learning_reference_memories`

记录：

- 用户
- 题目版本
- 文档
- 推荐次数
- 最近推荐时间

推荐时优先避开最近推过的文档，不够再 fallback。

### 解决了什么

- 推荐开始具备“短期记忆”
- 用户体验更像持续辅导，而不是重复刷相同资料

## 七、专项练习做了，但画像里看不出有没有效果

### 问题

以前系统只知道：

- 某个标签弱

但不知道：

- 为这个标签做过几次专项练习
- 练了以后有没有改善

### 解决方案

后来把专项练习效果回流到 mastery 画像中，新增：

- `practice_assignment_count`
- `practice_avg_score_rate`
- `practice_delta_rate`
- `practice_signal`

### 解决了什么

- 用户能看到“练习后提升/稳定/仍波动”
- 系统决策也能利用这些信号
- 学习闭环开始有“反馈后的反馈”

## 八、最初很多判断藏在 use case 里，后续难扩展

### 问题

如果检索和学习推荐的判断逻辑都散在 use case 中，后面会越来越难：

- 扩展
- 调参
- 评测
- 替换策略

### 解决方案

后来引入：

- `DecisionContext`
- `PolicyEngine`

### 解决了什么

- 把输入信号和决策逻辑分开
- 把“系统自行判断”变成结构化策略输出
- 为未来接 LLM planner 留出位置

## 九、当前还没有完全解决的问题

这些不是忘了做，而是有意分阶段处理：

- 还没有基于 top-k 分数分布自动扩召
- 还没有完整长期 memory
- 还没有资料已读状态
- 还没有 recommendation outcome 的因果追踪
- 代码题还没有执行沙箱

当前策略是：

- 先把结构打稳
- 先有评测
- 再把更复杂的自适应逻辑逐步接进来

## 十、这个复盘在面试里的价值

很多项目只讲“用了什么技术”，但更能拉开差距的是：

- 你为什么改方案
- 原方案哪里不够
- 新方案解决了什么
- 又引入了什么代价

TalentHub 现在的价值，不只是功能多，而是它已经积累了一批真实的工程问题与取舍。

## 总结

TalentHub 不是一开始就设计得很完美，而是在真实问题驱动下不断收敛：

- 抓取不稳，就接浏览器渲染和 fallback
- 检索过重，就调整默认 embedding 路线
- 向量层不适合演进，就切到 Qdrant
- 重操作阻塞请求，就引入异步 worker
- 推荐总是重复，就加推荐记忆
- 练习效果不可见，就把练习反馈回流到 mastery

这些问题与解决方式，正是这个项目最有工程含量的部分之一。
