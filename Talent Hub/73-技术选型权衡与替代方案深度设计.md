# 技术选型权衡与替代方案深度设计

## 目标

TalentHub 目前已经有一套能跑通业务闭环的技术栈，但“能跑通”不等于“选型已经没有争议”。这篇文档不再只回答“用了什么”，而是重点回答：

- 为什么选它
- 它解决了什么问题
- 它引入了什么成本
- 为什么没有选另一个方案
- 什么时候应该继续坚持当前方案
- 什么时候应该考虑切换

这类 trade-off 不是附属说明，而是系统设计里最能体现判断力的部分。

## 一、总的选型原则

TalentHub 不是一个追求极致吞吐的通用平台，也不是一个为了展示炫技而堆满 AI 组件的 demo。它当前的主要目标是：

- 做出完整、可解释、可扩展的企业培训闭环
- 保持本地开发和学习成本可控
- 让每一次引入的新技术，都能服务真实业务能力
- 尽量把复杂度放在“值得复杂”的地方

因此，TalentHub 的选型原则可以归纳成五条：

1. 先让业务闭环成立，再让底层技术演进
2. 优先选择能解释清楚的方案，而不是最时髦的方案
3. 默认方案优先“稳和轻”，高级方案保留为可切换能力
4. 核心业务数据和检索/AI 基础设施要适度解耦
5. 所有复杂方案都要有退出机制和降级路径

## 二、整体架构：模块化单体，而不是微服务

### 当前选择

- `DDD + 模块化单体`

### 解决了什么问题

- 在从 `0 -> 1` 的阶段保持统一事务边界
- 避免过早拆服务带来的联调、运维和数据一致性复杂度
- 更容易围绕领域模型而不是网络边界组织代码
- 让知识库、题库、考试、学习闭环能快速串起来

### 为什么不是微服务

微服务的优势在于：

- 团队并行开发
- 独立部署
- 流量热点单独扩容

但 TalentHub 当前并不满足这些前提：

- 团队规模并不大
- 业务边界还在持续演进
- 很多操作需要跨模块一致性
- 如果一开始拆服务，会把大量精力花在：
  - 接口契约
  - 分布式事务
  - 调用链排错
  - 配置与部署

### 当前代价

- 模块之间仍在一个进程里，天然隔离不如服务级边界强
- 某些模块如果流量单独膨胀，会被单体部署形态限制

### 未来什么时候该拆

当出现下面情况时，才值得认真考虑拆服务：

- 检索与索引流量远大于业务 API
- 判卷和出题任务已形成单独计算集群需求
- 团队开始按领域分团队并独立发布

当前结论：

- 模块化单体是有意选择，不是“还没来得及拆微服务”

## 三、后端：FastAPI + Python，而不是 Java / Spring Boot

### 当前选择

- `Python + FastAPI`

### 解决了什么问题

- AI SDK、embedding、reranker、评测工具链天然在 Python 生态里更成熟
- 出题、判卷、网页抓取、评测脚本、数据处理可以共享一套语言和依赖
- `Pydantic` 让接口输入输出和类型约束更自然
- 对个人学习和快速迭代非常友好

### 为什么不是 Spring Boot

Spring Boot 的优势很强：

- 企业级工程化成熟
- 依赖注入、事务、测试、治理生态完整
- 适合大型中后台和团队协作

但 TalentHub 当前核心难点不是传统业务 CRUD，而是：

- AI provider 接入
- 检索与评测
- 异步任务与 worker
- 网页抓取与数据加工

在这条主线上，Python 的综合开发效率更高。

### 为什么不是 Django

Django 的优势：

- 全家桶
- Admin 后台
- ORM 和权限体系成熟

但它的问题也明显：

- TalentHub 当前以 API 驱动为主
- 需要更清晰地拆 `domain / application / infrastructure / interfaces`
- Django 容易诱导系统朝“模型驱动 CRUD”而不是“领域驱动用例”收敛

### 当前代价

- Python 性能上限不如 Java/Go
- 高并发 CPU 密集任务不适合直接堆在 API 进程里
- 类型系统和大规模代码约束力弱于 Java

### 未来升级点

- 继续把重操作下沉到 worker
- 更明确的服务层与 query 层
- 如果未来有超强并发诉求，可以把部分能力拆成专门服务，而不是整体迁移语言

## 四、前端：React + TypeScript + Vite + Ant Design，而不是 Next.js / Vue

### 当前选择

- `React + TypeScript + Vite + Ant Design`

### 解决了什么问题

- 工作台式后台交互成熟
- 表单、表格、分页、弹窗、树形、筛选器能快速搭建
- TypeScript 让前后端 API 类型更稳定
- Vite 开发迭代快

### 为什么不是 Next.js

Next.js 更适合：

- SEO
- SSR / SSG
- 面向公网内容站点

TalentHub 当前更像：

- 后台工作台
- 作者工作区
- 成员考试与学习中心

它并不依赖首屏 SEO 和复杂 SSR。

### 为什么不是 Vue

Vue 完全可以做成这个系统，但当前选择 React 的原因主要是：

- 生态成熟
- 后台场景组件和示例多
- 便于未来继续细拆组件与 hooks 层

这更多是团队和项目语境选择，而不是绝对技术高低。

### 当前代价

- Ant Design 容易把页面带成“标准后台味”
- React 页面如果不收口，容易变成超大组件

### 未来升级点

- 继续按领域拆组件
- 把工作台逻辑从页面层剥离
- 逐步减少过重的单页面职责

## 五、主数据库：PostgreSQL，而不是 MySQL

### 当前选择

- `PostgreSQL`

### 解决了什么问题

- 复杂关系模型表达能力强
- JSON / 复杂索引 / 全文检索支持好
- 适合用户、部门、题目版本、试卷冻结、作答、判卷、学习进度这类强关系数据

### 为什么不是 MySQL

MySQL 当然也能做业务库，但 TalentHub 更看重：

- 复杂查询与分析扩展性
- 更灵活的 JSON 和索引能力
- 和 AI / 检索周边生态的适配经验

### 当前代价

- 如果团队对 MySQL 更熟，PostgreSQL 的运维和 SQL 写法会有学习成本

### 未来升级点

- 对高频分析报表引入 snapshot / materialized view 思路
- 对 learning/analytics 做更明确的读写分层

## 六、检索引擎：Qdrant，而不是继续用 pgvector

### 当前选择

- `Qdrant`

### 解决了什么问题

- 把检索从业务数据库中独立出来
- 更适合：
  - dense retrieval
  - hybrid retrieval
  - payload filtering
  - rerank 前候选召回
- 为后续多阶段检索留出空间

### 为什么不是继续用 pgvector

`pgvector` 的优点很明显：

- 最省事
- 跟 PostgreSQL 同库
- 原型很快

它适合：

- 轻量 dense 检索
- 检索不是主能力的业务系统

但 TalentHub 的检索已经不是“顺手加一个向量列”了，而是在承担：

- 跨知识库召回
- 出题依据
- 学习资料推荐
- reranker 之前的候选召回

这时继续把所有检索逻辑塞在 PostgreSQL 里，会让：

- 检索演进越来越别扭
- 业务表和检索表耦合过重
- 后续 hybrid / 多路召回路线难做

### 为什么不是 Milvus / Weaviate / Pinecone

#### Milvus

优点：

- 更偏专业向量检索基础设施
- 扩展性和规模上限高

缺点：

- 对 TalentHub 这种本地优先、学习导向项目来说偏重
- 单机开发和整体运维复杂度高于 Qdrant

#### Weaviate

优点：

- 一体化能力强
- AI-native 数据库心智清晰

缺点：

- 概念层更重
- 当前项目并不需要那么重的平台化抽象

#### Pinecone

优点：

- 托管省心

缺点：

- 不符合 TalentHub 的本地优先、开源优先路线
- 不利于学习“自建检索系统”

### 当前代价

- 文档新增、更新、删除时要和 Qdrant 同步
- 需要回填和全量重建索引脚本
- 引入了新的运行基础设施

### 当前结论

- `pgvector` 是很好的起点
- `Qdrant` 更适合 TalentHub 当前阶段及后续演进

## 七、Embedding：为什么默认 OpenAI，同时保留 BGE-M3

### 当前默认选择

- `OpenAI text-embedding-3-large`

### 同时保留的高级路线

- `BGE-M3`

### 为什么默认不是 BGE-M3

`BGE-M3` 的优势很强：

- 开源
- 多语言
- dense / sparse / multi-vector 路线统一
- 很适合长期 hybrid 检索

但它当前的代价也很现实：

- 本地运行时更重
- 首次模型下载和加载更慢
- 容器和开发环境依赖更复杂
- 默认链路会明显拖慢开发体验

TalentHub 当前不是专门的搜索产品，而是一个业务闭环系统，所以默认路径优先考虑：

- 轻
- 稳
- 易运行

于是默认 embedding 切成：

- `OpenAI text-embedding-3-large`

### 为什么不是 Gemini / Voyage 作为当前默认

#### Gemini embedding

优点：

- retrieval task type 很友好
- 中英混合效果较强

缺点：

- TalentHub 当前 OpenAI 使用面已经更广
- 切到 Gemini 不能明显降低整体系统复杂度

#### Voyage embedding

优点：

- retrieval 导向清晰
- 价格和检索体验不错

缺点：

- 当前项目默认 LLM 主线并不在 Voyage
- 新增 provider 的整体收益暂时不如先把 reranker 接好

### 当前结论

- 默认 `OpenAI` 是现实选择
- 保留 `BGE-M3` 是为了后续更深 hybrid，不是历史遗留

## 八、Reranker：为什么默认不用本地 BGE，而是更偏线上 Voyage

### 当前选择

- 默认 `disabled`
- 推荐线上：`Voyage rerank-2.5-lite`
- 本地可切换：`bge-reranker-v2-m3`

### 解决了什么问题

reranker 负责的是：

- 候选召回后的精排

它不是 sparse 的替代品，也不是 embedding 的替代品，而是：

- 先召回
- 再重排

### 为什么不默认开本地 BGE reranker

原因很现实：

- 本地 reranker 首次冷启动更重
- 开发环境启动和检索路径都会变慢
- 当前阶段默认体验不够轻

### 为什么选 Voyage

- 线上调用更轻
- 不需要本地模型运行时
- 多语言支持和长文本能力适合 TalentHub
- 与当前 `OpenAI embedding + Qdrant` 组合很自然

### 当前代价

- 对外部服务有依赖
- 成本需要控制
- 网络和配额抖动需要 fallback

### 当前结论

- reranker 对质量有帮助
- 但默认链路不一定必须强开
- TalentHub 当前更偏“可切换能力”，而不是“一刀切默认开启”

## 九、Chunking：为什么做父子 chunk，而不是简单平铺切块

### 当前选择

- 父子 chunk

### 解决了什么问题

- 检索需要细粒度 child chunk 命中具体知识点
- 出题和学习建议又需要 parent chunk 提供更完整上下文

### 为什么不是简单 flat chunk

flat chunk 的问题是：

- 切小了，上下文不足
- 切大了，检索不准
- 展示和引用时缺少稳定结构归属

父子结构本质上是在平衡：

- 检索精度
- 上下文完整度
- 业务可解释性

### 当前代价

- 索引构建更复杂
- 回填和重建逻辑更重

## 十、异步任务：为什么先做数据库任务表 + worker，而不是 Celery / MQ

### 当前选择

- `background_jobs + PostgreSQL + 独立 worker`

### 解决了什么问题

- 网页抓取、出题、判卷这些重操作不再阻塞请求链路
- 前端可以轮询任务结果
- 先把生产化最危险的同步重操作拆下来

### 为什么不是一开始就上 Celery / RabbitMQ / Kafka

这些方案的优势很强，但当前会引入额外复杂度：

- 新的消息基础设施
- 任务确认、重试、死信、幂等治理
- 部署和运维复杂度

TalentHub 当前阶段首先要解决的是：

- 同步链路太重

而不是：

- 构建一个超强分布式任务平台

### 当前代价

- 当前默认 worker 还偏轻量
- 吞吐上限不如真正 MQ 体系

### 当前结论

- 数据库任务表是合理的第一阶段
- 如果任务量继续上升，再演进到 MQ 才有意义

## 十一、缓存：为什么只做选择性缓存，而不是一开始大面积缓存

### 当前选择

- 只缓存可重算读结果

目前典型包括：

- 标签列表
- 网页抓取结果

### 解决了什么问题

- 避免把缓存变成新的事实来源
- 避免过早进入复杂一致性问题

### 为什么不是全站缓存

因为 TalentHub 当前的核心复杂度不在“每个页面都慢”，而在：

- 出题
- 判卷
- 检索
- 重操作异步化

如果一开始就全站缓存，会把注意力从真正瓶颈上移开。

### 当前结论

- 缓存是加速器
- 不是系统正确性的组成部分

## 十二、观测：为什么选 Loki + Grafana，而不是 ELK

### 当前选择

- `Loki + Promtail + Grafana`

### 解决了什么问题

- 本地日志采集和查询
- 让业务语义日志有可视化入口
- 支撑本地研发和面试演示

### 为什么不是 ELK

ELK 很强，但对 TalentHub 当前阶段来说：

- 更重
- 配置和资源成本更高
- 并不能显著提高当前的主要收益

### 当前结论

- TalentHub 现在更需要“清晰的业务日志语义”
- 而不是最重的日志平台

## 十三、认证：为什么先做自有登录体系，而不是先接 OAuth/SSO

### 当前选择

- 自有账号体系
- Bearer access token

### 解决了什么问题

- 能快速把部门、角色、权限边界跑通
- 适合本地和学习环境

### 为什么不是直接 OAuth / 企业 SSO

因为 TalentHub 当前阶段更需要先验证：

- 权限模型是否合理
- 组织隔离是否合理
- 产品闭环是否成立

如果一开始就接企业级统一身份平台，复杂度会被过早拉高。

### 当前代价

- 会话治理还不完整
- refresh token / 密码重置 / 邀请流程后续仍要补

## 十四、删除语义：为什么题目、考试、知识文档都改成永久删除

### 当前选择

- 永久删除优先

### 为什么不是继续归档

归档的优点是：

- 历史可恢复
- 更安全

但 TalentHub 当前实际场景里，归档引入了明显问题：

- 前端列表容易混入历史状态
- 用户心智是“删了就应该没了”
- 维护 archived 兼容语义增加复杂度

### 当前结论

- 在当前项目阶段，永久删除更符合产品心智
- 如果未来真有强审计诉求，再引入更正式的回收站或审计删除机制

## 十五、文档：为什么要把设计文档和面试文档分开

### 当前选择

- `docs/` 保存正式设计文档
- `interview-docs/` 保存项目表达与面试导向文档

### 解决了什么问题

- 设计文档不再被“怎么面试讲”污染
- 面试文档可以更强调：
  - trade-off
  - 踩坑
  - 取舍
  - 表达方式

### 当前结论

- 这是为了让系统设计和项目表达各自保持清晰，不是重复劳动

## 十六、最重要的 trade-off 总结

TalentHub 的技术选型并不是“哪个更强就选哪个”，而是不断回答下面这些问题：

- 当前主要矛盾是什么
- 这个技术是不是恰好解决当前主要矛盾
- 它带来的新复杂度值不值得
- 如果以后系统阶段变化，能不能平滑切换

因此，TalentHub 的很多核心选择都遵循同一条主线：

- 架构先保持模块化单体，不急着微服务
- 默认技术先选更轻更稳的路线
- 高阶技术保留可切换和可演进能力
- 复杂度优先放在：
  - 检索
  - 学习闭环
  - 业务可解释性

而不是一开始就把整套系统堆成“技术展示架”。
