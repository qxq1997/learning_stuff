# 基于现有知识库的检索对照测试报告

## 目标

这份报告回答的是一个比“系统能不能跑”更实际的问题：

- 如果不再引入 synthetic 文档，只用 TalentHub 当前知识库里已经存在的真实文档，系统的检索能力到底如何？
- 在当前默认链路下，`dense-only` 和 `dense + reranker` 的差异到底有多大？
- 长文档 chunking 到底带来了什么收益，又带来了什么精度损失？
- 当前最容易失败的场景是什么，失败根因在哪里？

## 一、测试上下文

### 1. 当前运行链路

本次评测使用的是当前默认运行链路：

- `OpenAI text-embedding-3-large`
- `Qdrant dense retrieval`
- `Voyage rerank-2.5-lite`

但为了看清 reranker 的真实收益，本次做了两组对照：

1. `dense-only`
2. `dense + Voyage rerank-2.5-lite`

两组评测共享同一组 benchmark、同一组知识文档和同一组检索参数：

- `retrieval_candidate_limit = 18`
- `retrieval_chunk_limit = 6`

### 2. 当前没有 BM25 基线

这点需要先说清楚：

- 当前默认链路**没有**经典 `BM25` 或 PostgreSQL FTS 参与
- 当前默认也**不是** `BGE-M3 sparse/hybrid`

所以这份报告里的对照不是：

- `dense vs dense+BM25`

而是：

- `dense-only vs dense+reranker`

换句话说，这份报告回答的是：

- 当前默认生产链路在真实知识库上的检索能力如何
- reranker 是否真的有帮助

但它**还不能**回答：

- 稀疏召回是否能系统性补足 dense 的短板

## 二、语料范围

### 1. 当前知识库概况

评测时，知识库中共有 `13` 篇 `indexed` 文档。

这次 benchmark 覆盖了其中 `12` 篇，覆盖比例已经足够代表当前知识库主流形态。没有纳入 benchmark 的主要是：

- `Software Engineer, Machine Learning & AI`

它更像一篇职位描述，而不是当前 RAG 出题和学习推荐的主力知识源。

### 2. 纳入 benchmark 的文档

本次 benchmark 覆盖的文档包括：

- `NASA`
- `MacBook Neo`
- `Pied butcherbird`
- `Postgres.app`
- `第一章 初识智能体`
- `第二章 智能体发展史`
- `第四章 智能体经典范式构建`
- `第五章 基于低代码平台的智能体搭建`
- `大模型语言基础(Agent开发)`
- `梁思成`
- `独乐寺`
- `托马斯·R·马歇尔`

### 3. 代表性文档规模

这组真实语料的一个重要特点是：**文档长度差异非常大**。

| 文档 | 字符数 | 父块数 | 子块数 |
| --- | ---: | ---: | ---: |
| `NASA` | `185699` | `88` | `350` |
| `大模型语言基础(Agent开发)` | `42503` | `20` | `79` |
| `第四章 智能体经典范式构建` | `38046` | `18` | `71` |
| `第五章 基于低代码平台的智能体搭建` | `27235` | `13` | `51` |
| `Postgres.app` | `9103` | `5` | `17` |

这里有两个直接影响检索结果的事实：

1. `NASA` 是典型的长维基页面，前部有大量 infobox、列表、导航和历史沿革混杂内容。
2. AI 课程文档虽然没有 NASA 那么长，但章节里混有代码、示例、模板、FAQ、章节总结等多种内容形态。

这意味着这次 benchmark 不只是测“能不能找对”，也在真实测：

- 父子 chunk 是否把长文档切得够稳
- dense retrieval 是否会被近邻噪声和相邻 chunk 干扰
- reranker 能否在同一篇长文内部做更细的精排

## 三、Benchmark 是怎么构建的

### 1. 不是重新上传新文档

这次 benchmark **没有**再往 repo 里放新的长文，也没有再人为灌入一套新的 synthetic 文档。

做法是：

- 直接引用当前知识库中已有的文档标题
- 为每个 query 标注：
  - `expected_document_title`
  - `expected_anchor`

也就是说，这是一套“**基于现有知识库真实语料的 gold 标注 benchmark**”。

### 2. 不是让 LLM 自己判自己

这里没有让 LLM 做 judge。

评测流程是：

1. 用 `expected_document_title` 找到真实文档
2. 在当前最新版本的真实 chunk 中，用 `expected_anchor` 找到 gold chunk
3. 对 query 跑真实检索链路
4. 把检索结果返回的真实 `chunk_id` 和 gold `chunk_id` 对比
5. 用 `ranx + Ragas ID-based metrics` 算指标

所以这次评测测到的是：

- 真文档
- 真 chunk
- 真 embedding
- 真 Qdrant
- 真 reranker

而不是 mock 结果。

### 3. 为什么这轮要补 anchor 归一化

在第一次跑这组 benchmark 时，评测并不是直接成功的。

第一个真实问题就是：

- 网页导入后的文本里存在不换行空格、全半角差异、特殊 Unicode 间距
- 直接用原始 anchor 字符串做精确包含匹配，会导致 gold chunk 定位失败

因此这轮我把评测脚本补成了：

- 先做 `NFKC` 归一化
- 再压缩连续空白
- 再做 anchor 匹配

这不是“评测技巧”，而是长网页/维基内容进入 RAG 后非常真实的工程问题。

## 五、全局结果

### 1. dense-only

| 指标 | 结果 |
| --- | ---: |
| `MRR@5` | `0.664394` |
| `NDCG@5` | `0.469715` |
| `Recall@5` | `0.454545` |
| `IDBasedContextPrecision` | `0.333333` |
| `IDBasedContextRecall` | `0.500000` |

### 2. dense + reranker

| 指标 | 结果 |
| --- | ---: |
| `MRR@5` | `0.827273` |
| `NDCG@5` | `0.702537` |
| `Recall@5` | `0.659091` |
| `IDBasedContextPrecision` | `0.454545` |
| `IDBasedContextRecall` | `0.670455` |

### 3. 直接差分

| 指标 | 提升 |
| --- | ---: |
| `MRR@5` | `+0.162879` |
| `NDCG@5` | `+0.232822` |
| `Recall@5` | `+0.204546` |
| `IDBasedContextPrecision` | `+0.121212` |
| `IDBasedContextRecall` | `+0.170455` |

**结论：**

- reranker 对当前真实知识库语料是有明显收益的
- 收益不是“锦上添花”，而是实打实把一批 dense-only 失败或后排命中的 case 拉回来了

## 六、按场景看差异

| 场景 | dense `MRR@5` | rerank `MRR@5` | dense `Recall@5` | rerank `Recall@5` | dense `Hit@1` | rerank `Hit@1` |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `definition` | `1.000000` | `1.000000` | `0.562500` | `0.750000` | `1.000000` | `1.000000` |
| `entity_fact` | `0.755556` | `1.000000` | `0.500000` | `0.833333` | `0.666667` | `1.000000` |
| `long_document_noise` | `0.666667` | `0.666667` | `0.416667` | `0.416667` | `0.666667` | `0.666667` |
| `numeric_fact` | `0.395833` | `1.000000` | `0.437500` | `0.812500` | `0.250000` | `1.000000` |
| `term_and_alias` | `0.500000` | `0.440000` | `0.350000` | `0.400000` | `0.400000` | `0.400000` |

这里最值得注意的不是平均分，而是结构性差异：

### 1. reranker 对 `numeric_fact` 和 `entity_fact` 非常有效

说明在真实长文里：

- dense 先把大概相关的块找出来
- reranker 很擅长把“真正回答问题的那一块”顶上去

### 2. reranker 对 `long_document_noise` 没有系统性改善

这很关键。

说明：

- 当前 hardest case 不是“相关块排序略差”
- 而是“同一篇长文里，多个近邻 chunk 都很像，甚至 gold 标注和模型最想选的 chunk 不完全一致”

### 3. `term_and_alias` 仍然是最弱的场景

这说明：

- 术语名
- 中英混合表达
- 范式别名
- 标题 / 正文 / 示例之间的跨段匹配

依然是当前系统的短板。

## 七、按领域和语言看差异

### 1. 语言维度

| 语言 | dense `MRR@5` | rerank `MRR@5` | dense `Recall@5` | rerank `Recall@5` | dense `Hit@1` | rerank `Hit@1` |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `en` | `0.800000` | `1.000000` | `0.625000` | `0.812500` | `0.750000` | `1.000000` |
| `zh` | `0.866667` | `1.000000` | `0.500000` | `0.775000` | `0.800000` | `1.000000` |
| `zh-en` | `0.343750` | `0.525000` | `0.312500` | `0.437500` | `0.250000` | `0.500000` |

**结论：**

- 纯中文和纯英文都已经明显可用
- 中英混合 query 仍然是当前最脆弱的一档

### 2. 领域维度

几类最明显的现象：

- `history_culture`：reranker 带来了最明显提升
  - `MRR@5: 0.555556 -> 1.000000`
- `software_tooling`：`Postgres.app` 这种短文档、强事实语句受益非常明显
  - `MRR@5: 0.250000 -> 1.000000`
- `space_history`：改进明显，但还没过线
  - `MRR@5: 0.066667 -> 0.666667`
  - 仍然不是稳定可用态

这正好说明：

- 短文档 + 明确事实：现在已经比较稳
- 超长维基页面 + 多段近似事实：当前仍然很容易互相干扰

## 八、代表性改进案例

这轮最有价值的不是总分，而是这些具体 case 的变化。

### 1. `nasa-budget-2026`

- dense-only：`rank=None`
- rerank：`rank=1`

这是一个非常典型的“dense 召回到相关主题，但没有把真正的数值块顶上来”的例子。

reranker 把：

- `NASA 2026 财年预算是多少`

对应到：

- `Annual budget US$24.4 billion (2026)`

说明 reranker 对“明确问数值”的问题很有效。

### 2. `nasa-headquarters-name`

- dense-only：`rank=5`
- rerank：`rank=1`

说明 dense-only 能找到 NASA 相关块，但容易被其他同主题 chunk 压住；reranker 能把真正命中的“总部大楼名称”块提前。

### 3. `postgres-default-port`

- dense-only：`rank=4`
- rerank：`rank=1`

这是短技术文档里的典型 hard case。  
dense-only 会被：

- 连接字符串
- 代码片段
- 其他端口相关上下文

干扰；reranker 则把真正的默认配置块拉回了 top-1。

### 4. `dule-temple-year`

- dense-only：`rank=3`
- rerank：`rank=1`

这个 case 很能说明中文长历史文的真实情况：

- 文档里有多个“统和二年（984年）”附近的表述
- dense-only 会被相邻历史叙述和后续考证段落干扰
- reranker 能把更直接回答问题的那块前推

### 5. `marshall-president`

- dense-only：`rank=3`
- rerank：`rank=1`

这说明人物传记这种长条目里，reranker 对“人物—职位—总统”这类结构化事实很有帮助。

## 九、顽固失败和反例

### 1. `nasa-preceding-agency` 仍然失败

- dense-only：失败
- rerank：仍失败

这是当前最值得重视的失败例子。

问题不是找不到 NASA，而是：

- dense 和 reranker 都更偏向“NASA 成立于 1958”“National Aeronautics and Space Act”这一组强相关块
- 但 gold anchor 对应的是更早的 infobox/前身机构信息：
  - `National Advisory Committee for Aeronautics (1915–1958)`

这说明当前链路对：

- 超长维基页前部 infobox
- “前身 / predecessor / preceding agency”这类半结构化字段

还不够稳。

### 2. `react-paradigm` 仍然失败

### 3. `reflection-paradigm` 仍然失败

这两个 case 说明当前课程型长文还有一个问题：

- 范式名虽然出现了
- 但正文附近混有代码、模板、示例、总结、练习题
- 检索很容易把“提到 ReAct/Reflection 的旁边内容”排上来，却不一定是定义性句子

这是典型的：

- 章节文档结构复杂
- 定义、案例、代码、练习混在一起
- dense 和 reranker 都还不够“定义敏感”

### 4. `macbook-neo-chip` 被 reranker 反向拉坏

- dense-only：`rank=2`
- rerank：`rank=5`

这说明 reranker 不是总会更好。

当前这个 query 在 `MacBook Neo` 页面里有多个相邻强候选：

- A-series 芯片
- 性能测试
- 硬件介绍
- 发布时间

reranker 把几个非常相近的技术描述排得更近了，但没有把真正最完整回答问题的那一块排到最高。

这属于：

- 候选文本之间差异太小
- 术语相关但语义边界不够清晰

因此 reranker 发生了误排。

## 十、Chunk 带来的收益和精度损失

这次最值得看的不是“chunking 好不好”，而是**它到底在哪些地方帮了我们，在哪些地方也制造了损失**。

### 1. 收益

父子 chunk 让我们能做到：

- 用 child chunk 做细粒度召回
- 用 parent chunk 保留上下文
- 在长文档里不必把整篇都喂给 embedding

对于：

- `独乐寺`
- `梁思成`
- `托马斯·R·马歇尔`
- `Postgres.app`

这类文档，父子 chunk 基本是正收益。

### 2. 损失

但 chunking 也带来了精度损失，主要体现在三类场景。

#### 第一类：infobox / 列表 / 导航污染

`NASA` 最典型。

问题不是文档太长本身，而是页面开头有大量：

- 多语言导航
- infobox
- 机构信息
- 坐标
- 预算
- 管理层

这些信息都在短距离内堆叠，导致：

- 很多 child chunk 都“看起来相关”
- dense 和 reranker 都容易在同一组近邻 chunk 内漂移

#### 第二类：定义和代码混在一起

`第四章 智能体经典范式构建` 最典型。

问题是：

- 定义性段落
- 示例输出
- 代码模板
- 案例总结
- 练习题

被一起切成了相邻 child chunks。  
对于 `ReAct` / `Reflection` 这种 query，系统很容易“找到提到这个词的块”，但未必找到“最适合作为定义答案的块”。

#### 第三类：一个 query 对应多个 gold chunk

这次 benchmark 里很多 anchor 在真实 chunking 下会映射出多个 gold chunk。  
这说明：

- 父子 chunk 并没有把答案完全切成一个唯一块
- 一个事实往往会同时出现在：
  - infobox
  - 开头概述
  - 正文展开

所以当前 `precision` 低并不总等于结果差，有时只是：

- 正确答案被多个块重复表达
- top-6 里有不少“半对”的相邻 chunk

## 十一、现在怎么判断系统检索能力

### 可以说“已经可用”的地方

- 中文课程定义类问题
- 中文历史文化类事实问题
- 英文短技术文档的明确配置项
- dense-only 召回后，reranker 能显著提升大部分数值和实体问题

### 还不能说“已经稳定”的地方

- 超长英文维基页的前身/别名/结构化字段问题
- `ReAct`、`Reflection` 这类术语型定义问题
- 中英混合 query
- 同一长文中多个相邻候选都很强的场景

### 一句话结论

当前 TalentHub 的真实检索能力可以概括为：

**基础 dense 召回已经可用，线上 reranker 有明显收益，但长维基页的结构化字段和课程文档里的术语定义仍然不够稳。**

## 十二、下一步最值得做什么

### 1. 补 BM25 / sparse 基线

当前最明显的缺口是：

- 还没有 `BM25` 或 `sparse` 召回对照

如果补上这条线，就能更准确回答：

- `term_and_alias`
- `preceding agency`
- `ReAct / Reflection`

这些失败到底是不是 dense 本身的短板。

### 2. 给 benchmark 增加多 gold chunk 标注

当前很多 case 明显不是“只有一个唯一正确块”。  
如果继续只标单 gold chunk，会低估系统真实可用性，也不利于解释 precision。

### 3. 单独增加“术语定义 benchmark”

目前最弱的其实不是纯 fact，而是：

- 范式名
- 英文术语
- 中英混合问法
- 标题与正文分离的定义性回答

这一块值得单独拆 benchmark。

### 4. 给长维基页做结构清洗

像 `NASA` 这种页面，光靠当前网页导入和 chunking 还不够。

后续更值得做的是：

- 弱化多语言导航
- 弱化 infobox 噪声
- 给 section title 更强的结构权重

### 5. 把评测分成三层

后续最理想的评测体系是：

1. `dense-only`
2. `dense + rerank`
3. `dense + sparse/hybrid + rerank`

这样才能真正看清每一层带来的净收益。

## 十三、最终结论

如果只基于这次“现有知识库真实语料 benchmark”来判断：

- 当前系统**已经不是 demo 级检索**
- 但也**还远没到可以宣称长文档检索已经完全成熟**

更客观的判断是：

- `OpenAI dense + Qdrant` 已经打下了可用底座
- `Voyage reranker` 明显提高了排序质量
- 但对于长维基页和课程型术语定义，系统仍然需要：
  - 更好的结构化预处理
  - sparse/BM25 补充
  - 更细的 benchmark 标注

这才是当前 TalentHub 检索能力最真实、最有工程价值的结论。
