# gRPC 学习路径与知识图谱

## 学习定位

很多人学 gRPC，停留在“会用脚手架”的层面：

- 会写 `.proto`，会 `protoc` 生成代码，会 `stub.sayHello(req)`
- 知道 gRPC「基于 HTTP/2」「用 Protobuf」「比 REST 快」
- 知道有四种流（unary / server / client / 双向）
- 知道要配 deadline、知道有 interceptor、知道返回的是 status code

但一被深挖就发懵：

- gRPC 到底「基于 HTTP/2」是怎么个基于法？抓包看到的 `POST /pkg.Svc/Method`、一堆二进制、`grpc-status` 跑到 trailer 里去了，这套映射是怎么设计的
- Protobuf 为什么快、为什么兼容性那么强？字段号删了会怎样、`int32` 改 `int64` 行不行、`required` 为什么在 proto3 被砍掉
- 为什么我用 K8s Service / 四层 ELB 给 gRPC 做负载均衡，流量全打到一个 Pod 上
- 为什么生产上动不动来个 `UNAVAILABLE`、`DEADLINE_EXCEEDED`、`RESOURCE_EXHAUSTED`，它们到底各自代表链路的哪一段出了问题
- gRPC 的重试、hedging、超时传播、cancellation 这套「调用治理」是怎么协同的，service config 又是什么
- 客户端负载均衡、xDS、proxyless mesh 这些词，和我平时用的 Nginx / Envoy 是什么关系
- 双向流里如果消费端慢了，数据会堆在哪、谁来背压、会不会 OOM

这条学习路径的目标，不是把你训练成「gRPC 脚手架操作员」或「gRPC 八股背诵机」，而是把 gRPC 拆回它的**三块基石**——Protobuf（契约 + 序列化）、HTTP/2（传输）、四种调用模型（服务语义）——再把它放回真实的微服务体系里，从「一个 RPC 框架」提升成「一套跨语言、强契约、面向长连接与流式的远程调用基础设施」来理解。

学完这一套，你应该能做到：

- 把一次 gRPC 调用在 HTTP/2 上的真实字节流画出来，并解释每一帧（HEADERS / DATA / TRAILERS）在干什么
- 讲清楚 Protobuf 的 wire format（varint、zigzag、tag-length-value）和它的兼容性规则是怎么互相支撑的
- 解释 gRPC 为什么必须是 HTTP/2，而不是「HTTP/2 顺便快一点」
- 把 deadline 传播、cancellation、重试、hedging、负载均衡这套客户端治理串成一张图
- 说清楚「gRPC 在四层 LB 下流量倾斜」的根因，并给出客户端 LB / headless / mesh 三种解法
- 面对线上 gRPC 故障（连接风暴、LB 不均、流控打满、偶发 UNAVAILABLE），有一套链路化的排障思路，而不是「重启 + 加机器」

## 知识图谱（从零到专家）

整张图分成 **7 层**，从外到里、从概念到字节：

### 第 1 层：整体认知
- gRPC 是什么、不是什么；为什么会出现（Google Stubby 的开源版 + REST/JSON 在大规模微服务下的痛点）
- 三块基石心智模型：**Protobuf（契约 + 序列化）+ HTTP/2（传输）+ 四种调用模型（服务语义）**
- gRPC 在架构里的典型位置：东西向内部服务调用、强契约微服务、流式 / 双向通信、多语言异构系统
- 边界：什么时候**不该**用 gRPC（浏览器直连、对外公开 API、要 curl 调试、老旧网络）

### 第 2 层：两块基石
- **Protocol Buffers**：IDL 语法、wire format（varint / zigzag / TLV / packed）、proto3 语义、前后向兼容规则、字段号为什么是“契约的主键”
- **HTTP/2**：为什么 gRPC 离不开它；Stream / Frame / HPACK / 流控 / 多路复用；队头阻塞如何从 HTTP/1 的“连接级”降到“TCP 级”，以及它带来的新麻烦

### 第 3 层：gRPC 协议本身
- 一次调用在 HTTP/2 上的真实样子：`:method=POST`、`:path=/pkg.Svc/Method`、`content-type: application/grpc`、**Length-Prefixed-Message**（5 字节前缀）、HEADERS → DATA → TRAILERS、`grpc-status` 为什么在 trailer
- 四种调用类型：Unary / Server streaming / Client streaming / 双向流的语义、半关闭、与 HTTP/2 stream 的对应
- Channel / Stub / 代码生成：从 `.proto` 到一次远程调用，blocking / async / future stub 的差异，Channel 与 Subchannel

### 第 4 层：调用语义与治理
- Deadline / 超时传播 / Cancellation：一条调用链的生命周期，为什么 deadline 是绝对时间而不是相对超时
- Metadata / Interceptor：横切面、上下文透传、鉴权与埋点
- 错误模型：status code 全集、`grpc-status` / `grpc-message`、`google.rpc.Status` rich error model、异常该怎么映射
- 重试 / hedging / service config：客户端容错的正确打开方式，幂等性的边界

### 第 5 层：连接、负载均衡与服务发现
- 名字解析（Name Resolver）+ service config 下发
- 负载均衡：**客户端 LB**、pick_first / round_robin、lookaside（gRPC-LB）/ xDS、为什么 gRPC 天然不爱 L4 代理
- 连接生命周期：keepalive / idle / GOAWAY / 流控 / 背压 / 优雅上下线

### 第 6 层：安全与互操作
- TLS / mTLS / ALTS；channel credentials vs call credentials；per-RPC 鉴权（token / JWT）
- 互操作：grpc-gateway / transcoding（gRPC ↔ JSON REST）、gRPC-Web（浏览器）、Server Reflection、Health Checking 协议

### 第 7 层：工程实战与排障
- 可观测性：OpenTelemetry tracing、metrics、channelz、日志
- 性能调优：消息大小上限、压缩、线程 / 事件循环模型、多语言实现差异（Java-Netty / Go 原生 / C-core 系）
- gRPC 在 K8s / Service Mesh 里的坑：L4 LB 破坏长连接、headless service、sidecar、proxyless gRPC + xDS
- 线上排障专题：UNAVAILABLE / DEADLINE_EXCEEDED / RESOURCE_EXHAUSTED / 连接风暴 / LB 不均
- 选型与边界：gRPC vs Thrift vs Dubbo vs REST/JSON vs GraphQL

## 为什么这样排序

这条路线刻意不是「先讲 proto 怎么写、再讲 server/client 怎么起」，而是先把 gRPC 拆回它的三块基石，再层层往上长：

- **第 1 课**先建立「gRPC = Protobuf + HTTP/2 + 四种调用模型」这个心智坐标系，并把边界划清楚。后面所有内容都是在这三块基石上展开。
- **第 2、3 课**是整个专栏的“地基”：Protobuf 和 HTTP/2。
  gRPC 自己其实很“薄”——它真正的复杂度，一半在 Protobuf 的 wire format 与兼容性，一半在 HTTP/2 的流与流控。这两块不啃下来，后面的「length-prefixed message」「流控背压」「HPACK 头压缩」全是悬空的名词。
- **第 4 课**把前两块合起来：一次调用在 HTTP/2 上到底长什么样。
  这是「面过 gRPC」和「真懂 gRPC」的第一道分水岭——能不能说清 `grpc-status` 为什么必须放在 trailer、5 字节前缀里那个字节是干嘛的。
- **第 5、6 课**讲调用模型与客户端骨架：四种流、Channel/Stub/代码生成。到这里你能完整跑通并理解一次调用。
- **第 7 到 10 课**是“调用治理”四件套：deadline 传播、metadata/interceptor、错误模型、重试/hedging。
  这四块决定了 gRPC 在真实微服务里「能不能扛住抖动、能不能链路透传、能不能优雅失败」，也是中间件团队最关心的部分。
- **第 11、12 课**进入 gRPC 最反直觉、也最容易在生产踩坑的地方：**负载均衡与连接生命周期**。
  「为什么 L4 LB 对 gRPC 会失效」「客户端 LB 是什么」「keepalive / GOAWAY / 背压怎么配”——这两课是 gRPC 区别于 HTTP/1 微服务的核心，也是线上事故高发区。
- **第 13、14 课**收口安全与互操作：TLS/mTLS、和 REST/浏览器世界怎么打通。
- **第 15 到 18 课**是工程纵深：可观测性与调优、K8s/Mesh 里的坑、线上排障手册、横向选型。
  到这一层，能不能讲到「Java gRPC 是 Netty 事件循环、Python 是 C-core 后台线程，所以阻塞行为完全不同」「proxyless gRPC 是怎么直接吃 xDS 的」，就是「用过」和「治理过」的分水岭。

## 学习方法建议

- **每篇先问一个问题**：这一篇在三块基石（Protobuf / HTTP/2 / 调用模型）的哪一块，或者在调用治理的哪一环。
  比如 length-prefixed message 是“gRPC 在 HTTP/2 DATA 帧里自己加的一层分帧”，deadline 是“把超时从相对时间换成绝对时间好跨进程传播”，客户端 LB 是“因为 gRPC 是长连接 + 多路复用，L4 代理拦不住”。
- **学协议时一定要落到字节**：看到「gRPC 基于 HTTP/2」不要停在这句话，要能画出 HEADERS 帧里有哪些伪头部、DATA 帧里 5 字节前缀的结构、TRAILERS 里的 `grpc-status`。
- **学 Protobuf 时盯住兼容性**：每个特性都问「改了它，老客户端 / 老服务端会怎样」。字段号、字段类型、`oneof`、`optional`、默认值、未知字段——这些都是线上灰度发布会踩的雷。
- **学治理时要看客户端和服务端怎么配合**：deadline 不是客户端单方面的事，服务端要检查并主动 cancel；重试不是无脑重发，要和幂等、hedging、service config 一起看。
- **学负载均衡和连接时建立“长连接思维”**：gRPC 的几乎所有反直觉坑，根因都是“它是少量长连接 + HTTP/2 多路复用”，而不是 HTTP/1 那种“一请求一短连接”。把这条主线抓住，K8s LB 倾斜、连接风暴、优雅下线、GOAWAY 这些都会串起来。
- **学排障时要建立“status code → 链路段”的映射**：见到 UNAVAILABLE 先想“连接 / 解析 / 服务端不可达”，见到 DEADLINE_EXCEEDED 先想“超时是谁设的、是不是被传播下来的、慢在哪一跳”，见到 RESOURCE_EXHAUSTED 先想“消息超限 / 流控 / 服务端限流”。

## 学完后你应该能回答的问题

- 一次 unary 调用在 HTTP/2 上经过几个帧，`grpc-status` 为什么不能放在响应的 HEADERS 而要放 TRAILERS
- Protobuf 的 varint 和 zigzag 各解决什么问题，为什么 `sint32` 比 `int32` 适合存负数
- proto3 砍掉 `required`、把字段默认非空，背后是出于什么兼容性考虑；删字段为什么要“保留字段号”
- gRPC 为什么必须 HTTP/2 而不是 HTTP/1.1，多路复用解决了什么、又把队头阻塞推到了哪一层
- deadline 是绝对时间还是相对时间，它怎么跨服务传播，服务端不检查 deadline 会有什么后果
- 客户端 LB 和 L4/L7 代理 LB 的区别，为什么 K8s 默认的 Service ClusterIP 会让 gRPC 流量倾斜，有哪三类解法
- gRPC 的重试默认开不开，开了之后非幂等接口会有什么风险，hedging 和 retry 的区别是什么
- keepalive、idle timeout、GOAWAY 分别在解决什么，服务端滚动发布时怎么让 gRPC 客户端优雅切走
- 双向流里下游消费慢，数据堆在哪、谁来背压，HTTP/2 流控和应用层背压是什么关系
- UNAVAILABLE / DEADLINE_EXCEEDED / RESOURCE_EXHAUSTED 三个最常见的 status，分别先怀疑什么
- gRPC 和 Thrift / Dubbo / REST 的真实边界在哪，什么场景下你会**不**选 gRPC

## 推进规则

- 第一次系统学 gRPC：严格按顺序，从 01 推到 06，先把「一次调用怎么走完」吃透，再往治理和运维走。
- 已经会用 gRPC、但原理不扎实：从 02、03、04 这三篇「地基」补起，这是大多数人最虚的地方。
- 重点是面试：02、03、04、07、11 这五篇要能讲到字节级 / 链路级。
- 重点是线上稳定性：10、11、12、16、17 这五篇当排障与治理手册看。
- 重点是选型 / 架构：先看 01、05、18，再回头补地基。

## 和 RPC 专栏的关系

仓库里已有一个 `RPC/` 专栏，那条线讲的是**通用 RPC 的共性**：动态代理、序列化、注册中心、负载均衡、熔断降级，以及 gRPC / Dubbo / Thrift 的横向对比。

本专栏是对其中「gRPC」这一格的**纵向深挖**：不再停留在“gRPC 用 protobuf + HTTP/2”这种一句话结论，而是钻到 wire format、HTTP/2 帧、调用治理、客户端 LB、mesh 集成的字节与链路级别。

- 想建立 RPC 的整体认知、做框架横向选型 → 看 `RPC/` 专栏。
- 想把 gRPC 一个框架吃到生产级、能排障能调优能讲源码动机 → 看本专栏。
