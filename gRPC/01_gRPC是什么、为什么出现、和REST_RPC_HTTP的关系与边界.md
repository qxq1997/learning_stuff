# 01 gRPC 是什么、为什么出现、和 REST/RPC/HTTP 的关系与边界

> 这一篇不写代码，只干一件事：把 gRPC 在你脑子里的坐标系建好。坐标系歪了，后面学 wire format、HTTP/2 帧、客户端负载均衡全是散点。

## 一、一句话定义，和它背后的三块基石

先给一句话定义，然后立刻把它拆开：

> **gRPC 是一个由 Google 开源的、跨语言的 RPC 框架；它用 Protocol Buffers 作为接口契约和序列化格式，用 HTTP/2 作为传输层，并原生支持四种调用模型（一元 + 三种流式）。**

这句话里藏着 gRPC 的**三块基石**，整个专栏都是围着这三块转的：

```
                    ┌───────────────────────────────────────────┐
                    │                  gRPC                       │
                    ├───────────────┬──────────────┬─────────────┤
        基石 1      │  Protobuf     │  基石 2       │  基石 3      │
                    │  契约 + 序列化 │  HTTP/2       │  四种调用模型 │
                    │  （“说什么”）  │  传输（“怎么传”）│ （“怎么调”）  │
                    └───────────────┴──────────────┴─────────────┘
```

- **基石 1 — Protobuf（说什么 / What）**：`.proto` 文件既是**接口契约**（IDL，定义服务、方法、消息），又是**序列化格式**（二进制 wire format）。这一块决定了 gRPC“强契约、跨语言、传得快”。
- **基石 2 — HTTP/2（怎么传 / How to transport）**：gRPC 不自造传输协议，而是把 HTTP/2 当底座。多路复用、二进制分帧、头压缩、流控、服务端推流，这些 HTTP/2 的能力直接被 gRPC 拿来用。这一块决定了 gRPC“长连接、能多路复用、能双向流”。
- **基石 3 — 四种调用模型（怎么调 / How to call）**：unary（一问一答）、server streaming（一问多答）、client streaming（多问一答）、bidirectional streaming（多问多答、全双工）。这一块决定了 gRPC“不只是请求-响应，还能做流”。

**记住这个拆法。** 以后任何一个 gRPC 的特性，你都能把它归到这三块里的某一块：

- “gRPC 兼容性为什么强” → 基石 1（Protobuf 字段号 + 兼容规则）
- “gRPC 为什么一条 TCP 连接能跑很多并发请求” → 基石 2（HTTP/2 多路复用）
- “gRPC 怎么做实时推送” → 基石 3（server / bidi streaming）

> 顺带说一句无聊但常被问的八卦：gRPC 里的 “g” **不固定**代表某个词。官方在每个版本的 release note 里给 “g” 编一个不同的词（gRPC Remote Procedure Calls、good、green……），是个递归式的玩笑。别在面试里一本正经地说“g 代表 Google”——虽然它确实出自 Google。

## 二、为什么会出现 gRPC：它在替谁解决什么痛

技术不会凭空出现。gRPC 的直接前身是 Google 内部用了十几年的 RPC 框架 **Stubby**。Google 内部有海量微服务、海量语言（C++/Java/Go/Python…）、海量东西向调用，Stubby 解决得很好，但它和 Google 内部基础设施绑死，没法直接开源。等到 HTTP/2 在 2015 年标准化（RFC 7540），Google 就基于 HTTP/2 重新做了一个能开源的版本，这就是 gRPC。

那它到底在替谁解决痛？答案是：**替“REST + JSON + HTTP/1.1”这套组合，在大规模内部微服务场景下解决痛。** 注意限定词是“大规模内部微服务”，不是“所有场景”——这点到第六节讲边界时会回扣。

REST/JSON/HTTP-1.1 在内部服务间调用时的几个真实痛点：

| 痛点 | REST + JSON + HTTP/1.1 的问题 | gRPC 的回应 |
|---|---|---|
| **没有强契约** | 接口长什么样靠文档 / Swagger，字段加错删错全靠人盯；前后端、跨团队对不齐 | `.proto` 是机器可校验的强契约，改契约会编译报错 |
| **序列化笨重** | JSON 是文本，字段名重复出现、数字也是字符串，体积大、解析慢 | Protobuf 二进制，字段名不传（只传字段号），体积小、编解码快 |
| **连接开销 / 队头阻塞** | HTTP/1.1 一条连接同一时刻只能跑一个请求；要并发就开很多连接，或者排队（队头阻塞） | HTTP/2 一条连接多路复用，几十上百个请求并发跑 |
| **没有原生流** | HTTP/1.1 做服务端推送要靠 SSE / long-polling / WebSocket，各搞各的 | 四种调用模型里三种是流式，统一在一套机制里 |
| **多语言一致性差** | 每种语言手写或半自动生成客户端，行为不一致 | 一份 `.proto`，`protoc` 给十几种语言生成一致的 stub |
| **手写客户端** | 调用方要自己拼 URL、序列化、处理状态码 | 生成的 stub 让远程调用“长得像本地方法调用” |

把这张表反过来读，就是 gRPC 的设计目标：**强契约、高效序列化、长连接多路复用、原生流、跨语言一致、调用透明**。

> 一个常见误区：很多人觉得 gRPC 主打的是“快”。**“快”只是结果，不是目标。** gRPC 真正的设计核心是「强契约 + 跨语言一致 + 长连接 / 流」。如果你只是想要“快”，HTTP/2 + Protobuf 自己拼也能快；gRPC 的价值是把契约、代码生成、调用治理（deadline / 重试 / 负载均衡 / 拦截器）打成一套体系。

## 三、gRPC 和 RPC 的关系：它是 RPC 思想的一套“工业级标准答案”

RPC（Remote Procedure Call）是一种**思想**：让调用远程服务像调用本地方法一样自然，把网络细节藏起来。任何 RPC 框架都要回答四个问题：

1. **接口怎么描述**（IDL）？
2. **数据怎么序列化**？
3. **数据怎么在网络上传**（传输协议）？
4. **调用怎么治理**（超时、重试、负载均衡、容错）？

不同框架给出不同答案。gRPC 的答案非常“成套”：

| RPC 要回答的问题 | gRPC 的答案 |
|---|---|
| 接口怎么描述 | Protocol Buffers IDL（`service` / `rpc` / `message`） |
| 数据怎么序列化 | Protocol Buffers 二进制 wire format |
| 数据怎么传 | HTTP/2 |
| 调用怎么治理 | deadline 传播、cancellation、metadata、interceptor、客户端负载均衡、重试 / hedging、service config |

所以一句话：**gRPC 是 RPC 思想的一个具体实现，而且是把这四个问题一次性给全的“标准答案”。** 它和 Dubbo、Thrift、bRPC 是“同类不同款”——都在回答这四个问题，只是选型不同（Dubbo 早期默认 Hessian + 自定义 TCP 协议 + 注册中心治理，Thrift 自带 IDL + 多种传输和协议组合）。横向对比放到第 18 篇细讲，这里你只要记住 gRPC 在“RPC 家族”里的位置。

## 四、gRPC 和 HTTP 的关系：它不是“绕开 HTTP”，而是“深绑 HTTP/2”

这是最容易被误解的一点，必须掰清楚。

很多人有个模糊印象：“RPC 是为了绕开 HTTP 的开销，所以自定义二进制协议。” 这话对 Dubbo 早期、Thrift 部分模式成立，但**对 gRPC 恰恰相反**——gRPC 是**深度绑定 HTTP/2** 的，它把自己整个架在 HTTP/2 之上。

但要注意：**gRPC 用的是 HTTP/2 的“传输能力”，不是 HTTP 的“RESTful 语义”。** 这是关键区别。一次 gRPC 调用在 HTTP/2 里长这样（第 04 篇会逐字节拆，这里先给个轮廓）：

```
HTTP/2 HEADERS 帧（请求头）:
    :method = POST                       ← 永远是 POST，不用 GET/PUT/DELETE
    :scheme = https
    :path   = /helloworld.Greeter/SayHello   ← 路径 = /包名.服务名/方法名
    content-type = application/grpc
    te = trailers                        ← 声明我能收 trailer
    grpc-timeout = 1S                    ← deadline 就藏在这里
    （还有 HPACK 压缩过的自定义 metadata）

HTTP/2 DATA 帧（请求体）:
    [1 字节 压缩标志][4 字节 消息长度][N 字节 Protobuf 序列化后的请求]
    ↑ 这 5 字节叫 Length-Prefixed-Message，是 gRPC 在 HTTP/2 之上自己加的一层分帧

────────────── 服务端返回 ──────────────

HTTP/2 HEADERS 帧（响应头，叫 initial metadata）:
    :status = 200                        ← 注意！HTTP 状态永远是 200，哪怕业务失败
    content-type = application/grpc

HTTP/2 DATA 帧（响应体）:
    [5 字节前缀][Protobuf 序列化后的响应]

HTTP/2 HEADERS 帧（带 END_STREAM，叫 Trailers）:
    grpc-status  = 0                     ← 真正的业务状态码在这！（0 = OK）
    grpc-message = ...                   ← 出错时的描述
```

从这个轮廓里，提前记住三个“反直觉但很重要”的点，它们都是面试高频、抓包必见：

1. **HTTP method 永远是 POST，path 永远是 `/服务/方法`。** gRPC 不用 HTTP method 表达语义（不像 REST 用 GET/POST/PUT/DELETE 表达增删改查），它把语义全放在 path 和 body 里。
2. **HTTP 状态码永远是 200（只要这条 HTTP 流本身正常）。** 业务上的成功 / 失败用 **`grpc-status`** 表达，而 `grpc-status` 在 **trailer**（流末尾的 HEADERS 帧）里，不在响应头里。为什么必须放 trailer？因为流式调用时，服务端要先把一大串 DATA 帧推给你，等全推完了才知道最终成功还是失败——所以最终状态只能放在“流的最后面”。这点第 04、09 篇会反复回扣。
3. **gRPC 在 HTTP/2 的 DATA 帧里又自己套了一层 5 字节前缀**（Length-Prefixed-Message）。为什么 HTTP/2 已经分帧了，gRPC 还要再分一次？因为一个 HTTP/2 DATA 帧和一条 gRPC 消息**不是一一对应**的——一条大消息可能拆到多个 DATA 帧，一个 DATA 帧也可能装多条小消息（流式时）。gRPC 需要自己的“消息边界”，所以加了这层前缀。

> 所以面试里如果被问“gRPC 和 HTTP 什么关系”，最精准的回答是：**“gRPC 把 HTTP/2 当传输底座，复用它的多路复用、二进制分帧、头压缩、流控、trailer，但不复用 HTTP 的 RESTful 语义——method 恒为 POST、状态走 grpc-status 而非 HTTP status。它是‘跑在 HTTP/2 上的 RPC’，不是‘RESTful HTTP’。”**

## 五、gRPC vs REST：一张对比表 + 一句心智总结

把 gRPC 和 “REST + JSON over HTTP/1.1” 这套主流对外 API 风格放一起对比（这是最常被要求对比的两者）：

| 维度 | gRPC | REST + JSON |
|---|---|---|
| 契约 | `.proto` 强契约，机器校验、自动生成 | OpenAPI/Swagger 可选，弱约束，靠人维护 |
| 序列化 | Protobuf 二进制，小而快 | JSON 文本，大而慢，但人类可读 |
| 传输 | HTTP/2（强制） | 通常 HTTP/1.1，也可 HTTP/2 |
| 流式 | 原生四种（含双向全双工） | 需 SSE / WebSocket 另搞 |
| 浏览器 | **不能直连**，需 gRPC-Web + 代理 | 原生支持，`fetch` 即可 |
| 可调试性 | 二进制，要 `grpcurl` / 反射，curl 不行 | curl / 浏览器直接看，可读性强 |
| 跨语言代码生成 | 一等公民，`protoc` 全语言 | 靠各语言 SDK / 手写 |
| 缓存（HTTP 缓存） | 基本用不上（POST + 二进制） | 天然适配 CDN / HTTP 缓存 |
| 典型场景 | 内部微服务东西向、低延迟、高吞吐、流式 | 对外公开 API、浏览器、第三方集成 |

**心智总结：gRPC 是“为机器与机器之间的内部调用优化的”，REST 是“为开放、可读、可缓存的对外接口优化的”。** 二者不是替代关系，绝大多数公司是**两者并存**：对外网关层用 REST/JSON（甚至 GraphQL），网关往后的内部服务之间用 gRPC。第 14 篇讲的 grpc-gateway 就是把这两个世界在边界上缝起来的胶水。

## 六、边界：什么时候**不该**用 gRPC

「力求全面」不等于「无脑安利」。能清楚说出一个技术的边界，比会用它更值钱。下面这些场景，gRPC 要么用不了，要么得不偿失：

1. **浏览器直接调用后端**。浏览器的 `fetch`/`XMLHttpRequest` **拿不到 HTTP/2 的 trailer，也无法精细控制帧**，所以**不能直接讲原生 gRPC**。必须用 **gRPC-Web**（裁剪版协议）+ 一个代理（Envoy / grpc-gateway）做转换。如果你的主要消费方是浏览器，REST/JSON 往往更省心。
2. **对外公开 API / 第三方集成**。外部开发者期望 curl 能调、文档能读、Postman 能点。给外部甩一个 `.proto` 让人家装 `protoc`，体验很差。对外用 REST，内部用 gRPC，是常见分层。
3. **需要 HTTP 缓存 / CDN**。gRPC 是 POST + 二进制，天然和 HTTP 缓存语义（GET + ETag + Cache-Control）不沾边。读多写少、想吃 CDN 缓存的接口，REST 更合适。
4. **老旧网络 / 中间设备对 HTTP/2 支持差**。一些老的负载均衡器、企业代理、防火墙对 HTTP/2（尤其是明文 h2c、长连接、trailer）支持不好，可能把连接掐断或降级，导致诡异故障。
5. **团队对二进制协议的可观测 / 调试没准备**。gRPC 抓包是二进制，排查要靠 `grpcurl` + 反射、channelz、专门的拦截器埋点。如果团队还没建立这套，初期排障会比 REST 痛苦（这也是为什么本专栏后半段花大篇幅讲可观测性与排障）。
6. **极简单、低频、内部脚本级调用**。杀鸡用牛刀。一个内部定时任务调一个接口，REST/JSON 甚至直接函数调用都行，没必要上 protoc 工具链。

> 一个好用的判断口诀：**“东西向、内部、强契约、高频 / 低延迟 / 流式”→ gRPC；“南北向、对外、要可读 / 可缓存 / 浏览器友好”→ REST。** 中间用网关缝合。

## 七、一张总览图：gRPC 在你系统里的位置

```
                       ┌────────── 浏览器 / 移动端 / 第三方 ──────────┐
   南北向（对外）       │   REST/JSON、GraphQL、gRPC-Web              │
   可读 / 可缓存        └───────────────────────┬────────────────────┘
                                               │
                                       ┌───────▼────────┐
                                       │   API 网关       │  ← grpc-gateway / Envoy
                                       │ (REST ↔ gRPC 转换)│     在这里缝合两个世界
                                       └───────┬────────┘
                                               │  gRPC (HTTP/2 + Protobuf)
        ┌──────────────────┬───────────────────┼───────────────────┬──────────────┐
        │                  │                   │                   │              │
   ┌────▼────┐       ┌─────▼────┐        ┌──────▼─────┐      ┌──────▼─────┐  ┌────▼─────┐
   │ 订单服务  │◄────►│ 库存服务   │◄──────►│  用户服务    │◄────►│  支付服务    │  │  风控服务  │
   │ (Java)  │ gRPC │ (Go)     │  gRPC  │  (Python)  │ gRPC │  (Java)    │  │  (C++)   │
   └─────────┘       └──────────┘        └────────────┘      └────────────┘  └──────────┘
        东西向（内部服务之间）：强契约、长连接多路复用、低延迟、跨语言一致、可流式
```

记住这张图：**gRPC 的主战场是网关之后、服务与服务之间的“东西向”流量。** 对外那一层交给 REST，浏览器那一段交给 gRPC-Web，跨语言由 Protobuf 抹平。后面所有章节，本质都是在把这张图里“服务之间那根 gRPC 箭头”讲透。

## 八、本章小结

- gRPC = **三块基石**：Protobuf（契约 + 序列化）+ HTTP/2（传输）+ 四种调用模型（语义）。后面所有内容都挂在这三块上。
- 它的出现是为了解决 **REST/JSON/HTTP-1.1 在大规模内部微服务下**的痛：无强契约、序列化笨重、连接 / 队头阻塞、无原生流、多语言不一致。
- 它是 **RPC 思想的一套成套标准答案**（IDL + 序列化 + 传输 + 治理一次给全），在家族里和 Dubbo / Thrift 同类不同款。
- 它和 HTTP 的关系是**深绑 HTTP/2 的传输能力、但不用 RESTful 语义**：method 恒为 POST、状态走 `grpc-status`（在 trailer 里）、DATA 帧里还套了 5 字节 length-prefix。
- 它**不是 REST 的替代**，而是分工：东西向内部用 gRPC，南北向对外用 REST，网关缝合。
- 边界很重要：浏览器直连、对外 API、要缓存 / 可读 / curl 调试、老旧网络——这些场景 gRPC 不占优。

## 生产常见问题

> 每章末尾固定这一节。这里先放几个“在还没深入原理时就会撞上、且根因都要靠后面章节解释”的真实问题，并标好它们会在哪一篇被讲透。这样你带着问题往下读。

**Q1：我用 K8s Service（ClusterIP）/ 四层 ELB 给 gRPC 做负载均衡，为什么压测时流量几乎全打到一个 Pod 上，扩容也没用？**

根因：gRPC 是**长连接 + HTTP/2 多路复用**。L4（四层）负载均衡只在**建立 TCP 连接那一刻**做一次分发，之后这条连接上的成百上千个请求全部复用同一条连接、打到同一个后端 Pod。客户端建连少、连接又长寿，于是流量严重倾斜。这和 HTTP/1.1“一请求一短连接、每次都重新被 LB 分发”完全不同。
解法方向有三类：客户端负载均衡（gRPC 自己在多个后端间轮询）、headless Service + 客户端解析所有 Pod IP、或上 L7 代理 / service mesh（Envoy 在请求级别 LB）。这是 gRPC **最经典的生产坑**，第 **11**、**16** 篇会专门拆。

**Q2：浏览器前端同学说“你给我个 gRPC 接口我直接调”，能直接调吗？**

不能直接调原生 gRPC。浏览器拿不到 HTTP/2 trailer、控制不了帧，所以社区搞了 **gRPC-Web**（裁剪协议），但它需要一个代理（Envoy 的 grpc-web filter 或 grpc-gateway）在中间把 gRPC-Web 转成原生 gRPC。直接把 `.proto` 甩给前端让他调原生 gRPC 是行不通的。详见第 **14** 篇。

**Q3：抓包（Wireshark / tcpdump）看 gRPC，为什么看到的是 `POST`，body 一堆乱码，HTTP 状态还都是 200，根本看不出业务是成功还是失败？**

因为：① gRPC 的 method 恒为 POST，语义在 path（`/服务/方法`）里；② body 是 Protobuf 二进制 + 5 字节长度前缀，不是文本，肉眼看就是“乱码”；③ HTTP status 恒为 200，真正的业务状态码是 **`grpc-status`**，它在**响应流末尾的 trailer**里，不在响应头里。要想看懂，得让抓包工具解 HTTP/2 + gRPC（Wireshark 有 gRPC dissector），或者用 `grpcurl` 调。逐字节解释见第 **04** 篇，status 体系见第 **09** 篇。

**Q4：服务明明是好的，客户端却偶发报 `UNAVAILABLE`，重试一下又好了，是不是 gRPC 不稳定？**

通常不是“不稳定”，而是**长连接的正常生命周期事件**没被正确处理。服务端滚动发布、idle 超时、keepalive 探活失败、或服务端主动发 **GOAWAY** 要优雅关闭连接时，客户端如果没处理好，就会在那一瞬间把正在/即将发出的请求报成 `UNAVAILABLE`。配好 keepalive、重试策略、优雅下线后，绝大多数偶发 UNAVAILABLE 会消失。连接生命周期见第 **12** 篇，重试见第 **10** 篇，排障见第 **17** 篇。

**Q5：我把 `.proto` 里一个字段从 `int32` 改成了 `int64`，或者删了一个字段，灰度发布后线上数据开始错乱 / 报解析错误，为什么？**

因为 Protobuf 的兼容性规则是有边界的，**不是“随便改都兼容”**。字段号（field number）才是契约的“主键”，字段名只是给人看的。`int32 → int64` 在 wire 上有微妙差异（都属于 varint，但取值范围和符号处理要小心）；删字段如果**字段号被后续新字段复用**，老数据就会被错误解读——所以删字段要“保留（reserve）字段号”。这类灰度事故几乎全出在不懂 Protobuf 兼容规则上，第 **02** 篇会把“能改什么、不能改什么、怎么安全演进”讲透。
