# 03 HTTP/2 深挖：为什么 gRPC 离不开它，Stream/Frame/HPACK/流控与多路复用

> 第二块地基。上一篇钻 Protobuf 的字节，这一篇钻 HTTP/2 的帧。一句话先抛出这章的灵魂：**HTTP/2 的多路复用是 gRPC 的超能力，也是 gRPC 几乎所有反直觉生产坑的根源。** L4 负载均衡为什么会失效、单连接为什么会突然排队、优雅下线为什么这么麻烦——根全在这一篇。把 02（Protobuf）和 03（HTTP/2）打牢，下一篇就能把一次 gRPC 调用逐字节拆开。

## 一、HTTP/1.1 的四宗罪：gRPC 为什么不能用它

要理解 gRPC 为什么**必须**是 HTTP/2，先得看清 HTTP/1.1 在“高频内部服务调用”这个场景下的硬伤：

1. **应用层队头阻塞（Head-of-Line Blocking）**。HTTP/1.1 一条连接，**同一时刻只能跑一个请求-响应**，必须一问一答串行。前一个响应没回来，后面的请求只能干等。这对“一条连接想跑很多并发 RPC”是致命的。
2. **并发只能靠堆连接**。要并发就得开多条 TCP 连接（浏览器对单域名通常限 6 条）。每条连接都要 TCP 握手 + TLS 握手，开销大；连接多了服务端文件描述符、内存也吃不消。
3. **头部冗长且不压缩**。HTTP/1.1 的头是纯文本，`Host`、`User-Agent`、`Cookie` 这些每个请求都重复发一遍，几百上千字节，没有任何压缩。
4. **没有原生的流 / 服务端推送**。想做服务端持续推数据，只能 SSE、long-polling、WebSocket 各搞一套，没有统一机制。

> 有人会说“HTTP/1.1 不是有 pipelining 吗”。有,但 pipelining 要求**响应必须按请求顺序返回**,一个慢响应照样堵死后面所有响应——队头阻塞没解决,反而更难处理,所以现实中基本没人用、浏览器默认关闭。

HTTP/2 就是来一次性解决这四宗罪的。而 gRPC 想要的“**少量长连接 + 海量并发 RPC + 原生双向流 + 紧凑头部**”，恰好就是 HTTP/2 解决方案的形状——所以 gRPC 直接把自己架在 HTTP/2 上，不另造轮子。

## 二、HTTP/2 的四层抽象：Connection / Stream / Message / Frame

HTTP/2 把通信拆成四层，这四层的关系是理解一切的前提：

```
┌─────────────────────────────────────────────────────────────┐
│  Connection（连接）  = 1 条 TCP（通常 + TLS）                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Stream 1（一个 RPC）   ┌── Message：请求 ──┐            │   │
│  │                       │ HEADERS帧 DATA帧 ..│            │   │
│  │                       └───────────────────┘            │   │
│  │                       ┌── Message：响应 ──┐             │   │
│  │                       │ HEADERS DATA TRAILERS│          │   │
│  │                       └───────────────────┘            │   │
│  ├──────────────────────────────────────────────────────┤   │
│  │ Stream 3（另一个 RPC，和 Stream 1 在同一连接上并发交错）   │   │
│  ├──────────────────────────────────────────────────────┤   │
│  │ Stream 5 ...                                          │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘

Connection  ⊃  Stream  ⊃  Message  ⊃  Frame
  一条TCP      一个虚拟      一个请求或      最小传输单位
              双向通道       一个响应        （二进制）
```

- **Connection**：一条 TCP 连接（gRPC 里通常带 TLS）。
- **Stream（流）**：连接内的一个**虚拟双向通道**，有唯一的 **Stream ID**。**一次 gRPC RPC = 一个 Stream**（记住这条，第 04、05 篇反复用）。
- **Message**：一个完整的请求或响应，由一个或多个帧组成。
- **Frame（帧）**：最小单位，二进制。每个帧都标着自己属于哪个 Stream ID。

最关键的转变：HTTP/1.1 是“一条连接=一个请求通道”，HTTP/2 是“**一条连接里塞很多个 Stream，它们的帧在连接上交错并发跑**”。这就是多路复用，第四节细讲。

## 三、帧：9 字节固定头 + payload，十种帧类型

HTTP/2 是**二进制协议**（HTTP/1.1 是文本）。所有数据都切成帧。每个帧 = **9 字节固定帧头 + 变长 payload**：

```
帧头（9 字节）：
+-----------------------------------------------+
|                Length (24 bits)               |  payload 长度，默认上限 16KB
+---------------+---------------+
|  Type (8)     |  Flags (8)    |                  帧类型 + 标志位
+-+-------------+---------------+-----------------+
|R|              Stream Identifier (31 bits)      |  这个帧属于哪个 Stream
+-+---------------------------------------------+-+
|                 Frame Payload ...               |
+-------------------------------------------------+
```

**十种帧类型**，gRPC 里你需要认识其中这些：

| Type | 帧名 | 作用 | 在 gRPC 里对应什么 |
|---|---|---|---|
| 0x0 | **DATA** | 传输请求/响应体 | gRPC 消息体（里面装 length-prefixed protobuf） |
| 0x1 | **HEADERS** | 传输头部 | 请求头 / 响应头 / **trailers**（grpc-status 在这） |
| 0x3 | **RST_STREAM** | 立即终止某个 stream | gRPC **取消单个调用**（cancellation） |
| 0x4 | **SETTINGS** | 连接参数协商 | 并发流上限、窗口大小等（见下） |
| 0x6 | **PING** | 探活 / 测 RTT | gRPC **keepalive** |
| 0x7 | **GOAWAY** | 优雅关闭整条连接 | gRPC **优雅下线**（滚动发布） |
| 0x8 | **WINDOW_UPDATE** | 流控窗口补充 | gRPC **流控 / 背压**的底层 |
| 0x5 | PUSH_PROMISE | 服务端推送 | gRPC **不用**（明确禁用 server push） |
| 0x2 / 0x9 | PRIORITY / CONTINUATION | 优先级 / 头部续帧 | 少用 |

把这张表记住：**gRPC 的很多“高级特性”其实就是某个 HTTP/2 帧的直接应用**——cancellation 是 RST_STREAM，keepalive 是 PING，优雅下线是 GOAWAY，背压是 WINDOW_UPDATE。后面好几章会回扣这张表。

**Stream ID 的规则**（有讲究）：

- **客户端发起的 stream 用奇数**（1, 3, 5…），**服务端发起的用偶数**。gRPC 里几乎都是客户端发起，所以你抓包看到的多是奇数。
- **Stream ID 0 保留给连接级控制帧**（SETTINGS、PING、GOAWAY、连接级 WINDOW_UPDATE）——这些不属于任何具体 stream。
- **Stream ID 单调递增、用完不复用**。一条连接的 ID 是 31 位，理论上能用 ~21 亿个；**用尽后必须新建连接**（长寿连接 + 超高频 RPC 的服务要注意，虽然很少触及）。GOAWAY 帧里会带“我处理到的最后一个 stream id”，告诉对端哪些 stream 还会被处理、哪些请重发。

## 四、多路复用：一条连接跑几百个 RPC（gRPC 的超能力）

这是 HTTP/2 的核心，也是 gRPC 的立身之本。

因为每个帧都标了 Stream ID，**多个 stream 的帧可以在同一条 TCP 连接上交错发送**，接收方按 Stream ID 重新归类组装：

```
一条 TCP 连接上的帧序列（时间从左到右）：

  [S1:HEADERS][S3:HEADERS][S1:DATA][S5:HEADERS][S3:DATA][S1:DATA,END][S5:DATA]...
       │           │          │         │          │          │          │
       └─ RPC-A    └─ RPC-B   └─ RPC-A  └─ RPC-C   └─ RPC-B   └─ RPC-A    └─ RPC-C
       
  接收方按 Stream ID 拆开 → 同时进行的 RPC-A、RPC-B、RPC-C 互不阻塞
```

意义对 gRPC 极其重大：

- **一条长连接就能并发跑几百上千个 RPC**，不用像 HTTP/1.1 那样为并发狂开连接。建连成本（TCP + TLS 握手）摊薄到趋近于零。
- **没有应用层队头阻塞**：RPC-A 的慢响应不会堵住 RPC-B、RPC-C。
- **双向流成为可能**：同一个 stream 上，请求方向和响应方向的 DATA 帧可以同时来回飞（第 05 篇讲四种调用模型时细说）。

**但是**——这个超能力有个 SETTINGS 参数卡着它，必须知道：

> **`SETTINGS_MAX_CONCURRENT_STREAMS`**：一条连接上**同时存活的 stream 数上限**。也就是“一条连接最多同时跑多少个 RPC”。

这个值各实现默认不同（grpc-go 服务端历史上默认 250，Envoy/nginx 常见 100/128，grpc-java 默认很大）。**一旦客户端在一条连接上的并发 RPC 数顶到这个上限，新的 RPC 就在客户端本地排队等待**——表现为 **P99 延迟尖刺，但服务端 CPU/负载看着一点都不高**，极度迷惑人。这是第十节“一体两面”和生产问题 Q3 的核心，先记住。

## 五、HPACK：把重复的头压到 1 个字节

HTTP/2 用 **HPACK** 压缩头部，解决 HTTP/1.1 “头部重复且不压缩”的第三宗罪。HPACK 三件套：

1. **静态表（Static Table）**：61 个预定义的常见头，固定编号。比如 `:method: GET` 是 2 号、`:status: 200` 是 8 号。命中静态表，**一个字节（索引）就发完一个头**。
2. **动态表（Dynamic Table）**：连接维度维护，把本连接出现过的头**动态加进去**、FIFO 淘汰。第一次发 `grpc-timeout: 1S` 要发全量并存入动态表，**之后再发就只发一个索引**。大小由 `SETTINGS_HEADER_TABLE_SIZE`（默认 4096 字节）控制。
3. **Huffman 编码**：对没进表的字符串值做静态 Huffman 压缩，再省一道。

对 gRPC 的意义：gRPC 每个 RPC 都带一堆头——`:method`、`:path`、`:authority`、`content-type: application/grpc`、`te: trailers`，外加你的 metadata（trace id、token、租户 id…）。这些头在同一条连接上**高度重复**，HPACK 动态表能把它们压到几乎不占带宽。**gRPC 的 metadata 之所以“便宜”，靠的就是 HPACK。**

> 一个安全注意点：HPACK 动态表是按连接共享、且基于“重复内容压缩”的,历史上 HTTP 压缩(SPDY/TLS 压缩)出过 CRIME 这类**压缩侧信道攻击**。HPACK 在设计上做了缓解(头字段值可标记“不索引”),但把高熵敏感值(如完整 auth token)反复塞进会被压缩的头里时,仍要意识到这层。gRPC 实践中 auth token 走 metadata,通常依赖 TLS 整体加密来兜底。

## 六、流控（Flow Control）：两级窗口，背压的地基

这是为 gRPC streaming 打的最重要的地基。

**问题**：多路复用下,一条连接跑很多 stream。如果发送方猛灌数据、接收方处理不过来,数据会堆在接收方内存里,可能 OOM,而且一个贪婪的 stream 会饿死其他 stream。**需要一个机制让接收方能喊“慢点,我还没处理完”**——这就是流控。

HTTP/2 的流控基于**窗口（window）+ WINDOW_UPDATE 帧**，而且是**两级**：

```
┌──── 连接级窗口（Stream 0）───────────────────────┐
│  整条连接所有 DATA 帧共享的总额度                    │
│   ┌─ 流级窗口 Stream1 ─┐  ┌─ 流级窗口 Stream3 ─┐   │
│   │ 这个 stream 的额度  │  │ 这个 stream 的额度  │   │
│   └───────────────────┘  └───────────────────┘   │
└─────────────────────────────────────────────────┘

发送方每发 N 字节 DATA → 连接级窗口和该流级窗口各扣 N
接收方处理完数据 → 发 WINDOW_UPDATE 把窗口补回去
窗口归零 → 发送方必须停下,直到收到 WINDOW_UPDATE
```

关键点：

- **只有 DATA 帧受流控**，HEADERS、SETTINGS 等控制帧不受（保证控制信令永远能发出去）。
- **初始窗口 `SETTINGS_INITIAL_WINDOW_SIZE` 默认只有 65535 字节（64KB）**。这个默认值在**高带宽 × 高延迟**链路上**严重限制吞吐**——这就是带宽时延积（BDP）问题：链路能容纳的在途数据远大于 64KB,但窗口卡在 64KB,发送方发满 64KB 就得停下等 ACK 回来的 WINDOW_UPDATE,带宽利用率上不去。
- 所以 grpc-go、grpc-java 都实现了 **BDP 自动探测 / 动态窗口（auto-tuning）**,根据实测 RTT 和带宽动态把窗口调大。跨地域、大消息、流式传输时这个特性很关键(调优见第 15 篇)。

**流控 = 背压（backpressure）的底层机制**。第 05、12 篇讲流式调用“消费端慢了数据堆哪”,答案就落在这两级窗口上：消费端不读 → 不发 WINDOW_UPDATE → 窗口耗尽 → 发送端被迫停下,压力沿链路反向传导。**HTTP/2 流控让背压能自动传播,这是 gRPC 流式不 OOM 的保障。**

## 七、流的生命周期，与 RST_STREAM / GOAWAY

每个 stream 有自己的状态机（简化版）：

```
        发送/接收 HEADERS
 idle ──────────────────► open
                            │
              一方发 END_STREAM（半关闭）
                            ▼
                      half-closed
                            │
              另一方也发 END_STREAM
                            ▼
                         closed

  RST_STREAM 可从任意状态强制 ──► closed（异常终止该 stream）
```

- **`END_STREAM` 标志**：标在最后一个帧上,表示“我这个方向发完了”。双向都发完 → stream 正常关闭。**gRPC 的“半关闭”就是用它**:client streaming 时,客户端发完所有请求消息后,用 END_STREAM 告诉服务端“我说完了,该你了”(第 05 篇细讲)。
- **`RST_STREAM`**:立即终止**单个** stream,不影响连接上其他 stream。**gRPC 的调用取消(cancellation)、deadline 超时,底层就是发 RST_STREAM**(第 07 篇)。
- **`GOAWAY`**:优雅关闭**整条连接**。发送方说“我不再接收新 stream 了,但已经在处理的(id ≤ last-stream-id 的)我会处理完”。**gRPC 服务端滚动发布 / 优雅下线,靠的就是 GOAWAY**:先发 GOAWAY 让客户端别再往这条连接发新 RPC、去重新建连到新实例,存量 RPC 处理完再真正关闭(第 12、16 篇)。

把这三个和第三节的帧表连起来：**gRPC 调用治理的“取消、超时、优雅下线”这些动作,落到 HTTP/2 就是 RST_STREAM / GOAWAY 这几个帧**。这是“gRPC 很薄、复杂度在 HTTP/2”的又一例证。

## 八、h2 vs h2c：加密的和明文的

gRPC 跑在 HTTP/2 上,而 HTTP/2 有两种形态,生产里都会遇到:

- **h2**:HTTP/2 **over TLS**。通过 TLS 握手时的 **ALPN（Application-Layer Protocol Negotiation）** 扩展协商出 “h2”。**浏览器只支持这种**(浏览器不支持明文 HTTP/2)。
- **h2c**:HTTP/2 **cleartext**,明文,不走 TLS。gRPC 内网服务之间为了省 TLS 开销常用 h2c。建连方式通常是 **prior knowledge**——客户端直接假设服务端讲 h2c,上来就发 HTTP/2 连接前言,不走 HTTP/1.1 Upgrade 协商。

**连接前言（Connection Preface）**:HTTP/2 连接建立后,客户端必须先发一段固定的 magic 字节串:

```
PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n      ← 24 字节魔数,然后紧跟一个 SETTINGS 帧
```

(这串字符拼起来是 “PRISM”的梗。)服务端也要回 SETTINGS 帧。双方交换 SETTINGS 后,连接才真正可用。

**h2c 是个生产大坑**:很多老式四层/七层 LB、企业代理、API 网关对**明文 HTTP/2** 支持很差——它们可能把 h2c 连接误当 HTTP/1.1 处理、或者干脆掐断长连接。所以“本地 h2c 跑得好好的,一上某些云 LB / 代理就各种诡异 UNAVAILABLE”是常见现象(生产问题 Q1)。

## 九、队头阻塞的转移：从应用层到 TCP 层（HTTP/3 的动机）

这是 HTTP/2 最深刻、也最常被面试追问到底的一点。

HTTP/2 的多路复用**消除了应用层（HTTP 层）的队头阻塞**——一个 stream 慢不会堵其他 stream。**但它没有、也不可能消除 TCP 层的队头阻塞**:

```
所有 stream 共享同一条 TCP 连接,而 TCP 保证“字节流严格有序、不丢”。
若某个 TCP 段在网络中丢失：

  发送端: [S1段][S3段][S1段(丢!)][S5段][S3段] ...
                          ✗
  接收端内核: 收到了 S5、S3 后续的段,但因为 S1 那个段没到,
             TCP 必须等它重传补齐 ── 在此之前,后面所有段
             (哪怕是别的 stream 的!)都被堵在内核缓冲区,
             应用层一个字节都拿不到。
```

也就是说:**HTTP/2 把队头阻塞从“应用层”下推到了“TCP 层”。** 网络丢包率越高、连接上 stream 越多,这个 TCP 级队头阻塞的杀伤越大——一个包丢失,连累一整条连接上所有并发 RPC 集体卡顿。

**这正是 HTTP/3 改用 QUIC（基于 UDP）的根本动机**:QUIC 在用户态把“流”做成了**真正独立**的传输单元,一个 stream 丢包只影响它自己,不连累其他 stream,从而**彻底消除队头阻塞**。gRPC over HTTP/3 已在演进中(grpc-go 等有实验支持),但目前生产主力仍是 HTTP/2,所以本专栏以 HTTP/2 为主、HTTP/3 作为演进方向提及。

> 面试标准答法:**“HTTP/2 用多路复用消除了 HTTP 层队头阻塞,但因为所有 stream 复用一条 TCP、而 TCP 是有序字节流,丢包时会触发 TCP 层队头阻塞,连累整条连接。HTTP/3 用基于 UDP 的 QUIC、让每个 stream 独立传输来根治这一点。”**

## 十、一体两面：多路复用既是超能力,也是所有坑的根源

把这章收束成一张“因果图”,这是理解后续所有 gRPC 运维章节的总纲:

```
            HTTP/2 = 少量长连接 + 多路复用
                         │
        ┌────────────────┼─────────────────────┐
        │ 超能力(正面)                坑(反面)    │
        ▼                                       ▼
 · 一条连接跑几百 RPC         · L4 LB 只在建连时分发 → 连接少且长寿
 · 建连成本摊薄到零             → 流量倾斜,扩容无效(第11、16篇)
 · 无应用层队头阻塞           · 单连接 MAX_CONCURRENT_STREAMS 顶满
 · 原生双向流                   → RPC 本地排队、P99 尖刺(本章第四节)
 · HPACK 让 metadata 便宜      · 连接是长寿的有状态实体
 · 流控让背压自动传播           → 优雅下线要 GOAWAY、滚更要重均衡(第12篇)
                              · 所有 stream 共享一条 TCP
                                → 丢包触发 TCP 队头阻塞(第九节)
```

**记住这条主线:gRPC 的几乎每一个反直觉坑,根因都能追到“它是少量长连接 + HTTP/2 多路复用”,而不是 HTTP/1.1 那种“一请求一短连接”。** 把这句话刻进去,后面 L4 LB 倾斜、连接风暴、优雅下线、流控背压这些章节,你会发现它们其实是同一个根长出来的不同枝。

## 十一、本章小结

- gRPC **必须** HTTP/2,因为它要的“少量长连接 + 海量并发 RPC + 双向流 + 紧凑头”正是 HTTP/2 解决 HTTP/1.1 四宗罪的形状。
- HTTP/2 四层:**Connection ⊃ Stream ⊃ Message ⊃ Frame**;**一次 RPC = 一个 Stream**。
- 帧 = 9 字节头 + payload;**gRPC 的取消=RST_STREAM、keepalive=PING、优雅下线=GOAWAY、背压=WINDOW_UPDATE**。
- **多路复用**是超能力(一条连接并发几百 RPC),但被 `MAX_CONCURRENT_STREAMS` 卡上限,顶满会本地排队。
- **HPACK** 让重复头压到 1 字节,这是 gRPC metadata 便宜的原因。
- **两级流控窗口**是背压的地基;默认 64KB 在高 BDP 链路是瓶颈,靠动态窗口调优。
- HTTP/2 消除了**应用层**队头阻塞,但把它**下推到 TCP 层**;HTTP/3/QUIC 才根治。
- 总纲:**gRPC 的坑大多来自“长连接 + 多路复用”这一个根**。

## 生产常见问题

**Q1:本地 / 测试环境 gRPC 跑得好好的,一上云(经过 ELB/SLB 或某些 API 网关)就频繁 `UNAVAILABLE` 或连接被重置,为什么?**

高度怀疑 **h2c(明文 HTTP/2)经过了不支持它的中间设备**。很多四层/七层 LB、老式代理对明文 HTTP/2 支持差,会把 h2c 长连接误判、降级或直接掐断。排查方向:① 确认链路上 LB/网关是否支持 HTTP/2(以及是 h2 还是 h2c);② 内网若用 h2c,优先选明确支持 gRPC/HTTP2 的 LB(如配置正确的 Envoy、支持 gRPC 的 ALB/网关);③ 跨越不可控网络时改用 **h2(TLS)**,中间设备对 TLS+ALPN 的兼容性通常更好。

**Q2:大消息或流式传输,带宽明明很富裕,吞吐却上不去、延迟还高,CPU 也不忙。**

经典 **HTTP/2 流控窗口 / BDP** 问题。默认初始窗口 64KB,在高带宽 × 高延迟(如跨地域)链路上,发送方发满 64KB 就得停下等对端 WINDOW_UPDATE,RTT 越大停得越久,带宽利用率越低。解法:① 确认所用 gRPC 实现/版本开启了 **BDP 自动调窗(auto-tuning)**;② 手动调大 `InitialWindowSize` / `InitialConnWindowSize`(grpc-go 的 `WithInitialWindowSize`、grpc-java 的 `flowControlWindow`);③ 检查链路上的代理(Envoy 等)是否也得同步调大窗口,否则瓶颈卡在代理。

**Q3:客户端 P99 延迟周期性尖刺,但服务端 CPU、GC、下游都正常,像是“凭空变慢”。**

很可能撞上了 **`SETTINGS_MAX_CONCURRENT_STREAMS`**:客户端往往**只对一个目标维持一条(或很少几条)连接**,当瞬时并发 RPC 数超过这条连接的并发流上限时,**多出来的 RPC 在客户端本地排队**,等前面的 stream 关闭才放行——表现就是“客户端测到慢、服务端看着闲”。解法:① 调大服务端 `MaxConcurrentStreams`;② 客户端开**多条连接 / 多 subchannel**分摊(第 06、11 篇);③ 监控连接上的活跃 stream 数(channelz,第 15 篇)。

**Q4:压测时给 gRPC 后端加机器、配了 K8s Service 做负载均衡,流量却几乎全压在一两个 Pod 上,扩容无效。**

**本章第十节的“头号坑”**。L4 负载均衡(K8s ClusterIP / 四层 LB)**只在 TCP 建连那一刻分发一次**,而 gRPC 是**少量长连接 + 多路复用**:连接建好后成百上千个 RPC 全复用同一条连接、打到同一个后端。连接又长寿、又少,于是严重倾斜。这是 HTTP/2 长连接的必然结果,不是 bug。解法是**客户端负载均衡 / headless Service / L7 代理或 mesh**——第 11、16 篇专门拆。先在这里记住根因:**坑出在“长连接 + 多路复用”,L4 LB 拦不住。**

**Q5:跨公网或弱网环境下,gRPC 偶发整条连接上所有请求一起卡顿几百毫秒,然后恢复。**

典型 **TCP 层队头阻塞**(本章第九节)。弱网丢包时,共享同一条 TCP 的所有 stream 都得等那个丢失的段重传,期间集体卡住。缓解:① 弱网/高丢包场景适当**增加连接数**,把 stream 分散到多条 TCP,降低单连接被一个丢包连累的范围;② 评估 gRPC over HTTP/3(QUIC)的可行性(目前多为实验性);③ 这本质是 TCP 的特性,应用层只能缓解、不能根治,根治要靠 QUIC。
