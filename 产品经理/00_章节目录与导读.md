# 后端工程师转产品经理 · 学习与作战手册

## 本手册的目标

这不是一本"产品经理岗位介绍",也不是"互联网黑话词典"。它是为**一个有 5 年后端经验、希望真正能做产品经理工作**的人写的系统化学习路径。

写作原则与 EMF 手册一脉相承:

- 每一章都讲清楚**为什么这样做**、**不这样做会发生什么**、**对应到具体的产出物 / 决策路径长什么样**。
- 拒绝"以用户为中心"、"小步快跑"、"价值闭环"这种缺乏可操作性的口号——所有方法论都要能落到一份 PRD、一次访谈、一个指标、一次评审上。
- 全程显式标记**"工程师视角的陷阱"**——后端工程师转 PM 有非常具体的认知盲区,这些盲区不是靠"努力换位思考"能消解的,需要刻意训练。
- 凡是涉及具体公司案例 / 行业惯例的数字与实践,会在文中标注**"经验值 / 业内通行做法"**,不会冒充科学定理。

## 一句话定位:产品经理到底是干什么的

> 产品经理是**在不确定的市场、不完整的信息、有限的资源**下,**定义"做什么以及为什么做"的人**——以及**承担这个定义带来的全部后果**的人。

这句话里有三个反直觉的点,后端工程师初学时往往会忽略:

1. **不确定性是 PM 的工作主体,而不是异常情况。** 工程师习惯把模糊需求"打回去澄清",PM 必须**在永远澄清不完的状态下做决定**。
2. **PM 的价值不是"想出点子",而是"在众多点子中选出该做的那一个,并说服团队去做"。** 创意是廉价的,排序与说服是稀缺的。
3. **承担后果**:一个功能上线没人用,工程师可能会说"PM 没想清楚",但 PM 不能说"工程师没做好"——结果归属是 PM 的边界。

## 阅读顺序建议

- **想先建立全局心智模型**:01 → 02 → 04 → 07 → 09。读完这五章,你就具备"看一个产品决策能说出门道"的能力。
- **目标内部转岗 / 已有现成产品要负责**:01 → 06 → 04 → 09 → 07。先解决"上手能跑起来"的问题。
- **目标跳槽 PM / 想拿一份 offer**:全章 + 13 章作品集 + 反复演练 12 章 AI 产品 case。
- **只想做 AI / Dev Tools / B 端产品**:01 → 02 → 11 → 12 → 08 → 13。后端背景在这条线上是巨大资产。

## 关于"后端思维 vs 产品思维":本手册的核心矛盾

整套手册的隐藏主线,是不断对齐**两套思维方式**的差异——这也是为什么它和一份普通的"产品经理入门书"读起来不一样:

| 维度 | 后端工程师默认思维 | 产品经理需要的思维 |
|------|------------------|------------------|
| 问题来源 | 收到需求,先问"是否合理" | 主动找问题,先问"是否值得做" |
| 解决方案 | 找最优解 | 找在约束下"最有价值的解"——常常不是最优的 |
| 不确定性 | 视为风险,要消除 | 视为常态,要管理 |
| 完成定义 | "通过测试 / 上线 / SLO 达标" | "用户用了 / 留存了 / 回头买了" |
| 沟通方式 | 准确、严谨,默认对方理解上下文 | 重复、对齐、调动情绪,假设对方没有上下文 |
| 决策依据 | 数据 + 规范 + 既有系统 | 数据 + 用户语言 + 商业判断 + 直觉 |
| 时间观 | 几个迭代(weeks) | 季度 / 半年 / 三年 |

这张表会在 §02 展开,并且每一章都会回头打卡:这一章训练的是哪一种"切换"。

## 关于本手册的几条立场(读之前请先看)

1. **没有唯一正确的产品经理画像。** 字节、阿里、腾讯、Google、Meta、SaaS 创业公司的 PM,日常工作可以差出 10 倍。本手册讲的是**底层能力**,不是某一家公司的岗位说明书。
2. **方法论本身没有价值,场景匹配才有价值。** 同一份 PRD 模板在小创业公司可能是过度工程,在大厂可能还不够细。每个方法论都会标注它的"适用半径"。
3. **AI 时代的 PM 正在被重新定义。** 12 章不是凑数的,是这套手册区别于 2020 年前 PM 教材的关键。后端工程师在这一波最有机会"换道超车"——不是因为更懂 LLM 原理,而是因为**对系统不确定性、可观测性、降级链路的本能**正好是 AI 产品最缺的产品经理素养。
4. **本手册接受被打脸。** 凡是写得过于绝对的论断,欢迎用真实工作中的反例反驳——反驳本身就是 §03 用户访谈的一种应用。

## 术语对照表(贯穿全手册)

| 术语 | 含义 | 后端工程师可类比的概念 |
|------|------|---------------------|
| PRD | Product Requirements Document,产品需求文档 | 类似详细设计文档(DDD),但读者是"未来的所有相关方",不只是研发 |
| MRD / BRD | 市场需求文档 / 商业需求文档 | 比 PRD 更上游,更接近"立项申请书" |
| MVP | Minimum Viable Product,最小可行产品 | 不是"砍版本",是"用最小代价验证假设" |
| PMF | Product-Market Fit,产品市场契合 | 大致对应"产品已经被市场接受,接下来该规模化",标志:留存曲线企稳 + 自然增长 + NPS 上升 |
| 北极星指标 | North Star Metric,衡量产品长期价值的核心指标 | 类似系统的"主 SLO",但不能等同于 SLO(§05 会专门讲) |
| AARRR | Acquisition / Activation / Retention / Revenue / Referral,海盗指标 | C 端产品的漏斗框架 |
| HEART | Happiness / Engagement / Adoption / Retention / Task success,Google 用户体验指标 | 偏体验类产品的细化框架 |
| OKR | Objectives and Key Results,目标与关键结果 | 不要等同于"团队的 SLO 表",§05 详述 |
| JTBD | Jobs To Be Done,用户雇用产品来"完成什么任务" | 区别于"用户画像"的另一种用户建模 |
| Persona | 用户画像 | 用结构化的方式描述目标用户 |
| Cohort | 同期群,按某个时间或属性切分的用户群 | 类似按"上线日期"分桶的用户群,做留存分析的基本单位 |
| LTV / CAC | 用户生命周期价值 / 获客成本 | 商业化必看比值,§08 详述 |
| Land & Expand | 先小范围切入,再扩大 | B 端经典打法 |
| GTM | Go-To-Market,市场进入策略 | 不仅是"上线",还包括渠道、定价、销售支持 |
