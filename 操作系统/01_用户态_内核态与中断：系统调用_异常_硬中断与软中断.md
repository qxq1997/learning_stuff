# 用户态、内核态与中断：系统调用、异常、硬中断与软中断

## 为什么要区分用户态和内核态

CPU 通常提供不同特权级。应用程序运行在用户态，操作系统内核运行在内核态。这个设计不是为了增加复杂度，而是为了建立一条硬边界：

- 用户态不能直接访问任意物理内存、设备寄存器、页表和调度器。
- 内核态可以执行特权指令，管理 CPU、内存、磁盘、网卡、文件系统和进程。
- 用户程序需要访问受保护资源时，通过系统调用进入内核，由内核检查参数、权限和资源状态。

如果没有这条边界，一个普通进程写坏页表、关中断、覆盖别的进程内存，就可能让整个系统崩掉。用户态把故障影响限制在进程内；内核态把硬件访问集中到受控路径里。

## 用户态进入内核态的几种路径

| 路径 | 触发方 | 典型例子 | 特点 |
| --- | --- | --- | --- |
| 系统调用 | 应用主动触发 | `read`、`write`、`mmap`、`fork`、`epoll_wait` | 用户程序请求内核服务 |
| 异常 | CPU 执行指令时触发 | 缺页异常、除零、非法指令、写保护异常 | 常常和当前指令有关 |
| 硬中断 | 外设触发 | 网卡收包、磁盘完成 I/O、时钟中断 | 和当前进程不一定有关 |
| 软中断/下半部 | 内核延后执行 | 网络协议栈收包处理、定时器回调 | 把中断中不宜做太久的工作延后 |

系统调用、异常、中断都会让 CPU 切到内核态，但语义不同：

- 系统调用是“我需要内核帮忙”。
- 异常是“当前执行出了一个需要内核处理的情况”。
- 中断是“外部事件来了，请内核响应一下”。

## 一次系统调用做了什么

以 `read(fd, buf, n)` 为例，粗略过程是：

1. 用户程序把系统调用号和参数放到约定位置。
2. CPU 执行专门的陷入指令，切换到内核态。
3. 内核根据系统调用号找到对应处理函数。
4. 内核检查 `fd` 是否有效、缓冲区地址是否可写、权限是否允许。
5. 如果数据未就绪，进程可能阻塞并触发调度；如果就绪，内核把数据拷到用户缓冲区。
6. 内核返回用户态，用户程序继续执行。

这里至少有两个成本：

- 模式切换成本：从用户态进入内核态，再返回用户态。
- 数据拷贝成本：普通文件或 socket I/O 往往需要在内核缓冲区和用户缓冲区之间复制数据。

所以高性能 I/O 会尽量减少系统调用次数、减少拷贝次数，或者让一次系统调用处理更多事件。

## 异常和缺页为什么不是坏事

异常不一定代表程序错误。缺页异常就是虚拟内存机制的正常组成部分。

当进程访问某个虚拟地址时，MMU 发现页表项不存在、没有物理页、权限不满足，就会触发异常进入内核。内核根据原因决定：

- 如果只是物理页尚未分配：分配物理页，更新页表，然后让进程继续。
- 如果是文件映射页不在内存：从磁盘读取页面。
- 如果是写时复制页被写入：复制物理页，改权限。
- 如果访问非法地址：发送信号，常见结果是进程崩溃。

也就是说，异常是 CPU 把“我解决不了的地址或指令问题”交给内核。缺页能让程序先拥有很大的虚拟地址空间，实际用到时再分配物理内存。

## 中断为什么要拆成上半部和下半部

硬中断处理应该越短越好，因为中断处理期间会影响系统响应其他事件。以网卡收包为例，硬中断里适合做：

- 确认设备事件。
- 把必要信息放进内核队列。
- 唤醒后续处理机制。

协议解析、唤醒 socket、把数据交给应用这些工作可能比较耗时，通常放到软中断或内核线程等延后路径中做。

这就是上半部/下半部的基本思想：

- 上半部：快速响应硬件，少做事。
- 下半部：延后处理复杂逻辑，降低中断关闭时间。

## 时钟中断和调度

时钟中断是抢占式多任务的基础之一。操作系统会周期性获得 CPU 控制权，检查当前任务是否用完时间片、是否有更高优先级任务就绪。

如果需要切换任务，内核会保存当前任务的 CPU 上下文，再恢复下一个任务的上下文。于是即使单核 CPU 同一时刻只能跑一个执行流，用户也会感觉多个进程“同时”运行。

## 面试常见追问

**为什么系统调用比普通函数调用慢？**

普通函数调用只在用户态内跳转，主要是压栈、跳转、返回。系统调用要切换特权级、进入内核入口、检查参数和权限，可能还要调度、阻塞、拷贝数据，因此更重。

**用户态程序崩溃为什么一般不会拖垮系统？**

因为每个进程有独立虚拟地址空间，页表和权限位阻止它访问别的进程或内核的敏感内存。崩溃后内核回收该进程资源即可。

**内核态代码出错为什么更危险？**

内核态拥有全局资源和硬件访问能力。一旦内核越界写、死循环或破坏关键结构，影响的可能是整个系统，而不是单个进程。
