# CPU 调度与死锁：FCFS、SJF、HRRN、RR、多级反馈队列与死锁处理

## 调度器在解决什么问题

系统里就绪任务数量常常大于 CPU 核心数。调度器要决定：

- 谁先运行。
- 每次运行多久。
- 运行中是否可以被抢占。
- 如何兼顾吞吐、响应时间、公平性和优先级。

不同调度算法本质是在这些目标之间取舍。

## 常见 CPU 调度算法

| 算法 | 核心规则 | 优点 | 缺点 | 适合回答时强调 |
| --- | --- | --- | --- | --- |
| FCFS | 先来先服务 | 简单、公平直观 | 长作业会拖慢后续短作业，产生护航效应 | 非抢占，适合批处理但响应差 |
| SJF | 优先运行预计时间最短的任务 | 平均等待时间可能很低 | 很难准确知道运行时长，长作业可能饥饿 | 对短作业友好，对长作业不公平 |
| HRRN | 响应比最高者优先 | 兼顾等待时间和服务时间 | 仍需估算服务时间 | 等待越久优先级越高 |
| RR | 时间片轮转 | 公平，适合交互式系统 | 时间片太短切换开销大，太长退化为 FCFS | 时间片大小是关键 |
| 优先级调度 | 优先级高者先运行 | 可表达业务重要性 | 低优先级可能饥饿 | 可用动态优先级/老化缓解 |
| 多级反馈队列 | 高优先级短时间片，运行不完逐级下沉 | 兼顾交互和长任务 | 参数复杂，调优难 | 现代调度思想的重要来源 |

## HRRN 的公式

高响应比优先的响应比：

```text
响应比 = (等待时间 + 要求服务时间) / 要求服务时间
```

它有两个效果：

- 服务时间短的任务响应比天然较高，所以短作业容易先运行。
- 长作业等待越久，响应比越高，最终也能获得运行机会。

因此 HRRN 比 FCFS 更照顾短作业，又比 SJF 更不容易让长作业饿死。

## 时间片轮转的核心取舍

时间片太短：

- 上下文切换次数多。
- CPU cache 被频繁打乱。
- 真正执行业务代码的比例下降。

时间片太长：

- 交互任务响应变差。
- 算法更接近先来先服务。

所以 RR 的关键不是“轮流”两个字，而是时间片长度和上下文切换成本之间的折中。

## 多级反馈队列怎么工作

多级反馈队列通常有几个规则：

1. 新任务先进入高优先级队列。
2. 高优先级队列时间片短，低优先级队列时间片长。
3. 任务在当前时间片内没跑完，会被下放到更低优先级队列。
4. 高优先级队列不为空时，低优先级队列不运行。
5. 为防止饥饿，等待太久的低优先级任务可以被提升。

它的直觉是：

- 交互式短任务通常很快让出 CPU，留在高优先级，响应快。
- CPU 密集长任务会逐级下沉，但每次获得更长时间片，减少切换开销。

## 并发为什么需要锁

多线程共享地址空间。如果多个线程同时读写同一份共享数据，而中间操作不是原子的，就可能出现竞态条件。

例如 `count++` 看起来是一条语句，底层可能是：

1. 从内存读 `count` 到寄存器。
2. 寄存器加一。
3. 写回内存。

两个线程交错执行，就可能丢更新。锁的作用是让临界区同一时刻只被一个执行流进入。

## 常见锁与同步工具

| 工具 | 语义 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| 互斥锁 | 只有一个线程能进入临界区 | 临界区可能较长，竞争时愿意睡眠 | 阻塞和唤醒有上下文切换成本 |
| 自旋锁 | 获取不到锁就忙等 | 临界区极短、多核、不能睡眠的路径 | 持锁久会浪费 CPU |
| 读写锁 | 多读共享，写独占 | 读多写少 | 写饥饿或锁升级复杂 |
| 条件变量 | 等待某条件成立并被唤醒 | 生产者消费者、状态变化通知 | 必须搭配互斥锁，注意虚假唤醒 |
| 信号量 | 计数型资源许可 | 限流、连接池、生产消费协作 | 容易忘记释放或释放次数错误 |

## 死锁的四个必要条件

死锁必须同时满足：

1. 互斥：资源不能被多个线程同时占用。
2. 持有并等待：线程持有一部分资源，同时等待新资源。
3. 不可剥夺：资源不能被外部强制抢走，只能由持有者释放。
4. 循环等待：多个线程形成环形资源等待关系。

破坏任意一个条件，都能避免死锁。工程里最常见的是破坏循环等待：给资源编号，所有线程按同一顺序申请锁。

## 死锁处理策略

| 策略 | 思路 | 典型方式 |
| --- | --- | --- |
| 预防 | 破坏死锁必要条件 | 一次性申请全部资源、锁排序、允许抢占 |
| 避免 | 分配前判断系统是否仍处于安全状态 | 银行家算法 |
| 检测 | 允许死锁发生，周期性检查等待图 | 数据库死锁检测、线程 dump 分析 |
| 恢复 | 发现死锁后解除 | 回滚事务、杀进程、超时释放 |

## 银行家算法的核心

银行家算法关注“系统如果现在分配资源，会不会进入不安全状态”。它维护几类信息：

- Available：当前可用资源数。
- Max：每个进程最大需求。
- Allocation：已分配给每个进程的资源。
- Need：还需要多少资源才能完成，`Need = Max - Allocation`。

如果存在一个执行序列，让每个进程都能在当前和未来释放的资源帮助下完成，那么系统处于安全状态。否则，当前分配请求应被拒绝或延迟。

银行家算法适合作为理解模型，但在通用操作系统中很难要求所有进程提前声明最大资源需求。真实工程里更常见的是锁顺序、超时、检测、事务回滚和隔离拆分。

## 排查死锁的思路

1. 先确认卡在哪里：线程状态、阻塞栈、等待锁、等待 I/O。
2. 找等待关系：线程 A 持有哪些锁，等待哪些锁；线程 B 同理。
3. 判断是否成环：A 等 B，B 等 C，C 等 A。
4. 看是否有外部资源：数据库锁、分布式锁、文件锁、远程调用也可能参与死锁。
5. 修复时优先统一锁顺序、缩小临界区、加超时、避免持锁调用外部服务。
