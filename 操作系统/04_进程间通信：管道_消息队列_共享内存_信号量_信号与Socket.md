# 进程间通信：管道、消息队列、共享内存、信号量、信号与 Socket

## IPC 在解决什么问题

进程之间默认隔离：一个进程不能直接读写另一个进程的地址空间。这提升了安全性和稳定性，但也带来协作问题。

进程间通信要解决三件事：

- 数据怎么传。
- 谁来同步。
- 故障边界和权限怎么控制。

## 常见 IPC 对比

| 方式 | 数据形态 | 是否经过内核 | 适用场景 | 主要限制 |
| --- | --- | --- | --- | --- |
| 匿名管道 | 字节流 | 是 | 父子进程单向通信，shell 管道 | 半双工、无结构、容量有限 |
| 命名管道 | 字节流 | 是 | 无亲缘关系进程通信 | 仍是流式，通常不适合复杂协议 |
| 消息队列 | 消息体 | 是 | 结构化消息、异步解耦 | 拷贝成本和内核队列限制 |
| 共享内存 | 内存读写 | 建立映射后少走内核 | 大量数据、高吞吐 | 必须额外同步 |
| 信号量 | 计数 | 是 | 互斥、同步、资源计数 | 不传业务数据 |
| 信号 | 事件编号 | 是 | 异步通知、进程控制 | 信息量小，处理模型受限 |
| Socket | 字节流或报文 | 是 | 跨主机、本机通用通信 | 协议栈和拷贝成本更高 |

## 管道：最朴素的流式通信

匿名管道通常只适合有亲缘关系的进程，比如父子进程。shell 中的：

```bash
ps aux | grep redis
```

就是把前一个进程的输出接到后一个进程的输入。

管道的特点：

- 数据是无格式字节流，读写双方要约定协议。
- 通常是单向通信，双向要建两个管道。
- 内核维护缓冲区，写端写入，读端读出。
- 数据遵循先进先出。

命名管道把管道暴露为文件系统中的特殊文件，使无亲缘关系进程也能通信。

## 消息队列：给数据加边界

管道是字节流，没有天然消息边界。消息队列则以一个个消息为单位，发送方写入消息，接收方按类型或顺序读取。

优点：

- 消息有边界，适合传结构化数据。
- 可异步解耦发送方和接收方。

缺点：

- 数据仍要在用户态和内核态之间拷贝。
- 队列容量有限。
- 大消息或高频消息场景可能不如共享内存。

## 共享内存：最快但最危险

共享内存的核心是：多个进程把各自虚拟地址空间中的一段地址映射到同一片物理内存。

这样进程 A 写入共享区域，进程 B 立刻能读到，不需要通过内核缓冲区反复拷贝。它常用于大数据量、高性能通信。

但共享内存只解决“数据在哪里”，不解决“谁先读写”。如果没有同步机制，多个进程同时写同一块内存，会产生数据竞争。

常见组合：

- 共享内存 + 信号量。
- 共享内存 + 互斥锁。
- 共享内存 + 环形队列。
- 共享内存 + futex。

## 信号量和信号不是一回事

信号量是同步工具，通常表示资源数量。经典 P/V 操作：

- P：申请资源，计数减一；如果不够，等待。
- V：释放资源，计数加一；如有等待者，唤醒。

信号是异步通知机制，比如 `SIGINT`、`SIGTERM`、`SIGKILL`。进程收到信号后可以：

- 执行默认动作。
- 捕获并执行处理函数。
- 忽略信号。

`SIGKILL` 和 `SIGSTOP` 不能被捕获或忽略，这是为了保证系统总有办法终止或暂停进程。

## Socket：统一本机和跨主机通信

Socket 不只用于网络。Unix Domain Socket 也可以用于同一台机器上的进程通信，少走完整网络协议栈，常用于本机服务间通信。

Socket 的优势：

- 跨机器通信能力强。
- 语义统一，生态成熟。
- 可基于 TCP 提供可靠字节流，基于 UDP 提供报文。

代价：

- 比共享内存更重。
- 要处理连接、缓冲区、网络异常、协议边界。

## 线程间通信和进程间通信的边界

线程在同一进程内共享地址空间，所以线程间通信通常不需要 IPC，而是靠共享变量加同步机制：

- 互斥锁保护临界区。
- 条件变量做等待和通知。
- 信号量控制资源数量。
- 队列封装生产者消费者模型。

进程间通信更强调隔离和边界，需要显式通信机制。线程间通信更强调数据竞争控制。

## 面试选型口诀

- 父子进程简单串流：管道。
- 无亲缘关系简单串流：命名管道。
- 小消息、结构化、异步：消息队列。
- 大数据、高吞吐：共享内存加同步。
- 只通知事件：信号。
- 控制并发资源：信号量。
- 跨机器或通用本机通信：Socket。
