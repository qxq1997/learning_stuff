# 页面置换算法：OPT、FIFO、LRU、Clock 与工作集

## 页面置换在解决什么问题

当进程访问一个不在物理内存中的页面时，会发生缺页异常。如果此时没有足够空闲物理页，内核必须选择一个现有页面换出，再把需要的页面换入。

页面置换算法的目标是：

- 尽量减少未来缺页次数。
- 尽量避免频繁换入换出。
- 兼顾实现成本。

理想算法不一定能落地，真实系统常用近似算法。

## 常见算法对比

| 算法 | 选择淘汰谁 | 优点 | 缺点 |
| --- | --- | --- | --- |
| OPT | 未来最长时间不会被访问的页 | 理论最优，可做基准 | 无法预知未来，不能实际实现 |
| FIFO | 最早进入内存的页 | 简单 | 可能淘汰热点页，有 Belady 异常 |
| LRU | 最近最久未使用的页 | 符合局部性原理 | 精确实现成本高 |
| Clock | 引用位为 0 的页 | 近似 LRU，成本低 | 精度不如 LRU |
| LFU | 访问次数最少的页 | 适合频次稳定场景 | 历史热点可能污染，需衰减 |

## OPT：只能当理想上界

OPT 会淘汰未来最长时间不会被访问的页面。例如页面序列为：

```text
7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2
```

如果只有 3 个物理页框，OPT 会尽量保留未来快要访问的页。它的缺页次数最少，但真实系统不知道未来访问序列。

所以 OPT 的用途是衡量其他算法：越接近 OPT，说明置换效果越好。

## FIFO：简单但可能很糟

FIFO 淘汰最早进入内存的页。它不关心页面最近是否被频繁访问。

问题是：一个很早进入内存的页面可能一直是热点，FIFO 也会淘汰它。更反直觉的是，FIFO 会出现 Belady 异常：增加物理页框数量，缺页次数反而可能上升。

这说明“更大缓存一定更好”不总成立，算法也很关键。

## LRU：用历史预测未来

LRU 的思路是：如果一个页面很久没被访问，它未来短期内大概率也不会被访问。

它利用的是程序局部性：

- 时间局部性：刚访问过的数据，近期可能再次访问。
- 空间局部性：访问某地址后，附近地址也可能被访问。

精确 LRU 通常需要维护访问顺序。每次访问都更新链表或时间戳，在高频内存访问路径上成本太高。因此操作系统常用近似 LRU，而不是严格 LRU。

## Clock：低成本近似 LRU

Clock 算法为每个页维护访问位，页面组织成环。

淘汰过程：

1. 指针检查当前页。
2. 如果访问位为 1，清零并跳过，给它第二次机会。
3. 如果访问位为 0，淘汰该页。
4. 指针继续向前转。

它不像 LRU 那样维护精确最近访问顺序，但成本更低，能避免直接淘汰刚被访问过的页面。

## 工作集和内存抖动

工作集指进程在一段时间内活跃访问的页面集合。如果物理内存不足以容纳工作集，进程会频繁缺页：

- 刚换入一个页，又淘汰了马上要用的页。
- CPU 大量时间花在等待磁盘换页。
- 系统吞吐急剧下降。

这就是内存抖动。它不是单个页面置换策略的小问题，而是“活跃数据集大于可用内存”的系统性问题。

常见缓解手段：

- 减少并发进程数量。
- 增加内存或降低工作集大小。
- 避免无界缓存。
- 监控 major fault、swap in/out、直接回收、OOM。
- 针对热点数据做更好的访问局部性设计。

## 页面置换和缓存淘汰的关系

页面置换和 Redis 缓存淘汰、数据库 Buffer Pool 淘汰很像，都在回答：

> 空间不够时，淘汰谁最不影响未来命中率？

差别在于：

- OS 页面置换面对的是虚拟内存页，缺页可能触发磁盘 I/O。
- 应用缓存淘汰面对的是业务对象，通常还要考虑 TTL、成本、热点、容量和一致性。

理解页面置换能帮助理解很多系统里的 LRU、Clock、二次机会、active/inactive list。
