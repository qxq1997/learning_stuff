# 零拷贝：read/write、mmap、sendfile 与 DMA

## 普通文件传输为什么慢

以“从磁盘读文件，再通过 socket 发出去”为例，传统 `read` + `write` 路径大致是：

```text
磁盘 -> 内核页缓存 -> 用户缓冲区 -> socket 缓冲区 -> 网卡
```

其中：

- 磁盘到内核页缓存：DMA 拷贝。
- 内核页缓存到用户缓冲区：CPU 拷贝。
- 用户缓冲区到 socket 缓冲区：CPU 拷贝。
- socket 缓冲区到网卡：DMA 拷贝。

还会有系统调用导致的用户态/内核态切换：

- `read` 进入内核并返回。
- `write` 进入内核并返回。

所以普通路径的主要成本是：多次上下文切换和多次数据拷贝，尤其是两次 CPU 拷贝。

## 零拷贝的“零”是什么意思

零拷贝不是一次拷贝都没有，而是尽量避免数据在用户态和内核态之间来回复制，减少 CPU 参与的数据搬运。

更准确地说：

- 用户态不再拿到完整文件数据。
- 数据尽量在内核缓冲区之间流转。
- DMA 负责设备和内存之间的数据搬运。

这样 CPU 可以少做内存复制，把时间留给协议处理和业务逻辑。

## mmap + write

`mmap` 可以把文件映射到进程虚拟地址空间。应用访问映射区域时，页面由内核按需加载到页缓存。

传输路径变成：

```text
磁盘 -> 内核页缓存
用户态通过映射访问页缓存
内核页缓存 -> socket 缓冲区 -> 网卡
```

它减少了 `read` 从内核缓冲区拷到用户缓冲区这一步，但 `write` 仍可能把数据从页缓存拷到 socket 缓冲区。

适合：

- 文件随机访问。
- 多进程共享文件映射。
- 需要用户态读取或处理文件内容。

风险：

- 缺页会引入不可预测延迟。
- 写映射需要处理一致性和刷盘语义。
- 文件过大或访问模式差时可能带来内存压力。

## sendfile

`sendfile` 把“读文件”和“写 socket”合并到一个系统调用里。应用只告诉内核：

> 把这个文件的某段内容发到这个 socket。

数据路径大致是：

```text
磁盘 -> 内核页缓存 -> socket 缓冲区 -> 网卡
```

优势：

- 系统调用次数减少。
- 数据不需要完整经过用户态。
- 减少用户态和内核态之间的 CPU 拷贝。

如果硬件支持 scatter-gather DMA，内核还可以让网卡直接从页缓存相关内存区域读取数据，只传递描述符和长度，进一步减少内核内部拷贝。

## splice 和管道

Linux 还提供 `splice`，可以在文件描述符之间移动数据，常借助管道作为中转。它适合构建更通用的内核态数据转发路径。

但工程中更常见的面试重点仍是：

- 普通 `read/write` 有哪些拷贝。
- `mmap` 减少了哪次拷贝。
- `sendfile` 为什么适合静态文件传输。
- DMA 在其中扮演什么角色。

## 零拷贝的限制

零拷贝适合“数据从文件到网络，不需要用户态修改内容”的场景，例如静态文件服务、消息日志文件传输。

它不适合或会复杂化的场景：

- 数据需要在用户态压缩、加密、过滤、改写。
- 传输链路需要应用层逐字节处理。
- TLS 在用户态实现时，明文数据往往仍要进入用户态加密。
- 小文件很多时，系统调用、元数据和网络包开销可能比拷贝更重要。

所以零拷贝不是万能优化，而是针对特定数据路径减少无意义搬运。

## 和 Page Cache 的关系

无论普通 I/O 还是 `sendfile`，文件数据通常都会经过页缓存。页缓存的价值是：

- 多次读取同一文件时减少磁盘 I/O。
- 让内核统一管理文件缓存和回收。
- 支持 `mmap`、`sendfile` 等机制复用同一批缓存页。

这也是 Kafka、Nginx 等系统能高效发送文件或日志段的重要原因之一：顺序写、页缓存、批量发送、零拷贝组合起来，CPU 不必反复搬运同一份数据。

## 面试回答模板

解释零拷贝可以按三步：

1. 普通路径：磁盘到页缓存、页缓存到用户缓冲区、用户缓冲区到 socket 缓冲区、socket 缓冲区到网卡。
2. 优化点：减少系统调用，减少用户态/内核态之间的数据拷贝，让 DMA 搬设备数据。
3. 边界：适合文件到网络的直通传输，不适合需要用户态改写内容的场景。
