# 消息队列专题学习路径

## 学习定位

这个专题不是再背一遍“解耦、异步、削峰”三件套。真正难的是：当系统引入 MQ 之后，调用关系、可靠性边界、一致性语义、失败恢复方式都会变。

已有内容里：

- `系统设计/05_消息队列、异步化与最终一致性.md` 已经从系统设计角度讲了异步边界、Outbox、最终一致。
- `Kafka/` 目录已经在深入拆 Kafka 的日志模型、Partition、PageCache、ISR、KRaft 等底层机制。

所以这个 `消息队列/` 专题补的是更通用的一层：**如何从 MQ 这个基础设施本身出发，理解消息模型、可靠性、重复消费、顺序、积压、事务、延迟、死信、以及 Kafka / RocketMQ / RabbitMQ 的选型边界**。

## 知识图谱

### 第 1 层：心智模型

- MQ 的三段链路：生产者、Broker、消费者
- 队列模型、日志模型、发布订阅模型分别在解决什么问题
- 观察者模式和发布订阅模式的区别
- push 与 pull 的差异
- “消息”到底是命令、事件，还是任务

### 第 2 层：可靠性

- 生产者如何确认消息已经被 Broker 接收
- Broker 如何通过持久化、副本、刷盘、主从复制避免消息丢失
- 消费者为什么必须“处理成功后再 ACK / commit offset”
- 至少一次、至多一次、精确一次分别在什么边界内成立

### 第 3 层：消费正确性

- 为什么重复消费不可避免
- 消费幂等怎么设计：唯一键、状态机、去重表、乐观锁、业务流水
- 顺序消息的真实边界：全局有序、分区有序、同 key 有序
- 消息积压的排查与应急扩容

### 第 4 层：高级能力

- 事务消息：本地事务和消息发送的一致性
- RocketMQ 半消息与事务回查
- Outbox / CDC 和事务消息的关系
- 延迟消息：时间轮、延迟 Topic、调度服务、RabbitMQ TTL + DLX
- 死信队列：失败兜底、人工修复、重试治理

### 第 5 层：主流 MQ 选型

- Kafka：分布式日志、高吞吐、可重放、数据流
- RocketMQ：业务消息、事务消息、延迟消息、海量 Topic
- RabbitMQ：AMQP、Exchange、Routing Key、Binding、灵活路由
- 选型不是问“哪个最好”，而是问“你的主矛盾是什么”

## 学完后应该能回答的问题

- 为什么 MQ 能解耦，但也会降低系统整体可用性？
- 消息从生产到消费，哪些位置会丢？每一段怎么兜？
- 为什么“消费端幂等”是 MQ 系统的必修课，而不是可选项？
- Kafka 为什么用 offset commit，RabbitMQ / RocketMQ 为什么更像 ACK 签收？
- 延迟队列和死信队列各自解决什么问题，订单超时取消为什么常把两者放一起？
- 事务消息到底保证了什么，不保证什么？
- 面试问“设计一个 MQ”时，应该从 RPC、存储、消费关系、可靠性、高可用、扩展性哪些维度展开？
