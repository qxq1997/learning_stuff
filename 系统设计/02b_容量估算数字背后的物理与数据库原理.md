# 系统设计 - 第 2 课补充：容量估算数字背后的物理与数据库原理

## 学习目标（本节结束后你能做到什么）

1. 理解 [02](./02_容量估算与性能指标.md) 和 [02h](./02h_容量估算指标到技术选型逐项拆解.md) 那张判断表里的数字，不是“背下来的经验线”，而是从硬件和数据库原理里推导出来的临界点。
2. 能把每个阈值还原到四大物理资源之一：磁盘 IO、内存、网络、CPU/锁。
3. 能在面试里多说一句“这个数从哪来”，并说明它在什么条件下会变（机械盘换 SSD、行变宽、内存变大、网卡升级）。
4. 避免两种极端：一种是只会背“2000 万行要分表”，另一种是张口就“看情况，没有固定数字”。正确姿态是：给出量级 + 说清它的物理来源 + 说清它的浮动条件。

## 一、先记住四组物理常数

所有阈值最终都落在四种资源上。先把基准值刻进脑子，后面所有推导都引用它们。

### 1. 网络

| 项目 | 数值 | 备注 |
| --- | --- | --- |
| 1 Gbps 网卡 | 125 MB/s | 注意 Gbps 是 bit，除以 8 才是 Byte |
| 10 Gbps 网卡 | 1.25 GB/s | 目前服务器主流 |
| 25 / 100 Gbps 网卡 | 3.1 / 12.5 GB/s | 高端/存储节点 |
| 实际可用带宽 | 标称的 70%-90% | 协议头、重传、中断处理有损耗 |
| 同机房 RTT | 0.1 - 0.5 ms | 一次往返 |
| 同城跨可用区 RTT | 1 - 2 ms | |
| 跨区域（国内）RTT | 10 - 40 ms | |
| 跨洲 RTT | 100 - 300 ms | 受光速限制，无法靠堆机器解决 |
| 单条 TCP 连接内核缓冲 | 默认读写各几十 KB，可调到 4 - 16 KB | 决定百万连接吃多少内存 |

一个最常被记错的换算：**带宽单位 Gbps 是比特，存储/响应体单位 MB 是字节，差 8 倍。** 算出口带宽时如果忘了除 8，会高估一个数量级。

### 2. 磁盘 IO

| 介质 | 随机 IOPS | 顺序吞吐 | 单次 fsync 延迟 |
| --- | --- | --- | --- |
| 机械盘 HDD | 100 - 200 | 100 - 200 MB/s | 几 ms - 10 ms |
| SATA SSD | 几万 | ~500 MB/s | 0.1 - 1 ms |
| NVMe SSD | 几十万 - 百万 | 2 - 7 GB/s | < 0.1 ms |

这里有两个决定性事实：

- **机械盘随机 IOPS 只有一两百。** 因为一次随机读要寻道（机械臂移动 ~10ms）+ 旋转延迟。这是“数据装不进内存、查询要打到磁盘”时最致命的天花板——再多 CPU 也没用，磁盘一秒就给你一两百次随机读。
- **顺序 vs 随机，机械盘差 100 倍以上，SSD 差距小但仍在。** 这就是为什么“顺序写日志（append-only）”是高吞吐写入的万能解，而“随机更新二级索引”是写入瓶颈的常见元凶。

### 3. 内存

| 项目 | 数值 |
| --- | --- |
| 单机内存常见规格 | 64 / 128 / 256 GB，高端 TB 级 |
| 内存访问延迟 | ~100 ns |
| 内存带宽 | 10 - 100 GB/s |
| DB buffer pool 一般配 | 物理内存的 50% - 75% |
| Redis 单实例建议上限 | 几十 GB（受 fork、RDB/AOF rewrite、主从全量同步制约） |

内存是“热数据能不能驻留”的总开关。一旦工作集（常被访问的数据 + 索引）超过内存，命中率掉下来，请求就开始打磁盘，于是磁盘 IOPS 成为新瓶颈。**很多“数据量”阈值，本质是在问“还装得进内存吗”。**

### 4. CPU / 锁 / syscall + Jeff Dean 延迟表

| 操作 | 量级 | 相对内存的倍数 |
| --- | --- | --- |
| L1 缓存访问 | ~1 ns | 0.01x |
| 分支预测失败 | ~3 ns | |
| 互斥锁 lock/unlock | ~17 ns | |
| 主存访问 | ~100 ns | 1x（基准） |
| 一次系统调用 | 几百 ns - 1 μs | ~10x |
| 线程上下文切换 | 1 - 5 μs | ~30x |
| SSD 随机读 1 次 | ~16 μs | ~160x |
| 同机房网络往返 | ~500 μs（0.5 ms） | ~5000x |
| 机械盘寻道 1 次 | ~10 ms | ~100000x |
| 跨洲往返 | ~150 ms | ~1500000x |

这张表（业界俗称 “Latency Numbers Every Programmer Should Know”，Jeff Dean 早年版本 + 后人按现代硬件校准）是**所有延迟预算的物理底座**。记住一个直觉：内存访问是 1，一次同机房网络往返是 5000，一次机械盘寻道是 10 万。所以“能在内存里算完就别走网络，能走一次网络就别走磁盘”。

> Redis 单实例补一条：单线程，简单命令（GET/SET）单实例约 **8 万 - 12 万 QPS**。这个数后面会反复用到。

## 二、逐个阈值的物理导出

下面把 02/02h 判断表里的每条线，逐个还原到上面的物理常数。每条都按“**这个数 = 哪个物理量算出来的 → 不这样会怎样 → 什么时候这个数会变**”来讲。

### 1. 写 QPS：100/s 开始注意，10k/s 要削峰或分区

**物理来源：fsync 延迟 + 二级索引随机 IO。**

一笔写事务要落盘才算真正提交，核心成本是 redo log 的 `fsync`。

```text
单线程、无组提交时：
写 TPS 上限 ≈ 1 / fsync延迟
机械盘 fsync ~5ms  -> 单线程 ~200 TPS
SSD    fsync ~0.5ms -> 单线程 ~2000 TPS
```

现代数据库有 **组提交（group commit）**：把多个并发事务的 redo 攒一批一次 fsync，所以并发够高时单机 SSD 能到几万 TPS。但还有第二个成本压着——**每多一个二级索引，写入就多一次随机 IO 和可能的页分裂**。一张表五六个二级索引，写放大就把“几万”打回“几千”。

所以：
- `100/s` 这条线：单机随便扛，但开始要意识到“每个索引、每次更新、每个触发器都在加 fsync 和随机写成本”。
- `10k/s` 这条线：已经逼近单机 SSD + 组提交的舒适区上限，再上去就得 **削峰（队列攒批）、批量写、分库分表、或改成顺序写的 append-only 结构（LSM）**。

**什么时候这个数会变：** 机械盘 → SSD → NVMe，每跳一档上限涨约一个数量级；表上二级索引越多、行越宽、越是随机主键（如 UUID 导致页分裂），这条线越往下掉。

### 2. 读 QPS：1k/s 开始认真，50k/s 要专门优化

**物理来源：buffer pool 命中率，决定走内存还是走磁盘。**

读 QPS 的天花板是分裂的，取决于数据在不在内存：

```text
命中 buffer pool（走内存）：一次点查 ~微秒级 CPU+内存
  -> 单机几万到十几万 QPS
未命中（走磁盘随机读）：
  机械盘 ~100-200 IOPS  -> 读 QPS 被摁死在几百
  SSD    ~几万 IOPS     -> 还能撑几万
```

这就是为什么同一句 SQL，有人压测出 10 万 QPS，有人只有几百——差别在工作集装没装进内存。

- `1k/s` 这条线：如果是命中内存的简单点查，单机毫无压力；这条线真正的意义是提醒你检查 **有没有 JOIN、聚合、回表、未命中索引**，这些会让单次读的成本从“一次内存查”变成“多次随机读 + CPU 排序”。
- `50k/s` 这条线：超过单机数据库舒适区，必须把读引到 **缓存（Redis 单实例约 10 万 QPS）、只读副本、读模型/物化视图、CDN**。

**什么时候这个数会变：** 工作集相对内存越小，命中率越高，这条线越高；机械盘换 SSD，未命中时的下限从几百抬到几万，整张表的读容量天花板都跟着抬。**很多“老经验值偏保守”，就是因为它们诞生在机械盘时代。**

### 3. 单表行数：2000 万行注意，10 亿行必分片

**物理来源：B+ 树层数 + 索引能否驻留内存。** 这是最经典、最值得手算一遍的导出。

InnoDB 一个数据页 16 KB。B+ 树的非叶节点只存「索引键 + 指向下层的指针」：

```text
非叶节点一项 ≈ 主键(BIGINT 8B) + 页指针(6B) ≈ 14B
一个 16KB 页能放的指针数 ≈ 16384 / 14 ≈ 1170

叶子节点存真实行。假设一行 1KB：
一个 16KB 页能放的行数 ≈ 16

那么不同层数能撑的行数：
2 层：1170 × 16            ≈ 1.9 万行
3 层：1170 × 1170 × 16     ≈ 2190 万行   <- “2000 万”这条线
4 层：1170 × 1170 × 1170 × 16 ≈ 256 亿行
```

看到了吗？“2000 万行”不是玄学，它就是 **B+ 树从 3 层涨到 4 层的临界点**。意义是：树每多一层，一次点查最坏就多一次磁盘 IO（如果该层节点不在内存）。

但这里有个更深的点，恰恰是面试加分项：**如果所有非叶节点（即索引）都能驻留在 buffer pool 里，层数多一层其实没那么可怕——真正打磁盘的只有最后一跳取叶子行。** 所以严格说，“2000 万行性能下降”的真正判据不是行数本身，而是：

```text
索引大小 > buffer pool？ -> 索引都装不进内存，每次查询多次打磁盘 -> 真的慢
索引大小 < buffer pool？ -> 4 层也还好，慢的只是写入维护成本
```

- `2000 万` 这条线：到这个量级该认真评估索引内存占用、写入维护成本、单表备份/DDL 时间，开始规划分区或分片。
- `10 亿+` 这条线：4 层 B+ 树、索引大概率超内存、单表 DDL 和备份痛苦，必须水平分片。

**什么时候这个数会变（这正是“背数据”和“懂原理”的分界）：**

```text
行宽 1KB  -> 叶子 16 行/页  -> 3 层 ≈ 2190 万
行宽 4KB  -> 叶子 4 行/页   -> 3 层 ≈ 550 万   （行变胖，线下移）
行宽 100B -> 叶子 160 行/页 -> 3 层 ≈ 2.2 亿  （行变瘦，线上移）
```

所以正确的说法是“**千字节行宽下大约两千万**”，而不是“一律两千万”。行里塞了大 JSON/TEXT 的表，这条线会塌到几百万。

> 这里讲的是“为什么到这条线该考虑分片”的物理原因；真正要分片时，分片键怎么选、跨分片查询/事务、扩容重平衡，见 [04 数据库、索引、分片与复制](./04_数据库、索引、分片与复制.md) 第四到六节。

### 4. 在线热数据：100GB 注意，10TB 必分片

**物理来源：单机内存容量。**

- `100GB` 这条线：正好卡在单机内存量级（128/256GB）。热数据小于 100GB 时，buffer pool 基本能把工作集兜住，命中率高、磁盘压力小，单机能扛。
- `10TB` 这条线：远超任何单机内存和单盘舒适区，命中率必然崩，必须水平分片把数据和内存一起摊开。

本质上，“热数据多大”这个指标是在问“**你的工作集还能不能被一台机器的内存罩住**”。罩得住，单机加缓存就行；罩不住，再多优化也救不了命中率，只能分片。

**什么时候这个数会变：** 服务器内存从 128GB 走向 512GB/1TB 常态化，这条单机分界线就往上抬；反过来，如果访问很分散（没有热点、几乎全量随机访问），就算总量不大，内存也兜不住，这条线要下调。

### 5. 日新增数据：10GB/day 规划分区，1TB/day 必须冷热分层

**物理来源：累积总量撞上单表行数/单机内存上限的速度。**

这条线本身是“速度”，要换算成“多久撞墙”：

```text
10GB/day × 365 ≈ 3.6 TB/年   -> 一年就超单机内存，要按时间分区
1TB/day × 365  ≈ 365 TB/年   -> 几天就超单盘，必须冷热分层 + 归档
```

它和第 3、4 条是联动的：日新增决定了你 **多快** 会撞到“2000 万行”或“100GB 热数据”那两堵墙。增长快，就要提前按时间分区（天/月）、设 TTL、把冷数据归档到对象存储/数据湖，别让在线表无限膨胀。

### 6. 并发连接数：10 万注意，100 万要平台化

**物理来源：内存（每连接 socket buffer）+ CPU（上下文切换、心跳唤醒）。** 这就是经典的 C10K → C10M 问题。

两个成本：

```text
内存：每条 TCP 连接内核读写缓冲默认几十 KB
  100 万连接 × 200KB = 200 GB   <- 不可行！
  必须把每连接缓冲调小到 8KB：100 万 × 8KB = 8 GB  <- 可行
  外加应用层每连接对象（会话状态、读写缓冲）

CPU：每连接一线程模型
  线程栈默认 1MB，100 万线程 = 1TB 栈内存 + 海量上下文切换  <- 不可行
  必须用 epoll/IO 多路复用：一个线程管几万连接，O(1) 就绪通知
```

- `10 万` 这条线：单机调优（文件描述符上限、socket buffer、epoll）能扛，但已经要专门设计接入层。
- `100 万` 这条线：逼近单机 C10M 极限，要内核深度调优甚至用户态网络栈，通常直接横向拆成接入网关集群。心跳本身也是成本——100 万连接每 30 秒一次心跳 ≈ 3.3 万次/秒的定时唤醒和读写。

**什么时候这个数会变：** 调小每连接 buffer、用更省内存的协程/事件模型、内核优化（io_uring、SO_REUSEPORT），都能往上抬；反过来，每连接要维护大量状态（如实时协作的文档快照）会往下压。详见 [02f](./02f_连接数高系统的连接网关与接入层方法论.md)。

### 7. 出口带宽：100MB/s 关注，1GB/s 决定架构

**物理来源：单机网卡物理带宽。** 这条最直白。

```text
100 MB/s = 800 Mbps  -> 逼近 1Gbps 网卡（125MB/s）上限
1 GB/s   = 8 Gbps    -> 逼近 10Gbps 网卡（1.25GB/s）上限
```

- `100MB/s`：开始撞普通 1G 网卡，要关注。
- `1GB/s`：单台 10G 网卡基本到顶，**物理上一台机器吐不出去**，必须多机分摊 + CDN 把流量挡在源站之外。

这是个硬物理墙，没有“优化代码”能绕过，只能加网卡、加机器、加 CDN。

### 8. 单响应大小：10KB 注意，200KB 要控制

**物理来源：响应大小 × QPS = 带宽，外加序列化 CPU。**

单响应大小本身不可怕，可怕的是乘上 QPS：

```text
10KB  × 5 万 QPS = 500 MB/s   -> 撞 10G 网卡一半
200KB × 5000 QPS = 1 GB/s     -> 撞满 10G 网卡
```

外加一笔常被忘的成本：**JSON 序列化单核约几百 MB/s**，大响应体会吃掉可观的 CPU。所以 `200KB+` 的响应要做分页、字段裁剪、压缩，大二进制内容走对象存储/CDN，API 只返回 URL 和元数据。

### 9. 单 key / 单对象 QPS：1k/s 警觉，10k/s 拆分

**物理来源：单分片 = 单核 + 单网卡，无法靠分片分散。**

正常情况下加分片能线性扩容，但**热点 key 破坏了这个前提**：不管你有多少分片，同一个 key 永远落在同一个分片、同一个 CPU 核、同一张网卡上。

```text
Redis 单实例约 10 万 QPS（多 key 分散时）
但单个 hot key 全压到一个分片：
  1k/s  -> 已经是明显热点，要警觉
  10k/s -> 占掉单实例 ~10% 且无法靠加分片缓解，要拆
```

应对就不是“加缓存”而是“把单点摊开”：key 拆分（加随机后缀分散到多 key）、多副本读、本地缓存挡在 Redis 前面、请求合并。

### 10. 单行写热点：100/s 就有锁竞争

**物理来源：行锁持有时间。** 这条最能体现“数字 = 物理量”。

一笔更新事务持有这一行的行锁，直到事务提交：

```text
持锁时间 ≈ 应用到 DB 的 RTT + 加锁 + 修改 + redo fsync ≈ 1ms 量级
理论串行上限 = 1 / 1ms = 1000 次/s
```

但理论值要打折：MVCC 版本冲突、锁等待排队、偶发死锁回滚，让**实际在 100/s 量级就开始明显竞争**。所以经典的库存扣减、计数器、点赞数这类“单行高频更新”，要用 **分段计数（把一行拆成 N 行分摊）、异步聚合、队列串行化** 来破。

**什么时候这个数会变：** 事务越短（少跨网络、少做别的事）、fsync 越快（NVMe），持锁时间越短，上限越高；反之事务里夹了远程调用、慢查询，持锁时间拉长，几十/s 就开始堵。

### 11. P99 延迟目标：300ms 是坎，50-100ms 要全内存

**物理来源：RTT 累加 + 慢查询（磁盘）+ GC 停顿 + 跨区域。** 用第一节的延迟表把预算拆开看：

```text
同机房一次 RPC 往返 ~0.5ms（实际含排队/序列化/GC，常到几 ms）
一次走磁盘的慢查询 ~10ms+
JVM 一次 Full GC 停顿 ~几十 ms 到上百 ms
跨区域往返 ~几十 ms 到 100ms+
```

把它们累加：

```text
串行 10 个下游 × 各 30ms = 300ms       <- “300ms 这个坎”的来源之一
全内存路径 + 1-2 次同机房调用 ≈ 几 ms   <- 才够 50-100ms 目标
```

- `< 300ms`：允许有限聚合，但要控住下游数量、避免慢查询、压制 GC。
- `< 50-100ms`：基本要求全内存路径、不能有跨区域调用、不能有打磁盘的慢查询、要管好 GC（甚至换低停顿 GC）。
- 跨洲 RTT ~150ms 是光速决定的，**无法优化**，所以全球低延迟只能靠就近部署（多区域 + CDN），见 [11](./11_多区域部署、容灾与可观测性.md)。

### 12. 同步下游调用数：4-10 变差，10+ 不宜串行

**物理来源：延迟累加 + 可用性乘法 + 尾延迟概率放大。** 三个独立的物理效应叠在一起：

```text
延迟：串行 total = Σ各下游      并行 total ≈ max(各下游) + 聚合
可用性：每个 99.9%，串行 10 个 = 0.999^10 ≈ 99% （直接掉一个 9）
尾延迟：调的下游越多，越大概率至少命中一个慢的，整体 P99 被放大
```

最后这条“尾延迟放大”最反直觉：每个下游各自 P99 都很好，但你同时等 10 个，只要任意一个慢，你就慢。下游越多，“至少一个慢”的概率越高。所以 `10+` 下游必须改 **并行调用、聚合服务/读模型、批量接口**，并给每个下游设 timeout、熔断、降级。

### 13. 峰值/平均 与 放大系数

这两条不是直接的硬件常数，而是“**把业务量换算成真实打到硬件上的量**”的放大镜，最终还是落回前面那些物理墙。

**峰值/平均（3x / 10x）的物理含义是扩容速度跟不上洪峰：** 自动扩容是分钟级（拉机器、预热缓存、建连接），而瞬时洪峰是秒级。所以 `10x+` 的尖峰你来不及扩容，只能 **按峰值预留容量（贵）或在入口削峰/限流/排队**。

**放大系数（写放大 20x / 100x）的作用是修正入口 QPS：**

```text
真实内部写 = 入口写 QPS × 写放大
入口 100/s × 写放大 20 = 内部 2000/s   -> 才是真正打到第 1 条那个 fsync 瓶颈的量
```

也就是说，放大系数本身没有独立的物理墙，它是个乘数——先用它把入口流量换算成内部真实 IO/调用，再去套第 1、2、12 条的物理上限。这就是 02 里反复强调“别用入口 QPS 设计”的根本原因。

## 四、面试里怎么用这一层

普通回答和高手回答的差别，就在能不能把“数字 + 物理来源 + 浮动条件”这个三件套说出来。

**反面（背数据）：**

> 单表超过 2000 万行就要分库分表。

**正面（懂原理）：**

> 我把 2000 万行当作认真考虑分片的参考线。这个数来自 InnoDB 三层 B+ 树在大约 1KB 行宽下的容量——再大就要涨到第四层，点查多一跳磁盘 IO。但更本质的判据是索引能不能驻留 buffer pool：如果索引超过内存，每次查询多次打磁盘，机械盘随机 IO 才一两百，立刻就崩；如果索引还在内存里，四层其实也能忍，主要痛的是写入维护成本。所以如果这张表行很宽、塞了大字段，这条线会下移到几百万；如果行很窄，撑到上亿也没问题。

后者用同样的时间，展示了：知道临界点的来源、知道真正的判据、知道它的浮动条件。这才是面试官想确认的“你是否真的理解”。

通用表达模板：

```text
我把 ______ 当作 ______ 的参考线。
这个数本质上来自 ______（哪个物理资源/原理）。
如果 ______（条件变化），这条线会往上/往下移。
所以真正要确认的判据其实是 ______，而不是死记这个数字。
```

## 小结（关键点）

1. 02/02h 判断表里的数字不是要背的经验线，而是从四大物理资源推导出的临界点：磁盘 IO、内存、网络、CPU/锁。
2. 先记住四组常数：网卡 10G≈1.25GB/s（注意 bit/Byte 差 8 倍）；机械盘随机 IOPS 才一两百、SSD 几万；单机内存几十~几百 GB 是热数据驻留的总开关；内存访问=1、同机房往返=5000、机械盘寻道=10 万。
3. 几个最该会手算的导出：单表 2000 万行 = 三层 B+ 树（1170×1170×16，1KB 行）；单行写 100/s = 1/持锁时间；出口 1GB/s = 10G 网卡 ÷ 8；读 QPS 天花板 = buffer pool 命中率决定走内存还是磁盘。
4. 每个阈值都有“浮动条件”：机械盘换 SSD、行变宽变窄、内存变大、网卡升级，都会让数字移动。能说清浮动条件，是“懂原理”和“背数据”的分界。
5. 放大系数和峰均比没有独立的物理墙，它们是乘数——先把入口流量换算成内部真实 IO/调用，再去套那些硬件上限。
6. 面试表达的三件套：给出量级 + 说清物理来源 + 说清浮动条件。比单纯背数字或者“看情况没有固定值”都更能体现理解。
