# 系统设计 - 第 8b 课：搜索、推荐与自动补全面试案例：电商发现系统推演

## 这一节怎么用

第 8 课讲的是发现类系统的知识框架：搜索、推荐、自动补全分别在解决什么问题。  
这一节模拟真实面试现场，把这三个方向放到一个电商发现系统里推演：

- 面试官怎么从“设计搜索”开题
- 候选人怎么澄清搜索对象、展示单元、新鲜度和排序目标
- 容量估算怎么算
- 写链路和读链路怎么拆
- 搜索建议为什么不是主搜索的附属小功能
- 面试官切到推荐时，候选人怎么切换主矛盾
- 最后如何用 2-3 分钟收束答案

这节的目标不是替代后面的 `19_搜索系统真题模拟.md` 和 `20_推荐系统真题模拟.md`。  
它更像第 8 课的口头输出训练：先判断题目到底在考搜索、推荐，还是自动补全，再选择不同回答框架。

## 案例：设计电商 App 的发现系统

### 1. 面试官开题

**面试官：**  
我们来设计一个电商 App 的搜索系统。用户可以搜索商品，也会在搜索框里看到搜索建议。你会怎么设计？

### 2. 候选人的第一反应

**候选人思考：**  
这题不能直接说 Elasticsearch。第 8 课里说过，搜索不是数据库查询，自动补全也不是简单 Trie。我要先确认搜索对象、展示单位、字段新鲜度和排序目标。

另外，面试官说的是“搜索系统”，但提到了搜索建议，后面很可能会追问推荐。我要把搜索、补全、推荐三条链路区分开。

---

## 第一轮：澄清题目语义

### 1. 候选人澄清问题

**候选人：**  
我先确认几个边界：

1. 搜索对象是商品，还是也包括店铺、内容、直播间？
2. 搜索结果页展示的是 `SPU` 还是 `SKU`？
3. 是否支持品牌、类目、价格、属性筛选？
4. 排序主要按文本相关性，还是要混入销量、CTR、转化率、价格、上新时间？
5. 商品价格、库存、上下架变化后，希望多久在搜索结果中生效？
6. 搜索建议是只做热门 query，还是要结合用户历史和个性化？
7. 是否需要广告、复杂推荐、向量搜索和多语言？

**面试官：**  
先做商品搜索。结果页按商品聚合展示，可以理解成 SPU。支持品牌、类目、价格筛选。排序要综合相关性、销量和转化。价格、库存、上下架要尽快生效，标题和详情可以有秒级到分钟级延迟。搜索建议要支持热门词和一点用户历史。先不做广告和多语言。

### 2. 候选人收敛题目

**候选人回答：**  
那我把题目收敛成电商商品发现系统：

- 搜索对象是商品，主展示单元是 `SPU`
- 交易和真实库存仍以 `SKU` 为准
- 支持标题、品牌、类目、属性检索
- 支持品牌、类目、价格等过滤
- 排序综合文本相关性、销量、CTR、转化和新鲜度
- 价格、库存、上下架是强新鲜字段，需要更快收敛
- 标题、描述、属性可以接受秒级到分钟级索引延迟
- 搜索建议要比主搜索更低延迟，并支持热门词和用户历史

这里我会先强调一个边界：  
搜索索引、推荐特征、补全词典都不是交易真相源，它们是由商品主数据派生出来的读优化结构。

## 第二轮：容量估算

### 1. 面试官追问

**面试官：**  
你估一下规模。

### 2. 候选人思考

**候选人思考：**  
搜索题估算要覆盖四类量：文档数、查询 QPS、搜索建议 QPS、索引更新 QPS。还要指出搜索内部会有 shard 放大，外部 QPS 不等于内部查询量。

### 3. 候选人计算过程

**候选人回答：**  
我先给一组合理假设：

- 商品 `SPU`：`1 亿`
- 商品 `SKU`：`3 亿`
- DAU：`2000 万`
- 每个 DAU 每天搜索：`5 次`
- 搜索建议请求约为主搜索的 `3-5 倍`
- 商品更新峰值：`5000 QPS`
- 搜索峰值放大系数：`10 倍`

主搜索 QPS：

```text
搜索请求 / day = 2000 万 * 5 = 1 亿次 / day
平均搜索 QPS = 1 亿 / 86400 ≈ 1157 QPS
峰值按 10 倍估算 ≈ 1 万 - 1.5 万 QPS
```

搜索建议 QPS：

```text
搜索建议 QPS ≈ 主搜索 QPS * 3-5
峰值 ≈ 3 万 - 7.5 万 QPS
```

索引规模：

```text
SPU 文档数 ≈ 1 亿
每个搜索文档含标题、品牌、类目、属性、排序字段、过滤字段
如果每个文档索引后占 5-8KB
主索引大小 ≈ 500GB - 800GB
加副本后 ≈ 1TB - 1.6TB
```

内部查询放大：

```text
如果索引分成 50 个 shard
一次 query 可能打多个 shard
外部 1 万 QPS 可能变成几十万级 shard query / second
```

### 4. 候选人的估算结论

**候选人回答：**  
从估算看，这题不是单机搜索引擎问题。主要 trade-off 是：

- 搜索读链路要低延迟，但一次查询会打多个 shard
- 搜索建议 QPS 比主搜索更高，必须强缓存或预计算
- 商品更新 QPS 不低，但索引新鲜度要分字段处理
- 价格、库存、上下架比标题描述更影响交易正确性，需要更快收敛

## 第三轮：数据模型和写链路

### 1. 面试官追问

**面试官：**  
商品数据怎么进入搜索系统？

### 2. 候选人回答

**候选人回答：**  
我会把商品主库和搜索索引拆开。

商品主库是真相源：

```text
product_spu
- spu_id
- title
- brand_id
- category_id
- attributes
- status
- created_at
- updated_at
```

SKU 和交易字段：

```text
product_sku
- sku_id
- spu_id
- price
- stock_status
- sale_status
- region
```

搜索文档是派生视图：

```text
search_document
- spu_id
- title_tokens
- brand
- category
- attributes
- min_price
- max_price
- stock_summary
- sale_status
- sales_score
- ctr_score
- updated_at
```

写链路：

```text
商品中心变更
-> 商品主库提交
-> CDC / Outbox / MQ 产生 ProductChanged
-> Index Builder 清洗、分词、字段展开
-> 写入搜索索引
-> 更新搜索建议词典和部分排序特征
```

这里我会按字段新鲜度分层：

1. 标题、描述、类目、属性  
   可以走异步索引构建，接受秒级到分钟级延迟。

2. 价格、库存、上下架  
   要更快收敛。可以走实时字段更新、实时过滤服务，或者在返回前查轻量可售状态兜底。

3. 销量、CTR、CVR  
   多数是近线或离线特征，允许分钟级延迟，用于排序，不应该阻塞交易正确性。

## 第四轮：搜索读链路

### 1. 面试官追问

**面试官：**  
用户搜“耐克黑色跑鞋 42 码”，读链路怎么走？

### 2. 候选人思考

**候选人思考：**  
这里要把 Query 解析、召回、过滤、排序分清。不能说“查 ES 返回结果”。还要提结构化属性和可售过滤。

### 3. 候选人回答

**候选人回答：**  
读链路我会拆成几步：

```text
用户输入 query
-> Query Understanding
-> 召回
-> 过滤
-> 排序
-> 结果组装
-> 返回分页
```

对于 `耐克黑色跑鞋 42 码`，Query Understanding 会尝试解析：

```text
品牌：Nike / 耐克
颜色：黑色
品类：跑鞋
尺码：42
```

召回层：

- 用标题、品牌、类目、属性的倒排索引召回候选 SPU
- 可以结合品牌/类目字段 boost
- 对 query 改写、同义词、拼写纠错做轻量支持

过滤层：

- 过滤下架商品
- 过滤无权限、不可售区域商品
- 过滤没有 42 码可售 SKU 的 SPU
- 根据品牌、类目、价格区间做 facet 过滤

排序层：

- 第一层：文本相关性，比如 BM25 或字段权重
- 第二层：销量、CTR、CVR、评分、价格、新鲜度等特征
- 最终重排：去重、多样性、业务规则

结果组装：

- 批量加载商品卡片需要的字段
- 返回 `spu_id`、标题、图片、价格区间、可售摘要、评分
- 分页使用 cursor 或 search_after，尽量避免深分页 offset

### 4. 面试官追问

**面试官：**  
为什么过滤要放在排序前？

**候选人回答：**  
因为过滤是硬约束，排序是软优化。

比如商品下架、无库存、区域不可售，这些结果即使文本分数很高，也不应该展示。  
所以我会先用可售性、权限、类目等条件缩小候选，再在合法候选里排序。

一句话：  
搜索结果里，`可见性和可售性决定生死，相关性决定顺序`。

## 第五轮：搜索建议 / 自动补全

### 1. 面试官切换问题

**面试官：**  
搜索框输入时的自动补全怎么做？

### 2. 候选人思考

**候选人思考：**  
这里要主动说：自动补全不是主搜索的简化版。它的输入是前缀，发生在每个字符输入后，QPS 更高、延迟更敏感、结果更少。

### 3. 候选人回答

**候选人回答：**  
自动补全和主搜索目标不同。

主搜索回答的是：

```text
用户完整表达意图后，系统如何找出相关商品？
```

自动补全回答的是：

```text
用户还没输入完时，系统如何预测高质量 query？
```

所以我会单独设计 Suggestion Service。

候选来源：

- 热门 query
- 品牌词、类目词、商品词典
- 用户历史搜索
- 区域和季节热门词
- 运营配置词

写链路：

```text
搜索日志 / 点击日志
-> 清洗、去作弊、去敏感词
-> 统计 query 热度
-> 构建 prefix -> top suggestions 词典
-> 推送到内存服务 / Redis / 搜索建议索引
```

读链路：

```text
用户输入前缀 "iph"
-> Suggestion Service 查 prefix cache
-> 合并热门建议 + 用户历史建议
-> 过滤敏感词、下架词、低质量词
-> 返回 top 10
```

因为搜索建议峰值可能是主搜索的 `3-5 倍`，我会重点优化：

- 热门前缀预计算
- 内存缓存
- 极短 timeout
- 结果数量很少，比如 top 10
- 降级到热门词榜

### 4. 面试官追问

**面试官：**  
为什么不直接每次拿用户输入的前缀去搜索商品标题？

**候选人回答：**  
有几个问题：

1. QPS 太高  
   用户每输入一个字符都会触发请求，直接打主搜索会拖垮搜索集群。

2. 目标不同  
   自动补全要返回 query 建议，不一定是商品结果。

3. 延迟要求更苛刻  
   用户打字时对卡顿很敏感，建议服务应该几十毫秒内返回。

4. 质量控制不同  
   补全要过滤敏感词、作弊词、低质量 query，还要考虑热门度和用户历史。

所以自动补全应该有独立的前缀词典和缓存，不能简单复用主搜索链路。

## 第六轮：面试官切到推荐

### 1. 面试官开题

**面试官：**  
如果用户没有输入 query，只是在首页看推荐商品，你会怎么设计？

### 2. 候选人的第一反应

**候选人思考：**  
主矛盾又变了。搜索有显式 query，推荐没有显式 query。搜索强调相关性和可解释性，推荐强调候选召回、特征、排序和反馈闭环。不能继续按倒排索引查询来讲。

### 3. 候选人回答

**候选人回答：**  
推荐和搜索最大的区别是输入信号不同。

搜索是：

```text
用户有明确 query -> 系统匹配意图
```

推荐是：

```text
用户没有明确 query -> 系统根据画像、上下文和内容特征做在线决策
```

推荐系统我会拆成：

```text
多路召回
-> 粗排
-> 精排
-> 重排
-> 反馈回流
```

候选召回来源：

- 用户历史相似商品
- 协同过滤候选
- 热门商品
- 新品探索
- 类目偏好候选
- 向量相似候选
- 运营池

排序依赖特征：

- 用户特征：兴趣类目、价格偏好、历史点击购买
- 商品特征：类目、价格、销量、评分、新鲜度
- 交叉特征：用户对某类商品的点击/购买倾向
- 实时上下文：时间、地域、设备、入口

在线链路：

```text
首页请求
-> 获取用户画像和上下文
-> 多路召回几千个候选
-> 粗排到几百个
-> 精排打分
-> 重排做去重、多样性、新鲜度、业务规则
-> 返回 20 个商品
-> 记录曝光日志
```

反馈回流：

```text
曝光、点击、收藏、加购、购买、停留
-> 日志流
-> 近线特征更新
-> 离线训练样本
-> 模型和规则迭代
```

### 4. 面试官追问

**面试官：**  
为什么不能直接对全量商品用模型打分？

**候选人回答：**  
因为全量商品太大，在线延迟不允许。

假设有 `1 亿` SPU，如果每次首页请求都对全量商品打分，即使单个打分很快，总成本也不可接受。

推荐必须分层：

- 召回层：从全库里用低成本方法拿几千个候选
- 粗排层：用轻量特征筛到几百个
- 精排层：对少量候选用复杂模型
- 重排层：处理多样性、去重、业务规则

这本质上是用分层计算，在延迟预算内做尽量好的决策。

### 5. 面试官追问

**面试官：**  
如果排序模型挂了怎么办？

**候选人回答：**  
推荐系统一定要有降级路径。

可以按层降级：

- 精排模型失败：退回粗排分数
- 个性化特征缺失：退回类目偏好或热门榜
- 某一路召回失败：其他召回源补齐
- 推荐服务整体异常：返回区域热门商品或运营兜底池

同时要监控：

- 召回源成功率
- 模型服务 RT 和错误率
- 特征缺失率
- 推荐结果曝光、点击、转化指标
- 降级率

推荐结果不是交易真相源，所以它可以降级，但不能返回下架、封禁、不可售商品。可见性和可售性过滤仍然是硬约束。

## 第七轮：候选人 3 分钟完整回答

**面试官：**  
你用几分钟总结一下这个发现系统。

**候选人回答：**  
我会把电商发现系统拆成搜索、搜索建议和推荐三条链路，因为它们输入信号和目标不同。搜索有显式 query，重点是 query 解析、召回、过滤、排序；搜索建议发生在用户输入过程中，QPS 更高、延迟更敏感，重点是前缀词典、热门 query 和用户历史；推荐没有显式 query，重点是多路召回、特征、排序和反馈闭环。

数据上，商品主库是交易真相源，搜索索引、推荐特征、补全词典都是派生视图。商品变更通过 CDC 或 Outbox/MQ 进入索引构建、特征更新和补全词典更新。字段新鲜度要分层：标题、描述、属性可以接受秒级到分钟级延迟；价格、库存、上下架需要更快收敛，必要时在返回前用实时可售状态兜底过滤。

搜索读链路里，用户 query 先做解析，比如识别品牌、类目、颜色、尺码，再通过倒排索引召回候选，过滤下架、无库存、权限不符合的商品，然后结合文本相关性、销量、CTR、转化率等特征排序，最后组装商品卡片返回。搜索建议不直接复用主搜索，而是用预计算的 `prefix -> top suggestions` 词典，融合热门 query、用户历史和运营词，并做敏感词过滤和强缓存。

推荐链路则按多路召回、粗排、精排、重排来设计。在线请求从用户画像、热门池、相似商品、类目偏好、新品探索等来源召回候选，再在延迟预算内排序和重排。曝光、点击、加购、购买等行为通过日志回流到近线特征和离线训练，形成闭环。整体上，搜索强调相关性和过滤正确性，推荐强调在线决策和反馈闭环，补全强调极低延迟和 query 质量。

## 面试复盘：这一题真正考什么

### 1. 搜索部分得分点

- 不把搜索答成“上 ES”
- 能区分商品主库和搜索派生索引
- 能先澄清 `SPU / SKU`
- 能把写链路和读链路拆开
- 能解释 query 解析、召回、过滤、排序
- 能说明不同字段的新鲜度要求不同

### 2. 自动补全部分得分点

- 能说明自动补全不是主搜索的简化版
- 能估到搜索建议 QPS 通常高于主搜索
- 能设计前缀词典、热门 query、用户历史融合
- 能说出敏感词、作弊词、低质量 query 过滤
- 能给出缓存和降级方案

### 3. 推荐部分得分点

- 能从“有 query”切换到“无 query”的主矛盾
- 能讲多路召回、粗排、精排、重排
- 能说明为什么不能全量商品直接精排
- 能把特征服务、模型服务、反馈日志串起来
- 能处理模型失败、特征缺失、召回源失败的降级

### 4. 常见失误

1. 把搜索、推荐、自动补全混成一个系统  
   结果输入信号、排序目标和延迟要求都说不清。

2. 一上来报组件名  
   例如 Elasticsearch、Redis、Trie、向量库，但不讲对象和链路。

3. 不区分真相源和派生视图  
   把搜索索引当成商品主库，会导致下架、库存、价格问题说不清。

4. 不区分字段新鲜度  
   标题修改、价格变化、库存变化、上下架不应该用同一种一致性要求回答。

5. 推荐只讲模型  
   不讲候选召回、特征、反馈回流、降级和治理。

## 最后记忆模板

发现类题型面试时，可以先问自己：

```text
用户有没有显式输入？
```

如果有完整 query，是搜索：

```text
对象：商品主库 / 搜索文档 / 索引 / 排序特征
链路：写入建索引 / Query 解析 / 召回 / 过滤 / 排序
关键：SPU-SKU、新鲜度、可售过滤、分页、热点 query
```

如果只有输入前缀，是自动补全：

```text
对象：prefix 词典 / 热门 query / 用户历史 / 运营词
链路：日志统计 / 词典构建 / 前缀查询 / 过滤 / 缓存
关键：极低延迟、高 QPS、敏感词过滤、热门前缀预计算
```

如果没有 query，是推荐：

```text
对象：用户画像 / 内容特征 / 候选池 / 排序模型 / 反馈日志
链路：多路召回 / 粗排 / 精排 / 重排 / 日志回流
关键：延迟预算、特征新鲜度、多样性、冷启动、降级
```

## 口头练习题

1. 用 3 分钟解释：为什么搜索索引不是商品真相源。
2. 用 5 分钟算一遍：`2000 万 DAU`、每天搜索 `5 次`、搜索建议是主搜索 `3-5 倍` 时，主搜索和建议峰值 QPS 大概是多少。
3. 用 3 分钟解释：为什么价格、库存、上下架的新鲜度要求高于标题和描述。
4. 用 5 分钟解释：自动补全为什么不应该直接复用主搜索链路。
5. 用 5 分钟解释：推荐系统为什么必须分召回、粗排、精排、重排，而不能全量商品直接打分。
