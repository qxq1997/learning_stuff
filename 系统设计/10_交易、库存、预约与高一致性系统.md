# 系统设计 - 第 10 课：交易、库存、预约与高一致性系统

## 学习目标（本节结束后你能做到什么）

1. 理解交易、库存、预约这类题为什么属于“正确性优先”的系统设计，而不是单纯高并发题。
2. 能先定义业务不变量，再据此决定同步边界、并发控制和补偿策略。
3. 能区分可数库存、唯一资源、时间段资源这三类高一致性对象，并知道它们各自更适合的建模方式。
4. 能结合电商库存、座位预订、预约挂号等案例，讲出一套更像真实工程的高一致性设计。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

系统设计题里，最容易暴露工程判断力的，不是“首页怎么加缓存”，而是这类题：

- 设计库存系统
- 设计预约系统
- 设计抢票系统
- 设计酒店房态系统
- 设计高并发下单与扣库存系统

因为这些题的核心矛盾不是“读快不快”，而是：

`错一次就很贵。`

你在 Feed 里出现一条重复内容，用户可能只是觉得烦；  
你在库存里多卖一单，或者在预约系统里把同一时间段卖给两个人，用户体验和业务后果都完全不一样。

所以这类题最重要的第一步，不是选数据库，不是上 Redis，不是上 MQ，而是先定义：

`这个系统最关键的不变量是什么？`

### 一、什么叫“不变量”

不变量就是系统无论遇到多高并发、多复杂故障，都不能破坏的业务事实。

例如：

#### 电商库存

- 已卖数量不能超过总可售数量
- 同一个订单不能重复占库存
- 超时未支付的锁定库存最终要释放

#### 座位/票务预约

- 同一个座位在同一场次只能卖给一个人
- 同一时间段不能被双重预约

#### 医生/会议室预约

- 同一资源在同一时间片最多只属于一个预约
- 被取消后才能重新开放

你会发现，系统设计一旦先把不变量说清楚，后面的架构讨论就不容易跑偏。

### 二、这类题和普通高并发题有什么本质不同

正确性优先系统的一个很大特点是：

- 允许慢一点
- 不允许错

这句话不是绝对，但它很适合当默认立场。

举个例子：

- 商品详情页慢 100ms，通常还能接受
- 扣库存错一次，多卖 100 单，就不是简单性能问题

所以这类题的系统设计通常更强调：

- 状态机
- 幂等
- 条件写
- 唯一约束
- 锁定与过期
- 对账和补偿

而不是一上来先谈 CDN、分片和异步化。

### 三、先分资源类型：可数资源、唯一资源、时间段资源

很多候选人把库存、座位、预约都当成同一种问题，其实它们差异很大。

#### 1. 可数资源

例如：

- 电商某 SKU 库存 100 件
- 优惠券名额 1000 份

特点是资源是“同质”的，只关心数量够不够。

#### 2. 唯一资源

例如：

- 演唱会座位 A-12
- 酒店房间 1808
- 车牌号、序列号、某个唯一工位

特点是每个资源有唯一身份，不能用简单计数替代。

#### 3. 时间段资源

例如：

- 医生 10:00-10:30 预约
- 会议室 14:00-15:00 预约
- 某辆车某个时间窗口的预订

特点是冲突不是“数量超卖”，而是“时间区间重叠”。

这三类资源的建模和并发控制方式都不同。  
你在面试里如果能主动先分类，会显得非常成熟。

### 四、库存系统先别急着谈 Redis，先把状态拆开

库存题里最经典的误区，是只维护一个数字：

- `stock = 100`

然后大家去做加减。  
这在真正的交易链路里通常不够解释复杂情况。

更成熟的库存模型往往会区分：

- 可售库存 `available`
- 锁定库存 `reserved`
- 已售库存 `sold`

为什么要拆？

因为下单和支付不是同一个时刻完成：

1. 用户提交订单
2. 系统先锁库存
3. 用户去支付
4. 支付成功再确认扣减
5. 如果超时未支付，再释放锁定库存

如果你只维护一个总库存字段，就很难清楚表达“已经预占但还没支付”的中间状态。

### 五、高一致性系统最重要的第一刀：同步边界

和前面 MQ 一课一样，这里也必须先划同步边界。  
但这类题的同步边界通常更保守。

以“下单 + 扣库存”为例，更合理的同步主链路通常是：

1. 幂等校验
2. 检查库存可售
3. 预占或锁定库存
4. 创建订单
5. 返回“订单创建成功，待支付”

而这些更适合异步：

- 发短信
- 积分
- 营销统计
- 推荐画像
- 搜索索引

你会发现，这类题里真正必须同步完成的，通常就是保护不变量的最小闭环。

### 六、并发控制到底控制什么：不是“所有写都加锁”

这类题经常会被问到：

- 你怎么防超卖？
- 乐观锁还是悲观锁？
- Redis 预扣和数据库怎么配合？

回答这些问题时，最稳的方式不是先选工具，而是先说：

`我要保护的是哪个不变量。`

#### 1. 可数资源：常见手段是条件更新

例如库存表有：

- `available`
- `reserved`

扣减时用类似思想：

- 只有 `available >= need` 时才允许更新
- 更新写成原子条件写

这类做法通常配合数据库事务或单条条件更新即可。

一个很典型的思路是：

```sql
update inventory
set available = available - :n,
    reserved = reserved + :n
where sku_id = :sku_id
  and available >= :n;
```

这类 conditional update 很适合讲清“不会把库存扣成负数”。

#### 2. 唯一资源：更常见的是唯一约束或行锁

如果是座位 A-12，只能卖给一个人。  
这种场景比“库存数量减一”更像：

- 插入一条唯一占用记录
- 或对该资源行加锁后更新状态

所以唯一资源问题，很多时候主武器不是计数器，而是：

- 唯一索引
- 行锁
- 资源状态机

#### 3. 时间段资源：要考虑区间冲突

会议室和医生预约，不是简单主键冲突，而是时间重叠。  
常见做法包括：

- 预先切成固定时间片
- 每个时间片作为唯一资源
- 或维护冲突检测逻辑 + 事务

如果面试时资源粒度够小，切片通常更好讲；  
如果是复杂区间预订，则要说明冲突检测和索引策略。

### 七、乐观锁、悲观锁，到底怎么选

这也是高频追问点。

#### 1. 乐观锁

更适合：

- 冲突概率不高
- 单次更新比较轻
- 希望减少锁等待

典型做法是版本号或条件更新。  
例如：

- 带 `version`
- 或带 `available >= n`

#### 2. 悲观锁

更适合：

- 资源非常稀缺
- 冲突概率高
- 错误代价极高
- 业务实体比较小且热点可控

例如：

- 抢最后几个票
- 锁定唯一座位
- 对某个预约时段的唯一资源做占用

#### 3. 真正工程里，常常不是二选一

更常见的是：

- 入口先限流和资格筛选
- 热点资源再做更强的串行化或条件写
- 下游用数据库约束兜底

也就是说，“锁策略”只是整个防超卖方案的一部分。

### 八、为什么很多库存系统会先在缓存/内存里预扣，再在数据库里落账

这是一个很经典但也很容易讲浅的点。

如果是秒杀场景：

- 入口瞬时百万请求
- 底层数据库不可能直接同步扛住所有竞争

这时候常见做法是：

1. 前面先做资格校验与限流
2. 再在 Redis/内存里做预扣或令牌控制
3. 只有拿到资格的请求才进入下游创单与数据库落账

这类做法的本质是：

- 前层做“流量筛选”
- 后层做“账务真相”

你必须主动说一句：

`缓存预扣不是最终真相，数据库或账务表才是最终真相源。`

否则这个设计就会显得不稳。

### 九、预约系统为什么比库存题更像“状态机 + 唯一约束”问题

预约题很容易被答成“就是库存减一”。  
其实很多预约题的本质更接近：

- 某个资源
- 某个时间段
- 只能被一个人成功占用

例如会议室预约：

- 会议室 A
- 2026-04-18 14:00-15:00

这更像：

- 资源 ID
- 时间片
- 唯一占用记录

一个成熟的设计会主动问：

- 时间粒度是 15 分钟、30 分钟还是小时？
- 是否允许重叠预订？
- 是否需要 hold 住一段时间等待确认？
- 过期未确认如何释放？

这些问题会直接决定你的数据模型。

### 十、预约和抢票系统为什么常常需要“hold”状态

无论是电影票、机票、门诊号源还是酒店房态，都常见一个中间状态：

- 已 hold / 已锁定
- 待支付 / 待确认

这是因为用户确认、支付、身份校验不是瞬间完成的。  
如果没有 hold，你很难解释“我明明刚点进去为什么座位瞬间被人买走”。

但 hold 也带来新问题：

- hold 多久过期
- 过期释放谁来做
- 重复释放如何防止

常见方式包括：

- 延迟消息
- 定时扫描
- 基于过期时间的状态推进

### 十一、补偿不是为了“让错单消失”，而是为了收敛到正确状态

正确性优先系统里，补偿非常重要。  
但补偿不该理解成“出了错以后人工改一改”。

更成熟的理解是：

- 本地事务保护最小闭环
- 跨系统通过事件驱动推进
- 某一步失败后，通过幂等补偿把系统拉回到合法状态

例如：

1. 订单已创建
2. 库存已锁定
3. 但支付长时间未成功

这时系统应该自动：

- 关闭订单
- 释放锁定库存

补偿的目标不是“回到什么都没发生”，而是“进入一个业务上合法可解释的终态”。

### 十二、对账为什么是高一致性系统的最后一道保险

只靠在线链路不能假设永远没 bug。  
因此高一致性系统常常需要离线或准实时对账。

例如：

- 订单状态和支付状态是否一致
- 库存锁定、已售和订单数量是否平
- 预约记录和资源状态是否一致

如果发现不一致：

- 自动修复
- 标记异常
- 人工介入

面试里主动提到对账，会让你的方案看起来明显更像真实系统。

### 十三、案例一：电商库存系统怎么讲

如果面试官问“设计高并发库存系统”，一个比较稳的节奏可以是：

1. 先定义不变量  
   已售不能超过总可售；同一订单不能重复占库存。

2. 再拆库存状态  
   `available / reserved / sold`

3. 划同步边界  
   下单时先锁库存并创建订单；支付成功后再确认扣减。

4. 讲并发控制  
   条件更新、幂等键、数据库真相源。

5. 再讲热点  
   秒杀场景前面加令牌、限流、缓存预扣，后面数据库落账。

6. 最后讲释放与对账  
   超时释放库存、异常对账修复。

这样的回答很像真实电商系统，而不是只会说“Redis 扣减库存”。

### 十四、案例二：电影院选座或会议室预约怎么讲

这类题的主线通常是：

1. 先把资源切成唯一可占用单元  
   如座位 ID、会议室 ID + 时间片。

2. 进入选择页面时，可以展示可用状态，但不要承诺已经锁定

3. 用户确认后，进入 hold 状态  
   例如锁定 5 分钟待支付。

4. 支付或确认成功后，状态变为已售/已预约

5. 超时则释放 hold

这类题最值得主动说的 trade-off 是：

- 展示层可用状态可能略滞后
- 真正的成功只在 hold 或 commit 成功后才成立

这会让你的回答更像真实抢票/预约产品。

### 十五、案例三：挂号系统和库存系统的最大差别

挂号系统虽然也像“资源被占用”，但和库存系统有个关键不同：

- 它常常带有严格时间片
- 对资源冲突容忍度极低
- 人工操作、取消规则、迟到规则很多

所以如果你把挂号题答成“库存减一”，会显得太粗。  
更成熟的说法是：

- 号源本质上是“医生 + 日期 + 时间段”的唯一资源
- 预约更像“时间片占用系统”

### 十六、面试里怎么把这类题讲得成熟

一个很稳的回答顺序是：

1. 先定义不变量  
   什么绝对不能错。

2. 再分类资源  
   可数资源、唯一资源还是时间段资源。

3. 设计状态机  
   available / reserved / sold / expired 或 hold / confirmed / cancelled。

4. 划同步边界  
   哪一步必须同步完成，哪一步可以异步。

5. 再讲并发控制  
   条件写、唯一约束、锁、幂等。

6. 最后补热点与失败恢复  
   限流、预扣、超时释放、补偿、对账。

这样一来，交易与预约题就会从“组件题”变成“正确性设计题”。

## 小结（3-5 条关键点）

1. 交易、库存、预约题的核心不是先选组件，而是先定义业务不变量，也就是系统绝对不能破坏的事实。
2. 不同资源类型需要不同建模：可数资源更像条件计数，唯一资源更像唯一占用，时间段资源更像区间冲突控制。
3. 高一致性系统通常会显式建模 `available / reserved / sold` 或 `hold / confirmed / cancelled` 这类中间状态。
4. 缓存预扣和令牌控制可以做前层流量筛选，但数据库或账务表通常仍然是最终真相源。
5. 真正成熟的方案一定会补上超时释放、幂等、补偿和对账，而不是只讲“防超卖”。
