# 36. 多模态文档理解平台真题模拟

## 题目

> 设计一个企业级 **多模态文档理解平台（Document Intelligence Platform）**：
>
> - 用户上传 PDF / 扫描件 / 图片 / Office 文档 / 邮件附件，平台要能：
>   1. **抽取文本**（含扫描件 OCR）；
>   2. **还原版面结构**（段落、标题、列表、表格、图、页眉页脚、阅读顺序、多栏）；
>   3. **理解语义**（这是一张发票 / 合同 / 身份证 / 财报 / 病历 / 货运单）；
>   4. **结构化抽取**成业务字段（发票号、金额、日期、条款、图表数字……）；
>   5. **支持自然语言问答**（基于文档内容回答用户问题，并给出逐字可追溯的 citation）；
> - 面向场景：金融（开户材料审核、合同风控）、法律（合同对比）、保险（理赔材料）、医疗（病历结构化）、跨境电商（海关单据）；
> - 要求支持 **批量异步**（每天百万页）和 **实时交互**（用户在 UI 上框选一块区域立即抽取）两种负载；
> - 结果要能被下游消费：
>   - 进入 28 章的 RAG 作为知识源；
>   - 进入 BI / 数据仓库做报表；
>   - 被 29 章 Agent 当作工具结果读取；
> - 需要对结构化抽取的**正确性**负责：金融行业字段级准确率 ≥ 99%，并能提供“不确定”信号触发人工复核。
>
> 请按系统设计面试的标准完整方案：语义收敛、容量估算、核心不变量、方案推演、核心对象模型、高层架构、API 设计、数据模型、核心链路、演进路径、追问、失分点、自测。

这类题对应 Amazon Textract、Azure Document Intelligence、Google DocAI、Unstructured.io、LlamaParse、Reducto、合合 TextIn、金智维、百度文心 OCR 等产品。2025-2026 VLM（Qwen-VL、GPT-4o、Gemini、Claude 等）已经能端到端读图，但**纯 VLM 方案在生产上仍然扛不住准确率和成本**，真实平台是 **传统 CV + OCR + 版面分析 + VLM + LLM** 的混合管线。这正是这题的工程魅力所在。

---

## 为什么这题值得深讲

它和 28 章 RAG 看起来像，但其实根本不同：

- RAG 把文档当成**文本**，只关心 chunk / embedding / retrieval；
- 文档理解平台把文档当成**带版面的结构化对象**，要负责“先把 PDF 正确地读懂”，再把结果交给 RAG / BI / Agent；
- 这题的主要风险**不是可用性而是正确性**：一张发票金额识别错 1 块钱，下游会计系统直接对不上账；
- 面试官想看的是——候选人能不能意识到“端到端一个大模型读完”在 2026 还没替代传统管线，以及**何时该用 VLM、何时该用专有模型、何时必须人工复核**。

真正有经验的候选人，会在三个地方暴露水平：

1. **版面分析（layout）作为一等对象**；
2. **字段级可信度（confidence）作为输出的一部分**；
3. **回退 / 复核 / 评测 / 漂移**四件事同时讲。

---

## 面试官真正想看什么

- 能不能区分 **“OCR / 版面 / 语义 / 抽取 / 问答”** 五个阶段，而不是混成一团；
- 能不能讨论 **VLM vs OCR+LLM** 的 trade-off，而不是无脑选 VLM；
- 能不能处理 **异步批处理 + 实时交互** 两种负载走同一套“工件（artifact）”存储；
- 能不能讲清 **schema-first 抽取**：每类文档对应一个抽取 schema，不是自由问答；
- 能不能设计 **confidence → 人工复核闭环**，而不是“模型说多少就是多少”；
- 能不能把 28 章 RAG、34 章评测、31 章向量库、29 章 Agent 作为已存在的依赖正确复用。

---

## 收敛题目语义

第一件事依然是边界对齐。

1. **平台不是 OCR 服务**：OCR 只是内部一个 stage，对外暴露的是“文档 → 结构化对象 + 问答接口”；
2. **不保证 100% 正确**：目标是“99%+ 字段级准确率 + 低置信度时标记人工复核”，绝不是零错误；
3. **只支持少数一等文档类型**：平台自带模板（发票 / 身份证 / 合同 / 财报 / 病历 / 海关单据 / 自由文档），其他类型走“通用版面 + LLM 抽取”降级路径；
4. **schema 是用户定义的**：用户通过 33 章 prompt 管理里的文档抽取 schema（字段 + 类型 + 必填 + 验证规则）定义自己想要什么；
5. **不负责业务一致性**：例如“发票金额和订单金额要一致”是业务系统的事，不是文档平台的事；
6. **批量任务是一等对象**：用户提交一次，得到 `job_id`，可以 poll 可以订阅事件，不要求长连接。

讲清这六条后，面试过程会稳定很多。

## 容量估算

- 峰值入口：每天 100 万页 PDF / 1000 万张图片，相当于 **平均 150 QPS 页**，峰值 500 QPS 页；
- 实时交互：同时在线交互会话 1 万个，平均每会话 2 QPS → **2 万 QPS 小请求**（区域抽取 / 问答）；
- 平均每页存储：原始 PDF 页 200 KB + 渲染成图片 1 MB + layout JSON 50 KB + 结构化结果 JSON 20 KB + 嵌入向量 4 KB（每 chunk）；
- 每页成本估算：
  - OCR：100ms × 1 GPU 小核（或 CPU）= $0.001 / 页；
  - 版面模型：50ms × 1 GPU = $0.0005 / 页；
  - VLM（可选）：1-3s × A100 片段 = $0.01-$0.03 / 页；
  - LLM 抽取（文本）：每页 500 tokens prompt + 200 tokens output = $0.005 / 页；
- 100 万页全跑一遍 → **~$15,000-$40,000 / 天**，这就是为什么要做**分层路由**（不是所有页都喂 VLM）；
- 存储：100 万页 × 1.3 MB / 页 × 保留 30 天 = **~40 TB** 热存储，冷存储对象存储即可；
- 并发 GPU 规模：OCR ~ 100 张、版面 ~ 50 张、VLM ~ 100-200 张、LLM 抽取走 32 网关共享。

**核心结论：成本是这题的主要矛盾**。架构上要围绕“分层（轻模型粗筛 → 重模型精修）+ 缓存（模板命中不重算）+ 采样（人工标注只抽低置信样本）”来组织。

## 不变量

1. **“原始工件不可变”**：上传的原始文件有内容哈希，系统所有阶段都基于这个哈希，同一哈希永远产生同一 lineage；
2. **每个阶段都有独立可重放的产物**（OCR 结果、版面结果、语义分类、抽取字段、问答），阶段间**只通过工件传递**，不做内存状态；
3. **每个字段必须带 `source` 和 `confidence`**：`source` 指向原始文档的 bbox（哪一页第几个框），`confidence` ∈ [0,1]；
4. **schema 版本化**：同一份抽取 schema 升级后，旧任务的结果仍可回放验证；
5. **结果不可被下游消费者“修改写回”**：下游只能读或发起“纠错 request”，真相源是平台内的工件；
6. **敏感字段（身份证号、银行卡、病历号）默认走脱敏通道**，明文只存在内存流水线里，落盘一律加密或哈希。

这六条是面试里**一条都不能省的不变量**，尤其第 3 条——很多候选人忽略了 `source bbox` 就无法提供 citation、无法做人工复核、无法做回归评测。

## 方案推演

### 方案 A：纯 VLM 端到端

一张图扔给 GPT-4o / Qwen-VL / Gemini，直接让它输出 JSON。

- 优点：流程最短，能处理任意版面；
- 问题：
  - 成本每页 $0.02-$0.05，百万页 = 几万美金；
  - 长文档（50 页合同）受 context / 图像 token 限制；
  - 幻觉严重：金额多一个零、日期错一位，字段级准确率到不了 99%；
  - 无法提供 bbox 级 citation，出错了没法追源；
  - 无法批改：模型升级后行为变化，回归难；
  - 不擅长复杂表格合并单元格。
- 结论：作为 **兜底 / 降级 / 新品类快速上线** 有用，不能作为主路径。

### 方案 B：纯传统 OCR + 规则抽取（老派）

- 优点：成本低、确定性强、bbox 信息齐全；
- 问题：任何版面变化（新模板、手写、印章遮挡）都要写新规则；扩展性差；处理自由文本合同不行。
- 结论：作为发票 / 身份证这种强模板场景的**快速路径**很好，但单靠它撑不起整个平台。

### 方案 C：`Ingest → Preprocess → OCR → Layout → Classify → Extract → Verify → Publish` 分阶段管线（本题目标方案）

把文档理解拆成 8 个明确 stage，每 stage 都是一等工件，有独立模型、独立扩容、独立评测。VLM 作为其中的“重火力”，**只在需要时启用**。

核心想法：
- **轻重分层**：轻量 OCR + 版面打底保证 bbox / citation / 成本，VLM / LLM 做语义升华；
- **模板优先**：已知文档类型（发票、身份证、海关单、常见合同）走“模板抽取 + 校验”快路径；
- **LLM fallback**：未识别或低置信的页面/字段走 VLM/LLM 重路径；
- **schema-first**：所有 extraction 都受 schema 约束，不是自由对话；
- **人工复核 in the loop**：低 confidence 字段进人工队列，复核结果反哺训练集。

## 核心对象模型

```
Artifact（工件，不可变）
├─ artifact_id（content hash 派生）
├─ tenant_id
├─ kind（raw_file / page_image / ocr_result / layout / classification
│        / extraction / chunk / embedding / qa_answer）
├─ parent_artifact_id（lineage）
├─ stage_version（产生该工件的 stage 版本）
├─ mime / bytes_uri（对象存储地址）
└─ metadata（尺寸、页数、语言、生成时间）

Document
├─ document_id
├─ raw_artifact_id
├─ pages[] -> Page
├─ doc_class（invoice / id_card / contract / ...）
├─ schema_id（抽取 schema）
├─ status（queued / running / verified / failed）
├─ extraction_artifact_id
└─ confidence_summary

Page
├─ page_no
├─ image_artifact_id
├─ ocr_artifact_id
├─ layout_artifact_id
├─ blocks[]（由 layout 产生）
└─ reading_order

Block
├─ block_id
├─ page_no
├─ bbox (x,y,w,h)
├─ type（heading / paragraph / list / table / image / stamp / signature / figure / footer）
├─ text（来自 OCR）
├─ confidence
└─ structure（若 type=table，则含 rows/cols/cells[][] 及合并单元格）

ExtractionField
├─ name（"invoice_no"）
├─ value（"INV-2026-0001"）
├─ type（string / number / date / currency / enum / list / table）
├─ confidence
├─ source[]（list of block_id + bbox + page_no）
├─ validation_status（passed / failed / skipped）
└─ needs_review（bool）

Schema
├─ schema_id / version
├─ doc_class
├─ fields[] -> (name, type, required, regex, enum, cross_rules)
└─ post_validation[]（"sum(items.amount) == total" 这类）

Job
├─ job_id
├─ tenant_id / user_id
├─ mode（batch / realtime / selection）
├─ document_ids[]
├─ schema_id
├─ status / progress / error
└─ callbacks

ReviewTask（人工复核）
├─ task_id
├─ field_ref (document_id, field_name)
├─ reason（low_confidence / validation_failed / policy）
├─ assignee
├─ correction
└─ timestamps
```

要特别强调两点：

1. **Artifact 作为 lineage 一等对象**：整条管线可以用 artifact tree 可视化，出错时回溯容易；
2. **ExtractionField.source 是必填**：没有 source 的字段不允许进 DB，用这条硬规则防止 LLM 幻觉被误当事实。

几个架构决策必须在面试里讲清：

1. **控制面 vs 执行面分离**：Job Manager / Planner / Router 是控制面，具体 OCR/Layout/Extractor 是执行面 workers，两者通过事件驱动；
2. **Stage Planner 不是一个大模型**：它是一段规则 + 分类器，负责“根据 doc_class 和 schema 决定要跑哪些 stage、走哪条路径”；
3. **VLM 是一条支路，不是主路**：只有 Planner 判断“版面复杂 / 无模板 / 上一阶段失败”时才启用；
4. **Review 队列和管线是反向耦合**：人工纠错后，纠错结果会作为训练/评测样本回到管线；
5. **和 33 章 Schema Registry 复用**：schema 定义、版本、上线门槛跟 prompt 走同一套版本治理。

## API 设计

对外暴露两类 API：**批量任务**和**实时交互**。

### 批量任务

```
POST /v1/documents          上传原始文件（小文件直传，大文件走预签名分块）
  => { document_id, artifact_id }

POST /v1/jobs
  {
    mode: "batch",
    document_ids: [...],
    schema_id: "invoice.v3",
    options: { language: "zh", force_vlm: false, review_threshold: 0.9 }
  }
  => { job_id }

GET  /v1/jobs/{id}          查进度与结果摘要
GET  /v1/jobs/{id}/events   SSE 推送阶段事件
GET  /v1/documents/{id}/extraction     取结构化结果（带 source/confidence）
GET  /v1/documents/{id}/layout         取版面结果
GET  /v1/documents/{id}/ocr            取 OCR 结果
POST /v1/documents/{id}/reprocess      重跑（升级 schema 或模型后）
```

### 实时交互

```
// 用户在页面上拉了一个框，立即抽取
POST /v1/realtime/region
  { document_id, page_no, bbox, task: "extract_text" | "extract_table" | "ask" }
  => { result, confidence, source }

// 基于文档问答（走 RAG）
POST /v1/realtime/qa
  { document_id, question }
  => { answer, citations: [{page_no, bbox, text}] }
```

### 复核

```
GET  /v1/review/tasks?assignee=me
POST /v1/review/tasks/{id}/correct
  { field, value, note }
```

几个细节：

- `review_threshold` 暴露给用户，让客户可以配置“金融类字段要求高、备注类字段放宽”；
- 结果接口永远返回 `source` 和 `confidence`，即使前端不显示，也必须落库；
- `reprocess` 支持“保留老结果做 diff”，这是合规客户常见诉求；
- 实时区域抽取必须走 **工件缓存**：如果 OCR/layout 已经做过，就用缓存，不重复处理。

## 数据模型

```sql
-- 工件（核心 lineage 表）
CREATE TABLE artifacts (
  artifact_id    UUID PRIMARY KEY,
  tenant_id      BIGINT NOT NULL,
  content_hash   BYTEA NOT NULL,
  kind           VARCHAR(32) NOT NULL,
  parent_id      UUID,
  stage_version  VARCHAR(32),
  uri            TEXT,              -- 对象存储地址
  mime           VARCHAR(64),
  size_bytes     BIGINT,
  metadata       JSONB,
  created_at     TIMESTAMP
);
CREATE INDEX ON artifacts(tenant_id, content_hash);
CREATE INDEX ON artifacts(parent_id);

-- 文档
CREATE TABLE documents (
  document_id      UUID PRIMARY KEY,
  tenant_id        BIGINT NOT NULL,
  raw_artifact_id  UUID REFERENCES artifacts,
  page_count       INT,
  doc_class        VARCHAR(64),
  schema_id        VARCHAR(64),
  schema_version   INT,
  status           ENUM('queued','running','verified','needs_review','failed'),
  confidence_min   NUMERIC(4,3),
  confidence_avg   NUMERIC(4,3),
  extraction_id    UUID REFERENCES artifacts,
  created_at       TIMESTAMP,
  updated_at       TIMESTAMP
);

-- 页（冗余便于按页查询/抽取/展示）
CREATE TABLE pages (
  document_id      UUID,
  page_no          INT,
  image_artifact   UUID,
  ocr_artifact     UUID,
  layout_artifact  UUID,
  PRIMARY KEY (document_id, page_no)
);

-- 字段（核心输出）
CREATE TABLE extraction_fields (
  document_id      UUID,
  field_name       VARCHAR(64),
  value            JSONB,          -- 多类型，统一 JSON
  type             VARCHAR(16),
  confidence       NUMERIC(4,3),
  source           JSONB,          -- [{page_no, bbox, block_id}]
  validation       VARCHAR(16),
  needs_review     BOOLEAN,
  updated_at       TIMESTAMP,
  PRIMARY KEY (document_id, field_name)
);

-- 任务
CREATE TABLE jobs (
  job_id         UUID PRIMARY KEY,
  tenant_id      BIGINT,
  mode           VARCHAR(16),
  schema_id      VARCHAR(64),
  doc_total      INT,
  doc_done       INT,
  doc_failed     INT,
  status         VARCHAR(16),
  started_at     TIMESTAMP,
  finished_at    TIMESTAMP
);

-- 复核
CREATE TABLE review_tasks (
  task_id        UUID PRIMARY KEY,
  document_id    UUID,
  field_name     VARCHAR(64),
  reason         VARCHAR(32),
  state          VARCHAR(16),
  assignee       BIGINT,
  original_value JSONB,
  corrected_value JSONB,
  corrected_at   TIMESTAMP
);
```

**注意力放在 artifacts 表**：它是整个系统的“事实账本”。后面任何 reprocessing、debug、evaluation 都从 artifacts 开始查。

## 核心链路

### 1. 入库（Ingest）

1. 客户端通过预签名分块上传原始文件到对象存储；
2. `Sniffer` 识别 MIME，校验反病毒、反嵌入脚本（尤其 PDF JS / Office 宏）；
3. `Normalizer` 统一到 PDF；Office 走 headless LibreOffice / Aspose；邮件附件递归展开；加密 PDF 需要 passphrase 则触发失败事件（不猜密码）；
4. 按页渲染成图片（300 DPI 彩图，灰度可选），生成 page-level artifact；
5. **内容哈希去重**：同一 `content_hash` 的工件已存在则跳过重新 OCR / layout，只拷 artifact 引用。

这一步的关键是：**尽早产生稳定 artifact_id**，后续所有 stage 都以 artifact 为主键，而不是 document_id。

### 2. OCR

1. OCR Workers 从 page_image_artifact 读图；
2. 内部按“印刷体 / 手写 / 公式 / 多语言”四个模型分流（例如 PaddleOCR / Tesseract / 自研多语 OCR / TrOCR）；
3. 输出带 bbox 的文本块（block-level）和字符级 confidence；
4. 产出 `ocr_artifact` JSON；
5. 记录每个 block 的 confidence，后面 Verifier 会用。

Trade-off：

- **CPU vs GPU**：中文印刷体 CPU 已够；手写 / 复杂版面需要 GPU 模型；Router 按页面特征决定；
- **高精度 vs 速度**：表单 / 证件类走高精度（高分辨率 + 大模型），自由文本走快速 OCR。

### 3. 版面分析（Layout）

这是这题的“隐藏主角”，很多候选人忽略它：

1. 对整页图像使用版面模型（如 LayoutLMv3、DiT、RT-DETR、PaddleStructure），识别 block 类型：`heading / paragraph / list / table / figure / stamp / footer`；
2. 读顺序分析：基于列检测 + block 层级给出 reading_order；
3. Table 检测走专门的 Table Parser（TableTransformer / PP-Structure），输出 cell 级结构包括合并单元格；
4. 把 OCR 文本按 block 对齐，产出 `layout_artifact`。

为什么版面是一等对象？因为：

- 结构化抽取的 `source` bbox 来自 layout；
- RAG chunking 必须按版面切，而不是按字符切（不然表格被切碎）；
- 问答时 citation 要精确到 block。

### 4. 分类（Classify）

用一个轻量多标签分类器（文本 + 首页缩略图）给 `doc_class`（invoice / contract / medical_record / …）。这步决定**走模板路径还是通用路径**。

### 5. 抽取（Extract）

这是核心 stage，要按**三层路径**讲：

**快路径：模板抽取**

- 匹配 template（字段 bbox 相对位置 + regex），对印刷体发票、证件、快递单等极快（每页 <50ms，成本几近 0）；
- 只适用于版式稳定的文档；
- 命中率评估（template-hit-rate）是平台核心指标。

**主路径：Schema + LLM 文本抽取**

- 把 OCR + layout 的文本按 schema 组装 prompt；
- LLM 输出 JSON，要求“字段不存在就返回 null，不要编造”，并同时输出 `source_block_id`；
- 后处理按 `source_block_id` 反查 layout 拿到真实 bbox；
- 每字段 confidence = LLM self-report confidence ⊕ block OCR confidence ⊕ schema validation 结果（三者加权）；
- 走 32 章网关，schema 换算成结构化 output 调用（function calling / JSON mode）。

**重路径：VLM 路径**

- 复杂表格、手写、印章遮挡、版面异常时启用；
- 直接把页面图像喂 VLM，要求输出 JSON + 近似 bbox；
- 由于 VLM bbox 精度不高，需要再用 OCR 的 bbox 做二次对齐；
- 最贵，按需启用。

面试里要强调：**三层不是互斥，而是组合**。一个发票可能绝大多数字段走模板，单独一个“备注”字段走 LLM，只有印章处走 VLM。Stage Planner 负责决策，Extractor 负责执行。

### 6. 校验（Verify）

Schema 上挂验证规则：

- **类型/格式**：日期、金额、身份证号；
- **跨字段规则**：`sum(items.amount) + tax == total`；
- **外部一致性**（可选）：和订单系统对账；
- **敏感词/合规**：合同里 PII、隐私字段触发脱敏。

验证失败、或 confidence 低于 `review_threshold`，触发 ReviewTask。这是 **confidence → 人工复核** 闭环的核心。

### 7. 索引与发布

- 结构化结果写入 Metadata DB（OLTP），供业务直接查询；
- 原文按版面 chunk 化，写入 31 章向量库（给 28 章 RAG 用）；
- 事件流发给 34 章；
- 可选发给下游 BI ETL（Kafka / CDC）。

### 8. 问答（QA）

- 本质上是 28 章 RAG 的一个调用；
- 不同点：检索必须 **重权版面和结构**，比如合同条款要能“沿条款编号”精确召回；
- 答案必须带 citation（page_no + bbox + 文本片段）；
- 对结构化字段的问题（“这份发票金额多少？”）优先从 `extraction_fields` 直接回答，不走向量检索——这是精度提升的关键。

### 9. 复核回流

- Reviewer 在 UI 里看到“原图 + 候选字段 + 高亮 bbox”；
- 修改后写入 `extraction_fields.corrected_value`；
- 审核 diff 会：
  - 更新最终值；
  - 作为 **评测样本** 进入 34 章评测集；
  - 作为 **训练样本** 定期刷新模板 / 微调 LLM。

这一步是“平台能持续提高准确率”的根本动力，不能漏讲。

## 失败 / 异常路径

| 异常 | 策略 |
| --- | --- |
| PDF 加密 / 损坏 | 上传期立即失败，记录理由 |
| OCR 全页空白（扫描极差） | 自动提升分辨率重试，仍失败则走 VLM 兜底 |
| Layout 识别表格失败 | 降级到纯文本抽取，字段标 `low_confidence` |
| LLM 输出非合法 JSON | 使用 JSON 模式 + 重试两次，再失败走模板或 VLM |
| VLM 产生幻觉（如编造金额） | 强制要求 `source bbox`，后处理校验 bbox 内 OCR 文本一致，不一致视为幻觉，字段置 null |
| 跨字段校验失败 | 全部相关字段 needs_review = true，进入人工队列 |
| Schema 升级 | 老任务状态保留，提供 `POST /reprocess` 按 schema 版本重跑 |
| GPU 扩容跟不上 | 限流 + 任务降级（把 VLM 路径临时关闭，只走模板+文本 LLM） |
| 对象存储抖动 | 工件读取加本地 SSD 缓存；写入失败走 WAL 重试 |

## 性能与成本优化

这题在面试里经常被追问到“你怎么省成本”，下面这几条是标准答案：

1. **Content hash 去重**：企业发票、合同模板大量重复，同 hash 跳过；
2. **Template cache**：已识别为“同源模板”的文档用统一 bbox 配置直接抽字段；
3. **分辨率自适应**：大多数页 200 DPI 够，只有表格和小字走 300+ DPI；
4. **VLM 按需启用**：Planner 在 classify/layout 阶段根据置信度和表格复杂度决定；
5. **Batch prefill**：LLM 抽取同 schema 多文档时共享 system prompt prefix，走 27 章 prefix cache；
6. **双模型路由**：低复杂度走小模型抽取（Qwen-7B 量级），复杂走大模型；
7. **推理硬件分层**：OCR / layout / VLM / LLM 分别选最适合的硬件（T4 / L4 / A100 / H100），不混用；
8. **离线批处理夜间跑**：非实时任务排在便宜时段，用抢占式算力；
9. **向量只索引有检索价值的内容**：表格值不做向量，只入 OLAP；
10. **评测驱动裁剪**：34 章评测显示某些 stage 对某类文档无增益时，直接关掉该 stage。

## 评测与质量治理

文档理解平台的评测 **不是 BLEU / perplexity**，而是：

- **字段级准确率（Field F1）**：每种字段单独算 precision / recall / F1；
- **完整抽取率（Doc Exact Match）**：所有字段都正确的文档占比；
- **表格单元格 F1**：结构 + 内容两项；
- **Citation Accuracy**：问答答案的 bbox 是否确实包含答案；
- **低置信触达召回**：真错中有多少被 `needs_review` 标记出来（这是**复核体系的真正 KPI**）。

评测集是 **持续生长**的：人工复核结果会成为新样本。每次模型 / schema / 模板升级，都会跑一次**全量回归评测**，不达标不上线。这一点直接复用 33 章的门槛机制和 34 章的评测平台。

### 数据漂移监控

- 文档分布漂移：`doc_class` 月环比 >20% 波动触发告警；
- 置信度漂移：`confidence_avg` 下滑 2σ 触发告警；
- 模板命中率下滑：说明客户开始上传新模板，需要生产扩展；
- 人工复核通过率下滑：说明抽取模型质量可能退化。

## 演进路径

**阶段 1（3 个月）：MVP**
- 支持 PDF / 图片上传、OCR + 简单 layout；
- 只支持 1-2 类模板（发票 / 身份证）；
- 抽取结果同步接口返回；
- 无人工复核，低 confidence 直接返回失败。

**阶段 2（6 个月）：通用管线**
- 引入 layout model + table parser；
- 加 schema-first 抽取 + LLM fallback；
- 引入异步 job 模型；
- 上线人工复核闭环。

**阶段 3（12 个月）：多租户 + RAG 集成**
- 接入 31 章向量库做 chunk / embedding；
- 开放 RAG QA API（citations）；
- 引入 tenant-scoped schema；
- 上线 34 章评测与漂移监控。

- 合同比对、财报跨页表格；
- VLM 深度集成；
- 私有化部署（On-Prem / VPC）；
- 行业化：金融 / 医疗 / 法律三个垂直 SKU。

**阶段 5：下游一等集成**

- 数据管道直接订阅事件流；

## 面试讲法

建议按 25-30 分钟节奏：

1. **语义收敛（1 分钟）**：文档 → 结构化对象 + 问答；不是 OCR，不是纯 LLM；
2. **容量与成本（2 分钟）**：百万页 / 天 + 成本 $15k-40k，明确“成本是主矛盾”；
3. **不变量（1.5 分钟）**：工件不可变 / 字段必有 source+confidence / schema 版本化；
4. **方案推演（3 分钟）**：纯 VLM → 纯规则 → 分层管线；
5. **对象模型（3 分钟）**：Artifact / Document / Page / Block / Field / Schema / Job / Review；
6. **架构图（5 分钟）**：画出 Control / Pipeline / Storage / Review / Downstream 五面；
7. **核心链路（6 分钟）**：Ingest → OCR → Layout → Classify → Extract（三层路径！）→ Verify → Publish → QA → Review；
8. **异常与成本优化（3 分钟）**：幻觉检测、模板缓存、VLM 按需；
9. **评测与漂移（2 分钟）**：Field F1、low-confidence recall、漂移监控；
10. **演进（2 分钟）**：五阶段。

讲解技巧：

- **讲 Extract 阶段一定要分三层路径**，这是这题的核心亮点；
- **强调 source/confidence 是一等字段**，和 RAG 的 citation 对齐；
- **讲 Review 闭环是产品能持续变好的发动机**，别当成附属流程；
- **主动提 VLM 并非银弹**，展现技术判断力。

## 常见追问

**Q1：为什么不全用 VLM 做端到端？Gemini / Claude 已经很强了。**

答：三个原因：
1. 成本：百万页每页 $0.02 vs 分层管线 $0.002-$0.005，一年差几百万美金；
2. 准确率：VLM 对数字 / 日期 / 金额幻觉概率仍高于 1%，金融场景无法接受；
3. 可追溯：VLM bbox 精度不足，无法生成生产级 citation。
VLM 更适合**新模板快速冷启动**和**重路径兜底**，不是主路径。

**Q2：表格合并单元格怎么处理？**

答：Table Parser 产出 cell 矩阵 + merge info（`rowspan`, `colspan`）。抽取时按 schema 定义——比如“表格某列求和”用结构化路径；自由表格用 LLM 读 HTML 化结果。人工复核界面必须能按 cell 做最小粒度修正。

**Q3：多语言、混排、竖排（日/阿拉伯文）怎么办？**

答：OCR 按 language-aware 多模型分流，layout 前加一个 script detector。竖排需要单独 reading-order 模型。schema 在多语言间共享字段定义但允许 `value` 是多语言字符串。抽取 prompt 也要按语言选。

**Q4：手写签名、印章遮挡怎么办？**

答：印章检测作为 layout 的一种 block type；如果印章覆盖文字区域，OCR 置信度自动下降触发 VLM 重路径或人工复核。手写走 TrOCR 或 VLM，confidence 阈值更高。

**Q5：长合同（500 页）怎么切？**

答：按章节/条款切（layout 给出的 heading block），再在条款内按段落切。向量入库时 chunk id 保留 hierarchy（章→节→条款→段），这样 RAG 检索可以回溯上下文。抽取时按字段“可能在哪一章”的先验做定位，不是全文喂 LLM。

**Q6：同一份文档，用户陆续改了 schema，怎么保证历史数据一致？**

答：schema 版本化，每次运行记录 `schema_version`；旧数据保留旧结果；用户如果要“按新 schema 重跑”走 `POST /documents/{id}/reprocess`，老结果保留做 diff。 这和 33 章 prompt 版本管理思路完全一致。

**Q7：如果一页里既有表格又有自由文本，模板覆盖不到自由文本字段怎么办？**

答：模板和 LLM 抽取可以共存在一次 extraction：模板覆盖稳定字段，剩余字段走 LLM。Planner 和 Extractor 之间的接口允许“部分字段完成、部分留空”。

**Q8：怎么防止抽取 LLM 输出 bbox 不存在的内容？**

答：
1. 强制 schema 要求 `source_block_id`；
2. 后处理拿 source_block 的 OCR 文本与 LLM 输出 value 做 fuzzy match（如编辑距离或数字完全匹配）；
3. 不通过就标 `needs_review`；
4. 记录此类事件，模型质量下降会在 34 章告警。

**Q9：医疗 / 金融数据怎么合规？**

答：
- 上传链路 TLS + 客户端可选加密；
- 对象存储按 tenant + KMS key 加密；
- PII 字段默认脱敏（在 ExtractionField 里存 hash + 最后 4 位）；
- 可选 On-Prem / VPC 部署，模型全在用户 VPC 内；
- 审计日志覆盖每次访问 artifact；
- 与 34 章可观测分开两条链路（可观测只记指标不记内容）。

**Q10：实时区域抽取和批量跑的流水线可以共用吗？**

答：共用 stage 执行，但走不同调度优先级；实时走 sync 通道（同步调用 OCR/layout/LLM，目标 <2s），批量走事件驱动异步。Artifact 层完全共享（一份文档先做批量 OCR，再做实时区域抽取，可以直接复用 OCR artifact，不重算）。

## 常见失分点

- **把 OCR 当终点**：没讲 layout、classify、schema、review，只答到“OCR + LLM 就行”；
- **没有 source/confidence**：字段裸值返回，面试官追问“怎么证明没幻觉”即翻车；
- **VLM 无脑主路径**：成本、幻觉、追溯三个问题一个都没考虑；
- **忽视 schema-first**：答得像通用 chatbot，没有“结构化约束”的概念；
- **没有 Review 闭环**：模型出错只会重跑，没有人工纠错反哺数据；
- **评测只讲准确率**：漏掉 low-confidence recall、漂移监控、schema 版本回归；
- **没讲 lineage / artifact**：导致回放、调试、重跑都说不清；
- **把 RAG 当成独立系统重写**：没有复用 28/31 章，反映出缺乏平台观；
- **忽略长文档 / 表格 / 多语言**：这三类是真实客户的主要投诉来源；
- **忽略成本**：百万页规模下，成本架构直接决定商业可行性。

## 总结

多模态文档理解平台的本质是：

1. **把“文档”升级为一个可观测、可回放、可追溯的工件链**（Artifact lineage）；
2. **把“抽取”限制在 schema 之内**，用规则 + 小模型 + 大模型 + VLM 分层协作，而不是交给一个大模型；
3. **把“正确性”交给 confidence + 人工复核**，让系统可以承认“我不确定”，并有机制把不确定变成下一轮的训练集。

做到这三点，平台就能从“AI demo”变成“可以承诺 99% 字段准确率、可审计、可长期维护的生产系统”。
