# 系统设计 - 案例 43：分布式锁与热点资源协调真题模拟

## 题目

设计一个可重入的分布式锁，用于多个服务实例之间协调同一资源的互斥访问。

要求支持：

- 互斥
- 可重入
- 自动过期，避免死锁
- 只有锁持有者可以释放
- 续租
- 异常恢复
- 尽量避免过期锁继续写入造成脏结果

先不做：

- 用分布式锁替代数据库事务
- 跨地域强一致锁服务
- 所有业务都默认加锁
- 高并发核心库存扣减完全依赖锁

## 为什么这题值得深讲

分布式锁是面试高频题，但很多回答只停在：

```text
SET key value NX EX 30
```

这只能算最基础的互斥锁。

一旦题目要求“可重入”，问题会立刻变复杂：

- 同一个线程重复拿锁怎么识别？
- 重入次数存在哪里？
- 释放几次才真正删除 key？
- 锁过期后老持有者继续执行怎么办？
- 进程暂停、GC、网络抖动、Redis 主从切换怎么办？

这题真正考的是：

`你是否能把锁状态、持有者身份、重入计数、TTL、续租和 fencing token 一起建模。`

## 面试官真正想看什么

这题通常在看下面几件事：

1. 你是否知道释放锁必须校验 owner，不能直接 DEL。
2. 你是否知道可重入需要记录 owner 和 count。
3. 你是否能用 Lua 保证加锁和释放的原子性。
4. 你是否知道 TTL 只能避免死锁，不能保证业务一定安全。
5. 你是否能解释 watchdog 续租。
6. 你是否知道锁过期后老请求可能继续写，需要 fencing token 防护。
7. 你是否能比较 Redis 锁和 Zookeeper / etcd 锁的适用边界。

## 一开始先收敛题目语义

我会先问：

1. 锁保护的资源是什么？订单、文件、任务、库存，还是配置？
2. 临界区通常执行多久？
3. 同一个线程是否需要递归调用同一段逻辑？
4. 业务写入的下游是否支持 fencing token 校验？
5. 锁失败时是直接失败，还是等待重试？
6. 可接受的锁服务依赖是 Redis、Zookeeper，还是 etcd？

如果面试官不补充，我会收敛成：

- 单区域内部服务使用
- 锁用于低频协调，例如定时任务抢占、资源重建、避免重复执行
- Redis 实现锁状态
- Lua 保证原子加锁和释放
- owner 使用 `app_id + instance_id + thread_id + request_id`
- 支持 TTL 和 watchdog 续租
- 对关键写入引入 fencing token

## 第一步：先说明什么时候不该用分布式锁

分布式锁不是通用解药。

不适合：

- 高频库存扣减主链路
- 账户余额更新
- 支付扣款
- 可以用数据库唯一约束或条件更新解决的问题

这些场景通常更适合：

- 数据库条件更新
- 唯一索引
- 乐观锁版本号
- 状态机幂等
- MQ 串行化

分布式锁更适合：

- 只有一个实例执行某个后台任务
- 避免多个实例同时重建缓存
- 同一资源的低频互斥操作
- 外部系统不支持事务时做协调保护

一句话：

`能用数据不变量解决的，不要优先用锁；锁是协调手段，不是真相源。`

## 第二步：Redis 可重入锁的数据结构

普通锁只需要一个 key：

```text
lock:resource:10086 -> owner
```

可重入锁需要更多状态：

```text
Key: lock:resource:10086
Type: Hash

Fields:
- owner: appA:instance3:thread9:requestX
- count: 2
- token: 12345
```

含义：

- `owner`：当前持有者身份
- `count`：重入次数
- `token`：本次锁持有期的 fencing token
- TTL：存在 Redis key 上，不一定作为 hash 字段

为什么 owner 要足够细？

因为只用 user_id 或 service_name 不够。

同一个服务可能有多个实例、多个线程、多个请求。owner 必须能区分真正的执行上下文。

## 第三步：加锁逻辑

加锁有三种情况：

1. key 不存在，创建锁，count = 1。
2. key 存在且 owner 是自己，说明重入，count + 1。
3. key 存在且 owner 不是自己，加锁失败。

必须用 Lua 保证原子性。

伪 Lua：

```lua
-- KEYS[1] = lock key
-- KEYS[2] = token key
-- ARGV[1] = owner
-- ARGV[2] = ttl_ms

if redis.call('EXISTS', KEYS[1]) == 0 then
  local token = redis.call('INCR', KEYS[2])
  redis.call('HSET', KEYS[1], 'owner', ARGV[1], 'count', 1, 'token', token)
  redis.call('PEXPIRE', KEYS[1], ARGV[2])
  return {1, token}
end

if redis.call('HGET', KEYS[1], 'owner') == ARGV[1] then
  redis.call('HINCRBY', KEYS[1], 'count', 1)
  redis.call('PEXPIRE', KEYS[1], ARGV[2])
  local token = redis.call('HGET', KEYS[1], 'token')
  return {1, token}
end

return {0, redis.call('PTTL', KEYS[1])}
```

注意：

- 第一次加锁才生成新的 token
- 重入时复用同一个 token
- 每次重入可以刷新 TTL

## 第四步：释放锁逻辑

释放也有三种情况：

1. key 不存在，说明锁已经过期或被释放。
2. key 存在但 owner 不是自己，拒绝释放。
3. key 存在且 owner 是自己，count - 1；如果 count 为 0 才删除 key。

伪 Lua：

```lua
-- KEYS[1] = lock key
-- ARGV[1] = owner
-- ARGV[2] = ttl_ms

if redis.call('EXISTS', KEYS[1]) == 0 then
  return 0
end

if redis.call('HGET', KEYS[1], 'owner') ~= ARGV[1] then
  return -1
end

local count = redis.call('HINCRBY', KEYS[1], 'count', -1)
if count > 0 then
  redis.call('PEXPIRE', KEYS[1], ARGV[2])
  return count
else
  redis.call('DEL', KEYS[1])
  return 0
end
```

这个设计保证：

- 非持有者不能误删锁
- 重入几次就要释放几次
- 最后一次释放才真正删除锁

## 第五步：为什么必须有 TTL

如果没有 TTL：

- 持有锁的进程崩溃
- 锁永远不释放
- 所有后续请求都会卡住

TTL 是为了避免死锁。

但 TTL 也带来一个问题：

- 如果业务执行时间超过 TTL
- 锁已经过期
- 另一个请求拿到了新锁
- 老请求恢复后继续写

这就会出现“两个持有者都以为自己合法”的情况。

所以 TTL 只能解决“锁不永久占用”，不能解决“过期后老请求继续写”的所有问题。

## 第六步：watchdog 续租

如果临界区执行时间不确定，可以引入 watchdog：

1. 加锁成功后启动后台续租任务。
2. 每隔 TTL 的三分之一检查一次。
3. 如果 owner 仍然是自己，则延长 TTL。
4. 业务完成或释放锁后停止续租。

续租 Lua 也必须校验 owner：

```lua
if redis.call('HGET', KEYS[1], 'owner') == ARGV[1] then
  redis.call('PEXPIRE', KEYS[1], ARGV[2])
  return 1
else
  return 0
end
```

但 watchdog 不是万能的。

如果进程长时间 Stop-The-World、网络隔离、调度暂停，续租可能来不及。

所以关键写入仍然需要下游保护。

## 第七步：fencing token 防止过期锁写脏数据

fencing token 是分布式锁里非常重要但常被漏掉的一层。

思路是：

- 每次首次成功获得锁时，生成一个单调递增 token
- 业务写入下游时带上 token
- 下游只接受 token 更大的写入

例如：

```text
Request A 获取锁，token = 100
Request A 卡住，锁过期
Request B 获取锁，token = 101
Request B 写数据库成功，last_token = 101
Request A 恢复，尝试带 token = 100 写入
数据库拒绝，因为 100 < 101
```

数据库表可以有：

```text
resource_state
- resource_id
- value
- last_lock_token
```

写入时：

```sql
UPDATE resource_state
SET value = ?, last_lock_token = ?
WHERE resource_id = ?
  AND last_lock_token < ?;
```

这样即使锁过期，旧持有者也不能覆盖新结果。

如果下游不支持 fencing token，Redis 锁只能提供互斥的尽力而为，不能保证所有异常情况下都绝对安全。

## 第八步：等待和重试策略

加锁失败后有几种选择：

1. 直接失败  
   适合用户请求，不要无限等待。

2. 短时间自旋 + 退避  
   适合临界区很短的场景。

3. 进入队列  
   适合后台任务或可排队工作。

4. 尝试异步补偿  
   适合最终一致任务。

不要让请求线程无限阻塞。

锁等待也要有：

- wait_timeout
- retry_interval
- jitter
- 最大重试次数

否则高并发下会形成锁竞争风暴。

## 第九步：Redis 锁和 Zookeeper 锁怎么选

| 维度 | Redis 锁 | Zookeeper / etcd 锁 |
| --- | --- | --- |
| 性能 | 高 | 较低 |
| 延迟 | 低 | 较高 |
| 实现复杂度 | 中等 | 中等 |
| 一致性语义 | 依赖部署和实现细节 | 更强 |
| 适合场景 | 低延迟、短临界区、尽力互斥 | 强协调、任务主备、元数据一致 |
| 典型机制 | SET/Lua/TTL/watchdog | 临时节点、租约、顺序节点 |

如果是高频、低延迟、可容忍少量异常后补偿的场景，Redis 更常见。

如果是主备选举、任务调度 leader、强协调元数据，Zookeeper / etcd 更稳。

## 第十步：监控指标

关键指标：

- 加锁成功率
- 加锁等待耗时
- 锁持有时间
- 释放失败次数
- owner mismatch 次数
- watchdog 续租失败次数
- 锁过期次数
- fencing token 拒绝写次数
- 单资源锁竞争 QPS
- 临界区异常率

如果一个锁长期高竞争，通常要反思：

- 这个资源是不是不该用锁串行
- 是否需要分片
- 是否可以用状态机或唯一约束替代
- 是否可以改成队列串行消费

## 面试版回答

如果让我设计一个可重入分布式锁，我会先说明它适合低频协调，不适合替代数据库事务或高频库存扣减。Redis 实现时，我会用一个 hash key 保存锁状态，字段包括 owner、count 和 token，TTL 放在 key 上。owner 用 `app_id + instance_id + thread_id + request_id` 标识真实持有者。

加锁用 Lua 原子判断：如果 key 不存在，就创建 hash，owner 为当前请求，count 为 1，并生成单调递增的 fencing token；如果 key 存在且 owner 是自己，就把 count 加一，表示重入；如果 owner 不是自己，就返回失败和剩余 TTL。释放锁也用 Lua：先校验 owner，只有持有者才能释放；count 减一后如果大于 0 只刷新 TTL，等 count 到 0 才删除锁。

为了避免进程崩溃导致死锁，锁必须有 TTL；为了避免临界区超过 TTL，可以加 watchdog 续租。但 TTL 和续租都不能完全解决“锁过期后老请求继续写”的问题，所以关键写入要带 fencing token，下游只接受 token 更大的写入。这样即使旧持有者恢复，也不能覆盖新持有者的结果。

## 高频追问

### 追问 1：为什么释放锁不能直接 DEL

因为锁可能已经过期并被别人重新获得。直接 DEL 会误删别人的锁。释放前必须校验 owner。

### 追问 2：可重入怎么实现

同一个 owner 重复加锁时，不重新创建锁，而是把 count 加一。释放时 count 减一，只有减到 0 才删除锁。

### 追问 3：TTL 能保证安全吗

TTL 只能避免死锁，不能保证业务一定安全。如果锁过期后老持有者继续写，仍可能污染结果。关键写入要用 fencing token。

### 追问 4：watchdog 有什么风险

watchdog 依赖进程能正常调度和访问 Redis。如果发生长时间 GC、网络隔离或进程暂停，续租可能失败。

### 追问 5：什么时候不用 Redis 锁

强一致协调、leader 选举、元数据一致性要求高时，Zookeeper / etcd 更合适。高频库存扣减、余额更新通常应该用数据库条件更新、唯一约束或状态机，而不是锁。

## 常见失分点

1. 只说 `SET NX EX`，没有 owner 校验。
2. 不支持重入计数。
3. 释放锁不用 Lua，存在竞态。
4. 认为 TTL 可以解决所有安全问题。
5. 不讲 watchdog 续租失败。
6. 不知道 fencing token。
7. 什么场景都想加分布式锁。
