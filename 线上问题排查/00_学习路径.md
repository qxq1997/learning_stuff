# 线上问题排查 学习路径与知识图谱

## 学习定位

很多后端工程师面对线上故障，常见的两种状态：

- **第一种**：知道一堆零散命令——`top`、`jstack`、`tcpdump`、`mysql slow log`、`redis-cli --bigkeys`，但遇到真问题不知道**先看哪个、后看哪个、怎么把它们串起来**。
- **第二种**：只懂自家业务代码层面的日志，遇到"接口慢"只会盯着应用日志看半天，不会向下穿透到 JVM、操作系统、网络。

这条学习路径的目标，不是把你变成"会背工具名字的人"，而是建立一套**可复用的全栈排障 SOP**，让你能做到：

- 看到告警/现象，能在 30 秒内说出"先看什么、再看什么"的决策树
- 在 CPU / 内存 / 接口 RT / 数据库 / 缓存 / MQ / 网络 / 磁盘 / 容器这 9 个层面，每一层都有一套自己的**工具组合 + 解读姿势**
- 把症状（"CPU 100%"）、工具输出（`top -H -p` + `jstack`）、根因（业务死循环 / GC 风暴 / 锁竞争）三者对得上
- 故障处理时区分"先止血"和"再定位"——能做应急预案、回滚、降级，而不是死磕根因让故障扩大

学完之后，你应该能够：

- 接到任何"线上 XX 异常"的告警，能用一张决策树推到具体工具和具体命令
- 看 `top`、`vmstat`、`iostat`、`pidstat`、`ss -s`、`jstack`、慢查询日志，不靠 Google 也能读懂关键列
- 知道哪些问题应该**优先回滚**而不是死磕（比如刚发版后 OOM、CPU 飙升）
- 复盘时能写出像样的时间线、5Why 和系统性改进项

## 与 JVM / Linux 文件夹的关系

| 文件夹 | 定位 |
| --- | --- |
| `JVM/` | JVM 内部原理与 GC 调优：CMS / G1 / ZGC 内部细节、GC 日志解读、OOM 原理、堆 dump 分析 |
| `Linux/` | Linux 基础：epoll、文件系统、权限、网络 IO 模型 |
| **`线上问题排查/`（本文件夹）** | **故障现场的端到端 SOP**：从告警到根因，跨 OS / 网络 / JVM / DB / 缓存 / MQ / 容器的全栈排障 |

JVM 内部细节（比如 G1 To-space Exhausted 的内部机制）链回 `JVM/`，这里只讲**遇到现象时该看什么、怎么决策、怎么落地处理**。

## 知识图谱（从入门到老司机）

### 第一部分：方法论先行

1. **线上排障方法论**：从告警到根因的标准流程
   - 故障四阶段：发现 → 止血 → 定位 → 复盘
   - **止血优先原则**：先回滚 / 降级 / 限流，再慢慢看根因
   - 现场保留：日志、堆 dump、线程 dump、coredump、监控截图
   - 决策树思维：CPU、内存、IO、网络、依赖 五选一进入

### 第二部分：工具地图（按层）

2. **Linux 系统层工具**：`top`、`htop`、`vmstat`、`mpstat`、`pidstat`、`sar`、`iostat`、`uptime`
   - 怎么从 load average / CPU / mem / IO wait 四个维度读"机器忙不忙"
   - `pidstat -t` 看线程级 CPU
3. **网络排查工具**：`ss`、`netstat`、`tcpdump`、`iftop`、`nload`、`telnet`、`nc`、`curl -v`、`dig` / `nslookup`
   - TCP 状态机：SYN_SENT、ESTABLISHED、TIME_WAIT、CLOSE_WAIT 各自意味着什么
   - 抓包姿势：怎么用 `tcpdump` + Wireshark 看握手、RST、重传
4. **文件、磁盘、句柄**：`lsof`、`df -h`、`du -sh`、`iotop`、`fdisk`、`inode` 占用
   - "磁盘没满但写不进"——inode 满了
   - "服务起不来"——句柄泄漏、`ulimit` 限制
5. **Java 进程现场工具**：`jps`、`jstack`、`jmap`、`jcmd`、`jstat`、Arthas、async-profiler
   - 这一节做工具索引和 SOP，JVM 内部原理链回 `JVM/09`、`JVM/11`
   - Arthas 的核心命令：`dashboard`、`thread`、`trace`、`watch`、`profiler`
6. **可观测性三件套**：Metrics（Prometheus + Grafana）、Log（ELK / Loki）、Trace（SkyWalking / Jaeger）
   - 三者各自能回答什么问题、不能回答什么问题
   - 排障时三件套的进入顺序：Metrics 看面 → Trace 找点 → Log 看细节

### 第三部分：典型问题专题（按现象组织）

7. **CPU 飙高**：从 `top` 到 `jstack` 的完整定位 SOP
   - `top -Hp <pid>` → `printf "%x\n" <tid>` → `jstack <pid> | grep <hex_tid>`
   - 区分用户态 CPU 高（业务死循环 / 序列化 / 加解密 / 正则灾难）vs 内核态 CPU 高（系统调用 / GC / 上下文切换风暴）
8. **内存暴涨与 OOM**：堆内 / 堆外 / Native 三层定位
   - 堆内 OOM → Heap Dump + MAT（链回 `JVM/11`）
   - 堆外 OOM → NMT、`pmap`、`gdb`
   - Native 内存泄漏 → glibc malloc / Netty DirectByteBuffer / 第三方 .so
9. **接口 RT 抖动与超时**：四层链路拆解
   - 应用层（线程池 / 锁 / GC）→ 中间件（DB / Redis / MQ）→ 网络（带宽 / 丢包）→ OS（CPU 调度 / 内存换页）
   - 怎么用 Trace 快速定位"慢在哪一段"
10. **线程池打满与拒绝**：Tomcat、Dubbo、业务自定义池
    - 区分 `RejectedExecutionException`、`PoolExhaustedException`
    - 怎么从堆栈看到"卡在哪个外部调用上"
11. **FullGC 频繁与 STW**：现象层 SOP（链回 `JVM/12`–`JVM/19` 看内部原理）
    - GCeasy 怎么用、`-Xlog:gc*` 怎么开
    - 区分"GC 真的有问题"还是"业务分配速率本身就高"
12. **数据库相关**：慢 SQL、连接池耗尽、行锁 / 表锁、死锁、主从延迟
    - 慢查询日志 + `EXPLAIN` + `SHOW PROCESSLIST` + `INNODB STATUS`
    - HikariCP / Druid 池子打满怎么看（链回 `MySQL/`）
13. **Redis 相关**：慢命令、热 Key、大 Key、连接打满、缓存击穿/雪崩/穿透
    - `redis-cli --bigkeys` / `--hotkeys` / `MONITOR`（慎用） / `SLOWLOG GET`
    - 客户端 jedis / lettuce 连接池排查
14. **MQ 相关**：消费堆积、消息丢失、消费抖动、重复消费
    - Kafka：consumer lag、rebalance 风暴、`__consumer_offsets`
    - RocketMQ：消费位点、广播 vs 集群消费、订阅关系不一致
15. **网络问题**：丢包、连接重置（RST）、TIME_WAIT / CLOSE_WAIT 堆积、DNS 抖动
    - CLOSE_WAIT 堆积 = 业务忘记关连接
    - TIME_WAIT 堆积 = 短连接 + 客户端主动关
    - DNS 解析慢导致接口 P99 突刺
16. **磁盘问题**：写满、IO 瓶颈、inode 用完、日志暴写
    - `find / -size +1G` 查大文件
    - 日志切割 / 异步写 / 删除策略
17. **文件句柄泄漏**：`too many open files`
    - `lsof -p <pid>` 看泄漏来源
    - `ulimit -n` / `/proc/<pid>/limits` / `/proc/<pid>/fd`
18. **容器与 K8s 环境特殊问题**：cgroup 限制、`OOMKilled`、HPA 抖动、`kubectl describe pod`、init container 卡死、JVM 不感知 cgroup
    - 为什么容器里 `Runtime.availableProcessors()` 经常不对
    - JDK 8u191 / JDK 10+ 的 cgroup 感知

### 第四部分：流程与预案

19. **应急预案、回滚、降级、限流、熔断**
    - 发布期故障的"3 分钟回滚"窗口
    - Sentinel / Hystrix / Resilience4j 的兜底打法
20. **故障复盘**：时间线、5Why、根因分类、行动项
    - 区分 contributing factor 和 root cause
    - 行动项要可验证、有 owner、有截止时间

## 推荐学习顺序

- **新手**（处理过 0–3 次线上故障）：1 → 2 → 3 → 7 → 8 → 9 → 11 → 19
  - 先把"方法论 + 系统/网络基础工具 + 几个最常见现象 + 应急预案"打通
- **中阶**（独立处理过若干次线上故障）：补全 4–6 工具层、12–14 中间件层、18 容器层
- **老司机**（已经能 cover 大部分故障）：重点在 15、16、17 这种偏门但要命的问题，以及 20 复盘的系统性沉淀

## 写作风格约定

- 每一篇专题文都按这个结构：
  1. **现象描述**：业务和监控上看到什么
  2. **快速止血动作**：回滚 / 限流 / 摘节点 等可立即做的动作
  3. **定位 SOP**：从粗到细的命令链，配合"看到 X 就往 Y 方向"的决策点
  4. **常见根因**：每一类根因配一个真实/拟真案例
  5. **改进与预防**：监控告警怎么加、代码层怎么防、容量怎么规划
- 命令都给完整可粘贴版本，关键列加中文注释
- 跨文件夹引用用 `[JVM/11](../JVM/11_线上故障实战：死锁检测、HeapDump与OutOfMemoryError排查.md)` 这种形式
