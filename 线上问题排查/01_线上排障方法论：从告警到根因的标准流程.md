# 线上排障方法论：从告警到根因的标准流程

## 这一章想解决什么

工具谁都会用。`top`、`jstack`、`tcpdump` 这些命令网上一搜一大把。

但真到线上炸了，新手和老司机的差距不在于"知道哪个命令"，而在于：

- **先做什么、后做什么的顺序感**
- **什么时候停止定位、先做止血**
- **看到一个症状，下一步应该往哪个方向走，而不是把所有工具都跑一遍**
- **故障没结束就开始喊人、还是把现场摸清楚再说**

这一章先把方法论钉死，后面 02–20 篇所有工具和专题，都是在这个框架里的具体填充。**没有方法论，工具再熟也是瞎打**。

## 一、故障四阶段：发现 → 止血 → 定位 → 复盘

线上故障的完整生命周期是四段，每段目标不同，常犯的错也不同：

| 阶段 | 时间窗口 | 核心目标 | 反模式 |
| --- | --- | --- | --- |
| **发现（Detect）** | 0 ~ 1 分钟 | 确认"是不是真有问题"+"影响面多大" | 告警一响就慌着重启、还没看监控就喊人 |
| **止血（Mitigate）** | 1 ~ 10 分钟 | **先让业务恢复**，再考虑根因 | 死磕日志找根因，故障扩大到 30 分钟 |
| **定位（Diagnose）** | 10 分钟 ~ 几小时 | 找到根因 + 留好证据 | 现场没保留，事后想分析啥都没有 |
| **复盘（Postmortem）** | 24 ~ 72 小时 | 沉淀改进项、防止再发 | 写成"个人检讨书"、没有可验证的行动项 |

**关键转换点**：

- **发现 → 止血**：一旦确认是真故障（不是误报），立刻进入止血动作，**不要在止血前先研究根因**。
- **止血 → 定位**：业务恢复以后，**才**开始正经查根因，而且要尽量保留现场。
- **定位 → 复盘**：根因清楚以后，把现象、时间线、根因、改进项写成正式文档。

这个四段式有个反直觉的地方：**止血永远在定位之前**。新手最常见的错误就是看到 CPU 100% 之后立刻去拉 jstack 分析栈——而那 10 分钟里，业务已经损失了几万订单。正确顺序是先把这台机器从 LB 摘掉（止血），再去单独留一台慢慢分析（定位）。

## 二、发现阶段：怎么判断"是不是真出事了"

### 告警来源的优先级

线上信号来源大致五类，可信度从高到低：

1. **黄金指标告警**（RT P99 突刺、错误率上升、QPS 异常下降）——最可信，直接进入止血
2. **业务指标告警**（订单成功率下降、支付失败率上升）——非常可信，但要小心是不是上游来的流量本身变了
3. **资源指标告警**（CPU > 90%、内存 > 90%、磁盘 > 85%）——**先确认对业务有没有影响**，资源高但 RT 没事，可能是夜间任务正常跑
4. **依赖方告警**（"DB 主从延迟 > 10s"）——可能你这边还没有体感，但要立刻确认
5. **用户/客服反馈**——最不可信也最重要：意味着监控漏报了，需要先确认现象再补告警

**反模式**：告警一响就动手。先问自己三个问题：

- **现象是真的吗**？登监控大盘看一眼，别只信告警短信（告警可能因为 Prometheus 抓取抖动误报）
- **影响面多大**？是单机、单 AZ、整个集群、整条业务线？
- **是不是误报**？比如某个边缘接口本来 QPS 就 0，错误率从 0/0 变成 1/1 就是 100%，但根本没有用户受影响

### 影响面评估的"三个维度"

故障要不要立刻喊人、要不要走 P0 流程，看三个维度：

| 维度 | 问 |
| --- | --- |
| **范围** | 单机？单可用区？整集群？跨服务？ |
| **业务** | 影响哪些业务流（下单/支付/登录/搜索）？ |
| **趋势** | 还在恶化？平稳？正在恢复？ |

**举个例子**：

- "10% 流量 502，单 AZ，趋势平稳" → P2，先摘 AZ，慢慢查
- "5% 订单创建失败，全集群，且失败率每分钟涨 1%" → P0，立刻拉群，先回滚

## 三、止血优先原则：为什么先止血再定位

### 为什么这么强调"先止血"

工程师天然有一种"想搞清楚为什么"的本能，看到 CPU 100% 就想拉 jstack 看是哪个线程。但**业务的成本时钟在跑**：

- 一个支付接口故障每分钟损失订单的金额，可能是你年薪的 N 倍
- 故障扩散是非线性的：5 分钟和 30 分钟的爆炸面差几个数量级（用户开始重试、缓存雪崩、上游线程池打满）

**所以止血永远是第一优先**。定位有的是时间，业务损失没有时间。

### 常用止血手段（按优先级）

1. **回滚**——发布期故障最有效的止血。Git revert + 重新发布，或者直接切到上一个镜像/版本。
   - **判断信号**：故障开始时间 与 上次发布时间高度吻合（5 分钟内）
   - **代价**：可能丢失本次发布的功能，但故障损失远大于此
2. **摘节点 / 摘 AZ**——单机或单 AZ 异常时最有效
   - 从 LB / 注册中心摘掉异常节点
   - 注意保留**一台**异常节点不重启，用于事后定位（关键现场）
3. **限流**——上游流量异常、下游扛不住时
   - Sentinel / 网关限流，把 QPS 压到能扛住的水平
4. **降级**——核心链路依赖出问题时
   - 关掉非核心功能（推荐、广告、个性化），保住主流程
   - 切到本地缓存 / mock 数据
5. **熔断**——依赖服务挂了，避免拖死自己
   - Hystrix / Resilience4j 自动熔断，快速失败而不是排队等
6. **扩容**——容量真的不够时
   - 加机器、加副本、加 Pod
   - **慎用**：扩容是最慢的止血手段，新机器拉镜像启动可能 5–10 分钟，比回滚慢得多
7. **重启**——最后的手段
   - 适用于"已经 hang 死、回滚也没用"的极端情况
   - **代价**：现场全没了，事后没法定位
   - **保留至少一个节点不重启**

**关键的反模式**：上来就重启。重启之后现场没了，jstack、jmap、coredump、内存里的状态全部丢失，事后只能靠日志反推，效率低十倍。

### 止血和定位的"分裂大脑"

老司机处理 P0 故障的姿势是：

- **一只手做止血**（回滚 / 摘节点 / 限流）
- **另一只手保留现场**（dump、抓包、jstack 留档）

这两件事**并行**做，不能因为要保留现场就拖住止血，也不能因为要止血就把现场清光。

具体执行上：

- 集群有 N 台机器异常，**摘掉 N-1 台**，留 1 台不摘（既不接流量，也不重启），用于慢慢查
- 容器环境下，可以把异常 Pod 标记为 `kubectl cordon` + 流量摘除，但**不要 kubectl delete pod**（Pod 一删现场全没）

## 四、现场保留：第一时间要抓哪些东西

故障一旦确认是真的，**在止血动作开始的同时**，下面这些东西要立刻留档。错过了就再也拿不回来：

### 必须保留的现场（按优先级）

1. **线程 dump（jstack）**：3 次，每次间隔 5 秒
   - 为什么 3 次：单次 jstack 是瞬时快照，3 次对比才能区分"卡在某个调用上"和"瞬间经过"
   - 命令：`for i in 1 2 3; do jstack <pid> > jstack-$i.log; sleep 5; done`
2. **堆 dump（jmap）**：内存类问题必须留
   - 命令：`jmap -dump:live,format=b,file=heap.hprof <pid>`
   - **注意**：jmap dump 会触发一次 Full GC + STW，业务还在受影响时慎用——但故障节点已经摘掉，那就放心 dump
   - 文件可能 4–8 GB，要确认磁盘空间，且**用 `live` 参数减小体积**
3. **关键监控截图**：CPU / 内存 / GC / RT / QPS / 错误率，至少覆盖故障前 30 分钟 ~ 故障中
   - 监控数据通常有保留期（Prometheus 默认 15 天），但**告警截图**是事后复盘最直观的素材
4. **错误日志样本**：故障期间的异常堆栈、慢日志、accesslog
   - `grep -A 50 "Exception" app.log | head -1000` 留几条典型样本
5. **环境快照**：
   - `ps -ef | grep java`
   - `netstat -anp | grep <port>` 或 `ss -tnp`
   - `lsof -p <pid> | wc -l`（句柄数）
   - `cat /proc/<pid>/status`（VmRSS、VmPeak、Threads 等）
   - `free -h`、`df -h`、`top -bn1`

### 容器环境下的特殊保留

容器环境 Pod 被 OOMKilled 后会自动重启，**现场会丢**。所以：

- 关键 Pod 加上 `terminationGracePeriodSeconds`，给 PreStop hook 留时间做 dump
- 把 dump 目录挂载到 PV 而不是 Pod 内本地盘
- `kubectl logs --previous` 拿到上一次容器的日志（重启后默认看到的是新容器日志）
- `kubectl describe pod <name>` 看 OOMKilled 的 reason 和 lastState

### 不推荐保留的（白费力气）

- **完整的 `tcpdump` 抓包**：除非问题就是网络层的，否则抓包文件巨大且后期分析成本高
- **完整的火焰图**：火焰图建议在故障节点上跑，不要等事后重放——业务流量复现不出原始火焰图

## 五、定位阶段：从症状到根因的决策树

止血完成、业务恢复、现场已留档——这时候才开始正经定位。

### 五大入口决策树

线上故障 90% 以上的现象，最终都能归到下面五条路径之一：

```
                  ┌─ CPU 高 ──→ §07 CPU 飙高
                  │
                  ├─ 内存高/OOM ──→ §08 内存暴涨
   告警/现象 ──→ ─┤
                  ├─ IO 慢/磁盘满 ──→ §04 §16 §17
                  │
                  ├─ 网络异常 ──→ §03 §15
                  │
                  └─ 依赖慢/超时 ──→ §09 §12 §13 §14
```

进入决策树的姿势：**先看症状，不要先看代码**。

具体的入口判别命令组合：

| 看什么 | 命令 | 判断点 |
| --- | --- | --- |
| 整机负载 | `uptime` | load average / 核数 > 1，说明跑不动了 |
| CPU 分布 | `top -bn1`、`mpstat -P ALL 1 3` | %us 高 = 业务忙；%sy 高 = 内核忙（GC/上下文切换）；%wa 高 = IO 等 |
| 内存 | `free -h`、`cat /proc/<pid>/status` | available 是不是接近 0；VmRSS 是不是异常增长 |
| IO | `iostat -x 1 3` | `%util` 接近 100、`await` 飙升 |
| 网络 | `ss -s`、`netstat -nat \| awk '{print $6}' \| sort \| uniq -c` | TIME_WAIT / CLOSE_WAIT 是否堆积 |
| Java 进程 | `jstat -gc <pid> 1s 10` | YGC / FGC 频率、Eden / Old 占比 |
| 线程 | `jstack <pid>` | BLOCKED / WAITING / RUNNABLE 分布 |

### 决策树的"分裂点"——多个症状同时出现

线上真实情况经常是**多个症状同时出现**，比如 CPU 高 + RT 抖动 + GC 频繁。这时候要分辨**因果链**：

- **GC 频繁 → STW → RT 抖动**：GC 是因，RT 是果
- **业务突增 → 分配速率高 → GC 频繁 → CPU 高**：业务流量是因，其他都是果
- **慢 SQL → 线程池阻塞 → 请求堆积 → 内存涨**：DB 是因

**定位手法**：找最早出现的那个症状作为入口。监控大盘按时间轴看，**第一个突变的指标就是因，后面的都是果**。

### 一个具体的决策树例子：接口 RT P99 突刺

```
RT P99 突刺
    │
    ├─ 同时 CPU 飙高？──→ §07（业务/序列化/GC）
    │
    ├─ 同时 GC 频繁？──→ §11（FullGC）+ JVM/16-19
    │
    ├─ 同时 DB QPS 没变但 RT 涨？──→ §12（慢 SQL / 锁）
    │
    ├─ 同时网络 RT 涨？──→ §15（丢包/RTT）
    │
    ├─ 同时下游服务 RT 涨？──→ 调用链上溯
    │
    └─ 全都没变？──→ 看 Trace，多半是某个偶发的慢调用
```

这种决策树会在 §07–§18 里每一个专题各画一张，**形式统一**：现象 → 同步看的指标 → 分叉到下一级。

## 六、定位过程中的几条铁律

### 铁律 1：先看面，再看点

监控大盘 → Trace 链路 → Log 详细堆栈。

**不要直接 grep 日志**。日志是面"看到细节"的工具，不是面"找问题"的工具。直接 grep 日志找根因的人，会浪费大量时间在不相关的日志噪音上。

### 铁律 2：用数据说话，不要"我觉得"

排障时最危险的一句话是"我觉得应该是 XXX 导致的"。**只有数据能证明因果**。

- "我觉得是 GC 问题" → 给我 GC 日志 + GCeasy 分析
- "我觉得是 DB 慢" → 给我 `EXPLAIN` + 慢查询日志
- "我觉得是网络问题" → 给我 `tcpdump` + RTT 统计

### 铁律 3：变量控制——只动一个

排查中常想"试一下重启""试一下扩容""试一下加缓存"——**一次只动一个**，否则恢复了也不知道是哪个起作用的。

更糟的是**多个动作同时改**，恢复了之后无法复盘根因，下次再发还是不会。

### 铁律 4：写下你的假设

每次怀疑一个根因，**写在群里或文档里**：

> 假设：本次 RT 抖动是 G1 Mixed GC 单次停顿过长导致
> 验证方法：拉 GC 日志，找停顿 > 500ms 的事件，确认是否在 RT 突刺时刻

写下来好处有三：

- 强制你想清楚怎么验证（如果想不出验证方法，假设就是空的）
- 让其他人能看到你的思路，避免重复劳动
- 复盘时可以回溯哪些假设对、哪些错

## 七、沟通规范：群里怎么同步

线上 P0/P1 拉群之后，最大的问题不是"没人干活"，而是"信息混乱"：

- A 在群里问"现在 RT 怎么样"
- B 跟一句"我看到 jstack 里有锁"
- C 说"我重启一台试试"
- 一片混乱，谁都不知道当前状态

老司机带群的姿势：**指定一个 IC（Incident Commander）**，所有动作必须经过 IC 确认才能做，且每 5 分钟由 IC 发一次状态同步：

```
【状态同步 21:35】
现象：订单创建接口 P99 1.2s（基线 80ms），错误率 0.3%
影响面：约 5% 用户，10 分钟内未恢复
当前动作：已回滚版本 v2.3.4 → v2.3.3，等待 5 分钟观察
负责人：@张三 回滚 / @李四 定位 / @王五 业务侧通知客服
```

每条同步包含：**现象、影响面、当前动作、下一步、负责人**。

## 八、复盘阶段：不是写检讨书

故障结束 24–72 小时内必须出复盘文档。复盘的目的**不是追责，是沉淀改进项**。

### 一个像样的复盘必须包含

1. **故障概述**：一句话说清"什么时候、什么业务、影响多大、持续多久"
2. **时间线**：精确到分钟，谁在什么时候做了什么
3. **直接原因**：技术层面的直接触发因素（"某个 SQL 走了全表扫描"）
4. **根因（5Why）**：往后追 5 层
   - Why 1：为什么这个 SQL 走全表扫描？因为没走索引
   - Why 2：为什么没走索引？因为数据倾斜，优化器选错了
   - Why 3：为什么会数据倾斜？因为某个新业务集中写入
   - Why 4：为什么新业务上线没评估写入量？因为流程里没这一步
   - Why 5：为什么流程里没这一步？因为大数据量场景上线评估清单缺失
5. **改进项**：每一项必须有 **owner + 截止时间 + 可验证标准**
   - ❌ "加强 SQL 评审" ——不可验证
   - ✅ "所有新表上线前必须由 DBA 在 wiki 模板 V2 上签字，2026-06-01 前生效，由 @张三 owner"

### 复盘的反模式

- **个人检讨书**："是我考虑不周……"——没用，要的是制度性改进
- **责备文化**："都是张三写的代码"——下次张三或别人还会犯，制度没变
- **没有 owner 的行动项**：写在文档里没人推进，三个月后再发同样故障

### 根因分类（用于长期统计）

公司层面如果做故障数据沉淀，建议把根因分类：

| 大类 | 子类 |
| --- | --- |
| 变更引入 | 代码 bug / 配置错误 / 数据变更 / 容量变更 |
| 容量不足 | 流量突增 / 资源规划失误 / 依赖容量不足 |
| 依赖故障 | 上游 / 下游 / 中间件 / 基础设施 |
| 设计缺陷 | 缺乏熔断 / 缺乏限流 / 缺乏降级 / 单点 |
| 人为操作 | 误删 / 误改配置 / 误发布 |

跑一年，统计**变更引入占比 70%+**几乎是必然的——这就是为什么"灰度发布 + 可观测的发布流程"是减少故障最高 ROI 的事。

## 九、几个常见反模式（新手最容易踩）

| 反模式 | 正确做法 |
| --- | --- |
| 告警一响就重启 | 先看监控确认现象，留至少一台不重启 |
| 死磕根因不止血 | 止血永远优先，业务在烧钱 |
| 上来就 grep 日志 | 先看监控面，再看 Trace，最后看日志 |
| 多个动作同时做 | 一次只动一个变量 |
| 复盘写检讨 | 复盘是为了改进项，不是追责 |
| 现场没留就重启/删 Pod | dump、jstack、监控截图、`lsof` 至少留一份 |
| 不写下假设硬猜 | 每个怀疑都写成可验证的假设 |
| 告警泛滥不分级 | P0/P1/P2 分级 + 黄金指标优先 |

## 十、本章小结与下一步

### 小结

线上排障的核心是**方法论 > 工具**。一套完整的排障流程：

1. 发现：告警 → 监控大盘确认 → 影响面评估 → 拉群分级
2. 止血：回滚 / 摘节点 / 限流 / 降级 → 业务恢复 → 留一台现场
3. 定位：现场留档（jstack / jmap / 监控截图 / 环境快照）→ 五大入口决策树 → 假设-验证循环
4. 复盘：时间线 → 5Why → 制度性改进项（owner + 截止时间 + 可验证）

四条铁律：**先看面再看点、用数据说话、一次只动一个变量、写下你的假设**。
