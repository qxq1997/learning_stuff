# 行为面试 - 第 1 课补充：从 AI 考试平台和 AWS 商户入驻项目出发练 Behavior 题

## 学习目标（本节结束后你能做到什么）

- 能把 `AI考试平台` 和 `AWS商户入驻与数据发布系统` 这两个项目，整理成可复用的 behavior 素材
- 知道同一个项目如何覆盖“复杂项目推进”“分歧处理”“失败复盘”“Owner意识”等多类题目
- 能直接拿着示范答案开口说，而不是继续停留在看模板阶段

## 内容讲解（核心概念，用类比、例子、图示说清楚）

很多人准备行为题时，最大的问题不是没项目，而是不会“拆故事”。  
一个做过的项目，明明信息很多，但一到面试就只会说：

- 我做了一个 AI 平台
- 我做了一个 AWS 工作流
- 用了 Step Functions、DynamoDB、LLM

这类回答的问题是，它们更像项目介绍，不像 behavior answer。  
behavior question 需要的不是“你做了什么系统”，而是：

- 这个系统里最难的矛盾是什么
- 你在那个矛盾里承担了什么责任
- 你做了什么判断和推动
- 最终结果是什么
- 你后来沉淀了什么方法

所以这节补充课不再泛讲，而是直接把两个项目拆成可复用主故事。

### 1. 先把两个项目分别压缩成一句“主故事定义”

#### 1.1 AI 考试平台项目主故事

这个项目最适合包装成：

`我把一个不稳定、不可审计的大模型能力，治理成一个可用于正式考试的出题和判卷平台。`

这个定义为什么强？  
因为它天然带出几个面试官喜欢追问的点：

- 模型不确定性
- 评分公平性
- 可审计性
- 成本与降级
- 线上治理能力

也就是说，这个项目很适合回答：

- 讲一个你主导的复杂项目
- 讲一个你做过的重要技术取舍
- 讲一次你和产品有分歧
- 讲一次你如何处理高风险问题
- 讲一次失败和复盘

#### 1.2 AWS 商户入驻项目主故事

这个项目最适合包装成：

`我把一个跨服务、带人工审核、还要同步分析数据的长流程入驻系统，做成了可推进、可恢复、可审计的工作流。`

这个定义为什么强？  
因为它天然带出：

- 长流程编排
- 跨团队协作
- 一致性与补偿
- 幂等和异常恢复
- 主链路与分析链路解耦

也就是说，这个项目特别适合回答：

- 讲一个复杂项目推进经历
- 讲一次跨团队协作
- 讲一次没有职权但推动事情发生
- 讲一次线上风险治理
- 讲一次取舍

### 2. 高频题一：讲一个你主导推进的复杂项目

这道题里，面试官最想判断的是：  
你是不是只能写一个模块，还是能把复杂事情真正往前推动。

#### 从 AI 考试平台角度回答

“我做过一个比较典型的复杂项目，是 AI 考试平台里的出题和判卷能力治理。这个项目难点不在于接一个大模型接口，而在于要把模型的不确定性，变成一个可以用于正式考试的能力。因为考试场景比普通 AI 产品敏感得多，题目质量、评分一致性、申诉追溯都不能出问题。  
我当时负责的重点不是某个单点接口，而是把整个链路从‘模型直接输出结果’改造成‘模型生成候选结果，平台做规则校验、版本冻结和人工复核兜底’。具体我做了三件事。第一，把题目、评分 Rubric、模型版本和 Prompt 版本全部做成可冻结、可追溯的版本化对象，避免同一场考试在不同时间得到不同结果；第二，把 AI 判卷改造成异步链路，用户提交答卷后先安全入库，再进入待判卷状态，避免外部模型故障阻断考试提交；第三，我补了人工复核和申诉需要的证据链，保存评分理由、关键证据片段和模型调用信息。  
最终这个项目把风险重点从‘能不能生成结果’转成了‘结果能不能被审计和复核’，整个系统的上线形态也更像正式平台而不是 demo。  
这件事让我最大的收获是，AI 项目最难的不是模型效果本身，而是如何定义平台最终认可什么、拒绝什么、出错后怎么兜底。” 

#### 从 AWS 商户入驻角度回答

“我做过一个很适合讲复杂度控制的项目，是 AWS 商户入驻工作流。这个场景表面上像一个提单系统，但本质上是一个跨服务长流程，里面有基础资料校验、主体去重、风控、合规、仓库校验、人工审核、正式开户，还要把状态同步到分析链路。  
我负责的核心不是某一个 Lambda，而是把这条分散在多个服务里的流程收敛成一个可推进、可恢复的工作流。我的动作主要有三块。第一，用 Step Functions Standard 把流程状态、分支、并行步骤、等待回调和重试逻辑显式化，让流程从‘藏在代码里’变成‘状态机里可见’；第二，把幂等和补偿机制补齐，比如入口防重复提交、工作流实例唯一、下游落正式数据时做条件写和去重，避免半成功状态失控；第三，把交易链路和分析链路解耦，主流程只负责推进申请状态，数据发布通过 DynamoDB Streams 到 S3、Glue、Redshift，避免报表系统反向拖垮入驻。  
最终这个项目最大的价值不是技术栈更好看，而是流程可观测性和异常恢复能力明显提升，卡单、重复提交、部分成功这种问题更容易被定位和收敛。  
这件事让我更明确，复杂项目的关键不是把组件堆出来，而是先把状态、边界和失败路径设计清楚。” 

### 3. 高频题二：讲一次你和产品或其他团队有分歧的经历

这道题最怕答成“我坚持技术正确，最后证明我是对的”。  
更好的答法是：先讲对方目标，再讲你的风险判断，最后讲你怎么提出折中方案。

#### 从 AI 考试平台角度回答

“在 AI 考试平台里，我和产品有过一次比较典型的分歧。产品一开始希望用户提交主观题后尽量实时出分，理由是这样体验更完整，尤其在练习场景下用户会觉得反馈更快。  
但我当时反对把所有场景都做成同步实时判卷，因为正式考试和练习场景的要求不一样。正式考试里如果把提交成功强绑定到模型实时评分完成，一旦 Provider 超时、限流或者评分结果需要复核，整个考试提交体验就会被外部依赖绑架，而且公平性和可审计性也很难保障。  
我没有直接否掉产品诉求，而是先把场景拆开。对于练习模式，可以接受低风险的快速反馈，就保留较快的自动评分体验；但对于正式考试模式，我坚持把‘提交成功’定义为‘答卷已安全入库并进入待判卷状态’，评分改为异步完成，必要时进入人工复核。  
为了推动这个方案落地，我还把申诉、复评、Provider 故障和评分边缘样本这几类异常场景列出来，让大家看到如果强行追求全场景实时，会把风险转嫁给业务。  
最后双方形成的方案是分模式设计，练习追求时效，正式考试优先公平和可追溯。  
这次分歧让我更明确，跨团队沟通时不要只强调技术限制，而要把不同目标下的风险和责任边界讲清楚。” 

#### 从 AWS 商户入驻角度回答

“在 AWS 商户入驻项目里，比较典型的一次分歧出现在多仓库处理上。业务侧一开始希望哪个仓库先校验通过，就先落一个正式绑定，这样看起来处理更快、结果更实时。  
但我当时认为这个做法风险很大。因为一个商户可能绑定多个仓库，如果前面 4 个仓库已经落正式态，第 5 个仓库后面又因为合规或状态问题失败，系统就会进入半成功状态。此时不管是回滚、补偿还是业务语义解释都会变得很麻烦。  
我没有简单说不能这么做，而是把流程拆成两个阶段来沟通：第一阶段是并发校验，尽可能提升效率；第二阶段是汇总结果后统一落正式态，保证业务含义清晰。如果业务上允许部分通过，也应该在汇总后统一标记哪些仓库成功、哪些待补充，而不是边校验边直接写正式数据。  
最后这个方案被接受，因为它既保留了并发校验的效率，又避免了过早写正式态带来的补偿复杂度。  
这件事让我觉得，很多分歧表面上是效率和稳定性的冲突，实际上是有没有把过程态和结果态分清楚。” 

### 4. 高频题三：讲一次失败经历，或者一次做得不够好的经历

失败题一定要选那种：  
你确实判断上有不足，但最后又形成了更成熟的方法。

#### 从 AI 考试平台角度回答

“如果讲一个做得不够好的经历，我会讲 AI 考试平台前期设计时，我一开始低估了评分解释和申诉链路的重要性。那时我更关注的是模型能不能把题判出来、结构化结果能不能回写成功，觉得只要总分准确，后面再慢慢补解释也可以。  
但后来在梳理正式考试场景时，我意识到这个判断不够成熟。因为考试结果不是普通推荐结果，用户是会追问‘为什么扣了这几分’的。如果没有评分理由、证据片段、Rubric 版本和模型版本留痕，后续申诉和人工复评会非常被动。  
我后面做了两方面修正。第一，把评分结果从一个简单分数字段，扩成带评分理由、证据和版本信息的完整结果对象；第二，把人工复核和申诉流程前置进设计，而不是把它当上线之后再补的运营功能。  
这次经历让我学到，AI 场景里‘结果能生成’和‘结果能承担责任’是两回事。后者如果不在设计期考虑，后面补起来成本会很高。” 

#### 从 AWS 商户入驻角度回答

“如果讲一次做得不够好的经历，我会讲 AWS 商户入驻项目里，我前期一度把重点放在主流程推进能跑通，低估了可观测性设计的重要性。也就是说，状态机、校验、审核这些核心步骤都设计了，但对‘卡在哪、为什么卡、谁来接手’这件事考虑得还不够系统。  
这个问题早期不一定体现成故障，但一旦流程涉及人工审核和外部依赖，缺少清晰状态字段、失败原因码和重试次数，排查成本会迅速升高。  
后面我做了修正，把申请单状态表设计补完整，明确记录当前步骤、当前状态、更新时间、失败原因和重试信息；同时把 Step Functions 执行历史和业务状态结合起来看，不再只依赖日志。  
这件事让我意识到，长流程系统里，‘流程能推进’只是第一步，‘卡住时所有人都知道该怎么定位和恢复’才算真的设计完整。” 

### 5. 高频题四：讲一次你没有直接职权，但推动了事情发生

这道题的重点不是“我帮了别人”，而是：

- 你看到了公共风险
- 你把风险讲明白了
- 你提出了别人愿意接受的落地方案

#### 从 AI 考试平台角度回答

“在 AI 考试平台里，我做过一次比较典型的横向推动。当时大家的关注点更多放在题目生成和判卷效果上，但我发现如果没有统一的模型网关、版本记录和调用留痕，后面做回归评估、问题追溯和 Provider 切换都会很痛苦。  
这件事并不完全在我单一模块的职责范围里，因为它涉及出题、判卷、平台治理和运维视角。我的做法不是先提一个大而全的改造，而是先拿几个具体风险说明问题，比如模型升级后评分漂移难定位、不同链路各自调模型导致版本不一致、Provider 限流时缺少统一降级入口。  
基于这些问题，我推动大家先统一了最核心的几项能力：模型调用统一走网关、关键请求记录模型和 Prompt 版本、正式考试链路默认保留审计日志。因为这几项都直接对应后续风险，团队接受度比较高。  
这件事让我认识到，横向推动不是靠职位，而是靠你能不能把系统性风险从‘感觉有问题’讲成‘不做会出什么问题，先做哪几步最划算’。” 

#### 从 AWS 商户入驻角度回答

“在 AWS 商户入驻项目里，我有一次没有直接管理权限，但推动了一个很关键的统一动作。当时各服务对于幂等和重复提交的处理并不统一，入口有入口的去重方式，下游落库又是另一套逻辑，数据发布链路还有自己的事件去重规则。  
短期看每个团队都能解释自己的方案，但从系统角度看，这会形成责任断层，一旦出现重复申请、重复开户或者报表重复计数，就很难快速收敛。  
我当时先整理了几个典型风险场景，包括前端重复点击、工作流重试、下游事件重复消费和补数回放，然后把这些场景映射到现有实现方式里，指出哪些地方缺口最大。之后我推动先统一三件事：申请单唯一键、工作流实例唯一标识、数据发布事件去重规则。  
因为我不是要求所有团队一次性重构，而是先解决最危险的公共风险，所以这件事比较顺利地推进下去了。  
这件事让我形成一个判断，没有职权时，推动事情最有效的方式不是喊统一，而是先把最小可落地的公共规则收敛出来。” 

### 6. 两个项目分别最适合打哪类行为题

#### AI 考试平台更适合打的题

- 讲一个高风险项目
- 讲一次技术与业务目标之间的取舍
- 讲一次你和产品的分歧
- 讲一次你如何处理质量、公平性或责任边界问题
- 讲一次失败和复盘

这个项目的强点在于：

- AI 不确定性
- 公平性和审计
- Provider 故障与降级
- 模型治理

这些关键词会让你的回答显得更有判断力。

#### AWS 商户入驻更适合打的题

- 讲一个复杂项目推进
- 讲一次跨团队协作
- 讲一次一致性和补偿设计
- 讲一次资源有限下的取舍
- 讲一次没有职权但推动事情发生

这个项目的强点在于：

- 长流程编排
- 状态机
- 幂等和补偿
- 主链路与分析链路解耦
- 可观测性和审计

这些关键词会让你的回答显得更像成熟后端工程师。

### 7. 最实用的面试策略：两个项目，各准备一主一副

如果你现在就要开始准备，我建议你这样搭配：

#### 主故事 1：AWS 商户入驻

把它当成你的“复杂项目推进”主故事。  
优先覆盖：

- 复杂项目
- 跨团队协作
- 没有职权但推动事情发生
- 技术取舍

#### 主故事 2：AI 考试平台

把它当成你的“高风险治理与判断力”主故事。  
优先覆盖：

- 分歧处理
- 失败复盘
- 高风险系统设计
- Owner 意识

这样搭配的好处是，你不会两个项目都讲成系统设计题。  
一个偏流程与协作，一个偏治理与判断，信息密度会更高，也更不容易重复。

## 小结（3-5 条关键点）

- `AI考试平台` 最适合讲“模型不确定性如何被治理成正式业务能力”。
- `AWS商户入驻` 最适合讲“长流程、跨服务系统如何被做成可推进、可恢复、可审计的工作流”。
- 同一个项目不要试图回答所有题，而是要先定义它最适合打哪几类 behavior question。
- 行为题的强弱不取决于技术名词多少，而取决于你能不能把矛盾、判断、动作、结果、复盘讲完整。
- 对你来说，最稳的策略是把 `AWS onboarding` 当复杂项目主故事，把 `AI exam platform` 当高风险治理主故事。
