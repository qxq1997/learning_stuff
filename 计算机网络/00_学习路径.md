# 计算机网络学习路径与知识图谱

## 学习定位

计算机网络面试最容易出现一种错觉：很多题看起来都听过，真正追问时却会卡在细节上。

比如：

- 知道 TCP 三次握手，但说不清为什么不是两次。
- 知道 HTTPS 比 HTTP 安全，但说不清证书链和密钥协商。
- 知道 DNS 是域名解析，但说不清递归查询、迭代查询、缓存和 TCP/UDP。
- 知道 TIME_WAIT，但说不清为什么要等 2MSL，以及大量 TIME_WAIT 到底该不该优化。
- 知道慢启动、拥塞避免、快重传、快恢复，但不会把拥塞窗口变化讲成一条完整曲线。

这套专题的目标不是背协议名，而是把“应用请求如何一步步变成网络报文，又如何被对端还原”的链路讲清楚。

## 知识图谱

1. 协议栈主线
   - OSI 七层模型
   - TCP/IP 四层模型
   - 应用数据的封装与解封装
   - 每一层到底解决什么问题
2. 应用层
   - HTTP 报文结构
   - 请求方法与幂等性
   - 状态码
   - 长连接、断点续传、无状态
   - Cookie、Session、JWT、WebSocket
3. 安全与解析
   - HTTPS 在 HTTP 与 TCP 之间加入 TLS
   - TLS 1.2 与 TLS 1.3 的握手差异
   - 证书链、CA、公钥、会话密钥
   - DNS 层级、递归查询、迭代查询、缓存
4. 传输层
   - TCP 头部字段
   - 三次握手
   - 半连接队列、全连接队列、accept
   - 四次挥手
   - TIME_WAIT、CLOSE_WAIT、FIN_WAIT2
   - TCP Keepalive 与应用层心跳
5. 可靠性与性能
   - 序列号、确认应答
   - 超时重传、快速重传
   - 流量控制与接收窗口
   - 拥塞控制与拥塞窗口
   - 粘包不是 TCP 的错误，而是字节流协议的自然结果
6. 工程排查与安全
   - 页面加载慢如何拆链路
   - ping 不通但 HTTP 能访问为什么可能成立
   - DDoS、SYN Flood、DNS 劫持
   - 抓包、状态码、日志、内核连接状态如何结合判断

## 面试表达原则

答网络题时，尽量不要只背结论。更好的结构是：

1. 先说明这个协议或机制解决什么问题。
2. 再说明它在协议栈哪一层。
3. 然后按请求链路讲关键步骤。
4. 最后补异常场景和工程边界。

例如问“三次握手为什么不是两次”，不要只说“为了确认双方收发能力”，还要继续说：

- 一次握手只能让服务端知道客户端能发。
- 两次握手会让服务端过早建立连接，旧 SYN 或第二次握手丢失时容易产生半开连接或资源浪费。
- 三次握手让双方都确认对方的发送和接收能力，并同步双方初始序列号。
