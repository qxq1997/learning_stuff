# 第 8 课：TCP 可靠传输与粘包：序列号、ACK、重传、流量控制与消息边界

## 学习目标

- 理解 TCP 如何实现可靠、有序、无重复的字节流传输。
- 掌握序列号、ACK、超时重传、快速重传。
- 区分流量控制和拥塞控制。
- 解释 TCP 粘包为什么不是 TCP 的 bug，以及应用层如何解决边界问题。

## TCP 为什么可靠

TCP 的可靠性来自一组机制：

- 连接管理：三次握手和四次挥手管理连接生命周期。
- 序列号：每个字节都有逻辑序号。
- 确认应答：接收方通过 ACK 告诉发送方收到了哪里。
- 重传机制：超时或重复 ACK 触发重传。
- 接收窗口：接收方告诉发送方自己还能接收多少数据。
- 拥塞控制：发送方根据网络状况调节发送速度。
- 校验和：检测报文在传输中是否损坏。

## 序列号与确认号

TCP 是字节流协议，序列号不是“第几个包”，而是“字节编号”。

如果发送方从 Seq=1000 开始发送 500 字节，接收方成功收到后，会回复：

```text
Ack = 1500
```

含义是：1500 之前的数据我都收到了，下次请从 1500 开始发。

这种累计确认让 ACK 更紧凑，但也意味着接收方通常确认的是连续收到的最大边界。

## 超时重传

发送方发出数据后会启动重传计时器。如果在 RTO 内没有收到 ACK，就认为可能丢包，重传对应数据。

RTO 不能太短，否则网络只是稍慢也会误重传；也不能太长，否则真实丢包恢复慢。TCP 会根据 RTT 动态估算 RTO。

## 快速重传

如果接收方发现中间缺了一段数据，会持续 ACK 已经连续收到的位置。

发送方如果收到多个重复 ACK，说明后面的数据可能已经到了，只是中间某段丢了。此时可以不等超时，直接重传缺失段。

示例：

```text
发送: 1 2 3 4 5
丢失: 3
接收方收到 4、5 后仍然回复 ACK=3
发送方收到多个 ACK=3，快速重传 3
```

快速重传能显著缩短丢包恢复时间。

## 流量控制

流量控制解决的是“接收方处理不过来”的问题。

接收方通过 TCP 窗口字段告诉发送方自己的接收缓冲区剩余空间。发送方最多只能发送窗口允许的数据量。

如果接收方窗口变成 0，发送方要暂停发送，并通过窗口探测机制等待窗口恢复。

流量控制关注接收端能力，不是网络拥塞。

## 拥塞控制与流量控制的区别

| 机制 | 解决的问题 | 由谁反馈 | 核心变量 |
| --- | --- | --- | --- |
| 流量控制 | 接收方缓冲区是否扛得住 | 接收方 | 接收窗口 rwnd |
| 拥塞控制 | 网络链路是否扛得住 | 网络表现间接反馈 | 拥塞窗口 cwnd |

实际发送窗口通常受二者共同限制：

```text
发送窗口 = min(rwnd, cwnd)
```

## TCP 粘包是什么

TCP 提供的是连续、有序、可靠的字节流，不保留应用层消息边界。

应用写入：

```text
write("hello")
write("world")
```

接收端可能读到：

```text
"helloworld"
```

也可能读到：

```text
"hel"
"lowor"
"ld"
```

这不是 TCP 出错，而是 TCP 从来没有承诺“write 一次，对方 read 一次”。

## 如何解决粘包

应用层协议必须自己定义消息边界。

常见方式：

1. 固定长度：每条消息固定 N 字节。
2. 分隔符：例如文本协议用 `\r\n` 或特殊分隔符。
3. 长度字段：消息头里写 body 长度。
4. TLV：Type-Length-Value，类型、长度、值。

HTTP 就是很好的例子：

- 请求行和 Header 用 CRLF 分隔。
- Header 和 body 用空行分隔。
- body 长度通过 `Content-Length` 或 chunked 机制判断。

所以“TCP 粘包怎么解决”的标准答案不是“关掉 Nagle 就行”。Nagle 可能影响小包合并，但消息边界必须由应用协议定义。

## UDP 为什么不可靠

UDP 是面向数据报的协议，保留报文边界，但不保证：

- 到达。
- 顺序。
- 不重复。
- 拥塞控制。
- 流量控制。

UDP 的优点是轻量、低延迟、无连接。实时音视频、DNS、游戏、QUIC 都会在不同方式下利用 UDP。

QUIC 基于 UDP 实现可靠性、拥塞控制、多路复用和 TLS 集成，所以不能简单说“UDP 一定不适合可靠传输”，准确说是 UDP 协议本身不提供可靠性。

## 小结

- TCP 靠序列号、ACK、重传、窗口和拥塞控制实现可靠传输。
- ACK 通常表示“期望收到的下一个字节序号”。
- 超时重传靠 RTO，快速重传靠重复 ACK。
- 流量控制看接收方，拥塞控制看网络。
- TCP 是字节流协议，没有应用层消息边界，粘包要靠应用协议解决。
