# 认证与授权 - 第 4 课：SSO、权限模型与其他常见认证方案：企业系统怎么做统一控制

## 学习目标（本节结束后你能做到什么）

- 理解 `SSO` 解决的是多系统共享登录状态，而不是单纯换个 token 格式。
- 说清企业统一身份里 `IdP`、业务系统、网关、权限中心之间的大致关系。
- 分清 `RBAC`、`ABAC`、`ACL`、数据权限各自适合什么问题。
- 知道 `Basic Auth`、`API Key`、`HMAC`、`mTLS` 这些其他常见认证方式的适用场景。
- 面试时能从“统一身份 + 统一授权 + 数据隔离”的角度回答权限系统设计题。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

### 1. SSO 到底在解决什么问题

`SSO` 全称是 `Single Sign-On`，单点登录。  
它解决的问题不是“怎么生成 token”，而是：

**一个用户登录一次后，能否在多个彼此独立的系统之间共享登录状态。**

比如公司里有这些系统：

- OA
- CRM
- 报销系统
- 数据平台
- 审批系统

如果没有 SSO，用户要在每个系统里分别登录一次。  
不仅体验差，还会带来：

- 账号体系重复建设
- 密码策略不统一
- 用户离职后多系统清理麻烦
- 安全审计分散

所以 SSO 的核心是“统一身份入口”。

### 2. SSO 不是 OAuth2 的同义词

这一点非常重要。

很多公司口头会说“我们做 OAuth 登录，所以就有 SSO 了”。  
这不一定错，但也不够准确。

更准确的说法是：

- `SSO` 是一种系统能力或产品能力
- 它底层可以用多种协议实现
- `OIDC`、`SAML` 都常被用来承载 SSO

所以 SSO 更像目标，协议更像实现手段。

### 3. 企业统一登录里常见角色是什么

最常见的一套角色可以先记成这样：

- `IdP`：Identity Provider，身份提供方
- `SP`：Service Provider，业务系统
- 用户浏览器或客户端
- 可能还会有网关、权限中心、用户中心

一个简化后的思路是：

1. 用户访问业务系统 A
2. A 发现用户没登录
3. A 跳转到统一身份平台 `IdP`
4. 用户在 `IdP` 完成认证
5. `IdP` 返回身份断言或 token
6. A 建立本地登录态
7. 用户再访问业务系统 B 时，B 也跳到 `IdP`
8. 因为 `IdP` 已经有统一会话，所以用户无需再次输入密码

这就是“单点登录”的核心体验。

### 4. SSO 的关键不只是单点登录，还有单点登出

很多系统做登录共享时，容易只做前半段，不做后半段。

问题是：

- A 系统退出了
- B 系统还保持登录
- `IdP` 的全局会话也还在

这会让用户和安全团队都很困惑。

所以真正完整的企业 SSO，通常还要考虑：

- 全局登录会话
- 各业务系统本地会话
- 单点登出传播
- 会话过期和续期

这也是为什么 SSO 一旦进入企业级，复杂度会明显上升。

### 5. 登录统一了，不代表权限也自动统一了

这是很多系统设计里第二个大坑。

统一身份平台解决的是：

- 你是谁
- 你是否已登录

但真正业务权限还包括：

- 你是不是管理员
- 你能访问哪个菜单
- 你能操作哪个 API
- 你能看哪个部门的数据
- 你是否处在某个审批节点

所以企业系统通常至少拆成两块：

- 身份认证中心
- 权限控制中心

登录打通只是第一步，权限统一往往更难。

### 6. RBAC 为什么最常见

`RBAC` 全称 `Role-Based Access Control`，基于角色的访问控制。

它的思路很朴素：

- 给用户分配角色
- 给角色分配权限
- 用户通过角色继承权限

比如：

- 普通员工
- 部门主管
- 财务
- 超级管理员

RBAC 的优点非常明显：

- 好理解
- 好配置
- 好做后台管理
- 很适合菜单、按钮、API 粒度的权限

所以绝大多数后台系统，第一版权限模型几乎都是 RBAC 起步。

### 7. 为什么 RBAC 用着用着就不够了

因为很多真实世界限制，根本不是“角色”三个字能表达完的。

例如：

- 只能看自己创建的数据
- 只能看本部门数据
- 跨部门借调期间临时多一组权限
- 某个功能只在工作时间开放
- 某个操作需要订单状态为“待审核”才能执行

这时你会发现，权限判断开始依赖：

- 用户属性
- 资源属性
- 环境属性
- 业务状态

RBAC 还是有用，但不够用了。

### 8. ABAC 在解决什么

`ABAC` 全称 `Attribute-Based Access Control`，基于属性的访问控制。

它不只看角色，而是看多个维度的属性组合。

例如一条规则可能长这样：

- 用户部门 = 资源所属部门
- 用户职级 >= 资源要求职级
- 当前时间在工作日 9:00-18:00
- 操作类型 = 只读

ABAC 更灵活，但代价是：

- 规则表达更复杂
- 调试更困难
- 管理后台更难做
- 出问题时不如 RBAC 直观

所以现实里常见做法不是“全 RBAC”或“全 ABAC”，而是：

**RBAC 打底，ABAC 做细化补充。**

### 9. ACL 和数据权限又是什么位置

#### ACL

`ACL` 可以理解成针对具体资源对象挂权限列表。

比如某篇文档：

- Alice 可读
- Bob 可编辑
- Carol 无权限

ACL 很适合对象级共享场景，比如：

- 文档协同
- 文件分享
- 某条记录定向授权

#### 数据权限

数据权限常常是企业系统里最复杂的那层。

因为它不是简单“能不能进页面”，而是：

- 能看哪些行
- 能看哪些列
- 能不能导出
- 能不能看脱敏前数据

典型数据权限模型可能包括：

- 仅本人
- 本部门
- 本部门及子部门
- 指定组织范围
- 全部数据

很多系统最开始只做菜单和接口权限，等真的上财务、人事、CRM、运营平台之后，才发现最难的是数据权限。

### 10. 一个更贴近真实系统的权限分层

你可以把权限控制先粗分成四层：

1. 菜单权限
   - 决定看不看得到入口
2. 按钮 / 操作权限
   - 决定能不能点“新增”“删除”“导出”
3. API 权限
   - 决定请求打到后端时是否放行
4. 数据权限
   - 决定放行之后到底能看到哪些数据

这四层不能互相替代。

比如一个按钮前端隐藏了，如果后端接口不拦，抓包照样能调。  
又比如 API 放行了，如果 SQL 不做数据范围过滤，也可能越权看到不该看的记录。

### 11. 其他常见认证方式应该放在哪个语境理解

除了用户名密码、Session、JWT、OAuth、SSO，工程上还常见这些方案。

#### 11.1 Basic Auth

就是 HTTP 里最经典的用户名密码头。  
简单直接，但通常只适合：

- 内部调试
- 临时保护
- 简单接口

不适合复杂用户系统。

#### 11.2 API Key

常见于开放平台、服务接口。  
它更像“应用身份”，不是“用户身份”。

适合：

- 服务调用
- 平台开放 API
- 区分调用方配额与来源

#### 11.3 HMAC / 签名认证

请求里除了身份凭证，还要对请求内容做签名，防止篡改和伪造。  
常见于：

- 支付回调
- 云服务开放接口
- 高安全要求的服务间调用

#### 11.4 mTLS

双向 TLS。  
不仅客户端验证服务端证书，服务端也验证客户端证书。

它常用于：

- 服务到服务身份认证
- Service Mesh
- 高安全内网通信

这里的重点不是“登录哪个用户”，而是“这个调用方服务到底是谁”。

### 12. 面试里怎么把这一整套讲得像工程而不是背书

如果面试官问你“讲讲权限系统怎么设计”，一个很稳的答法是：

1. 先分身份认证和权限授权
2. 身份统一通常通过登录中心 / IdP / SSO 解决
3. 权限控制通常先做 RBAC，复杂场景再叠加 ABAC 和数据权限
4. 前端菜单控制只是体验层，真正安全要落在 API 与数据层
5. 如果有开放平台或服务间通信，再补 API Key、签名、mTLS 等机制

这样讲，层次会非常清楚。

## 小结（3-5 条关键点）

- SSO 解决的是多系统共享登录，不是单纯一种 token 技术。
- 企业统一身份和统一权限通常不是同一回事，登录打通后仍然要单独设计权限体系。
- RBAC 适合做基础权限骨架，ABAC 适合处理更细粒度、更动态的条件判断。
- ACL 适合对象级共享，数据权限则常常是企业系统里最复杂、最容易越权的一层。
- Basic Auth、API Key、HMAC、mTLS 各有语境，很多时候服务身份认证和用户身份认证不是一回事。
