# 设计模式 - 第 2 课：设计原则、SOLID 与代码坏味道

## 学习目标（本节结束后你能做到什么）

1. 用工程语言解释 SOLID，而不是只背五个英文缩写。
2. 能从一段后端业务代码里识别职责混乱、依赖扩散、接口臃肿、继承滥用等坏味道。
3. 能判断什么时候应该抽象，什么时候应该保持朴素实现。
4. 能把设计原则和生产中的扩展、测试、发布、排障联系起来。

## 先建立一个判断：原则不是规矩，是降低变更成本的工具

设计原则听起来很像“代码道德规范”，比如类要单一职责、接口要小、依赖抽象、对扩展开放。实际工作里不能这么理解。原则不是为了让代码显得标准，而是为了降低未来变更的成本。

比如你写一个订单服务，第一版只是创建订单、扣库存、发消息。后来促销、风控、履约、售后、对账全都加进来。如果所有逻辑都堆在 `OrderService.createOrder` 里，这个方法会越来越像一个小型系统：它知道太多业务规则，依赖太多外部组件，也承担太多失败处理。问题不只是“方法长”，而是以后任何一条规则变化，都可能影响整条链路。

所以设计原则背后的核心问题是：

1. 哪些代码应该放在一起变化？
2. 哪些变化应该被隔离在局部？
3. 哪些依赖方向会让高层业务被底层技术绑死？
4. 哪些抽象是真变化，哪些抽象只是想象出来的复杂度？

你可以把设计原则理解成“模式之前的诊断工具”。模式是药方，原则是体检指标。没有原则，模式容易变成套模板；没有模式，原则又可能停留在口号。

## 几个最底层的设计直觉

在进入 SOLID 前，先抓住几条更朴素的直觉。

### 高内聚、低耦合

高内聚是说：一个模块内部的代码应该围绕同一个目的工作。比如 `PaymentCallbackVerifier` 只负责验签，`PaymentStatusMapper` 只负责把三方状态映射成内部状态。

低耦合是说：模块之间知道彼此越少越好。订单模块不应该知道微信支付返回字段叫 `trade_state`，也不应该知道某个 MQ SDK 的异常类型。它应该依赖业务含义清晰的接口，比如 `PaymentGateway`、`EventPublisher`。

适用场景：业务会长期演进，多个团队协作，一个模块改动可能影响多个调用方。

不适用场景：一次性脚本、临时数据修复、非常小的工具函数。为了“低耦合”把十行脚本拆成五个类，就是把维护成本提前花掉。

### 封装变化

设计模式里最重要的一句话是：找到变化点，然后封装它。

支付渠道会变，就封装成 `PaymentStrategy`。第三方接口字段会变，就放到 adapter 里。订单状态流转会变，就集中在状态机或状态服务里。封装变化的目标不是消灭变化，而是让变化发生时只改一个清晰的位置。

不适用场景：你还没看清变化维度，只是觉得“以后可能会变”。这时先保留简单实现，等第二次、第三次变化出现后再抽象，通常更稳。

### 组合优于继承

继承适合表达稳定的“is-a”关系，组合适合表达可替换的能力。后端业务里，大多数变化不是“它是什么”，而是“它能搭配哪些规则”。比如订单可以组合不同的定价策略、风控规则、通知方式，而不应该继承出 `VipCouponFlashSaleOrder` 这类越来越长的子类。

适用场景：能力可以独立替换、组合、装配，比如策略、装饰器、责任链。

不适用场景：框架扩展点明确要求继承，或者领域层确实存在稳定层级。即便如此，也要避免把业务分支全塞进父类。

### 面向接口编程

面向接口不是“每个类都配一个 interface”。它的本质是：调用方依赖自己真正需要的能力，而不是依赖某个具体实现的全部细节。

如果一个类只有一个实现，而且短期没有替换、测试隔离、插件扩展的需求，不一定要立刻抽接口。接口应该从调用方需求长出来，而不是从实现类影子里复制出来。

## SOLID 逐条拆解

### S：单一职责原则

单一职责原则不是“一个类只能有一个方法”，而是一个模块应该只有一个主要变化原因。

坏味道：

- `OrderService` 同时做参数校验、价格计算、库存扣减、支付下单、事务控制、消息发送、日志告警。
- `User` 实体里既有用户属性，又有数据库查询、HTTP 调用和缓存刷新。
- 一个方法里同时出现业务规则、技术细节和监控埋点。

更好的拆法：

- 应用服务负责流程编排，比如 `OrderApplicationService`。
- 领域对象或领域服务负责业务规则，比如 `PricingService`、`InventoryPolicy`。
- 基础设施适配器负责外部系统，比如 `PaymentGatewayAdapter`、`MessagePublisher`。
- 横切能力通过拦截器、装饰器、AOP 或框架能力处理，比如日志、指标、鉴权。

适用场景：一个类越来越难命名，或者每次需求变化都要改它。

不适用场景：为了追求“纯粹”，把一个本来很连贯的业务动作拆成一堆只有一行代码的小类。单一职责的目标是让变化更清晰，不是让文件数量变多。

生产问题与解决方法：

| 生产问题 | 常见表现 | 解决方法 |
| --- | --- | --- |
| God Service 难维护 | 一个服务类几千行，新人不敢改 | 先按变化原因切方法，再提取策略、适配器、领域服务 |
| 事故定位慢 | 业务逻辑、重试、告警混在一起 | 把横切能力移到统一组件，业务代码只表达业务分支 |
| 测试很难写 | 测一个规则必须 mock 十几个外部依赖 | 把纯业务规则从应用服务里抽出来，优先做无外部依赖的单元测试 |

### O：开闭原则

开闭原则是对扩展开放，对修改关闭。它不是说旧代码永远不能改，而是说当某个变化维度反复出现时，新增变化最好通过新增实现完成，而不是频繁修改稳定流程。

典型例子是支付方式：

```java
interface PaymentStrategy {
    PaymentResult pay(PaymentCommand command);
}
```

当微信、支付宝、银行卡、余额支付的差异集中在各自策略里，新增一种支付方式就主要是新增一个策略实现，而不是反复修改一大段 `if-else`。

适用场景：

- 变化维度清楚，比如渠道、规则、算法、状态、处理步骤。
- 新增类型的频率高于修改主流程的频率。
- 主流程稳定，变化点可以被接口表达清楚。

不适用场景：

- 需求还在探索期，业务边界每天都变。
- 只有一个实现，且短期没有替换需求。
- 抽象出来的接口需要不断改签名，说明抽象还没稳定。

生产问题与解决方法：

| 生产问题 | 常见表现 | 解决方法 |
| --- | --- | --- |
| 策略漏注册 | 新增实现类后线上找不到处理器 | 启动时校验策略表完整性，枚举和实现一一对应 |
| 默认分支吞错误 | 未知类型走了默认策略，产生错误订单 | 未知类型默认失败，打告警，不要静默兜底 |
| 抽象频繁变更 | 每加一个渠道都要改接口参数 | 重新识别稳定输入，使用 command/context 承载扩展字段 |

### L：里氏替换原则

里氏替换原则说的是：任何使用父类型或接口的地方，都应该能安全替换成它的子类型或实现类。

这条原则经常被背得很抽象，落到代码里就是一句话：实现类不能偷偷破坏接口承诺。

坏例子：

- `DiscountStrategy.calculate()` 约定返回非负金额，某个实现返回了负数。
- `StorageClient.save()` 约定保存失败抛业务异常，某个实现失败时直接返回 `null`。
- `PaymentGateway.pay()` 约定幂等，某个渠道适配器重复调用会重复扣款。

适用场景：所有策略、适配器、插件、仓储接口、客户端接口。

不适用场景：不要为了“复用代码”硬造继承层级。LSP 约束的是替换关系，不是鼓励继承。

生产问题与解决方法：

| 生产问题 | 常见表现 | 解决方法 |
| --- | --- | --- |
| 同一接口语义不一致 | A 实现抛异常，B 实现返回空，调用方到处特判 | 写清接口契约，统一异常、空值、幂等、超时语义 |
| 新实现破坏主流程 | 新渠道上线后某些订单状态异常 | 给接口加 contract test，所有实现共享同一套行为测试 |
| 子类覆盖父类规则 | 父类校验被绕过 | 少用继承复用业务逻辑，优先组合公共组件 |

### I：接口隔离原则

接口隔离原则是说：调用方不应该被迫依赖自己不需要的方法。

坏味道：

```java
interface UserService {
    User getUser(String id);
    void updateProfile(User user);
    void resetPassword(String id);
    void sendMarketingMessage(String id);
    void freezeAccount(String id);
}
```

如果订单模块只需要查询用户，却被迫依赖一个大而全的 `UserService`，那它就会被用户模块的很多无关变化影响。更好的方式是按调用场景拆成小接口，比如 `UserQueryPort`、`AccountRiskPort`。

适用场景：

- 一个接口被多个模块使用，而不同模块只需要其中一小部分能力。
- 接口变更经常牵连无关调用方。
- 你在 mock 接口时必须实现很多无关方法。

不适用场景：

- 接口天然就是一个小而完整的能力，比如 `Clock.now()`、`IdGenerator.nextId()`。
- 刚拆出来的小接口没有稳定语义，只是把一个大接口机械拆碎。

生产问题与解决方法：

| 生产问题 | 常见表现 | 解决方法 |
| --- | --- | --- |
| 大接口牵连发布 | 改一个方法，很多服务都要重新编译或适配 | 按消费者视角拆 port，保留兼容层逐步迁移 |
| mock 很重 | 单测为了一个查询方法 mock 十几个无关方法 | 拆出测试对象真正需要的小接口 |
| 权限边界模糊 | 一个接口暴露太多能力，被不该调用的模块调用 | 接口按业务权限和调用场景拆分 |

### D：依赖倒置原则

依赖倒置原则是 SOLID 里最容易改变代码结构的一条：高层策略不应该依赖低层细节，二者都应该依赖抽象。

在后端里，高层策略通常是业务流程，比如下单、退款、审核、结算。低层细节是数据库、缓存、MQ、HTTP SDK、第三方支付 SDK。

坏味道：

```java
class RefundService {
    private final WeChatPayClient client = new WeChatPayClient();
}
```

这样 `RefundService` 被微信 SDK 绑住了。你要测试退款规则，就必须处理真实 SDK 的构造、配置和异常。更好的方式是依赖业务接口：

```java
class RefundService {
    private final RefundGateway refundGateway;

    RefundService(RefundGateway refundGateway) {
        this.refundGateway = refundGateway;
    }
}
```

适用场景：

- 高层业务逻辑需要独立测试。
- 外部依赖有替换可能，比如三方服务、存储、消息、搜索。
- 你希望通过 mock、fake、stub 做边界测试。

不适用场景：

- 对稳定标准库的简单调用，比如 `Math`、`String`、本地不可变值对象。
- 为了依赖倒置，把所有工具函数都包装成接口，反而让代码更绕。

生产问题与解决方法：

| 生产问题 | 常见表现 | 解决方法 |
| --- | --- | --- |
| 三方 SDK 污染业务层 | 业务代码里到处是 vendor model 和错误码 | 用 adapter/anti-corruption layer，把外部模型转成内部模型 |
| 单测依赖真实环境 | 跑测试需要 Redis、MQ、真实支付配置 | 业务层依赖 port，测试注入 fake 实现 |
| 替换基础设施成本高 | 从一个 MQ 换到另一个 MQ 要改大量业务代码 | 业务层依赖 `EventPublisher`，具体 MQ 放在基础设施层 |

## SOLID 之外，生产里也很重要的原则

### DRY：不要重复自己，但别急着消灭所有重复

重复有两种：一种是同一知识的重复，另一种只是长得像的代码。

比如两个支付渠道都有“验签、映射状态、更新订单”，这可能是同一流程骨架，适合模板方法或组合复用。可如果两个业务只是刚好都写了三行类似判断，但业务含义完全不同，强行抽公共方法会造成错误耦合。

实战经验：第一次重复可以忍，第二次重复要警觉，第三次重复再认真抽象。抽象前先问：它们未来会一起变化，还是只是现在长得像？

### KISS：保持简单

简单不是简陋，而是让读代码的人能直接看懂主路径。能用清晰方法命名解决的问题，不要先上复杂模式。能用一个小策略表解决的问题，不要引入完整规则引擎。

适用场景：业务还小，团队成员水平不一，线上问题需要快速定位。

不适用场景：复杂度已经客观存在。比如订单状态有几十种转移，继续说“保持简单”然后堆 `if-else`，其实是在把复杂度藏起来。

### YAGNI：你现在不需要它

YAGNI 反对为想象中的未来提前建设复杂框架。它不是反对设计，而是提醒你：抽象应该有证据。

证据包括：

- 已经出现第二个或第三个实现。
- 产品路线明确会新增同类能力。
- 不抽象会影响多个模块、多个团队或多个发布窗口。
- 这个位置一旦改错，线上损失很大。

### 最少知识原则

一个对象不应该穿透太多层对象去拿内部细节。

坏味道：

```java
order.getCustomer().getAccount().getLevel().isVip()
```

更好的方式是让领域对象提供业务语义：

```java
order.isVipCustomerOrder()
```

这不是为了少写点链式调用，而是为了避免调用方了解太多内部结构。内部结构一变，外面全跟着变。

## 如何判断“该抽象”还是“过度设计”

可以用这张表快速判断：

| 判断维度 | 倾向保持简单 | 倾向引入抽象 |
| --- | --- | --- |
| 实现数量 | 只有一个实现 | 已有多个实现，或短期明确新增 |
| 变化频率 | 很少变化 | 同一维度反复变化 |
| 影响范围 | 只影响一个小方法 | 牵动多个模块、多个团队、多个测试用例 |
| 业务稳定性 | 需求还在探索 | 业务语义稳定，只是实现扩展 |
| 发布风险 | 改错影响有限 | 改错会造成资损、错单、不可逆数据 |
| 测试成本 | 手工验证即可 | 需要隔离外部依赖做自动化测试 |

一个很实用的顺序是：

1. 先写朴素实现，让业务跑通。
2. 第二次出现类似变化时，提取方法或小对象。
3. 第三次出现同类变化时，再引入接口、策略、工厂、责任链等稳定结构。
4. 如果这个位置涉及资损、权限、安全，可以比普通业务更早抽象和测试。

## 生产中最常见的设计问题

| 问题 | 表现 | 根因 | 解决方法 |
| --- | --- | --- | --- |
| 过度抽象 | 一个接口一个实现，跳转链路很长 | 只有模式意识，没有变化证据 | 合并无意义接口，保留真正稳定的扩展点 |
| 抽象泄漏 | 业务层到处出现三方字段、SDK 异常 | adapter 边界没做好 | 外部模型只在基础设施层出现，进入业务层前转成内部模型 |
| 策略失控 | 策略类越来越多，但规则归属不清 | 变化维度没分清 | 按业务维度分包，策略命名包含场景和规则 |
| 依赖环 | A 服务依赖 B，B 又回调 A | 职责边界不清，流程编排散落 | 抽出应用服务、领域事件或中介流程，重新定义依赖方向 |
| 接口语义漂移 | 同一接口不同实现行为不一致 | 没有契约测试 | 写接口契约、共享测试、统一异常和返回语义 |
| 横切逻辑污染 | 每个业务方法都写日志、鉴权、重试 | 技术能力侵入业务 | 用 interceptor、decorator、AOP、middleware 统一处理 |
| 状态判断散落 | 订单状态规则到处复制 | 状态转移没有中心 | 建状态转移表、状态服务或状态模式，配合乐观锁 |

## 实战经验：后端项目里怎么用设计原则

1. 接口从调用方长出来，不从实现类复制出来。
2. 抽象要用业务语言命名，比如 `RefundGateway` 比 `HttpClientWrapper` 更稳定。
3. 业务层尽量不要出现三方 SDK 类型、数据库表字段名、MQ 原生消息对象。
4. 新增策略、责任链节点、状态转移时，要有启动校验、单元测试和监控埋点。
5. 不要把所有变化都塞进配置。配置能降低发布频率，也会降低可读性和类型安全。
6. 资损、安全、权限相关逻辑要少靠“约定”，多靠显式模型、状态机和自动化测试。
7. code review 里不要只说“违反 SOLID”，要说清楚“这个设计会导致哪类变化成本上升”。
8. 发现过度设计时，不要羞于删接口。好的设计会随着证据演进，不是一次性猜中未来。

## 小结（3-5 条关键点）

1. 设计原则的目标是降低变更成本，不是让代码形式上更“面向对象”。
2. SOLID 分别处理职责、扩展、替换、接口粒度和依赖方向问题。
3. DRY、KISS、YAGNI 经常互相制衡：不要重复，但也不要过早抽象。
4. 生产落地时，要特别关注抽象泄漏、策略漏注册、接口语义不一致、依赖环和状态判断散落。
5. 真正成熟的设计不是模式越多越好，而是每个抽象背后都有明确变化证据。
