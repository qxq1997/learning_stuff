# 设计模式 - 第 9 课：适用边界、生产问题与实战经验

## 学习目标（本节结束后你能做到什么）

1. 能说清楚高频设计模式分别适合什么场景、不适合什么场景。
2. 能预判模式落地后在生产里常见的问题，比如漏注册、顺序错误、状态并发、事件重复。
3. 能给出工程化解决方法，而不是只停留在“用了某某模式”。
4. 能在面试和 code review 里表达取舍：为什么这里要用模式，为什么那里不该用。

## 先说结论：设计模式落地要同时回答四个问题

任何一个模式进入生产代码前，都要回答四个问题：

1. 它封装的变化点是什么？
2. 它带来的额外复杂度是什么？
3. 它失败时线上会怎么坏？
4. 我准备用什么测试、监控、校验和降级手段兜住它？

只回答前两个问题，是写 demo。四个问题都能回答，才是工程落地。

## 高频模式适用边界速查

| 模式 | 适用场景 | 不适用场景 | 生产常见坑 | 解决方法 |
| --- | --- | --- | --- | --- |
| 策略模式 | 支付、计费、折扣、路由、风控规则等同一维度多实现 | 分支很少且稳定，或者每个分支流程差异巨大 | 策略漏注册、默认策略误吞错误、接口参数越来越膨胀 | 启动校验、未知类型失败告警、使用 command/context 承载参数、契约测试 |
| 模板方法 | 主流程稳定，个别步骤变化，如导入流程、三方对接流程 | 每个实现连流程顺序都不同 | 父类变胖，子类靠 override 改坏流程 | 父类只保留骨架，变化点用组合策略，关键步骤设为不可跳过 |
| 责任链 | 过滤器、审批、风控、校验、网关处理链 | 顺序强事务、每一步都必须强一致提交 | 链路顺序混乱、异常被吞、排障困难 | 显式声明顺序、链路 trace、每个节点记录输入输出和耗时 |
| 状态模式 | 订单、工单、审批、履约等状态复杂且转移受规则约束 | 只有两三个简单状态，或者状态只是展示字段 | 并发状态跳变、非法转移、补偿困难 | 状态转移表、乐观锁、幂等键、状态变更日志 |
| 观察者/事件 | 下单后发通知、写审计、刷新缓存、触发异步任务 | 核心主交易必须同步强一致 | 事件丢失、重复、乱序、消费者失败 | Outbox、幂等消费、死信队列、事件版本和 traceId |
| 工厂模式 | 创建对象逻辑复杂，需要按类型选择实现 | `new` 一个简单值对象即可 | 工厂变成大 switch，新增类型还要多处修改 | 注册表、枚举映射、和策略/适配器配合 |
| 建造者模式 | 构造参数多、可选项多、对象需要校验不变量 | 参数很少，构造过程直接清晰 | builder 绕过校验，构造出非法对象 | 在 build 阶段集中校验，必要字段强制传入 |
| 适配器模式 | 对接三方系统、遗留系统、外部协议 | 内部新代码之间接口本来可以统一设计 | 外部字段和错误码泄漏进业务层 | 建防腐层，统一错误码、超时、重试和领域模型转换 |
| 装饰器模式 | 给对象叠加日志、缓存、重试、限流等能力 | 只需要一次性简单包装 | 装饰顺序错误、重复包装、异常语义改变 | 明确装饰链顺序，统一异常契约，给链路加指标 |
| 代理模式 | AOP、远程调用、权限、懒加载、缓存代理 | 直接调用足够清晰且无横切需求 | 代理隐藏真实成本，调用方以为是本地轻量方法 | 命名和文档暴露远程/缓存语义，设置超时、熔断、监控 |
| 外观模式 | 给复杂子系统提供稳定入口，如支付门面、库存门面 | 把所有能力都塞进一个万能服务 | Facade 变成 God Service | 门面只做编排，不吞掉领域模型和子模块边界 |
| 单例模式 | 无状态工具、全局配置、线程安全共享资源 | 可变业务状态、需要按租户/请求隔离的对象 | 测试困难、隐藏依赖、并发污染 | 优先用 DI 容器管理生命周期，避免可变全局状态 |

## 模式落地的真实判断

### 策略模式：最常用，也最容易被滥用

策略模式适合“同一件事有多种算法或规则”。比如不同支付渠道都要完成支付，不同会员等级有不同折扣，不同租户有不同路由规则。

它不适合“每个分支本质上是不同业务流程”的场景。如果微信支付和线下转账除了名字相同，输入、流程、状态、回调都完全不同，强行塞进同一个 `PaymentStrategy` 反而会得到一个巨大的 `PaymentContext`，里面全是可空字段。

生产里最常见的问题是漏注册。代码写了 `ApplePayStrategy`，但没有放进策略表，线上请求一来就找不到处理器。解决方法是启动时做完整性校验：枚举里的每个 `PaymentType` 必须有唯一策略实现，多了少了都启动失败。

另一个坑是默认策略。很多人会写“找不到策略就用默认策略”，这在资损场景很危险。未知支付方式应该失败、告警、阻断，而不是悄悄走默认逻辑。

### 责任链：不是把 if-else 换个写法

责任链适合“多个处理节点按顺序处理请求，每个节点可决定继续、拦截或补充信息”。网关鉴权、参数校验、风控规则、审批流程、消息消费前置检查都很典型。

它不适合所有步骤都必须在一个数据库事务里强一致提交的流程。责任链的优势是可插拔和顺序编排，但如果每一步都有复杂回滚，应该考虑事务边界、Saga、状态机，而不是简单链式调用。

生产问题通常有三个：

1. 顺序被改坏。比如先扣库存再校验风控，事故味道已经出来了。
2. 异常被某个节点吞掉，主流程误以为成功。
3. 排障看不到请求经过了哪些节点。

解决方法是显式顺序配置、节点级别指标、链路 trace、统一异常语义。核心链路不要只靠 Spring bean 加载顺序这种隐式机制。

### 状态模式：订单类系统的高价值工具

状态模式适合状态多、转移规则复杂、不同状态下行为不同的业务。比如订单从待支付、已支付、待发货、已发货、已完成、已取消、退款中、已退款之间转移，每一步都有前置条件和副作用。

它不适合只有简单状态展示的场景。比如一个任务只有 `enabled/disabled`，用枚举加清晰判断就够了。

生产里最大的问题是并发。两个请求同时操作同一订单，一个取消，一个支付成功，如果没有状态版本控制，就可能出现非法终态。解决方法是状态转移表 + 乐观锁 + 幂等键 + 状态变更日志。状态机不是只解决代码优雅，它还要解决线上并发和可追溯。

### 观察者/事件：解耦之后要补可靠性

观察者或事件机制适合把主流程和附属动作解耦。如下单成功后发短信、写审计、刷新推荐、推送积分任务，这些动作不应该阻塞主交易太久。

但不要把它误用在必须强一致的核心动作里。比如支付成功后更新订单状态，这通常不能只靠一个可能丢失的异步事件完成，除非你已经设计好可靠消息、补偿和对账。

生产里事件系统的四个经典问题是：丢失、重复、乱序、延迟。解决方法不是“用了观察者模式”，而是 Outbox、事务消息、幂等消费、死信队列、重试退避、事件版本、消费者监控和对账任务。

### 适配器：三方对接必须有防腐层

适配器模式在后端很常见，尤其是支付、物流、短信、实名认证、风控、地图、云服务。它的价值不是“换个接口名”，而是隔离外部系统的字段、错误码、超时语义和奇怪边界。

不适合的场景是内部新模块之间本来就能对齐模型，却为了模式而套一层 adapter。内部边界如果能直接统一，就直接统一。

生产里最怕外部模型泄漏。比如业务层到处判断 `trade_state == "SUCCESS"`，将来换渠道或接口升级时会改穿一片。解决方法是防腐层：外部响应进入系统边界后，立刻转换成内部 `PaymentStatus`、`PaymentErrorCode` 和统一异常。

### 外观模式：门面要薄，边界要清楚

外观模式适合给复杂子系统提供一个稳定入口。比如库存系统内部有锁库存、释放库存、扣减库存、库存流水、补偿任务，对外可以暴露 `InventoryFacade`。

它不适合做万能服务。很多项目里的 `OrderFacade` 最后会变成几千行，什么都知道，什么都调用。这不是外观模式，这是边界坍塌。

实战做法是：门面负责对外协议和流程编排，核心规则仍然放在领域服务、策略、状态机或子模块里。门面可以薄，但不能空；它应该表达用例流程，而不是吞掉所有模型。

### 单例：慎用可变全局状态

单例适合无状态、线程安全、生命周期确实全局唯一的对象。比如配置读取器、枚举型注册中心、某些无状态工具。

不适合保存业务运行时状态。把当前租户、当前用户、请求上下文、订单缓存放进单例，迟早会遇到并发污染和测试隔离问题。

现代后端里，很多“单例需求”应该交给 DI 容器，比如 Spring singleton bean。这样生命周期、依赖注入、测试替换都更清楚。

## 生产落地四步

1. 识别变化维度
   先说清楚变化的是渠道、算法、流程步骤、状态、外部协议，还是横切能力。

2. 定义稳定契约
   接口要写调用方真正需要的业务语义，包括返回值、异常、幂等、超时、是否可重试。

3. 补工程护栏
   策略要启动校验，状态要乐观锁，事件要幂等和死信，责任链要顺序和 trace，适配器要错误码映射。

4. 渐进迁移
   不要为了模式一次性重写整块业务。可以先把最痛的分支抽出来，新旧逻辑并行灰度，配合监控和回滚开关。

## 线上问题处理手册

| 线上症状 | 可能相关模式 | 常见根因 | 处理方法 |
| --- | --- | --- | --- |
| 新增渠道后部分请求失败 | 策略/工厂 | 策略未注册，类型映射缺失 | 启动校验、枚举覆盖测试、失败告警 |
| 某些订单状态不可能但出现了 | 状态模式 | 并发更新、非法转移没拦住 | 乐观锁、转移矩阵、状态变更审计 |
| 下单成功但通知没发 | 观察者/事件 | 事件丢失或消费者失败 | Outbox、重试、死信队列、补偿扫描 |
| 风控规则顺序错导致误拦截 | 责任链 | 链路顺序隐式、配置无人校验 | 显式优先级、链路快照、变更审批 |
| 业务层出现大量三方错误码 | 适配器 | 防腐层缺失 | 建统一错误模型，外部错误只在 adapter 内出现 |
| 单测特别难写 | 依赖倒置/代理 | 业务类直接 new 外部客户端 | 抽 port，构造注入 fake/mock |
| 看代码需要跳十几层 | 过度设计 | 没有真实变化点却硬套模式 | 合并无意义抽象，保留关键扩展点 |

## 不要用模式的信号

- 你说不清它封装的变化点，只是觉得“这样更高级”。
- 接口只有一个实现，而且测试、替换、扩展都不需要它。
- 新人读代码必须跳很多层，才能明白一个简单分支。
- 抽象后的接口比原来的业务方法更不稳定。
- 你需要给每个类都写很长注释解释为什么存在。
- 业务还没稳定，但你已经设计了完整插件体系、规则引擎和多层工厂。

## 实战经验

1. 模式优先解决“稳定流程中的变化点”，不要试图解决“需求还没想清楚”。
2. 能用业务枚举表达的类型，就不要用散落的字符串。
3. 所有基于类型路由的模式，都要有启动时完整性校验。
4. 所有异步解耦，都要补幂等、重试、死信和对账。
5. 所有状态流转，都要补并发控制和状态变更日志。
6. 所有三方适配，都要在系统边界完成模型转换和错误映射。
7. 所有横切增强，都要明确顺序，比如缓存、限流、重试、熔断的顺序会影响结果。
8. 不要让配置成为第二套代码。配置化规则也需要版本、审批、回滚和测试。
9. code review 时先问“这里未来怎么变”，再讨论要不要上模式。
10. 面试里讲设计模式，不要只说“我用了策略模式”，要说“我识别了支付渠道这个变化维度，用策略隔离渠道差异，并用启动校验和契约测试防止漏注册和语义不一致”。

## 面试表达模板

可以按这个顺序讲：

1. 原来痛点是什么：分支膨胀、状态判断散落、三方模型泄漏、横切逻辑重复。
2. 变化点是什么：渠道、规则、状态、处理步骤、外部协议。
3. 选择了什么模式：策略、责任链、状态、适配器、事件等。
4. 为什么不用更复杂方案：当前变化维度清楚，不需要规则引擎或工作流引擎。
5. 生产护栏是什么：启动校验、契约测试、幂等、乐观锁、监控、灰度、回滚。

## 小结（3-5 条关键点）

1. 设计模式不是只看“代码长什么样”，而是看它封装了哪个变化点。
2. 每个模式都有适用边界，不适合的场景硬套会制造新的复杂度。
3. 生产落地必须补测试、监控、启动校验、幂等、并发控制、错误映射等工程护栏。
4. 高频模式里，策略、责任链、状态、事件、适配器最贴近后端业务，但也最容易踩坑。
5. 成熟表达不是“我用了模式”，而是“我知道为什么用、哪里不能用、上线后怎么兜底”。
