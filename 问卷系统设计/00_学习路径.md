# 问卷系统设计学习路径

## 学习目标

这个主题不是为了背几个“问卷”“草稿”“锁”的名词，而是为了把你带到这样一种水平：

- 你能把一个“多人可打开、单人可编辑、其他人只读、退出后可恢复”的问卷系统讲成完整方案，而不是只说“加个 Redis 锁”
- 你能分清问卷模板、问卷实例、作答草稿、编辑会话、提交结果这些对象的边界
- 你能解释这个题真正的技术矛盾是`单写多读协同 + 草稿持久化恢复`，而不是把它误答成 Google Docs 那种多人实时协同编辑
- 你能在系统设计面试里把需求澄清、核心链路、技术选型和 trade-off 讲得像一个真正落过系统的人

## 重要说明

这道题表面看起来像“问卷系统”，但如果你直接按普通表单系统回答，通常会丢很多分。因为题目里其实塞了三个很关键的约束：

1. 同一份问卷可以被多人在不同时间打开。
2. 同一时刻只能有一个人编辑，其他打开者必须是灰白只读态。
3. 系统关闭后再次打开，要能恢复上一次保存的内容。

这意味着它不是一个纯 CRUD 系统，也不是一个真正的多人实时编辑系统，而是一个`单写者租约锁 + 多读者查看 + 自动保存草稿 + 崩溃后恢复`的业务系统。  
如果你在面试里一上来就讲 CRDT、Operational Transform、分布式协同算法，往往是过度设计。因为题目没有要求多个人同时改同一个字段，也没有要求冲突合并。真正需要解决的是：谁现在有编辑权、别人如何感知自己只能看不能改、用户中途退出后如何恢复、锁丢失和保存失败时怎么兜底。

## 知识图谱

这个主题建议按 4 个层次理解：

1. 业务与对象层
   - 问卷模板是什么
   - 问卷实例是什么
   - 草稿和最终提交结果有什么区别
   - 为什么“同一份问卷”通常指同一条待填写记录，而不是同一个模板

2. 协同与状态层
   - 编辑权限怎么判定
   - 只读状态怎么同步给其他用户
   - 锁过期、续租、释放、接管怎么做
   - 为什么这里更适合悲观锁思路而不是乐观冲突合并

3. 存储与恢复层
   - 自动保存频率怎么定
   - 草稿存数据库还是浏览器本地
   - 系统重开后如何恢复最近一次内容
   - 提交后是否还允许回到草稿态

4. 架构与表达层
   - 技术栈怎么选
   - 哪些组件必须在主链路
   - 哪些异步能力只是增强项
   - 面试里应该怎么回答才不会显得空泛

## 课程安排

1. `01_需求澄清与总体架构.md`
   - 先把题目边界讲透
   - 重点建立全局视角，避免把“问卷系统”答成普通后台表单

2. `02_并发编辑控制与只读协同.md`
   - 这是这道题最核心的一课
   - 重点讲单写者锁、只读降级、锁续租、锁过期和接管流程

3. `03_草稿持久化、恢复与提交流程.md`
   - 这是用户体验和可靠性的核心
   - 重点讲自动保存、版本号、异常退出恢复和最终提交状态机

4. `04_技术选型、容量估算与面试表达.md`
   - 把前面内容工程化
   - 重点讲技术选择、量级估算、为什么不用更复杂方案，以及面试时怎么组织答案

## 推荐学习方式

- 先完整读第 1 课，把“这题到底在设计什么”定清楚
- 第 2 课要重点理解“为什么是单写多读锁，而不是多人实时协同编辑”
- 第 3 课要重点理解“自动保存成功”和“用户以为保存成功”不是一回事，恢复逻辑必须讲清楚
- 第 4 课用来把前面内容收束成一个可表达的系统设计答案

## 学完后你应该能回答的问题

- 为什么这个问卷系统的核心矛盾不是高并发写入，而是编辑权控制和状态恢复
- 为什么更适合用`租约锁 + 心跳续租`，而不是数据库长事务锁住整条记录
- 为什么草稿通常要区分服务端持久化和浏览器本地兜底两层
- 为什么这个题大多数时候不需要 Kafka 进主链路，也不需要 CRDT
- 如果用户编辑过程中断网、浏览器崩溃、服务重启，系统分别该如何恢复
