# 问卷系统设计 - 第 4 课：技术选型、容量估算与面试表达

## 学习目标（本节结束后你能做到什么）

1. 能根据这个题的核心矛盾给出不过度设计的技术选型。
2. 能做一个量级合理的容量估算，支撑自己的组件选择。
3. 能说清 MySQL/PostgreSQL、Redis、WebSocket/SSE、消息队列分别为什么选或为什么先不选。
4. 能把整道题组织成一段面试里可直接输出的答案。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

技术选型一定要围绕矛盾，不要围绕名词。  
这道题的核心不是每秒百万写入，也不是全球多活，更不是复杂富文本协同。它的重点是：

- 同一问卷实例同一时刻只能有一个编辑者
- 非编辑者要实时或准实时感知自己是只读
- 草稿要可靠保存并能恢复
- 提交后要进入稳定终态

基于这个目标，一个很务实的首版技术栈可以是：

- 前端：`React + TypeScript`
- 后端：`Spring Boot`
- 数据库：`MySQL` 或 `PostgreSQL`
- 缓存与分布式锁：`Redis`
- 在线状态通知：`WebSocket` 或 `SSE`
- 审计 / 统计异步链路：必要时再加 `Kafka` 或其他 MQ

为什么这套组合比较稳？

前端层，React 或 Vue 都能做，但如果你站在企业级问卷系统场景，React + TypeScript 的生态、表单组件成熟度和状态管理能力都不错。重点不在框架本身，而在你要有能力把字段级校验、只读态切换、保存状态提示和本地草稿兜底做清楚。

后端层，如果你的背景是 Java 后端，Spring Boot 是很自然的选择。  
因为这道题并没有复杂到必须用事件驱动微服务拆成很多块。首版完全可以是一个问卷服务，把模板读取、草稿保存、提交、锁校验这些逻辑放在一起。只有当模板管理、作答服务、统计分析规模明显上来时，再考虑拆分。面试里主动说“这个题初版我会保持单体或轻服务化，避免为了微服务而微服务”，通常比上来就画十几个服务更成熟。

存储层，MySQL 和 PostgreSQL 都能胜任。  
如果你更看重团队熟悉度、线上经验、运维成熟度，MySQL 很稳。  
如果你希望更自然地存 JSON、做部分字段索引，PostgreSQL 的 JSONB 体验更好。  
这里的重点不是“谁更先进”，而是问卷答案本身通常是半结构化数据，所以你要么用 MySQL 的 JSON 字段配合少量冗余索引字段，要么用 PostgreSQL JSONB。  
我一般会这样说：`模板、实例元数据、提交状态仍然是强结构化关系数据；答案内容采用 JSON 存储，以适应题目结构灵活变化。`

Redis 的角色非常明确：  
它不是为了缓存整个问卷答案主数据，而是为了实现带 TTL 的编辑锁、心跳续租和短时在线状态。  
如果有人说“我把锁直接放数据库里也可以”，理论上能做，但工程体验通常更差，因为数据库不擅长这种高频续租、自动过期的短周期状态。

WebSocket 和 SSE 怎么选？  
如果只是服务端单向通知“锁状态变了，你现在可以编辑了”，SSE 就够了，实现也更轻。  
如果未来还想做更复杂的双向在线能力，比如在线提示、强制接管确认、协同评论，那 WebSocket 更灵活。  
所以你可以这样讲：`这题的最小闭环不是一定要 WebSocket，轮询甚至都能工作；但如果希望锁切换体验更及时，我会优先引入 SSE 或 WebSocket 做状态通知。`

消息队列要不要上？  
默认不要放在主链路。  
因为保存草稿、获取编辑权、提交问卷，都是用户强交互动作，用户希望立刻知道结果。如果你把“保存成功”这件事改成“先入 MQ，稍后异步处理”，用户体验和一致性都会变差。  
MQ 更适合放在旁路，比如：

- 提交后的审计日志
- 统计埋点
- 搜索索引更新
- 报表同步

也就是说，这题的主链路更偏同步写库，异步只做增强。

再做一个简单容量估算。  
假设这是一个中后台问卷系统：

- 日活填写用户：2 万
- 高峰同时在线打开问卷：2000
- 其中真正处于编辑态：500
- 每个编辑用户平均 5 秒自动保存一次

那么草稿保存的写入 QPS 大约是：`500 / 5 = 100 QPS`。  
心跳续租如果也是 10 秒一次，大约是：`500 / 10 = 50 QPS`。  
这说明什么？说明这个题默认不是吞吐极限题，普通的关系数据库 + Redis 完全扛得住。真正需要你讲清楚的不是“怎么把 QPS 从 10 万做到 100 万”，而是“如何保证锁不乱、草稿不丢、状态不脏”。

当然，如果规模继续放大，比如几万活跃编辑者同时在线，就要进一步考虑：

- 保存请求限流和批量化
- 热点实例的锁争抢保护
- 读写分离与分库分表
- 模板与答案的冷热分层
- 多机房部署下锁的一致性策略

但这已经是后续扩展题，不是首答重点。

最后给你一段面试里可以直接说的回答骨架：

“这道题我会先把对象定清楚，设计的是问卷作答实例，不是模板后台。题目的核心要求是多人可以同时打开同一份问卷，但同一时刻只能有一个用户编辑，其他人完全只读，所以我不会把它设计成 Google Docs 那样的多人实时协同，而会采用单写多读模型。总体上前端负责渲染模板和只读态切换，后端问卷服务负责加载草稿、申请编辑锁、自动保存和提交。草稿和提交结果落 MySQL 或 PostgreSQL，编辑锁放 Redis，用 TTL 和心跳续租避免死锁。用户打开问卷时先读草稿，再尝试申请锁；拿到锁就可编辑，拿不到就灰白只读。编辑过程中做自动保存，服务端同时校验锁归属和草稿版本，避免乱序覆盖。系统关闭后再次打开时，优先恢复服务端草稿，前端本地缓存只做异常兜底。提交时做全量校验、状态切换并释放锁。这个方案比实时协同编辑更贴合题目，也比数据库长事务锁住整份问卷更可恢复。”  

这段话的价值不在于组件名，而在于它把题目的关键矛盾、主链路和技术取舍一次说清楚了。

## 小结（3-5 条关键点）

1. 技术选型要围绕“单写多读 + 草稿恢复”，而不是盲目堆协同编辑或大数据组件。
2. MySQL/PostgreSQL 负责持久化，Redis 负责编辑锁，SSE/WebSocket 负责锁状态通知，是这题比较稳的组合。
3. 消息队列通常不该进入保存和提交流程主链路，更适合做审计、统计、索引等旁路异步任务。
4. 量级估算往往会发现这不是高吞吐极限题，真正难点是状态控制和可靠恢复。
5. 面试回答要先定义题目边界，再讲主链路，再讲选型和 trade-off，不要上来堆组件。
