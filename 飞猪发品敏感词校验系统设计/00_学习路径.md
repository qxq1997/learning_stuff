# 飞猪发品敏感词校验系统学习路径

## 学习目标

这个专题不是为了背一个"敏感词过滤"的标准答案，而是把你带到下面这个水平：

- 你能先把题目从"做个关键词过滤"纠正为"内容安全治理系统"——它的本质是合规风险管控，技术只是手段
- 你能讲清楚为什么不能只靠集团第三方服务，自建词库 + 白名单 + 第三方兜底的混合架构到底解决了什么痛点
- 你能像真正做过这套系统的人一样，把 AC 自动机、词库热更新、变体对抗、白名单优先级、降级与埋点这几个硬骨头讲透
- 你能把改进空间讲出深度——尤其是规则系统升级为"规则 + LLM"混合内容安全管线这条线，这是和你 RAG/Agent 方向最契合的简历亮点
- 你能把这道题稳定压缩成一段 5 到 8 分钟的面试回答，并扛住"性能怎么扛""词库怎么更新""怎么防绕过""线上出过什么问题"这类追问

## 重要说明

这是一道很典型、也很适合面试展开的**内容安全 / 合规风控**题：

- 平台定位：飞猪是 C2C/B2C 的旅游交易平台，商家发布旅游商品（机票、酒店、跟团游、签证等）
- 业务痛点：商家发品的标题和图文描述里如果出现违规词（广告法绝对化用语、涉政涉黄涉赌、虚假宣传、医疗承诺等），会带来合规风险、平台处罚甚至监管约谈
- 系统诉求：发品前端实时校验、精确定位命中位置和命中词、规则可运营、命中数据可统计

这里有一个关键的面试动作：**先把"为什么不直接用集团第三方服务"说清楚**。如果上来就讲 AC 自动机，会显得只懂技术不懂业务。正确的开场是先点出第三方方案的四个痛点（链路长延迟高、反馈粒度粗、规则不可定制、数据不沉淀），把题目从"实现一个过滤器"抬升到"设计一套可运营、可观测的内容安全系统"。

另外要强调：下面的内容是**基于通用内容安全、合规风控和文本匹配系统的架构抽象**，用于学习和系统设计训练，不代表任何公司内部真实实现。涉及的中间件（HSF、TDDL、Tair、Diamond、MetaQ、Sunfire 等）只是典型阿里系技术栈的代称，用于说明选型理由。

## 知识图谱

这个主题建议按四层来理解。

### 1. 业务层

- 为什么发品要校验敏感词，违规会带来什么后果（合规、处罚、约谈）
- 为什么集团统一的第三方服务不够用（误伤行业词、漏判飞猪特有违规、黑盒不可运营）
- 强拦截 vs 弱提醒：什么词该硬拦，什么词只给商家提个醒
- 词库为什么要分类、分等级、分场景、分行业

### 2. 核心系统层

- 双层校验：前端实时校验（体验）+ 后端提交兜底（安全，防绕过）
- 词库分层：自建词库（可运营）+ 白名单（防误伤）+ 第三方（兜底 + 发现新词）
- 检测流程：归一化预处理 → 自建词库 AC 扫描 → 未命中调第三方 → 合并 → 写埋点 → 返回带 offset 的命中详情
- 命中统计闭环：埋点 → 聚合 → 运营后台看词的有效性 → 指导词库治理

### 3. 分布式工程层

- 为什么用 AC 自动机而不是朴素 contains 循环（性能）
- 词库热更新为什么要全量重建 + 原子切换（一致性 + 可用性）
- 第三方调用为什么必须超时 + 熔断 + 降级（稳定性）
- 命中埋点为什么走 MQ 异步而不是同步写 DB（不阻塞主链路）
- 前端词库和后端词库为什么必须同源、要带版本号（数据一致性）

### 4. 面试表达层

- 如何先讲业务痛点、再给混合架构，而不是直接甩算法
- 如何挑 1 到 2 个点深挖（AC 自动机 + 变体对抗，或热更新 + 降级）
- 如何把改进空间讲出层次（语义检测 / 主动学习闭环 / 词库治理自动化）
- 如何回答"性能怎么扛""怎么防绕过""第三方挂了怎么办""误判申诉怎么处理"

## 课程安排

1. `01_背景、痛点与整体方案.md`
   - 先讲清楚为什么不直接用第三方，自建到底解决什么
   - 建立双层校验 + 词库分层 + 命中统计的整体架构
   - 走通一条完整的检测流程

2. `02_核心算法与技术栈.md`
   - AC 自动机为什么是这道题的核心，它怎么把 O(n×m×k) 降到 O(n+命中数)
   - 节点上怎么挂词的元数据，命中时一次性带出
   - 典型技术栈选型理由（存储、缓存、配置中心、MQ、监控）

3. `03_技术挑战与解决方案.md`
   - 六个硬骨头：匹配性能、词库热更新、变体绕过、白名单优先级、第三方稳定性、埋点不阻塞主链路
   - 每个挑战讲清楚"不这样会怎样"

4. `04_改进空间与演进方向.md`
   - 规则系统升级为规则 + LLM 混合管线（和 RAG 方向契合，简历亮点）
   - 主动学习闭环、变体识别系统化、词库治理自动化、多语言支持

5. `05_生产问题与应对.md`
   - 换成"真做这个系统"的工程视角
   - 七个真实会爆的故障形态：词库膨胀内存压力、重建 CPU 抖动、误判申诉慢、第三方误判、高峰 QPS 冲击、前后端词库不一致、对抗性绕过

## 推荐学习方式

- 第 1 课一定要认真看，很多候选人是从"答成关键词过滤"开始丢分的，业务抬升要先到位
- 第 2 课要把 AC 自动机的复杂度和"为什么不用 contains"讲到能上手画图的程度
- 第 3 课是分布式系统味道最浓的一课，建议反复看，系统成熟度都体现在这些边界处理上
- 第 4 课用来准备"如果重做你会怎么改"这类追问，规则 + LLM 那条线要能展开讲
- 第 5 课用来准备"线上出过什么问题"，这类追问最能体现你真做过而不是背的

## 学完后你应该能回答的问题

- 为什么内容安全不能只靠一个第三方接口，自建混合架构的必要性在哪
- 为什么前端实时校验和后端提交校验都要做，缺一个会怎样
- AC 自动机相比朴素匹配的复杂度优势，节点上挂元数据的设计
- 词库热更新为什么选全量重建 + volatile 原子切换，代价是什么
- 商家用谐音、间隔符、全半角绕过，归一化预处理怎么做
- 一段文本同时命中敏感词和白名单时，优先级怎么定、怎么实现
- 第三方抖动时怎么降级，降级了运营怎么知道
- 命中埋点为什么不能同步写库，MQ 异步聚合的链路
- 规则系统的天花板在哪，为什么需要引入语义级检测
