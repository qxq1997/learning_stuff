# 第 1 课：背景、痛点与整体方案

## 一、先把题目抬升到正确的层次

面试里被问"做个发品敏感词校验"，最容易丢分的回答是直接开始讲"我用一个 Set 存敏感词，然后 text.contains 一遍"。这把一道**内容安全治理**题答成了一道字符串题。

正确的开场，是先讲清楚业务背景和它真正要解决的风险：

飞猪是 C2C/B2C 的旅游交易平台，商家发布旅游商品（机票、酒店、跟团游、签证等）时，标题和图文描述里如果出现违规词，会带来一连串后果：

- **合规风险**：广告法绝对化用语（"最便宜""第一""国家级"）、涉政、涉黄、涉赌、虚假宣传、医疗承诺（"包治百病""根治"）等
- **平台处罚**：商品下架、商家扣分、店铺降权
- **监管风险**：严重的会引来监管约谈

所以这道题的本质不是"过滤关键词"，而是"在发品入口建一道可运营、可观测的内容安全闸门"。技术（AC 自动机那些）只是实现手段，先想清楚这一层，后面的架构决策才有依据。

## 二、为什么不直接用集团第三方服务

这是面试中最关键的一个"为什么"。飞猪所在的集团一般都有统一的内容安全服务（绿网 / 内容安全那套），那为什么还要自建？

原方案完全依赖第三方，有四个痛点，**每一个痛点都直接对应后面架构里的一个设计**：

### 痛点 1：链路长、延迟高

第三方是面向集团所有 BU 的中心化服务，调用链路长、延迟高，只能在**提交发品时同步校验**。商家辛辛苦苦填了一大段描述，点提交后才告诉他"有问题"，体验非常差。

→ 对应设计：**前端实时校验**（本地词库毫秒级返回），不等到提交。

### 痛点 2：反馈粒度粗

第三方只告诉你"命中敏感词"，不告诉你**具体哪个词、在哪一段**。商家得自己一个字一个字找，根本不知道改哪里。

→ 对应设计：返回**带 offset 的命中详情**（命中词、起始位置、长度、风险等级、提示文案），前端能精确高亮。

### 痛点 3：规则不可定制

飞猪有自己的业务场景。比如"包机""切位"这种是旅游行业正常术语，不该被拦；但"包治百病"这种医疗承诺一定要拦。公共敏感词库要么误伤行业词，要么漏判飞猪特有的违规。

→ 对应设计：**自建词库 + 白名单**，规则由飞猪自己的运营/合规团队维护。

### 痛点 4：没有数据沉淀

命中了什么词、被哪些商家高频触发、规则到底有没有用，第三方那边完全是黑盒，运营无法迭代规则。

→ 对应设计：**命中埋点 + 统计闭环**，让规则可度量、可治理。

把这四点讲完，自建的必要性就立住了：**不是不信任第三方，而是第三方解决不了"实时、精确、可定制、可运营"这四件事**。但第三方也不是不要——它的价值在于兜底和发现新词，所以最终是个混合架构。

## 三、整体方案

### 3.1 双层校验架构

这是整个系统的骨架，一句话：**前端实时校验保体验，后端提交校验保安全。**

- **前端**：在标题 / 描述输入框 `onBlur` 或防抖触发时，异步调用敏感词检测接口，实时高亮命中片段并弹出具体提示，例如"第 12 字'最便宜'涉嫌广告法绝对化用语"。
- **后端**：在发品提交链路做最终校验，作为安全兜底。

为什么两层都要？**前端校验是体验，不是安全边界**——前端可以被绕过（直接调发品接口、改请求）。所以后端提交时必须再校验一次，这才是真正的拦截点。前端的价值是让 99% 的正常商家在写的时候就把问题改掉，而不是等到提交被拒。

### 3.2 词库分层

词库不是一张大表，而是三层，职责清晰：

| 层 | 谁维护 | 作用 |
| --- | --- | --- |
| 自建敏感词库 | 运营 / 合规团队 | 核心规则。带分类（广告法、涉政、涉黄、行业自定义……）、风险等级（强拦截 / 弱提醒）、生效场景（标题 / 描述 / SKU 名）、生效行业（机票 / 酒店 / 门票） |
| 白名单 | 运营 | 针对第三方或自建词的误伤，放行合法词。比如"绝对零度"是合法商品名，但含"绝对"这个广告法敏感词 |
| 第三方接口 | 集团 | 兜底 + 发现新词的来源 |

自建词库的元数据维度（分类 / 等级 / 场景 / 行业）很重要，它让规则能精细化运营：同一个词在标题里强拦截、在描述里只弱提醒；某个词只在医美类目生效。

### 3.3 第三方的角色：兜底 + 新词发现

第三方命中后有两种处理：

- 如果运营判断这个词**确实该拦** → 沉淀到自建库（下次本地就能命中，不用再调第三方）
- 如果是**误伤** → 加白名单

这就形成了一个良性循环：第三方负责"发现"，自建库负责"沉淀"，白名单负责"纠偏"。随着运营，越来越多的判断本地化，对第三方的依赖逐渐降低。

### 3.4 命中统计闭环

每次命中写埋点，按词、按商家、按行业聚合。运营后台能看到：

- 哪些词**高频命中** → 说明规则有效，保留
- 哪些词**从来不命中** → 可以下线，减小词库
- 哪些词**频繁被加白名单** → 说明规则有问题（误伤多），需要调整或下线

这一步把"规则好不好"从拍脑袋变成了**数据驱动**，是自建相比第三方黑盒最大的价值之一。

## 四、一条完整的检测流程

把上面的部件串起来，一次检测请求的链路是：

```
请求进来
  → 归一化预处理（去标点、繁转简、全半角、大小写归一）
  → 走自建词库（本地内存 AC 自动机匹配，毫秒级）
     → 命中 → 直接返回（高 precision、低延迟）
     → 未命中 → 调第三方（异步 / 200ms 超时 / 熔断降级）
  → 合并自建 + 第三方结果，应用白名单过滤
  → 命中事件丢 MQ（异步写埋点，不阻塞主链路）
  → 返回带 offset 的命中详情（词、起始位置、长度、风险等级、提示文案）
```

这条流程里藏着几个关键的工程权衡，都会在第 3 课展开：

- 为什么**先本地后第三方**：本地快且准（自建库是精挑的），能挡掉大部分；第三方慢且可能误伤，只在本地没命中时才补一刀。
- 为什么第三方要**超时 + 熔断**：不能让一个外部依赖拖垮发品主链路。
- 为什么埋点走 **MQ 异步**：命中要落库统计，但同步写 DB 会拖慢校验响应。

## 检查站

回答下面几个问题，确认你真的理解了这一课：

1. 如果面试官说"集团不是有现成的内容安全服务吗，为什么还要自建"，你怎么用四个痛点回答？
2. 前端已经做了实时校验，为什么后端提交时还要再校验一遍？只做前端会有什么风险？
3. "包机"是行业词不该拦，"包治百病"是医疗承诺要拦——你的系统靠哪一层来区分这两种情况？
4. 第三方命中了一个词，运营接下来可能怎么处理？这体现了第三方在这套架构里的什么角色？
5. 运营后台看到某个词"从来不命中"和"频繁被加白名单"，分别说明什么、该怎么处理？

答完后我们进入第 2 课，讲核心算法 AC 自动机和技术栈选型。
