# 第 2 课：核心算法与技术栈

## 一、为什么 AC 自动机是这道题的核心

### 1.1 朴素做法为什么扛不住

最直觉的做法：把词库存成一个集合，对输入文本，每个词都 `text.contains(word)` 一遍。

问题出在规模上：

- 词库规模大：几万到几十万词
- 输入文本长：商家的图文描述可能上千字
- QPS 高：发品高峰（大促前商家集中改品）

朴素做法的复杂度是 **O(n × m × k)**（n = 文本长度，m = 平均词长，k = 词库大小）。词库几十万、文本上千字、QPS 又翻倍，这个量级直接打满 CPU，完全扛不住。

### 1.2 AC 自动机怎么解决

AC 自动机（Aho-Corasick）是**多模式串匹配**算法：把所有敏感词预先构建成一棵带 fail 指针的 Trie 树，然后对输入文本**只扫描一遍**，就能找出所有命中的词。

复杂度降到 **O(n + 命中数)**——和词库大小**几乎无关**（词库只影响构建时间和内存，不影响单次匹配的扫描成本）。实测毫秒级返回。

直观理解它为什么快：朴素做法是"拿每个词去文本里找"，AC 自动机是"拿文本在词库树上走一遍"。fail 指针的作用是匹配失败时不用回退到文本开头重来，而是跳到"最长可复用的后缀"继续，这样指针在文本上单调向前，一遍走完。

> 面试时如果能顺手画一下 Trie + fail 指针，并解释 fail 指针让扫描"不回头"，这个点的深度就到位了。

### 1.3 节点上挂元数据：命中即带出

一个工程上的关键设计：AC 自动机的**每个终止节点（代表一个完整的词）上，挂这个词的元数据**——分类、风险等级、生效场景、提示文案。

这样匹配命中的瞬间，不需要再回查一次词库表，直接从节点上把元数据一次性带出来，组装成返回结果（词、offset、长度、等级、提示）。一次扫描既得到"命中了什么"，也得到"该怎么处理、怎么提示商家"。

## 二、整体技术栈（典型阿里系）

下面是一套典型的阿里系技术栈，重点不是记住中间件名字，而是**每个选型解决什么问题**。

| 维度 | 选型 | 解决什么 |
| --- | --- | --- |
| 语言 / 框架 | Java + Spring Boot | 业务主体 |
| RPC | HSF | 集团内调第三方敏感词服务 |
| 存储 | MySQL（TDDL 分库分表） | 存词库元数据 + 命中流水（流水量大要分表） |
| 缓存 | Tair（集团 Redis） | 缓存词库，二级缓存 |
| 配置中心 | Diamond / Nacos | 推送词库变更，触发热更新 |
| 消息 | MetaQ / RocketMQ | 异步写命中埋点，解耦主链路 |
| 核心算法 | AC 自动机 | 多模式串匹配 |
| 监控 | Sunfire / 鹰眼 | 链路追踪 + 指标监控 |
| 前端 | React | debounce 触发 + 富文本高亮命中片段 |

### 2.1 几个选型的"为什么"

**为什么词库元数据放 MySQL，运行时却用内存里的 AC 自动机？**
MySQL 是词库的**真相源**（运营增删改、审计、版本管理都在这），但它**不在请求链路上**。请求链路用的是从 MySQL 构建出来、常驻内存的 AC 自动机。MySQL 改了，通过配置中心通知服务重建 AC 自动机（见第 3 课热更新）。

**为什么命中流水要分库分表（TDDL）？**
命中埋点是高频写入，每次校验命中都可能产生多条记录，长期累积量很大。分库分表是为了让写入和后续聚合查询不被单表拖垮。

**为什么配置变更用 Diamond，而不是定时轮询 DB？**
词库更新要**实时生效**且**对全部服务实例广播**。配置中心的推送模型天然适合"一处改、处处生效"，比每个实例定时轮询 DB 更及时、对 DB 压力也更小。

**为什么埋点走 MetaQ 而不是同步写库？**
校验是在发品主链路上的，响应要快。埋点是旁路的统计需求，不能让"写统计"拖慢"给商家返回结果"。MQ 把这两件事解耦——校验只管把命中事件丢进队列就返回，消费者在另一侧慢慢聚合落库（第 3 课展开）。

## 三、前端这一侧

前端不是简单调个接口，有两个工程点：

1. **debounce / onBlur 触发**：商家每敲一个字就调一次接口太浪费，用防抖（停止输入若干毫秒后才调）或失焦时调，减少无效请求。这也是高峰期降 QPS 的手段之一。
2. **富文本高亮命中片段**：拿到带 offset 的命中详情后，在描述里把对应区间高亮，并在旁边给出具体提示。这正是为了解决第 1 课说的"第三方反馈粒度粗、商家找不到问题在哪"的痛点。
