# 飞猪延期支付系统设计学习路径

## 学习目标

这个专题不是为了背一个“延期支付”的标准答案，而是把你带到下面这个水平：

- 你能先把题目从“支付功能”纠正为“授信 + 下单占额 + 履约确认 + 出账还款 + 逾期催收”的闭环系统
- 你能看出这道题的核心矛盾不是单纯支付成功率，而是成交转化、资金风险、供应商履约和账务一致性的平衡
- 你能像真正做过交易系统的人一样，把额度模型、账单模型、状态机、一致性和风控分层讲清楚
- 你能把这道题稳定压缩成一段 5 到 8 分钟的面试回答，并扛住进一步追问

## 重要说明

你给出的场景是一个很典型、也很适合面试展开的本地生活/旅游交易题：

- 平台定位：飞猪场景下，服务 B 端商户的短途旅游产品平台
- 产品特点：很多是日内游、半日游、小团、门票加套餐这类短链路产品
- 业务诉求：部分商户资金链紧张，不能在下单时立即付款，希望申请延期支付

这里有一个非常关键的面试动作：**先把业务假设说清楚**。  
如果面试官没有补充细节，我会主动说明：

“这里我先做一个合理假设，延期支付指的是商户向平台下单时不立即付款，而是平台基于商户授信先放单、先占额度，等履约完成或到约定账期后再收款。平台可能需要继续对上游供应商按正常周期结算，因此平台承担一部分信用风险和资金协同压力。”

这句话很重要，因为它把题目的本质从“加一个支付方式”改成了“设计一个 B 端信用交易系统”。

另外也要强调：  
下面的内容是**基于通用互联网交易平台、OTA、本地生活和账期支付系统的架构抽象**，用于学习和系统设计训练，不代表任何公司内部真实实现。

## 知识图谱

这个主题建议按四层来理解。

### 1. 业务层

- 为什么商户要延期支付
- 平台为什么愿意给延期支付
- 风险和收益分别是什么
- 一笔短途旅游订单从下单到履约完成，哪些时点会影响“应不应该收钱”

### 2. 核心系统层

- 商户准入和授信
- 下单校验和额度占用
- 履约确认与应收确认
- 周期出账与还款
- 逾期冻结、催收和降额
- 平台与供应商结算协同

### 3. 分布式工程层

- 额度扣减为什么要强一致
- 订单状态变化和账单变化为什么通常只能做最终一致
- 为什么必须做幂等、补偿、对账、重放
- 为什么取消、退款、改期、部分退款会让系统复杂度明显上升

### 4. 面试表达层

- 如何先澄清边界，再给出高层方案
- 如何挑 1 到 2 个关键点深挖，而不是把所有组件都点一遍
- 如何讲容量估算和 trade-off
- 如何回答“为什么不用全局事务”“为什么不每笔都人工审批”“逾期怎么办”这类追问

## 课程安排

1. `01_需求澄清、业务边界与核心矛盾.md`
   - 先把业务假设讲清楚
   - 建立参与方、主链路和系统边界
   - 识别这道题真正难的地方

2. `02_授信、额度账户、账单与状态机.md`
   - 这是系统主体
   - 重点讲额度怎么管、订单怎么占额、何时出账、何时释放和冲减

3. `03_一致性、风控、逾期与对账.md`
   - 这是最有分布式系统味道的一课
   - 重点讲强一致与最终一致怎么分层，风控如何前中后台联动

4. `04_面试表达、容量估算与Trade_off.md`
   - 把前三课压缩成面试答案
   - 重点讲怎么表达、怎么算规模、怎么说设计取舍

## 推荐学习方式

- 第 1 课一定要认真看，因为很多候选人是从边界没讲清楚开始丢分的
- 第 2 课要重点理解“额度三态 + 订单状态 + 账单状态”这三套状态不要混
- 第 3 课建议反复看，因为系统成熟度通常就体现在异常路径和逾期处理上
- 第 4 课主要用来练表达，尽量把答案讲成一段有节奏的话，而不是罗列术语

## 学完后你应该能回答的问题

- 为什么延期支付本质上是授信系统，而不是普通支付方式扩展
- 为什么下单时通常是“占额”，而不是立刻形成最终应收
- 为什么短途旅游场景里，履约确认、取消、改期、退款会直接影响账单生成
- 为什么额度系统偏强一致，而账单和对账链路偏最终一致
- 为什么平台要把风控拆成准入、交易中、贷后三个阶段
- 为什么面试里要主动讲供应商结算策略，否则答案会显得过于理想化
