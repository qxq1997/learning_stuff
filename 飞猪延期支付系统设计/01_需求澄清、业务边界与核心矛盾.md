# 飞猪延期支付系统设计 - 第 1 课：需求澄清、业务边界与核心矛盾

## 学习目标（本节结束后你能做到什么）

1. 能先把“延期支付”这道题讲成人话，解释它到底在解决什么问题。
2. 能主动澄清业务假设，避免把题答偏成“新增一种支付方式”。
3. 能看出这道题的核心参与方、主链路和关键冲突。
4. 能说明为什么这道题本质上更像“B 端信用交易系统”，而不是单点支付模块。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

系统设计题里最怕的不是不会，而是**一开始就把问题定义错了**。  
这道题表面上叫“延期支付”，很多人会本能地往支付中心、支付方式枚举、支付状态回调上想。但如果你真这么答，面试通常很快就会卡住，因为你会发现：商户不是支付链路里的一次性付款用户，而是需要被平台长期管理的信用主体。

所以这道题，第一步不是画架构图，而是澄清场景。

我会先这样说：

“我先确认一下业务边界。这里我理解的是一个服务 B 端商户的短途旅游交易平台，商户在平台上采购或代订日内游、小团游、门票套餐等产品。正常情况下商户下单后要立即付款，但现在部分商户由于现金流压力，希望申请账期，也就是先下单锁定资源，后续在约定时间内再统一付款。平台需要决定是否允许这类商户使用延期支付，以及如何控制风险、生成账单和处理逾期。”

这段话做了三件事。

第一，它把“延期支付”解释成了**先交易、后付款**。  
第二，它说明了这不是 C 端白条，而是 B 端商户账期。  
第三，它点出了后面系统必然会出现的三个能力：授信、账单、逾期。

### 1. 先把参与方讲清楚

这道题至少有五类核心角色：

- `商户`：在平台上采购旅游产品，申请延期支付的主体
- `平台`：飞猪场景下的交易撮合和信用管理方
- `供应商`：景区、地接社、车队、活动组织方等履约资源提供者
- `游客/终端消费者`：最终实际出行的人，虽然不直接申请账期，但他的取消、改期、退款会影响应收
- `风控/财务/运营`：负责授信审批、账务核对、催收和异常干预的人

一旦参与方讲清楚，面试官就会知道你没有把它当作孤立技术模块，而是已经把资金关系放进了业务上下文。

### 2. 这道题真正解决什么问题

一句话概括：

**平台希望通过“给商户账期”来提升交易转化和商户粘性，但又不能因此失去对资金风险、履约风险和账务风险的控制。**

这句话背后其实有三重矛盾。

第一重矛盾是**成交率和坏账风险的矛盾**。  
不给账期，商户可能因为现金流不足放弃下单，平台损失交易。给了账期，又可能出现逾期甚至坏账。

第二重矛盾是**供应商履约和商户回款节奏的矛盾**。  
供应商通常不愿意无限等钱。也就是说，平台可能要对商户放账期，但仍然需要按较短周期对供应商结算。这个时候平台要么垫资，要么和供应商协商账期。

第三重矛盾是**订单实时性和账务准确性的矛盾**。  
旅游订单不是静态的。它会取消、改期、部分退款、临时加项、售后赔付。你如果下单时就把整笔应收认死，后面会非常难修。

### 3. 为什么短途旅游场景会让问题更微妙

如果这是标准电商，很多人会直接套“下单占额，发货出账”的模型。  
但短途旅游有几个特殊性。

第一，**履约时间近**。  
很多产品是日内、次日或周末出行，交易和履约时间相隔不长，这意味着账单确认可能更依赖“核销/出游完成”这类事件，而不是传统发货签收。

第二，**取消和改期频繁**。  
天气、人数不足、景区临时调整、用户计划变更，都可能导致取消和改期。一个订单从下单成功到最终确认应收，中间会经历多个状态。

第三，**上游资源具备强时效和强库存属性**。  
商户拿到订单确认，并不意味着平台可以无限期等钱。很多资源位、车位、场次、门票都是即时锁定的，平台要非常谨慎地控制哪些商户可以享受账期。

第四，**客单价未必极高，但订单频次可能不低**。  
这意味着你不能每单都走人工审核，否则业务根本跑不起来；但你也不能完全无脑放单，否则风险暴露会很快。

### 4. 先明确系统边界，避免答散

这道题我会主动划清系统边界。

`本题重点包含的范围`：

- 商户授信申请和开通
- 下单时是否允许延期支付的判断
- 延期支付订单的额度冻结与释放
- 履约确认后的应收生成
- 周期账单、还款、逾期和催收
- 风控、对账和异常补偿

`本题可简化处理的范围`：

- 不展开第三方支付渠道内部细节
- 不展开复杂会计准则，只讨论工程化账务抽象
- 不展开供应商 ERP 内部实现，只考虑平台与其结算接口

面试里能主动划边界很加分，因为这说明你知道系统设计不是“把所有公司系统都介绍一遍”。

### 5. 主链路应该怎么理解

你可以先把整个流程压成六步：

1. 商户申请开通延期支付
2. 风控评估后授予额度和账期
3. 商户下单时校验是否可用并冻结额度
4. 订单履约完成或过关键售后窗口后，确认应收并进入账单
5. 到期后商户统一还款或自动扣款
6. 逾期则冻结权限、催收、降额甚至关停

用图来表示更直观：

```mermaid
flowchart LR
    A["商户申请账期"] --> B["风控授信"]
    B --> C["下单校验"]
    C --> D["冻结额度"]
    D --> E["履约完成/过退款窗口"]
    E --> F["确认应收并出账"]
    F --> G["到期还款"]
    G --> H["逾期处理/催收/降额"]
```

注意这里有一个特别重要的工程判断：  
**下单成功不等于立刻形成最终应收。**

为什么？因为旅游订单可能取消或变更。  
所以更稳妥的做法通常是：

- 下单时先占额度
- 履约完成或过关键可退窗口后再确认应收
- 如果取消或退款，就释放或冲减额度/应收

这就是后面状态机设计的基础。

### 6. 这道题为什么不是“给支付中心加个字段”

很多候选人会说：“支付方式新增一个 `DEFERRED_PAY` 枚举值不就好了？”  
这其实只解决了最表面的一层。

真正系统里，延期支付至少会引入这些能力：

- `商户信用账户`：记录总额度、冻结额度、已用额度、逾期状态
- `准入与审批`：不是所有商户都能开通
- `订单占额引擎`：并发下不能超额
- `账单系统`：要把多笔订单聚合成待还账单
- `贷后管理`：提醒、扣款、逾期、催收、降额
- `财务对账`：商户欠款、平台实收、供应商应付必须核对

所以更成熟的说法应该是：

**延期支付不是支付中心的一个配置项，而是一套围绕 B 端信用交易构建的子系统。**

### 7. 面试里你应该主动讲的核心矛盾

如果我要在前两分钟就建立深度，我会直接把题目总结成下面这句：

“这道题的难点不在支付接口本身，而在于如何让商户先下单后付款，同时保证额度不超发、账单可追踪、逾期可处置、供应商可结算，以及订单取消退款后整个账务链路还能收得回来。”

这句话为什么有效？  
因为它一次性把五个关键点点出来了：

- 额度
- 账单
- 逾期
- 结算
- 逆向交易

如果面试官听到这里，通常会愿意继续往下深挖，而不是怀疑你是不是只会讲常规电商支付。

## 小结（3-5 条关键点）

1. 这道题本质上是 B 端信用交易系统，不是普通支付方式扩展。
2. 回答前一定要先明确业务假设：商户先下单锁资源，后续按账期付款。
3. 核心参与方至少包括商户、平台、供应商、游客以及风控财务运营。
4. 真正难点是成交转化、资金风险、履约协同和账务一致性的平衡。
5. 下单时更合理的动作通常是“占额”，而不是直接形成最终应收。
