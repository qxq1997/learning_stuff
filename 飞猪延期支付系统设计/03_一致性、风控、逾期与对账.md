# 飞猪延期支付系统设计 - 第 3 课：一致性、风控、逾期与对账

## 学习目标（本节结束后你能做到什么）

1. 能区分这类系统里哪些地方要强一致，哪些地方适合最终一致。
2. 能讲清延期支付链路中最重要的异常场景和补偿策略。
3. 能从准入、交易中、贷后三个阶段说明风控如何介入。
4. 能解释为什么逾期、对账和补偿是这类系统成熟度的关键体现。

## 内容讲解（核心概念，用类比、例子、图示说清楚）

很多系统设计题，讲到高层架构就差不多了。  
但延期支付不是这种题。  
你如果只讲“下单占额、履约出账、到期还款”，面试官通常会马上追问：

- 并发下单会不会超额？
- 订单取消了，额度怎么释放？
- 出账消息丢了怎么办？
- 商户一直不还钱怎么办？
- 供应商已经结算了，但商户后面退款怎么办？

这些问题一追，就会把候选人区分得很明显。  
因为真正成熟的系统，强项不在“正常流程”，而在“脏现实里的稳定性”。

### 1. 先定一致性原则，不要上来就说全链路强事务

我会先给出一个原则：

**额度账户要偏强一致，订单、账单、通知、催收链路可以接受最终一致，但必须可追踪、可补偿、可对账。**

为什么？

因为额度是风控底线。  
如果额度错了，系统可能在一秒钟内放出大量风险订单。  
而账单通知晚几秒、催收任务晚几分钟，一般不会立刻造成灾难。

所以系统里不同链路的一致性要求并不一样。

`需要尽量强一致的地方`：

- 下单占额
- 释放额度
- 冻结额度转已用额度
- 还款成功后冲减已用和逾期金额

`适合最终一致的地方`：

- 账单汇总
- 还款提醒通知
- 催收任务生成
- BI 报表
- 风险画像异步更新

这时候你就能顺势说出一个成熟观点：  
**不要为了追求“全都一致”去强行上分布式全局事务，工程上更稳的是局部强一致 + 事件驱动 + 补偿 + 对账。**

### 2. 下单占额怎么保证正确

额度是账户类数据，它最怕三件事：

- 并发超扣
- 重复扣减
- 失败后没回滚

所以我通常会这样设计额度服务：

1. 商户信用账户单独建模
2. 占额接口要求带幂等键，例如 `merchant_id + order_id + action_type`
3. 更新账户时用数据库行锁或乐观锁 CAS 控制版本
4. 占额成功后写入额度流水
5. 账户变更和流水写入放在同一个本地事务里

这样做的好处是，即使调用方重试了三次，额度服务也能识别这是同一笔占额，而不是重复冻结。

### 3. 订单状态变化为什么要靠事件驱动

旅游订单的后续状态很多：

- 确认成功
- 商户取消
- 游客取消
- 改期
- 部分退款
- 全额退款
- 已核销
- 售后赔付

如果额度系统去轮询订单表判断“现在该不该释放”，会非常脆弱。  
更稳的办法是：订单服务和履约服务在关键状态变化时发布领域事件，账单中心和额度服务订阅这些事件。

例如：

- `ORDER_CANCELLED` -> 释放冻结额度
- `ORDER_CONSUMED` -> 冻结转已用
- `ORDER_REFUNDED_PARTIAL` -> 冲减应收
- `ORDER_RESCHEDULED` -> 可能重新计算应收窗口

这里通常可以采用：

- 业务库本地事务提交订单状态
- 同时写 Outbox 事件表
- 后台任务可靠投递 MQ
- 下游按事件幂等处理

这样比“先发消息再改库”要稳得多，因为你不会出现消息发了但主状态没落库的尴尬。

### 4. 为什么幂等和补偿必须设计成系统能力

交易系统里，重复和乱序不是偶发，而是常态。  
比如一个订单取消事件可能因为网络抖动被重投两次，一个退款事件可能比核销事件先到，一个下游服务处理到一半超时后又被重试。

所以系统需要天然支持：

- `幂等`
  - 同一事件重复到来，处理结果不变

- `顺序隔离`
  - 不能假设所有事件天然有序
  - 必要时按订单维度做状态版本校验

- `补偿`
  - 处理失败后可重试
  - 错误处理后能人工触发修复任务

- `审计`
  - 每一次账户金额变化都有流水
  - 每一次账单状态变化都有记录

这是为什么资金相关系统通常偏好“追加流水”，而不是直接更新一堆聚合字段。  
聚合字段可以有，但它们最好是当前快照；真正的真相来源应该是可审计流水。

### 5. 风控不能只放在准入阶段

很多人说风控时，只提“商户申请时审批一下”。  
这远远不够。  
延期支付至少需要三层风控。

#### 5.1 准入风控

目标是决定“这个商户能不能开通，给多少额度，给多长账期”。

可参考的输入包括：

- 商户资质、经营年限、保证金情况
- 历史交易额、退款率、投诉率、履约率
- 历史逾期记录
- 产品品类风险
- 城市和供应商稳定性

#### 5.2 交易中风控

目标是决定“这单能不能放、是否要降额、是否要人工复核”。

典型规则包括：

- 单笔金额是否异常高
- 当日累计占额是否快速飙升
- 某个高风险产品是否不允许走账期
- 商户是否存在近期退款激增
- 是否命中黑名单供应商/线路

#### 5.3 贷后风控

目标是尽早发现“这个商户可能还不上钱”。

可观测指标包括：

- 即将到期金额
- 最近 7 天新增应收增长率
- 最近 7 天退款冲减比率
- 连续提醒未读
- 自动扣款失败次数
- 已出现小额逾期但仍继续下单

这三层风控一起形成闭环，才是真实世界里能跑的系统。

### 6. 逾期处置要有明确节奏

逾期处理不能只有一句“冻结商户”。  
更成熟的设计通常会分阶段。

1. `到期前提醒`
   - D-3、D-1 站内信、短信、钉钉或客户经理通知

2. `到期日处理`
   - 自动扣款
   - 扣款失败则立即标记风险

3. `轻度逾期`
   - 冻结新下单能力
   - 降低额度
   - 通知商户经理跟进

4. `中度逾期`
   - 进入人工催收
   - 补充要求保证金、担保或提前回款

5. `重度逾期`
   - 关闭账期资格
   - 拉黑风控名单
   - 法务或更重流程介入

这套节奏的价值，在于它告诉面试官：  
你知道逾期不是一个布尔值，而是一条贷后运营链路。

### 7. 对账为什么是最后的保险丝

任何声称“我们设计得很好，所以不需要对账”的系统，基本都不可信。  
因为真实系统里一定会有边角异常。

延期支付至少要做三类对账。

1. `订单对账`
   - 订单系统状态 vs 订单信用流水状态
   - 看是否有订单已取消但额度没释放

2. `账单对账`
   - 应收明细汇总 vs 商户账单金额
   - 看是否有漏出账、重复出账、冲减错误

3. `资金对账`
   - 商户还款流水 vs 实际渠道扣款结果
   - 看是否有扣款成功但账没销、账已销但扣款失败

对账的目标不是“证明系统没问题”，而是尽快把问题暴露出来，并形成补单、冲正、重试、人工处理的闭环。

### 8. 监控应该盯什么

这类系统除了常规技术指标，还要盯业务指标。

`技术指标`：

- 占额接口成功率、延迟、超时率
- MQ 堆积、消费失败数、死信数
- Outbox 未投递事件数
- 对账任务失败率

`业务指标`：

- 商户延期支付使用率
- 额度命中率、额度耗尽率
- 当日新增应收金额
- 账单逾期率、逾期金额
- 自动扣款成功率
- 取消退款释放额度时延

真正成熟的答案，技术监控和业务监控都会讲到。

## 小结（3-5 条关键点）

1. 延期支付系统里，额度账户要偏强一致，账单和通知链路更适合最终一致。
2. 订单状态变化更适合通过领域事件驱动额度释放、应收确认和账单更新。
3. 幂等、补偿、审计流水和对账不是补充项，而是核心能力。
4. 风控至少分为准入、交易中和贷后三层，不能只做一次开通审批。
5. 逾期要有阶段化处置策略，对账则是最后的保险丝。
