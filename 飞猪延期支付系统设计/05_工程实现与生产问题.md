# 飞猪延期支付系统设计 - 第 5 课：工程实现、技术栈与生产问题

## 学习目标（本节结束后你能做到什么）

1. 能在面试里说出一套自洽的技术选型，并解释每个选型为什么这么定。
2. 能把"下单占额、事件驱动、出账、代扣"这几条链路落到伪代码级别。
3. 能讲出这类系统在生产上真正会爆的故障形态，而不是只讲正常流程。
4. 能把每个生产问题对应到一条具体的工程兜底手段（幂等 / Outbox / 对账 / 风控）。

## 内容讲解

前四课偏业务和面试表达。这一课换一个视角：**假设你真的要带团队把这个系统做出来，会怎么落地，上线后哪里会半夜被叫起来。** 面试官一旦追问"你们线上出过什么问题"，能答好这一节的人会立刻和背答案的人拉开差距。

### 1. 模块划分（落到服务粒度）

骨架仍是五段闭环：**授信 → 占额 → 履约确认 → 出账 → 还款/逾期**。落到服务这么切，粒度刚好，别拆成微服务目录树。

| 模块 | 职责 | 一致性要求 |
| --- | --- | --- |
| 商户中心 | 商户资质、合同、保证金、经营概览 | 普通 |
| 授信/风控服务 | 准入审批、授额、账期配置、三层风控规则 | 普通，但规则版本要可追溯 |
| 额度账户服务 | 总额/可用/冻结/已用/逾期五金额，占额/释放/转应收 | **强一致（核心）** |
| 订单服务 | 下单、调用占额、发履约领域事件 | 本地强一致 + Outbox |
| 账单中心 | 应收明细聚合成周期账单 | 最终一致 |
| 还款/扣款服务 | 主动还款、到期代扣、对接支付渠道 | 资金强一致 |
| 催收/通知服务 | 提醒、逾期分级处置、外呼任务派发 | 最终一致 |
| 对账/补偿平台 | 订单/账单/资金三类对账 + 差异补单 | 离线批处理 |

一句话原则：**额度账户是全系统唯一需要拼命保强一致的地方，其余链路靠事件驱动 + 对账兜底。**

### 2. 技术栈与选型理由

不要只报一串技术名词，每个都要能说"为什么"。

- **语言/框架**：Java + Spring Boot / Spring Cloud，或公司内 RPC 框架（阿里系 HSF/Dubbo）。
  - 为什么：资金交易团队基本是 Java 栈，生态成熟、事务和并发工具齐全、招人好招。

- **核心存储：MySQL（InnoDB）**
  - 额度账户、账单、各类流水都放关系库。
  - 为什么不用 NoSQL：额度扣减需要行级锁 + 事务 + 唯一约束做幂等，这是关系库的强项，NoSQL 反而要自己造一致性轮子。
  - 分库分表用 **ShardingSphere**，按 `merchant_id` 分片。
  - 为什么按商户分片：占额、出账、对账几乎都是商户维度操作，按商户分片能让单商户的操作落在同一个库，避免跨库事务。

- **缓存：Redis**
  - 只做查询加速、热点商户限流、分布式锁。
  - **关键纪律：额度的真相来源永远是 MySQL，不要用 Redis 做扣减真相。** Redis 宕机或主从切换丢数据时，绝不能让额度算错。

- **消息：RocketMQ**（或 Kafka）
  - 为什么 RocketMQ：阿里系首选，原生支持事务消息和顺序消息，正好对应"本地事务 + 可靠发事件"和"按订单维度有序消费"两个诉求。
  - 履约事件、出账触发、异步通知都走 MQ。

- **可靠事件投递：Outbox 表**
  - 业务库一张 `outbox_event`，业务变更和事件写入同一个本地事务，再由后台任务投递到 MQ。
  - 也可以直接用 RocketMQ 事务消息省掉 Outbox，但 Outbox 更通用、可审计、不绑定 MQ 实现。

- **定时/批处理：XXL-Job / ElasticJob**
  - 跑周期出账、三类对账、到期代扣、逾期扫描、冻结泄漏扫描。

- **分布式锁：Redisson**
  - 作为账户级串行化的兜底，但优先用 DB 乐观锁（CAS），锁只在极端热点时叠加。

- **可观测：Prometheus + Grafana + 公司数仓/BI**
  - 技术指标走 Prometheus，业务指标（逾期率、额度耗尽率）走数仓离线算。

### 3. 关键链路的落地实现

#### 3.1 下单占额：强一致 + 幂等的核心

```
订单服务 -> 额度账户服务.freeze(merchantId, orderId, amount, idemKey)
  idemKey = merchantId + ":" + orderId + ":FREEZE"

额度账户服务内部（一个本地事务）:
  -- 1. 幂等：先查流水，存在则直接返回上次结果
  SELECT * FROM order_credit_record WHERE idem_key = :idemKey;
  if exists: return 上次结果;   -- 重试三次也只冻结一次

  -- 2. CAS 扣减，WHERE 同时承担乐观锁 + 余额校验
  UPDATE merchant_credit_account
     SET available = available - :amount,
         frozen    = frozen + :amount,
         version   = version + 1
   WHERE merchant_id = :merchantId
     AND version = :oldVersion
     AND available >= :amount;

  affected = 0 ?
     -> 版本冲突：重试（重新读 version 再来）
     -> 余额不足：返回失败

  -- 3. 同事务写流水
  INSERT INTO order_credit_record(idem_key, merchant_id, order_id, action='FREEZE', amount, ...);
  COMMIT;
```

精髓只有一句：**那条 UPDATE 的 WHERE 子句同时做了乐观锁和余额校验，一条 SQL 就堵死了并发超扣。** 两个并发请求只有一个能命中 `version`，另一个 affected=0，重试时会看到新余额。

#### 3.2 履约状态变化：事件驱动

订单/履约服务在本地事务里改状态 + 写 outbox，再投递 MQ。下游订阅：

- `ORDER_CANCELLED` → 释放冻结额度（frozen 减、available 加）
- `ORDER_CONSUMED` → 冻结转已用（frozen 减、used 加）
- `ORDER_REFUNDED_PARTIAL` → 冲减应收
- `ORDER_RESCHEDULED` → 重算应收窗口

下游消费两道防线：
- **幂等**：每条事件带唯一 `event_id`，消费前查去重表，处理过就跳过。
- **状态版本校验**：防乱序。比如退款事件比核销事件先到，就靠订单维度的状态版本号拒绝"从后面的状态往前跳"。

#### 3.3 周期出账

XXL-Job 按账期扫 `receivable_item`，按商户 + 周期聚合成 `receivable_bill`，状态 `DRAFT → ISSUED`。出账动作本身也要幂等（同一商户同一周期只能出一张账单，用唯一索引 `merchant_id + period` 兜底）。

#### 3.4 到期代扣

定时任务拉到期账单 → 调支付渠道代扣 → 成功则冲减 used、账单转 PAID；失败则标风险、进逾期流程。**代扣结果不确定（超时）时绝不能凭一次返回销账**，必须发起渠道侧状态查询补偿。

### 4. 生产上真正会遇到的问题（重点）

这是面试加分、也是线上最疼的部分。每个问题都配一条兜底手段。

1. **额度账户行锁热点**
   - 现象：大商户大促时同一行被狂更新，DB 连接被占满，占额 TPS 上不去甚至雪崩。
   - 兜底：账户拆桶（一个商户拆成 N 个子额度账户，请求散列到子账户）；或 Redis 预扣 + 异步落库对平（复杂度上升，要权衡）。

2. **消息丢失 / 重复导致额度漂移**
   - 现象：出账消息丢了商户账没出；取消事件重投两次，额度被多释放。
   - 兜底：Outbox 保证不丢 + 消费幂等保证不重 + **每日对账**兜最后。对账不是可选项，是保险丝。

3. **冻结额度泄漏（占额没释放）**
   - 现象：订单卡在中间态（下单成功但后续 MQ 消费失败），冻结额度一直占着，商户可用额度越用越少，却查不到对应欠款，客诉。
   - 兜底：定时扫描超时未确认的 `order_credit_record`，结合订单状态对账，自动补释放或转人工。

4. **退款慢于结算造成的信用风险**
   - 现象：平台已按账期把钱结算给上游供应商，商户随后发起大额退款，平台等于垫了钱。
   - 兜底：这是**信用风险，不是纯技术问题**。靠保证金、贷后风控降额、账期设计（结算节奏晚于退改窗口）来兜。面试讲到这一层会显得很真实。

5. **代扣渠道结果不确定**
   - 现象：代扣请求超时，到底扣没扣不知道。直接当失败可能重复扣，直接当成功可能漏扣。
   - 兜底：渠道对账 + 主动状态查询 + 幂等代扣单号，绝不凭单次返回销账。

6. **对账差异爆量没人处理**
   - 现象：差异表每天堆积上千条，运营处理不过来，对账形同虚设。
   - 兜底：差异分级（金额、类型）+ 高频可自动补单的规则化 + 人工兜底工作台，把人力只用在真正异常上。

7. **改期 / 部分退款引发的账单冲减错乱**
   - 现象：已出账账单要冲减，涉及跨周期调整，出现负账单、重复冲减、冲减到已结清账单。
   - 兜底：冲减也走流水 + 幂等，跨周期调整生成"调整项"而不是直接改历史账单，保证账单可重算、可审计。

### 5. 怎么把这节塞进面试回答

面试时不需要全讲，但被追问"线上出过什么问题"时，挑 2 个讲透最有效：
- 讲 **冻结额度泄漏** + 定时扫描补偿（体现你想过中间态）。
- 讲 **代扣结果不确定** + 渠道对账（体现你懂资金系统的脏现实）。

这两个都不是背书能背出来的，一听就知道是不是真做过。

## 小结（3-5 条关键点）

1. 额度账户是全系统唯一死保强一致的地方，MySQL 行级 CAS + 流水 + 幂等是它的三件套。
2. 技术选型每一项都要能说"为什么"，尤其要强调 Redis 不做额度真相、按商户分片避免跨库事务。
3. 关键链路（占额、事件、出账、代扣）都要落到幂等键和状态版本这一层，否则只是 PPT。
4. 生产故障的核心形态是：热点锁、消息丢重、冻结泄漏、退款信用风险、代扣不确定、对账积压、冲减错乱。
5. 每个生产问题都要能对应到一条具体兜底（Outbox / 幂等 / 对账 / 扫描补偿 / 风控保证金）。
